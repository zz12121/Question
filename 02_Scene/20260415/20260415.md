# 业务场景题 - 第7天（181-210）

## 高并发场景（6题）

181. 你负责的电商系统要做"双十一"当晚的"实时大屏"：展示全站 GMV、订单量、各品类 TOP10 实时排行榜、各省份热力图。数据源来自订单服务的 Kafka 消息流。大屏要求"亚秒级延迟"（< 1 秒），高峰期每秒可能有 50 万笔订单。计算引擎怎么选？（Flink / Spark Streaming / Redis + Lua / ClickHouse）维度聚合怎么做？

182. 你的外卖系统要做"骑手实时位置追踪"功能：每个骑手每 2 秒上报一次 GPS 位置，高峰期 10 万骑手同时在线。前端需要展示骑手在地图上的实时位置和预计到达时间（ETA）。位置数据怎么存储？历史轨迹怎么压缩？（Douglas-Peucker 算法）ETA 预算怎么实现？10 万 TCP 长连接的服务端怎么撑住？

183. 你的在线教育平台要做"万人同时在线考试"功能：所有考生在同一时间开始答题、提交试卷时有"最后 5 分钟集中提交"的瞬时峰值。如何保证考试过程中页面不白屏、交卷请求不丢失？试卷答案实时自动保存怎么实现？防止作弊（切屏检测、人脸识别、题目乱序）的技术方案怎么设计？

184. 你的社交平台要做"用户动态 Feed 流"功能：每个用户的 Feed 流需要展示"关注的人 + 系统推荐"混合内容，且要支持"拉新"（新用户冷启动）和"拉活"（沉睡用户召回）。推模式（写扩散）和拉模式（读扩散）怎么选？推拉结合的架构怎么设计？推荐内容的"插入位"怎么确定才能不影响核心社交内容的体验？

185. 你的系统要支持"跨平台同步"：同一个用户在手机 App、Web 端、PC 客户端同时登录，任何一端的操作（如修改备注、标记已读、删除消息）需要在其他端实时同步。冲突怎么处理？（最后写入获胜 vs CRDT）离线期间的操作怎么合并？消息去重怎么保证？WebSocket / MQTT / SSE 的技术选型？

186. 你的系统在做"全站搜索"功能，当前用的是 MySQL LIKE 模糊查询，数据量到了 2000 万条后查询已经到了 3-5 秒。需要引入 Elasticsearch，但面临几个问题：历史 2000 万条数据怎么全量同步到 ES？增量同步怎么保证实时性（binlog → Canal → ES）？ES 的索引分片数和副本数怎么规划？搜索结果的排序怎么兼顾"相关性"和"时效性"？

## 数据一致性场景（5题）

187. 你的支付系统在执行"退款"操作时涉及多个步骤：修改订单状态、扣减商户余额、增加用户余额、通知下游（物流系统回仓、营销系统回退优惠券）。其中"增加用户余额"调用了第三方银行接口，银行返回"处理中"（异步结果）。这个场景下怎么保证各步骤最终一致？补偿机制怎么设计？重试次数和重试间隔怎么定？

188. 你的系统从"单库单表"迁移到了"分库分表"（ShardingSphere），但发现了一个棘手问题：历史数据中有一些"跨库关联查询"（如订单表按 user_id 分片，商品表按 shop_id 分片，需要 JOIN 查询）。分库分表后的"跨库 JOIN"怎么解决？（数据冗余 / ES 中间层 / 应用层组装）数据迁移期间新旧两套查询逻辑怎么并行？

189. 你的系统用了 Redis 做分布式锁，但发现"锁超时释放后，业务还没执行完"导致并发问题。于是用了 Redisson 的看门狗续期机制，但最近又出现了"看门狗线程本身 OOM 了"的极端情况。有没有比 Redis 分布式锁更可靠的方案？（ZooKeeper / etcd 的临时顺序节点方案）各自的优缺点和适用场景？锁的"安全等级"怎么评估？

190. 你的系统有"定时批量对账"需求：每天凌晨把 MySQL 的订单数据、支付渠道的对账文件、Redis 中的缓存数据进行三方比对。但 MySQL 中的数据可能在这期间被修改（夜间有批处理任务），对账结果不准确。如何设计一个"一致性快照"级别的对账方案？`START TRANSACTION WITH CONSISTENT SNAPSHOT` 能解决吗？大批量数据的逐行比对性能怎么优化？

191. 你的系统在做"数据归档"：把 6 个月前的订单数据从线上 MySQL 迁移到归档库（Hive / ClickHouse）。归档过程中需要保证：线上业务不受影响、归档数据不丢不重、归档完成后能从归档库查询历史数据。归档策略怎么定？（按月 / 按量 / 按表）归档任务失败了怎么恢复断点续传？归档后的查询接口怎么统一？

## 系统稳定性场景（5题）

192. 你的系统刚上线了一个新功能，上线 30 分钟后监控系统开始报警：P99 延迟从 200ms 飙升到 5 秒，CPU 使用率 90%+，但没有任何错误日志。需要你在 10 分钟内定位并回滚。排查思路是什么？（先看是什么变化导致的 → 看监控指标时间线 → Arthas 在线诊断 → 灰度回滚 / 配置降级）你有哪些"快速止血"的手段？

193. 你的服务部署在 K8s 上，使用了 HPA（Horizontal Pod Autoscaler）根据 CPU 自动扩缩容。但有一天流量突增时 HPA 扩容太慢（从 5 个 Pod 扩到 50 个花了 5 分钟），导致大量请求超时。HPA 的扩容决策延迟来自哪些环节？（指标采集延迟 + 冷启动 + 镜像拉取 + 应用初始化）如何优化？（Pod 预热 / PDB 优先级 / 自定义指标 / KEDA）

194. 你的服务依赖了 3 个下游微服务（A、B、C），其中 C 服务是"非核心路径"（用于记录操作日志和推送通知）。某天 C 服务挂了导致你的服务整体 5xx 率飙升。问题出在虽然业务逻辑上 C 不是必须的，但代码层面是同步调用且没有做好异常隔离。如何改造？（异步化 + 熔断 + 降级 + 超时控制）如何用"架构级别"而非"代码级别"解决这个问题？

195. 你的系统有一个"定时任务调度中心"（类似 XXL-JOB），每天凌晨跑数百个批处理任务。某天一个任务的 SQL 写错导致全表扫描，跑了 3 小时还没完，把数据库 CPU 打满了，影响了线上业务查询。如何从架构上防止"批处理任务影响在线业务"？（读写分离 / 限流 / DB 资源隔离 / 任务超时熔断 / 慢 SQL 自动 kill）XXL-JOB 的"路由策略"和"失败重试"机制怎么合理配置？

196. 你的系统刚做了 JDK 从 8 升级到 17 的迁移，上线后发现 Young GC 频率明显增高了（从每 30 秒一次变成每 5 秒一次），但每次 GC 时间很短（10ms 左右）。这是什么原因？G1 GC 在 JDK 17 中的默认 Region 大小和 JDK 8 有什么不同？JDK 17 的 `ZGC` 是否可以作为替代方案？从 G1 切换到 ZGC 需要改哪些 JVM 参数？

## 业务设计场景（5题）

197. 设计一个"抽奖系统"，支持：多种抽奖玩法（大转盘 / 刮刮卡 / 九宫格 / 盲盒）、奖品池管理（实物奖品 / 虚拟奖品 / 优惠券 / 空奖）、概率动态调整（运营可以随时改中奖概率）、防薅羊毛（一人只能抽 N 次、防止内幕中奖）、高并发下库存扣减不超卖。奖品库存存在哪里？概率计算在 Redis 还是应用层做？

198. 设计一个"运费计算引擎"：电商平台的运费规则非常复杂（按重量 / 按体积 / 按件数 / 按距离 / 满额包邮 / 偏远地区加价 / 多个包裹合并），且规则会频繁变更（运营每天可能改好几次）。运费计算必须是实时的（不能让用户等 3 秒才能看到运费）。规则引擎怎么选？（Drools / LiteFlow / 自研）规则热更新怎么做？计算结果怎么缓存？

199. 设计一个"实时风控系统"：检测用户登录的异常行为（异地登录、设备指纹异常、登录频率异常）、交易行为异常（短时间内大额转账、频繁小额试探）、账号被盗风险。要求：毫秒级决策（不能让正常用户感知到延迟）、规则引擎灵活配置（风控团队可以自己加规则）、误判率低（不能误杀正常用户）。流式计算怎么选？（Flink / Spark Streaming）特征计算怎么优化？

200. 设计一个"商品价格管理系统"：电商平台上有数百万 SKU，每个 SKU 可能有多种价格（原价 / 促销价 / 会员价 / 渠道价 / 阶梯价），价格变更需要实时生效到前端展示和下单流程。同时要支持"价格审批流"（运营修改价格后需要主管审核）、"价格历史"（审计溯源）、"价格策略"（满减 / 满折 / 限时特价的自动叠加计算）。价格数据存在哪里？读写频率怎么平衡？

201. 设计一个"分布式任务调度平台"（类似 XXL-JOB / Elastic-Job）：支持可视化配置定时任务、Cron 表达式、任务分片（一个任务并行跑在多个节点上）、失败重试、超时控制、任务依赖（DAG 有向无环图）、任务日志收集与查看。核心架构怎么设计？（Master-Worker / 无中心化 / ZooKeeper 注册中心）任务分片的实现原理是什么？故障转移怎么做？

## 排查优化场景（5题）

202. 你的服务最近频繁出现"OOM（OutOfMemoryError）：Java heap space"，但重启后过一段时间又复现。heap dump 文件有 4GB，用什么工具分析？（MAT / VisualVM / Arthas heapdump）分析后发现 `ConcurrentHashMap$Node` 对象占了 80%，怎么确认是哪个业务对象导致的？如何在不重启服务的情况下"在线排查"内存泄漏？

203. 你的系统在做"压力测试"时发现：单机 QPS 到 3000 后，响应时间开始急剧上升（P99 从 50ms 飙到 500ms），但 CPU 使用率只有 60%，内存充足，网络带宽也未打满。可能的瓶颈在哪里？（线程上下文切换过多 / 锁竞争 / GC 频率 / 数据库连接池耗尽 / 日志同步写入阻塞）你用什么工具和指标来逐一排查？

204. 你的 Spring Boot 应用启动时间从最初的 15 秒慢慢增长到了 90 秒，开发体验很差。哪些因素会导致 Spring Boot 启动变慢？（Bean 数量膨胀 / 类路径扫描 / JPA Entity 扫描 / 大量 @PostConstruct 初始化 / 数据库连接池初始化 / 健康检查超时）如何定位"哪个 Bean 初始化最慢"？Spring Boot 3 的"Observability 启动追踪"能帮你做什么？

205. 你的微服务系统中，偶尔会出现"请求链路超时"：用户发起的请求经过 API Gateway → Service A → Service B → Service C，整个链路 P99 耗时 2 秒，但单独测每个服务都在 100ms 以内。链路中有哪些"隐藏耗时"？（序列化/反序列化 / 网络抖动 / DNS 解析 / 连接池获取 / 线程池排队 / GC STW）如何用分布式追踪（Zipkin / Jaeger / SkyWalking）来定位耗时最长的环节？

206. 你的系统用了 Kafka 做消息解耦，最近发现某个 Topic 的"消费延迟"（Consumer Lag）一直在增长，从 0 涨到了 100 万条。消费者没有报错，但就是消费不过来。可能的原因有哪些？（消费者处理逻辑变慢 / 消费者数量不够 / 分区分配不均 / 消费者 GC 频繁 / Rebalance 频繁）如何用 Kafka 的 `Consumer Lag` 监控来预警？Rebalance 为什么会导致"消费暂停"几十秒？

## 架构演进场景（4题）

207. 你的系统最初是单体架构，后来拆成了 20+ 个微服务。现在发现微服务数量太多导致新的痛点：一个需求要改 5-6 个服务、服务间依赖关系复杂（A 依赖 B、B 依赖 C、C 又依赖 A 的循环依赖）、新人上手成本高。这种"过度拆分"的问题怎么解决？哪些微服务应该合并回去？怎么评估"合理的微服务粒度"？

208. 你的系统要做"服务网格"（Service Mesh）改造：当前每个微服务都自己集成了 SDK（Spring Cloud LoadBalancer、Resilience4j、Sleuth 等），SDK 版本升级需要所有服务一起发版。引入 Istio 后，sidecar 代理（Envoy）接管了流量管理、熔断降级、可观测性。改造过程中有哪些坑？（sidecar 资源开销 / gRPC 协议兼容 / 多集群服务发现 / 遗留服务渐进式接入）Istio 和 Linkerd 怎么选型？

209. 你的系统要支持"多语言微服务"：Java 做核心业务、Go 做高并发网关、Python 做机器学习推荐服务。不同语言的服务之间怎么通信？（gRPC + Protobuf）服务注册发现怎么做？（Consul / Nacos 多语言 SDK）不同语言的"链路追踪上下文"怎么透传？（OpenTelemetry 统一标准）多语言 CI/CD 流水线怎么统一管理？

210. 你的系统当前部署在国内 3 个机房（北京、上海、广州），现在要扩展到海外（新加坡、美东、法兰克福），服务全球用户。跨洋延迟（中国到美国 RTT 200ms+）怎么解决？（CDN 边缘节点 / 就近部署 / 异步化）数据合规问题（GDPR、数据本地化存储）怎么满足？多区域部署的 CI/CD 和配置管理怎么搞？（K8s Federation / 多集群 GitOps）
