## 先说结论

第三方 API 超时重试要遵循**有限重试 + 指数退避 + 熔断降级**原则——重试3次以内，间隔指数增长（1s→2s→4s），配合熔断器防止雪崩。无限重试会导致**级联故障**（拖垮自身连接池）、**流量放大**（压垮第三方）、**数据不一致**（重复提交），必须避免。

---

## 🔥 核心概念

**1. 解决什么问题**
- 网络抖动、第三方服务瞬时故障导致偶发超时
- 简单重试可能加剧问题，需要智能重试策略

**2. 基本原理/思想**
- **有限重试**：设置最大重试次数，避免无限循环
- **指数退避**：重试间隔逐渐增加，给第三方恢复时间
- **熔断降级**：连续失败时停止重试，直接走降级逻辑

## 🔥 详细展开（原理/实现/使用）

### 一、无限重试的灾难

```
场景：调用支付接口，设置无限重试

┌─────────────────────────────────────────────────────────────┐
│  问题1：级联故障（拖垮自己）                                   │
│  ─────────────────────                                       │
│  重试线程占用连接池 → 连接池耗尽 → 其他请求无法处理              │
│  重试线程堆积 → 内存暴涨 → OOM                                 │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  问题2：流量放大（压垮第三方）                                  │
│  ─────────────────────                                       │
│  1个请求 → 重试100次 = 100倍流量                               │
│  第三方本来只是慢，被重试打挂                                   │
│  雪崩效应：A重试打挂B，B的请求也重试打挂C...                    │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  问题3：数据不一致（重复提交）                                  │
│  ─────────────────────                                       │
│  第一次请求实际已处理，只是响应超时                              │
│  重试导致第二次执行 → 重复扣款/重复下单                          │
│  没有幂等保护，数据错乱                                        │
└─────────────────────────────────────────────────────────────┘
```

### 三、具体实现方案

#### 方案1：Spring Retry（简单场景）

```java
@Configuration
@EnableRetry
public class RetryConfig {
    
    @Bean
    public RetryTemplate retryTemplate() {
        // 重试策略：最多3次
        SimpleRetryPolicy retryPolicy = new SimpleRetryPolicy();
        retryPolicy.setMaxAttempts(3);
        
        // 退避策略：指数退避 1s → 2s → 4s
        ExponentialBackOffPolicy backOffPolicy = new ExponentialBackOffPolicy();
        backOffPolicy.setInitialInterval(1000);   // 初始1秒
        backOffPolicy.setMultiplier(2.0);          // 倍数2
        backOffPolicy.setMaxInterval(10000);       // 最大10秒
        
        RetryTemplate template = new RetryTemplate();
        template.setRetryPolicy(retryPolicy);
        template.setBackOffPolicy(backOffPolicy);
        
        return template;
    }
}

@Service
public class PaymentService {
    
    @Autowired
    private RetryTemplate retryTemplate;
    
    @Autowired
    private ThirdPartyPaymentClient paymentClient;
    
    public PaymentResult pay(PaymentRequest request) {
        return retryTemplate.execute(
            context -> {
                // 第几次重试
                int retryCount = context.getRetryCount();
                log.info("支付请求，第{}次尝试", retryCount + 1);
                
                return paymentClient.charge(request);
            },
            context -> {
                // 重试耗尽后的恢复逻辑
                log.error("支付重试{}次后仍失败", context.getRetryCount());
                throw new PaymentException("支付服务暂时不可用，请稍后重试");
            }
        );
    }
}
```

    }
}

@Service
public class PaymentService {
    
    @Autowired
    private ThirdPartyPaymentClient paymentClient;
    
    // 组合：重试 + 熔断 + 超时
    @Retry(name = "paymentRetry")
    @CircuitBreaker(name = "paymentCircuit", fallbackMethod = "payFallback")
    @TimeLimiter(name = "paymentTimeout")
    public CompletableFuture<PaymentResult> payAsync(PaymentRequest request) {
        return CompletableFuture.supplyAsync(() -> {
            return paymentClient.charge(request);
        });
    }
    
    // 降级方法
    private CompletableFuture<PaymentResult> payFallback(PaymentRequest request, Exception ex) {
        log.error("支付降级，原因：{}", ex.getMessage());
        
        // 降级策略：记录到本地消息表，稍后异步重试
        return CompletableFuture.completedFuture(
            PaymentResult.builder()
                .status("PENDING")
                .message("支付请求已接收，稍后处理")
                .build()
        );
    }
}

### 四、幂等性保证（防止重复提交）

```java
@Service
public class IdempotentPaymentService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private PaymentLogRepository paymentLogRepository;
    
    /**
     * 带幂等保护的支付
     */
    public PaymentResult payWithIdempotency(PaymentRequest request) {
        String idempotencyKey = request.getIdempotencyKey();  // 客户端生成的唯一键
        
        // 1. 先查是否处理过（防重）
        String cachedResult = redisTemplate.opsForValue().get("pay:" + idempotencyKey);
        if (cachedResult != null) {
            log.info("检测到重复请求，直接返回缓存结果");
            return JSON.parseObject(cachedResult, PaymentResult.class);
        }
        
        // 2. 加分布式锁（防并发重复）
        Boolean locked = redisTemplate.opsForValue()
            .setIfAbsent("lock:pay:" + idempotencyKey, "1", 30, TimeUnit.SECONDS);
        
        if (!Boolean.TRUE.equals(locked)) {
            throw new ConcurrentException("请求处理中，请勿重复提交");
        }
        
        try {
            // 3. 再查DB（双重检查）
            PaymentLog log = paymentLogRepository.findByIdempotencyKey(idempotencyKey);
            if (log != null) {
                return log.getResult();
            }
            
            // 4. 执行支付
            PaymentResult result = doPay(request);
            
            // 5. 记录结果（幂等点）
            savePaymentLog(idempotencyKey, request, result);
            redisTemplate.opsForValue().set("pay:" + idempotencyKey, 
                JSON.toJSONString(result), 24, TimeUnit.HOURS);
            
            return result;
            
        } finally {
            redisTemplate.delete("lock:pay:" + idempotencyKey);
        }
    }
}
```

## 🔥 扩展追问 1：重试的异常类型如何选择？

| 异常类型 | 是否重试 | 说明 |
|---------|---------|------|
| `TimeoutException` | ✅ 重试 | 网络瞬时问题，可能下次成功 |
| `IOException` | ✅ 重试 | 连接断开，可恢复 |
| `HttpServerErrorException(5xx)` | ✅ 重试 | 服务端错误，可能恢复 |
| `HttpClientErrorException(4xx)` | ❌ 不重试 | 客户端错误，重试无用 |
| `BusinessException` | ❌ 不重试 | 业务逻辑错误，如余额不足 |
| `CircuitBreakerOpenException` | ❌ 不重试 | 已熔断，直接降级 |

```java
public class SmartRetryPolicy implements RetryPolicy {
    
    @Override
    public boolean canRetry(RetryContext context) {
        Throwable lastException = context.getLastThrowable();
        
        // 只重试特定异常
        if (lastException instanceof TimeoutException) {
            return context.getRetryCount() < 3;
        }
        
        if (lastException instanceof IOException) {
            return context.getRetryCount() < 3;
        }
        
        // 业务异常不重试
        return false;
    }
}
```

## 🔥 扩展追问 2：异步重试与消息队列

```java
@Service
public class AsyncRetryService {
    
    @Autowired
    private RabbitTemplate rabbitTemplate;
    
    /**
     * 同步调用失败，转异步MQ重试
     */
    public PaymentResult payWithAsyncRetry(PaymentRequest request) {
        try {
            // 先尝试同步调用（1次）
            return paymentClient.charge(request);
        } catch (Exception e) {
            log.warn("同步调用失败，转异步重试");
            
            // 发送到延迟队列，稍后重试
            rabbitTemplate.convertAndSend("pay.retry.exchange", 
                "pay.retry.routing", 
                request,
                message -> {
                    // 延迟30秒后重试
                    message.getMessageProperties().setDelay(30000);
                    return message;
                });
            
            return PaymentResult.builder()
                .status("PROCESSING")
                .message("支付处理中，请稍后查询")
                .build();
        }
    }
}

// 消费者处理重试
@Component
public class PayRetryConsumer {
    
    @RabbitListener(queues = "pay.retry.queue")
    public void handleRetry(PaymentRequest request, Message message, Channel channel) 
            throws IOException {
        
        try {
            PaymentResult result = paymentClient.charge(request);
            // 成功，确认消息
            channel.basicAck(message.getMessageProperties().getDeliveryTag(), false);
        } catch (Exception e) {
            // 失败，判断重试次数
            Integer retryCount = (Integer) message.getMessageProperties()
                .getHeaders().getOrDefault("x-retry-count", 0);
            
            if (retryCount < 3) {
                // 重新发送，增加重试次数
                message.getMessageProperties().setHeader("x-retry-count", retryCount + 1);
                rabbitTemplate.send("pay.retry.exchange", "pay.retry.routing", message);
                channel.basicAck(message.getMessageProperties().getDeliveryTag(), false);
            } else {
                // 重试耗尽，转入死信队列人工处理
                channel.basicReject(message.getMessageProperties().getDeliveryTag(), false);
            }
        }
    }
}
```

## 一句话收尾

**重试是双刃剑，用不好雪上加霜**——有限次数防雪崩、指数退避给恢复时间、熔断降级保系统存活、幂等设计防数据错乱，配合监控告警才能用得安心。
这道题和之前的熔断降级有呼应，但更聚焦于**第三方调用**这个特定场景。

几个面试加分点：
- **异常分类意识**：4xx不重试、5xx才重试，体现对HTTP协议的深入理解
- **幂等设计**：重试必须配合幂等，否则就是定时炸弹
- **工具选型**：Spring Retry简单、Resilience4j功能全、Feign整合好，根据场景选
