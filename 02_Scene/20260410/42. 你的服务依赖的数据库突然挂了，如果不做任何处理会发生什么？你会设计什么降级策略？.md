## 先说结论

数据库挂了不做任何处理会导致**级联雪崩**：连接池耗尽 → 线程池打满 → 全链路超时 → 服务僵死。降级策略核心是**快速失败+兜底数据+流量隔离**，通过熔断器自动切换只读缓存、Mock数据或排队等待，保证核心链路可用。

---

## 🔥 核心概念

**1. 解决什么问题**
- 数据库是系统的核心依赖，单点故障会导致整个服务不可用
- 慢查询或连接池耗尽会拖垮整个应用（线程阻塞）
- 需要保证"部分可用"而非"完全不可用"

**2. 基本原理/思想**
- **熔断（Circuit Breaker）**：故障时快速失败，防止资源耗尽
- **降级（Degrade）**：牺牲非核心功能，保证核心功能可用
- **兜底（Fallback）**：缓存数据、Mock数据、排队提示等备用方案

### 一、不做处理的后果（级联故障链）

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   用户请求   │────▶│  获取DB连接   │────▶│  执行SQL    │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                                │
                    ┌───────────────────────────┘
                    ▼
┌─────────────────────────────────────────────────────────┐
│  DB挂了 / 网络不通 / 慢查询                                │
│  ↓                                                      │
│  连接池等待队列堆积（HikariCP默认connectionTimeout=30s）   │
│  ↓                                                      │
│  线程池被占满（Tomcat默认200线程）                        │
│  ↓                                                      │
│  新请求无法处理，服务僵死（假死状态）                      │
│  ↓                                                      │
│  调用方超时重试 → 流量放大 → 级联雪崩                     │
└─────────────────────────────────────────────────────────┘
```

**关键指标恶化：**
- RT（响应时间）从几十ms飙升到30s+
- 错误率从0%飙升到100%
- CPU可能不高（线程都在等待），但服务完全不可用

### 三、具体实现方案

#### 方案1：熔断器自动降级（Sentinel/Hystrix）

```java
@RestController
@RequestMapping("/api/order")
public class OrderController {
    
    @Autowired
    private OrderService orderService;
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    /**
     * 查询订单详情 - 带熔断降级
     */
    @GetMapping("/{orderId}")
    @SentinelResource(
        value = "getOrderDetail",
        fallback = "getOrderDetailFallback",      // 业务异常降级
        blockHandler = "getOrderDetailBlock"      // 熔断/限流降级
    )
    public Result<OrderDTO> getOrderDetail(@PathVariable Long orderId) {
        // 正常走数据库查询
        OrderDTO order = orderService.getByIdFromDB(orderId);
        return Result.ok(order);
    }
    
    /**
     * 业务异常降级（如DB挂了抛异常）
     */
    public Result<OrderDTO> getOrderDetailFallback(Long orderId, Throwable ex) {
        log.warn("订单查询降级, orderId={}, 原因: {}", orderId, ex.getMessage());
        
        // 1. 先查缓存
        String cacheKey = "order:" + orderId;
        String cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return Result.ok(JSON.parseObject(cached, OrderDTO.class));
        }
        
        // 2. 缓存也没有，返回降级数据
        OrderDTO mockOrder = new OrderDTO();
        mockOrder.setOrderId(orderId);
        mockOrder.setStatus("QUERY_UNAVAILABLE");
        mockOrder.setRemark("系统繁忙，请稍后查看");
        
        return Result.ok(mockOrder, "服务降级中，显示缓存数据");
    }
    
    /**
     * 熔断/限流降级
     */
    public Result<OrderDTO> getOrderDetailBlock(Long orderId, BlockException ex) {
        return Result.fail(503, "系统繁忙，请稍后重试");
    }
}
```

**Sentinel熔断规则配置：**

```java
@Configuration
public class SentinelConfig {
    
    @PostConstruct
    public void initRules() {
        List<DegradeRule> rules = new ArrayList<>();
        
        DegradeRule rule = new DegradeRule();
        rule.setResource("getOrderDetail");
        rule.setGrade(CircuitBreakerStrategy.ERROR_RATIO);  // 按错误率熔断
        rule.setCount(0.5);                                  // 错误率超过50%
        rule.setTimeWindow(30);                              // 熔断30秒
        rule.setMinRequestAmount(10);                        // 最小请求数
        rule.setStatIntervalMs(1000);                        // 统计周期1s
        
        rules.add(rule);
        DegradeRuleManager.loadRules(rules);
    }
}
```

#### 方案3：多级缓存兜底

```java
@Service
public class OrderService {
    
    @Autowired
    private OrderMapper orderMapper;
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    // 本地缓存（Caffeine）- 最后一道防线
    private final Cache<Long, OrderDTO> localCache = Caffeine.newBuilder()
        .maximumSize(10000)
        .expireAfterWrite(5, TimeUnit.MINUTES)
        .build();
    
    /**
     * 多级缓存查询（DB → Redis → LocalCache → 降级）
     */
    public OrderDTO getOrderWithFallback(Long orderId) {
        // L1: 本地缓存（进程内，不依赖网络）
        OrderDTO cached = localCache.getIfPresent(orderId);
        if (cached != null) {
            return cached;
        }
        
        // L2: Redis
        String redisKey = "order:" + orderId;
        String redisData = redisTemplate.opsForValue().get(redisKey);
        if (redisData != null) {
            OrderDTO order = JSON.parseObject(redisData, OrderDTO.class);
            localCache.put(orderId, order);  // 回填本地缓存
            return order;
        }
        
        // L3: 数据库（带熔断保护）
        try {
            OrderDTO order = orderMapper.selectById(orderId);
            if (order != null) {
                // 异步回填缓存
                redisTemplate.opsForValue().set(redisKey, 
                    JSON.toJSONString(order), 10, TimeUnit.MINUTES);
                localCache.put(orderId, order);
            }
            return order;
        } catch (DataAccessException e) {
            // DB异常，返回友好提示
            log.error("数据库查询异常, orderId={}", orderId, e);
            throw new ServiceException("系统繁忙，请稍后重试");
        }
    }
}
```

## 🔥 扩展追问 1：熔断器三种状态如何流转？

```
┌─────────────┐    失败率<阈值     ┌─────────────┐
│   CLOSED    │◀─────────────────│  HALF_OPEN  │
│   (关闭)     │                  │  (半开)      │
└──────┬──────┘                  └──────▲──────┘
       │ 失败率≥阈值                    │
       │ 或慢调用比例超标                 │ 探测成功
       ▼                                  │
┌─────────────┐                          │
│    OPEN     │──────────────────────────┘
│   (打开)     │   熔断时长结束，进入半开探测
└─────────────┘
```

**Sentinel状态流转细节：**
- **CLOSED**：正常放行，统计错误率/慢调用比例
- **OPEN**：熔断打开，所有请求直接走降级逻辑
- **HALF_OPEN**：经过熔断时长（如30s）后，放行少量请求探测恢复情况

```java
// 慢调用熔断配置（更适合DB慢查询场景）
DegradeRule slowCallRule = new DegradeRule();
slowCallRule.setGrade(CircuitBreakerStrategy.SLOW_REQUEST_RATIO);
slowCallRule.setCount(1000);      // 慢调用阈值：超过1秒算慢
slowCallRule.setSlowRatioThreshold(0.8);  // 慢调用比例超过80%
slowCallRule.setTimeWindow(30);   // 熔断30秒
```

## 🔥 扩展追问 2：降级数据一致性如何保证？

| 场景 | 策略 | 说明 |
|-----|------|------|
| 读降级 | 缓存过期时间 + 异步刷新 | 允许短暂不一致 |
| 写降级 | 本地消息表 + 最终一致 | 保证数据不丢 |
| 库存类 | 预扣缓存 + 异步对账 | 定时校准差异 |

```java
/**
 * 写降级：本地消息表保证最终一致
 */
@Service
public class OrderWriteFallbackService {
    
    @Autowired
    private OrderMessageRepository messageRepository;
    
    @Transactional
    public void asyncCreateOrder(OrderRequest request) {
        // 1. 写入业务表（本地事务）
        Order order = new Order();
        order.setStatus("PENDING");
        orderMapper.insert(order);
        
        // 2. 写入消息表（同一事务）
        OrderMessage msg = new OrderMessage();
        msg.setOrderId(order.getId());
        msg.setStatus("UNSENT");
        msg.setRetryCount(0);
        messageRepository.insert(msg);
        
        // 3. 事务提交后，异步发送MQ
        TransactionSynchronizationManager.registerSynchronization(
            new TransactionSynchronizationAdapter() {
                @Override
                public void afterCommit() {
                    rabbitTemplate.convertAndSend("order.create", msg);
                }
            }
        );
    }
}
```

```java
/**
 * 混沌工程：模拟DB故障
 */
@Component
@Profile("chaos")
public class DBChaosMonkey {
    
    @Autowired
    private DataSource dataSource;
    
    private final AtomicBoolean dbBroken = new AtomicBoolean(false);
    
    /**
     * 触发DB故障（用于压测验证降级效果）
     */
    public void breakDB() {
        dbBroken.set(true);
        // 关闭连接池，模拟DB不可达
        if (dataSource instanceof HikariDataSource) {
            ((HikariDataSource) dataSource).suspendPool();
        }
    }
    
    /**
     * 恢复DB
     */
    public void fixDB() {
        dbBroken.set(false);
        if (dataSource instanceof HikariDataSource) {
            ((HikariDataSource) dataSource).resumePool();
        }
    }
}
```
