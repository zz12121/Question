## 先说结论

TPS上不去但资源利用率低，典型瓶颈是**线程阻塞**——数据库连接池耗尽、同步调用下游超时、锁竞争、Full GC停顿。排查思路：**看线程状态**（jstack找BLOCKED/WAITING）、**看调用链**（Arthas trace看耗时分布）、**看资源限制**（连接池/队列满），常见根因是连接池配置不当或下游依赖拖慢。

### 二、具体排查步骤

#### 第1步：线程状态分析

```bash
# 1. 打印线程栈
jstack -l <pid> > thread_dump.txt

# 2. 统计线程状态
grep java.lang.Thread.State thread_dump.txt | sort | uniq -c

# 典型输出：
#  50 java.lang.Thread.State: RUNNABLE          # 正在运行
# 150 java.lang.Thread.State: WAITING           # 无限等待（如wait()）
#  50 java.lang.Thread.State: TIMED_WAITING     # 限时等待（如sleep）
#  10 java.lang.Thread.State: BLOCKED           # 阻塞（等锁）

# 3. 分析BLOCKED线程
grep -A 5 "java.lang.Thread.State: BLOCKED" thread_dump.txt

# 典型场景：
# - 等数据库连接池：at com.alibaba.druid.pool.DruidDataSource.getConnection
# - 等锁：at com.example.Service.method (Service.java:123)
```

**线程状态解读：**

| 状态 | 含义 | 可能原因 |
|-----|------|---------|
| RUNNABLE | 正在运行 | 正常，但数量少说明线程被阻塞 |
| BLOCKED | 阻塞等锁 | synchronized、ReentrantLock竞争 |
| WAITING | 无限等待 | wait()、join()、LockSupport.park() |
| TIMED_WAITING | 限时等待 | sleep、wait(timeout)、join(timeout) |

### 三、常见瓶颈根因

#### 根因1：数据库连接池过小

```yaml
# 错误配置
spring:
  datasource:
    hikari:
      maximum-pool-size: 10  # 太小！
      connection-timeout: 30000  # 30秒太长

# 正确配置
spring:
  datasource:
    hikari:
      maximum-pool-size: 50  # 根据并发调整
      minimum-idle: 10
      connection-timeout: 3000  # 3秒快速失败
      idle-timeout: 600000
      max-lifetime: 1800000
```

**连接池大小公式：**
```
连接数 = ((核心数 × 2) + 有效磁盘数)

例如：8核服务器，1块SSD
连接数 = (8 × 2) + 1 = 17（最小）
实际可配 20-50，根据压测调整
```

#### 根因2：同步调用下游服务

```java
@Service
public class OrderService {
    
    @Autowired
    private PaymentClient paymentClient;  // Feign客户端
    
    /**
     * 问题：同步调用下游，阻塞Tomcat线程
     */
    public Order createOrder(OrderRequest request) {
        // 1. 创建订单
        Order order = orderMapper.insert(request);
        
        // 2. 同步调用支付服务（阻塞！）
        PaymentResult result = paymentClient.charge(order);  // 耗时200ms
        
        // 3. 更新状态
        order.setStatus(result.isSuccess() ? "PAID" : "FAILED");
        orderMapper.update(order);
        
        return order;
    }
    
    /**
     * 优化：异步化，释放线程
     */
    @Async("taskExecutor")
    public CompletableFuture<Order> createOrderAsync(OrderRequest request) {
        Order order = orderMapper.insert(request);
        
        return CompletableFuture.supplyAsync(() -> {
            PaymentResult result = paymentClient.charge(order);
            order.setStatus(result.isSuccess() ? "PAID" : "FAILED");
            orderMapper.update(order);
            return order;
        });
    }
}
```

#### 根因3：锁竞争

```java
@Service
public class InventoryService {
    
    /**
     * 问题：synchronized导致串行化
     */
    public synchronized void deductStock(Long skuId, Integer quantity) {
        // 只有一个线程能执行，其他线程BLOCKED
        Integer stock = inventoryMapper.getStock(skuId);
        if (stock >= quantity) {
            inventoryMapper.updateStock(skuId, stock - quantity);
        }
    }
    
    /**
     * 优化：分段锁，提高并发
     */
    private final ConcurrentHashMap<Long, Lock> locks = new ConcurrentHashMap<>();
    
    public void deductStockOptimized(Long skuId, Integer quantity) {
        Lock lock = locks.computeIfAbsent(skuId, k -> new ReentrantLock());
        
        lock.lock();
        try {
            Integer stock = inventoryMapper.getStock(skuId);
            if (stock >= quantity) {
                inventoryMapper.updateStock(skuId, stock - quantity);
            }
        } finally {
            lock.unlock();
        }
    }
    
    /**
     * 更优：数据库乐观锁
     */
    public void deductStockBest(Long skuId, Integer quantity) {
        int updated = inventoryMapper.updateStockWithVersion(
            skuId, quantity, version);
        
        if (updated == 0) {
            throw new ConcurrentModificationException("库存并发冲突");
        }
    }
}
```

```java
/**
 * 线程池大小计算公式（参考）
 */
public class ThreadPoolCalculator {
    
    /**
     * CPU密集型：线程数 = CPU核心数 + 1
     */
    public int cpuBoundPoolSize() {
        return Runtime.getRuntime().availableProcessors() + 1;
    }
    
    /**
     * IO密集型：线程数 = CPU核心数 × (1 + 等待时间/计算时间)
     */
    public int ioBoundPoolSize(double waitTime, double computeTime) {
        int cores = Runtime.getRuntime().availableProcessors();
        return (int) (cores * (1 + waitTime / computeTime));
    }
    
    /**
     * 实际压测确定
     */
    public void findOptimalSizeByTest() {
        // 1. 从小线程数开始压测
        // 2. 逐步增加线程数，观察TPS和RT变化
        // 3. 找到TPS拐点，即为最优线程数
        // 4. 超过拐点后，RT上升，TPS下降（线程切换开销）
    }
```
