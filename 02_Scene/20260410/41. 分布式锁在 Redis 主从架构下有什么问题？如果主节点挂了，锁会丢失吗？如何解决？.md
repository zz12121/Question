## 先说结论

Redis 主从架构下分布式锁的核心问题是**主从异步复制导致锁丢失**：主节点加锁成功后立即宕机，从节点晋升为主节点，但锁数据还未同步，导致多个客户端可以同时获得锁。解决方案：**RedLock 算法**（多节点加锁）或 **Redisson 的看门狗自动续期**，生产环境推荐 **Redisson + 单节点/哨兵模式**（权衡性能与可靠性）。

---

## 🔥 核心概念

### 主从架构锁丢失场景

```
┌─────────────────────────────────────────────────────────┐
│              Redis 主从锁丢失问题                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  正常流程：                                              │
│  Client ──→ SET lock:order:123 NX EX 30 ──→ Master     │
│                   └── 锁创建成功                          │
│                                                         │
│  故障场景：                                              │
│  1. Client A 向 Master 加锁成功                          │
│     Master ──[异步复制]──→ Slave（延迟 1ms~1s）          │
│                                                         │
│  2. Master 立即宕机                                      │
│     └── 锁数据还未同步到 Slave                           │
│                                                         │
│  3. Slave 晋升为新的 Master                              │
│     └── 新 Master 上没有 lock:order:123                  │
│                                                         │
│  4. Client B 向新 Master 加锁                            │
│     └── SET lock:order:123 NX EX 30 ──→ 成功！           │
│                                                         │
│  结果：Client A 和 Client B 同时持有锁！                   │
│        分布式锁的互斥性被破坏！                           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 分布式锁的基本要求

| 要求 | 说明 |
|------|------|
| **互斥性** | 同一时刻只有一个客户端能持有锁 |
| **防死锁** | 锁必须有过期时间，防止客户端崩溃导致永久占用 |
| **可重入** | 同一线程可以重复获取锁 |
| **高可用** | 锁服务不能单点故障 |
| **自动续期** | 业务执行时间长时自动延长锁有效期 |

## 🔥 详细展开：Redis 分布式锁演进

### 方案一：SET NX EX（单节点，最简单）

```java
// 加锁：SET key value NX EX seconds
// NX - 不存在才设置（互斥）
// EX - 设置过期时间（防死锁）
String result = redisTemplate.opsForValue()
    .setIfAbsent("lock:order:" + orderId, 
                 clientId, 
                 30, 
                 TimeUnit.SECONDS);

if (Boolean.TRUE.equals(result)) {
    try {
        // 执行业务
        processOrder(orderId);
    } finally {
        // 释放锁（检查是否是自己加的锁）
        String currentValue = redisTemplate.opsForValue().get("lock:order:" + orderId);
        if (clientId.equals(currentValue)) {
            redisTemplate.delete("lock:order:" + orderId);
        }
    }
}
```

**问题：**
- 主从切换时锁丢失
- 业务执行超过 30s，锁自动过期，其他客户端可以获取锁
- 释放锁时非原子操作（GET + DEL 之间可能锁已过期被其他客户端获取）

### 方案二：Lua 脚本原子释放（解决误删）

```java
// Lua 脚本：原子检查 + 删除
private static final String UNLOCK_SCRIPT = 
    "if redis.call('get', KEYS[1]) == ARGV[1] then " +
    "    return redis.call('del', KEYS[1]) " +
    "else " +
    "    return 0 " +
    "end";

public boolean unlock(String key, String clientId) {
    DefaultRedisScript<Long> script = new DefaultRedisScript<>();
    script.setScriptText(UNLOCK_SCRIPT);
    script.setResultType(Long.class);
    
    Long result = redisTemplate.execute(script, 
        Collections.singletonList(key), 
        clientId);
    
    return result != null && result == 1L;
}
```

**仍存在的问题：**
- 主从切换锁丢失
- 无自动续期机制

### 方案三：Redisson（生产推荐）

```
Redisson 核心特性：
┌─────────────────────────────────────────────────────────┐
│  1. 看门狗自动续期                                       │
│     └── 默认锁有效期 30s，看门狗每 10s 检查一次           │
│     └── 业务未执行完，自动续期到 30s                      │
│     └── 业务执行完，立即释放锁                            │
│                                                         │
│  2. 可重入锁                                             │
│     └── 同一线程可以多次获取锁，维护重入计数               │
│                                                         │
│  3. 红锁 RedLock（多节点）                                │
│     └── 向 N 个独立 Redis 节点加锁                        │
│     └── 大多数节点（N/2+1）加锁成功才算成功                │
│                                                         │
│  4. 读写锁、信号量、CountDownLatch 等高级锁               │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

```java
        //     .setMasterName("mymaster")
        //     .addSentinelAddress("redis://sentinel1:26379", "redis://sentinel2:26379");

        //     .addNodeAddress("redis://node1:7000", "redis://node2:7000");

}

@Service
public class OrderService {
    
    @Autowired
    private RedissonClient redissonClient;
    
    public void processOrder(Long orderId) {
        // 获取锁（可重入锁）
        RLock lock = redissonClient.getLock("lock:order:" + orderId);
        
        try {
            // 尝试加锁，最多等待 10s，锁有效期 30s（看门狗自动续期）
            boolean locked = lock.tryLock(10, 30, TimeUnit.SECONDS);
            
            if (!locked) {
                throw new BusinessException("获取锁失败，请稍后重试");
            }
            
            // 执行业务（锁会自动续期，不用担心过期）
            doProcess(orderId);
            
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException(e);
        } finally {
            // 释放锁（检查是否当前线程持有）
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

### 方案四：RedLock 红锁（解决主从问题）

```java
/**
 * RedLock：向多个独立 Redis 节点加锁
 * 大多数节点（N/2+1）加锁成功才算成功
 */
public class RedLockDemo {
    
    public boolean tryRedLock(String resource) {
        // 5 个独立 Redis 节点（不是主从，是 5 个独立实例）
        List<RedissonClient> redissonNodes = Arrays.asList(
            Redisson.create(new Config().useSingleServer().setAddress("redis://node1:6379")),
            Redisson.create(new Config().useSingleServer().setAddress("redis://node2:6379")),
            Redisson.create(new Config().useSingleServer().setAddress("redis://node3:6379")),
            Redisson.create(new Config().useSingleServer().setAddress("redis://node4:6379")),
            Redisson.create(new Config().useSingleServer().setAddress("redis://node5:6379"))
        );
        
        String value = UUID.randomUUID().toString();
        int lockCount = 0;
        long startTime = System.currentTimeMillis();
        
        for (RedissonClient node : redissonNodes) {
            try {
                RLock lock = node.getLock(resource);
                // 向每个节点尝试加锁，超时时间要短
                if (lock.tryLock(100, 10, TimeUnit.MILLISECONDS)) {
                    lockCount++;
                }
            } catch (Exception e) {
                // 某个节点失败，继续尝试其他节点
            }
        }
        
        // 大多数节点加锁成功（5个节点至少3个成功）
        if (lockCount >= 3) {
            return true;
        }
        
        // 失败，释放已加的锁
        redissonNodes.forEach(node -> {
            try {
                node.getLock(resource).unlock();
            } catch (Exception e) {
                // 忽略
            }
        });
        
        return false;
    }
}
```

**RedLock 问题：**
- 需要至少 3 个独立 Redis 节点，运维成本高
- 性能比单节点差（需要向多个节点请求）
- 时钟漂移问题（各节点时间不一致）
- **Redis 作者与分布式专家对此有争议**

## 🔥 扩展追问 2：锁续期原理

```java
// Redisson 看门狗源码核心逻辑
public class WatchDog {
    
    private static final long LOCK_EXPIRATION_INTERVAL = 30 * 1000;  // 锁有效期 30s
    private static final long RENEWAL_INTERVAL = LOCK_EXPIRATION_INTERVAL / 3;  // 10s 续期一次
    
    /**
     * 启动看门狗
     */
    private void scheduleExpirationRenewal(long threadId) {
        Timeout task = commandExecutor.getConnectionManager().newTimeout(timeout -> {
            // 执行续期 Lua 脚本
            RFuture<Boolean> future = renewExpirationAsync(threadId);
            
            future.thenAccept(res -> {
                if (res) {
                    // 续期成功，递归调度下一次
                    scheduleExpirationRenewal(threadId);
                }
                // 续期失败（锁已被释放或不是当前线程持有），停止看门狗
            });
            
        }, RENEWAL_INTERVAL, TimeUnit.MILLISECONDS);
        
        expirationRenewalMap.put(getEntryName(), task);
    }
    
    /**
     * 续期 Lua 脚本
     * 检查锁是否仍被当前线程持有，如果是则续期到 30s
     */
    private static final String RENEW_SCRIPT = 
        "if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then " +
        "    redis.call('pexpire', KEYS[1], ARGV[1]); " +
        "    return 1; " +
        "end; " +
        "return 0;";
}
```

## 🔥 扩展追问 4：Redis 锁 vs ZooKeeper 锁

| 特性 | Redis 锁 | ZooKeeper 锁 |
|------|---------|-------------|
| **实现** | SET NX / Redisson | 临时顺序节点 |
| **一致性** | 最终一致（主从可能丢失）| 强一致（ZAB 协议）|
| **性能** | 极高（10万+/s） | 中等（1万+/s）|
| **可靠性** | 中（需 RedLock） | 高 |
| **自动续期** | 看门狗 | 会话保持 |
| **适用** | 高并发，可接受偶尔失效 | 强一致要求 |
| **运维** | 简单 | 复杂 |

**选型建议：**
- 追求性能：**Redis + Redisson**
- 追求强一致：**ZooKeeper / etcd**

## 一句话收尾

Redis 主从架构下分布式锁的**核心风险是异步复制导致锁丢失**，解决方案：**Redisson 看门狗自动续期**解决单节点锁过期问题，**RedLock 多节点加锁**解决主从切换问题；生产环境推荐 **Redisson + 哨兵模式**（高可用）或 **单节点 + DB 唯一索引兜底**（简单可靠），根据业务对一致性的要求权衡选择。
