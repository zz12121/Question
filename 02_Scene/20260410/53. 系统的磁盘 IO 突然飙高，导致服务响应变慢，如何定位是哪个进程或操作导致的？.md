## 先说结论

磁盘IO飙高定位的核心是**先确认是读还是写，再定位具体进程和文件**——用`iostat`看读写比例和等待时间，用`iotop`/`pidstat`定位高IO进程，用`lsof`/`strace`追踪具体文件和操作。常见原因：日志狂写、大查询落盘、定时任务、内存不足触发swap。

## 🔥 核心概念

**1. 解决什么问题**
- 磁盘IO是系统瓶颈，IO等待会导致进程阻塞，服务响应变慢
- 需要快速定位是哪个进程、什么操作导致的IO压力

**2. 基本原理/思想**
- **读写分离**：读IO和写IO的优化方向不同
- **进程定位**：从系统级到进程级到文件级逐层深入
- **关联分析**：IO高往往伴随CPU、内存异常，综合判断

### 三、常见根因及处理

| 根因 | 现象 | 定位方法 | 解决方案 |
|-----|------|---------|---------|
| **日志狂写** | 写IO高，日志文件增长快 | `lsof`看到日志文件FD | 调整日志级别，异步日志，日志分割 |
| **大查询落盘** | 读IO高，数据库进程IO高 | MySQL `show processlist` | 优化SQL，加索引，限制大查询 |
| **内存不足swap** | IO高，内存使用率高 | `free -m`看swap使用率 | 扩容内存，关闭swap，优化内存使用 |
| **定时任务** | IO周期性飙高 | `crontab -l`，对比时间 | 调整定时任务时间，分散执行 |
| **文件备份** | 读IO高，特定时间点 | 查看备份脚本执行时间 | 调整备份时间，限速备份 |
| **GC日志** | 写IO高，GC频繁 | 查看GC日志配置 | 关闭GC日志或调整输出路径 |

#### 案例1：日志级别配置错误

```bash
# 现象：应用写IO 50MB/s，日志文件每小时增长10GB

# 定位：
lsof -p 12345 | grep log
# 输出：application.log 大小 50G

# 查看日志内容：
tail -f /var/log/app/application.log
# 发现大量DEBUG日志

# 解决：
# 1. 调整日志级别为INFO/ERROR
# 2. 使用异步日志（logback-async）
# 3. 配置日志滚动（按天/按大小）
```

**Logback异步化配置：**
```xml
<configuration>
    <!-- 异步appender -->
    <appender name="ASYNC" class="ch.qos.logback.classic.AsyncAppender">
        <queueSize>5000</queueSize>
        <discardingThreshold>0</discardingThreshold>
        <appender-ref ref="FILE" />
    </appender>
    
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>/var/log/app/application.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>/var/log/app/application.%d{yyyy-MM-dd}.%i.log</fileNamePattern>
            <maxHistory>30</maxHistory>
            <maxFileSize>100MB</maxFileSize>
        </rollingPolicy>
    </appender>
    
    <root level="INFO">
        <appender-ref ref="ASYNC" />
    </root>
</configuration>
```

#### 案例3：内存不足触发swap

```bash
# 现象：IO高，但找不到明显的高IO进程

# 定位：
free -m
# 输出：
#              total       used       free     shared    buffers     cached
# Mem:         16000      15800        200        100         50        500
# Swap:         8000       7500        500

# 发现内存几乎用完，swap使用率很高

# 查看哪些进程在swap中
for file in /proc/*/status ; do awk '/VmSwap|Name/{printf $2 " " $3}END{ print ""}' $file; done | sort -k 2 -n -r | head -10

# 输出：
# java 5000 MB
# mysql 2000 MB
```

**解决：**
```bash
# 1. 临时关闭swap（谨慎使用）
swapoff -a

# 2. 长期方案：扩容内存或优化应用内存使用
# - JVM调优：调整堆大小
# - MySQL调优：调整Buffer Pool大小

# 3. 配置swappiness，减少swap倾向
sysctl vm.swappiness=10  # 默认60，越小越不倾向swap
```

## 一句话收尾

**磁盘IO问题定位三板斧：iostat看状况、iotop找进程、lsof/strace追文件**——先区分读写类型，再逐层定位到具体进程和文件，常见根因是日志、大查询、swap、定时任务，预防靠监控告警和代码优化。
第53题完成！磁盘IO排查是运维和开发的必备技能，重点掌握那几个Linux命令。

几个核心命令速记：
- `iostat -x 1`：看整体IO状况（%util、await）
- `iotop`/`pidstat -d`：定位高IO进程
- `lsof -p <pid>`：看进程打开的文件
- `strace -e trace=file -p <pid>`：追踪文件操作

常见根因：
- **日志**：级别配错、同步写、没分割
- **数据库**：大查询、全表扫描、buffer pool小
- **内存**：不足触发swap，磁盘当内存用
- **定时任务**：备份、清理、统计任务
