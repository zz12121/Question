## 先说结论

内存持续增长 + 重启恢复 = **典型的内存泄漏特征**。排查核心思路是：**抓堆Dump分析大对象 → 分析GC Root引用链 → 定位泄漏代码 → 验证修复**。

---

## 🔥 核心概念

### 1. 内存泄漏 vs 内存溢出

| 对比 | 内存泄漏 (Memory Leak) | 内存溢出 (OOM) |
|------|------------------------|----------------|
| **本质** | 对象不再使用但无法被GC回收 | 申请内存超过JVM上限 |
| **表现** | 内存缓慢/持续增长，最终可能OOM | 瞬间报错 `OutOfMemoryError` |
| **重启** | 重启后恢复，但问题会重现 | 重启后可能暂时解决 |
| **常见原因** | 静态集合持有对象、未关闭资源、ThreadLocal未清理 | 大对象分配、堆设置过小 |

### 2. 本题特征分析

```
内存持续增长 + 重启恢复 + 几天后复现
        ↓
    典型的"慢性"内存泄漏
        ↓
可能原因：
- 静态集合无限增长（缓存、监听器列表）
- 未关闭的资源（流、连接、线程）
- ThreadLocal未清理（线程池场景）
- 类加载器泄漏（热部署、OSGi）
```

### 步骤4：代码修复示例

```java
// ❌ 问题代码：静态缓存无限增长
@Component
public class CacheManager {
    // 静态集合，永远不会被回收
    private static final Map<String, Object> CACHE = new HashMap<>();
    
    public void put(String key, Object value) {
        CACHE.put(key, value);  // 只进不出！
    }
}

// ✅ 修复方案1：使用带过期策略的缓存
@Component
public class CacheManager {
    // Caffeine自动淘汰，避免内存泄漏
    private final Cache<String, Object> cache = Caffeine.newBuilder()
        .maximumSize(10000)           // 最大条目数
        .expireAfterWrite(1, TimeUnit.HOURS)  // 写入后1小时过期
        .recordStats()
        .build();
}

// ✅ 修复方案2：使用弱引用（适合临时缓存）
private final Map<String, WeakReference<Object>> weakCache = new WeakHashMap<>();

// ✅ 修复方案3：手动清理 + 定时任务
@Scheduled(fixedRate = 3600000)  // 每小时清理
public void cleanExpired() {
    Iterator<Map.Entry<String, CacheEntry>> it = cache.entrySet().iterator();
    while (it.hasNext()) {
        if (it.next().getValue().isExpired()) {
            it.remove();
        }
    }
}
```

## 🔥 扩展追问 1：常见内存泄漏场景

### 场景1：ThreadLocal未清理（线程池场景）

```java
// ❌ 问题代码：线程池复用线程，ThreadLocal不清理
@Service
public class UserService {
    private static final ThreadLocal<UserContext> contextHolder = new ThreadLocal<>();
    
    public void process() {
        contextHolder.set(new UserContext());  // 设置
        // 业务逻辑...
        // 忘记remove()！
    }
}

// ✅ 修复：finally中清理
public void process() {
    contextHolder.set(new UserContext());
    try {
        // 业务逻辑
    } finally {
        contextHolder.remove();  // 必须清理！
    }
}

// ✅ 或使用TransmittableThreadLocal（TTL）自动透传和清理
```

### 场景2：未关闭资源

```java
// ❌ 问题代码：流未关闭
public String readFile(String path) throws IOException {
    BufferedReader reader = new BufferedReader(new FileReader(path));
    return reader.readLine();  // 流未关闭，句柄泄漏
}

// ✅ 修复：try-with-resources
public String readFile(String path) throws IOException {
    try (BufferedReader reader = new BufferedReader(new FileReader(path))) {
        return reader.readLine();
    }  // 自动关闭
}
```

### 场景3：监听器/回调未注销

```java
// ❌ 问题代码：添加监听器但不移除
public class EventBus {
    private static final List<Listener> listeners = new ArrayList<>();
    
    public void register(Listener listener) {
        listeners.add(listener);  // 只增不减
    }
}

// ✅ 修复：提供注销方法 + 弱引用
public void register(Listener listener) {
    listeners.add(new WeakReference<>(listener));  // 弱引用
}

// 或
public void unregister(Listener listener) {
    listeners.remove(listener);
}
```

## 🔥 扩展追问 2：MAT分析技巧

### 关键报告解读

| 报告类型 | 用途 | 关键指标 |
|---------|------|---------|
| **Histogram** | 查看各类对象数量和大小 | Retained Heap（实际占用） |
| **Dominator Tree** | 找到对象的支配者（谁持有它） | 从大到小排查 |
| **Leak Suspects** | 自动分析疑似泄漏点 | 直接给出可疑对象 |
| **Top Consumers** | 内存消耗大户 | 快速定位 |
| **Path to GC Roots** | 引用链分析 | 找到为什么无法回收 |

## 一句话收尾

内存泄漏排查三板斧：**`jstat`看趋势 → `jmap`抓Dump → `MAT`找引用链**。预防胜于治疗：静态集合必须有界、ThreadLocal必须清理、资源必须try-with-resources。
