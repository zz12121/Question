## 先说结论

预约抢购系统的核心矛盾是**瞬时流量洪峰 vs 极少量库存**，设计思路是：**前端削峰 + Redis 原子扣库存 + 异步下单 + DB 兜底防超卖**，让 99% 的请求在 Redis 层快速失败，只有抢到库存的少量请求才进入后端处理。

### 2. 核心：Redis Lua 原子扣库存

```lua
-- seckill.lua
-- KEYS[1] = stock:activityId    库存 key
-- KEYS[2] = sold:activityId     已购用户集合 key
-- ARGV[1] = userId
-- ARGV[2] = activityId

-- 1. 检查用户是否已抢过
if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 1 then
    return -1  -- 已购买，重复提交
end

-- 2. 检查库存
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil or stock <= 0 then
    return -2  -- 库存不足
end

-- 3. 原子扣减库存
redis.call('DECRBY', KEYS[1], 1)

-- 4. 记录购买用户（防重复）
redis.call('SADD', KEYS[2], ARGV[1])

-- 5. 返回成功
return 1
```

```java
@Service
public class SeckillService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private RocketMQTemplate mqTemplate;
    
    // 预加载 Lua 脚本（避免每次传输脚本，用 SHA 摘要）
    private static final DefaultRedisScript<Long> SECKILL_SCRIPT;
    static {
        SECKILL_SCRIPT = new DefaultRedisScript<>();
        SECKILL_SCRIPT.setLocation(new ClassPathResource("scripts/seckill.lua"));
        SECKILL_SCRIPT.setResultType(Long.class);
    }
    
    /**
     * 秒杀下单主入口
     */
    public SeckillResult doSeckill(Long activityId, Long userId) {
        String stockKey = "stock:" + activityId;
        String soldKey  = "sold:"  + activityId;
        
        // Lua 脚本原子执行
        Long result = redisTemplate.execute(
            SECKILL_SCRIPT,
            Arrays.asList(stockKey, soldKey),
            String.valueOf(userId),
            String.valueOf(activityId)
        );
        
        if (result == null || result == -1L) {
            return SeckillResult.fail("您已抢过，请勿重复提交");
        }
        if (result == -2L) {
            return SeckillResult.fail("手慢了，库存不足");
        }
        
        // 抢购成功，发 MQ 异步创建订单
        String orderNo = generateOrderNo(userId, activityId);
        SeckillMessage msg = SeckillMessage.builder()
            .userId(userId)
            .activityId(activityId)
            .orderNo(orderNo)
            .build();
        
        mqTemplate.asyncSend("seckill-order-topic", msg);
        
        // 立即返回，告知用户"排队中"
        return SeckillResult.queuing(orderNo, "正在为您处理，请稍候查询结果");
    }
}
```

### 3. 异步下单消费者

```java
@Component
@RocketMQMessageListener(
    topic = "seckill-order-topic",
    consumerGroup = "seckill-order-group",
    consumeMode = ConsumeMode.CONCURRENTLY,
    consumeThreadMax = 8  // 适当并发，不要太高
)
public class SeckillOrderConsumer implements RocketMQListener<SeckillMessage> {
    
    @Autowired
    private OrderMapper orderMapper;
    
    @Autowired
    private InventoryMapper inventoryMapper;
    
    @Override
    @Transactional(rollbackFor = Exception.class)
    public void onMessage(SeckillMessage msg) {
        // 幂等检查：防止 MQ 重投
        if (orderMapper.existsByOrderNo(msg.getOrderNo())) {
            return;
        }
        
        // 1. 扣 DB 库存（有唯一索引兜底）
        int affected = inventoryMapper.decrementStock(msg.getActivityId(), 1);
        if (affected == 0) {
            // DB 库存不足（极端情况：Redis 与 DB 数据不一致）
            // 回补 Redis 库存
            redisTemplate.opsForValue().increment("stock:" + msg.getActivityId());
            // 通知用户失败
            notifyUser(msg.getUserId(), "很遗憾，抢购失败");
            return;
        }
        
        // 2. 创建订单
        Order order = Order.builder()
            .orderNo(msg.getOrderNo())
            .userId(msg.getUserId())
            .activityId(msg.getActivityId())
            .status(OrderStatus.PENDING_PAY)
            .expireTime(LocalDateTime.now().plusMinutes(15))  // 15分钟支付
            .build();
        orderMapper.insert(order);
        
        // 3. 通知用户抢购成功
        notifyUser(msg.getUserId(), "恭喜您抢购成功！请15分钟内完成支付");
    }
}
```

### 4. DB 层：唯一索引兜底防超卖

```sql
-- 库存表（乐观锁版本号）
CREATE TABLE activity_inventory (
    activity_id  BIGINT      NOT NULL,
    total_stock  INT         NOT NULL COMMENT '总库存',
    sold_stock   INT         NOT NULL DEFAULT 0 COMMENT '已售库存',
    version      INT         NOT NULL DEFAULT 0 COMMENT '乐观锁版本号',
    PRIMARY KEY (activity_id)
);

-- 订单表（唯一索引防重复下单）
CREATE TABLE seckill_order (
    id           BIGINT      NOT NULL AUTO_INCREMENT,
    order_no     VARCHAR(32) NOT NULL,
    user_id      BIGINT      NOT NULL,
    activity_id  BIGINT      NOT NULL,
    status       TINYINT     NOT NULL DEFAULT 0,
    PRIMARY KEY (id),
    UNIQUE KEY uk_order_no (order_no),          -- 防重复创建
    UNIQUE KEY uk_user_activity (user_id, activity_id)  -- 防一人多单
);

-- 扣库存 SQL（原子更新，sold_stock 不超 total_stock）
UPDATE activity_inventory
SET sold_stock = sold_stock + 1,
    version    = version + 1
WHERE activity_id = #{activityId}
  AND sold_stock < total_stock;
-- affected rows = 0 时，表示库存不足，拒绝下单
```

        }

## 一句话收尾

预约抢购系统的核心是**漏斗模型**：通过前端限流、网关削峰、Redis Lua 原子扣库存，让绝大部分请求在最外层快速失败，只把少量有效请求透传到 DB；配合 **MQ 异步下单 + DB 唯一索引兜底**，实现高性能、零超卖的抢购系统。
早上好！这道题是典型的**系统设计题**，面试官考察的是整体架构思维，几个最关键的点总结一下：

1. **漏斗模型**：把请求挡在越外层越好，只让极少量请求打到 DB
2. **Redis Lua 原子扣减**：防超卖的核心，check + decr 必须原子
3. **MQ 异步下单**：峰值流量和处理能力解耦，快速返回排队状态
4. **DB 唯一索引兜底**：`uk_user_activity` 是最后一道防线
5. **医院挂号特殊性**：分时段库存 + 实名就诊 + 爽约机制，是加分项

如果面试官追问"Redis 挂了怎么办"，答：**降级到 DB 限流模式**，通过 DB 行锁控制并发，牺牲性能保证可用。
