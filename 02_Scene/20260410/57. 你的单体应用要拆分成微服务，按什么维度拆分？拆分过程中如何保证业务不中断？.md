## 先说结论

微服务拆分遵循**"先垂直拆分（按业务域），后水平拆分（按读写/冷热）"**的原则。保证业务不中断的核心是**"逐步演进、灰度发布、双写兼容"**：通过Strangler Fig模式渐进式替换，配合数据双写、流量灰度、快速回滚机制实现平滑迁移。

## 🔥 核心概念

### 拆分维度对比

| 拆分维度 | 适用场景 | 优点 | 缺点 |
|---------|---------|------|------|
| **业务领域**（DDD） | 首选方式 | 高内聚低耦合，团队自治 | 需要清晰的领域边界 |
| **读写分离** | 读多写少场景 | 提升读性能，独立扩容 | 数据一致性挑战 |
| **冷热数据** | 数据量大的场景 | 降低存储成本 | 查询复杂度增加 |
| **用户维度** | 多租户场景 | 数据隔离好 | 跨用户查询困难 |

### 康威定律

> "设计系统的架构受制于产生这些设计的组织的沟通结构"

```
单体应用                    微服务架构
┌─────────────────┐        ┌─────────┐ ┌─────────┐ ┌─────────┐
│   一个大团队     │   →    │ 订单服务 │ │ 库存服务 │ │ 用户服务 │
│   维护所有代码   │        │  小团队  │ │  小团队  │ │  小团队  │
└─────────────────┘        └─────────┘ └─────────┘ └─────────┘
        ↓                          ↓
   沟通成本高                  团队自治，独立发布
```

### 详细迁移步骤

#### 步骤1：数据库拆分（最复杂）

```java
// 1.1 数据同步（CDC）
@Component
public class DataSynchronizer {
    
    @Autowired
    private CanalClient canalClient;
    
    // 监听MySQL Binlog，实时同步到新库
    @EventListener
    public void onBinlogEvent(CanalEntry entry) {
        if (entry.getTableName().equals("order")) {
            // 同步到订单服务的数据库
            orderSyncService.sync(entry);
        }
    }
}

// 1.2 双写逻辑（兼容期）
@Service
public class OrderService {
    
    @Transactional
    public void createOrder(Order order) {
        // 先写旧库（保证兼容）
        legacyOrderDao.save(order);
        
        // 异步写新库（最终一致）
        asyncExecutor.submit(() -> {
            try {
                newOrderService.save(order);
            } catch (Exception e) {
                // 写入失败队列，后续补偿
                failedSyncQueue.offer(order);
            }
        });
    }
}
```

#### 步骤2：流量灰度切换

```java
// 基于用户ID的灰度路由
@Component
public class GrayRouter {
    
    public ServiceInstance route(String userId, List<ServiceInstance> instances) {
        // 灰度用户走新服务
        if (isGrayUser(userId)) {
            return selectNewService(instances);
        }
        // 普通用户走旧服务
        return selectOldService(instances);
    }
    
    private boolean isGrayUser(String userId) {
        // 1. 白名单用户
        // 2. 按用户ID取模（如 userId % 100 < 5，5%灰度）
        int hash = Math.abs(userId.hashCode()) % 100;
        return hash < 5;  // 5%灰度
    }
}
```

#### 步骤3：数据一致性校验

```java
@Component
public class DataConsistencyChecker {
    
    @Scheduled(fixedRate = 60000)  // 每分钟校验
    public void checkConsistency() {
        // 抽样对比新旧库数据
        List<Long> sampleIds = fetchSampleIds();
        
        for (Long id : sampleIds) {
            Order oldOrder = legacyDao.findById(id);
            Order newOrder = newService.findById(id);
            
            if (!Objects.equals(oldOrder, newOrder)) {
                // 记录不一致，触发补偿
                inconsistencyLog.warn("Data inconsistent: id={}", id);
                compensationService.compensate(id);
            }
        }
    }
}
```

```java
    orderDao.save(order);      // 本地事务

// 微服务：分布式事务（最终一致性）
@Service
public class OrderSagaService {
    
    public void createOrder(OrderCmd cmd) {
        // Saga编排
        sagaOrchestrator.start()
            .step("createOrder", () -> orderService.create(cmd))
            .compensate("cancelOrder", () -> orderService.cancel(cmd.getOrderId()))
            .step("deductInventory", () -> inventoryService.deduct(cmd.getSku()))
            .compensate("releaseInventory", () -> inventoryService.release(cmd.getSku()))
            .execute();
    }
}
```

## 🔥 扩展追问 3：回滚策略

```java
// 一键回滚机制
@Component
public class RollbackManager {
    
    public void rollback(String serviceName) {
        // 1. 切流量回旧服务
        gatewayService.setRouteWeight(serviceName, 0);  // 新服务0%
        
        // 2. 停止双写
        dualWriteManager.disable(serviceName);
        
        // 3. 数据补偿（如有需要）
        compensationService.run(serviceName);
        
        // 4. 通知告警
        alertService.send("服务{}已回滚", serviceName);
    }
}

// 自动熔断回滚
@EventListener
public void onErrorRateHigh(ErrorRateHighEvent event) {
    if (event.getErrorRate() > 0.1) {  // 错误率>10%
        rollbackManager.rollback(event.getServiceName());
    }
}
```

## 一句话收尾

微服务拆分的黄金法则是**"先治域、后治库、再治服"**：先用DDD识别限界上下文，再逐步拆分数据库（双写兼容），最后切流量灰度发布。记住：**"大单体是罪，微服务过度也是罪，找到适合团队的粒度才是关键"**。
