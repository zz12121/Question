## 先说结论

Full GC 停顿 5 秒通常是**老年代空间不足**或**内存泄漏**导致。快速定位三步：**看GC日志确认类型**（是CMS/CMS失败/Major GC还是G1 Mixed）→ **堆Dump分析大对象** → **代码定位内存大户**。解决思路：扩容老年代、优化对象生命周期、消除内存泄漏，长期用G1/ZGC降低停顿。

---

## 🔥 核心概念

**1. 解决什么问题**
- Full GC 导致应用停顿（STW），5秒意味着这段时间服务完全不可用
- 频繁Full GC会拖垮整个系统响应，甚至OOM崩溃

**2. 基本原理/思想**
- **分代收集**：新生代（Young GC快）vs 老年代（Full GC慢）
- **GC根搜索**：从GC Roots遍历所有存活对象，对象越多越慢
- **内存泄漏**：无用对象被长期引用，无法回收，老年代持续增长

### 二、常见Full GC原因及解决

| 场景 | GC日志特征 | 根因 | 解决方案 |
|-----|-----------|------|---------|
| **老年代空间不足** | `[Full GC (Ergonomics)]` | 对象晋升过快/老年代太小 | 增大老年代(-Xmx/-Xmn比例)、优化对象生命周期 |
| **元空间不足** | `[Full GC (Metadata GC Threshold)]` | 动态生成类过多（CGLIB/反射） | 增大Metaspace(`-XX:MaxMetaspaceSize=512m`) |
| **System.gc()** | `[Full GC (System.gc())]` | 代码显式调用（如NIO DirectBuffer） | 加`-XX:+DisableExplicitGC` |
| **CMS失败** | `concurrent mode failure` | 晋升速度超过回收速度 | 调低触发阈值/增大老年代/换G1 |
| **大对象分配** | `[Full GC (Allocation Failure)]` | 超大对象直接进入老年代 | 拆分大对象/增大新生代 |

**典型问题日志：**

```log
2024-04-12T10:23:45.123+0800: 12345.678: [Full GC (Ergonomics) 
  [PSYoungGen: 2048K->0K(458752K)] 
  [ParOldGen: 1398144K->1398144K(1398144K)]  # 老年代满了！
  1400192K->1398144K(1843200K),
```

**关键指标解读：**
- `ParOldGen: 1398144K->1398144K`：老年代回收前后无变化 → **内存泄漏或真的不够用**


**MAT分析步骤：**

```
1. File → Open Heap Dump
2. Overview → Histogram
   查看占用内存最多的类（按Retained Heap排序）
   
3. 右键可疑类 → Merge Shortest Paths to GC Roots
   找到是谁在引用这些对象（泄漏源头）
   
4. Leak Suspects Report
   自动分析最可能的内存泄漏点
```

**常见内存大户：**

| 类 | 可能原因 | 解决 |
|---|---------|------|
| `byte[]` | 大文件缓存、NIO DirectBuffer | 流式处理、限制缓存大小 |
| `char[]` | 大字符串（SQL/JSON/XML） | 分页查询、压缩存储 |
| `HashMap$Node[]` | 超大HashMap/ConcurrentHashMap | 限制容量、本地缓存换Caffeine |
| `Object[]` | ArrayList扩容残留、线程池队列堆积 | 指定初始容量、队列大小限制 |

#### 3. 代码级定位案例

```java
// 典型案例1：静态Map无限增长
@Service
public class OrderService {
    // 内存泄漏！永不被回收
    private static final Map<Long, Order> CACHE = new HashMap<>();
    
    public void process(Order order) {
        CACHE.put(order.getId(), order);  // 只put不remove
    }
}

// 修复：用Caffeine自动淘汰
private final Cache<Long, Order> cache = Caffeine.newBuilder()
    .maximumSize(10000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .build();
```

```java
// 典型案例2：ThreadLocal未清理
public class UserContext {
    private static final ThreadLocal<User> CONTEXT = new ThreadLocal<>();
    
    public void set(User user) {
        CONTEXT.set(user);
    }
    // 缺少remove()！线程池复用线程时导致泄漏
}

// 修复：用完必remove，或用TransmittableThreadLocal
public void clear() {
    CONTEXT.remove();
}
```

```java
public List<Order> queryAll() {
```

```java
/**
 * 对象池复用（针对频繁创建的大对象）

public class ByteBufferPool {
    private static final ThreadLocal<SoftReference<byte[]>> BUFFER_POOL = 
        ThreadLocal.withInitial(() -> new SoftReference<>(new byte[8192]));
    
    public static byte[] getBuffer() {
        byte[] buffer = BUFFER_POOL.get().get();
        if (buffer == null) {
            buffer = new byte[8192];
            BUFFER_POOL.set(new SoftReference<>(buffer));
        }
        return buffer;
    }
}
```

## 🔥 扩展追问 1：G1 GC的Mixed GC是什么？

```
┌─────────────────────────────────────────────────────────────┐
│  G1 GC 设计                                                  │
│  ─────────                                                   │
│  堆划分为多个Region（1~32MB），Eden/Survivor/Old角色动态变化    │
│                                                              │
│  Young GC: 只回收Eden区（快）                                 │
│       ↓                                                      │
│  Concurrent Marking: 后台标记老年代存活对象（并发，不停顿）      │
│       ↓                                                      │
│  Mixed GC: 回收Young + 部分Old（控制停顿时间）                 │
│       ↓                                                      │
│  Full GC: 老年代真的满了，串行回收（慢，要避免）                │
└─────────────────────────────────────────────────────────────┘
```

**频繁Mixed GC调优：**

```bash
# 提高触发并发标记的阈值（默认45%）
-XX:InitiatingHeapOccupancyPercent=35

# 增加Mixed GC回收的Old比例
-XX:G1MixedGCLiveThresholdPercent=85
-XX:G1MixedGCCountTarget=8
```

```yaml
  expr: increase(jvm_gc_collection_count{gc="G1 Old Generation"}[1m]) > 1
  for: 2m
  severity: critical

- alert: LongGCPause
  expr: |
    (
      jvm_gc_collection_time_seconds{gc="G1 Old Generation"} 
      / 
      jvm_gc_collection_count{gc="G1 Old Generation"}
    ) > 1
  severity: warning
  
# 堆内存使用率 > 80%
- alert: HighHeapUsage
  expr: jvm_memory_heap_used_bytes / jvm_memory_heap_max_bytes > 0.8
  severity: warning
```

**Full GC是结果不是原因**——5秒停顿说明老年代已不堪重负，先用`jstat -gcutil`看内存趋势，再用MAT找内存大户，最后根治代码里的缓存泄漏或大对象问题，长期用G1/ZGC把停顿控制在200ms以内。

几个容易踩的坑：
- **System.gc()**：NIO的DirectBuffer会触发，加`-XX:+DisableExplicitGC`屏蔽
- **容器环境**：JDK 8默认不看容器limit，要用`UseCGroupMemoryLimitForHeap`
- **MAT分析**：重点关注`Retained Heap`而不是`Shallow Heap`，找真正占用内存的大户
