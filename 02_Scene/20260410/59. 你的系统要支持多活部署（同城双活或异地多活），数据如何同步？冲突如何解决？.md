## 先说结论

多活部署的核心是**"单元化架构"**：按用户维度划分单元，每个单元内闭环（应用+数据），单元间通过**异步复制**同步。冲突解决采用**"先分区避免，后规则仲裁"**——优先通过用户归属避免冲突，不可避免时用**时间戳/版本号/业务规则**仲裁。

## 🔥 核心概念

### 多活类型对比

| 类型 | 同城双活 | 异地多活 |
|------|---------|---------|
| **距离** | 同城（<50km） | 异地（>1000km） |
| **延迟** | <5ms | 30-100ms |
| **RPO** | 0（同步复制） | 秒级（异步复制） |
| **RTO** | 分钟级 | 分钟级 |
| **数据一致性** | 强一致 | 最终一致 |
| **适用场景** | 机房级容灾 | 城市级容灾 |

### CAP与多活

```
多活场景下，P（分区容错）必选，在C（一致性）和A（可用性）间权衡：

┌─────────────────────────────────────────────────────────────┐
│  同城双活：追求CP（强一致）                                   │
│  - 同步复制，RPO=0                                           │
│  - 写操作需等待所有副本确认                                    │
│  - 牺牲部分可用性（脑裂时拒写）                                │
├─────────────────────────────────────────────────────────────┤
│  异地多活：追求AP（高可用）                                   │
│  - 异步复制，RPO>0                                           │
│  - 本地写成功即返回                                           │
│  - 接受最终一致，需处理冲突                                    │
└─────────────────────────────────────────────────────────────┘
```

```java
/**
 * 单元化路由规则
 */
@Component
public class UnitRouter {
    
    // 用户ID → 单元映射
    private static final Map<String, String> USER_UNIT_MAP = new HashMap<>();
    
    /**
     * 根据用户ID路由到对应单元
     */
    public String route(String userId) {
        // 1. 查路由表
        String unit = USER_UNIT_MAP.get(userId);
        if (unit != null) {
            return unit;
        }
        
        // 2. 按哈希路由（默认）
        int hash = Math.abs(userId.hashCode()) % 2;
        return hash == 0 ? "unit-a" : "unit-b";
    }
    
    /**
     * 网关层路由（根据Cookie/Header）
     */
    public String routeByRequest(HttpServletRequest request) {
        // 优先从Cookie读取
        String userId = getUserIdFromCookie(request);
        return route(userId);
    }
}
```

```java
/**
 * Canal数据同步
 */
@Component
public class MultiSiteSyncService {
    
    @Autowired
    private CanalConnector canalConnector;
    
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    
    @PostConstruct
    public void startSync() {
        canalConnector.connect();
        canalConnector.subscribe(".*\\..*");  // 订阅所有表
        
        while (running) {
            Message message = canalConnector.getWithoutAck(1000);
            
            for (Entry entry : message.getEntries()) {
                // 解析Binlog
                RowChange rowChange = RowChange.parseFrom(entry.getStoreValue());
                
                // 转换为同步消息
                SyncMessage syncMsg = new SyncMessage();
                syncMsg.setDatabase(entry.getHeader().getSchemaName());
                syncMsg.setTable(entry.getHeader().getTableName());
                syncMsg.setEventType(rowChange.getEventType().toString());
                syncMsg.setData(parseRowData(rowChange));
                syncMsg.setTimestamp(System.currentTimeMillis());
                
                // 发送到Kafka（按表分区，保证顺序）
                String topic = "sync-" + entry.getHeader().getTableName();
                kafkaTemplate.send(topic, entry.getHeader().getSchemaName(), 
                    jsonMapper.writeValueAsString(syncMsg));
            }
            
            canalConnector.ack(message.getId());
        }
    }
}
```

## 🔥 详细展开：冲突解决

### 冲突类型

```
┌─────────────────────────────────────────────────────────────┐
│  1. 主键冲突（插入冲突）                                       │
│     场景：两个单元同时插入相同ID的记录                          │
│     解决：全局唯一ID（雪花算法）                                │
├─────────────────────────────────────────────────────────────┤
│  2. 更新冲突（数据不一致）                                     │
│     场景：同一记录在不同单元被同时修改                          │
│     解决：时间戳/版本号仲裁                                    │
├─────────────────────────────────────────────────────────────┤
│  3. 外键冲突（关联数据不一致）                                  │
│     场景：订单和库存跨单元，状态不一致                          │
│     解决：业务补偿、对账修复                                   │
└─────────────────────────────────────────────────────────────┘
```

### 冲突解决策略

#### 策略1：时间戳优先（Last-Write-Wins）

```java
/**
 * 时间戳冲突解决
 */
@Component
public class TimestampConflictResolver {
    
    public <T extends TimestampedEntity> T resolve(T local, T remote) {
        // 时间戳大的覆盖小的
        if (remote.getTimestamp() > local.getTimestamp()) {
            return remote;  // 接受远程数据
        }
        return local;  // 保留本地数据
    }
}

// 表结构设计
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,
    status INT,
    data VARCHAR(1000),
    version INT DEFAULT 0,      -- 版本号
    update_time BIGINT,          -- 更新时间戳（毫秒）
    unit_id VARCHAR(20)          -- 所属单元
);
```

#### 策略2：版本号仲裁（Vector Clock）

```java
/**
 * 向量时钟冲突检测
 */
public class VectorClock {
    private Map<String, Integer> clocks = new HashMap<>();
    
    // 更新时递增本地时钟
    public void increment(String unitId) {
        clocks.merge(unitId, 1, Integer::sum);
    }
    
    // 比较两个向量时钟
    public CompareResult compare(VectorClock other) {
        boolean greater = false, less = false;
        
        for (String unit : clocks.keySet()) {
            int v1 = clocks.getOrDefault(unit, 0);
            int v2 = other.clocks.getOrDefault(unit, 0);
            
            if (v1 > v2) greater = true;
            if (v1 < v2) less = true;
        }
        
        if (greater && !less) return CompareResult.GREATER;
        if (!greater && less) return CompareResult.LESS;
        if (!greater && !less) return CompareResult.EQUAL;
        return CompareResult.CONCURRENT;  // 并发冲突
    }
}
```

#### 策略3：业务规则仲裁

```java
/**
 * 业务规则冲突解决（以库存为例）
 */
@Component
public class InventoryConflictResolver {
    
    public Inventory resolve(Inventory local, Inventory remote) {
        // 规则1：库存扣减优先（保证不超卖）
        if (local.getQuantity() < remote.getQuantity()) {
            return local;  // 本地库存更少，优先
        }
        
        // 规则2：预占库存优先
        if (local.getReserved() > remote.getReserved()) {
            return local;
        }
        
        // 默认：时间戳优先
        return local.getUpdateTime() > remote.getUpdateTime() ? local : remote;
    }
}
```

```java
@Component
public class CrossUnitReconciliation {
    
    @Scheduled(cron = "0 0 2 * * ?")  // 每天凌晨2点对账
    public void reconcile() {
        // 1. 获取昨日所有变更记录

        List<Order> unitBOrders = fetchUnitBOrders(yesterday);
        
        // 2. 比对差异
        Map<String, OrderDiff> diffs = compare(unitAOrders, unitBOrders);
        
        for (OrderDiff diff : diffs.values()) {
            if (diff.isCritical()) {
                // 关键差异，立即告警
                alertService.send("Critical data inconsistency: " + diff);
            }
            
            // 3. 自动修复（按规则）
            if (canAutoFix(diff)) {
                fix(diff);
            } else {
                // 人工介入
                createTicket(diff);
            }
        }
    }
}
```

```java
@Component
public class UnitFailover {
    
    /**
     * 单元A故障，流量切换到单元B
     */
    public void failover(String failedUnit) {
        // 1. 停止向故障单元写流量
        trafficController.block(failedUnit);
        
        // 2. 修改路由规则（故障单元用户→备用单元）
        List<String> users = getUsersInUnit(failedUnit);
        for (String userId : users) {
            routeService.updateRoute(userId, "unit-b");
        }
        
        // 3. 异步数据补偿（故障单元恢复后）
        compensationService.scheduleSync(failedUnit);
    }
}
```

## 一句话收尾

多活部署的核心是**"单元化隔离+异步复制+冲突仲裁"**：按用户维度划分单元实现闭环，通过Canal/Otter等工具异步同步数据，用时间戳/版本号/业务规则解决冲突，配合对账机制保证最终一致。记住**"能分区解决的冲突，就不要仲裁解决"**，良好的单元划分是避免冲突的最佳实践。
