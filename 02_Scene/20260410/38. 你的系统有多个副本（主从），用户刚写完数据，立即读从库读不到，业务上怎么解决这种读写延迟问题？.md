## 先说结论

主从延迟导致"写后读不到"，本质是**主从复制延迟**。解决方案：**强一致性读走主库**、**缓存补偿**、**延迟双删**、或**业务容忍短暂不一致**；推荐做法是**关键业务强制读主库，非关键业务读从库+缓存兜底**。

## 🔥 核心概念

### 主从延迟原因

```
┌─────────────────────────────────────────────────────────┐
│                  主从复制延迟来源                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  主库              网络传输              从库            │
│  ┌─────┐         ┌──────┐            ┌─────┐          │
│  │ 写  │ ───────→│ 中继  │ ─────────→ │ 应用 │          │
│  │ Binlog│         日志   │            │ Relay │          │
│  └─────┘         └──────┘            │ Log   │          │
│     │                                  │     │          │
│     │  1. 主库写 Binlog（毫秒级）        │     │          │
│     │                                  │     │          │
│     │  2. 网络传输延迟（同机房<1ms，      │     │          │
│     │     跨机房 10-100ms）              │     │          │
│     │                                  │     │          │
│     │  3. 从库 SQL 线程串行应用          │     │          │
│     │     （大事务可能延迟秒级）          │     │          │
│     │                                  ▼     │          │
│     │                            ┌─────────┐ │          │
│     └──────────────────────────→ │ 数据可见 │ │          │
│                                  │ 有延迟！ │ │          │
│                                  └─────────┘ │          │
│                                                         │
│  延迟时间：正常 1ms ~ 1s，异常时可达秒级甚至分钟级          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 方案一：强制读主库（强一致性）

```java
@Service
public class UserService {
    
    @Autowired
    @Qualifier("masterDataSource")
    private DataSource masterDataSource;
    
    @Autowired
    @Qualifier("slaveDataSource")
    private DataSource slaveDataSource;
    
    /**
     * 注册后查询：强制走主库
     */
    public User getUserAfterRegister(Long userId) {
        // 注册后立即查询，必须读主库
        try (Connection conn = masterDataSource.getConnection()) {
            return queryUser(conn, userId);
        }
    }
    
    /**
     * 普通查询：走从库
     */
    public User getUser(Long userId) {
        try (Connection conn = slaveDataSource.getConnection()) {
            return queryUser(conn, userId);
        }
    }
}
```

### 方案二：缓存补偿（推荐）

```
┌─────────────────────────────────────────────────────────┐
│                  缓存补偿方案                              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  写操作：                                                │
│  1. 写主库                                               │
│  2. 删缓存（或更新缓存）                                  │
│  3. 返回成功                                             │
│                                                         │
│  读操作：                                                │
│  1. 先读缓存                                             │
│     ├── 命中 → 直接返回（一定是最新）                      │
│     └── 未命中 → 读从库 → 回填缓存                        │
│                                                         │
│  优势：                                                  │
│  - 写后读一定命中缓存，不受主从延迟影响                    │
│  - 读从库压力减轻                                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

```java
@Service
public class UserServiceWithCache {
    
    @Autowired
    private UserMapper userMapper;
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    private static final long CACHE_TTL = 30;  // 30分钟
    
    /**
     * 注册：写主库 + 删缓存
     */
    @Transactional
    public User register(RegisterRequest request) {
        // 1. 写主库
        User user = new User();
        user.setUsername(request.getUsername());
        userMapper.insert(user);
        
        // 2. 删除缓存（或设置缓存）
        String cacheKey = "user:" + user.getId();
        redisTemplate.delete(cacheKey);
        
        // 3. 可选：直接写缓存（保证立即可读）
        redisTemplate.opsForValue().set(
            cacheKey, 
            JSON.toJSONString(user),
            CACHE_TTL, 
            TimeUnit.MINUTES
        );
        
        return user;
    }
    
    /**
     * 查询：先读缓存，再读从库
     */
    public User getUser(Long userId) {
        String cacheKey = "user:" + userId;
        
        // 1. 读缓存
        String cached = redisTemplate.opsForValue().get(cacheKey);
        if (StringUtils.isNotBlank(cached)) {
            return JSON.parseObject(cached, User.class);
        }
        
        // 2. 缓存未命中，读从库
        User user = userMapper.findById(userId);
        
        // 3. 回填缓存
        if (user != null) {
            redisTemplate.opsForValue().set(
                cacheKey,
                JSON.toJSONString(user),
                CACHE_TTL,
                TimeUnit.MINUTES
            );
        }
        
        return user;
    }
}
```

### 方案三：延迟双删

```
┌─────────────────────────────────────────────────────────┐
│                  延迟双删策略                              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  问题：先删缓存 → 写主库 → 从库同步前，                   │
│        并发读从库旧数据回填缓存，导致缓存脏数据             │
│                                                         │
│  解决：延迟双删                                          │
│                                                         │
│  1. 先删缓存                                             │
│  2. 写主库                                               │
│  3. 延迟一段时间（如500ms）                               │
│  4. 再删一次缓存（确保从库同步完成后的脏缓存被删）          │
│                                                         │
│  延迟时间 > 主从延迟时间（通常 500ms ~ 1s）                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

```java
@Service
public class DelayDoubleDeleteService {
    
    @Autowired
    private UserMapper userMapper;
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private ThreadPoolExecutor executor;  // 异步执行延迟删除
    
    /**
     * 更新用户：延迟双删
     */
    @Transactional
    public void updateUser(User user) {
        String cacheKey = "user:" + user.getId();
        
        // 1. 先删缓存
        redisTemplate.delete(cacheKey);
        
        // 2. 写主库
        userMapper.update(user);
        
        // 3. 延迟删除（异步）
        executor.execute(() -> {
            try {
                Thread.sleep(500);  // 延迟500ms，确保从库同步完成
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
            // 4. 再次删除缓存
            redisTemplate.delete(cacheKey);
        });
    }
}
```

### 方案四：业务层等待/重试

```java
@Service
public class RetryReadService {
    
    @Autowired
    private UserMapper userMapper;
    
    /**
     * 带重试的查询（简单场景用）
     */
    public User getUserWithRetry(Long userId, int maxRetry) {
        for (int i = 0; i < maxRetry; i++) {
            User user = userMapper.findById(userId);
            if (user != null) {
                return user;
            }
            
            // 没查到，等待后重试
            try {
                Thread.sleep(100);  // 等待100ms
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
        
        // 重试多次仍查不到，可能真的不存在
        return null;
    }
    
    /**
     * 注册后查询（业务层等待）
     */
    public User getUserAfterRegister(Long userId) {
        // 注册后立即查询，最多等 500ms
        return getUserWithRetry(userId, 5);
    }
}
```

## 🔥 扩展追问 1：方案对比与选型

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|---------|
| **强制读主库** | 强一致 | 主库压力大 | 低 | 关键业务（支付、订单状态）|
| **缓存补偿** | 强一致 | 高 | 中 | **推荐，通用方案** |
| **延迟双删** | 最终一致 | 高 | 中 | 高并发写场景 |
| **业务重试** | 强一致 | 低（有等待） | 低 | 简单场景，延迟敏感 |
| **业务容忍** | 最终一致 | 最高 | 低 | 非关键数据 |

## 🔥 扩展追问 2：如何监控主从延迟？

```sql
-- MySQL 查看主从延迟（从库执行）
SHOW SLAVE STATUS\G

-- 关键字段：
-- Seconds_Behind_Master: 从库落后主库的秒数
-- Master_Log_File / Read_Master_Log_Pos: 主库 binlog 位置
-- Relay_Log_File / Relay_Log_Pos: 从库 relay log 位置

-- 如果 Seconds_Behind_Master = NULL，表示复制已停止
```

```java
// 应用层监控
@Component
public class ReplicationLagMonitor {
    
    @Autowired
    private SlaveDataSource slaveDataSource;
    
    @Scheduled(fixedRate = 60000)  // 每分钟检查
    public void checkLag() {
        try (Connection conn = slaveDataSource.getConnection();
             Statement stmt = conn.createStatement();
             ResultSet rs = stmt.executeQuery("SHOW SLAVE STATUS")) {
            
            if (rs.next()) {
                long lag = rs.getLong("Seconds_Behind_Master");
                
                if (lag > 5) {  // 延迟超过5秒告警
                    alertService.send(AlertLevel.P1, 
                        "主从延迟告警: " + lag + "秒");
                }
            }
        }
    }
}
```

## 一句话收尾

主从延迟的**根本解法是避免写后立即读从库**：关键业务**强制读主库**或**缓存补偿**保证强一致，非关键业务可**容忍最终一致**；生产环境推荐**缓存+延迟双删**组合方案，配合**主从延迟监控**及时发现异常。
