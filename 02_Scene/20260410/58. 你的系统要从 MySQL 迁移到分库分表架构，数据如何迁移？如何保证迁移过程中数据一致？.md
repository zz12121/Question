## 先说结论

MySQL迁移分库分表的核心是**"双写过渡"**：通过**全量迁移+增量同步+实时校验+灰度切流**四步走，实现平滑迁移。一致性保证依赖**数据双写+对账补偿**，最终达到新旧库数据一致后切换读流量。

---

## 🔥 核心概念

### 分库分表 vs 单体MySQL

| 对比项 | 单体MySQL | 分库分表（Sharding） |
|-------|-----------|---------------------|
| **数据量** | < 5000万 | > 5000万或快速增长 |
| **并发** | < 1000 QPS | > 1000 QPS |
| **扩容** | 垂直扩容（硬件升级） | 水平扩容（加节点） |
| **复杂度** | 低 | 高（分布式事务、跨库查询） |
| **维护成本** | 低 | 高 |

### 分片策略选择

```java
// 1. 哈希分片（推荐：数据分布均匀）
shard = hash(user_id) % 8  // 8个分片

// 2. 范围分片（适合时间序列）
shard = create_time.month  // 按月分表

// 3. 混合分片（哈希+范围）
// 先按user_id哈希到库，再按时间范围分表
```

### 阶段1：全量迁移（离线）

```java
/**
 * 全量数据迁移
 * 使用分页查询 + 批量插入，避免内存溢出
 */
@Component
public class FullDataMigrator {
    
    @Autowired
    private JdbcTemplate oldJdbcTemplate;
    
    @Autowired
    private ShardingDataSource newDataSource;
    
    public void migrate(String tableName) {
        long lastId = 0;
        int batchSize = 1000;
        
        while (true) {
            // 分页查询旧库
            List<Order> orders = oldJdbcTemplate.query(
                "SELECT * FROM " + tableName + " WHERE id > ? ORDER BY id LIMIT ?",
                new BeanPropertyRowMapper<>(Order.class),
                lastId, batchSize
            );
            
            if (orders.isEmpty()) {
                break;
            }
            
            // 批量写入新库（分片路由自动处理）
            for (Order order : orders) {
                String sql = "INSERT INTO " + tableName + " (...) VALUES (...)";
                // 根据分片键路由到对应分片
                newDataSource.getConnection(order.getUserId())
                    .prepareStatement(sql)
                    .executeUpdate();
            }
            
            lastId = orders.get(orders.size() - 1).getId();
            
            // 记录迁移进度（断点续传）
            migrationProgressService.save(tableName, lastId);
        }
    }
}
```

### 阶段2：增量同步（实时）

```java
/**
 * 基于Canal的Binlog实时同步
 */
@Component
public class CanalSyncService {
    
    @Autowired
    private CanalConnector canalConnector;
    
    @PostConstruct
    public void startSync() {
        canalConnector.connect();
        canalConnector.subscribe("db.order");  // 订阅订单表
        
        while (running) {
            Message message = canalConnector.getWithoutAck(100);
            for (Entry entry : message.getEntries()) {
                // 解析Binlog事件
                RowChange rowChange = RowChange.parseFrom(entry.getStoreValue());
                
                for (RowData rowData : rowChange.getRowDatasList()) {
                    // 根据事件类型处理
                    switch (rowChange.getEventType()) {
                        case INSERT:
                            handleInsert(rowData.getAfterColumnsList());
                            break;
                        case UPDATE:
                            handleUpdate(rowData.getAfterColumnsList());
                            break;
                        case DELETE:
                            handleDelete(rowData.getBeforeColumnsList());
                            break;
                    }
                }
            }
            canalConnector.ack(message.getId());  // 确认消费
        }
    }
    
    private void handleInsert(List<Column> columns) {
        Order order = convertToOrder(columns);
        // 写入新库（幂等处理，避免重复）
        newOrderService.saveWithIdempotency(order);
    }
}
```

```java
/**
 * 数据一致性校验
 */
@Component
public class DataConsistencyChecker {
    
    @Scheduled(fixedRate = 300000)  // 每5分钟执行
    public void checkConsistency() {
        // 1. 抽样校验（最近1小时的数据）
        List<Long> sampleIds = fetchRecentIds(1, TimeUnit.HOURS);
        
        for (Long id : sampleIds) {
            Order oldOrder = oldOrderDao.findById(id);
            Order newOrder = newOrderDao.findById(id);
            
            if (!isConsistent(oldOrder, newOrder)) {
                // 记录不一致
                inconsistencyLog.warn("Inconsistent: id={}", id);
                
                // 触发补偿
                compensationService.compensate(id);
            }
        }
    }
    
    /**
     * 全量校验（低峰期执行）
     */
    public void fullCheck() {
        // 使用Checksum比对
        String oldChecksum = oldOrderDao.calculateChecksum();
        String newChecksum = newOrderDao.calculateChecksum();
        
        if (!oldChecksum.equals(newChecksum)) {
            // 逐条比对找出差异
            findAndFixDifferences();
        }
    }
}
```

### 阶段4：灰度切流

```java
/**
 * 读写流量切换
 */
@Component
public class TrafficSwitcher {
    
    private volatile boolean readFromNew = false;
    private volatile boolean writeToOld = true;
    private volatile boolean writeToNew = true;
    
    /**
     * 阶段1：双写（旧库主读，新旧双写）
     */
    public void phase1_DualWrite() {
        writeToOld = true;
        writeToNew = true;
        readFromNew = false;
    }
    
    /**
     * 阶段2：灰度读（部分流量读新库）
     */
    public void phase2_GrayRead(int grayPercent) {
        // 根据用户ID灰度
        readFromNew = (userId % 100) < grayPercent;
    }
    
    /**
     * 阶段3：全量读（全部流量读新库，旧库只写）
     */
    public void phase3_FullRead() {
        readFromNew = true;
        writeToOld = true;   //  still write to old for rollback
        writeToNew = true;
    }
    
    /**
     * 阶段4：单写（只写新库）
     */
    public void phase4_SingleWrite() {
        readFromNew = true;
        writeToOld = false;  //  stop writing to old
        writeToNew = true;
    }
    
    // DAO层使用
    public void save(Order order) {
        if (writeToOld) {
            oldOrderDao.save(order);
        }
        if (writeToNew) {
            newOrderDao.save(order);
        }
    }
    
    public Order findById(Long id) {
        if (readFromNew) {
            return newOrderDao.findById(id);
        }
        return oldOrderDao.findById(id);
    }
}
```

## 🔥 扩展追问 2：数据不一致处理

### 不一致原因

```
1. 全量迁移期间的数据变更丢失
   → 解决方案：全量迁移前记录Binlog位点，增量从该位点开始
   
2. 增量同步延迟
   → 解决方案：同步延迟监控，延迟过大时暂停切流
   
3. 双写失败（新库写入失败）
   → 解决方案：失败队列+重试+告警
   
4. 主从延迟（如果新库有从库）
   → 解决方案：读主库或强制走主库Hint
```

### 补偿机制

```java
/**
 * 数据补偿服务
 */
@Component
public class CompensationService {
    
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    
    /**
     * 发现不一致时，发送补偿消息
     */
    public void compensate(Long orderId) {
        // 发送到补偿队列
        kafkaTemplate.send("compensation-topic", 
            jsonMapper.writeValueAsString(new CompensateMsg(orderId)));
    }
    
    /**
     * 消费补偿消息
     */
    @KafkaListener(topics = "compensation-topic")
    public void onCompensate(CompensateMsg msg) {
        // 1. 从旧库读取最新数据
        Order source = oldOrderDao.findById(msg.getOrderId());
        
        // 2. 覆盖写入新库
        newOrderDao.save(source);
        
        // 3. 再次校验
        if (!checkConsistent(msg.getOrderId())) {
            // 告警，人工介入
            alertService.send("Compensation failed: " + msg.getOrderId());
        }
    }
}
```

```java
/**
 * 一键回滚机制
 */
@Component
public class MigrationRollback {
    
    public void rollback() {
        // 1. 切回读旧库
        trafficSwitcher.setReadFromNew(false);
        
        // 2. 停止写新库
        trafficSwitcher.setWriteToNew(false);
        
        // 3. 检查旧库数据完整性
        if (!checkOldDbIntegrity()) {
            // 从备份恢复
            restoreFromBackup();
        }
        
        // 4. 通知告警
        alertService.send("Migration rolled back!");
    }
    
    /**
     * 自动回滚触发条件
     */
    @EventListener
    public void onErrorRateHigh(ErrorRateHighEvent event) {
        if (event.getErrorRate() > 0.05) {  // 错误率>5%
            rollback();
        }
    }
}
```

## 一句话收尾

分库分表迁移的核心是**"离线全量+实时增量+双写过渡"**：先用Canal等工具做历史数据全量迁移，再通过Binlog实时同步增量数据，期间双写保证一致性，最后灰度切流。记住**"宁可慢不可错"**，每个阶段都要有回滚能力和数据校验机制。
