## 先说结论

"重复下单"本质是**幂等性问题**，常见原因是：前端重复点击、网络超时重试、MQ 消息重投、客户端异常重连。防止重复提交的核心方案是**唯一标识 + 幂等校验**，通过 Token 机制或业务唯一键在最外层拦截。

### 方案二：Token 机制（推荐）

```
┌─────────────────────────────────────────────────────────┐
│                  Token 幂等机制流程                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 打开下单页面                                         │
│     ├── 前端请求：GET /order/token                       │
│     ├── 后端生成 UUID，存入 Redis（TTL = 10分钟）         │
│     └── 返回 token 给前端                                │
│                                                         │
│  2. 提交订单                                             │
│     ├── 前端携带 token 提交：Header: Idempotency-Key     │
│     │                                                   │
│     ├── 后端校验 token（Lua 原子 get+del）：             │
│     │   ├── token 存在 → 删除 + 继续处理                 │
│     │   └── token 不存在 → 返回"重复提交"                │
│     │                                                   │
│     └── 处理完成                                         │
│                                                         │
│  3. 网络超时重试                                         │
│     ├── 前端重试，但 token 已被删除                       │
│     └── 后端直接返回"重复提交"，不再处理                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**代码实现：**

```java
// 1. 生成 Token 接口
@RestController
public class IdempotentController {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @GetMapping("/order/token")
    public String getToken(@CurrentUser Long userId) {
        String token = UUID.randomUUID().toString().replace("-", "");
        String key = "idempotent:order:" + userId + ":" + token;
        redisTemplate.opsForValue().set(key, "1", 10, TimeUnit.MINUTES);
        return token;
    }
}

// 2. 幂等性 AOP 切面
@Aspect
@Component
public class IdempotentAspect {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    private static final DefaultRedisScript<Long> TOKEN_SCRIPT;
    static {
        TOKEN_SCRIPT = new DefaultRedisScript<>();
        // GET + DEL 原子操作：存在则删除返回1，不存在返回0
        TOKEN_SCRIPT.setScriptText(
            "if redis.call('get', KEYS[1]) == ARGV[1] then " +
            "    return redis.call('del', KEYS[1]) " +
            "else " +
            "    return 0 " +
            "end"
        );
        TOKEN_SCRIPT.setResultType(Long.class);
    }
    
    @Around("@annotation(Idempotent)")
    public Object check(ProceedingJoinPoint pjp) throws Throwable {
        HttpServletRequest request = getCurrentRequest();
        String token = request.getHeader("Idempotency-Key");
        Long userId = getCurrentUserId();
        
        if (StringUtils.isBlank(token)) {
            throw new BusinessException("缺少幂等 Token");
        }
        
        String key = "idempotent:order:" + userId + ":" + token;
        
        // 原子 check + delete
        Long result = redisTemplate.execute(TOKEN_SCRIPT, 
            Collections.singletonList(key), "1");
        
        if (result == null || result == 0L) {
            throw new BusinessException(ErrorCode.REPEAT_SUBMIT, "请勿重复提交");
        }
        
        return pjp.proceed();
    }
}

// 3. 业务接口使用
@PostMapping("/order/create")
@Idempotent  // 加注解即可
public OrderVO createOrder(@RequestBody CreateOrderRequest request) {
    return orderService.createOrder(request);
}
```

### 方案四：Redis SETNX 分布式锁

```java
@Service
public class OrderService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    public OrderVO createOrder(CreateOrderRequest request) {
        // 业务唯一键：userId + activityId
        String lockKey = "lock:order:" + request.getUserId() 
                       + ":" + request.getActivityId();
        
        // SETNX + TTL（原子操作）
        Boolean locked = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, "1", 30, TimeUnit.SECONDS);
        
        if (Boolean.FALSE.equals(locked)) {
            // 已有相同请求在处理
            throw new BusinessException("请勿重复提交");
        }
        
        try {
            // 二次校验：DB 中是否已存在
            if (orderMapper.existsByUserAndActivity(
                    request.getUserId(), request.getActivityId())) {
                // 幂等返回已有订单
                return convert(orderMapper.findByUserAndActivity(
                    request.getUserId(), request.getActivityId()));
            }
            
            // 创建订单
            return doCreateOrder(request);
            
        } finally {
            redisTemplate.delete(lockKey);  // 释放锁
        }
    }
}
```

### 方案五：MQ 消费幂等

```java
@Component
@RocketMQMessageListener(topic = "order-topic", consumerGroup = "order-group")
public class OrderConsumer implements RocketMQListener<OrderMessage> {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private OrderMapper orderMapper;
    
    @Override
    public void onMessage(OrderMessage message) {
        String msgId = message.getMsgId();  // MQ 消息唯一 ID
        String consumeKey = "consumed:order:" + msgId;
        
        // 方式1：Redis SETNX 判重
        Boolean isFirst = redisTemplate.opsForValue()
            .setIfAbsent(consumeKey, "1", 24, TimeUnit.HOURS);
        
        if (Boolean.FALSE.equals(isFirst)) {
            log.info("消息已消费，跳过: {}", msgId);
            return;  // 幂等跳过
        }
        
        try {
            processOrder(message);
        } catch (Exception e) {
            // 消费失败，删除标记，允许重试
            redisTemplate.delete(consumeKey);
            throw e;
        }
    }
}
```

## 🔥 扩展追问 2：幂等性设计原则

```
┌─────────────────────────────────────────────────────────┐
│                  幂等性 vs 重试安全                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  幂等：同一请求执行多次，结果与执行一次相同               │
│                                                         │
│  HTTP 方法的天然幂等性：                                  │
│  ├── GET    → 天然幂等（查询不改变状态）                  │
│  ├── DELETE → 幂等（删两次和删一次结果一样）              │
│  ├── PUT    → 幂等（整体替换）                           │
│  └── POST   → 非幂等（需要主动设计幂等）                  │
│                                                         │
│  业务层幂等设计：                                         │
│  ├── 使用业务唯一键（非 ID）                              │
│  ├── 状态机检查（已处理不再处理）                          │
│  └── 幂等返回（重复时返回已有结果，不报错）                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 3：如何排查线上重复下单问题？

```
排查步骤：

1. 查日志 → 找两条相同订单的请求日志
   ├── 时间差多少？（毫秒级=前端重复点击，秒级=超时重试）
   ├── 请求 IP 是否相同？（相同=同一客户端，不同=网关重试）
   └── traceId 是否相同？（相同=同一请求链路，不同=两次独立请求）

2. 看 orderNo 生成逻辑
   ├── 两个订单的 orderNo 是否相同？
   └── orderNo 是否包含唯一性因素（userId + timestamp + random）

3. 检查 DB 唯一索引
   └── 有没有唯一索引？有没有触发 DuplicateKeyException？

4. 检查 Feign/HTTP 重试配置
   └── ribbon.MaxAutoRetries / feign.client.config.default.retryer

5. 检查 MQ 消费逻辑
   └── 是否有幂等消费？消费者重启时是否会重消费？
```

## 一句话收尾

重复下单的根本原因是**接口缺乏幂等性**，最佳实践是**双重保障**：业务层用 **Token 机制**（一次性令牌）拦截重复请求，数据层用 **DB 唯一索引**兜底防止数据污染；MQ 消费场景额外需要**消费幂等标记**，确保消息重投时不重复处理。
