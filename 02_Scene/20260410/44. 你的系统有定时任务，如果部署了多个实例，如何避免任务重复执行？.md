## 先说结论

多实例定时任务防重复，核心思路是**分布式锁抢执行权**——只有抢到锁的实例才能执行任务。常用方案：轻量级用Redis SETNX/Redisson，强一致用DB唯一索引，复杂调度用Elastic-Job/XXL-JOB等分布式调度框架，最省事直接上ShedLock注解驱动。

---

## 🔥 核心概念

**1. 解决什么问题**
- 单机定时任务（@Scheduled）部署多实例后，每个实例都会触发，导致任务重复执行
- 重复执行可能造成数据混乱（重复发送通知、重复对账、重复扣款等）

**2. 基本原理/思想**
- **互斥执行**：同一时刻只有一个实例在执行任务
- **锁超时**：防止某个实例挂掉后锁永远不解，其他实例无法执行
- **可重入**：同一个实例内线程安全，避免死锁

### 二、具体实现方案

#### 方案1：Redis SETNX（最轻量）

```java
@Component
public class OrderCheckJob {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    private static final String LOCK_KEY = "job:order:check";
    private static final long LOCK_TTL = 60; // 锁60秒
    
    @Scheduled(cron = "0 0 2 * * ?") // 每天2点执行
    public void checkTimeoutOrders() {
        String lockValue = UUID.randomUUID().toString();
        
        // 1. 尝试获取锁
        Boolean locked = redisTemplate.opsForValue()
            .setIfAbsent(LOCK_KEY, lockValue, LOCK_TTL, TimeUnit.SECONDS);
        
        if (!Boolean.TRUE.equals(locked)) {
            log.info("未获取到锁，其他实例正在执行");
            return;
        }
        
        try {
            log.info("获取锁成功，开始执行订单超时检查");
            doCheck();
        } finally {
            // 2. 释放锁（先判断是不是自己加的）
            String currentValue = redisTemplate.opsForValue().get(LOCK_KEY);
            if (lockValue.equals(currentValue)) {
                redisTemplate.delete(LOCK_KEY);
            }
        }
    }
}
```

**问题**：判断+删除不是原子操作，可能误删别人的锁。

**优化：Lua脚本原子释放**

```java
@Component
public class RedisLockJob {
    
    private static final String RELEASE_SCRIPT = 
        "if redis.call('get', KEYS[1]) == ARGV[1] then " +
        "    return redis.call('del', KEYS[1]) " +
        "else " +
        "    return 0 " +
        "end";
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    public boolean tryLock(String key, String value, long seconds) {
        return Boolean.TRUE.equals(
            redisTemplate.opsForValue()
                .setIfAbsent(key, value, seconds, TimeUnit.SECONDS)
        );
    }
    
    public void unlock(String key, String value) {
        redisTemplate.execute(
            new DefaultRedisScript<>(RELEASE_SCRIPT, Long.class),
            Collections.singletonList(key),
            value
        );
    }
}
```

#### 方案2：Redisson（推荐，生产首选）

```java
@Component
public class RedissonJob {
    
    @Autowired
    private RedissonClient redissonClient;
    
    @Scheduled(cron = "0 0 2 * * ?")
    public void dailyReport() {
        RLock lock = redissonClient.getLock("job:daily:report");
        
        try {
            // 尝试获取锁，等待10秒，锁自动续期30秒
            boolean locked = lock.tryLock(10, 30, TimeUnit.SECONDS);
            if (!locked) {
                log.info("获取锁失败，跳过执行");
                return;
            }
            
            log.info("获取锁成功，执行日报生成");
            generateReport();
            
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            // 只有当前线程持有锁才释放
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

**Redisson优势：**
- **看门狗自动续期**：业务执行时间长时自动续期，避免锁过期
- **可重入锁**：同一线程可多次获取锁
- **红锁RedLock**：多Redis节点部署时保证强一致

#### 方案3：数据库唯一索引（强一致）

```sql
-- 任务锁表
CREATE TABLE distributed_lock (
    lock_name VARCHAR(64) PRIMARY KEY COMMENT '锁名称',
    lock_holder VARCHAR(64) NOT NULL COMMENT '持有者标识',
    expire_time DATETIME NOT NULL COMMENT '过期时间',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB;
```

```java
@Component
public class DBLockJob {
    
    @Autowired
    private DistributedLockMapper lockMapper;
    
    private final String instanceId = InetAddress.getLocalHost().getHostName() 
        + ":" + System.currentTimeMillis();
    
    @Scheduled(cron = "0 0 2 * * ?")
    @Transactional
    public void dataSync() {
        String lockName = "job:data:sync";
        LocalDateTime expireTime = LocalDateTime.now().plusMinutes(5);
        
        // 1. 尝试插入（利用唯一索引）
        int inserted = lockMapper.tryLock(lockName, instanceId, expireTime);
        if (inserted == 0) {
            // 插入失败，检查是否过期
            DistributedLock lock = lockMapper.selectByLockName(lockName);
            if (lock != null && lock.getExpireTime().isBefore(LocalDateTime.now())) {
                // 锁已过期，强制抢占
                lockMapper.forceLock(lockName, instanceId, expireTime);
                log.info("抢占过期锁成功");
            } else {
                log.info("锁被其他实例持有，跳过执行");
                return;
            }
        }
        
        try {
            log.info("获取锁成功，instanceId={}", instanceId);
            doSync();
        } finally {
            // 只有持有者才能释放
            lockMapper.unlock(lockName, instanceId);
        }
    }
}
```

```xml
<!-- Mapper -->
<insert id="tryLock">
    INSERT INTO distributed_lock (lock_name, lock_holder, expire_time)
    VALUES (#{lockName}, #{holder}, #{expireTime})
</insert>

<update id="forceLock">
    UPDATE distributed_lock 
    SET lock_holder = #{holder}, expire_time = #{expireTime}
    WHERE lock_name = #{lockName}
</update>

<delete id="unlock">
    DELETE FROM distributed_lock 
    WHERE lock_name = #{lockName} AND lock_holder = #{holder}
</delete>
```

## 🔥 扩展追问 1：任务执行时间超过定时间隔怎么办？

```
场景：任务每5分钟执行一次，但执行需要10分钟

问题：
- 实例A获取锁，开始执行（预计10分钟）
- 5分钟后，实例B尝试获取锁，发现锁未释放
- 结果：本次调度被跳过，下次调度时实例A可能还没执行完

解决方案：
```

```java
@Component
public class LongRunningJob {
    
    @Autowired
    private RedissonClient redissonClient;
    
    // 方案1：延长锁时间，确保覆盖执行周期
    @Scheduled(fixedDelay = 5 * 60 * 1000) // 上次执行完5分钟后再执行
    public void process() {
        RLock lock = redissonClient.getLock("job:long:task");
        
        // 锁30分钟，确保覆盖执行时间
        if (lock.tryLock(0, 30, TimeUnit.MINUTES)) {
            try {
                doLongTask(); // 可能执行10分钟
            } finally {
                lock.unlock();
            }
        }
    }
    
    // 方案2：使用fixedDelay而非cron
    // fixedDelay = 上次执行结束时间 + 间隔
    // cron = 固定时间点触发，不管上次是否执行完
}
```

**关键区别：**
- `cron`：固定时间点触发，可能堆积
- `fixedRate`：固定频率触发，可能重叠
- `fixedDelay`：上次结束后再等间隔，不会重叠（推荐）

## 🔥 扩展追问 2：任务执行到一半实例挂了怎么办？

```java
@Component
public class SafeJob {
    
    @Autowired
    private RedissonClient redissonClient;
    
    @Autowired
    private JobProgressRepository progressRepository;
    
    @Scheduled(cron = "0 0 2 * * ?")
    public void batchProcess() {
        RLock lock = redissonClient.getLock("job:batch:process");
        
        try {
            if (!lock.tryLock(0, 30, TimeUnit.MINUTES)) {
                return;
            }
            
            // 1. 查询上次执行进度
            JobProgress progress = progressRepository.get("batchProcess");
            long lastId = progress != null ? progress.getLastProcessedId() : 0;
            
            // 2. 分页处理，每页记录进度
            while (true) {
                List<Data> list = dataMapper.selectByIdGt(lastId, 1000);
                if (list.isEmpty()) break;
                
                for (Data data : list) {
                    process(data);
                    lastId = data.getId();
                }
                
                // 3. 每页保存进度（幂等点）
                progressRepository.save("batchProcess", lastId);
            }
            
        } catch (Exception e) {
            log.error("任务执行异常", e);
            // 不删除进度，下次从断点继续
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

**断点续传设计：**
- 任务可幂等（重复执行同一数据无影响）
- 定期保存进度（处理完一批数据就保存）
- 下次执行从上次进度继续

## 🔥 扩展追问 3：如何监控定时任务执行？

```java
@Component
public class JobMetrics {
    
    private final MeterRegistry meterRegistry;
    
    @Autowired
    public JobMetrics(MeterRegistry meterRegistry) {
        this.meterRegistry = meterRegistry;
    }
    
    public void recordJob(String jobName, Runnable task) {
        Timer.Sample sample = Timer.start(meterRegistry);
        
        try {
            task.run();
            meterRegistry.counter("job.success", "name", jobName).increment();
        } catch (Exception e) {
            meterRegistry.counter("job.failure", "name", jobName).increment();
            throw e;
        } finally {
            sample.stop(meterRegistry.timer("job.duration", "name", jobName));
        }
    }
}
```

**监控指标：**
- 任务执行次数（成功/失败）
- 任务执行耗时
- 锁竞争情况（获取锁失败次数）
- 任务堆积（上次执行时间距离现在多久）

## 一句话收尾

**多实例定时任务防重，锁是核心**——简单场景Redisson一把锁搞定，复杂调度上XXL-JOB/Elastic-Job，最省事用ShedLock注解，关键是锁要有超时和续期机制，避免死锁和误释放。
第44题完成！定时任务防重是分布式系统的经典问题。

几个选型建议：
- **简单场景**（任务少、不要求分片）→ **ShedLock**，注解一把梭
- **中等规模**（任务较多）→ **Redisson**，灵活可控
- **大规模/需分片**（数据量大、要并行处理）→ **XXL-JOB/Elastic-Job**

容易踩的坑：
- **锁没释放**：实例挂了锁还在，要设置合理超时+看门狗续期
- **任务重叠**：用`fixedDelay`代替`cron`，等上次执行完再触发
- **执行中断**：设计断点续传，定期保存进度
