## 先说结论

数据库连接池被打满，本质是**连接泄漏**或**连接数配置不合理**。排查思路：**先确认是慢 SQL 导致连接长时间占用，还是连接未释放导致泄漏**；解决方案：**优化慢 SQL + 合理配置连接池大小 + 启用连接泄漏检测 + 缩短连接超时时间**。

```
│                                                         │

│   └── max-lifetime: 30m        ← 连接最大存活时间         │
│                                                         │
│   连接池状态：                                           │
│   ┌─────────────────────────────────────────────┐      │
│   │  总连接: 20                                  │      │

│   │                                              │      │
│   │  等待队列: 50个请求在等待获取连接              │      │
│   │  → 最终超时报错：ConnectionTimeoutException   │      │
│   └─────────────────────────────────────────────┘      │
│                                                         │
│   常见原因：                                             │
│   1. SQL 执行慢，连接长时间不释放                        │

│   3. 连接获取后未关闭（代码 bug）                         │
│   4. 连接池配置过小，并发量超过承载能力                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 方案二：控制事务范围（关键！）

```java
// ❌ 问题代码：事务范围过大，连接长时间占用
@Service
public class OrderService {
    
    @Transactional  // 整个方法都在事务中
    public void processOrder(OrderDTO dto) {
        // 1. 查订单（需要连接）
        Order order = orderMapper.findById(dto.getId());
        
        // 2. 调用外部 HTTP 接口（5秒）
        paymentService.charge(order);  // 连接被占用等待！
        
        // 3. 更新订单（需要连接）
        orderMapper.updateStatus(order.getId(), PAID);
        
        // 4. 发送 MQ（不需要连接，但在事务中）
        mqTemplate.send("order-paid", order);
    }
}

// ✅ 优化后：缩小事务范围
@Service
public class OrderService {
    
    @Autowired
    private OrderProcessor orderProcessor;
    
    // 非事务方法
    public void processOrder(OrderDTO dto) {
        // 1. 查订单（无事务，用完即还）
        Order order = orderMapper.findById(dto.getId());
        
        // 2. 调用外部 HTTP 接口（无连接占用）
        paymentService.charge(order);
        
        // 3. 事务方法：只包含 DB 操作
        orderProcessor.updateAndNotify(order);
    }
}

@Component
public class OrderProcessor {
    
    @Transactional  // 小事务
    public void updateAndNotify(Order order) {
        orderMapper.updateStatus(order.getId(), PAID);
        mqTemplate.send("order-paid", order);
    }
}
```

### 方案四：连接池监控告警

```java
@Component
public class ConnectionPoolMonitor {
    
    @Autowired
    private HikariDataSource dataSource;
    
    @Scheduled(fixedRate = 60000)  // 每分钟检查
    public void checkPoolStatus() {
        HikariPoolMXBean poolMXBean = dataSource.getHikariPoolMXBean();
        
        int active = poolMXBean.getActiveConnections();
        int total = poolMXBean.getTotalConnections();
        int waiting = poolMXBean.getThreadsAwaitingConnection();
        
        double usageRate = (double) active / total;
        
        // 使用率超过 80% 告警
        if (usageRate > 0.8) {
            alertService.send(AlertLevel.P2, 
                String.format("连接池使用率过高: %d/%d (%.1f%%), 等待线程: %d",
                    active, total, usageRate * 100, waiting));
        }
        
        // 有等待线程告警
        if (waiting > 0) {
            alertService.send(AlertLevel.P1,
                String.format("连接池已满，%d 个线程等待获取连接", waiting));
        }
    }
}
```

## 🔥 扩展追问 1：连接池大小怎么定？

```
HikariCP 作者推荐的公式：
连接数 = ((核心数 × 2) + 有效磁盘数)

示例：
- 4核 CPU，SSD 磁盘 → (4×2)+1 = 9
- 8核 CPU，SSD 磁盘 → (8×2)+1 = 17
- 16核 CPU，SSD 磁盘 → (16×2)+1 = 33

实际配置建议：
- 低配服务器（2-4核）：10-20
- 中配服务器（8核）：20-30
- 高配服务器（16核+）：30-50

注意：
- 不是越大越好！连接数超过 CPU 核心数，上下文切换开销反而降低性能
- 多个服务共用数据库时，总连接数 = 各服务连接池之和 ≤ 数据库 max_connections
```

## 🔥 扩展追问 2：连接泄漏常见代码模式

```java
// ❌ 泄漏模式 1：异常时未关闭连接
public void leakyMethod() {
    Connection conn = dataSource.getConnection();
    Statement stmt = conn.createStatement();
    ResultSet rs = stmt.executeQuery("SELECT ...");
    
    // 如果这里抛异常，下面的 close 不会执行！
    process(rs);
    
    rs.close();      // 可能不执行
    stmt.close();    // 可能不执行
    conn.close();    // 可能不执行 → 泄漏！
}

// ✅ 正确写法：try-with-resources
public void safeMethod() {
    try (Connection conn = dataSource.getConnection();
         Statement stmt = conn.createStatement();
         ResultSet rs = stmt.executeQuery("SELECT ...")) {
        
        process(rs);
        
    } // 自动关闭，即使抛异常
}

// ❌ 泄漏模式 2：Spring 事务中异步操作
@Transactional
public void asyncInTx() {
    orderMapper.update(order);  // 开启事务，占用连接
    
    // 异步线程需要连接，但主线程事务未提交，连接不释放
    CompletableFuture.runAsync(() -> {
        inventoryMapper.deduct(skuId);  // 等待连接...
    });
    
    // 主线程等待异步结果，死锁！
}

// ✅ 正确写法：异步操作在事务外
public void safeAsync() {
    // 事务内只同步操作
    txTemplate.execute(status -> {
        orderMapper.update(order);
        return null;
    });
    
    // 事务提交后再异步
    CompletableFuture.runAsync(() -> {
        inventoryMapper.deduct(skuId);
    });
}
```

## 一句话收尾

连接池打满的核心解决思路是**"快用快还"**：通过 **HikariCP 泄漏检测**定位问题代码，**缩小事务范围**减少连接持有时间，**优化慢 SQL**降低单连接占用时长，配合**合理的连接池大小配置**和**实时监控告警**，从根本上避免连接池耗尽。
