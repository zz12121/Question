## 先说结论

消息重复消费的本质是**至少一次投递（At-least-once）** 的副作用。保证幂等性的核心是**业务唯一标识 + 去重机制**：消费前检查该消息是否已处理，已处理则跳过，未处理则执行业务并记录处理状态。

## 🔥 核心概念

### 为什么消息会重复？

```
┌─────────────────────────────────────────────────────────┐
│                  消息重复消费场景                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  场景1：消费者处理成功，但 ACK 失败                        │
│  ───────────────────────────────                        │
│  消费者 ──→ 处理业务 ──→ 业务成功                        │
│              │                                          │
│              └──→ 发送 ACK ──→ 网络中断                  │
│                                └──→ Broker 未收到 ACK    │
│                                └──→ 消息重新投递          │
│                                                         │
│  场景2：消费者处理超时                                    │
│  ─────────────────────                                  │
│  Broker 等待 ACK 超时（如 30s）                          │
│  └──→ 认为消费失败                                       │
│  └──→ 消息重新投递给该消费者或其他消费者                   │
│                                                         │
│  场景3：消费者重启                                        │
│  ─────────────────                                      │
│  消费者处理到一半，系统重启                                │
│  重启后从上次 Offset 继续消费                             │
│  └──→ 部分已处理的消息被重新消费                          │
│                                                         │
│  结论：MQ 只保证 At-least-once，不保证 Exactly-once       │
│        幂等性必须由消费者业务层保证                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 🔥 详细展开：五种幂等方案

### 方案一：数据库唯一索引（推荐，简单可靠）

```
┌─────────────────────────────────────────────────────────┐
│              唯一索引幂等方案                              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  核心思想：利用数据库唯一约束，重复插入会报错               │
│                                                         │
│  表设计：                                                │
│  CREATE TABLE idempotent_log (                          │
│      msg_id VARCHAR(64) PRIMARY KEY,  -- 消息唯一ID      │
│      biz_id VARCHAR(64),               -- 业务唯一ID      │
│      status TINYINT,                   -- 处理状态        │
│      create_time DATETIME              -- 创建时间        │
│  );                                                     │
│                                                         │
│  消费流程：                                              │
│  1. 先插入幂等表（msg_id 唯一）                           │
│     ├── 成功 → 继续处理业务                               │
│     └── 失败（DuplicateKey）→ 已处理，直接返回             │
│  2. 处理业务                                             │
│  3. 更新幂等表状态为成功                                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

```java
@Component
@KafkaListener(topics = "order-topic", groupId = "order-group")
public class OrderConsumer {
    
    @Autowired
    private IdempotentLogMapper idempotentLogMapper;
    
    @Autowired
    private OrderMapper orderMapper;
    
    @Transactional
    public void onMessage(ConsumerRecord<String, String> record) {
        String msgId = record.key();  // 消息唯一ID
        
        // 1. 幂等检查：插入幂等表
        try {
            IdempotentLog log = new IdempotentLog();
            log.setMsgId(msgId);
            log.setBizId(extractBizId(record.value()));
            log.setStatus(0);  // 处理中
            log.setCreateTime(LocalDateTime.now());
            
            idempotentLogMapper.insert(log);
            
        } catch (DuplicateKeyException e) {
            // 已存在，说明已处理或正在处理
            IdempotentLog exist = idempotentLogMapper.findByMsgId(msgId);
            
            if (exist.getStatus() == 1) {
                // 已处理成功，直接返回
                log.info("消息已处理，跳过: {}", msgId);
                return;
            }
            
            // 状态=0（处理中），可能是上次消费中断，继续处理
            log.warn("消息处理中，可能上次中断，继续处理: {}", msgId);
        }
        
        try {
            // 2. 处理业务
            OrderMessage message = JSON.parseObject(record.value(), OrderMessage.class);
            processOrder(message);
            
            // 3. 更新幂等表为成功
            idempotentLogMapper.updateStatus(msgId, 1);
            
        } catch (Exception e) {
            // 业务失败，不更新幂等表，让 MQ 重试
            throw new RuntimeException("业务处理失败", e);
        }
    }
}
```

### 方案二：Redis SETNX（高性能，短时效）

```java
@Component
public class RedisIdempotentConsumer {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private OrderService orderService;
    
    @KafkaListener(topics = "order-topic")
    public void onMessage(ConsumerRecord<String, String> record) {
        String msgId = record.key();
        String key = "idempotent:" + msgId;
        
        // 1. Redis SETNX：只有第一次设置成功
        Boolean isFirst = redisTemplate.opsForValue()
            .setIfAbsent(key, "1", 24, TimeUnit.HOURS);  // 24小时过期
        
        if (Boolean.FALSE.equals(isFirst)) {
            log.info("消息已处理，跳过: {}", msgId);
            return;
        }
        
        try {
            // 2. 处理业务
            OrderMessage message = JSON.parseObject(record.value(), OrderMessage.class);
            orderService.createOrder(message);
            
        } catch (Exception e) {
            // 业务失败，删除 Redis 标记，允许重试
            redisTemplate.delete(key);
            throw e;
        }
    }
}
```

**优缺点：**
- ✅ 性能高，无数据库压力
- ❌ Redis 可能丢数据（需持久化配置）
- ❌ 过期时间难定（业务处理时间不确定）

### 方案三：业务状态机（天然幂等）

```java
/**
 * 订单状态机：同一状态只能流转一次，天然幂等
 */
@Service
public class OrderStateMachineService {
    
    @Autowired
    private OrderMapper orderMapper;
    
    /**
     * 支付回调处理
     */
    @Transactional
    public void onPaymentCallback(String orderNo, String paymentId) {
        Order order = orderMapper.findByOrderNo(orderNo);
        
        // 状态机检查：只有 PENDING 才能变为 PAID
        if (order.getStatus() != OrderStatus.PENDING) {
            log.info("订单状态已变更，跳过: {} -> {}", orderNo, order.getStatus());
            return;  // 已处理过，幂等返回
        }
        
        // 更新状态（乐观锁防止并发）
        int affected = orderMapper.updateStatus(
            orderNo, 
            OrderStatus.PENDING,  // 期望当前状态
            OrderStatus.PAID,     // 目标状态
            paymentId
        );
        
        if (affected == 0) {
            // 被其他线程更新，幂等返回
            log.info("订单状态已被其他线程更新: {}", orderNo);
            return;
        }
        
        // 继续后续业务...
    }
}

// SET status = #{newStatus}, payment_id = #{paymentId}
// WHERE order_no = #{orderNo} AND status = #{expectedStatus}
```

### 方案四：Token 机制（生产者保证唯一）

```java
// 生产者：生成唯一 Token
@Service
public class MessageProducer {
    
    public void sendOrderMessage(Order order) {
        // 生成业务唯一 Token
        String idempotentToken = generateToken(order.getUserId(), order.getSkuId());
        
        OrderMessage message = OrderMessage.builder()
            .orderNo(order.getOrderNo())
            .userId(order.getUserId())
            .idempotentToken(idempotentToken)  // 幂等 Token
            .build();
        
        kafkaTemplate.send("order-topic", idempotentToken, JSON.toJSONString(message));
    }
    
    private String generateToken(Long userId, Long skuId) {
        // userId + skuId + 时间窗口（如小时级）
        String timeWindow = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyyMMddHH"));
        return DigestUtils.md5Hex(userId + ":" + skuId + ":" + timeWindow);
    }
}

// 消费者：Token 去重
@Component
public class TokenIdempotentConsumer {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @KafkaListener(topics = "order-topic")
    public void onMessage(ConsumerRecord<String, String> record) {
        String token = record.key();  // 业务 Token
        
        // SETNX 去重
        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent("token:" + token, "1", 1, TimeUnit.HOURS);
        
        if (!success) {
            log.info("Token 已存在，重复消息: {}", token);
            return;
        }
        
        // 处理业务...
    }
}
```

### 方案五：数据库乐观锁（更新场景）

```java
/**
 * 库存扣减：乐观锁保证幂等
 */
@Service
public class InventoryService {
    
    @Autowired
    private InventoryMapper inventoryMapper;
    
    /**
     * 扣减库存（幂等）
     * @param orderNo 订单号，用于幂等判断
     */
    @Transactional
    public boolean deduct(Long skuId, Integer quantity, String orderNo) {
        // 1. 检查是否已扣过（幂等表或业务表唯一索引）
        if (deductLogMapper.existsByOrderNo(orderNo)) {
            return true;  // 已扣过，幂等成功
        }
        
        // 2. 乐观锁更新库存
        int retry = 3;
        while (retry-- > 0) {
            Inventory inventory = inventoryMapper.findBySkuId(skuId);
            
            if (inventory.getStock() < quantity) {
                return false;  // 库存不足
            }
            
            // version 作为乐观锁
            int affected = inventoryMapper.decrement(
                skuId, 
                quantity, 
                inventory.getVersion()  // 期望当前版本
            );
            
            if (affected > 0) {
                // 3. 记录扣减日志（幂等）
                deductLogMapper.insert(new DeductLog(orderNo, skuId, quantity));
                return true;
            }
            
            // 版本冲突，重试
        }
        
        throw new RuntimeException("扣减库存失败，重试耗尽");
    }
}

// SQL
// UPDATE inventory 
// SET stock = stock - #{quantity}, version = version + 1
// WHERE sku_id = #{skuId} AND version = #{version}
```

## 🔥 扩展追问 1：方案对比与选型

| 方案 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **数据库唯一索引** | 通用，推荐 | 可靠，持久化 | 有数据库压力 |
| **Redis SETNX** | 高并发，短时效 | 性能极高 | 可能丢数据，过期难定 |
| **业务状态机** | 有状态流转的业务 | 无额外存储 | 需设计状态机 |
| **Token 机制** | 生产者可控 | 业务语义清晰 | 需生产者配合 |
| **乐观锁** | 更新操作 | 天然防并发 | 需重试机制 |

**生产推荐组合：**
- 高并发场景：**Redis SETNX + 数据库唯一索引**（双保险）
- 一般场景：**数据库唯一索引**（简单可靠）
- 有状态业务：**业务状态机**（最优雅）

## 🔥 扩展追问 3：消费端配置优化

```yaml
# Kafka 消费者配置（减少重复消费概率）
spring:
  kafka:
    consumer:
      enable-auto-commit: false     # 关闭自动提交，手动控制
      max-poll-records: 100         # 每次拉取100条，批量处理
      max-poll-interval-ms: 300000  # 5分钟，处理慢时不会被认为是消费者失效
      
    listener:
      ack-mode: manual_immediate    # 处理完立即提交 offset
      concurrency: 5                # 5个消费者实例
```

```java
@Component
public class ReliableConsumer {
    
    @KafkaListener(topics = "order-topic")
    public void onMessage(ConsumerRecord<String, String> record, Acknowledgment ack) {
        try {
            // 处理业务
            process(record);
            
            // 业务成功，提交 offset
            ack.acknowledge();
            
        } catch (Exception e) {
            // 业务失败，不提交 offset，消息会重投
            // 或根据异常类型决定是否提交（如业务校验失败，提交避免无限重试）
            if (e instanceof BusinessException) {
                ack.acknowledge();  // 业务错误，不再重试
            }
            throw e;
        }
    }
}
```

## 一句话收尾

消息幂等的核心是**"先检查，后处理"**：通过**数据库唯一索引**或**Redis SETNX**在去重表中标记消息处理状态，业务处理前检查标记，已处理则直接返回，配合**业务状态机**或**乐观锁**实现端到端幂等，确保重复消息不会导致业务异常。
