## 先说结论

10万/秒的异步任务处理，核心方案是 **Kafka 分区队列 + 多消费者组 + 批量消费**：Kafka 单机可达百万级吞吐，通过分区实现水平扩展，消费者批量拉取消息减少网络开销，配合**异步发送 + 背压控制 + 死信队列**实现高可靠、高性能的任务处理。

```java
@Service
public class TaskProducer {
    
    @Autowired
    private KafkaTemplate<String, TaskMessage> kafkaTemplate;
    
    /**
     * 异步发送任务（非阻塞，高吞吐）
     */
    public void sendTaskAsync(TaskMessage task) {
        // 按 userId 分区，保证同一用户的任务顺序处理
        String key = String.valueOf(task.getUserId() % 10000);
        
        ListenableFuture<SendResult<String, TaskMessage>> future = 
            kafkaTemplate.send("task-queue", key, task);
        
        // 异步回调，记录失败
        future.addCallback(
            result -> {
                // 成功，记录 offset
            },
            ex -> {
                // 失败，记录日志，可入 DB 补偿
                log.error("发送失败: {}", task, ex);
                fallbackToDb(task);
            }
        );
    }
    
    /**
     * 批量发送（更高吞吐）
     */
    public void sendBatch(List<TaskMessage> tasks) {
        // 分批异步发送
        List<List<TaskMessage>> batches = Lists.partition(tasks, 500);
        
        for (List<TaskMessage> batch : batches) {
            batch.forEach(this::sendTaskAsync);
        }
    }
    
    private void fallbackToDb(TaskMessage task) {
        // 发送失败，落入 DB，定时任务补偿重发
        failedTaskMapper.insert(task);
    }
}
```

```java
@Component
public class TaskConsumer {
    
    @Autowired
    private EmailService emailService;
    
    @Autowired
    private SmsService smsService;
    
    /**
     * 批量消费任务
     */
    @KafkaListener(
        topics = "task-queue",
        groupId = "task-consumer-group",
        containerFactory = "batchFactory"
    )
    public void onMessage(List<ConsumerRecord<String, TaskMessage>> records,
                          Acknowledgment ack) {
        
        // 1. 按任务类型分组
        Map<TaskType, List<TaskMessage>> grouped = records.stream()
            .map(ConsumerRecord::value)
            .collect(Collectors.groupingBy(TaskMessage::getType));
        
        // 2. 并发处理不同类型（邮件和短信互不阻塞）
        CompletableFuture<Void> emailFuture = CompletableFuture.runAsync(() -> 
            processBatch(grouped.getOrDefault(TaskType.EMAIL, Collections.emptyList()),
                        emailService::send));
        
        CompletableFuture<Void> smsFuture = CompletableFuture.runAsync(() -> 
            processBatch(grouped.getOrDefault(TaskType.SMS, Collections.emptyList()),
                        smsService::send));
        
        // 3. 等待全部完成
        CompletableFuture.allOf(emailFuture, smsFuture).join();
        
        // 4. 手动确认（所有消息处理完才提交 offset）
        ack.acknowledge();
    }
    
    /**
     * 批量处理 + 异常隔离
     */
    private <T> void processBatch(List<T> tasks, Consumer<T> processor) {
        for (T task : tasks) {
            try {
                processor.accept(task);
            } catch (Exception e) {
                // 单条失败不影响其他，记录后入死信队列
                log.error("处理失败: {}", task, e);
                sendToDLQ(task);
            }
        }
    }
    
    private void sendToDLQ(Object task) {
        // 发送到死信队列，后续重试
        kafkaTemplate.send("task-queue-dlq", task);
    }
}
```

## 🔥 扩展追问 2：背压控制（防止消费者被压垮）

```java
@Component
public class BackpressureConsumer {
    
    private final Semaphore semaphore = new Semaphore(100);  // 最大并发100
    
    @KafkaListener(topics = "task-queue", groupId = "backpressure-group")
    public void onMessage(ConsumerRecord<String, TaskMessage> record) {
        try {
            // 获取许可，满了则阻塞（自然背压）
            semaphore.acquire();
            
            processAsync(record.value())
                .whenComplete((result, ex) -> semaphore.release());
                
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
    
    // 或使用 Reactor 背压
    @Bean
    public ReactiveKafkaConsumerTemplate<String, TaskMessage> reactiveConsumer() {
        return new ReactiveKafkaConsumerTemplate<>(
            ReceiverOptions.<String, TaskMessage>create(props)
                .subscription(Collections.singleton("task-queue"))
                .addAssignListener(partitions -> 
                    System.out.println("Assigned: " + partitions))
        );
    }
    
    // 背压消费
    public void consumeWithBackpressure() {
        reactiveConsumer.receive()
            .onBackpressureBuffer(1000)  // 缓冲区1000
            .flatMap(record -> 
                processReactive(record)
                    .doOnSuccess(r -> record.receiverOffset().acknowledge()),
                50  // 并发度50
            )
            .subscribe();
    }
}
```

## 🔥 扩展追问 3：延迟任务处理（如定时发送）

```java
// 方案1：Kafka 延迟消息（RocketMQ 原生支持，Kafka 需自实现）
@Component
public class DelayedTaskScheduler {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private KafkaTemplate<String, TaskMessage> kafkaTemplate;
    
    /**
     * 投递延迟任务到 Redis ZSet
     */
    public void schedule(TaskMessage task, long delayMs) {
        long executeTime = System.currentTimeMillis() + delayMs;
        redisTemplate.opsForZSet().add(
            "delayed:tasks",
            JSON.toJSONString(task),
            executeTime
        );
    }
    
    /**
     * 定时扫描，到期任务发送到 Kafka
     */
    @Scheduled(fixedRate = 1000)  // 每秒扫描
    public void scanAndDispatch() {
        Set<ZSetOperations.TypedTuple<String>> tasks = 
            redisTemplate.opsForZSet()
                .rangeByScoreWithScores("delayed:tasks", 
                    0, System.currentTimeMillis());
        
        for (ZSetOperations.TypedTuple<String> task : tasks) {
            TaskMessage message = JSON.parseObject(task.getValue(), TaskMessage.class);
            kafkaTemplate.send("task-queue", message);
            
            // 从延迟队列移除
            redisTemplate.opsForZSet().remove("delayed:tasks", task.getValue());
        }
    }
}
```

## 一句话收尾

10万/秒异步任务的核心设计是 **Kafka 分区并行 + 批量消费 + 背压控制**：通过合理设置分区数实现水平扩展，批量拉取消息提升吞吐，手动提交 offset + 死信队列保证可靠性，配合**异步发送、连接池复用、限流熔断**实现高可用、高性能的任务队列系统。
这道题是**高并发架构设计题**，重点考察：

1. **技术选型**：为什么选 Kafka 而不是 RabbitMQ/Redis
2. **吞吐优化**：批量发送、批量消费、压缩、分区并行
3. **可靠性保障**：at-least-once、死信队列、手动 ack
4. **生产运维**：背压控制、监控告警、延迟任务

如果面试官追问"Kafka 消费者挂了，消息积压怎么办"，答：
- 临时扩容消费者（不超过分区数）
- 如果超过分区数，临时增加分区（需要停服或在线扩容）
- 或者跳过部分消息（业务允许时），重置 offset
