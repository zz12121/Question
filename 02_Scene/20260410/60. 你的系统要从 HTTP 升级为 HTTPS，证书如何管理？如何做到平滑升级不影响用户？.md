## 先说结论

HTTP升级HTTPS的核心是**"证书托管+反向代理+渐进切换"**：证书使用Let's Encrypt或云厂商托管实现自动续期，通过Nginx/SLB统一卸载TLS，业务层无感知。平滑升级采用**"双协议并存→301跳转→HSTS强制"**三步走，配合CDN回源协议适配，确保用户零中断。

## 🔥 核心概念

### HTTPS vs HTTP

| 对比项 | HTTP | HTTPS |
|-------|------|-------|
| **安全性** | 明文传输，易劫持 | TLS加密，防窃听篡改 |
| **端口** | 80 | 443 |
| **性能** | 无握手开销 | TLS握手增加1-RTT |
| **SEO** | 无优势 | 搜索引擎优先收录 |
| **成本** | 无 | 证书费用（免费证书已普及） |

### TLS握手过程

```
客户端                    服务端
  │                         │
  │ ───── ClientHello ─────>│  支持的TLS版本、加密套件
  │                         │
  │ <──── ServerHello ───── │  选定的TLS版本、加密套件
  │ <──── Certificate ───── │  服务端证书（含公钥）
  │ <──── ServerKeyExchange │  密钥交换参数（非必须）
  │ <──── ServerHelloDone ─ │  
  │                         │
  │ ───── ClientKeyExchange─>│ 预主密钥（用公钥加密）
  │ ───── [ChangeCipherSpec]│
  │ ───── Finished ────────>│
  │                         │
  │ <──── [ChangeCipherSpec]│
  │ <──── Finished ─────────│
  │                         │
  │ ═══════════════════════ │  加密通道建立，应用数据传输
```

### 方案1：Let's Encrypt免费证书（推荐）

```bash
# 使用Certbot自动申请和续期
# 1. 安装Certbot
sudo apt-get install certbot

# 2. 申请证书（DNS验证或HTTP验证）
sudo certbot certonly --standalone -d example.com -d www.example.com

# 3. 自动续期（crontab）
0 0 * * * /usr/bin/certbot renew --quiet --deploy-hook "nginx -s reload"

# 证书位置
/etc/letsencrypt/live/example.com/
├── fullchain.pem  # 证书链（证书+中间证书）
├── privkey.pem    # 私钥
├── cert.pem       # 证书
└── chain.pem      # 中间证书
```

## 🔥 详细展开：平滑升级方案

### 升级流程

```
阶段1：准备期
┌─────────────────────────────────────────────────────────────┐
│  1. 申请并配置HTTPS证书                                        │
│  2. Nginx/SLB配置443端口，支持HTTP+HTTPS双协议                  │
│  3. 测试HTTPS访问，验证证书有效性                               │
│  4. 更新CDN回源配置（支持HTTPS回源）                           │
└─────────────────────────────────────────────────────────────┘
                              ↓
阶段2：灰度期（双协议并存）
┌─────────────────────────────────────────────────────────────┐
│  HTTP(80) ──┐                                               │
│             ├──► Nginx/SLB ──► 后端服务                       │
│  HTTPS(443)─┘                                               │
│                                                             │
│  - 用户可自由选择协议访问                                      │
│  - 监控HTTPS流量占比、错误率                                   │
│  - 修复兼容性问题（混合内容、Cookie安全等）                     │
└─────────────────────────────────────────────────────────────┘
                              ↓
阶段3：切换期（301跳转）
┌─────────────────────────────────────────────────────────────┐
│  HTTP(80) ──► Nginx ──► 301 Redirect ──► HTTPS(443)         │
│                                                             │
│  - 强制HTTP跳转到HTTPS                                        │
│  - 保留部分API路径允许HTTP（如健康检查）                       │
│  - 监控跳转成功率、SEO收录变化                                 │
└─────────────────────────────────────────────────────────────┘
                              ↓
阶段4：强制期（HSTS）
┌─────────────────────────────────────────────────────────────┐
│  HTTPS(443) + HSTS头                                         │
│  Strict-Transport-Security: max-age=31536000; includeSubDomains│
│                                                             │
│  - 浏览器强制使用HTTPS（防SSL剥离攻击）                        │
│  - 提交到HSTS Preload List                                   │
│  - 关闭80端口（可选）                                         │
└─────────────────────────────────────────────────────────────┘
```

### Nginx配置示例

```nginx
# /etc/nginx/nginx.conf

server {
    listen 80;
    server_name example.com www.example.com;
    
    # 阶段2：双协议并存（注释掉跳转）
    # location / {
    #     proxy_pass http://backend;
    # }
    
    # 阶段3：301跳转到HTTPS
    location / {
        return 301 https://$server_name$request_uri;
    }
    
    # 例外：允许某些路径HTTP访问（如健康检查）
    location /health {
        proxy_pass http://backend;
    }
}

server {
    listen 443 ssl http2;
    server_name example.com www.example.com;
    
    # 证书配置
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;
    
    # TLS版本和加密套件（安全性配置）
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
    ssl_prefer_server_ciphers off;
    
    # 阶段4：HSTS（强制HTTPS）
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    
    # OCSP Stapling（提升性能）
    ssl_stapling on;
    ssl_stapling_verify on;
    ssl_trusted_certificate /etc/letsencrypt/live/example.com/chain.pem;
    
    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;  # 告知后端是HTTPS
    }
}
```

### Spring Boot应用适配

```java
/**
 * 应用层感知HTTPS
 */
@Configuration
public class HttpsConfig {
    
    /**
     * 强制Session Cookie安全属性
     */
    @Bean
    public CookieSerializer cookieSerializer() {
        DefaultCookieSerializer serializer = new DefaultCookieSerializer();
        serializer.setUseSecureCookie(true);  // 仅HTTPS传输
        serializer.setSameSite("Strict");      // 防止CSRF
        return serializer;
    }
    
    /**
     * 安全头配置
     */
    @Bean
    public FilterRegistrationBean<Filter> securityHeadersFilter() {
        FilterRegistrationBean<Filter> registration = new FilterRegistrationBean<>();
        registration.setFilter((request, response, chain) -> {
            HttpServletResponse res = (HttpServletResponse) response;
            
            // HSTS
            res.setHeader("Strict-Transport-Security", 
                "max-age=31536000; includeSubDomains");
            
            // 防止混合内容
            res.setHeader("Content-Security-Policy", 
                "upgrade-insecure-requests");
            
            chain.doFilter(request, response);
        });
        return registration;
    }
}
```

## 🔥 扩展追问 2：常见问题处理

### 问题1：混合内容（Mixed Content）

```
现象：HTTPS页面加载HTTP资源被浏览器拦截
解决：
1. 全站资源改为HTTPS（推荐）
2. 使用相对协议（//example.com/resource.js）
3. CSP upgrade-insecure-requests自动升级
```

```html
<!-- 修改前 -->
<img src="http://cdn.example.com/image.jpg">

<!-- 修改后 -->
<img src="https://cdn.example.com/image.jpg">
<!-- 或 -->
<img src="//cdn.example.com/image.jpg">
```

### 问题2：SEO影响

```java
/**
 * SEO适配：Canonical标签指向HTTPS
 */
@Component
public class SeoFilter extends OncePerRequestFilter {
    
    @Override
    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain chain) 
            throws ServletException, IOException {
        
        // 添加Canonical Link
        String canonicalUrl = "https://" + request.getServerName() 
            + request.getRequestURI();
        response.setHeader("Link", "<" + canonicalUrl + ">; rel=\"canonical\"");
        
        chain.doFilter(request, response);
    }
}
```
