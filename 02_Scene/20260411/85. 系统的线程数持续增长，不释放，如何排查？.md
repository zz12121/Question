### 3.2 线程池配置问题

```java
// ❌ 问题代码 1：FixedThreadPool + 任务不均衡
@Configuration
public class ThreadPoolConfig {
    @Bean("taskExecutor")
    public ExecutorService taskExecutor() {
        return Executors.newFixedThreadPool(50);  // ⚠️ 固定 50 线程
        // 问题：如果任务少，50 个线程空转
        // 问题：如果任务异常未处理，线程卡死
    }
}

// ❌ 问题代码 2：CachedThreadPool 无上限
public ExecutorService badExecutor() {
    return Executors.newCachedThreadPool();  // ⚠️ 无上限！并发高时疯狂创建线程
}

// ❌ 问题代码 3：拒绝策略不当
threadPoolExecutor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
// ⚠️ CallerRunsPolicy 会在主线程执行，高并发时可能 OOM

// ❌ 问题代码 4：动态扩展后不回收
@Configuration
public class DynamicThreadPool {
    private ThreadPoolExecutor executor = new ThreadPoolExecutor(
        10, 100, 60L, TimeUnit.SECONDS, 
        new LinkedBlockingQueue<>(1000)
    );
    
    public void submit(Task task) {
        // ⚠️ 核心线程 10，最大 100
        // ⚠️ keepAliveTime=60s，但核心线程永不回收
        // ⚠️ 高峰期扩展到 100，低谷期不缩回
    }
}
```

```bash
# 查看线程池状态
# Arthas
thread -b    # 查看阻塞线程
thread --state BLOCKED   # 查看 blocked 线程

# Spring Boot Actuator
curl http://localhost:8080/actuator/metrics/tomcat.threads.busy
curl http://localhost:8080/actuator/metrics/tomcat.threads.current

# Dynamic TP 监控（如果用了 dynamic-tp）
curl http://localhost:8080/dynamic/tp
```

**解决方案**：
```java
// ✅ 正确配置：可伸缩线程池
@Configuration
public class ProperThreadPoolConfig {
    
    @Bean("taskExecutor")
    public ThreadPoolExecutor taskExecutor() {
        return new ThreadPoolExecutor(
            10,                      // 核心线程数
            50,                      // 最大线程数（不是无上限）
            60L,                     // 空闲回收时间
            TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(1000),  // 有界队列
            new ThreadFactoryBuilder()
                .setNameFormat("task-%d")
                .setDaemon(false)
                .build(),
            new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
        );
    }
}

// ✅ 优雅关闭
@PreDestroy
public void shutdown() {
    threadPool.shutdown();
    try {
        if (!threadPool.awaitTermination(30, TimeUnit.SECONDS)) {
            threadPool.shutdownNow();
        }
    } catch (InterruptedException e) {
        threadPool.shutdownNow();
    }
}
```

### 3.3 CompletableFuture 泄漏

```java
// ❌ 问题代码：CompletableFuture 未处理
@Service
public class BadAsyncService {
    
    public void processOrder(Long orderId) {
        // ⚠️ 不等待结果，异常也不会被发现
        CompletableFuture.runAsync(() -> {
            // 如果这里抛异常，线程会怎样？
            throw new RuntimeException("Oops!");
        });
        // ⚠️ 任务丢失，没有结果，没有异常日志
    }
    
    // ⚠️ 问题场景：thenCompose 嵌套过深
    public CompletableFuture<Result> badChain(Long id) {
        return CompletableFuture.supplyAsync(() -> get1(id))
            .thenCompose(r1 -> CompletableFuture.supplyAsync(() -> get2(r1)))
            .thenCompose(r2 -> CompletableFuture.supplyAsync(() -> get3(r2)));
            // ⚠️ ForkJoinPool.common() 默认并行度 = CPU-1
            // ⚠️ 如果某个链超时或卡住，线程会累积
    }
    
    // ⚠️ 问题场景：allOf 任一异常
    public void badAllOf(List<Long> ids) {
        CompletableFuture.allOf(
            ids.stream()
                .map(id -> CompletableFuture.runAsync(() -> process(id)))
                .toArray(CompletableFuture[]::new)
        ).join();
        // ⚠️ 如果其中一个失败，异常被吞掉
        // ⚠️ 线程状态不可控
    }
}
```

```bash

# jstack 中搜索
grep -A 5 "ForkJoinPool" jstack.log

#     java.lang.Thread.State: WAITING

#     at java.util.concurrent.ForkJoinPool.awaitWork(ForkJoinPool.java:...)

# 监控 CompletableFuture
# 使用 micrometer
@Configuration
public class AsyncConfig {
    @Bean
    public ExecutorBuilderCustomizer asyncExecutorCustomizer(MeterRegistry registry) {
        return executor -> {
            if (executor instanceof ThreadPoolTaskExecutor) {

```

**解决方案**：
```java
// ✅ 正确使用 CompletableFuture
@Service
public class GoodAsyncService {
    
    // 方案1：显式处理异常
    public CompletableFuture<Result> processWithHandler(Long id) {
        return CompletableFuture.supplyAsync(() -> process(id))
            .exceptionally(ex -> {
                log.error("Process failed for id={}", id, ex);
                return Result.defaultError();
            })
            .thenApply(r -> transform(r));
    }
    
    // 方案2：自定义线程池，避免影响全局 ForkJoinPool
    @Bean("asyncExecutor")
    public ExecutorService asyncExecutor() {
        return new ThreadPoolExecutor(
            10, 50, 60L, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(500)
        );
    }
    
    public CompletableFuture<Result> processOnCustomPool(Long id) {
        return CompletableFuture.supplyAsync(() -> process(id), asyncExecutor)
            .thenApply(r -> transform(r));
    }
    
    // 方案3：使用 @Async 注解（Spring 管理）
    @Async("asyncExecutor")
    public void processAsync(Long id) {
        // Spring 会处理异常和线程生命周期
    }
}
```

### 3.5 定时任务异常

```java
// ❌ 问题代码：Scheduled 任务异常未处理
@Service
public class BadScheduler {
    
    // ⚠️ 任务抛出异常后，ScheduledThreadPool 会怎样？
    @Scheduled(fixedDelay = 5000)
    public void syncData() {
        // 如果这里抛异常
        throw new RuntimeException("Data sync failed!");
        // ⚠️ 整个ScheduledExecutor 可能会停止调度！
    }
    
    // ⚠️ 异常导致线程被回收，不断创建新线程
    @Scheduled(fixedDelayString = "${schedule.interval:5000}")
    public void processTask() {
        throw new RuntimeException();  // 每次抛异常
    }
    // 现象：ScheduledThreadPool 的线程数持续增长
}

// ❌ 问题代码：动态创建定时任务
public class DynamicScheduler {
    
    private ScheduledExecutorService scheduler = 
        Executors.newScheduledThreadPool(5);
    
    // ⚠️ 每次调用创建新任务，但不清理
    public void scheduleNewTask(Runnable task) {
        scheduler.scheduleAtFixedRate(task, 0, 1, TimeUnit.HOURS);
        // ⚠️ 调用 100 次 = 100 个定时任务
    }
}
```

```java
// ✅ 正确使用 @Scheduled
@Configuration
@EnableScheduling
public class SchedulerConfig implements SchedulingConfigurer {
    
    @Override
    public void configureTasks(ScheduledTaskRegistrar taskRegistrar) {
        taskRegistrar.setTaskScheduler(
            new ConcurrentTaskScheduler(
                Executors.newScheduledThreadPool(5)  // 显式配置线程池
            )
        );
    }
}

@Service
public class GoodScheduler {
    
    @Autowired private TaskScheduler taskScheduler;
    
    // ✅ 任务加 try-catch
    @Scheduled(fixedDelay = 5000)
    public void syncData() {
        try {
            doSync();
        } catch (Exception e) {
            log.error("Sync failed", e);
            // ✅ 捕获异常，防止线程崩溃
        }
    }
    
    // ✅ 动态任务用完后取消
    public void scheduleOnce(Runnable task) {
        ScheduledFuture<?> future = taskScheduler.schedule(task, 
            Instant.now().plusSeconds(60));
        
        // 用完后取消
        future.cancel(false);
    }
}
```

## 🔥 扩展追问 1：如何快速定位是哪个线程池在泄漏？

> **A:** 通过线程名称 + 创建栈定位：

```bash
# 1. 统计所有线程池的线程数量
jstack <PID> | grep -E '"pool-|"Dubbo-|"http-|"RxComputation|' | \
    sed 's/.*"\(pool-[0-9]*\)".*/\1/' | sort | uniq -c | sort -rn

# 2. 找到泄漏的线程池后，搜索创建栈
jstack <PID> | grep -B 30 "pool-10-thread"

# 3. 输出所有 "Created by" 标记（线程创建者）
jstack <PID> | grep -A 2 "Created by"

# 输出示例：
# "pool-10-thread-50"
#    java.lang.Thread.State: RUNNABLE
#    ...
# "pool-10-thread-51"
#    java.lang.Thread.State: RUNNABLE
# ...
# "pool-10-thread-200"
#    java.lang.Thread.State: WAITING
# Created by:
#    at java.util.concurrent.ThreadPoolExecutor$Worker.<init>
#    at java.util.concurrent.ThreadPoolExecutor.addWorker
#    at java.util.concurrent.ThreadPoolExecutor.execute

# 4. 对比正常时和异常时的线程池数量
# 在压测开始前记录：thread_count_baseline = 200
# 压测中记录：thread_count_current = 800
# 差值 = 600 = 泄漏的线程数
```

## 🔥 扩展追问 2：为什么线程池的线程不会自动回收？

> **A:** 核心是线程池的回收机制：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    线程池回收机制解析                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  ThreadPoolExecutor 线程回收条件：                                        │
│                                                                          │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  corePoolSize  vs  maximumPoolSize                              │   │
│  │                                                                 │   │
│  │  ┌───────────────────────────────────────────────────────────┐ │   │
│  │  │                                                           │ │   │
│  │  │   核心线程数                          最大线程数          │ │   │
│  │  │   ┌─────────────────────────┐   ┌─────────────────────┐ │ │   │
│  │  │   │  corePoolSize = 10     │   │  maxPoolSize = 100  │ │ │   │
│  │  │   │                        │   │                      │ │ │   │
│  │  │   │  [T1][T2]...[T10]       │   │  [T11]...[T100]     │ │ │   │
│  │  │   │                        │   │   ← 临时线程        │ │ │   │
│  │  │   │  ← 永不回收            │   │   ← 空闲后回收     │ │ │   │
│  │  │   │                        │   │                      │ │ │   │
│  │  │   └─────────────────────────┘   └─────────────────────┘ │ │   │
│  │  │                                                           │ │   │
│  │  └───────────────────────────────────────────────────────────┘ │   │
│  │                                                                 │   │
│  │  回收公式：                                                       │   │
│  │  ┌───────────────────────────────────────────────────────────┐ │   │
│  │  │  当前线程数 > corePoolSize                                │ │   │
│  │  │  且 空闲时间 > keepAliveTime                              │ │   │
│  │  │  → 线程被回收                                              │ │   │
│  │  └───────────────────────────────────────────────────────────┘ │   │
│  │                                                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                          │
│  常见陷阱：                                                             │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  ❌ newFixedThreadPool(50)                                     │   │
│  │     core = max = 50，且 LinkedBlockingQueue 无界               │   │
│  │     → 永远 > corePoolSize，keepAliveTime 无效                   │   │
│  │                                                                 │   │
│  │  ❌ allowCoreThreadTimeOut(false)                              │   │
│  │     → 核心线程永不回收                                          │   │
│  │                                                                 │   │
│  │  ❌ 任务永不结束                                                │   │
│  │     → while(true) { Thread.sleep(100); doTask(); }            │   │
│  │     → 线程永远 busy，无法触发回收                               │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 3：如何在线上环境快速止血？

> **A:** 分级应急处理：

| 级别 | 操作 | 适用场景 |
|------|------|---------|
| **L1** | 重启应用 | 紧急止血，保留现场前先 `jstack -dump` |
| **L2** | 关闭可疑定时任务 | `@Scheduled` 任务导致，动态关闭 |
| **L3** | 熔断外部依赖 | Feign/Hystrix 熔断，避免线程阻塞 |
| **L4** | 手动触发 GC | 如果是 OOM 导致的线程创建失败 |
| **L5** | 切流量到其他节点 | 保留故障节点，流量摘除 |

```bash
# L1: 先保留现场
jstack <PID> > thread_dump_$(date +%s).log
jmap -heap <PID> > heap_$(date +%s).log

# L2: 如果是 Spring Boot
# 通过 Actuator 关闭定时任务
curl -X POST http://localhost:8080/actuator/scheduledtasks

# L3: 如果是 Dubbo/Hystrix
# 调用管理接口触发熔断
curl -X PUT http://localhost:8080/actuator/hystrix.stream

# L4: 优雅重启（K8s）
kubectl rollout restart deployment/your-app -n production
```

## 一句话总结

> **线程数持续增长的根因是"线程泄漏"：线程池配置不当（FixedThreadPool 无上限）、CompletableFuture 异常未处理、外部依赖超时过长阻塞、@Scheduled 任务抛异常未捕获、第三方 SDK 连接泄漏。定位方法是 jstack 按线程名称分组统计，搜索 `waiting on` 和 `BLOCKED`，找到创建栈后定位到代码。**

## 面试加分话术

> "之前遇到过一个问题，线上线程数从 300 慢慢涨到 2000，花了两天才 OOM。一开始以为是正常业务增长，后来发现线程名都是 `DubboServerHandler-xxx`，而且都在 `WAITING`。jstack 了一把，全是 `waiting on <0x...>`，搜了下这个对象是个 MQ 连接。原来是运维改了 MQ 的心跳配置，把超时从 30 秒改成了 0——即永不断开。结果消费者线程卡在等待响应上，数量就一直涨。改成合理超时后，线程数稳定在 400。所以这类问题关键是**先分类**——看看哪类线程在涨，再定位是池配置、异常处理还是外部依赖的问题。"
