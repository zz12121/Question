### 4.2 网络抓包分析

```bash
# 1️⃣ tcpdump 基础抓包
tcpdump -i eth0 -nn -s0 -w traffic.pcap 'port 8080'
```

### 4.3 业务代码层面埋点

```java
// 在关键接口加流量统计
@RestController
@RequestMapping("/api")
public class NetworkMonitorController {
    
    // 统计每个接口的入出流量
    private static final MicrometerRegistry registry = MeterRegistry;
    
    @GetMapping("/download/{fileId}")
    public void download(@PathVariable String fileId, 
                         HttpServletResponse response) {
        
        // 1. 记录入向流量（请求大小，通常很小）
        Counter.builder("http.request.bytes")
            .tag("uri", "/api/download")
            .register(registry)
            .increment(requestSize);
        
        // 2. 记录出向流量（响应大小）
        FileResource resource = fileService.getFile(fileId);
        
        // 包装 Response，通过 Filter 统计出向流量
        ContentCachingResponseWrapper wrapped = 
            new ContentCachingResponseWrapper(response);
        
        // ... 业务逻辑 ...
        
        wrapped.copyBodyToResponse();
        
        // 3. 记录响应大小
        Counter.builder("http.response.bytes")
            .tag("uri", "/api/download")
            .tag("file_type", resource.getType())
            .register(registry)
            .increment(resource.getSize());
    }
}

// 网络 IO 拦截器
@Component
public class NetworkIOInterceptor implements HandlerInterceptor {
    
    @Autowired private NetworkIOService networkIOService;
    
    @Override
    public void afterCompletion(HttpServletRequest request, 
            HttpServletResponse response, Object handler, Exception ex) {
        
        // 记录每个请求的网络IO
        networkIOService.record(
            request.getRequestURI(),
            request.getContentLength(),      // 入向
            response.getStatus(),            // 响应码
            response.getBufferSize()         // 出向
        );
    }
}
```

## 🔥 扩展追问 1：如何区分正常流量和异常流量（攻击）？

> **A:** 核心是**行为基线 + 特征识别**：

| 特征 | 正常流量 | 攻击流量 |
|------|---------|---------|
| 来源分布 | 分散，多地区 | 集中，单一 IP 段 |
| 请求规律 | 有业务规律 | 短时间内请求相同接口 |
| 请求头 | 完整 UA/Referer | 缺失或伪造 |
| 连接特征 | 正常 TCP 时序 | 短连接 + 高并发 |

```bash
# CC 攻击特征识别
# 1. 大量相同请求路径
awk '{print $7}' access.log | sort | uniq -c | sort -rn | head -10

# 2. 短时间单一 IP 请求量
awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -10
# 如果某个 IP > 10000 次/分钟 = 异常

# 3. 异常 User-Agent
awk -F'"' '{print $6}' access.log | sort | uniq -c | sort -rn | head -10
# 无 UA 或 UA 为 python/curl = 可疑
```

## 🔥 扩展追问 4：MySQL 的网络 IO 高如何排查？

> **A:** 核心是**慢查询 + 连接数 + 大结果集**：

```sql
-- 1️⃣ 查看当前连接数
SHOW STATUS LIKE 'Threads_connected';
SHOW STATUS LIKE 'Max_used_connections';

-- 2️⃣ 查看流量最大的查询
SHOW FULL PROCESSLIST;
-- 找到 Command != Sleep 且 Time 长的

-- 3️⃣ 慢查询日志分析
-- 开启慢查询
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;

-- 分析慢查询
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log

-- 4️⃣ 查看大结果集查询
SELECT 
    id, user, host, db, command, time, 
    SUBSTRING(info, 1, 100) as query
FROM information_schema.processlist
WHERE info IS NOT NULL
ORDER BY LENGTH(info) DESC
LIMIT 10;

-- 5️⃣ 全表扫描查询
SELECT * FROM performance_schema.events_statements_summary_by_digest
ORDER BY SUM_ROWS_EXAMINED DESC LIMIT 10;
```

```bash
# MySQL 流量分析
# 查看客户端连接的网络流量
mysqladmin ext -i 1 | grep -E "Bytes_received|Bytes_sent"
```

## 一句话总结

> **网络 IO 飙高排查的核心是分层定位：sar/iftop 确认是入向还是出向 → ss/tcpdump 定位目标 IP → Arthas/SkyWalking 定位到具体接口 → 最终在代码层找到大数据传输的根因。**

## 面试加分话术

> "之前遇到过一次网络 IO 告警，带宽飙到 90%。一开始以为是用户量涨了，但翻了业务曲线是平稳的。后来用 tcpdump 抓包发现大量请求都打到同一个 IP，一看是个数据同步接口。再用 Arthas trace 发现，是 DBA 写的定时任务每次同步全量用户头像 Base64，一跑起来就是 10GB 流量打出去。后来改成增量同步 + 头像走 CDN，流量降了 80%。这类问题的关键是**不要只看总量，要拆解到每个 IP、每个接口**，才能定位到根因。"
