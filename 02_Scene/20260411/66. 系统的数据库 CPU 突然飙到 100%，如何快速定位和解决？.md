## 2. 核心排查命令与代码

### 2.1 第一现场：查看正在执行的 SQL

```sql
-- ========== 1. 查看所有连接（MySQL 命令行）==========
-- 这是排查的第一步，立刻执行！
SHOW FULL PROCESSLIST;

-- 或者查询 information_schema
SELECT 
    ID,
    USER,
    HOST,
    DB,
    COMMAND,
    TIME,          -- 执行时间（秒），越大的越可疑
    STATE,
    LEFT(INFO, 100) AS SQL_TEXT
FROM information_schema.processlist
WHERE COMMAND != 'Sleep'
ORDER BY TIME DESC;

-- ========== 2. 查找执行时间超过 30 秒的查询 ==========
SELECT 
    ID,
    USER,
    TIME,
    STATE,
    INFO
FROM information_schema.processlist
WHERE TIME > 30
  AND COMMAND = 'Query'
ORDER BY TIME DESC;

-- ========== 3. 查看锁等待 ==========
SELECT 
    r.trx_id AS waiting_trx_id,
    r.trx_mysql_thread_id AS waiting_thread,
    r.trx_query AS waiting_query,
    b.trx_id AS blocking_trx_id,
    b.trx_mysql_thread_id AS blocking_thread,
    b.trx_query AS blocking_query
FROM information_schema.INNODB_LOCK_WAITS w
JOIN information_schema.INNODB_TRX b ON b.trx_id = w.blocking_trx_id
JOIN information_schema.INNODB_TRX r ON r.trx_id = w.requesting_trx_id;

-- ========== 4. 查看当前锁 ==========
SELECT 
    ENGINE,
    LOCK_TYPE,
    LOCK_TABLE,
    LOCK_INDEX,
    LOCK_DATA
FROM information_schema.INNODB_LOCKS;

-- ========== 5. 查看事务 ==========
SELECT 
    trx_id,
    trx_state,
    trx_started,
    trx_rows_locked,
    trx_mysql_thread_id,
    LEFT(trx_query, 100) AS current_query
FROM information_schema.INNODB_TRX
WHERE trx_state = 'LOCK WAIT';
```

### 2.2 慢查询分析

```sql
-- ========== 1. 查看慢查询配置 ==========
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';
SHOW VARIABLES LIKE 'log_queries_not_using_indexes';

-- ========== 2. 查看慢查询日志（文件模式）==========
-- 设置慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 超过 1 秒记录
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';

-- 查看慢查询
SHOW GLOBAL STATUS LIKE 'Slow_queries';

-- ========== 3. performance_schema 分析（MySQL 5.6+）==========
-- 开启监控
UPDATE performance_schema.setup_consumers 
SET ENABLED = 'YES' WHERE NAME LIKE '%events_statements%';

UPDATE performance_schema.setup_instruments 
SET ENABLED = 'YES' WHERE NAME LIKE 'statement/sql/select';

-- 查看最慢的 SQL
SELECT 
    DIGEST_TEXT AS sql_query,
    COUNT_STAR AS exec_count,
    SUM_TIMER_WAIT / 1000000000000 AS total_time_sec,
    AVG_TIMER_WAIT / 1000000000000 AS avg_time_sec,
    SUM_ROWS_EXAMINED AS rows_scanned,
    SUM_ROWS_SENT AS rows_returned
FROM performance_schema.events_statements_summary_by_digest
ORDER BY SUM_TIMER_WAIT DESC
LIMIT 10;

-- ========== 4. 查看哪个表最慢 ==========
SELECT 
    OBJECT_SCHEMA,
    OBJECT_NAME,
    COUNT_STAR,
    SUM_TIMER_WAIT / 1000000000000 AS total_time_sec
FROM performance_schema.objects_summary_by_table
WHERE OBJECT_SCHEMA = 'your_database'
ORDER BY SUM_TIMER_WAIT DESC
LIMIT 10;
```

### 2.3 EXPLAIN 分析执行计划

```sql
-- ========== EXPLAIN 分析慢查询 ==========
-- 找到慢 SQL 后，用 EXPLAIN 分析执行计划
EXPLAIN SELECT * FROM orders 
WHERE status = 'PENDING' 
  AND create_time > '2026-04-01'
ORDER BY create_time DESC;

-- EXPLAIN ANALYZE（MySQL 8.0+）实际执行并分析
EXPLAIN ANALYZE SELECT * FROM orders 
WHERE status = 'PENDING'
ORDER BY create_time DESC;

-- ========== 常见问题识别 ==========
-- type=ALL：全表扫描（最慢）
-- type=index：索引扫描
-- type=range：范围扫描（可接受）
-- type=ref：非唯一索引扫描（可接受）
-- type=eq_ref：唯一索引扫描（好）
-- type=const：常量扫描（最好）

-- Extra 字段：
-- Using filesort：需要额外排序（慢）
-- Using temporary：需要临时表（慢）
-- Using index：覆盖索引（好）
-- Using where：需要回表（可接受）
```

### 2.4 索引分析与优化

```sql
-- ========== 1. 查看表的索引 ==========
SHOW INDEX FROM orders;

-- ========== 2. 检查未使用索引 ==========
SELECT 
    OBJECT_SCHEMA,
    OBJECT_NAME,
    INDEX_NAME,
    COUNT_STAR,
    SUM_TIMER_WAIT
FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE OBJECT_SCHEMA = 'your_database'
  AND INDEX_NAME IS NOT NULL
  AND COUNT_STAR = 0;  -- 未使用的索引

-- ========== 3. 检查大表 ==========
SELECT 
    TABLE_NAME,
    TABLE_ROWS,
    ROUND(DATA_LENGTH / 1024 / 1024, 2) AS data_size_mb,
    ROUND(INDEX_LENGTH / 1024 / 1024, 2) AS index_size_mb
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'your_database'
ORDER BY DATA_LENGTH DESC
LIMIT 20;

-- ========== 4. 重建索引/分析表 ==========
-- 定期执行，更新统计信息
ANALYZE TABLE orders;

-- 如果统计信息不准确，强制刷新
OPTIMIZE TABLE orders;  -- 会锁表，慎用！
```

## 🔥 扩展追问 1：突然 CPU 100% 的常见原因有哪些？

> **A:** 快速分类：外部流量暴涨 + 内部查询退化。

| 原因类型 | 具体场景 | 特征 |
|---------|---------|------|
| 流量暴涨 | 秒杀/活动导致并发飙升 | CPU 和连接数同时涨 |
| 慢查询 | 突然执行了大范围查询 | TIME 很大的查询 |
| 锁等待 | 某条 SQL 持锁太久 | 大量 Lock wait |
| 索引失效 | 突然走了全表扫描 | type=ALL |
| 连接池满 | 连接被耗尽 | 新请求排队等待 |
| 数据量突增 | 大数据任务 | 特定表肥大 |
| 配置变更 | buffer_pool 变小 | 缓存命中率下降 |

**排查优先级：**
```
1. 先用 SHOW PROCESSLIST 定位正在跑的 SQL
2. 如果有 TIME>30 的慢查询，先 KILL
3. 如果有 Lock wait，查 INNODB_LOCK_WAITS
4. 如果都不是，查慢查询日志
5. 最后分析 EXPLAIN 执行计划
```

## 🔥 扩展追问 2：如何紧急止血？

> **A:** 快速止血三板斧：KILL + 限流 + 从库切换。

```bash
# ========== 1. 紧急 KILL 慢查询 ==========
# 先查出来
SELECT ID, TIME, INFO FROM information_schema.processlist WHERE TIME > 30;

# 然后杀掉（假设 ID 是 123）
KILL 123;

# 如果有很多，可以批量 KILL
# 杀掉所有执行超过 60 秒的查询
SELECT CONCAT('KILL ', ID, ';') FROM information_schema.processlist 
WHERE TIME > 60 AND COMMAND = 'Query'
INTO OUTFILE '/tmp/kill_slow.sql';

# 然后执行
SOURCE /tmp/kill_slow.sql;

# ========== 2. 紧急限流 ==========
# 应用层：Sentinel 熔断
@SentinelResource(value = "queryOrders", blockHandler = "fallback")
public List<Order> queryOrders(Long userId) {
    return orderMapper.selectByUserId(userId);
}

public List<Order> fallback(Long userId, BlockException e) {
    return Collections.emptyList();  // 降级返回空
}

# ========== 3. 紧急切从库 ==========
# 读写分离，临时把读流量切到从库
# 写流量先暂停

# MySQL 主从切换（需要提前配置好）
STOP SLAVE;
CHANGE MASTER TO MASTER_HOST='slave_ip';
START SLAVE;
```

## 🔥 扩展追问 3：如何预防数据库被打爆？

> **A:** 三层防线：架构 + 限流 + 监控。

```java
// ========== 1. 读写分离 ==========
@Configuration
public class DataSourceConfig {
    
    @Bean
    public DataSource routingDataSource() {
        RoutingDataSource routingDataSource = new RoutingDataSource();
        routingDataSource.setDefaultTargetDataSource(masterDataSource());
        
        Map<Object, Object> targetDataSources = new HashMap<>();
        targetDataSources.put("master", masterDataSource());
        targetDataSources.put("slave", slaveDataSource());
        
        routingDataSource.setTargetDataSources(targetDataSources);
        return routingDataSource;
    }
}

// ========== 2. 分布式限流 ==========
@Service
public class DatabaseProtector {
    
    // 单机限流
    private RateLimiter dbLimiter = RateLimiter.create(500);  // 每秒 500 QPS
    
    // 分布式限流（Redis）
    public boolean tryAcquire(String key, int qps) {
        String lua = 
            "local current = redis.call('INCR', KEYS[1]) " +
            "if current == 1 then redis.call('EXPIRE', KEYS[1], 1) end " +
            "return current";
        
        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(lua, Long.class),
            Collections.singletonList("ratelimit:" + key)
        );
        
        return result != null && result <= qps;
    }
    
    public <T> T executeWithLimit(String sql, Supplier<T> supplier) {
        if (!tryAcquire(sql, 100)) {
            throw new BizException("系统繁忙，请稍后重试");
        }
        return supplier.get();
    }
}

// ========== 3. 监控告警 ==========
# Prometheus + Grafana 监控
# 关键指标：
# - MySQL Threads_connected / max_connections
# - MySQL CPU usage
# - MySQL Questions / Queries per second
# - Slow_queries 数量
# - InnoDB Row lock wait time
# - Buffer pool hit rate
```

## 🔥 扩展追问 4：死锁问题怎么排查和解决？

> **A:** 死锁是锁等待的极端情况，MySQL 会自动检测并回滚一条事务。

```sql
-- ========== 1. 查看死锁日志 ==========
-- MySQL 错误日志中会记录死锁信息
SHOW ENGINE INNODB STATUS;

-- ========== 2. 最近一次死锁详情 ==========
-- 输出示例：
-- LATEST DETECTED DEADLOCK
-- Transaction 1: ...
--   LOCK WAIT ... 
--   ...
-- Transaction 2: ...
--   ...
-- WE ROLL BACK TRANSACTION (1)  -- 回滚了事务 1

-- ========== 3. 死锁原因分析 ==========
-- 典型场景：
-- T1: BEGIN; UPDATE orders SET status=1 WHERE id=1;  -- 锁住 id=1
-- T2: BEGIN; UPDATE orders SET status=1 WHERE id=2;  -- 锁住 id=2
-- T1: UPDATE orders SET status=2 WHERE id=2;  -- 等待 id=2
-- T2: UPDATE orders SET status=2 WHERE id=1;  -- 等待 id=1 → 死锁！

-- ========== 4. 解决方案 ==========
-- 方案一：按固定顺序访问资源
-- 始终先更新 id 小的，再更新 id 大的

-- 方案二：减少锁范围
UPDATE orders SET status=1 WHERE id=1 AND status=0;  -- 加条件，减少锁

-- 方案三：降低事务隔离级别
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 方案四：使用乐观锁
UPDATE orders SET status=1, version=version+1 
WHERE id=1 AND version=1;  -- CAS 模式
```

## 🔥 扩展追问 5：Buffer Pool 命中率低也会导致 CPU 高吗？

> **A:** 会，缓存命中率低意味着大量磁盘 IO 和随机读写，CPU 等待 IO。

```sql
-- ========== 1. 查看 Buffer Pool 状态 ==========
SHOW ENGINE INNODB STATUS\G

-- 看 Buffer pool hit rate
-- Innodb_buffer_pool_read_requests: 总读请求
-- Innodb_buffer_pool_reads: 物理磁盘读（缓存未命中）

-- 计算命中率
SELECT 
    FORMAT(1 - (a.variable_value / b.variable_value), 4) AS hit_rate
FROM performance_schema.global_status a
JOIN performance_schema.global_status b
ON a.variable_name = 'Innodb_buffer_pool_reads'
AND b.variable_name = 'Innodb_buffer_pool_read_requests';

-- ========== 2. 查看 Buffer Pool 使用情况 ==========
SHOW STATUS LIKE 'Innodb_buffer_pool_pages%';

-- ========== 3. 优化建议 ==========
-- 如果命中率 < 95%，需要增大 buffer_pool

-- 设置 buffer_pool 大小（建议 OS 内存的 60-80%）
SET GLOBAL innodb_buffer_pool_size = 13421772800;  -- 10GB

-- 多实例 buffer_pool（MySQL 5.7+）
innodb_buffer_pool_instances = 4  -- 分成 4 个实例，减少锁竞争
```

## 一句话总结

> **数据库 CPU 100% 的排查顺序是：先 `SHOW PROCESSLIST` 找正在执行的慢 SQL、再查 `INNODB_LOCK_WAITS` 定位锁等待、然后 `EXPLAIN` 分析执行计划找全表扫描，紧急处理用 `KILL` 杀掉慢查询、限流熔断保护、切换从库分压，预防靠读写分离 + 索引优化 + 监控告警。**

> "之前凌晨两点被报警叫醒，线上数据库 CPU 100%。我先用 `SHOW FULL PROCESSLIST` 扫了一眼，发现有个 SQL 执行了 3 分钟还没完，看了下是 `SELECT * FROM orders WHERE create_time > 'xxx'` 查了全表。立即 `KILL 123` 杀掉，然后去查慢查询日志，发现这个查询平时只要 0.5 秒，今晚突然慢是因为 `create_time` 字段没建索引，数据量涨到 5000 万后变全表扫描了。后来紧急加索引（`ALTER TABLE orders ADD INDEX idx_create_time(create_time)`），同时把查询条件加上 `LIMIT 100` 限制返回行数，CPU 5 分钟内恢复正常。第二天复盘，加了查询超时配置（`SET GLOBAL max_execution_time = 3000`），超过 3 秒自动 kill，防止类似问题再发生。"
