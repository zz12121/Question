### 完整排查流程

```bash
# ========== 第一步：快速定位重启原因 ==========

# 1.1 查看 Pod 基本状态和重启次数
kubectl get pod <pod-name> -n <namespace> -o wide

# 输出示例：
# NAME                    READY   STATUS          RESTARTS   AGE
# order-service-5d8f9     1/1     Running         3          15m
# ⚠️ RESTARTS 列显示已重启 3 次！

# 1.2 查看详细事件（核心！）
kubectl describe pod <pod-name> -n <namespace>

# 关键输出区域：
/*
Last State: Terminated
  Reason: OOMKilled
  Exit Code: 137
  Started: Mon, 14 Apr 2026 10:00:00 +0800
  Finished: Mon, 14 Apr 2026 10:05:00 +0800

⚠️ Exit Code 137 = OOMKilled（内存超限）
⚠️ Exit Code 1  = 业务异常退出
⚠️ Exit Code 143 = SIGTERM 正常终止
⚠️ Exit Code 0  = 正常退出（可能主动重启）
*/


# ========== 第二步：查看日志 ==========

# 2.1 查看当前日志
kubectl logs <pod-name> -n <namespace>

# 2.2 查看上次容器的日志（关键！上次崩溃前的日志）
kubectl logs <pod-name> -n <namespace> --previous

# 2.3 实时查看日志
kubectl logs -f <pod-name> -n <namespace>

# 2.4 如果是多容器 Pod
kubectl logs <pod-name> -n <namespace> -c <container-name>


# ========== 第三步：排查具体原因 ==========

# 3.1 资源使用情况
kubectl top pod <pod-name> -n <namespace>

# 3.2 Pod 的资源限制
kubectl get pod <pod-name> -n <namespace> -o jsonpath='{.spec.containers[0].resources}'

# 3.3 查看 Events（K8s 事件）
kubectl get events -n <namespace> --sort-by='.lastTimestamp'

# 3.4 查看 Node 状态（可能是 Node 问题）
kubectl get nodes
kubectl describe node <node-name>


# ========== 第四步：进入容器调试 ==========

# 4.1 临时进入容器
kubectl exec -it <pod-name> -n <namespace> -- /bin/sh

# 4.2 如果镜像没有 sh，用 debug 临时容器（K8s 1.23+）
kubectl debug <pod-name> -it --image=busybox --share-processes --copy-to=debug-pod

# 4.3 查看进程和内存
top
free -h
ps aux
```

### 常见 Exit Code 分析

```
┌─────────────────────────────────────────────────────────────────┐
│                  常见 Exit Code 含义                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Code 0                                                         │
│  含义：正常退出                                                  │
│  场景：主动 System.exit(0)，或者进程正常完成                      │
│  处理：检查是否误配置了 restartPolicy                            │
│                                                                 │
│  Code 1                                                         │
│  含义：业务异常退出                                              │
│  场景：未捕获的 RuntimeException、main() 抛出异常                  │
│  处理：kubectl logs --previous 查看错误日志                       │
│                                                                 │
│  Code 137 (128 + 9)                                            │
│  含义：SIGKILL，99% 是 OOMKilled                                 │
│  场景：容器内存超过 limits，被 K8s kill 掉                        │
│  处理：增加 memory limits，或优化内存使用                         │
│                                                                 │
│  Code 143 (128 + 15)                                            │
│  含义：SIGTERM，正常终止                                         │
│  场景：kubectl delete、更新 Deployment、滚动发布                  │
│  处理：正常行为，无需干预                                        │
│                                                                 │
│  Code 139 (128 + 11)                                            │
│  含义：SIGSEGV，段错误                                           │
│  场景：JVM crash、Native 代码问题                                 │
│  处理：查 dmesg 找 core dump，JVM 配 -XX:+HeapDumpOnOutOfMemory  │
│                                                                 │
│  Code 134 (128 + 6)                                            │
│  含义：SIGABRT，异常终止                                         │
│  场景：assert 失败、FATAL 级别日志                               │
│  处理：检查代码异常日志                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```java
    @GetMapping("/health")
```

## 🔥 扩展追问 1：OOMKilled 如何定位和解决？

> **A:** OOMKilled 是容器被 K8s kill 掉，分两种情况：

```bash
# 1. 查看 Node 级别 OOM
dmesg | grep -i "killed process"
# 输出：[Mon Apr 14 10:00:00] Memory cgroup out of memory: Killed process 12345

# 2. 查看 Pod 内存使用趋势
kubectl top pod <pod-name> -n <namespace> --sort-by=memory

# 3. 如果是 Java 服务，检查 JVM 堆外内存
# JVM 堆外内存 = 直接内存(NIO) + 元空间 + 线程栈 + JNI + MappedFile
# 默认 JVM 只限制堆内存，容器限制的是整个进程内存
```

```yaml
# ========== JVM + K8s 内存配置最佳实践 ==========
apiVersion: v1
kind: Pod
metadata:
  name: java-app
spec:
  containers:
  - name: java-app
    image: openjdk:11
    
    # 容器限制 1G
    resources:
      limits:
        memory: "1Gi"
        cpu: "500m"
    
    env:
    # 方式1：固定堆内存（保守）
    - name: JAVA_OPTS
      value: "-Xmx768m -Xms768m -XX:+UseG1GC"  
      # 768m < 1Gi，给非堆内存留 256m
    
    # 方式2：动态计算（推荐）
    - name: JAVA_OPTS
      value: >-
        -XX:MaxRAMPercentage=75.0
        -XX:InitialRAMPercentage=75.0
        -XX:+UseG1GC
        -XX:+HeapDumpOnOutOfMemoryError
      # 75% 留给堆，25% 留给元空间/线程/GC 开销
```

```java
// ========== 排查内存泄漏（常见 OOM 原因）==========

@Service
public class MemoryLeakService {
    
    // ⚠️ 坑1：静态集合不断增长
    private static final Map<String, Object> cache = new HashMap<>();
    
    public void addToCache(String key, Object value) {
        // 没有清理逻辑，会 OOM
        cache.put(key, value);  // ⚠️ 内存泄漏
    }
    
    // ✅ 修复：用 WeakHashMap 或 LRU 缓存
    private static final Map<String, Object> safeCache = 
        new ConcurrentHashMap<>();
    private static final int MAX_SIZE = 10000;
    
    public void addToCacheSafe(String key, Object value) {
        if (safeCache.size() >= MAX_SIZE) {
            // LRU 淘汰
            String oldestKey = safeCache.keySet().iterator().next();
            safeCache.remove(oldestKey);
        }
        safeCache.put(key, value);
    }
    
    // ⚠️ 坑2：ThreadLocal 未清理
    @RequestMapping("/test")
    public void test() {
        // ThreadLocal 用完没 remove()
        threadLocal.set(new byte[10 * 1024 * 1024]);  // ⚠️ 每次请求 10M
        // 方法结束，threadLocal.set() 不会被 GC
    }
    
    // ✅ 修复：用完清理
    public void testFixed() {
        try {
            threadLocal.set(new byte[10 * 1024 * 1024]);
            // 业务逻辑
        } finally {
            threadLocal.remove();  // ⚠️ 必须清理
        }
    }
}
```

## 🔥 扩展追问 2：Liveness Probe 失败导致重启怎么处理？

> **A:** Liveness Probe 失败是 K8s 认为容器不健康，主动重启 Pod：

```yaml
# ========== 探针配置不当的场景 ==========

livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 5     # ⚠️ 启动 5 秒就开始探
  periodSeconds: 10          # 每 10 秒探一次
  failureThreshold: 1         # ⚠️ 1 次失败就重启
  successThreshold: 1
  timeoutSeconds: 1

# 问题：应用启动要 30 秒 → 前 25 秒探针都失败 → Pod 一启动就被 kill
# 正确：initialDelaySeconds 要大于应用启动时间
```

```yaml
# ========== 正确的探针配置 ==========

livenessProbe:
  httpGet:
    path: /health/live
    port: 8080
  # 启动 60 秒后才开始探活（给应用充足启动时间）
  initialDelaySeconds: 60
  periodSeconds: 15           # 每 15 秒探一次（不要太频繁）
  timeoutSeconds: 5           # 5 秒超时
  failureThreshold: 3          # 连续 3 次失败才重启（容忍短暂抖动）
  
readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080
  initialDelaySeconds: 10     # 启动 10 秒后开始探就绪
  periodSeconds: 5
  timeoutSeconds: 3
  failureThreshold: 2          # 2 次失败标为不就绪
  successThreshold: 1
```

```java
// ========== 健康检查接口实现 ==========

@RestController
@RequestMapping("/health")
public class HealthController {
    
    // Liveness Probe：只检查进程是否活着，不依赖外部
    @GetMapping("/live")
    public ResponseEntity<?> live() {
        // ✅ 只检查 JVM 进程
        RuntimeMXBean runtime = ManagementFactory.getRuntimeMXBean();
        long uptime = runtime.getUptime();
        
        if (uptime > 0) {
            return ResponseEntity.ok(Map.of(
                "status", "UP",
                "uptime", uptime
            ));
        }
        return ResponseEntity.status(503).body(Map.of("status", "DOWN"));
    }
    
    // Readiness Probe：检查能否接收流量（可以依赖外部，但要快）
    @GetMapping("/ready")
    public ResponseEntity<?> ready() {
        List<String> checks = new ArrayList<>();
        boolean allHealthy = true;
        
        // 检查数据库连接
        try {
            jdbcTemplate.queryForObject("SELECT 1", Integer.class);
            checks.add("database:OK");
        } catch (Exception e) {
            checks.add("database:FAIL");
            allHealthy = false;
        }
        
        // 检查 Redis
        try {
            redisTemplate.opsForValue().get("health:check");
            checks.add("redis:OK");
        } catch (Exception e) {
            checks.add("redis:FAIL");
            // ⚠️ Redis 不是必需的，可以降级
        }
        
        if (allHealthy) {
            return ResponseEntity.ok(Map.of(
                "status", "READY",
                "checks", checks
            ));
        }
        
        // 不健康，但不要返回 500，否则 K8s 会杀 Pod
        return ResponseEntity.status(503).body(Map.of(
            "status", "NOT_READY",
            "checks", checks
        ));
    }
}
```

## 🔥 扩展追问 3：CrashLoopBackOff 是什么？怎么处理？

> **A:** CrashLoopBackOff 是 K8s 检测到容器反复崩溃，进入**退避重启**模式：

```
┌─────────────────────────────────────────────────────────────────┐
│              CrashLoopBackOff 退避策略                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   重启次数   │   等待时间                                        │
│   ────────────────────────────────────                          │
│      1       │    10 秒                                         │
│      2       │    20 秒                                         │
│      3       │    40 秒                                         │
│      4       │    80 秒                                         │
│      5       │   160 秒                                         │
│     ...      │   逐渐增加，最长 5 分钟                            │
│                                                                 │
│   原因：防止崩溃的 Pod 疯狂重启消耗资源                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```bash
# 查看重启详情
kubectl describe pod <pod-name> | grep -A 10 "Last State"

# 输出示例：
/*
Last State: Terminated
  Reason: Error
  Exit Code: 1
  Started: Sun, 13 Apr 2026 09:00:00
  Finished: Sun, 13 Apr 2026 09:00:01
  
  Last State: Terminated
  Reason: Error
  Exit Code: 1
  Started: Sun, 13 Apr 2026 09:00:30
  Finished: Sun, 13 Apr 2026 09:00:31
*/
```

```java
// ========== 常见 CrashLoopBackOff 原因 ==========

// 原因1：应用启动失败但容器没有退出
public static void main(String[] args) {
    // ❌ 启动失败，但 JVM 继续运行
    try {
        initDatabase();
    } catch (Exception e) {
        // 只打印日志，没有退出
        log.error("数据库连接失败", e);
        // ❌ 应该退出，让 K8s 重启
    }
    
    // 应用继续运行，但功能不完整
    SpringApplication.run(App.class, args);
}

// ✅ 修复：启动失败时主动退出
public static void main(String[] args) {
    try {
        initDatabase();
    } catch (Exception e) {
        log.error("启动失败，数据库不可用，进程退出");
        System.exit(1);  // ⚠️ 主动退出，让 K8s 知道启动失败
    }
    SpringApplication.run(App.class, args);
}

// 原因2：健康检查超时（应用启动慢）
// → 增加 initialDelaySeconds 或增加探针超时时间

// 原因3：依赖服务不可用
// → 增加启动重试 + initialDelaySeconds
```

## 🔥 扩展追问 4：Node 压力导致 Pod 被驱逐怎么办？

> **A:** Pod Eviction（驱逐）是 Node 资源不足时，K8s 自动迁移 Pod：

```bash
# 1. 查看驱逐事件
kubectl get events -n <namespace> | grep -i evict

# 2. 查看 Node 资源情况
kubectl describe node <node-name> | grep -A 10 "Conditions"

# 3. 常见驱逐原因
# - MemoryPressure：内存不足
# - DiskPressure：磁盘不足
# - PIDPressure：进程数过多
# - NetworkUnavailable：网络不可用
```

```yaml
# ========== Pod 优先级配置（防止核心 Pod 被驱逐）==========
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 100000
globalDefault: false
description: "高优先级 Pod，不会被轻易驱逐"

---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  template:
    spec:
      priorityClassName: high-priority  # ⚠️ 高优先级
      containers:
      - name: order-service
        image: order-service:1.0.0
```

## 🔥 扩展追问 5：如何通过 Prometheus + Grafana 监控 Pod 重启？

> **A:** 提前发现 Pod 重启趋势：

```yaml
# ========== Prometheus 告警规则 ==========
groups:
- name: pod-restart-alert
  rules:
  # Pod 重启次数告警
  - alert: PodRestartHigh
    expr: |
      sum by (namespace, pod) (
        increase(kube_pod_container_status_restarts_total[5m])
      ) > 3
    for: 2m
    labels:
      severity: warning
    annotations:
      summary: "Pod 重启次数过多"
      description: "Pod {{ $labels.namespace }}/{{ $labels.pod }} 在 5 分钟内重启超过 3 次"

  # OOMKilled 告警
  - alert: PodOOMKilled
    expr: |
      sum by (namespace, pod) (
        kube_pod_container_status_last_terminated_reason{reason="OOMKilled"}
      ) > 0
    labels:
      severity: critical
    annotations:
      summary: "Pod 被 OOMKilled"
      description: "Pod {{ $labels.namespace }}/{{ $labels.pod }} 内存超限被杀"
```

```java
// ========== 自定义指标暴露 Pod 健康状态 ==========

@Component
public class PodMetricsExporter {
    
    private static final Counter RESTART_COUNTER = Counter.build()
        .name("app_pod_restart_total")
        .labelNames("reason")
        .register();
    
    @Autowired
    private MeterRegistry meterRegistry;
    
    @PostConstruct
    public void init() {
        // 暴露 Pod 内存使用
        meterRegistry.gauge("app_memory_usage_bytes", 
            Collections.singletonList(Tag.of("pod", getPodName())),
            this, PodMetricsExporter::getMemoryUsage);
    }
    
    /**
     * 健康检查时记录重启原因
     */
    public void recordRestart(String reason) {
        RESTART_COUNTER.labels(reason).inc();
    }
}
```

> **总结：K8s Pod 重启排查 → kubectl describe 看事件 + logs --previous 看日志 + Exit Code 定原因 + 资源/探针/依赖三排查**
>
> **核心是：Exit Code 137 = OOM，Exit Code 1 = 业务异常，Exit Code 143 = 正常终止；OOM 调内存/JVM 参数，业务异常查日志，探针失败调配置。**

> "我之前遇到过一次诡异的 Pod 频繁重启事件。每次重启间隔 5 分钟，重启了 8 次。第一时间 describe pod 看到 Exit Code 137，典型的 OOMKilled。但奇怪的是应用的内存使用监控一直很正常，没超过 limits。后来深入排查发现是 JVM 的 MetaSpace（元空间）不断增长导致的 OOM——不是堆内存，是元空间。原因是动态类加载没有卸载，再加上 Arthas 热加载插件泄漏。后来加上 -XX:MaxMetaspaceSize=256m，并且禁止了 Arthas 的热加载，问题才彻底解决。这个经验告诉我：**Java 进程的内存不只是堆内存**，元空间、直接内存、线程栈都要在容器 limits 的考量范围内。"
