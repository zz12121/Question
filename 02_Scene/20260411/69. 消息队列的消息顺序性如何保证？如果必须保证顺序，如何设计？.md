## 1. 基础流程

### 为什么顺序难以保证？

```
┌─────────────────────────────────────────────────────────────────┐
│                    消息乱序的根源                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   生产者              Broker                 消费者               │
│                                                                 │
│   P1 ─────────┐                                               │
│              │                                                │
│   P2 ────────┼───▶ [Queue/Topic] ─────────▶ [Consumer 1]      │
│              │       (多分区/多队列)          (并行处理)          │
│   P3 ────────┘                                                │
│              │                                                │
│              ▼                                                │
│       可能出现的情况：                                         │
│       1. P1、P2、P3 网络延迟不同，到达顺序: P3 → P1 → P2         │
│       2. 多分区并行消费，各自独立消费进度                       │
│       3. Consumer 1 消费 P1 时 GC 暂停，P2 先处理完            │
│       4. Broker 副本同步不一致，Leader 切换后消息顺序变化       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 核心解决方案：按业务 Key 分区

```
┌─────────────────────────────────────────────────────────────────┐
│                    顺序保证的架构设计                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   订单消息:                                                    │
│   ┌─────────┐   ┌─────────┐   ┌─────────┐                     │
│   │ order   │   │ order   │   │ order   │                     │
│   │ 1001    │   │ 1002    │   │ 1003    │                     │
│   │ create  │   │ create  │   │ create  │                     │
│   └────┬────┘   └────┬────┘   └────┬────┘                     │
│        │              │              │                          │
│        ▼              ▼              ▼                          │
│   ┌─────────────────────────────────────────┐                   │
│   │         Hash(orderId) % partitionNum     │                   │
│   └─────────────────────────────────────────┘                   │
│              │              │              │                    │
│              ▼              ▼              ▼                    │
│        ┌────────┐     ┌────────┐     ┌────────┐                 │
│        │ Partition│   │ Partition│   │ Partition│                │
│        │   0      │   │   1      │   │   2      │                │
│        └────┬────┘     └────┬────┘     └────┬────┘                │
│             │               │               │                    │
│             ▼               ▼               ▼                    │
│        ┌─────────────────────────────────────────┐              │
│        │   同一 Partition → 同一 Consumer → 顺序处理  │             │
│        └─────────────────────────────────────────┘              │
│                                                                 │
│   ⚠️ 注意: 单 Partition 单 Consumer = 顺序 + 性能下降             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 代码实现

```java
// RocketMQ 示例
@Service
public class OrderMessageProducer {
    
    @Autowired
    private RocketMQTemplate rocketMQTemplate;
    
    /**
     * 生产订单消息，保证同一订单的消息路由到同一 Partition
     * 关键：使用 orderId 作为 MessageQueueSelector 的参数
     */
    public void sendOrderMessage(Long orderId, String eventType, Object payload) {
        OrderMessage message = new OrderMessage(orderId, eventType, payload);
        
        // 方式1：使用 MessageQueueSelector 自动路由
        rocketMQTemplate.asyncSend("order-topic:" + eventType, message, new MessageQueueSelector() {
            @Override
            public MessageQueue select(List<MessageQueue> mqs, Message msg, Object arg) {
                // arg 就是 orderId，保证同一 orderId 落到同一队列
                Long orderId = (Long) arg;
                int index = (int) (orderId % mqs.size());
                return mqs.get(index);
            }
        }, orderId);  // 第三个参数会传给 selector 的 arg
        
        // 方式2：业务简单时，直接用 orderId 做 shardingKey（RocketMQ 4.9+）
        // rocketMQTemplate.asyncSendOrderly("order-topic:" + eventType, message, String.valueOf(orderId));
    }
}

// Kafka 示例
@Service
public class OrderMessageProducer {
    
    @Autowired
    private KafkaTemplate<String, OrderMessage> kafkaTemplate;
    
    public void sendOrderMessage(Long orderId, String eventType, OrderMessage message) {
        // Kafka 天然支持按 key 哈希分区
        // 同一 orderId → 同一 Partition → 顺序保证
        kafkaTemplate.send("order-topic", String.valueOf(orderId), message)
            .addCallback(
                success -> log.info("订单消息发送成功: orderId={}", orderId),
                failure -> log.error("订单消息发送失败: orderId={}", orderId, failure)
            );
    }
}

// Kafka Consumer 消费配置
@Configuration
public class KafkaConsumerConfig {
    
    @Bean
    public ConsumerFactory<String, OrderMessage> consumerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ConsumerConfig.GROUP_ID_CONFIG, "order-consumer-group");
        // ⚠️ 关键：同一 Partition 只能被同一 Consumer 实例消费
        props.put(ConsumerConfig.MAX_POLL_RECORDS_CONFIG, "1");  // 控制每次拉取量
        return new DefaultKafkaConsumerFactory<>(props);
    }
    
    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, OrderMessage> kafkaListenerContainerFactory() {
        ConcurrentKafkaListenerContainerFactory<String, OrderMessage> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory());
        // ⚠️ 多分区多消费者并发消费 = 乱序风险
        // 如果必须保证顺序，只能单 Consumer 或按 Partition 分配固定 Consumer
        factory.setConcurrency(1);  // 强制单线程消费（性能代价大）
        return factory;
    }
}
```

## 🔥 扩展追问 1：消费端失败重试时怎么保证不乱序？

> **A:** 核心思路：**暂停消费 → 本地重试 → 恢复消费**，不丢消息也不乱序。

```java
@Service
public class OrderMessageConsumer {
    
    @Autowired
    private RocketMQTemplate rocketMQTemplate;
    
    private static final int MAX_RETRY = 3;
    
    @RocketMQMessageListener(topic = "order-topic", consumerGroup = "order-consumer-group")
    public void onMessage(OrderMessage message, Context context) {
        Long orderId = message.getOrderId();
        
        try {
            // 业务处理
            processOrder(message);
            
        } catch (RetryableException e) {
            // 可重试异常：本地重试，不扔回队列
            int retryCount = getRetryCount(message);
            if (retryCount < MAX_RETRY) {
                // 本地延迟重试（用线程池，不影响消费进度）
                retryWithBackoff(message, retryCount);
                return;  // ⚠️ 关键：不 return ACK，消息还在 Pending
            }
            // 超过重试次数，扔到死信队列
            sendToDeadLetterQueue(message, "MAX_RETRY_EXCEEDED");
            
        } catch (NonRetryableException e) {
            // 不可重试异常：直接扔死信
            sendToDeadLetterQueue(message, e.getMessage());
        }
        
        // 正常处理完或扔死信后 ACK
        context.ack();
    }
    
    /**
     * 本地重试机制：保证消息不乱序
     * 原理：消息不离开 Consumer 内存，线程等待后重试，同一消息不干扰其他消息
     */
    private void retryWithBackoff(OrderMessage message, int retryCount) {
        long delayMs = (long) Math.pow(2, retryCount) * 1000;  // 1s, 2s, 4s
        
        scheduledExecutor.schedule(() -> {
            try {
                processOrder(message);
                log.info("订单消息重试成功: orderId={}", message.getOrderId());
            } catch (Exception e) {
                log.error("订单消息重试失败: orderId={}", message.getOrderId(), e);
                // 触发下一次重试逻辑
                incrementRetryCountAndRetry(message);
            }
        }, delayMs, TimeUnit.MILLISECONDS);
    }
}
```

## 🔥 扩展追问 2：多阶段业务流程如何保证全局顺序？

> **A:** 实际业务中，一个订单有多个阶段（创建 → 支付 → 发货 → 收货 → 评价），全局顺序几乎不可能，需要**分层有序**。

```java
// 分阶段顺序保证：每阶段内部有序，阶段间通过状态机协调
public class OrderStateMachine {
    
    // 订单状态流转
    // CREATE → PAYING → PAID → SHIPPING → SHIPPED → COMPLETED
    //                  ↘ TIMEOUT ←↗
    
    private final ConcurrentHashMap<Long, AtomicInteger> orderSequence = new ConcurrentHashMap<>();
    
    public boolean transition(Order order, OrderEvent event) {
        Long orderId = order.getId();
        
        // 1. 维护每个订单的消息序列号
        AtomicInteger seq = orderSequence.computeIfAbsent(orderId, k -> new AtomicInteger(0));
        int expectedSeq = seq.get();
        int actualSeq = event.getSequence();
        
        // 2. 序列号校验：消息顺序必须严格递增
        if (actualSeq != expectedSeq) {
            log.warn("订单[{}]消息乱序: expected={}, actual={}, 延迟处理", 
                orderId, expectedSeq, actualSeq);
            // 扔到延迟队列，稍后重试
            throw new OutOfOrderException("消息乱序，延迟处理");
        }
        
        // 3. CAS 更新序列号
        if (!seq.compareAndSet(expectedSeq, expectedSeq + 1)) {
            log.warn("订单[{}]并发更新序列号，重试", orderId);
            throw new ConcurrentUpdateException("并发更新序列号");
        }
        
        // 4. 执行状态流转
        return doTransition(order, event);
    }
}
```

## 🔥 扩展追问 3：Kafka 和 RocketMQ 顺序方案有何不同？

> **A:** 两者设计哲学不同：

| 维度 | Kafka | RocketMQ |
|------|-------|----------|
| **顺序类型** | Partition 内有序 | MessageQueue 内有序 |
| **保证方式** | 按 key 哈希到同一 Partition | 业务指定 MessageQueueSelector |
| **消费模式** | 单 Partition 单 Consumer 实例（通过 partition.assignment.strategy） | 顺序消费监听器 MessageListenerOrderly |
| **乱序场景** | Partition 扩缩容时 rebalance 可能短暂乱序 | Broker 切换时可能短暂乱序 |
| **性能** | 高（批量拉取 + 零拷贝） | 中等（逐条处理 + 事务支持） |
| **死信处理** | 需手动实现 | 支持配置死信队列 |

```java
// Kafka 顺序消费：固定 Partition 分配
@Configuration
public class KafkaOrderConfig {
    
    @Bean
    public PartitionAssignor orderBasedAssignor() {
        // 自定义分配策略：按 orderId 哈希固定分配到特定 Consumer
        return (partitions, subscription) -> {
            // 固定分配策略，确保同一 orderId 始终分配给同一 Consumer
            Map<String, List<TopicPartition>> assignment = new HashMap<>();
            for (String memberId : subscription.values()) {
                assignment.put(memberId, new ArrayList<>());
            }
            for (int i = 0; i < partitions.size(); i++) {
                TopicPartition tp = partitions.get(i);
                int consumerIndex = i % assignment.size();
                String memberId = new ArrayList<>(assignment.keySet()).get(consumerIndex);
                assignment.get(memberId).add(tp);
            }
            return assignment;
        };
    }
}
```

## 🔥 扩展追问 4：性能与顺序必须二选一吗？

> **A:** 不一定，关键在于**拆分粒度**。

```
┌─────────────────────────────────────────────────────────────────┐
│              顺序保证 vs 性能的权衡策略                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  方案A: 全局顺序（性能最差）                                       │
│  ┌─────────────────────────────────────────┐                    │
│  │ 1个 Partition ───▶ 1个 Consumer ───▶ 顺序 │                    │
│  │              吞吐量 = 1台机器上限           │                    │
│  └─────────────────────────────────────────┘                    │
│                                                                 │
│  方案B: 按订单顺序（推荐）                                         │
│  ┌─────────────────────────────────────────┐                    │
│  │ 16个 Partition ──▶ 4个 Consumer           │                   │
│  │                                          │                    │
│  │ 同一订单 ──▶ 同一 Partition ──▶ 同一 Consumer  ──▶ 顺序         │
│  │ 不同订单 ──▶ 不同 Partition ──▶ 并行消费 ──▶ 高吞吐             │
│  │                                          │                    │
│  │ 吞吐量 ≈ 4台机器 × 单机吞吐                │                    │
│  └─────────────────────────────────────────┘                    │
│                                                                 │
│  方案C: 分组顺序（折中）                                          │
│  ┌─────────────────────────────────────────┐                    │
│  │ 用户维度分区: userId % 16                │                    │
│  │ 同一用户的所有操作有序                     │                    │
│  │ 不同用户可并行                            │                    │
│  └─────────────────────────────────────────┘                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**实际建议：**
- **支付场景**：必须按订单有序 → 订单 ID 分区
- **日志收集**：不需要有序 → 随机分区 + 最大吞吐
- **库存扣减**：同一 SKU 有序 → SKU ID 分区
- **用户画像更新**：同一用户有序 → userId 分区

---

> **总结：消息顺序保证 → 业务 Key 分区 + 消费端幂等 + 异常延迟处理**
>
> **核心是：同一业务 Key 的消息路由到同一 Consumer 消费，保证局部有序；跨 Key 的消息天然并行，不保证全局顺序。**

> "我之前做过一个支付通道的消息队列改造。原来用的是单队列单 Consumer，吞吐量只有 2000 TPS/秒。后来改成按订单 ID 哈希分区，16 个分区配 16 个 Consumer，吞吐量直接拉到 3 万 TPS，同时保证了同一订单的支付回调、扣款、发券消息有序。这个优化的关键是我们评估了业务场景：用户感知到的是单个订单的状态一致性，不需要全局跨订单的顺序保证，所以按订单分区是完全合理的。"
