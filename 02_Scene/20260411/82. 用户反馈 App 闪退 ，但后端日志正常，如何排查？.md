```java
        .build();
```
```bash
}
```

### 5.2 网络波动导致的崩溃

```java
// 场景：弱网环境下，JSON 解析不完整数据导致崩溃
// 后端返回：{"code":200,"data":{... incomplete json
// App 端：JSON.parse 直接抛异常

// 防御代码
public class SafeJsonParser {
    
    private static Gson gson = new GsonBuilder()
        .serializeNulls()        // 保留 null 序列化
        .setLenient()            // 容错解析
        .create();
    
    public static <T> T parse(String json, Class<T> clazz) {
        try {
            return gson.fromJson(json, clazz);
        } catch (JsonSyntaxException e) {
            // 记录异常数据，但不要让 App 崩溃
            CrashReport.postException("JSON_PARSE_ERROR", e, json);
            return null;
        }
    }
}
```

## 🔥 扩展追问 3：iOS 闪退和 Android 有何不同？

> **A:** 核心差异对比：

| 维度 | iOS | Android |
|------|-----|---------|
| 崩溃日志 | Xcode → Devices → View Logs | Bugly 平台 / adb logcat |
| 热修复 | 静默更新（App Store审核）| Tinker/RSx |
| 系统限制 | 严格，超时直接杀 | 可配置，后台存活 |
| 内存限制 | 软限制，触发 OOM | 硬限制，杀进程 |
| 闪退特征 | EXC_BAD_ACCESS 多 | ANR + Native 崩溃多 |

> **iOS 特有**：
> - WatchDog 超时（主线程阻塞 > 10s）
> - Memory Warning 收到后不释放 → OOM
> - 低内存杀进程（ Jetsam）

## 一句话总结

> **App 闪退排查的核心是：先从崩溃监控平台获取堆栈信息，判断崩溃类型（ANR/OOM/Native/空指针），结合设备分布缩小范围，再联动后端日志排查是否由异常数据触发，最后建立端到端监控体系防患于未然。**

> "我之前处理过一次 App 闪退投诉，用户说每次进入个人中心就崩，但后台日志完全正常。排查了一圈发现是后端有个接口在用户没设置头像时返回了 `avatar: null`，iOS 端没有做空判断，直接把 null 当作 URL 去加载，就崩了。后来我们在 App 端加了防御，后端也统一规范——永远不返回 null，改成返回空字符串或默认头像，彻底解决了。这类问题关键是**崩溃日志和时间点必须和后端请求关联起来**，才能定位到是客户端还是服务端的问题。"
