## 二、Cache Aside（旁路缓存）- 最常用

### 2.1 原理

```java
/**
 * Cache Aside 策略
 * 读：Cache 命中返回，miss 则查 DB 并写入 Cache
 * 写：先写 DB，再删除 Cache（而非更新）
 */
@Service
public class CacheAsideService {
    
    @Autowired private CacheManager cacheManager;
    @Autowired private UserMapper userMapper;
    
    private static final String USER_CACHE = "user_cache";
    
    /**
     * 读操作
     */
    public User getUser(Long userId) {
        Cache cache = cacheManager.getCache(USER_CACHE);
        
        // 1. 先查缓存
        User user = cache.get(userId, User.class);
        
        if (user != null) {
            log.debug("Cache hit: userId={}", userId);
            return user;
        }
        
        // 2. Cache miss，查数据库
        log.debug("Cache miss: userId={}", userId);
        user = userMapper.selectById(userId);
        
        if (user != null) {
            // 3. 写入缓存（注意：这里用 put，而非 get 再 put）
            cache.put(userId, user);
        }
        
        return user;
    }
    
    /**
     * 写操作 - 核心：先写 DB，再删 Cache
     */
    public void updateUser(Long userId, UserUpdateDTO dto) {
        // ⚠️ 顺序很重要！
        
        // 1. 先写 DB
        User user = userMapper.selectById(userId);
        BeanUtils.copyProperties(dto, user);
        user.setUpdatedAt(LocalDateTime.now());
        userMapper.updateById(user);
        
        // 2. 删除缓存（不是更新缓存！）
        Cache cache = cacheManager.getCache(USER_CACHE);
        cache.evict(userId);
        
        // ⚠️ 为什么是删除而不是更新？
        // 如果更新缓存失败 → DB 和 Cache 数据不一致
        // 删除缓存 → 下次读取时重新加载，最坏情况是 miss
    }
    
    /**
     * 删除操作
     */
    public void deleteUser(Long userId) {
        // 1. 先删 DB
        userMapper.deleteById(userId);
        
        // 2. 再删 Cache
        cache.evict(userId);
    }
}
```

### 2.2 为什么 Cache Aside 用"删除"而非"更新"？

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    删除 vs 更新 - 数据一致性对比                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  场景：用户 A 更新 user_id=1 的数据                                      │
│                                                                          │
│  ════════════════════════════════════════════════════════════════════   │
│                                                                          │
│  ❌ 错误方案：更新缓存                                                    │
│                                                                          │
│  T1: Thread-A 查 DB → user={name:"张三", age:20}                       │
│  T2: Thread-B 更新 DB → user={name:"李四", age:25}                     │
│  T3: Thread-B 更新 Cache → user={name:"李四", age:25}                   │
│  T4: Thread-A 更新 Cache → user={name:"张三", age:20}  ← 老数据！       │
│                                                                          │
│  结果：Cache 里是旧数据（张三 20），DB 里是新数据（李四 25）              │
│                                                                          │
│  ════════════════════════════════════════════════════════════════════   │
│                                                                          │
│  ✓ 正确方案：删除缓存                                                    │
│                                                                          │
│  T1: Thread-A 查 DB → user={name:"张三", age:20}                       │
│  T2: Thread-B 更新 DB → user={name:"李四", age:25}                     │
│  T3: Thread-B 删除 Cache                                               │
│  T4: Thread-A 查 Cache → miss → 重新查 DB → user={name:"李四", age:25} │
│                                                                          │
│  结果：最终一致，Cache 里是最新的                                        │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

## 三、Read Through（读穿透）

### 3.1 原理

```java
/**
 * Read Through 策略
 * 缓存自动加载数据，应用只访问缓存，缓存自己负责从 DB 加载
 */
@Service
public class ReadThroughService {
    
    // 方式1：使用 Spring Cache（推荐）
    @Service
    @Cacheable(value = "users", unless = "#result == null")
    public User getUser(Long userId) {
        // ⚠️ 实际查询 DB 的代码
        // 第一次调用：查 DB，结果自动写入缓存
        // 后续调用：直接走缓存
        return userMapper.selectById(userId);
    }
    
    // 方式2：自定义 CacheLoader
    public class UserCacheLoader {
        
        private LoadingCache<Long, User> cache;
        
        public UserCacheLoader() {
            // 创建 LoadingCache，指定 loader
            cache = CacheBuilder.newBuilder()
                .maximumSize(10000)
                .expireAfterWrite(30, TimeUnit.MINUTES)
                .build(new CacheLoader<Long, User>() {
                    @Override
                    public User load(Long userId) throws Exception {
                        // Cache miss 时自动调用此方法
                        return userMapper.selectById(userId);
                    }
                });
        }
        
        public User getUser(Long userId) {
            // 应用只调用 cache.get()
            // 不需要判断 miss，CacheLoader 自动处理
            try {
                return cache.get(userId);
            } catch (ExecutionException e) {
                throw new RuntimeException(e);
            }
        }
    }
}
```

### 3.2 与 Cache Aside 的区别

| 维度 | Cache Aside | Read Through |
|------|------------|--------------|
| 数据加载 | 业务代码控制 | 缓存自动加载 |
| 代码复杂度 | 较高（需判断 miss） | 较低（无感知）|
| 灵活性 | 高（可自定义加载逻辑） | 低（受限于 Loader）|
| 适用场景 | 复杂查询、关联加载 | 简单 Key-Value |

## 四、Write Through / Write Behind

### 4.1 Write Through（写穿透）

```java
/**
 * Write Through 策略
 * 写操作同时写 Cache 和 DB，Cache 和 DB 始终一致
 */
@Service
public class WriteThroughService {
    
    @Autowired private RedisTemplate<String, Object> redisTemplate;
    @Autowired private UserMapper userMapper;
    
    public void updateUser(User user) {
        // 1. 同时写 Cache 和 DB
        redisTemplate.opsForValue().set("user:" + user.getId(), user);
        userMapper.updateById(user);
        
        // 2. 如果 Cache 写入失败，删除 Cache（降级）
        // 这样下次读取会从 DB 加载，保证最终一致
    }
}
```

### 4.2 Write Behind（写回）

```java
/**
 * Write Behind 策略（又称 Write Back）
 * 写操作只写 Cache，异步批量写 DB
 */
@Service
public class WriteBehindService {
    
    private ConcurrentLinkedQueue<UserUpdateRequest> updateQueue = 
        new ConcurrentLinkedQueue<>();
    
    @Autowired private RedisTemplate<String, Object> redisTemplate;
    
    // 写操作：只写缓存
    public void updateUser(UserUpdateRequest request) {
        // 1. 写入缓存
        redisTemplate.opsForValue().set(
            "counter:" + request.getKey(), 
            request.getValue()
        );
        
        // 2. 加入写队列（异步批量写 DB）
        updateQueue.offer(request);
    }
    
    // 后台线程：批量写 DB
    @Scheduled(fixedDelay = 1000)
    public void flushToDB() {
        List<UserUpdateRequest> batch = new ArrayList<>();
        UserUpdateRequest request;
        
        // 批量取出（最多 100 条或 1 秒内最多）
        int count = 0;
        while ((request = updateQueue.poll()) != null && count++ < 100) {
            batch.add(request);
        }
        
        if (!batch.isEmpty()) {
            // 批量写入 DB
            userMapper.batchUpdate(batch);
            log.info("Write Behind: 批量写入 {} 条到 DB", batch.size());
        }
    }
}
```

## 七、缓存常见问题与解决方案

### 7.1 缓存穿透

```java
/**
 * 缓存穿透：查询不存在的数据，每次都打到 DB
 */
@Service
public class CachePenetrationService {
    
    // 方案1：缓存空值
    private static final String NULL_VALUE = "NULL";
    private static final long NULL_TTL = 5 * 60; // 空值缓存 5 分钟
    
    public User getUser(Long userId) {
        User user = cache.get(userId, User.class);
        if (user != null) {
            if (NULL_VALUE.equals(user)) {
                return null;  // 命中空值，返回 null
            }
            return user;
        }
        
        user = userMapper.selectById(userId);
        
        if (user == null) {
            // 缓存空值，防止穿透
            cache.put(userId, NULL_VALUE);
        } else {
            cache.put(userId, user);
        }
        
        return user;
    }
    
    // 方案2：布隆过滤器
    @Autowired private BloomFilter<Long> userIdFilter;
    
    public User getUser(Long userId) {
        // 布隆过滤器判断是否存在
        if (!userIdFilter.mightContain(userId)) {
            return null;  // 一定不存在
        }
        
        // 可能存在，查缓存和 DB
        User user = cache.get(userId, User.class);
        if (user != null) return user;
        
        user = userMapper.selectById(userId);
        if (user != null) cache.put(userId, user);
        
        return user;
    }
}
```

### 7.2 缓存击穿

```java
/**
 * 缓存击穿：热点 key 过期，瞬间大量请求打到 DB
 */
@Service
public class CacheBreakdownService {
    
    // 方案1：互斥锁
    public User getUserWithLock(Long userId) {
        String key = "user:" + userId;
        
        User user = cache.get(key, User.class);
        if (user != null) return user;
        
        // 获取分布式锁
        String lockKey = "lock:" + key;
        String lockValue = UUID.randomUUID().toString();
        
        Boolean locked = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, lockValue, 10, TimeUnit.SECONDS);
        
        if (Boolean.TRUE.equals(locked)) {
            try {
                // 双重检查
                user = cache.get(key, User.class);
                if (user != null) return user;
                
                // 查 DB
                user = userMapper.selectById(userId);
                if (user != null) cache.put(key, user);
                
                return user;
            } finally {
                // 释放锁
                if (lockValue.equals(redisTemplate.opsForValue().get(lockKey))) {
                    redisTemplate.delete(lockKey);
                }
            }
        } else {
            // 未获取到锁，短暂等待后重试
            try {
                Thread.sleep(50);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
            return getUserWithLock(userId);
        }
    }
    
    // 方案2：逻辑过期（热点数据永不过期，定期异步更新）
    public User getUserWithLogicExpire(Long userId) {
        String key = "user:" + userId;
        
        UserCache userCache = cache.get(key, UserCache.class);
        
        if (userCache == null) {
            // 缓存不存在，加载数据
            return loadAndCache(userId);
        }
        
        // 检查逻辑过期
        if (userCache.isExpired()) {
            // 逻辑过期，异步更新
            threadPool.submit(() -> loadAndCache(userId));
            
            // 返回旧数据（可能稍旧，但不阻塞）
            return userCache.getUser();
        }
        
        return userCache.getUser();
    }
    
    private User loadAndCache(Long userId) {
        User user = userMapper.selectById(userId);
        if (user != null) {
            UserCache userCache = new UserCache(user, 
                LocalDateTime.now().plusHours(1));  // 逻辑过期 1 小时
            cache.put("user:" + userId, userCache);
        }
        return user;
    }
}
```

### 7.3 缓存雪崩

```java
/**
 * 缓存雪崩：大量缓存同时过期，或缓存服务宕机
 */
@Configuration
public class CacheAvalancheConfig {
    
    // 方案1：随机过期时间
    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {
        // 给过期时间加随机值，避免同时过期
        Map<String, RedisCacheConfiguration> configs = new HashMap<>();
        
        configs.put("user", RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(30 + new Random().nextInt(10))));
        
        configs.put("product", RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(60 + new Random().nextInt(20))));
        
        return RedisCacheManager.builder(factory)
            .cacheDefaults(RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofMinutes(10)))  // 默认 10 分钟
            .withInitialCacheConfigurations(configs)
            .build();
    }
    
    // 方案2：Redis 高可用 + 本地缓存兜底
    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {
        // 两级缓存：Redis + Caffeine（本地）
        // 本地缓存作为兜底，Redis 挂了还能用
        
        return RedisCacheManager.builder(factory)
            .cacheDefaults(RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofMinutes(30)))
            .withCacheConfiguration("local", 
                CaffeineCacheConfiguration.defaultCacheConfig()
                    .maximumSize(10000)
                    .expireAfterWrite(Duration.ofMinutes(5)))  // 本地缓存 5 分钟
            .build();
    }
}
```

## 🔥 扩展追问 1：如何设计缓存的过期策略？

> **A:** 多层过期策略组合：

| 策略 | 实现 | 适用场景 |
|------|------|---------|
| **TTL 过期** | 所有 key 设置最大存活时间 | 通用，数据量可控 |
| **LRU/LFU** | 内存满时淘汰最少使用 | 本地缓存（Guava/Caffeine）|
| **主动过期** | 定时任务扫描删除 | 精确控制过期时间 |
| **惰性删除** | 访问时检查并删除 | 节省资源，有延迟 |

```java
// 分层过期策略
@Configuration
public class TieredCacheStrategy {
    
    // L1：本地缓存（Caffeine）- 热点数据，永不过期
    private Cache<Long, User> localCache = Caffeine.newBuilder()
        .maximumSize(10000)
        .recordStats()
        .build();
    
    // L2：Redis 缓存 - 适中过期时间
    // 默认 30 分钟
    private Duration redisTTL = Duration.ofMinutes(30);
    
    // L3：DB - 最终数据源
    
    public User getUser(Long userId) {
        // 1. 查本地缓存
        User user = localCache.getIfPresent(userId);
        if (user != null) {
            return user;
        }
        
        // 2. 查 Redis
        user = redisTemplate.opsForValue().get("user:" + userId);
        if (user != null) {
            localCache.put(userId, user);  // 回填 L1
            return user;
        }
        
        // 3. 查 DB
        user = userMapper.selectById(userId);
        if (user != null) {
            localCache.put(userId, user);
            redisTemplate.opsForValue().set("user:" + userId, user, redisTTL);
        }
        
        return user;
    }
}
```

## 🔥 扩展追问 2：Redis Cluster 模式下如何做缓存？

> **A:** Key 设计与路由策略：

```java
@Configuration
public class RedisClusterCacheConfig {
    
    // 问题：同一条数据的缓存 key，可能分散在多个节点
    // 但我们的业务代码需要同时操作多个字段
    
    // 方案1：Key 按业务聚合
    // ❌ 不用：user:id:profile, user:id:setting, user:id:stat
    // ✓ 用：user:id（一次获取所有关联数据）
    
    // 方案2：使用 Hash 结构，减少 Key 数量
    public void cacheUserProfile(Long userId, UserProfile profile) {
        String key = "user:profile:" + userId;
        
        // 用 Hash 存储，一次操作获取所有字段
        redisTemplate.opsForHash().putAll(key, Map.of(
            "name", profile.getName(),
            "avatar", profile.getAvatar(),
            "bio", profile.getBio()
        ));
        
        redisTemplate.expire(key, Duration.ofMinutes(30));
    }
    
    // 方案3：跨节点操作（Pipeline）
    public void updateUserProfile(Long userId, Map<String, Object> updates) {
        String key = "user:profile:" + userId;
        
        // Pipeline 批量操作，减少网络往返
        redisTemplate.executePipelined((RedisCallback<Object>) connection -> {
            for (Map.Entry<String, Object> entry : updates.entrySet()) {
                connection.hashCommands().hSet(
                    key.getBytes(),
                    entry.getKey().getBytes(),
                    entry.getValue().toString().getBytes()
                );
            }
            return null;
        });
    }
}
```

## 🔥 扩展追问 3：热点数据如何优化？

> **A:** 多级缓存 + 本地缓存热点兜底：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        热点数据访问优化                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│   请求 ───▶ Nginx 本地缓存 ───▶ Redis ───▶ Redis Cluster ───▶ DB      │
│              (L1, 热点兜底)  (L2, 分布式缓存)  (数据分片)    (最终数据)  │
│              TTL=100ms        TTL=5min                                    │
│                                                                          │
│   热点数据识别：                                                          │
│   ┌─────────────────────────────────────────────────────────────────┐  │
│   │  1. 实时热点检测                                                  │  │
│   │     - Redis 计数器：每次访问 +1                                   │  │
│   │     - 超过阈值 → 推送到 Nginx + 本地缓存                        │  │
│   │                                                                  │  │
│   │  2. 历史热点预加载                                               │  │
│   │     - 基于历史数据，提前加载到本地缓存                            │  │
│   │     - 如：明星热搜、商品大促                                     │  │
│   │                                                                  │  │
│   │  3. 热点数据隔离                                                 │  │
│   │     - 单独集群部署，不与其他数据竞争                             │  │
│   │     - 专门的内存配额                                             │  │
│   └─────────────────────────────────────────────────────────────────┘  │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

## 一句话总结

> **缓存策略选型核心是：读多写少用 Cache Aside（最灵活）、简单 KV 用 Read Through（代码最简）、高并发计数用 Write Behind（写性能最高）。一致性要求高的场景（库存）慎用缓存，优先用数据库 + 原子操作。**

## 面试加分话术

> "我在上家公司做过商品详情的缓存优化，一开始用的是 Write Behind，想着写性能高嘛。但上线后发现问题：用户下单时显示有库存，Redis 里也是这个数，结果订单创建成功去扣库存时，发现 DB 里库存已经被另一个用户先扣了——Redis 还没来得及同步到 DB。这才发现 Write Behind 的弱一致性问题根本不适合交易链路。后来改成 Cache Aside + 延迟双删，才解决了一致性问题。所以选缓存策略，**先看业务对一致性的要求**，交易类场景必须强一致，宁可慢一点也不能错；统计类场景才适合 Write Behind。"
