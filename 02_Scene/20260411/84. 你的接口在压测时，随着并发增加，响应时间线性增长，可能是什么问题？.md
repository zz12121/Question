```java
// 方案1：减小锁粒度
@Service
public class OrderService {
    
    // 分段锁：根据订单ID hash 分桶
    private final ConcurrentHashMap<Long, ReentrantLock> orderLocks = 
        new ConcurrentHashMap<>();
    
    public Order createOrder(OrderDTO dto) {
        Long orderId = dto.getId();
        
        // 只锁当前订单，不影响其他订单
        ReentrantLock lock = orderLocks.computeIfAbsent(
            orderId % 100, k -> new ReentrantLock());
        
        lock.lock();
        try {
            // 业务逻辑
        } finally {
            lock.unlock();
        }
    }
}

// 方案2：分布式锁替代 JVM 锁（多实例部署）
@Service
public class DistributedOrderService {
    
    @Autowired private RedissonClient redisson;
    
    public Order createOrder(OrderDTO dto) {
        String lockKey = "order:lock:" + dto.getProductId();
        RLock lock = redisson.getLock(lockKey);
        
        lock.lock(10, TimeUnit.SECONDS);
        try {
            // 业务逻辑
        } finally {
            lock.unlock();
        }
    }
}
```

### 3.4 外部依赖瓶颈

```java
// 问题代码：同步调用，阻塞线程
@Service
public class OrderService {
    
    @Autowired private RestTemplate restTemplate;
    @Autowired private UserService userService;
    
    public OrderVO createOrder(Long userId, Long productId) {
        // ⚠️ 串行调用下游服务
        UserDTO user = userService.getUser(userId);      // RT 50ms
        ProductDTO product = productService.get(productId); // RT 30ms
        InventoryDTO inventory = inventoryService.check(productId); // RT 20ms
        
        // 累计 RT = 50 + 30 + 20 = 100ms
        // 并发 100 → RT = 100 × 100 = 10s ❌
        
        return buildOrder(user, product, inventory);
    }
}

// ❌ 反面案例：嵌套同步调用
@RestController
public class ProductController {
    
    public ProductVO getProduct(Long id) {
        // A → B → C → D（级联超时）
        ProductDTO product = productService.getProduct(id);
        return convert(product);
    }
}

@Service
public class ProductService {
    public ProductDTO getProduct(Long id) {
        // 每个调用都同步等待，100 并发 = 每层都排队
        CategoryDTO category = categoryClient.get(id);  // 阻塞等待
        return productMapper.selectById(id);
    }

```

**解决方案**：
```java
// 方案1：并行调用 CompletableFuture
public OrderVO createOrder(Long userId, Long productId) {
    // 并行获取，不等待
    CompletableFuture<UserDTO> userFuture = 
        CompletableFuture.supplyAsync(() -> userService.getUser(userId));
    CompletableFuture<ProductDTO> productFuture = 
        CompletableFuture.supplyAsync(() -> productService.get(productId));
    CompletableFuture<InventoryDTO> inventoryFuture = 
        CompletableFuture.supplyAsync(() -> inventoryService.check(productId));
    
    // 等待所有结果（取最长的时间，而非累加）
    CompletableFuture.allOf(userFuture, productFuture, inventoryFuture).join();
    
    // RT = max(50, 30, 20) = 50ms ✓
    return buildOrder(
        userFuture.get(), 
        productFuture.get(), 
        inventoryFuture.get()
    );
}

// 方案2：服务降级 + 超时控制
```

```java
@Service
public class OrderService {
    
    // ❌ 每次创建新对象
    public List<OrderItem> buildItems(List<Long> productIds) {
        List<OrderItem> items = new ArrayList<>();
        for (Long id : productIds) {
            items.add(new OrderItem(id)); // ⚠️ 频繁 new
        }
        return items;
    }
    
    // ✅ 对象池复用
    private final Recycler<OrderItem> itemRecycler = 
        Recycler.builder(OrderItem.class).build();
    
    public List<OrderItem> buildItems(List<Long> productIds) {
        List<OrderItem> items = new ArrayList<>();
        for (Long id : productIds) {
            items.add(itemRecycler.get().setProductId(id)); // 复用
        }
        // 用完归还
        return items;
    }
```

## 🔥 扩展追问 1：拐点增长和线性增长有何不同？

> **A:** 两者根因不同：

| 类型 | 增长模式 | 根因 | 典型场景 |
|------|---------|------|---------|
| **拐点增长** | 初期平稳，到达拐点后陡增 | 资源池耗尽（连接池/线程池） | 100 并发时正常，200 并发时 RT 暴增 |
| **线性增长** | 从一开始就是线性 | 串行瓶颈（锁/同步调用/全表扫描）| 并发 10 → 100ms，并发 100 → 1000ms |

## 🔥 扩展追问 2：如何快速验证是哪个瓶颈？

> **A:** 用「排除法」快速定位：

```bash
# 1. 先看线程状态
jstack <PID> | grep -c "BLOCKED"

# BLOCKED 数量 > 50 → 锁竞争问题

# 2. 看 DB 连接
curl http://localhost:8080/druid/datasource-stat.json | jq '.DataSourceStatList[0].activeCount'

# activeCount == maxActive → 连接池耗尽

# 3. 看 GC
jstat -gcutil <PID> 1000 10 | awk '{print $8}' 

# Old 区持续 > 80% → 内存问题

# 4. 看下游 RT
# 在接口里埋点记录：downstream_rt
# 如果 downstream_rt 总和 ≈ 接口 RT → 下游是瓶颈
```

## 一句话总结

> **响应时间线性增长的本质是「串行瓶颈」：连接池耗尽导致排队等待、synchronized 全局锁强制串行、全表扫描随并发累积、下游同步调用 RT 累加、频繁 FGC 导致 STW。定位方法是 jstack 看 BLOCKED、慢日志看 SQL、GC 日志看 Old 区占用、链路追踪看下游 RT。**

## 面试加分话术

> "之前压测一个订单接口，并发从 10 加到 100，RT 从 50ms 涨到了 500ms，基本就是 10 倍关系。一开始怀疑是数据库，但单独压 SQL 很快。后来 jstack 了一把，发现大量线程卡在同一个 synchronized 锁上——是个历史遗留代码，运营后台的配置读取用了全局锁，而订单和配置完全不相关。改成 ConcurrentHashMap 分段锁之后，同样的 100 并发，RT 稳定在 55ms。所以线性增长基本上就是**串行瓶颈**，要么是锁，要么是池，要么是单点。"
