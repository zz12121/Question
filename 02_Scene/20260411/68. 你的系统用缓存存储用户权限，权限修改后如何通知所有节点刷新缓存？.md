## 1. 基础流程

### 核心问题

多节点部署时，权限变更只写入了数据库，其他节点的 Redis 缓存还是旧数据，**缓存与数据库不一致**。

```
┌─────────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────────┤

│                                                                 │
```

### 方案一：Redis Pub/Sub（轻量级）

```java
// 1. 权限变更时，发布通知
@Service
public class PermissionService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    public void updatePermission(Long userId, List<String> newPermissions) {
        // 先更新数据库
        permissionMapper.updateByUserId(userId, newPermissions);
        
        // 发布权限变更通知
        redisTemplate.convertAndSend("permission:update", 
            JSON.toJSONString(new PermissionUpdateEvent(userId, "UPDATE")));
        
        // 如果是角色变更，影响该角色下所有用户
        if (roleChanged) {
            List<Long> affectedUserIds = getUsersByRole(roleId);
            for (Long uid : affectedUserIds) {
                redisTemplate.convertAndSend("permission:update",
                    JSON.toJSONString(new PermissionUpdateEvent(uid, "UPDATE")));
            }
        }
    }
}

// 2. 每个节点订阅通知，清除本地缓存
@Component
public class PermissionCacheSubscriber {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @PostConstruct
    public void init() {
        redisTemplate.getConnectionFactory().getConnection()
            .subscribe(new MessageListener() {
                @Override
                public void onMessage(Message message, byte[] pattern) {
                    String body = new String(message.getBody());
                    PermissionUpdateEvent event = JSON.parseObject(body, 
                        PermissionUpdateEvent.class);
                    
                    // 清除该用户的权限缓存
                    String cacheKey = "permission:user:" + event.getUserId();
                    redisTemplate.delete(cacheKey);
                    
                    log.info("已清除用户[{}]的权限缓存", event.getUserId());
                }
            }, "permission:update".getBytes());
    }
}
```

### 方案二：MQ 广播（可靠、跨语言）

```java
// 1. 使用 RocketMQ 广播模式
@Configuration
public class RocketMQConfig {
    
    @Bean
    public DefaultMQProducer permissionProducer() {
        DefaultMQProducer producer = new DefaultMQProducer("permission-update-group");
        producer.setNamesrvAddr("192.168.1.100:9876");
        producer.start();
        return producer;
    }
}

// 2. 权限变更时发送消息
@Service
public class PermissionService {
    
    @Autowired
    private DefaultMQProducer permissionProducer;
    
    public void updatePermission(Long userId, List<String> permissions) {
        permissionMapper.updateByUserId(userId, permissions);
        
        // 发送广播消息
        Message message = new Message("permission-topic", "update",
            String.valueOf(userId), JSON.toJSONString(permissions).getBytes());
        permissionProducer.send(message);
    }
}

// 3. 每个节点消费消息
@Component
@RocketMQMessageListener(
    topic = "permission-topic",
    consumerGroup = "permission-consumer-group",  // 注意：广播模式每节点独立消费
    messageModel = MessageModel.BROADCASTING
)
public class PermissionCacheConsumer implements RocketMQListener<String> {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Override
    public void onMessage(String body) {
        PermissionUpdateEvent event = JSON.parseObject(body, PermissionUpdateEvent.class);
        redisTemplate.delete("permission:user:" + event.getUserId());
        log.info("消费权限更新消息，清除缓存: userId={}", event.getUserId());
    }
}
```

### 方案三：定时轮询 + 版本号兜底

```java
// 定时任务：对比数据库版本号，有变化则刷新缓存
@Scheduled(fixedDelay = 30000)  // 每30秒
public void syncPermissionCache() {
    List<UserPermission> allPermissions = permissionMapper.selectAllWithVersion();
    
    for (UserPermission up : allPermissions) {
        String cacheKey = "permission:user:" + up.getUserId();
        String cachedVersion = redisTemplate.opsForValue().get(cacheKey + ":version");
        
        // 版本不一致，刷新缓存
        if (!String.valueOf(up.getVersion()).equals(cachedVersion)) {
            List<String> permissions = getUserPermissions(up.getUserId());
            redisTemplate.opsForValue().set(cacheKey, JSON.toJSONString(permissions));
            redisTemplate.opsForValue().set(cacheKey + ":version", String.valueOf(up.getVersion()));
        }
    }
}
```

## 🔥 扩展追问 1：MQ 挂了或消息丢失怎么办？

> **A:** 这是个很好的问题。我会设计多层兜底：

```java
public class PermissionUpdateHandler {
    
    // 1. 主流程：MQ 广播
    public void updatePermission(Long userId) {
        try {
            // 发 MQ 消息
            mqProducer.send(buildMessage(userId));
        } catch (Exception e) {
            log.warn("MQ发送失败，降级到定时任务", e);
            // 2. 降级：写入本地重试表
            saveToRetryTable(userId);
        }
    }
    
    // 3. 兜底：定时任务扫描未刷新缓存的记录
    @Scheduled(fixedDelay = 60000)
    public void retryFailedUpdates() {
        List<Long> failedUserIds = retryMapper.selectUnprocessedUsers();
        for (Long userId : failedUserIds) {
            try {
                redisTemplate.delete("permission:user:" + userId);
                retryMapper.markProcessed(userId);
            } catch (Exception e) {
                log.error("重试刷新权限缓存失败: userId={}", userId);
            }
        }
    }
    
    // 4. 版本号兜底：任何请求时先检查版本
    public List<String> getPermissions(Long userId) {
        String cached = redisTemplate.opsForValue().get("permission:user:" + userId);
        Long dbVersion = permissionMapper.selectVersionByUserId(userId);
        
        if (cached == null || isVersionExpired(userId, dbVersion)) {
            // 缓存过期/版本不一致，直接查库并回填缓存
            return refreshFromDB(userId);
        }
        return JSON.parseArray(cached);
    }
}
```

**三保险：**
- **第一层**：MQ 实时推送
- **第二层**：MQ 失败时写重试表 + 定时任务兜底
- **第三层**：版本号对比，请求时被动刷新

## 🔥 扩展追问 2：如何避免广播风暴？

> **A:** 权限变更往往是批量操作（比如修改角色权限影响 10000 个用户），直接广播 10000 条消息会打爆 MQ。

```java
public void updateRolePermission(Long roleId, List<String> newPermissions) {
    // ❌ 错误：逐个发消息
    // for (Long userId : affectedUsers) {
    //     mqProducer.send(buildMessage(userId));
    // }
    
    // ✅ 正确：合并为一条消息，携带用户列表
    List<Long> affectedUserIds = userRoleMapper.selectUsersByRoleId(roleId);
    
    // 合并策略：分批广播，每批最多 500 人
    List<List<Long>> batches = Lists.partition(affectedUserIds, 500);
    for (List<Long> batch : batches) {
        Message message = new Message("permission-topic", "batch-update",
            null, JSON.toJSONString(batch).getBytes());
        mqProducer.send(message);
    }
    
    log.info("权限变更广播，角色ID={}，影响用户数={}，分{}批",
        roleId, affectedUserIds.size(), batches.size());
}

// 消费端处理批量消息
@RocketMQMessageListener(topic = "permission-topic", selectorExpression = "batch-update")
public void onBatchMessage(String body) {
    List<Long> userIds = JSON.parseArray(body, Long.class);
    for (Long userId : userIds) {
        redisTemplate.delete("permission:user:" + userId);
    }
}
```

**或者更优：用版本号广播替代用户列表广播**

```java
// 发消息时只发角色版本号，消费端自己判断哪些用户受影响
Message message = new Message("permission-topic", "role-version",
    String.valueOf(roleId), 
    JSON.toJSONString(new RoleVersionUpdate(roleId, newVersion())).getBytes());

// 消费端：根据角色版本号计算需要清除的用户
public void onRoleVersionUpdate(RoleVersionUpdate update) {
    List<Long> affectedUserIds = userRoleMapper.selectUsersByRoleId(update.getRoleId());
    for (Long userId : affectedUserIds) {
        // 标记该用户需要刷新（不立即执行，延迟批量处理）
        redisTemplate.opsForSet().add("permission:refresh:pending", String.valueOf(userId));
    }
}
```

## 🔥 扩展追问 3：权限数据量很大，如何优化？

> **A:** 权限数据量大时，缓存策略需要分层：

```java
// 分层缓存架构
public class PermissionCacheManager {
    
    // L1: 本地缓存（Guava/Caffeine），TTL 5分钟，热点数据
    private Cache<Long, List<String>> localCache = Caffeine.newBuilder()
        .maximumSize(10000)
        .expireAfterWrite(5, TimeUnit.MINUTES)
        .build();
    
    // L2: Redis 分布式缓存，TTL 30分钟
    // L3: 数据库
    
    public List<String> getPermissions(Long userId) {
        // 先查本地缓存
        List<String> result = localCache.getIfPresent(userId);
        if (result != null) {
            return result;
        }
        
        // 再查 Redis
        String cached = redisTemplate.opsForValue().get("permission:user:" + userId);
        if (cached != null) {
            result = JSON.parseArray(cached);
            localCache.put(userId, result);  // 回填 L1
            return result;
        }
        
        // 最后查 DB
        result = loadFromDB(userId);
        redisTemplate.opsForValue().set(
            "permission:user:" + userId, 
            JSON.toJSONString(result),
            30, TimeUnit.MINUTES);
        localCache.put(userId, result);
        
        return result;
    }
    
    // 通知刷新时，需要同时清除 L1 和 L2
    public void invalidate(Long userId) {
        localCache.invalidate(userId);  // 清除 L1
        redisTemplate.delete("permission:user:" + userId);  // 清除 L2
    }
    
    // 批量刷新用 Redis Pipeline
    public void batchInvalidate(List<Long> userIds) {
        localCache.invalidateAll(userIds);  // 批量清除 L1
        // 批量清除 L2
        List<String> keys = userIds.stream()
            .map(id -> "permission:user:" + id)
            .collect(Collectors.toList());
        redisTemplate.delete(keys);
    }
}
```

**关键配置：**

```yaml
# application.yml
spring:
  cache:
    type: caffeine
  caffeine:
    spec: maximumSize=10000,expireAfterWrite=300s

permission:
  cache:
    local-ttl-minutes: 5
    redis-ttl-minutes: 30
    batch-size: 500
```

## 🔥 扩展追问 4：通知失败导致缓存长期不一致怎么办？

> **A:** 加一个缓存版本号机制，结合请求时被动刷新：

```sql
-- 权限表加版本号
ALTER TABLE sys_user_permission ADD COLUMN version BIGINT DEFAULT 0;
ALTER TABLE sys_user_permission ADD COLUMN update_time DATETIME DEFAULT CURRENT_TIMESTAMP;

-- 每次更新权限，版本号 +1
UPDATE sys_user_permission 
SET permissions = #{permissions}, 
    version = version + 1,
    update_time = NOW()
WHERE user_id = #{userId};
```

```java
// 请求时检查版本，过期就刷新
public List<String> getPermissions(Long userId) {
    String cached = redisTemplate.opsForValue().get("permission:user:" + userId);
    Long dbVersion = permissionMapper.selectVersion(userId);
    String cachedVersion = redisTemplate.opsForValue().get("permission:user:version:" + userId);
    
    // 缓存不存在 或 版本不一致 → 刷新
    if (cached == null || !String.valueOf(dbVersion).equals(cachedVersion)) {
        log.info("权限缓存过期，userId={}，dbVersion={}，cachedVersion={}", 
            userId, dbVersion, cachedVersion);
        List<String> fresh = loadFromDB(userId);
        redisTemplate.opsForValue().set("permission:user:" + userId, JSON.toJSONString(fresh));
        redisTemplate.opsForValue().set("permission:user:version:" + userId, String.valueOf(dbVersion));
        return fresh;
    }
    
    return JSON.parseArray(cached);
}
```

**版本号的妙用：**
- 发通知时只需要发 `userId + newVersion`，消费端按版本号判断是否需要刷新
- 即使漏发通知，下一次请求时也会被动刷新
- 版本号天然支持 TTL，去掉无效版本号占用

## 🔥 扩展追问 5：Canal 监听数据库变更可以吗？

> **A:** 可以，这是 CDC（Change Data Capture）方案，**适合不想改业务代码的场景**：

```
┌─────────┐    ┌─────────┐    ┌─────────────────┐    ┌─────────────┐
│  MySQL  │───▶│  Canal  │───▶│   Canal Server  │───▶│  各节点消费  │
│  Binlog │    │  Client │    │   (伪装成从库)   │    │  清除缓存    │
└─────────┘    └─────────┘    └─────────────────┘    └─────────────┘
```

```java
// Canal 消费端（伪代码）
public class CanalPermissionListener {
    
    public void onChange(RowChange rowChange) {
        for (Entry entry : rowChange.getRowEntries()) {
            if ("sys_user_permission".equals(entry.getTableName())) {
                RowData rowData = entry.getAfterColumnsMap();
                Long userId = Long.valueOf(rowData.get("user_id").getValue());
                
                // 权限表有变更，清除缓存
                redisTemplate.delete("permission:user:" + userId);
                log.info("Canal 监听到权限变更，清除缓存: userId={}", userId);
            }
        }
    }
}
```

**优缺点：**
- ✅ **优点**：对业务代码零侵入；实时性高
- ❌ **缺点**：引入额外组件；只支持 MySQL；Canal 本身高可用需额外配置

---

> **总结：多节点权限缓存一致性 → 推送+拉取双保险**
>
> **核心思路：实时推送（MQ/Redis Pub/Sub）做主动通知 + 版本号机制做被动兜底 + 分层缓存优化性能。** 实际生产中推荐 MQ 广播 + 定时任务兜底 + 请求时版本校验三合一。

---

> "我们项目里用的是 MQ 广播 + 定时兜底方案。之前踩过一个坑：运维有一次重启 MQ 导致广播消息积压，权限缓存没及时刷新。后来加了请求时版本号校验，即使 MQ 完全挂了，下一次用户操作时也会强制从数据库拉取最新权限，用户无感知。这个版本号的 trick 后来被我们推广到了所有缓存数据同步的场景。"
