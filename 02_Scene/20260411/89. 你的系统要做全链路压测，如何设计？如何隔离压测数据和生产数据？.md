### 1.3 核心代码实现

#### 1.3.1 流量染色Filter

```java
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class PerfTestTrafficDyeFilter implements GlobalFilter {

    private static final String PERF_TEST_HEADER = "X-Perf-Test";
    private static final String PERF_TEST_KEY = "X-Perf-Key";
    private static final String PERF_TEST_TRUE = "true";

    @Autowired
    private PerfTestContextHolder contextHolder;

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        HttpHeaders headers = exchange.getRequest().getHeaders();
        
        // 提取压测标记
        String perfTest = headers.getFirst(PERF_TEST_HEADER);
        String perfKey = headers.getFirst(PERF_TEST_KEY);
        
        if (PERF_TEST_TRUE.equalsIgnoreCase(perfTest)) {
            // 设置上下文
            PerfTestContext context = new PerfTestContext();
            context.setPerfTest(true);
            context.setPerfKey(perfKey);
            context.setTraceId(generatePerfTraceId(perfKey));
            contextHolder.setContext(context);
            
            // 追加压测标记到响应头，透传给下游
            ServerHttpResponse response = exchange.getResponse();
            response.getHeaders().add(PERF_TEST_HEADER, PERF_TEST_TRUE);
            response.getHeaders().add(PERF_TEST_KEY, perfKey);
            response.getHeaders().add("X-Trace-Id", context.getTraceId());
            
            // 继续传递（不再等待响应）
            return chain.filter(exchange);
        }
        
        return chain.filter(exchange);
    }
    
    private String generatePerfTraceId(String perfKey) {
        return String.format("PERF-%s-%s", perfKey, UUID.randomUUID().toString().substring(0, 8));
    }
}
```

#### 1.3.2 压测上下文Holder（线程传递）

```java
public class PerfTestContextHolder {
    
    private static final ThreadLocal<PerfTestContext> CONTEXT_HOLDER = new ThreadLocal<>();
    
    public static void setContext(PerfTestContext context) {
        CONTEXT_HOLDER.set(context);
    }
    
    public static PerfTestContext getContext() {
        return CONTEXT_HOLDER.get();
    }
    
    public static boolean isPerfTest() {
        PerfTestContext context = CONTEXT_HOLDER.get();
        return context != null && Boolean.TRUE.equals(context.isPerfTest());
    }
    
    public static String getPerfKey() {
        PerfTestContext context = CONTEXT_HOLDER.get();
        return context != null ? context.getPerfKey() : null;
    }
    
    public static void clear() {
        CONTEXT_HOLDER.remove();
    }
}

// RPC透传拦截器（以Dubbo为例）
@Activate(group = Constants.CONSUMER)
public class PerfTestDubboConsumerFilter implements Filter {
    
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcResult {
        if (PerfTestContextHolder.isPerfTest()) {
            // 透传压测标记到RPC上下文
            RpcContext.getContext()
                .setAttachment("perf_test", "true")
                .setAttachment("perf_key", PerfTestContextHolder.getPerfKey())
                .setAttachment("trace_id", PerfTestContextHolder.getContext().getTraceId());
        }
        return invoker.invoke(invocation);
    }
}

// 服务端接收
@Activate(group = Constants.PROVIDER)
public class PerfTestDubboProviderFilter implements Filter {
    
    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) {
        String perfTest = invocation.getAttachment("perf_test");
        if ("true".equals(perfTest)) {
            PerfTestContext context = new PerfTestContext();
            context.setPerfTest(true);
            context.setPerfKey(invocation.getAttachment("perf_key"));
            context.setTraceId(invocation.getAttachment("trace_id"));
            PerfTestContextHolder.setContext(context);
        }
        try {
            return invoker.invoke(invocation);
        } finally {
            PerfTestContextHolder.clear();
        }
    }
}
```

```java
// 数据源路由逻辑
public class DataSourceRouteDataSource extends AbstractDataSource {
    
    private DataSource primaryDataSource;
    private DataSource shadowDataSource;

    @Override
    public Connection getConnection() throws SQLException {
        if (PerfTestContextHolder.isPerfTest()) {
            // 压测流量路由到影子库
            ShardingConnection shadowConn = shadowDataSource.getConnection();
            return wrapConnection(shadowConn, true);
        }
        return primaryDataSource.getConnection();
    }
    
    // 影子库连接增强：自动添加表前缀/后缀
    private Connection wrapConnection(Connection conn, boolean isShadow) {
        // 返回装饰后的连接，压测SQL自动路由到 shadow.xxx 表
        return new ShadowConnectionWrapper(conn, isShadow);
    }
}
```

```java
@Component
public class PerfTestRedisTemplate extends RedisTemplate<String, Object> {

    private static final String SHADOW_PREFIX = "perf_test:";

    @Override
    public ValueOperations<String, Object> opsForValue() {
        if (PerfTestContextHolder.isPerfTest()) {
            return new ShadowValueOperations(super.opsForValue());
        }
        return super.opsForValue();
    }
    
    @Override
    public <T> T get(Object key) {
        if (PerfTestContextHolder.isPerfTest()) {
            return super.get(SHADOW_PREFIX + key);
        }
        return super.get(key);
    }
    
    @Override
    public void put(Object key, Object value) {
        if (PerfTestContextHolder.isPerfTest()) {
            super.put(SHADOW_PREFIX + key, value);
        } else {
            super.put(key, value);
        }
    }
}
```

```java
@Service
@Slf4j
public class PerfTestDataService {

    @Autowired
    private JdbcTemplate primaryJdbcTemplate;
    
    @Autowired
    private JdbcTemplate shadowJdbcTemplate;

    /**
     * 全量数据同步（影子库）
     */
    @Transactional
    public void syncFullDataToShadow(String tableName, Long batchSize) {
        String countSql = String.format("SELECT COUNT(*) FROM %s", tableName);
        Long total = primaryJdbcTemplate.queryForObject(countSql, Long.class);
        
        Long offset = 0L;
        while (offset < total) {
            String querySql = String.format(
                "SELECT * FROM %s LIMIT %d OFFSET %d", 
                tableName, batchSize, offset
            );
            List<Map<String, Object>> rows = primaryJdbcTemplate.queryForList(querySql);
            
            // 批量插入影子库
            batchInsertToShadow(tableName, rows);
            offset += batchSize;
            log.info("已同步 {}/{} 条数据到影子库", offset, total);
        }
    }

    /**
     * 构造压测数据（根据业务模型生成）
     */
    public List<User> generatePerfTestUsers(int count) {
        List<User> users = new ArrayList<>();
        for (int i = 0; i < count; i++) {
            User user = new User();
            user.setId(1000000000L + i);  // 使用大区间ID，避免与真实用户冲突
            user.setUsername("perf_test_user_" + i);
            user.setPhone("13800000000");  // 特殊前缀标识
            user.setEmail("perf_test_" + i + "@perf-test.com");
            user.setStatus(1);
            user.setCreatedAt(new Date());
            users.add(user);
        }
        return users;
    }

    /**
     * 清理压测数据
     */
    @Transactional
    public void cleanupPerfTestData(String tableName, Long minId) {
        String deleteSql = String.format(
            "DELETE FROM %s WHERE id >= %d", tableName, minId
        );
        shadowJdbcTemplate.execute(deleteSql);
        log.info("已清理 {} 表中id>= {} 的压测数据", tableName, minId);
    }
}
```

### 2.1 🔥 扩展追问 1：压测数据如何构造？有哪些策略？

> **A:**

**压测数据构造策略对比：**

| 策略 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **真实数据脱敏** | 数据量小、对准确性要求高 | 真实度高 | 脱敏成本高、合规风险 |
| **影子表全量同步** | 需要完整数据验证 | 覆盖全、功能验证完整 | 同步耗时长、存储成本高 |
| **构造数据生成** | 大规模数据、基准压测 | 生成快、数据量可控 | 真实性差、可能触发边界bug |
| **流量回放** | 复现生产真实场景 | 场景真实 | 需要流量录制基础设施 |
| **增量构造+缓存预热** | 大促前压测 | 兼顾速度和数据真实性 | 实现复杂 |

**推荐方案：影子库 + 增量数据生成**

```java
@Service
@Slf4j
public class PerfTestDataGenerator {

    /**
     * 分层数据构造策略
     */
    public void constructPerfTestData(PerfTestScene scene) {
        // 1. 基础数据同步（用户、商品等核心表）

            String cacheKey = "product:" + productId;
            redisTemplate.opsForValue().set(cacheKey, productService.getById(productId));
        });
    }
}
```

### 2.3 🔥 扩展追问 3：如何评估压测结果？有哪些关键指标？

> **A:**

**核心指标体系：**

| 指标类别 | 指标名称 | 优秀 | 良好 | 合格 | 压测重点 |
|---------|---------|------|------|------|---------|
| **性能指标** | 响应时间 P50 | <50ms | <100ms | <200ms | 是否满足SLA |
| | 响应时间 P99 | <200ms | <500ms | <1s | 长尾请求 |
| | QPS | 达标 | 接近目标 | - | 系统容量 |
| | TPS | 达标 | 接近目标 | - | 业务处理能力 |
| **资源指标** | CPU使用率 | <70% | <85% | <95% | 资源饱和度 |
| | 内存使用率 | <70% | <80% | <85% | OOM风险 |
| | JVM堆使用率 | <70% | <80% | <85% | GC压力 |
| | DB连接池 | <60% | <80% | <90% | 连接耗尽 |
| | Redis连接池 | <60% | <80% | <90% | 连接耗尽 |
| **稳定性指标** | 错误率 | <0.01% | <0.1% | <1% | 系统可靠性 |
| | 成功率 | >99.9% | >99% | >99% | 业务正确性 |

**压测报告自动生成：**

```java
@Service
@Slf4j
public class PerfTestReportGenerator {

    public PerfTestReport generateReport(Long taskId, Path resultFile) {
        PerfTestReport report = new PerfTestReport();
        
        // 1. 解析JMeter结果文件
        JtlParser parser = new JtlParser();
        JtlResult jtl = parser.parse(resultFile);
        
        // 2. 计算聚合指标
        report.setRtP50(calculatePercentile(jtl.getResponseTimes(), 0.5));
        report.setRtP90(calculatePercentile(jtl.getResponseTimes(), 0.9));
        report.setRtP99(calculatePercentile(jtl.getResponseTimes(), 0.99));
        report.setQps(calculateQps(jtl));
        report.setErrorRate(calculateErrorRate(jtl));
        
        // 3. 采集系统资源指标
        Map<String, Object> metrics = prometheusClient.queryRange(
            "node_cpu_seconds_total",
            report.getStartTime(),
            report.getEndTime()
        );
        report.setCpuAvg(calculateAvg(metrics));
        
        // 4. 生成瓶颈分析
        List<Bottleneck> bottlenecks = analyzeBottlenecks(report);
        report.setBottlenecks(bottlenecks);
        
        // 5. 输出优化建议
        List<Suggestion> suggestions = generateSuggestions(bottlenecks);
        report.setSuggestions(suggestions);
        
        return report;
    }
    
    private List<Bottleneck> analyzeBottlenecks(PerfTestReport report) {
        List<Bottleneck> bottlenecks = new ArrayList<>();
        
        // CPU瓶颈检测
        if (report.getCpuAvg() > 85) {
            bottlenecks.add(Bottleneck.builder()
                .type("CPU")
                .level("HIGH")
                .description("CPU使用率过高: " + report.getCpuAvg() + "%")
                .suggestion("考虑扩容或优化计算密集型代码")
                .build());
        }
        
        // GC瓶颈检测
        if (report.getGcPauseMs() > 500) {
            bottlenecks.add(Bottleneck.builder()
                .type("GC")
                .level("MEDIUM")
                .description("GC停顿时间过长")
                .suggestion("检查内存泄漏或调整GC参数")
                .build());
        }
        
        return bottlenecks;
    }
}
```

### 2.4 🔥 扩展追问 4：全链路压测与接口级压测的区别？

> **A:**

**对比表格：**

| 维度 | 接口级压测 | 全链路压测 |
|------|-----------|-----------|
| **压测范围** | 单个服务/接口 | 完整调用链路（前端→网关→服务→DB→缓存→MQ） |
| **数据真实性** | 模拟数据 | 接近生产真实数据 |
| **问题发现** | 单点性能问题 | 端到端性能问题 + 架构瓶颈 |
| **资源消耗** | 低 | 高（需要完整压测环境） |
| **实现复杂度** | 低（JMeter直接压接口） | 高（需要流量染色、数据隔离等） |
| **适用场景** | 日常开发测试 | 大促前核心链路上线前 |
| **执行频率** | 每次发版前 | 季度/大促前 |
| **压测工具** | JMeter、Postman | JMeter + PTS + 自研平台 |

**全链路压测补充场景：**

```
接口压测能发现的：
├── 单接口响应时间
├── 单接口QPS上限
├── 单接口内存/CPU消耗
└── 单接口数据库慢查询

全链路压测额外发现的：
├── 网关层瓶颈（限流、降级）
├── 服务调用链路超时
├── 分布式锁竞争
├── 缓存热点问题
├── MQ积压问题
├── 上下游服务能力不匹配
└── 架构设计不合理（同步调用链路过长）
```

### 2.6 🔥 扩展追问 6：压测环境与生产环境的差异如何处理？

> **A:**

**差异处理策略：**

| 差异维度 | 常见差异 | 处理方案 |
|---------|---------|---------|
| **机器配置** | 压测环境CPU/内存比生产低 | 按比例折算性能指标，或使用等比缩容公式 |
| **数据量** | 压测数据量远小于生产 | 影子库同步+数据量等比测试 |
| **流量特征** | 压测流量分布与生产不同 | 流量模型复现（根据生产日志分析） |
| **网络拓扑** | 网络延迟/带宽不同 | 模拟真实网络拓扑 |
| **依赖服务** | 下游服务能力不同 | Mock或压测下游服务 |
| **缓存预热** | 冷启动 vs 热缓存 | 提前预热后再压测 |

**性能指标折算公式：**

```
生产预估QPS = 压测QPS × (生产机器数 / 压测机器数) × (生产CPU配置 / 压测CPU配置)

示例：
- 压测环境: 4台 8核16G → QPS=10000
- 生产环境: 16台 16核32G
- 预估生产QPS = 10000 × (16/4) × (16/8) = 10000 × 4 × 2 = 80000
```

## 3. 一句话总结

> **全链路压测通过流量染色+六层数据隔离（网关→服务→缓存→数据库→MQ→外部依赖），在压测环境中完整复现生产流量，验证系统真实容量上限，同时通过影子库、影子缓存、影子MQ等机制确保压测数据与生产数据完全隔离。**

## 4. 面试加分话术

> "我们公司在双十一大促前两周，会进行全链路压测。我负责的是压测数据隔离方案的设计。
> 
> 核心思路是'流量染色 + 链路透传 + 数据分层隔离'：流量从JMeter发出时打上 `X-Perf-Test: true` 的标记，网关层通过 Filter 识别后写入 ThreadLocal，后续整个链路（RPC、DB、Redis、MQ）都能感知到这个标记，自动路由到影子库/影子缓存/影子MQ。
> 
> 数据层面我们做了三件事：
> 1. **影子库**：用 ShardingSphere 的数据源路由，所有压测SQL自动路由到 `shadow_xxx` 表
> 2. **影子缓存**：Redis Key 加上 `perf_test:` 前缀，与生产数据完全隔离
> 3. **影子MQ**：Kafka Topic 路由到 `perf_test_xxx`，消费组也是独立的
> 
> 压测数据构造用的是'影子库全量同步 + 增量数据生成'的混合策略——核心表（用户、商品）从生产库同步并脱敏，业务表（订单）根据业务模型构造。
> 
> 大促压测中我们发现了一个隐藏的瓶颈：Redis集群在 8 万 QPS 时出现了热点Key问题，定位到是某个活动缓存没有加前缀导致数据集中在一个Slot上，通过调整Key的Hash Tag解决了这个问题。"
