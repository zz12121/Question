# 业务场景题 - 消息队列（B235-B264）

> 以下为消息队列领域的业务场景题，编号从235起，覆盖消息队列在实际项目中的业务场景、问题排查、架构选型。

## 消息积压与消费处理（6题）

235. 双十一大促期间，你的订单系统突然出现了严重的 Kafka 消息积压，Consumer Lag 从几百飙到了 500 万。你作为值班开发，需要快速定位和解决。首先怎么排查积压原因？（消费者处理慢 / 消费者异常退出 / Rebalance 频繁触发 / 突发流量洪峰）如果确认是流量洪峰导致的，消费者已经满负荷运转，最快的恢复方案是什么？（临时扩容消费者 + 如果分区数不够则先增加分区再扩容）扩容后的消费者会从哪里开始消费？（从最新提交的 offset 开始，已消费的不会重复）积压的旧消息怎么处理？

236. 你的系统用 Kafka 做日志采集，Topic 有 50 个 Partition，平时消费很正常。某天发现消费者频繁 Rebalance，每次 Rebalance 持续 30 秒以上，导致消费时断时续。可能的原因有哪些？（心跳超时 / 消费者 GC 停顿 / 网络波动 / 消费者处理时间超过 `max.poll.interval.ms`）如何排查 Rebalance 的频率和原因？（Kafka Consumer 的 `jmx` 指标 `rebalance-rate-per-hour` / 日志中的 `rebalance` 关键字）怎么减少不必要的 Rebalance？（调大 `session.timeout.ms` 和 `max.poll.interval.ms` / 减少每次 poll 的消息数量 / 升级到 CooperativeStickyAssignor）

237. 你的 Kafka 消费者处理一条消息平均需要 200ms，但某个 Topic 的 Partition 只有 6 个，导致消费速度远跟不上生产速度。在不增加 Partition 数的情况下，如何提高消费吞吐量？（批量消费 / 多线程异步处理 / 消费者内部线程池并行处理同批次消息）如果使用多线程处理，如何保证同一 Key 的消息顺序？（按 Key hash 到固定线程 / 内存队列分组串行）批量消费的 `max.poll.records` 参数怎么设置比较合理？批量处理中如果某条消息处理失败，整批都要回滚吗？

238. 你的 RocketMQ 消费者在运行过程中突然"不再消费新消息"了，但服务本身没有报错，进程也活着。可能的原因有哪些？（消费被 Block（Thread.sleep 太长）/ 消费线程死锁 / 消费者达到了 Prefetch 上限但 ACK 未发送 / 频繁重试导致线程池耗尽）RocketMQ 的消费者处理耗时如果超过 15 分钟会怎样？（消费锁过期，Broker 认为消费者已掉线，重新投递）如何设置合理的消费超时时间？消费端的线程池配置（`consumeThreadMin` / `consumeThreadMax`）怎么调优？

239. 你的系统需要消费 RabbitMQ 中的消息并写入数据库，但由于数据库偶尔变慢（比如锁等待、连接池耗尽），导致消费者处理速度下降，消息在 RabbitMQ 中持续堆积。RabbitMQ 的内存使用率到了告警阈值 80%，甚至触发了内存告警。RabbitMQ 的内存告警后默认行为是什么？（`vm_memory_high_watermark` 到达后，阻塞所有连接的发布）如何缓解？（增加消费者实例 / 设置 `prefetch_count` 限流 / 开启惰性队列 Lazy Queue / 将消息转储到磁盘）Lazy Queue 和 Classic Queue 的区别是什么？

240. 你的系统有一个"批量数据同步"场景：从 MySQL 读取变更数据，发送到 Kafka，下游消费者根据消息更新 Redis 缓存。某天发现下游 Redis 缓存和 MySQL 数据不一致了，但 Kafka 的消息并没有丢失（Consumer Lag 正常）。可能的原因是什么？（消息消费顺序不对：先消费了"旧版本"的更新消息，再消费了"新版本"的更新消息）如果多条消息更新同一行数据，如何保证最终一致性？（按主键做版本号或时间戳比较 / 使用 Redis 的乐观锁 CAS / 下游消费者按时间戳排序后处理）

## 消息可靠性保障（5题）

241. 你的电商系统用 RocketMQ 做订单状态变更通知，但偶尔会出现"消息丢失"的情况：订单状态已经从"待支付"变为"已支付"，但下游的积分服务没有收到消息。可能丢失的节点在哪里？（Producer 发送失败未重试 / Broker 存储异常 / Consumer 消费成功但 ACK 失败）如何从架构层面避免消息丢失？（Producer: 同步发送 + 重试 + 事务消息 / Broker: 同步刷盘 + 多副本 / Consumer: 手动 ACK + 幂等消费）RocketMQ 的"同步刷盘"（`flushDiskType=SYNC_FLUSH`）和"异步刷盘"（`ASYNC_FLUSH`）有什么区别？性能差异有多大？

242. 你的支付系统用 Kafka 做交易流水异步处理，要求"消息绝对不能丢"。但 Kafka 的默认配置是异步刷盘，Broker 宕机可能丢失部分消息。如何配置 Kafka 做到"不丢消息"？（`acks=all` + `min.insync.replicas=2` + Broker 端 `unclean.leader.election.enable=false` + `min.insync.replicas` 配合 `replication.factor=3`）这些配置对性能的影响有多大？（吞吐量可能下降 30%-50%）在"性能"和"可靠性"之间如何取舍？实际项目中你是怎么配置的？

243. 你的系统用 RabbitMQ 做任务分发，但最近频繁出现"消费者重启后部分消息丢失"的现象。排查发现消费者的 `auto-ack=true`，消费者拿到消息后还没处理完就宕机了。如何改为手动确认？（`channel.basicAck()` / `channel.basicNack()`）手动确认模式下，如果消费者处理完消息后忘记调 `basicAck()` 会怎样？（消息一直处于 Unacked 状态，消费者重启后重新投递）RabbitMQ 的 `message_ttl` 和 `queue_ttl` 分别是什么？如果队列设置了 TTL，消息会怎样？

244. 你的系统在做"分布式事务"改造：订单服务创建订单后，需要同时通知库存服务扣减库存、通知积分服务增加积分、通知物流服务创建运单。使用消息队列做最终一致性，但如何保证"消息要么全部成功投递，要么全部回滚"？（RocketMQ 事务消息 / 本地消息表 + 定时任务扫描 / Saga 模式编排）如果用 RocketMQ 事务消息，回查逻辑执行时间过长（比如需要调用第三方接口查询），会影响事务消息的正常流程吗？（Broker 等待回查结果期间，半消息对消费者不可见）回查超时后消息会怎样？

245. 你的系统用 Kafka 做跨服务数据同步，但发现偶尔会出现"重复消息"导致下游数据重复处理（比如重复扣减库存）。重复消息的来源可能是哪里？（Producer 发送成功但没收到 ACK 于是重发 / Consumer 处理成功但提交 offset 前宕机 / Rebalance 导致新消费者重新消费未提交 offset 的消息）消费端如何实现幂等？（数据库唯一约束 / Redis SETNX 去重 / 业务状态机判断 / 全局消息ID去重表）幂等表的设计要点是什么？（消息ID做唯一索引 + 处理状态字段 + 定期清理已处理记录）

## 消息顺序性与一致性（4题）

246. 你的电商系统用 Kafka 做订单状态流转通知：订单创建 → 支付成功 → 发货 → 签收。这些消息必须按顺序消费，否则会出现"先收到发货通知再收到支付通知"的异常。如何保证同一订单的消息顺序？（订单ID作为 Key 路由到同一 Partition）但如果同一个 Topic 里还有其他业务的消息（比如商品更新、用户操作日志），不同业务的消息混在同一个 Partition 里会影响顺序性吗？（不会，因为 Key 不同路由到不同 Partition）如果订单消息量特别大，单 Partition 成为瓶颈怎么办？（按订单ID分片到多个 Partition，但同一订单ID的路由规则固定）

247. 你的系统用 RocketMQ 做订单流水处理，要求"同一笔交易的创建、支付、退款消息必须顺序消费"。RocketMQ 的顺序消息怎么使用？（`MessageQueueSelector` 按 Key 选择 Queue + `ConsumeMessageOrderlyService` 顺序消费）如果顺序消费模式下，某条消息处理失败了（抛出异常），RocketMQ 会怎么处理？（当前消息会被反复重试，后续消息全部阻塞等待）这种"一条失败全部阻塞"的问题怎么解决？（设置重试次数上限 / 失败后转入死信队列 / 降级为异步重试而不是同步阻塞）

248. 你的系统需要从 MySQL 实时同步数据到 Elasticsearch，使用 Canal 监听 MySQL binlog，将变更事件发送到 Kafka，消费者消费 Kafka 消息后写入 ES。但发现 ES 中的数据和 MySQL 不一致：某条记录在 MySQL 中是"已删除"状态，但 ES 中仍然存在。可能的原因有哪些？（binlog 事件的消费顺序不对：先消费了 INSERT 再消费了 DELETE / Canal 发送了但 Kafka 丢失 / 消费者写入 ES 失败但 offset 已提交）如何保证 binlog 事件的严格有序？（Canal 按 table 分发到不同 Partition / 消费者按 binlog position 排序）Canal 的 HA 模式（ZooKeeper 协调）怎么保证不丢不重？

249. 你的系统用消息队列做"缓存更新"：数据库变更后发送消息，消费者消费消息后更新 Redis 缓存。但出现了"数据库已经是新值，Redis 缓存还是旧值"的情况。可能的原因？（消息消费延迟 / 消费失败但消息已 ACK / 重复消息覆盖了新值）如何设计一个更可靠的"缓存更新"方案？（消息携带数据版本号 / 消费者比较版本号只接受更新的数据 / Redis 的 WATCH + MULTI 乐观更新 / 使用 Canal 直接监听 binlog 而不是应用层发消息）

## 架构设计与选型（6题）

250. 你要从零搭建一个"消息通知中心"，需要支持以下功能：站内信、短信、邮件、App 推送四种通知渠道，每种渠道的发送方（短信服务商、邮件服务商等）不同，且各渠道的发送速率有限制（比如短信 100条/秒、邮件 50条/秒）。如何用消息队列来解耦？（发送请求 → 入统一 Topic → 按渠道路由到不同 Queue → 各渠道独立消费者限速消费）如何实现"发送失败自动重试"+"重试次数达到上限后转入人工处理"？（RocketMQ 的重试机制 + 死信队列 + 死信消费者转工单系统）如何监控各渠道的发送成功率？

251. 你的公司现有系统使用 RabbitMQ，但新的业务需求需要"事务消息"和"延迟消息"功能，RabbitMQ 的原生支持不够好。你面临一个选择：继续用 RabbitMQ 做改造（死信队列实现延迟 / 本地消息表实现事务）还是迁移到 RocketMQ？从改造成本、学习成本、运维成本、功能完整度四个维度分析。如果选择迁移，数据迁移方案怎么做？（双写过渡 / 灰度切换 / 监控对比）如果选择改造 RabbitMQ，延迟消息的精度能达到多少？（受限于 TTL 检查间隔，通常有秒级误差）

252. 你的系统需要做"日志采集和分析"，每天产生约 50GB 的日志数据，要求实时写入、支持批量消费和流处理（Flink）。选 Kafka 还是 RabbitMQ？为什么？如果后续还需要对接 Kafka Connect + Schema Registry 做数据管道，答案会变吗？如果业务要求"每条日志必须可靠到达不丢失"，Kafka 的配置建议是什么？（`acks=all` + `min.insync.replicas=2` + `compression.type=producer`）日志采集的 Topic Partition 数量怎么规划？（按日志量估算，单个 Partition 约 10MB/s 写入上限）

253. 你的系统需要实现一个"分布式延迟任务"调度平台，支持"订单创建后 30 分钟未支付自动取消"、"优惠券到期前 1 小时提醒"等需求。延迟精度要求秒级。用 RocketMQ 的延迟消息（18 个固定延迟级别）、RabbitMQ 的死信队列+TTL、Redis ZSet 轮询、时间轮（Timing Wheel）四种方案分别有什么优缺点？如果延迟时间不固定（用户自定义），哪种方案更合适？高并发场景下（每秒 10 万个延迟任务创建），哪种方案的性能最好？

254. 你的微服务系统有 20 个服务，目前服务间调用全部是同步 HTTP/RPC，调用链路平均 3-4 层深。现在需要将部分非核心链路改为"异步消息驱动"，减少调用延迟和提高系统可用性。哪些链路适合改为消息驱动？（通知类链路：短信/邮件/push / 数据同步类：缓存更新/搜索索引更新 / 日志审计类 / 异步任务类：报表生成/数据导出）哪些链路不适合？（需要实时返回结果的查询链路 / 强一致性事务链路 / 需要同步等待结果的业务链路）如何设计消息的"关联查询"——即异步处理完成后如何通知发起方结果？（回调 + Correlation ID / 结果查询接口 / WebSocket 推送）

255. 你的系统需要设计一个"消息追踪"系统，用于排查"消息发送后为什么没被消费"的问题。消息从 Producer 发出到 Consumer 消费完成，中间经过了哪些环节？每个环节需要记录什么信息？（Producer: 消息ID、发送时间、Topic、Partition / Broker: 存储时间、投递时间 / Consumer: 拉取时间、处理时间、ACK时间）如何实现消息的全链路追踪？（消息携带 TraceID / Broker 端的日志 / Apache Kafka 的 MirrorMaker / RocketMQ 的 msgTrace）如果 Kafka 自带的消息追踪不够用，如何自定义实现？

## 问题排查与性能调优（5题）

256. 你的 Kafka 集群突然出现"某个 Broker 磁盘使用率 95% 以上"的告警，但其他 Broker 磁盘正常。可能的原因是什么？（某个 Partition 的数据倾斜——这个 Broker 上的 Partition 消息量远大于其他 Broker / 日志清理策略 Log Retention 配置问题 / 某个消费者长期停止消费导致 Segment 文件堆积）如何确认是哪个 Topic/Partition 导致的？（`kafka-log-dirs.sh --describe` 查看各 Broker 的 Partition 大小分布）如何解决数据倾斜？（重新分配 Partition `kafka-reassign-partitions.sh` / 调整 Partition 数量 / 检查生产者的 Key 分布是否均匀）

257. 你的 Kafka Producer 发送消息偶尔出现"消息发送超时"（`TimeoutException: Failed to update metadata after xxx ms`）。可能的原因有哪些？（Broker 负载过高响应慢 / 网络抖动 / DNS 解析问题 / Controller 所在 Broker 压力大）如何排查？（查看 Broker 的 JMX 指标 `request-latency-avg` / 查看 Producer 的 `metadata-age` / 查看 Controller 的切换日志）Producer 的 `metadata.max.age.ms` 和 `request.timeout.ms` 参数分别控制什么？如何调整？

258. 你的 RabbitMQ 集群在高峰期出现了"连接数暴增"的问题，从正常的 5000 个连接突然涨到了 50000 个。可能的原因是什么？（消费者不断重连但没有正确释放旧连接 / 连接泄露（Connection Leak）/ 客户端连接池配置不当 / 大量短连接没有复用）如何排查？（`rabbitmqctl list_connections` 查看连接列表 / 按状态分组统计 / 查看连接的 `recv_oct` 和 `send_oct` 判断是否活跃）如何解决？（修复连接泄露 / 启用连接池 / 设置 `connection.max-age` 自动回收空闲连接）

259. 你的 RocketMQ Broker 偶尔出现"Full GC 停顿 3 秒以上"的问题，导致发送消息的 RT 偶尔飙到 5 秒。RocketMQ 的 Broker 是基于 Java 的，JVM 调优怎么做的？（堆内存不要太大、避免大对象 / 使用 G1 收集器 / 调整 `CommitLog` 和 `ConsumeQueue` 的刷盘策略减少内存压力）RocketMQ 的 `transientStorePoolEnable`（堆外内存缓冲）参数的作用是什么？（消息先写入堆外内存再异步刷盘，减少堆内存压力和 GC 频率）Broker 的 JVM 参数一般怎么配置？

260. 你的 Kafka 集群有 6 个 Broker，12 个 Partition，3 个副本。某天运维误操作关闭了 3 个 Broker，导致部分 Partition 只剩 1 个副本（ISR 缩小到了 1）。虽然后续恢复了 Broker，但发现"部分 Partition 的数据有丢失"。为什么？（ISR 中只剩 Leader，写入后 Leader 宕机，新 Leader 从非 ISR 副本中选出（如果 `unclean.leader.election.enable=true`），数据截断）如何预防这种情况？（`min.insync.replicas=2` + `unclean.leader.election.enable=false` + `acks=all`）如何检查是否有数据丢失？（对比各副本的 LEO / 使用 `kafka-run-class.sh tools.LogDumpTool` 分析日志）

## 延迟消息与特殊场景（4题）

261. 你的电商系统需要实现"订单超时自动取消"功能，要求延迟时间可变（不同商品的超时时间不同：有的 15 分钟、有的 30 分钟、有的 1 小时）。如果用 RocketMQ 的 18 个固定延迟级别，精度不够。如何实现"任意延迟时间"？（使用 RocketMQ 开源版的 Timer 特性（需要从源码编译）/ 使用 Kafka + 延迟表轮询 / 使用 Redis ZSet + 定时扫描 / 使用时间轮 Timing Wheel）如果选择 Redis ZSet 方案，高并发下（每秒创建 10 万个延迟任务）如何保证性能？（分片 ZSet / 分桶 / 异步批量写入）

262. 你的系统需要实现一个"消息广播"功能：一条消息需要被所有服务实例同时消费（比如配置变更通知）。Kafka 的消费者组机制下，同一条消息只能被一个消费者消费——如何实现广播？（每个实例使用不同的 Group ID / 使用 Kafka 的 `Wildcards` 模式 / 每个实例独立订阅所有 Partition）RocketMQ 的广播消费模式是怎么实现的？（每个 Consumer 实例都消费所有 MessageQueue / 本地存储 Offset 而不是远程存储）广播消费模式下，某个实例宕机重启后，会消费它宕机期间错过的消息吗？（不会，广播模式不持久化消费进度）

263. 你的系统用 Kafka 做数据管道，需要将同一个 Topic 的消息同时发送给"实时计算"和"离线存储"两套下游系统。实时计算要求低延迟（秒级），离线存储要求高吞吐但不要求实时性。如何设计？（同一个 Topic 不同消费者组 / Kafka Connect 导出 + Flink 消费 / 用 Kafka Streams 做 Branch 分流）如果离线消费者消费速度慢导致 Lag 越来越大，会影响实时消费者吗？（不会，不同消费者组独立消费）但如果离线消费者堆积太多导致 Broker 磁盘不足（因为 Log Retention 未清理），会影响其他 Topic 吗？

264. 你的系统需要做"消息溯源"（Message Audit）：每一笔交易的消息从创建到最终消费的完整链路都需要可追溯，用于满足审计合规要求。如何设计消息的全生命周期管理？（消息ID全局唯一 / 每个环节记录操作日志：发送、存储、投递、消费、ACK / 消息内容持久化到审计库 / 消息状态机：待发送→已发送→已投递→已消费/消费失败）如果消息被消费了 7 天后才发现有问题，需要"重新消费"历史消息，怎么做？（Kafka 设置较长的 Log Retention / 将消费过的消息备份到对象存储 / 消费者使用时间戳重置 offset）
