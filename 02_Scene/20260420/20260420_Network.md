## TCP连接问题场景（6题）

295. 你们的支付系统调用银行接口，每次请求都新建TCP连接再关闭。高峰期QPS 500，服务器上TIME_WAIT连接达到5万个，新连接偶尔失败。怎么排查和解决？（`netstat -an | grep TIME_WAIT | wc -l` 确认数量 / 每个请求创建+关闭=1个TIME_WAIT / QPS 500 × 60秒=3万 / 2MSL=60秒所以持续堆积 / 解决：1）HTTP Keep-Alive复用连接 / 2）连接池：PoolingHttpClientConnectionManager / 3）`tcp_tw_reuse=1` 允许复用 / 4）增大端口范围 / 5）让服务端主动关闭（TIME_WAIT出现在客户端）/ 最佳方案：连接池+Keep-Alive，从每请求一连接变成N连接复用）

296. 你们的IM系统用TCP长连接，用户反馈消息延迟3-5秒才收到。排查发现部分连接经过NAT设备，NAT超时时间2分钟，但心跳间隔3分钟。连接被NAT静默断开后客户端不知道，发了消息服务端收不到。怎么解决？（NAT静默断连：NAT设备清理映射表但不通知两端 / 心跳间隔必须小于NAT超时 / 调整心跳到30秒 / 确保NAT映射不会过期 / 同时解决：1）降低心跳间隔到30秒 / 2）客户端检测连续3次心跳无响应则重连 / 3）服务端检测读空闲60秒主动关闭 / 4）重连后补发离线消息 / 5）消息ACK确认机制 / 6）TCP KeepAlive参数调小：`tcp_keepalive_time=30` / `tcp_keepalive_intvl=10` / `tcp_keepalive_probes=3`）

297. 你们的RPC服务突然大量报 `Connection refused`，但服务端进程还在运行。重启服务后恢复正常。排查发现是连接队列满了。怎么理解TCP的连接队列？（TCP有两个队列：SYN队列（半连接）和Accept队列（全连接）/ `ss -lnt` 查看 `Recv-Q` 和 `Send-Q` / `Recv-Q` > 0 表示Accept队列有积压 / 积压原因：应用Accept太慢 / 突然大量连接 / Accept队列满后新连接被RST / `net.core.somaxconn` 控制Accept队列上限 / `tcp_max_syn_backlog` 控制SYN队列上限 / 解决：1）增大 `somaxconn` / 2）Tomcat的 `acceptCount` 参数 / 3）应用层优化Accept速度 / 4）避免在Accept线程做耗时操作 / 5）监控Accept队列长度告警）

298. 你们的微服务间通过HTTP通信，某个下游服务响应变慢，上游服务的连接池被占满。原因是下游服务GC停顿导致请求积压，上游连接全部阻塞在等响应。怎么从TCP层面优化超时和熔断？（连接池被占满→新请求被拒绝→级联故障 / 超时设置：连接超时200ms（快速发现不可达）/ 读取超时1秒（发现慢响应）/ 熔断器：Hystrix/Sentinel / 超时比例>50%触发熔断 / 连接池隔离：核心服务和非核心服务用不同连接池 / 重试策略：只对读操作重试 / 幂等要求 / TCP层面：`SO_TIMEOUT` 设置读取超时 / 连接池的 `validateAfterInactivity` 检查连接可用性 / 异步调用：CompletableFuture + 超时 / 监控：连接池使用率 / 平均RT / 超时率 / 熔断状态）

299. 你们的服务部署在K8s上，通过Nginx Ingress暴露。发现每4个请求中就有1个被路由到不同的Pod，导致有状态的WebSocket会话丢失。怎么解决？（K8s Service默认轮询 / Nginx Ingress默认轮询 / WebSocket需要会话保持 / 方案1：Nginx `ip_hash` 基于客户端IP路由 / 方案2：Cookie-based session affinity / 方案3：Redis存储WebSocket会话状态，任意Pod可恢复 / 方案4：Sticky Session + `nginx.ingress.kubernetes.io/affinity: cookie` / 方案5：客户端重连时带sessionID，服务端查找对应Pod / 最佳方案：无状态设计（Redis存会话）+ IP Hash兜底 / 注意：IP Hash在客户端NAT下可能不均匀 / Cookie affinity更可靠）

300. 你们的系统调用第三方支付接口，偶尔报 `java.net.SocketTimeoutException: Read timed out`。但第三方确认他们的服务正常，响应时间在100ms以内。怎么排查这种偶发超时？（偶发超时可能原因：1）网络抖动：中间链路丢包重传 / 2）DNS解析慢：JVM DNS缓存过期后重新解析 / 3）TCP连接复用问题：连接被中间设备断开但客户端不知道 / 4）GC停顿：STW期间连接超时 / 5）连接池中的死连接 / 排查步骤：1）设置DNS缓存 `networkaddress.cache.ttl=60` / 2）连接池健康检查 `validateAfterInactivity` / 3）TCP KeepAlive保持连接存活 / 4）GC日志确认是否有长时间STW / 5）开启HTTP客户端的详细日志 / 6）网络层面：`mtr` 检查到目标的路由丢包 / 7）对比不同时间段的成功/失败率）

## HTTP优化场景（5题）

301. 你们的首页加载需要请求10个API，每个API平均RT 100ms，串行加载完需要1秒。用户体验差，怎么优化？（串行请求总RT=100ms×10=1秒 / 优化1：并行请求，`CompletableFuture.allOf()` 总RT=100ms / 优化2：HTTP/2多路复用，一个连接并行10个请求 / 优化3：聚合API，BFF层合并多个请求为一个 / 优化4：缓存，静态数据本地缓存 / 优化5：预加载，首屏数据服务端渲染SSR / 优化6：CDN加速静态资源 / 优化7：Gzip压缩减少传输量 / 实际效果：并行+聚合后RT降到200ms / 加上缓存可降到50ms / 注意：并行请求不要超过连接池上限 / 浏览器对同一域名有并发限制HTTP/1.1为6个）

302. 你们的移动端App首屏加载慢，抓包发现每个请求都要300ms的TLS握手。每天活跃用户100万，TLS握手消耗大量服务端CPU。怎么优化？（TLS 1.2每次连接2-RTT握手 / 会话复用：Session ID/Session Ticket / TLS 1.3降到1-RTT / 0-RTT：首次连接后后续连接0-RTT / 服务端Session Cache / OpenSSL的Session Cache / 连接池：客户端复用连接避免重复握手 / CDN卸载：静态资源走CDN / HTTP/2：一个连接多路复用减少连接数 / 会话票证加密密钥轮换：安全+性能 / 监控：TLS握手次数 / 握手耗时 / 会话复用率 / 优化后：会话复用率>80% / 新连接握手从300ms降到100ms / TLS 1.3进一步降到50ms）

303. 你们的前后端分离项目，前端部署在Nginx，后端是Spring Boot。前端调用后端API报跨域错误：`Access-Control-Allow-Origin` 不匹配。后端加了CORS配置但OPTIONS预检请求还是很慢。怎么优化？（CORS预检请求：非简单请求先发OPTIONS / 每个跨域请求多一次RT / 优化1：`Access-Control-Max-Age: 86400` 缓存预检结果24小时 / 优化2：Nginx层处理CORS，不到达后端 / 优化3：同源部署：前端和API部署在同一域名下 / 优化4：反向代理：Nginx将API请求代理到后端 / 优化5：简化请求为简单请求（避免触发预检）/ Nginx CORS配置：

```nginx
add_header Access-Control-Allow-Origin $http_origin;
add_header Access-Control-Allow-Methods "GET, POST, OPTIONS";
add_header Access-Control-Max-Age 86400;
if ($request_method = OPTIONS) { return 204; }
```

最佳方案：同源部署+反向代理，彻底避免跨域）

304. 你们的API网关QPS峰值5万，但Nginx的 `worker_connections` 默认1024，偶尔出现502 Bad Gateway。怎么调优Nginx的网络参数？（502原因：Nginx无法连接到后端或后端返回无效响应 / `worker_connections` 太小：1024×worker数=总连接数 / 调优参数：`worker_processes auto` / `worker_connections 65535` / `worker_rlimit_nofile 65535` / `keepalive_timeout 65` / `keepalive_requests 10000` / 后端Keep-Alive连接池：`upstream` 块的 `keepalive 128` / `proxy_connect_timeout 5s` / `proxy_read_timeout 60s` / `proxy_send_timeout 60s` / 系统层面：`ulimit -n 65535` / `net.core.somaxconn=65535` / `net.ipv4.tcp_max_syn_backlog=65535` / 监控：Nginx Stub Status / 连接数/请求数/等待数）

305. 你们的系统需要从HTTP迁移到HTTPS，但担心TLS加解密影响性能。怎么评估和优化？（TLS性能影响：1）握手开销：首次连接+1-2RTT / 2）加解密开销：对称加密AES-GCM很快 / 3）非对称加密：ECDHE密钥交换较慢但只在握手时 / 评估：AES-GCM加密1GB数据约10ms / 握手约100-300ms / 优化1：TLS会话复用 / 优化2：HTTP/2减少连接数 / 优化3：OCSP Stapling减少证书验证时间 / 优化4：硬件加速：AES-NI指令集 / 优化5：CDN卸载TLS / 优化6：证书链优化：减少中间证书 / 优化7：ALPN协商HTTP/2 / 实际影响：CPU增加5-10% / RT增加50-100ms（首次握手）/ 会话复用后影响<10ms / 结论：性能影响可以接受，安全性更重要）

## HTTPS与安全场景（4题）

306. 你们的支付系统用了HTTPS，但安全团队发现SSL Labs评级为B，原因是支持TLS 1.0和弱密码套件。怎么升级到A级？（TLS 1.0/1.1不安全：POODLE/BEAST攻击 / 弱密码套件：RC4/3DES/CBC模式 / 升级步骤：1）禁用TLS 1.0/1.1，只保留TLS 1.2+1.3 / 2）密码套件只保留AES-GCM/CHACHA20 / 3）启用HSTS：`Strict-Transport-Security: max-age=31536000; includeSubDomains` / 4）启用OCSP Stapling / 5）DH参数至少2048位 / Nginx配置：`ssl_protocols TLSv1.2 TLSv1.3` / `ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256` / `ssl_prefer_server_ciphers on` / 升级后SSL Labs A+ / 注意：禁用TLS 1.0前确认没有老客户端依赖 / 监控TLS版本分布）

307. 你们的系统需要对接银行接口，银行要求双向TLS认证（mTLS）。你们的服务端需要验证银行的证书，银行也要验证你们的客户端证书。怎么实现？（mTLS：双方都验证对方证书 / 服务端配置：`ssl_client_certificate bank-ca.crt` / `ssl_verify_client on` / 客户端配置：携带客户端证书和私钥 / HttpClient配置：`SSLContext` 加载KeyStore（客户端证书+私钥）和TrustStore（银行CA证书）/ KeyStore类型：PKCS12 / 证书申请：向银行申请客户端证书 / 银行CA根证书导入TrustStore / 证书过期监控 / 自动续期 / Spring Boot配置：`server.ssl.client-auth=need` / `server.ssl.trust-store` / Nginx配置：`ssl_verify_client on` / `ssl_client_certificate` / 注意：私钥安全存储 / HSM/Vault管理私钥）

308. 你们的API接口被恶意刷接口，每天有几百万次异常请求。已有限流但攻击者不断换IP。怎么从网络层面加强防护？（攻击特征：大量不同IP / 高频请求 / 模拟正常请求 / 防护层次：1）WAF：规则过滤（User-Agent/请求模式）/ 2）CDN：边缘清洗，隐藏源站IP / 3）IP信誉库：封禁已知恶意IP段 / 4）验证码：可疑请求弹验证码 / 5）行为分析：请求频率/路径/参数模式 / 6）TLS指纹：JA3指纹识别恶意客户端 / 7）速率限制：按IP+按账号+按接口 / 8）GeoIP：限制非业务地区访问 / 9）API Gateway：请求签名验证 / 10）蜜罐接口：检测爬虫 / 架构：CDN→WAF→API Gateway→应用 / 攻击者换IP→行为分析+TLS指纹比IP更可靠 / 监控：异常请求占比/验证码触发率/拦截率）

309. 你们的系统需要存储用户密码和敏感数据。HTTPS只保护传输安全，数据落盘后如果数据库被拖库怎么办？（HTTPS只保护传输 / 数据落盘后明文存储=裸奔 / 密码存储：bcrypt/Argon2/SCrypt 加盐哈希 / 绝对不能MD5/SHA1 / 绝对不能明文 / 敏感数据加密：AES-256-GCM 对称加密 / 密钥管理：KMS/Vault/HSM / 数据脱敏：日志中只显示后4位 / 传输加密：HTTPS + 证书锁定（Certificate Pinning）/ 字段级加密：身份证/银行卡号加密存储 / 加密搜索：保留格式加密/哈希索引 / 密钥轮换：定期更换加密密钥 / 数据分类：公开/内部/敏感/机密 / 合规：GDPR/等保/PCI DSS / 审计：数据访问日志 / 密钥不与应用部署在一起）

## DNS与CDN场景（4题）

310. 你们的系统偶发出现服务调用超时，排查发现是DNS解析耗时2-3秒。JVM默认DNS缓存策略是什么？怎么优化？（JVM默认DNS缓存：成功30秒 / 失败10秒 / 缓存过期后重新解析 / 2-3秒的解析耗时说明DNS服务器响应慢 / 优化1：`networkaddress.cache.ttl=300` 延长缓存到5分钟 / 优化2：`networkaddress.cache.negative.ttl=0` 失败不缓存 / 优化3：配置 `/etc/hosts` 跳过DNS查询 / 优化4：本地DNS缓存服务：dnsmasq / 优化5：HTTPDNS：绕过Local DNS / 优化6：连接预热：启动时解析所有域名并缓存 / 优化7：监控DNS解析耗时 / 优化8：多DNS服务器配置 / Spring Boot：RestTemplate + HttpClient设置DNS缓存 / 注意：TTL过大→IP变更后连不上 / 结合服务发现：Nacos/Eureka直接获取IP不需要DNS）

311. 你们的图片服务用了CDN，但用户反馈图片更新后5分钟内看到的还是旧图片。怎么优化CDN缓存策略？（CDN缓存未及时刷新 / 原因：TTL设置为5分钟 / 解决方案1：更新时主动刷新CDN（Purge）/ 解决方案2：版本化URL：`image.jpg?v=2` 绕过缓存 / 解决方案3：缩短TTL到60秒 / 解决方案4：CDN预热：上传后主动推送 / 方案5：回源验证：`stale-while-revalidate` / 不同资源不同策略：静态资源长TTL（1天）/ 用户头像短TTL（5分钟）/ 实时数据不缓存 / CDN刷新API：阿里云CDN `RefreshObjectCaches` / 注意：Purge有频率限制 / URL版本化最可靠但不利于CDN命中率 / 最佳实践：长期不变资源长TTL / 会变化资源短TTL+Purge / 版本化URL强制刷新）

312. 你们的系统需要支持全球用户访问，国内用户访问正常但海外用户RT很高（500ms+）。怎么优化海外访问？（海外访问慢的原因：物理距离远 / 多次RTT / TCP慢启动 / 国际线路拥塞 / 优化1：海外CDN节点：CloudFlare/Akamai / 优化2：海外部署：多区域部署+就近接入 / 优化3：边缘计算：CloudFlare Workers / 优化4：静态资源CDN全球分发 / 优化5：TCP优化：BBR拥塞控制 / 优化6：HTTP/2+QUIC减少RTT / 优化7：Gzip/Brotli压缩 / 优化8：减少请求数：合并/内联 / 架构方案：DNS智能解析→就近CDN→海外Region→国内源站 / 数据同步：异地多活 / 全球负载均衡：GSLB / 注意：数据合规：GDPR / 海外部署成本 / 冷启动延迟）

313. 你们的系统上线后，部分用户报"无法访问"。排查发现这些用户用的公司网络，公司内部DNS解析你们的域名到了内网IP。怎么处理这种DNS劫持？（公司内部DNS劫持：企业DNS服务器将外部域名解析到内网地址 / 可能是安全策略 / 可能是误配置 / 处理1：联系对方IT部门修复DNS / 处理2：提供HTTPDNS方案给App用户 / 处理3：Web端无法用HTTPDNS，提供IP直连备选 / 处理4：提供备用域名 / 处理5：SSL证书问题：IP直连时证书域名不匹配 / 处理6：SNI问题：IP访问时无法发送SNI / 根本方案：联系对方网络管理员 / HTTPDNS SDK：客户端直接向HTTPDNS服务器查询 / 回退机制：HTTPDNS失败→本地DNS / 监控：用户报错中DNS解析失败的比例 / 报错收集：客户端上报解析结果和耗时）

## WebSocket实时通信场景（4题）

314. 你们的在线协作编辑系统用WebSocket推送编辑内容。用户反馈偶尔出现内容丢失，排查发现WebSocket连接断开后重连时丢失了断连期间的消息。怎么保证消息不丢失？（WebSocket断连期间的消息丢失 / 原因：服务端发送消息时客户端已断开 / 方案1：消息持久化+重连后补发 / 方案2：消息序号：每条消息带递增ID / 客户端重连时上报最后收到的ID / 服务端补发后续消息 / 方案3：消息ACK：客户端收到消息后回ACK / 服务端未收到ACK则重发 / 方案4：操作日志（OT/CRDT）：记录操作而非状态 / 方案5：心跳检测快速发现断连 / 重连策略：指数退避 / 补发消息的顺序保证 / 消息去重：客户端根据ID去重 / 和MQ的区别：WebSocket推送是推模式 / MQ是拉模式 / 可以结合：消息先写MQ再推送）

315. 你们的股票行情推送系统用WebSocket推送实时价格。行情数据每秒更新10次，单机需要支撑5万连接。当前实现每个连接一个线程，内存和CPU都无法支撑。怎么优化？（每个连接一个线程→5万线程→内存5万×1MB=50GB / CPU上下文切换开销巨大 / 优化1：Netty NIO：少量IO线程+Selector / 优化2：连接分组：多个EventLoop分组处理 / 优化3：数据共享：行情数据只需写一次，广播到所有连接 / 优化4：批量写入：攒一批数据一起flush / 优化5：压缩：Protobuf替代JSON / 优化6：连接分级：VIP用户单独分组保证质量 / Netty架构：bossGroup(1线程) + workerGroup(CPU×2线程) + 业务线程 / 推送优化：`writeAndFlush` 批量 / `isWritable()` 背压 / 监控：连接数/推送延迟/丢包率 / 单机5万连接只需4核8GB）

316. 你们的聊天系统用WebSocket，需要支持"消息已读"功能。客户端发送已读回执，服务端需要通知发送者。已读回执量很大（每秒1万+），怎么设计？（已读回执：每条消息一个回执 / 高频小消息 / 方案1：批量回执：攒100ms的回执一起处理 / 方案2：合并回执：只保留每个会话的最新已读位置 / 方案3：回执走MQ异步处理 / 方案4：Redis记录已读位置：`read:conversation:{id}` → `messageId` / 推送策略：只推送会话级别的已读位置而非逐条 / WebSocket推送格式：`{type: "read", conversationId: "xxx", lastReadMsgId: 123}` / 存储：HBase/Cassandra存历史已读 / Redis存最新已读 / 对比：逐条回执→1万QPS / 合并回执→100QPS / 架构：WebSocket接收→MQ→消费处理→Redis更新→WebSocket推送通知）

317. 你们的直播间有10万观众，主播操作需要实时推送给所有观众。当前每条消息逐个连接推送，推送完需要3秒。怎么优化推送延迟？（10万连接逐个推送→3秒 / 优化1：Netty的 `writeAndFlush` 批量：遍历所有Channel一次性写入 / 优化2：Channel分组：`ChannelGroup` 广播 / 优化3：多EventLoop并行推送：每个EventLoop负责一部分连接 / 优化4：消息编码只做一次：编码为ByteBuf后共享给所有Channel / 优化5：零拷贝：`FileRegion` / 优化6：UDP组播（局域网）/ 优化7：CDN边缘推送：观众就近连接CDN / 架构：主播消息→消息服务→Redis Pub/Sub→各推送节点→Netty ChannelGroup广播 / 优化后：10万连接推送延迟<100ms / 关键：编码一次+并行推送+批量flush / 监控：推送延迟P99/推送成功率/连接存活率）

## 网络排查与性能场景（7题）

318. 你们的API接口偶发出现504 Gateway Timeout，但后端服务日志显示请求处理只花了100ms。Nginx到后端之间到底发生了什么？（504是Nginx判断后端超时 / 后端处理100ms但Nginx认为超时 / 可能原因：1）Nginx的 `proxy_read_timeout` 设置过短 / 2）Nginx到后端的连接建立慢（TCP握手+TLS）/ 3）后端 Accept 队列满导致连接等待 / 4）网络抖动导致TCP重传 / 排查：1）Nginx error.log 看具体超时时间 / 2）`proxy_read_timeout` 当前配置 / 3）后端 Accept 队列 `Recv-Q` / 4）`tcpdump` 抓包看TCP握手和响应时间 / 5）Nginx `$upstream_response_time` vs `$request_time` / 如果 `$request_time` >> `$upstream_response_time` 说明瓶颈在Nginx到后端的连接建立 / 解决：调大 `proxy_read_timeout` / 优化连接建立 / 连接池 keepalive）

319. 你们的系统大促期间突然大量502 Bad Gateway。Nginx日志显示 `upstream prematurely closed connection`。后端是Java服务，JVM正在做Full GC。怎么从网络和应用层面解决？（502 + `prematurely closed connection` = 后端在处理过程中连接被关闭 / Full GC时应用线程暂停 / Nginx等不到响应超时 / Nginx的 `proxy_read_timeout` 默认60秒 / Full GC如果<60秒应该不会502 / 可能是：1）Full GC > 60秒 / 2）Full GC导致连接队列溢出 / 3）OOM Killer杀进程 / 解决方案：1）优化GC：G1/ZGC减少停顿 / 2）增大 `proxy_read_timeout` / 3）优雅关闭：Spring Boot graceful shutdown / 4）连接池检测死连接 / 5）熔断：Sentinel/GC停顿时触发熔断 / 6）隔离：核心服务独立部署 / 7）监控：GC日志+JVM监控+Nginx 502告警 / 8）兜底：502时返回缓存数据）

320. 你们的微服务间调用偶尔出现 `java.net.UnknownHostException`，持续几秒后自动恢复。排查发现是DNS服务器偶发不可达。怎么保证DNS解析的高可用？（DNS服务器不可达→解析失败→UnknownHostException / JVM默认失败缓存10秒 / 解决方案1：多DNS服务器配置：主备 / 解决方案2：本地DNS缓存：dnsmasq / 解决方案3：JVM参数：`networkaddress.cache.ttl=300` / `networkaddress.cache.negative.ttl=0` / 解决方案4：配置 `/etc/hosts` 作为兜底 / 解决方案5：应用层DNS缓存：Guava Cache / 解决方案6：HTTPDNS / 解决方案7：服务发现替代DNS：Nacos/Eureka / 解决方案8：IP直连+配置中心 / 最佳方案：服务发现（Nacos）+ DNS兜底 + 本地缓存 / 监控：DNS解析成功率/耗时 / 告警：解析失败率>1% / 客户端重试：DNS失败时重试1-2次）

321. 你们的系统需要对接一个外部HTTPS接口，但对方用的自签名证书。Java客户端报 `PKIX path building failed`。怎么处理？（自签名证书不被Java信任库信任 / 临时方案：将证书导入JDK的cacerts / `keytool -import -alias xxx -file xxx.crt -keystore $JAVA_HOME/jre/lib/security/cacerts` / 代码方案：自定义TrustManager信任所有证书（不安全）/ 更好的方案：自定义TrustManager只信任特定证书 / 代码：

```java
KeyStore trustStore = KeyStore.getInstance("PKCS12");
trustStore.load(new FileInputStream("truststore.p12"), password);
SSLContext sslContext = SSLContextBuilder.create()
    .loadTrustMaterial(trustStore, null)
    .build();
```

最佳方案：让对方用CA签名证书 / 如果是测试环境可以导入cacerts / 生产环境禁止信任所有证书 / 证书过期监控）

322. 你们的系统上线后，运维发现TCP连接数持续增长不下降。`netstat` 显示大量 `ESTABLISHED` 状态的空闲连接。怎么排查连接泄漏？（连接持续增长→泄漏 / 排查步骤：1）`netstat -anp | grep ESTABLISHED | wc -l` 统计 / 2）`lsof -i :8080` 查看连接详情 / 3）按进程分组：哪些连接最多 / 4）检查代码：HTTP客户端是否正确关闭连接 / 5）连接池配置：maxTotal是否过大 / 6）idle连接清理：`setValidateAfterInactivity` / `setMaxIdleTime` / 7）Keep-Alive超时：服务端 `keep-alive timeout` / 8）TCP KeepAlive：检测死连接 / 9）应用层心跳 / 常见原因：连接池没有idle清理 / 没有设置maxIdle / 没有调用close / try-with-resources / 监控：连接池活跃数/空闲数/等待数 / 告警：连接数超过阈值 / HttpClient的 `IdleConnectionEvictor`）

323. 你们的支付回调接口要求高可用，但银行回调你们的服务时偶尔失败。银行那边看到的错误是 `Connection reset`。你们的Nginx日志没有对应请求记录。怎么排查？（银行发请求→Nginx没记录→请求没到达Nginx / 可能1：防火墙/安全组拦截 / 可能2：DNS解析到错误IP / 可能3：中间网络设备RST / 可能4：TCP连接被拒绝 / 排查：1）确认银行的DNS解析结果是否正确 / 2）检查防火墙/安全组是否放行银行IP / 3）`tcpdump` 在服务器上抓包看是否有银行IP的SYN / 4）如果SYN都没到→防火墙拦截 / 5）如果SYN到了但回RST→服务未监听/端口错误 / 6）如果TCP建立成功但HTTP被RST→WAF拦截 / 解决：开放银行IP白名单 / 确认DNS解析 / 检查WAF规则 / 银行回调IP可能是多个，需要全部放行 / 回调重试机制：银行通常会重试3次）

324. 你们的系统需要从HTTP/1.1迁移到HTTP/2。迁移后部分客户端报 `ERR_HTTP2_PROTOCOL_ERROR`。怎么排查和解决？（HTTP/2协议错误 / 可能原因：1）中间代理不支持HTTP/2 / 2）HTTP/2头部压缩HPACK不兼容 / 3）TLS ALPN协商失败 / 4）HTTP/2的伪头部 `:method/:path/:authority/:scheme` 缺失 / 5）请求头包含HTTP/2不允许的头部（如Connection）/ 排查：1）确认客户端和Nginx的HTTP/2配置 / 2）`curl -v --http2` 测试 / 3）Chrome `chrome://net-internals/#http2` / 4）Nginx `error_log` 看具体错误 / 解决：1）Nginx配置 `http2 on` + `listen 443 ssl http2` / 2）确保ALPN配置：`ssl_alpn http1.1 http2` / 3）移除不兼容的中间头 / 4）降级：检测到不支持HTTP/2时自动降级HTTP/1.1 / 5）逐步迁移：先在非核心接口测试）
