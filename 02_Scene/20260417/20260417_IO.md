# 业务场景题 - Java IO/NIO（B265-B294）

> 以下为 Java IO/NIO 领域的业务场景题，编号从265起，覆盖高并发IO、文件处理、网络通信、序列化、零拷贝、Netty框架等实际业务场景中的问题与优化。

## 高并发IO场景（6题）

265. 你们系统有个文件上传接口，用户上传10MB~500MB的文件，当前用BIO方式读取请求体再写入磁盘。高峰期30个并发上传，Tomcat线程池就打满了，接口RT从200ms飙到5s+。你怎么排查和优化？（BIO读取大文件时线程一直阻塞在 `InputStream.read()` / 每个上传占用一个Tomcat线程30s+ / 30个上传就把线程池吃光了）优化方案有哪些？（NIO + Chunked传输：Netty接收分块写入 / 异步Servlet：`AsyncContext` 释放Tomcat线程 / 直接流式写入磁盘不做内存缓存 / 限制并发上传数用Semaphore控制）

266. 你们有个实时日志推送服务，客户端通过WebSocket连接服务端，服务端把应用日志实时推给前端展示。当前单机支撑500个WebSocket连接，CPU就到80%了。你觉得问题在哪？怎么优化？（BIO模型下每个连接一个线程 / 500个连接=500个线程 / 线程上下文切换开销大 / 大部分时间线程在等IO / NIO + Selector：一个线程管理多个连接 / Netty的WebSocket支持 / 连接分组多Reactor线程 / 心跳检测断开空闲连接释放资源）

267. 你们的RPC框架底层用BIO做网络通信，调用下游服务超时时间3秒。某次下游服务Full GC停顿5秒，上游所有调用线程全部阻塞在 `InputStream.read()` 等响应，导致上游线程池耗尽级联故障。怎么从IO层面避免这种级联？（BIO的read无法中断是核心问题 / 设置SO_TIMEOUT但BIO下粒度不够 / 改NIO：读操作非阻塞，超时直接取消 / 连接池+超时熔断：单连接超时不影响其他请求 / Hystrix/Sentinel信号量隔离限制并发）

268. 你们有个IM消息网关，10万在线用户，每人平均每30秒收发1条消息。当前用Tomcat WebSocket实现，单机只能支撑3000连接就要扩容。估算一下需要多少台机器？怎么大幅减少机器数量？（10万连接 / 单机3000 / 需要34台 / 切Netty NIO后单机可支撑5万+连接 / 只需2-3台 / 核心区别：Netty用少量IO线程+Selector管理大量连接 / Tomcat每连接一线程）

269. 你们的数据同步服务每分钟从数据库查出5万条记录，序列化后通过HTTP推给下游10个服务。当前用BIO HttpClient逐条发送，5万条处理完要3分钟，经常超时。怎么优化？（逐条发送效率极低 / 批量发送：每批500条 / 连接池复用：PoolingHttpClientConnectionManager / 并行发送：CompletableFuture + 10个下游并行 / NIO异步发送：Netty HTTP Client / 序列化优化：JSON换Protobuf减少传输量）

270. 你们系统需要同时连接3个外部支付渠道（微信/支付宝/银联），每个渠道有独立的超时和重试策略。当前用BIO方式顺序调用，一笔支付要等3个渠道都确认才返回，RT经常超2秒。怎么从IO模型上优化？（顺序调用改为并行：CompletableFuture.allOf / 每个渠道独立HTTP连接池 / NIO异步调用+各自超时控制 / 最快确认即可返回+降级策略 / 熔断器保护：某渠道慢不影响其他渠道的独立调用）

## 文件处理场景（6题）

271. 你们系统每天凌晨需要处理一个2GB的CSV对账文件，逐行解析后写入数据库。当前用 `Files.readAllLines()` 一次性加载，经常OOM。怎么优化？（2GB文件全部加载到内存 / `Files.readAllLines()` 返回 `List<String>` / 2GB文件→堆内存至少需要4GB+ / 改为 `Files.lines()` 流式读取：逐行处理不全部加载 / BufferedReader逐行读 / 数据库批量INSERT：每1000条提交 / 内存占用从4GB降到几十MB）

272. 你们有个报表导出功能，用户选择条件后导出50万行Excel。当前用POI的 `XSSFWorkbook` 在内存中构建整个Excel，50万行数据直接OOM。怎么解决？（XSSFWorkbook是DOM模型，50万行内存爆炸 / 改SXSSFWorkbook流式写入：滑动窗口只在内存保留100行 / 改CSV格式：更轻量 / 分页导出：每10万行一个文件 / 异步导出：后台生成后通知下载 / 限制最大导出行数）

273. 你们系统需要实时监控某个目录下的新增文件，文件到达后立即解析入库。当前用定时任务每5秒扫描一次目录，但5秒延迟不满足业务要求，而且文件多时扫描耗时增加。怎么优化？（定时轮询有延迟且效率低 / JDK 7 WatchService：操作系统级文件变更通知 / Apache Commons IO的FileAlterationMonitor / Linux的inotify / WatchService注意：Windows可靠但Linux下EVENT_OVERFLOW可能丢事件 / 严重场景用MQ替代：上传服务直接发消息通知）

274. 你们有个大文件上传功能，支持断点续传。用户上传到一半网络断开了，重新上传时从断点继续。当前方案是把文件分片上传到临时目录，断点信息存在Redis。如果Redis断点信息丢了怎么办？怎么保证断点续传的可靠性？（Redis断点信息丢失→客户端从0重新上传 / 方案1：断点信息双写Redis+本地文件 / 方案2：服务端根据已接收分片计算断点 / 方案3：分片上传完成后立即落盘，不依赖Redis / 分片校验：每个分片上传后做MD5校验 / 合并时校验整个文件完整性）

275. 你们系统需要把MySQL中的100万条用户数据导出为JSON文件供第三方使用。当前用 `SELECT *` 一次性查出所有数据再JSON序列化，内存直接爆了。怎么做？（禁止一次性加载全量数据 / 流式查询：MyBatis的Cursor / JDBC的 `setFetchSize(Integer.MIN_VALUE)` MySQL流式结果集 / 逐行序列化写入文件：不全部加载到内存 / 分批查询：每批1万条，用ID做游标 / JSON流式写入：Jackson的JsonGenerator / 内存占用控制在百MB以内）

276. 你们的应用日志目录每天产生500MB日志，需要定期归档压缩。当前用Java的 `ZipOutputStream` 压缩，500MB日志压缩完要2分钟，CPU占用很高。怎么优化？（ZipOutputStream是单线程压缩 / 用 `gzip` 命令行工具比Java快 / 并行压缩：分块读入多线程压缩后合并 / NIO零拷贝：FileChannel.transferTo直接传输 / 归档策略：按小时滚动，单文件更小压缩更快 / 考虑logback的RollingPolicy自动压缩 / 压缩级别调低：速度优先于压缩率）

## 网络通信场景（6题）

277. 你们的微服务间用gRPC通信，某次发布后客户端报 `INTERNAL: HTTP/2 error code: INTERNAL_ERROR`，但服务端日志没有异常。怎么排查？（gRPC基于HTTP/2 / 可能是服务端发送了RST_STREAM / 检查gRPC的maxMessageSize配置 / 可能是header过大超限 / 检查服务端是否OOM导致连接被杀 / gRPC Trailer中的错误信息 / Wireshark抓包分析HTTP/2帧 / 检查是否是gRPC版本不兼容）

278. 你们系统需要对接一个第三方短信服务，通过HTTPS调用。高峰期QPS 200，当前每个请求都新建HTTPS连接，RT中TLS握手占了200ms，总RT 350ms。怎么优化TLS握手开销？（每次新建连接都要TLS握手 / 连接池：PoolingHttpClientConnectionManager复用连接 / HTTP Keep-Alive：复用TCP连接避免重复握手 / 服务端session ticket：恢复TLS会话跳过完整握手 / 连接池配置：maxTotal=200 / maxPerRoute=50 / 预热：启动时创建连接池并初始化连接 / RT从350ms降到150ms）

279. 你们的支付网关需要同时对接10+支付渠道，每个渠道的HTTP超时、重试策略都不同。当前用一个统一的HttpClient，经常出现A渠道超时导致B渠道的连接池被占满。怎么设计？（统一HttpClient的问题：共享连接池，一个渠道慢影响所有 / 方案1：每个渠道独立连接池，独立配置 / 方案2：按渠道分组，核心渠道和非核心渠道隔离 / 方案3：Netty异步HTTP Client，按渠道分配EventLoop / 超时分级：快速渠道1秒 / 普通渠道3秒 / 慢渠道5秒 / 熔断器独立：每渠道独立熔断状态）

280. 你们系统有个长连接推送服务，客户端通过TCP长连接接收实时消息。运维发现服务器上有大量 `CLOSE_WAIT` 状态的连接，几天累积到上万个。这是什么问题？怎么解决？（CLOSE_WAIT说明对端关闭了连接，本端没有调用close / 应用层没有正确处理连接关闭事件 / 检查是否缺少对 `-1` 返回值的处理 / Netty的 `channelInactive` 是否正确释放资源 / 心跳机制：定时检测空闲连接主动关闭 / 连接泄漏排查：是否在业务异常时未关闭连接 / 设置SO_LINGER避免大量CLOSE_WAIT堆积）

281. 你们的HTTP服务偶发出现大量 `Connection reset by peer` 错误，尤其在高并发场景下。这是什么原因？怎么处理？（客户端在服务端响应前关闭了连接 / 服务端还在写数据时客户端已断 / 高并发下客户端超时断开 / 负载均衡器健康检查断开 / 代码层面：捕获IOException忽略即可，不需要重试 / 网络层面：调整keepalive和timeout / 监控层面：区分正常reset和异常reset / 如果reset率>5%需要排查是否服务端响应太慢）

282. 你们的服务部署在K8s上，通过Service暴露。Service后面有3个Pod，但发现请求分布严重不均，某个Pod接收了70%的流量。从网络层面怎么排查？（K8s Service默认iptables随机转发 / 检查iptables规则是否正确 / 检查Pod的readiness探针 / 客户端连接复用导致长连接粘滞 / 旧版本kube-proxy的已知bug / 如果用Nginx Ingress检查upstream配置 / 连接池+长连接导致流量固定路由到一个Pod / 短连接或定期重建连接来均衡）

## 序列化与数据格式场景（5题）

283. 你们的微服务间用JSON序列化通信，一个订单对象序列化后2KB，日均1亿次调用，带宽成本很高。怎么优化？（JSON文本格式冗余 / Protobuf：二进制序列化，通常缩小3-5倍 / 字段名缩短 / 压缩传输：gzip / MsgPack：兼容JSON但二进制 / 评估改造代价：Protobuf需要IDL定义和代码生成 / 渐进方案：先加gzip压缩，再逐步迁移Protobuf / 带宽从200GB/天降到60GB/天）

284. 你们的支付系统用JSON存储交易报文，需要保留10年。当前单个报文5KB，每天100万笔交易，10年存储成本极高。怎么从序列化角度优化存储成本？（JSON文本存储效率低 / Avro/Protobuf二进制存储：压缩3-5倍 / 列式存储：Parquet/ORC，适合归档查询 / 冷热分层：近期JSON在线查询 / 远期二进制归档冷存 / 压缩：gzip后JSON约1KB / Protobuf约0.5KB / 估算：5KB × 100万 × 365 × 10 = 18TB → 压缩后6TB → Protobuf+压缩2TB）

285. 你们的订单服务用Jackson序列化返回给前端，但前端经常报字段类型错误。后端 `BigDecimal` 序列化成了字符串 `"12.34"` 而非数字 `12.34`，日期序列化成了时间戳 `1713369600000` 而非 `"2024-04-17"`。怎么统一规范？（Jackson的序列化行为可配置 / `@JsonSerialize` 注解自定义序列化器 / 全局ObjectMapper配置 / BigDecimal：`configure(Feature.WRITE_BIGDECIMAL_AS_PLAIN, false)` / 日期：`@JsonFormat(pattern="yyyy-MM-dd HH:mm:ss")` / 统一ObjectMapper Bean注入 / API文档：Swagger生成字段类型说明 / 前后端约定序列化规范）

286. 你们的缓存系统用Java原生序列化存储对象到Redis，反序列化时经常报 `InvalidClassException: local class incompatible: stream classdesc serialVersionUID = 1, local class serialVersionUID = 2`。怎么彻底解决？（Java原生序列化依赖serialVersionUID / 类修改后UID变化导致不兼容 / 方案1：显式声明serialVersionUID / 方案2：改用JSON/Protobuf序列化，不依赖UID / 方案3：缓存对象用DTO隔离，与业务类解耦 / 最佳实践：Redis存JSON字符串 / 加上@class类型信息方便反序列化 / 原生序列化的其他坑：大小膨胀3-5倍 / 跨语言不兼容 / 安全漏洞）

287. 你们的服务间通信用Protobuf，新增了一个字段后，老版本的服务反序列化报错。Protobuf不是号称前后兼容吗？为什么还会出错？（Protobuf默认前后兼容：新增字段老版本忽略 / 报错原因可能是：修改了已有字段的编号 / 删除了字段但编号被复用 / 修改了字段类型不兼容 / required字段在proto3已移除但proto2中使用 / 检查字段编号是否被复用 / Protobuf兼容规则：只能新增字段 / 不能修改编号 / 不能修改类型 / 使用reserved保护已删除字段的编号）

## 零拷贝与高性能传输场景（4题）

288. 你们有个文件下载服务，用户下载100MB~1GB的文件。当前用 `FileInputStream.read() → ServletOutputStream.write()` 循环拷贝，下载速度只有50MB/s，CPU占用30%。怎么优化到接近网卡极限？（传统IO：磁盘→内核缓冲区→用户缓冲区→Socket缓冲区→网卡，4次拷贝 / `FileChannel.transferTo()` 零拷贝：磁盘→内核缓冲区→网卡，2次拷贝 / Linux sendfile系统调用 / 效果：速度提升到200MB+，CPU降到5% / Nginx的 `sendfile on` 同理 / 注意：transferTo在Linux上用sendfile，Windows上退化）

289. 你们的消息队列需要把消息从磁盘发送到网络。Kafka是怎么做到单机百万级TPS的？从IO层面分析（Kafka使用零拷贝：`FileChannel.transferTo()` / 数据写入用mmap：`MappedByteBuffer` / 批量发送：多条消息攒批减少系统调用 / 顺序写磁盘：追加写，避免随机写 / PageCache：写入先到PageCache，操作系统异步刷盘 / 操作系统层面：预读、合并写 / 对比自己系统：是否也能用mmap+transferTo / Netty的FileRegion封装transferTo）

290. 你们系统需要定期把1GB的数据库备份文件从A服务器传输到B服务器。当前用SCP传输需要40秒，有没有更快的方式？（SCP加密传输有CPU开销 / 方案1：rsync增量传输只传差异 / 方案2：NC+tar不加密直传 / 方案3：零拷贝+压缩：先压缩再传输 / 方案4：mmap+sendfile减少拷贝 / 方案5：多线程分块并行传输 / 实际场景：内网不加密传输10Gbps网卡理论1.25GB/s / 1GB文件0.8秒 / SCP慢因为加密+单线程 / 根据安全要求选方案）

291. 你们的图片服务需要在两台服务器之间同步图片文件，日均同步10万个图片文件，总大小约50GB。当前每张图片单独建立TCP连接传输，同步完要2小时。怎么优化？（每张图片单独连接：TCP三次握手+慢启动 / 连接复用：长连接+HTTP Keep-Alive / 批量传输：打包tar后单连接传输 / 零拷贝：FileChannel.transferTo / 并行传输：多个连接并行传不同文件 / rsync+inotify：增量实时同步 / 压缩：相似图片去重/差异传输 / 优化后：连接复用+并行10连接+零拷贝，预计15分钟）

## Netty与框架场景（3题）

292. 你们用Netty做网关，某次压测发现QPS到5万后上不去了，CPU只有40%。分析发现是业务Handler中有数据库查询操作阻塞了IO线程。怎么解决？（Netty的IO线程（EventLoop）不能做阻塞操作 / 阻塞操作会让整个EventLoop上的所有连接排队 / 方案1：阻塞操作提交到独立业务线程池 / 方案2：增加EventLoop线程数（不推荐，违背NIO设计） / 方案3：异步数据库驱动 / 正确做法：自定义线程池 `executorService.submit(() -> { ... })` / 结果提交回IO线程：`ctx.writeAndFlush()`）

293. 你们的Netty服务经常出现 `OutOfDirectMemoryError: failed to allocate` 错误，尤其是大文件上传场景。怎么排查和解决？（Netty默认使用堆外直接内存 / 直接内存不受-Xmx控制 / `-Dio.netty.maxDirectMemory` 默认和-Xmx一样 / 大文件上传：每请求分配大ByteBuf / 泄漏：ByteBuf没有release / 方案1：限制单请求ByteBuf大小 / 方案2：设置 `-Dio.netty.maxDirectMemory=512m` / 方案3：用 `PooledByteBufAllocator` + 引用计数 / 方案4：文件上传用FileRegion零拷贝不经过用户空间 / 监控：`PlatformDependent.directMemoryUsed()`）

294. 你们用Netty实现了一个自定义协议的通信框架。上线后客户端频繁断连重连，Netty报 `DecoderException: java.io.StreamCorruptedException: invalid stream header`。这通常是什么原因？（自定义协议解码问题 / 客户端发的是JSON但服务端用Protobuf解码器 / 半包/粘包：消息没有完整帧边界 / 缺少LengthFieldBasedFrameDecoder / 字节序不匹配：大端小端 / 解决：加LengthField帧解码器 / 统一编解码器配置 / 检查是否漏加了SSL Handler / 确保客户端和服务端Pipeline配置一致）
