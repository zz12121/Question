## 先说结论

多租户数据隔离有三种方案：**独立数据库、共享库独立表、共享库共享表加 `tenant_id` 字段**，主流 SaaS 系统用的是第三种——共享表 + `tenant_id` 列级隔离，再配合**拦截器自动注入租户上下文**，做到代码无感知、数据强隔离。

---

## 🔥 核心概念

- **解决什么问题**：多个客户（租户）共用一套系统，数据之间绝对不能互相看到，同时资源利用率要高、运维成本要低
- **三种隔离粒度**：
  - 数据库级隔离（最强，成本最高）
  - Schema 级隔离（中等）
  - 行级隔离（最轻，主流选择）

---

## 🔥 详细展开

### 1. 三种方案对比

| 方案 | 原理 | 隔离性 | 成本 | 适用场景 |
|------|------|--------|------|----------|
| **独立数据库** | 每个租户一个 DB | 最强，物理隔离 | 极高，租户多了扛不住 | 政企大客户、高安全要求 |
| **共享库独立 Schema** | 同一 DB，每租户一套表 | 较强，逻辑隔离 | 中等，表数量爆炸 | 租户数量不多（几十个） |
| **共享库共享表 + tenant_id** | 所有租户同一张表，靠字段区分 | 行级隔离，代码层保证 | 最低，最省资源 | 中小 SaaS，租户数量大 |

> 💡 **绝大多数互联网 SaaS 用第三种**，本文重点讲这个。

---

### 2. 行级隔离核心设计

**① 所有业务表加 `tenant_id` 字段**

```sql
CREATE TABLE orders (
  order_id    BIGINT PRIMARY KEY,
  tenant_id   BIGINT NOT NULL,     -- 🔑 租户ID，每张表都要有
  user_id     BIGINT NOT NULL,
  amount      DECIMAL(10,2),
  created_at  DATETIME,
  INDEX idx_tenant_user (tenant_id, user_id)  -- 联合索引，tenant_id 放最左
);
```

> 💡 **每张表的索引都要以 `tenant_id` 打头**，否则跨租户查询会扫全表。

---

**② 租户上下文用 ThreadLocal 传递**

```java
// 租户上下文持有者
public class TenantContext {
    private static final ThreadLocal<Long> TENANT_ID = new ThreadLocal<>();
    
    public static void set(Long tenantId) { TENANT_ID.set(tenantId); }
    public static Long get() { return TENANT_ID.get(); }
    public static void clear() { TENANT_ID.remove(); }  // 用完必须清！
}
```

---

**③ 拦截器在请求入口自动注入 tenant_id**

```java
@Component
public class TenantInterceptor implements HandlerInterceptor {
    
    @Override
    public boolean preHandle(HttpServletRequest request, ...) {
        // 从 JWT Token 或请求头里解析租户ID
        String token = request.getHeader("Authorization");
        Long tenantId = jwtUtil.extractTenantId(token);
        TenantContext.set(tenantId);
        return true;
    }
    
    @Override
    public void afterCompletion(...) {
        TenantContext.clear();  // 🔑 请求结束必须清理，防止线程复用导致串租户
    }
}
```

---

**④ MyBatis 拦截器自动拼 WHERE tenant_id = ?**

这是最关键的一步，让业务代码完全感知不到多租户：

```java
@Intercepts({
    @Signature(type = StatementHandler.class, method = "prepare", 
               args = {Connection.class, Integer.class})
})
@Component
public class TenantSqlInterceptor implements Interceptor {
    
    // 不需要加 tenant_id 的表（系统表、字典表等）
    private static final Set<String> SKIP_TABLES = 
        Set.of("tenant", "sys_config", "data_dict");
    
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        StatementHandler handler = (StatementHandler) invocation.getTarget();
        BoundSql boundSql = handler.getBoundSql();
        String originalSql = boundSql.getSql();
        
        Long tenantId = TenantContext.get();
        if (tenantId == null) return invocation.proceed();
        
        // 用 JSqlParser 解析 SQL，自动追加 tenant_id 条件
        String newSql = appendTenantCondition(originalSql, tenantId);
        // 反射替换 SQL（略）
        return invocation.proceed();
    }
}
```

> 💡 实际项目中直接用 **MyBatis-Plus 的多租户插件**（`TenantLineInnerInterceptor`），原理一样，一行配置搞定，不用自己写拦截器。

---

**⑤ MyBatis-Plus 多租户插件配置（实际项目写法）**

```java
@Configuration
public class MybatisPlusConfig {
    
    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
        
        // 多租户插件
        interceptor.addInnerInterceptor(new TenantLineInnerInterceptor(
            new TenantLineHandler() {
                @Override
                public Expression getTenantId() {
                    // 从 ThreadLocal 拿当前租户ID
                    return new LongValue(TenantContext.get());
                }
                
                @Override
                public boolean ignoreTable(String tableName) {
                    // 系统表跳过
                    return SKIP_TABLES.contains(tableName);
                }
            }
        ));
        
        // 分页插件
        interceptor.addInnerInterceptor(new PaginationInnerInterceptor());
        return interceptor;
    }
}
```

这样配完之后，业务代码里写：
```java
// 业务代码完全不感知多租户，就像普通查询一样
List<Order> orders = orderMapper.selectList(
    new LambdaQueryWrapper<Order>().eq(Order::getUserId, userId)
);
// 实际执行 SQL：SELECT * FROM orders WHERE tenant_id = 100 AND user_id = 123
```

### 3. 整体架构流程

```
HTTP 请求
    ↓
JWT 解析 → 获取 tenant_id → TenantContext.set(tenantId)
    ↓
业务代码（无感知，正常写 SQL）
    ↓
MyBatis-Plus 拦截器 → 自动追加 WHERE tenant_id = ?
    ↓
数据库返回当前租户数据
    ↓
afterCompletion → TenantContext.clear()
```

## 🔥 扩展追问 1：如果不用 MyBatis-Plus，怎么做到自动注入？

两种方案：
1. **自定义 MyBatis Interceptor**：拦截 `StatementHandler.prepare()`，用 JSqlParser 解析 SQL AST，向 WHERE 子句追加 `tenant_id = ?` 条件
2. **数据库视图 + Row Level Security**：PostgreSQL 支持 RLS（Row Level Security），在 DB 层做策略，业务完全无感。MySQL 不支持，只能应用层解决

---

## 🔥 扩展追问 2：ThreadLocal 不是线程安全的吗？异步线程怎么传递 tenant_id？

**ThreadLocal 只绑定当前线程**，一旦用了线程池（`@Async`、`CompletableFuture`），子线程拿不到父线程的 `tenant_id`，会是 `null`。

解决方案：
```java
// 方案1：用 TransmittableThreadLocal（TTL，阿里开源）
private static final TransmittableThreadLocal<Long> TENANT_ID 
    = new TransmittableThreadLocal<>();

// 方案2：手动传递
Long tenantId = TenantContext.get();
CompletableFuture.runAsync(() -> {
    TenantContext.set(tenantId);  // 子线程手动 set
    try {
        doWork();
    } finally {
        TenantContext.clear();   // 用完清理
    }
});
```

> 💡 生产环境推荐 **TTL（TransmittableThreadLocal）**，配合 `TtlExecutors.getTtlExecutorService()` 包装线程池，自动透传所有 ThreadLocal 值，省心。

## 🔥 扩展追问 3：索引怎么设计才高效？

所有联合索引都把 `tenant_id` 放**最左前缀**：

```sql
-- ✅ 正确：tenant_id 在最左，走索引
INDEX idx_tenant_status_time (tenant_id, status, created_at)

-- ❌ 错误：tenant_id 不在最左，范围查询时退化
INDEX idx_status_time (status, created_at)
```

原因：查询几乎所有都带 `tenant_id = ?` 等值条件，放最左可以先用它缩小范围，再走后面的字段。

## 🔥 扩展追问 4：超级管理员（平台方）要查所有租户数据怎么办？

专门给平台运营人员提供**跳过租户过滤**的入口：

```java
// 方案1：设置特殊 tenantId = -1，拦截器识别后跳过注入
// 方案2：MyBatis-Plus 提供 TenantIgnore 注解（需要自己实现）
// 方案3：直接绕过拦截器，走原生 SQL 或独立的 DAO 层

// MyBatis-Plus 官方推荐：用 InterceptorIgnoreHelper
InterceptorIgnoreHelper.handle(IgnoreStrategy.builder().tenantLine(true).build());
try {
    // 这段查询不加 tenant_id 过滤
    return adminMapper.selectAllTenants();
} finally {
    InterceptorIgnoreHelper.clearIgnoreStrategy();
}
```

## 🔥 扩展追问 5：租户级别的数据量差异很大怎么处理？（大租户问题）

某些大客户数据量是小客户的几百倍，共享表会产生**数据倾斜**，导致这几个大租户的分区热点。

解法：
- **大租户单独分库**（混合方案）：识别出头部租户，给他们分配独立数据库，中小租户继续共享
- **分表路由**：按 `tenant_id` 取模分表，大租户分到的分片更多
- 阿里的 AHAS、PolarDB 等云数据库支持**租户级别资源隔离**，直接买就行

## 一句话收尾

> 多租户的本质是 **"共享资源 + 逻辑隔离"**，核心套路就是：**所有表加 `tenant_id` + ThreadLocal 传递上下文 + 拦截器自动注入 SQL 条件**，业务代码无感知，隔离靠框架保证。面试时加一句"异步场景用 TTL 解决透传问题"，直接拉开差距。
