## 先说结论

"不停机迁移"的核心是 **"双写 + 增量同步 + 灰度切流"**——先在阿里云搭建新环境，通过 DTS（数据传输服务）做全量+增量数据同步，业务层改造支持双写（同时写自建机房和阿里云），然后按用户维度灰度切流，最后下线旧机房。整个过程要 **"数据先行、流量渐进、随时回滚"**。

---

## 🔥 核心概念

- **解决什么问题**：机房迁移过程中业务不中断，数据零丢失，迁移后可回滚
- **核心挑战**：数据一致性、流量切换平滑、故障回滚能力
- **关键原则**：先并行运行，再逐步切流，最后下线；任何步骤都要可回滚

#### 阶段2：应用改造（双写支持）

**代码层改造**：
```java
@Component
public class DualWriteService {
    
    @Autowired
    @Qualifier("oldDataSource")
    private DataSource oldDataSource;
    
    @Autowired
    @Qualifier("newDataSource")  
    private DataSource newDataSource;
    
    @Value("${migration.dual-write.enabled:false}")
    private boolean dualWriteEnabled;
    
    @Value("${migration.dual-write.percentage:0}")
    private int dualWritePercentage;
    
    public void write(Order order) {
        // 1. 写旧库（必须成功）
        writeToOld(order);
        
        // 2. 按百分比双写新库（异步，不阻塞主流程）
        if (dualWriteEnabled && shouldDualWrite(order.getUserId())) {
            CompletableFuture.runAsync(() -> {
                try {
                    writeToNew(order);
                } catch (Exception e) {
                    // 双写失败打日志，不抛异常影响主流程
                    log.error("Dual write to new failed", e);
                    // 发 MQ 补偿
                    mqTemplate.send("dual-write-fail", order);
                }
            });
        }
    }
    
    private boolean shouldDualWrite(Long userId) {
        return userId % 100 < dualWritePercentage;
    }
}
```

**配置中心动态调整**：
```yaml
migration:
  dual-write:
    enabled: true
    percentage: 10    # 先 10% 用户双写，逐步放大
```

#### 阶段4：灰度切流（2-3周）

**切流策略（按用户维度）**：

| 阶段 | 灰度比例 | 观察期 | 检查项 |
|------|---------|--------|--------|
| 白名单 | 内部员工 | 3天 | 功能正常、性能达标 |
| 小流量 | 1% 用户 | 3天 | 错误率、RT、业务指标 |
| 中流量 | 10% 用户 | 5天 | 峰值压力、数据库负载 |
| 大流量 | 50% 用户 | 7天 | 长时间稳定性 |
| 全量 | 100% 用户 | - | 最终切换 |

**DNS/GSLB 切流配置**：
```nginx
# 按用户ID哈希分流
split_clients "${arg_user_id}" $backend {
    1%     aliyun;      # 1% 走阿里云
    10%    aliyun;      # 10% 走阿里云  
    100%   old;         # 其余走旧机房
}

upstream aliyun {
    server aliyun-lb.internal;
}

upstream old {
    server old-lb.internal;
}
```

#### 阶段5：反向同步 + 回滚能力

**双向同步（防止要回滚）**：
```
阿里云 RDS ──DTS──► 自建机房 MySQL（反向同步）
```

**回滚预案**：
```java
// 一键回滚脚本
@PostMapping("/admin/migration/rollback")
public Result rollback(@RequestParam int percentage) {
    // 1. 切 DNS 回旧机房
    dnsService.updateWeight("aliyun", percentage);
    
    // 2. 停止双写
    configService.set("migration.dual-write.enabled", "false");
    
    // 3. 数据补偿：把阿里云增量数据同步回旧库
    dtsService.startReverseSync();
    
    return Result.ok();
}
```

3. 旧数据库设为只读，保留 30 天

## 🔥 扩展追问 1：数据同步延迟怎么处理？

**DTS 延迟监控**：
```yaml
# 告警规则
- alert: DTSLatencyHigh
  expr: dts_sync_latency_seconds > 1
  for: 5m
  labels:
    severity: warning
```

**延迟处理**：
- 延迟 < 1 秒：正常，继续切流
- 延迟 1~5 秒：暂停切流，排查 DTS 性能
- 延迟 > 5 秒：回滚到旧机房，修复后再切

**业务层补偿**：
```java
// 读数据时如果新库没有，回查旧库
public Order getOrder(Long orderId) {
    Order order = newOrderDao.selectById(orderId);
    if (order == null) {
        // 可能同步延迟，查旧库
        order = oldOrderDao.selectById(orderId);
        if (order != null) {
            log.warn("Order not found in new db, fallback to old: {}", orderId);
        }
    }
    return order;
}
```

## 🔥 扩展追问 2：怎么保证事务一致性？

**分布式事务问题**：
```
用户下单 → 扣库存 → 创建订单
    ↓
旧库扣库存成功，新库创建订单失败（网络抖动）
```

**解法**：

| 方案 | 实现 | 适用 |
|------|------|------|
| **最终一致性** | 本地事务 + MQ 异步同步 | 大部分业务 |
| **TCC** | Try-Confirm-Cancel 两阶段 | 强一致必须 |
| **Seata AT** | 自动代理数据源 | 简单场景，性能一般 |

**推荐：MQ 异步最终一致**
```java
@Transactional
public void createOrder(OrderRequest req) {
    // 1. 旧库本地事务
    stockService.deduct(req.getSkuId(), req.getCount());
    Order order = orderService.create(req);
    
    // 2. 发 MQ 同步到新库
    rocketMQTemplate.sendMessageInTransaction("order-topic", 
        new OrderMessage(order), 
        null
    );
}

// 新库消费消息，失败重试，最终一致
@RocketMQMessageListener(topic = "order-topic")
public class OrderConsumer implements RocketMQListener<OrderMessage> {
    public void onMessage(OrderMessage msg) {
        newOrderService.create(msg.getOrder());  // 幂等
    }
}
```

## 🔥 扩展追问 4：怎么验证新环境性能？

**影子流量（Shadow Traffic）**：
```java
// 旧环境处理请求，同时复制一份请求到新环境，不返回结果
@Aspect
@Component
public class ShadowTrafficAspect {
    
    @Around("@annotation(ShadowTraffic)")
    public Object around(ProceedingJoinPoint pjp) throws Throwable {
        Object result = pjp.proceed();  // 旧环境执行
        
        // 异步复制请求到新环境
        CompletableFuture.runAsync(() -> {
            try {
                // 调用阿里云相同接口
                aliyunClient.call(pjp.getArgs());
            } catch (Exception e) {
                // 只记录，不影响主流程
                metrics.counter("shadow.error").increment();
            }
        });
        
        return result;
    }
}
```

**对比指标**：
- RT（响应时间）：阿里云 vs 旧机房
- 错误率：两者差异 < 0.1%
- 资源使用率：CPU、内存、数据库连接

## 🔥 扩展追问 5：迁移过程中出故障怎么办？

**故障分级处理**：

| 级别 | 场景 | 处理 |
|------|------|------|
| P0 | 阿里云完全不可用 | 立即 DNS 切回 100% 旧机房，停止双写 |
| P1 | 阿里云部分接口错误率高 | 降级：该接口切回旧机房 |
| P2 | 数据不一致 | 暂停切流，DTS 修复，校验通过后继续 |
| P3 | 性能下降 | 限流、扩容，观察后再决策 |

**应急预案文档**：
```markdown
## 阿里云故障回滚 SOP

1. 执行回滚脚本：POST /admin/migration/rollback?percentage=0
2. 确认 DNS 已切回旧机房：dig @8.8.8.8 api.example.com
3. 确认旧机房流量 100%：看监控大盘
4. 阿里云问题修复后，重新评估切流时间
5. 复盘故障原因，更新迁移方案
```
