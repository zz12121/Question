## 先说结论

实时排行榜的核心方案是 **Redis ZSet（有序集合）**——用打赏金额作为 score，用户 ID 作为 member，`ZINCRBY` 原子累加分数，`ZREVRANGE` 直接拿 Top N，天然有序、读写都是 O(log N)，每秒几万次更新完全没压力。数据库只做异步持久化，不参与实时排名计算。

---

## 🔥 核心概念

- **解决什么问题**：高频写入（每秒 N 次打赏）+ 高频读取（每秒刷新榜单展示）+ 实时排名准确，三者同时满足
- **为什么不用数据库**：`ORDER BY score DESC LIMIT 10` 在高并发写的同时执行，锁竞争严重、每次都全表扫描，根本扛不住
- **Redis ZSet 为什么合适**：底层是跳表（SkipList）+ 哈希表，写入 `ZINCRBY` O(log N)，排名查询 `ZREVRANGE` O(log N + K)，天然排序，支持原子操作

## 🔥 详细展开

### 1. 核心数据结构设计

```
Redis ZSet Key：rank:live:{room_id}:{date}
  Member：user_id（用户ID）
  Score：累计打赏金额（分，避免浮点）
```

```bash
# 用户 1001 在直播间 888 打赏 500 分
ZINCRBY rank:live:888:20260410  500  "1001"

# 查 Top 10（从高到低，带分数）
ZREVRANGE rank:live:888:20260410  0  9  WITHSCORES

# 查某个用户的排名（0-based，+1 即排名）
ZREVRANK rank:live:888:20260410  "1001"

# 查某个用户的总分
ZSCORE rank:live:888:20260410  "1001"
```

### 3. 核心代码

```java
@Service
public class RankService {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    private static final String RANK_KEY = "rank:live:%s:%s";
    
    // 打赏：原子累加分数
    public void addScore(Long roomId, Long userId, long amount) {
        String key = String.format(RANK_KEY, roomId, LocalDate.now());
        redisTemplate.opsForZSet().incrementScore(key, userId.toString(), amount);
        // key 设置过期时间，直播结束后 7 天自动清理
        redisTemplate.expire(key, 7, TimeUnit.DAYS);
    }
    
    // 获取 Top N 榜单
    public List<RankItem> getTopN(Long roomId, int n) {
        String key = String.format(RANK_KEY, roomId, LocalDate.now());
        Set<ZSetOperations.TypedTuple<String>> tuples = 
            redisTemplate.opsForZSet().reverseRangeWithScores(key, 0, n - 1);
        
        return tuples.stream().map(t -> RankItem.builder()
            .userId(Long.parseLong(t.getValue()))
            .score(t.getScore().longValue())
            .build()
        ).collect(Collectors.toList());
    }
    
    // 查某用户排名（1-based）
    public Long getUserRank(Long roomId, Long userId) {
        String key = String.format(RANK_KEY, roomId, LocalDate.now());
        Long rank = redisTemplate.opsForZSet()
            .reverseRank(key, userId.toString());
        return rank == null ? -1 : rank + 1;
    }
}
```

## 🔥 扩展追问 1：Redis 挂了怎么办？排行榜数据会丢吗？

**三层保障：**

```
① Redis 开启 AOF 持久化（appendfsync everysec）
   → 最多丢 1 秒的数据

② Redis Sentinel / Cluster 主从复制
   → 主挂了，从顶上，数据基本不丢

③ 定时任务每分钟把 Redis 榜单快照写入 DB
   → Redis 彻底挂了，从 DB 最近快照恢复，最多差 1 分钟
```

恢复流程：
```java
// 重启时从 DB 快照恢复 Redis
List<RankSnapshot> snapshots = rankSnapshotMapper.getLatest(roomId);
for (RankSnapshot s : snapshots) {
    redisTemplate.opsForZSet().add(key, s.getUserId().toString(), s.getTotalScore());
}
```

## 🔥 扩展追问 2：同分怎么处理？打赏一样多谁排前面？

ZSet 的 score 是 double 类型，当两个用户分数相同时，Redis 默认按 **member 字典序**排，并不是"先达到的排前面"。

**业务要求先到先得**，有两个方案：

**方案一：score 编码时间戳（推荐）**
```
实际 score = 打赏金额 * 10^10 + (Long.MAX_VALUE - 当前时间戳)
```
分数高的在前，同分时时间戳小（更早）的在前，一个字段搞定。

**方案二：Lua 脚本原子判断**
```lua
-- 只有新分数 > 旧分数才更新，同分保持原来的时间顺序
local old = redis.call('ZSCORE', KEYS[1], ARGV[1])
if old == false or tonumber(ARGV[2]) > tonumber(old) then
    redis.call('ZADD', KEYS[1], ARGV[2], ARGV[1])
end
```

## 🔥 扩展追问 3：直播间几万人同时刷礼物，Redis 单节点扛得住吗？

**单节点 Redis 写入 QPS 约 10 万/秒**，普通直播间完全够用。但头部直播间（百万在线）要做**分片聚合**：

```
直播间 888 → 按用户ID分3片：
  rank:live:888:shard:0  （userId % 3 == 0）
  rank:live:888:shard:1  （userId % 3 == 1）
  rank:live:888:shard:2  （userId % 3 == 2）

查排行榜时：
  ZREVRANGE 三个分片各拿 Top N
  → 业务层合并排序
  → 返回最终 Top N
```

另外可以加**本地缓存（Caffeine）**：排行榜读请求先打本地缓存（TTL 500ms），大幅降低 Redis 请求量。

## 🔥 扩展追问 4：历史总榜（跨天、跨周）怎么做？

ZSet 的 Key 设计支持多粒度：

```
rank:live:{room_id}:day:{yyyyMMdd}    → 日榜
rank:live:{room_id}:week:{yyyyWW}     → 周榜
rank:live:{room_id}:month:{yyyyMM}    → 月榜
rank:live:{room_id}:all               → 总榜
```

打赏时同时往多个 ZSet 里写：
```java
// 一次打赏，同时更新日榜 + 周榜 + 月榜 + 总榜
redisTemplate.executePipelined((RedisCallback<?>) connection -> {
    connection.zIncrBy(dayKey, amount, member);
    connection.zIncrBy(weekKey, amount, member);
    connection.zIncrBy(monthKey, amount, member);
    connection.zIncrBy(totalKey, amount, member);
    return null;
});  // Pipeline 批量执行，一次网络往返
```

> 💡 用 **Pipeline** 批量提交，4 个 ZINCRBY 合并成一次网络请求，吞吐量提升 3~4 倍。

## 🔥 扩展追问 5：前端每秒刷新，后端接口能扛住吗？

**绝对不能让前端直接轮询业务接口**，标准做法：

```
方案一：服务端 + 本地缓存
  Redis ZREVRANGE → Caffeine 本地缓存 500ms
  → 同一时刻大量请求命中本地缓存，Redis 零压力

方案二：WebSocket 服务端推送
  打赏事件 → MQ → 排行榜聚合服务 → WebSocket 推送给直播间所有在线用户
  → 前端被动接收，彻底干掉轮询

方案三：SSE（Server-Sent Events）
  比 WebSocket 轻量，单向推送，适合排行榜这种只需要服务端推的场景
```

> 💡 生产环境推荐 **WebSocket + 本地缓存双保险**：打赏变化时主动推，推送间隔限流（如最快 500ms 推一次，防止消息风暴）。
