## 先说结论

国际化低延迟的核心是 **"就近部署 + 数据分区 + 智能路由"**——在美、欧、亚三地各部署一套完整服务（单元化），用户请求通过 DNS/Anycast 路由到最近的机房，数据按用户归属地分区存储，跨地域只读不写。全球数据通过异步复制最终一致，关键业务（如支付）走全球共识强一致。

---

## 🔥 核心概念

- **解决什么问题**：全球用户访问单机房延迟高（欧美访问亚洲机房 RT > 200ms），需要就近服务
- **核心挑战**：数据一致性、跨地域复制延迟、用户归属地判定
- **关键原则**：读本地、写本地，跨地域异步同步；用户数据分区归属明确

### 3. 数据分区策略

**用户归属地判定**（注册时确定，不可变）：
```java
public enum Region {
    US(0),      // 美国 0-33
    EU(33),     // 欧洲 33-66  
    ASIA(66);   // 亚洲 66-99
    
    public static Region of(Long userId) {
        int hash = (int) (userId % 100);
        if (hash < 33) return US;
        if (hash < 66) return EU;
        return ASIA;
    }
}
```

**数据存储规则**：
| 数据类型 | 存储位置 | 复制策略 |
|---------|---------|---------|
| 用户基础信息 | 归属地域 | 全球异步复制 |
| 订单数据 | 归属地域 | 全球异步复制 |
| 商品信息 | 全球各单元 | 主从复制，写主读从 |
| 配置/字典 | 全球各单元 | 全量复制 |
| 支付流水 | 全球共识（如美国） | 强一致，同步复制 |

用户登录在美国 → 读美国 Redis 会话
用户下单在美国 → 订单写美国 RDS，异步同步到欧/亚

**跨地域访问限制**：
```java
@Service
public class OrderService {
    
    public Order createOrder(Long userId, OrderRequest req) {
        Region userRegion = Region.of(userId);
        Region currentRegion = getCurrentRegion();
        
        // 写操作必须在用户归属地
        if (!userRegion.equals(currentRegion)) {
            // 302 重定向到正确地域
            throw new RegionMismatchException(userRegion);
        }
        
        // 本地写入
        return orderDao.insert(req);
    }
    
    public Order getOrder(Long orderId) {
        // 读操作可以本地读（可能读到旧数据）
        Order order = orderDao.selectById(orderId);
        
        // 如果数据不存在，可能正在同步，查全球视图
        if (order == null) {
            order = globalOrderView.selectById(orderId);
        }
        
        return order;
    }
}
```

## 🔥 扩展追问 1：用户旅行到其他国家怎么办？

**场景**：中国用户（归属亚洲）到美国出差，打开 App

**解法**：
```
1. DNS 路由到美国机房（就近接入）
2. 美国机房识别用户归属地 = 亚洲
3. 读操作：本地读（可能有延迟）
4. 写操作：代理到亚洲机房 或 提示"当前网络下无法下单"
```

**实现**：
```java
@RestController
public class ProxyController {
    
    @PostMapping("/proxy/{region}/**")
    public ResponseEntity<?> proxy(
            @PathVariable Region region,
            HttpServletRequest request) {
        
        // 获取目标地域服务地址
        String targetUrl = regionService.getEndpoint(region);
        
        // HTTP 转发到正确地域
        return httpProxy.forward(targetUrl, request);
    }
}
```

## 🔥 扩展追问 2：全球唯一 ID 怎么生成？

**方案对比**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 雪花算法 + 地域标识 | 本地生成，无依赖 | 时钟回拨问题 |
| 号段模式（Leaf） | 趋势递增，性能好 | 需要中央服务 |
| UUID | 全球唯一，简单 | 无序，索引效率低 |

**推荐：改进版雪花算法**
```java
public class GlobalIdGenerator {
    
    // 41位时间戳 | 5位地域 | 5位机器 | 12位序列
    public long nextId(Region region) {
        long timestamp = System.currentTimeMillis() - EPOCH;
        long regionId = region.getCode();  // 0~31
        
        return (timestamp << 22) 
             | (regionId << 17) 
             | (machineId << 12) 
             | sequence;
    }
}
```

ID 本身包含地域信息，便于路由和数据归档。

## 🔥 扩展追问 3：支付等强一致业务怎么处理？

**全球共识架构**：
```
支付订单 → 统一路由到"主地域"（如美国）
         → 美国机房写本地 + 同步复制到欧/亚（Paxos/Raft）
         → 返回成功
```

**实现**：使用 TiDB / CockroachDB / Spanner 等全球分布式数据库

**性能权衡**：
- 强一致：RT 高（跨地域 RTT），用于支付、库存
- 最终一致：RT 低，用于订单查询、日志
