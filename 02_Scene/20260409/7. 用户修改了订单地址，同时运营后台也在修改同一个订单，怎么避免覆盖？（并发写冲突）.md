这和之前Q19关于ACID和Q20关于幻读的题目有联系，但这里是关于应用层面的并发控制问题。
## 先说结论

这是经典的 **并发更新冲突** 问题，解决思路有三种：

| 方案 | 原理 | 适用场景 | 复杂度 |
|-----|------|---------|-------|
| **乐观锁** | 版本号/CAS，冲突后重试 | 冲突概率低 | ⭐⭐ |
| **悲观锁** | SELECT FOR UPDATE 锁住 | 冲突概率高 | ⭐⭐ |
| **分布式锁** | Redis/ZK 互斥 | 多节点部署 | ⭐⭐⭐ |

**一句话答案**：用 **乐观锁（版本号）** 作为默认方案，配合 **分布式锁** 处理极端并发场景。

---

## 一、问题本质分析

```
用户端                              运营后台
   │                                    │
   │  ① 读取订单（version=1）           │  ① 读取订单（version=1）
   │                                    │
   │  ② 修改地址，version 1→2           │  ② 修改价格，version 1→2
   │    UPDATE SET address=新地址        │    UPDATE SET price=新价格
   │    WHERE id=xxx AND version=1      │    WHERE id=xxx AND version=1
   │                                    │
   │  ③ 更新成功（影响行数=1）           │  ④ 更新失败（version已变，影响行数=0）
   │                                    │
   │  ⑤ 订单地址已改                     │  ⑥ 提示"订单已被修改，请刷新重试"
   │                                    │
```

**问题**：用户修改的地址被运营后台的更新覆盖（丢失更新）

## 二、方案一：乐观锁（最推荐）

### 2.1 数据库层实现

```sql
-- 订单表加 version 字段
ALTER TABLE orders ADD COLUMN version INT DEFAULT 0 NOT NULL;

-- 每次更新 version + 1
UPDATE orders 
SET address = #{address},
    version = version + 1,
    updated_at = NOW()
WHERE id = #{id} 
  AND version = #{currentVersion};

-- 影响行数 = 0 时，说明有冲突
```

```java
public class Order {
```

```java
@Service
public class OrderService {
    
    @Autowired private OrderRepository orderRepository;
    
    /**
     * 用户端修改地址（乐观锁保护）
     */
    @Transactional
    public Result updateAddress(Long orderId, String newAddress, Integer clientVersion) {
        // 1. 查询当前订单
        Order order = orderRepository.findById(orderId)
            .orElseThrow(() -> new BizException("订单不存在"));
        
        // 2. 版本检查
        if (!order.getVersion().equals(clientVersion)) {
            return Result.error("订单已被其他人修改，请刷新后重试");
        }
        
        // 3. 乐观更新
        int affected = orderRepository.updateAddressOptimistic(
            orderId, newAddress, clientVersion);
        
        if (affected == 0) {
            throw new OptimisticLockException("订单已被修改，请重试");
        }
        
        return Result.success("修改成功");
    }
    
    /**
     * 运营后台修改订单（通用更新方法）
     */
    @Transactional
    public Result updateOrder(Long orderId, OrderUpdateDTO updateDTO) {
        Order order = orderRepository.findById(orderId)
            .orElseThrow(() -> new BizException("订单不存在"));
        
        // 使用 JPA 的 @Version 自动乐观锁
        // 如果 version 不匹配，会抛出 OptimisticLockingFailureException
        try {
            BeanUtils.copyProperties(updateDTO, order);
            orderRepository.save(order);  // JPA 自动检查 version
            return Result.success("修改成功");
        } catch (OptimisticLockingFailureException e) {
            return Result.error("订单已被其他人修改，请刷新后重试");
        }
    }
}

@Repository
public interface OrderRepository extends JpaRepository<Order, Long> {
    
    // 自定义乐观锁更新
    @Modifying
    @Query("UPDATE Order o SET o.address = :address, " +
           "o.version = o.version + 1, o.updatedAt = CURRENT_TIMESTAMP " +
           "WHERE o.id = :orderId AND o.version = :version")
    int updateAddressOptimistic(@Param("orderId") Long orderId,
                                @Param("address") String address,
                                @Param("version") Integer version);
}
```

## 三、方案二：分布式锁（多节点场景）

### 3.1 Redis 分布式锁实现

```java
@Service
public class DistributedLockOrderService {
    
    @Autowired private RedissonClient redisson;
    @Autowired private OrderMapper orderMapper;
    
    private static final String ORDER_LOCK_PREFIX = "order:lock:";
    private static final long LOCK_WAIT_TIME = 3;  // 等待 3 秒
    private static final long LOCK_LEASE_TIME = 10; // 锁自动释放 10 秒
    
    /**
     * 修改订单（加分布式锁）
     */
    @Transactional
    public Result updateOrder(Long orderId, OrderUpdateDTO updateDTO) {
        String lockKey = ORDER_LOCK_PREFIX + orderId;
        RLock lock = redisson.getLock(lockKey);
        
        boolean locked = false;
        try {
            // 尝试获取锁，最多等 3 秒
            locked = lock.tryLock(LOCK_WAIT_TIME, LOCK_LEASE_TIME, TimeUnit.SECONDS);
            
            if (!locked) {
                return Result.error("系统繁忙，请稍后重试");
            }
            
            // 获取锁后，再执行业务逻辑
            Order order = orderMapper.selectById(orderId);
            if (order == null) {
                return Result.error("订单不存在");
            }
            
            // 执行更新
            BeanUtils.copyProperties(updateDTO, order);
            order.setUpdatedAt(LocalDateTime.now());
            orderMapper.updateById(order);
            
            return Result.success("修改成功");
            
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return Result.error("操作被中断，请重试");
        } finally {
            // 释放锁
            if (locked && lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
    
    /**
     * 修改地址（Redis 乐观锁 + 分布式锁双重保护）
     */
    @Transactional
    public Result updateAddress(Long orderId, String newAddress, Integer version) {
        String lockKey = ORDER_LOCK_PREFIX + orderId;
        RLock lock = redisson.getLock(lockKey);
        
        try {
            lock.tryLock(LOCK_WAIT_TIME, LOCK_LEASE_TIME, TimeUnit.SECONDS);
            
            // 乐观锁更新
            int affected = orderMapper.updateAddressOptimistic(
                orderId, newAddress, version);
            
            if (affected == 0) {
                return Result.error("订单已被修改，请刷新后重试");
            }
            
            return Result.success("修改成功");
            
        } catch (InterruptedException e) {
            return Result.error("操作被中断");
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

### 3.2 分布式锁 + 版本号的完整方案

```java
/**
 * 最佳实践：分布式锁 + 乐观锁双重保护
 */
@Service
public class BestPracticeOrderService {
    
    @Autowired private RedissonClient redisson;
    @Autowired private OrderMapper orderMapper;
    @Autowired private OrderEventLogMapper eventLogMapper;
    
    private static final String LOCK_PREFIX = "order:update:";
    
    /**
     * 修改订单 - 完整版
     */
    @Transactional
    public OrderUpdateResult updateOrder(Long orderId, OrderUpdateRequest request) {
        String lockKey = LOCK_PREFIX + orderId;
        RLock lock = redisson.getLock(lockKey);
        
        try {
            // 1. 获取分布式锁
            if (!lock.tryLock(3, 10, TimeUnit.SECONDS)) {
                return OrderUpdateResult.busy();  // 系统繁忙
            }
            
            // 2. 查询当前订单状态
            Order current = orderMapper.selectById(orderId);
            if (current == null) {
                return OrderUpdateResult.notFound();
            }
            
            // 3. 检查版本号（乐观锁）
            if (!current.getVersion().equals(request.getVersion())) {
                return OrderUpdateResult.conflict(
                    "订单已被修改", 
                    current.getVersion(),      // 返回最新版本
                    current.getUpdatedAt()     // 返回修改时间
                );
            }
            
            // 4. 执行更新
            int affected = orderMapper.updateWithVersion(
                orderId,
                request.getAddress(),
                request.getPrice(),
                current.getVersion() + 1
            );
            
            if (affected == 0) {
                return OrderUpdateResult.conflict("更新失败，请重试");
            }
            
            // 5. 记录操作日志（方便审计）
            OrderEventLog log = new OrderEventLog();
            log.setOrderId(orderId);
            log.setOperatorType(request.getOperatorType());  // USER / ADMIN
            log.setOperatorId(request.getOperatorId());
            log.setBeforeValue(JSON.toJSONString(current));
            log.setAfterValue(JSON.toJSONString(request));
            log.setCreateTime(LocalDateTime.now());
            eventLogMapper.insert(log);
            
            return OrderUpdateResult.success();
            
        } catch (InterruptedException e) {
            return OrderUpdateResult.error("操作被中断");
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

## 四、方案三：消息队列（最终一致）

```java
@Service
public class OrderEventSource {
    
    @Autowired private RocketMQTemplate rocketMQ;
    
    /**
     * 发布订单更新事件
     */
    public void publishUpdateEvent(OrderUpdateEvent event) {
        // 消息幂等键：orderId + version
        String idempotentKey = event.getOrderId() + ":" + event.getVersion();
        
        rocketMQ.asyncSend("order-update-topic", event, new SendCallback() {
            @Override
            public void onSuccess(SendResult result) {
                log.info("[订单更新消息发送成功] key={}", idempotentKey);
            }
            
            @Override
            public void onException(Throwable e) {
                log.error("[订单更新消息发送失败] key={}", idempotentKey, e);
                // 发送失败，需要补偿或告警
            }
        });
    }
}

@Service
public class OrderEventConsumer {
    
    @Autowired private OrderMapper orderMapper;
    
    @RocketMQMessageListener(
        topic = "order-update-topic",
        consumerGroup = "order-update-consumer",
        concurrency = "5"
    )
    public void handleOrderUpdate(OrderUpdateEvent event) {
        String idempotentKey = event.getOrderId() + ":" + event.getVersion();
        
        // 1. 幂等检查
        if (isProcessed(idempotentKey)) {
            log.info("[重复消息跳过] key={}", idempotentKey);
            return;
        }
        
        // 2. 版本号校验（乐观锁）
        int affected = orderMapper.updateWithVersion(
            event.getOrderId(),
            event.getNewAddress(),
            event.getVersion()
        );
        
        if (affected > 0) {
            // 3. 标记已处理
            markProcessed(idempotentKey);
        }
    }
}
```

## 五、运营后台特殊处理

```java
/**
 * 运营后台专属：强制覆盖模式
 * 适用于：客服紧急修改、异常数据修复
 */
@Service
public class AdminOrderService {
    
    @Autowired private OrderMapper orderMapper;
    @Autowired private RedissonClient redisson;
    
    /**
     * 强制更新（管理员权限）
     * 会覆盖其他人的修改，需要记录操作日志
     */
    @Transactional
    public Result forceUpdate(Long orderId, OrderUpdateDTO updateDTO, Long adminId) {
        String lockKey = "order:admin:lock:" + orderId;
        RLock lock = redisson.getFairLock(lockKey);  // 公平锁，按顺序获取
        
        try {
            lock.lock(30, TimeUnit.SECONDS);
            
            // 1. 查询最新订单
            Order order = orderMapper.selectById(orderId);
            
            // 2. 强制更新（不检查版本）
            OrderUpdateDTO forced = new OrderUpdateDTO();
            BeanUtils.copyProperties(updateDTO, forced);
            forced.setVersion(order.getVersion() + 1);  // 版本号仍然递增
            
            orderMapper.updateById(forced);
            
            // 3. 记录强制覆盖日志
            logOrderOverride(orderId, order, updateDTO, adminId);
            
            return Result.success("强制更新成功");
            
        } finally {
            lock.unlock();
        }
    }
    
    private void logOrderOverride(Long orderId, Order oldOrder, 
                                  OrderUpdateDTO newDTO, Long adminId) {
        // 记录谁在什么时候覆盖了什么
        log.warn("[订单强制覆盖] orderId={}, adminId={}, " +
                 "oldPrice={}, newPrice={}, oldAddress={}, newAddress={}",
            orderId, adminId,
            oldOrder.getPrice(), newDTO.getPrice(),
            oldOrder.getAddress(), newDTO.getAddress());
    }
}
```

| 维度 | 乐观锁 | 悲观锁 | 分布式锁 | 消息队列 |
|-----|-------|-------|---------|---------|
| **并发能力** | 高 | 低 | 中 | 高 |
| **实现复杂度** | 低 | 低 | 中 | 高 |
| **用户体验** | 需前端配合重试 | 无感知 | 无感知 | 无感知 |
| **适用场景** | 读多写少 | 写多读少 | 多节点 | 异步解耦 |
| **冲突处理** | 提示重试 | 排队等待 | 提示重试 | 自动重试 |

## 面试加分话术

> "这个问题的核心是 **并发更新的数据一致性**，我的方案是 **'乐观锁 + 分布式锁双重保障'**。
> 
> 日常场景用乐观锁就够了：
> - 前端每次请求带上 `version` 字段
> - 后端更新时 `WHERE version = #{version}`，影响行数为 0 就说明冲突
> - 冲突了返回给前端，让用户决定是刷新还是合并
> 
> 极端场景（比如大促期间）用分布式锁兜底，防止并发击穿。
> 
> 之前我们线上就碰到过用户端和运营后台同时改地址的问题，后来加了版本号 + 冲突提示，投诉率直接降了 80%。"

### Q7-1: 如何实现自动合并冲突？

```java
/**
 * 字段级别的冲突合并
 */
@Service
public class ConflictMergeService {
    
    /**
     * 智能合并冲突
     * 规则：不同字段可以合并，相同字段需要人工处理
     */
    public MergeResult merge(Order localOrder, Order serverOrder) {
        Order merged = new Order();
        merged.setId(localOrder.getId());
        merged.setVersion(serverOrder.getVersion());
        
        List<String> conflicts = new ArrayList<>();
        
        // 地址：取最新的
        merged.setAddress(localOrder.getAddress());
        if (!localOrder.getAddress().equals(serverOrder.getAddress())) {
            merged.setAddress(localOrder.getAddress());  // 以本地为准
            // 也可以选择以服务器为准，或记录冲突
        }
        
        // 价格：不同字段冲突，需要人工处理
        if (!localOrder.getPrice().equals(serverOrder.getPrice())) {
            conflicts.add("价格");
            merged.setPrice(null);  // 标记冲突，等待人工处理
        }
        
        // 备注：可以合并
        if (StringUtils.isNotBlank(localOrder.getRemark())) {
            merged.setRemark(localOrder.getRemark());
        }
        
        return new MergeResult(merged, conflicts);
    }
}
```
