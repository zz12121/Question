用户问的是微服务级联故障的问题：A→B→C三级调用，C变慢导致B也慢，最终A的线程池被打满。这是一个经典的"级联故障"或"雪崩"问题。

6. 监控告警

# 15. 微服务级联故障：从线程池打满到雪崩的全链路防护

## 结论先行

**雪崩链条：C 慢 → B 等 C → B 线程阻塞 → A 等 B → A 线程池打满 → 全链路超时 → 系统崩溃。**

**防护核心：超时熔断 + 限流降级 + 舱壁隔离，三管齐下。**

```

│                        级联故障雪崩链                                │
│                                                                     │
│  T=0    C 服务变慢（数据库慢、GC、外部依赖超时）                      │
│         ┌──────────────────────────────────────────────────────┐   │
│         │  C 服务正常 RT: 50ms → 变慢 RT: 5000ms              │   │
│         └──────────────────────────────────────────────────────┘   │
│                               │                                      │
│                               ▼                                      │
│  T=1    B 服务调用 C，设置超时 3s                                   │
│         ┌──────────────────────────────────────────────────────┐   │
│         │  B 的线程池: 50 个                                   │   │
│         │  → 10% 的请求卡在等待 C (5个线程)                     │   │
│         │  → 30% 的请求卡在等待 C (15个线程)                   │   │
│         │  → 50% 的请求卡在等待 C (25个线程) ← 危险！          │   │
│         └──────────────────────────────────────────────────────┘   │
│                               │                                      │
│                               ▼                                      │
│  T=3    B 服务线程池开始耗尽                                        │
│         ┌──────────────────────────────────────────────────────┐   │
│         │  B 的线程池: 50 个                                   │   │
│         │  → 100% 线程都在等 C 返回                             │   │
│         │  → 新请求进来 → 无可用线程 → 拒绝                     │   │
│         └──────────────────────────────────────────────────────┘   │
│                               │                                      │
│                               ▼                                      │
│  T=5    A 服务调用 B，同样开始堆积                                  │
│         ┌──────────────────────────────────────────────────────┐   │
│         │  A 的线程池: 100 个                                   │   │
│         │  → 全部卡在等 B → 线程池耗尽                          │   │
│         │  → 新请求进来 → 直接拒绝                              │   │
│         └──────────────────────────────────────────────────────┘   │
│                               │                                      │
│                               ▼                                      │
│  T=10   全链路超时 → 用户请求全部失败 → 服务不可用                   │
│                                                                     │
│  ⚠️ 从 C 变慢到系统崩溃：可能只需要 5~30 秒！                        │
```

### 2. Sentinel 熔断器（核心）

```java
/**
 * Sentinel 熔断 + 降级
 */
@Service
@Slf4j
public class ServiceBCallingC {

    @Autowired
    private ServiceCClient serviceCClient;

    /**
     * 调用 C 服务，带熔断保护
     */
    @SentinelResource(
        value = "serviceC:getData",
        blockHandler = "getDataBlockHandler",    // 熔断触发时调用
        fallback = "getDataFallback"              // 业务异常时调用
    )
    public String getDataFromC(Long id) {
        return serviceCClient.getData(id);
    }

    /**
     * 熔断降级：返回兜底数据
     */
    public String getDataBlockHandler(Long id, BlockException e) {
        log.warn("触发熔断，降级返回, id={}, error={}", id, e.getClass().getSimpleName());
        // 返回兜底数据
        return getDefaultData(id);
    }

    /**
     * 业务异常降级
     */
    public String getDataFallback(Long id, Throwable t) {
        log.error("C 服务调用失败，降级处理, id={}", id, t);
        return getDefaultData(id);
    }

    private String getDefaultData(Long id) {
        // 返回默认数据 / 缓存数据 / 空数据
        return "{\"code\": -1, \"msg\": \"服务繁忙，请稍后重试\"}";
    }
}

/**
 * Sentinel 熔断规则配置
 */
@Configuration
public class Sentinel熔断规则 {

    @PostConstruct
    public void init熔断规则() {
        // 慢调用比例熔断
        List<DegradeRule> rules = Collections.singletonList(
            new DegradeRule("serviceC:getData")
                .setGrade(CircuitBreakerStrategy.SLOW_RATIO_GRADE.getType())  // 慢调用比例
                .setCount(2.0)                      // RT 超过 2s 算慢调用
                .setSlowRatioThreshold(0.5)        // 50% 慢调用触发熔断
                .setMinRequestAmount(5)            // 最少 5 个请求统计
                .setStatIntervalMs(10000)          // 10 秒内统计
                .setTimeWindow(10)                 // 熔断 10 秒
        );

        // 异常比例熔断
        List<DegradeRule> errorRules = Collections.singletonList(
            new DegradeRule("serviceC:getData")
                .setGrade(CircuitBreakerStrategy.ERROR_RATIO_GRADE.getType())
                .setCount(0.5)                     // 50% 异常比例
                .setMinRequestAmount(5)
                .setStatIntervalMs(10000)
                .setTimeWindow(30)                 // 熔断 30 秒
        );

        DegradeRuleManager.loadRules(rules);
    }
}
```

```
│  │   C3 慢 → 只影响自己的 10 个线程，其他业务不受影响          │  │
│  └─────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────┘
```

```java
/**
 * 线程池隔离（Resilience4j）
 */
@Configuration
public class ThreadPoolIsolation {

    /**
     * C3 服务的专属线程池（慢服务隔离）
     */
    @Bean("c3ThreadPool")
    public ThreadPoolExecutor c3ThreadPool() {
        return new ThreadPoolExecutor(
            10,                          // 核心线程
            10,                          // 最大线程
            60, TimeUnit.SECONDS,         // 空闲回收
            new LinkedBlockingQueue<>(100),  // 队列大小
            new ThreadFactoryBuilder()
                .setNameFormat("c3-pool-%d")
                .build(),
            // 拒绝策略：队列满了直接拒绝，不阻塞主线程
            new ThreadPoolExecutor.AbortPolicy()
        );
    }
}

/**
 * 使用隔离线程池调用慢服务
 */
@Service
@Slf4j
public class ServiceBCallingC3 {

    @Autowired
    @Qualifier("c3ThreadPool")
    private ThreadPoolExecutor c3ThreadPool;

    @Autowired
    private ServiceC3Client serviceC3Client;

    /**
     * 同步调用：提交到隔离线程池
     */
    public String getDataFromC3(Long id) {
        try {
            // 提交到隔离线程池
            Future<String> future = c3ThreadPool.submit(
                () -> serviceC3Client.getData(id)
            );

            // 最多等待 2 秒
            return future.get(2, TimeUnit.SECONDS);

        } catch (TimeoutException e) {
            log.warn("C3 服务超时，降级处理, id={}", id);
            return getDefaultData(id);
        } catch (Exception e) {
            log.error("C3 服务调用失败", e);
            return getDefaultData(id);
        }
    }

    /**
     * 异步调用：立即返回 Future，不阻塞
     */
    public CompletableFuture<String> getDataFromC3Async(Long id) {
        return CompletableFuture.supplyAsync(
            () -> serviceC3Client.getData(id),
            c3ThreadPool
        ).orTimeout(2, TimeUnit.SECONDS)
         .exceptionally(ex -> {
             log.warn("C3 服务异常, id={}", id);
             return getDefaultData(id);
         });
    }
}
```

### 4. 信号量隔离（Sentinel 默认）

```java
/**
 * 信号量隔离（轻量级，不创建新线程）
 */
@Service
@Slf4j
public class SemaphoreIsolation {

    // 信号量：最多 20 个并发调用
    private final Semaphore semaphore = new Semaphore(20, true);

    public String callServiceC(Long id) {
        // 获取信号量许可
        if (!semaphore.tryAcquire(2, TimeUnit.SECONDS)) {
            log.warn("信号量获取失败，C 服务限流, id={}", id);
            return getDefaultData(id);
        }

        try {
            return serviceCClient.getData(id);
        } catch (Exception e) {
            log.error("C 服务调用失败", e);
            return getDefaultData(id);
        } finally {
            semaphore.release();
        }
    }

    /**
     * Sentinel 信号量隔离（更推荐）
     */
    @SentinelResource(
        value = "semaphore:serviceC",
        blockHandler = "semaphoreBlockHandler"
    )
    public String callServiceCWithSentinel(Long id) {
        return serviceCClient.getData(id);
    }
}
```

## 五、重试策略（谨慎使用）

```java
/**
 * 重试策略（⚠️ 注意：重试可能加剧雪崩，谨慎使用）
 */
@Configuration
public class RetryConfig {

    @Bean
    public RetryTemplate retryTemplate() {
        SimpleRetryPolicy retryPolicy = new SimpleRetryPolicy();
        retryPolicy.setMaxAttempts(2);  // 最多重试 2 次（别太多！）

        // 只在特定异常时重试
        Map<Class<? extends Throwable>, Boolean> retryableExceptions = new HashMap<>();
        retryableExceptions.put(ConnectTimeoutException.class, true);
        retryableExceptions.put(SocketTimeoutException.class, true);
        retryableExceptions.put(ServiceUnavailableException.class, true);  // 5xx 才重试
        retryableExceptions.put(ClientException.class, false);  // 4xx 不重试

        retryPolicy.setRetryableExceptions(retryableExceptions);

        ExponentialBackOffPolicy backOffPolicy = new ExponentialBackOffPolicy();
        backOffPolicy.setInitialInterval(100);   // 初始 100ms
        backOffPolicy.setMultiplier(2);          // 2 倍退避
        backOffPolicy.setMaxInterval(1000);       // 最多 1s

        RetryTemplate template = new RetryTemplate();
        template.setRetryPolicy(retryPolicy);
        template.setBackOffPolicy(backOffPolicy);

        return template;
    }
}
```

| 防护手段 | 作用 | 性能影响 | 配置难度 | 适用场景 |
|---------|------|---------|---------|---------|

| **重试** | 临时故障兜底 | 中 | 中 | 网络抖动 |

## 八、面试加分话术

> **面试官问：重试会不会加剧雪崩？**
>
> 会！所以重试要谨慎：
> 1. **设置最大重试次数**（最多 2~3 次，别无限重试）
> 2. **指数退避**（100ms → 200ms → 400ms，避免集中重试）
> 3. **只对临时故障重试**（超时、网络抖动），业务异常不重试
> 4. **熔断时禁止重试**（都熔断了还重试，只会加重负担）

> **追问：线程池隔离和信号量隔离怎么选？**
>
> 看场景：
> - **线程池隔离**：下游调用耗时长（如 5s+），需要异步并行，用线程池
> - **信号量隔离**：下游调用快（如 < 500ms），用信号量（开销小）

> **再问：服务 B 重启时，积压的请求怎么处理？**
>
> 三个方向：
> 1. **启动预热**：启动初期限流小流量，逐步放开（Sentinel warmUp）
> 2. **请求缓存**：MQ 缓存积压请求，缓慢消费
> 3. **优雅关闭**：关闭前不再接收新请求，等待积压处理完再停（`@PreDestroy`）
