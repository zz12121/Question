1. 可能的原因：
- 数据库问题：慢查询、锁等待、连接池耗尽
- 网络问题：网络抖动、延迟增加
- 外部依赖：下游服务变慢、第三方接口超时
- GC问题：虽然内存正常，但可能有频繁的GC暂停
- Redis/缓存问题：缓存失效、击穿
- 线程池问题：线程池耗尽、队列积压
- 磁盘IO问题
- 分布式锁问题：锁等待
- 数据库连接池配置问题

2. 排查步骤：
- 查看日志
- 查看数据库慢查询
- 查看线程池状态
- 查看数据库连接
- 查看GC日志
- 查看网络延迟
- 查看下游服务状态
- 查看分布式追踪
## 先说结论

CPU 和内存正常，但响应时间暴涨，说明 **瓶颈不在计算资源**，而在 **等待资源**：数据库锁、连接池、外部依赖、网络。

| 排查方向 | 可能原因 | 排查优先级 |
|---------|---------|-----------|
| **数据库** | 慢查询、锁等待、连接池耗尽 | ⭐⭐⭐⭐⭐ |
| **外部依赖** | 下游服务超时、第三方接口慢 | ⭐⭐⭐⭐⭐ |
| **GC 暂停** | STW 导致线程暂停 | ⭐⭐⭐⭐ |
| **网络** | 网络抖动、延迟增加 | ⭐⭐⭐ |
| **线程池** | 线程耗尽、队列积压 | ⭐⭐⭐ |

**一句话答案**：先看日志定位慢在哪一段，再查数据库慢查询、外部依赖耗时、GC 暂停、连接池状态，四个方向同时排查。

### 2.2 第二步：数据库排查（最高优先级）

```sql
-- 1. 查看当前正在执行的慢查询
SELECT 
    id,
    user,
    host,
    db,
    command,
    time,
    state,
    LEFT(info, 100) as query
FROM information_schema.processlist
WHERE command != 'Sleep'
  AND time > 5  -- 执行超过 5 秒
ORDER BY time DESC;

-- 2. 查看锁等待情况
SELECT 
    trx_id,
    trx_state,
    trx_started,
    trx_rows_locked,
    trx_rows_modified,
    LEFT(trx_query, 50) as query,
    lp.lock_index,
    lp.lock_type
FROM information_schema.innodb_trx trx
JOIN information_schema.innodb_lock_waits lw ON trx.trx_id = lw.blocking_trx_id
JOIN information_schema.innodb_lock_pages lp ON lw.lock_id = lp.lock_id;

-- 3. 查看锁等待
SELECT 
    r.trx_mysql_thread_id AS waiting_thread,
    r.trx_query AS waiting_query,
    b.trx_query AS blocking_query
FROM information_schema.innodb_lock_waits w
JOIN information_schema.innodb_trx b ON b.trx_id = w.blocking_trx_id
JOIN information_schema.innodb_trx r ON r.trx_id = w.requesting_trx_id;

-- 4. 查看表锁情况
SHOW STATUS LIKE 'Table_locks%';

-- 5. 查看慢查询日志（MySQL 配置 long_query_time = 1）
SHOW VARIABLES LIKE 'slow_query_log%';
SHOW VARIABLES LIKE 'long_query_time';

-- 6. 分析慢查询日志
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log
```

```sql
-- 7. 查看索引使用情况
SELECT 
    t.table_name,
    i.index_name,
    i.cardinality,
    s.selectivity,
    s.avg_relative_selectivity
FROM information_schema.statistics i
JOIN information_schema.tables t ON i.table_schema = t.table_schema 
    AND i.table_name = t.table_name
JOIN mysql.index_statistics s ON i.table_name = s.table_name 
    AND i.index_name = s.index_name
WHERE t.table_schema = 'your_database'
ORDER BY s.avg_relative_selectivity DESC;

-- 8. 查看表大小和数据量变化
SELECT 
    table_name,
    table_rows,
    data_length / 1024 / 1024 AS data_mb,
    index_length / 1024 / 1024 AS index_mb
FROM information_schema.tables
WHERE table_schema = 'your_database'
ORDER BY data_length DESC;
```

```java
@RestController
public class PoolMonitor {
    
    @Autowired private DataSource dataSource;

    public Map<String, Object> getDruidStats() {
        Map<String, Object> stats = new HashMap<>();

        if (dataSource instanceof DruidDataSource) {
            DruidDataSource druid = (DruidDataSource) dataSource;
            stats.put("activeCount", druid.getActiveCount());
            stats.put("poolingCount", druid.getPoolingCount());
            stats.put("waitThreadCount", druid.getWaitThreadCount());
            stats.put("errorCount", druid.getErrorCount());
            stats.put("maxActive", druid.getMaxActive());
        }
        
        // HikariCP
        if (dataSource instanceof HikariDataSource) {
            HikariDataSource hikari = (HikariDataSource) dataSource;
            stats.put("activeConnections", hikari.getHikariPoolMXBean().getActiveConnections());
            stats.put("idleConnections", hikari.getHikariPoolMXBean().getIdleConnections());
            stats.put("totalConnections", hikari.getHikariPoolMXBean().getTotalConnections());
            stats.put("threadsAwaitingConnection", hikari.getHikariPoolMXBean().getThreadsAwaitingConnection());
        }
        
        return stats;
    }
}
```

```java
@Component
public class ExternalCallMonitor implements IRemoteService {
    
    @Autowired private RestTemplate restTemplate;
    
    @Autowired private ObjectMapper objectMapper;
    
    public String callExternalApi(String url) {
        long start = System.currentTimeMillis();
        
        try {
            // 添加详细日志
            log.info("[外部调用开始] url={}", url);
            
            ResponseEntity<String> response = restTemplate.getForEntity(url, String.class);
            
            long cost = System.currentTimeMillis() - start;
            log.info("[外部调用结束] url={}, cost={}ms, status={}", 
                url, cost, response.getStatusCode());
            
            // 超时告警
            if (cost > 1000) {
                log.warn("[外部调用慢] url={}, cost={}ms", url, cost);
                metricsService.recordExternalCallSlow(url, cost);
            }
            
            return response.getBody();
        } catch (Exception e) {
            long cost = System.currentTimeMillis() - start;
            log.error("[外部调用异常] url={}, cost={}ms, error={}", 
                url, cost, e.getMessage());
            throw e;
        }
    }
}
```

```java
// 添加线程池监控
@Configuration
public class ThreadPoolMonitorConfig {
    
    @Bean
    public ThreadPoolExecutor businessExecutor() {
        ThreadPoolExecutor executor = new ThreadPoolExecutor(
            20, 50, 60L, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(1000),
            new ThreadFactoryBuilder().setNameFormat("biz-%d").build(),
            (r, e) -> {
                // 拒绝策略：记录告警
                log.error("[线程池拒绝] active={}, queue={}", 
                    e.getActiveCount(), e.getQueue().size());
                metricsService.recordThreadPoolReject("business");
            }
        );
        
        // 添加监控
        return new ThreadPoolExecutor(
            executor.getCorePoolSize(),
            executor.getMaximumPoolSize(),
            executor.getKeepAliveTime(TimeUnit.SECONDS),
            TimeUnit.SECONDS,
            executor.getQueue(),
            executor.getThreadFactory(),
            executor.getRejectedExecutionHandler()
        ) {
            @Override
            public void execute(Runnable command) {
                long start = System.currentTimeMillis();
                super.execute(() -> {
                    long waitTime = System.currentTimeMillis() - start;
                    if (waitTime > 1000) {
                        log.warn("[线程池等待慢] waitTime={}ms, queueSize={}", 
                            waitTime, this.getQueue().size());
                    }
                    command.run();
                });
            }
        };
    }
}
```

## 四、常见场景对照

| 场景 | 特征 | 快速判断 |
|-----|------|---------|
| **数据库慢查询** | 响应时间稳定在 5s | SHOW PROCESSLIST 有长查询 |
| **数据库锁等待** | 偶发超时，有时正常 | SHOW INNODB STATUS 有锁等待 |
| **连接池耗尽** | 大流量时变慢 | activeCount == maxActive |
| **外部服务超时** | 响应时间阶梯式 | 日志中有 HTTP 超时 |
| **GC 停顿** | 规律性变慢（每分钟） | GC 日志有 Full GC |
| **网络抖动** | 偶发变慢 | Ping 延迟增加 |
| **Redis 慢查询** | 偶发 | slowlog 有 KEYS 命令 |

## 五、面试加分话术

> "这个问题我遇到过很多次，排查思路是 **'先定位慢在哪，再找为什么'**。
> 
> 首先看日志，如果大量请求同时超时，基本就是 **数据库或外部依赖** 的问题。
> 
> 如果只是单个接口变慢，我会分几步：
> 1. **数据库层**：先查 `SHOW PROCESSLIST`，看有没有锁等待或慢查询
> 2. **连接池层**：看连接池是否耗尽，有没有排队
> 3. **GC 层**：用 jstat 看有没有 Full GC
> 4. **依赖层**：看日志里第三方调用的耗时
> 
> 之前碰到过一次，是 **定时任务和业务接口抢连接池**，导致业务高峰期连接池被打爆。后来把定时任务改成低峰期执行就好了。"

### Q6-1: 如何预防这类问题？

```java
// 1. 完善监控告警
@Configuration
public class MetricsConfig {
    
    @Bean
    public MeterRegistry meterRegistry() {
        return new PrometheusMeterRegistry(PrometheusConfig.DEFAULT);
    }
    
    // 接口响应时间监控
    @Aspect
    @Component
    public class ResponseTimeAspect {
        
        @Autowired private MeterRegistry registry;
        
        @Around("execution(* com.xxx.controller..*.*(..))")
        public Object recordResponseTime(ProceedingJoinPoint point) throws Throwable {
            Timer.Sample sample = Timer.start(registry);
            try {
                return point.proceed();
            } finally {
                String uri = getUri(point);
                Timer timer = Timer.builder("http_server_requests_seconds")
                    .tag("uri", uri)
                    .register(registry);
                sample.stop(timer);
            }
        }
    }
}

// 2. 慢查询自动告警
@Scheduled(fixedRate = 60000)
public void checkSlowQueries() {
    List<Map<String, Object>> slowQueries = jdbcTemplate.queryForList(
        "SHOW PROCESSLIST WHERE Time > 10");
    
    if (!slowQueries.isEmpty()) {
        alertService.sendAlert("数据库慢查询", slowQueries);
    }
}
```

**一句话收尾**：接口响应暴涨但 CPU/内存正常，瓶颈一定在 **等待资源** 上——90% 的情况是数据库（慢查询/锁/连接池），剩下 10% 是外部依赖和 GC。按照 **日志 → 数据库 → 连接池 → GC → 外部依赖** 的顺序排查，5 分钟内一定能定位到问题。
