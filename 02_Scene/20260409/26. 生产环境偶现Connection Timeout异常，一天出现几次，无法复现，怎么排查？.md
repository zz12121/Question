## 先说结论

偶现的 Connection Timeout 是典型的 **"长尾问题"**，最可能的原因是 **"连接池配置不当"**（连接超时时间 < 数据库/下游服务实际响应时间）或 **"连接泄漏"**（拿了连接没归还）。排查思路是 **"先完善监控埋点，再分析时间分布，最后缩小范围定位"**：给超时异常加上下文日志（当时连接池状态、SQL/URL、耗时分布），用 Prometheus 看 P99 延迟，结合连接池监控判断是偶发慢查询还是连接池耗尽。

---

## 🔥 核心概念

- **解决什么问题**：偶现、无法复现的连接超时，需要系统化监控和分析找到根因
- **关键线索**：超时发生的时机、当时系统负载、连接池状态、下游服务状态
- **排查原则**：偶现问题靠监控和数据，不是靠复现

## 🔥 详细展开

### 1. 常见原因分类

| 原因 | 表现 | 排查方向 |
|------|------|----------|
| **连接池配置不合理** | 超时时间设太短（如 1s），偶发慢查询触发 | 对比 `connection-timeout` 和 SQL P99 延迟 |
| **连接泄漏** | 连接拿了没 close，池慢慢耗尽 | 监控 active 连接数只增不减 |
| **下游服务偶发慢** | 下游依赖抖动，RT 偶尔飙高 | 链路追踪看下游 P99 |
| **数据库偶发慢** | 慢 SQL、锁等待、刷盘 | 数据库慢查询日志 |
| **网络抖动** | 跨可用区、跨机房调用 | 网络监控、TCP 重传率 |
| **GC 停顿** | STW 期间连接等待超时 | GC 日志看停顿时间 |

### 2. 排查步骤（系统化）

#### 第一步：给异常加上下文日志

**改造前**（无用）：
```
java.sql.SQLTimeoutException: Connection timeout
```

**改造后**（有上下文）：
```
Connection timeout after 3000ms
  - Pool: HikariPool-1 (active=10, idle=0, waiting=5, total=10)
  - SQL: SELECT * FROM orders WHERE user_id = ?  (未执行)
  - Thread: http-nio-8080-exec-25
  - Time: 2024-01-15 14:23:15.123
  - Current QPS: 150, Avg RT: 50ms, P99 RT: 200ms
```

**实现方式**：
```java
@Component
public class ConnectionTimeoutLogger {
    
    @Autowired
    private HikariDataSource dataSource;
    
    @Autowired
    private MeterRegistry registry;
    
    public void logTimeoutException(SQLException e, String sql) {
        HikariPoolMXBean poolMXBean = dataSource.getHikariPoolMXBean();
        
        log.error("Connection timeout | pool={} | active={} | idle={} | waiting={} | total={} | sql={} | thread={}",
            dataSource.getPoolName(),
            poolMXBean.getActiveConnections(),
            poolMXBean.getIdleConnections(),
            poolMXBean.getThreadsAwaitingConnection(),
            dataSource.getMaximumPoolSize(),
            sql,
            Thread.currentThread().getName()
        );
        
        // 上报监控
        registry.counter("db.connection.timeout", 
            "pool", dataSource.getPoolName(),
            "sql_hash", HashUtil.hash(sql)
        ).increment();
    }
}
```

---

#### 第二步：看连接池监控趋势

**关键指标**：
```
hikaricp_connections_active     # 活跃连接数
hikaricp_connections_idle       # 空闲连接数
hikaricp_connections_pending    # 等待连接的线程数（>0 说明池耗尽）
hikaricp_connections_timeout    # 超时次数
```

**判断场景**：

| 场景 | active | idle | pending | 结论 |
|------|--------|------|---------|------|
| 连接池耗尽 | ≈ max | 0 | > 0 | 连接泄漏或池太小 |
| 偶发慢查询 | 波动 | 有 | 偶尔 > 0 | 个别 SQL 慢，池够用 |
| 正常 | < max | > 0 | 0 | 健康状态 |

#### 第三步：分析超时时间分布

```sql
-- 假设你把超时日志存了 Elasticsearch
SELECT 
  histogram(@timestamp, '1h') as hour,
  count(*) as timeout_count,
  avg(wait_time_ms) as avg_wait
FROM logs 
WHERE message LIKE 'Connection timeout%'
GROUP BY hour
ORDER BY hour
```

**如果发现集中在某几个小时** → 找那个时间段的系统变更、定时任务、业务高峰

#### 第四步：链路追踪看下游

```java
// 在连接获取处加 Span
@Around("execution(* javax.sql.DataSource.getConnection(..))")
public Object traceConnection(ProceedingJoinPoint pjp) throws Throwable {
    Span span = tracer.nextSpan().name("getConnection").start();
    try (Tracer.SpanInScope ws = tracer.withSpanInScope(span)) {
        return pjp.proceed();
    } catch (Exception e) {
        span.error(e);
        throw e;
    } finally {
        span.end();
    }
}
```

**SkyWalking 看调用链**：
```
你的接口 (3500ms)  ← 超时
  ├── getConnection (3000ms)  ← 等连接花了 3 秒
  └── SQL执行 (0ms)           ← 没执行到
```

### 3. 典型案例分析

**案例一：连接池配置太小 + 偶发慢查询**

```yaml
# ❌ 配置
spring:
  datasource:
    hikari:
      maximum-pool-size: 5
      connection-timeout: 1000   # 1秒太短

# 现象：平时正常，偶发超时
# 原因：某个 SQL 偶尔慢（如 800ms），占住连接，其他请求等 1 秒就超时
```

**解法**：
```yaml
# ✅ 调大池子和超时时间
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      connection-timeout: 5000   # 5秒，覆盖 P99 延迟
      leak-detection-threshold: 60000  # 连接泄漏检测，60秒没归还打日志
```

---

**案例二：连接泄漏**

```java
// ❌ 泄漏：连接没 close
public List<Order> getOrders() {
    Connection conn = dataSource.getConnection();
    // ... 查询
    return orders;  // 异常时没走到 close
    // conn.close() 没执行
}

// ✅ 正确：try-with-resources
public List<Order> getOrders() {
    try (Connection conn = dataSource.getConnection();
         PreparedStatement ps = conn.prepareStatement(sql)) {
        // ... 查询
        return orders;
    }  // 自动 close
}
```

**症状**：`active` 连接数持续增长，直到 max，然后大量 timeout

---

**案例三：下游 HTTP 调用阻塞导致连接占用**

```java
// ❌ 在事务里调外部 HTTP，连接被占用
@Transactional
public void process() {
    Order order = orderMapper.selectById(1);  // 拿连接
    // 连接保持中...
    httpClient.callSlowApi();  // 阻塞 5 秒
    orderMapper.update(order);   // 还在用这个连接
}  // 才释放连接
```

**解法**：事务最小化，HTTP 调用放事务外
```java
public void process() {
    Order order = orderService.getOrder(1);  // 事务内只查
    
    // 事务外调 HTTP
    ApiResult result = httpClient.callSlowApi();
    
    orderService.update(order, result);  // 新事务更新
}
```

## 🔥 扩展追问 1：怎么判断是连接池问题还是数据库问题？

**对比测试**：
```bash
# 直接用 mysql 客户端连，执行相同 SQL
time mysql -h db-host -e "SELECT * FROM orders WHERE user_id = 123"

# 如果客户端也慢 → 数据库问题
# 如果客户端快，只有应用慢 → 连接池或网络问题
```

**看数据库侧**：
```sql
-- MySQL 看当前连接和正在执行的 SQL
SHOW PROCESSLIST;
SELECT * FROM information_schema.PROCESSLIST WHERE TIME > 1;

-- 看慢查询日志
SELECT * FROM mysql.slow_log WHERE start_time > NOW() - INTERVAL 1 HOUR;
```

## 🔥 扩展追问 2：连接泄漏怎么快速发现？

**HikariCP 泄漏检测**：

    hikari:

## 🔥 扩展追问 3：偶发问题怎么抓现场？

**Arthas 定时抓线程状态**：
```bash
# 每 10 秒抓一次线程状态，抓 100 次
for i in {1..100}; do
  java -jar arthas-boot.jar -c "thread -n 20" <pid> >> thread.log
  sleep 10
done
```

**分析 thread.log**：
- 看哪些线程长期在 `getConnection`
- 看它们在执行什么 SQL

## 🔥 扩展追问 4：超时时间到底设多少合理？

**公式**：
```
connection-timeout ≥ 业务可接受的最大 RT + 网络抖动余量
```

**参考**：
| 场景 | 建议 timeout |
|------|-------------|
| 内网 MySQL，RT 10ms 级 | 1000~3000ms |
| 跨可用区，RT 50ms 级 | 3000~5000ms |
| 公网/云数据库，RT 不稳定 | 5000~10000ms |

**不要设太短**（如 100ms），网络抖动一下就超时。

## 🔥 扩展追问 5：怎么预防这类问题？

**监控告警**：
```yaml
# Prometheus 告警规则
- alert: HighConnectionWait
  expr: hikaricp_connections_pending > 0
  for: 1m
  labels:
    severity: warning
  
- alert: ConnectionTimeoutRate
  expr: rate(hikaricp_connections_timeout_total[5m]) > 0.01
  labels:
    severity: critical
```

**代码规范**：
1. 所有数据库连接必须用 try-with-resources
2. 事务范围最小化，禁止在事务里调外部 HTTP
3. 慢 SQL 必须优化（加索引、分页、异步化）
