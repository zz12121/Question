## 先说结论

这是一个典型的 **分布式事务不一致** 问题，核心原因是 **扣钱（下单）和更新订单状态不在同一个事务中**，中间任何一步失败或超时都会导致数据不一致。

| 环节 | 操作 | 问题点 |
|-----|------|-------|
| 支付服务 | 扣用户钱（余额/银行卡） | ✅ 成功 |
| 第三方回调 | 通知支付成功 | ⚠️ 超时/丢失 |
| 订单服务 | 更新订单状态 | ❌ 未执行/失败 |

**一句话答案**：支付系统和订单系统是两个独立服务，调用链路是"扣钱 → 回调通知 → 订单状态更新"，中间任何一步网络抖动或服务宕机都会导致"钱扣了但订单未支付"。

### 1.1 链路图解

```
用户下单
    │
    ▼
┌──────────────────────────────────────────────────────────┐
│                    订单服务                                │
│  1. 创建订单（状态=待支付）                                │
│  2. 调用支付服务                                           │
└────────────────────────┬─────────────────────────────────┘

                         ▼
┌──────────────────────────────────────────────────────────┐
│                    支付服务                                │
│  3. 扣减用户余额/银行卡                                  │
│  4. 生成支付流水                                          │
└────────────────────────┬─────────────────────────────────┘
                         │ 异步回调（POST 通知）
                         ▼
┌──────────────────────────────────────────────────────────┐
│                    订单服务                                │
│  5. 接收支付回调                                          │
│  6. 更新订单状态为"已支付"                                │
└──────────────────────────────────────────────────────────┘
```

### 1.2 失败的 6 种场景

| # | 失败阶段 | 具体原因 | 表现 |
|---|---------|---------|------|
| 1 | 支付请求 | 网络超时，订单服务没收到响应 | 订单"待支付"，钱没扣 |
| 2 | 支付成功，回调 | 网络丢包，回调没发出去 | 钱扣了，订单"待支付" |
| 3 | 支付成功，回调 | 回调超时，订单服务返回失败 | 钱扣了，订单"待支付" |
| 4 | 订单服务处理 | 宕机/GC 暂停，处理到一半 | 钱扣了，订单"处理中" |
| 5 | 订单服务更新 | 数据库锁等待超时，更新失败 | 钱扣了，订单"待支付" |
| 6 | 幂等校验失败 | 重复回调被错误拒绝 | 钱扣了，订单"待支付" |

## 三、解决方案

### 3.1 方案一：TCC 事务（Try-Confirm-Cancel）

**核心思想**：把业务拆成 3 个阶段，全程可回滚

```
┌─────────┐    Try      ┌─────────┐   Confirm   ┌─────────┐
│  订单   │ ──────────→ │  支付   │ ──────────→ │  完成   │
│  服务   │             │  服务   │             │         │
└─────────┘             └─────────┘             └─────────┘
     │                       │
     │   Cancel（失败回滚）    │
     └───────────────────────┘
```

```java
// TCC 分布式事务实现
@LocalTCC
public interface PaymentTccService {
    
    @TwoPhaseBusinessAction(
        name = "deduct", 
        commitMethod = "confirm", 
        rollbackMethod = "cancel"
    )
    void deduct(BusinessActionContext context, 
                @BusinessActionContextParameter(paramName = "userId") Long userId,
                @BusinessActionContextParameter(paramName = "amount") BigDecimal amount,
                @BusinessActionContextParameter(paramName = "orderId") String orderId);
    
    void confirm(BusinessActionContext context);
    
    void cancel(BusinessActionContext context);
}

@Service
public class PaymentTccServiceImpl implements PaymentTccService {
    
    @Autowired private AccountMapper accountMapper;
    @Autowired private TransactionMapper transactionMapper;
    
    @Override
    public void deduct(BusinessActionContext context, Long userId, 
                       BigDecimal amount, String orderId) {
        // Try 阶段：冻结金额（不真正扣）
        accountMapper.freezeAmount(userId, amount);
        transactionMapper.createPending(orderId, userId, amount);
    }
    
    @Override
    public void confirm(BusinessActionContext context) {
        // Confirm 阶段：真正扣款
        Long userId = Long.valueOf(context.getActionContext("userId").toString());
        BigDecimal amount = new BigDecimal(context.getActionContext("amount").toString());
        String orderId = context.getActionContext("orderId").toString();
        
        accountMapper.confirmDeduct(userId, amount);
        transactionMapper.updateStatus(orderId, "SUCCESS");
    }
    
    @Override
    public void cancel(BusinessActionContext context) {
        // Cancel 阶段：解冻金额
        Long userId = Long.valueOf(context.getActionContext("userId").toString());
        BigDecimal amount = new BigDecimal(context.getActionContext("amount").toString());
        
        accountMapper.unfreezeAmount(userId, amount);
        transactionMapper.updateStatus(orderId, "CANCELLED");
    }
}
```

### 3.2 方案二：可靠消息 + 最终一致性（推荐）

**核心思想**：MQ 消息可靠投递 + 消费者幂等 + 定时对账

```
扣钱成功 ──→ 发送"支付成功"消息 ──→ 订单服务消费 ──→ 更新订单状态
    │                                    │
    │         ┌──────────────────────────┘
    │         ▼ 消费失败（重试）
    │      消息进入死信队列
    │         │
    │         ▼ 人工处理 / 定时任务补偿
    └────────────────────────────────────────
```

```java
// 1. 支付服务：可靠消息发送
@Service
public class PaymentService {
    
    @Autowired private RocketMQTemplate rocketMQ;
    
    public void pay(Order order) {
        // 1. 执行本地扣款
        accountService.deduct(order.getUserId(), order.getAmount());
        
        // 2. 发送支付成功消息（事务消息）
        PaymentMessage message = new PaymentMessage();
        message.setOrderId(order.getId());
        message.setUserId(order.getUserId());
        message.setAmount(order.getAmount());
        message.setPayTime(System.currentTimeMillis());
        
        // 使用事务消息，确保扣款和发消息在同一事务
        TransactionSendResult result = rocketMQ.sendMessageInTransaction(
            "order-pay-topic", 
            MessageBuilder.withPayload(message).build(),
            order);  // transaction executor 参数
        
        // 3. 记录消息发送状态
        log.info("事务消息发送结果: orderId={}, status={}", 
            order.getId(), result.getSendStatus());
    }
    
    // RocketMQ 事务监听器
    @RocketMQTransactionListener
    public class PaymentTransactionListener implements RocketMQLocalTransactionListener {
        
        @Override
        public RocketMQLocalTransactionState executeLocalTransaction(
            Message msg, Object arg) {
            try {
                // 这里执行扣款逻辑（与发送消息在同一事务）
                Order order = (Order) arg;
                accountService.deduct(order.getUserId(), order.getAmount());
                return RocketMQLocalTransactionState.COMMIT;
            } catch (Exception e) {
                log.error("本地事务失败，回滚消息", e);
                return RocketMQLocalTransactionState.ROLLBACK;
            }
        }
        
        @Override
        public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
            // 回查：检查扣款是否成功
            String orderId = parseOrderId(msg);
            Boolean success = paymentMapper.isPaySuccess(orderId);
            return success ? COMMIT : ROLLBACK;
        }
    }
}

// 2. 订单服务：幂等消费
@Service
public class OrderService {
    
    @Autowired private OrderMapper orderMapper;
    
    @RocketMQMessageListener(
        topic = "order-pay-topic",
        consumerGroup = "order-pay-consumer",
        concurrency = "10"
    )
    public void handlePaySuccess(PaymentMessage message) {
        // 幂等校验：检查订单是否已处理
        String lockKey = "pay:lock:" + message.getOrderId();
        Boolean acquired = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, "1", 30, TimeUnit.SECONDS);
        
        if (!Boolean.TRUE.equals(acquired)) {
            log.info("订单已处理，跳过: orderId={}", message.getOrderId());
            return;
        }
        
        try {
            // 更新订单状态
            int affected = orderMapper.updateStatusByIdempotentKey(
                message.getOrderId(), 
                message.getPayTime(),
                "PAID",
                message.getIdempotentKey()  // 幂等键
            );
            
            if (affected == 0) {
                log.warn("订单状态更新失败，可能已处理: orderId={}", 
                    message.getOrderId());
            }
        } finally {
            redisTemplate.delete(lockKey);
        }
    }
}
```

### 3.3 方案三：定时任务对账补偿

```java
// 兜底机制：每天凌晨对账，修复异常数据
@Component
public class AccountReconciliation {
    
    @Scheduled(cron = "0 0 3 * * ?")  // 每天凌晨3点
    public void reconcile() {
        // 1. 找出"钱扣了但订单未支付"的异常
        List<AbnormalOrder> abnormalOrders = findAbnormalOrders();
        
        for (AbnormalOrder order : abnormalOrders) {
            try {
                // 2. 自动修复：更新订单状态
                orderMapper.updateStatus(order.getOrderId(), "PAID", order.getPaidAt());
                log.info("自动修复异常订单: orderId={}", order.getOrderId());
            } catch (Exception e) {
                // 3. 修复失败，告警人工处理
                alertService.sendAlert("订单异常待人工处理", order);
            }
        }
    }
    
    private List<AbnormalOrder> findAbnormalOrders() {
        // 查出"账户流水有扣款，但订单未支付"的数据
        String sql = """
            SELECT o.order_id, o.user_id, o.amount, 
                   af.create_time as paid_time
            FROM orders o
            INNER JOIN account_flow af ON o.order_id = af.order_id
            WHERE o.status = 'PENDING'
              AND af.flow_type = 'DEDUCT'
              AND af.create_time > DATE_SUB(NOW(), INTERVAL 7 DAY)
            """;
        return jdbcTemplate.query(sql, (rs, row) -> {
            AbnormalOrder ao = new AbnormalOrder();
            ao.setOrderId(rs.getString("order_id"));
            ao.setUserId(rs.getLong("user_id"));
            ao.setAmount(rs.getBigDecimal("amount"));
            ao.setPaidAt(rs.getTimestamp("paid_time"));
            return ao;
        });
    }
}
```

## 四、完整幂等方案设计

```java
// 支付回调幂等：多级保障
@RestController
@RequestMapping("/pay/callback")
public class PayCallbackController {
    
    @Autowired private PaymentService paymentService;
    @Autowired private IdempotentService idempotentService;
    
    @PostMapping("/notify")
    public Result handleCallback(@RequestBody PayCallbackRequest request) {
        String idempotentKey = request.getTradeNo();  // 支付流水号
        
        // 第一层：Redis 快速幂等
        if (idempotentService.checkAndSet(idempotentKey, 5, TimeUnit.MINUTES)) {
            return Result.success("success");  // 已处理过，直接返回
        }
        
        // 第二层：DB 幂等（唯一索引）
        try {
            paymentService.processCallback(request);
            return Result.success("success");
        } catch (DuplicateKeyException e) {
            return Result.success("success");  // 重复回调
        } catch (Exception e) {
            // 第三层：失败重试
            idempotentService.remove(idempotentKey);  // 移除缓存，允许重试
            throw e;
        }
    }
}
```

## 五、问题对比总结

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|-----|------|------|------|---------|
| **TCC** | 3阶段可回滚 | 强一致 | 侵入性大 | 金融、账务 |
| **可靠消息** | MQ + 幂等 + 重试 | 低侵入 | 最终一致 | 电商订单 |
| **本地消息表** | 本地表 + MQ | 实现简单 | 增加 DB 负担 | 小规模 |
| **对账补偿** | 定时兜底 | 兜底保障 | 有延迟 | 所有场景 |

## 面试加分话术

> "这种'扣钱成功但订单未支付'的问题，本质是 **分布式事务一致性** 问题。
> 
> 我之前线上遇到过类似情况，排查后发现是 **支付回调超时导致消息丢失**。
> 
> 解决方案：
> 1. **事前**：支付回调加幂等校验，MQ 消息可靠投递
> 2. **事中**：RocketMQ 事务消息，保证扣款和发消息原子性
> 3. **事后**：定时对账任务，每天凌晨扫描异常数据并补偿
> 
> 现在我们线上是 **事务消息 + 幂等消费 + 定时对账** 三重保障，这种问题基本不会再出现了。"
