## 先说结论

从单体到支撑千万用户、十亿数据，核心思路是 **"先垂直拆分再水平拆分，先读写分离再分库分表"**——先按业务模块垂直拆成独立服务（用户服务、订单服务），单库扛不住后做读写分离，最后对大表做水平分片（Sharding）。整个过程要 **"业务低耦合优先，技术拆分跟上"**，避免过度设计，每一步都有回滚方案。

---

## 🔥 核心概念

- **解决什么问题**：单体应用在用户量和数据量暴涨后的扩展性问题，包括数据库性能瓶颈、代码维护复杂度、发布风险
- **拆分原则**：先垂直（按业务）后水平（按数据），先读写分离后分库分表，数据迁移优先双写
- **关键决策点**：什么时候拆？拆多细？怎么保证数据一致性？

---

## 🔥 详细展开

### 1. 演进路线图（按阶段）

```
阶段1：单体应用 + 单库（1万用户，100万行）
    ↓ 用户量涨，读压力大
阶段2：单体应用 + 读写分离（10万用户，1000万行）
    ↓ 写压力也大了，数据量涨
阶段3：垂直拆分（服务化 + 分库）（100万用户，1亿行）
    ↓ 单表数据量过大
阶段4：水平分片（分库分表）（1000万用户，10亿行）
    ↓ 全球化/多地域
阶段5：单元化/多活架构
```

---

### 2. 阶段详解

#### 阶段1：单体应用 + 单库（现状）

```
┌─────────────────┐
│   单体应用       │
│  (Spring Boot)  │
└────────┬────────┘
         │
    ┌────┴────┐
    │  MySQL  │
    │  单库单表 │
    └─────────┘
```

**痛点开始出现**：
- 查询变慢（没索引的全表扫描）
- 连接数打满（默认 151，高并发不够）

**优化（不拆）**：
- 加索引、优化慢 SQL
- 加缓存（Redis）抗读
- 连接池调大（HikariCP max 100）

---

#### 阶段2：读写分离（读压力大时）

```
┌─────────────────┐
│   单体应用       │
└────────┬────────┘
         │
    ┌────┴────┐
    │  Proxy  │  ← MyCat / ShardingSphere / 自研
    │ (读写分离) │
    └────┬────┘
    ┌────┴────┐
    ▼         ▼
┌───────┐  ┌───────┐
│ Master │  │ Slave │
│  (写)  │  │  (读)  │
└───────┘  └───────┘
```

**实现方式**：
- 中间件：ShardingSphere-JDBC、MyCat
- 或代码层：Spring 动态数据源，注解切换

**注意点**：
- 主从延迟（几十毫秒~几秒）：读自己的写要强制走主库
- 从库挂了的降级：自动切主库读

---

#### 阶段3：垂直拆分（服务化 + 分库）

**按业务模块拆服务**：

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 用户服务  │  │ 订单服务  │  │ 商品服务  │
│  (User)  │  │ (Order)  │  │ (Product) │
└────┬─────┘  └────┬─────┘  └────┬─────┘
     │             │             │
     ▼             ▼             ▼
┌─────────┐   ┌─────────┐   ┌─────────┐
│ user_db │   │order_db │   │prod_db  │
└─────────┘   └─────────┘   └─────────┘
```

**拆分原则**：
- 按业务边界：用户、订单、商品独立
- 每个服务有自己的数据库，禁止跨库 JOIN
- 服务间用 RPC（Dubbo / gRPC / OpenFeign）通信

**数据一致性**：
- 最终一致性：MQ 异步通知（用户注册后发优惠券）
- 分布式事务：Seata AT 模式（少用，性能差）

---

#### 阶段4：水平分片（分库分表）

**单表超过 5000 万行，必须水平拆分**：

```
order_db_0          order_db_1
├─ order_0          ├─ order_0
├─ order_1          ├─ order_1
├─ order_2          ├─ order_2
└─ order_3          └─ order_3

分片键：user_id
路由：user_id % 8 → 确定库和表
```

**分片策略**：

| 策略 | 公式 | 适用场景 |
|------|------|----------|
| 哈希取模 | `user_id % N` | 数据分布均匀，但扩容需迁移 |
| 范围分片 | `user_id [0-100w)` | 扩容方便，但可能热点 |
| 一致性哈希 | 虚拟节点 | 扩容迁移数据量少 |
| 混合分片 | 先按时间范围，再哈希 | 时间序列数据 |

**ShardingSphere 配置示例**：
```yaml
spring:
  shardingsphere:
    datasource:
      names: ds0, ds1
      ds0: jdbc:mysql://host0:3306/order_db_0
      ds1: jdbc:mysql://host1:3306/order_db_1
    rules:
      sharding:
        tables:
          order:
            actual-data-nodes: ds$->{0..1}.order_$->{0..3}
            table-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: order-hash
        sharding-algorithms:
          order-hash:
            type: INLINE
            props:
              algorithm-expression: order_$->{user_id % 8}
```

**全局 ID 生成**：
- 雪花算法（Snowflake）：分布式、趋势递增
- 号段模式（Leaf）：批量取号，性能高

---

#### 阶段5：单元化/多活架构（全球化）

```
        ┌─────────────┐
        │   全局路由   │
        │  (GSLB/DNS) │
        └──────┬──────┘
    ┌─────────┼─────────┐
    ▼         ▼         ▼
┌───────┐ ┌───────┐ ┌───────┐
│ 华北单元 │ │ 华东单元 │ │ 华南单元 │
│(北京)  │ │(上海)  │ │(广州)  │
└───────┘ └───────┘ └───────┘
```

**单元化原则**：
- 用户按地域划分单元（如 user_id 尾号 0-3 走华北）
- 单元内闭环：读写都在本单元，跨单元只读
- 全局数据同步：CockroachDB / TiDB / 自研同步

---

### 3. 数据迁移方案（关键！）

**双写方案（不停机）**：

```
阶段1：旧库读 + 旧库写
    ↓ 开始双写
阶段2：旧库读 + 旧库写 + 新库写（异步）
    ↓ 校验数据一致
阶段3：旧库读 + 双库写 + 新库读（灰度）
    ↓ 全部切新库
阶段4：新库读 + 新库写
```

**校验工具**：
- 对比旧库新库数据哈希
- 不一致用 MQ 补偿

## 🔥 扩展追问 1：分库分表后怎么 JOIN？

**禁止跨库 JOIN**，解法：

| 方案 | 实现 | 适用 |
|------|------|------|
| **宽表冗余** | 把需要的字段冗余到主表 | 读多写少 |
| **应用层组装** | 先查主表，再批量查关联表 | 关联数据少 |
| **ES 异构查询** | 数据同步到 Elasticsearch，复杂查询走 ES | 多维筛选 |
| **分布式 SQL** | ShardingSphere 支持跨片 JOIN（性能差） | 临时查询 |

**示例：订单表冗余用户昵称**
```sql
-- 订单表
CREATE TABLE order (
  order_id    BIGINT,
  user_id     BIGINT,
  user_nickname VARCHAR(64),  -- 冗余，避免查用户服务
  amount      DECIMAL,
  ...
);
```

## 🔥 扩展追问 2：分片键怎么选？

**原则**：
1. **高频查询条件**：80% 查询都带这个字段
2. **数据分布均匀**：避免热点（如按时间分片，最新数据集中）
3. **不可变**：分片键变更 = 数据迁移

**常见选择**：
| 业务 | 分片键 | 原因 |
|------|--------|------|
| 订单 | `user_id` | 按用户查订单最多 |
| 用户 | `user_id` | 主键查询 |
| 日志 | `create_time` | 按时间范围查询 |

**非分片键查询怎么办？**
- 建立映射表：`phone → user_id` 存在 Redis
- 扫全表：ES 异构，或并行扫所有分片（慢）

---

## 🔥 扩展追问 3：扩容时数据怎么迁移？

**一致性哈希减少迁移量**：

```
传统哈希：4 库 → 8 库，50% 数据要迁移
一致性哈希：虚拟节点，只迁移 1/N 数据
```

**迁移步骤**：
1. 双写新旧分片规则
2. 历史数据按新规则迁移
3. 校验一致性
4. 切读流量到新规则

---

## 🔥 扩展追问 4：分布式事务怎么解决？

**优先最终一致性，少用强一致性**：

| 场景 | 方案 |
|------|------|
| 下单扣库存 | 本地事务 + MQ 异步扣库存，失败补偿 |
| 转账 | TCC（Try-Confirm-Cancel），性能较好 |
| 强一致必须 | Seata AT 模式（性能差，慎用） |

**示例：下单异步扣库存**
```java
@Transactional
public Order createOrder(CreateOrderRequest req) {
    // 1. 本地事务：创建订单
    Order order = orderMapper.insert(req);
    
    // 2. 发 MQ 异步扣库存
    rocketMQTemplate.asyncSend("deduct-stock", 
        new StockDeductMessage(order.getOrderId(), req.getSkuId(), req.getCount()),
        new SendCallback() {...}
    );
    
    return order;
}

// 库存服务消费消息扣库存，失败发补偿消息
```

## 🔥 扩展追问 5：拆分过程中怎么保证业务连续性？

**关键原则**：

```
1. 先拆分代码，再拆分数据库
   → 代码层面先模块化，接口隔离
   
2. 数据库先双写，再切读，最后切写
   → 任何步骤可回滚
   
3. 灰度发布
   → 先 1% 流量，观察监控，再全量
   
4. 监控先行
   → 拆分前埋好指标，对比拆分前后 RT、错误率
```

## 一句话收尾

> 单体应用扩展的演进路线是 **"读写分离 → 垂直拆分 → 水平分片 → 单元化"**，每一步都遵循 **"先优化再拆分，先双写再切换，监控先行灰度发布"**。面试时说一句"我们按 user_id 哈希分 8 库 16 表，用 ShardingSphere 做路由，双写方案迁移数据"，立刻显得有实战经验。
