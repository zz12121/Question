## 先说结论

大文件上传的核心是 **"分片 + 秒传 + 断点续传"** 三件套：先把文件切成 4MB/8MB 的小块并行上传，用 MD5 做秒传判断（已存在直接返回成功），服务端合并时支持断点续传（只传失败的分片）。存储走对象存储（OSS/S3），元数据存 MySQL/Redis，上传状态用 Redis 缓存。

---

## 🔥 核心概念

- **解决什么问题**：大文件上传超时、网络抖动导致失败、重复上传浪费带宽、服务器内存扛不住大文件
- **核心思路**：
  - **分片上传**：把 10GB 切成 N 个 4MB 块，浏览器并行上传，失败只重传单个块
  - **秒传**：文件 MD5 已存在 → 直接返回成功，0 流量
  - **断点续传**：记录已上传的分片序号，刷新页面/断网后只传未完成的块

## 🔥 详细展开

### 1. 整体架构流程

```
用户选择 10GB 文件
    ↓
前端计算文件 MD5 + 分片（4MB/块）
    ↓
① 秒传检查：POST /upload/check {fileMd5, fileSize}
    → 查 Redis/MySQL，文件已存在？→ 直接返回秒传成功
    ↓
② 初始化上传：返回 uploadId + 已上传分片列表（断点续传用）
    ↓
③ 并行上传分片：POST /upload/chunk {uploadId, chunkIndex, chunkMd5, data}
    → 每个分片独立上传，失败只重传该块
    → 分片存临时目录（OSS 分片上传 API / 本地磁盘）
    ↓
④ 全部完成：POST /upload/merge {uploadId}
    → 服务端合并所有分片 → 移入正式目录
    → 写文件元数据表 → 返回文件 URL
```

### 2. 数据库表设计

**① 文件元数据表 `file_record`**
```sql
CREATE TABLE file_record (
  file_id       BIGINT PRIMARY KEY,
  file_md5      VARCHAR(32) NOT NULL,    -- 文件整体 MD5，秒传关键
  file_name     VARCHAR(256),
  file_size     BIGINT NOT NULL,         -- 字节
  chunk_size    INT DEFAULT 4194304,     -- 4MB
  chunk_count   INT NOT NULL,            -- 总分片数
  storage_path  VARCHAR(512),            -- OSS 路径或本地路径

  created_at    DATETIME,
  completed_at  DATETIME,
  UNIQUE KEY uk_file_md5 (file_md5)      -- 秒传去重
);
```

**② 分片上传记录表 `file_chunk`**
```sql
CREATE TABLE file_chunk (
  id            BIGINT PRIMARY KEY,
```

### 3. 核心接口设计

**① 秒传检查 + 初始化**
```java
@PostMapping("/upload/check")
public UploadCheckResult check(@RequestBody CheckRequest req) {
    // 1. 查文件是否已存在（MD5 唯一）
    FileRecord exist = fileMapper.selectByMd5(req.getFileMd5());
    if (exist != null && exist.getStatus() == 1) {
        return UploadCheckResult.builder()
            .skipUpload(true)           // 秒传！
            .fileUrl(exist.getStoragePath())
            .build();
    }
    
    // 2. 生成 uploadId，初始化分片记录
    String uploadId = UUID.randomUUID().toString();
    int chunkCount = (int) Math.ceil(req.getFileSize() / (double) CHUNK_SIZE);
    
    // 3. 查该 uploadId 已上传的分片（断点续传）
    List<Integer> uploadedChunks = chunkMapper
        .selectUploadedChunks(uploadId);
    
    return UploadCheckResult.builder()
        .skipUpload(false)
        .uploadId(uploadId)
        .chunkSize(CHUNK_SIZE)
        .chunkCount(chunkCount)
        .uploadedChunks(uploadedChunks)  // 告诉前端哪些不用传了
        .build();
}
```

---

**② 分片上传**
```java
@PostMapping("/upload/chunk")
public Result uploadChunk(
        @RequestParam("uploadId") String uploadId,
        @RequestParam("chunkIndex") int chunkIndex,
        @RequestParam("chunkMd5") String chunkMd5,
        @RequestParam("file") MultipartFile file) {
    
    // 1. 校验分片 MD5（防篡改）
    String realMd5 = DigestUtils.md5Hex(file.getInputStream());
    if (!realMd5.equals(chunkMd5)) {
        return Result.fail("分片校验失败");
    }
    
    // 2. 上传分片到 OSS（使用 OSS 分片上传 API）
    String chunkKey = String.format("temp/%s/%d", uploadId, chunkIndex);
    ossClient.putObject(bucket, chunkKey, file.getInputStream());
    
    // 3. 记录分片已上传
    chunkMapper.updateStatus(uploadId, chunkIndex, 1, chunkKey);
    redisTemplate.opsForSet().add("upload:" + uploadId, chunkIndex);
    
    return Result.ok();
}
```

> 💡 **OSS 分片上传**：阿里云 OSS 有 `InitiateMultipartUpload` → `UploadPart` → `CompleteMultipartUpload` 原生 API，直接用，比自己管理分片更稳。

---

**③ 合并分片**
```java
@PostMapping("/upload/merge")
public Result merge(@RequestBody MergeRequest req) {
    String uploadId = req.getUploadId();
    
    // 1. 检查是否全部分片上传完成
    Long uploadedCount = chunkMapper.countByUploadId(uploadId);
    FileRecord file = fileMapper.selectByUploadId(uploadId);
    if (uploadedCount < file.getChunkCount()) {
        return Result.fail("还有分片未上传");
    }
    
    // 2. OSS 合并分片（或本地合并）
    List<String> partETags = chunkMapper.selectChunkETags(uploadId);
    CompleteMultipartUploadRequest completeRequest = 
        new CompleteMultipartUploadRequest(bucket, objectKey, uploadId, partETags);
    ossClient.completeMultipartUpload(completeRequest);
    
    // 3. 清理临时分片，更新文件状态
    chunkMapper.deleteByUploadId(uploadId);
    fileMapper.updateStatus(file.getFileId(), 1, finalUrl);
    redisTemplate.delete("upload:" + uploadId);
    
    return Result.ok(finalUrl);
}
```

```javascript
const retryUpload = async (chunk, index, retryCount = 0) => {
  try {
    return await uploadChunk(chunk, index);
  } catch (e) {
    if (retryCount < 3) {
      await sleep(1000 * Math.pow(2, retryCount));
      return retryUpload(chunk, index, retryCount + 1);
    }
    throw e;
  }
};
```

## 🔥 扩展追问 1：秒传是怎么实现的？MD5 冲突怎么办？

**秒传原理**：
```
用户上传 10GB 文件 → 前端算 MD5 → 后端查 file_record 表
  → MD5 存在且 status=1 → 直接返回已有文件的 URL，不上传
  → MD5 不存在 → 正常走分片上传
```

**MD5 冲突（极小概率，但存在）**：
- 方案一：**MD5 + 文件大小** 联合唯一，降低冲突概率
- 方案二：**MD5 + SHA256** 双哈希，几乎不可能冲突
- 方案三：**抽样校验**，秒传后前端再抽样对比几个字节，确认一致

## 🔥 扩展追问 2：断点续传刷新页面后怎么恢复？

```
刷新页面 → 重新选择文件 → 前端重新计算 MD5
  → 调用 /upload/check {fileMd5}
    → 后端返回 uploadId + uploadedChunks=[0,1,3,5]（已上传的分片序号）
    → 前端只上传缺失的 [2,4,6...]
```

关键点：**uploadId 和已上传分片列表持久化到 DB**，不是纯内存，页面刷新不丢。

## 🔥 扩展追问 3：分片上传的并发控制怎么做？

浏览器同时发 100 个请求会卡死，需要**并发控制**：

```javascript
// Promise.all 控制并发（最多 3 个同时上传）
async function uploadWithLimit(chunks, limit = 3) {
  const pool = new Set();
  for (const [index, chunk] of chunks.entries()) {
    const promise = uploadChunk(chunk, index).then(() => pool.delete(promise));
    pool.add(promise);
    if (pool.size >= limit) {
      await Promise.race(pool);  // 有一个完成就继续
    }
  }
  await Promise.all(pool);  // 等全部完成
}
```

服务端也要做限流（Sentinel / Nginx limit_req），防止恶意并发打爆服务器。

## 🔥 扩展追问 4：超大文件（10GB+）前端算 MD5 卡死怎么办？

**分片计算 MD5**，不是一次性读整个文件：

```javascript
// SparkMD5 分片增量计算
const spark = new SparkMD5.ArrayBuffer();
let offset = 0;

while (offset < file.size) {
  const chunk = file.slice(offset, offset + chunkSize);
  const buffer = await chunk.arrayBuffer();
  spark.append(buffer);  // 增量追加
  offset += chunkSize;
  // 每算完一块更新进度条，UI 不卡
}

const fileMd5 = spark.end();
```

或者：**算抽样 MD5**（头 1MB + 中间 1MB + 尾 1MB），牺牲一点准确性换取速度。

## 🔥 扩展追问 5：分片存在哪里？OSS 还是本地磁盘？

| 场景 | 存储位置 | 原因 |
|------|----------|------|
| 生产环境 | **OSS 分片上传** | 高可用、自动合并、生命周期管理 |
| 测试/小文件 | 本地磁盘 / MinIO | 成本低，自己控制 |

**OSS 分片上传 API**（阿里云示例）：
```java
// 1. 初始化分片上传
InitiateMultipartUploadRequest initRequest = 
    new InitiateMultipartUploadRequest(bucket, objectKey);
InitiateMultipartUploadResult initResult = 
    ossClient.initiateMultipartUpload(initRequest);
String uploadId = initResult.getUploadId();  // OSS 返回的 uploadId

// 2. 上传每个分片（前端直接调 OSS 临时授权 URL，不走服务端）
// 3. 完成合并
CompleteMultipartUploadRequest completeRequest = 
    new CompleteMultipartUploadRequest(bucket, objectKey, uploadId, partETags);
ossClient.completeMultipartUpload(completeRequest);
```

> 💡 **最佳实践**：服务端只生成**临时授权 URL**（STS），前端直传 OSS，大文件流量不经过业务服务器。
