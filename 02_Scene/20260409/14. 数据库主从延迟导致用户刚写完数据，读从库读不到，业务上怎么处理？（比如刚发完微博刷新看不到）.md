## 结论先行

**核心矛盾：主从同步有延迟，写主库后立即读从库可能失败。**

**解决方案：强制读主库 / 延迟读 / 业务兜底，三层应对。**

---

## 一、问题根源：主从同步原理

```
┌─────────────────────────────────────────────────────────────────────┐
│                    MySQL 主从复制原理                                │
│                                                                     │
│  主库 (Master)                      从库 (Slave)                     │
│  ┌─────────────────┐               ┌─────────────────┐             │
│  │   ┌─────────┐   │   Binlog     │   ┌─────────┐   │             │
│  │   │ 用户写  │───▶│ ───────────▶ │   │  Relay  │   │             │
│  │   └─────────┘   │   异步同步     │   │  Log   │   │             │
│  │                 │               │   └────┬────┘   │             │
│  │                 │               │        │        │             │
│  │                 │               │        ▼        │             │
│  │                 │               │   ┌─────────┐  │             │
│  │                 │               │   │ 应用    │  │             │
│  │                 │               │   │ Binlog │  │             │
│  │                 │               │   └─────────┘  │             │
│  └─────────────────┘               └─────────────────┘             │
│                                                                     │
│  延迟来源：                                                          │
│  • Binlog 写入 + 传输 + 从库回放（毫秒~秒级）                          │
│  • 从库积压过多（从库慢、网络抖动）                                     │
│  • 大事务（1万条更新可能延迟几分钟）                                    │
│  • 从库查询压力大（读写都在从库拉取）                                   │
│                                                                     │
│  ⚠️ 正常延迟：毫秒级；异常延迟：秒~分钟级                              │
└─────────────────────────────────────────────────────────────────────┘
```

## 三、解决方案

### 方案一：强制读主库（最简单）

```
┌─────────────────────────────────────────────────────────────────────┐
│                    强制读主库策略                                    │
│                                                                     │
│  【核心思路】写操作后的短时间窗口内，强制读主库                        │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  适用场景：                                                     │ │
│  │  • 刚写入的数据必须立刻读到（订单、支付、微博）                  │ │
│  │  • 数据一致性要求 > 性能要求                                    │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

```java
/**
 * 读写分离 + 强制读主库
 */
@Service
public class UserService {

    @Autowired
    private MasterDataSource masterDataSource;   // 主库

    @Autowired
    private SlaveDataSource slaveDataSource;      // 从库

    @Autowired
    private DataSourceRouter dataSourceRouter;     // 动态路由

    /**
     * 写操作 → 强制读主库（时间窗口内）
     */
    @Transactional
    public void updateUser(Long userId, String nickname) {
        // 1. 写主库
        userMapper.updateNickname(userId, nickname);

        // 2. 标记：接下来 5 秒内强制读主库
        DataSourceContextHolder.setForceMaster(true);

        // 3. 立即读取验证（可选）
        User user = userMapper.selectById(userId);
    }

    /**
     * 读操作（自动路由）
     */
    public User getUser(Long userId) {
        // 检查是否强制读主库
        if (DataSourceContextHolder.isForceMaster()) {
            return masterUserMapper.selectById(userId);
        }
        // 默认读从库
        return slaveUserMapper.selectById(userId);
    }

    /**
     * 清理标记（建议用 AOP + @After 增强）
     */
    @AfterReturning(pointcut = "execution(* com.example..*Service.update*(..))",
                    returning = "result")
    public void clearForceMaster() {
        DataSourceContextHolder.setForceMaster(false);
    }
}

/**
 * 数据源路由上下文
 */
public class DataSourceContextHolder {

    private static final ThreadLocal<Boolean> forceMaster = new ThreadLocal<>();

    public static void setForceMaster(boolean value) {
        forceMaster.set(value);
    }

    public static boolean isForceMaster() {
        Boolean value = forceMaster.get();
        return value != null && value;
    }

    public static void clear() {
        forceMaster.remove();
    }
}

/**
 * Dynamic DataSource（切换主从）
 */
@Configuration
public class DynamicDataSource extends AbstractRoutingDataSource {

    @Override
    protected Object determineCurrentLookupKey() {
        if (DataSourceContextHolder.isForceMaster()) {
            return "master";
        }
        return "slave";
    }
}
```

### 方案二：延迟读取（用户体验兜底）

```
┌─────────────────────────────────────────────────────────────────────┐
│                    延迟读取 + 重试策略                                │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  核心思路：                                                     │ │
│  │  • 读从库失败？不慌，延迟 N ms 再试                              │ │
│  │  • 最多重试 M 次                                                  │ │
│  │  • 全部失败 → 降级读主库 / 返回友好提示                          │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

```java
/**
 * 延迟读取 + 重试兜底
 */
@Service
public class SafeReadService {

    @Autowired
    private SlaveUserMapper slaveMapper;

    @Autowired
    private MasterUserMapper masterMapper;

    private static final int MAX_RETRY = 3;
    private static final long RETRY_DELAY_MS = 100;

    /**
     * 安全读取：失败自动重试，最终兜底读主库
     */
    public User getUserSafely(Long userId) {
        // 第一次尝试读从库
        User user = tryReadFromSlave(userId);
        if (user != null) {
            return user;
        }

        // 重试 1~2 次
        for (int i = 1; i < MAX_RETRY; i++) {
            try {
                Thread.sleep(RETRY_DELAY_MS * i);  // 指数退避
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }

            user = tryReadFromSlave(userId);
            if (user != null) {
                log.info("重试成功, userId={}, retryCount={}", userId, i);
                return user;
            }
        }

        // 最终兜底：强制读主库
        log.warn("从库读取失败，兜底读主库, userId={}", userId);
        return masterMapper.selectById(userId);
    }

    private User tryReadFromSlave(Long userId) {
        try {
            return slaveMapper.selectById(userId);
        } catch (Exception e) {
            log.warn("从库读取失败, userId={}", userId, e);
            return null;
        }
    }
}
```

### 方案三：半同步复制（数据层面解决）

```
┌─────────────────────────────────────────────────────────────────────┐
│                    MySQL 半同步复制                                  │
│                                                                     │
│  【原理】主库事务提交 → 等待至少一个从库确认收到 → 才返回客户端       │
│                                                                     │
│  主库                              从库                              │
│  ┌─────────────────┐               ┌─────────────────┐             │
│  │ 写事务提交      │               │                 │             │
│  │       │        │   同步写入    │                 │             │
│  │       ├────────┼──────────────▶│ 写入 Relay Log   │             │
│  │       │        │               │                 │             │
│  │       │◀───────┼───────────────┤  ACK 确认       │             │
│  │       │ 收到ACK │               │                 │             │
│  │  返回成功      │               │                 │             │
│  └─────────────────┘               └─────────────────┘             │
│                                                                     │
│  配置：rpl_semi_sync_master_wait_point = AFTER_SYNC (默认)          │
│                                                                     │
│  ✅ 优点：数据强一致，从库不会丢失主库数据                             │
│  ⚠️ 缺点：增加事务延迟（等待 ACK 的时间）                             │
│  ⚠️ 缺点：从库挂了会退化为异步（有警告但继续）                         │
└─────────────────────────────────────────────────────────────────────┘
```

### 方案四：业务层兜底（最佳实践）

```java
/**
 * 业务层综合方案：写后直读 + 延迟队列 + 兜底
 */
@Service
public class BlogService {

    @Autowired
    private BlogMapper blogMapper;

    @Autowired
    private BlogCacheService cacheService;

    @Autowired
    private MQTemplate mqTemplate;

    /**
     * 发微博（写操作）
     */
    @Transactional
    public Blog publishBlog(Long userId, String content) {
        // 1. 写主库
        Blog blog = new Blog();
        blog.setUserId(userId);
        blog.setContent(content);
        blog.setCreateTime(new Date());
        blogMapper.insert(blog);

        // 2. 写缓存（重要！自己先缓存一份）
        cacheService.setBlog(blog.getId(), blog);

        // 3. 发送延迟消息（延迟读兜底）
        mqTemplate.send("blog.read.backfill", blog.getId());

        return blog;
    }

    /**
     * 读微博（带多层兜底）
     */
    public Blog getBlog(Long blogId, Long currentUserId) {
        // 第一层：读缓存（最新）
        Blog blog = cacheService.getBlog(blogId);
        if (blog != null) {
            return blog;
        }

        // 第二层：读从库
        blog = blogMapper.selectByIdFromSlave(blogId);
        if (blog != null) {
            // 回填缓存
            cacheService.setBlog(blogId, blog);
            return blog;
        }

        // 第三层：兜底读主库
        log.warn("从库读取失败，兜底读主库, blogId={}", blogId);
        blog = blogMapper.selectByIdFromMaster(blogId);
        return blog;
    }
}

/**
 * 延迟消息消费者：主动回填缓存（补偿延迟）
 */
@RabbitListener(queues = "blog.read.backfill")
public class BlogBackfillConsumer {

    @Autowired
    private BlogCacheService cacheService;

    @RabbitHandler
    public void handleBackfill(Long blogId) {
        // 等待 2 秒（给主从同步留时间）
        try {
            Thread.sleep(2000);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }

        // 再次读取（此时从库应该同步完成）
        Blog blog = blogMapper.selectByIdFromSlave(blogId);
        if (blog != null) {
            cacheService.setBlog(blogId, blog);
            log.info("博客已回填缓存, blogId={}", blogId);
        }
    }
}
```

## 四、缓存 + 数据库双写（终极方案）

```
┌─────────────────────────────────────────────────────────────────────┐
│                    缓存写优先 + 数据库兜底                            │
│                                                                     │
│  【核心思路】写入时同时写缓存，读取永远走缓存                         │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │   写操作                                                       │ │
│  │   ① 写数据库（主库）                                          │ │
│  │   ② 写缓存（Redis）                                           │ │
│  │   → 成功返回                                                   │ │
│  │                                                               │ │
│  │   读操作                                                       │ │
│  │   ① 读缓存 → 命中 → 返回 ✅                                    │ │
│  │   ② 未命中 → 读从库 → 回填缓存 → 返回                          │ │
│  │                                                               │ │
│  │   优点：读写都不依赖主从一致性                                  │ │
│  │   缺点：缓存有短暂不一致（可接受）                              │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

```java
/**
 * 写操作：数据库 + 缓存双写
 */
public void updateBlog(Long blogId, String content) {
    // 1. 写主库
    blogMapper.updateContent(blogId, content);

    // 2. 写缓存（延迟双删策略）
    cacheService.deleteBlog(blogId);
    // 延迟 300ms 再删除（等待主从同步）
    CompletableFuture.runAsync(() -> {
        try {
            Thread.sleep(300);
            cacheService.deleteBlog(blogId);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    });
}

/**
 * 读操作：缓存优先
 */
public Blog getBlog(Long blogId) {
    // 1. 读缓存
    Blog blog = cacheService.getBlog(blogId);
    if (blog != null) {
        return blog;
    }

    // 2. 缓存未命中，读从库
    blog = blogMapper.selectByIdFromSlave(blogId);

    if (blog != null) {
        // 3. 回填缓存
        cacheService.setBlog(blogId, blog, Duration.ofMinutes(10));
    }

    return blog;
}
```

## 五、综合对比表

| 方案 | 一致性 | 性能影响 | 实现成本 | 适用场景 |
|------|-------|---------|---------|---------|
| **强制读主库** | 强一致 | 高（主库压力大） | 低 | 订单、支付等核心链路 |
| **延迟重试** | 最终一致 | 中 | 中 | 非核心链路，可接受短暂失败 |
| **半同步复制** | 强一致 | 中（增加事务延迟） | 中 | 数据一致性要求极高的场景 |
| **缓存写优先** | 最终一致 | 低（读走缓存） | 高 | 读多写少，高并发场景 |
| **延迟消息补偿** | 最终一致 | 低 | 高 | 容忍延迟的异步场景 |

## 七、面试加分话术

> **面试官问：主从延迟监控怎么做？**
>
> 两层监控：
> 1. **MySQL 层**：`SHOW SLAVE STATUS` 的 `Seconds_Behind_Master`
> 2. **应用层**：埋点统计"从从库读到数据 vs 兜底读主库"的比例，超过 5% 告警

> **追问：强制读主库会不会把主库打爆？**
>
> 会的，所以要控制：
> 1. **只对关键路径强制读主库**（订单、支付），其他路径默认从库
> 2. **加本地缓存兜底**，减少对主库的冲击
> 3. **主库从库规格一致**，主库能扛读压力

> **再问：能不能从架构上根本解决主从延迟？**
>
> 两个方向：
> 4. **分库分表**：数据分散到多个库，减少单库写入压力
> 5. **读写都走同一数据源**（如 TiDB/CockroachDB，天然强一致）
> 6. **业务层不依赖主从一致性**：用 MQ + 缓存异步同步，读永远走缓存

---

## 一句话收尾

**"读写分离不是为了强一致，而是为了分摊压力"—— 强一致需求走主库，允许延迟走从库，业务层兜底才是王道。**
