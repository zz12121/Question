## 先说结论

有索引但走全表扫描，最常见的原因是**"索引失效"**——要么查询条件对索引列做了函数运算/类型转换导致无法命中，要么使用了 `!=`、`NOT IN`、`LIKE '%xx%'` 等无法使用索引的操作符，要么数据分布倾斜导致优化器认为全表扫描更划算。还有可能是索引统计信息过期，或者表太小优化器觉得没必要走索引。

---

## 🔥 核心概念

- **解决什么问题**：明明建了索引，但查询还是慢，需要定位索引失效的根因
- **索引失效的本质**：MySQL 优化器基于成本（CBO）选择执行计划，当估算走索引的成本 > 全表扫描时，就会放弃索引
- **排查核心**：看 `EXPLAIN` 的 `key`、`type`、`Extra` 字段，结合 SQL 写法分析

---

## 🔥 详细展开

### 1. 索引失效的常见场景（按发生频率排序）

| 场景 | 示例 | 解决方案 |
|------|------|----------|
| **对索引列做函数运算** | `WHERE DATE(create_time) = '2024-01-01'` | 改成范围查询 `WHERE create_time BETWEEN '2024-01-01 00:00:00' AND '2024-01-01 23:59:59'` |
| **隐式类型转换** | `WHERE user_id = '123'`（user_id 是 BIGINT） | 保证类型一致，或显式转换 `WHERE user_id = 123` |
| **LIKE 左模糊** | `WHERE name LIKE '%张三%'` | 改用全文索引，或右模糊 `LIKE '张三%'` |
| **使用了 `!=` 或 `<>`** | `WHERE status != 1` | 改为 `IN (0,2,3)` 或考虑反向索引 |
| **使用了 `NOT IN`** | `WHERE id NOT IN (1,2,3)` | 改为 `LEFT JOIN` 或 `NOT EXISTS` |
| **使用了 `OR` 且条件不全走索引** | `WHERE id = 1 OR name = 'xxx'`（name 无索引） | 拆成 `UNION`，或给 name 也加索引 |
| **索引列参与计算** | `WHERE age + 1 = 18` | 改成 `WHERE age = 17` |
| **数据分布倾斜** | 某值占 90% 数据，`WHERE status = 1` | 考虑分区、或该值单独处理 |

---

### 2. 具体案例分析

**场景一：函数导致索引失效（最常见）**

```sql
-- ❌ 索引失效：对 create_time 做 DATE() 函数
SELECT * FROM orders 
WHERE DATE(create_time) = '2024-01-01';

-- ✅ 正确写法：范围查询，索引生效
SELECT * FROM orders 
WHERE create_time >= '2024-01-01 00:00:00' 
  AND create_time < '2024-01-02 00:00:00';
```

**EXPLAIN 对比**：
```
❌ type=ALL, key=NULL, Extra=Using where; Using filesort
✅ type=range, key=idx_create_time, Extra=Using index condition
```

---

**场景二：隐式类型转换**

```sql
-- 表结构：user_id BIGINT，有索引 idx_user_id

-- ❌ 索引失效：字符串和数字比较，MySQL 隐式转换
SELECT * FROM orders WHERE user_id = '123456';

-- ✅ 正确写法：类型一致
SELECT * FROM orders WHERE user_id = 123456;
```

> 💡 可以用 `EXPLAIN WARNINGS` 看优化器警告，会提示隐式转换。

---

**场景三：LIKE 左模糊**

```sql
-- ❌ 索引失效：% 在左边
SELECT * FROM users WHERE name LIKE '%张三%';

-- ✅ 索引生效：% 在右边（前缀匹配）
SELECT * FROM users WHERE name LIKE '张三%';

-- 🔥 需要全文检索：用 MATCH AGAINST
SELECT * FROM users WHERE MATCH(name) AGAINST('张三');
```

---

**场景四：OR 条件部分无索引**

```sql
-- 假设只有 idx_user_id，没有 idx_status

-- ❌ 索引失效：status 无索引，整体走全表
SELECT * FROM orders 
WHERE user_id = 100 OR status = 1;

-- ✅ 方案一：拆成 UNION
SELECT * FROM orders WHERE user_id = 100
UNION ALL
SELECT * FROM orders WHERE status = 1;

-- ✅ 方案二：给 status 也建索引，或建联合索引
```

---

### 3. 优化器"故意"不走索引的情况

**数据分布倾斜（选择性太低）**：

```sql
-- orders 表 1000 万行，status=1 的有 950 万行（95%）

SELECT * FROM orders WHERE status = 1;

-- EXPLAIN 显示 type=ALL，key=NULL
```

优化器估算：走索引需要回表 950 万次，成本 > 全表扫描 1000 万行，所以选全表。

**解法**：
- 该值单独处理（如归档到历史表）
- 使用覆盖索引（`SELECT status FROM ...` 只查索引列）
- 强制索引（不推荐）：`FORCE INDEX(idx_status)`

---

**表太小**：

```sql
-- 表只有 100 行
SELECT * FROM small_table WHERE id = 1;
```

优化器认为全表扫描 100 行比走索引（查索引 + 回表）更快，直接全表。

---

### 4. 统计信息过期

```sql
-- 查看索引统计信息
SHOW INDEX FROM orders;

-- 更新统计信息（MySQL 8.0）
ANALYZE TABLE orders;
```

当表数据变化很大（如批量导入），统计信息没及时更新，优化器可能做出错误选择。

## 🔥 扩展追问 1：怎么确认是索引失效还是根本没用到索引？

**EXPLAIN 关键字段解读**：

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 100\G
```

| 字段 | 含义 | 判断 |
|------|------|------|
| `type` | 访问类型 | `ALL`=全表扫描，`range`=范围索引，`ref`=等值索引，`const`=主键 |
| `key` | 实际使用的索引 | `NULL`=没用索引 |
| `key_len` | 索引使用长度 | 越短用的越少，为 0 表示没用 |
| `rows` | 扫描行数估算 | 和实际差距大说明统计信息不准 |
| `Extra` | 额外信息 | `Using index`=覆盖索引，`Using where`=回表过滤，`Using filesort`=需要排序 |

---

## 🔥 扩展追问 2：强制走索引有用吗？什么情况下用？

```sql
-- 强制使用索引（不推荐常规使用）
SELECT * FROM orders FORCE INDEX(idx_user_id) WHERE user_id = 100;
```

**适用场景**：
- 确认是优化器统计信息问题，临时应急
- AB 测试对比性能

**风险**：
- 数据分布变化后，强制索引可能变得更慢
- 代码维护性差

**更好的做法**：
- 定期 `ANALYZE TABLE` 更新统计信息
- 使用 `OPTIMIZER HINT`（MySQL 5.6+）：
```sql
SELECT /*+ INDEX(orders idx_user_id) */ * FROM orders WHERE user_id = 100;
```

---

## 🔥 扩展追问 3：联合索引为什么没走？

**最左前缀原则**：

```sql
-- 联合索引：idx_a_b_c (a, b, c)

-- ✅ 走索引
WHERE a = 1
WHERE a = 1 AND b = 2
WHERE a = 1 AND b = 2 AND c = 3
WHERE a = 1 AND c = 3          -- 只走 a，c 不走（中间断了）

-- ❌ 不走索引
WHERE b = 2                    -- 缺少最左列 a
WHERE b = 2 AND c = 3          -- 缺少最左列 a
WHERE a = 1 OR b = 2           -- OR 导致索引失效
```

---

## 🔥 扩展追问 4：覆盖索引是什么？怎么用？

**覆盖索引**：查询的所有列都在索引里，不需要回表。

```sql
-- 索引：idx_user_status (user_id, status)

-- ❌ 需要回表：SELECT * 包含了不在索引里的列
SELECT * FROM orders WHERE user_id = 100;

-- ✅ 覆盖索引：只查索引列
SELECT user_id, status FROM orders WHERE user_id = 100;

-- EXPLAIN 显示 Extra=Using index，性能最好
```

**强制走覆盖索引解决选择性低的问题**：
```sql
-- 即使 status=1 有 95% 数据，只查索引列也很快
SELECT status FROM orders WHERE status = 1;
```

## 🔥 扩展追问 5：怎么预防这类问题？

**开发规范**：
1. 禁止对索引列使用函数（`DATE()`、`UPPER()` 等）
2. 字段类型和查询值类型必须一致
3. LIKE 查询必须右模糊，左模糊走全文索引
4. 联合索引设计遵循最左前缀，把等值条件放左边
5. 定期 `EXPLAIN` 检查慢 SQL

**监控告警**：
- 慢查询日志阈值设为 1 秒，自动告警
- 每周分析慢 SQL，检查索引使用情况

## 一句话收尾

> 索引失效的排查口诀是 **"先看 EXPLAIN 的 type 和 key，再检查 SQL 是否对索引列做函数/转换/左模糊，最后看数据分布是否倾斜"**。最常见的是 `DATE(create_time)` 和 `LIKE '%xxx%'`，面试时直接举这两个例子，面试官就知道你踩过坑。
