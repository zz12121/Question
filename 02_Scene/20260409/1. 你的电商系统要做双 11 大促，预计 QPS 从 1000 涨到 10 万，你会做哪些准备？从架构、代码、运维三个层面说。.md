## 先说结论

扩容的本质是 **"让非核心链路扛流量，核心链路保准确"**：

| 维度 | 核心思路 | 目标效果 |
|-----|---------|---------|
| **架构层** | 分层 + 隔离 + 降级 | 流量分层拦截，系统不雪崩 |
| **代码层** | 缓存 + 异步 + 幂等 | 减少数据库压力，数据不错乱 |
| **运维层** | 扩容 + 压测 + 监控 | 提前发现瓶颈，运行时可观测 |

**一句话答案**：架构上做读写分离 + 服务拆分 + 限流降级；代码上缓存预热 + 异步削峰 + 幂等保护；运维上弹性扩容 + 全链路压测 + 实时大盘监控。

## 二、代码层：性能优化的 6 把刀

### 2.1 缓存：多级缓存 + 预热

```java
// 多级缓存：Caffeine(L1) + Redis(L2) + 数据库
@Configuration
public class CacheConfig {
    
    @Bean
    public Cache<Long, ProductVO> localCache() {
        return Caffeine.newBuilder()
            .maximumSize(10_000)
            .expireAfterWrite(10, TimeUnit.SECONDS)  // 短过期，避免数据不一致
            .recordStats()
            .build();
    }
}

@Service
public class ProductService {
    
    @Autowired private Cache<Long, ProductVO> localCache;
    @Autowired private RedisTemplate<Long, ProductVO> redisCache;
    @Autowired private ProductMapper productMapper;
    
    public ProductVO getProduct(Long productId) {
        // L1: 本地缓存（极低延迟）
        ProductVO vo = localCache.getIfPresent(productId);
        if (vo != null) {
            return vo;
        }
        
        // L2: Redis
        vo = redisCache.opsForValue().get("product:" + productId);
        if (vo != null) {
            localCache.put(productId, vo);
            return vo;
        }
        
        // L3: 数据库
        vo = productMapper.selectById(productId).toVO();
        redisCache.opsForValue().set("product:" + productId, vo, 1, TimeUnit.HOURS);
        localCache.put(productId, vo);
        return vo;
    }
    
    // 大促前预热缓存
    @PostConstruct
    public void warmUpCache() {
        List<Product> hotProducts = productMapper.selectHotProducts();
        hotProducts.forEach(p -> {
            redisCache.opsForValue().set("product:" + p.getId(), p.toVO());
            localCache.put(p.getId(), p.toVO());
        });
    }
}
```

### 2.3 幂等：Token 机制防重复提交

```java
// 秒杀场景的幂等保护
@Service
public class IdempotentService {
    
    private static final String TOKEN_PREFIX = "idempotent:token:";
    
    // 1. 下单前申请 Token
    public String generateToken(Long userId, Long productId) {
        String token = UUID.randomUUID().toString();
        redis.opsForValue().set(TOKEN_PREFIX + token, 
            userId + ":" + productId, 5, TimeUnit.MINUTES);
        return token;
    }
    
    // 2. 扣减库存前校验 Token
    public boolean tryConsumeToken(String token, Long userId, Long productId) {
        String script = """
            local value = redis.call('get', KEYS[1])
            if value == nil then
                return -1  -- token 不存在或已过期
            end
            if value ~= ARGV[1] then
                return 0   -- token 不匹配
            end
            redis.call('del', KEYS[1])
            return 1
            """;
        
        Long result = redis.execute(script, 
            List.of(TOKEN_PREFIX + token),
            userId + ":" + productId);
        
        return result == 1;
    }
}
```

### 2.4 热点检测与隔离

```java
// 热点商品自动识别 + 隔离
@Service
public class HotProductDetector {
    
    private static final int HOT_THRESHOLD = 1000;  // 1000 QPS 判定为热点
    
    @Autowired private RedisTemplate<String, Object> redis;
    
    // 实时统计商品 QPS
    public void recordAccess(Long productId) {
        String key = "hot:product:" + productId;
        Long count = redis.opsForValue().increment(key);
        redis.expire(key, 60, TimeUnit.SECONDS);  // 60 秒滑动窗口
        
        // 达到热点阈值，触发隔离
        if (count != null && count == HOT_THRESHOLD) {
            hotProductService.isolateProduct(productId);
        }
    }
}

// 热点商品走独立线程池，避免拖累其他商品
@Configuration
public class ThreadPoolConfig {
    
    @Bean("hotProductExecutor")
    public ThreadPoolExecutor hotProductExecutor() {
        return new ThreadPoolExecutor(50, 100, 
            60, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(10000),
            new ThreadFactoryBuilder().setNameFormat("hot-prod-%d").build(),
            new ThreadPoolExecutor.CallerRunsPolicy());  // 满了由调用方执行
    }
}
```

## 面试加分话术

> "大促备战的核心是 **提前发现瓶颈，提前解决**，而不是等大促当天救火。
> 
> 我的习惯是把问题往前推：
> 1. **压测前置**：提前 2 周做全链路压测，找出真正的瓶颈点
> 2. **缓存为王**：大促 90% 的流量其实是读请求，让缓存挡住大部分
> 3. **优雅降级**：即使系统扛不住，也要保证核心链路（下单、支付）可用
> 4. **可观测**：出问题不怕，怕的是不知道问题在哪，监控和链路追踪必须到位"

---

**一句话收尾**：从 1000 到 10 万 QPS，不是简单加机器就行的——**架构上做分层隔离，代码上做缓存异步，运维上做压测监控**，三者缺一不可。
