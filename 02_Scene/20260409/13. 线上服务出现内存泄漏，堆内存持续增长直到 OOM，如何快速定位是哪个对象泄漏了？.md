## 一、问题识别：内存泄漏 vs 内存溢出

│  │  │ ╱                                                             │ │  │
│  │  │╱                                                             │ │  │

## 二、快速定位：四步排查法

### 第一步：确认 OOM 类型和堆内存状态

```bash
# 1. 查看 JVM 内存使用情况（无需重启服务）
jstat -gcutil <pid> 1000 10

# 输出示例：
#   S0     S1     E      O      M     YGC     YGCT    FGC    FGCT     GCT   
#  75.0   0.00  85.0   98.5   95.2  12345  123.456  567    89.012  212.468
#  S0/S1=Survivor区  E=Eden区  O=老年代  M=Metaspace  YGC=年轻代GC  FGC=FullGC

# 关键指标：
# • O(Old区) 接近 100% → 对象无法晋升，反复 Full GC
# • FGC 频繁 → 内存压力极大
# • YGC 很频繁但 O 区持续增长 → 内存泄漏（对象过早进入老年代）
```

```bash
# 2. 查看进程内存详情
jps -lvm
# 输出：12345 com.example.Application --Xms4g --Xmx4g

# 3. 查看系统内存占用
free -h
ps aux | grep <pid> | grep -v grep
```

### 第二步：保留堆快照（关键时刻）

```bash
# ⚠️ 线上操作要谨慎！建议在低峰期 + 保留多个快照

# 方式1：jmap（保留堆快照，瞬时暂停）
# 生成 HPROF 格式堆 dump
jmap -dump:format=b,file=/tmp/heap_$(date +%Y%m%d_%H%M%S).hprof <pid>

# -XX:HeapDumpPath=/data/logs/oom/ \
# -XX:+ExitOnOutOfMemoryError

# 方式3：G1 堆Dump（推荐，更快）
jcmd <pid> GC.heap_dump /tmp/heap_live.hprof

# 4. 实时查看大对象
jmap -histo <pid> | head -50
# 输出：num #instances #bytes class name
#      1:       12345  5678900  com.example.cache.BigCacheObject

```

## 三、常见内存泄漏场景与根因分析

### 场景一：静态集合类持有对象（最常见）

```java
/**
 * ❌ 典型泄漏：静态 Map 不断添加，永不清理
 */
@Service
public class BadCacheService {

    // 静态集合 = GC Root（类变量）
    // 不断添加，永不清理 → 内存持续增长
    private static Map<String, Object> cache = new HashMap<>();

    public void process(Object data) {
        // 每次请求都往里塞
        cache.put(UUID.randomUUID().toString(), data);
        // ❌ 没有清理逻辑
    }
}

/**
 * ✅ 修复1：使用 WeakHashMap（弱引用，GC 可回收）
 */
private static Map<String, Object> cache = new WeakHashMap<>();

/**
 * ✅ 修复2：使用带 TTL 的缓存框架
 */
private static Cache<String, Object> cache = Caffeine.newBuilder()
    .maximumSize(10_000)           // 最多 1 万条
    .expireAfterWrite(Duration.ofMinutes(10))  // 10 分钟过期
    .build();
```

### 场景二：未关闭的资源（连接、Stream、线程）

```java
/**
 * ❌ 典型泄漏：连接/流未关闭
 */
@Service
public class BadFileService {

    public String readFile(String path) {
        FileInputStream fis = null;
        try {
            fis = new FileInputStream(path);
            return readAsString(fis);
            // ❌ 如果抛异常，fis 永远不会 close
        } catch (IOException e) {
            // ...
        }
        // ❌ 方法结束，fis 资源泄漏
    }
}

/**
 * ✅ 修复：try-with-resources（自动关闭）
 */
public String readFile(String path) {
    try (FileInputStream fis = new FileInputStream(path)) {
        return readAsString(fis);
    } catch (IOException e) {
        throw new RuntimeException(e);
    }
    // 自动关闭，无需 finally
}

/**
 * ❌ 典型泄漏：线程未正确停止
 */
public class BadTimerTask {

    private static final Timer timer = new Timer();

    public void start() {
        // 每秒执行一次
        timer.scheduleAtFixedRate(new TimerTask() {
            @Override
            public void run() {
                // ⚠️ 匿名内部类持有外部类引用（如果是 Activity/Service → 内存泄漏）
            }
        }, 0, 1000);
        // ❌ 页面销毁时未 cancel() → TimerTask 泄漏
    }
}

/**
 * ✅ 修复：ScheduledExecutorService
 */
private static ScheduledExecutorService executor = 
    Executors.newScheduledThreadPool(2);

public void start() {
    executor.scheduleAtFixedRate(() -> {
        // 业务逻辑
    }, 0, 1, TimeUnit.SECONDS);
}

public void stop() {
    executor.shutdown();  // 停止时调用
}
```

### 场景三：监听器和回调未注销

```java
/**
 * ❌ 典型泄漏：EventListener 注册后未注销
 */
@Service
public class BadEventService {

    @Autowired
    private ApplicationEventPublisher publisher;

    public void registerUser(User user) {
        // 注册事件监听器
        publisher.addListener(event -> {
            // 处理事件
        });
        // ❌ 每次注册都加一个监听器，永不删除
    }
}

/**
 * ✅ 修复：使用 @EventListener + 手动注销
 */
@Component
public class UserRegistrationListener {

    @EventListener
    public void onUserRegistered(UserRegisteredEvent event) {
        // Spring 自动管理生命周期
    }
}

// 或手动管理
private List<Consumer<Event>> listeners = new CopyOnWriteArrayList<>();

public void register(Consumer<Event> listener) {
    listeners.add(listener);
}

public void unregister(Consumer<Event> listener) {
    listeners.remove(listener);
}
```

### 场景四：ThreadLocal 泄漏（经典！）

```java
/**
 * ❌ 典型泄漏：ThreadLocal 未清理
 */
public class BadThreadLocalService {

    // ThreadLocal 变量
    private static ThreadLocal<UserContext> contextHolder = new ThreadLocal<>();

    public void process() {
        // 设置上下文
        contextHolder.set(new UserContext());
        // ... 业务逻辑 ...
        // ❌ 请求结束未清理
        // → 如果线程池复用，ThreadLocal 数据残留
    }
}

/**
 * ✅ 修复1：使用完手动清理
 */
public void process() {
    try {
        contextHolder.set(new UserContext());
        // ... 业务逻辑 ...
    } finally {
        contextHolder.remove();  // ✅ 清理
    }
}

/**
 * ✅ 修复2：包装为可清理的 ThreadLocal（推荐）
 */
public class CleanableThreadLocal<T> {

    private final ThreadLocal<T> delegate = new ThreadLocal<>();

    public void set(T value) {
        delegate.set(value);
    }

    public T get() {
        return delegate.get();
    }

    /**
     * 清理（配合框架使用）
     */
    public void remove() {
        delegate.remove();
    }
}

/**
 * ✅ 修复3：使用 Alibaba TransmittableThreadLocal
 * （阿里开源，解决线程池复用时 ThreadLocal 传递问题）
 */
import com.alibaba.ttl.TtlRunnable;

ExecutorService executor = Executors.newFixedThreadPool(4);

public void submitTask(Runnable task) {
    // 自动传递 ThreadLocal
    executor.execute(TtlRunnable.get(task));
}
```

### 场景五：String.intern() 误用

```java
/**
 * ❌ 典型泄漏：频繁 intern() 占用 Metaspace
 */
public class BadStringService {

    public void process() {
        // ❌ 频繁 intern()，大量字符串进入字符串常量池
        // 字符串常量池在 Metaspace，Metaspace 默认无限（但可配置）
        String result = someLargeString.intern();
    }
}

/**
 * ✅ 修复：减少 intern() 使用，或扩大 Metaspace
 */
-XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m
```

## 七、面试加分话术

> **面试官问：如何在不停服务的情况下定位 OOM？**
>
> 三步走：
> 1. **先看监控**：JMX / Grafana 看哪个区域爆了
> 2. **在线诊断**：Arthas `heapdump` 只 dump 存活对象，GC 期间采样
> 3. **离线分析**：上传 GCEasy 或用 MAT 分析

> **追问：OOM 发生在 Metaspace 还是 Heap？**
>
> 两回事：
> - **Heap OOM**：对象太多，`-Xmx` 不够 → 看 Histogram 找大对象
> - **Metaspace OOM**：类加载太多，字符串常量池爆了 → 查 `jstat -gc 0` 的 M 列

> **再问：如何避免线上 OOM？**
>
> 预防为主：
> 1. **正确配置 JVM 参数**（-Xmx、MetaspaceSize）
> 2. **开启自动 Dump**（-XX:+HeapDumpOnOutOfMemoryError）
> 3. **监控告警**（老年代 > 80%、Full GC 频率 > 阈值）
> 4. **代码Review**：静态集合、ThreadLocal、连接池必须过审

## 一句话收尾

**OOM 不慌：监控先发现苗头 → Arthas 实时诊断 → 堆 dump 离线分析 → MAT 找引用链 → 定位 GC Root 根因。**
