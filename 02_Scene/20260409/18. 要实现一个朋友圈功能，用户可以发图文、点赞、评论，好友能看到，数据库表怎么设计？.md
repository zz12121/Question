## 先说结论

朋友圈核心是**"写扩散 vs 读扩散"的 Feed 流设计选型**，数据库层面需要设计：帖子表、点赞表、评论表、好友关系表，再配合 Feed 流分发策略（推模式/拉模式/推拉结合），才能同时兼顾写性能和读性能。

---

## 🔥 核心概念

- **解决什么问题**：多用户之间社交内容的发布、存储、分发和查询，要求写入低延迟、好友 Feed 流聚合高效、点赞/评论实时可见
- **核心难点**：好友 Feed 流聚合（不是简单查自己的帖子，而是聚合所有好友的帖子按时间排序）、高并发点赞的计数一致性、评论楼层结构

---

## 🔥 详细展开

### 1. 核心表设计

**① 用户表 `user`**
```sql
CREATE TABLE user (
  user_id     BIGINT PRIMARY KEY,
  nickname    VARCHAR(64),
  avatar_url  VARCHAR(256),
  created_at  DATETIME
);
```

---

**② 帖子表 `post`**
```sql
CREATE TABLE post (
  post_id     BIGINT PRIMARY KEY,   -- 分布式ID，如雪花算法
  user_id     BIGINT NOT NULL,      -- 发帖人
  content     VARCHAR(2000),        -- 文字内容
  media_urls  JSON,                 -- 图片/视频URL数组，存CDN地址
  location    VARCHAR(128),         -- 可选：发帖位置
  visible     TINYINT DEFAULT 0,    -- 0=所有好友可见 1=私密 2=部分好友
  like_count  INT DEFAULT 0,        -- 冗余计数，避免count(*)
  comment_count INT DEFAULT 0,
  created_at  DATETIME,
  INDEX idx_user_time (user_id, created_at DESC)
);
```

> 💡 `media_urls` 用 JSON 字段存数组，避免额外的图片关联表；`like_count` 冗余存储，直接展示无需聚合查询。

---

**③ 好友关系表 `friendship`**
```sql
CREATE TABLE friendship (
  id          BIGINT PRIMARY KEY,
  user_id     BIGINT NOT NULL,      -- 主动方
  friend_id   BIGINT NOT NULL,      -- 被关注方
  status      TINYINT,              -- 1=单向关注 2=互为好友
  created_at  DATETIME,
  UNIQUE KEY uk_user_friend (user_id, friend_id),
  INDEX idx_user_id (user_id)
);
```

> 💡 微信是双向好友，两边都插一条记录，查自己的好友列表只需 `WHERE user_id = ?` 即可，避免 OR 查询。

---

**④ 点赞表 `post_like`**
```sql
CREATE TABLE post_like (
  id          BIGINT PRIMARY KEY,
  post_id     BIGINT NOT NULL,
  user_id     BIGINT NOT NULL,      -- 点赞人
  created_at  DATETIME,
  UNIQUE KEY uk_post_user (post_id, user_id),  -- 防重复点赞
  INDEX idx_post_id (post_id)
);
```

> 💡 点赞用独立表存储，`UNIQUE KEY` 天然防幂等；点赞总数走 `post.like_count` 冗余字段，不做 `COUNT(*)`。

---

**⑤ 评论表 `comment`**
```sql
CREATE TABLE comment (
  comment_id    BIGINT PRIMARY KEY,
  post_id       BIGINT NOT NULL,
  user_id       BIGINT NOT NULL,     -- 评论人
  parent_id     BIGINT DEFAULT 0,    -- 0=一级评论，非0=回复某条评论
  reply_to_uid  BIGINT DEFAULT 0,    -- 被回复人ID（展示"回复@xxx"）
  content       VARCHAR(500),
  created_at    DATETIME,
  INDEX idx_post_parent (post_id, parent_id, created_at)
);
```

> 💡 `parent_id` 支持二级楼层结构（主流方案，微信朋友圈只有两级）。如果要支持无限嵌套，换成 `path` 字段存祖先路径（闭包表方案）。

---

**⑥ Feed 流收件箱表 `feed_inbox`（推模式专用）**
```sql
CREATE TABLE feed_inbox (
  id          BIGINT PRIMARY KEY,
  owner_id    BIGINT NOT NULL,   -- 收件人（好友）
  post_id     BIGINT NOT NULL,
  author_id   BIGINT NOT NULL,
  created_at  DATETIME,
  INDEX idx_owner_time (owner_id, created_at DESC)
);
```

---

### 2. Feed 流分发策略选型

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **推模式（写扩散）** | 发帖时主动往所有好友收件箱写一条记录 | 读 Feed 极快，一张表直接查 | 大V发帖写放大严重（1000个粉丝=写1000条） | 好友数量有上限（如微信5000人） |
| **拉模式（读扩散）** | 读 Feed 时实时聚合所有好友的最新帖子 | 写性能好，无写放大 | 读放大严重，好友多时查询慢 | 关注数量无上限的场景 |
| **推拉结合** | 普通用户推，大V拉 | 均衡写读性能 | 实现复杂 | 微博、抖音等 |

> 💡 **微信朋友圈** 好友上限5000，用**推模式**即可。发帖时往好友的 `feed_inbox` 表写记录，用户刷朋友圈就是 `SELECT * FROM feed_inbox WHERE owner_id = ? ORDER BY created_at DESC LIMIT 20`。

---

### 3. 高并发点赞方案

```
用户点赞请求
    ↓
Redis SETNX post:like:{post_id}:{user_id} 1  （去重判断）
    ↓（成功）
Redis INCR post:like:count:{post_id}       （计数器）
    ↓（异步）
MQ 消息 → 消费者写 post_like 表 + 更新 post.like_count
```

> 点赞的**实时计数**走 Redis，**持久化**走 MQ 异步写库，避免高并发下 DB 压力。

## 🔥 扩展追问 1：帖子表要不要分库分表？

**要的，但时机很关键。**

早期单表扛得住就不拆，等到 post 表超过 5000 万行再考虑。分表策略：
- **按 `user_id` 分片**：同一个用户的帖子在同一个分片，查某人主页很快；但查 Feed 流（聚合多个好友）需要跨分片 —— 所以配合 `feed_inbox` 推模式，Feed 查询走收件箱表，收件箱可以按 `owner_id` 分片。
- 帖子详情页只查 `post_id`，可以用 `post_id % N` 再建一层路由。

---

## 🔥 扩展追问 2：评论的二级结构查询怎么做？

一次查帖子详情时，分两步：
```sql
-- 第一步：查所有一级评论
SELECT * FROM comment WHERE post_id = ? AND parent_id = 0 ORDER BY created_at LIMIT 20;

-- 第二步：根据一级评论ID，批量查二级回复
SELECT * FROM comment WHERE parent_id IN (?, ?, ...) ORDER BY created_at;
```
在业务层组装成树形结构，**不用递归 SQL**，避免 N+1 查询。

---

## 🔥 扩展追问 3：点赞了哪些人怎么展示？

点赞列表展示的是"头像墙"，一般只展示最新 N 个点赞人：
```sql
SELECT u.avatar_url, u.nickname 
FROM post_like pl JOIN user u ON pl.user_id = u.user_id
WHERE pl.post_id = ?
ORDER BY pl.created_at DESC LIMIT 10;
```
也可以把最新10个点赞人的 `user_id` 直接 JSON 冗余存进 `post` 表，省一次 JOIN。

---

## 🔥 扩展追问 4：可见性（仅好友/部分好友/私密）怎么实现？

```sql
-- post 表的 visible 字段
0 = 所有好友可见
1 = 仅自己可见
2 = 部分好友可见（需要额外的 post_visible_user 白名单表）
```

Feed 分发时在推送环节过滤：`visible=0` 推给所有好友，`visible=2` 只推给白名单好友。读取时二次校验权限，防止越权。

---

## 🔥 扩展追问 5：图片/视频怎么存储？

**绝对不存数据库，存 CDN。**
- 上传时走对象存储（OSS/COS），拿到 URL
- `post.media_urls` 存 CDN URL 数组（JSON）
- 展示时客户端直连 CDN，不经过业务服务器

视频还要加转码流程（HLS 切片），上传完成后异步触发转码 MQ，转码完成再更新 `post` 状态为可见。

---

## 一句话收尾

> 朋友圈的表设计核心是 **"帖子表 + 好友关系表 + 点赞/评论独立表 + Feed 收件箱表"** 四件套，配合 **推模式 Feed 分发 + Redis 点赞计数 + 冗余字段避免聚合查询**，就能在高并发下稳住性能。面试官问到这里，顺手说一句"大V场景要推拉结合"，立刻加分。

---
答案给完了，几个**最容易被追问的高频坑**总结一下：

| 追问点 | 一句话记住 |
|--------|-----------|
| Feed 流为啥要推模式？ | 读多写少，收件箱查询 O(1)，比实时聚合快得多 |
| 点赞为啥不直接写库？ | 高并发下 DB 扛不住，Redis 先抗，MQ 异步落盘 |
| 图片为啥不存数据库？ | BLOB 数据库性能差、带宽贵，CDN 才是正解 |
| 评论树形结构怎么查？ | 两次 SQL + 业务层组装，别用递归 SQL |
| 帖子表什么时候分？ | 超 5000 万行再考虑，按 `user_id` 分片 |
