## 先说结论

CPU 低但 TPS 上不去，典型的 **"非 CPU 瓶颈"**——最常见的是 **线程阻塞**（等数据库连接、等锁、等 IO）、**线程数配置不足**（Tomcat/业务线程池太小）、或者 **下游服务慢**（同步调用外部 HTTP/RPC 阻塞）。排查思路是 **"先看线程状态，再看资源等待，最后看调用链路"**：用 `jstack` 或 Arthas 看线程 dump，结合链路追踪找阻塞点。

---

## 🔥 核心概念

- **解决什么问题**：压测时系统资源没打满，但吞吐量上不去，需要定位真正的瓶颈
- **关键指标**：CPU 使用率、线程状态（RUNNABLE/BLOCKED/WAITING）、接口 RT（响应时间）、连接池使用率
- **排查原则**：CPU 低说明不是计算密集型，瓶颈在 IO 或并发度

---

## 🔥 详细展开

### 1. 常见瓶颈分类（CPU 低时）

| 瓶颈类型 | 典型表现 | 定位方法 |
|----------|----------|----------|
| **线程池打满** | Tomcat 线程 200/200，请求排队 | `jstack` 看线程状态，或监控 `tomcat.threads.busy` |
| **数据库连接池耗尽** | 等连接，线程状态 `WAITING` | 监控 `hikaricp.connections.active` / `pending` |
| **同步调用下游慢** | 线程 `RUNNABLE` 但卡在 socket read | 链路追踪看下游 RT，或 `jstack` 看堆栈 |
| **锁竞争** | 线程 `BLOCKED`，堆栈显示 `synchronized` | `jstack` 找 `BLOCKED` 线程，看持有锁的线程 |
| **磁盘 IO 慢** | 日志刷盘、大文件读写 | `iostat -x 1` 看 `%util` 和 `await` |
| **网络带宽满** | 网卡流量打满 | `iftop`、`nload` 看流量 |
| **GC 停顿** | 频繁 Young GC 或 Full GC | GC 日志看停顿时间 |

### 2. 排查步骤（按优先级）

#### 第一步：看线程状态分布

```bash
# 打印线程 dump
jstack <pid> > thread.dump

# 统计线程状态
grep java.lang.Thread.State thread.dump | sort | uniq -c
```

**关键状态**：
```
RUNNABLE      # 正常执行中
BLOCKED       # 等锁，锁竞争严重
WAITING       # 无限等待（如 wait()、join()）
TIMED_WAITING # 限时等待（如 sleep、wait(timeout)）
```

**如果 BLOCKED + WAITING 占比高** → 线程阻塞是瓶颈

---

#### 第二步：Arthas 快速定位

```bash
#  attach 进程
java -jar arthas-boot.jar

# 看线程状态统计
thread --state BLOCKED

# 看最忙的线程（按 CPU）
thread -n 5

# 看线程详细堆栈
thread <tid>
```

**示例：发现线程都在等数据库连接**

#### 第三步：看资源使用率

**数据库连接池**：
```yaml
# 开启 HikariCP 监控
management:
  metrics:
    enable:
      hikaricp: true
```

看指标：
- `hikaricp.connections.active` = 最大连接数 → 连接池太小
- `hikaricp.connections.pending` > 0 → 有线程在等连接

**Tomcat 线程池**：
```yaml
server:
  tomcat:
    threads:
      max: 200        # 默认 200
      min-spare: 10
    accept-count: 100 # 队列长度
```

监控 `tomcat.threads.busy` / `tomcat.threads.config.max`，如果 busy ≈ max，调大线程数。

#### 第四步：链路追踪看下游

如果线程显示 `RUNNABLE` 但卡在 `socketRead`：


```
你的接口 (100ms)
  └── 调用下游服务 A (80ms)   ← 瓶颈在这里
      └── 下游服务 A 内部处理
```

### 3. 典型案例分析

**案例一：数据库连接池太小**

```yaml
# ❌ 连接池只有 10，Tomcat 线程 200
spring:
  datasource:
    hikari:
      maximum-pool-size: 10

# ✅ 调大连接池（一般 = Tomcat 线程数）
spring:
  datasource:
    hikari:
      maximum-pool-size: 50
      minimum-idle: 20
```

**症状**：`jstack` 显示大量线程 `WAITING` 在 `HikariPool.getConnection`，连接池监控显示 active = 10（打满）。

---

**案例二：同步调用下游 HTTP 阻塞**

```java
// ❌ 同步调用，线程阻塞等待
RestTemplate restTemplate = new RestTemplate();
ResponseEntity<String> resp = restTemplate.getForEntity("http://slow-service/api", String.class);

// ✅ 改用异步或缓存
@Async
public CompletableFuture<String> callAsync() {
    return CompletableFuture.supplyAsync(() -> 
        restTemplate.getForObject("http://slow-service/api", String.class)
    );
}

// 或加本地缓存
@Cacheable("slowData")
public String callWithCache() { }
```

**症状**：线程状态 `RUNNABLE`，但堆栈卡在 `socketRead`，下游 RT 高。

---

**案例三：锁竞争严重**

```java
// ❌ 全局锁，并发度为 1
private static final Object LOCK = new Object();

public void process() {
    synchronized (LOCK) {  // 所有线程排队
        // 业务逻辑
    }
}

// ✅ 分段锁，提高并发
private final ConcurrentHashMap<String, Object> locks = new ConcurrentHashMap<>();

public void process(String key) {
    synchronized (locks.computeIfAbsent(key, k -> new Object())) {
        // 只锁相同 key 的请求
    }
}
```

**症状**：`jstack` 显示大量 `BLOCKED` 线程，等待同一个 monitor。

## 🔥 扩展追问 1：怎么确定是线程池瓶颈还是连接池瓶颈？

**对比指标**：

| 指标 | 线程池瓶颈 | 连接池瓶颈 |
|------|-----------|-----------|
| Tomcat busy threads | ≈ max | < max |
| 连接池 active | < max | ≈ max |
| 线程状态 | `RUNNABLE` 正常 | `WAITING` 等连接 |
| 响应时间 | 正常 | 变长（等连接排队） |

**如果两者都满了** → 先调大连接池，再调大线程池（线程数 > 连接数没意义）。

---

## 🔥 扩展追问 2：Tomcat 线程数配多少合适？

**公式**：
```
线程数 = (CPU 核数 * 目标 CPU 利用率 * (1 + 等待时间/计算时间))
```

**示例**：
- 4 核 CPU，目标利用率 80%
- 接口平均 RT 100ms，其中 80ms 等 DB，20ms 计算
- 线程数 = 4 * 0.8 * (1 + 80/20) = 16 * 5 = 80

**实际配置**：
```yaml
server:
  tomcat:
    threads:
      max: 100        # 根据公式计算，留 20% 余量
    max-connections: 10000  # NIO 模式可以很大
```

## 🔥 扩展追问 3：异步化能提升 TPS 吗？

**同步模型**（Tomcat 线程阻塞等 IO）：
```
线程数 = 并发上限，线程阻塞 = 资源浪费
```

**异步模型**（WebFlux / Servlet 3.0 Async）：
```
少量线程处理大量连接，IO 等待时不占线程
```

**适用场景**：
- IO 密集型（大量下游调用、数据库查询）→ 异步提升大
- CPU 密集型 → 异步没帮助，反而增加复杂度

**Servlet 异步示例**：
```java
@GetMapping("/async")
public Callable<String> async() {
    return () -> {
        // 在另一个线程执行，Tomcat 线程立即释放
        return slowService.call();
    };
}
```

```bash
# 实时看网卡流量
iftop -i eth0

# 或

# 看连接状态
```

**推荐工具链**：

```
压测工具：JMeter / k6 / Gatling
    ↓
指标采集：Prometheus + Micrometer
    ↓
可视化：Grafana 看板
    ↓

```

**关键看板**：
- JVM：CPU、内存、GC、线程状态
- 应用：QPS、RT、错误率、Tomcat 线程、连接池
- 系统：磁盘 IO、网络流量、文件描述符

## 一句话收尾

> CPU 低但 TPS 上不去的排查口诀是 **"先看线程状态找阻塞，再看连接池和线程池配置，最后用链路追踪找下游慢调用"**——最常见的是数据库连接池太小、同步调用下游阻塞、或者锁竞争。面试时说一句"我们用异步化和连接池调优，把 TPS 从 1000 提升到 5000"，立刻加分。
