## 结论先行

**核心思路：业务侧做幂等，不依赖 MQ 可靠性保证。**

消费流程：`接收消息 → 业务处理 → 幂等校验 → 提交 ACK`

## 二、幂等性保障方案

### 方案一：唯一消息 ID + 幂等表（最通用）

```
┌─────────────────────────────────────────────────────────────────┐
│                         幂等表设计方案                            │
│                                                                  │
│  表结构：consumer_idempotent                                    │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ msg_id (PK)  │ biz_id  │ status │ create_time │ update_time│  │
│  ├──────────────┼─────────┼────────┼─────────────┼────────────┤  │
│  │ uuid-12345   │ order-1 │  1     │ 2026-04-10  │ 2026-04-10 │  │
│  │ uuid-12346   │ order-2 │  1     │ 2026-04-10  │ 2026-04-10 │  │
│  │ uuid-12347   │ order-3 │  0     │ 2026-04-10  │            │  │
│  └──────────────────────────────────────────────────────────┘  │
│  status: 0=处理中, 1=成功                                         │
│  msg_id: 消息唯一标识（MQ 投递时携带）                             │
│  biz_id: 业务主键（订单号、用户ID等）                              │
└─────────────────────────────────────────────────────────────────┘
```

**代码实现：**

```java
@Service
public class IdempotentMessageConsumer {

    @Autowired
    private IdempotentRecordMapper idempotentMapper;

    @Autowired
    private OrderService orderService;

    /**
     * 消费消息（带幂等保障）
     */
    @RabbitListener(queues = "order.create.queue")
    public void consumeOrderMessage(Message message, Channel channel) {
        long deliveryTag = message.getMessageProperties().getDeliveryTag();

        try {
            // 1. 解析消息，获取幂等 key
            String msgId = message.getMessageProperties().getMessageId();
            OrderMessage orderMsg = JSON.parseObject(message.getBody(), OrderMessage.class);

            // 2. 幂等检查 + 防重处理（核心！）
            boolean processed = handleWithIdempotency(msgId, orderMsg);

            if (!processed) {
                // 已处理过，跳过（打印日志即可，不要报错）
                log.info("消息已处理，跳过, msgId={}", msgId);
            }

            // 3. 手动 ACK
            channel.basicAck(deliveryTag, false);

        } catch (Exception e) {
            log.error("消费失败，重试, deliveryTag={}", deliveryTag, e);
            // NACK + 重新入队（触发重试）
            channel.basicNack(deliveryTag, false, true);
        }
    }

    /**
     * 幂等处理核心逻辑
     * @return true=本次处理了业务, false=之前已处理过
     */
    private boolean handleWithIdempotency(String msgId, OrderMessage orderMsg) {
        // 开启事务
        TransactionTemplate transaction = new TransactionTemplate(
            dataSource.getTransactionManager()
        );

        return transaction.execute(status -> {
            try {
                // ① 尝试插入幂等记录（唯一索引保证原子性）
                int inserted = idempotentMapper.insertIfNotExists(msgId, orderMsg.getBizId());
                if (inserted == 0) {
                    // 记录已存在，说明之前处理过了
                    return false;
                }

                // ② 执行业务逻辑
                orderService.createOrder(orderMsg);

                // ③ 更新幂等记录状态为成功
                idempotentMapper.updateStatus(msgId, 1);

                return true;

            } catch (DuplicateKeyException e) {
                // 唯一键冲突 = 重复消息
                log.warn("重复消息检测到, msgId={}", msgId);
                return false;
            }
        });
    }
}
```

**Mapper 层（MyBatis）：**

### 方案二：业务状态机 + 乐观锁（业务层面天然幂等）

```
┌──────────────────────────────────────────────────────────────────┐
│                     业务幂等：订单状态机                           │
│                                                                  │
│  ┌──────┐    pay_suc    ┌─────────┐   deliver   ┌───────────┐  │
│  │CREATED│──────────────▶│PAID     │─────────────▶│DELIVERING │  │
│  └──┬───┘               └────┬────┘              └─────┬─────┘  │
│     │                        │                         │        │
│     │ 重复支付               │ 重复回调                  │ 完成   │
│     │ (幂等)                 │ (幂等)                   │        │
│     ▼                        ▼                         ▼        │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  CREATE → PAID → DELIVERING → COMPLETED (单向不可逆)    │   │
│  │                                                          │   │
│  │  UPDATE orders SET status = :newStatus                  │   │
│  │  WHERE id = :orderId                                    │   │
│  │    AND status = :currentStatus  ← 乐观锁条件            │   │
│  │    AND version = :version;                              │   │
│  │                                                          │   │
│  │  if (affectedRows == 0) → 幂等返回（状态已是目标值）     │   │
│  └──────────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────┘
```

```java
/**
 * 业务幂等：订单状态流转（天然幂等）
 */
public boolean updateOrderStatus(Long orderId, OrderStatus targetStatus) {
    // 1. 查询当前状态
    Order order = orderMapper.selectById(orderId);
    if (order == null) {
        return false;
    }

    // 2. 校验状态流转合法性（状态机）
    if (!isValidTransition(order.getStatus(), targetStatus)) {
        log.warn("非法状态流转, orderId={}, from={}, to={}",
                 orderId, order.getStatus(), targetStatus);
        return true; // 幂等返回，不报错
    }

    // 3. 乐观锁更新
    int rows = orderMapper.updateStatusWithVersion(
        orderId,
        order.getStatus(),    // 当前状态（乐观锁条件）
        targetStatus,         // 目标状态
        order.getVersion()    // 版本号
    );

    if (rows == 0) {
        // 更新失败 = 并发修改或状态已变化 = 重复消息
        log.info("订单状态更新冲突（幂等跳过）, orderId={}", orderId);
        return true;
    }

    // 4. 执行业务后置动作
    afterStatusChanged(orderId, order.getStatus(), targetStatus);
    return true;
}

/**
 * 乐观锁 SQL
 */
@Update("UPDATE orders SET status = #{targetStatus}, version = version + 1 " +
        "WHERE id = #{orderId} AND status = #{currentStatus} AND version = #{version}")
int updateStatusWithVersion(@Param("orderId") Long orderId,
                            @Param("currentStatus") Integer currentStatus,
                            @Param("targetStatus") Integer targetStatus,
                            @Param("version") Long version);
```

### 方案三：Redis 幂等（高性能场景）

```java
/**
 * Redis 幂等：适合高并发、允许短暂不一致的场景
 */
@Service
public class RedisIdempotentHandler {

    @Autowired
    private StringRedisTemplate redisTemplate;

    private static final String IDEMPOTENT_KEY_PREFIX = "msg:idempotent:";
    private static final Duration EXPIRE = Duration.ofHours(24);

    public boolean tryAcquire(String msgId, String bizType) {
        String key = IDEMPOTENT_KEY_PREFIX + bizType + ":" + msgId;

        // SETNX 原子操作，成功=未处理，失败=已处理
        Boolean success = redisTemplate.opsForValue()
                .setIfAbsent(key, "1", EXPIRE);

        return Boolean.TRUE.equals(success);
    }

    public void consumeWithIdempotency(String msgId, String bizType, Runnable businessLogic) {
        // 1. 尝试获取幂等锁
        if (!tryAcquire(msgId, bizType)) {
            log.info("消息已处理，跳过, msgId={}", msgId);
            return;
        }

        try {
            // 2. 执行业务
            businessLogic.run();
        } catch (Exception e) {
            // 3. 失败则释放锁（允许重试）
            redisTemplate.delete(IDEMPOTENT_KEY_PREFIX + bizType + ":" + msgId);
            throw e;
        }
    }
}
```

## 三、综合对比

| 方案 | 实现复杂度 | 性能 | 数据可靠性 | 适用场景 |
|------|-----------|------|-----------|---------|
| **幂等表（DB）** | 中 | ⭐⭐⭐ | 强（持久化） | 通用场景，订单、支付等核心链路 |
| **业务状态机** | 低 | ⭐⭐⭐⭐ | 强 | 状态流转类业务（订单、库存） |
| **Redis** | 低 | ⭐⭐⭐⭐⭐ | 中（可能丢数据） | 高并发、允许短暂不一致 |
| **去重表 + Redis 组合** | 高 | ⭐⭐⭐⭐ | 强 | 极致高性能 + 高可靠需求 |

**兜底补偿机制：**

```java
/**
 * 兜底：扫描处理中的消息，处理超时未 ACK 的
 */
@Scheduled(fixedDelay = 60000)
public void compensateUnfinishedMessages() {
    // 1. 找出"处理中"超过 N 分钟的记录
    List<IdempotentRecord> pendingRecords = idempotentMapper.selectProcessingOverdue(
        Duration.ofMinutes(5).toMillis()
    );

    for (IdempotentRecord record : pendingRecords) {
        // 2. 重新投递到 MQ（带重试标记）
        mqTemplate.send("order.create.retry",
            MessageBuilder.withBody(record.getBizContent())
                .setMessageId(record.getMsgId())
                .setHeader("x-retry-count", record.getRetryCount() + 1)
                .build()
        );

        log.warn("补偿重试消息, msgId={}, retryCount={}",
                 record.getMsgId(), record.getRetryCount());
    }
}
```

## 五、面试加分话术

> **面试官问：除了幂等表，还有什么思路？**
>
> 可以分三层：
> 1. **入口去重**：MQ 本身支持消息 ID 去重（开启 RabbitMQ 的 `x-message-ttl` + `x-dead-letter-exchange`）
> 2. **业务幂等**：业务表加状态机 + version 字段，更新时用乐观锁
> 3. **兜底补偿**：定时任务扫描"处理中"记录，防止极端情况漏处理

> **追问：Redis 挂了怎么办？**
>
> Redis 只做快速拦截，**不作为唯一幂等依据**。核心幂等依赖 DB 唯一索引，Redis 挂了最多导致少量重复消费，不会产生数据问题。

> **再问：如何控制重试次数，防止死循环？**
>
> 4. 消息头带 `x-retry-count`，超过阈值转入死信队列
> 5. DB 幂等表记录重试次数，达到上限报警 + 人工介入
> 6. 指数退避：`delay = baseDelay * 2^retryCount`，避免频繁重试打爆系统

---

## 一句话收尾

**幂等性 = 业务层兜底 + DB 唯一索引 + Redis 快速拦截 + 兜底补偿，四层防护缺一不可。**
