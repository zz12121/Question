## 先说结论

三个操作在不同服务，**不存在原子性，只能保证最终一致性**：

| 方案 | 原理 | 一致性 | 复杂度 | 适用场景 |
|-----|------|-------|-------|---------|
| **TCC** | Try-Confirm-Cancel 三阶段 | 强一致 | ⭐⭐⭐⭐ | 金融、库存敏感 |
| **Saga** | 编排 + 补偿事务 | 最终一致 | ⭐⭐⭐ | 长流程、异步 |
| **可靠消息** | MQ + 本地表 + 定时扫表 | 最终一致 | ⭐⭐⭐ | 量大、允许延迟 |
| **Seata AT** | 自动回滚 | 强一致 | ⭐⭐ | 微服务框架集成 |

**一句话答案**：用 **可靠消息 + 本地表 + 定时补偿** 或 **Saga 编排** 实现最终一致性，牺牲瞬时一致性换取性能和可用性。

## 一、问题本质分析

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   库存服务    │     │   余额服务    │     │   订单服务    │
│              │     │              │     │              │
│ 扣减库存     │     │  扣减余额     │     │  创建订单     │

       │                    │                    │
       │                    │                    │
       └────────────────────┼────────────────────┘
                           │

                    要么全回滚？
```

**核心矛盾**：每个服务只能保证自己的本地事务，无法感知其他服务是否成功。

### 2.3 手写 TCC 实现（了解原理）

```java
/**
 * TCC 分布式事务接口定义
 */
public interface TccTransaction {
    
    /**
     * Try 阶段：预留资源
     */
    void tryOperation();
    
    /**
     * Confirm 阶段：确认执行
     */
    void confirm();
    
    /**
     * Cancel 阶段：取消释放
     */
    void cancel();
}

// ===== 库存服务 TCC =====
@Service
public class StockTccServiceImpl implements StockTccService {
    
    @Autowired private StockMapper stockMapper;
    @Autowired private StockFreezeMapper freezeMapper;
    
    @Override
    public void tryOperation(Long productId, Integer quantity) {
        // 1. 检查库存是否足够
        Stock stock = stockMapper.selectByProductId(productId);
        if (stock.getAvailableStock() < quantity) {
            throw new BizException("库存不足");
        }
        
        // 2. 冻结库存（预留）
        stockMapper.freezeStock(productId, quantity);
        
        // 3. 记录冻结流水（用于回滚）
        StockFreeze freeze = new StockFreeze();
        freeze.setProductId(productId);
        freeze.setQuantity(quantity);
        freeze.setStatus("FROZEN");
        freeze.setExpireTime(LocalDateTime.now().plusMinutes(30));  // 30分钟超时
        freezeMapper.insert(freeze);
    }
    
    @Override
    public void confirm() {
        // 真正扣减库存
        List<StockFreeze> freezes = freezeMapper.selectPendingFreezes();
        for (StockFreeze freeze : freezes) {
            stockMapper.decreaseStock(freeze.getProductId(), freeze.getQuantity());
            freezeMapper.updateStatus(freeze.getId(), "CONFIRMED");
        }
    }
    
    @Override
    public void cancel() {
        // 解冻库存
        List<StockFreeze> freezes = freezeMapper.selectPendingFreezes();
        for (StockFreeze freeze : freezes) {
            stockMapper.unfreezeStock(freeze.getProductId(), freeze.getQuantity());
            freezeMapper.updateStatus(freeze.getId(), "CANCELLED");
        }
    }
}
```

### 3.1 Saga 原理

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         Saga 编排模式                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   ┌─────────┐                                                          │
│   │ 开始    │                                                          │
│   └────┬────┘                                                          │
│        │                                                               │
│        ▼                                                               │
│   ┌─────────┐     成功      ┌─────────┐     成功      ┌─────────┐     │
│   │ 扣余额   │ ──────────→ │ 扣库存   │ ──────────→ │ 创建订单  │     │
│   └────┬────┘              └────┬────┘              └────┬────┘     │
│        │                        │                        │          │
│        │ 失败                    │ 失败                    │ 失败    │
│        ▼                        ▼                        ▼          │
│   ┌─────────┐              ┌─────────┐              ┌─────────┐     │
│   │ 补偿余额 │              │ 补偿库存 │              │ 删除订单 │     │
│   │ (失败)  │              │ (回滚)  │              │ (回滚)  │     │
│   └─────────┘              └─────────┘              └─────────┘     │
│                                                                         │
│   正向操作：A → B → C                                                  │
│   补偿操作：C.cancel() → B.cancel() → A.cancel()                       │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.2 Saga 状态机实现

```java
/**
 * Saga 订单创建器
 */
@Service
@Slf4j
public class SagaOrderCreator {
    
    @Autowired private AccountService accountService;
    @Autowired private StockService stockService;
    @Autowired private OrderService orderService;
    @Autowired private SagaLogMapper sagaLogMapper;
    
    /**
     * Saga 编排创建订单
     */
    @Transactional
    public SagaResult createOrderSaga(CreateOrderRequest request) {
        String sagaId = UUID.randomUUID().toString();
        SagaContext context = new SagaContext(sagaId, request);
        
        try {
            // Step 1: 扣减余额（正向）
            log.info("[Saga开始] sagaId={}, step=扣余额", sagaId);
            SagaStep step1 = new SagaStep("DEDUCT_BALANCE");
            boolean deducted = accountService.deductBalance(
                request.getUserId(), request.getTotalAmount());
            
            if (!deducted) {
                throw new BizException("余额不足");
            }
            step1.setStatus("SUCCESS");
            
            // Step 2: 扣减库存（正向）
            log.info("[Saga] sagaId={}, step=扣库存", sagaId);
            SagaStep step2 = new SagaStep("DEDUCT_STOCK");
            boolean stockOk = stockService.deductStock(
                request.getProductId(), request.getQuantity());
            
            if (!stockOk) {
                // 补偿 Step 1
                compensateStep(step1, context);
                throw new BizException("库存不足");
            }
            step2.setStatus("SUCCESS");
            
            // Step 3: 创建订单（正向）
            log.info("[Saga] sagaId={}, step=创建订单", sagaId);
            SagaStep step3 = new SagaStep("CREATE_ORDER");
            Order order = orderService.createOrder(request);
            step3.setStatus("SUCCESS");
            
            // 全部成功
            context.addStep(step1);
            context.addStep(step2);
            context.addStep(step3);
            saveSagaLog(sagaId, "COMPLETED", context);
            
            return SagaResult.success(order);
            
        } catch (Exception e) {
            log.error("[Saga失败] sagaId={}, error={}", sagaId, e.getMessage());
            // 执行补偿
            compensate(context);
            saveSagaLog(sagaId, "COMPENSATED", context);
            return SagaResult.fail(e.getMessage());
        }
    }
    
    /**
     * 补偿：逆序回滚已完成的步骤
     */
    private void compensate(SagaContext context) {
        // 逆序执行补偿
        List<SagaStep> steps = context.getSteps();
        for (int i = steps.size() - 1; i >= 0; i--) {
            SagaStep step = steps.get(i);
            if ("SUCCESS".equals(step.getStatus())) {
                try {
                    compensateStep(step, context);
                    step.setCompensateStatus("SUCCESS");
                } catch (Exception e) {
                    log.error("[补偿失败] step={}", step.getName(), e);
                    step.setCompensateStatus("FAILED");
                    step.setError(e.getMessage());
                }
            }
        }
    }
    
    private void compensateStep(SagaStep step, SagaContext context) {
        log.info("[执行补偿] step={}", step.getName());
        
        switch (step.getName()) {
            case "DEDUCT_BALANCE":
                // 补偿：退回余额
                accountService.refundBalance(
                    context.getUserId(), context.getTotalAmount());
                break;
                
            case "DEDUCT_STOCK":
                // 补偿：返还库存
                stockService.refundStock(
                    context.getProductId(), context.getQuantity());
                break;
                
            case "CREATE_ORDER":
                // 补偿：删除订单
                orderService.deleteOrder(context.getOrderId());
                break;
        }
    }
    
    private void saveSagaLog(String sagaId, String status, SagaContext context) {
        SagaLog log = new SagaLog();
        log.setSagaId(sagaId);
        log.setStatus(status);
        log.setContext(JSON.toJSONString(context));
        log.setCreateTime(LocalDateTime.now());
        sagaLogMapper.insert(log);
    }
}

/**
 * 定时任务：扫描失败的 Saga，尝试重新补偿
 */
@Component
@Slf4j
public class SagaRetryTask {
    
    @Autowired private SagaLogMapper sagaLogMapper;
    @Autowired private SagaOrderCreator sagaOrderCreator;
    
    @Scheduled(fixedDelay = 60000)  // 每分钟执行
    public void retryFailedSagas() {
        List<SagaLog> failedSagas = sagaLogMapper.selectFailedSagas();
        
        for (SagaLog saga : failedSagas) {
            log.info("[Saga重试] sagaId={}", saga.getSagaId());
            
            // 最多重试 3 次
            if (saga.getRetryCount() >= 3) {
                log.warn("[Saga重试次数超限，告警] sagaId={}", saga.getSagaId());
                alertService.sendAlert("Saga补偿失败", saga);
                continue;
            }
            
            try {
                SagaContext context = JSON.parseObject(
                    saga.getContext(), SagaContext.class);
                
                // 重新执行补偿
                sagaOrderCreator.compensate(context);
                sagaLogMapper.updateStatus(saga.getSagaId(), "COMPENSATED");
                
            } catch (Exception e) {
                log.error("[Saga重试失败] sagaId={}", saga.getSagaId(), e);
                sagaLogMapper.incrementRetryCount(saga.getSagaId());
            }
        }
    }
}
```

```java
/**
 * 本地消息表：存储待发送消息

@Entity
@Table(name = "local_message")
public class LocalMessage {

    private String messageId;
    
    private String topic;           // MQ Topic
    private String messageBody;    // 消息内容
    
    private String bizType;         // 业务类型：ORDER_CREATE
    private String bizId;           // 业务ID：订单号
    
    private String status;          // PREPARE/SENT/CONSUMED/FAILED
    private Integer retryCount;     // 重试次数
    
    private LocalDateTime createTime;
    private LocalDateTime updateTime;
    private LocalDateTime sentTime;
    private LocalDateTime consumedTime;
}

/**
 * 订单服务：本地消息表实现可靠消息
 */
@Service
@Slf4j
public class OrderServiceWithLocalMessage {
    
    @Autowired private OrderMapper orderMapper;
    @Autowired private LocalMessageMapper messageMapper;
    @Autowired private RabbitTemplate rabbitTemplate;
    
    /**
     * 创建订单（本地事务 + 消息表）
     */
    @Transactional
    public Order createOrderWithMessage(CreateOrderRequest request) {
        
        // 1. 创建订单
        Order order = new Order();
        order.setUserId(request.getUserId());
        order.setTotalAmount(request.getTotalAmount());
        order.setStatus("PENDING");
        order.setCreateTime(LocalDateTime.now());
        orderMapper.insert(order);
        
        // 2. 生成消息 ID
        String messageId = UUID.randomUUID().toString();
        
        // 3. 构造扣减消息
        StockDeductMessage stockMsg = new StockDeductMessage();
        stockMsg.setOrderId(order.getId());
        stockMsg.setProductId(request.getProductId());
        stockMsg.setQuantity(request.getQuantity());
        
        // 4. 写入本地消息表（与订单在同一事务）
        LocalMessage message = new LocalMessage();
        message.setMessageId(messageId);
        message.setTopic("stock.deduct.topic");
        message.setMessageBody(JSON.toJSONString(stockMsg));
        message.setBizType("ORDER_CREATE");
        message.setBizId(order.getId().toString());
        message.setStatus("PREPARE");
        message.setRetryCount(0);
        message.setCreateTime(LocalDateTime.now());
        messageMapper.insert(message);
        
        // 事务提交后，消息状态仍为 PREPARE
        
        // 5. 发送库存扣减消息（同步发送）
        try {
            rabbitTemplate.convertAndSend(
                "stock.deduct.topic", 
                stockMsg,
                m -> {
                    m.getMessageProperties().setMessageId(messageId);
                    return m;
                });
            
            // 6. 更新消息状态为 SENT
            messageMapper.updateStatus(messageId, "SENT");
            
        } catch (Exception e) {
            log.error("[发送消息失败] messageId={}", messageId, e);
            // 发送失败，不抛异常，依赖定时任务补偿
            messageMapper.updateStatus(messageId, "FAILED");
        }
        
        return order;
    }
}

/**
 * 库存服务：消费消息
 */
@Service
@Slf4j
public class StockConsumerService {
    
    @Autowired private StockMapper stockMapper;
    @Autowired private LocalMessageMapper messageMapper;
    @Autowired private RabbitTemplate rabbitTemplate;
    
    @RabbitListener(queues = "stock.deduct.queue", concurrency = "5")
    public void handleStockDeduct(StockDeductMessage message, Channel channel,
                                  @Header(AmqpHeaders.DELIVERY_TAG) long tag) {
        String messageId = message.getMessageId();
        
        try {
            // 1. 幂等检查
            LocalMessage localMsg = messageMapper.findByMessageId(messageId);
            if (localMsg != null && "CONSUMED".equals(localMsg.getStatus())) {
                log.info("[重复消费，跳过] messageId={}", messageId);
                channel.basicAck(tag, false);
                return;
            }
            
            // 2. 扣减库存
            int affected = stockMapper.deductStock(
                message.getProductId(), message.getQuantity());
            
            if (affected == 0) {
                // 库存不足，发补偿消息
                log.error("[库存不足] messageId={}", messageId);
                rabbitTemplate.convertAndSend(
                    "order.rollback.topic", 
                    new OrderRollbackMessage(message.getOrderId(), "STOCK_NOT_ENOUGH"));
                channel.basicAck(tag, false);
                return;
            }
            
            // 3. 更新消息状态为已消费
            messageMapper.updateStatus(messageId, "CONSUMED");
            messageMapper.updateConsumedTime(messageId, LocalDateTime.now());
            
            // 4. 发送库存扣减确认
            rabbitTemplate.convertAndSend(
                "stock.deduct.confirm.topic", 
                new StockDeductConfirm(message.getOrderId(), message.getProductId()));
            
            channel.basicAck(tag, false);
            log.info("[库存扣减成功] messageId={}, productId={}", 
                messageId, message.getProductId());
            
        } catch (Exception e) {
            log.error("[库存扣减失败] messageId={}", messageId, e);
            // 拒绝消息，重新入队重试
            channel.basicNack(tag, false, true);
        }
    }
}

/**
 * 定时任务：扫描未发送的消息
 */
@Component
@Slf4j
public class MessageRetryTask {
    
    @Autowired private LocalMessageMapper messageMapper;
    @Autowired private RabbitTemplate rabbitTemplate;
    
    @Scheduled(fixedDelay = 30000)  // 每 30 秒执行
    public void retryUnsentMessages() {
        // 1. 找出 PREPARE 或 FAILED 状态的消息
        List<LocalMessage> messages = messageMapper.selectPendingMessages();
        
        for (LocalMessage message : messages) {
            try {
                // 2. 检查重试次数
                if (message.getRetryCount() >= 5) {
                    log.warn("[消息重试次数超限] messageId={}", message.getMessageId());
                    messageMapper.updateStatus(message.getMessageId(), "FAILED");
                    alertService.sendAlert("消息发送失败", message);
                    continue;
                }
                
                // 3. 重新发送
                rabbitTemplate.convertAndSend(
                    message.getTopic(), 
                    JSON.parseObject(message.getMessageBody(), Object.class));
                
                // 4. 更新状态
                messageMapper.updateStatus(message.getMessageId(), "SENT");
                messageMapper.incrementRetryCount(message.getMessageId());
                
                log.info("[消息重发成功] messageId={}", message.getMessageId());
                
            } catch (Exception e) {
                log.error("[消息重发失败] messageId={}", message.getMessageId(), e);
                messageMapper.incrementRetryCount(message.getMessageId());
            }
        }
    }
}
```

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           方案选择决策树                                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   Q: 对一致性要求多高？                                                  │
│                                                                         │
│   高一致性                                                              │
│     │                                                                  │
│     ├── 微服务框架集成 Seata？ → AT 模式（自动回滚）                       │
│     │                                                                  │
│     └── 需要精细控制？ → TCC 模式（Try-Confirm-Cancel）                   │
│                                                                         │
│   最终一致性（可接受短暂不一致）                                          │
│     │                                                                  │
│     ├── 流程长（> 5 步）？ → Saga 编排模式                               │
│     │                                                                  │
│     └── 流程短（3-5 步）？ → 可靠消息 + 本地表                            │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

| 维度 | Seata AT | TCC | Saga | 可靠消息 |
|-----|---------|-----|------|---------|
| **一致性** | 强一致 | 强一致 | 最终一致 | 最终一致 |
| **侵入性** | 低 | 高 | 中 | 低 |
| **性能损耗** | 中 | 低 | 低 | 低 |
| **复杂度** | 低 | 高 | 中 | 中 |
| **失败处理** | 自动回滚 | 自动回滚 | 补偿 | 重试 |
| **适用场景** | 常规业务 | 金融/库存 | 长流程 | 大流量 |

## 六、面试加分话术

> "分布式事务没有银弹，关键是 **根据业务场景选择合适的方案**。
> 
> 我们线上用的是 **可靠消息 + 本地表** 的方案：
> 1. 在订单服务开启本地事务，订单写入和消息记录在同一个 DB 事务
> 2. 事务提交后，消息异步发送到 MQ
> 3. 库存服务、余额服务消费消息，幂等处理
> 4. 定时任务扫表，补偿发送失败的消息
> 
> 如果业务对一致性要求极高（比如余额），我会推荐 **Seata AT 模式**，加个 `@GlobalTransactional` 注解就够了，框架帮你管理回滚。
> 
> 关键点：**不要追求强一致性而牺牲可用性**，大部分业务场景最终一致性就够了。"

## 扩展追问

### Q9-1: 如果 MQ 消息丢失怎么办？

```java
// 方案：RocketMQ 事务消息
@Service
public class TransactionalMessageService {
    
    @Autowired private RocketMQTemplate rocketMQ;
    
    /**
     * 事务消息：扣减余额
     */
    public void sendDeductBalanceMessage(Long userId, BigDecimal amount) {
        DeductBalanceMessage message = new DeductBalanceMessage();
        message.setUserId(userId);
        message.setAmount(amount);
        
        // 发送事务消息
        TransactionSendResult result = rocketMQ.sendMessageInTransaction(
            "balance.deduct.topic",
            MessageBuilder.withPayload(message).build(),
            userId  // 透传参数，用于本地事务
        );
        
        log.info("[事务消息发送] userId={}, status={}", 
            userId, result.getSendStatus());
    }
    
    /**
     * RocketMQ 事务监听器
     */
    @Component
    @RocketMQTransactionListener
    public class BalanceTransactionListener 
        implements RocketMQLocalTransactionListener {
        
        @Override
        public RocketMQLocalTransactionState executeLocalTransaction(
            Message msg, Object arg) {
            Long userId = (Long) arg;
            
            try {
                // 执行本地扣款（在事务中）
                accountService.deductBalanceLocal(userId);
                return RocketMQLocalTransactionState.COMMIT;
                
            } catch (Exception e) {
                log.error("[本地事务失败] userId={}", userId, e);
                return RocketMQLocalTransactionState.ROLLBACK;
            }
        }
        
        @Override
        public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
            // 回查：检查扣款是否成功
            DeductBalanceMessage message = parseMessage(msg);
            
            Boolean deducted = accountService.isBalanceDeducted(message.getUserId());
            
            return deducted 
                ? RocketMQLocalTransactionState.COMMIT
                : RocketMQLocalTransactionState.ROLLBACK;
        }
    }
}
```

**一句话收尾**：分布式事务的本质是 **"接受最终一致，用补偿换可用"**——TCC 适合强一致场景，Saga 适合长流程，可靠消息适合大流量高并发。
