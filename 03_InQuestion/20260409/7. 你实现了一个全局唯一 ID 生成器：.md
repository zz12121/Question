**7. 你实现了一个"全局唯一 ID 生成器"：**

```java
public class IdGenerator {
    private static long sequence = 0;
    public static synchronized long nextId() {
        return System.currentTimeMillis() * 10000 + (sequence++ % 10000);
    }
}
```

**问题：** 高并发下有什么问题？ID 会重复吗？性能如何？如何改进？
## 先说结论

这段代码在**单机**下 ID 不会重复，但有三个致命问题：`synchronized` 全局锁性能差、时钟回拨导致 ID 重复、多实例部署必然冲突。生产环境直接上**雪花算法（Snowflake）**或美团 **Leaf**。

---

## 🔥 核心概念——逐行拆解这段代码的问题

```java
public static synchronized long nextId() {
    //          ↑ 问题①：全局锁，所有线程串行，高并发下吞吐量惨烈
    
    return System.currentTimeMillis() * 10000 + (sequence++ % 10000);
    //     ↑ 问题②：时钟回拨时 currentTimeMillis 变小 → ID 变小 → 重复
    //                                          ↑ 问题③：sequence 是 static long，
    //                                            多实例部署时各自从0开始 → 不同机器可能生成相同ID
}
```

---

## 🔥 详细展开

### 问题一：`synchronized` 性能瓶颈

```
单线程基准：nextId() 约 50~100 ns/次 → 理论 TPS ~ 1000万/s

但加了 synchronized 之后：
  - 线程竞争锁 → 大量线程挂起/唤醒（OS 上下文切换）
  - 实际测试：100 个线程并发 → 吞吐量跌至 50~200万/s
  - CPU 利用率高，但有效计算少（都在等锁）

更严重的是：这是一个"热点方法"
  每一次业务操作（下单、创建用户）都要经过这里
  → 整个系统的性能天花板被这一把锁卡死
```

### 问题二：时钟回拨导致 ID 重复 ⚠️ 最严重

```
正常时序：
  T=1000ms → ID: 10000000, 10000001, ..., 10009999
  T=1001ms → ID: 10010000, 10010001, ...

时钟回拨（NTP 校时 / 闰秒 / 服务器漂移）：
  T=1001ms → ID: 10010000
  T=1000ms ← NTP 回拨！currentTimeMillis 变小
  T=1000ms → ID: 10000000  ← 和之前的 ID 重复！
                               → 数据库唯一索引冲突
                               → 分布式系统数据错乱
```

这不是小概率事件，IDC 机房每天都有 NTP 同步，**每次都可能回拨几毫秒到几百毫秒**。

### 问题三：多实例部署 ID 必然冲突

```
服务部署了 3 个实例：
  实例A: sequence=0 → nextId() = 1000 * 10000 + 0 = 10000000
  实例B: sequence=0 → nextId() = 1000 * 10000 + 0 = 10000000  ← 完全相同！
  实例C: sequence=0 → nextId() = 1000 * 10000 + 0 = 10000000  ← 完全相同！
```

三台机器在同一毫秒内生成的 ID 完全一样，微服务架构下这是必然发生的。

### 问题四：sequence 溢出边界

```java
sequence++ % 10000
// 同一毫秒内最多支持 10000 个 ID（0~9999）
// 如果同一毫秒并发超过 10000 次 nextId() 调用：
//   sequence = 10000 → 10000 % 10000 = 0 → 和 sequence=0 时的 ID 重复！
// 且 sequence 是 long，永远递增，最终溢出（虽然需要很久）
```

---

## 🔥 扩展追问 1：雪花算法（Snowflake）如何解决这些问题？

### 雪花算法结构（64 bit）

```
 0 | 0000000000 0000000000 0000000000 0000000000 0 | 0000000000 | 000000000000
 ↑        41 位时间戳（毫秒）                         10位机器ID    12位序列号
符号位
（始终为0）

时间戳：当前毫秒 - 自定义纪元（如 2020-01-01）
  → 可用约 69 年

机器ID（10bit = 1024台）：
  → 高5位=数据中心ID，低5位=机器ID
  → 解决多实例冲突问题

序列号（12bit = 4096）：
  → 同一毫秒内最多生成 4096 个不重复 ID
  → 超出则等待下一毫秒

时钟回拨处理：
  → 记录上次时间戳，若 currentTime < lastTime 则抛异常或等待
```

### 标准 Snowflake 实现

```java
public class SnowflakeIdGenerator {
    // 自定义纪元（2020-01-01 00:00:00 UTC）
    private static final long EPOCH = 1577836800000L;

    private static final long MACHINE_ID_BITS   = 10L;
    private static final long SEQUENCE_BITS     = 12L;
    private static final long MAX_MACHINE_ID    = ~(-1L << MACHINE_ID_BITS);  // 1023
    private static final long MAX_SEQUENCE      = ~(-1L << SEQUENCE_BITS);    // 4095

    private static final long MACHINE_ID_SHIFT  = SEQUENCE_BITS;              // 12
    private static final long TIMESTAMP_SHIFT   = SEQUENCE_BITS + MACHINE_ID_BITS; // 22

    private final long machineId;
    private long lastTimestamp = -1L;
    private long sequence = 0L;

    public SnowflakeIdGenerator(long machineId) {
        if (machineId > MAX_MACHINE_ID || machineId < 0) {
            throw new IllegalArgumentException("machineId 超出范围: " + MAX_MACHINE_ID);
        }
        this.machineId = machineId;
    }

    public synchronized long nextId() {
        long now = System.currentTimeMillis();

        // ① 时钟回拨检测
        if (now < lastTimestamp) {
            long offset = lastTimestamp - now;
            if (offset <= 5) {
                // 回拨小于5ms，等一等
                try { Thread.sleep(offset * 2); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
                now = System.currentTimeMillis();
            }
            if (now < lastTimestamp) {
                throw new RuntimeException("时钟回拨超过5ms，拒绝生成ID: " + offset + "ms");
            }
        }

        // ② 同一毫秒内序列号递增
        if (now == lastTimestamp) {
            sequence = (sequence + 1) & MAX_SEQUENCE; // 用 & 代替 %，更快
            if (sequence == 0) {
                // 序列号溢出，等到下一毫秒
                now = waitNextMillis(lastTimestamp);
            }
        } else {
            // 新的毫秒，序列号重置
            sequence = 0L;
        }

        lastTimestamp = now;

        // ③ 组装 64 位 ID
        return ((now - EPOCH) << TIMESTAMP_SHIFT)
             | (machineId << MACHINE_ID_SHIFT)
             | sequence;
    }

    private long waitNextMillis(long lastTimestamp) {
        long ts = System.currentTimeMillis();
        while (ts <= lastTimestamp) {
            ts = System.currentTimeMillis();
        }
        return ts;
    }
}
```

### 单机理论上限

```
每毫秒最多 4096 个 ID
每秒 = 4096 × 1000 = 409.6 万 ID/s
实际加上 synchronized 竞争，约 200~300 万/s
→ 对绝大多数业务完全够用
```

## 🔥 扩展追问 2：如何去掉 synchronized，进一步提升性能？

### 方案：分段 ID + 线程本地序列（无锁）

```java
// 思路：给每个线程分配独立的序列号段，彻底消除锁竞争
public class SegmentIdGenerator {
    private final AtomicLong globalSequence = new AtomicLong(0);
    private static final int SEGMENT_SIZE = 1000; // 每次从全局申请1000个

    // 每个线程持有自己的号段
    private final ThreadLocal<long[]> localSegment = ThreadLocal.withInitial(
        () -> allocateSegment()
    );

    private long[] allocateSegment() {
        long start = globalSequence.getAndAdd(SEGMENT_SIZE);
        return new long[]{start, start + SEGMENT_SIZE, start}; // [min, max, current]
    }

    public long nextId() {
        long[] seg = localSegment.get();
        if (seg[2] >= seg[1]) {
            // 当前号段用完，重新申请
            long[] newSeg = allocateSegment();
            seg[0] = newSeg[0]; seg[1] = newSeg[1]; seg[2] = newSeg[2];
        }
        return seg[2]++;
    }
}
```

## 🔥 扩展追问 3：机器 ID 怎么分配？（分布式部署时）

```
问题：部署 100 台服务，每台机器 ID 怎么保证唯一？

方案一：手动配置（简单但运维成本高）
  → application.yml: snowflake.machine-id: 1
  → 容器化部署时 K8s StatefulSet 用 Pod 序号

方案二：ZooKeeper 自动分配
  → 服务启动时在 ZK 上创建临时顺序节点
  → 节点序号就是机器ID（范围0~1023）
  → 服务下线时临时节点自动删除，ID 被回收

方案三：Redis INCR
  → 服务启动时 INCR machine:id:counter
  → 取模 1024 作为机器ID
  → 简单，但需要保证 Redis 高可用

方案四：IP 后两段取模
  → machineId = (ip3 * 256 + ip4) % 1024
  → 存在碰撞风险（IP 相同尾段时）
```

## 🔥 扩展追问 4：美团 Leaf、百度 UidGenerator 解决了哪些额外问题？

| 特性 | 原始Snowflake | 美团 Leaf | 百度 UidGenerator |
|------|--------------|-----------|------------------|
| 时钟回拨 | 抛异常/等待 | ✅ 借用未来时间 | ✅ 离线预分配，完全规避 |
| 机器ID管理 | 手动/ZK | ✅ DB自动分配 | ✅ DB自动分配 |
| 号段模式 | ❌ | ✅ DB号段双buffer | ❌ |
| 性能 | ~200万/s | ~500万/s | ~600万/s |
| 依赖 | 无 | DB + ZK | DB |

**Leaf 双 Buffer 号段模式：**

```
DB 中存一张表：
  biz_tag | max_id  | step  | version
  order   | 1000000 | 10000 | 1

Leaf 从 DB 取号段 [990001, 1000000]（step=10000）
当用到 50% 时，异步预加载下一个号段
→ 始终有两个号段在内存（双 Buffer）
→ DB 每次只在号段用半时才访问，QPS 极低
→ 即使 DB 短暂不可用，还有一个号段可以撑着
```

## 一句话收尾

> 原始方案是"能用但禁不起推敲"——单机用**雪花算法**（解决时钟回拨 + 无符号位冲突），多机用**美团 Leaf**（DB 管机器 ID + 双 Buffer 号段），核心是把"全局串行"变成"预分配 + 本地生成"。
