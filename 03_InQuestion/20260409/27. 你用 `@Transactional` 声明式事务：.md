**27. 你用 `@Transactional` 声明式事务：**

```java
@Transactional
public void createOrder(Order order) {
    orderDao.save(order);
    inventoryService.deduct(order.getProductId());
    if (order.getAmount() > 1000) {
        throw new IllegalArgumentException("金额超限");
    }
}
```

**问题：** 如果 `inventoryService.deduct()` 是远程调用，事务能回滚吗？为什么？如何设计才能保证一致性？

---
## 先说结论

`inventoryService.deduct()` 是远程调用时，**本地事务能回滚，但远程库存扣减不会回滚**，导致数据不一致。根本原因是本地事务（如 JDBC）和远程调用（如 HTTP/RPC）不在同一个分布式事务中。要保证一致性，需引入 **分布式事务方案**（Seata AT/TCC/Saga）或采用 **最终一致性设计**（本地事务 + 消息队列异步补偿）。

---

## 🔥 核心概念

**解决什么问题：**
- 单体应用内多数据源操作的原子性
- 分布式环境下跨服务操作的一致性保证

**基本原理：**
- `@Transactional` 基于 Spring AOP + ThreadLocal 管理事务上下文
- 只作用于当前线程内的数据库连接，无法传播到远程服务

---

## 🔥 详细展开

### 1. 为什么远程调用无法回滚？

```java
@Transactional
public void createOrder(Order order) {
    orderDao.save(order);  // ① 本地DB操作，在事务中
    
    // ② 远程调用，不在本地事务上下文！
    inventoryService.deduct(order.getProductId());  
    
    if (order.getAmount() > 1000) {
        throw new IllegalArgumentException("金额超限");  // ③ 抛异常
    }
}
```

**执行流程：**

| 步骤 | 操作 | 是否可回滚 |
|------|------|-----------|
| ① | `orderDao.save()` | ✅ 可以，在本地事务中 |
| ② | `inventoryService.deduct()` | ❌ 不可以，已提交到远程服务 |
| ③ | 抛异常 | 只触发本地事务回滚 |

**结果：** 订单没创建，但库存已扣减 → **数据不一致！**

### 2. 事务传播的本质限制

```java
// Spring 事务管理器
public class DataSourceTransactionManager {
    protected void doBegin(Object transaction, TransactionDefinition definition) {
        // 只绑定当前线程的数据库连接
        Connection con = dataSource.getConnection();
        TransactionSynchronizationManager.bindResource(dataSource, con);
    }
}
```

**关键限制：**
- 事务上下文通过 `ThreadLocal` 存储
- 只作用于当前 JVM 内的数据库连接
- 无法跨进程、跨网络传播事务状态

### 3. 分布式事务解决方案

#### 方案 A：Seata AT 模式（推荐，对业务侵入小）

```java
@GlobalTransactional  // Seata 全局事务
public void createOrder(Order order) {
    orderDao.save(order);  // 分支事务1
    
    // 远程调用，Seata 自动传播 XID
    inventoryService.deduct(order.getProductId());  // 分支事务2
}
```

**原理：**
- TC（Transaction Coordinator）协调全局事务
- TM（Transaction Manager）开启全局事务
- RM（Resource Manager）管理分支事务
- 基于 Undo Log 实现回滚

**架构：**
```
┌─────────┐     ┌─────────┐     ┌─────────┐
│   TM    │────→│   TC    │←────│   RM1   │ (订单库)
│(业务应用)│     │(协调器) │     └─────────┘
└─────────┘     └────┬────┘     ┌─────────┘
                     │          │   RM2   │ (库存服务)
                     └─────────→│(库存库) │
                                └─────────┘
```

#### 方案 B：TCC 模式（性能更好，业务侵入大）

```java
public interface InventoryTccAction {
    
    @TwoPhaseBusinessAction(name = "deductInventory")
    boolean tryDeduct(@BusinessActionContextParameter(paramName = "productId") String productId,
                      @BusinessActionContextParameter(paramName = "count") int count);
    
    boolean commit(BusinessActionContext context);
    
    boolean rollback(BusinessActionContext context);
}
```

**三阶段：**
- **Try：** 预留资源（如冻结库存）
- **Confirm：** 确认执行（真正扣减）
- **Cancel：** 取消回滚（释放预留）

#### 方案 C：Saga 模式（长事务，适合业务流程）

```java
@SagaStart
public void createOrder(Order order) {
    sagaOrchestrator.start()
        .step("saveOrder", () -> orderService.save(order))
        .step("deductInventory", () -> inventoryService.deduct(order))
        .compensate("saveOrder", () -> orderService.cancel(order))
        .compensate("deductInventory", () -> inventoryService.rollback(order))
        .execute();
}
```

### 4. 最终一致性方案（更常用）

**核心思想：** 本地事务保证原子性，通过消息队列异步保证最终一致

```java
@Transactional
public void createOrder(Order order) {
    // 1. 保存订单
    orderDao.save(order);
    
    // 2. 发送扣减库存消息（本地事务表保证消息不丢）
    messageService.sendAfterCommit(
        "deduct_inventory", 
        new InventoryMessage(order.getProductId(), order.getCount())
    );
}

// 消息消费者
@RocketMQMessageListener(topic = "deduct_inventory")
public class InventoryConsumer {
    @Override
    public void onMessage(Message message) {
        // 幂等性保证
        inventoryService.deduct(message.getProductId(), message.getCount());
    }
}
```

**本地事务表模式：**

```java
@Transactional
public void createOrder(Order order) {
    orderDao.save(order);
    
    // 同一事务中写入消息表
    messageDao.insert(new MessageRecord(
        UUID.randomUUID().toString(),
        "deduct_inventory",
        json(order),
        "PENDING"
    ));
}

// 定时扫描发送
@Scheduled(fixedRate = 5000)
public void scanPendingMessages() {
    List<MessageRecord> pending = messageDao.selectByStatus("PENDING");
    pending.forEach(msg -> {
        rocketMQTemplate.asyncSend(msg.getTopic(), msg.getPayload(), new SendCallback() {
            @Override
            public void onSuccess(SendResult result) {
                messageDao.updateStatus(msg.getId(), "SENT");
            }
        });
    });
}
```

## 🔥 扩展追问 1：`@Transactional` 方法内部调用另一个 `@Transactional` 方法，事务怎么传播？

```java
@Service
public class OrderService {
    
    @Transactional
    public void createOrder(Order order) {
        orderDao.save(order);
        
        // 内部调用，事务传播行为？
        this.updateStatistics(order);
    }
    
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void updateStatistics(Order order) {
        statisticsDao.update(order);
    }
}
```

**答案：`REQUIRES_NEW` 不生效！**

原因还是 AOP 代理问题，内部调用不走代理，直接调用目标方法，传播行为失效。

**解决：**
```java
@Autowired
private OrderService self;  // 注入自身代理

@Transactional
public void createOrder(Order order) {
    orderDao.save(order);
    self.updateStatistics(order);  // 走代理，传播行为生效
}
```

## 🔥 扩展追问 2：事务超时怎么设置？什么场景会触发？

```java
@Transactional(timeout = 5)  // 5秒超时
public void createOrder(Order order) {
    // 模拟慢查询
    Thread.sleep(10000);  // 超时！
    orderDao.save(order);
}
```

**超时判断时机：**
- 在 SQL 执行前检查是否超时
- 不是方法执行总时长限制

**常见超时场景：**
- 慢 SQL（没索引、大表扫描）
- 锁等待（行锁、间隙锁）
- 网络延迟（数据库连接慢）

**最佳实践：**
```java
@Transactional(timeout = 3, rollbackFor = Exception.class)
public void createOrder(Order order) {
    // 业务逻辑尽量前置，减少事务范围
    validate(order);
    
    // 事务内只保留必要的数据库操作
    orderDao.save(order);
}
```

## 🔥 扩展追问 3：Spring 事务失效的常见场景有哪些？

| 场景 | 原因 | 解决 |
|------|------|------|
| 同类内部调用 | 不走代理 | 注入自身或使用 AopContext |
| 方法非 public | AOP 无法代理 | 改为 public |
| 异常被 catch | 未抛给事务管理器 | catch 后重新抛出或手动回滚 |
| 异步方法 `@Async` | 不同线程，事务不传播 | 合并到同一线程或分布式事务 |
| 多数据源 | 事务管理器不匹配 | 指定正确的事务管理器 |
| 数据库不支持事务 | MyISAM 等引擎 | 改用 InnoDB |

**手动回滚示例：**
```java
@Transactional
public void createOrder(Order order) {
    try {
        riskyOperation();
    } catch (Exception e) {
        // 手动标记回滚
        TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
        // 或者抛出自定义异常
        throw new BusinessException("操作失败", e);
    }
}
```
