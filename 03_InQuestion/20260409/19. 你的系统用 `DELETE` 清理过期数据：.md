**19. 你的系统用 `DELETE` 清理过期数据：**

```sql
DELETE FROM logs WHERE create_time < DATE_SUB(NOW(), INTERVAL 30 DAY);
```

**问题：** 如果 `logs` 表有 10 亿条数据，这条 SQL 会有什么问题？如何安全地清理大量历史数据？
## 先说结论

单条DELETE清理10亿数据会**锁表阻塞**、**undo log暴涨**、**主从延迟**，甚至**OOM**。安全方案：**分批删除**、**分区表直接DROP分区**、**归档后TRUNCATE**、或**pt-archiver工具**。

---

## 🔥 核心概念——大DELETE的问题

```
DELETE执行过程：
  1. 扫描满足条件的记录（全表扫描或索引扫描）
  2. 对每行加X锁（排他锁）
  3. 写undo log（用于回滚）
  4. 写redo log
  5. 标记删除（实际空间不释放，只是标记）
  6. 提交后，purge线程异步清理

10亿数据的问题：
  - 锁持有时间极长（小时级）→ 业务阻塞
  - undo log文件暴涨 → 磁盘满
  - 主从复制延迟 → 从库数据不一致
  - buffer pool被污染 → 热点数据被挤出
```

---

## 🔥 详细展开

### 问题一：锁表阻塞

```sql
-- 10亿数据的DELETE，可能执行几小时
DELETE FROM logs WHERE create_time < '2024-01-01';
-- 期间：
--   - 所有写操作被阻塞（waiting for lock）
--   - 读操作如果走锁也可能被阻塞
--   - 业务完全不可用
```

### 问题二：Undo Log暴涨

```
InnoDB Undo Log机制：
  - DELETE操作需要记录undo log，用于事务回滚
  - 10亿条记录 → 大量undo log
  - undo log文件可能涨到几百GB
  - 磁盘满 → 数据库崩溃
```

### 问题三：主从延迟

```sql
-- 主库执行几小时的DELETE
-- binlog记录所有删除操作
-- 从库单线程重放，延迟可能几小时甚至天级
-- 主从切换时数据不一致
```

### 问题四：空间不释放

```sql
-- DELETE只是标记删除，空间不立即释放
-- 需要OPTIMIZE TABLE或重建表才能回收空间
-- 10亿数据的表，OPTIMIZE可能需要几天
```

## 🔥 扩展追问 1：分批删除（LIMIT）

```sql
-- 分批删除，每次删1000条
REPEAT
    DELETE FROM logs 
    WHERE create_time < '2024-01-01' 
    LIMIT 1000;
    SLEEP 0.1;  -- 暂停100ms，让出资源
UNTIL ROW_COUNT() = 0;
```

**Java实现：**

```java
@Component
public class DataCleaner {
    
    @Scheduled(cron = "0 0 2 * * *")  // 每天凌晨2点执行
    public void cleanExpiredLogs() {
        int batchSize = 1000;
        int deleted;
        
        do {
            deleted = jdbcTemplate.update(
                "DELETE FROM logs WHERE create_time < ? LIMIT ?",
                LocalDateTime.now().minusDays(30),
                batchSize
            );
            
            // 每批暂停，避免压力过大
            if (deleted > 0) {
                Thread.sleep(100);
            }
        } while (deleted == batchSize);  // 删满一批说明还有数据
    }
}
```

**优点：**
- 每次只锁1000条，时间短
- undo log量可控
- 主从延迟小

**缺点：**
- 整体清理时间长（10亿/1000 = 100万次循环）
- 需要长时间运行

## 🔥 扩展追问 2：分区表直接DROP（推荐）

```sql
-- 按时间分区，直接删除整个分区（瞬间完成）
CREATE TABLE logs (
    id BIGINT,
    create_time DATETIME,
    message TEXT
) PARTITION BY RANGE (YEAR(create_time) * 100 + MONTH(create_time)) (
    PARTITION p202401 VALUES LESS THAN (202402),
    PARTITION p202402 VALUES LESS THAN (202403),
    PARTITION p202403 VALUES LESS THAN (202404),
    -- ...
    PARTITION p_future VALUES LESS THAN MAXVALUE
);

-- 清理2024年1月数据（瞬间完成，不锁表）
ALTER TABLE logs DROP PARTITION p202401;

-- 原理：
--   - 分区是独立的物理文件
--   - DROP PARTITION = 删除文件，瞬间完成
--   - 不扫描数据，不写undo log
```

**优点：**
- 毫秒级完成
- 不锁表
- 空间立即释放

**缺点：**
- 需要提前设计分区表
- 只能按分区粒度删除（不能精确到行）

## 🔥 扩展追问 3：归档后TRUNCATE

```sql
-- 方案：先归档到历史库，再TRUNCATE
-- 1. 创建归档表（不同实例或不同库）
CREATE TABLE archive.logs_2023 LIKE logs;

-- 2. 导出数据（用mysqldump或SELECT INTO OUTFILE）
-- 或增量同步到归档库

-- 3. 重命名表（原子操作）
RENAME TABLE logs TO logs_old, logs_new TO logs;

-- 4. 异步清理旧表
DROP TABLE logs_old;  -- 或TRUNCATE
```

## 🔥 扩展追问 4：逻辑删除替代物理删除

```sql
-- 不DELETE，只UPDATE标记
ALTER TABLE logs ADD COLUMN is_deleted TINYINT DEFAULT 0;

-- 清理时
UPDATE logs SET is_deleted = 1 
WHERE create_time < '2024-01-01';

-- 查询时
SELECT * FROM logs WHERE is_deleted = 0;

-- 定期重建表回收空间
-- 或按is_deleted分区，直接DROP已删除分区
```

## 一句话收尾

> 10亿数据DELETE是"自杀式操作"，会锁表、涨undo、主从延迟；安全方案用**分区表DROP**（毫秒级）、**分批LIMIT删除**（可控）、或**pt-archiver归档**，核心原则是**小步快跑、避免大事务**。
