**15. 你的订单表有 5000 万数据，查询很慢，你加了索引：**

```sql
CREATE INDEX idx_user_time ON orders(user_id, create_time);
```

**查询：** `SELECT * FROM orders WHERE user_id = 100 AND create_time > '2024-01-01'` **问题：** 这个索引一定能用到吗？什么情况下会失效？如果查询变成 `SELECT order_id` 性能有变化吗？
## 先说结论

这个联合索引**大概率能用**，但存在失效风险：数据类型隐式转换、函数操作、like 左模糊、数据倾斜导致优化器放弃索引。`SELECT order_id` 比 `SELECT *` 快，因为**覆盖索引**避免了回表。

---

## 🔥 核心概念——联合索引与最左前缀

```
索引结构：idx_user_time (user_id, create_time)

B+ 树排序规则：
  先按 user_id 排序
  user_id 相同，再按 create_time 排序

┌─────────────────────────────────────┐
│  (1, 2024-01-01)                    │
│  (1, 2024-01-02)                    │
│  (1, 2024-01-03)                    │
│  (2, 2024-01-01)  ← user_id=2 开始  │
│  (2, 2024-01-05)                    │
│  ...                                │
└─────────────────────────────────────┘

查询条件：user_id = 100 AND create_time > '2024-01-01'
  1. 先定位 user_id = 100 的位置（最左前缀命中）
  2. 在该范围内扫描 create_time > '2024-01-01'
  3. 符合最左前缀原则，索引有效
```

---

## 🔥 详细展开

### 问题一：索引失效的几种情况

```sql
-- 1. 隐式类型转换（user_id 是 int，传入字符串）
SELECT * FROM orders WHERE user_id = '100' AND create_time > '2024-01-01';
-- MySQL 会把 '100' 转成 100，但可能不走索引（取决于版本和优化器）

-- 2. 对索引列使用函数
SELECT * FROM orders WHERE CAST(user_id AS CHAR) = '100';
-- 函数操作导致索引失效

-- 3. 对索引列进行运算
SELECT * FROM orders WHERE user_id + 0 = 100;
-- 索引列参与运算，失效

-- 4. LIKE 左模糊（如果查询变成这样）
SELECT * FROM orders WHERE user_id LIKE '%100';
-- 左模糊无法使用索引

-- 5. 数据倾斜（某个 user_id 数据量极大）
-- 如果 user_id = 100 有 1000 万条记录
-- 优化器可能认为全表扫描比索引+回表更快
-- → 放弃索引，走全表扫描

-- 6. OR 条件（如果查询变成）
SELECT * FROM orders 
WHERE user_id = 100 OR user_id = 200;
-- OR 条件可能走索引，也可能优化器选择全表扫描
```

### 问题二：EXPLAIN 验证索引使用

```sql
EXPLAIN SELECT * FROM orders 
WHERE user_id = 100 AND create_time > '2024-01-01';

-- 关键字段：
-- type: range（范围扫描，好）| ALL（全表扫描，坏）
-- key: idx_user_time（使用的索引）
-- rows: 预估扫描行数（越小越好）
-- Extra: Using index condition（ICP，好）| Using where; Using filesort（坏）
```

## 🔥 扩展追问 1：SELECT * vs SELECT order_id 性能差异

```sql
-- 查询 A：SELECT *
SELECT * FROM orders 
WHERE user_id = 100 AND create_time > '2024-01-01';

执行过程：
  1. 在 idx_user_time 索引找到满足条件的记录
  2. 拿到主键 id
  3. 回表（根据主键 id 查聚簇索引）获取所有字段
  4. 返回结果
-- 需要回表，IO 次数多

-- 查询 B：SELECT order_id
SELECT order_id FROM orders 
WHERE user_id = 100 AND create_time > '2024-01-01';

执行过程：
  1. 在 idx_user_time 索引找到满足条件的记录
  2. 如果 order_id 也在索引中 → 直接返回（覆盖索引）
  3. 不需要回表
-- 如果 order_id 不在索引中，仍需回表
```

### 覆盖索引优化

```sql
-- 方案 1：调整索引顺序，包含需要查询的字段
CREATE INDEX idx_user_time_order ON orders(user_id, create_time, order_id);
-- 现在 (user_id, create_time, order_id) 都在索引中
-- SELECT order_id 成为覆盖索引，无需回表

-- 方案 2：包含更多常用字段（索引下推 ICP）
CREATE INDEX idx_user_time_cover ON orders(user_id, create_time, order_id, status, amount);
-- 常用查询字段都在索引中，避免回表
-- 代价：索引变大，写入变慢
```

## 🔥 扩展追问 2：5000万大表的特殊问题

### 问题一：索引选择度

```sql
-- 查看 user_id 的选择度
SELECT COUNT(DISTINCT user_id) / COUNT(*) FROM orders;
-- 结果接近 1：选择度高，索引效果好
-- 结果接近 0：选择度低，可能全表扫描更好

-- 如果某个 user_id 数据量特别大（如系统用户）
-- 可以单独处理，不走索引
SELECT * FROM orders 
WHERE user_id = 100 AND create_time > '2024-01-01'
  AND user_id != 0;  -- 排除系统用户，或单独处理
```

### 问题二：索引维护成本

```sql
-- 5000万数据的索引 B+ 树高度约 3-4 层
-- 每次插入/更新需要维护索引，成本较高

-- 优化：如果 create_time 更新频繁，考虑去掉
CREATE INDEX idx_user ON orders(user_id);
-- 只保留 user_id 索引，create_time 用于排序和过滤
-- 代价：需要扫描更多行，但减少了索引维护成本
```

### 问题三：分页查询深度分页问题

```sql
-- 深度分页，索引失效
SELECT * FROM orders 
WHERE user_id = 100 
ORDER BY create_time DESC 
LIMIT 1000000, 10;
-- 需要扫描 1000010 行，再丢弃前 1000000 行

-- 优化：延迟关联
SELECT * FROM orders o
JOIN (
    SELECT order_id FROM orders 
    WHERE user_id = 100 
    ORDER BY create_time DESC 
    LIMIT 1000000, 10
) tmp ON o.order_id = tmp.order_id;
-- 子查询只查索引，拿到 id 后再回表取完整数据
```

## 🔥 扩展追问 3：分区表优化

```sql
-- 5000万数据建议按时间分区
CREATE TABLE orders (
    order_id BIGINT PRIMARY KEY,
    user_id INT,
    create_time DATETIME,
    ...
) PARTITION BY RANGE (YEAR(create_time)) (
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);

-- 查询自动命中分区
SELECT * FROM orders 
WHERE user_id = 100 AND create_time > '2024-01-01';
-- 只扫描 p2024 和 p_future 分区
```

## 🔥 扩展追问 4：读写分离与归档

```sql
-- 1. 读写分离：查询走从库
-- 2. 冷热分离：历史数据归档
CREATE TABLE orders_archive LIKE orders;

-- 定时任务迁移历史数据
INSERT INTO orders_archive 
SELECT * FROM orders WHERE create_time < DATE_SUB(NOW(), INTERVAL 1 YEAR);

DELETE FROM orders WHERE create_time < DATE_SUB(NOW(), INTERVAL 1 YEAR);
-- 主表只保留 1 年数据，查询更快
```

## 一句话收尾

> 联合索引遵循**最左前缀**，但可能因**隐式转换、函数、数据倾斜**失效；`SELECT order_id` 如果命中**覆盖索引**则无需回表；5000万大表要配合**分区、归档、延迟关联**才能扛住高并发查询。
