**8. 你用 `CompletableFuture` 并行调用两个接口：**

```java
CompletableFuture<ResultA> fa = CompletableFuture.supplyAsync(() -> callA());
CompletableFuture<ResultB> fb = CompletableFuture.supplyAsync(() -> callB());
return fa.thenCombine(fb, (a, b) -> merge(a, b)).get();
```

**问题：** 如果 `callA()` 或 `callB()` 抛出异常，会发生什么？如何优雅处理超时和异常？
## 先说结论

默认情况下，任一接口抛出异常会导致整个 `CompletableFuture` 链异常终止，`get()` 会抛出 `ExecutionException` 包装原始异常。超时必须用 `orTimeout/completeOnTimeout`，异常用 `exceptionally/handle` 兜底，否则调用方永远挂起或报错。

---

## 🔥 核心概念——异常传播机制

```
CompletableFuture 链的异常传播规则：

1. 任一阶段抛出异常 → 后续所有阶段跳过正常逻辑，直接传播异常
2. 异常被包装在 ExecutionException 中（调用 get/join 时抛出）
3. 如果没有调用 get/join，异常被"吞掉"（不会打印堆栈！）

示例：
  fa.thenCombine(fb, merge) 中：
    - fa 异常 → thenCombine 直接返回异常 CF，不会执行 merge
    - fb 异常 → 同上
    - 两者都正常 → 执行 merge
```

## 🔥 详细展开

### 问题一：默认异常行为

```java
CompletableFuture<ResultA> fa = CompletableFuture.supplyAsync(() -> {
    if (true) throw new RuntimeException("A 接口挂了");
    return callA();
});

CompletableFuture<ResultB> fb = CompletableFuture.supplyAsync(() -> callB());

// 任一异常 → thenCombine 返回异常 CF
CompletableFuture<Result> fc = fa.thenCombine(fb, (a, b) -> merge(a, b));

try {
    Result result = fc.get();  // ← 抛 ExecutionException，cause = RuntimeException("A 接口挂了")
} catch (ExecutionException e) {
    Throwable cause = e.getCause();  // RuntimeException
    // 需要手动判断是哪个接口的问题
}
```

**痛点：**
- 不知道具体哪个接口异常
- 一个接口挂了，另一个接口的结果也拿不到（全部失败）
- 没有超时控制，`get()` 可能永久阻塞

### 问题二：没有超时 = 潜在死锁

```java
// 如果 callA() 内部网络请求卡死（比如对端服务无响应）
// supplyAsync 默认用 ForkJoinPool.commonPool()
// 线程被占着不释放，最终线程池耗尽
// 更严重：get() 永久阻塞，调用方超时
```

## 🔥 扩展追问 1：如何优雅处理单个接口异常？

### 方案一：exceptionally 兜底（返回默认值）

```java
CompletableFuture<ResultA> fa = CompletableFuture
    .supplyAsync(() -> callA())
    .exceptionally(ex -> {
        log.error("callA 异常", ex);
        return ResultA.empty();  // 返回空对象或默认值
    });

CompletableFuture<ResultB> fb = CompletableFuture
    .supplyAsync(() -> callB())
    .exceptionally(ex -> {
        log.error("callB 异常", ex);
        return ResultB.empty();
    });

// 即使其中一个异常，也能拿到另一个的结果
return fa.thenCombine(fb, (a, b) -> merge(a, b)).get();
```

### 方案二：handle 统一处理（同时处理正常和异常结果）

```java
CompletableFuture<ResultA> fa = CompletableFuture
    .supplyAsync(() -> callA())
    .handle((result, ex) -> {
        if (ex != null) {
            log.error("callA 失败，使用降级数据", ex);
            return ResultA.fallback();
        }
        return result;
    });
```

### 方案三：allOf + 手动检查（需要知道每个 CF 的状态）

```java
CompletableFuture<ResultA> fa = CompletableFuture.supplyAsync(() -> callA());
CompletableFuture<ResultB> fb = CompletableFuture.supplyAsync(() -> callB());

// 等待两个都完成（无论成功失败）
CompletableFuture<Void> all = CompletableFuture.allOf(fa, fb);
all.get();  // 等待完成

// 手动检查每个结果
ResultA a = fa.isCompletedExceptionally() ? ResultA.empty() : fa.getNow(null);
ResultB b = fb.isCompletedExceptionally() ? ResultB.empty() : fb.getNow(null);

return merge(a, b);
```

## 🔥 扩展追问 2：如何优雅处理超时？

### 方案一：orTimeout（Java 9+）——超时抛异常

```java
CompletableFuture<ResultA> fa = CompletableFuture
    .supplyAsync(() -> callA())
    .orTimeout(2, TimeUnit.SECONDS)  // 2秒超时，抛 TimeoutException
    .exceptionally(ex -> {
        if (ex instanceof TimeoutException) {
            log.warn("callA 超时");
        }
        return ResultA.empty();
    });
```

### 方案二：completeOnTimeout（Java 9+）——超时返回默认值

```java
CompletableFuture<ResultA> fa = CompletableFuture
    .supplyAsync(() -> callA())
    .completeOnTimeout(ResultA.empty(), 2, TimeUnit.SECONDS);
    // 超时自动返回 ResultA.empty()，不抛异常
```

### 方案三：Java 8 兼容方案——自定义超时包装

```java
public static <T> CompletableFuture<T> withTimeout(
        CompletableFuture<T> future, 
        long timeout, 
        TimeUnit unit) {
    
    CompletableFuture<T> timeoutFuture = failAfter(timeout, unit);
    
    // 两个 CF 竞争，谁先完成用谁的结果
    return future.applyToEither(timeoutFuture, Function.identity())
                 .exceptionally(ex -> {
                     // 区分是业务异常还是超时
                     if (ex.getCause() instanceof TimeoutException) {
                         return null; // 或默认值
                     }
                     throw new RuntimeException(ex);
                 });
}

private static <T> CompletableFuture<T> failAfter(long timeout, TimeUnit unit) {
    CompletableFuture<T> result = new CompletableFuture<>();
    Delayer.delay(() -> result.completeExceptionally(new TimeoutException()), timeout, unit);
    return result;
}
```

## 🔥 扩展追问 3：完整生产级代码

```java
public class ParallelService {
    
    private static final ExecutorService executor = Executors.newFixedThreadPool(
        Runtime.getRuntime().availableProcessors() * 2,
        new ThreadFactoryBuilder().setNameFormat("async-pool-%d").build()
    );

    public Result mergeParallel() {
        // ① 带超时 + 异常兜底的并行调用
        CompletableFuture<ResultA> fa = CompletableFuture
            .supplyAsync(this::callA, executor)
            .orTimeout(2000, TimeUnit.MILLISECONDS)
            .exceptionally(this::handleAException);

        CompletableFuture<ResultB> fb = CompletableFuture
            .supplyAsync(this::callB, executor)
            .orTimeout(2000, TimeUnit.MILLISECONDS)
            .exceptionally(this::handleBException);

        // ② 合并结果（任一失败也能拿到另一个的结果）
        try {
            return fa.thenCombine(fb, this::merge)
                     .get(3000, TimeUnit.MILLISECONDS);  // 整体超时兜底
        } catch (TimeoutException e) {
            log.error("整体合并超时");
            return Result.fallback();
        } catch (Exception e) {
            log.error("合并异常", e);
            return Result.fallback();
        }
    }

    private ResultA handleAException(Throwable ex) {
        if (ex instanceof TimeoutException || ex.getCause() instanceof TimeoutException) {
            Metrics.counter("callA.timeout").increment();
        } else {
            Metrics.counter("callA.error").increment();
        }
        log.warn("callA 失败，使用降级", ex);
        return ResultA.fallback();
    }

    // 优雅关闭线程池
    @PreDestroy
    public void shutdown() {
        executor.shutdown();
        try {
            if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
                executor.shutdownNow();
            }
        } catch (InterruptedException e) {
            executor.shutdownNow();
            Thread.currentThread().interrupt();
        }
    }
}
```

## 🔥 扩展追问 4：anyOf 和 allOf 的区别

```java
// allOf: 等待所有完成（无论成功失败）
CompletableFuture.allOf(fa, fb, fc).get();  // 阻塞直到三个都结束

// anyOf: 任意一个完成就返回（最快响应）
CompletableFuture<Object> any = CompletableFuture.anyOf(fa, fb, fc);
Object result = any.get();  // 拿到最先完成的那个结果

// 适用场景：
// allOf → 需要聚合多个接口结果（如页面渲染）
// anyOf → 多活容灾，多个服务同时调用，用最快返回的
```

## 一句话收尾

> `CompletableFuture` 异常会打断整个链条，必须用 `exceptionally/handle` 在每个阶段兜底，用 `orTimeout/completeOnTimeout` 防挂死，最后用 `get(timeout)` 做整体兜底——三层防护才能上生产。
