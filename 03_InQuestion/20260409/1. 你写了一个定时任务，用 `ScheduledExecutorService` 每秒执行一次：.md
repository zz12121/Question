**1. 你写了一个定时任务，用 `ScheduledExecutorService` 每秒执行一次：**

```java
executor.scheduleAtFixedRate(() -> {
    // 查询数据库，处理数据
    processData();
}, 0, 1, TimeUnit.SECONDS);
```

**问题：** 如果 `processData()` 某次执行超过 1 秒，会发生什么？有什么隐患？如何改进？
## 先说结论

`scheduleAtFixedRate` 的特点是**"固定频率执行"**——如果某次任务执行超过 1 秒，下一次任务会**立即补执行**，可能导致任务堆积、线程池耗尽、数据库连接打满。隐患是**任务重叠执行 + 资源雪崩**。改进方案是用 `scheduleWithFixedDelay`（固定间隔）或加**任务执行时间保护 + 分布式锁防重入**。

---

## 🔥 核心概念

- **解决什么问题**：定时任务执行时间不稳定时，防止任务堆积和资源竞争
- **scheduleAtFixedRate vs scheduleWithFixedDelay**：
  - `AtFixedRate`：固定频率，超时补执行，可能重叠
  - `WithFixedDelay`：固定间隔，前一次结束后再等 delay，不会重叠
- **关键隐患**：任务重叠执行、线程池耗尽、数据重复处理

---

## 🔥 详细展开

### 1. 问题演示

```java
ScheduledExecutorService executor = Executors.newScheduledThreadPool(1);

executor.scheduleAtFixedRate(() -> {
    System.out.println("开始执行: " + System.currentTimeMillis());
    
    // 模拟某次执行超过 1 秒
    try {
        Thread.sleep(1500);  // 执行 1.5 秒
    } catch (InterruptedException e) {
        e.printStackTrace();
    }
    
    System.out.println("结束执行: " + System.currentTimeMillis());
}, 0, 1, TimeUnit.SECONDS);
```

**输出**：
```
开始执行: 1000
开始执行: 2000    ← 1秒后，上一个还没结束，新任务又来了！
结束执行: 2500
开始执行: 3000    ← 继续堆积
...
```

**结果**：任务重叠执行，单线程池还好（排队），多线程池会并发执行同一任务。

---

### 2. 隐患分析

| 隐患 | 说明 | 后果 |
|------|------|------|
| **任务重叠** | 上一次没结束，下一次已经开始 | 数据重复处理、逻辑冲突 |
| **线程池耗尽** | 任务持续堆积，占用所有线程 | 其他定时任务无法执行 |
| **数据库连接打满** | 每个任务都查 DB，并发增加 | `Connection timeout` |
| **数据不一致** | 同一批数据被多个任务处理 | 重复计算、资金错误 |
| **内存溢出** | 任务队列无限增长 | `OutOfMemoryError` |

---

### 3. 改进方案

#### 方案一：改用 `scheduleWithFixedDelay`（推荐简单场景）

```java
// 前一次结束后再等 1 秒，保证不重叠
executor.scheduleWithFixedDelay(() -> {
    processData();
}, 0, 1, TimeUnit.SECONDS);
```

**输出**：
```
开始执行: 1000
结束执行: 2500
开始执行: 3500    ← 等 1 秒后再执行，不重叠
结束执行: 5000
```

> 💡 适合：任务可以串行执行，不需要严格固定频率

---

#### 方案二：加执行时间保护（推荐生产环境）

```java
@Component
public class SafeScheduledTask {
    
    private final AtomicBoolean running = new AtomicBoolean(false);
    
    @Scheduled(fixedRate = 1000)
    public void process() {
        // 1. 防重入：上一个没结束，跳过本次
        if (!running.compareAndSet(false, true)) {
            log.warn("上一次任务未结束，跳过本次执行");
            return;
        }
        
        try {
            // 2. 超时控制：最多执行 5 秒
            Future<?> future = executor.submit(this::doProcess);
            future.get(5, TimeUnit.SECONDS);  // 超时抛异常
            
        } catch (TimeoutException e) {
            log.error("任务执行超时", e);
            // 告警通知
        } catch (Exception e) {
            log.error("任务执行异常", e);
        } finally {
            running.set(false);
        }
    }
    
    private void doProcess() {
        // 业务逻辑
    }
}
```

#### 方案三：分布式锁（集群环境必须）

```java
@Component
public class DistributedScheduledTask {
    
    @Autowired
    private RedissonClient redisson;
    
    @Scheduled(fixedRate = 1000)
    public void process() {
        RLock lock = redisson.getLock("scheduled:processData");
        
        // 尝试获取锁，最多等 0 秒（不阻塞）
        boolean acquired = false;
        try {
            acquired = lock.tryLock(0, 30, TimeUnit.SECONDS);
            if (!acquired) {
                log.info("其他节点正在执行，跳过");
                return;
            }
            
            doProcess();
            
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            if (acquired && lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

> 💡 适合：多实例部署，保证同一时刻只有一个节点执行

#### 方案四：异步化 + MQ（大数据量场景）

```java
@Scheduled(fixedDelay = 60000)  // 每分钟触发一次，不是每秒
public void trigger() {
    // 1. 查询待处理数据 ID 列表（只查 ID，不查全量）
    List<Long> ids = taskDao.selectPendingIds(1000);  // 限制 1000 条
    
    // 2. 发 MQ 异步处理
    for (Long id : ids) {
        rocketMQTemplate.asyncSend("process-topic", id);
    }
}

// 消费者异步处理
@RocketMQMessageListener(topic = "process-topic")
public class ProcessConsumer implements RocketMQListener<Long> {
    public void onMessage(Long id) {
        // 单机消费，天然串行
        processOne(id);
    }
}
```

> 💡 适合：数据量大、处理慢，需要削峰填谷

## 🔥 扩展追问 1：Spring 的 `@Scheduled` 默认是什么线程池？

```java
// Spring 默认使用单线程池：ScheduledThreadPoolExecutor(1)
// 所有 @Scheduled 任务共享这一个线程！

@Component
public class TaskA {
    @Scheduled(fixedRate = 1000)
    public void taskA() { sleep(5000); }  // 执行 5 秒
}

@Component
public class TaskB {
    @Scheduled(fixedRate = 1000)  
    public void taskB() { }  // 想每秒执行，但被 TaskA 阻塞
}
```

**解法：自定义线程池**
```java
@Configuration
public class ScheduleConfig implements SchedulingConfigurer {
    
    @Override
    public void configureTasks(ScheduledTaskRegistrar taskRegistrar) {
        taskRegistrar.setScheduler(Executors.newScheduledThreadPool(10));
    }
}
```

## 🔥 扩展追问 2：任务执行异常会怎样？

`scheduleAtFixedRate` / `scheduleWithFixedDelay`：**任务抛异常后，后续任务不再执行！**

```java
executor.scheduleAtFixedRate(() -> {
    System.out.println("执行");
    throw new RuntimeException("出错了");  // 抛异常
}, 0, 1, TimeUnit.SECONDS);

// 输出一次 "执行"，然后没了
```

**必须加 try-catch**：
```java
executor.scheduleAtFixedRate(() -> {
    try {
        processData();
    } catch (Exception e) {
        log.error("定时任务异常", e);  // 捕获，不抛
    }
}, 0, 1, TimeUnit.SECONDS);
```

## 🔥 扩展追问 3：如何监控定时任务？

```java
@Component
public class MonitoredScheduledTask {
    
    @Autowired
    private MeterRegistry registry;
    
    @Scheduled(fixedRate = 1000)
    public void process() {
        Timer.Sample sample = Timer.start(registry);
        
        try {
            doProcess();
            registry.counter("scheduled.success").increment();
        } catch (Exception e) {
            registry.counter("scheduled.failure").increment();
            throw e;
        } finally {
            sample.stop(registry.timer("scheduled.duration"));
        }
    }
}
```

**告警规则**：
- `scheduled.failure` > 0 立即告警
- `scheduled.duration` P99 > 阈值告警

## 🔥 扩展追问 4：Cron 表达式任务错过执行时间怎么办？

```java
// 应用重启期间错过的任务，是否补执行？
@Scheduled(cron = "0 0 2 * * ?")  // 每天凌晨 2 点
public void dailyTask() { }
```

**解法：持久化 + 补偿**
```java
// 1. 任务触发记录存表
INSERT INTO scheduled_log (task_name, plan_time, status) 
VALUES ('dailyTask', '2024-01-15 02:00:00', 'PENDING');

// 2. 启动时检查未执行的任务
@PostConstruct
public void compensate() {
    List<ScheduledLog> missed = logDao.selectMissedTasks();
    for (ScheduledLog log : missed) {
        execute(log);  // 补偿执行
    }
}
```

## 一句话收尾

> `scheduleAtFixedRate` 的隐患是**"超时补执行导致任务重叠"**，生产环境要用 `scheduleWithFixedDelay` 或加**防重入标志 + 超时控制 + 分布式锁**，同时务必捕获异常防止任务终止。面试时说一句"我们用 Redisson 分布式锁保证集群单点执行，超时 5 秒自动中断"，立刻显得有生产经验。
