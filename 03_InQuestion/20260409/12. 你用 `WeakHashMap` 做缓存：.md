**12. 你用 `WeakHashMap` 做缓存：**

```java
WeakHashMap<String, byte[]> cache = new WeakHashMap<>();
cache.put(key, new byte[1024 * 1024]); // 1MB
```

**问题：** 什么情况下缓存的数据会被回收？如果 key 被强引用持有，value 会被回收吗？有什么使用陷阱？
## 先说结论

`WeakHashMap` 的 **key 是弱引用**，当 key 不再被强引用持有时，下次 GC 会回收整个 entry（key + value）。但如果外部强引用持有 key，value 永远不会被回收，可能导致内存泄漏。使用陷阱：value 不能间接引用 key，否则形成强引用环，key 永不被回收。

---

## 🔥 核心概念——弱引用与 WeakHashMap 结构

```
WeakHashMap 内部结构：

┌─────────────────────────────────────┐
│  Entry[] table                      │  ← 数组，存储 Entry
│  ├─ Entry 0                         │
│  │   ├─ key: WeakReference<String>  │  ← key 是弱引用！
│  │   ├─ value: byte[] (强引用)       │  ← value 是强引用
│  │   └─ next: Entry                 │
│  └─ ...                             │
└─────────────────────────────────────┘

回收条件：
  - 当 key 的弱引用是唯一的引用时（没有外部强引用）
  - 下次 GC 时，key 被回收
  - WeakHashMap 的 expungeStaleEntries() 方法会清理 value
```

---

## 🔥 详细展开

### 问题一：什么情况下数据会被回收？

```java
// 场景1：key 没有外部强引用 → 会被回收
WeakHashMap<String, byte[]> cache = new WeakHashMap<>();

void method() {
    String key = "data_" + System.currentTimeMillis();  // 局部变量
    cache.put(key, new byte[1024 * 1024]);
    
    // method 结束，key 引用消失
    // 下次 GC 时，WeakHashMap 中的 entry 被回收
}

// 场景2：key 被外部强引用持有 → 不会被回收
private static final String CACHE_KEY = "important_data";  // 静态常量

void init() {
    cache.put(CACHE_KEY, new byte[1024 * 1024]);
    // CACHE_KEY 一直存在，所以 value 永远不会被回收
    // 等同于普通 HashMap！
}
```

### 问题二：value 被回收的时机

```
WeakHashMap 的 value 是强引用，但会在以下时机被清理：

1. 主动调用 WeakHashMap 的方法时（get/put/size 等）
   → 内部调用 expungeStaleEntries() 清理失效 key 对应的 value

2. ReferenceQueue 机制
   → key 被 GC 回收后，进入 ReferenceQueue
   → 下次访问 WeakHashMap 时，从 queue 中取出失效 key，清理对应 entry

关键：value 的回收是"延迟"的，不是立即的
```

### 问题三：最严重的陷阱——value 间接引用 key

```java
// ❌ 致命错误：value 持有 key 的强引用
class CacheValue {
    private String key;      // 强引用！
    private byte[] data;
    
    CacheValue(String key, byte[] data) {
        this.key = key;      // 保存了 key 的强引用
        this.data = data;
    }
}

WeakHashMap<String, CacheValue> cache = new WeakHashMap<>();

void put(String key, byte[] data) {
    cache.put(key, new CacheValue(key, data));
    // 问题：
    // 1. key 被 WeakHashMap 的 Entry 弱引用持有
    // 2. key 同时被 CacheValue 强引用持有
    // 3. 即使外部没有强引用 key，CacheValue 里的 key 字段还在强引用它
    // 4. → key 永远不会被 GC 回收
    // 5. → 整个 entry 永远不会被清理
    // 6. → 内存泄漏！
}
```

**示意图：**

```
外部引用 ──→ key ──(弱引用)──→ WeakHashMap.Entry
              ↑
              └────(强引用)──→ CacheValue.key ──→ CacheValue ──→ value
              
形成闭环：即使外部引用断开，key 仍被 value 强引用
         → key 永不被回收 → entry 永不被清理
```

## 🔥 扩展追问 1：正确使用 WeakHashMap 的姿势

```java
// ✅ 正确用法1：key 和 value 完全独立
WeakHashMap<String, byte[]> cache = new WeakHashMap<>();

// key 是临时生成的，方法结束就释放
void process(String id) {
    String cacheKey = "temp:" + id;
    byte[] data = cache.computeIfAbsent(cacheKey, k -> loadFromDB(id));
    // 使用 data...
    // method 结束，cacheKey 引用消失，下次 GC 时 entry 被回收
}

// ✅ 正确用法2：配合 SoftReference 做 value
WeakHashMap<String, SoftReference<byte[]>> cache = new WeakHashMap<>();
// 双重保护：key 弱引用 + value 软引用
// 内存不足时，value 也会被回收
```

## 🔥 扩展追问 2：WeakHashMap vs SoftHashMap（Caffeine 实现）

```java
// Java 标准库没有 SoftHashMap，但 Caffeine 支持
Caffeine.newBuilder()
    .weakKeys()      // key 弱引用（同 WeakHashMap）
    .weakValues()    // value 弱引用（GC 时回收）
    .softValues()    // value 软引用（内存不足时回收）
    .build();

// 对比：
// ┌──────────────┬─────────────────┬─────────────────┐
// │   类型        │   回收时机       │   适用场景       │
// ├──────────────┼─────────────────┼─────────────────┤
// │  WeakKey     │  无强引用时 GC   │  临时缓存        │
// │  WeakValue   │  GC 时          │  可被重建的数据   │
// │  SoftValue   │  内存不足时      │  大对象缓存      │
// └──────────────┴─────────────────┴─────────────────┘
```

## 🔥 扩展追问 3：WeakHashMap 的线程安全问题

```java
// WeakHashMap 不是线程安全的！
WeakHashMap<String, Object> cache = new WeakHashMap<>();

// ❌ 错误：多线程并发访问
executor.submit(() -> cache.put("a", 1));
executor.submit(() -> cache.get("a"));  // 可能 ConcurrentModificationException

// ✅ 正确：使用 Collections.synchronizedMap 或 ConcurrentHashMap
Map<String, Object> syncCache = Collections.synchronizedMap(new WeakHashMap<>());

// 但 synchronizedMap 锁粒度粗，性能差
// 更好的方案：Caffeine（线程安全 + 高性能）
```

## 🔥 扩展追问 4：实际应用场景

```java
// 场景1：监听器/回调的自动清理
class EventBus {
    // key = 监听器对象，value = 回调函数
    // 当监听器对象不再被使用时，自动从 map 中移除
    private final WeakHashMap<Object, Runnable> listeners = new WeakHashMap<>();
    
    public void register(Object listener, Runnable callback) {
        listeners.put(listener, callback);
    }
    
    public void fireEvent() {
        // 遍历时自动清理已失效的监听器
        for (Runnable callback : listeners.values()) {
            callback.run();
        }
    }
}

// 场景2：ClassLoader 缓存（Tomcat 热部署）
// 旧的 ClassLoader 不再被引用时，相关的类定义自动释放
// 防止 PermGen/Metaspace 溢出
```

## 一句话收尾

> `WeakHashMap` 的 key 是弱引用，**外部无强引用时 GC 回收**，但 value 不能间接引用 key 否则内存泄漏；生产环境用 **Caffeine（weakKeys/softValues）** 替代，线程安全且功能更强。
