**16. 你实现了一个分页查询：**

```java
// 第 10000 页，每页 20 条
String sql = "SELECT * FROM orders ORDER BY id LIMIT 200000, 20";
```

**问题：** 这条 SQL 有什么问题？随着页码增大性能如何变化？如何优化深分页？
## 先说结论

`LIMIT 200000, 20` 需要扫描 **200020 行** 再丢弃前 20 万行，随着页码增大呈**线性性能下降**。优化方案：**延迟关联（子查询先取 id）**、**覆盖索引**、**游标分页（记录上次位置）**、**搜索引擎（ES）**。

---

## 🔥 核心概念——LIMIT 的工作原理

```
执行流程：
  1. 根据 ORDER BY id 排序（可能用索引，也可能 filesort）
  2. 扫描并计数，跳过前 OFFSET = 200000 行
  3. 取接下来的 20 行
  4. 返回结果

问题：
  - 需要扫描 OFFSET + LIMIT = 200020 行
  - 前 200000 行被白白扫描后丢弃
  - 页码越大，扫描行数越多，性能越差

时间复杂度：O(N)，N = OFFSET + LIMIT
```

---

## 🔥 详细展开

### 问题一：性能随页码线性下降

```sql
-- 测试不同 OFFSET 的耗时（假设 5000 万数据）
SELECT * FROM orders ORDER BY id LIMIT 0, 20;      -- 0.001s，扫描 20 行
SELECT * FROM orders ORDER BY id LIMIT 10000, 20;  -- 0.1s，扫描 10020 行
SELECT * FROM orders ORDER BY id LIMIT 100000, 20; -- 1s，扫描 100020 行
SELECT * FROM orders ORDER BY id LIMIT 200000, 20; -- 2s，扫描 200020 行
SELECT * FROM orders ORDER BY id LIMIT 500000, 20; -- 5s+，扫描 500020 行

-- 第 10000 页需要 2 秒，用户无法容忍
```

### 问题二：内存和临时表

```sql
-- 如果排序字段没有索引，MySQL 需要：
-- 1. 全表扫描，加载到内存或临时表
-- 2. 排序（filesort）
-- 3. 再 LIMIT

EXPLAIN SELECT * FROM orders ORDER BY id LIMIT 200000, 20;
-- Extra: Using filesort（坏，需要优化）
```

## 🔥 扩展追问 1：延迟关联（Deferred Join）

```sql
-- 优化方案 1：子查询先取 id，再关联取完整数据
SELECT o.* 
FROM orders o
JOIN (
    SELECT id FROM orders 
    ORDER BY id 
    LIMIT 200000, 20
) tmp ON o.id = tmp.id;

执行过程：
  1. 子查询只在索引上扫描（覆盖索引，无需回表）
  2. 拿到 20 个 id
  3. 根据 id 回表取完整数据（只需 20 次回表）
  
对比原 SQL：
  - 原 SQL：扫描 200020 行，回表 200020 次
  - 延迟关联：扫描 200020 行索引，回表 20 次
  - 性能提升：10~100 倍（取决于行大小）
```

### 覆盖索引版本

```sql
-- 如果 id 是主键，上面的子查询已经是覆盖索引
-- 如果不是，创建覆盖索引
CREATE INDEX idx_id_cover ON orders(id, order_no, status);

-- 子查询
SELECT id, order_no, status FROM orders 
ORDER BY id 
LIMIT 200000, 20;
-- 这三个字段都在索引中，无需回表
```

## 🔥 扩展追问 2：游标分页（推荐）

```sql
-- 优化方案 2：记录上次位置，避免 OFFSET
-- 第一页
SELECT * FROM orders 
ORDER BY id 
LIMIT 20;
-- 返回最后一条 id = 10020

-- 下一页（传入上次最后一条的 id）
SELECT * FROM orders 
WHERE id > 10020 
ORDER BY id 
LIMIT 20;

执行过程：
  1. 直接定位 id > 10020 的位置（索引范围扫描）
  2. 取 20 条
  3. 扫描行数 = 20，与页码无关！

时间复杂度：O(1)，与深度无关
```

**Java 实现：**

```java
public class CursorPage<T> {
    private List<T> data;
    private T lastCursor;  // 最后一条记录的排序字段值
    private boolean hasMore;
}

public CursorPage<Order> queryNextPage(Long lastId, int pageSize) {
    String sql = lastId == null 
        ? "SELECT * FROM orders ORDER BY id LIMIT ?"
        : "SELECT * FROM orders WHERE id > ? ORDER BY id LIMIT ?";
    
    List<Order> orders = jdbcTemplate.query(sql, 
        lastId == null 
            ? new Object[]{pageSize} 
            : new Object[]{lastId, pageSize},
        new OrderRowMapper()
    );
    
    CursorPage<Order> page = new CursorPage<>();
    page.setData(orders);
    page.setHasMore(orders.size() == pageSize);
    if (!orders.isEmpty()) {
        page.setLastCursor(orders.get(orders.size() - 1).getId());
    }
    return page;
}
```

**优点：**
- 性能与页码深度无关，永远是 O(1)
- 适合"加载更多"场景（手机 APP 下拉刷新）

**缺点：**
- 不支持跳转到任意页码（只能上一页/下一页）
- 数据变动时可能重复或遗漏（需要版本号或时间戳辅助）

## 🔥 扩展追问 3：业务层优化

### 方案一：限制最大页码

```java
// 产品层面限制，不允许查太深
if (pageNum > 1000) {
    throw new BusinessException("不支持查询超过 1000 页，请使用筛选条件缩小范围");
}
// 或者强制使用搜索功能
```

### 方案二：倒序查询（如果业务允许）

```sql
-- 用户通常关心最新数据
-- 倒序查询，前几页就是最新数据，很少需要深分页
SELECT * FROM orders 
ORDER BY id DESC 
LIMIT 20;
```

### 方案三：搜索引擎（ES）

```java
// 深分页场景交给 Elasticsearch
SearchRequest request = new SearchRequest("orders");
request.source().query(QueryBuilders.matchAllQuery())
       .from(200000)  // ES 的 from + size 也有深度限制（默认 10000）
       .size(20);

// ES 深分页用 scroll 或 search_after
// 类似游标分页，性能更好
```

## 🔥 扩展追问 4：分区表优化

```sql
-- 按时间分区，查询指定分区
CREATE TABLE orders (
    id BIGINT,
    create_time DATETIME,
    ...
) PARTITION BY RANGE (YEAR(create_time)) (
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025)
);

-- 带分区条件的查询，数据量减少，LIMIT 更快
SELECT * FROM orders 
WHERE create_time >= '2024-01-01' 
ORDER BY id 
LIMIT 200000, 20;
-- 只扫描 p2024 分区，数据量从 5000 万降到 1000 万
```

## 一句话收尾

> `LIMIT OFFSET` 是**深度分页杀手**，页码越大性能越差；优化用**延迟关联**减少回表、**游标分页**避免 OFFSET、**限制页码**或上**搜索引擎**，核心是把 O(N) 降到 O(1)。
