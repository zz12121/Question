**23. 你用 Redis List 做消息队列：**

```java
// 生产者
jedis.lpush("queue", message);
// 消费者
while (true) {
    String msg = jedis.rpop("queue");
    if (msg != null) process(msg);
}
```

**问题：** 如果消费者挂了，消息会丢失吗？如何实现可靠消费？和专业的 MQ 比有什么缺陷？
## 先说结论

消费者挂了会**丢失消息**（已 POP 但未 ACK）。可靠消费需要**BRPOPLPUSH 备份队列**或**Stream 类型**，但相比专业 MQ（RocketMQ/Kafka），Redis 缺乏**消息持久化、顺序保证、消费组、死信队列**等核心能力，只适合轻量级场景。

---

## 🔥 核心概念——当前代码的问题

```
执行流程：
  生产者 LPUSH queue msg    → 消息入队
  消费者 RPOP queue          → 消息出队并删除
  消费者 process(msg)        → 处理消息

消费者挂了的问题：
  T1: 消费者 RPOP 拿到消息 msg1
  T2: 消费者正在处理 msg1
  T3: 消费者挂了
  T4: msg1 已出队但未处理完成 → 消息丢失！
  
问题本质：
  RPOP 是"破坏性读取"，消息出队即删除
  没有 ACK 机制，无法保证可靠消费
```

---

## 🔥 详细展开

### 问题一：消息丢失

```java
// 消费者代码
while (true) {
    String msg = jedis.rpop("queue");  // 消息立即从队列删除
    if (msg != null) {
        // 如果在这里挂了，消息永久丢失
        process(msg);
    }
}
```

### 问题二：空轮询

```java
// 当前实现是忙等待
while (true) {
    String msg = jedis.rpop("queue");  // 队列为空时返回 null
    // 不断循环，浪费 CPU 和网络资源
}
```

### 问题三：无消费组支持

```
多个消费者时：
  - 无法做负载均衡（可能重复消费）
  - 无法做广播消费
  - 无法追踪消费进度
```

## 🔥 扩展追问 1：BRPOPLPUSH 实现可靠消费

```java
// 方案：原子操作 + 备份队列
// BRPOPLPUSH：从 queue 弹出，同时推入 processing 队列

while (true) {
    // 阻塞等待，有消息才返回，避免空轮询
    String msg = jedis.brpoplpush("queue", "processing", 0);
    
    try {
        process(msg);
        // 处理成功，从备份队列删除
        jedis.lrem("processing", 1, msg);
    } catch (Exception e) {
        // 处理失败，消息还在 processing 队列
        // 可以重试或移入死信队列
        log.error("处理失败", e);
    }
}
```

**可靠性保障：**

```
正常流程：
  LPUSH queue msg
  BRPOPLPUSH queue processing → 原子操作，消息在 processing
  process(msg)
  LREM processing 1 msg → 确认完成

消费者挂了：
  消息仍在 processing 队列
  重启后从 processing 恢复，继续处理
```

**缺点：**
- 需要额外维护 processing 队列
- 消息去重困难（同一消息可能处理多次）
- 无消费进度追踪

## 🔥 扩展追问 2：Redis Stream（Redis 5.0+）

```java
// Stream 是 Redis 专门的消息队列类型
// 支持消费组、ACK、消息持久化

// 生产者
jedis.xadd("mystream", StreamEntryID.NEW_ENTRY, 
           Collections.singletonMap("field", "value"));

// 消费者组
jedis.xgroupCreate("mystream", "mygroup", StreamEntryID.LAST_ENTRY, true);

// 消费者
while (true) {
    // 阻塞读取，支持 ACK
    List<Map.Entry<String, List<StreamEntry>>> messages = 
        jedis.xreadGroup("mygroup", "consumer1", 1, 0, true, 
                         new AbstractMap.SimpleEntry<>("mystream", ">"));
    
    for (Map.Entry<String, List<StreamEntry>> entry : messages) {
        for (StreamEntry msg : entry.getValue()) {
            try {
                process(msg);
                // ACK 确认
                jedis.xack("mystream", "mygroup", msg.getID());
            } catch (Exception e) {
                // 未 ACK，消息保留在 Pending List，可重试
            }
        }
    }
}
```

**Stream 优势：**
- 消息持久化（RDB/AOF）
- 消费组支持（负载均衡）
- ACK 机制（可靠消费）
- 消息 ID 自增（顺序保证）

**Stream 劣势：**
- 单节点容量有限（内存限制）
- 无高可用（主从切换可能丢消息）
- 消费组实现简单（对比 Kafka）

## 🔥 扩展追问 3：Redis vs 专业 MQ

| 特性 | Redis List/Stream | RocketMQ | Kafka |
|------|-------------------|----------|-------|
| 消息持久化 | 内存（RDB/AOF） | 磁盘（CommitLog） | 磁盘（PageCache） |
| 吞吐量 | 10万/s | 10万+/s | 100万+/s |
| 消息回溯 | Stream 支持 | 支持 | 支持（时间戳/Offset） |
| 消费组 | Stream 支持 | 完善 | 完善 |
| 死信队列 | 需自建 | 内置 | 需自建 |
| 顺序消息 | 单分区支持 | 支持 | 单分区支持 |
| 延迟消息 | 需自建 | 内置 18 级 | 需自建 |
| 高可用 | 主从（可能丢） | 多副本 | 多副本 |
| 适用场景 | 轻量/缓存 | 企业级业务 | 大数据日志 |

public class RedisStreamQueue {

            try (Jedis jedis = jedisPool.getResource()) {
                while (!Thread.interrupted()) {
                    List<Map.Entry<String, List<StreamEntry>>> msgs = 
                        jedis.xread(1, 0, new AbstractMap.SimpleEntry<>(topic, "$"));
                    // 处理并 ACK...
                }
            }
        });
    }
}

@Component
public class HybridQueue {
    
    public void send(Message msg) {

            }

    }
}

## 一句话收尾

> Redis List 做队列会**丢消息**（无 ACK），改进用 **BRPOPLPUSH 备份队列** 或 **Stream 类型**，但生产环境**高可靠场景**必须用 **RocketMQ/Kafka**，Redis 只适合做**轻量级缓冲**或**缓存层**。
