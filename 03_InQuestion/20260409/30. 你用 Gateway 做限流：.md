**30. 你用 Gateway 做限流：**

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: order_route
          uri: lb://order-service
          predicates:
            - Path=/order/**
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 10
                redis-rate-limiter.burstCapacity: 20
```

**问题：** 这个限流是基于什么维度（IP？用户？全局？）？如果 Redis 挂了，限流会失效吗？如何实现自定义限流策略（比如按用户 ID）？

---
## 先说结论

默认基于 **KeyResolver 返回的 Key** 限流，通常是 IP 或 Hostname；**Redis 挂了限流会失效**（请求直接放行或报错，取决于配置）；自定义策略需实现 `KeyResolver` 接口，按用户 ID、接口路径等维度限流。

---

## 🔥 核心概念

**解决什么问题：**
- 防止突发流量压垮后端服务
- 保护系统可用性，实现流量整形

**基本原理：**
- 基于 **令牌桶算法**（Token Bucket）
- `replenishRate`：每秒补充令牌数（QPS）
- `burstCapacity`：桶容量（允许突发流量）
- 使用 Redis 存储令牌状态，支持集群共享

---

## 🔥 详细展开

### 1. 默认限流维度

```yaml
filters:
  - name: RequestRateLimiter
    args:
      redis-rate-limiter.replenishRate: 10   # 每秒10个请求
      redis-rate-limiter.burstCapacity: 20   # 桶容量20
```

**默认 KeyResolver（按 IP）：**

```java
// Spring Cloud Gateway 默认实现
public class RemoteAddrKeyResolver implements KeyResolver {
    
    @Override
    public Mono<String> resolve(ServerWebExchange exchange) {
        // 默认按客户端 IP 限流
        return Mono.just(exchange.getRequest()
            .getRemoteAddress()
            .getAddress()
            .getHostAddress());
    }
}
```

**其他常见维度：**

| 维度 | KeyResolver 实现 | 适用场景 |
|------|------------------|----------|
| IP 地址 | `RemoteAddrKeyResolver` | 防刷、防爬虫 |
| Hostname | `HostNameKeyResolver` | 按域名限流 |
| 用户 ID | 自定义（见下文） | 登录用户限流 |
| 接口路径 | `PathKeyResolver` | 按 API 限流 |
| 全局 | 返回固定值 | 全局限流 |

### 2. Redis 挂了会怎样？

**默认行为：请求直接放行！**

```java
public class RedisRateLimiter {
    
    public Mono<Response> isAllowed(String routeId, String id) {
        return redisTemplate.execute(redisScript, keys, args)
            .onErrorResume(throwable -> {
                // Redis 异常时，默认放行！
                log.warn("Error determining if user allowed", throwable);
                return Flux.just(Arrays.asList(1L, -1L));
            })
            .reduce(new ArrayList<Long>(), (list, results) -> {
                list.addAll(results);
                return list;
            })
            .map(results -> {
                boolean allowed = results.get(0) == 1L;
                Long tokensLeft = results.get(1);
                return new Response(allowed, getHeaders(tokensLeft));
            });
    }
}
```

**风险：** Redis 挂了 = 限流失效 = 系统暴露在高流量下！

**安全降级配置：**

```java
@Bean
public RedisRateLimiter redisRateLimiter(ReactiveRedisTemplate<String, String> template) {
    return new RedisRateLimiter(template) {
        @Override
        public Mono<Response> isAllowed(String routeId, String id) {
            return super.isAllowed(routeId, id)
                .onErrorResume(e -> {
                    // Redis 异常时，拒绝请求（保守策略）
                    log.error("Redis rate limiter failed, denying request", e);
                    return Mono.just(new Response(false, 
                        Collections.singletonMap("X-RateLimit-Error", "Service Unavailable")));
                });
        }
    };
}
```

**或配置本地兜底：**

```java
@Component
public class LocalRateLimiterFallback {
    
    private final Map<String, AtomicInteger> localCounters = new ConcurrentHashMap<>();
    
    public boolean isAllowed(String key, int limit) {
        AtomicInteger counter = localCounters.computeIfAbsent(key, k -> new AtomicInteger(0));
        int current = counter.incrementAndGet();
        
        // 每秒重置
        ScheduledExecutorService executor = Executors.newSingleThreadScheduledExecutor();
        executor.schedule(() -> counter.set(0), 1, TimeUnit.SECONDS);
        
        return current <= limit;
    }
}
```

### 3. 自定义限流策略（按用户 ID）

```java
@Component
public class UserIdKeyResolver implements KeyResolver {
    
    @Override
    public Mono<String> resolve(ServerWebExchange exchange) {
        // 从 Header 获取用户 ID
        String userId = exchange.getRequest()
            .getHeaders()
            .getFirst("X-User-Id");
        
        if (userId == null || userId.isEmpty()) {
            // 未登录用户按 IP 限流
            userId = exchange.getRequest()
                .getRemoteAddress()
                .getAddress()
                .getHostAddress();
        }
        
        return Mono.just(userId);
    }
}
```

**配置使用：**

```java
@Configuration
public class GatewayConfig {
    
    @Bean
    public KeyResolver userKeyResolver() {
        return new UserIdKeyResolver();
    }
}
```

### 4. 更灵活的限流策略（多维度组合）

```java
@Component
public class CompositeKeyResolver implements KeyResolver {
    
    @Override
    public Mono<String> resolve(ServerWebExchange exchange) {
        String path = exchange.getRequest().getPath().value();
        String userId = exchange.getRequest().getHeaders().getFirst("X-User-Id");
        String clientType = exchange.getRequest().getHeaders().getFirst("X-Client-Type");
        
        // 组合维度：路径 + 用户类型 + 用户ID
        // 例如：/order/api:APP:user123
        String key = String.format("%s:%s:%s", 
            path, 
            clientType != null ? clientType : "UNKNOWN",
            userId != null ? userId : getClientIP(exchange));
        
        return Mono.just(key);
    }
}
```

## 🔥 扩展追问 1：令牌桶、漏桶、滑动窗口算法对比？

| 算法 | 原理 | 突发流量 | 适用场景 |
|------|------|----------|----------|
| **令牌桶** | 匀速产生令牌，请求消耗令牌 | 允许一定突发 | 大多数场景（Gateway 默认） |
| **漏桶** | 请求进入桶，匀速流出处理 | 不允许突发，平滑输出 | 严格限速（如支付接口） |
| **滑动窗口** | 统计时间窗口内请求数 | 精确控制 | 精确限流（如短信验证码） |
| **滑动日志** | 记录每个请求时间戳 | 最精确 | 低流量精确场景 |

**漏桶实现：**

```java
public class LeakyBucketRateLimiter {
    
    private final long capacity;      // 桶容量
    private final long leakRate;      // 漏出速率（每秒）
    private long water;               // 当前水量
    private long lastLeakTime;
    
    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        long leaked = (now - lastLeakTime) * leakRate / 1000;
        water = Math.max(0, water - leaked);
        lastLeakTime = now;
        
        if (water < capacity) {
            water++;
            return true;
        }
        return false;
    }
}
```

## 🔥 扩展追问 2：Gateway 限流和 Sentinel 限流怎么选？

| 维度 | Gateway 限流 | Sentinel 限流 |
|------|--------------|---------------|
| 位置 | 网关层 | 网关/服务层均可 |
| 功能 | 简单 QPS 限流 | 丰富（QPS/并发/热点/系统保护） |
| 熔断降级 | 不支持 | 支持 |
| 热点参数 | 不支持 | 支持 |
| 自适应保护 | 不支持 | 支持（系统负载保护） |
| 控制台 | 无 | 可视化控制台 |

**最佳实践：双层防护**

```
用户请求
    ↓
Gateway 限流（粗粒度，按 IP/用户）  ← 防刷、第一层保护
    ↓
服务层
    ↓
Sentinel 限流（细粒度，按接口/参数） ← 业务级保护、熔断降级
    ↓
数据库
```

## 🔥 扩展追问 3：如何实现全局限流（集群共享）vs 本地限流？

**本地限流（Guava RateLimiter）：**

```java
@Service
public class LocalRateLimiterService {
    
    // 每个实例独立计数
    private final RateLimiter rateLimiter = RateLimiter.create(100);
    
    public boolean tryAcquire() {
        return rateLimiter.tryAcquire();
    }
}
```

**问题：** 3 个实例，每个 100 QPS，集群实际 300 QPS，可能压垮下游。

**全局限流（Redis + Lua）：**

```lua
-- ratelimiter.lua
local key = KEYS[1]
local rate = tonumber(ARGV[1])
local capacity = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

local fill_time = capacity / rate
local ttl = math.floor(fill_time * 2)

local last_tokens = redis.call('get', key)
if last_tokens == false then
    last_tokens = capacity
end

local last_updated = redis.call('get', key .. ':last_updated')
if last_updated == false then
    last_updated = 0
end

local delta = math.max(0, now - tonumber(last_updated))
local filled_tokens = math.min(capacity, tonumber(last_tokens) + (delta * rate))
local allowed = filled_tokens >= 1
local new_tokens = filled_tokens
if allowed then
    new_tokens = filled_tokens - 1
end

redis.call('setex', key, ttl, new_tokens)
redis.call('setex', key .. ':last_updated', ttl, now)

return {allowed, new_tokens}
```

**选型建议：**

| 场景 | 推荐方案 |
|------|----------|
| 单机部署 | 本地限流（性能更好） |
| 集群部署 | Redis 全局限流 |
| 混合场景 | 本地限流 + Redis 全局限流（本地先挡一波） |

---

## 一句话收尾

Gateway 默认按 IP 限流，Redis 挂了会失效需配置降级，自定义维度实现 `KeyResolver` 接口；生产环境建议 Gateway + Sentinel 双层防护，关键接口用漏桶严格平滑限流，秒杀场景用令牌桶允许合理突发！
