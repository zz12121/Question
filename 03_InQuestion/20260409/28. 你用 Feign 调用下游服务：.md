**28. 你用 Feign 调用下游服务：**

```java
@FeignClient(name = "inventory-service")
public interface InventoryClient {
    @GetMapping("/stock/{productId}")
    int getStock(@PathVariable Long productId);
}
```

**问题：** 如果下游服务响应慢或挂了，会发生什么？如何配置超时、重试、熔断？重试会带来什么问题？

---
## 先说结论

下游服务响应慢或挂了会导致**调用线程阻塞、级联雪崩**。需配置 **超时（Timeout）** 防止无限等待，**重试（Retry）** 提高成功率，**熔断（Circuit Breaker）** 快速失败保护系统。但重试可能引发**幂等性问题**（重复扣款）、**放大故障**（压垮下游），需谨慎设计。

---

## 🔥 核心概念

**解决什么问题：**
- 分布式调用中的不确定性（网络抖动、服务故障）
- 防止局部故障扩散为系统级雪崩

**基本原理：**
- **超时：** 限制等待时间，避免线程长期阻塞
- **重试：** 短暂故障时自动重试，提高可用性
- **熔断：** 故障率过高时快速失败，给下游恢复时间

## 🔥 详细展开

### 1. 无保护时的灾难场景

```java
// 用户下单接口
@PostMapping("/order")
public Result createOrder(@RequestBody Order order) {
    // 如果 inventory-service 挂了，这里会阻塞多久？
    int stock = inventoryClient.getStock(order.getProductId());
    // 默认超时可能是 60 秒甚至更久！
    
    // 线程池逐渐耗尽...
    // 所有请求都卡在这里，系统雪崩！
}
```

**风险链：**
```
下游服务响应慢 
    → 调用线程阻塞 
    → 线程池耗尽 
    → 新请求被拒绝 
    → 级联故障扩散到整个系统
```

### 2. 超时配置（Timeout）

**Feign + Ribbon 配置：**

```yaml
# application.yml
feign:
  client:
    config:
      default:
        connectTimeout: 2000    # 连接超时：2秒
        readTimeout: 5000       # 读取超时：5秒
      inventory-service:        # 针对特定服务
        connectTimeout: 1000
        readTimeout: 3000
        
ribbon:
  ReadTimeout: 5000
  ConnectTimeout: 2000
```

**Feign + LoadBalancer（Spring Cloud 2020+）：**

```yaml
spring:
  cloud:
    openfeign:
      client:
        config:
          inventory-service:
            connectTimeout: 2000
            readTimeout: 5000
            loggerLevel: full
```

**代码方式配置：**

```java
@Configuration
public class FeignConfig {
    
    @Bean
    public Request.Options feignOptions() {
        return new Request.Options(2, TimeUnit.SECONDS, 5, TimeUnit.SECONDS, true);
```

### 3. 重试配置（Retry）

**Ribbon 重试（传统方式）：**

```yaml
ribbon:
  MaxAutoRetries: 1           # 同一实例重试次数
  MaxAutoRetriesNextServer: 2  # 切换实例重试次数
  OkToRetryOnAllOperations: false  # 只对 GET 重试（默认）
  retryableStatusCodes: 503,504    # 特定状态码才重试
```

**Spring Retry（推荐，更灵活）：**

```java
@Configuration
public class FeignRetryConfig {
    
    @Bean
    public Retryer feignRetryer() {
        // 初始间隔100ms，最大间隔1s，最多重试3次
        return new Retryer.Default(100, TimeUnit.SECONDS.toMillis(1), 3);
    }
}

// 或使用 Spring Retry 注解
@Retryable(
    value = {FeignException.class},
    maxAttempts = 3,
    backoff = @Backoff(delay = 100, multiplier = 2)
)
public int getStock(Long productId) {
    return inventoryClient.getStock(productId);
}
```

**自定义重试策略（区分幂等性）：**

```java
public class IdempotentRetryer implements Retryer {

        // 非幂等方法不重试
        if (!idempotentMethods.contains(e.request().httpMethod())) {
            throw e;
        }
        // ... 重试逻辑
    }
}
```

### 4. 熔断配置（Circuit Breaker）

**Resilience4j（Spring Cloud Circuit Breaker）：**

```yaml
resilience4j:
  circuitbreaker:
    instances:
      inventoryService:
        slidingWindowSize: 10          # 统计窗口大小
        failureRateThreshold: 50       # 失败率阈值 50%
        waitDurationInOpenState: 10000 # 熔断后等待10秒
        permittedNumberOfCallsInHalfOpenState: 3  # 半开状态允许3个请求
        slowCallRateThreshold: 80      # 慢调用比例阈值
        slowCallDurationThreshold: 2000 # 慢调用阈值 2秒
        
  timelimiter:
    instances:
      inventoryService:
        timeoutDuration: 3s            # 超时时间
        
  retry:
    instances:
      inventoryService:
        maxAttempts: 3
        waitDuration: 100ms
```

**代码使用：**

```java
@Service
public class OrderService {
    
    @Autowired
    private InventoryClient inventoryClient;
    
    private final CircuitBreaker circuitBreaker;
    private final Retry retry;
    
    public OrderService(CircuitBreakerRegistry cbRegistry, RetryRegistry retryRegistry) {
        this.circuitBreaker = cbRegistry.circuitBreaker("inventoryService");
        this.retry = retryRegistry.retry("inventoryService");
    }
    
    public int getStock(Long productId) {
        // 装饰调用
        Supplier<Integer> decorated = CircuitBreaker
            .decorateSupplier(circuitBreaker, 
                Retry.decorateSupplier(retry, 
                    () -> inventoryClient.getStock(productId)));
        
        return Try.ofSupplier(decorated)
            .recover(throwable -> {
                // 熔断或重试耗尽后的降级逻辑
                log.warn("Get stock failed, using cache", throwable);
                return getStockFromCache(productId);
            })
            .get();
    }
    
    private int getStockFromCache(Long productId) {
        // 返回缓存值或默认值
        return redisTemplate.opsForValue().get("stock:" + productId);
    }
}
```

**注解方式（更简洁）：**

```java
@CircuitBreaker(name = "inventoryService", fallbackMethod = "getStockFallback")
@Retry(name = "inventoryService")
@TimeLimiter(name = "inventoryService")
public CompletableFuture<Integer> getStockAsync(Long productId) {
    return CompletableFuture.supplyAsync(() -> 
        inventoryClient.getStock(productId));
}

public CompletableFuture<Integer> getStockFallback(Long productId, Exception ex) {
    return CompletableFuture.completedFuture(getStockFromCache(productId));
}
```

### 5. 重试带来的问题

#### 问题 1：幂等性破坏

```java
// 非幂等操作重复执行 = 灾难
@PostMapping("/deduct")
public Result deductStock(@RequestBody DeductRequest request) {
    // 第一次调用：已扣减成功，但响应超时
    // 重试第二次：又扣减一次！
    stockService.deduct(request.getProductId(), request.getCount());
}
```

**解决：幂等性设计**

```java
// 客户端生成唯一幂等键
String idempotencyKey = UUID.randomUUID().toString();

// 服务端幂等处理
@PostMapping("/deduct")
public Result deductStock(@RequestBody DeductRequest request,
                          @RequestHeader("Idempotency-Key") String key) {
    // Redis 原子检查
    Boolean success = redisTemplate.opsForValue()
        .setIfAbsent("idempotent:" + key, "1", 24, TimeUnit.HOURS);
    
    if (!success) {
        // 重复请求，直接返回上次结果
        return getPreviousResult(key);
    }
    
    // 执行业务
    return stockService.deduct(request);
}
```

#### 问题 2：故障放大

```
用户重试 → 网关重试 → 服务A重试 → 服务B重试 → 数据库
         每层都重试3次 = 3×3×3×3 = 81 倍流量！
```

**解决：重试风暴防护**

```yaml
# 限制重试层级
retry:
  maxAttempts: 1  # 只在最外层重试
  
# 或标记重试次数
feign:
  client:
    config:
      default:
        requestInterceptors:
          - com.example.RetryCountInterceptor  # 传递重试计数
```

#### 问题 3：超时叠加

```
每层超时 5 秒，3 层调用链 = 最外层需要等 15 秒！
```

**解决：超时传递（Deadline）**

```java
// 上游传递剩余超时时间
public class TimeoutPropagationInterceptor implements RequestInterceptor {
    
    @Override
    public void apply(RequestTemplate template) {
        // 计算剩余时间
        long remaining = getRemainingTimeout();
        template.header("X-Remaining-Timeout", String.valueOf(remaining));
    }
}
```

```java
public class NacosWeightRule extends AbstractLoadBalancerRule {
    
    @Autowired
    private NacosDiscoveryProperties nacosProperties;
    
    @Override
    public Server choose(Object key) {
        DynamicServerListLoadBalancer loadBalancer = (DynamicServerListLoadBalancer) getLoadBalancer();
        String serviceName = loadBalancer.getName();

                .selectOneHealthyInstance(serviceName);

        } catch (NacosException e) {
            return null;
        }
    }
}
```

## 🔥 扩展追问 3：Feign 和 Dubbo 的对比，什么时候选哪个？

| 维度 | Feign | Dubbo |
|------|-------|-------|
| 协议 | HTTP/REST | 自定义二进制（Dubbo协议） |
| 性能 | 较低（HTTP开销） | 高（序列化高效） |
| 生态 | Spring Cloud 生态 | 阿里生态，云原生支持好 |
| 服务治理 | 依赖 Spring Cloud 组件 | 内置完善 |
| 跨语言 | 好（HTTP通用） | 一般（需 Triple 协议） |
| 适用场景 | 对外 API、跨团队 | 内部微服务、高性能要求 |

**选型建议：**
- 对外暴露、跨团队、跨语言 → Feign/HTTP
- 内部服务、追求性能、复杂治理 → Dubbo

## 一句话收尾

Feign 调用必须配超时防阻塞、重试提可用（注意幂等）、熔断保系统，三者配合才能在高可用和稳定性之间找到平衡，面试要强调重试的双刃剑效应和分布式系统的"超时传递"设计！
