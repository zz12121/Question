**6. 你用 `ReentrantReadWriteLock` 优化读多写少的缓存：**

```java
public V get(K key) {
    readLock.lock();
    try {
        return cache.get(key);
    } finally {
        readLock.unlock();
    }
}
```

**问题：** 如果读操作里需要缓存未命中时从数据库加载，怎么实现？直接加写锁有什么问题？
## 先说结论

读锁里**不能直接升级成写锁**——Java 的 `ReentrantReadWriteLock` 不支持锁升级，强行加写锁会**死锁**。正确姿势是：释放读锁 → 加写锁 → **二次检查（DCL）** → 加载数据。

---

## 🔥 核心概念

```
ReentrantReadWriteLock 的两个铁律：
  ✅ 支持锁降级：写锁 → 读锁（持有写锁时可以再加读锁）
  ❌ 不支持锁升级：读锁 → 写锁（会死锁！）
```

**为什么不支持升级？**

```
线程A：持有读锁，等待写锁
线程B：持有读锁，等待写锁
  ↓
A 等 B 释放读锁，B 等 A 释放读锁
  ↓
死锁！
```

---

## 🔥 详细展开

### ❌ 错误写法一：读锁中直接加写锁

```java
public V get(K key) {
    readLock.lock();
    try {
        V value = cache.get(key);
        if (value == null) {
            // ❌ 致命错误：持有读锁的情况下再加写锁
            writeLock.lock(); // → 永久阻塞，死锁！
            try {
                value = loadFromDB(key);
                cache.put(key, value);
            } finally {
                writeLock.unlock();
            }
        }
        return value;
    } finally {
        readLock.unlock();
    }
}
```

### ❌ 错误写法二：释放读锁后直接加写锁（无 DCL）

```java
public V get(K key) {
    readLock.lock();
    V value = cache.get(key);
    readLock.unlock();          // 先释放读锁

    if (value == null) {
        writeLock.lock();       // 再加写锁
        try {
            // ❌ 没有二次检查！
            // 此时可能已有其他线程完成了加载
            // → 重复查 DB，重复写缓存（轻则浪费，重则数据不一致）
            value = loadFromDB(key);
            cache.put(key, value);
        } finally {
            writeLock.unlock();
        }
    }
    return value;
}
```

**问题：** 在"释放读锁"到"加写锁"的窗口期，10 个线程同时发现 `value == null`，全部去查数据库 → **缓存击穿**。

---

### ✅ 正确写法：释放读锁 → 加写锁 → DCL 二次检查

```java
private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
private final Lock readLock  = rwLock.readLock();
private final Lock writeLock = rwLock.writeLock();
private final Map<K, V> cache = new HashMap<>();

public V get(K key) {
    // ① 先用读锁快速检查缓存
    readLock.lock();
    try {
        V value = cache.get(key);
        if (value != null) {
            return value;  // 命中，直接返回，无竞争
        }
    } finally {
        readLock.unlock();  // ② 必须先释放读锁
    }

    // ③ 缓存未命中，升级为写锁
    writeLock.lock();
    try {
        // ④ DCL：二次检查！（防止重复加载）
        // 原因：步骤②~③之间可能已有线程完成了加载
        V value = cache.get(key);
        if (value == null) {
            value = loadFromDB(key);   // 只有真正为 null 时才查 DB
            cache.put(key, value);
        }
        return value;
    } finally {
        writeLock.unlock();
    }
}
```

**执行时序图：**

```
时间轴 ──────────────────────────────────────────▶

线程A: [readLock]─未命中─[unlock]──[writeLock]─loadDB─put─[unlock]
线程B:            [readLock]─未命中─[unlock]────────等待写锁────[writeLock]─DCL命中─return─[unlock]
线程C:                              [readLock]─────────────────────────────────命中─return─[unlock]
                                                   ↑                              ↑
                                              A释放写锁前B被阻塞             A写完后C直接读命中
```

DCL 保证：**只有第一个拿到写锁的线程查 DB，后续线程直接读缓存**。

## 🔥 扩展追问 1：为什么 HashMap 配读写锁，而不直接用 ConcurrentHashMap？

| | `ConcurrentHashMap` | `HashMap + ReadWriteLock` |
|---|---|---|
| **读-读** | 无锁（volatile读） | 共享读锁，无阻塞 |
| **读-写** | 分段锁，细粒度 | 写锁独占，读全阻塞 |
| **写-写** | 分段锁 | 全局写锁 |
| **复合操作** | 需用 `computeIfAbsent` | 写锁内可任意操作 |
| **适用场景** | 高频读写，细粒度 | **读极多写极少**，需原子复合操作 |

**结论：** 如果只是 `get/put`，直接用 `ConcurrentHashMap.computeIfAbsent` 更简单；读写锁适合**读比写多 10 倍以上**，且需要在锁内执行复杂复合操作的场景。

## 🔥 扩展追问 2：`computeIfAbsent` 能替代上面的写法吗？

```java
// ConcurrentHashMap 版本，更简洁
private final ConcurrentHashMap<K, V> cache = new ConcurrentHashMap<>();

public V get(K key) {
    return cache.computeIfAbsent(key, k -> loadFromDB(k));
}
```

**注意两个坑：**

```java
// ⚠️ 坑1：computeIfAbsent 持有 bin 锁期间调用 loadFromDB
// 如果 loadFromDB 很慢（几百ms），该 hash 桶上其他 key 也会被阻塞
// → JDK 8 bug（同一桶内死锁），JDK 9+ 已修复但仍有性能问题

// ⚠️ 坑2：不能缓存 null 值
// computeIfAbsent 返回 null 时不会存入 map
// → 每次都穿透到 DB（缓存穿透）
// 解决：用哨兵值代替 null
private static final Object NULL_SENTINEL = new Object();

public V get(K key) {
    Object val = cache.computeIfAbsent(key, k -> {
        V v = loadFromDB(k);
        return v != null ? v : NULL_SENTINEL;
    });
    return val == NULL_SENTINEL ? null : (V) val;
}
```

## 🔥 扩展追问 3：本地缓存终极方案——Caffeine

```java
// 直接用 Caffeine，内置异步加载 + 过期 + 容量限制 + 统计
LoadingCache<K, V> cache = Caffeine.newBuilder()
    .maximumSize(10_000)               // LRU/LFU 淘汰
    .expireAfterWrite(5, TimeUnit.MINUTES)
    .refreshAfterWrite(1, TimeUnit.MINUTES)  // 后台异步刷新，不阻塞读
    .recordStats()                     // 命中率统计
    .build(key -> loadFromDB(key));    // 未命中自动加载

// 使用：自动处理未命中加载、并发控制、过期
V value = cache.get(key);
```

**Caffeine 的并发控制原理：**
- 底层用 `ConcurrentHashMap` + `StampedLock`（读走乐观锁路径）
- 未命中时同一 key 只有一个线程去加载（内部做了 DCL）
- 其他线程等待或返回旧值（`refreshAfterWrite` 模式）

## 🔥 扩展追问 4：锁降级的使用场景

```java
// 锁降级：写锁 → 持有写锁的同时加读锁 → 释放写锁
// 用途：写完数据后，想继续用读锁读数据（保证看到自己写的值）

writeLock.lock();
try {
    cache.put(key, newValue);
    
    readLock.lock();  // ✅ 降级：持有写锁时可以加读锁
} finally {
    writeLock.unlock();  // 释放写锁，其他读线程可以进来了
}
// 此时仍持有读锁，可以安全读取 newValue
try {
    return cache.get(key);  // 读到自己刚写的值
} finally {
    readLock.unlock();
}
```

**使用场景：** 写完配置后立即读取并返回给调用方，防止"刚写完被其他写线程覆盖后读到脏数据"。

## 一句话收尾

> 读写锁不支持锁升级，缓存未命中要**先释放读锁再加写锁**，写锁内必须做 **DCL 二次检查**防缓存击穿；生产环境直接上 **Caffeine**，它已经帮你把这些坑全踩过了。
