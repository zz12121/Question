**4. 你用 `ThreadPoolExecutor` 创建了一个线程池：**

```java
new ThreadPoolExecutor(10, 20, 60L, TimeUnit.SECONDS, 
    new LinkedBlockingQueue<>(100));
```

**问题：** 当任务数达到多少时会触发拒绝策略？如果任务持续暴增，会出现什么问题？如何配置更合理？
## 先说结论

这个线程池配置下，当**同时运行 20 个线程 + 队列积压 100 个任务 = 共 120 个任务**时触发拒绝策略。任务持续暴增时，默认的 `AbortPolicy` 会直接抛异常导致任务丢失。更合理的配置需要根据业务场景调整核心线程数、队列容量、拒绝策略，并配套监控告警。

---

## 🔥 核心概念

- **线程池工作流程**：核心线程 → 队列 → 最大线程 → 拒绝策略（注意：不是核心线程满了就直接扩容！）
- **关键参数**：corePoolSize(10)、maximumPoolSize(20)、队列容量(100)
- **拒绝策略**：默认 `AbortPolicy`（抛异常），其他还有 `CallerRunsPolicy`、`DiscardPolicy`、`DiscardOldestPolicy`

---

## 🔥 详细展开

### 1. 任务执行的完整流程

很多人以为是"核心线程满了就扩容"，其实是**先排队再扩容**：

```
提交任务
    │
    ▼
核心线程有空闲？ ──是──→ 直接执行
    │ 否
    ▼
队列未满？ ──是──→ 放入队列等待
    │ 否
    ▼
线程数 < maximumPoolSize？ ──是──→ 创建新线程执行
    │ 否
    ▼
触发拒绝策略！
```

---

### 2. 具体数字推导

```java
new ThreadPoolExecutor(
    10,                          // corePoolSize
    20,                          // maximumPoolSize  
    60L, TimeUnit.SECONDS,       // 空闲线程存活时间
    new LinkedBlockingQueue<>(100) // 队列容量
);
```

| 阶段 | 任务数 | 行为 |
|------|--------|------|
| 阶段1 | 1~10   | 每个任务创建一个核心线程直接执行 |
| 阶段2 | 11~110 | 核心线程忙，**新任务进队列**（不扩核心线程） |
| 阶段3 | 111~120 | 队列满了，才开始扩到 10~20 个线程 |
| 阶段4 | **121+** | 线程数已达 20 且队列满，**触发拒绝策略** |

> 💡 **面试高频坑**：很多人以为任务数超过 `corePoolSize` 就扩线程，实际上是**先填满队列，再扩线程**。只有 `SynchronousQueue`（无容量）才会立即扩线程。

---

### 3. 触发拒绝策略后会发生什么

**默认策略 `AbortPolicy`**：
```java
// 直接抛 RejectedExecutionException
executor.submit(task);
// → java.util.concurrent.RejectedExecutionException!
// 任务丢失，调用方崩溃
```

**四种拒绝策略对比**：

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| `AbortPolicy`（默认） | 抛 `RejectedExecutionException` | 不能丢任务时，让上游感知 |
| `CallerRunsPolicy` | 由**调用方线程**直接执行任务 | 能接受降级，减少任务丢失 |
| `DiscardPolicy` | 静默丢弃新任务 | 可以接受丢失日志等非核心任务 |
| `DiscardOldestPolicy` | 丢弃队列最老的任务，再尝试提交 | 新任务比老任务重要 |

**`CallerRunsPolicy` 的巧妙之处**：
```java
// 调用方线程被迫执行任务 → 暂时停止向线程池提交新任务
// 相当于自然限流，防止任务无限堆积
new ThreadPoolExecutor.CallerRunsPolicy()
```

---

### 4. 任务持续暴增的隐患

**场景：每秒提交 200 个任务，线程池每秒只能处理 80 个**

```
第 1 秒：队列积压 120 个
第 2 秒：队列满了，线程扩到 20，触发拒绝 → 任务开始丢失
第 3 秒：持续丢失，业务数据不完整
......
```

**更严重的情况（无界队列）**：
```java
// ❌ 危险！无界队列
new ThreadPoolExecutor(10, 20, 60L, TimeUnit.SECONDS, 
    new LinkedBlockingQueue<>());  // 没有容量限制

// 任务无限堆积在队列中 → 内存持续增长 → OOM
```

### 5. 合理的配置方案

#### 方案一：IO 密集型场景（数据库查询、网络调用）

```java
int cpuCores = Runtime.getRuntime().availableProcessors();

ThreadPoolExecutor executor = new ThreadPoolExecutor(
    cpuCores * 2,           // corePoolSize：IO 密集，多开
    cpuCores * 4,           // maximumPoolSize
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(500),    // 队列适当大一点
    new ThreadFactory() {              // 自定义线程名，便于排查
        AtomicInteger idx = new AtomicInteger(0);
        public Thread newThread(Runnable r) {
            Thread t = new Thread(r);
            t.setName("io-task-" + idx.getAndIncrement());
            t.setDaemon(false);  // 非守护线程，保证任务执行完
            return t;
        }
    },
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝时降级
);
```

---

#### 方案二：CPU 密集型场景（计算、加密）

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    cpuCores + 1,           // corePoolSize：CPU 密集，不超过核心数
    cpuCores + 1,           // maximumPoolSize 等于 core，避免过多切换
    0L, TimeUnit.SECONDS,   // 不需要回收非核心线程
    new SynchronousQueue<>(),          // 不排队，任务来了立刻执行或拒绝
    factory,
    new ThreadPoolExecutor.AbortPolicy() // 拒绝时明确报错
);
```

---

#### 方案三：完整生产配置（带监控）

```java
@Bean("taskExecutor")
public ThreadPoolExecutor taskExecutor() {
    ThreadPoolExecutor executor = new ThreadPoolExecutor(
        10, 20, 60L, TimeUnit.SECONDS,
        new LinkedBlockingQueue<>(1000),
        new CustomThreadFactory("business"),
        (r, e) -> {
            // 自定义拒绝策略：告警 + 降级
            log.error("线程池任务被拒绝！active={} queue={}", 
                e.getActiveCount(), e.getQueue().size());
            Metrics.counter("threadpool.rejected").increment();
            
            // 降级处理（可以选择：发 MQ / 同步执行 / 丢弃+记录）
            mqTemplate.send("fallback-topic", r);
        }
    );
    
    // 允许核心线程超时（服务空闲时回收资源）
    executor.allowCoreThreadTimeOut(true);
    
    return executor;
}
```

## 🔥 扩展追问 1：`LinkedBlockingQueue` vs `SynchronousQueue` 怎么选？

| 队列 | 特点 | 适用场景 |
|------|------|---------|
| `LinkedBlockingQueue(n)` | 有界缓冲，先排队后扩线程 | IO 密集、可以接受等待 |
| `SynchronousQueue` | 无缓冲，任务来了立刻扩线程 | 实时性高，CPU 密集 |
| `ArrayBlockingQueue(n)` | 有界，FIFO，比 Linked 稍小内存 | 需要严格 FIFO |
| `PriorityBlockingQueue` | 优先级队列 | 任务有优先级 |

> 💡 **关键区别**：`LinkedBlockingQueue` 会导致 maximumPoolSize 形同虚设——因为队列先填满，队列满之前线程数不会超过 core；而 `SynchronousQueue` 才能让线程快速扩到 max。

## 🔥 扩展追问 3：怎么优雅关闭线程池？

```java
// 注册 JVM 关闭钩子
Runtime.getRuntime().addShutdownHook(new Thread(() -> {
    log.info("开始关闭线程池...");
    
    executor.shutdown();  // 不再接受新任务，等待正在执行的任务完成
    
    try {
        // 等待最多 60 秒
        if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
            log.warn("线程池超时未完成，强制关闭");
            executor.shutdownNow();  // 发送 interrupt
        }
    } catch (InterruptedException e) {
        executor.shutdownNow();
        Thread.currentThread().interrupt();
    }
    
    log.info("线程池已关闭");
}));
```

## 🔥 扩展追问 4：线程池异常怎么捕获？

```java
// submit 的异常被 Future 吞掉了！
executor.submit(() -> {
    throw new RuntimeException("出错了");  // 不打日志，不抛出！
});

// ✅ 方案1：execute 替代 submit
executor.execute(() -> {
    throw new RuntimeException("出错了");  // 触发 UncaughtExceptionHandler
});

// ✅ 方案2：submit 后调 Future.get()
Future<?> f = executor.submit(() -> { throw new RuntimeException(); });
try {
    f.get();  // 这里才会抛 ExecutionException
} catch (ExecutionException e) {
    log.error("任务异常", e.getCause());
}

// ✅ 方案3：自定义 afterExecute 钩子
@Override
protected void afterExecute(Runnable r, Throwable t) {
    super.afterExecute(r, t);
    if (t == null && r instanceof Future<?>) {
        try {
            ((Future<?>) r).get(0, TimeUnit.NANOSECONDS);
        } catch (ExecutionException e) {
            t = e.getCause();
        } catch (Exception ignored) {}
    }
    if (t != null) {
        log.error("线程池任务执行异常", t);
    }
}
```

## 一句话收尾

> `ThreadPoolExecutor` 的执行顺序是 **"核心线程 → 队列 → 最大线程 → 拒绝"**（先进队列，不是先扩线程），这题触发拒绝是在第 121 个任务。配置要点是 **"有界队列 + CallerRunsPolicy 降级 + 监控告警"**，千万不能用无界队列，OOM 就是这么来的。
