**2. 你用 `synchronized` 实现了一个简单的缓存：**

```java
public synchronized V get(K key) {

    if (value == null) {
        value = loadFromDB(key);
        cache.put(key, value);
    }
    return value;
}
```

**问题：** 高并发下有什么问题？如何优化性能同时保证线程安全？
## 先说结论

这个缓存实现的问题是 **"锁粒度太粗"**——`get` 方法整个被 `synchronized` 修饰，高并发时所有读写串行，性能极差。更严重的是 **"缓存穿透时，多个线程会重复查 DB"**（经典的缓存击穿问题）。优化方向是 **"缩小锁粒度 + 双重检查锁 + 异步加载"**，生产环境直接用 `Caffeine` 或 `Guava Cache`，它们已经完美解决了这些问题。

### 2. 优化方案演进

#### 方案一：缩小锁粒度（ConcurrentHashMap）

```java
public class ConcurrentCache<K, V> {
    private final ConcurrentHashMap<K, V> cache = new ConcurrentHashMap<>();
    
    // 读不加锁，写才加锁（ConcurrentHashMap 分段锁）
    public V get(K key) {
        V value = cache.get(key);
        if (value != null) {
            return value;  // 缓存命中，无锁返回
        }
        
        // 只有未命中才加锁，且只锁这个 key
        synchronized (this) {  // 还可以优化：锁 key，不是锁整个 cache
            value = cache.get(key);  // 双重检查
            if (value == null) {
                value = loadFromDB(key);
                cache.put(key, value);
            }
        }
        return value;
    }
}
```

**改进点**：
- 缓存命中时无锁，并发读性能极好
- 但未命中时还是串行，缓存击穿问题还在

#### 方案二：双重检查锁（DCL）+ 细粒度锁

```java
public class OptimizedCache<K, V> {
    private final ConcurrentHashMap<K, V> cache = new ConcurrentHashMap<>();
    private final ConcurrentHashMap<K, Object> locks = new ConcurrentHashMap<>();
    
    public V get(K key) {
        V value = cache.get(key);
        if (value != null) {
            return value;
        }
        
        // 每个 key 独立的锁对象
        Object keyLock = locks.computeIfAbsent(key, k -> new Object());
        
        synchronized (keyLock) {
            // 双重检查：其他线程可能已加载
            value = cache.get(key);
            if (value != null) {
                return value;
            }
            
            // 只有一个线程查 DB
            value = loadFromDB(key);
            cache.put(key, value);
        }
        
        // 清理锁对象（可选）
        locks.remove(key, keyLock);
        return value;
    }
}
```

**改进点**：
- 不同 key 的查询并行，互不影响
- 同一 key 只有一个线程查 DB（解决缓存击穿）

#### 方案三：异步加载（防止阻塞）

```java
public class AsyncLoadingCache<K, V> {
    private final ConcurrentHashMap<K, CompletableFuture<V>> loading = new ConcurrentHashMap<>();
    
    public V get(K key) {
        // 1. 先检查是否正在加载
        CompletableFuture<V> future = loading.get(key);
        if (future != null) {
            return future.join();  // 等待加载完成
        }
        
        // 2. 尝试启动加载
        CompletableFuture<V> newFuture = CompletableFuture.supplyAsync(() -> {
            try {
                return loadFromDB(key);
            } finally {
                loading.remove(key);  // 加载完成清理
            }
        });
        
        CompletableFuture<V> existing = loading.putIfAbsent(key, newFuture);
        if (existing != null) {
            // 其他线程先启动了，用它的
            newFuture.cancel(true);
            return existing.join();
        }
        
        // 3. 等待加载完成，放入缓存
        V value = newFuture.join();
        cache.put(key, value);
        return value;
    }
}
```

**改进点**：
- 加载过程异步，不阻塞其他 key
- 同一 key 只有一个线程执行加载

### 2. 优化方案演进

#### 方案一：双重检查锁定（DCL）—— 解决穿透问题

```java
public class Cache<K, V> {
    private final Map<K, V> cache = new ConcurrentHashMap<>();
    private final Object[] locks = new Object[128];  // 分段锁
    
    {
        for (int i = 0; i < locks.length; i++) {
            locks[i] = new Object();
        }
    }
    
    public V get(K key) {
        // 1. 先无锁读（绝大多数情况命中）
        V value = cache.get(key);
        if (value != null) {
            return value;
        }
        
        // 2. 未命中，加锁加载（只锁相同 hash 的 key）
        synchronized (locks[hash(key) % locks.length]) {
            // 3. 双重检查：可能其他线程已加载
            value = cache.get(key);
            if (value != null) {
                return value;
            }
            
            // 4. 只有一个线程去查 DB
            value = loadFromDB(key);
            cache.put(key, value);
            return value;
        }
    }
}
```

**优点**：
- 读缓存无锁，并行度高
- 缓存未命中时，同 hash 的 key 串行，不同 key 并行
- 只有一个线程查数据库

#### 方案二：ConcurrentHashMap.computeIfAbsent（推荐）

```java
public class Cache<K, V> {
    private final Map<K, V> cache = new ConcurrentHashMap<>();
    
    public V get(K key) {
        return cache.computeIfAbsent(key, this::loadFromDB);
    }
    
    private V loadFromDB(K key) {
        // 这个方法只会被调用一次（原子性保证）
        System.out.println("Loading from DB: " + key);
        return db.query(key);
    }
}
```

**原理**：
- `ConcurrentHashMap` 内部用分段锁（Java 8 是 CAS + synchronized）
- `computeIfAbsent` 保证**原子性**：只有一个线程执行 `mappingFunction`
- 其他线程阻塞等待，完成后直接返回结果

#### 方案三：ReadWriteLock —— 读多写少极致优化

```java
public class Cache<K, V> {
    private final Map<K, V> cache = new HashMap<>();
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Lock readLock = rwLock.readLock();
    private final Lock writeLock = rwLock.writeLock();
    
    public V get(K key) {
        // 1. 读锁获取
        readLock.lock();
        try {
            V value = cache.get(key);
            if (value != null) {
                return value;
            }
        } finally {
            readLock.unlock();
        }
        
        // 2. 未命中，升级写锁
        writeLock.lock();
        try {
            // 3. 双重检查
            V value = cache.get(key);
            if (value != null) {
                return value;
            }
            
            value = loadFromDB(key);
            cache.put(key, value);
            return value;
        } finally {
            writeLock.unlock();
        }
    }
    
    public void put(K key, V value) {
        writeLock.lock();
        try {
            cache.put(key, value);
        } finally {
            writeLock.unlock();
        }
    }
}
```

**特点**：
- 读读不互斥，读写互斥，写写互斥
- 适合读极多、写极少的场景
- 写锁升级要小心（必须先释放读锁）
