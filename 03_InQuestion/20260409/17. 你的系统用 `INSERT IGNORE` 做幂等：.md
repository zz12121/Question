**17. 你的系统用 `INSERT IGNORE` 做幂等：**

```sql
INSERT IGNORE INTO order_unique (order_id) VALUES ('12345');
```

**问题：** 高并发下会出现什么问题？如果表上有唯一索引，和 `INSERT ... ON DUPLICATE KEY UPDATE` 有什么区别？
## 先说结论

`INSERT IGNORE` 高并发下会出现**死锁**（gap lock 冲突）和**主从延迟**（statement 格式 binlog），且无法获取"是否已存在"的状态。相比之下 `ON DUPLICATE KEY UPDATE` 可以更新字段并返回受影响行数，但同样有锁冲突问题。生产环境推荐**数据库唯一索引 + 业务层分布式锁**或**Redis 去重**。

---

## 🔥 核心概念——INSERT IGNORE 的执行流程

```sql
INSERT IGNORE INTO order_unique (order_id) VALUES ('12345');

执行过程：
  1. 尝试插入 order_id='12345'
  2. 如果唯一索引冲突 → 忽略错误，返回 0 行受影响
  3. 如果没有冲突 → 插入成功，返回 1 行受影响

问题：
  - 无法区分"插入成功"和"冲突忽略"
  - 高并发时多个事务同时尝试插入同一值 → 死锁
```

---

## 🔥 详细展开

### 问题一：高并发死锁

```sql
-- 表结构
CREATE TABLE order_unique (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    order_id VARCHAR(32) UNIQUE,  -- 唯一索引
    create_time DATETIME DEFAULT NOW()
) ENGINE=InnoDB;

-- 场景：两个事务同时插入相同的 order_id
-- 事务 A
BEGIN;
INSERT IGNORE INTO order_unique (order_id) VALUES ('12345');  -- 成功插入

-- 事务 B（同时执行）
BEGIN;
INSERT IGNORE INTO order_unique (order_id) VALUES ('12345');  -- 等待 A 的锁

-- 事务 A 再插入另一个值
INSERT IGNORE INTO order_unique (order_id) VALUES ('67890');  -- 可能死锁！

-- 死锁原因：
-- 1. 唯一索引的 gap lock 和 insert intention lock 冲突
-- 2. 多个事务以不同顺序插入多个值时极易死锁
```

**InnoDB 锁分析：**

```
唯一索引插入时的锁：
  1. 先加 gap lock（防止幻读）
  2. 检查唯一性
  3. 加 insert intention lock（插入意向锁）
  4. 插入成功，锁升级为记录锁

高并发下：
  - 多个事务同时检查唯一性 → 都拿到 gap lock
  - 都尝试加 insert intention lock → 互相等待 → 死锁
```

### 问题二：主从延迟（statement 格式 binlog）

```sql
-- binlog_format=STATEMENT 时
INSERT IGNORE ... 
-- 主库执行：冲突忽略
-- 从库执行：可能时序不同，先插入了，不冲突
-- → 主从数据不一致！

-- 解决：改用 ROW 格式 binlog
SET GLOBAL binlog_format = 'ROW';
-- 或 MIXED 格式
```

### 问题三：无法获取插入状态

```java
// JDBC 执行
int affectedRows = stmt.executeUpdate(sql);
// INSERT IGNORE 时：
//   插入成功 → affectedRows = 1
//   冲突忽略 → affectedRows = 0
// 但无法知道是"已存在"还是"其他错误"

// 需要查 last_insert_id 或再查询一次，增加一次 IO
```

## 🔥 扩展追问 1：ON DUPLICATE KEY UPDATE 对比

```sql
INSERT INTO order_unique (order_id, status) 
VALUES ('12345', 'NEW')
ON DUPLICATE KEY UPDATE 
    status = VALUES(status),
    update_time = NOW();

-- 行为：
--   无冲突 → 插入新记录，返回 affected_rows = 1
--   有冲突 → 更新指定字段，返回 affected_rows = 2（MySQL 特殊定义）

-- 获取状态
int affected = stmt.executeUpdate();
if (affected == 1) {
    // 插入成功
} else if (affected == 2) {
    // 更新成功（已存在）
}
```

| 特性 | INSERT IGNORE | ON DUPLICATE KEY UPDATE |
|------|---------------|------------------------|
| 冲突时行为 | 忽略，不报错 | 执行 UPDATE |
| 返回值 | 0（冲突）或 1（成功） | 1（插入）或 2（更新） |
| 可更新字段 | ❌ 不能 | ✅ 可以 |
| 死锁风险 | 有 | 有（更大，因为 UPDATE 加 X 锁）|
| 主从安全 | STATEMENT 格式有风险 | 同样有 |
| 使用场景 | 简单去重 | 需要更新字段（如计数器）|

## 🔥 扩展追问 2：高并发幂等的正确方案

### 方案一：数据库唯一索引 + 分布式锁

```java
@Service
public class OrderService {
    
    @Autowired
    private StringRedisTemplate redis;
    
    @Autowired
    private OrderUniqueMapper orderUniqueMapper;
    
    public void createOrder(String orderId) {
        // 1. 分布式锁防并发
        String lockKey = "lock:order:" + orderId;
        Boolean locked = redis.opsForValue()
            .setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS);
        
        if (!Boolean.TRUE.equals(locked)) {
            throw new DuplicateException("订单处理中，请勿重复提交");
        }
        
        try {
            // 2. 查重（可选，减少 DB 压力）
            if (orderUniqueMapper.exists(orderId)) {
                return;  // 已存在，幂等返回
            }
            
            // 3. 插入唯一索引表
            orderUniqueMapper.insert(orderId);
            
            // 4. 执行业务逻辑
            doCreateOrder(orderId);
            
        } catch (DuplicateKeyException e) {
            // 唯一索引冲突，说明已处理
            log.info("订单已存在，幂等返回: {}", orderId);
        } finally {
            redis.delete(lockKey);
        }
    }
}
```

### 方案二：Redis SETNX 去重

```java
public boolean idempotentCreate(String orderId) {
    // Redis SET key value NX EX 10
    // NX = 不存在才设置，EX = 过期时间
    Boolean success = redis.opsForValue()
        .setIfAbsent("idempotent:" + orderId, "1", 24, TimeUnit.HOURS);
    
    if (!Boolean.TRUE.equals(success)) {
        // 已存在，幂等返回
        return false;
    }
    
    try {
        doCreateOrder(orderId);
        return true;
    } catch (Exception e) {
        // 失败时删除 Redis key，允许重试
        redis.delete("idempotent:" + orderId);
        throw e;
    }
}
```

**优点：**
- 完全避免数据库锁冲突
- 性能高（Redis 10万 QPS）
- 可设置过期时间自动清理

**缺点：**
- Redis 挂了会丢失幂等性（需配合数据库唯一索引兜底）
- 需要处理 Redis 和 DB 的一致性

### 方案三：Token 机制（前端配合）

```java
// 1. 用户提交前申请 Token
@GetMapping("/token")
public String applyToken() {
    String token = UUID.randomUUID().toString();
    redis.opsForValue().set("token:" + token, "0", 5, TimeUnit.MINUTES);
    return token;
}

// 2. 提交时携带 Token
@PostMapping("/order")
public void createOrder(@RequestParam String token, @RequestBody OrderRequest req) {
    // 使用 Token 做幂等（Lua 原子操作）
    String lua = "if redis.call('get', KEYS[1]) == '0' then " +
                 "  redis.call('set', KEYS[1], '1'); " +
                 "  return 1; " +
                 "else " +
                 "  return 0; " +
                 "end";
    
    Long result = redis.execute(new DefaultRedisScript<>(lua, Long.class), 
                                Collections.singletonList("token:" + token));
    
    if (result == 0) {
        throw new DuplicateException("重复提交");
    }
    
    // 执行业务
    doCreateOrder(req);
}
```

## 🔥 扩展追问 3：批量插入的幂等

```java
// 批量插入时，INSERT IGNORE 更高效
INSERT IGNORE INTO order_unique (order_id) 
VALUES ('A'), ('B'), ('C'), ('D');

// 但冲突时不知道哪个冲突
// 方案：先查重，再批量插入未存在的

// 1. 查询已存在的
SELECT order_id FROM order_unique 
WHERE order_id IN ('A', 'B', 'C', 'D');

// 2. 过滤后批量插入
INSERT INTO order_unique (order_id) VALUES ('C'), ('D');  -- A,B 已存在
```

## 一句话收尾

> `INSERT IGNORE` 高并发会**死锁**且**无法感知冲突状态**，生产环境用**分布式锁 + 唯一索引**或**Redis SETNX**做幂等，数据库只作为最终兜底，把锁冲突转移到性能更高的 Redis。
