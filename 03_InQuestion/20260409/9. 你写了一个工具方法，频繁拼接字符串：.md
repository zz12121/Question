**9. 你写了一个工具方法，频繁拼接字符串：**

```java
public String buildLog(User user, Order order) {
    String result = "";
    result += "user=" + user.getName();
    result += ", order=" + order.getId();
    result += ", time=" + System.currentTimeMillis();
    return result;
}
```

**问题：** 高并发下这段代码有什么问题？JVM 层面会发生什么？如何优化？
## 先说结论

这段代码每次拼接都创建新的 `String` 对象，触发大量年轻代 GC，高并发下 CPU 飙高、吞吐量暴跌。优化用 `StringBuilder`（单线程）或 `StringBuffer`（线程安全），日志场景直接用占位符让框架延迟计算。

---

## 🔥 核心概念——字符串拼接的底层真相

```java
result += "user=" + user.getName();
// 编译后等价于：
result = new StringBuilder(result)
         .append("user=")
         .append(user.getName())
         .toString();
```

**问题：**
- 每次 `+=` 都新建 `StringBuilder` → 新建 `char[]` → 拷贝旧内容 → 追加 → `toString()` 再新建 `String`
- 3 次拼接 = 3 个 `StringBuilder` + 3 个 `char[]` + 3 个 `String` 对象

---

## 🔥 详细展开

### 问题一：对象创建爆炸

```
假设 buildLog 每秒调用 10 万次：

原始代码：
  每次调用：3 次 += → 9 个临时对象（3 SB + 3 char[] + 3 String）
  每秒创建：90 万个对象
  
  年轻代 Eden 区（默认几百 MB）很快被填满
  → 频繁触发 Young GC（每秒几十次）
  → GC 线程占用 CPU
  → 业务线程停顿（STW）
  → 吞吐量下降 30%~50%
```

### 问题二：char[] 拷贝开销

```java
// StringBuilder 扩容机制
char[] value;  // 默认初始容量 16

// 每次 append 可能触发扩容：
if (count + len > value.length) {
    value = Arrays.copyOf(value, newCapacity(count + len));  // ← 数组拷贝，O(n)
}
```

原始代码中 `StringBuilder` 刚创建就 `toString()`，根本**无法复用扩容后的数组**。

### 问题三：`System.currentTimeMillis()` 的隐藏成本

```java
", time=" + System.currentTimeMillis()
// 先调用 currentTimeMillis() → 返回 long
// long 转 String → 创建新的 String 对象
// 再拼接
```

虽然单次很快，但高频调用下也是负担。

---

## 🔥 扩展追问 1：如何优化？

### 方案一：手动 StringBuilder（推荐）

```java
public String buildLog(User user, Order order) {
    // 预估容量，避免扩容（user.name 约 20，order.id 约 20，time 约 13）
    StringBuilder sb = new StringBuilder(64);
    sb.append("user=").append(user.getName())
      .append(", order=").append(order.getId())
      .append(", time=").append(System.currentTimeMillis());
    return sb.toString();
}
```

**效果：**
- 1 个 `StringBuilder` + 1 个 `char[]` + 1 个最终 `String`
- 对象创建减少 **88%**（从 9 个降到 1 个）

### 方案二：StringBuffer（线程安全，但一般不需要）

```java
// 如果 buildLog 会被多线程并发调用，且需要共享同一个 buffer
// 但通常不会这样设计，每个线程用自己的 StringBuilder 即可
StringBuffer sb = new StringBuffer(64);  // 所有方法 synchronized，性能差
```

**结论：** 除非真的需要线程安全，否则用 `StringBuilder`。

### 方案三：日志框架占位符（最佳实践）

```java
// 不要自己拼接，让日志框架来做
// SLF4J / Logback
log.info("user={}, order={}, time={}", 
         user.getName(), 
         order.getId(), 
         System.currentTimeMillis());

// 优势：
// 1. 如果当前日志级别是 WARN，INFO 日志不会执行 toString() 和拼接
//    → 零开销（延迟计算）
// 2. 日志框架内部用 StringBuilder，且会复用（ThreadLocal）
// 3. 代码可读性更好
```

### 方案四：Java 15+ 的 Text Blocks（多行文本）

```java
// 如果日志格式固定，用文本块更清晰
String log = """
    user=%s, order=%s, time=%d
    """.formatted(user.getName(), order.getId(), System.currentTimeMillis());
// formatted 内部也是用 StringBuilder，但语法更优雅
```

## 🔥 扩展追问 2：JVM 层面的优化——字符串常量池与 intern

```java
// 字符串常量池复用
String a = "hello";  // 放入常量池
String b = "hello";  // 直接从常量池取，a == b 为 true

// 但拼接结果不会自动入池
String c = "hel" + "lo";  // 编译期优化，等价于 "hello"，会入池
String d = new String("hello");  // 堆上新建对象，d != a

// 手动入池
String e = new String("hello").intern();  // e == a
```

**对当前问题的启示：**
- 日志中的固定前缀 `"user="`、`", order="` 会被编译器优化为常量
- 但动态拼接结果每次都在堆上新建，无法复用

## 🔥 扩展追问 3：编译器优化——Java 9+ 的 indy 字符串拼接

```java
// Java 9 之前：编译为 StringBuilder.append 链
// Java 9+：使用 invokedynamic + StringConcatFactory

// 编译后的字节码变成：
invokedynamic makeConcatWithConstants:(Ljava/lang/String;J)Ljava/lang/String;
  StringConcatFactory.makeConcatWithConstants(
      MethodHandles.lookup(),
      "makeConcatWithConstants",
      MethodType.methodType(String.class, String.class, long.class),
      "user=\u0001, order=\u0002, time=\u0001",  // 配方
      new Object[]{}
  );

// 优势：
// - JVM 可以选择最优策略（StringBuilder / StringConcatHelper / 数组拷贝）
// - 可能复用 byte[] 而不是 char[]（Java 9 String 底层改为 byte[] + coder）
// - 但高频调用下，手动 StringBuilder 仍然更可控
```

public void test() {

## 一句话收尾

> `+=` 拼接是"对象创建炸弹"，高并发下 GC 压垮系统；优化用 `StringBuilder` 预分配容量，日志场景直接用占位符让框架延迟计算——**不要拼接，要追加**。
