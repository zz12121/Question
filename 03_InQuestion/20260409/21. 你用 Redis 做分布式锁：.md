**21. 你用 Redis 做分布式锁：**

```java
String result = jedis.set(lockKey, requestId, "NX", "EX", 30);
if ("OK".equals(result)) {
    try {
        doBusiness();
    } finally {
        jedis.del(lockKey);
    }
}
```

**问题：** 如果 `doBusiness()` 执行超过 30 秒，会发生什么？如果服务挂了，锁会怎样？如何改进？
## 先说结论

`doBusiness()` 超过 30 秒会导致**锁过期被其他线程获取**，出现**并发安全问题**；服务挂了锁自动过期是正常设计，但业务可能未完成。改进方案：**Watchdog 自动续期**、**RedLock 多节点**、或**数据库唯一索引兜底**。

## 🔥 核心概念——当前代码的问题

```
执行流程：
  1. SET lockKey requestId NX EX 30  → 获取锁，30秒过期
  2. doBusiness() 执行
  3. DEL lockKey  → 释放锁

问题场景：
  ├─ doBusiness() 执行 40 秒
  │   30秒时锁自动过期
  │   线程B获取锁，执行业务
  │   40秒时线程A执行DEL，删除了线程B的锁！
  │   → 并发安全问题
  │
  └─ 服务在 doBusiness() 中挂了
      锁30秒后自动过期，其他线程可以获取
      但业务可能未完成（部分成功）
      → 需要业务幂等
```

## 🔥 扩展追问 1：Watchdog 自动续期

```java
// Redisson 实现（生产推荐）
RLock lock = redisson.getLock(lockKey);
try {
    // 尝试获取锁，最多等待10秒，锁30秒后自动释放
    boolean locked = lock.tryLock(10, 30, TimeUnit.SECONDS);
    if (locked) {
        doBusiness();
    }
} finally {
    lock.unlock();
}

// Redisson Watchdog 机制：
//   - 获取锁后，启动后台线程
//   - 每 10 秒（过期时间的1/3）检查锁是否仍被持有
//   - 如果持有，自动续期到 30 秒
//   - 业务完成或崩溃后，不再续期，锁自然过期
```

**自己实现 Watchdog：**

```java
@Component
public class DistributedLock {
    
    @Autowired
    private StringRedisTemplate redis;
    
    private final ScheduledExecutorService scheduler = 
        Executors.newScheduledThreadPool(10);
    
    public boolean tryLock(String lockKey, String requestId, int expireSeconds) {
        Boolean success = redis.opsForValue()
            .setIfAbsent(lockKey, requestId, expireSeconds, TimeUnit.SECONDS);
        
        if (Boolean.TRUE.equals(success)) {
            // 启动续期任务
            startRenewal(lockKey, requestId, expireSeconds);
            return true;
        }
        return false;
    }
    
    private void startRenewal(String lockKey, String requestId, int expireSeconds) {
        scheduler.scheduleAtFixedRate(() -> {
            String currentValue = redis.opsForValue().get(lockKey);
            if (requestId.equals(currentValue)) {
                // 锁仍被自己持有，续期
                redis.expire(lockKey, expireSeconds, TimeUnit.SECONDS);
            }
        }, expireSeconds / 3, expireSeconds / 3, TimeUnit.SECONDS);
    }
    
    public void unlock(String lockKey, String requestId) {
        // Lua 脚本保证原子性
        String lua = "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                     "return redis.call('del', KEYS[1]) else return 0 end";
        redis.execute(new DefaultRedisScript<>(lua, Long.class), 
                      Collections.singletonList(lockKey), requestId);
        
        // 取消续期任务
        stopRenewal(lockKey);
    }
}
```

## 🔥 扩展追问 3：RedLock 多节点方案

```
单节点 Redis 的问题：
  - Master 挂了，锁可能丢失（异步复制延迟）
  - 需要 Redisson RedLock 或 Redis 6.0 的 ACL

RedLock 算法：
  1. 向 N 个独立的 Redis 节点申请锁
  2. 计算获取锁的总耗时
  3. 如果成功获取超过半数节点，且总耗时小于锁过期时间，则获取成功
  4. 释放时向所有节点发送释放命令

Redisson 实现：
```java
Config config = new Config();
config.useRedLock()
    .addNodeAddress("redis://192.168.0.1:7000")
    .addNodeAddress("redis://192.168.0.2:7000")
    .addNodeAddress("redis://192.168.0.3:7000");

RedissonClient redisson = Redisson.create(config);
RLock lock = redisson.getRedLock(lock1, lock2, lock3);
```
