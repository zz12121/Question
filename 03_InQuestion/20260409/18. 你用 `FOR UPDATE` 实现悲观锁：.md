**18. 你用 `FOR UPDATE` 实现悲观锁：**

```sql
BEGIN;
SELECT * FROM inventory WHERE product_id = 100 FOR UPDATE;
UPDATE inventory SET stock = stock - 1 WHERE product_id = 100;
COMMIT;
```

**问题：** 如果 `product_id = 100` 不存在，会发生什么？如果库存扣减逻辑复杂（涉及多个表），锁粒度如何控制？
## 先说结论

`product_id = 100` 不存在时，InnoDB 会加**间隙锁（Gap Lock）**，阻塞范围内其他插入，可能导致死锁。复杂扣减逻辑应**缩小锁粒度**（单行记录锁）、**减少锁持有时间**（事务内不做 IO）、或改用**乐观锁**（版本号/CAS）。

---

## 🔥 核心概念——FOR UPDATE 的锁行为

```
InnoDB 锁类型：
  1. 记录锁（Record Lock）：锁定存在的记录
  2. 间隙锁（Gap Lock）：锁定索引记录之间的间隙
  3. 临键锁（Next-Key Lock）：记录锁 + 间隙锁（默认）

SELECT ... FOR UPDATE 行为：
  - 记录存在 → 加 X 记录锁（排他锁）
  - 记录不存在 → 加 Gap Lock（间隙锁）
```

## 🔥 详细展开

### 问题一：记录不存在时的 Gap Lock

```sql
-- 表数据：product_id = 1, 5, 10
-- 查询不存在的 product_id = 100
BEGIN;
SELECT * FROM inventory WHERE product_id = 100 FOR UPDATE;
-- 结果：Empty set
-- 但加了 Gap Lock！锁定 (10, +∞) 的间隙

-- 另一个事务
BEGIN;
INSERT INTO inventory (product_id, stock) VALUES (50, 100);
-- 被阻塞！因为 50 在 (10, +∞) 间隙内
-- 直到第一个事务提交/回滚
```

**Gap Lock 的影响：**

```
场景：高并发创建新商品库存
  事务A：SELECT * WHERE product_id = 100 FOR UPDATE;  -- 不存在，加 (10, +∞) Gap Lock
  事务B：SELECT * WHERE product_id = 200 FOR UPDATE;  -- 不存在，加 (10, +∞) Gap Lock
  事务A：INSERT product_id = 100;  -- 等待 B 释放 Gap Lock
  事务B：INSERT product_id = 200;  -- 等待 A 释放 Gap Lock
  → 死锁！
```

### 问题二：复杂扣减逻辑的锁粒度问题

```java
@Service
public class InventoryService {
    
    @Transactional
    public void deductStock(Long productId, Integer quantity) {
        // 1. 加锁查询
        Inventory inv = inventoryMapper.selectForUpdate(productId);
        
        // 2. 复杂校验（涉及多个表查询）
        checkSupplierStatus(inv.getSupplierId());      // 查供应商表
        checkPromotionRules(productId);                // 查促销规则表
        calculateFinalPrice(inv, quantity);            // 复杂计算
        
        // 3. 扣减库存
        inventoryMapper.updateStock(productId, inv.getStock() - quantity);
        
        // 4. 记录日志
        insertStockLog(productId, quantity);           // 插入日志表
        
        // 问题：
        // - 步骤1的锁一直持有到事务提交
        // - 步骤2~4 执行时间长，锁持有时间长
        // - 其他事务阻塞，吞吐量下降
    }
}
```

## 🔥 扩展追问 1：优化方案——缩小锁粒度

### 方案一：先查后锁（避免 Gap Lock）

```java
public void deductStock(Long productId, Integer quantity) {
    // 1. 先查（不加锁），确认存在
    Inventory inv = inventoryMapper.selectById(productId);
    if (inv == null) {
        throw new NotFoundException("商品不存在");
    }
    
    // 2. 再锁（只锁存在的记录，避免 Gap Lock）
    // 使用 UPDATE 代替 SELECT FOR UPDATE，减少一次查询
    int affected = inventoryMapper.updateStock(productId, quantity, inv.getVersion());
    if (affected == 0) {
        throw new ConcurrentException("库存不足或并发冲突");
    }
}
```

### 方案二：事务内不做 IO

```java
@Transactional
public void deductStock(Long productId, Integer quantity) {
    // 1. 只保留最小锁粒度操作
    int affected = inventoryMapper.updateStock(productId, quantity);
    if (affected == 0) {
        throw new StockNotEnoughException();
    }
    
    // 2. 其他操作移到事务外
    // 不要在这里查供应商、算价格、插日志！
}

// 事务外异步处理
@Async
public void asyncProcess(Long productId, Integer quantity) {
    // 记录日志、发送 MQ、计算价格等
}
```

## 🔥 扩展追问 2：乐观锁替代方案

```java
// 乐观锁：版本号控制
public void deductStockWithOptimisticLock(Long productId, Integer quantity) {
    int retry = 3;
    while (retry-- > 0) {
        // 1. 查版本号（不加锁）
        Inventory inv = inventoryMapper.selectById(productId);
        
        // 2. 校验库存
        if (inv.getStock() < quantity) {
            throw new StockNotEnoughException();
        }
        
        // 3. 更新（CAS）
        int affected = inventoryMapper.updateStockWithVersion(
            productId, 
            inv.getStock() - quantity,
            inv.getVersion(),      // WHERE version = #{version}
            inv.getVersion() + 1   // SET version = version + 1
        );
        
        if (affected == 1) {
            return;  // 成功
        }
        // 失败则重试
    }
    throw new ConcurrentException("并发冲突，请重试");
}
```

**适用场景对比：**

| 场景 | 悲观锁 | 乐观锁 |
|------|--------|--------|
| 写冲突概率 | 高 | 低 |
| 重试成本 | 阻塞等待 | 轻量重试 |
| 适用 | 库存强一致性 | 读多写少 |
| 性能 | 低（锁竞争） | 高（无锁） |

## 🔥 扩展追问 3：Redis 分布式锁优化

```java
// 高并发场景，先用 Redis 限流，再操作数据库
public void deductStock(Long productId, Integer quantity) {
    String lockKey = "stock:lock:" + productId;
    String lockValue = UUID.randomUUID().toString();
    
    try {
        // 1. Redis 加锁（快速失败）
        Boolean locked = redis.opsForValue()
            .setIfAbsent(lockKey, lockValue, 5, TimeUnit.SECONDS);
        if (!Boolean.TRUE.equals(locked)) {
            throw new BusyException("系统繁忙，请稍后再试");
        }
        
        // 2. 查缓存库存（快速判断）
        Integer stock = getStockFromCache(productId);
        if (stock < quantity) {
            throw new StockNotEnoughException();
        }
        
        // 3. 异步扣减数据库（MQ 或线程池）
        asyncDeductStock(productId, quantity);
        
        // 4. 扣减缓存
        decrementCacheStock(productId, quantity);
        
    } finally {
        // 释放锁（Lua 保证原子性）
        unlock(lockKey, lockValue);
    }
}
```

## 一句话收尾

> `FOR UPDATE` 查不到记录会加 **Gap Lock** 引发死锁，复杂业务要**缩小锁粒度**（UPDATE 代替 SELECT FOR UPDATE）、**事务内不做 IO**、或改用**乐观锁/Redis 分布式锁**，核心原则是**锁范围最小、持有时间最短**。
