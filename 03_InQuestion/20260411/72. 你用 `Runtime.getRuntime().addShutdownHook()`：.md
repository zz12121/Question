72. 你用 `Runtime.getRuntime().addShutdownHook()`：
```java
Runtime.getRuntime().addShutdownHook(new Thread(() -> {
    // 清理资源
}));
```
**问题：** ShutdownHook 在什么情况下会执行？什么情况下不会执行？

---
## 先说结论

ShutdownHook 在 **JVM 正常关闭时** 执行（如 `System.exit()`、所有非守护线程结束、收到 SIGTERM 信号），但 **不会执行** 的情况包括：被 `kill -9` 强制终止、JVM 崩溃（如段错误）、电源断电、调用 `Runtime.halt()`。Hook 的执行无超时保证，且多个 Hook 是**并发无序执行**的。

---

## 🔥 会执行 ShutdownHook 的场景

| 场景 | 触发方式 | 说明 |
|---|---|---|
| **程序正常结束** | `main()` 执行完毕，且无非守护线程 | 最常见 |
| **显式退出** | `System.exit(0)` / `System.exit(1)` | 主动调用 |
| **信号触发** | `kill -15` (SIGTERM)、Ctrl+C (SIGINT) | 优雅关闭 |
| **IDE 停止** | 点击"停止"按钮（发送 SIGTERM） | 开发环境 |
| **容器关闭** | Docker/K8s 的 `docker stop`（默认 10s 宽限期） | 云原生环境 |

```java
Runtime.getRuntime().addShutdownHook(new Thread(() -> {
    System.out.println("ShutdownHook 执行：关闭连接池、刷盘日志...");
    dataSource.close();
    logger.shutdown();
}));
```

---

## 🔥 不会执行 ShutdownHook 的场景

| 场景 | 触发方式 | 说明 |
|---|---|---|
| **强制杀死** | `kill -9` (SIGKILL) | 操作系统直接终止，无处理机会 |
| **JVM 崩溃** | 段错误（SIGSEGV）、堆外内存访问违规 | 进程直接死亡 |
| **电源故障** | 断电、硬件故障 | 物理层面终止 |
| **强制 halt** | `Runtime.halt()` | 立即终止，不执行 Hook |
| **其他进程杀死** | `taskkill /F` (Windows 强制) | 类似 `kill -9` |

```java
// ❌ 这些情况下 ShutdownHook 不会执行：

// 1. kill -9 <pid>
// 操作系统直接回收进程资源，JVM 来不及反应

// 2. Runtime.halt(0)
Runtime.getRuntime().halt(0);  // 立即终止，Hook 不执行

// 3. 段错误（JNI 代码导致）
// JVM 崩溃，生成 hs_err_pid.log，Hook 不执行
```

## 🔥 实战：正确的资源清理模式

```java
public class GracefulShutdown {
    
    private final ExecutorService executor;
    private final DataSource dataSource;
    
    public GracefulShutdown() {
        this.executor = Executors.newFixedThreadPool(10);
        this.dataSource = createDataSource();
        
        // 注册 ShutdownHook
        Runtime.getRuntime().addShutdownHook(new Thread(this::onShutdown));
    }
    
    private void onShutdown() {
        System.out.println("开始优雅关闭...");
        
        // 1. 停止接收新任务
        executor.shutdown();
        
        try {
            // 2. 等待现有任务完成（最多 30 秒）
            if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
                System.err.println("任务未在 30 秒内完成，强制关闭");
                List<Runnable> dropped = executor.shutdownNow();
                System.err.println("丢弃了 " + dropped.size() + " 个未执行任务");
            }
        } catch (InterruptedException e) {
            executor.shutdownNow();
            Thread.currentThread().interrupt();
        }
        
        // 3. 关闭数据库连接池
        dataSource.close();
        
        // 4. 刷盘日志
        LoggerContext.getContext().stop();
        
        System.out.println("优雅关闭完成");
    }
}
```

## 🔥 扩展追问 1：多个 ShutdownHook 的执行顺序

```java
Runtime.getRuntime().addShutdownHook(new Thread(() -> 
    System.out.println("Hook 1")));
Runtime.getRuntime().addShutdownHook(new Thread(() -> 
    System.out.println("Hook 2")));
Runtime.getRuntime().addShutdownHook(new Thread(() -> 
    System.out.println("Hook 3")));
```

**输出（无序）：**
```
Hook 2
Hook 1
Hook 3
```

**特点：**
- 多个 Hook **并发执行**，顺序完全不保证
- 一个 Hook 的异常**不影响**其他 Hook 执行
- 不要依赖 Hook 之间的执行顺序

---

## 🔥 扩展追问 2：ShutdownHook 里的注意事项

```java
// ❌ 危险：Hook 里再调用 System.exit()
Runtime.getRuntime().addShutdownHook(new Thread(() -> {
    System.exit(0);  // 死锁！ShutdownHook 里调 exit 会卡住
}));

// ❌ 危险：长时间阻塞
Runtime.getRuntime().addShutdownHook(new Thread(() -> {
    Thread.sleep(60000);  // 等 60 秒？容器可能只给 10 秒就 kill -9 了
}));

// ❌ 危险：访问已销毁的对象
Runtime.getRuntime().addShutdownHook(new Thread(() -> {
    // 其他 Hook 可能已经关了连接池，这里再用会报错
    jdbcTemplate.query("SELECT ...");  
}));
```

**最佳实践：**
- Hook 执行要快（建议 < 10 秒）
- 每个 Hook 只处理自己的资源
- 做好超时和异常处理

## 🔥 扩展追问 3：Docker/K8s 的优雅关闭

```java
// Dockerfile 要用 exec 形式，确保信号传递给 JVM
ENTRYPOINT ["java", "-jar", "app.jar"]  // ✅ 正确
ENTRYPOINT java -jar app.jar             // ❌ 错误，shell 会拦截信号

// 代码里处理 SIGTERM
Runtime.getRuntime().addShutdownHook(new Thread(() -> {
    System.out.println("收到 SIGTERM，开始优雅关闭...");
    // K8s 默认给 30 秒宽限期（terminationGracePeriodSeconds）
}));
```

**K8s 配置：**
```yaml
spec:
  terminationGracePeriodSeconds: 60  # 给 60 秒关闭时间
  containers:
    - name: app
      lifecycle:
        preStop:
          exec:
            command: ["/bin/sh", "-c", "sleep 10"]  # 先等 10 秒，让流量切走
```

## 🔥 扩展追问 4：Spring Boot 的优雅关闭

```java
// Spring Boot 2.3+ 内置优雅关闭
server:
  shutdown: graceful  # 开启优雅关闭

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s  # 每个阶段最多等 30 秒
```

**Spring 自动处理：**
- 停止接收新请求
- 等待活跃请求完成
- 执行 `@PreDestroy` 方法
- 发布 `ContextClosedEvent`

```java
@Component
public class GracefulShutdownHandler {
    
    @PreDestroy
    public void onDestroy() {
        // Spring 会自动调用，比 ShutdownHook 更可控
        System.out.println("Spring 容器关闭，清理资源...");
    }
    
    @EventListener(ContextClosedEvent.class)
    public void onContextClosed() {
        // 也可以监听事件
    }
}
```

## 🔥 扩展追问 5：Hook 执行超时怎么办？

```java
// 容器只给 10 秒，但关闭逻辑需要 30 秒
// 方案：异步执行 + 超时控制
Runtime.getRuntime().addShutdownHook(new Thread(() -> {
    ExecutorService shutdownExecutor = Executors.newSingleThreadExecutor();
    
    Future<?> future = shutdownExecutor.submit(() -> {
        // 实际的关闭逻辑
        doShutdown();
    });
    
    try {
        // 只等 8 秒（留 2 秒缓冲）
        future.get(8, TimeUnit.SECONDS);
    } catch (TimeoutException e) {
        System.err.println("关闭超时，强制退出");
        future.cancel(true);
    } catch (Exception e) {
        e.printStackTrace();
    } finally {
        shutdownExecutor.shutdownNow();
    }
}));
```

## 一句话收尾

> ShutdownHook 在 **JVM 正常关闭** 时执行（`exit()`、SIGTERM、程序结束），**`kill -9`、JVM 崩溃、`halt()` 时不执行**；多个 Hook 并发无序，执行要快，配合 Docker/K8s 的宽限期设置，Spring Boot 项目优先用内置的 `server.shutdown=graceful`。
