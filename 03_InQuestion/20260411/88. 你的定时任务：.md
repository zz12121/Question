88. 你的定时任务：
```java
@Scheduled(fixedRate = 60000)
public void task() {
    // 执行任务
}
```
**问题：** 如果任务执行时间超过 60 秒，会发生什么？如何避免任务重叠执行？

---
## 先说结论

`fixedRate = 60000` 模式下，**任务执行时间超过间隔时间会导致任务重叠执行**（多个任务线程并发跑）。避免方案：改用 `fixedDelay`（等上次完成）、加同步锁、或配置线程池为单线程。

---

## 🔥 fixedRate 的问题演示

### 默认行为：任务重叠

```java
@Scheduled(fixedRate = 60000)  // 每 60 秒触发一次
public void task() {
    log.info("开始执行: {}", System.currentTimeMillis());
    Thread.sleep(80000);  // 模拟执行 80 秒
    log.info("执行结束");
}
```

**执行时间线：**
```
T+0s:   任务1开始
T+60s:  任务2开始（任务1还没完！）→ 重叠执行
T+80s:  任务1结束
T+120s: 任务2结束，任务3开始
```

**问题：**
- 线程池默认 1 个线程时，任务 2 会**排队等待**（不会重叠，但堆积）
- 自定义线程池后，多个任务**并发执行** → 数据竞争、资源冲突

---

## 🔥 四种解决方案

### 方案一：fixedDelay（推荐简单场景）

```java
@Scheduled(fixedDelay = 60000)  // 上次执行完毕后，间隔 60 秒再执行
public void task() {
    // 执行 80 秒 → 等 60 秒 → 下次开始
}
```

**时间线：**
```
T+0s:   开始 → T+80s 结束 → 等60s → T+140s 下次开始
```

| 特性 | fixedRate | fixedDelay |
|------|-----------|------------|
| 触发时机 | 固定频率，不管上次 | 上次完成后延迟 |
| 重叠风险 | ⚠️ 有 | ✅ 无 |
| 适用场景 | 独立无状态任务 | 有状态/资源竞争任务 |

---

### 方案二：同步锁（防重叠执行）

```java
@Scheduled(fixedRate = 60000)
public void task() {
    if (!lock.tryLock()) {
        log.warn("上次任务未结束，跳过本次");
        return;
    }
    try {
        // 执行任务
    } finally {
        lock.unlock();
    }
}

private final ReentrantLock lock = new ReentrantLock();
```

**或分布式场景用 Redis 锁：**
```java
@Scheduled(fixedRate = 60000)
public void task() {
    String lockKey = "task:lock";
    Boolean locked = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, "1", 5, TimeUnit.MINUTES);
    
    if (!Boolean.TRUE.equals(locked)) {
        return;  // 已有实例在执行
    }
    try {
        // 执行
    } finally {
        redisTemplate.delete(lockKey);
    }
}
```

---

### 方案三：单线程线程池（强制串行）

```java
@Configuration
public class ScheduleConfig implements SchedulingConfigurer {
    
    @Override
    public void configureTasks(ScheduledTaskRegistrar taskRegistrar) {
        // 单线程：任务排队执行，不会重叠
        taskRegistrar.setScheduler(
            Executors.newSingleThreadScheduledExecutor(r -> {
                Thread t = new Thread(r, "single-scheduler");
                t.setDaemon(true);
                return t;
            })
        );
    }
}
```

**注意：** 单线程下 `fixedRate` 也会变成排队模式，失去固定频率意义。

---

### 方案四：@SchedulerLock（专业方案）

```java
@Scheduled(fixedRate = 60000)
@SchedulerLock(name = "orderSyncTask", lockAtMostFor = "5m", lockAtLeastFor = "30s")
public void task() {
    // 分布式锁保证集群只有一个实例执行
    // lockAtMostFor: 锁最长持有时间（防止崩溃死锁）
    // lockAtLeastFor: 锁最短持有时间（防止任务太短频繁切换）
}
```

**依赖：**
```xml
<dependency>
    <groupId>net.javacrumbs.shedlock</groupId>
    <artifactId>shedlock-spring</artifactId>
    <version>5.10.0</version>
</dependency>
```

## 🔥 扩展追问

### 追问 1：cron 表达式会重叠吗？

```java
@Scheduled(cron = "0 * * * * ?")  // 每分钟执行
public void task() {
    Thread.sleep(80000);  // 执行 80 秒
}
```

**答案：会！** cron 也是按触发时间调度，与 fixedRate 类似。解决方案同上。

---

### 追问 2：多实例部署怎么防重复执行？

```java
// 方案：分布式锁 + 任务分片
@Scheduled(fixedRate = 60000)
@SchedulerLock(name = "dataSync", lockAtMostFor = "5m")
public void task() {
    // 只有拿到锁的实例执行
    List<Data> list = fetchData();
    
    // 按实例 ID 取模分片
    int shard = instanceId % totalInstances;
    List<Data> myShard = list.stream()
        .filter(d -> d.getId() % totalInstances == shard)
        .collect(Collectors.toList());
    
    process(myShard);
}
```

### 追问 3：任务执行超时怎么监控？

```java
@Scheduled(fixedDelay = 60000)
public void task() {
    StopWatch watch = new StopWatch();
    watch.start();
    
    try {
        // 执行
    } finally {
        watch.stop();
        long seconds = watch.getTotalTimeSeconds();
        
        if (seconds > 55) {  // 接近间隔，告警
            alertService.warn("任务执行时间过长: {}s", seconds);
        }
        
        metrics.record("task.duration", seconds);
    }
}
```

## 一句话收尾

> **`fixedRate` 任务超时会导致重叠执行**，简单场景改 `fixedDelay`，分布式环境用 `@SchedulerLock`，关键任务加监控告警。集群部署必须考虑分布式锁防重复。
