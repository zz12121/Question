82. 你的缓存删除代码：
```java
public void deleteUser(Long id) {
    userDao.delete(id);
    redis.del("user:" + id);
}
```
**问题：** 如果删除缓存失败（Redis 挂了），会出现什么问题？如何保证最终一致？

---
## 先说结论

`先删 DB 再删缓存` 的代码存在**数据不一致风险**：Redis 删除失败时，缓存中残留旧数据，后续读取会返回已删除的用户。更隐蔽的是**并发竞态**——删缓存和读缓存的时序导致脏数据。保证最终一致的核心方案：**延迟双删**、**消息队列异步删**、或**Canal 订阅 Binlog** 删。

---

## 🔥 问题分析：删除缓存失败的后果

```java
public void deleteUser(Long id) {
    userDao.delete(id);           // 1. DB 删除成功
    redis.del("user:" + id);      // 2. Redis 删除失败（网络超时/Redis挂了）
}
```

**执行结果：**
```
DB：用户已删除
Redis：缓存还在，key="user:100", value=User@100

后续查询：
  get("user:100") → 返回已删除的用户（脏数据！）
  业务逻辑继续处理，可能报错或产生错误数据
```

---

## 🔥 更隐蔽的问题：并发竞态

即使 Redis 没挂，时序问题也会导致不一致：

```
时间线：
  T1: 线程 A 删除 DB
  T2: 线程 B 查询缓存（未命中）
  T3: 线程 B 查 DB（读到旧数据，因为 A 的事务未提交）
  T4: 线程 A 删除缓存
  T5: 线程 B 写入缓存（把旧数据又写回去了！）

结果：缓存中是旧数据，DB 已删除，长期不一致
```

---

## 🔥 优化方案 1：延迟双删（简单有效）

```java
public void deleteUser(Long id) {
    String key = "user:" + id;
    
    // 1. 先删缓存（防旧数据）
    redis.del(key);
    
    // 2. 删 DB
    userDao.delete(id);
    
    // 3. 延迟再次删缓存（异步）
    CompletableFuture.delayedExecutor(500, TimeUnit.MILLISECONDS)
        .execute(() -> redis.del(key));
}
```

**原理：**
- 第一次删：清理可能的旧缓存
- 删 DB：主操作
- 延迟删：覆盖 T1~T5 时间窗口内的并发写脏

**缺点：** 延迟时间难确定（500ms 是经验值），极端并发仍可能不一致。

## 🔥 优化方案 2：消息队列异步删（推荐）

```java
public void deleteUser(Long id) {
    // 1. 删 DB（本地事务）
    userDao.delete(id);
    
    // 2. 发送删缓存消息（事务消息，确保 DB 提交后才发送）
    transactionMQProducer.sendMessage(
        new DeleteCacheMessage("user", id)
    );
}

// MQ 消费者
@Component
public class CacheDeleteConsumer {
    
    @KafkaListener(topics = "cache-delete")
    public void onMessage(DeleteCacheMessage msg) {
        String key = msg.getCacheType() + ":" + msg.getId();
        
        // 重试 3 次
        for (int i = 0; i < 3; i++) {
            try {
                redis.del(key);
                return;
            } catch (Exception e) {
                if (i == 2) {
                    // 入死信队列，人工处理
                    deadLetterQueue.send(msg);
                }
                Thread.sleep(100 * (i + 1));
            }
        }
    }
}
```

**优势：**
- 解耦缓存删除和主流程
- MQ 保证消息必达
- 失败可重试，最终一致

## 🔥 优化方案 3：Canal 订阅 Binlog（最可靠）

```
架构：
  MySQL → Binlog → Canal Server → MQ → 缓存删除服务 → Redis

流程：
  1. 业务代码只删 DB
  2. Canal 监听 Binlog，捕获 DELETE 操作
  3. 解析出被删的 id，发送 MQ
  4. 消费者删 Redis
```

```java
// 业务代码极简，只关心 DB
@Transactional
public void deleteUser(Long id) {
    userDao.delete(id);  // 就这一行，缓存由 Canal 处理
}

// Canal 消费者
@Component
public class BinlogCacheDeleteListener {
    
    @KafkaListener(topics = "canal.user")
    public void onBinlog(CanalMessage msg) {
        if ("DELETE".equals(msg.getEventType())) {
            List<Long> ids = msg.getPrimaryKeys();  // 被删的主键
            for (Long id : ids) {
                redis.del("user:" + id);
            }
        }
    }
}
```

**优势：**
- 业务代码无侵入
- Binlog 是最终事实，不会漏
- 天然支持批量、异步

// 无论怎么删，都可能不一致，设置短过期时间兜底
redis.opsForValue().set(key, user,

## 🔥 扩展追问 1：先删缓存还是先删 DB？

| 顺序 | 问题 | 结论 |
|---|---|---|
| **先删缓存，再删 DB** | 删缓存后、删 DB 前，其他线程读旧数据写入缓存 | 短暂不一致，但可接受 |
| **先删 DB，再删缓存** | 删 DB 后、删缓存前，其他线程读缓存（旧数据） | 同样短暂不一致 |

**推荐：先删缓存**，因为：
- DB 删除是主操作，失败可回滚
- 缓存删除失败影响小（可重建）
- 配合延迟双删，覆盖时间窗口

## 🔥 扩展追问 2：更新操作的一致性

```java
// 更新比删除更复杂，因为缓存需要最新值

// ❌ 错误：先更新 DB，再更新缓存
public void updateUser(User user) {
    userDao.update(user);           // DB 更新
    redis.set("user:" + user.getId(), user);  // 缓存更新
    // 如果缓存更新失败，不一致
}

// ✅ 正确：先删缓存，再更新 DB，再延迟删缓存
public void updateUser(User user) {
    String key = "user:" + user.getId();
    
    redis.del(key);           // 1. 删缓存
    userDao.update(user);     // 2. 更新 DB
    asyncDel(key, 500);       // 3. 延迟删（防并发写脏）
}

// 或：更新 DB 后，不主动写缓存，让下次查询懒加载
```

## 🔥 扩展追问 3：分布式锁保证强一致

```java
// 极端场景：要求强一致，用分布式锁串行化
public void deleteUser(Long id) {
    String key = "user:" + id;
    String lockKey = "lock:user:" + id;
    
    RLock lock = redisson.getLock(lockKey);
    try {
        lock.lock();
        
        userDao.delete(id);
        redis.del(key);
        
    } finally {
        lock.unlock();
    }
}

// 读操作也加锁（性能差，慎用）
public User getUser(Long id) {
    String lockKey = "lock:user:" + id;
    RLock lock = redisson.getLock(lockKey);
    try {
        lock.lock();
        // 查缓存 → 查 DB → 写缓存
    } finally {
        lock.unlock();
    }
}
```

**缺点：** 性能极差，只适用于极少数强一致场景。

## 🔥 扩展追问 4：多级缓存的一致性

```java
@Component
public class UserCache {
    
    private final LoadingCache<Long, User> caffeine = Caffeine.newBuilder()...;
    
    @Autowired
    private RedisTemplate<String, User> redis;
    
    public void delete(Long id) {
        // 1. 删本地缓存
        caffeine.invalidate(id);
        
        // 2. 删 Redis
        redis.del("user:" + id);
        
        // 3. 广播本地缓存失效（其他 JVM 实例）
        redis.convertAndSend("cache:invalidate", "user:" + id);
    }
    
    @EventListener
    public void onInvalidate(CacheInvalidateEvent event) {
        caffeine.invalidate(event.getKey());
    }
}
```

## 一句话收尾

> 缓存删除失败会导致**长期脏数据**，`先删缓存再删 DB + 延迟双删` 是简单方案，生产环境推荐 **MQ 异步删** 或 **Canal 订阅 Binlog** 实现最终一致，配合**短过期时间**兜底；极端强一致场景用**分布式锁**，但会牺牲性能。
