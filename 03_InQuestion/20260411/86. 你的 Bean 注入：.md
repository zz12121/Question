86. 你的 Bean 注入：
```java
@Service
public class OrderService {
    @Autowired
    private UserService userService;
}
```
**问题：** 字段注入有什么问题？推荐用什么注入方式？

---
## 先说结论

**字段注入（Field Injection）** 是 Spring 的"懒人写法"，生产环境推荐 **构造器注入（Constructor Injection）**。核心问题在于：字段注入隐藏了类的依赖、无法保证不可变性、测试困难，且对循环依赖的检测滞后。

---

## 🔥 字段注入的问题

### 1. 隐藏依赖，违反单一职责原则

```java
@Service
public class OrderService {
    @Autowired
    private UserService userService;  // 依赖藏在字段里，不看源码不知道
    @Autowired
    private PayService payService;
    @Autowired
    private InventoryService inventoryService;
    // ... 还能继续注入十几个
}
```

**问题：** 类的依赖数量没有约束，容易膨胀成"上帝类"。构造器注入强制你面对依赖过多的问题。

---

### 2. 无法声明为 final，失去不可变性保障

```java
@Service
public class OrderService {
    @Autowired
    private UserService userService;  // 不能加 final！
}
```

- 字段注入发生在对象创建**之后**（通过反射设置）
- 无法配合 `final` 使用，对象状态可能被意外修改

---

### 3. 单元测试困难

```java
// 字段注入的类，测试时需要反射或 Spring 容器
public class OrderServiceTest {
    @Test
    public void test() {
        OrderService service = new OrderService();  // userService = null！
        // 必须用反射注入，或启动 Spring 上下文
    }
}
```

---

### 4. 循环依赖检测滞后

```java
@Service
public class AService {
    @Autowired
    private BService b;  // 启动时不报错，运行时才可能 NPE
}

@Service
public class BService {
    @Autowired
    private AService a;
}
```

字段注入的循环依赖在启动时**不报错**，可能潜伏到运行时爆发。

---

## 🔥 推荐：构造器注入

```java
@Service
public class OrderService {
    
    private final UserService userService;
    private final PayService payService;
    
    // Spring 4.3+ 单构造器可省略 @Autowired
    public OrderService(UserService userService, PayService payService) {
        this.userService = userService;
        this.payService = payService;
    }
}
```

### 或者用 Lombok 简化：

```java
@Service
@RequiredArgsConstructor  // 生成包含 final 字段的构造器
public class OrderService {
    
    private final UserService userService;
    private final PayService payService;
}
```

---

## 🔥 三种注入方式对比

| 特性 | 字段注入 | Setter 注入 | 构造器注入 |
|------|---------|-------------|-----------|
| **final 支持** | ❌ 不支持 | ❌ 不支持 | ✅ 支持 |
| **依赖显式性** | ❌ 隐藏 | ⚠️ 可选 | ✅ 强制声明 |
| **不可变性** | ❌ 可变 | ❌ 可变 | ✅ 不可变 |
| **测试友好** | ❌ 需反射/Spring | ⚠️ 需 Setter | ✅ 直接 new |
| **循环依赖检测** | ❌ 启动不报错 | ⚠️ 可能滞后 | ✅ 启动即报错 |
| **NPE 风险** | ⚠️ 运行时 | ⚠️ 运行时 | ✅ 编译期保证 |
| **代码简洁度** | ✅ 最简洁 | ⚠️ 中等 | ⚠️ 稍繁琐（Lombok 解决）|

## 🔥 扩展追问

### 追问 1：构造器注入怎么检测循环依赖？

```java
@Service
public class AService {
    private final BService bService;
    public AService(BService bService) { this.bService = bService; }
}

@Service
public class BService {
    private final AService aService;
    public BService(AService aService) { this.aService = aService; }
}
```

**启动直接报错：**
```
BeanCurrentlyInCreationException: Error creating bean with name 'AService': 
Requested bean is currently in creation: Is there an unresolvable circular reference?
```

**原理：** 构造器注入在实例化阶段就需要依赖，Spring 无法完成代理，只能抛异常。

---

### 追问 2：循环依赖一定要解决吗？怎么解决？

**方案一：重构，消除循环依赖（推荐）**

```java
// 提取公共逻辑到 CService，A、B 都依赖 C
@Service
public class CService { /* 公共逻辑 */ }
```

**方案二：Setter/字段注入 + @Lazy**

```java
@Service
public class AService {
    private final BService bService;
    
    public AService(@Lazy BService bService) {  // 延迟注入代理对象
        this.bService = bService;
    }
}
```

**方案三：Spring Boot 2.6+ 默认禁止循环依赖**

```yaml
spring:
  main:
    allow-circular-references: false  # 强制良好设计
```

---

### 追问 3：Setter 注入什么时候用？

**场景：可选依赖**

```java
@Service
public class OrderService {
    private Optional<AuditService> auditService = Optional.empty();
    
    @Autowired(required = false)  // 可选，没有也能跑
    public void setAuditService(AuditService auditService) {
        this.auditService = Optional.ofNullable(auditService);
    }
}
```

---

## 一句话收尾

> **构造器注入是 Spring 官方推荐的方式**：依赖显式、不可变、测试友好、循环依赖早发现。配合 Lombok 的 `@RequiredArgsConstructor`，代码简洁又健壮。
