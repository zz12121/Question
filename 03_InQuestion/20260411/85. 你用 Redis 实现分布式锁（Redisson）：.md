85. 你用 Redis 实现分布式锁（Redisson）：
```java
RLock lock = redissonClient.getLock("myLock");
lock.lock();
try {
    doBusiness();
} finally {
    lock.unlock();
}
```
**问题：** Redisson 的看门狗机制是什么？锁的续期是如何实现的？

---
## 先说结论

Redisson 的**看门狗（Watchdog）机制**是自动续期的核心：客户端获取锁后，启动一个**后台定时任务**（默认每 10 秒检查），如果锁还在持有状态且业务未执行完，就自动将锁的过期时间**重置为 30 秒**，防止业务执行超时导致锁提前释放。业务执行完毕或崩溃时，看门狗停止，锁自然过期释放。

---

## 🔥 为什么需要看门狗

```java
// 没有看门狗的简单实现
SET myLock randomValue NX EX 10  // 10 秒后自动过期

// 问题场景：
T0: 获取锁，设置 10 秒过期
T1: 开始执行业务逻辑
T5: 业务还没执行完，锁已经过期被其他线程获取！
T6: 两个线程同时执行业务，分布式锁失效
```

**核心矛盾：** 锁的过期时间很难预估，业务执行时间不确定。

---

## 🔥 Redisson 看门狗机制详解

### 获取锁时的逻辑

```java
// RedissonLock.lock() 简化逻辑
public void lock() {
    // 1. 尝试获取锁
    Long ttl = tryAcquire(-1, null);
    
    if (ttl == null) {
        // 获取成功，启动看门狗
        scheduleExpirationRenewal(threadId);
    }
}
```

### 看门狗续期实现

```java
// Redisson 内部实现（简化）
private void scheduleExpirationRenewal(long threadId) {
    // 定时任务：每 10 秒执行一次（看门狗间隔 = 过期时间 / 3）
    Timeout task = commandExecutor.getConnectionManager()
        .newTimeout(timeout -> {
            // 检查锁是否仍被当前线程持有
            RFuture<Boolean> future = renewExpirationAsync(threadId);
            
            future.thenAccept(res -> {
                if (res) {
                    // 续期成功，递归调度下一次
                    scheduleExpirationRenewal(threadId);
                }
                // 续期失败（锁已被释放或易主），停止看门狗
            });
        }, 
        internalLockLeaseTime / 3,  // 默认 30/3 = 10 秒
        TimeUnit.MILLISECONDS);
}
```

### Lua 脚本续期

```lua
-- renewExpiration.lua
if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then
    -- 锁存在且是当前线程持有，续期到 30 秒
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return 1;
end;
return 0;
```

## 🔥 关键参数

| 参数 | 默认值 | 说明 |
|---|---|---|
| `lockWatchdogTimeout` | 30000 ms (30 秒) | 锁的过期时间，也是看门狗续期目标 |
| 看门狗检查间隔 | 10000 ms (10 秒) | `lockWatchdogTimeout / 3` |
| 续期时机 | 剩余时间 < 10 秒 | 确保锁不会过期 |

## 🔥 执行流程图解

```
线程 A 获取锁成功
    ↓
启动看门狗（后台线程）
    ↓
每 10 秒检查：
    锁还在持有？ → 是 → 续期到 30 秒 → 继续检查
                  ↓ 否（已 unlock 或崩溃）
                停止看门狗，锁自然过期
    ↓
业务执行完毕，调用 unlock()
    ↓
停止看门狗 + 删除锁
```

## 🔥 手动指定过期时间 vs 看门狗

```java
// 方式 1：不指定时间，启用看门狗（推荐）
RLock lock = redisson.getLock("myLock");
lock.lock();  // 默认 30 秒，看门狗自动续期
try {
    doBusiness();  // 执行 1 小时也没事，看门狗一直续期
} finally {
    lock.unlock();
}

// 方式 2：手动指定时间，禁用看门狗
lock.lock(10, TimeUnit.SECONDS);  // 10 秒后强制过期，无看门狗
// 业务超过 10 秒，锁自动释放，可能不安全
```

**建议：** 除非明确知道业务最大执行时间，否则用默认看门狗模式。

## 🔥 扩展追问 1：看门狗续期失败怎么办？

```java
// 场景：Redis 主从切换、网络抖动，续期失败
// Redisson 处理：
// 1. 续期失败，看门狗停止
// 2. 锁在 30 秒后自动过期（兜底）
// 3. 业务线程还在执行，但锁已释放，其他线程可能获取锁

// 防御：业务幂等性
if (!lock.isHeldByCurrentThread()) {
    // 锁已丢失，检查业务状态，避免重复处理
    return;
}
```

## 🔥 扩展追问 2：RedLock 红锁（多节点）

```java
// 单 Redis 实例故障时，锁可能丢失
// RedLock：在 N 个独立 Redis 节点上加锁

RLock lock1 = redisson1.getLock("myLock");
RLock lock2 = redisson2.getLock("myLock");
RLock lock3 = redisson3.getLock("myLock");

RedissonRedLock redLock = new RedissonRedLock(lock1, lock2, lock3);
redLock.lock();
try {
    doBusiness();
} finally {
    redLock.unlock();
}
```

**原理：** 在多数节点（N/2+1）上成功获取锁，才算加锁成功。

**争议：** RedLock 被 Redis 作者和分布式系统专家质疑，生产环境建议用 **ZooKeeper/Etcd** 或 **数据库唯一索引** 替代。

## 🔥 扩展追问 3：锁的可重入性

```java
// Redisson 锁是可重入的（同一线程多次获取）
RLock lock = redisson.getLock("myLock");

lock.lock();    // 第一次获取，holdCount = 1
try {
    lock.lock();  // 第二次获取，holdCount = 2
    try {
        doBusiness();
    } finally {
        lock.unlock();  // holdCount = 1，锁不释放
    }
} finally {
    lock.unlock();  // holdCount = 0，锁真正释放
}
```

**实现：** Hash 结构存储，`key = 锁名，field = 线程标识，value = 重入次数`。

## 🔥 扩展追问 4：锁的公平性

```java
// 非公平锁（默认）：抢锁，可能饥饿
RLock lock = redisson.getLock("myLock");

// 公平锁：按请求顺序获取
RLock fairLock = redisson.getFairLock("myLock");
// 内部维护一个等待队列，新请求排在队尾
```

**公平锁代价：** 性能略低，需要维护队列。

## 🔥 扩展追问 5：读写锁

RReadWriteLock rwLock = redisson.getReadWriteLock("myLock");
RLock readLock = rwLock.readLock();
RLock writeLock = rwLock.writeLock();

// 读锁：共享，多个线程可同时获取

    writeData();

    writeLock.unlock();
}

## 一句话收尾

> Redisson 看门狗是**自动续期机制**，后台每 10 秒检查，将锁重置为 30 秒过期，确保业务执行完才释放；**默认启用**（不指定过期时间），业务执行时间不确定时比手动设过期更安全；极端场景配合 **RedLock 多节点** 或 **数据库唯一索引** 兜底，但 RedLock 有争议，强一致场景建议用 **ZooKeeper/Etcd**。
