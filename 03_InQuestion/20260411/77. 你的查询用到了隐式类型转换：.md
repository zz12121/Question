77. 你的查询用到了隐式类型转换：
```sql
-- phone 是 varchar 类型
SELECT * FROM user WHERE phone = 13800138000;
```
**问题：** 这条 SQL 有什么问题？索引会失效吗？

---
## 先说结论

`phone` 是 `varchar` 类型，但查询条件传了数字 `13800138000`，MySQL 会**隐式把字符串列转为数字**再比较，导致**索引失效、全表扫描**。这是"函数作用于列"的变种，优化方案：**传入字符串类型参数**，或**修改表结构统一类型**。

---

## 🔥 隐式转换的执行过程

```sql
-- 原始 SQL
SELECT * FROM user WHERE phone = 13800138000;

-- MySQL 实际执行的转换
SELECT * FROM user WHERE CAST(phone AS SIGNED) = 13800138000;
-- 对每一行的 phone 调用 CAST() 转为数字，再比较
```

**为什么索引失效：**

```
索引存储的是字符串 '13800138000'
查询需要比较数字 13800138000

CAST(phone AS SIGNED) 的结果无法直接用字符串索引匹配
→ 只能全表扫描，逐行转换后比较
```

**执行计划对比：**

```sql
-- 隐式转换：索引失效
EXPLAIN SELECT * FROM user WHERE phone = 13800138000;
-- type: ALL（全表扫描）
-- Extra: Using where

-- 正确写法：走索引
EXPLAIN SELECT * FROM user WHERE phone = '13800138000';
-- type: ref（索引引用）
-- key: idx_phone
```

## 🔥 隐式转换的更多陷阱

### 1. 精度丢失

```sql
-- phone = '13800138001' 和 phone = '13800138002' 转换后都是 13800138000
SELECT * FROM user WHERE phone = 13800138000;
-- 可能匹配到多条记录（如果 BIGINT 溢出或精度丢失）
```

### 2. 前导零问题

```sql
-- 手机号 01234567890 存为字符串 '01234567890'
SELECT * FROM user WHERE phone = 1234567890;  -- 数字，前导零丢失
-- 实际匹配 '1234567890'，'01234567890' 匹配不到
```

### 3. 非数字字符串转 0

```sql
-- 某条记录 phone = 'abc'
SELECT * FROM user WHERE phone = 0;
-- 'abc' 转为 0，这条记录会被匹配到！
```

## 🔥 优化方案

### 方案 1：传入正确类型（推荐）

```java
// ✅ MyBatis：参数加引号，或明确用 String
@Select("SELECT * FROM user WHERE phone = #{phone}")
User findByPhone(@Param("phone") String phone);  // 传入 String

// 调用
userMapper.findByPhone("13800138000");  // ✅ 正确
userMapper.findByPhone(13800138000L);   // ❌ 错误，Long 类型
```

### 方案 2：MyBatis 类型处理器

```xml
<!-- mybatis-config.xml -->
<typeHandlers>
    <typeHandler javaType="java.lang.Long" 
                 jdbcType="VARCHAR"
                 handler="com.example.LongToVarcharHandler"/>
</typeHandlers>
```

### 方案 3：统一表结构（根治）

```sql
-- 如果业务上 phone 就是数字，改成 BIGINT
ALTER TABLE user MODIFY phone BIGINT;

-- 但注意：手机号前导零、国际区号 +86 等问题
-- 通常建议保持 VARCHAR，代码里传 String
```

## 🔥 扩展追问 1：哪些场景会触发隐式转换？

| 场景 | 示例 | 结果 |
|---|---|---|
| 字符串列 = 数字 | `varchar_col = 123` | 列转数字，索引失效 |
| 数字列 = 字符串数字 | `int_col = '123'` | 字符串转数字，**可以走索引** |
| 不同字符集比较 | `utf8mb4_col = utf8_col` | 列转字符集，索引失效 |
| 时间列 = 字符串 | `datetime_col = '2024-01-01'` | 字符串转时间，**可以走索引** |

**规律：MySQL 倾向于把"简单类型"转为"复杂类型"，字符串比数字复杂，所以字符串列会被转。**

## 🔥 扩展追问 3：其他数据库的行为

| 数据库 | 隐式转换行为 | 索引影响 |
|---|---|---|
| **MySQL** | 灵活，自动转换 | 字符串列转数字会失效 |
| **PostgreSQL** | 严格，报错 | `operator does not exist: character varying = bigint` |
| **Oracle** | 灵活，自动转换 | 类似 MySQL，索引可能失效 |
| **SQL Server** | 严格，报错 | 需显式转换 `CAST` 或 `CONVERT` |

**PostgreSQL 的严格是优势：**
```sql
-- PostgreSQL 直接报错，不会静默失效
SELECT * FROM user WHERE phone = 13800138000;
-- ERROR: operator does not exist: character varying = bigint
```

## 🔥 扩展追问 4：连接查询中的隐式转换

```sql
-- 更隐蔽的问题：JOIN 条件类型不匹配
SELECT u.*, o.*
FROM user u
JOIN orders o ON u.phone = o.user_phone;
-- u.phone 是 varchar，o.user_phone 是 bigint
-- 两边都转换，两个索引都失效！
```

**修复：**
```sql
-- 方案 1：统一表结构
ALTER TABLE orders MODIFY user_phone VARCHAR(20);

-- 方案 2：显式转换（但索引还是失效）
SELECT u.*, o.*
FROM user u
JOIN orders o ON u.phone = CAST(o.user_phone AS CHAR);

-- 方案 3：应用层保证类型一致
```

---

## 一句话收尾

> **类型不匹配是索引杀手！** `varchar` 列传数字会导致隐式 `CAST`，索引完全失效；**代码里严格保持类型一致**，数据库用 `VARCHAR` 存手机号，Java 用 `String` 传参；PostgreSQL 的严格类型检查能提前暴露问题，MySQL 需靠代码规范和 SQL 审查预防。
