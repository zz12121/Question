70. 你用 `Thread.sleep(0)`：
```java
while (running) {
    doWork();
    Thread.sleep(0);
}
```
**问题：** `sleep(0)` 有什么作用？和 `yield()` 有什么区别？

---
## 先说结论

`Thread.sleep(0)` 在**大多数 JVM 实现上会触发线程切换**，让出 CPU 时间片，但语义上只是"睡眠 0 毫秒"，不保证切换。`Thread.yield()` 是**明确提示调度器让出 CPU**，但同样不保证一定切换。两者都不能替代正确的并发控制，现代代码中**基本不用**，直接用 `LockSupport.parkNanos()` 或线程池更可控。

---

## 🔥 `sleep(0)` 的实际行为

```java
while (running) {
    doWork();
    Thread.sleep(0);  // 暗示：我忙完了，可以切换了
}
```

**底层发生了什么：**

```
Linux/Windows 调度器：
  sleep(0) → 系统调用 nanosleep(0) 或 Sleep(0)
           → 内核检查：有没有同优先级或更高优先级的线程就绪？
           → 有 → 当前线程让出 CPU，进入就绪队列尾部
           → 没有 → 继续执行（没切换）
```

**关键点：**
- 不是"空操作"，会进入内核态
- 不释放锁（如果持有锁，不会释放）
- 不保证一定切换，取决于调度器
- 比 `sleep(1)` 快，但仍有系统调用开销

---

## 🔥 `sleep(0)` vs `yield()` 对比

| 特性 | `Thread.sleep(0)` | `Thread.yield()` |
|---|---|---|
| **语义** | 睡眠 0 毫秒 | 提示调度器：我愿意让出 CPU |
| **JVM 实现** | 调用 OS 的 sleep 系统调用 | 调用 OS 的 sched_yield() |
| **保证切换？** | ❌ 不保证 | ❌ 不保证 |
| **进入内核？** | ✅ 是 | ✅ 是 |
| **精度** | 毫秒级（实际可能更高） | 无时间参数 |
| **可移植性** | Windows 效果明显，Linux 不一定 | 各平台行为差异大 |
| **现代建议** | 避免使用 | 避免使用 |

## 🔥 为什么代码里会出现 `sleep(0)`？

### 历史原因：自旋锁的"退避"

```java
// 古老的自旋锁实现
while (!tryLock()) {
    Thread.sleep(0);  // 忙等太久，让一下 CPU
}
```

**现代替代：**

```java
// Java 5+：用 LockSupport，更轻量
while (!tryLock()) {
    LockSupport.parkNanos(1);  // 纳秒级，不进入内核（短时间）
}

// Java 5+：直接用 j.u.c 锁
Lock lock = new ReentrantLock();
lock.lock();  // 内部用 AQS，park/unpark 实现，比 sleep/yield 高效
```

### 另一个场景：防止线程饿死

```java
// 低优先级线程长时间占用 CPU，高优先级线程饿死
while (hasWork()) {
    process();
    Thread.sleep(0);  // 给同优先级线程机会
}

// 现代替代：调整线程优先级，或用线程池
executor.submit(this::process);  // 让调度器管理
```

## 🔥 为什么不推荐用这两个方法？

### 1. 行为不可预测

```java
// 这段代码在不同 JVM、不同 OS 表现完全不同
Thread.yield();
// Windows：可能立即切换
// Linux：可能被忽略
// macOS：行为又不同
```

### 2. 无法解决并发问题

```java
// ❌ 错误：用 sleep/yield 解决可见性问题
while (!flag) {
    Thread.yield();  // 不保证能看到其他线程的修改！
}

// ✅ 正确：用 volatile 或锁
while (!volatileFlag) {
    LockSupport.parkNanos(1000);  // 有条件的短暂等待
}
```

### 3. 性能不如专业工具

```java
// 低效：sleep(0) 每次都要系统调用
while (!queue.isEmpty()) {
    process(queue.poll());
    Thread.sleep(0);
}

// 高效：用 BlockingQueue，没数据时自动阻塞
while (running) {
    Task task = queue.take();  // 空时自动 park，有数据时 unpark
    process(task);
}
```

## 🔥 扩展追问 1：`sleep(0)` 和 `sleep(1)` 的区别

| 特性 | `sleep(0)` | `sleep(1)` |
|---|---|---|
| 最小时间 | 可能立即返回 | 至少 1ms（实际可能 10ms+） |
| 精度 | 低 | 更低 |
| 系统调用 | 有 | 有 |
| 适用 | 快速让出 | 明确延迟 |

**Windows 上 `sleep(1)` 可能休眠 15ms+**（受时钟粒度影响），高精度定时用 `LockSupport.parkNanos()`。

## 🔥 扩展追问 2：正确的线程等待姿势

```java
// 场景：等待某个条件成立

// ❌ 古老做法：忙等 + yield
while (!condition) {
    Thread.yield();
}

// ✅ Java 5+：用 LockSupport
while (!condition) {
    LockSupport.parkNanos(100_000);  // 100μs，可精确控制
}

// ✅ 更好的做法：用条件变量
Lock lock = new ReentrantLock();
Condition condition = lock.newCondition();

lock.lock();
try {
    while (!ready) {
        condition.await();  // 自动释放锁并阻塞，被唤醒后重新获取锁
    }
} finally {
    lock.unlock();
}
```

## 🔥 扩展追问 3：`yield()` 在 JUC 中的实际应用

```java
// java.util.concurrent 里唯一用 yield() 的地方：ForkJoinPool
// ForkJoinPool 的工作窃取算法
final void runTask(ForkJoinTask<?> task) {
    if (task != null) {
        task.doExec();
        // 任务执行完，提示调度器：我可以让出 CPU 给其他工作线程
        Thread.yield();  // 只在特定条件下调用
    }
}
```

**即使是 Doug Lea，也只在特定场景、经过严格测试后才用 `yield()`。**

## 🔥 扩展追问 4：现代并发编程的最佳实践

```java
// 1. 需要延迟 → ScheduledExecutorService
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);
scheduler.schedule(this::task, 100, TimeUnit.MILLISECONDS);

// 2. 需要定时 → 也用 scheduler
scheduler.scheduleAtFixedRate(this::task, 0, 1, TimeUnit.SECONDS);

// 3. 需要等待条件 → CountDownLatch / CompletableFuture
CountDownLatch latch = new CountDownLatch(1);
// 其他线程 countDown()，当前线程 await()

// 4. 需要限流 → RateLimiter
RateLimiter limiter = RateLimiter.create(100);  // 每秒 100 个
limiter.acquire();  // 阻塞等待令牌

// 5. 需要背压 → BlockingQueue
BlockingQueue<Task> queue = new LinkedBlockingQueue<>(1000);
queue.put(task);  // 满时自动阻塞
```

## 一句话收尾

> `Thread.sleep(0)` 和 `yield()` 都是**弱提示、不保证行为**的过时 API，现代 Java 并发应使用 `j.u.c` 包的工具（`LockSupport`、`BlockingQueue`、`CompletableFuture` 等），它们提供更精确的语义和更好的跨平台一致性。
