68. 你用 `BlockingQueue` 的 `drainTo` 批量消费：
```java
BlockingQueue<Task> queue = new LinkedBlockingQueue<>();
List<Task> batch = new ArrayList<>();
queue.drainTo(batch, 100);
```
**问题：** `drainTo` 有什么优势？如果队列中不足 100 条，会阻塞吗？

---
## 先说结论

`drainTo` 的核心优势是**批量转移零拷贝**——直接把队列内部节点批量移动到目标集合，避免逐条 `poll()` 的锁竞争和 CAS 开销。如果队列不足 100 条，**不会阻塞**，有多少拿多少，立即返回实际转移数量。

---

## 🔥 `drainTo` vs 逐条 `poll()` 的性能对比

```java
// ❌ 低效写法：逐条 poll，每次都要加锁/解锁
List<Task> batch = new ArrayList<>();
for (int i = 0; i < 100; i++) {
    Task task = queue.poll();  // 每次 poll 都竞争锁
    if (task == null) break;
    batch.add(task);
}

// ✅ 高效写法：批量 drain，一次加锁，批量转移
List<Task> batch = new ArrayList<>();
int count = queue.drainTo(batch, 100);  // 单次批量操作
```

**性能差异（以 `LinkedBlockingQueue` 为例）：**

| 操作 | 锁竞争次数 | 时间复杂度 | 适用场景 |
|---|---|---|---|
| `poll()` 100 次 | 100 次加锁/解锁 | O(n) 锁开销 | 单条消费 |
| `drainTo(batch, 100)` | **1 次加锁** | O(n) 无锁竞争 | 批量消费 |

`drainTo` 内部实现（简化）：
```java
// LinkedBlockingQueue.drainTo
public int drainTo(Collection<? super E> c, int maxElements) {
    fullyLock();  // 只加一次锁
    try {
        int n = 0;
        Node<E> p = head.next;
        while (p != null && n < maxElements) {
            c.add(p.item);      // 直接转移，无 CAS
            p.item = null;      // 帮助 GC
            Node<E> next = p.next;
            p.next = p;         // 标记已删除
            p = next;
            n++;
        }
        head.next = p;          // 一次性调整头指针
        return n;
    } finally {
        fullyUnlock();  // 只解锁一次
    }
}
```

## 🔥 不足 100 条会阻塞吗？

**不会！`drainTo` 是非阻塞的。**

```java
// 队列只有 3 条数据
BlockingQueue<Task> queue = new LinkedBlockingQueue<>();
queue.add(task1);
queue.add(task2);
queue.add(task3);

List<Task> batch = new ArrayList<>();
int count = queue.drainTo(batch, 100);  // count = 3，立即返回

// batch 里有 3 条，不会等，不会阻塞
```

**行为总结：**
| 场景 | 结果 |
|---|---|
| 队列 ≥ 100 条 | 转移 100 条，返回 100 |
| 队列 < 100 条 | 转移全部，返回实际数量（0~99） |
| 队列为空 | 返回 0，空列表 |

## 🔥 实战：批量消费 + 超时兜底

```java
public class BatchConsumer {
    private final BlockingQueue<Task> queue = new LinkedBlockingQueue<>(10000);
    
    public void start() {
        Thread consumer = new Thread(() -> {
            while (!Thread.currentThread().isInterrupted()) {
                List<Task> batch = new ArrayList<>(100);
                
                // 先尝试批量拿，不阻塞
                int count = queue.drainTo(batch, 100);
                
                // 如果没拿到，等一会儿再拿（避免忙等待）
                if (count == 0) {
                    try {
                        Task first = queue.poll(100, TimeUnit.MILLISECONDS);
                        if (first != null) {
                            batch.add(first);
                            // 继续拿剩下的
                            queue.drainTo(batch, 99);
                        }
                    } catch (InterruptedException e) {
                        Thread.currentThread().interrupt();
                        break;
                    }
                }
                
                // 处理批次
                if (!batch.isEmpty()) {
                    processBatch(batch);
                }
            }
        });
        consumer.start();
    }
    
    private void processBatch(List<Task> batch) {
        // 批量插入数据库、批量发送 MQ 等
        jdbcTemplate.batchUpdate("INSERT ...", batch);
    }
}
```

**策略：**
1. `drainTo` 先批量拿，零阻塞
2. 空队列时 `poll(timeout)` 等待，避免 CPU 空转
3. 拿到一条后，继续 `drainTo` 补满批次

## 🔥 扩展追问 1：`drainTo` 的线程安全问题

```java
// drainTo 本身是线程安全的，但目标集合需要外部同步
List<Task> batch = new ArrayList<>();  // 局部变量，无竞争
queue.drainTo(batch, 100);

// ❌ 危险：多线程共享同一个 batch
List<Task> sharedBatch = new ArrayList<>();
executor.submit(() -> queue.drainTo(sharedBatch, 100));  // 线程1
executor.submit(() -> queue.drainTo(sharedBatch, 100));  // 线程2
// ArrayList 非线程安全，并发修改会丢数据或抛异常
```

**修复：每个线程用自己的 batch，或加同步。**

## 🔥 扩展追问 2：`drainTo` 和 `take` 怎么配合？

```java
// 场景：必须至少有一条才处理，但尽量批量
while (running) {
    List<Task> batch = new ArrayList<>();
    
    // 阻塞等第一条
    Task first = queue.take();
    batch.add(first);
    
    // 非阻塞拿剩下的（最多 99 条）
    queue.drainTo(batch, 99);
    
    processBatch(batch);  // 至少 1 条，最多 100 条
}
```

**适用：** 消息队列消费者，保证有消息才处理，同时尽量批量提升吞吐。

## 🔥 扩展追问 3：不同 `BlockingQueue` 的 `drainTo` 实现

| 队列类型 | `drainTo` 特点 |
|---|---|
| `LinkedBlockingQueue` | 双锁分离，`drainTo` 需全锁，但仍是单次加锁 |
| `ArrayBlockingQueue` | 单锁，循环数组，`drainTo` 批量移动指针 |
| `PriorityBlockingQueue` | 堆结构，`drainTo` 按优先级顺序取出 |
| `DelayQueue` | 只转移已到期的元素，未到期的不拿 |
| `SynchronousQueue` | 容量为 0，`drainTo` 永远返回 0 |

## 🔥 扩展追问 4：批量消费的背压控制

```java
// 生产者：批量生产，批量入队
List<Task> tasks = fetchFromDB(1000);
// ❌ 逐条 offer，1000 次锁竞争
for (Task t : tasks) queue.offer(t);

// ✅ 批量入队（Java 没有原生 API，自己封装）
public <E> int batchOffer(BlockingQueue<E> queue, List<E> list, int maxBatch) {
    int count = 0;
    for (E e : list) {
        if (!queue.offer(e)) {  // 队列满时快速失败
            return count;  // 返回实际入队数量
        }
        count++;
        if (count >= maxBatch) break;
    }
    return count;
}
```

**双向批量：生产者批量入队，消费者批量出队，减少锁竞争 100 倍。**

## 一句话收尾

> `drainTo` 是**批量消费的零拷贝神器**，单次加锁批量转移，性能碾压逐条 `poll()`；**非阻塞，有多少拿多少**，配合 `poll(timeout)` 可实现"至少一条 + 尽量批量"的高效消费模式。
