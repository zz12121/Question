83. 你用 Redis 做分布式限流（滑动窗口）：
```java
public boolean allow(String key, int limit, int window) {
    long now = System.currentTimeMillis();
    redis.zadd(key, now, String.valueOf(now));
    redis.zremrangeByScore(key, 0, now - window * 1000);
    Long count = redis.zcard(key);
    redis.expire(key, window);
    return count <= limit;
}
```
**问题：** 这段代码有什么问题？如何保证原子性？

---
## 先说结论

这段代码有**四个严重问题**：多条 Redis 命令非原子（竞态条件）、无异常回滚、时间精度依赖系统时钟、ZSet 内存持续增长。保证原子性的核心方案：**Lua 脚本** 或 **Redis + Redisson 的 RRateLimiter**，生产环境推荐直接用成熟方案。

---

## 🔥 代码问题分析

```java
public boolean allow(String key, int limit, int window) {
    long now = System.currentTimeMillis();
    redis.zadd(key, now, String.valueOf(now));           // 1. 添加当前请求
    redis.zremrangeByScore(key, 0, now - window * 1000); // 2. 清理过期窗口
    Long count = redis.zcard(key);                        // 3. 统计窗口内数量
    redis.expire(key, window);                            // 4. 设置过期
    return count <= limit;
}
```

### 问题 1：非原子性（竞态条件）

```
时间线（两个线程同时执行）：

T1: 线程 A zadd，窗口内有 100 条
T2: 线程 B zadd，窗口内有 101 条
T3: 线程 A zremrange（清理后 99 条）
T4: 线程 B zremrange（清理后 100 条）
T5: 线程 A zcard = 100，判断通过（实际已超 100）
T6: 线程 B zcard = 100，判断通过

结果：limit=100，实际通过了 101 条，限流失效！
```

### 问题 2：异常无回滚

```java
// 如果执行到一半 Redis 挂了
redis.zadd(key, now, String.valueOf(now));  // 成功
redis.zremrangeByScore(...);                 // 失败
// zadd 的数据残留，窗口统计永远不准
```

### 问题 3：时钟依赖

```java
long now = System.currentTimeMillis();  // 各节点时钟可能不同步
// 节点 A 的时间比节点 B 快 5 秒，限流窗口错位
```

### 问题 4：ZSet 内存问题

```
高并发场景：每秒 10 万请求，窗口 60 秒
ZSet 需要存储 600 万个成员
内存占用：600万 × (8字节score + 8字节member指针) ≈ 100MB+

且 zremrangeByScore 是 O(logN) + M，M 是删除数量，越来越慢
```

## 🔥 优化方案 1：Lua 脚本保证原子性

```lua
-- rate_limiter.lua
local key = KEYS[1]
local now = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local limit = tonumber(ARGV[3])

-- 1. 清理过期成员
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)

-- 2. 获取当前窗口内的请求数
local count = redis.call('ZCARD', key)

-- 3. 判断是否允许
if count < limit then
    -- 允许，添加当前请求
    redis.call('ZADD', key, now, now)
    redis.call('EXPIRE', key, window / 1000)
    return 1  -- 允许
else
    return 0  -- 拒绝
end
```

```java
public boolean allow(String key, int limit, int windowMs) {
    long now = System.currentTimeMillis();
    
    DefaultRedisScript<Long> script = new DefaultRedisScript<>();
    script.setScriptText(luaScript);
    script.setResultType(Long.class);
    
    Long result = redisTemplate.execute(script, 
        Collections.singletonList(key),
        String.valueOf(now),
        String.valueOf(windowMs),
        String.valueOf(limit)
    );
    
    return result == 1;
}
```

**Lua 脚本优势：**
- 单线程执行，天然原子
- 所有操作在一个命令完成，无竞态
- 减少网络往返（RTT）

```java
// ✅ 成熟方案，无需自己实现
@Configuration
public class RedissonConfig {
    
    @Bean
    public RedissonClient redissonClient() {
        Config config = new Config();
        config.useSingleServer().setAddress("redis://localhost:6379");
        return Redisson.create(config);
    }
}

@Service
public class OrderService {
    
    @Autowired
    private RedissonClient redisson;
    
    public void createOrder(Long userId) {
        // 获取限流器：每用户每秒最多 10 个请求
        RRateLimiter limiter = redisson.getRateLimiter("rate:order:" + userId);
        
        // 初始化（只执行一次，幂等）
        limiter.trySetRate(RateType.OVERALL, 10, 1, RateIntervalUnit.SECONDS);
        
        // 尝试获取许可
        if (limiter.tryAcquire()) {
            // 处理业务
        } else {
            throw new RateLimitException("请求过于频繁");
        }
    }
}
```

## 🔥 扩展追问 1：固定窗口 vs 滑动窗口

| 算法 | 实现 | 问题 | 适用 |
|---|---|---|---|
| **固定窗口** | `INCR key` + `EXPIRE` | 窗口边界突刺（最后1秒 + 开始1秒 = 2倍流量） | 简单场景 |
| **滑动窗口** | ZSet 按时间滑动 | 内存占用大，实现复杂 | 精确限流 |
| **令牌桶** | Redisson / Redis Cell | 平滑限流，允许突发 | 生产推荐 |
| **漏桶** | 队列实现 | 严格匀速，无突发 | 流量整形 |

## 🔥 扩展追问 3：限流降级策略

```java
public enum LimitStrategy {
    BLOCK,      // 阻塞等待
    FAIL_FAST,  // 快速失败
    FALLBACK    // 降级处理
}

public void handleRequest(String userId, LimitStrategy strategy) {
    RRateLimiter limiter = redisson.getRateLimiter("rate:" + userId);
    
    switch (strategy) {
        case BLOCK:
            limiter.acquire();  // 阻塞直到获得许可
            process();
            break;
            
        case FAIL_FAST:
            if (!limiter.tryAcquire()) {
                throw new RateLimitException();
            }
            process();
            break;
            
        case FALLBACK:
            if (!limiter.tryAcquire(100, TimeUnit.MILLISECONDS)) {
                fallback();  // 走降级逻辑
                return;
            }
            process();
            break;
    }
}
```
