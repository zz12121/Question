79. 你的事务隔离级别：
```java
@Transactional(isolation = Isolation.READ_UNCOMMITTED)
public void query() {
    // 查询操作
}
```
**问题：** `READ_UNCOMMITTED` 有什么问题？生产环境一般用什么隔离级别？

---
## 先说结论

`READ_UNCOMMITTED`（读未提交）是**最低的隔离级别**，允许读取其他事务未提交的数据（**脏读**），会导致数据不一致、业务逻辑错误。生产环境一般使用 **`READ_COMMITTED`**（Oracle/SQL Server 默认）或 **`REPEATABLE_READ`**（MySQL InnoDB 默认），根据业务需求选择，金融场景可能需要 `SERIALIZABLE` 或乐观锁补偿。

---

## 🔥 四种隔离级别对比

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 适用场景 |
|---|---|---|---|---|
| `READ_UNCOMMITTED` | ✅ 允许 | ✅ 允许 | ✅ 允许 | 几乎不用 |
| `READ_COMMITTED` | ❌ 不允许 | ✅ 允许 | ✅ 允许 | 通用 OLTP（Oracle/SQL Server 默认） |
| `REPEATABLE_READ` | ❌ 不允许 | ❌ 不允许 | ⚠️ 部分允许 | MySQL 默认，事务内一致性 |
| `SERIALIZABLE` | ❌ 不允许 | ❌ 不允许 | ❌ 不允许 | 强一致，性能最差 |

## 🔥 `READ_UNCOMMITTED` 的脏读问题

```java
// 事务 A：转账操作
@Transactional
public void transfer(Long from, Long to, BigDecimal amount) {
    // 扣减 A 账户
    accountMapper.deduct(from, amount);  // 余额 1000 → 500，未提交
    
    // 模拟其他操作（或异常）
    Thread.sleep(1000);
    
    // 异常回滚
    throw new RuntimeException("系统异常，回滚");
}

// 事务 B：查询操作（READ_UNCOMMITTED）
@Transactional(isolation = Isolation.READ_UNCOMMITTED)
public BigDecimal queryBalance(Long accountId) {
    return accountMapper.selectBalance(accountId);  // 读到 500（脏数据）
}
```

**执行时序：**
```
时间线：
  T1: 事务 A 扣减余额，余额变为 500（未提交）
  T2: 事务 B 查询，读到 500（脏读！）
  T3: 事务 A 异常回滚，余额恢复 1000
  T4: 事务 B 拿着 500 做业务决策，数据完全错误
```

## 🔥 生产环境隔离级别选择

### MySQL InnoDB 默认：`REPEATABLE_READ`

```java
@Transactional  // 默认 REPEATABLE_READ
public void processOrder(Long orderId) {
    // 第一次查询
    Order order1 = orderMapper.selectById(orderId);  // 状态 = PENDING
    
    // 其他操作...
    
    // 第二次查询（同一事务内）
    Order order2 = orderMapper.selectById(orderId);  // 状态还是 PENDING（可重复读）
    
    // 即使其他事务修改了，这里读到的是快照
}
```

**优势：** 事务内数据一致，适合复杂业务逻辑。

**注意：** InnoDB 的 RR 通过 MVCC + Gap Lock 基本解决幻读，但不是完全避免。

---

### Oracle/SQL Server 默认：`READ_COMMITTED`

```java
@Transactional(isolation = Isolation.READ_COMMITTED)
public void processOrder(Long orderId) {
    Order order1 = orderMapper.selectById(orderId);  // 状态 = PENDING
    
    // 其他事务提交，状态变为 PAID
    
    Order order2 = orderMapper.selectById(orderId);  // 状态 = PAID（读到最新）
}
```

**优势：** 每次读取都是最新已提交数据，减少锁竞争。

---

## 🔥 为什么 MySQL 用 RR，Oracle 用 RC？

| 数据库 | 默认级别 | 原因 |
|---|---|---|
| **MySQL InnoDB** | `REPEATABLE_READ` | 历史原因，RR 配合 MVCC 性能好，且主从复制（Statement 格式）需要 |
| **Oracle** | `READ_COMMITTED` | RC 实现高效，RR 需要额外开销 |
| **PostgreSQL** | `READ_COMMITTED` | 类似 Oracle |
| **SQL Server** | `READ_COMMITTED` | 可配置，默认 RC |

**MySQL 建议：**
- 普通业务：保持默认 `REPEATABLE_READ`
- 高并发读：可改为 `READ_COMMITTED`（减少 Gap Lock 死锁）

```sql
-- 会话级别修改
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 或 Spring 注解
@Transactional(isolation = Isolation.READ_COMMITTED)
```

## 🔥 扩展追问 1：幻读和解决方案

```java
// REPEATABLE_READ 下的幻读
@Transactional(isolation = Isolation.REPEATABLE_READ)
public void checkOrders(Long userId) {
    // 第一次查询：3 条订单
    List<Order> list1 = orderMapper.selectByUserId(userId);
    
    // 其他事务插入新订单并提交
    
    // 第二次查询：4 条订单（幻读！）
    List<Order> list2 = orderMapper.selectByUserId(userId);
}
```

**解决方案：**

```java
// 方案 1：SELECT FOR UPDATE（加间隙锁）
@Select("SELECT * FROM orders WHERE user_id = #{userId} FOR UPDATE")
List<Order> selectWithLock(@Param("userId") Long userId);

// 方案 2：SERIALIZABLE（性能差）
@Transactional(isolation = Isolation.SERIALIZABLE)

// 方案 3：乐观锁（推荐）
@Update("UPDATE orders SET status = 'PAID', version = version + 1 " +
        "WHERE id = #{id} AND version = #{version}")
int updateWithVersion(Order order);
```

## 🔥 扩展追问 2：Spring 事务隔离级别传播

```java
@Service
public class OrderService {
    
    @Transactional(isolation = Isolation.READ_COMMITTED)
    public void parent() {
        child();  // 子方法的事务隔离级别？
    }
    
    @Transactional(isolation = Isolation.REPEATABLE_READ)
    public void child() {
        // 实际使用父事务的隔离级别（READ_COMMITTED）
        // 子方法的隔离级别注解被忽略！
    }
}
```

**规则：** 事务隔离级别以**外层事务**为准，内层设置无效。

## 🔥 扩展追问 3：隔离级别与锁的关系

| 隔离级别 | 锁行为 |
|---|---|
| `READ_UNCOMMITTED` | 读不加锁，直接读最新（可能脏读） |
| `READ_COMMITTED` | 读加共享锁，读完立即释放 |
| `REPEATABLE_READ` | 读加共享锁，事务结束才释放；写加排他锁 + Gap Lock |
| `SERIALIZABLE` | 所有操作加排他锁，完全串行 |

## 一句话收尾

> `READ_UNCOMMITTED` 允许**脏读**，生产环境**绝对禁用**；MySQL 默认 **`REPEATABLE_READ`**（事务内一致），Oracle/SQL Server 默认 **`READ_COMMITTED`**（减少锁竞争）；金融强一致场景用乐观锁或 `SERIALIZABLE`，高并发读可考虑降隔离级别 + 补偿机制。
