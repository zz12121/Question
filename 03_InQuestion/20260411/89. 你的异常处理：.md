89. 你的异常处理：
```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(Exception.class)
    public Result handle(Exception e) {
        return Result.fail(e.getMessage());
    }
}
```
**问题：** 这样处理异常有什么问题？如何区分业务异常和系统异常？

---
## 先说结论

这种"一刀切"的异常处理会**暴露敏感信息**、**无法区分业务错误和系统故障**、**丢失堆栈不利于排查**。正确做法是分层处理：业务异常返回友好提示，系统异常脱敏后记录详细日志，并配合 HTTP 状态码区分严重程度。

---

## 🔥 当前代码的问题

### 1. 暴露敏感信息

```java
@ExceptionHandler(Exception.class)
public Result handle(Exception e) {
    return Result.fail(e.getMessage());  // 直接把异常信息返回给前端！
}
```

**危险场景：**
- SQL 异常 → 暴露表结构：`Table 'user' doesn't exist`
- 文件操作 → 暴露服务器路径：`/home/admin/config/password.txt (No such file)`
- 空指针 → 暴露代码结构：`Cannot invoke "User.getName()" because "user" is null`

---

### 2. 无法区分异常类型

| 异常类型 | 用户该看到什么 | 实际处理 |
|---------|--------------|---------|
| 参数校验失败 | "手机号格式错误" | 同上 |
| 余额不足 | "账户余额不足" | 同上 |
| 数据库挂了 | "系统繁忙，请稍后" | 暴露 `Communications link failure` |

---

### 3. 丢失关键排查信息

- 没有记录堆栈，线上出问题无法定位
- 没有请求上下文（URL、参数、用户ID）
- 没有错误码，前端无法做针对性处理

---

## 🔥 正确的分层异常处理

### 第一步：定义异常体系

```java
// 业务异常（已知错误，用户可感知）
public class BusinessException extends RuntimeException {
    private final String code;
    private final Object[] args;  // 国际化参数
    
    public BusinessException(String code, Object... args) {
        super(MessageFormat.format(code, args));
        this.code = code;
        this.args = args;
    }
}

// 系统异常（未知错误，用户不感知）
public class SystemException extends RuntimeException {
    public SystemException(String message, Throwable cause) {
        super(message, cause);
    }
}
```

### 第二步：完善的全局处理器

```java
@Slf4j
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    // ========== 业务异常（4xx 客户端错误）==========
    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<Result<?>> handleBusiness(BusinessException e) {
        log.warn("业务异常: code={}, message={}", e.getCode(), e.getMessage());
        
        return ResponseEntity
            .status(HttpStatus.BAD_REQUEST)
            .body(Result.fail(e.getCode(), e.getMessage()));
    }
    
    // 参数校验异常
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Result<?>> handleValidation(MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getFieldErrors().stream()
            .map(error -> error.getField() + ": " + error.getDefaultMessage())
            .collect(Collectors.joining(", "));
        
        return ResponseEntity
            .status(HttpStatus.BAD_REQUEST)
            .body(Result.fail("PARAM_ERROR", message));
    }
    
    // ========== 系统异常（5xx 服务端错误）==========
    @ExceptionHandler(Exception.class)
    public ResponseEntity<Result<?>> handleSystem(Exception e) {
        // 生成错误追踪ID
        String errorId = UUID.randomUUID().toString();
        
        // 详细日志：包含堆栈、请求信息
        log.error("系统异常[{}]: ", errorId, e);
        
        // 返回脱敏信息
        return ResponseEntity
            .status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(Result.fail("SYSTEM_ERROR", 
                "系统繁忙，请稍后重试（错误码：" + errorId + "）"));
    }
    
    // ========== 特定第三方异常 ==========
    @ExceptionHandler(FeignException.class)
    public ResponseEntity<Result<?>> handleFeign(FeignException e) {
        log.error("服务调用异常: status={}, content={}", 
            e.status(), e.contentUTF8());
        
        return ResponseEntity
            .status(HttpStatus.SERVICE_UNAVAILABLE)
            .body(Result.fail("SERVICE_ERROR", "依赖服务暂时不可用"));
    }
}
```

### 第三步：统一返回结构

```java
@Data
public class Result<T> {
    private boolean success;
    private String code;
    private String message;
    private T data;
    private Long timestamp;
    
    public static <T> Result<T> ok(T data) {
        Result<T> r = new Result<>();
        r.success = true;
        r.code = "SUCCESS";
        r.data = data;
        r.timestamp = System.currentTimeMillis();
        return r;
    }
    
    public static <T> Result<T> fail(String code, String message) {
        Result<T> r = new Result<>();
        r.success = false;
        r.code = code;
        r.message = message;
        r.timestamp = System.currentTimeMillis();
        return r;
    }
}
```

## 🔥 使用示例

```java
@Service
public class OrderService {
    
    public void createOrder(CreateOrderRequest request) {
        // 业务校验 → 抛业务异常
        if (request.getAmount().compareTo(BigDecimal.ZERO) <= 0) {
            throw new BusinessException("ORDER_AMOUNT_INVALID", request.getAmount());
        }
        
        User user = userService.getById(request.getUserId());
        if (user == null) {
            throw new BusinessException("USER_NOT_FOUND", request.getUserId());
        }
        
        // 系统错误 → 抛系统异常或让上层捕获
        try {
            paymentClient.charge(request);
        } catch (Exception e) {
            throw new SystemException("支付服务调用失败", e);
        }
    }
}
```

### 追问 2：怎么记录请求上下文？

```java
@Component
public class RequestContextFilter extends OncePerRequestFilter {
    
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {
        
        MDC.put("traceId", UUID.randomUUID().toString());
        MDC.put("userId", getCurrentUserId());
        MDC.put("uri", request.getRequestURI());
        
        try {
            chain.doFilter(request, response);
        } finally {
            MDC.clear();
        }
    }
}
```

**日志格式：**
```
[%d{yyyy-MM-dd HH:mm:ss}] [%X{traceId}] [%X{userId}] [%X{uri}] - %msg%n
```

### 追问 3：异步方法的异常怎么处理？

```java
@Async
public CompletableFuture<Void> asyncTask() {
    // 异常不会走到 @RestControllerAdvice！
    throw new RuntimeException("异步异常");
}

// 正确做法：在异步方法内部捕获，或返回 CompletableFuture 处理
@Async
public CompletableFuture<Result<Void>> asyncTask() {
    return CompletableFuture.supplyAsync(() -> {
        try {
            // 业务逻辑
            return Result.ok(null);
        } catch (BusinessException e) {
            return Result.fail(e.getCode(), e.getMessage());
        } catch (Exception e) {
            log.error("异步任务异常", e);
            return Result.fail("SYSTEM_ERROR", "系统繁忙");
        }
    });
}
```

public Result handleSystem(Exception e) {

## 🔥 异常处理分层总结

| 层级 | 异常类型 | HTTP 状态码 | 用户看到 | 日志记录 |
|-----|---------|------------|---------|---------|
| 业务层 | BusinessException | 400 | 具体错误信息 | WARN |
| 参数校验 | ValidationException | 400 | 字段错误详情 | WARN |
| 认证授权 | AuthenticationException | 401/403 | "请登录"/"无权限" | INFO |
| 服务调用 | FeignException | 503 | "服务暂时不可用" | ERROR |
| 系统未知 | Exception | 500 | "系统繁忙+错误码" | ERROR+堆栈 |

---

## 一句话收尾

> **业务异常给用户看，系统异常给日志看**。分层处理 + 脱敏返回 + 错误追踪 ID，是生产环境异常处理的黄金法则。
