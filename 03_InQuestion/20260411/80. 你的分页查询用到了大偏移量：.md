80. 你的分页查询用到了大偏移量：
```sql
SELECT * FROM orders ORDER BY id LIMIT 5000000, 10;
```
**问题：** 这条 SQL 的执行过程是什么？为什么慢？如何优化？

---
## 先说结论

`LIMIT 5000000, 10` 是**深分页**，MySQL 需要**扫描 5000010 行**，丢弃前 500 万行，只返回最后 10 行，时间复杂度 O(n)，越往后越慢。优化方案：**延迟关联**（先查 ID 再回表）、**覆盖索引**、**游标分页**（记录上次位置）、或**搜索引擎**（ES）。

---

## 🔥 执行过程分析

```sql
SELECT * FROM orders ORDER BY id LIMIT 5000000, 10;
```

**MySQL 实际执行：**

```
1. 从索引 idx_id 的第一行开始扫描
2. 逐行读取，直到第 5000010 行
3. 对每行的主键 id 回表查 *（回表 5000010 次！）
4. 丢弃前 5000000 行
5. 返回最后 10 行

总扫描：5000010 行
总回表：5000010 次
实际返回：10 行
```

**执行计划：**

```sql
EXPLAIN SELECT * FROM orders ORDER BY id LIMIT 5000000, 10;
-- type: index（全索引扫描）
-- rows: 5000010
-- Extra: Using index; Using filesort（可能）
```

---

## 🔥 为什么慢

| 问题 | 说明 |
|---|---|
| **大量扫描** | 需要遍历 500 万行索引 |
| **大量回表** | 每行都要回主键索引查完整数据 |
| **内存排序** | 如果无法走索引排序，还要 filesort |
| **网络传输** | 中间结果占用内存和临时表空间 |

**耗时对比：**

| 偏移量 | 耗时 |
|---|---|
| `LIMIT 0, 10` | 1 ms |
| `LIMIT 10000, 10` | 50 ms |
| `LIMIT 100000, 10` | 500 ms |
| `LIMIT 5000000, 10` | **30+ 秒** |

---

## 🔥 优化方案 1：延迟关联（推荐）

```sql
-- ✅ 先查 ID，再回表
SELECT o.* 
FROM orders o
JOIN (
    SELECT id 
    FROM orders 
    ORDER BY id 
    LIMIT 5000000, 10
) tmp ON o.id = tmp.id;
```

**执行过程：**

```
1. 子查询：只查 id，走覆盖索引
   - 扫描 5000010 行，但不用回表，很快
2. 拿到 10 个 id
3. 主查询：10 次回表查完整数据
4. 返回 10 行

总回表：10 次（vs 原方案 5000010 次）
```

**效果：从 30 秒降到 100 ms。**

## 🔥 优化方案 3：游标分页（最佳实践）

```java
// ✅ 记录上次位置，避免大偏移量
// 第一页
SELECT * FROM orders ORDER BY id LIMIT 10;
// 返回 id: [1, 2, 3, ..., 10]

// 下一页（记录 last_id = 10）
SELECT * FROM orders 
WHERE id > 10 
ORDER BY id LIMIT 10;
// 只扫描 10 行，O(1) 复杂度
```

**通用封装：**

```java
@Data
public class CursorPage<T> {
    private List<T> list;
    private Long nextCursor;  // 下一页起始位置
    private Boolean hasMore;
}

@Service
public class OrderService {
    
    public CursorPage<Order> queryByCursor(Long cursor, int size) {
        List<Order> list;
        if (cursor == null) {
            // 第一页
            list = orderMapper.selectFirstPage(size);
        } else {
            // 下一页
            list = orderMapper.selectByCursor(cursor, size);
        }
        
        Long nextCursor = list.isEmpty() ? null : list.get(list.size() - 1).getId();
        
        CursorPage<Order> page = new CursorPage<>();
        page.setList(list);
        page.setNextCursor(nextCursor);
        page.setHasMore(list.size() == size);
        return page;
    }
}

// Mapper
@Select("SELECT * FROM orders WHERE id > #{cursor} ORDER BY id LIMIT #{size}")
List<Order> selectByCursor(@Param("cursor") Long cursor, @Param("size") int size);
```

**限制：** 不支持跳页（只能上一页/下一页），但大多数场景（无限滚动）不需要跳页。

## 🔥 优化方案 4：搜索引擎（ES）

```java
// ✅ 大数据量分页，用 Elasticsearch
SearchRequest request = new SearchRequest("orders");
request.source()
    .sort("id", SortOrder.ASC)
    .from(5000000)  // ES 的 from + size 也有深分页问题
    .size(10);

// 更好的方案：search_after
request.source()
    .sort("id", SortOrder.ASC)
    .searchAfter(new Object[]{lastId})  // 游标方式
    .size(10);
```

**ES 深分页优化：**
- `from + size`：默认限制 10000，超过报错
- `search_after`：游标方式，无深度限制
- `scroll`：快照方式，适合导出

## 一句话收尾

> `LIMIT offset, size` 的**深分页是 O(n) 复杂度**，偏移量越大越慢；生产环境**优先用游标分页**（`WHERE id > #{cursor}`），必须跳页时用**延迟关联**（先查 ID 再回表），大数据量场景直接上 **Elasticsearch**，从根本上避免深分页问题。
