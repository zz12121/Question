62. 你用 `CyclicBarrier` 实现多线程分阶段执行：
```java
CyclicBarrier barrier = new CyclicBarrier(3);
for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        phase1();
        barrier.await();
        phase2();
    }).start();
}
```
**问题：** `CyclicBarrier` 和 `CountDownLatch` 有什么区别？如果某个线程中断，会发生什么？

---
## 先说结论

`CyclicBarrier` 是"大家一起等，到齐再继续"的栅栏；`CountDownLatch` 是"倒计时炸弹，归零即触发"。两者方向相反、语义不同。**一旦某个线程在 `await()` 时被中断，整个栅栏会进入 Broken 状态，所有等待线程都会收到 `BrokenBarrierException`，这是 CyclicBarrier 特有的连锁雪崩机制。**

---

## 🔥 核心对比

| 维度 | `CyclicBarrier` | `CountDownLatch` |
|---|---|---|
| **方向** | N 个线程互相等待（汇聚） | 1个（或多个）线程等待 N 个事件完成 |
| **计数方式** | `await()` 一次减一，到 0 触发 | `countDown()` 减一，`await()` 等归零 |
| **可复用** | ✅ 可重置（`reset()`），循环使用 | ❌ 一次性，归零后不可重置 |
| **参与者角色** | 所有线程平等，都既是参与者又是等待者 | 分两类：等待者（`await`）和计数者（`countDown`） |
| **异常传播** | 中断/超时 → `BrokenBarrierException` 广播给所有等待线程 | 无 broken 概念，只影响自身 |
| **barrierAction** | ✅ 支持最后一个到达时执行汇总任务 | ❌ 不支持 |
| **适用场景** | 多阶段并行计算（分治/MapReduce） | 主线程等待子任务完成 |

---

## 🔥 详细展开：线程中断时发生了什么

### 正常流程

```
Thread-0: phase1() → await() ─┐
Thread-1: phase1() → await() ─┼─ 第3个到达，栅栏打开 → 所有人继续 phase2()
Thread-2: phase1() → await() ─┘
```

### 中断场景（Thread-1 在 await() 时被中断）

```
Thread-0: await() ... 等待中
Thread-1: await() → 被 interrupt() → 抛出 InterruptedException
                  → 栅栏自动标记为 BROKEN
Thread-2: await() → 检测到 BROKEN → 立刻抛出 BrokenBarrierException

Thread-0: 也立刻被唤醒 → 抛出 BrokenBarrierException
```

**关键结论：一个线程中断，所有等待线程全部被惊醒并失败——这叫"栅栏雪崩"。**

---

## 🔥 源码级原理

`CyclicBarrier.await()` 内部用 `ReentrantLock` + `Condition` 实现：

```java
// 简化版核心逻辑
private int dowait(boolean timed, long nanos) throws ... {
    final ReentrantLock lock = this.lock;
    lock.lock();
    try {
        final Generation g = generation;
        if (g.broken) throw new BrokenBarrierException(); // 已破损直接抛

        if (Thread.interrupted()) {
            breakBarrier(); // 标记 broken，唤醒所有等待线程
            throw new InterruptedException();
        }

        int index = --count;
        if (index == 0) { // 最后一个到达
            // 执行 barrierCommand（如果有）
            nextGeneration(); // 重置计数，进入下一代
            return 0;
        }

        // 未到齐，等待
        for (;;) {
            try {
                trip.await(); // 阻塞在此
            } catch (InterruptedException ie) {
                if (g == generation && !g.broken) {
                    breakBarrier(); // 传染给所有人
                    throw ie;
                }
            }
            if (g.broken) throw new BrokenBarrierException();
        }
    } finally {
        lock.unlock();
    }
}
```

**`breakBarrier()` 做了两件事：把 `generation.broken = true`，然后 `trip.signalAll()` 唤醒所有线程。**

## 🔥 正确的异常处理姿势

### 方案一：捕获两种异常，记录并降级

```java
CyclicBarrier barrier = new CyclicBarrier(3, () -> {
    System.out.println("本阶段汇总完成，进入下一阶段");
});

List<Exception> errors = new CopyOnWriteArrayList<>();

for (int i = 0; i < 3; i++) {
    final int id = i;
    new Thread(() -> {
        try {
            phase1(id);
            barrier.await();   // 可能抛 InterruptedException / BrokenBarrierException
            phase2(id);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt(); // 恢复中断标志
            errors.add(e);
            System.out.println("线程 " + id + " 被中断");
        } catch (BrokenBarrierException e) {
            errors.add(e);
            System.out.println("线程 " + id + " 检测到栅栏已破损，放弃执行");
        }
    }).start();
}
```

### 方案二：超时 await，防止永久阻塞

```java
try {
    barrier.await(5, TimeUnit.SECONDS); // 超时抛 TimeoutException
} catch (TimeoutException e) {
    // 超时也会触发 breakBarrier，影响其他线程！
    System.out.println("等待超时，栅栏已破损");
}
```

> ⚠️ 注意：**`await(timeout)` 超时同样会触发 `breakBarrier()`**，连锁影响其他线程！

### 方案三：栅栏破损后重置（谨慎使用）

```java
catch (BrokenBarrierException e) {
    if (barrier.isBroken()) {
        barrier.reset(); // 重置到初始状态（可能导致其他线程再次抛异常）
    }
}
```

> ⚠️ `reset()` 时若还有线程在 `await()`，那些线程会收到 `BrokenBarrierException`，需慎用。

## 🔥 扩展追问 1：barrierAction 是什么？执行在哪个线程？

```java
CyclicBarrier barrier = new CyclicBarrier(3, () -> {
    // 由最后一个到达栅栏的线程执行！
    System.out.println("执行汇总：" + Thread.currentThread().getName());
    mergeResults();
});
```

**执行线程是"最后一个调用 `await()` 的线程"**，不是主线程，也不是固定线程。如果 barrierAction 抛异常，同样会 `breakBarrier()`。

## 🔥 扩展追问 2：CyclicBarrier 的"可复用"体现在哪？

```java
CyclicBarrier barrier = new CyclicBarrier(3);

// 第一轮
thread.run() → await() // 3个都到了，打开，generation 自动切换到下一代

// 第二轮（同一个 barrier 对象！）
thread.run() → await() // 又等 3 个，复用

// CountDownLatch 做不到这一点，归零后 count 永远是 0
```

底层通过 `Generation` 对象实现"换代"：每次栅栏打开，就 `new Generation()`，旧代的 broken 状态不会污染新代。

## 🔥 扩展追问 3：什么场景用 CyclicBarrier，什么场景用 CountDownLatch？

```
CountDownLatch：
  ✅ 主线程等待多个子任务完成
  ✅ 服务启动等待多个依赖就绪
  ✅ 一次性触发，不需要复用

CyclicBarrier：
  ✅ 多线程分阶段计算（MapReduce、并行模拟）
  ✅ 所有参与线程需要同步推进
  ✅ 需要多轮循环的并发任务
  ✅ 需要在每阶段结束时执行汇总逻辑（barrierAction）
```

## 🔥 扩展追问 4：用 Phaser 代替 CyclicBarrier 更灵活

```java
// Phaser 支持动态注册/注销参与者
Phaser phaser = new Phaser(3);

for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        phase1();
        phaser.arriveAndAwaitAdvance(); // 等所有人完成 phase1

        phase2();
        phaser.arriveAndAwaitAdvance(); // 等所有人完成 phase2

        phaser.arriveAndDeregister();   // 中途退出，不影响其他线程
    }).start();
}
```

`Phaser` 是 CyclicBarrier 的超集——支持动态参与者数量，适合参与者会中途加入/退出的场景。

## 一句话收尾

> `CyclicBarrier` 是"等齐了再走"的可复用栅栏，核心风险是**中断/超时会触发 BrokenBarrierException 广播给所有等待线程**，生产中必须同时捕获 `InterruptedException` 和 `BrokenBarrierException`，并在 finally 中做好降级处理。

答案已经在上面了，重点梳理一下：

---

**核心差异一句话**：`CountDownLatch` 是"倒计时触发"（单向，一次性）；`CyclicBarrier` 是"等齐了再走"（双向汇聚，可复用）。

**最容易踩的坑——中断雪崩**：

```
Thread-1 在 await() 时被中断
  → 内部调用 breakBarrier()
    → generation.broken = true
    → trip.signalAll() 唤醒所有等待线程
      → Thread-0、Thread-2 全部抛 BrokenBarrierException
```

**生产中正确写法，必须同时处理两种异常**：

```java
try {
    barrier.await(5, TimeUnit.SECONDS);
} catch (InterruptedException e) {
    Thread.currentThread().interrupt(); // 恢复中断标志
    // 降级处理
} catch (BrokenBarrierException e) {
    // 其他线程出问题了，本线程跟着放弃
} catch (TimeoutException e) {
    // 超时也会 breakBarrier，影响其他人！
}
```

> 还有一个细节：`barrierAction` 是由**最后一个到达栅栏的线程**执行的，不是主线程，这是高频追问点。
