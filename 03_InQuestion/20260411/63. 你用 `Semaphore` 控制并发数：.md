63. 你用 `Semaphore` 控制并发数：
```java
Semaphore semaphore = new Semaphore(10);
public void handleRequest() {
    semaphore.acquire();
    try {
        process();
    } finally {
        semaphore.release();
    }
}
```
**问题：** 如果 `release()` 调用次数多于 `acquire()`，会发生什么？如何防止？

---
## 先说结论

`Semaphore` 内部是**无上界计数器**，`release()` 比 `acquire()` 多调一次，许可证数量就会**超出初始值**，导致原本应该被限流的请求"漏进来"——这是一个**静默的逻辑错误，不会抛异常，但会破坏并发控制语义**。

---

## 🔥 核心原理：Semaphore 是什么

`Semaphore` 本质是 AQS 的共享模式实现，内部维护一个 `state` 计数器：

```
初始 state = 10（许可证数量）

acquire()：state - 1，若 state < 0 则阻塞
release()：state + 1，无条件执行，没有上界检查！
```

**关键：`release()` 只是 `state + 1`，它不检查你是否曾经 `acquire()` 过，也不检查 state 是否超过了初始值。**

---

## 🔥 多次 release() 的后果演示

```java
Semaphore semaphore = new Semaphore(2); // 初始许可证 = 2

semaphore.acquire(); // state = 1
semaphore.acquire(); // state = 0，此时已满负荷

semaphore.release(); // state = 1
semaphore.release(); // state = 2，正常归还
semaphore.release(); // state = 3 ！！超出初始值

// 此时有 3 个许可证，3 个线程可以同时进入
// 原来限制 2 并发，现在变成 3——限流被悄悄突破
```

继续多次 release：

```java
// 极端情况：
for (int i = 0; i < 1000; i++) semaphore.release();
// state = 1002，限流形同虚设，大量请求涌入
```

## 🔥 三种常见的多释放场景

### 场景 1：异常路径提前 release（题目代码是安全的）

```java
// 题目中的写法是正确的：
semaphore.acquire();
try {
    process();
} finally {
    semaphore.release(); // acquire 成功才能进入 try，release 只执行一次 ✅
}
```

### 场景 2：忘记判断 acquire 是否成功

```java
// 危险写法：
boolean acquired = semaphore.tryAcquire();
try {
    process();
} finally {
    semaphore.release(); // ❌ acquired = false 时也 release！多了一次
}
```

**修复：**

```java
boolean acquired = semaphore.tryAcquire();
try {
    if (!acquired) {
        // 降级处理，不进入临界区
        return fallback();
    }
    return process();
} finally {
    if (acquired) {        // ✅ 只有真正 acquire 成功，才 release
        semaphore.release();
    }
}
```

### 场景 3：并发中手动多次调用 release

```java
// 比如重试逻辑写错：
public void doWithRetry() {
    semaphore.acquire();
    try {
        process();
    } catch (Exception e) {
        semaphore.release(); // ❌ 异常时手动 release 一次
        throw e;
    } finally {
        semaphore.release(); // ❌ finally 里又 release 一次，共 2 次！
    }
}
```

**修复：只在 finally 里 release，catch 里不要 release：**

```java
public void doWithRetry() {
    semaphore.acquire();
    try {
        process();
    } finally {
        semaphore.release(); // ✅ 只在这里，统一释放
    }
}
```

## 🔥 如何防止：封装 + 有界 Semaphore

### 方案一：封装成不可多释放的工具类

```java
public class BoundedSemaphore {
    private final Semaphore semaphore;
    private final int maxPermits;

    public BoundedSemaphore(int permits) {
        this.semaphore = new Semaphore(permits);
        this.maxPermits = permits;
    }

    public void acquire() throws InterruptedException {
        semaphore.acquire();
    }

    public void release() {
        // 检查当前可用许可，超出上界则拒绝 release
        if (semaphore.availablePermits() >= maxPermits) {
            throw new IllegalStateException(
                "Release called more times than acquire! permits=" 
                + semaphore.availablePermits());
        }
        semaphore.release();
    }
}
```

### 方案二：用 tryAcquire + 标志位，彻底隔离 acquire/release 配对

```java
public <T> T execute(Callable<T> task) throws Exception {
    boolean acquired = semaphore.tryAcquire(500, TimeUnit.MILLISECONDS);
    if (!acquired) {
        throw new RejectedExecutionException("限流：服务繁忙，请稍后重试");
    }
    try {
        return task.call();
    } finally {
        semaphore.release(); // acquired 为 true 才到这里，配对保证
    }
}
```

### 方案三：用 Guava RateLimiter 替代（更适合限流场景）

```java
// RateLimiter 基于令牌桶，天然无"多释放"问题
RateLimiter limiter = RateLimiter.create(10.0); // 每秒10个请求

public void handleRequest() {
    limiter.acquire(); // 等待令牌，不需要手动 release
    process();         // 令牌消耗后自动恢复，不存在多释放
}
```

## 🔥 扩展追问 1：公平模式 vs 非公平模式有什么区别？

```java
new Semaphore(10, true);  // 公平：FIFO 队列，先等先得，防饥饿
new Semaphore(10, false); // 非公平（默认）：新来的线程可能插队抢到许可证
```

- **非公平**：吞吐量更高，但长时间等待的线程可能一直抢不到（饥饿）
- **公平**：延迟更稳定，适合对等待时间敏感的场景（如数据库连接池）

## 🔥 扩展追问 2：Semaphore 和连接池有什么关系？

```
数据库连接池本质上就是一个 Semaphore：
  - acquire() = 从池中获取连接
  - release() = 归还连接到池中
  - 多次 release = 池中连接数溢出（等价于凭空多出连接）

HikariCP、Druid 内部都有连接数上界校验，
防止归还的连接数超过池的最大容量。
```

## 🔥 扩展追问 3：`acquire()` 和 `tryAcquire()` 怎么选？

| 方法 | 行为 | 适用场景 |
|---|---|---|
| `acquire()` | 阻塞等待，直到拿到许可 | 任务必须执行，可以排队 |
| `tryAcquire()` | 立即返回 true/false | 快速失败，直接降级 |
| `tryAcquire(timeout)` | 等待一段时间 | 有超时容忍度的场景 |

**生产中推荐 `tryAcquire(timeout)` + 降级，避免线程无限阻塞撑满线程池。**

---

## 一句话收尾

> `Semaphore.release()` 没有上界校验，多调一次就多出一个许可证——**核心防护原则是：`acquire` 和 `release` 必须一一配对，`release` 只放在 `finally` 中，且只有 `acquire` 成功后才能 `release`。**
