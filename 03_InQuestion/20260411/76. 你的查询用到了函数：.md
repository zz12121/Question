76. 你的查询用到了函数：
```sql
SELECT * FROM orders WHERE DATE(create_time) = '2024-01-01';
```
**问题：** 这条 SQL 有什么问题？索引会失效吗？如何优化？

---
## 先说结论

`DATE(create_time)` 是**函数作用于列**，会导致**索引失效**，全表扫描。优化方案：**避免函数，用范围查询** `create_time >= '2024-01-01' AND create_time < '2024-01-02'`，或**函数索引**（MySQL 8.0.13+/PostgreSQL 支持）。

---

## 🔥 为什么索引会失效

```sql
-- 原始 SQL
SELECT * FROM orders WHERE DATE(create_time) = '2024-01-01';

-- 实际执行
DATE(create_time) = '2024-01-01'
-- 对每一行数据调用 DATE() 函数，再比较
```

**B+Tree 索引存储的是 `create_time` 的原始值**，不是 `DATE(create_time)` 的结果：

```
索引中的值：2024-01-01 10:30:00, 2024-01-01 14:20:00, 2024-01-02 09:00:00...
函数结果：  2024-01-01, 2024-01-01, 2024-01-02...

索引无法匹配函数计算后的值，只能全表扫描！
```

**执行计划对比：**

```sql
-- 函数导致索引失效
EXPLAIN SELECT * FROM orders WHERE DATE(create_time) = '2024-01-01';
-- type: ALL（全表扫描）
-- rows: 1000000

-- 范围查询走索引
EXPLAIN SELECT * FROM orders WHERE create_time >= '2024-01-01' 
  AND create_time < '2024-01-02';
-- type: range（索引范围扫描）
-- rows: 1000
```

---

## 🔥 优化方案 1：范围查询（推荐）

```sql
-- ✅ 最通用，所有数据库支持
SELECT * FROM orders 
WHERE create_time >= '2024-01-01' 
  AND create_time < '2024-01-02';

-- 等效写法（注意边界）
SELECT * FROM orders 
WHERE create_time BETWEEN '2024-01-01 00:00:00' 
                      AND '2024-01-01 23:59:59.999';
```

**优势：**
- 完全走索引
- 无函数计算开销
- 可复用现有索引

---

## 🔥 优化方案 2：函数索引（MySQL 8.0.13+）

```sql
-- ✅ MySQL 8.0.13+ 支持函数索引
ALTER TABLE orders ADD INDEX idx_date_create ((DATE(create_time)));

-- 然后原 SQL 就能走索引了
SELECT * FROM orders WHERE DATE(create_time) = '2024-01-01';
-- 会使用 idx_date_create
```

**注意：**
- 只适用于 MySQL 8.0.13+
- 需要额外存储空间
- 插入/更新时有维护开销

---

## 🔥 优化方案 3：虚拟列 + 索引（MySQL 5.7+）

```sql
-- ✅ MySQL 5.7+ 虚拟列
ALTER TABLE orders 
ADD COLUMN create_date DATE AS (DATE(create_time)) VIRTUAL,
ADD INDEX idx_create_date (create_date);

-- 查询用虚拟列
SELECT * FROM orders WHERE create_date = '2024-01-01';
```

**优势：** 兼容旧版本 MySQL，虚拟列不占用存储空间（5.7+）。

## 🔥 优化方案 4：应用层处理

// ✅ 代码里计算好范围，传入参数
public List<Order> findByDate(LocalDate date) {
    LocalDateTime start = date.atStartOfDay();           // 2024-01-01 00:00:00
    LocalDateTime end = date.plusDays(1).atStartOfDay(); // 2024-01-02 00:00:00
    
    return orderMapper.selectByTimeRange(start, end);
}

// Mapper

## 🔥 扩展追问 1：还有哪些操作会导致索引失效？

| 操作 | 示例 | 结果 |
|---|---|---|
| **函数** | `DATE(col)`, `UPPER(col)` | 失效 |
| **运算** | `col + 1 = 10` | 失效 |
| **隐式转换** | `varchar_col = 123`（数字） | 失效 |
| **LIKE 左模糊** | `LIKE '%abc'` | 失效 |
| **NOT 条件** | `NOT IN`, `!=`, `<>` | 可能失效 |
| **OR 条件** | `col1 = 1 OR col2 = 2` | 可能失效（除非都有索引） |

## 🔥 扩展追问 2：时间查询的更多场景

```sql
-- 查某月
SELECT * FROM orders 
WHERE create_time >= '2024-01-01' 
  AND create_time < '2024-02-01';

-- 查某时
SELECT * FROM orders 
WHERE create_time >= '2024-01-01 14:00:00' 
  AND create_time < '2024-01-01 15:00:00';

-- 查最近 N 天（动态）
SELECT * FROM orders 
WHERE create_time >= DATE_SUB(NOW(), INTERVAL 7 DAY);

-- 注意：NOW() 是确定性的，可以走索引
-- 但避免在列上用函数：WHERE DATE_SUB(create_time, INTERVAL 1 DAY) = ...
```

## 🔥 扩展追问 3：分区表优化

```sql
-- 大数据量时，按时间分区
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    create_time DATETIME,
    ...
) PARTITION BY RANGE (YEAR(create_time)) (
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION pfuture VALUES LESS THAN MAXVALUE
);

-- 查 2024 年数据，只扫描 p2024 分区
SELECT * FROM orders WHERE create_time >= '2024-01-01' 
                      AND create_time < '2025-01-01';
```

## 一句话收尾

> **不要在索引列上用函数！** `DATE(create_time)` 导致全表扫描，优化为**范围查询** `>= AND <`，或 MySQL 8.0+ 用**函数索引**，大数据量配合**分区表** pruning，性能提升百倍。
