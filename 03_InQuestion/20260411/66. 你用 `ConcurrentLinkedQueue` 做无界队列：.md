66. 你用 `ConcurrentLinkedQueue` 做无界队列：
```java
ConcurrentLinkedQueue<Task> queue = new ConcurrentLinkedQueue<>();

// 生产者
queue.offer(task);

// 消费者
while (true) {
    Task task = queue.poll();
    if (task != null) process(task);
}
```
**问题：** 如果生产者速度远快于消费者，会出现什么问题？和 `BlockingQueue` 有什么区别？

---
## 先说结论

`ConcurrentLinkedQueue` 是**无界非阻塞队列**，生产者速度超过消费者时会导致 **OOM（内存溢出）**，因为它会无限增长直到撑爆堆内存。和 `BlockingQueue` 的核心区别在于：前者"满了就崩"，后者"满了就阻塞"，生产环境无界队列是**高危操作**。

---

## 🔥 核心对比

| 特性 | `ConcurrentLinkedQueue` | `BlockingQueue` (如 `LinkedBlockingQueue`) |
|---|---|---|
| **容量** | 无界，无限增长 | 可设界（有界时）或无界 |
| **阻塞行为** | 非阻塞，`offer/poll` 立即返回 | `put/take` 会阻塞等待 |
| **内存风险** | 生产者快 → OOM | 有界时阻塞，保护内存 |
| **适用场景** | 内存充足、生产者消费者速度匹配 | 需要背压控制、防止内存溢出 |
| **实现机制** | CAS + 链表，无锁 | ReentrantLock + Condition |

---

## 🔥 无界队列的 OOM 灾难

```java
// 生产者疯狂生产
ExecutorService producer = Executors.newSingleThreadExecutor();
producer.submit(() -> {
    while (true) {
        queue.offer(new Task(1024)); // 每个任务 1KB
        // 没有背压，没有限制，一直塞
    }
});

// 消费者慢慢消费
while (true) {
    Task task = queue.poll();
    if (task != null) {
        Thread.sleep(100); // 处理慢
        process(task);
    }
}
```

**结果：**
```
队列长度: 10万 → 100万 → 1000万...
堆内存: 100MB → 1GB → 8GB...
最终: java.lang.OutOfMemoryError: Java heap space
```

**更隐蔽的场景：突发流量**

```java
// 平时没问题，大促时流量激增 100 倍
for (Request req : burstTraffic) {  // 100万请求瞬间涌入
    queue.offer(req);  // 全部进内存，JVM 直接崩
}
```

## 🔥 正确的背压控制：有界 BlockingQueue

```java
// ✅ 有界队列，满了就阻塞生产者
BlockingQueue<Task> queue = new LinkedBlockingQueue<>(1000);

// 生产者
try {
    queue.put(task);  // 队列满时阻塞，自然背压
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
    // 降级处理
}

// 消费者
while (true) {
    try {
        Task task = queue.take();  // 空时阻塞，有数据才唤醒
        process(task);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        break;
    }
}
```

**背压效果：**
```
生产者速度 > 消费者速度
  → 队列增长到 1000
  → put() 阻塞
  → 上游线程池饱和 → 拒绝新请求
  → 系统自我保护，不会 OOM
```

## 🔥 扩展方案：拒绝策略 + 降级

```java
BlockingQueue<Task> queue = new ArrayBlockingQueue<>(1000);

// 非阻塞 offer + 拒绝策略
boolean offered = queue.offer(task, 100, TimeUnit.MILLISECONDS);
if (!offered) {
    // 队列满，快速失败或降级
    metrics.recordReject();
    return fallbackResponse();  // 返回缓存或友好提示
}
```

## 🔥 扩展追问 1：`ConcurrentLinkedQueue` 什么时候能用？

```java
// ✅ 场景 1：生产者消费者速度匹配，且内存充足
// 比如：单生产者单消费者，处理速度差不多

// ✅ 场景 2：需要极高并发，且能控制上游流量
// 比如：Disruptor 框架内部，配合 RingBuffer 使用

// ✅ 场景 3：配合外部限流，确保不会无限增长
RateLimiter limiter = RateLimiter.create(1000);  // 每秒 1000
limiter.acquire();
queue.offer(task);
```

**原则：能用有界就别用无界。**

## 🔥 扩展追问 2：`poll()` 的忙等待问题

```java
// ❌ 问题代码：CPU 空转
while (true) {
    Task task = queue.poll();  // 无数据时立即返回 null
    if (task != null) {
        process(task);
    }
    // 没有 sleep，CPU 100%
}

// ✅ 修复：BlockingQueue 的 take() 会阻塞，不耗 CPU
while (true) {
    Task task = queue.take();  // 空时阻塞，有数据才唤醒
    process(task);
}
```

`ConcurrentLinkedQueue` 的 `poll()` 是**非阻塞的**，空队列时返回 `null`，如果消费者无脑循环，会导致 CPU 飙高。

## 🔥 扩展追问 3：`LinkedBlockingQueue` vs `ArrayBlockingQueue`

| 特性 | `LinkedBlockingQueue` | `ArrayBlockingQueue` |
|---|---|---|
| 底层结构 | 链表 | 循环数组 |
| 默认容量 | 无界（`Integer.MAX_VALUE`） | 必须指定 |
| 内存分配 | 每次插入新建节点 | 预分配，复用 |
| 性能 | 插入删除 O(1)，但有 GC 压力 | 插入删除 O(1)，无 GC 压力 |
| 锁 | 两把锁（入队/出队分离） | 一把锁 |

**建议：**
- 需要高吞吐且能控制内存 → `ArrayBlockingQueue`
- 不确定容量但想要链表语义 → `LinkedBlockingQueue`（但务必设界！）

## 🔥 扩展追问 4：Disruptor 为什么更快？

```java
// Disruptor：基于 RingBuffer 的有界队列，无锁 CAS
Disruptor<Event> disruptor = new Disruptor<>(
    Event::new, 
    1024,           // 必须是 2 的幂次方
    Executors.defaultThreadFactory(),
    ProducerType.SINGLE,    // 单/多生产者
    new BlockingWaitStrategy()  // 等待策略
);
```

| 特性 | `BlockingQueue` | Disruptor |
|---|---|---|
| 锁 | ReentrantLock | 无锁 CAS |
| 伪共享 | 有 | 消除（缓存行填充） |
| 批量消费 | 不支持 | 支持批量读取 |
| 延迟 | 微秒级 | 纳秒级 |

**金融交易、高频场景首选 Disruptor。**

## 🔥 扩展追问 5：线程池的队列选择

```java
// ❌ 危险：Executors 默认无界队列
ExecutorService executor = Executors.newFixedThreadPool(10);
// 内部是 LinkedBlockingQueue(Integer.MAX_VALUE) → OOM 风险

// ✅ 正确：自定义有界队列 + 拒绝策略
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    4, 8, 60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(100),           // 有界队列
    new ThreadFactoryBuilder().setNameFormat("pool-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy() // 队列满时，提交者自己执行
);
```

## 一句话收尾

> `ConcurrentLinkedQueue` 无界非阻塞，生产者快就会 OOM，**生产环境必须用有界 `BlockingQueue`**，配合 `put/take` 实现自然背压，或用 `offer` 超时 + 降级拒绝，保护系统内存。
