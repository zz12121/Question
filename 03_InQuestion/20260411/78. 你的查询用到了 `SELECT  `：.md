78. 你的查询用到了 `SELECT *`：
```sql
SELECT * FROM orders WHERE user_id = 100;
```
**问题：** 如果表有 100 个字段，有什么问题？如何优化？

---
## 先说结论

`SELECT *` 在**字段多、数据量大、网络带宽受限**的场景下是性能杀手，会导致**网络传输膨胀、内存占用飙升、缓存命中率下降、索引覆盖失效**。优化方案：**只查需要的字段**、配合**覆盖索引**、或用**垂直拆分**减少单表宽度。

---

## 🔥 `SELECT *` 的四大问题

### 1. 网络传输膨胀

```
表结构：100 个字段，平均行大小 2KB
查询 10 万行：

SELECT *：    10万 × 2KB = 200MB 网络传输
SELECT id,name：10万 × 0.1KB = 10MB 网络传输

差距：20 倍！
```

**实际场景：**
- 微服务间调用，200MB 数据在 100ms 内传完？不可能
- 结果：超时、重试、服务雪崩

---

### 2. 内存占用飙升

```java
// MyBatis 查询结果映射
List<Order> orders = mapper.selectAll();  // 10万条 × 100字段

// JVM 堆内存：
// - 数据库结果集缓存
// - MyBatis 对象列表  
// - 可能触发 Young GC 甚至 Full GC
```

```
内存占用估算：
  每行 2KB × 10万行 = 200MB 原始数据
  Java 对象头 + 引用开销 ≈ 2-3 倍
  实际占用：400-600MB
```

---

### 3. 缓存命中率下降

```
MySQL Buffer Pool 有限（通常 8GB）：

SELECT * 读取大字段（text/blob）：
  - 把大量数据加载到 Buffer Pool
  - 热数据被挤出缓存
  - 后续查询命中率下降，磁盘 IO 增加
```

---

### 4. 索引覆盖失效

```sql
-- 有联合索引 idx_user_id_name(user_id, name)

SELECT * FROM orders WHERE user_id = 100;
-- 需要回表查所有字段，索引覆盖失效

SELECT user_id, name FROM orders WHERE user_id = 100;
-- 索引包含所有需要字段，无需回表，性能提升 10 倍
```

---

## 🔥 优化方案 1：只查需要的字段（推荐）

```sql
-- ❌ 避免
SELECT * FROM orders WHERE user_id = 100;

-- ✅ 明确字段
SELECT id, order_no, amount, status, create_time 
FROM orders 
WHERE user_id = 100;
```

```java
// MyBatis 只映射需要的字段
@Select("SELECT id, order_no, amount, status, create_time " +
        "FROM orders WHERE user_id = #{userId}")
List<OrderSummary> selectByUserId(@Param("userId") Long userId);

// 用 DTO 替代完整 Entity
@Data
public class OrderSummary {
    private Long id;
    private String orderNo;
    private BigDecimal amount;
    private Integer status;
    private LocalDateTime createTime;
}
```

---

## 🔥 优化方案 2：配合覆盖索引

```sql
-- 高频查询：用户订单列表（只需要 id, order_no, amount, status, create_time）
CREATE INDEX idx_user_id_covering 
ON orders(user_id, id, order_no, amount, status, create_time);

-- 查询
SELECT id, order_no, amount, status, create_time 
FROM orders 
WHERE user_id = 100;

-- 执行计划：Using index（覆盖索引，无需回表）
```

**覆盖索引优势：**
- 索引通常比表小（InnoDB 二级索引叶子节点是主键）
- 顺序读取索引，比随机回表快 10-100 倍

## 🔥 优化方案 3：垂直拆分

```sql
-- 大字段拆到扩展表
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    order_no VARCHAR(32) NOT NULL,
    amount DECIMAL(18,2) NOT NULL,
    status TINYINT NOT NULL,
    create_time DATETIME NOT NULL,
    -- 核心字段，频繁查询
    INDEX idx_user_id(user_id)
);

CREATE TABLE orders_ext (
    order_id BIGINT PRIMARY KEY,
    remark TEXT,           -- 大文本
    invoice_info JSON,     -- JSON 字段
    logistics_detail TEXT, -- 物流详情
    -- 大字段、低频访问字段
    FOREIGN KEY (order_id) REFERENCES orders(id)
);
```

```java
// 列表查询：只查主表
List<Order> list = orderMapper.selectByUserId(userId);

// 详情查询：主表 + 扩展表 JOIN
OrderDetail detail = orderMapper.selectDetailById(orderId);
```

## 🔥 优化方案 4：分页 + 流式查询

```java
// ❌ 一次性加载大量数据
List<Order> all = orderMapper.selectAll();  // 内存爆炸

// ✅ 分页查询
PageHelper.startPage(1, 100);
List<Order> page = orderMapper.selectByUserId(userId);

// ✅ 流式查询（MyBatis）
@Select("SELECT id, order_no FROM orders WHERE user_id = #{userId}")
@Options(resultSetType = ResultSetType.FORWARD_ONLY, fetchSize = 1000)
void selectStream(@Param ResultHandler<Order> handler);
```

## 🔥 扩展追问 1：什么场景可以用 `SELECT *`？

```sql
-- ✅ 小表（< 1000 行，字段 < 10 个）
SELECT * FROM config WHERE key = 'xxx';

-- ✅ 临时查询、数据分析（非生产环境）
-- 但要 LIMIT 防止拖垮数据库

-- ✅ 配合 LIMIT 1 查单条详情
SELECT * FROM orders WHERE id = 100 LIMIT 1;
-- 只返回一行，字段多也没关系
```

## 🔥 扩展追问 2：代码生成器的陷阱

```java
// MyBatis-Plus 代码生成器默认生成
@Select("SELECT * FROM orders")
List<Order> selectList();

// 改进：生成时指定字段，或手写 SQL
@Select("SELECT ${OrderSummary.COLUMNS} FROM orders")
List<OrderSummary> selectList();
```

## 🔥 扩展追问 3：ORM 的延迟加载

```java
// JPA/Hibernate 的解决方案：实体拆分 + 延迟加载
@Entity
public class Order {
    @Id
    private Long id;
    
    private String orderNo;
    private BigDecimal amount;
    
    @OneToOne(fetch = FetchType.LAZY)  // 延迟加载
    @PrimaryKeyJoinColumn
    private OrderDetail detail;  // 大字段放这里
}

// 查询时只加载主表
Order order = orderRepository.findById(1L);  
// detail 字段是代理对象，未加载

// 访问时才加载
String remark = order.getDetail().getRemark();  // 触发第二次查询
```

**缺点：** N+1 查询问题，需谨慎使用。

## 一句话收尾

> `SELECT *` 是**懒惰的代价**，100 个字段的表会导致**网络、内存、缓存、索引**全面劣化；生产环境**必须明确字段列表**，配合**覆盖索引**和**垂直拆分**，列表用 DTO、详情才用全字段，从源头杜绝性能隐患。
