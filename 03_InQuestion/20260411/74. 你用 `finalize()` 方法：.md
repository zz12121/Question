74. 你用 `finalize()` 方法：
```java
protected void finalize() throws Throwable {
    // 清理资源
}
```
**问题：** `finalize()` 有什么问题？为什么不推荐使用？替代方案是什么？

---
## 先说结论

`finalize()` 是 Java 的**终结者方法**，用于对象被 GC 回收前执行清理逻辑。但它有**致命缺陷**：执行时机不确定、执行顺序不确定、执行线程是单线程串行、可能复活对象导致内存泄漏。Java 9 已标记为废弃，Java 18 正式移除。**替代方案：try-with-resources、`Cleaner`、或 `PhantomReference`。**

---

## 🔥 `finalize()` 的致命问题

### 1. 执行时机不确定

```java
public class Resource {
    @Override
    protected void finalize() throws Throwable {
        System.out.println("资源释放了");
        connection.close();
    }
}

// 使用
Resource r = new Resource();
r = null;  // 不再引用
System.gc();  // 建议 GC（还不一定执行）

// finalize() 什么时候执行？
// 答案：不知道！可能几秒后，可能几分钟后，可能永远不执行
```

**问题：资源长时间不释放，连接池耗尽、文件句柄泄漏。**

---

### 2. 执行顺序不确定

```java
class A {
    @Override
    protected void finalize() {
        System.out.println("A finalized");
    }
}

class B extends A {
    @Override
    protected void finalize() {
        System.out.println("B finalized");
        // 父类的 finalize 可能还没执行！
    }
}

// 子类 finalize 先执行，父类后执行？还是反过来？
// 答案：不确定，取决于 JVM 实现
```

---

### 3. 单线程串行执行

```java
// JVM 只有一个 Finalizer 线程，负责执行所有 finalize()
// 如果某个 finalize() 耗时或阻塞：

@Override
protected void finalize() throws Throwable {
    Thread.sleep(10000);  // 睡 10 秒
}

// 结果：其他对象的 finalize() 全部排队等待
// 队列堆积 → 对象无法回收 → OOM
```

**Finalizer 线程栈：**
```
"Finalizer" #3 daemon prio=8 os_prio=0
  java.lang.Thread.State: RUNNABLE
  at java.lang.ref.Finalizer.invokeFinalizeMethod(Native Method)
  at java.lang.ref.Finalizer.runFinalizer(Finalizer.java:100)
```

---

### 4. 对象复活（Resurrection）

```java
public class Zombie {
    static Zombie instance;  // 静态引用
    
    @Override
    protected void finalize() throws Throwable {
        System.out.println("finalize 执行");
        instance = this;  // 把自己又引用起来！
    }
}

// 使用
Zombie z = new Zombie();
z = null;
System.gc();

// finalize 执行后，instance 又指向了对象
// 对象复活了！但 finalize 不会再次执行
// 这个对象永远不会被真正回收，内存泄漏！
```

---

### 5. 性能开销

```
对象有 finalize() 时：
  1. 创建对象：额外注册到 Finalizer 队列
  2. GC 时：不直接回收，先加入 Finalizer 队列
  3. Finalizer 线程：逐个执行 finalize()
  4. 下次 GC：才真正回收对象

结果：对象至少多活一轮 GC，延迟回收，增加 GC 压力
```

---

## 🔥 替代方案

### 方案 1：try-with-resources（推荐）

```java
// ✅ Java 7+：自动关闭，确定性强
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql)) {
    
    ps.executeQuery();
    
} // 自动调用 close()，即使抛异常也会执行
```

**原理：编译器生成 finally 块，保证关闭。**

---

### 方案 2：`Cleaner`（Java 9+）

```java
// ✅ 替代 finalize() 的官方方案
public class NativeResource implements AutoCloseable {
    private static final Cleaner cleaner = Cleaner.create();
    
    private final State state;
    private final Cleaner.Cleanable cleanable;
    
    // 清理动作（不能引用外部对象！）
    private static class State implements Runnable {
        private long nativePtr;  // 堆外内存指针
        
        State(long nativePtr) {
            this.nativePtr = nativePtr;
        }
        
        @Override
        public void run() {
            // 清理堆外内存
            freeNative(nativePtr);
            nativePtr = 0;
            System.out.println("Native 资源已清理");
        }
    }
    
    public NativeResource(long nativePtr) {
        this.state = new State(nativePtr);
        // 注册清理器：对象被 GC 时触发
        this.cleanable = cleaner.register(this, state);
    }
    
    @Override
    public void close() {
        cleanable.clean();  // 手动触发清理
    }
}
```

**Cleaner 的优势：**
| 特性 | `finalize()` | `Cleaner` |
|---|---|---|
| 执行时机 | 不确定 | 对象不可达后触发 |
| 执行线程 | 单线程串行 | `ReferenceHandler` 线程，多 Cleaner 并行 |
| 对象复活 | 可能 | 不可能（State 不能引用外部对象） |
| 性能 | 差（延迟 GC） | 好（不影响 GC） |
| 手动触发 | 不能 | 可以（`clean()`） |

---

### 方案 3：`PhantomReference`（最底层）

```java
// ✅ 最灵活，但最复杂
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantom = new PhantomReference<>(obj, queue);

// 另一个线程等待回收通知
new Thread(() -> {
    try {
        Reference<?> ref = queue.remove();  // 阻塞等待
        // obj 已经被 GC，可以安全清理关联资源
        cleanup();
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
}).start();
```

**适用场景：** 需要精确控制清理时机的框架级代码。

## 🔥 扩展追问 1：`finalize()` 的执行流程

```
对象创建（有 finalize()）
    ↓
    注册到 Finalizer 引用队列
    ↓
对象不可达
    ↓
GC 发现：有 finalize() 且未执行过
    ↓
对象从堆移到 Finalizer 队列（对象又可达了！）
    ↓
Finalizer 线程执行 finalize()
    ↓
对象再次不可达
    ↓
下次 GC：真正回收对象（finalize 不会再次执行）
```

## 🔥 扩展追问 2：Java 18 移除 `finalize()`

```java
// Java 18 之前：能用，但有警告
@Override
protected void finalize() throws Throwable {
    super.finalize();  // 必须调用父类！
}

// Java 18 之后：编译错误！
// error: finalize() has been deprecated and marked for removal
```

**迁移路径：**
```
Java 8/11：finalize() 能用，但建议换 try-with-resources
Java 17：finalize() 标记为 deprecated for removal
Java 18+：finalize() 彻底移除，代码无法编译
```

## 一句话收尾

> `finalize()` 有**执行不确定、单线程瓶颈、对象复活**等致命缺陷，Java 18 已移除；资源清理优先用 **try-with-resources**（确定性强），堆外内存或复杂场景用 **`Cleaner`**（Java 9+），框架级需求用 **`PhantomReference`**，彻底告别 `finalize()`。
