64. 你用 `Future` 获取异步任务结果：
```java
ExecutorService executor = Executors.newSingleThreadExecutor();
Future<Result> future = executor.submit(() -> callRemote());
Result result = future.get();
```
**问题：** `get()` 方法有什么问题？如何设置超时？超时后如何取消任务？

---
## 先说结论

`future.get()` 是**阻塞式无限等待**，如果异步任务死循环、网络超时或线程池队列堆积，调用线程会永远卡住，最终导致线程池耗尽、服务雪崩。**必须设置超时 + 超时后主动取消任务释放资源。**

---

## 🔥 `get()` 的三大问题

| 问题 | 后果 | 场景 |
|---|---|---|
| **无限阻塞** | 调用线程永久 WAITING | 下游服务无响应、死循环 |
| **异常包装** | `ExecutionException` 包裹真实异常，需 `getCause()` | 排查问题多一层 unwrap |
| **无超时控制** | 无法优雅降级，只能干等 | 高并发下线程被占满 |

---

## 🔥 正确姿势：带超时的 `get(timeout, unit)`

```java
ExecutorService executor = Executors.newFixedThreadPool(10);

Future<Result> future = executor.submit(() -> callRemote());

try {
    // ✅ 最多等 3 秒，超时抛 TimeoutException
    Result result = future.get(3, TimeUnit.SECONDS);
    return result;
} catch (TimeoutException e) {
    // 超时处理：取消任务 + 降级
    future.cancel(true);  // true = 发送中断信号
    return fallbackResult();
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
    return fallbackResult();
} catch (ExecutionException e) {
    // 异步任务内部抛的异常，包装在这里
    Throwable cause = e.getCause();
    log.error("任务执行失败", cause);
    return fallbackResult();
} finally {
    executor.shutdown();  // 优雅关闭
}
```

---

## 🔥 `cancel(true)` 的深层机制

```java
future.cancel(true);  // 参数 mayInterruptIfRunning
```

| 参数值 | 行为 | 适用场景 |
|---|---|---|
| `false` | 只取消未开始的任务，运行中的不管 | 不需要中断，自然结束即可 |
| `true` | 发送 `Thread.interrupt()` 给执行线程 | 需要打断 I/O 或 sleep 等待 |

**关键：cancel(true) 只是发中断信号，任务必须响应中断才能停下来！**

### 可取消的任务写法

```java
public Result callRemote() {
    while (!Thread.currentThread().isInterrupted()) {  // ✅ 检查中断标志
        // 业务逻辑
        try {
            doSomething();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();  // ✅ 恢复标志，退出
            throw new RuntimeException("任务被取消", e);
        }
    }
    return result;
}
```

**常见坑：不检查中断标志的任务，cancel(true) 也停不下来**

```java
// ❌ 无法被取消的"毒任务"
Future<?> future = executor.submit(() -> {
    while (true) {
        // 不检查中断标志，死循环到底
    }
});
future.cancel(true);  // 发了中断，但任务无视，继续跑
```

## 🔥 超时后的资源清理

### 场景：任务在执行 I/O 操作时被取消

```java
public Result callRemote() {
    HttpClient client = HttpClient.newBuilder()
        .connectTimeout(Duration.ofSeconds(2))
        .build();
    
    try {
        return client.send(request, handler);
    } catch (IOException e) {
        if (Thread.currentThread().isInterrupted()) {
            // 被中断，清理资源
            client.close();
            throw new RuntimeException("请求被取消");
        }
        throw e;
    }
}
```

### 更健壮的封装：带强制中断的取消

```java
public class CancellableTask {
    
    public static <T> T executeWithTimeout(
            Callable<T> task, 
            long timeout, 
            TimeUnit unit) throws Exception {
        
        ExecutorService executor = Executors.newSingleThreadExecutor();
        Future<T> future = executor.submit(task);
        
        try {
            return future.get(timeout, unit);
        } catch (TimeoutException e) {
            future.cancel(true);
            // 额外：强制等待线程结束（防止毒任务）
            executor.shutdownNow();
            if (!executor.awaitTermination(1, TimeUnit.SECONDS)) {
                log.warn("任务未能在 1 秒内终止");
            }
            throw new TimeoutException("任务执行超时");
        }
    }
}
```

## 🔥 扩展追问 1：`cancel(false)` 什么时候用？

```java
// 场景：任务已提交但还没开始执行，想撤销它
Future<?> future = executor.submit(longRunningTask);

// 线程池队列里有 100 个任务，这个还没轮到
future.cancel(false);  // 从队列移除，不执行了，不会中断任何线程
```

**适用于**：任务还没开始，取消只是"撤销排队"，不需要打断。

## 🔥 扩展追问 2：`Future.get()` vs `CompletableFuture.get()` 超时处理

```java
// Future - 需要手动 try-catch + cancel
Future<Result> future = executor.submit(task);
try {
    return future.get(3, TimeUnit.SECONDS);
} catch (TimeoutException e) {
    future.cancel(true);
    return fallback;
}

// CompletableFuture - 更优雅的 API
CompletableFuture<Result> cf = CompletableFuture.supplyAsync(() -> task.call());
return cf.orTimeout(3, TimeUnit.SECONDS)           // 超时自动抛异常
         .exceptionally(ex -> fallback)             // 异常降级
         .join();
```

`CompletableFuture.orTimeout()` 内部自动处理取消，代码更简洁。

## 🔥 扩展追问 3：批量任务超时控制

```java
List<Future<Result>> futures = new ArrayList<>();
for (Task task : tasks) {
    futures.add(executor.submit(task));
}

// 等待所有任务，但只给 5 秒总时间
for (Future<Result> future : futures) {
    try {
        Result result = future.get(5, TimeUnit.SECONDS);  // ❌ 每个都等5秒，总时间会超
    } catch (TimeoutException e) {
        future.cancel(true);
    }
}

// ✅ 正确做法：用 invokeAll 控制总超时
List<Future<Result>> futures = executor.invokeAll(
    callables, 5, TimeUnit.SECONDS  // 整体5秒超时
);
// 超时的任务自动取消，返回的 Future.isCancelled() = true
```

## 🔥 扩展追问 4：超时后任务还在跑怎么办？

`cancel(true)` 只是发中断，如果任务不响应：

```java
// 终极方案：用独立的线程池 + shutdownNow
ExecutorService executor = Executors.newSingleThreadExecutor();
Future<?> future = executor.submit(task);

try {
    future.get(3, TimeUnit.SECONDS);
} catch (TimeoutException e) {
    future.cancel(true);
    executor.shutdownNow();  // 强制中断所有线程
    
    // 等待线程真正结束
    if (!executor.awaitTermination(5, TimeUnit.SECONDS)) {
        log.error("任务无法终止，可能存在毒代码");
    }
}
```

**生产建议**：对第三方不可控的任务，用独立线程池隔离，超时后连池一起销毁。

---

## 一句话收尾

> `future.get()` 必须带超时，超时后 `cancel(true)` 发中断信号，但**任务必须响应中断才能真正停止**——封装成工具类统一处理，或用 `CompletableFuture.orTimeout()` 更省心。
