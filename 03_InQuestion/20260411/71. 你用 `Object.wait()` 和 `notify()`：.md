71. 你用 `Object.wait()` 和 `notify()`：
```java
synchronized (lock) {
    while (!condition) {
        lock.wait();
    }
    doWork();
}
```
**问题：** 为什么用 `while` 而不是 `if`？`wait()` 会释放锁吗？

---
## 先说结论

必须用 `while` 而不是 `if`，因为 `wait()` 被唤醒后**条件可能已经不满足**（虚假唤醒、多生产者竞争）。`wait()` 会**完全释放锁**，进入等待队列；被 `notify()` 唤醒后，需要**重新竞争锁**，拿到锁后才能继续执行。

---

## 🔥 为什么用 `while` 而不是 `if`

### 错误写法：用 `if`

```java
synchronized (lock) {
    if (!condition) {      // ❌ 只检查一次
        lock.wait();
    }
    doWork();              // 危险：条件可能又不满足了！
}
```

### 正确写法：用 `while`

```java
synchronized (lock) {
    while (!condition) {   // ✅ 唤醒后再次检查
        lock.wait();
    }
    doWork();              // 确定条件满足才执行
}
```

---

## 🔥 必须用 `while` 的三大原因

### 1. 虚假唤醒（Spurious Wakeup）

```
操作系统层面：
  线程调用 wait() → 进入等待队列
  没有调用 notify()，但线程被唤醒了！（操作系统信号、JVM 实现等原因）
  
如果用 if：
  唤醒后直接执行 doWork()，但 condition 还是 false！
  
用 while：
  唤醒后回到循环条件，发现 !condition 还是 true，继续 wait()
```

**Java 规范明确允许虚假唤醒，程序必须处理。**

### 2. 多消费者竞争（Lost Wake-up）

```java
// 生产者
synchronized (lock) {
    condition = true;
    lock.notify();  // 只唤醒一个等待线程
}

// 消费者 A 和 B 都在 wait()
// notify() 唤醒了 A，但 A 还没抢到锁之前：
//   另一个生产者进来，把 condition 又改成 false
//   或者 A 执行完，把 condition 改回 false
// A 拿到锁，执行 doWork()
// B 被唤醒（但 notify 只唤了一个，B 还在等？不，这里假设有多个 notify）
```

**更典型的场景：多个 wait() 线程，一个 notifyAll()**

```java
// 两个消费者 C1、C2 都在 wait()
// 生产者 notifyAll()，两个都被唤醒

// C1 先抢到锁，执行 doWork()，把 condition 改回 false
// C1 释放锁

// C2 抢到锁，如果用 if，直接执行 doWork()，但 condition 已经是 false 了！
// 用 while，C2 检查 condition，发现 false，继续 wait()
```

### 3. 代码健壮性

```java
// 即使现在只有一个生产者和一个消费者
// 未来代码改动，变成多生产者，用 if 就埋了坑
// 用 while 是防御性编程，永远安全
```

## 🔥 `wait()` 的锁机制详解

```java
synchronized (lock) {    // 1. 获取锁
    while (!condition) {
        lock.wait();      // 2. 释放锁，进入等待队列
                          // 3. 线程阻塞，不消耗 CPU
                          
                          // ... 被 notify() 唤醒后 ...
                          
                          // 4. 重新竞争锁（可能阻塞）
                          // 5. 拿到锁，继续执行
    }
    doWork();
}                        // 6. 释放锁
```

**状态转换：**

```
获取锁 → wait() → 释放锁 → 等待队列
                            ↓ notify()
                            ↓
                         就绪队列 → 竞争锁 → 获取锁 → 继续执行
```

---

## 🔥 完整示例：生产者-消费者

```java
public class Buffer<T> {
    private final List<T> list = new ArrayList<>();
    private final int capacity;
    private final Object lock = new Object();

    public Buffer(int capacity) {
        this.capacity = capacity;
    }

    // 生产者
    public void produce(T item) throws InterruptedException {
        synchronized (lock) {
            // 满了就等（while！）
            while (list.size() == capacity) {
                lock.wait();
            }
            
            list.add(item);
            lock.notifyAll();  // 唤醒所有等待的消费者
        }
    }

    // 消费者
    public T consume() throws InterruptedException {
        synchronized (lock) {
            // 空了就等（while！）
            while (list.isEmpty()) {
                lock.wait();
            }
            
            T item = list.remove(0);
            lock.notifyAll();  // 唤醒所有等待的生产者
            return item;
        }
    }
}
```

**关键点：**
- 生产者和消费者都用 `while` 检查条件
- 都用 `notifyAll()`（如果用 `notify()`，可能唤醒同类线程，导致死锁）

## 🔥 扩展追问 1：`wait()` 和 `sleep()` 的区别

| 特性 | `wait()` | `sleep()` |
|---|---|---|
| **所属类** | `Object` | `Thread` |
| **锁行为** | **释放锁** | **不释放锁** |
| **调用前提** | 必须在同步块内 | 任意地方 |
| **唤醒方式** | `notify()` / `notifyAll()` / 中断 | 时间到 / 中断 |
| **用途** | 线程间协调 | 延迟执行 |

```java
synchronized (lock) {
    lock.wait();    // ✅ 释放锁，其他线程可以进来
}

synchronized (lock) {
    Thread.sleep(1000);  // ❌ 抱着锁睡觉，其他线程进不来！
}
```

---

## 🔥 扩展追问 2：`notify()` vs `notifyAll()`

| 方法 | 行为 | 风险 |
|---|---|---|
| `notify()` | 随机唤醒**一个**等待线程 | 可能唤醒同类线程，导致死锁 |
| `notifyAll()` | 唤醒**所有**等待线程 | 所有线程竞争锁，上下文切换开销 |

**生产环境推荐 `notifyAll()`**，除非确定只有一个等待线程。

```java
// 危险：只有一个消费者，但用了 notify()
// 如果有多个生产者也在 wait()，notify() 可能唤醒另一个生产者
// 两个生产者互相等待，死锁
```

## 🔥 扩展追问 3：现代替代方案

```java
// 古老的 wait/notify（容易写错）
synchronized (lock) {
    while (!condition) lock.wait();
    doWork();
    lock.notifyAll();
}

// Java 5+：用 Condition（更清晰，支持多个条件队列）
Lock lock = new ReentrantLock();
Condition notFull = lock.newCondition();
Condition notEmpty = lock.newCondition();

lock.lock();
try {
    while (queue.isFull()) {
        notFull.await();  // 等价于 wait，但语义更清晰
    }
    queue.add(item);
    notEmpty.signal();     // 等价于 notify
} finally {
    lock.unlock();
}

// Java 5+：直接用 BlockingQueue（最简单）
BlockingQueue<T> queue = new LinkedBlockingQueue<>(capacity);
queue.put(item);   // 满了自动阻塞，内部用 Condition 实现
queue.take();      // 空了自动阻塞
```

## 🔥 扩展追问 4：`wait()` 的超时版本

```java
// 避免永久等待
synchronized (lock) {
    long deadline = System.currentTimeMillis() + 5000;  // 最多等 5 秒
    while (!condition) {
        long waitTime = deadline - System.currentTimeMillis();
        if (waitTime <= 0) {
            throw new TimeoutException("等待超时");
        }
        lock.wait(waitTime);  // 带超时
    }
    doWork();
}
```

**或用 `Lock` 的 `await(timeout)`：**

```java
if (!condition.await(5, TimeUnit.SECONDS)) {
    throw new TimeoutException();
}
```

## 一句话收尾

> `wait()` 必须用 `while` 循环检查条件，防止**虚假唤醒**和**多线程竞争导致条件失效**；`wait()` 会**完全释放锁**，唤醒后需**重新竞争锁**。现代代码优先用 `BlockingQueue` 或 `Condition`，比 `wait/notify` 更安全易维护。
