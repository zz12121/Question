**问题：** 如果用户量很大（1000 万），有什么问题？如何优化？

---
## 先说结论

1000 万用户全量预热会导致**内存溢出**（Redis 和 JVM）、**启动时间过长**、**数据库被打爆**、**缓存雪崩风险**。优化方向：**分批加载**、**懒加载**、**异步预热**、**本地缓存分层**，以及**淘汰策略**避免全量缓存。

---

## 🔥 全量预热的问题

```java
@PostConstruct
public void init() {
    List<User> users = userDao.findAll();  // 1000 万条，JVM 内存爆炸
    for (User user : users) {
        redis.set("user:" + user.getId(), user);  // 1000 万次网络请求
    }
}
```

| 问题 | 后果 |
|---|---|
| **JVM OOM** | `findAll()` 加载 1000 万到内存，堆内存撑爆 |
| **数据库崩溃** | 大查询导致数据库 CPU/IO 飙升，影响线上业务 |
| **Redis 内存不足** | 1000 万条 × 平均 1KB = 10GB，可能超出内存限制 |
| **启动超时** | 预热耗时 10 分钟+，健康检查失败，服务无法上线 |
| **缓存雪崩** | 所有 key 同时过期，下次重启又重复问题 |

---

## 🔥 优化方案 1：分批加载（防止 OOM）

```java
@PostConstruct
public void init() {
    int pageSize = 1000;
    int pageNo = 0;
    
    while (true) {
        // 分页查询，避免一次性加载
        Page<User> page = userDao.findPage(PageRequest.of(pageNo, pageSize));
        List<User> users = page.getContent();
        
        if (users.isEmpty()) break;
        
        // 批量写入 Redis（Pipeline）
        redisPipelineSet(users);
        
        pageNo++;
        
        // 批次间休眠，防止 Redis 被打满
        if (pageNo % 100 == 0) {
            Thread.sleep(100);
        }
    }
}

private void redisPipelineSet(List<User> users) {
    redisTemplate.executePipelined((RedisCallback<Object>) connection -> {
        for (User user : users) {
            byte[] key = ("user:" + user.getId()).getBytes();
            byte[] value = serialize(user);
            connection.set(key, value);
        }
        return null;
    });
}
```

**效果：**
- 内存占用：从 10GB → 1000 条（几 MB）
- Redis 写入：从 1000 万次 RTT → 1 万次 Pipeline 批量

## 🔥 优化方案 2：懒加载（按需缓存）

```java
// ✅ 不需要启动时全量加载，用的时候再缓存
@Service
public class UserService {
    
    @Autowired
    private UserDao userDao;
    
    @Autowired
    private RedisTemplate<String, User> redis;
    
    public User getUser(Long id) {
        String key = "user:" + id;
        
        // 1. 先查缓存
        User user = redis.opsForValue().get(key);
        if (user != null) {
            return user;
        }
        
        // 2. 缓存未命中，查数据库
        user = userDao.findById(id).orElse(null);
        
        // 3. 写入缓存（设置过期时间）
        if (user != null) {
            redis.opsForValue().set(key, user, 30, TimeUnit.MINUTES);
        }
        
        return user;
    }
}
```

**优势：**
- 启动快，无预热时间
- 只缓存热点数据，节省内存
- 配合**缓存击穿防护**（布隆过滤器、互斥锁）

## 🔥 优化方案 3：异步预热（不阻塞启动）

```java
@Component
public class CacheWarmUp implements ApplicationRunner {
    
    @Autowired
    private UserDao userDao;
    
    @Autowired
    private ThreadPoolTaskExecutor executor;
    
    @Override
    public void run(ApplicationArguments args) {
        // 异步执行，不阻塞主线程启动
        executor.submit(this::warmUp);
    }
    
    private void warmUp() {
        // 只预热热点数据（如最近 7 天活跃用户）
        LocalDateTime sevenDaysAgo = LocalDateTime.now().minusDays(7);
        List<Long> hotUserIds = userDao.findActiveUserIds(sevenDaysAgo);
        
        // 分批加载
        Lists.partition(hotUserIds, 1000).forEach(batch -> {
            List<User> users = userDao.findAllById(batch);
            redisPipelineSet(users);
        });
    }
}
```

**关键：**
- 用 `ApplicationRunner` 替代 `@PostConstruct`，Spring 启动完成后再执行
- 只预热**热点数据**，不是全量
- 异步线程池，不阻塞服务注册和健康检查

## 🔥 优化方案 4：多级缓存（本地 + Redis）

```java
@Component
public class UserCache {
    
    // Caffeine 本地缓存（JVM 内，QPS 10万+）
    private final LoadingCache<Long, User> localCache = Caffeine.newBuilder()
        .maximumSize(10000)                    // 最多 1 万条
        .expireAfterWrite(10, TimeUnit.MINUTES) // 10 分钟过期
        .build(this::loadFromRedis);
    
    @Autowired
    private RedisTemplate<String, User> redis;
    
    @Autowired
    private UserDao userDao;
    
    public User get(Long id) {
        return localCache.get(id);  // 先查本地，未命中查 Redis，再未命中查 DB
    }
    
    private User loadFromRedis(Long id) {
        String key = "user:" + id;
        User user = redis.opsForValue().get(key);
        
        if (user == null) {
            user = userDao.findById(id).orElse(null);
            if (user != null) {
                redis.opsForValue().set(key, user, 30, TimeUnit.MINUTES);
            }
        }
        
        return user;
    }
}
```

**架构：**

```
请求 → Caffeine 本地缓存（1% 热点数据，10万 QPS）
   ↓ 未命中
   Redis（10% 活跃数据，1万 QPS）
   ↓ 未命中
   MySQL（全量数据）
```

// 多实例同时启动，避免重复预热
@Component
public class DistributedWarmUp {
    
    @Autowired
    private RedissonClient redisson;
    
    public void warmUp() {
        RLock lock = redisson.getLock("cache:warmup:lock");
        
        // 只有一个实例执行预热
        if (lock.tryLock()) {
            try {
                doWarmUp();
            } finally {
                lock.unlock();
            }
        } else {

        }
    }
}

## 一句话收尾

> 1000 万全量预热是**架构反模式**，会导致 OOM、启动超时、数据库被打爆；**不要全量预热**，改用**懒加载**（按需缓存）、**异步分批**（只预热热点）、**多级缓存**（Caffeine + Redis），配合**随机过期**和**淘汰策略**，让缓存随流量自然收敛到热点数据。
