## 结论先行

**缓存击穿 = 缓存中没有但数据库中有。** 大量并发请求同时发现缓存 miss，全部涌向数据库，数据库被打爆。解法是**加锁让只有一个请求去查库**，或**永不过期+异步重建**。

## 一、问题分析

### 代码在干什么？

```java
public User getUser(Long id) {
    String key = "user:" + id;
    String json = redis.get(key);
    if (json == null) {  // 缓存 miss
        User user = db.query(id);  // 查数据库
        redis.setex(key, 3600, JSON.toJSONString(user));
        return user;
    }
    return JSON.parseObject(json, User.class);
}
```

### 问题：缓存击穿

```
并发场景：1000 个请求同时查询 user:100

T1:  Redis.get("user:100") → null（缓存 miss）
T2:  Redis.get("user:100") → null（缓存 miss）
T3:  Redis.get("user:100") → null（缓存 miss）
...
T1000: Redis.get("user:100") → null（缓存 miss）

然后 1000 个请求同时执行：
db.query(100) → db.query(100) → db.query(100) → ...
                    ↓
        数据库压力 × 1000 = 数据库爆炸 💥
```

### 三种经典缓存问题对比

| 问题 | 现象 | 原因 |
|------|------|------|
| **缓存击穿** | 某个热点 key 突然过期，大量请求打到 DB | 热点 key 过期的瞬间并发 |
| **缓存雪崩** | 大量 key 同时过期，整个缓存层集体 miss | 大量 key 设置了相同的 TTL |
| **缓存穿透** | 缓存和数据库都没有，恶意/异常请求穿透了 | 查询不存在的数据 |

```
┌─────────────────────────────────────────────────────────────┐
│                    三个经典问题对比                           │
│                                                             │
│  缓存击穿：  ████████████████░░░░░░░░░░░░░░░░  有缓存，中间过期 │
│  缓存雪崩：  ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░  大量同时过期  │
│  缓存穿透：  ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░  缓存没有    │
│             ↓                                              │
│           DB  ←─── 大量请求同时去查                          │
└─────────────────────────────────────────────────────────────┘
```

## 二、解决方案

### 方案一：互斥锁（最经典）✅

**核心思想：只让一个人去查库，其他人等着拿结果**

```java
public User getUser(Long id) {
    String key = "user:" + id;
    String json = redis.get(key);
    
    if (json != null) {
        return JSON.parseObject(json, User.class);
    }
    
    // ========== 缓存 miss，需要查库 ==========
    // 获取锁，只让一个请求查库
    String lockKey = "lock:user:" + id;
    String lockValue = UUID.randomUUID().toString();
    
    // 尝试加锁（SETNX）
    Boolean acquired = redis.setIfAbsent(lockKey, lockValue, Duration.ofSeconds(10));
    
    if (Boolean.TRUE.equals(acquired)) {
        try {
            // 双重检查（拿到锁后再查一次缓存）
            json = redis.get(key);
            if (json != null) {
                return JSON.parseObject(json, User.class);
            }
            
            // 真的 miss，查数据库
            User user = db.query(id);
            if (user != null) {
                redis.setex(key, 3600, JSON.toJSONString(user));
            }
            return user;
        } finally {
            // 释放锁（用 Lua 脚本保证原子性）
            redis.del(lockKey);
        }
    } else {
        // 没拿到锁，等一下再试（自旋/睡眠重试）
        try {
            Thread.sleep(50);
        } catch (InterruptedException e) {
            // ignore
        }
        return getUser(id);  // 递归重试
    }
}
```

**Redis 分布式锁释放（Lua 脚本）：**

```java
// 保证锁只能被持有者释放（防误删）
String luaScript = 
    "if redis.call('get', KEYS[1]) == ARGV[1] then " +
    "    return redis.call('del', KEYS[1]) " +
    "else " +
    "    return 0 " +
    "end";

redis.eval(luaScript, 
    Collections.singletonList(lockKey), 
    Collections.singletonList(lockValue)
);
```

### 方案二：Singleflight（Go Style，高并发推荐）

**核心思想：同一个 key 的多个请求，合并成一个请求去查库**

```
原始：1000 个请求 → 1000 次数据库查询
Singleflight：1000 个请求 → 1 次数据库查询 → 返回给 1000 个人
```

**Java 实现（可以用 Guava RateLimiter 或自己写）：**

```java
// 简易版 Singleflight
public class Singleflight<V> {
    
    private final ConcurrentHashMap<String, Future<V>> running = new ConcurrentHashMap<>();

    @SuppressWarnings("unchecked")
    public V execute(String key, Callable<V> fn) throws Exception {
        Future<V> future = running.get(key);
        
        if (future == null) {
            // 没人查，我来查
            FutureTask<V> ft = new FutureTask<>(() -> {
                try {
                    return fn.call();
                } finally {
                    running.remove(key);  // 完成后移除
                }
            });
            
            future = running.putIfAbsent(key, ft);
            if (future == null) {
                // 说明我成功 put 了，我来执行
                ft.run();
                return ft.get();
            }
        }
        
        // 说明有人已经在查了，我等他的结果
        return future.get();
    }
}

// 使用
@Service
public class UserService {
    
    private final Singleflight<User> singleflight = new Singleflight<>();

    public User getUser(Long id) {
        String cacheKey = "user:" + id;
        String json = redis.get(cacheKey);
        
        if (json != null) {
            return JSON.parseObject(json, User.class);
        }
        
        try {
            // 用 Singleflight 合并并发请求
            User user = singleflight.execute(cacheKey, () -> {
                User u = db.query(id);
                if (u != null) {
                    redis.setex(cacheKey, 3600, JSON.toJSONString(u));
                }
                return u;
            });
            return user;
        } catch (Exception e) {
            throw new RuntimeException("查询失败", e);
        }
    }
}
```

### 方案三：永不过期 + 异步重建 ✅

**核心思想：不设 TTL，用后台线程异步更新缓存**

```java
// 核心思路：缓存不设过期时间，用逻辑过期字段控制

public class UserCache {
    public User user;           // 用户数据
    public long logicalExpire;   // 逻辑过期时间
}

public User getUser(Long id) {
    String key = "user:" + id;
    String json = redis.get(key);
    
    if (json == null) {
        // 缓存真的不存在，异步重建
        rebuildCache(id);
        return db.query(id);  // 先返回旧数据（如果有）
    }
    
    UserCache cache = JSON.parseObject(json, UserCache.class);
    
    // 检查逻辑过期
    if (cache.logicalExpire < System.currentTimeMillis()) {
        // 已过期，异步重建
        rebuildCache(id);
    }
    
    return cache.user;
}

// 异步重建（用线程池，不阻塞主线程）
@Async
public void rebuildCache(Long id) {
    String lockKey = "rebuild:user:" + id;
    
    // 加锁，只让一个线程重建
    if (redis.setIfAbsent(lockKey, "1", Duration.ofSeconds(10))) {
        try {
            User user = db.query(id);
            UserCache cache = new UserCache();
            cache.user = user;
            cache.logicalExpire = System.currentTimeMillis() + 30 * 60 * 1000;  // 30分钟
            
            redis.set("user:" + id, JSON.toJSONString(cache));
        } finally {
            redis.del(lockKey);
        }
    }
}
```

## 三、生产级完整实现

### 推荐：互斥锁 + 双重检查

```java
@Service
@Slf4j
public class UserCacheService {

    private final RedisTemplate<String, String> redisTemplate;
    private final UserMapper userMapper;
    private final RedissonClient redissonClient;

    private static final String CACHE_PREFIX = "user:";
    private static final String LOCK_PREFIX = "lock:user:";
    private static final long CACHE_TTL = 30 * 60;  // 30 分钟

    public User getUser(Long id) {
        String cacheKey = CACHE_PREFIX + id;
        
        // 1. 查缓存
        String json = redisTemplate.opsForValue().get(cacheKey);
        if (json != null) {
            log.debug("缓存命中, userId={}", id);
            return JSON.parseObject(json, User.class);
        }
        
        // 2. 缓存 miss，用分布式锁查库
        return getUserWithLock(id);
    }

    private User getUserWithLock(Long id) {
        String cacheKey = CACHE_PREFIX + id;
        String lockKey = LOCK_PREFIX + id;
        
        RLock lock = redissonClient.getLock(lockKey);
        
        try {
            // 尝试加锁，最长等 3 秒，锁自动 10 秒过期
            boolean acquired = lock.tryLock(3, 10, TimeUnit.SECONDS);
            
            if (acquired) {
                try {
                    // 双重检查（拿到锁后再查一次缓存）
                    String json = redisTemplate.opsForValue().get(cacheKey);
                    if (json != null) {
                        return JSON.parseObject(json, User.class);
                    }
                    
                    // 真的 miss，查数据库
                    User user = userMapper.selectById(id);
                    if (user != null) {
                        redisTemplate.opsForValue().set(
                            cacheKey, 
                            JSON.toJSONString(user),
                            Duration.ofMinutes(CACHE_TTL)
                        );
                    }
                    return user;
                    
                } finally {
                    lock.unlock();
                }
            } else {
                // 没拿到锁，短暂等待后递归重试
                Thread.sleep(30);
                return getUser(id);
            }
        } catch (InterruptedException e) {
            log.warn("获取锁被中断, userId={}", id);
            // 降级：直接查库
            return userMapper.selectById(id);
        }
    }
}
```

## 四、三种问题综合防护

| 问题 | 解法 | 核心思路 |
|------|------|---------|
| 缓存击穿 | 互斥锁 / Singleflight / 永不过期 | 只让一个人查库 |
| 缓存雪崩 | 随机 TTL / 永不过期 / 预热 | 分散过期时间 |
| 缓存穿透 | 布隆过滤器 / 空值缓存 | 拦截不存在的数据 |

```java
// 综合防护版本
public User getUser(Long id) {
    String cacheKey = CACHE_PREFIX + id;
    
    // 1. 布隆过滤器检查（防穿透）
    if (!bloomFilter.mightContain(id)) {
        return null;  // 肯定不存在，不查库
    }
    
    // 2. 查缓存
    String json = redisTemplate.opsForValue().get(cacheKey);
    if (json != null) {
        return JSON.parseObject(json, User.class);
    }
    
    // 3. 互斥锁查库（防击穿）
    return getUserWithLock(id);
}
```

## 五、面试加分话术

> **面试官追问：互斥锁有什么缺点？**

> 缺点是有一定的性能损耗（加锁解锁的网络往返），在高并发下会损失一部分性能。但保证了数据库安全，是可以接受的 trade-off。另一个缺点是如果加锁后业务执行时间过长，可能锁自动过期，这时候多个请求可能同时拿到锁。不过可以通过 Redisson 的看门狗（Watchdog）自动续期来解决。

> **面试官追问：Singleflight 和互斥锁的区别？**

> 互斥锁是"一个人查，其他人等"；Singleflight 是"一个人查，其他人复用结果"。Singleflight 更适合读多写少、缓存命中率高的场景，性能更好。但实现复杂度更高。Go 语言内置了 singleflight，Java 可以用 Guava Cache 的 `LoadingCache` 或自己实现。

---

## 一句话总结

**缓存击穿 = 热点 key 过期瞬间，并发全部打 DB。** 解法是**互斥锁让一个人去查库**，或**永不过期+异步重建**。生产环境推荐 **Redisson 分布式锁 + 双重检查**，简单可靠。
