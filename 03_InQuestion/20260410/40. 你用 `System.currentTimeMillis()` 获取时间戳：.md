40. 你用 `System.currentTimeMillis()` 获取时间戳：
```java
for (int i = 0; i < 1000000; i++) {
    long time = System.currentTimeMillis();
    // 使用 time
}
```

**问题：** 频繁调用有什么问题？有什么替代方案？

---
## 先说结论

频繁调用`System.currentTimeMillis()`会触发**系统调用**，从用户态切换到内核态，**性能开销大**（约100ns/次）。替代方案：**① 缓存时间戳**（定时更新，适合精度要求不高的场景）；**② 使用`System.nanoTime()`**（纯用户态，适合计时）；**③ 使用Caffeine等框架的时间轮**；**④ Java 9+使用`Clock`API**。

## 🔥 核心概念

### System.currentTimeMillis() 原理

```
┌─────────────────────────────────────────────────────────────┐
│  调用流程：                                                    │
│                                                              │
│  Java代码                                                     │
│     ↓                                                        │
│  System.currentTimeMillis()                                   │
│     ↓                                                        │
│  JVM native方法                                               │
│     ↓                                                        │
│  系统调用 gettimeofday() / clock_gettime()                    │
│     ↓                                                        │
│  内核态读取硬件时钟（RTC/HPET/TSC）                           │
│     ↓                                                        │
│  返回结果                                                     │
│                                                              │
│  问题：每次调用都涉及用户态↔内核态切换，开销较大                 │
└─────────────────────────────────────────────────────────────┘
```

### 性能对比

| 方法 | 耗时 | 精度 | 适用场景 |
|------|------|------|---------|
| `System.currentTimeMillis()` | ~100ns | 毫秒 | 获取当前时间 |
| `System.nanoTime()` | ~20ns | 纳秒 | 计时、间隔测量 |
| 缓存时间戳 | ~1ns | 毫秒级 | 高并发时间戳获取 |
| `Instant.now()` (Java 8) | ~150ns | 毫秒 | 新时间API |

## 🔥 详细展开：优化方案

### 方案1：缓存时间戳（高并发场景）

```java
@Component
public class CachedTime {
    
    private volatile long currentTimeMillis;
    
    @PostConstruct
    public void init() {
        // 定时更新，每10ms更新一次
        ScheduledExecutorService scheduler = 
            Executors.newSingleThreadScheduledExecutor();
        scheduler.scheduleAtFixedRate(() -> {
            currentTimeMillis = System.currentTimeMillis();
        }, 0, 10, TimeUnit.MILLISECONDS);
    }
    
    /**
     * 获取缓存的时间戳（精度10ms）
     */
    public long currentTimeMillis() {
        return currentTimeMillis;
    }
}

// 使用
@Service
public class OrderService {
    @Autowired
    private CachedTime cachedTime;
    
    public void createOrder() {
        long time = cachedTime.currentTimeMillis();  // ~1ns，无系统调用
        // ...
    }
}
```

### 方案2：ThreadLocal缓存（更精确）

```java
public class ThreadLocalCachedTime {
    
    // 每个线程缓存自己的时间戳
    private static final ThreadLocal<TimeCache> CACHE = 
        ThreadLocal.withInitial(TimeCache::new);
    
    private static class TimeCache {
        long time;
        long lastUpdate;
        
        long get() {
            long now = System.nanoTime();
            // 每1ms更新一次（用nanoTime判断，避免递归）
            if (now - lastUpdate > 1_000_000) {
                time = System.currentTimeMillis();
                lastUpdate = now;
            }
            return time;
        }
    }
    
    public static long currentTimeMillis() {
        return CACHE.get().get();
    }
}
```

```java
public class ClockService {
    
    // 可替换的Clock，测试时可注入固定时间
    private final Clock clock;

    }
    
    public Instant now() {

    }
    
    public long millis() {
        return clock.millis();
    }
}
```

```java
@Component
public class CaffeineCachedTime {
    
    // Caffeine内部使用缓存的时间戳
    private final Cache<String, String> cache = Caffeine.newBuilder()
        .expireAfterWrite(1, TimeUnit.SECONDS)
        .build();
    
    /**
     * Caffeine使用类似方案缓存时间戳
     * 内部定时更新，外部读取无锁
     */
    public long currentTimeMillis() {
        // Caffeine的Ticker接口提供纳秒级时间
        return cache.policy().expireAfterWrite()
            .map(policy -> policy.getExpiresAfter(TimeUnit.MILLISECONDS))
            .orElse(0L);
    }
}
```

## 🔥 扩展追问 3：适用场景选择

| 场景 | 推荐方案 | 原因 |
|------|---------|------|

| 缓存过期判断 | `currentTimeMillis()` | 精度要求高 |

## 一句话收尾

频繁调用`System.currentTimeMillis()`的性能瓶颈是**系统调用开销**，优化方案是**缓存时间戳**（定时更新，精度换性能）、**`nanoTime()`计时**（纯用户态）、或使用**Java 8 Clock API**。高并发场景下缓存时间戳可提升**10-20倍**性能，但要注意**精度损失**是否在业务可接受范围内。
