31. 你用 `synchronized` 实现了一个单例模式：
```java
public class Singleton {
    private static Singleton instance;
    public static synchronized Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}
```
**问题：** 高并发下有什么问题？如何优化？

---
## 先说结论

当前代码的问题是**"锁粒度太粗"**：`static synchronized`锁的是整个Class对象，导致每次调用`getInstance()`都串行，高并发下性能极差。优化方案是**"双重检查锁定（DCL）+ volatile"**，将锁粒度缩小到仅初始化阶段，实例创建后无锁访问。

---

## 🔥 核心概念

### 问题分析

```
┌─────────────────────────────────────────────────────────────┐
│  原代码问题：                                                 │
│  public static synchronized Singleton getInstance()           │
│         ↑                                                    │
│    锁的是Singleton.class，锁粒度是整个方法                     │
│                                                              │
│  高并发场景：                                                 │
│  线程A ──► 获取锁 ──► 检查null ──► 创建实例 ──► 释放锁        │
│  线程B ──► 等待锁 ──► 等待 ──► 等待 ──► 获取锁 ──► 直接返回   │
│  线程C ──► 等待锁 ──► 等待 ──► 等待 ──► 等待 ──► 获取锁...    │
│                                                              │
│  即使instance已创建，后续所有线程仍需排队获取锁，性能瓶颈！      │
└─────────────────────────────────────────────────────────────┘
```

## 🔥 详细展开：优化方案

### 方案1：双重检查锁定（DCL）- 推荐

```java
public class Singleton {
    // volatile禁止指令重排序，保证可见性
    private static volatile Singleton instance;
    
    public static Singleton getInstance() {
        // 第一次检查（无锁）
        if (instance == null) {
            // 只有未初始化时才加锁
            synchronized (Singleton.class) {
                // 第二次检查（防止多个线程同时通过第一次检查）
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
    
    private Singleton() {}
}
```

**为什么需要volatile？**

```java
// instance = new Singleton() 实际分三步：
// 1. 分配内存空间
// 2. 初始化对象
// 3. 将引用指向内存地址

// 无volatile时，可能发生指令重排序：1 → 3 → 2
// 其他线程可能拿到未完全初始化的对象！

// volatile保证：
// - 可见性：一个线程修改，其他线程立即可见
// - 有序性：禁止指令重排序，保证2在3之前执行
```

### 方案2：静态内部类（最优雅）

```java
public class Singleton {
    
    // 静态内部类，延迟加载
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }
    
    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
    
    private Singleton() {}
}
```

**原理：**
- JVM保证类初始化是线程安全的（`<clinit>`方法加锁）
- 静态内部类只有被引用时才加载，实现延迟加载
- 无锁，无线程竞争，性能最好

### 方案3：枚举（最简洁，防反射攻击）

```java
public enum Singleton {
    INSTANCE;
    
    // 业务方法
    public void doSomething() {
        // ...
    }
}

// 使用
Singleton.INSTANCE.doSomething();
```

**优点：**
- 线程安全（枚举实例由JVM保证唯一）
- 序列化安全（自动处理）
- 防反射攻击（枚举构造器特殊处理）

## 🔥 扩展追问 2：反射攻击与防御

```java
// 问题：反射可以破坏单例
public class ReflectionAttack {
    public static void main(String[] args) throws Exception {
        Singleton instance1 = Singleton.getInstance();
        
        // 反射创建新实例
        Constructor<Singleton> constructor = Singleton.class.getDeclaredConstructor();
        constructor.setAccessible(true);
        Singleton instance2 = constructor.newInstance();
        
        System.out.println(instance1 == instance2);  // false！单例被破坏
    }
}

// 防御：在构造器中加判断
public class Singleton {
    private static volatile Singleton instance;
    
    private Singleton() {
        // 防止反射创建多个实例
        if (instance != null) {
            throw new RuntimeException("Singleton already exists!");
        }
    }
    
    // ... getInstance()
}
```

```java
public class SerializationAttack {

        Singleton instance1 = Singleton.getInstance();

        System.out.println(instance1 == instance2);  // false！
    }

// 防御：实现readResolve方法
public class Singleton implements Serializable {
    private static final long serialVersionUID = 1L;
    
    // ... 其他代码
    
    // 反序列化时替换为单例实例
    protected Object readResolve() {
        return getInstance();
    }
}
```

## 一句话收尾

`synchronized`单例的性能问题在于**锁粒度太粗**，优化核心是**"延迟加锁+缩小锁范围"**：用DCL+volatile将锁缩小到初始化阶段，或用静态内部类利用JVM类加载机制实现无锁线程安全。记住**"volatile防重排序，二次检查防重复创建"**。
