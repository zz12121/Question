37. 你用 `ThreadLocal` 存储用户信息：
```java
public class UserContext {
    private static final ThreadLocal<User> currentUser = new ThreadLocal<>();
    
    public static void set(User user) {
        currentUser.set(user);
    }
    
    public static User get() {
        return currentUser.get();
    }
}
```
**问题：** 如果在线程池中使用，会出现什么问题？如何正确清理？

---
## 先说结论

线程池中使用`ThreadLocal`会导致**"数据污染"**（线程复用导致旧数据残留）和**"内存泄漏"**（ThreadLocalMap的Entry无法回收）。解决方案：**① 每次使用完在`finally`中`remove()`**；**② 使用`TransmittableThreadLocal`（TTL）解决父子线程传递**；**③ 使用`InheritableThreadLocal`**（仅适用新建线程）；**④ 改用请求上下文传递**（如方法参数、MDC）。

## 🔥 核心概念

### ThreadLocal原理

```
┌─────────────────────────────────────────────────────────────┐
│  ThreadLocal结构：                                            │
│                                                              │
│  Thread ──► ThreadLocalMap ──► Entry[]                      │
│                │                                              │
│                ├── Entry[0]: key=ThreadLocal@1, value=user1  │
│                ├── Entry[1]: key=ThreadLocal@2, value=user2  │
│                └── ...                                       │
│                                                              │
│  注意：Entry是WeakReference，key是弱引用，value是强引用       │
│  如果ThreadLocal被回收，key变为null，但value还在！            │
│  → 内存泄漏风险                                               │
└─────────────────────────────────────────────────────────────┘
```

### 线程池场景的问题

```
┌─────────────────────────────────────────────────────────────┐
│  问题1：数据污染（线程复用导致）                               │
│                                                              │
│  请求A：Thread-1 set(userA) ──► 处理 ──► 线程归还线程池        │
│  请求B：Thread-1（复用）get() ──► 返回userA！（数据污染）      │
│                                                              │
│  结果：请求B拿到了请求A的用户信息，严重的安全问题！              │
├─────────────────────────────────────────────────────────────┤
│  问题2：内存泄漏                                               │
│                                                              │
│  线程池线程长期存活 ──► ThreadLocalMap一直存在                 │
│  如果remove()没被调用 ──► value一直被引用无法回收               │
│  大量请求 ──► ThreadLocalMap膨胀 ──► OOM                      │
└─────────────────────────────────────────────────────────────┘
```

### 数据污染场景

## 🔥 详细展开：解决方案

### 方案1：finally中remove()（最基础）

```java
@Component
public class UserService {
    
    @Autowired
    private ExecutorService executor;
    
    public void process(User user) {
        executor.submit(() -> {
            try {
                UserContext.set(user);
                // 业务逻辑...
                doBusiness();
            } finally {
                // 必须清理！
                UserContext.remove();
            }
        });
    }
}

// UserContext增加remove方法
public class UserContext {
    private static final ThreadLocal<User> currentUser = new ThreadLocal<>();
    
    public static void set(User user) {
        currentUser.set(user);
    }
    
    public static User get() {
        return currentUser.get();
    }
    
    public static void remove() {
        currentUser.remove();  // 关键！
    }
}
```

### 方案2：封装Runnable/Callable（自动化清理）

```java
@Component
public class CleanableTaskWrapper {
    
    /**
     * 包装Runnable，自动清理ThreadLocal
     */
    public static Runnable wrap(Runnable task) {
        return () -> {
            try {
                task.run();
            } finally {
                // 清理所有ThreadLocal
                UserContext.remove();
                TransactionContext.remove();
                RequestContext.remove();
                // ...
            }
        };
    }
    
    /**
     * 使用
     */
    public void submitCleanableTask(Runnable task) {
        executor.submit(wrap(task));
    }
}
```

### 方案3：TransmittableThreadLocal（TTL）- 推荐

```java
// 依赖：com.alibaba:transmittable-thread-local

@Component
public class TtlUserContext {
    // 使用TTL替代ThreadLocal
    private static final TransmittableThreadLocal<User> context = 
        new TransmittableThreadLocal<>();
    
    public static void set(User user) {
        context.set(user);
    }
    
    public static User get() {
        return context.get();
    }
    
    public static void remove() {
        context.remove();
    }
}

// 线程池包装（必须！）
@Configuration
public class ThreadPoolConfig {
    
    @Bean
    public ExecutorService ttlExecutor() {
        ExecutorService executor = Executors.newFixedThreadPool(10);
        // 包装为TTL线程池，自动传递ThreadLocal
        return TtlExecutors.getTtlExecutorService(executor);
    }
}

// 使用：父子线程自动传递，且线程复用时自动清理
@Service
public class OrderService {
    
    @Autowired
    private ExecutorService executor;  // TTL包装后的线程池
    
    public void createOrder() {
        TtlUserContext.set(new User("Alice"));
        
        // 提交到线程池，User自动传递
        executor.submit(() -> {
            User user = TtlUserContext.get();  // Alice
            // 处理...
        });
        
        // 主线程继续...
    }
}
```

**TTL原理：**

```
┌─────────────────────────────────────────────────────────────┐
│  提交任务时：                                                 │
│  1. 捕获当前线程的所有TTL值                                    │
│  2. 包装到Runnable中                                          │
│                                                              │
│  执行任务时：                                                 │
│  1. 从包装中恢复TTL值到当前线程                                │
│  2. 执行原始任务                                              │
│  3. finally中清理TTL（恢复线程干净状态）                       │
└─────────────────────────────────────────────────────────────┘
```

## 一句话收尾

线程池中使用`ThreadLocal`的核心问题是**"线程复用导致数据污染"**和**"未清理导致内存泄漏"**。解决方案：**`finally`中`remove()`**是基础，**`TransmittableThreadLocal`**解决父子线程传递和自动清理，生产环境推荐**显式参数传递**替代隐式ThreadLocal，避免维护噩梦。
