48. 你的查询用到了 OR 条件：
```sql
SELECT * FROM orders WHERE user_id = 100 OR status = 1;
```
**问题：** 这条 SQL 索引会怎么用？如何改写提高性能？

---
## 结论先行

**OR 条件是索引的"天敌"** —— 如果 OR 的两边字段没有同时建复合索引，MySQL 可能不走索引变成全表扫描。优化思路是**把 OR 拆成 UNION 或改写成 UNION ALL/改用 IN**，让每路都走索引。

---

## 一、OR 条件的索引问题

### 先看表结构和索引

```sql
-- 假设 orders 表有这些索引
SHOW INDEX FROM orders;

-- Index 1: idx_user_id (user_id)
-- Index 2: idx_status (status)
```

### MySQL 对 OR 的处理

```sql
SELECT * FROM orders WHERE user_id = 100 OR status = 1;
```

**MySQL 执行这条 SQL 时，会发生什么？**

```
┌──────────────────────────────────────────────────────────────┐
│  MySQL 优化器判断：                                          │
│                                                              │
│  user_id = 100  ──▶  走 idx_user_id ──▶  命中 1000 行       │
│                                                              │
│  status = 1    ───▶  走 idx_status   ──▶  命中 5000 行      │
│                                                              │
│  OR 条件要合并两个结果集 ──▶  可能要回表取 * ──▶  扫描成本大   │
└──────────────────────────────────────────────────────────────┘
```

### 什么时候 OR 会走索引？

**条件：OR 的所有字段都有单独索引，且 MySQL 估算后觉得值得**

```sql
-- 假设有这些索引
CREATE INDEX idx_user_id ON orders(user_id);
CREATE INDEX idx_status ON orders(status);

-- ✅ user_id = 100 走 idx_user_id，status = 1 走 idx_status
SELECT * FROM orders WHERE user_id = 100 OR status = 1;
```

**什么时候 OR 不走索引（变成全表扫描）？**

| 场景 | 结果 |
|------|------|
| OR 两侧字段都没有索引 | ❌ 全表扫描 |
| OR 两侧只有一个有索引 | ⚠️ 只走一个索引，另一个条件退化 |
| OR 两侧都有索引，但数据量很大 | ⚠️ MySQL 估算后觉得全表扫描更快 |
| OR 连接了不同类型的字段 | ❌ 索引失效 |

### 关键问题：OR 为什么不走索引？

**根本原因：OR 让索引"断链"**

```sql
-- 假设 user_id = 100 查到 1000 行，status = 1 查到 5000 行
-- MySQL 要取并集 = 最多 6000 行
-- 这 6000 行每行都要回表取 SELECT * 的字段

-- 如果全表只有 10000 行，MySQL 觉得"回表成本 > 全表扫描成本"
-- 结论：干脆全表扫描算了，不用索引
```

## 二、如何改写提高性能？

### 方案一：拆成 UNION（最常用）✅

**核心思想：把 OR 拆成两条查询，用 UNION 合并结果**

```sql
-- ❌ 原始写法（可能全表扫描）
SELECT * FROM orders WHERE user_id = 100 OR status = 1;

-- ✅ 改写：拆成 UNION，每路都走索引
(SELECT * FROM orders WHERE user_id = 100)
UNION
(SELECT * FROM orders WHERE status = 1);
```

**原理：**

```
原写法：查 A OR 查 B，MySQL 纠结要不要合并
       └──▶ 扫描全表

改写后：查 A ──▶ 走 idx_user_id
        UNION
        查 B ──▶ 走 idx_status
       └──▶ 每路都走索引，再合并结果
```

**UNION vs UNION ALL：**

| 类型 | 去重 | 性能 |
|------|------|------|
| UNION | ✅ 自动去重 | 略慢（有去重开销） |
| UNION ALL | ❌ 不去重 | 更快（直接合并） |

```sql
-- ✅ 如果确认不会重复，用 UNION ALL 更快
(SELECT * FROM orders WHERE user_id = 100)
UNION ALL
(SELECT * FROM orders WHERE status = 1);
```

### 方案二：改写成 IN（字段相同时）

**核心思想：当 OR 的是同一个字段时，IN 比 OR 更清晰**

```sql
-- ❌ 原始 OR
SELECT * FROM orders WHERE status = 1 OR status = 2 OR status = 3;

-- ✅ 改写成 IN
SELECT * FROM orders WHERE status IN (1, 2, 3);
```

**IN vs OR 性能对比：**

| 写法 | 索引使用 | 备注 |
|------|---------|------|
| status = 1 OR status = 2 OR status = 3 | ✅ 可以走索引 | MySQL 会改写成 IN 优化 |
| status IN (1, 2, 3) | ✅ 走索引 | 写法更简洁 |

### 方案三：复合索引覆盖（最优解）

**核心思想：把 OR 涉及的所有字段合成一个复合索引**

```sql
-- ❌ 分别建索引（OR 效率差）
CREATE INDEX idx_user_id ON orders(user_id);
CREATE INDEX idx_status ON orders(status);

-- ✅ 复合索引（覆盖所有 OR 条件）
CREATE INDEX idx_user_status ON orders(user_id, status);
```

**查询改写：**

```sql
-- 复合索引 (user_id, status) 可以覆盖这条查询
SELECT user_id, status, id, create_time FROM orders 
WHERE user_id = 100 OR status = 1;

-- 如果 SELECT * 回表多，优化成只查索引列
SELECT user_id, status, id, create_time FROM orders 
WHERE user_id = 100 
UNION ALL
SELECT user_id, status, id, create_time FROM orders 
WHERE status = 1;
```

### 方案四：强制索引提示（HINT）

**核心思想：告诉 MySQL 用哪个索引（不推荐但有效）**

```sql
-- 强制走 user_id 索引
SELECT * FROM orders FORCE INDEX (idx_user_id) 
WHERE user_id = 100 OR status = 1;
```

> **面试加分：** HINT 是"人工干预"，正常情况下让优化器自己判断。只有在确认优化器判断错误时才用。

## 三、完整优化示例

### 场景：订单列表查询

```sql
-- ❌ 低效写法
SELECT * FROM orders 
WHERE user_id = 100 OR status = 'PAID' OR create_time > '2024-01-01';

-- ✅ 优化方案1：拆成 UNION ALL
(SELECT * FROM orders WHERE user_id = 100)
UNION ALL
(SELECT * FROM orders WHERE status = 'PAID')
UNION ALL
(SELECT * FROM orders WHERE create_time > '2024-01-01');

-- ✅ 优化方案2：建立覆盖索引
CREATE INDEX idx_user_status_time ON orders(user_id, status, create_time);

-- ✅ 优化方案3：如果可以，用 AND 替代 OR
SELECT * FROM orders 
WHERE user_id = 100 
  AND (status = 'PAID' OR create_time > '2024-01-01');
```

### Java 代码实现

```java
// MyBatis-Plus 改写 OR
@Component
public class OrderQueryService {

    @Autowired
    private OrderMapper orderMapper;

    public List<Order> queryOrdersOptimized(Long userId, List<String> statuses) {
        LambdaQueryWrapper<Order> wrapper = new LambdaQueryWrapper<>();
        
        // 方案1：用 IN 改写同一字段的 OR
        if (statuses != null && !statuses.isEmpty()) {
            wrapper.eq(Order::getUserId, userId)
                   .in(Order::getStatus, statuses);  // status IN (...)
        }
        
        // 方案2：UNION ALL（不同字段的 OR）
        List<Order> result = new ArrayList<>();
        
        // 第一个条件
        result.addAll(orderMapper.selectList(
            new LambdaQueryWrapper<Order>()
                .eq(Order::getUserId, userId)
        ));
        
        // 第二个条件（UNION ALL）
        result.addAll(orderMapper.selectList(
            new LambdaQueryWrapper<Order>()
                .eq(Order::getStatus, "PAID")
        ));
        
        return result;
    }
}
```

## 四、索引设计最佳实践

### 为 OR 场景设计索引的原则

```
┌─────────────────────────────────────────────────────────────┐
│  OR 条件索引设计：                                           │
│                                                             │
│  1. 高频查询条件优先建索引                                    │
│     └── user_id 每天查 10 万次 ──▶ 建索引                    │
│     └── status   每天查 5 万次  ──▶ 建索引                    │
│                                                             │
│  2. OR 字段差异大时，拆 UNION 更优                            │
│     └── A 条件命中 10 行，B 条件命中 10 万行                  │
│     └── 拆成两个查询，每路按需查询                             │
│                                                             │
│  3. 避免 SELECT *，减少回表                                  │
│     └── 只查需要的字段，索引覆盖住                            │
└─────────────────────────────────────────────────────────────┘
```

### EXPLAIN 分析

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 100 OR status = 1;

-- 关键字段检查：
-- type: ALL = 全表扫描，ref/range = 走索引
-- rows: 估算扫描行数
-- Extra: Using where = 需要回表过滤
--        Using index = 覆盖索引，不需要回表
```

## 五、面试加分话术

> **面试官追问：UNION 和 UNION ALL 的区别？**

> UNION 会自动去重（相当于 SELECT DISTINCT），内部需要排序和合并操作。UNION ALL 直接追加结果，不去重，性能更好。如果确认两个查询不会有重复行（比如条件不重叠），就用 UNION ALL。

> **面试官追问：OR 条件和 AND 条件的索引有什么区别？**

> AND 条件可以"索引组合加速"：比如 (A, B) 复合索引，查询 WHERE A = x AND B = y 可以同时用上 A 和 B 两个列。而 OR 条件是"独立加速"：两边各自走自己的索引，最后合并。所以 AND 更适合复合索引，OR 更适合独立单列索引。

---

## 一句话总结

**OR 是索引的"坑"** —— 拆分 UNION 让每路独立走索引，或建覆盖索引减少回表，是优化 OR 查询的核心思路。
