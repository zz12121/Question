51. 你的缓存更新代码：
```java
public void updateUser(User user) {
    db.update(user);
    redis.del("user:" + user.getId());
}
```
**问题：** 这段代码有什么问题？如果删除缓存失败会怎样？如何优化？

---
## 结论先行

**这段代码的问题：删缓存失败了，数据库和缓存永远不一致。** 核心解法是**删除失败后异步重试（消息队列 / 延迟补偿），保证最终一致性**。

---

## 一、问题分析

### 代码流程

```java
public void updateUser(User user) {
    db.update(user);      // 1. 更新数据库 ✅
    redis.del("user:1");  // 2. 删除缓存
}
```

**两种失败场景：**

```
场景 A：先更库，后删缓存

T1: db.update(user)      →  数据库更新成功
T2: redis.del(...)       →  💥 删除失败（Redis 超时/异常）
T3: 后续请求              →  读到的还是旧缓存数据
T4: 缓存 TTL 过期         →  才能读到新数据
结果：短时间内数据不一致
```

```
场景 B：先删缓存，后更库

T1: redis.del(...)       →  缓存删除成功
T2: T2 请求来，读缓存     →  缓存 miss，查数据库读到旧值，写回缓存
T3: db.update(user)      →  数据库更新成新值
T4: 结果                  →  缓存是旧值，数据库是新值 💥
```

### 为什么删除会失败？

| 失败原因 | 频率 |
|---------|------|
| Redis 网络抖动 | 偶发 |
| Redis 连接池耗尽 | 高并发时 |
| Redis 主从切换 | 偶发 |
| 代码 Bug / Redis 满载 | 低 |

## 二、经典问题：先更库还是先删缓存？

### 业界最佳实践：Cache Aside（旁路缓存）

```
┌─────────────────────────────────────────────────────────────┐
│                     Cache Aside 模式                        │
│                                                             │
│  读操作：                                                    │
│    读缓存 ──▶ 命中 ──▶ 返回                                  │
│       │                                                   │
│       ├── 未命中 ──▶ 查数据库 ──▶ 写缓存 ──▶ 返回            │
│                                                             │
│  写操作（先删缓存，再更库）：                                  │
│    删缓存 ──▶ 更数据库                                       │
│    （即使删缓存失败，下次缓存过期后会自动从数据库加载新值）       │
└─────────────────────────────────────────────────────────────┘
```

### 为什么推荐"先删缓存，再更库"？

```
如果是"先更库，后删缓存"：

T1: T1 更新数据库 age=20
T2: T2 查询，缓存miss，查数据库读到 age=20，写入缓存
T3: T1 删除缓存（此时 T2 已经把旧值写回缓存了）
T4: T3 查询，缓存命中，返回 age=20（旧值）💥

这种情况叫"并发双写"问题，数据在 T2-T3 之间不一致。
```

```
如果是"先删缓存，再更库"：

T1: T1 删除缓存
T2: T2 查询，缓存miss，查数据库读到旧值（还没更新），尝试写缓存但缓存不存在
T3: T1 更新数据库
T4: T3 查询，缓存miss，查数据库读到新值，写入缓存
结果：最多读一次旧值，不会长期不一致
```

**结论：推荐"先删缓存，再更库"，更不容易出现长期不一致。**

## 三、删除失败的解法

### 方案一：延迟双删（简单有效）✅

**核心思想：删一次不够，那就删两次，中间等一下**

```java
public void updateUser(User user) {
    // 1. 先删缓存
    redis.del("user:" + user.getId());
    
    // 2. 更新数据库
    db.update(user);
    
    // 3. 延迟 500ms 再删一次（等 T2 把旧值写回，然后删掉）
    try {
        Thread.sleep(500);  // ⚠️ 实际用延迟队列，不用 Thread.sleep
    } catch (InterruptedException e) {
        // ignore
    }
    redis.del("user:" + user.getId());
}
```

**执行时序：**

```
T1: delete(user:1)       →  缓存删除
T2: 查询，读到旧值（数据库还是旧值），准备写缓存
T3: update db age=20      →  数据库更新
T4: delete(user:1)        →  再次删除（把 T2 刚写回的旧值删掉）
T5: T5 查询，缓存miss，读到新值，写入缓存
结果：一致性 ✅
```

### 方案二：消息队列异步重删（生产推荐）✅

**核心思想：删除失败后，扔进消息队列异步重试**

```java
public void updateUser(User user) {
    // 1. 删除缓存
    redis.del("user:" + user.getId());
    
    // 2. 缓存删除失败，发送延迟消息重试
    if (failed) {
        delayQueue.send(new CacheInvalidateMsg(
            "user:" + user.getId(),
            System.currentTimeMillis() + 1000  // 1秒后重试
        ));
    }
    
    // 3. 更新数据库
    db.update(user);
}

// 延迟队列消费者
@RabbitListener
public void handleCacheInvalidate(CacheInvalidateMsg msg) {
    try {
        redis.del(msg.getKey());
    } catch (Exception e) {
        // 重试失败，扔进死信队列告警
        deadLetterQueue.send(msg);
        alertService.notify("缓存删除失败，请人工处理");
    }
}
```

### 方案三：订阅 MySQL binlog 变更（最优雅）✅

**核心思想：不依赖应用层删除，让 Canal/Debezium 监听 binlog 来删缓存**

```
┌─────────────────────────────────────────────────────────────┐
│                  Canal 监听 MySQL binlog                     │
│                                                             │
│  MySQL ───▶ Binlog ──▶ Canal ──▶ 删缓存 ──▶ MQ              │
│                                                             │
│  优势：                                                      │
│  1. 不影响业务线程性能                                        │
│  2. 数据库更新和缓存删除完全解耦                               │
│  3. 天然保证顺序性                                            │
└─────────────────────────────────────────────────────────────┘
```

**Canal 配置：**

```yaml
server:
  port: 11111
bind_address: 0.0.0.0

canals:
  demo:
    host: mysql-host
    port: 3306
    user: canal
    password: xxx

```

**代码实现：**

```java
@Component
@Slf4j
public class CanalListener {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    @CanalListener(destination = "example", schema = "db", table = "user")
    public void onUserUpdate(RowChange rowChange) {
        for (RowData rowData : rowChange.getRowDatas()) {
            if (rowChange.getEventType() == EventType.UPDATE) {
                // 获取更新后的 id
                Long userId = getColumnValue(rowData, "id");
                
                // 删除缓存
                try {
                    redisTemplate.delete("user:" + userId);
                    log.info("Canal 删除缓存成功, userId={}", userId);
                } catch (Exception e) {
                    log.error("Canal 删除缓存失败, userId={}", userId, e);
                    // 告警 + 人工处理
                    alertService.notify("Canal 删缓存失败");
                }
            }
        }
    }
}
```

## 四、生产级完整实现

### 方案：消息队列 + 重试 + 告警

```java
@Service
@Slf4j
public class UserService {

    @Autowired
    private UserMapper userMapper;
    
    @Autowired
    private RedisTemplate<String, User> redisTemplate;
    
    @Autowired
    private CacheInvalidateProducer invalidateProducer;

    // 更新用户
    public void updateUser(User user) {
        // 1. 先删缓存
        String cacheKey = "user:" + user.getId();
        try {
            redisTemplate.delete(cacheKey);
        } catch (Exception e) {
            log.error("删除缓存失败, key={}", cacheKey, e);
            // 2. 删除失败，发送消息到延迟队列重试
            invalidateProducer.send(cacheKey, user.getId());
        }
        
        // 3. 更新数据库
        userMapper.updateById(user);
        
        log.info("用户更新成功, userId={}", user.getId());
    }

    // 查询用户（Cache Aside 读）
    public User getUser(Long userId) {
        String cacheKey = "user:" + userId;
        
        // 1. 查缓存
        User cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            log.debug("缓存命中, userId={}", userId);
            return cached;
        }
        
        // 2. 缓存 miss，查数据库
        User user = userMapper.selectById(userId);
        if (user != null) {
            // 3. 写缓存（设置 TTL，防止雪崩）
            redisTemplate.opsForValue().set(
                cacheKey, 
                user, 
                Duration.ofMinutes(30 + new Random().nextInt(10))
            );
        }
        
        return user;
    }
}
```

### 延迟队列生产者

```java
@Component
@RequiredArgsConstructor
public class CacheInvalidateProducer {

    private final RabbitTemplate rabbitTemplate;
    private static final String EXCHANGE = "cache.invalidate";
    private static final String ROUTING_KEY = "cache.delete";

    // 发送延迟消息（延迟 1 秒）
    public void send(String cacheKey, Long userId) {
        CacheInvalidateMsg msg = CacheInvalidateMsg.builder()
            .cacheKey(cacheKey)
            .bizId(userId)
            .retryCount(0)
            .build();
        
        rabbitTemplate.convertAndSend(
            EXCHANGE, 
            ROUTING_KEY, 
            JSON.toJSONString(msg),
            message -> {
                // 设置消息 TTL = 1 秒
                message.getMessageProperties().setExpiration("1000");
                return message;
            }
        );
        
        log.warn("缓存删除失败，发送延迟消息重试, key={}", cacheKey);
    }
}
```

### 延迟队列消费者

```java
@Component
@Slf4j
public class CacheInvalidateConsumer {

    private static final int MAX_RETRY = 3;

    @RabbitListener(queues = "cache.invalidate.queue")
    public void handle(CacheInvalidateMsg msg) {
        try {
            // 删除缓存
            // redisTemplate.delete(msg.getCacheKey());  // 伪代码
            
            log.info("缓存删除成功, key={}", msg.getCacheKey());
            
        } catch (Exception e) {
            log.error("缓存删除失败, key={}, retry={}", 
                msg.getCacheKey(), msg.getRetryCount(), e);
            
            if (msg.getRetryCount() < MAX_RETRY) {
                // 重试次数未满，发送新消息重试
                msg.setRetryCount(msg.getRetryCount() + 1);
                // delayQueue.send(msg);
            } else {
                // 超过最大重试次数，告警人工处理
                alertService.notify("P0告警: 缓存删除失败，请检查 key=" + msg.getCacheKey());
            }
        }
    }
}
```

## 六、面试加分话术

> **面试官追问：为什么不能用更新缓存代替删除缓存？**

> 更新缓存有并发问题。比如两个请求同时更新：请求 A 更新缓存为 v1，请求 B 更新缓存为 v2，但如果执行顺序是 A 先写但 B 后写，最终缓存是 v1，但数据库最新值可能是 v2。删除缓存更安全，因为缓存 miss 后下次会从数据库加载最新值。另外更新缓存还有额外开销（需要计算新值），而删除直接丢弃。

> **面试官追问：Redis 和 MySQL 双写一致，有没有强一致性方案？**

> 强一致性需要分布式锁或 2PC/3PC，但性能损耗很大，不适合高并发场景。一般用最终一致性 + 重试补偿就够了。读多写少的场景 Cache Aside 足够，读少写多的场景考虑直接查库，不走缓存。

---

## 一句话总结

**删缓存失败 = 数据不一致炸弹。** 最佳实践是**"先删缓存，再更库" + 延迟双删或 MQ 异步重删**，保证最终一致性。Canal 监听 binlog 是更优雅的方案，让删缓存和业务完全解耦。
