54. 你用 Redis 做排行榜：
```java
public void addScore(String userId, double score) {
    redis.zadd("rank", score, userId);
}

public Set<String> getTop10() {
    return redis.zrevrange("rank", 0, 9);
}
```
**问题：** 如果排行榜数据量很大（1000 万用户），有什么问题？如何优化？

---
1. **1000万数据在Redis Sorted Set中的问题**：
   - 内存问题：每个用户占一条记录，1000万 × (userId + score + overhead) = 内存巨大
   - 分页问题：ZREVRANGE在小数据量时O(log(N)+M)，但数据量大了性能也会下降
   - 实时性问题：每次更新分数都要操作Redis

2. **优化方案**：
   - 分桶：将排行榜按分数区间或用户ID哈希分到不同的key
   - 分页优化：用ZSCAN代替ZREVRANGE
   - 本地缓存：用Redis做增量，本地做聚合
   - 异步写入：分数变化先写到MQ，异步更新排行榜
   - 热数据分离：Top N用Redis，普通数据用数据库+缓存
   - 持久化：定期将Redis数据持久化到数据库
   - 大key问题：1000万用户的zset是大key，需要拆分

## 结论先行

**1000 万用户的排行榜，Redis 单 key 存不下也查不快。** 核心问题是**大 key 性能瓶颈 + 内存爆炸**，解法是**分桶 + 热冷分离 + 异步更新**。

## 二、解决方案

### 方案一：分桶策略（核心优化）✅

**核心思想：把一个大排行榜拆成多个小桶，降低单 key 压力**

#### 分桶方式 1：按分数区间分桶

```
rank:0-1000    → 分数在 0-1000 的用户
rank:1000-2000 → 分数在 1000-2000 的用户
rank:2000-3000 → 分数在 2000-3000 的用户
...

用户更新分数时：
1. 从旧桶删除
2. 根据新分数加入新桶

查询 TOP N 时：
遍历所有桶，每个桶取 TOP(K)
然后全局排序取 TOP N
```

```java
@Service
@Slf4j
public class RankingService {

    private final StringRedisTemplate redisTemplate;
    
    // 桶数量（根据数据量调整）
    private static final int BUCKET_COUNT = 100;
    private static final int BUCKET_CAPACITY = 100_000;  // 每个桶最多10万用户

    /**
     * 计算用户属于哪个桶（按分数范围）
     */
    private String getBucketKey(double score) {
        int bucketIndex = Math.min((int) (score / 10), BUCKET_COUNT - 1);
        return "rank:bucket:" + bucketIndex;
    }

    /**
     * 更新用户分数
     */
    public void updateScore(String userId, double score) {
        // 1. 找出当前分数所在桶（需要遍历，太慢）
        // 优化：用户信息里存当前桶，直接操作
        
        // 假设 user:info:{userId} 存了当前桶和分数
        String userBucket = userService.getUserBucket(userId);
        String oldBucket = userBucket;
        String newBucket = getBucketKey(score);
        
        // 2. 从旧桶移除
        if (oldBucket != null) {
            redisTemplate.opsForZSet().remove(oldBucket, userId);
        }
        
        // 3. 加入新桶
        redisTemplate.opsForZSet().add(newBucket, userId, score);
        
        // 4. 更新用户信息
        userService.updateUserBucket(userId, newBucket, score);
    }

    /**
     * 获取 TOP N
     */
    public List<UserRank> getTopN(int n) {
        List<UserRank> result = new ArrayList<>();
        
        // 1. 从每个桶取 TOP(n)（最多100桶）
        for (int i = BUCKET_COUNT - 1; i >= 0; i--) {
            String bucketKey = "rank:bucket:" + i;
            Set<ZSetOperations.TypedTuple<String>> top = 
                redisTemplate.opsForZSet().reverseRangeWithScores(
                    bucketKey, 0, n - 1
                );
            
            if (top != null) {
                for (ZSetOperations.TypedTuple<String> tuple : top) {
                    result.add(new UserRank(tuple.getValue(), tuple.getScore()));
                }
            }
            
            // 优化：已收集够 n 个，提前结束
            if (result.size() >= n) break;
        }
        
        // 2. 全局排序取 TOP N
        return result.stream()
            .sorted(Comparator.comparingDouble(UserRank::getScore).reversed())
            .limit(n)
            .collect(Collectors.toList());
    }
}
```

#### 分桶方式 2：按用户 ID 哈希分桶（更常用）✅

```
rank:shard:0 → userId hash % 100 = 0 的用户
rank:shard:1 → userId hash % 100 = 1 的用户
...

优点：用户固定在一个桶，不需要移动
缺点：查询 TOP N 时要合并所有桶
```

```java
public class HashShardingRanking {
    
    private static final int SHARD_COUNT = 100;
    
    private String getShardKey(String userId) {
        int shard = Math.abs(userId.hashCode() % SHARD_COUNT);
        return "rank:shard:" + shard;
    }
    
    public void updateScore(String userId, double score) {
        String shardKey = getShardKey(userId);
        redisTemplate.opsForZSet().add(shardKey, userId, score);
    }
    
    public List<UserRank> getTopN(int n) {
        List<UserRank> result = new ArrayList<>();
        
        // 并行从 100 个桶取 TOP N/100
        for (int i = 0; i < SHARD_COUNT; i++) {
            String shardKey = "rank:shard:" + i;
            Set<ZSetOperations.TypedTuple<String>> top = 
                redisTemplate.opsForZSet()
                    .reverseRangeWithScores(shardKey, 0, n / SHARD_COUNT);
            
            if (top != null) {
                top.forEach(t -> result.add(
                    new UserRank(t.getValue(), t.getScore())
                ));
            }
        }
        
        // 排序取 TOP N
        return result.stream()
            .sorted(Comparator.comparingDouble(UserRank::getScore).reversed())
            .limit(n)
            .collect(Collectors.toList());
    }
    
    // 获取用户的排名
    public Long getUserRank(String userId) {
        String shardKey = getShardKey(userId);
        // 获取该用户在桶内的排名
        Long rankInShard = redisTemplate.opsForZSet()
            .reverseRank(shardKey, userId);
        
        // 加上前面所有桶的元素总数
        long totalRank = 0;
        int shardIndex = Math.abs(userId.hashCode() % SHARD_COUNT);
        for (int i = 0; i < shardIndex; i++) {
            Long size = redisTemplate.opsForZSet().size("rank:shard:" + i);
            totalRank += (size != null ? size : 0);
        }
        
        return totalRank + (rankInShard != null ? rankInShard : -1);
    }
}
```

### 方案二：热冷分离 ✅

**核心思想：TOP 1000 用 Redis，1000 名以后用 MySQL + 缓存**

```
┌─────────────────────────────────────────────────────────────┐
│                    热冷分离架构                               │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  热数据层（Redis Sorted Set）                         │   │
│  │  rank:hot → 只存 TOP 10000 用户                       │   │
│  │  优点：查询极快，内存可控                              │   │
│  └─────────────────────────────────────────────────────┘   │
│                           │                                │
│                           ▼                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  冷数据层（MySQL + 索引）                             │   │
│  │  user_score 表 → 存所有用户分数                        │   │
│  │  优点：无限扩展，支持复杂查询                          │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  定时任务：每小时更新热数据层                                 │
└─────────────────────────────────────────────────────────────┘
```

```java
@Service
@Slf4j
public class HotColdRankingService {

    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private UserScoreMapper userScoreMapper;
    
    private static final int HOT_SIZE = 10_000;  // 热数据 TOP 1万

    // ========== 写入操作 ==========
    
    public void updateScore(String userId, double score) {
        // 1. 直接写 MySQL（主库）
        userScoreMapper.upsertScore(userId, score);
        
        // 2. 异步更新热数据层
        hotRankUpdater.submit(() -> {
            // 后面定时任务会同步
            log.debug("用户分数更新, userId={}, score={}", userId, score);
        });
    }

    // ========== 查询操作 ==========
    
    public List<UserRank> getTopN(int n) {
        // 查热数据层
        String hotKey = "rank:hot";
        Set<ZSetOperations.TypedTuple<String>> hotRanks = 
            redisTemplate.opsForZSet().reverseRangeWithScores(hotKey, 0, n - 1);
        
        return Optional.ofNullable(hotRanks)
            .map(set -> set.stream()
                .map(t -> new UserRank(t.getValue(), t.getScore()))
                .collect(Collectors.toList()))
            .orElse(new ArrayList<>());
    }
    
    public Long getUserRank(String userId) {
        // 先查热数据层
        Long rank = redisTemplate.opsForZSet().reverseRank("rank:hot", userId);
        if (rank != null) {
            return rank + 1;  // Redis rank 从 0 开始
        }
        
        // 不在热数据中，查 MySQL
        return userScoreMapper.countGreaterThanScore(userId);
    }

    // ========== 定时同步热数据 ==========
    
    @Scheduled(fixedRate = 3600000)  // 每小时
    public void syncHotData() {
        log.info("开始同步热数据...");
        
        // 1. 从 MySQL 取 TOP N
        List<UserScore> topUsers = userScoreMapper.selectTopN(HOT_SIZE);
        
        // 2. 全量替换热数据桶
        redisTemplate.delete("rank:hot");
        
        String hotKey = "rank:hot";
        for (UserScore us : topUsers) {
            redisTemplate.opsForZSet().add(hotKey, us.getUserId(), us.getScore());
        }
        
        redisTemplate.expire(hotKey, Duration.ofHours(2));
        
        log.info("热数据同步完成, count={}", topUsers.size());
    }
}
```

### 方案三：异步批量写入（解决高 QPS）✅

**核心思想：分数更新先写本地缓冲 + MQ，定时批量写 Redis**

```
用户更新分数
    │
    ├─▶ 本地 Buffer（内存）
    │       │
    │       ▼ 定时任务（每 5 秒）
    │   批量合并
    │       │
    │       ▼
    └─▶ Redis ZADD（批量）
```

```java
@Service
@Slf4j
public class AsyncRankingService {

    // 本地缓冲（Guava Cache）
    private final LoadingCache<String, Double> scoreBuffer = 
        CacheBuilder.newBuilder()
            .maximumSize(1_000_000)
            .expireAfterWrite(10, TimeUnit.SECONDS)
            .build(new CacheLoader<String, Double>() {
                @Override
                public Double load(String userId) {
                    return 0.0;  // 默认分数
                }
            });

    private final ConcurrentHashMap<String, Double> pendingUpdates = 
        new ConcurrentHashMap<>();

    // ========== 写入：先更新本地缓冲 ==========
    
    public void updateScore(String userId, double score) {
        // 1. 更新本地缓冲（内存操作，极快）
        scoreBuffer.put(userId, score);
        pendingUpdates.put(userId, score);
    }

    // ========== 定时批量同步到 Redis ==========
    
    @Scheduled(fixedDelay = 5000)  // 每 5 秒
    public void syncToRedis() {
        if (pendingUpdates.isEmpty()) return;
        
        log.info("开始同步分数到 Redis, count={}", pendingUpdates.size());
        
        // 1. 取出所有待更新（原子交换）
        ConcurrentHashMap<String, Double> toSync = 
            pendingUpdates.getAndSet(new ConcurrentHashMap<>());
        
        // 2. 批量写入 Redis
        RedisCallback<Object> callback = connection -> {
            for (Map.Entry<String, Double> entry : toSync.entrySet()) {
                connection.zSetCommands().zAdd(
                    "rank".getBytes(),
                    entry.getValue(),
                    entry.getKey().getBytes()
                );
            }
            return null;
        };
        
        redisTemplate.execute(callback);
        
        log.info("分数同步完成, count={}", toSync.size());
    }
}
```

## 三、方案对比

| 方案 | 内存占用 | 查询性能 | 写入性能 | 实现复杂度 | 适用场景 |
|------|---------|---------|---------|-----------|---------|
| 单 key | ❌ 10GB+ | O(log N) | O(log N) | 低 | ❌ 不推荐 |
| 哈希分桶 | ✅ 10GB/100 | O(log(N/100)) | O(log(N/100)) | 中 | ✅ 通用 |
| 分数区间分桶 | ✅ 可控 | 中 | 需迁移 | 高 | 分数范围有限 |
| 热冷分离 | ✅ 热数据少 | ✅ 极快 | ✅ 异步 | 中 | TOP N 场景 |
| 异步批量写 | ✅ 内存缓冲 | O(log N) | ✅ 批量 | 中 | 高 QPS 写入 |

## 四、面试加分话术

> **面试官追问：Redis Sorted Set 的底层原理？**

> Redis Sorted Set 底层是**跳表（Skip List）+ 哈希表**的组合。哈希表存 member → score 的映射，O(1) 查分。跳表存按分数排序的有序列表，O(log N) 范围查询。跳表的优势是插入/删除比平衡树简单，但内存占用略高。

> **面试官追问：为什么要用多个桶而不是一个大桶？**

> 主要解决两个问题：1）**大 key 问题**——单个 key 太大，RDB/AOF 备份和主从同步都慢。2）**热 key 问题**——单个 key 访问集中在某几个分片上，分桶可以分散热点。另外，分桶后每个桶的 ZRANGE 操作更快。

---

## 一句话总结

**1000 万用户的 Redis 排行，单 key 扛不住。** 核心优化是**哈希分桶降低单 key 大小 + 热冷分离只存 TOP N + 异步批量写入减少 QPS**。根据业务场景选组合，TOP 10 查询优先考虑热冷分离，高写入优先考虑异步缓冲。
