43. 你用 `substring()` 截取字符串：
```java
String str = new String(new char[1000000]);
String sub = str.substring(0, 10);
```
**问题：** JDK 6 和 JDK 7+ 的实现有什么区别？内存方面有什么影响？

---
## 结论先行

**JDK 6 的 `substring()` 共享底层 `char[]`，导致大字符串无法被 GC——这是一个著名的隐性内存泄漏。JDK 7 彻底重写，改为复制新数组。**

---

## 一、JDK 6 的实现（有坑）

```
┌─────────────────────────────────────────────────────────────────────┐
│                    JDK 6 substring() 内部结构                       │
│                                                                     │
│  String 对象三个字段：                                               │
│  • char[] value   ← 底层字符数组                                    │
│  • int   offset   ← 起始偏移量                                      │
│  • int   count    ← 长度                                            │
│                                                                     │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │  String str = new String(new char[1000000]);                 │  │
│  │                                                              │  │
│  │  str 对象                    底层 char[]                      │  │
│  │  ┌──────────────┐           ┌──────────────────────────┐    │  │
│  │  │ value ───────┼──────────▶│  1,000,000 个 '\0'       │    │  │
│  │  │ offset = 0   │           │  （占用约 2MB 内存）       │    │  │
│  │  │ count = 1M   │           └──────────────────────────┘    │  │
│  │  └──────────────┘                       ▲                   │  │
│  │                                         │ 共享同一个数组！   │  │
│  │  sub 对象                               │                   │  │
│  │  ┌──────────────┐                       │                   │  │
│  │  │ value ────────┼───────────────────────┘                   │  │
│  │  │ offset = 0   │                                           │  │
│  │  │ count = 10   │ ← 只使用 10 个字符                        │  │
│  │  └──────────────┘                                           │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                     │
│  str = null;  // 即使 str 置 null                                   │
│  // sub 仍持有对 char[1000000] 的强引用                               │
│  // → 1MB 的字节数组无法被 GC → 内存泄漏！                           │
└─────────────────────────────────────────────────────────────────────┘
```

**JDK 6 源码：**

```java
// JDK 6 String.substring()
public String substring(int beginIndex, int endIndex) {
    // ... 边界检查 ...
    return (beginIndex == 0 && endIndex == count) ? this
        : new String(offset + beginIndex,    // 偏移量
                     endIndex - beginIndex,  // 长度
                     value);                 // 共享原始数组！
}

// JDK 6 的特殊构造函数（包私有）
String(int offset, int count, char value[]) {
    this.value  = value;    // 直接引用，不复制
    this.offset = offset;
    this.count  = count;
}
```

## 二、JDK 7+ 的实现（修复）

```
┌─────────────────────────────────────────────────────────────────────┐
│                    JDK 7+ substring() 内部结构                      │
│                                                                     │
│  String 对象去掉了 offset 和 count 字段，只保留：                   │
│  • char[] value   ← 底层字符数组（每个 String 独立）                │
│  • int    hash    ← 缓存的哈希值                                    │
│                                                                     │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │  String str = new String(new char[1000000]);                 │  │
│  │                                                              │  │
│  │  str 对象              char[1000000]                         │  │
│  │  ┌──────────┐         ┌──────────────────────────┐          │  │
│  │  │ value ───┼────────▶│  1,000,000 个 '\0'       │          │  │
│  │  └──────────┘         └──────────────────────────┘          │  │
│  │                                                              │  │
│  │  String sub = str.substring(0, 10);                         │  │
│  │                                                              │  │
│  │  sub 对象              char[10]（全新数组）                  │  │
│  │  ┌──────────┐         ┌──────────────────────────┐          │  │
│  │  │ value ───┼────────▶│  '\0','\0',...,'\0'      │          │  │
│  │  └──────────┘         └──────────────────────────┘          │  │
│  │                                                              │  │
│  │  str = null;   // str 置 null                               │  │
│  │  // char[1000000] 没有任何强引用 → GC 正常回收 ✅            │  │
│  └──────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────┘
```

**JDK 7+ 源码：**

```java
// JDK 7+ String.substring()
public String substring(int beginIndex, int endIndex) {
    // ... 边界检查 ...
    int subLen = endIndex - beginIndex;
    return (beginIndex == 0 && endIndex == length()) ? this
        : new String(value, beginIndex, subLen);  // 复制新数组！
}

// JDK 7+ 使用标准构造函数
public String(char value[], int offset, int count) {
    this.value = Arrays.copyOfRange(value, offset, offset + count);
    // ✅ 复制新数组，不持有原始数组引用
}
```

## 三、内存影响对比

| 维度 | JDK 6 | JDK 7+ |
|------|-------|--------|
| 底层实现 | 共享原始 `char[]` | 复制新 `char[]` |
| `substring()` 性能 | 快（O(1)，只改偏移量） | 慢（O(n)，需要复制） |
| 内存占用 | 危险（原数组无法 GC） | 安全（独立数组） |
| 内存泄漏风险 | 高 | 无 |
| String 对象大小 | 含 offset+count 字段 | 只含 value+hash |

## 四、JDK 6 的修复方式（如果你被迫用 JDK 6）

```java
// ❌ JDK 6 泄漏写法
String sub = hugeString.substring(0, 10);  // 仍持有 1MB 数组

// ✅ JDK 6 修复写法：强制复制
String sub = new String(hugeString.substring(0, 10));
//           ↑ 通过 new String() 触发一次数组复制，释放对大数组的引用
```

```
┌─────────────────────────────────────────────────────────────────────┐
│                    JDK 6 修复原理                                    │
│                                                                     │
│  new String(sub) 调用的是：                                          │
│  public String(String original) {                                   │
│      this.value  = original.value;  // JDK 6 也直接共享…             │
│  }                                                                  │
│                                                                     │
│  ⚠️ 注意：JDK 6 的 new String(str) 实际上也共享数组！               │
│  真正的修复方式是：                                                   │
│                                                                     │
│  char[] chars = hugeString.substring(0, 10).toCharArray();         │
│  String sub   = new String(chars);   // ✅ 强制完全复制              │
│                                                                     │
│  或者更简单：                                                        │
│  String sub = new String(hugeString.toCharArray(), 0, 10);  // ✅   │
└─────────────────────────────────────────────────────────────────────┘
```

## 六、面试追问

> **JDK 6 的共享 char[] 设计初衷是什么？**
>
> 当时认为 `substring()` 是高频操作，共享数组可以 O(1) 完成，节省 CPU。但实际场景中，截取短子串 + 丢弃大字符串的情况很常见，导致大数组无法 GC，利大于弊，JDK 7 果断改掉。

> **`intern()` 和 `substring()` 都有历史坑，String 设计最大的教训是什么？**
>
> 共享可变底层资源（即使只读）在 GC 语言里依然危险，因为 GC 是引用计数逻辑——你看不到的地方只要有一个引用存着，内存就不会释放。**空间换时间的优化，必须考虑 GC 的副作用。**

> **用 `+` 拼接字符串在循环里为什么慢？**
>
> ```java
> // ❌ 循环里 + 拼接：每次都创建新 String 对象
> String result = "";
> for (int i = 0; i < 10000; i++) {
>     result += i;   // 等价于 new StringBuilder(result).append(i).toString()
> }
> // 产生 10000 个临时 String 对象，GC 压力巨大
>
> // ✅ StringBuilder 复用同一 char[]
> StringBuilder sb = new StringBuilder();
> for (int i = 0; i < 10000; i++) {
>     sb.append(i);
> }
> String result = sb.toString();  // 只 new 一次
> ```

---

## 一句话收尾

**JDK 6 的 `substring()` 是用 "偏移量共享数组" 换来的 O(1) 速度，却埋下了内存泄漏地雷；JDK 7 一刀切换成复制新数组，安全换速度，这笔账值得。**
