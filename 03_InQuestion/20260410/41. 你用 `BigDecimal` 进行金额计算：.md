41. 你用 `BigDecimal` 进行金额计算：
```java
BigDecimal a = new BigDecimal(0.1);
BigDecimal b = new BigDecimal(0.2);
System.out.println(a.add(b));
```
**问题：** 输出结果是什么？有什么问题？正确的写法是什么？

---
## 结论先行

**`new BigDecimal(0.1)` 是陷阱！double 本身就不精确，用它构造 BigDecimal 会把精度问题带进来。**

---

## 一、实际输出结果

```java
BigDecimal a = new BigDecimal(0.1);
BigDecimal b = new BigDecimal(0.2);
System.out.println(a.add(b));
```

**实际输出：**
```
0.3000000000000000166533453693773481063544750213623046875
```

不是你期望的 `0.3`，而是一串长得离谱的小数。

---

## 二、根因分析：double 的精度问题

```
┌─────────────────────────────────────────────────────────────────────┐
│                  浮点数二进制表示问题                                │
│                                                                     │
│  十进制 0.1 转二进制 → 无限循环小数（无法精确表示）                   │
│                                                                     │
│  0.1 (十进制)                                                        │
│   = 0.0001100110011001100110011... (二进制，无限循环)                │
│                                                                     │
│  double 只有 64 位存储，强行截断 → 精度损失                          │
│                                                                     │
│  double 实际存的 0.1：                                               │
│  = 0.1000000000000000055511151231257827021181583404541015625         │
│                                                                     │
│  double 实际存的 0.2：                                               │
│  = 0.200000000000000011102230246251565404236316680908203125          │
│                                                                     │
│  两者相加：                                                          │
│  ≈ 0.3000000000000000166533453693773481063544750213623046875         │
│                                                                     │
│  new BigDecimal(double) 的语义：                                    │
│  "把这个 double 精确地表示出来"                                      │
│  ↓                                                                   │
│  = 把已经不精确的 double 原样放大，误差被忠实保留                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

## 三、正确写法

### 方式一：`new BigDecimal(String)` ✅（首推）

```java
BigDecimal a = new BigDecimal("0.1");
BigDecimal b = new BigDecimal("0.2");
System.out.println(a.add(b));   // 0.3 ✅
```

### 方式二：`BigDecimal.valueOf(double)` ✅（也推荐）

```java
BigDecimal a = BigDecimal.valueOf(0.1);
BigDecimal b = BigDecimal.valueOf(0.2);
System.out.println(a.add(b));   // 0.3 ✅
```

**`BigDecimal.valueOf(double)` 内部做了什么？**

```java
// JDK 源码
public static BigDecimal valueOf(double val) {
    // 先 toString，再用字符串构造
    // Double.toString(0.1) = "0.1"（JDK 会做四舍五入到最短可辨认字符串）
    return new BigDecimal(Double.toString(val));
}
```

## 四、三种构造方式对比

```java
// 陷阱对比代码
BigDecimal d1 = new BigDecimal(0.1);         // ❌ double 构造
BigDecimal d2 = new BigDecimal("0.1");       // ✅ String 构造
BigDecimal d3 = BigDecimal.valueOf(0.1);     // ✅ valueOf

System.out.println(d1);  // 0.1000000000000000055511151231257827021181583404541015625
System.out.println(d2);  // 0.1
System.out.println(d3);  // 0.1
```

| 构造方式 | 结果 | 是否推荐 | 原因 |
|---------|------|---------|------|
| `new BigDecimal(0.1)` | 精度丢失 | ❌ | double 本身不精确 |
| `new BigDecimal("0.1")` | 精确 | ✅ | 字符串直接解析 |
| `BigDecimal.valueOf(0.1)` | 精确 | ✅ | 内部转 String 再解析 |

## 五、金额计算完整规范

```java
/**
 * 金额计算工具类（生产规范）
 */
public class MoneyCalculator {

    // 保留两位小数，四舍五入
    private static final int SCALE = 2;
    private static final RoundingMode ROUNDING = RoundingMode.HALF_UP;

    /**
     * 加法
     */
    public static BigDecimal add(String a, String b) {
        return new BigDecimal(a)
            .add(new BigDecimal(b))
            .setScale(SCALE, ROUNDING);
    }

    /**
     * 减法
     */
    public static BigDecimal subtract(String a, String b) {
        return new BigDecimal(a)
            .subtract(new BigDecimal(b))
            .setScale(SCALE, ROUNDING);
    }

    /**
     * 乘法（如：金额 × 税率）
     */
    public static BigDecimal multiply(String amount, String rate) {
        return new BigDecimal(amount)
            .multiply(new BigDecimal(rate))
            .setScale(SCALE, ROUNDING);
    }

    /**
     * 除法（必须指定精度，否则无限小数会抛 ArithmeticException）
     */
    public static BigDecimal divide(String a, String b) {
        return new BigDecimal(a)
            .divide(new BigDecimal(b), SCALE, ROUNDING);
    }
}

// 使用示例
BigDecimal price = new BigDecimal("99.90");
BigDecimal qty   = new BigDecimal("3");
BigDecimal total = price.multiply(qty)
                        .setScale(2, RoundingMode.HALF_UP);
System.out.println(total);  // 299.70 ✅
```

## 六、RoundingMode 选哪种？

| 模式 | 说明 | 典型场景 |
|------|------|---------|
| `HALF_UP` | 四舍五入（最常用） | 日常金额计算 |
| `HALF_EVEN` | 银行家舍入（四舍六入五取偶） | 统计/财务报表 |
| `DOWN` | 截断（向零舍入） | 优惠券计算（不多给用户钱） |
| `UP` | 向远离零方向 | 手续费（不能少收） |
| `CEILING` | 向正无穷 | 向上取整 |
| `FLOOR` | 向负无穷 | 向下取整 |

## 七、常见追问

> **除法不指定精度会怎样？**

```java
// ❌ 危险：1 ÷ 3 是无限小数，不指定精度直接抛异常
new BigDecimal("1").divide(new BigDecimal("3"));
// → ArithmeticException: Non-terminating decimal expansion

// ✅ 正确
new BigDecimal("1").divide(new BigDecimal("3"), 2, RoundingMode.HALF_UP);
// → 0.33
```

> **能用 `==` 比较 BigDecimal 吗？**

```java
BigDecimal a = new BigDecimal("1.0");
BigDecimal b = new BigDecimal("1.00");

System.out.println(a.equals(b));       // false ❌（精度不同）
System.out.println(a.compareTo(b));    // 0  ✅（数值相等）

// 金额比较必须用 compareTo！
if (price.compareTo(BigDecimal.ZERO) > 0) { ... }
```

> **数据库存金额用什么类型？**

```sql
-- MySQL: DECIMAL(19, 4)
-- 19位总长度，4位小数，覆盖绝大多数金融场景
price DECIMAL(19, 4) NOT NULL DEFAULT '0.0000'

-- 对应 Java 字段
@Column(precision = 19, scale = 4)
private BigDecimal price;
```

## 一句话收尾

**金额计算三要素：`new BigDecimal(String)` 构造、`setScale` 指定精度、`compareTo` 比较大小——少一样都是线上事故。**
