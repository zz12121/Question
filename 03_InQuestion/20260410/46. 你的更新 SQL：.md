46. 你的更新 SQL：
```sql
UPDATE account SET balance = balance - 100 WHERE user_id = 100;
```
**问题：** 高并发下会出现什么问题？如何保证余额不为负数？

---
## 结论先行

**高并发下，这句 SQL 会导致"余额被扣成负数"**。原因是 check-then-act 竞态条件（检查余额 → 扣款之间有时间窗口），核心解决思路是**把"先查后改"变成"原子扣减"**，让数据库天然保证余额不为负。

---

## 一、高并发下会出现什么问题？

### 问题一：余额扣成负数（超卖）

```
时间线：

T1: 线程A 查询余额 = 1000 ✅
T2: 线程B 查询余额 = 1000 ✅
T3: 线程A 执行 UPDATE balance = 900（扣100）✅
T4: 线程B 执行 UPDATE balance = 900（扣100）❌ 
    —— 应该扣成800，实际两个人都扣成了900
T5: 线程C 查询余额 = 900 ✅
T6: 线程C 执行 UPDATE balance = 800（扣100）✅
T7: 线程D 查询余额 = 800 ✅
T8: 线程D 执行 UPDATE balance = 700（扣100）✅
...
继续并发，余额越来越低 → 负数！
```

**根本原因：** `SELECT balance` 和 `UPDATE balance` 是两次独立的 SQL，存在竞态条件（Race Condition）。

### 问题二：重复扣款（幂等性问题）

用户点击支付按钮两次，假设网络重试发了两次请求：

```
请求1: UPDATE account SET balance = balance - 100  → 余额 900
请求2: UPDATE account SET balance = balance - 100  → 余额 800（重复扣了！）
```

### 问题三：数据不一致（ABA 问题）

线程 A 读余额 1000 → 线程 B 改成 900 → 线程 B 再改回 1000 → 线程 A 以为余额没变，继续扣款。

---

## 二、解决方案全解

### 方案一：余额条件更新（最简单 + 最推荐）✅

**核心思想：让数据库保证原子性，WHERE 条件限制余额不能为负**

```sql
-- ❌ 原始写法（不保护余额）
UPDATE account SET balance = balance - 100 WHERE user_id = 100;

-- ✅ 保护余额的写法（余额不足时影响行数为0）
UPDATE account 
SET balance = balance - 100 
WHERE user_id = 100 
  AND balance >= 100;  -- 关键：余额不足则更新0行
```

**Java 代码：**

```java
public int deductBalance(Long userId, BigDecimal amount) {
    // 返回影响行数，0表示余额不足
    int rows = accountMapper.updateBalance(userId, amount);
    if (rows == 0) {
        throw new BusinessException("余额不足");
    }
    return rows;
}

// Mapper
@Update("UPDATE account SET balance = balance - #{amount} " +
        "WHERE user_id = #{userId} AND balance >= #{amount}")
int updateBalance(@Param("userId") Long userId, @Param("amount") BigDecimal amount);
```

**效果：**

| 线程 | 初始余额 | WHERE 条件 | 执行后余额 | 结果 |
|------|---------|-----------|-----------|------|
| A | 100 | 100 >= 100 ✅ | 0 | 成功 |
| B | 100 | 100 >= 100 ✅ | 0 | 成功（不应该！） |
| C | 50 | 50 >= 100 ❌ | 50 | 更新0行，抛异常 |

> **等等，线程 A 和 B 都成功了？** 是的，条件更新只保证余额不为负，但不保证不超扣。余额 100 时，100 个人同时扣 1 元，还是会超扣。

---

### 方案二：乐观锁 + 版本号（适合并发不极端的场景）

**核心思想：加个版本号，更新时比对版本，防止并发修改**

```sql
-- 表结构加 version 字段
ALTER TABLE account ADD COLUMN version INT DEFAULT 0;

-- 查询时带上版本号
SELECT balance, version FROM account WHERE user_id = 100;
-- 返回: balance=1000, version=5

-- 更新时比对版本
UPDATE account 
SET balance = balance - 100, version = version + 1
WHERE user_id = 100 
  AND version = 5
  AND balance >= 100;
-- 影响行数: 1=成功, 0=被其他线程修改过，重试
```

**Java 实现（重试机制）：**

```java
public void deductWithOptimisticLock(Long userId, BigDecimal amount) {
    int retryCount = 3;
    while (retryCount > 0) {
        Account account = accountMapper.selectByUserId(userId);
        
        if (account.getBalance().compareTo(amount) < 0) {
            throw new BusinessException("余额不足");
        }
        
        int rows = accountMapper.updateBalanceWithVersion(
            userId, 
            amount, 
            account.getVersion()
        );
        
        if (rows > 0) {
            return;  // 成功
        }
        retryCount--;
        log.warn("乐观锁冲突，重试剩余次数={}", retryCount);
    }
    throw new BusinessException("并发更新失败，请稍后重试");
}

// Mapper
@Update("UPDATE account SET balance = balance - #{amount}, version = version + 1 " +
        "WHERE user_id = #{userId} AND version = #{version} AND balance >= #{amount}")
int updateBalanceWithVersion(@Param("userId") Long userId, 
                              @Param("amount") BigDecimal amount,
                              @Param("version") Integer version);
```

**乐观锁的问题：** 高并发下大量重试，性能急剧下降。

### 方案三：悲观锁（SELECT FOR UPDATE）✅ 适合强一致性场景

**核心思想：先锁住这行数据，再做操作，串行化并发**

```java
@Transactional
public void deductWithPessimisticLock(Long userId, BigDecimal amount) {
    // FOR UPDATE 加锁，事务提交/回滚后自动释放
    Account account = accountMapper.selectByUserIdForUpdate(userId);
    
    if (account.getBalance().compareTo(amount) < 0) {
        throw new BusinessException("余额不足");
    }
    
    account.setBalance(account.getBalance().subtract(amount));
    accountMapper.update(account);
}
```

**SQL：**

```sql
-- 加锁读取（阻塞其他事务的同一行）
SELECT * FROM account WHERE user_id = 100 FOR UPDATE;
```

**效果对比：**

| 锁类型 | 并发度 | 适用场景 |
|--------|--------|---------|
| 乐观锁 | 高（读多写少） | 低并发、冲突少 |
| 悲观锁 | 低（写多读多） | 高并发写、余额敏感 |
| 无锁 + 条件更新 | 最高 | 并发一般、只要求余额不超扣 |

**悲观锁的风险：**

```
线程A: SELECT ... FOR UPDATE (加锁，等待)
线程B: SELECT ... FOR UPDATE (排队等待)
线程C: SELECT ... FOR UPDATE (排队等待)
线程D: SELECT ... FOR UPDATE (排队等待)
...
长事务：线程A 执行 5 秒，后面全部阻塞 → 数据库连接耗尽 → 雪崩
```

### 方案四：分布式锁（跨 JVM 同步）✅ 分布式系统必备

**核心思想：在应用层加锁，防止多个服务实例同时操作同一用户余额**

```java
public void deductWithDistributedLock(Long userId, BigDecimal amount) {
    String lockKey = "account:lock:" + userId;
    
    try {
        // 尝试获取锁，最长等 3 秒，锁自动 10 秒过期
        Boolean acquired = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, "1", Duration.ofSeconds(10));
        
        if (!acquired) {
            throw new BusinessException("系统繁忙，请稍后重试");
        }
        
        // 拿到了锁，执行扣款
        deductInternal(userId, amount);
        
    } finally {
        // 释放锁
        redisTemplate.delete(lockKey);
    }
}
```

**分布式锁优化：可重入 + 自动续期（Redisson）：**

```java
RLock lock = redissonClient.getLock("account:lock:" + userId);

try {
    // 尝试加锁，等 3 秒，锁 10 秒自动释放（自动续期）
    if (!lock.tryLock(3, 10, TimeUnit.SECONDS)) {
        throw new BusinessException("系统繁忙");
    }
    deductInternal(userId, amount);
} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

## 三、生产级最佳实践：多层保护

```
┌─────────────────────────────────────────────────────────────┐
│  第1层：应用层分布式锁                                        │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  第2层：数据库条件更新                                │    │
│  │  ┌───────────────────────────────────────────────┐  │    │
│  │  │  第3层：业务层余额校验（兜底）                  │  │    │
│  │  │  balance.compareTo(amount) >= 0              │  │    │
│  │  └───────────────────────────────────────────────┘  │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

**完整代码：**

```java
@Service
@Slf4j
public class AccountService {

    @Transactional(rollbackFor = Exception.class)
    public void deduct(Long userId, BigDecimal amount) {
        // 1. 参数校验
        if (amount.compareTo(BigDecimal.ZERO) <= 0) {
            throw new BusinessException("扣款金额必须大于0");
        }

        // 2. 乐观锁重试扣款（最多3次）
        int retryCount = 3;
        while (retryCount > 0) {
            Account account = accountMapper.selectByUserId(userId);
            
            if (account.getBalance().compareTo(amount) < 0) {
                throw new BusinessException("余额不足，当前余额: " + account.getBalance());
            }

            // 3. 条件更新：余额 >= 扣款金额 才扣
            int rows = accountMapper.conditionalDeduct(
                userId, 
                amount, 
                account.getVersion()
            );

            if (rows > 0) {
                log.info("扣款成功, userId={}, amount={}, remaining={}", 
                    userId, amount, account.getBalance().subtract(amount));
                return;
            }
            
            retryCount--;
            log.warn("扣款乐观锁冲突, userId={}, 剩余重试={}", userId, retryCount);
        }

        throw new BusinessException("扣款失败，请稍后重试");
    }
}
```

**Mapper：**

```java
@Update("<script>" +
        "UPDATE account " +
        "SET balance = balance - #{amount}, version = version + 1 " +
        "WHERE user_id = #{userId} " +
        "  AND version = #{version} " +
        "  AND balance >= #{amount}" +
        "</script>")
int conditionalDeduct(@Param("userId") Long userId, 
                     @Param("amount") BigDecimal amount,
                     @Param("version") Integer version);
```

## 四、面试加分话术

> **面试官追问：条件更新 `balance >= 100` 能完全防止超扣吗？**

> 不能完全防止，在极端并发下（同一时刻100个请求，余额100），还是会有多个请求同时通过条件判断。条件更新只能**保证余额不为负**，但不能**保证扣款总额不超过余额**。真正要严格不超扣，还是要靠悲观锁或分布式锁串行化。

> **面试官追问：分布式锁和数据库锁的区别？**

> 数据库锁（FOR UPDATE）是数据库层面的，锁住在同一个数据库连接池内有效。但如果是微服务多实例部署，每个实例有自己的数据库连接，数据库锁管不了跨 JVM 的并发。分布式锁（Redis/Zookeeper）是在应用层做到跨 JVM 的同步，任何实例拿到锁就意味着其他实例不能操作。

---

## 一句话总结

**余额扣减的核心是"原子性"**：要么用数据库的原子操作（条件更新）保证余额不为负，要么用锁（悲观锁/分布式锁）串行化并发操作。生产环境通常是**多层保护 + 监控告警**。
