49. 你的表结构：
```sql
CREATE TABLE logs (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    content TEXT,
    create_time DATETIME,
    INDEX idx_time (create_time)
);
```
**问题：** 如果每天产生 1000 万条日志，这张表会出现什么问题？如何优化？

---
## 结论先行

**1000 万/天 × 365 = 36 亿行/年，MySQL 单表存不下也查不动。** 核心解法是**分区裁剪 + 冷热分离 + 定期归档**，让热数据快查询，冷数据低成本存储。

---

## 一、会出现什么问题？

### 问题一：存储爆炸

```
每天 1000 万条 × 每条 ~500 字节 = 5GB/天
1 个月 = 150GB
1 年 = 1.8TB（还没算索引）

MySQL 单表文件超过 50GB 后：
├── 备份/恢复慢（恢复 1TB 要几小时）
├── 迁移困难
└── 磁盘空间告警
```

### 问题二：索引查询变慢

```sql
-- 查询最近 7 天的日志
SELECT * FROM logs 
WHERE create_time >= '2024-01-01' 
ORDER BY create_time DESC 
LIMIT 100;

-- 问题：1 个月后的 idx_time 索引 B+ 树有 3 亿个节点
-- 索引深度增加，查询变慢
```

### 问题三：DDL 操作阻塞

```sql
ALTER TABLE logs ADD COLUMN level INT;  -- 💥 大表 DDL 要几小时
CREATE INDEX idx_level ON logs(level);   -- 💥 重建索引要几小时

-- DDL 期间表锁，线上业务不可用
```

### 问题四：主从延迟

```
单表太大 → 写入 QPS 高 → binlog 量大 → 从库同步慢 → 主从延迟
```

## 三、方案一：分区表（必做）✅

### 按月分区

```sql
-- 删除原表重建
DROP TABLE IF EXISTS logs;

CREATE TABLE logs (
    id BIGINT NOT NULL AUTO_INCREMENT,
    content TEXT,
    level INT DEFAULT 0,
    create_time DATETIME NOT NULL,
    PRIMARY KEY (id, create_time),  -- 主键必须包含分区键
    INDEX idx_time (create_time)
) ENGINE=InnoDB
PARTITION BY RANGE (TO_DAYS(create_time)) (
    PARTITION p202401 VALUES LESS THAN (TO_DAYS('2024-02-01')),
    PARTITION p202402 VALUES LESS THAN (TO_DAYS('2024-03-01')),
    PARTITION p202403 VALUES LESS THAN (TO_DAYS('2024-04-01')),
    PARTITION p202404 VALUES LESS THAN (TO_DAYS('2024-05-01')),
    PARTITION p_future VALUES LESS THAN MAXVALUE   -- 兜底未来数据
);
```

### 按天分区（数据量大时）

```sql
-- 每天一个分区，适合 1000 万/天的场景
CREATE TABLE logs (
    id BIGINT NOT NULL AUTO_INCREMENT,
    content TEXT,
    level INT DEFAULT 0,
    create_time DATETIME NOT NULL,
    PRIMARY KEY (id, create_time),
    INDEX idx_time (create_time)
) ENGINE=InnoDB
PARTITION BY RANGE (TO_DAYS(create_time)) (
    PARTITION p20240101 VALUES LESS THAN (TO_DAYS('2024-01-02')),
    PARTITION p20240102 VALUES LESS THAN (TO_DAYS('2024-01-03')),
    -- ... 依此类推 ...
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

### 分区表的优势

```sql
-- 查询最近 7 天的日志
SELECT * FROM logs 
WHERE create_time >= '2024-01-01' 
  AND create_time < '2024-01-08';

-- MySQL 自动只扫描 p20240101 ~ p20240107 这 7 个分区
-- 不扫描其他分区 = 分区裁剪（Partition Pruning）

-- 删除 30 天前的数据（毫秒级）
ALTER TABLE logs DROP PARTITION p20231201, p20231202, ...;
-- 相当于 DROP TABLE，比 DELETE 快 100 倍，不产生 binlog
```

### 自动建分区（存储过程）

```sql
DELIMITER //

CREATE PROCEDURE create_future_partitions()
BEGIN
    DECLARE partition_name VARCHAR(20);
    DECLARE partition_date DATE;
    
    SET partition_date = CURDATE() + INTERVAL 7 DAY;
    SET partition_name = CONCAT('p', DATE_FORMAT(partition_date, '%Y%m%d'));
    
    SET @sql = CONCAT(
        'ALTER TABLE logs REORGANIZE PARTITION p_future INTO (',
        'PARTITION ', partition_name, ' VALUES LESS THAN (TO_DAYS(''', partition_date + INTERVAL 1 DAY, ''')),',
        'PARTITION p_future VALUES LESS THAN MAXVALUE)'
    );
    
    PREPARE stmt FROM @sql;
    EXECUTE stmt;
    DEALLOCATE PREPARE stmt;
END //

DELIMITER ;

-- 定时任务：每天调用一次
CALL create_future_partitions();
```

## 五、方案三：数据归档与清理

### 归档到归档表

```sql
-- 创建归档表（独立存储，可放低配机器）
CREATE TABLE logs_archive LIKE logs;
ALTER TABLE logs_archive ENGINE=MyISAM;  -- 归档表可用 MyISAM（不支持事务，查询更快）

-- 归档存储过程（每晚执行）
DELIMITER //

CREATE PROCEDURE archive_logs()
BEGIN
    DECLARE archive_date DATE DEFAULT DATE_SUB(CURDATE(), INTERVAL 30 DAY);
    
    -- 1. 导出到归档表
    INSERT INTO logs_archive 
    SELECT * FROM logs 
    WHERE create_time < archive_date;
    
    -- 2. 删除原表数据（按分区删除更快）
    DELETE FROM logs WHERE create_time < archive_date;
    
    -- 3. 释放空间
    OPTIMIZE TABLE logs;
END //

DELIMITER ;
```

### 直接删除旧分区（最快）

```sql
-- 删除 30 天前所有分区
ALTER TABLE logs DROP PARTITION 
    p20231201, p20231202, p20231203, ...;

-- 比 DELETE 快 100 倍，不产生 binlog，不阻塞业务
```

## 七、生产级完整实现

### 定时归档任务（Spring Boot）

```java
@Component
@Slf4j
public class LogArchivalTask {

    @Autowired
    private DataSource dataSource;

    // 保留天数（热数据）
    private static final int RETAIN_DAYS = 7;

    // 每晚 2 点执行
    @Scheduled(cron = "0 0 2 * * ?")
    public void archiveOldLogs() {
        log.info("开始归档日志...");
        long start = System.currentTimeMillis();

        try {
            // 归档到历史表
            archiveToHistory();

            // 删除已归档的旧数据
            deleteOldData();

            // 清理归档表旧数据（超过 90 天）
            cleanHistoryData();

            long cost = System.currentTimeMillis() - start;
            log.info("日志归档完成, 耗时={}ms", cost);

        } catch (Exception e) {
            log.error("日志归档失败", e);
            alertService.notify("日志归档失败");
        }
    }

    private void archiveToHistory() {
        String archiveDate = LocalDate.now().minusDays(30).format("yyyy-MM-dd");
        String sql = String.format("""
            INSERT INTO logs_history 
            SELECT * FROM logs 
            WHERE create_time < '%s'
            """, archiveDate);

        JdbcTemplate jdbc = new JdbcTemplate(dataSource);
        int count = jdbc.update(sql);
        log.info("归档日志数量: {}", count);
    }

    private void deleteOldData() {
        String deleteDate = LocalDate.now().minusDays(RETAIN_DAYS).format("yyyy-MM-dd");
        String sql = String.format("""
            DELETE FROM logs 
            WHERE create_time < '%s'
            """, deleteDate);

        JdbcTemplate jdbc = new JdbcTemplate(dataSource);
        int count = jdbc.update(sql);
        log.info("删除旧日志数量: {}", count);
    }
}
```

## 八、面试加分话术

> **面试官追问：分区表和分表的区别？**

> 分区表是 MySQL 自带的功能，一张表逻辑上还是一张，但物理上分成多个文件。应用层不需要改 SQL，MySQL 自动分区裁剪。分表是应用层做的，比如按月份建 logs_202401、logs_202402，应用层根据时间路由到不同的表。分区表更简单，分表更灵活。

> **面试官追问：为什么要定期归档，直接存着不行吗？**

> 不行。MySQL 单表太大后，索引 B+ 树层数变深，查询变慢。DDL 操作（加索引、改字段）会锁全表几小时。备份恢复时间长（TB 级要几小时）。成本高（SSD 贵）。所以要冷热分离，热数据快存快取，冷数据低成本归档。

---

## 一句话总结

**1000 万/天的日志，MySQL 单表扛不住。** 核心套路：**分区表做热数据快速查询 → 定期归档到历史表/ES → 冷数据低成本保留 → 删除旧分区释放空间**。
