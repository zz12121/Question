50. 你的事务代码：
```java
@Transactional
public void transfer(Long from, Long to, BigDecimal amount) {
    accountDao.deduct(from, amount);
    // 发送消息通知
    kafkaTemplate.send("transfer", message);
    accountDao.add(to, amount);
}
```
**问题：** 这段代码有什么问题？如果 Kafka 发送失败会怎样？

---
## 结论先行

**这段代码藏着两个"定时炸弹"：**

1. **事务提交后 Kafka 发送失败** → 扣款成功但通知没发，消息丢失
2. **Kafka 发送成功但后续加款失败** → 钱扣了但没到账，数据不一致

核心问题是**本地事务和消息发送不是原子操作**，解法是**事务消息（本地消息表 / RocketMQ 事务消息）**。

---

## 一、问题分析

### 代码在事务内做了什么？

```java
@Transactional  // 整个方法在一个数据库事务内
public void transfer(Long from, Long to, BigDecimal amount) {
    accountDao.deduct(from, amount);      // ① 扣款（数据库）
    kafkaTemplate.send("transfer", msg);  // ② 发消息（Kafka）
    accountDao.add(to, amount);           // ③ 加款（数据库）
}
```

### 问题一：Kafka 发送失败

```
T1: BEGIN TRANSACTION
T2: accountDao.deduct(from, 100)  → 扣款 SQL 执行成功
T3: kafkaTemplate.send(...)        →  💥 Kafka 发送失败！
T4: 抛出异常
T5: ROLLBACK TRANSACTION           →  扣款回滚 ✅ 没问题
```

**结论：** Kafka 失败时事务回滚，OK，没问题。

### 问题二：事务提交后 JVM 崩溃（更严重！）

```
T1: BEGIN TRANSACTION
T2: accountDao.deduct(from, 100)  →  扣款成功
T3: kafkaTemplate.send(...)        →  Kafka 发送成功（消息在 Broker）
T4: accountDao.add(to, amount)     →  加款成功
T5: COMMIT TRANSACTION             →  数据库事务提交
T6: JVM 崩溃了！
```

**结果：** 钱扣了，加款了，消息也发了，一切正常？

**再来看另一个场景：**

```
T1: BEGIN TRANSACTION
T2: accountDao.deduct(from, 100)  →  扣款成功
T3: kafkaTemplate.send(...)        →  Kafka 发送成功
T4: accountDao.add(to, amount)    →  💥 数据库连接超时！
T5: ROLLBACK TRANSACTION          →  事务回滚，扣款回滚
T6: Kafka 消息已经发出去了
```

**结果：** 扣款回滚了，但消息已经发出去告诉对方"转账成功"，对方系统已经处理了。**数据不一致！**

### 问题三：Kafka 发送在事务内的问题

```java
@Transactional
public void transfer(...) {
    accountDao.deduct(from, amount);      // 扣款
    kafkaTemplate.send("transfer", msg);  // 如果在事务内发送
    accountDao.add(to, amount);           // 加款失败
}
```

**MySQL 默认隔离级别下：**

```
数据库事务提交  ────▶  释放行锁
Kafka 发送      ────▶  消息在 Broker

如果 Kafka 发送成功，但事务还没提交：
  → 消费者收到消息去查数据库 → 数据还没提交 → 查不到

如果事务回滚：
  → 消息已经发送了 → 消费者按消息处理了 → 但数据库数据回滚了
```

## 二、为什么不能把 Kafka 放在事务内？

```
┌─────────────────────────────────────────────────────────────┐
│  本地事务 + 消息发送 的执行顺序：                              │
│                                                             │
│  数据库事务 BEGIN                                            │
│      │                                                      │
│      ├─▶ 执行 SQL（扣款、加款）                               │
│      │                                                      │
│      ├─▶ 发送 Kafka 消息                                    │
│      │     └── 此时消息在内存，还没发送到 Broker             │
│      │                                                      │
│      └─▶ COMMIT / ROLLBACK                                  │
│            └── COMMIT 后：Kafka 消息才真正发送到 Broker      │
│                                                             │
│  问题：COMMIT 后如果 JVM 崩溃，消息不会发                     │
│       但如果 ROLLBACK，消息已经缓存在内存了                   │
└─────────────────────────────────────────────────────────────┘
```

## 三、解决方案

### 方案一：本地消息表（最通用）✅

**核心思想：把消息先存数据库，用数据库事务保证消息和业务原子性，再用定时任务补偿发送**

```
步骤：
1. 扣款 + 写消息表（在同一个事务内）
2. 定时任务扫描消息表，发送 Kafka
3. 发送成功后标记消息已发送
```

**表结构：**

```sql
CREATE TABLE transfer_message (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    transfer_id VARCHAR(64) NOT NULL UNIQUE,  -- 转账唯一标识
    from_user BIGINT NOT NULL,
    to_user BIGINT NOT NULL,
    amount DECIMAL(20,2) NOT NULL,
    status TINYINT DEFAULT 0,  -- 0=待发送, 1=已发送, 2=发送失败
    retry_count INT DEFAULT 0,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_status (status),
    INDEX idx_create_time (create_time)
);
```

**代码实现：**

```java
@Transactional
public void transfer(Long from, Long to, BigDecimal amount) {
    // 1. 扣款
    accountDao.deduct(from, amount);
    
    // 2. 加款
    accountDao.add(to, amount);
    
    // 3. 写消息表（和业务在同一个事务！）
    transferMessageDao.insert(TransferMessage.builder()
        .transferId(UUID.randomUUID().toString())
        .fromUser(from)
        .toUser(to)
        .amount(amount)
        .status(0)  // 待发送
        .build());
    
    // 4. 事务提交后，消息已经在数据库了
    // 定时任务会扫描并发送
}

@Slf4j
@Component
public class MessageSenderTask {

    @Autowired
    private TransferMessageDao messageDao;
    
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    // 每 5 秒扫描一次待发送消息
    @Scheduled(fixedDelay = 5000)
    public void sendPendingMessages() {
        // 1. 查出待发送消息（最多 100 条）
        List<TransferMessage> pending = messageDao.selectByStatus(0, 100);
        
        for (TransferMessage msg : pending) {
            try {
                // 2. 发送 Kafka
                kafkaTemplate.send("transfer", msg.getTransferId(), JSON.toJSONString(msg));
                
                // 3. 标记已发送
                messageDao.updateStatus(msg.getId(), 1);
                
                log.info("消息发送成功, transferId={}", msg.getTransferId());
                
            } catch (Exception e) {
                // 4. 发送失败，重试计数
                messageDao.incrementRetry(msg.getId());
                log.error("消息发送失败, transferId={}, retry={}", 
                    msg.getTransferId(), msg.getRetryCount() + 1);
                
                // 超过最大重试次数，标记失败，人工处理
                if (msg.getRetryCount() >= 3) {
                    messageDao.updateStatus(msg.getId(), 2);  // 2=发送失败
                    alertService.notify("转账消息发送失败，请人工处理");
                }
            }
        }
    }
}
```

### 方案二：RocketMQ 事务消息（最优雅）✅

**核心思想：RocketMQ 提供半消息机制，先发半消息，事务提交后再确认发送**

```
RocketMQ 事务消息流程：

1. 发送半消息（对消费者不可见）
2. 执行本地事务（扣款 + 加款）
3. 提交事务（消息对消费者可见）/ 回滚事务（消息删除）
4. RocketMQ 回调确认本地事务结果
```

**代码实现：**

```java
@Service
@Slf4j
public class TransferService {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    public void transfer(Long from, Long to, BigDecimal amount) {
        String transferId = UUID.randomUUID().toString();

        // 构建消息
        TransferMessage msg = TransferMessage.builder()
            .transferId(transferId)
            .fromUser(from)
            .toUser(to)
            .amount(amount)
            .build();

        // 发送事务消息
        rocketMQTemplate.sendMessageInTransaction(
            "transfer-topic",  // Topic
            MessageBuilder.withPayload(JSON.toJSONString(msg))
                .setHeader("transferId", transferId)
                .build(),
            // 额外参数
            null
        );
    }
}

// TransactionListener 实现
@Component
@Slf4j
public class TransferTransactionListener implements TransactionListener {

    @Autowired
    private AccountDao accountDao;

    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        String body = new String(msg.getPayload());
        TransferMessage transfer = JSON.parseObject(body, TransferMessage.class);

        try {
            // 1. 扣款
            accountDao.deduct(transfer.getFromUser(), transfer.getAmount());
            
            // 2. 加款
            accountDao.add(transfer.getToUser(), transfer.getAmount());
            
            // 3. 提交本地事务
            log.info("本地事务提交成功, transferId={}", transfer.getTransferId());
            return LocalTransactionState.COMMIT_MESSAGE;

        } catch (Exception e) {
            log.error("本地事务回滚, transferId={}", transfer.getTransferId(), e);
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }

    @Override
    public LocalTransactionState checkLocalTransaction(Message msg) {
        // RocketMQ 回调检查本地事务状态
        // 返回：COMMIT_MESSAGE / ROLLBACK_MESSAGE / UNKNOWN
        return LocalTransactionState.COMMIT_MESSAGE;
    }
}
```

```
│                                                                 │
│  Producer              Broker               Consumer             │
│     │                     │                    │               │
│     │──1. 发送半消息──────▶│                    │               │

```

## 四、方案对比

| 方案 | 复杂度 | 适用场景 | 缺点 |
|------|--------|---------|------|
| 本地消息表 | 中 | 所有场景 | 需要额外表 + 定时任务 |
| RocketMQ 事务消息 | 低 | RocketMQ 用户 | 强依赖 RocketMQ |
| Seata AT | 中 | 跨服务分布式事务 | 性能略有下降 |
| 最终一致性 | 高 | 可接受延迟的场景 | 需要完善的补偿机制 |

## 五、面试加分话术

> **面试官追问：本地消息表有什么缺点？**

> 缺点是延迟高一些（定时任务有扫描间隔）。还有就是需要维护额外的消息表，增加存储成本。另外定时任务本身也可能失败，所以消息表要设计重试机制。适合对实时性要求不那么高的场景。

> **面试官追问：RocketMQ 事务消息的原理？**

> 核心是"半消息"机制。发送时先发半消息，对消费者不可见。然后执行本地事务，提交后再确认消息可见。如果本地事务超时未返回，RocketMQ 会回查（check）本地事务状态，决定是提交还是删除。所以能保证本地事务和消息发送的原子性。

---

## 一句话总结

**数据库事务 + Kafka 发送 不是原子操作** —— 事务提交后消息可能没发，事务回滚后消息可能已发。解法是**本地消息表（业务 + 消息同事务） + 定时任务补偿发送**，或直接用 **RocketMQ 事务消息**。
