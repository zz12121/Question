44. 你用 `try-catch` 捕获所有异常：
```java
try {
    // 业务逻辑
} catch (Exception e) {
    // 忽略异常
}
```
**问题：** 这种做法有什么问题？如何正确记录异常信息？

---
## 结论先行

**这种写法是"隐形炸弹"** —— 业务异常被吞掉、问题无法复现、根因无从查起。正确做法是**分层处理异常 + 规范记录日志**，让异常"死得明白"。

---

## 一、这种写法有什么问题？

### 1. 异常被"吞掉"，问题消失于无形

```
用户反馈："我下单失败了"
工程师查日志：什么都没有 ❌
用户内心：？？？
```

异常被 `catch (Exception e)` 抓住后，`e` 对象躺在内存里等 GC，业务代码若无其事地继续往下走。用户看到的是"静默失败"。

### 2. 问题无法复现和定位

```
场景：用户下单时随机失败，每周出现几次
根因：某处 IOException 被吞掉
后果：无法知道在哪抛的、什么时候抛的、上下文是什么
```

没有日志的异常，就像没有监控的服务器 —— 出了事你根本不知道。

### 3. 掩盖真正的 Bug

```java
try {
    Integer result = map.get(key);  // key 可能为 null
    int value = result.intValue();  // NullPointerException 被吞
} catch (Exception e) {
    // ❌ NPE 被忽略了，真正的 bug 永远发现不了
}
```

### 4. 影响业务流程正确性

```java
try {
    // 下单扣库存
    orderService.create(order);
    stockService.deduct(stockId);
} catch (Exception e) {
    // 库存扣了，但订单没创建成功，还以为一切正常
}
```

### 5. 安全风险

异常信息可能包含敏感内容（如 SQL、堆栈），不规范的异常处理可能造成信息泄露。

---

## 二、正确的异常处理方式

### 原则一：永远不要空 catch

```java
// ❌ 错误
try {
    doSomething();
} catch (Exception e) {
    // 什么都没做
}

// ✅ 最低标准：至少记一条日志
try {
    doSomething();
} catch (Exception e) {
    log.error("下单失败, orderId={}", orderId, e);  // 把异常 e 传进去
}
```

### 原则二：分层处理异常

```
┌─────────────────────────────────────────────────────────┐
│  Controller 层：统一异常响应                              │
│  ┌─────────────────────────────────────────────────┐   │
│  │  Service 层：业务异常返回业务错误码                │   │
│  │  ┌─────────────────────────────────────────┐    │   │
│  │  │  DAO/基础层：记录原始异常日志              │    │   │
│  │  └─────────────────────────────────────────┘    │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 原则三：按异常类型分别处理

```java
try {
    // 业务逻辑
} catch (BusinessException e) {
    // 业务异常：可预期的业务错误，记录 info 或 warn 即可
    log.warn("业务校验失败, msg={}", e.getMessage());
    return Result.fail(e.getCode(), e.getMessage());
    
} catch (IOException e) {
    // IO 异常：基础设施问题，记录 error 并告警
    log.error("IO 操作失败, path={}", filePath, e);
    alertService.notify("IO异常告警");
    return Result.fail(500, "系统繁忙，请稍后重试");
    
} catch (Exception e) {
    // 未知异常：最高级别，必须告警
    log.error("未知异常, 请立即处理, orderId={}", orderId, e);
    alertService.notifyP0("P0告警: 未知异常");
    return Result.fail(500, "系统异常");
}
```

### 原则四：异常不能被吞，但可以被转换

```java
// ❌ 禁止：吞掉异常
try {
    userService.update(user);
} catch (Exception e) {
    // 什么日志都没有
}

// ✅ 推荐：转换异常类型，保留根因
try {
    userService.update(user);
} catch (DataAccessException e) {
    // 包装成业务异常，附加上下文
    throw new BusinessException("用户更新失败", "USER_UPDATE_FAILED", e);
}
```

### 原则五：打印异常时要把异常对象传进去

```java
// ❌ 错误：只打字符串，丢失堆栈
log.error("出错了: " + e.getMessage());

// ✅ 正确：把异常对象传入日志框架
log.error("出错了", e);

// 或者用占位符
log.error("订单创建失败, orderId={}", orderId, e);  // SLF4J 规范
```

## 三、最佳实践代码示例

### 统一异常处理（AOP 方式）

```java
// 全局异常处理器（Spring MVC）
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public Result handleBusiness(BusinessException e) {
        log.warn("业务异常: code={}, msg={}", e.getCode(), e.getMessage());
        return Result.fail(e.getCode(), e.getMessage());
    }

    @ExceptionHandler(IOException.class)
    public Result handleIO(IOException e) {
        log.error("IO异常", e);
        alertService.notify("IO异常");
        return Result.fail(500, "系统繁忙");
    }

    @ExceptionHandler(Exception.class)
    public Result handleUnknown(Exception e) {
        log.error("未预期异常", e);
        alertService.notifyP0("P0告警");
        return Result.fail(500, "系统异常");
    }
}
```

### Service 层规范

```java
public class OrderService {
    
    private final Logger log = LoggerFactory.getLogger(getClass());

    public void createOrder(OrderDTO dto) {
        try {
            validateOrder(dto);
            saveOrder(dto);
            sendMessage(dto);
        } catch (BusinessException e) {
            // 业务异常：已知风险，打 warn 不告警
            log.warn("订单创建业务校验失败, orderId={}, msg={}", dto.getOrderId(), e.getMessage());
            throw e;  // 继续抛出，让上层决定响应
        } catch (Exception e) {
            // 系统异常：未知风险，打 error 并附带完整上下文
            log.error("订单创建系统异常, orderId={}, userId={}", dto.getOrderId(), dto.getUserId(), e);
            throw new SystemException("订单创建失败", e);
        }
    }
}
```

## 四、面试加分话术

> **面试官追问：如果有些异常确实不需要处理怎么办？**

```java
// 比如 MDC 清理这种操作，失败也不影响业务
try {
    MDC.clear();
} catch (Exception e) {
    // 可以用 debug 级别记录，不影响业务
    log.debug("MDC清理失败", e);
}
```

> **面试官追问：异常记录有什么性能考虑？**

```java
// ❌ 错误：字符串拼接 + toString()
log.error("错误: " + e);  // e.toString() 可能触发不必要的操作

// ✅ 正确：让日志框架控制
log.error("订单失败", e);  // 延迟求值，关闭日志时不执行 toString()
```

---

## 一句话总结

**异常是系统的"尸体"，日志是"解剖报告"** —— 绝不能让异常"死得无声无息"，要让它"死得明明白白"。
