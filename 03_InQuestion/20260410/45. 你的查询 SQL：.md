# Q45 深分页问题：SQL 优化与最佳实践

## 结论先行

**LIMIT 1000000, 10 = 先扫描 1000010 行，再扔掉前 1000000 行**。数据量越大，性能越差，是典型的"杀敌一千自损八百"写法。

---

## 一、深分页有什么问题？

### 问题拆解

```
SELECT * FROM orders 
WHERE user_id = 100 
ORDER BY create_time DESC 
LIMIT 1000000, 10;
```

**数据库实际在做什么？**

```
第 1 步：全表扫描 / 索引扫描  →  找到所有满足 user_id=100 的行
第 2 步：回表取所有字段       →  每一行都要去主键索引拿完整数据
第 3 步：排序                 →  按 create_time 排序
第 4 步：跳过前100万行         →  ⚠️ 重点！这里在"数数"
第 5 步：取最后 10 行         →  返回给用户
```

**LIMIT 翻页的成本曲线：**

| 页码 | 偏移量 | 数据库扫描行数 | 实际返回 |
|------|--------|---------------|---------|
| 第 1 页 | 0 | 10 | 10 |
| 第 1000 页 | 9990 | 10000 | 10 |
| 第 10001 页 | 100000 | 100010 | 10 |
| **第 100001 页** | **999990** | **1000000** | **10** |

> **结论：** 偏移量越大，扫描的行数越多，性能呈线性衰退。

### 性能瓶颈在哪里？

1. **回表次数多** — 索引只覆盖 `user_id`，`SELECT *` 要回主键索引取所有字段
2. **排序代价高** — 百万级数据排序需要临时表或文件排序
3. **随机 IO 严重** — LIMIT 跳过的行分布不连续，每行都要单独访问

## 二、如何优化深分页？

### 方案一：游标分页（Cursor Pagination）✅ 推荐

**核心思想：不数数，用 ID 游标直接定位**

```sql
-- 首次查询
SELECT * FROM orders 
WHERE user_id = 100 
ORDER BY create_time DESC 
LIMIT 10;

-- 翻页：传入上一页最后一条的 create_time
SELECT * FROM orders 
WHERE user_id = 100 
  AND create_time < '2024-01-15 10:30:00'  -- 上一页的最小时间
ORDER BY create_time DESC 
LIMIT 10;
```

**Java 实现：**

```java
// 游标分页查询
public List<Order> queryOrders(Long userId, LocalDateTime cursor, int pageSize) {
    LambdaQueryWrapper<Order> wrapper = new LambdaQueryWrapper<>();
    wrapper.eq(Order::getUserId, userId)
           .lt(cursor != null, Order::getCreateTime, cursor)  // 游标条件
           .orderByDesc(Order::getCreateTime)
           .last("LIMIT " + pageSize);
    return orderMapper.selectList(wrapper);
}

// 调用方：记住上一页的游标
LocalDateTime cursor = null;
Page<Order> page1 = queryOrders(userId, null, 10);
if (!page1.isEmpty()) {
    cursor = page1.getLast().getCreateTime();  // 记住最后一个
}
Page<Order> page2 = queryOrders(userId, cursor, 10);
```

**优点：** 无论翻到第几页，性能恒定 O(1)
**缺点：** 不能跳页，只能顺序翻

---

### 方案二：延迟关联（Deferred Join）

**核心思想：先用索引定位 ID，再关联回表取字段**

```sql
-- ❌ 原写法（回表次数 = LIMIT偏移 + 条数）
SELECT * FROM orders 
WHERE user_id = 100 
ORDER BY create_time DESC 
LIMIT 1000000, 10;

-- ✅ 优化写法：先拿 ID，再关联
SELECT o.* FROM orders o
INNER JOIN (
    SELECT id FROM orders 
    WHERE user_id = 100 
    ORDER BY create_time DESC 
    LIMIT 1000000, 10
) AS t ON o.id = t.id;
```

**原理：**

```
原来：扫描 1000010 行，每行都回表
现在：子查询只扫描 id（覆盖索引），不回表
      再用 10 个 id 关联主表，只需 10 次回表
```

**性能对比：**

| 方式 | 扫描行数 | 回表次数 |
|------|---------|---------|
| 原写法 | 1,000,010 | 1,000,010 |
| 延迟关联 | 1,000,010 | 10 |

> **注意：** 本质没减少扫描行数，但**大幅减少回表次数**，对 IO 密集型场景有效。

---

### 方案三：范围条件替代 LIMIT 偏移

**核心思想：把"跳页"改成"查范围"**

```sql
-- ❌ 跳页（偏移量越来越大）
SELECT * FROM orders 
WHERE user_id = 100 AND create_time < '2024-01-01'  -- 人为设上界
ORDER BY create_time DESC 
LIMIT 10 OFFSET 1000000;  -- 越往后越慢

-- ✅ 范围条件（查询量恒定）
SELECT * FROM orders 
WHERE user_id = 100 
  AND create_time BETWEEN '2024-01-01' AND '2024-01-31'  -- 用上一页的最大时间
ORDER BY create_time DESC 
LIMIT 10;
```

### 方案四：ElasticSearch（适合超深分页）

当分页深度超过几万页时，MySQL 不管怎么优化都吃力，此时：

```
┌─────────────┐    ┌─────────────┐
│   业务表     │───▶│   ES 索引    │
│  (MySQL)    │ CDC │  (全量同步)  │
└─────────────┘    └─────────────┘
                        │
                        ▼
              ┌─────────────────┐
              │  ES 分页查询     │
              │  支持百万级深度   │
              │  延迟 < 100ms    │
              └─────────────────┘
```

ES 的 `search_after` 支持**无限制深分页**，原理和游标分页类似。

## 三、索引优化：让分页本身更快

确保分页查询命中复合索引：

```sql
-- 为分页查询量身定制索引
CREATE INDEX idx_user_time ON orders(user_id, create_time DESC);
```

**索引覆盖情况分析：**

| 查询字段 | 索引覆盖 |
|---------|---------|
| user_id | ✅ 在索引树中 |
| create_time | ✅ 在索引树中（复合索引第二列） |
| id | ✅ 主键隐含在索引中 |
| status（其他字段） | ❌ 需要回表 |

```sql
-- 如果 SELECT * 回表多，试试只查需要的字段
SELECT id, create_time, amount FROM orders  -- 字段都在索引里
WHERE user_id = 100 
ORDER BY create_time DESC 
LIMIT 1000000, 10;
```

## 四、完整代码示例

```java
/**
 * 分页查询优化：游标 + 延迟关联
 */
@Service
public class OrderService {

    public PageResult<OrderVO> queryOrders(Long userId, PageCursor cursor) {
        int pageSize = 10;
        
        // 方式1：游标分页（推荐，能满足绝大多数翻页需求）
        if (cursor != null && cursor.getCreateTime() != null) {
            return queryByCursor(userId, cursor.getCreateTime(), pageSize);
        }
        
        // 方式2：延迟关联（适合需要跳页的场景）
        return queryByDeferredJoin(userId, cursor != null ? cursor.getOffset() : 0, pageSize);
    }

    // 游标分页
    private PageResult<OrderVO> queryByCursor(Long userId, LocalDateTime cursor, int pageSize) {
        List<Order> orders = orderMapper.selectByUserIdAndCursor(userId, cursor, pageSize + 1);
        boolean hasMore = orders.size() > pageSize;
        if (hasMore) orders = orders.subList(0, pageSize);
        
        return PageResult.<OrderVO>builder()
            .data(convert(orders))
            .hasMore(hasMore)
            .nextCursor(hasMore ? orders.get(pageSize - 1).getCreateTime() : null)
            .build();
    }

    // 延迟关联
    private PageResult<OrderVO> queryByDeferredJoin(Long userId, int offset, int pageSize) {
        // 先拿 ID
        List<Long> ids = orderMapper.selectIdsByUserId(userId, offset, pageSize);
        if (ids.isEmpty()) return PageResult.empty();
        
        // 再用 ID 查完整数据
        List<Order> orders = orderMapper.selectByIds(ids);
        return PageResult.<OrderVO>builder()
            .data(convert(orders))
            .hasMore(orders.size() == pageSize)
            .offset(offset + pageSize)
            .build();
    }
}
```

## 五、面试加分话术

> **面试官追问：为什么游标分页性能好？**

> 因为 LIMIT 偏移量是"数数"操作，必须跳过前面的行才能开始返回。而游标分页用 `WHERE create_time < xxx` 直接定位到起始位置，无论翻到第几页，都只扫描索引树的一小段。

> **面试官追问：ES 的 search_after 和 MySQL 游标分页有什么区别？**

> 本质一样，都是"记住上一页最后一条的位置"。区别是 ES 能支持任意深度的排序后分页，因为它的分布式架构可以并行查多个分片。而 MySQL 单表几千万数据时，延迟关联能扛几万页，但再深就得上 ES 了。

---

## 一句话总结

**翻页越深，数据越少，MySQL 越累。** 解决方案的核心思路是：**不数数，用游标直接定位**。
