53. 你用 Redis 做限流：
```java
public boolean allowRequest(String userId) {
    String key = "limit:" + userId;
    Long count = redis.incr(key);
    if (count == 1) {
        redis.expire(key, 60);
    }
    return count <= 100;
}
```
**问题：** 这段代码有什么问题？`incr` 和 `expire` 之间如果挂了会怎样？

---
## 结论先行

**`incr` 和 `expire` 之间有个时间窗口，如果 Redis 挂了，key 永不过期，用户就能无限请求。** 解法是**用 Lua 脚本保证原子性**，或**用 Redis 的 SET + EXPIRE 二合一命令**。

---

## 一、问题分析

### 代码流程

```java
public boolean allowRequest(String userId) {
    String key = "limit:" + userId;
    Long count = redis.incr(key);     // ① 计数 +1
    if (count == 1) {                // ② 第一次访问，设置过期时间
        redis.expire(key, 60);       // ③ 60 秒过期
    }
    return count <= 100;             // ④ 判断是否超限
}
```

### 问题一：incr 和 expire 之间 Redis 挂了

```
正常流程：

T1: incr(key)     → count=1
T2: expire(key,60)→ 设置 60 秒过期
T3: incr(key)     → count=2
T4: ...           → 继续计数
T5: key 自然过期   → 重置计数

─────────────────────────────────────────────────

异常流程（incr 和 expire 之间 Redis 重启）：

T1: incr(key)     → count=1（写进去了）
T2: Redis 崩溃/重启（还没执行 expire）
T3: 重启完成，key 的 TTL 没了
T4: incr(key)     → count=2
T5: incr(key)     → count=3
T6: ...
T7: key 永远不过期，用户可以无限请求 💥
```

### 问题二：并发请求导致计数不准

```
假设用户第 1 次请求：

T1: incr(key) → count=1
T2: expire(key,60) → 设置过期
T3: expire(key,60) → 再次设置（OK，没问题）

假设两个请求同时到：

T1: 请求A incr → count=1
T2: 请求B incr → count=2
T3: 请求A expire(key,60)（count=2，不等于1，跳过）
T4: 请求B expire(key,60)（count=2，不等于1，跳过）

这种情况还好，最多是 key 没设置过期，但计数器是对的。

真正的坑：如果 incr 成功但 expire 没执行，key 永远不会过期。
```

### 问题三：SETNX + EXPIRE 的竞态

```java
// 有人会这样写（同样有问题）
if (redis.setnx(key, "1")) {   // ① 设置值（原子）
    redis.expire(key, 60);      // ② 设置过期（非原子！）
}
```

---

## 二、为什么单命令是原子的，但组合不是？

```
Redis 单命令（如 INCR, SETNX, EXPIRE）是原子的
Redis 组合命令不是原子的

INC + EXPIRE = 两条命令 = 两个网络往返 = 中间可能失败
```

---

## 三、解决方案

### 方案一：SET + EXPIRE 二合一（最简单）✅

**核心思想：用 SET 命令同时设置值和过期时间**

```java
public boolean allowRequest(String userId) {
    String key = "limit:" + userId;
    
    // 用 SET 的同时设置过期时间
    Long count = redisTemplate.opsForValue().increment(key);
    
    // 首次访问才设置过期（但仍有并发问题）
    if (count != null && count == 1) {
        redisTemplate.expire(key, Duration.ofSeconds(60));
    }
    
    return count != null && count <= 100;
}
```

**问题：** 还是有 incr 和 expire 的时间窗口。

---

### 方案二：Lua 脚本保证原子性 ✅

**核心思想：把 incr 和 expire 写到一个 Lua 脚本里，Redis 执行 Lua 是原子的**

```java
// Lua 脚本：原子执行计数 + 设置过期
String luaScript = 
    "local key = KEYS[1] " +
    "local limit = tonumber(ARGV[1]) " +
    "local expire = tonumber(ARGV[2]) " +
    " " +
    "local count = redis.call('INCR', key) " +
    "if count == 1 then " +
    "    redis.call('EXPIRE', key, expire) " +
    "end " +
    "return count";

// Java 调用
RedisScript<Long> script = RedisScript.of(luaScript, Long.class);
Long count = redisTemplate.execute(script, 
    Collections.singletonList("limit:" + userId),  // KEYS[1]
    "100",     // ARGV[1] 限流阈值
    "60"       // ARGV[2] 过期秒数
);

return count != null && count <= 100;
```

**执行流程：**

```
客户端发送 Lua 脚本
        │
        ▼
Redis 执行 Lua（原子操作）
  ├── INCR key
  ├── if count == 1 then EXPIRE key 60
  └── return count
        │
        ▼
返回结果给客户端

即使 Redis 在执行 Lua 过程中崩溃，命令要么全部执行，要么全部不执行
```

### 方案三：简化版固定窗口限流 ✅

**核心思想：用时间戳做窗口，SET 时直接带过期时间**

```java
public boolean allowRequest(String userId) {
    String key = "limit:" + userId;
    long now = System.currentTimeMillis() / 1000;  // 当前秒
    String windowKey = key + ":" + now;  // 按秒分窗口
    
    // 原子操作：SET + EXPIRE 合并（Redis 6.2+ 支持 SET EX）
    redisTemplate.opsForValue().set(
        windowKey, 
        "0", 
        Duration.ofSeconds(60)
    );
    
    Long count = redisTemplate.opsForValue().increment(windowKey);
    
    return count != null && count <= 100;
}
```

**问题：** 窗口边界的两秒内可以请求 200 次（上一秒 100 + 下一秒 100）。

---

### 方案四：滑动窗口限流（更精确）✅

**核心思想：用 ZSet 记录每个请求的时间戳，精确控制时间窗口**

```java
public boolean allowRequest(String userId) {
    String key = "limit:sliding:" + userId;
    long now = System.currentTimeMillis();
    long windowStart = now - 60_000;  // 60 秒前
    
    Jedis jedis = jedisPool.getResource();
    
    try {
        // 1. 删除窗口外的旧记录
        jedis.zremrangeByScore(key, 0, windowStart);
        
        // 2. 统计当前窗口内请求数
        long count = jedis.zcard(key);
        
        if (count < 100) {
            // 3. 未超限，添加当前请求
            jedis.zadd(key, now, String.valueOf(now));
            jedis.expire(key, 60);
            return true;
        }
        
        return false;
    } finally {
        jedis.close();
    }
}
```

## 四、生产级最佳实践

### 推荐：令牌桶算法（Redis + Lua）

```java
@Service
@Slf4j
public class RateLimiterService {

    private final StringRedisTemplate redisTemplate;

    // 令牌桶 Lua 脚本
    private static final String RATE_LIMIT_SCRIPT = """
        local key = KEYS[1]
        local rate = tonumber(ARGV[1])
        local capacity = tonumber(ARGV[2])
        local now = tonumber(ARGV[3])
        local requested = tonumber(ARGV[4])
        
        -- 获取当前令牌数
        local data = redis.call('HMGET', key, 'tokens', 'last_time')
        local tokens = tonumber(data[1]) or capacity
        local last_time = tonumber(data[2]) or now
        
        -- 计算补充的令牌数
        local elapsed = now - last_time
        local filled = math.min(capacity, tokens + (elapsed * rate))
        
        -- 是否能获取令牌
        if filled >= requested then
            local new_tokens = filled - requested
            redis.call('HMSET', key, 'tokens', new_tokens, 'last_time', now)
            redis.call('EXPIRE', key, math.ceil((capacity - new_tokens) / rate) + 1)
            return {1, new_tokens}
        else
            return {0, filled}
        end
        """;

    public boolean allowRequest(String userId, int limit, int windowSeconds) {
        String key = "ratelimit:token:" + userId;
        long now = System.currentTimeMillis() / 1000;
        double rate = (double) limit / windowSeconds;  // 每秒补充速率
        
        RedisScript<ArrayList> script = RedisScript.of(RATE_LIMIT_SCRIPT, ArrayList.class);
        
        ArrayList<Long> result = redisTemplate.execute(
            script,
            Collections.singletonList(key),
            String.valueOf(rate),     // 每秒生成令牌数
            String.valueOf(limit),   // 桶容量
            String.valueOf(now),     // 当前时间
            "1"                       // 请求 1 个令牌
        );
        
        boolean allowed = result.get(0) == 1;
        log.debug("限流检查, userId={}, allowed={}", userId, allowed);
        
        return allowed;
    }
}
```

### Controller 层使用

```java
@RestController
@Slf4j
public class ApiController {

    @Autowired
    private RateLimiterService rateLimiter;

    @GetMapping("/api/data")
    public Result getData(HttpServletRequest request) {
        String userId = request.getHeader("userId");
        
        // 1 分钟最多 100 次
        if (!rateLimiter.allowRequest(userId, 100, 60)) {
            return Result.fail(429, "请求太频繁，请稍后再试");
        }
        
        return Result.ok(dataService.getData());
    }
}
```

## 六、面试加分话术

> **面试官追问：为什么 Redis 执行 Lua 脚本是原子的？**

> Redis 使用单线程模型，执行 Lua 脚本时，Redis 会阻塞所有其他命令，直到脚本执行完成。所以 Lua 脚本内的所有命令天然是原子的，不会被其他命令插队。这也是 Lua 脚本比多个单独命令更安全的原因。

> **面试官追问：令牌桶和漏桶算法的区别？**

> 令牌桶是"有弹性"的——突发流量可以先把桶里积攒的令牌用完（允许一定程度的突发），然后按固定速率补充。漏桶是"严格平滑"的——无论来多少流量，输出速率恒定，不允许突发。Redis 限流用令牌桶更常见，因为它既能限流又能容忍一定的流量突增。

---

## 一句话总结

**INCR 和 EXPIRE 不是原子组合，中间可能 Redis 挂了，导致 key 永不过期。** 解法是**用 Lua 脚本把计数和过期写成原子操作**，或直接用 Redis 6.2+ 的 `SET key value EX 60 NX` 命令。
