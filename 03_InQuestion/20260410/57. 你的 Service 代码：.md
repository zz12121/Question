```java
@Service
public class OrderService {

    public void create(Order order) {
        orderDao.save(order);
        // 调用库存服务
        inventoryClient.deduct(order.getProductId());
    }
```

**问题：** 这段代码有什么问题？事务边界和远程调用混在一起会有什么后果？

---
用户问的是 Spring 事务和远程调用混在一起的问题。这是一个经典面试题，涉及：

1. 事务边界问题：远程调用在事务内，如果远程调用失败导致事务回滚，数据库操作也会回滚
2. 事务时间过长：远程调用耗时长，导致数据库事务持有的锁时间过长
3. 远程调用失败后的数据一致性问题
4. 解决方案：TCC、Saga、可靠消息等分布式事务模式，或者调整事务边界
## 结论先行

**远程调用放在 `@Transactional` 里，是两个经典问题：事务被拖长 + 远程失败导致回滚范围过大。** 核心问题是**本地事务管不了远程调用的成功与否**，解法是**拆事务或用分布式事务模式（TCC/Saga/消息）**。

---

## 一、问题分析

### 代码流程

```java
@Transactional
public void create(Order order) {
    orderDao.save(order);                    // ① 数据库操作
    inventoryClient.deduct(order.getProductId()); // ② 远程调用（RPC）
}
```

### 问题一：远程失败导致不该回滚的操作被回滚

```
成功路径：
T1: BEGIN TRANSACTION
T2: orderDao.save(order)    →  订单写入数据库
T3: inventoryClient.deduct() →  库存扣减成功
T4: COMMIT ✅

失败路径（问题所在）：
T1: BEGIN TRANSACTION
T2: orderDao.save(order)    →  订单写入数据库
T3: inventoryClient.deduct() →  💥 库存服务超时/异常
T4: 抛出异常                →  Spring 捕获异常
T5: ROLLBACK                →  订单数据也被回滚了 💥
```

**结果：** 库存扣减失败，订单也没了。但实际上可能库存只是临时不可用，用户重试就能成功，结果订单被回滚了。

### 问题二：事务时间被拖长

```
远程调用的耗时：50ms - 500ms（不稳定）
数据库事务持有锁时间 = 远程调用时间

T1: BEGIN TRANSACTION
T2: orderDao.save(order)    →  加行锁（订单行锁）
T3: inventoryClient.deduct() →  等 300ms（库存服务慢）
T4: COMMIT                   →  释放行锁

300ms 内，这行订单数据被锁住
高并发下 → 大量锁等待 → 数据库连接耗尽 → 性能雪崩
```

### 问题三：远程调用的"黑天鹅"

```
库存服务挂了（不可用）
     │
     ├── 事务回滚 → 订单也没了
     ├── 用户看到"下单失败"
     └── 用户重试 → 再失败 → 多次失败 → 用户流失

更糟糕的情况：
库存服务 timeout（而不是明确失败）
     │
     ├── 客户端：超时，当作失败 → 回滚
     ├── 库存服务：实际执行了扣减
     └── 订单：回滚了
     
结果：库存被扣了，但订单没了 → 数据不一致 💥💥💥
```

## 二、解决方案

### 方案一：调整事务边界（先提交，后远程）✅

**核心思想：本地事务先完成，远程调用单独处理**

```
原始：
@Transactional
save(order) ──▶ 远程调用（事务内）──▶ COMMIT

优化后：
save(order) ──▶ COMMIT ──▶ 远程调用（事务外）

如果远程调用失败，不影响已提交的数据
```

```java
@Service
@Slf4j
public class OrderService {

    @Transactional
    public void create(Order order) {
        // 1. 本地事务先完成
        orderDao.save(order);
        
        // 2. 事务已经提交，不在这里调用远程服务
        
        // 3. 发送消息，触发远程调用
        mqTemplate.send("order-created", order);
    }
}

// MQ 消费者处理库存扣减
@Component
@Slf4j
public class OrderMessageConsumer {

    @Autowired
    private InventoryClient inventoryClient;
    
    @RabbitListener(queues = "order-created-queue")
    public void handleOrderCreated(Order order) {
        int retry = 3;
        while (retry > 0) {
            try {
                inventoryClient.deduct(order.getProductId());
                log.info("库存扣减成功, orderId={}", order.getId());
                return;
            } catch (Exception e) {
                retry--;
                log.warn("库存扣减失败，重试, orderId={}, retry={}", order.getId(), retry, e);
                if (retry > 0) Thread.sleep(1000);
            }
        }
        // 超过重试次数，告警人工处理
        alertService.notify("订单库存扣减失败, orderId=" + order.getId());
    }
}
```

**问题：** MQ 消费者处理失败怎么办？需要幂等补偿。

### 方案二：TCC 模式（Try-Confirm-Cancel）✅

**核心思想：两阶段提交，Try 预留资源，Confirm 确认扣减，Cancel 回滚**

```
TCC 流程：
┌─────────────────────────────────────────────────────────────┐
│  Phase 1 - Try（预留资源）                                   │
│    orderService.try(order)                                  │
│      ├── 订单状态改为 "待确认"                               │
│      └── inventoryService.try(productId, count)            │
│            └── 库存预扣（冻结库存）                          │
│                                                             │
│  Phase 2a - Confirm（确认扣减）                              │
│    orderService.confirm(order)                               │
│      ├── 订单状态改为 "已确认"                               │
│      └── inventoryService.confirm(productId, count)         │
│            └── 冻结库存转为已扣减                            │
│                                                             │
│  Phase 2b - Cancel（回滚）                                   │
│    orderService.cancel(order)                                │
│      ├── 订单状态改为 "已取消"                               │
│      └── inventoryService.cancel(productId, count)          │
│            └── 解冻库存                                      │
└─────────────────────────────────────────────────────────────┘
```

**代码实现（Seata TCC 模式）：**

```java
@LocalTCC
public interface OrderTccService {

    @TwoPhaseBusinessAction(
        name = "createOrder",
        commitMethod = "confirm",
        rollbackMethod = "cancel"
    )
    void tryCreate(
        @BusinessActionContextParameter(paramName = "orderId") Long orderId,
        @BusinessActionContextParameter(paramName = "productId") Long productId,
        int count
    );

    void confirm(BusinessActionContext context);
    
    void cancel(BusinessActionContext context);
}

@Service
@Slf4j
public class OrderTccServiceImpl implements OrderTccService {

    @Autowired
    private OrderDao orderDao;
    
    @Autowired
    private InventoryTccService inventoryTccService;

    @Override
    @Transactional
    public void tryCreate(Long orderId, Long productId, int count) {
        // 1. 创建订单（待确认状态）
        Order order = new Order();
        order.setId(orderId);
        order.setStatus("PENDING");
        orderDao.save(order);
        
        // 2. 库存 TCC Try（预扣库存）
        inventoryTccService.tryDeduct(productId, count);
    }

    @Override
    public void confirm(BusinessActionContext context) {
        Long orderId = (Long) context.getActionContext("orderId");
        // 更新订单状态为已确认
        orderDao.updateStatus(orderId, "CONFIRMED");
        log.info("订单确认, orderId={}", orderId);
    }

    @Override
    public void cancel(BusinessActionContext context) {
        Long orderId = (Long) context.getActionContext("orderId");
        Long productId = (Long) context.getActionContext("productId");
        Integer count = (Integer) context.getActionContext("count");
        
        // 更新订单状态为已取消
        orderDao.updateStatus(orderId, "CANCELLED");
        
        // 库存 TCC Cancel（解冻库存）
        inventoryTccService.cancelDeduct(productId, count);
        
        log.info("订单取消, orderId={}", orderId);
    }
}
```

### 方案三：Saga 模式（长事务拆分）✅

**核心思想：把长事务拆成一系列本地事务，每个事务有对应的补偿操作**

```
Saga 模式流程：
┌─────────────────────────────────────────────────────────────┐
│  正向操作：                                                  │
│  createOrder → deductInventory → payment → confirm          │
│                                                             │
│  补偿操作：                                                  │
│  如果任意步骤失败，执行已完成的补偿：                          │
│  createOrder失败 → 无需补偿                                 │
│  deductInventory失败 → refundPayment（如果已付）             │
│  payment失败 → restoreInventory                             │
└─────────────────────────────────────────────────────────────┘
```

**适用场景：** 跨多个服务的操作，不要求强一致性，可以接受最终一致性。

---

### 方案四：最小事务原则 ✅（最简单）

**核心思想：远程调用不要放在事务里，只把最核心的数据库操作包事务**

```java
@Service
@Slf4j
public class OrderService {

    @Transactional
    public void create(Order order) {
        // 只把必须的数据库操作放事务
        orderDao.save(order);
    }

    public void createWithInventory(Order order) {
        // 事务只包含核心操作
        create(order);
        
        // 远程调用在事务外
        try {
            inventoryClient.deduct(order.getProductId(), order.getCount());
        } catch (Exception e) {
            // 远程调用失败，事务已提交，人工补偿或消息队列重试
            log.error("库存扣减失败，需要补偿, orderId={}", order.getId(), e);
            
            // 发送补偿消息
           补偿Service.compensate(order);
        }
    }
}
```

## 三、生产级最佳实践

### 推荐：消息队列 + 本地消息表 + 定时补偿

```
┌─────────────────────────────────────────────────────────────┐
│                   可靠消息最终一致性                          │
│                                                             │
│  1. 事务内：保存订单 + 保存本地消息表                          │
│     BEGIN                                                   │
│     INSERT INTO orders (...)                                 │
│     INSERT INTO messages (orderId, type, status) VALUES (...) │
│     COMMIT                                                  │
│                                                             │
│  2. 定时任务：扫描消息表，发送远程请求                         │
│     SELECT * FROM messages WHERE status=0                   │
│     inventoryClient.deduct(...)                              │
│     UPDATE messages SET status=1 WHERE id=?                 │
│                                                             │
│  3. 如果远程调用失败，重试                                    │
│     超过最大重试 → 告警 → 人工处理                            │
└─────────────────────────────────────────────────────────────┘
```

```java
@Service
@Slf4j
public class OrderService {

    @Autowired
    private OrderDao orderDao;
    
    @Autowired
    private OrderMessageDao messageDao;
    
    @Autowired
    private InventoryClient inventoryClient;

    @Transactional
    public void createOrder(Order order) {
        // 1. 创建订单
        orderDao.save(order);
        
        // 2. 记录消息（和订单在同一个事务！）
        OrderMessage msg = new OrderMessage();
        msg.setOrderId(order.getId());
        msg.setType("DEDUCT_INVENTORY");
        msg.setStatus(0);  // 待处理
        msg.setRetryCount(0);
        messageDao.save(msg);
        
        // 3. 事务提交后，消息已经在数据库了
    }

    // 定时任务处理消息
    @Scheduled(fixedDelay = 5000)
    public void processMessages() {
        List<OrderMessage> pending = messageDao.selectByStatus(0, 100);
        
        for (OrderMessage msg : pending) {
            try {
                if ("DEDUCT_INVENTORY".equals(msg.getType())) {
                    inventoryClient.deduct(msg.getOrderId());
                    messageDao.updateStatus(msg.getId(), 1);  // 已处理
                }
            } catch (Exception e) {
                messageDao.incrementRetry(msg.getId());
                log.error("消息处理失败, msgId={}", msg.getId(), e);
                
                if (msg.getRetryCount() >= 3) {
                    messageDao.updateStatus(msg.getId(), 2);  // 失败
                    alertService.notify("库存扣减失败, orderId=" + msg.getOrderId());
                }
            }
        }
    }
}
```

|------|--------|--------|------|---------|
| 拆事务（MQ） | 最终一致 | 中 | 高 | ✅ 大多数场景 |
| TCC | 强一致 | 高 | 中 | 跨服务资源预留 |
| Saga | 最终一致 | 高 | 高 | 长事务链路 |

## 五、面试加分话术

> **面试官追问：为什么不能用分布式事务解决所有问题？**

> 分布式事务（2PC/TCC）虽然保证强一致，但代价很大：性能损耗 30%-50%、实现复杂、运维成本高。所以不是所有场景都要强一致。比如下单+发消息，用最终一致性 + 补偿机制就够了。只有真正需要强一致的核心业务（如支付）才用分布式事务。

> **面试官追问：Saga 和 TCC 的区别？**

> TCC 是"预留-确认-取消"，需要业务提供 Try/Confirm/Cancel 三个接口，适合资源预留场景（如库存冻结）。Saga 是"正序执行 + 逆向补偿"，每个步骤有对应的补偿操作，没有 Try 阶段，实现更简单，适合不需要资源预留的长事务链路。

---

## 一句话总结

**远程调用放在 `@Transactional` 里，是"事务范围过大 + 远程失败污染本地"的双重问题。** 解法是**把远程调用拆到事务外面，用消息队列保证最终一致性**，只有真正需要强一致的核心业务才用 TCC/Saga。
