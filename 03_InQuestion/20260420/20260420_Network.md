303. 以下代码实现了一个简单的TCP连接池，但存在线程安全和连接泄漏问题。请改进：

```java
public class TcpConnectionPool {
    private List<Socket> freeConnections = new ArrayList<>();  // 非线程安全
    private List<Socket> usedConnections = new ArrayList<>();  // 非线程安全
    private int maxConnections = 100;
    
    public Socket getConnection(String host, int port) throws IOException {
        if (!freeConnections.isEmpty()) {
            Socket conn = freeConnections.remove(0);  // 可能已关闭
            usedConnections.add(conn);
            return conn;
        }
        if (usedConnections.size() < maxConnections) {
            Socket conn = new Socket(host, port);  // 无超时
            usedConnections.add(conn);
            return conn;
        }
        throw new RuntimeException("连接池已满");  // 应该等待而非直接抛异常
    }
    
    public void releaseConnection(Socket conn) {
        usedConnections.remove(conn);
        freeConnections.add(conn);  // 可能添加已关闭的连接
    }
    // 没有连接有效性检查 / 没有空闲连接清理 / 没有超时管理
}
```

问题分析：ArrayList非线程安全 / 连接有效性未检查 / 无超时 / 无空闲清理 / 泄漏风险 / 改进方案：

```java
public class TcpConnectionPool {
    private final BlockingQueue<Socket> pool = new LinkedBlockingQueue<>();
    private final AtomicInteger activeCount = new AtomicInteger(0);
    private final int maxConnections;
    private final int connectTimeout;
    private final int readTimeout;
    
    public Socket borrowConnection(String host, int port, long waitMs) 
            throws IOException, InterruptedException {
        Socket conn = pool.poll();
        if (conn != null && !conn.isClosed() && conn.isConnected()) {
            activeCount.incrementAndGet();
            return conn;
        }
        if (activeCount.get() < maxConnections) {
            Socket newConn = new Socket();
            newConn.setSoTimeout(readTimeout);
            newConn.connect(new InetSocketAddress(host, port), connectTimeout);
            activeCount.incrementAndGet();
            return newConn;
        }
        conn = pool.poll(waitMs, TimeUnit.MILLISECONDS);  // 等待归还
        if (conn != null && !conn.isClosed()) {
            activeCount.incrementAndGet();
            return conn;
        }
        throw new SocketTimeoutException("获取连接超时");
    }
    
    public void returnConnection(Socket conn) {
        activeCount.decrementAndGet();
        if (conn != null && !conn.isClosed()) {
            pool.offer(conn);
        } else {
            try { conn.close(); } catch (IOException e) { }
        }
    }
}
```

问题分析：异常路径没有关闭Socket / 连接泄漏 / try-with-resources修复：

```java
public void handleRequest(Socket socket) {
    try (Socket s = socket;  // try-with-resources
         InputStream in = s.getInputStream();
         OutputStream out = s.getOutputStream()) {
        byte[] data = readRequest(in);
        byte[] response = process(data);
        out.write(response);
        out.flush();
    } catch (IOException e) {
        log.error("处理请求失败", e);
        // try-with-resources自动关闭socket
    }
}
```

311. 以下代码实现了HTTP重试逻辑，但存在幂等性风险——POST请求也被重试了：

```java
public String httpPost(String url, String body) {
    int maxRetries = 3;
    for (int i = 0; i < maxRetries; i++) {
        try {
            HttpPost post = new HttpPost(url);
            post.setEntity(new StringEntity(body));
            try (CloseableHttpResponse resp = httpClient.execute(post)) {
                return EntityUtils.toString(resp.getEntity());
            }
        } catch (IOException e) {
            if (i == maxRetries - 1) throw new RuntimeException(e);
            // 无条件重试！POST请求可能已经执行成功了！
        }
    }
    return null;
}
```

问题分析：POST非幂等 / 网络超时不代表服务端没收到 / 重试可能导致重复执行 / 修正方案：

```java
public String httpPost(String url, String body) {
    // POST请求不自动重试，由调用方决定
    HttpPost post = new HttpPost(url);
    post.setEntity(new StringEntity(body, ContentType.APPLICATION_JSON));
    try (CloseableHttpResponse resp = httpClient.execute(post)) {
        int status = resp.getStatusLine().getStatusCode();
        if (status >= 200 && status < 300) {
            return EntityUtils.toString(resp.getEntity());
        }
        throw new RuntimeException("HTTP " + status);
    } catch (IOException e) {
        // 不重试POST，抛出异常让调用方处理
        throw new RuntimeException(e);
    }
}

// GET请求可以重试
public String httpGet(String url) {
    // 幂等操作可以重试
    HttpRequestRetryHandler retryHandler = (exception, retryCount, context) -> {
        if (retryCount > 2) return false;
        // 只对GET等幂等请求重试
        HttpRequest request = (HttpRequest) context.getAttribute("http.request");
        return request instanceof HttpGet;
    };
}
```

329. 设计一个WebSocket消息推送系统的网络架构，要求支持50万在线连接、消息不丢失、低延迟推送。

设计要点：
- 架构：客户端→CDN（可选）→L4负载均衡→WebSocket集群→消息服务→MQ
- L4负载均衡：LVS/HAProxy / TCP层转发 / 不解析HTTP / 连接数均衡
- WebSocket集群：Netty NIO / 每节点支撑5万连接 / 10个节点 / 主从Reactor模式
- 连接管理：Redis存储 uid→serverId 映射 / 用户上线/下线更新映射
- 消息路由：消息服务根据uid查Redis→找到目标server→推送
- 消息可靠性：消息先写MQ（Kafka）→消费→推送→ACK→删除 / 未ACK消息重试
- 离线消息：用户不在线时消息存Redis/MongoDB / 上线后拉取离线消息
- 心跳：30秒间隔 / 3次失败断开 / 自动重连
- 监控：在线连接数 / 推送延迟P99 / 消息到达率 / 消息丢失率
- 扩容：新增节点→新连接路由到新节点→旧连接自然过期 / 一致性哈希分配

330. 设计一个跨地域多活系统的网络架构，要求用户就近接入、数据最终一致性、故障自动切换。

设计要点：
- 整体架构：GSLB→Region1（北京）/ Region2（上海）/ Region3（广州）
- GSLB（全局负载均衡）：基于用户IP就近路由 / 健康检查自动切换 / DNS层面调度
- 每个Region内部：Nginx→API Gateway→微服务→数据库主从
- 数据同步：MySQL双向复制 / Canal监听binlog→MQ→异地域写入 / 冲突解决：时间戳+业务规则
- 一致性：最终一致性 / 关键操作同步复制 / 非关键操作异步复制
- 故障切换：GSLB检测Region不可用→DNS切换到备用Region / TTL=60秒快速切换
- 数据安全：跨Region数据加密传输 / 合规：数据不出境
- 挑战：数据冲突 / 同步延迟 / 网络分区 / 成本
- 监控：Region健康状态 / 同步延迟 / 冲突率 / 切换耗时
