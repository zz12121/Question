## 1. 基础流程

### 1.1 JIT 分层编译的整体架构

```
JVM 分层编译架构（Tiered Compilation）：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│   Java 代码 (.class)                                         │
│        │                                                     │
│        ▼                                                     │
│   ┌─────────────┐                                           │
│   │  解释执行   │  ← Tier 0（初始阶段，无编译开销）        │
│   │   (字节码)  │     方法被调用时直接由 Interpreter 执行   │
│   └──────┬──────┘                                           │
│          │ 调用计数器达到阈值                               │
│          ▼                                                  │
│   ┌─────────────┐                                           │
│   │  C1 编译    │  ← Tier 1（快速编译，适合启动阶段）      │
│   │  (Client)  │     无额外优化，仅做基础优化，编译快      │
│   └──────┬──────┘     生成机器码，立即生效                  │
│          │ 调用计数器 + 回边计数器双重阈值触发             │
│          ▼                                                  │
│   ┌─────────────┐                                           │
│   │ C1 + Profiling │  ← Tier 2（带性能剖析的编译）         │
│   │  (高级优化前) │     收集分支概率、类型信息等            │
│   └──────┬──────┘                                           │
│          │ 热点检测 + profiling 数据足够                     │
│          ▼                                                  │
│   ┌─────────────┐                                           │
│   │ C2 编译     │  ← Tier 3 / Tier 4（极限优化，适合稳态）│
│   │  (Server)  │     激进优化：内联、去虚拟化、逃逸分析    │
│   └──────┬──────┘     生成最高效机器码                      │
│          │                                                     │
│          ▼                                                     │
│      机器码执行 ◄── OSR（栈上替换，运行时替换正在执行的方法）│
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 C1 编译器 vs C2 编译器的区别

| 维度 | **C1（Client Compiler）** | **C2（Server Compiler）** |
|---|---|---|
| **全称** | C1 Compiler (JVM Client Compiler) | C2 Compiler (JVM Server Compiler) |
| **优化目标** | 快速编译，减少启动时间 | 峰值性能，极致优化 |
| **编译耗时** | 毫秒级（快） | 秒~分钟级（慢） |
| **生成代码质量** | 中等 | 极高 |
| **适用阶段** | 启动阶段、预热阶段 | 稳态运行阶段 |
| **优化技术** | 基础内联、常量折叠 | 激进内联、逃逸分析、热点版本化 |
| **JVM 参数** | `-client`（已被移除，JDK 10 后仅 Server） | `-server` |
| **编译触发阈值** | 1,500 次（简单阈值） | 10,000 次（双重阈值） |

```bash
# 分层编译相关参数（JDK 8+）
-XX:+TieredCompilation           # 开启分层编译（默认开启）
-XX:TieredStopAtLevel=1          # 最高只编译到 Tier 1（快速启动）
-XX:TieredCompilation            # 同上
-XX:CompileThreshold=10000       # C2 编译阈值（默认）
-XX:+PrintCompilation            # 打印编译日志（调试用）
-XX:+UnlockDiagnosticVMOptions
-XX:+LogCompilation              # 详细编译日志
```

```java
// 实际代码观察 JIT 编译行为
public class JITObserve {
    static long sum = 0;

    // 这个方法会被 JIT 优化
    public static long hotMethod(int n) {
        long s = 0;
        for (int i = 0; i < n; i++) {

        // $ jstack <pid> 或 arthas jad 命令可看到哪些方法被编译了

        Thread.sleep(1000);
        System.out.println("预热完成，JIT 已编译 hotMethod");
    }
}
```

### 1.3 OSR（On-Stack Replacement，栈上替换）

```
OSR 的核心问题：
┌──────────────────────────────────────────────────────────────┐
│  传统编译：方法整个执行完才能切换到优化版本                   │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  方法开始执行（解释器）                                │   │
│  │         │                                              │   │
│  │  调用 10000 次 ←── 没达到编译阈值                     │   │
│  │         │                                              │   │
│  │  第 10001 次：达到阈值 → 开始 C2 编译...             │   │
│  │                      (编译需要 0.5 秒)                 │   │
│  │                      (这 0.5 秒里还在解释执行！)      │   │
│  │  0.5 秒后：C2 版本可用，但方法已经在执行了！          │   │
│  │                      ↓                                 │   │
│  │         ← 需要 OSR 才能平滑过渡到 C2 版本              │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  OSR = 在方法执行中途，用优化版本替换解释版本                 │
│       不是等方法结束，而是在"栈上"直接替换                    │
└──────────────────────────────────────────────────────────────┘
```

```java
// OSR 的触发场景（解释执行 → 编译版本）
public class OSRDemo {
    public static void main(String[] args) {
        // 循环体是 OSR 的典型场景（方法入口只有一次）
        // 但 for 循环体内的热点可能非常多
        
        for (int i = 0; i < 100_000; i++) {
            // 这个循环本身不触发 OSR（方法未切换）
            // OSR 发生在：方法执行中，性能计数器达到阈值
            compute(i);
        }
    }

    static double compute(int x) {
        double result = 0;
        // 这里是真正的热点，循环 10 万次
        for (int i = 0; i < 10; i++) {
            result += Math.sin(x) * Math.cos(i);
        }
        return result;
    }

    // 实际上 JIT 会把这个方法内联到 main() 的循环里
    // 然后在内联版本上做 OSR
}
```

**OSR 的工作流程**：
1. 解释器执行时，热点计数器持续累加
2. 达到阈值，C2 开始在后台编译
3. 编译完成后，触发 OSR：拦截下一次循环/方法调用
4. 在栈帧层面，用新机器码替换旧解释器帧
5. 继续执行，但现在是优化后的机器码

## 2. 扩展追问

### 🔥 扩展追问 1：C2 编译器用了哪些激进优化技术？

> **A:** C2 的激进优化是面试重点，以下是核心技术的完整解析：

```java
// 技术1：方法内联（Method Inlining）
public class InliningDemo {
    // 无内联：方法调用有栈帧开销
    static int add(int a, int b) {
        return a + b;
    }
    static int compute(int x) {
        return add(add(x, 1), add(x, 2));  // 每次调用有开销
    }
    // 内联后：add() 的逻辑直接展开，消除了方法调用开销
    // 编译后等价于：
    static int compute_inlined(int x) {
        return (x + 1) + (x + 2);  // 无方法调用
    }
}
```

```
内联的决策规则（JVM）：
┌──────────────────────────────────────────────────────────────┐
│  满足以下条件才内联：                                        │
│  1. 方法体 < 35 字节（净机器码）→ 强制内联                 │
│  2. 方法体 < 325 字节 → 基于调用点热度决定                 │
│  3. 虚方法调用 → 看类型 profile（热点类型直接去虚化）       │
│  4. CHA（Class Hierarchy Analysis）确认只有一个子类 → 内联│
└──────────────────────────────────────────────────────────────┘
```

```java
// 技术2：去虚化（Devirtualization）—— 内联的孪生技术
abstract class Animal { abstract String speak(); }

class Dog extends Animal { 
    String speak() { return "汪"; } 
}

class Cat extends Animal { 
    String speak() { return "喵"; } 
}

Animal a = new Dog();
a.speak();  // 解释器：查虚表，多一次间接跳转

// JIT 收集 profiling：
// 发现 99% 的调用处，a 实际类型是 Dog
// → 热点版本化：直接调用 Dog.speak()（无虚表查找）
// → 性能提升 30%+（间接跳转 → 直接跳转）
```

```java
// 技术3：逃逸分析（Escape Analysis）
public String createString() {
    StringBuilder sb = new StringBuilder();  // 分配在堆上
    sb.append("hello");
    sb.append("world");
    return sb.toString();
}

// 逃逸分析发现：
// - sb 的作用域局限在 createString() 方法内
// - sb 从不逃逸出方法
// → 编译器做"标量替换"（Scalar Replacement）：
public String createString_optimized() {
    // sb 被拆解成独立的标量：s1, s2, s3...
    // 全部在栈上分配，无需堆分配
    String s1 = "hello";
    String s2 = "world";
    String s3 = s1 + s2;  // 栈上拼接
    return s3;
}

// JDK 14 之后的逃逸分析改进：
// - 同步消除（Lock Elision）：sb 在方法内不逃逸 → 内部锁无效 → 消除
// - 栈上分配（Stack Allocation）：sb 不逃逸 → 直接在栈上分配
```

**逃逸分析的 3 种状态**：

| 逃逸状态 | 含义 | JVM 优化 |
|---|---|---|
| **No Escape** | 不逃逸出方法 | 栈上分配 + 同步消除 |
| **Arg Escape** | 作为参数传给其他方法 | 保守优化 |
| **Global Escape** | 逃逸到堆/线程外 | 无优化 |

### 🔥 扩展追问 2：为什么 JDK 10 移除了 `-client` 参数？

> **A:** 原因有三个历史背景：

```
JDK Client/Server 模式的演变：
┌──────────────────────────────────────────────────────────────┐
│  JDK 5/6/7：                                                │
│    - client VM（轻量，启动快，优化差）                       │
│    - server VM（启动慢，优化强）                            │
│    → 桌面应用用 client，服务器用 server                    │
│                                                              │
│  JDK 8（分层编译引入后）：                                  │
│    - client 模式也支持分层编译                              │
│    - 分层编译的 Tier 1 ≈ 原 client 编译                    │
│    - 分层编译的 Tier 3/4 ≈ 原 server 编译                  │
│    - client 模式的独立存在价值大幅降低                      │
│                                                              │
│  JDK 10+（JVM 移除 client）：                               │
│    - GraalVM 作为实验性的 JIT 替代品出现                   │
│    - 统一架构：用 `-XX:+UseC2` 切换 C2                      │
│    - Graal: `-XX:+UseGraalCC`                             │
│    - 分层编译自动处理：C1 阶段快速，T2-T4 阶段深度优化     │
│    - 结论：单独的 client VM 变得多余                        │
└──────────────────────────────────────────────────────────────┘
```

**分层编译的收益**：
- 启动时间：比纯 server 模式快 30-50%（Tier 1 替代了解释器）
- 稳态性能：不输于传统 server 模式
- 代码量：一个 JVM 实现，维护成本降低

### 🔥 扩展追问 4：什么是逆优化（Deoptimization）？为什么 C2 可能需要回退？

> **A:** 逆优化是 C2 激进优化的"后悔药"——当优化假设被打破时，降级回解释器：

```
逆优化的触发场景：
┌──────────────────────────────────────────────────────────────┐
│  场景1：类型 profile 被打破（最常见）                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ C2 认为：a 99% 是 Dog 类型 → 热点版本化内联          │   │
│  │ 第 10001 次：a 突然变成了 Cat！                      │   │
│  │ → 热点假设打破 → 逆优化 → 回退解释执行              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  场景2：类加载打破 CHA                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ C2 认为：Animal 只有 Dog 一个子类 → 内联后去掉分支   │   │
│  │ 新加载了 Cat 类！→ CHA 不成立 → 逆优化              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  场景3：去优化计数达到阈值                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 频繁逆优化 → JIT 停止优化该方法 → 永久解释执行      │   │
│  │ 或降级到 Tier 1 编译                                  │   │
│  └─────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

```java
// 逆优化代码示例
public class DeoptExample {
    // 初始 profiling：100% 是 String
    public static void process(Object obj) {
        String s = (String) obj;  // C2 假设永远不抛 ClassCastException
        System.out.println(s.length());
    }

    public static void main(String[] args) {
        for (int i = 0; i < 100_000; i++) {
            process("hello");  // 一直都是 String → C2 优化
        }
        // 突然传入 Integer
        process(12345);  // ClassCastException！
        // → C2 检测到异常 → 逆优化标记
        // → 后续调用回到解释器执行
        // → 重新收集 profile → 未来某次可能再次 C2 优化
    }
}
```

## 3. 一句话总结

> **JVM 的分层编译让方法先快速解释执行，再逐步升级到 C1 快速编译、T2 profiling、C2 深度优化，通过 OSR 在方法执行中途无缝切换到优化版本——C2 的激进优化（内联、逃逸分析、去虚化）在带来极致性能的同时，也依赖 profiling 数据，一旦假设被打破就会触发逆优化回退到解释器。**
