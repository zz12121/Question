## 1. 基础流程

### 1.1 两个方法的定义与核心区别

```
CompletableFuture<T> orTimeout(long timeout, TimeUnit unit)
CompletableFuture<T> completeOnTimeout(T value, long timeout, TimeUnit unit)
```

两者都是 **Java 9** 引入的非阻塞超时手段，本质都是注册一个定时任务，到期后尝试"抢夺" Completion 的结果：

| | `orTimeout()` | `completeOnTimeout()` |
|---|---|---|
| **超时后行为** | 抛 `TimeoutException` | 用指定值完成 future |
| **返回类型** | `CompletableFuture<T>`（原对象） | `CompletableFuture<T>`（原对象） |
| **异常处理** | 触发 `exceptionally` / `handle` | 不抛异常，正常返回替代值 |
| **典型场景** | "超过 N 秒就算失败" | "超时就用默认值降级" |

```java
// 场景1：调用第三方接口，3秒超时则抛异常
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> callExternalApi())
    .orTimeout(3, TimeUnit.SECONDS);  // 超时 → TimeoutException

future.exceptionally(ex -> {
    // 降级逻辑
    return "default_response";
});

// 场景2：查缓存，2秒没查到就用空列表
CompletableFuture<List<Item>> cached = CompletableFuture
    .supplyAsync(() -> queryCache())
    .completeOnTimeout(Collections.emptyList(), 2, TimeUnit.SECONDS);
```

### 1.2 内部实现原理

两个方法的底层都借助了 **`DelimiterBasedFrameDecoder` 风格的定时线程**（实际上是 ForkJoinPool 的 `commonPool` 或默认 `Executor`）：

```java
// 伪代码，揭示核心逻辑
public CompletableFuture<T> orTimeout(long timeout, TimeUnit unit) {
    if (result == null) {  // 尚未完成才注册
        Timeout ts = new Timeout(); // 内部类，Runnable
        Delayer.delay(ts, timeout, unit);
    }
    return this;
}

// Timeout 任务核心逻辑
class Timeout implements Runnable {
    public void run() {
        // 关键：compareAndSetNode 尝试"抢先"完成
        // 如果此时原 future 已完成，cas 失败，timeout 任务静默失效
        completeExceptionally(new TimeoutException());
    }
}
```

> **核心结论**：定时任务是 **"尝试完成"** 而非 **"强制覆盖"**——如果原 future 已经完成，cas 失败，timeout 任务静默失效，不会产生任何影响。

### 1.3 和手动 `schedule + future.complete()` 的区别

```
手动方式（需自行管理线程池）:
┌─────────────────────────────────────────────────────────────┐
│  schedule(() -> future.complete(defaultVal), 3s)            │
│         ↓                                                   │
│  问题1: 手动创建定时器，需自行管理生命周期                   │
│  问题2: future 如果提前完成，定时任务仍会执行（需额外判断）   │
│  问题3: 若在原线程中 complete，定时任务可能"迟到"执行        │
└─────────────────────────────────────────────────────────────┘

orTimeout / completeOnTimeout:
┌─────────────────────────────────────────────────────────────┐
│  原future.orTimeout(3s) / completeOnTimeout(val, 3s)       │
│         ↓                                                   │
│  优点1: 利用 ForkJoinPool.commonPool()，无需额外线程池      │
│  优点2: 内部 CAS 判断，future 已完成则 timeout 任务自动失效 │
│  优点3: 代码简洁，意图清晰                                  │
└─────────────────────────────────────────────────────────────┘
```

```java
// 手动实现 orTimeout 的等价代码（理解原理用）
CompletableFuture<String> manualOrTimeout(CompletableFuture<String> cf, 
                                          long timeout, TimeUnit unit) {
    ScheduledFuture<?> scheduled = 
        SchedulableExecutor.schedule(
            () -> cf.completeExceptionally(new TimeoutException()),
            timeout, unit
        );
    
    cf.whenComplete((result, ex) -> {
        // 原 future 完成后取消定时任务（避免"迟到执行"）
        if (!scheduled.cancel(false)) {
            // 定时任务已执行，cf 已由 timeout 完成了，此处无需处理
        }
    });
    return cf;
}
```

| 对比维度 | `orTimeout()` / `completeOnTimeout()` | 手动 schedule + complete |
|---|---|---|
| **线程池管理** | 自动，复用 commonPool | 需手动管理，容易泄漏 |
| **已完成判断** | 内部 CAS 自动处理 | 需额外 if 判断或 cancel |
| **可取消性** | timeout 任务不可取消 | 可 `scheduled.cancel(true)` |
| **异常类型** | 固定 `TimeoutException` | 可自定义异常类型 |
| **JDK 版本** | Java 9+ | 任意版本 |

### 1.4 如果 CompletableFuture 已经完成，再调用 orTimeout 会怎样？

```java
CompletableFuture<String> cf = CompletableFuture.completedFuture("done");

// 已在完成状态，无论调用哪个方法都无任何效果
cf.orTimeout(3, TimeUnit.SECONDS);     // no-op
cf.completeOnTimeout("fallback", 3, TimeUnit.SECONDS);  // no-op

System.out.println(cf.join());  // 输出: done
```

**原因**：两个方法内部都有 `if (result == null)` 的守卫判断——future 已完成时 `result != null`，定时任务根本不会被注册，调用形同 no-op。

```java
// JDK 源码中的守卫逻辑
public CompletableFuture<T> orTimeout(long timeout, TimeUnit unit) {
    if (result == null) {  // ← 守卫：如果已 result != null，直接返回
        // 注册 Timeout 定时任务...
    }
    return this;
}
```

## 2. 扩展追问

### 🔥 扩展追问 1：如果 timeout 任务和原任务几乎同时完成，会发生什么？

> **A:** 这是一个典型的 **race condition** 问题。结果取决于 CAS 谁先成功：

```java
CompletableFuture<String> cf = new CompletableFuture<>();

cf.orTimeout(1, TimeUnit.MILLISECONDS);  // 注册 timeout 任务

// 模拟：原任务和 timeout 几乎同时触发
new Thread(() -> cf.complete("real_result")).start();
```

- 如果原任务 **先** `complete()` → `result` 被设为 "real_result"，后续 timeout 任务的 `completeExceptionally` CAS 失败，静默失效。**最终结果 = "real_result"**
- 如果 timeout **先** 完成 → `completeExceptionally` CAS 成功，设置 `TimeoutException`。原任务后续的 `complete("real_result")` **失败**（无法覆盖已完成的 future）。**最终结果 = TimeoutException**

> **结论**：`CompletableFuture` 的 completion 是 **不可逆的**，一旦完成就固化。哪个先到，哪个生效。

---

### 🔥 扩展追问 2：`completeOnTimeout` 的超时值能覆盖原始结果吗？

> **A:** 不能。和上面同理——超时任务是"尝试完成"，不是"强制覆盖"：

```java
CompletableFuture<String> cf = CompletableFuture.supplyAsync(() -> {
    sleep(500); // 原任务 0.5 秒后完成
    return "real_result";
});

cf.completeOnTimeout("fallback", 2, TimeUnit.SECONDS);  // 2 秒后超时
// 原任务 0.5s 完成 → cf 已固化 → 2s 时的 complete("fallback") 无效

System.out.println(cf.join());  // 输出: real_result
```

**但有一种情况可以覆盖**——用 `obtrudeValue()` / `obtrudeException()`，不过这两个方法是 **deprecated**，面试中可以主动提：

```java
// 强制覆盖（不推荐，破坏 future 语义）
cf.obtrudeValue("forced_value");  // 无论是否完成都强制设置
```

### 🔥 扩展追问 3：如果没有显式指定线程池，`orTimeout` 用的是哪个线程池？

> **A:** 默认使用 **ForkJoinPool.commonPool()**，这是一个 JVM 内置的共享线程池：

```java
// 内部实际调用路径
private static final Executor asyncPool = ForkJoinPool.commonPool();

public CompletableFuture<T> orTimeout(long timeout, TimeUnit unit) {
    if (result == null) {
        // 使用 commonPool 调度定时任务
        Delayer.delay(new Timeout(this, completion), timeout, unit);
    }
    return this;
}
```

```java
// ForkJoinPool.commonPool() 的特点
System.out.println(ForkJoinPool.commonPoolParallelism());  // CPU cores - 1

// 问题：如果所有任务把 commonPool 占满，timeout 任务也会被阻塞
// 解决：orTimeout 有自己的 Delayer 线程（ScheduledExecutorService）
//       timeout 任务是单独调度，不依赖 commonPool 的 worker 线程
```

**关键点**：`Delayer` 使用的是独立的 `ScheduledExecutorService`，**不依赖** `ForkJoinPool` 的工作线程，所以 timeout 任务本身不会被 commonPool 阻塞。

### 🔥 扩展追问 4：如何实现带超时的链式调用？（多个 CompletableFuture 串联）

> **A:** 结合 `anyOf()` 或 `allOf()` 可以优雅地实现：

```java
// 场景：同时查多个数据源，返回最快那个的结果
List<CompletableFuture<String>> futures = Arrays.asList(
    CompletableFuture.supplyAsync(() -> queryDB()).orTimeout(1, SECONDS),
    CompletableFuture.supplyAsync(() -> queryRedis()).orTimeout(500, MILLISECONDS),
    CompletableFuture.supplyAsync(() -> callHTTP()).orTimeout(3, SECONDS)
);

CompletableFuture.anyOf(futures.toArray(new CompletableFuture[0]))
    .exceptionally(ex -> "all_failed")
    .thenAccept(System.out::println);
```

### 🔥 扩展追问 5：`orTimeout` 和 `get(timeout, unit)` 的区别是什么？

> **A:** 两者都能实现超时，但本质不同：

```java
// 方式1：get + timeout（阻塞调用端）
try {
    future.get(3, TimeUnit.SECONDS);  // 阻塞等待，最多等3秒
} catch (TimeoutException e) {
    // 仅影响当前调用线程，future 本身仍然 incomplete
}

// 方式2：orTimeout（注册式，异步安全）
future.orTimeout(3, TimeUnit.SECONDS);  // 非阻塞，3秒后 future 自动失败
future.thenAccept(result -> { /* 后续处理 */ });
```

| 维度 | `get(timeout, unit)` | `orTimeout()` |
|---|---|---|
| **阻塞方式** | 阻塞调用线程等待结果 | 非阻塞，提前注册超时处理 |
| **超时后 future 状态** | 仍为 incomplete | 标记为 exceptional |
| **异常传播** | 抛 `TimeoutException` 给调用方 | future 变为 exceptional 状态 |
| **适用场景** | 同步 API 调用 | 异步流水线、链式调用 |

---

## 3. 一句话总结

> **orTimeout() 超时抛异常、completeOnTimeout() 超时降级值，两者通过内部 CAS 判断实现"已完成则忽略"，且底层借助独立 Delayer 调度，**不是**强制覆盖。**

## 4. 面试加分话术

> "我在项目里遇到过这样的场景：调用下游营销接口，RT 平均 200ms，但 P99 可能到 5 秒。当时就是用 `orTimeout(3, SECONDS)` 做快速失败，同时配了 Hystrix 熔断——超时率超过阈值就直接熔断 10 秒，保护上游服务不被拖死。另外我还注意到 `completeOnTimeout` 特别适合做**多级降级**：Redis 查不到用本地缓存，本地缓存也没有就给一个空列表兜底，比直接抛异常用户体验好很多。"

> "关于竞态问题，我之前踩过一个坑——有个定时任务在 timeout 后还要查日志，本来 timeout 任务已经设了 TimeoutException，但原任务 1ms 后也完成了，两个 completion 同时到达。最后排查下来发现 `orTimeout` 的 Delayer 是单独线程，而原任务在 commonPool 里，取决于调度顺序，结果不确定。后来加了 `whenComplete` 来做最终一致性确认。"
