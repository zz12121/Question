For MySQL 8.0+, internal temporary tables are stored in TempTable storage engine by default, but can fall back to InnoDB when needed, especially for certain operations like DISTINCT, UNION, or queries with GROUP BY/ORDER BY clauses that exceed memory thresholds.

The internal storage engine selection depends on several factors: if temp_tablespace runs out, large results overflow to InnoDB; for BLOBs and TEXT columns, InnoDB handles them better; older MySQL versions would use MyISAM, but since MySQL 5.7 and later, InnoDB is the primary engine for internal temporary tables. MySQL 8.0 introduced TempTable as the default engine instead, with InnoDB as a fallback option.

The key distinction is that MyISAM is a disk-based engine without transaction support, while InnoDB handles larger datasets and certain column types more efficiently. In modern MySQL, internal temporary tables created by the server use TempTable or InnoDB depending on the situation, whereas user-created temporary tables can be explicitly defined with any storage engine.

### 🔥 扩展追问 1：MySQL 8.0 的 TempTable 引擎和 InnoDB 在内部临时表上的核心区别是什么？

> **A:** TempTable 是 MySQL 8.0 专为临时数据设计的存储引擎，和 InnoDB 的定位完全不同：

```
TempTable vs InnoDB 用于内部临时表的对比：
┌──────────────────────────────────────────────────────────────┐
│  维度         │  TempTable           │  InnoDB              │
│  ─────────────────────────────────────────────────────────  │
│  存储位置      │  内存 + 磁盘溢出    │  仅磁盘              │
│  ─────────────────────────────────────────────────────────  │
│  内存占用      │  innodb_temp_tablespace_size（默认96MB）│  无限制  │
│  溢出触发      │  超过内存阈值       │  无内存              │
│  ─────────────────────────────────────────────────────────  │
│  BLOB/TEXT 支持│  ✅ 支持（溢出到磁盘）│  ✅ 支持              │
│  ─────────────────────────────────────────────────────────  │
│  事务支持      │  ❌ 无事务           │  ✅ 支持 ACID         │
│  ─────────────────────────────────────────────────────────  │
│  并发性能      │  较高（无锁优化）    │  中等（行锁开销）   │
│  ─────────────────────────────────────────────────────────  │
│  崩溃恢复      │  ❌ 不恢复           │  ✅ 依赖 redo log     │
│  ─────────────────────────────────────────────────────────  │
│  适用场景      │  短生命周期临时数据   │  需要持久化的临时表 │
└──────────────────────────────────────────────────────────────┘
```

```sql
-- TempTable 的内存管理特点
-- TempTable 使用"内存映射文件"（mmap）实现内存 + 溢出
-- 当数据量 < innodb_temp_tablespace_size → 完全在内存中
-- 当数据量 > innodb_temp_tablespace_size → 自动溢出到 .ibt 文件

-- 旧版本：MyISAM 处理 BLOB/TEXT → 不安全、不支持事务
```

### 🔥 扩展追问 2：什么情况下会触发磁盘临时表（Disk Temporary Table）？如何避免？

> **A:** 磁盘临时表的产生通常意味着查询需要优化，以下是完整触发条件和优化策略：

```
磁盘临时表的触发场景：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  触发条件（MySQL 8.0+）：                                    │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  1. 结果集大小 > innodb_temp_tablespace_size        │   │
│  │     → TempTable 溢出 → InnoDB 磁盘表                │   │
│  │                                                       │   │
│  │  2. 查询涉及 BLOB/TEXT 列                           │   │
│  │     → TempTable 列压缩存储（超过阈值溢出 InnoDB）  │   │
│  │                                                       │   │
│  │  3. 显式指定 ENGINE=MyISAM                          │   │
│  │     → （MySQL 5.7 默认 fallback）                   │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  MySQL 5.7 额外触发条件（已废弃）：                         │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  UNION 结果集 > tmp_table_size                       │   │
│  │  GROUP BY / ORDER BY 列总长度 > tmp_table_size       │   │
│  │  → 使用 MEMORY 引擎（内存），超限 → MyISAM（磁盘）  │   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

```sql
-- 监控磁盘临时表
SHOW GLOBAL STATUS LIKE 'Created_tmp%';
-- 
-- Created_tmp_tables: 1000   ← 总临时表数
-- Created_tmp_disk_tables: 100  ← 磁盘临时表数
-- 比例 = 100/1000 = 10% → 偏高，建议 < 5%

-- 典型优化案例：

-- ❌ 触发磁盘临时表的查询
EXPLAIN SELECT DISTINCT content FROM articles 
WHERE create_time >= '2026-01-01';
-- content = TEXT 列 → 强制磁盘临时表

-- ✅ 优化方案1：用主键 DISTINCT 代替
EXPLAIN SELECT DISTINCT id FROM articles 
WHERE create_time >= '2026-01-01';
-- id 是 INT → 可能走内存临时表

-- ✅ 优化方案2：改写查询，延迟加载 BLOB
EXPLAIN SELECT DISTINCT a.id FROM articles a
INNER JOIN (
    SELECT id FROM articles WHERE create_time >= '2026-01-01'
) t ON a.id = t.id;
-- 先在子查询中完成去重，再关联

-- ✅ 优化方案3：增加内存临时表空间
SET GLOBAL innodb_temp_tablespaces_size = 1073741824;  -- 1GB
```

## 3. 一句话总结

> **MySQL 的临时表分为用户显式创建的（`CREATE TEMP TABLE`，会话隔离，自动清理）和 MySQL 内部隐式创建的（用于 UNION/GROUP BY+ORDER BY/DISTINCT 等），MySQL 8.0 之前内部临时表会退化为 MyISAM 磁盘表，8.0 之后用 TempTable 引擎内存优先 + 溢出 InnoDB，比 MyISAM 更安全更快——但 BLOB/TEXT 列和超大数据量仍会触发磁盘溢出，此时应该优化查询而非单纯调参。**

## 4. 面试加分话术

> "我之前遇到过一个慢查询，排查发现一个 UNION 查询产生了大量的磁盘临时表，`Created_tmp_disk_tables` 占了 30%，每次查询慢 2 秒。分析后发现 UNION 的一个分支有个 TEXT 字段，强制触发了磁盘临时表。后来用 CTE 重写了那个查询（MySQL 8.0），把 TEXT 列的聚合放到子查询里，最后的结果集就只剩下 INT 列了，磁盘临时表消失，查询从 2 秒降到 80 毫秒。这个案例后来成了我们团队的查询优化教材。"

> "关于临时表在主从复制中的问题，我之前踩过一个坑：有个定时任务在主库存储过程里创建临时表做数据聚合，然后用 JOIN 临时表做最终结果——主库执行正常，但从库一直报'临时表不存在'。最后查文档才发现是因为主库的临时表对从库的 SQL 线程不可见。我们后来全面废弃了存储过程中的临时表，改用 CTE，复制问题彻底消失，而且 CTE 代码可读性也更好。"
