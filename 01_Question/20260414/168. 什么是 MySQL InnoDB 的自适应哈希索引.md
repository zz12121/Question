## 1. 基础流程

### 1.1 AHI 的核心定位

```
InnoDB 的索引结构全景图：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  InnoDB 表的存储 = B+ Tree 索引结构（聚簇索引）            │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │           B+ Tree Root Page (根节点)                  │   │
│  │                   │                                    │   │
│  │         ┌─────────┴─────────┐                         │   │
│  │    ┌────▼────┐         ┌───▼───┐                    │   │
│  │    │ 索引列值 │         │ 索引列值│                    │   │
│  │    │  [主键]  │         │ [主键] │                    │   │
│  │    └────┬────┘         └───┬───┘                    │   │
│  │         │                   │                          │   │
│  │    ┌────▼───────────────────▼────┐                   │   │
│  │    │     B+ Tree Leaf Pages       │                   │   │
│  │    │  [完整行数据 + 主键指针]       │                   │   │
│  │    └─────────────────────────────┘                   │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  Adaptive Hash Index（AHI）—— B+ 树之上的"加速层"   │   │
│  │                                                       │   │
│  │  ┌───────────────────────────────────────────────┐   │   │
│  │  │  key (索引列值)  →  value (B+ Tree Page指针) │   │   │
│  │  │  "张三" (name='张三') → Page #42 的 Slot 3   │   │   │
│  │  │  "李四" (name='李四') → Page #87 的 Slot 1   │   │   │
│  │  └───────────────────────────────────────────────┘   │   │
│  │                                                       │   │
│  │  本质：内存中的哈希表，key=索引列值，value=数据页地址│   │
│  │  作用：对于热点等值查询，直接 O(1) 定位，无需树查找│   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 AHI 的构建机制

```sql
-- AHI 是 InnoDB 自动管理的，无需手动创建
-- InnoDB 监控所有 B+ 树索引的使用情况
-- 如果某些热点被频繁访问 → 自动构建对应的 AHI 条目

-- 查看 AHI 状态
SHOW ENGINE INNODB STATUS\G

-- 关键输出：
-- -------------------------------------
-- INSERT BUFFER AND ADAPTIVE HASH INDEX
-- -------------------------------------
-- Ibuf: size 1, free list len 0, seg size 2, 0 merges
-- merged operations:
--  insert 0, delete mark 0, delete 0
-- discards 0, insert 0, delete mark 0, delete 0
-- Hash table size 553, node heap has 0 buffer(s)
-- Hash table size: 553      ← AHI 哈希表的大小（槽数）
-- 0.00 hash searches/s      ← 每秒哈希命中次数
-- 98.00 b-tree searches/s   ← 每秒 B+树命中次数
-- hash searches: 0, non-hash searches: 0
```

```
AHI 构建的工作原理：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  InnoDB 维护一个全局哈希表：                                │
│  - 命名空间：daph_hash_index_t                               │
│  - 存储位置：Buffer Pool 内部（不是独立的内存区域）        │
│  - 结构：链表数组（类似 Java 的 HashMap）                  │
│                                                              │
│  哈希函数的输入：                                           │
│  hash(index_id, index_key_value)                             │
│  - index_id：B+树索引的唯一标识                            │
│  - index_key_value：索引列的具体值                         │
│                                                              │
│  哈希表的 value = B+ Tree Page Pointer                      │
│  即：AHI 指向的是 B+ Tree Leaf Page 中的具体记录槽位        │
│                                                              │
│  查询流程：                                                 │
│  SELECT * FROM orders WHERE order_id = 12345                │
│       ↓                                                      │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  1. 计算 hash("order_id", 12345)                    │   │
│  │  2. 查 AHI 表：O(1) 找到 Page #42 Slot #7          │   │
│  │  3. 直接读取 Page #42 Slot #7 → 命中！               │   │
│  │  4. 全程无需遍历 B+ Tree                             │   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

### 1.3 AHI 和 B+ 树的关系

```
AHI 不是 B+ 树的副本，而是独立的哈希索引：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  常见误解：                                                 │
│  "AHI = B+ Tree 索引的副本，哈希化了整棵树"               │
│  ❌ 错误！                                                   │
│                                                              │
│  正确理解：                                                 │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  1. AHI 只索引被频繁访问的热点键                     │   │
│  │     （InnoDB 自动选择，不会索引整棵树）              │   │
│  │                                                       │   │
│  │  2. AHI 索引的是"值→页"的映射                      │   │
│  │     而非"值→完整行"                                 │   │
│  │                                                       │   │
│  │  3. AHI 属于 Buffer Pool 内部管理                   │   │
│  │     与 B+ 树文件结构完全独立                         │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  两者协作关系：                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  等值查询（WHERE idx_col = 'xxx'）                   │   │
│  │     → 优先走 AHI（O(1)）                             │   │
│  │     → 找不到则回退 B+ Tree（O(log N)）              │   │
│  │                                                       │   │
│  │  范围查询（WHERE idx_col > 'xxx'）                   │   │
│  │     → 必须走 B+ Tree（无法利用哈希）                 │   │
│  │     → AHI 完全没有帮助                               │   │
│  │                                                       │   │
│  │  前缀匹配（WHERE idx_col LIKE 'xxx%'）               │   │
│  │     → 可以走 B+ Tree（有序）                         │   │
│  │     → AHI 效率较低（哈希无序）                       │   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

## 2. 扩展追问

### 🔥 扩展追问 1：什么情况下 AHI 反而会拖慢性能？

> **A:** AHI 在高并发和频繁更新的场景下会产生严重的锁竞争和内存开销：

```
陷阱一：AHI 锁竞争（最严重）
┌──────────────────────────────────────────────────────────────┐
│  InnoDB 的 AHI 是全局共享的：                               │
│                                                              │
│  AHI 结构内部使用：                                          │
│  - rw_lock（读写锁）：保护 AHI 哈希表的并发访问             │
│  - btr_search_latch：全局 AHI 锁                             │
│                                                              │
│  高并发场景：                                                │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  1000 个并发连接，每个都在做等值查询                  │   │
│  │  所有线程都要竞争 AHI 的 btr_search_latch            │   │
│  │  → 全局锁成为瓶颈                                    │   │
│  │  → 锁等待时间 >> 查询本身的时间                       │   │
│  │  → QPS 反而下降                                      │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  症状：                                                       │
│  SHOW ENGINE INNODB STATUS 中：                             │
│  "Hash searches: XXX" 非常低（说明 AHI 被竞争抵消了优势） │
│  或者系统 CPU sys% 突然升高（锁竞争）                       │
└──────────────────────────────────────────────────────────────┘
```

```
陷阱二：AHI 维护开销（频繁更新导致重建）
┌──────────────────────────────────────────────────────────────┐
│  每次索引列值变化时：                                       │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  UPDATE orders SET order_id = 99999 WHERE id = 1     │   │
│  │       ↓                                              │   │
│  │  InnoDB 需要：                                       │   │
│  │  1. 删除 AHI 中旧值: (index_id, 1) → old_page        │   │
│  │  2. 插入 AHI 中新值: (index_id, 99999) → new_page  │   │
│  │  3. B+ Tree 的叶子页标记为 dirty（正常）             │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  如果是高频更新：                                           │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  UPDATE counter SET cnt = cnt + 1 WHERE id = 1       │   │
│  │       ↓                                              │   │
│  │  每次更新：cnt 从 1→2→3→4→5...                     │   │
│  │  AHI 每次都要删除旧条目、插入新条目                  │   │
│  │  → AHI 的维护开销 > 查询节省的开销                  │   │
│  │  → AHI 变成纯负担                                   │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  症状：                                                       │
│  "Innodb adaptive hash index size" 持续波动                 │
│  大量 "hash searches" 但系统 CPU 高                         │
└──────────────────────────────────────────────────────────────┘
```

```
陷阱三：AHI 内存占用
┌──────────────────────────────────────────────────────────────┐
│  AHI 存储在 Buffer Pool 中：                                 │
│  - 每个 AHI 条目 ≈ 56 字节（索引指针 + 链表节点）           │
│  - AHI 默认最大 = innodb_buffer_pool_size / 64              │
│  - 如果 Buffer Pool 80GB → AHI 最大 1.25GB                  │
│                                                              │
│  如果 AHI 索引了太多不必要的数据：                          │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  AHI 占用了 Buffer Pool 中宝贵的 Page 缓存空间        │   │
│  │  → 本来可以缓存更多数据页的空间被 AHI 占用           │   │
│  │  → Buffer Pool 命中率下降                             │   │
│  │  → 整体性能下降                                      │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  AHI 的空间开销公式：                                       │
│  AHI 大小 ≈ 索引列值数量 × 56 字节                         │
│  例如：order_id 列有 1 亿条记录                            │
│  AHI 最大 = 1亿 × 56B ≈ 5.6GB                             │
└──────────────────────────────────────────────────────────────┘
```

### 🔥 扩展追问 2：InnoDB 是如何决定哪些索引值得构建 AHI 的？

> **A:** InnoDB 使用自适应策略，基于访问模式动态决定：

```sql
-- InnoDB 内部有一个"搜索计数器"
-- 每个 B+ Tree 页面被访问时，计数器 +1
-- 如果某个页面的搜索次数超过阈值 → 为该页面构建 AHI 条目

-- 关键参数（可调）：
SHOW VARIABLES LIKE 'innodb_adaptive_hash_index';

-- 自适应策略的伪代码逻辑：
/*
  function onPageAccess(index_id, key_value, page_ptr):
      // 1. 尝试命中 AHI
      hash_entry = ahi_lookup(index_id, key_value)
      if hash_entry != null:
          return hash_entry.page_ptr  // AHI 命中
      
      // 2. AHI 未命中，查询 B+ Tree
      result = btree_search(index_id, key_value)
      
      // 3. 记录这次访问
      search_counter[page_ptr]++
      
      // 4. 如果页面访问次数达到阈值，添加到 AHI
      if search_counter[page_ptr] >= AHI_BUILD_THRESHOLD:
          ahi_insert(index_id, key_value, page_ptr)
      
      return result
*/
```

### 🔥 扩展追问 3：如何关闭或调整 AHI？

> **A:** 有几种方式，取决于你的需求：

```sql
-- 方式1：会话级动态关闭（MySQL 8.0+）
SET SESSION innodb_adaptive_hash_index = OFF;
SET GLOBAL innodb_adaptive_hash_index = OFF;

-- 注意：动态关闭后，已存在的 AHI 不会被立即清除
-- 需要重启或等待 AHI 自然过期

-- 方式2：启动参数关闭（永久生效）
-- my.cnf / my.ini
[mysqld]
innodb_adaptive_hash_index = 0

-- 方式3：观察后再决定是否关闭
-- 通过监控判断是否需要关闭：
SHOW ENGINE INNODB STATUS\G

-- 关键指标分析：
-- hash searches: 0, non-hash searches: 100000
-- 说明：AHI 使用率极低，B+ Tree 查询是主流
-- → 建议关闭 AHI

-- hash searches: 80000, non-hash searches: 20000
-- 说明：AHI 命中率 80%，效果很好
-- → 保持开启
```

```sql
-- AHI 调优参数（MySQL 8.0+）
SHOW VARIABLES LIKE 'innodb_adaptive_hash_index';

-- MySQL 8.0 引入了分区 AHI（Partitioned Adaptive Hash Index）
-- 目的：减少锁竞争
-- innodb_adaptive_hash_index_parts = 8（默认）
-- AHI 被分成 8 个分区，每个分区有独立的锁
-- 高并发下，锁竞争降低为 1/8

-- 如果仍然有 AHI 锁竞争问题：
SET GLOBAL innodb_adaptive_hash_index_parts = 16;
-- 或更高（最大 256）
SET GLOBAL innodb_adaptive_hash_index_parts = 256;
```

### 🔥 扩展追问 4：MySQL 8.0 的分区 AHI 解决了什么问题？

> **A:** MySQL 8.0 之前 AHI 是单点锁，分区 AHI 是关键性能改进：

```
MySQL 5.7 的 AHI 锁架构（单锁瓶颈）：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│           ┌─────────────────────────┐                       │
│           │   btr_search_sys_t      │                       │
│           │   (AHI 全局锁)           │                       │
│           │   btr_search_latch      │ ← 所有线程竞争这一把锁 │
│           └───────────┬─────────────┘                       │
│                       │                                       │
│         ┌─────────────┼─────────────┐                       │
│         ↓             ↓             ↓                       │
│    ┌─────────┐   ┌─────────┐   ┌─────────┐                  │
│    │ Thread1 │   │ Thread2 │   │Thread N │                  │
│    └─────────┘   └─────────┘   └─────────┘                  │
│                                                              │
│  问题：N 个线程竞争 1 把锁 → 并发度 = 1                    │
└──────────────────────────────────────────────────────────────┘

MySQL 8.0 的分区 AHI（锁分区）：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│           ┌─────────────────────────┐                       │
│           │   btr_search_sys_t      │                       │
│           │   (AHI 分区管理器)       │                       │
│           └───────────┬─────────────┘                       │
│         ┌─────────────┼─────────────┐                       │
│         ↓             ↓             ↓                       │
│   ┌───────────┐ ┌───────────┐ ┌───────────┐                 │
│   │Partition 0│ │Partition 1│ │Partition 7│  ← 8 个分区   │
│   │Latch[0]   │ │Latch[1]   │ │Latch[7]   │                 │
│   └───────────┘ └───────────┘ └───────────┘                 │
│         ↓             ↓             ↓                       │
│    ┌─────────┐   ┌─────────┐   ┌─────────┐                  │
│    │ Thread1 │   │ Thread2 │   │Thread N │                  │
│    └─────────┘   └─────────┘   └─────────┘                  │
│                                                              │
│  改进：每个分区有独立的锁                                    │
│  线程1 访问 Partition 0 的同时，线程2 访问 Partition 1    │
│  → 并发度 = 8                                               │
│  → innodb_adaptive_hash_index_parts 默认 = 8                │
└──────────────────────────────────────────────────────────────┘
```

### 🔥 扩展追问 5：AHI 和 InnoDB Buffer Pool 的关系是什么？

> **A:** AHI 占用 Buffer Pool 的内存，并且依赖 Buffer Pool 中的 Page：

```
AHI 的内存来源和生命周期：
┌──────────────────────────────────────────────────────────────┐
│  AHI 内存来源：                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  AHI 不是独立分配的内存                               │   │
│  │  AHI 条目存储在 Buffer Pool 的 Page 中               │   │
│  │                                                       │   │
│  │  Buffer Pool 的 Page 类型：                          │   │
│  │  ├─ Data Page：存储实际行数据                       │   │
│  │  ├─ Index Page：存储 B+ Tree 索引                   │   │
│  │  └─ AHI Page：存储 AHI 哈希表条目                   │   │
│  │                                                       │   │
│  │  AHI 最大内存 = Buffer Pool × innodb_buffer_pool_size│   │
│  │  / innodb_page_size × AHI_ENTRY_SIZE               │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  AHI Page 的淘汰：                                         │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  AHI Page 同样受 Buffer Pool LRU 淘汰机制管理        │   │
│  │                                                       │   │
│  │  如果一个 AHI Page 长时间未被访问 → 被 LRU 淘汰     │   │
│  │  该 Page 上的所有 AHI 条目全部失效                   │   │
│  │                                                       │   │
│  │  如果被淘汰的 Page 重新被访问：                       │   │
│  │  → InnoDB 重新为该 Page 构建 AHI 条目               │   │
│  │  → 形成"构建 → 淘汰 → 重建"的循环                  │   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

## 3. 一句话总结

> **AHI 是 InnoDB 在 Buffer Pool 内维护的一个内存哈希表，对热点索引列的等值查询提供 O(1) 定位、跳过 B+ 树搜索的能力，但它的维护本身有锁竞争和内存开销——高并发锁争抢、频繁更新导致频繁重建、以及占用 Buffer Pool 空间导致数据页缓存不足时，AHI 可能从加速器变成性能瓶颈，此时可以动态关闭或调大 `innodb_adaptive_hash_index_parts` 来缓解。**

## 4. 面试加分话术

> "我之前遇到过一个问题，线上 MySQL QPS 突然从 8 万降到 2 万，CPU 的 sys% 从 5% 飙升到 40%，但业务流量没有明显变化。排查了一圈发现是因为高峰期有个定时任务批量更新了 `user_login` 表的登录时间字段，这个字段恰好有索引，而每次更新都触发了 AHI 条目的删除和重建——AHI 的维护开销反而超过了它带来的收益。最后把 `innodb_adaptive_hash_index` 关闭了，QPS 恢复，同时后来改用低峰期执行批量更新，彻底解决。"

> "关于 AHI 的选型，我的经验是：等值查询多（JOIN、WHERE id = xxx）、数据分布均匀、高并发读多 → AHI 效果很好；范围查询多、高并发写、更新频率极高 → 关闭 AHI 或调小分区数。MySQL 8.0 的分区 AHI (`innodb_adaptive_hash_index_parts`) 是业界的一个好改进，如果还在用 MySQL 5.7 且遇到 AHI 锁竞争问题，升级到 8.0 是最省事的解法。"
