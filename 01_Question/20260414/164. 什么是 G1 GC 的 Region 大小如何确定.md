### 1.1 G1 的 Region 是什么

### 1.2 Region 大小如何确定

```java
// Region 大小的核心规则
// G1 的 Region 大小 = 2 的幂次方，必须满足以下约束：

// 约束1：Region 大小必须在 1MB ~ 32MB 之间
// 约束2：必须等于 2^N (N >= 0)
// 约束3：总堆大小必须能被 Region 大小整除

// Region 大小计算公式：
// region_size = max(1MB, round_down(heap_size / 2048))
// 
// 其中 2048 = G1 的最大 Region 数量（固定）
// 实际实现中，JVM 会选择最接近的 2 的幂次方
```

```bash
# Region 大小与堆大小的关系（实际示例）
┌─────────────────────────────────────────────────────────────┐
│  堆大小         | Region 大小 | Region 数量                  │
│  ─────────────────────────────────────────────────────────  │
│  256MB          |   1MB     |   2048 (256 / 1 * 1 = 256但实际2048?)  │
│  1GB (1024MB)   |   2MB     |   512                         │
│  4GB            |   2MB     |   2048 (4G / 2M = 2048)       │
│  8GB            |   4MB     |   2048 (8G / 4M = 2048)      │
│  16GB           |   8MB     |   2048 (16G / 8M = 2048)    │
│  32GB           |   16MB    |   2048 (32G / 16M = 2048)   │
│  64GB           |   32MB    |   2048 (64G / 32M = 2048)   │
│                                                              │
│  ⚠️ 注意：堆太小（< 1GB）时，Region 数量可能 < 2048        │
│  因为 Region 大小必须 >= 1MB                                │
└─────────────────────────────────────────────────────────────┘

# JVM 参数控制
-XX:G1HeapRegionSize=4MB   # 显式指定 Region 大小
# 注意：必须是 2 的幂次方，范围 1MB ~ 32MB
```

**Region 大小的选择策略**：

| 堆大小 | 推荐 Region 大小 | 理由 |
|---|---|---|
| < 4GB | 1MB 或 2MB | 大对象较多，小 Region 减少内存浪费 |
| 4GB - 16GB | 4MB 或 8MB | 平衡停顿时间和内存粒度 |
| 16GB - 64GB | 16MB | 大堆需要大 Region 减少 RSet 开销 |
| > 64GB | 32MB | Region 上限，减少 RSet 内存开销 |

### 1.3 CSet（Collection Set，回收集合）

```
CSet 的定义和作用：
┌──────────────────────────────────────────────────────────────┐
│  CSet = 在一次 GC 中将被回收的所有 Region 的集合            │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  Young GC 的 CSet：                                    │   │
│  │  ┌────┬────┬────┬────┬────┐                            │   │
│  │  │Eden│Eden│ S  │ S  │    │   ← Eden + Survivor 全部   │   │
│  │  │ ❌ │ ❌ │ ❌ │ ❌ │    │     进入 CSet                │   │
│  │  └────┴────┴────┴────┴────┘                            │   │
│  │  Old Gen 区域不在 Young GC 的 CSet 中                 │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  Mixed GC 的 CSet：                                    │   │
│  │  ┌────┬────┬────┬────┬────┬────┬────┐                │   │
│  │  │Eden│Eden│ S  │ S  │ O  │ O  │ O  │                │   │
│  │  │ ❌ │ ❌ │ ❌ │ ❌ │ ✅ │ ✅ │ ✅ │   ← Old Region   │   │
│  │  └────┴────┴────┴────┴────┴────┴────┘     按优先级选择│   │
│  │  Young + 部分的 Old Region 进入 CSet                 │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  CSet 的选择依据（Mixed GC）：                              │
│  1. Region 的垃圾密度（回收价值 = 垃圾量 / Region大小）    │
│  2. 目标停顿时间（由 -XX:MaxGCPauseMillis 控制）           │
│  3. Mixed GC 预算（每个 cycle 最多回收 Old 区的比例）       │
└──────────────────────────────────────────────────────────────┘
```

### 1.4 RSet（Remembered Set，记亿集）

```
RSet 的核心作用：跨 Region 引用的快速追踪
┌──────────────────────────────────────────────────────────────┐
│  G1 的并发标记需要知道：哪些对象引用了本 Region 的对象？   │
│  如果每次都扫描整个堆，GC 停顿时间不可控。                   │
│  RSet 就是来解决这个问题的！                                │
│                                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Region A (Old)        Region B (Eden)              │   │
│  │  ┌─────────┐            ┌─────────┐                 │   │
│  │  │ objectA │◄──────────│ objectB │  ← B → A 引用   │   │
│  │  └─────────┘            └─────────┘                 │   │
│  │       ↑                                                  │   │
│  │       │                                                  │   │
│  │  Region A 的 RSet 中记录：{ Region B }                 │   │
│  │  只需扫描 Region B → 找到对 Region A 的引用           │   │
│  │  无需扫描整个堆！                                       │   │
│  └─────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

## 2. 扩展追问

### 🔥 扩展追问 1：Mixed GC 是如何选择回收哪些 Old Region 的？

> **A:** G1 使用**基于垃圾密度的启发式算法**选择 Region，按回收价值排序后纳入 CSet：

```
Mixed GC Region 选择算法：
┌──────────────────────────────────────────────────────────────┐
│  每个 Old Region 都有一个回收价值分数：                      │
│                                                              │
│  回收价值 = (该 Region 中的垃圾字节数)                      │
│             / (回收该 Region 的预期停顿时间)                │
│                                                              │
│  G1 选择回收价值最高的 Region，直到满足：                    │
│  1. CSet 总大小触及停顿时间预算                              │
│  2. 回收的 Old Region 数量达到 Mixed GC 预算（默认 8 个）   │
│  3. Old 区回收比例达到目标（默认 5%）                       │
│                                                              │
│  价值排序示例：                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ Region A: 垃圾 80% → 价值 8.0  ← 优先回收            │   │
│  │ Region B: 垃圾 60% → 价值 6.0                       │   │
│  │ Region C: 垃圾 30% → 价值 3.0                       │   │
│  │ Region D: 垃圾 10% → 价值 1.0  ← 尽量避免回收        │   │
│  └─────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

```java
// G1MixedGCLiveThresholdPercent 和 G1OldCSetRegionThresholdPercent
// 控制 Mixed GC 的行为

// -XX:G1MixedGCLiveThresholdPercent=85
// → 只回收垃圾密度超过 85% 的 Old Region（最垃圾的才值得回收）

// -XX:G1OldCSetRegionThresholdPercent=10
// → 一次 Mixed GC 最多回收 10% 的 Old Region

// Mixed GC 的完整流程：
// 1. Young GC 触发时，顺便检查是否需要 Mixed GC
// 2. 如果 Old 区超过 IHOP（Initiating Heap Occupancy Percent，默认 45%）
// 3. 启动 Initial Mark（标记根对象）
// 4. 并发 Remark + Cleanup
// 5. Mixed GC：回收 Young + 选定的 Old Region
// 6. 重复 Mixed GC 直到 Old 区降到目标阈值
```

### 🔥 扩展追问 2：G1 的 Humongous Region 是什么？它有什么特殊问题？

> **A:** Humongous Region 是 G1 为大对象专门设计的超大 Region，它的处理方式有显著不同：

```
Humongous 对象（巨型对象）的处理：
┌──────────────────────────────────────────────────────────────┐
│  判断标准：对象大小 > Region 大小的 50%                      │
│  例如：Region 大小 = 4MB → > 2MB 的对象就是 Humongous        │
│                                                               │
│  Humongous 对象的特点：                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  · 不经过 Young 区，直接进入 Old 区                  │   │
│  │  · 可能占用多个连续的 Humongous Region               │   │
│  │  · Humongous 对象触发 Old 区 GC 的独立阈值（IHOP）   │   │
│  │  · RSet 密度高：引用它的其他 Region很多             │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                               │
│  ┌────┬────┬────┬────┬────┬────┬────┬────┐                   │
│  │ H1 │ H1 │ H1 │    │    │    │    │    │   ← 跨3个Region │
│  │    │    │    │    │    │    │    │    │     的 Humongous │
│  └────┴────┴────┴────┴────┴────┴────┴────┘                   │
└──────────────────────────────────────────────────────────────┘
```

```java
// Humongous 对象的问题：

// 问题1：内存碎片（最严重）
// Humongous 对象占满多个 Region，如果释放后形成空洞
// → G1 必须维护"可用 Humongous 区域"的链表
// → 大对象分配变慢

// 问题2：G1HeapWastePercent 浪费计算
// 如果 Humongous 对象尾部空间超过 G1HeapWastePercent（默认5%）
// → 整块 Region 被标记为浪费
// → 对大对象密集的堆（如 Spark/HBase）影响显著

// 问题3：Old 区回收压力大
// Humongous 对象直接进 Old 区 → Old 区增长快
// → 更频繁触发 Mixed GC

// 解决方案：
// 1. 调大 Region 大小，减少 Humongous 触发概率
//    -XX:G1HeapRegionSize=16MB
// 2. 减少 -XX:G1HeapWastePercent 减少浪费
//    -XX:G1HeapWastePercent=2
// 3. 在代码层面避免大对象：拆分成小对象集合
```

### 🔥 扩展追问 3：G1 的 RSet 内存开销有多大？如何调优？

> **A:** RSet 是 G1 最大的内存开销之一，需要仔细分析：

```
RSet 内存开销分析：
┌──────────────────────────────────────────────────────────────┐
│  RSet 内存占用公式（近似）：                                  │
│                                                              │
│  RSet 内存 ≈ Heap 大小 × RSet 开销系数                      │
│                                                              │
│  RSet 开销系数 ≈ 5% ~ 10%（取决于应用特点）                  │
│                                                              │
│  影响因素：                                                   │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  对象引用复杂度（跨 Region 引用越多，RSet越大）       │   │
│  │  Region 大小（Region 越小，Region 越多，RSet越多）    │   │
│  │  Card Table 精度（512B vs 1KB）                       │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  实测数据（估算）：                                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  堆大小 32GB + Region 4MB = 8192 个 Region            │   │
│  │  平均每个 Region 的 RSet ≈ 200KB（稀疏记录）          │   │
│  │  RSet 总开销 ≈ 1.6GB（占堆的 5%）                    │   │
│  └─────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

```bash
# RSet 调优参数
-XX:+PrintAdaptiveSizePolicy   # 打印自适应大小策略信息，观察 RSet 行为

# RSet 精度的权衡
# G1 使用三级 RSet 结构：

# 1. 稀疏 RSet（Sparse RSet）
#    - 精确记录指向本 Region 的外部指针
#    - 存储在 PerRegionTable 中
#    - 每个 Region 最多 16 个稀疏条目

# 2. 细粒度位图（Fine Grained Bitmap）
#    - 记录引用来自哪个其他 Region
#    - 每个其他 Region 占 1 bit

# 3. 粗粒度位图（Coarse Bitmap）
#    - 当引用来自太多 Region 时，启用
#    - 简化记录，但扫描时需要遍历更多 Region

# 调优建议：
# - Region 越大 → Region 数量越少 → RSet 条目越少
# - 所以大堆（> 16GB）用大 Region 可以减少 RSet 开销
# - 但 Region 越大，Minor GC 停顿时间会增加（Eden 变大）
```

### 🔥 扩展追问 4：G1 的并发标记阶段（Concurrent Marking）是如何工作的？

> **A:** G1 的并发标记是整个 GC 周期中最复杂的阶段，使用了 **SATB（Snapshot-At-The-Beginning）算法**：

```
G1 并发标记四步走：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  Phase 1: Initial Mark（初始标记）— Stop The World            │
│  ├─ 标记 GC Roots 直接引用的对象                            │
│  ├─ 暂停时间 ≈ Young GC 的停顿时间                         │
│  └─ 同时触发了 Young GC（Mixed GC 的第一次 YGC）            │
│                                                              │
│  Phase 2: Concurrent Mark（并发标记）— 与应用并发           │
│  ├─ 遍历堆，从 GC Roots 开始追踪所有可达对象                │
│  ├─ 利用 SATB 记录标记开始时的快照                          │
│  ├─ 应用线程和标记线程并发执行                              │
│  └─ 耗时 ≈ 堆大小 / 标记线程速度                            │
│                                                              │
│  Phase 3: Remark（重新标记）— Stop The World                  │
│  ├─ 处理 SATB 缓冲区中的所有变更（并发修改）                │
│  ├─ 处理 Dirty Card Queue                                    │
│  ├─ 暂停时间 ≈ 100ms ~ 500ms                                 │
│  └─ 结束后，所有存活对象已精确标记                           │
│                                                              │
│  Phase 4: Cleanup（清理）— Stop The World（部分）            │
│  ├─ 计算每个 Region 的存活对象比例                          │
│  ├─ 标记完全空闲的 Region                                    │
│  ├─ 回收空闲 Region（并发释放）                             │
│  └─ 为 Mixed GC 准备 CSet                                    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

```java
// SATB（Snapshot-At-The-Beginning）的核心思想：
// "标记开始时活着的对象，在标记期间变成死的，不回收"
// → 保证了在标记过程中新分配的对象至少存活到下次 GC

// SATB 的写屏障实现：
// 当应用线程修改一个引用时，JVM 记录旧引用值（被覆盖之前的值）
// 这些旧引用被加入 SATB 缓冲区
// Remark 阶段处理这些缓冲区，确保不漏标

// G1 调优参数
-XX:InitiatingHeapOccupancyPercent=45  
// → Old 区占用 45% 时触发并发标记周期

-XX:ConcGCThreads=4  
// → 并发标记线程数（默认约为 ParallelGCThreads 的一半）

-XX:G1ReservePercent=10  
// → 保留 10% 的堆作为"逃生通道"（避免 Mixed GC 后无空间分配）
```

## 3. 一句话总结

> **G1 的 Region 大小由堆大小除以最大 Region 数（2048）并取最近的 2 的幂次方决定；Mixed GC 通过计算每个 Old Region 的"垃圾密度/停顿时间"价值分数来选择回收目标；CSet 是被选中的待回收 Region 集合；RSet 通过 Card Table + 写屏障记录跨 Region 引用，让 G1 在并发标记时无需扫描全堆——但 Humongous 对象和 RSet 本身的空间开销是 G1 最需要关注的调优点。**

## 4. 面试加分话术

> "我在公司做过一次 G1 的专项调优，目标是把 P99 GC 停顿从 800ms 降到 200ms 以内。当时发现主要问题是 Mixed GC 回收了太多低价值的 Old Region（有些 Region 只有 30% 的垃圾），后来调了 `-XX:G1MixedGCLiveThresholdPercent=85` 让 G1 只回收垃圾密度超过 85% 的 Region，配合 `-XX:MaxGCPauseMillis=150` 的硬约束，Mixed GC 的回收效率大幅提升。另外还发现 Humongous 对象太多导致 Old 区碎片化严重，最后通过把某些大 List 拆分成多个小 Chunk，从根本上减少了 Humongous Region 的产生。"

> "关于 RSet 的开销，我之前用 `jstat -gcnew` 观察到 G1 的内存中 RSet 占用了将近 6% 的堆空间，这在 32GB 堆的机器上差不多是 2GB。后来通过把 Region 大小从 2MB 调大到 8MB，Region 数量从 16384 降到 4096，RSet 的条目数显著减少，但同时也导致了 Young GC 停顿时间略有增加——这就是 G1 调优的核心矛盾：Region 大 vs Region 小的权衡。"
