# Q174: Redis 内存计算与 MEMORY USAGE 深度解析

## 1. 基础流程

### 1.1 `MEMORY USAGE` 命令的工作原理

```
MEMORY USAGE 命令执行过程：

┌─────────────────────────────────────────────────────────┐
│  MEMORY USAGE user:100:name                            │
│                                                         │
│  Step 1: 定位 key                                       │
│    在全局 dict 中查找 "user:100:name"                   │
│    获取指向 redisObject 的指针                          │
│                                                         │
│  Step 2: 获取 redisObject 元信息                       │
│    type = REDIS_STRING (4)                             │
│    encoding = embstr (4)                               │
│    lru = LRU_CLOCK (4)                                 │
│    refcount = 1 (4)                                    │
│    *ptr → 指向实际数据的指针                            │
│                                                         │
│  Step 3: 递归计算所有关联内存                           │
│    sds = SDS_TYPE_5 / SDS_TYPE_8 / SDS_TYPE_64        │
│    + dictEntry（哈希表桶节点）                          │
│    + 全局 dictEntry 开销                                │
│    + 共享对象池（不计入，只计 refcount）                │
│                                                         │
│  Step 4: 汇总返回（单位：字节）                         │
│    total = dictEntry + redisObject + sds + 数据本身    │
└─────────────────────────────────────────────────────────┘
```

```bash
# 基础使用
MEMORY USAGE key              # 只计算 key 和 value，不含 dictEntry
MEMORY USAGE key SAMPLES 0    # SAMPLE=0 表示全量扫描（慢，精确）
MEMORY USAGE no_such_key       # nil（key 不存在）

# 实战：对比不同编码的内存
SET str:embstr "hello"     # embstr 编码（44 字节以内）
SET str:raw    "a very long string more than 44 bytes"  # raw 编码

MEMORY USAGE str:embstr   # 返回更小
MEMORY USAGE str:raw      # 返回更大
```

### 1.2 Redis 对象模型与内存结构

```
┌─────────────────────────────────────────────────────────┐
│  Redis Object（redisObject，固定 16 字节）              │
│                                                         │
│  typedef struct redisObject {                           │
│      unsigned type:4;        // 类型：STRING/LIST/HASH  │
│      unsigned encoding:4;   // 编码：embstr/raw/int/   │
│      unsigned lru:LRU_BITS; // LRU 或 LFU 信息         │
│      int refcount;          // 引用计数（共享对象）     │
│      void *ptr;             // 指向实际数据的指针       │
│  } robj;                                                  │
│                                                         │
│  固定大小：16 字节（64 位系统）                         │
│  压缩后：最小 String = 16 + sds_size + data_size       │
└─────────────────────────────────────────────────────────┘
```

```
Redis Object 的内存组件分解：

┌─────────────────────────────────────────────────────────┐
│  组件1: dictEntry（哈希表桶节点）                       │
│  ─────────────────────────────────────                  │
│  struct dictEntry {                                     │
│      void *key;        // SDS 字符串（key 本身）        │
│      union {                                           │
│          void *val;                                    │
│          uint64_t u64;                                 │
│          int64_t s64;                                  │
│      };                                                │
│      struct dictEntry *next; // 拉链法冲突链表           │
│  };                                                     │
│                                                         │
│  大小：sizeof(dictEntry) = 24 字节（64 位，8 字节指针） │
│  + key 的 SDS 字符串大小（按 key 长度）                │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  组件2: SDS（Simple Dynamic String）                    │
│  ─────────────────────────────────                      │
│  SDS_TYPE_5  (≤32 字节):  3 字节头部 + 数据            │
│  SDS_TYPE_8  (≤256B):     4 字节头部 + 数据            │
│  SDS_TYPE_16 (≤64KB):     6 字节头部 + 数据            │
│  SDS_TYPE_32 (≤4GB):     10 字节头部 + 数据            │
│  SDS_TYPE_64 (超大):     18 字节头部 + 数据            │
│                                                         │
│  头部结构（SDS_TYPE_8 为例）：                         │
│  char flags;        // 1 字节：类型 + 长度位           │
│  char buf[];        // 柔性数组，实际数据              │
│  // 隐含字段：len（1字节）+ free（1字节）             │
└─────────────────────────────────────────────────────────┘
```

### 1.3 空 String key 的内存占用

```
空 String key = "" 的内存计算：

┌─────────────────────────────────────────────────────────┐
│  MEMORY USAGE "" 等于一个不存在的 key？                │
│  不是！空字符串 "" 是合法值，不是 nil                   │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  空字符串 "" 的内存组成（SDS_TYPE_5）：                 │
│                                                         │
│  dictEntry:       24 字节（key="", 0字节 SDS 按优化存储）│
│  redisObject:     16 字节（type=STRING, encoding=embstr）│
│  SDS_TYPE_5:       3 字节（flags + 1字节len + 1字节free）│
│  数据指针:         0 字节（空字符串，数据区为空）       │
│  ─────────────────────────────────────────────────────  │
│  合计:            ~43 字节                              │
│                                                         │
│  注意：空字符串在 embstr 编码中，SDS 直接嵌入 robj 后  │
│  embstr 的最大长度：44 字节（64-16=48，但 Redis 留4B）│
└─────────────────────────────────────────────────────────┘

实际测试（redis-cli）：
  MEMORY USAGE key:empty   （假设 value=""）
  --> 56 字节 或 48 字节（因 Redis 版本和 key 长度略有不同）

┌─────────────────────────────────────────────────────────┐
│  有值 String key 的内存计算示例：                       │
│                                                         │
│  key = "name" (4字节)                                  │
│  value = "ZhangSan" (8字节)                            │
│                                                         │
│  dictEntry:              24 + 4(SDS key) = 28 字节     │
│  redisObject:             16 字节                        │
│  SDS_TYPE_5/8 (value):    4 + 8 = 12 字节              │
│  总计:                   ≈ 56 字节                     │
│                                                         │
│  如果 value 是整数 "12345"（可用 int 编码）：          │
│    redisObject.encoding = int                          │
│    ptr 直接指向共享整数对象（共享池），无额外分配       │
│    MEMORY USAGE ≈ dictEntry + redisObject ≈ 44 字节   │
└─────────────────────────────────────────────────────────┘
```

## 2. Hash 内存详细计算

### 2.1 Hash 的两种内部编码

```
Hash 的内部编码决策：

┌─────────────────────────────────────────────────────────┐
│  encoding = ziplist（压缩列表）                        │
│    条件：field + value 总数 ≤ 512（默认）              │
│    条件：每个 field 和 value 长度 ≤ 64 字节             │
│    特点：连续内存存储，内存紧凑                         │
│                                                         │
│  encoding = hashtable（哈希表）                        │
│    条件：超过 ziplist 阈值                             │
│    特点：dict 结构，支持 O(1) 查找，内存开销大          │
└─────────────────────────────────────────────────────────┘

Hash 相关参数：
hash-max-ziplist-entries: 512    # 超过此值切换到 hashtable
hash-max-ziplist-value: 64       # 单 field/value 超过此值切换

注意：阈值在 Redis 7.0 引入流控，可动态调整
```

### 2.2 Hash (ziplist 编码) 内存分析

```
1 个 field 的 Hash（ziplist 编码）：

┌─────────────────────────────────────────────────────────┐
│  ziplist 内存结构：                                     │
│                                                         │
│  ┌──────────────────────────────────────────────────┐  │
│  │ zlbytes(4) │ zltail(4) │ zllen(2) │ zlentry[]    │  │
│  │   总字节    │  尾偏移    │  元素数  │  实际数据    │  │
│  └──────────────────────────────────────────────────┘  │
│                                                         │
│  ziplist entry 编码（以 "name"="ZhangSan" 为例）：     │
│  prevrawlen: 变长编码（1-5字节）                       │
│    └─ 上一条 entry 的长度，用于反向遍历                 │
│  encoding: 1字节（ziplist 内部编码类型）               │
│    └─ 如 ZIP_STR_13B = 0x31（13字节短字符串）         │
│  contents: 实际数据 "nameZhangSan"（连续存储）         │
│                                                         │
│  ziplist 整体布局：                                     │
│  [zltail指向的位置]                                     │
│                                                         │
│  entry1: [prev=0] [enc] [name] [prev=12B] [enc] [ZhangSan] │
│                                                         │
│  计算：                                                 │
│  ziplist header: 4 + 4 + 2 = 10 字节                   │
│  entry1 field: 1(prev) + 1(enc) + 4(name) = 6 字节    │
│  entry1 value: 1(prev) + 1(enc) + 8(ZhangSan) = 10 字节│
│  ziplist end: 1 字节（0xFF）                           │
│  总计: 10 + 6 + 10 + 1 = 27 字节                       │
└─────────────────────────────────────────────────────────┘

MEMORY USAGE 的结果 ≈ dictEntry + redisObject + ziplist ≈ 57 字节
```

### 2.3 Hash (hashtable 编码) 内存分析

```
10000 个 field 的 Hash（hashtable 编码）：

┌─────────────────────────────────────────────────────────┐
│  hashtable 内存组成（每个 field=value 键值对）：       │
│                                                         │
│  1. 顶层 dict：                                        │
│     dictEntry 数组（桶数组）                           │
│     初始桶数 = 4（2^2），自动扩容                     │
│     负载因子 = size/used                               │
│     扩容阈值：used > buckets * load_factor             │
│                                                         │
│  2. 每个 field 的 dictEntry：                          │
│     struct dictEntry {                                  │
│         void *key;     // field 的 SDS                 │
│         void *val;     // value 的 redisObject         │
│         dictEntry *next;  // 哈希冲突链表               │
│     };                                                  │
│     每个 dictEntry = 24 字节 + SDS 大小                │
│                                                         │
│  3. field 本身的 SDS：                                 │
│     SDS_TYPE_8: 4字节头部 + field 长度                 │
│                                                         │
│  4. value 的 redisObject + SDS：                       │
│     redisObject = 16 字节                               │
│     SDS_TYPE_8: 4字节头部 + value 长度                 │
│                                                         │
│  5. 扩容/缩容的空闲 dictEntry 数组                     │
│     通常为 used 的 1-2 倍                              │
└─────────────────────────────────────────────────────────┘

10000 个 field 的内存估算（hashtable）：

┌─────────────────────────────────────────────────────────┐
│  假设：field 平均 8 字节，value 平均 16 字节            │
│                                                         │
│  dictEntry 数组（hashtable 主结构）：                  │
│    used = 10000                                         │
│    buckets ≈ 16384（负载因子≈0.61）                   │
│    每个 bucket = 8 字节（64位指针）                    │
│    dictEntry = 24 + 8(next指针在数组内) = 24 字节      │
│    主表内存 = 16384 * 8 = 131072 字节 ≈ 128 KB        │
│                                                         │
│  dictEntry 节点（10000 个 field）：                    │
│    每个 dictEntry = 24 字节                            │
│    field SDS: 4 + 8 = 12 字节                         │
│    value robj + SDS: 16 + 4 + 16 = 36 字节            │
│    每个 field 总计：24 + 12 + 36 = 72 字节            │
│    10000 个 field = 720000 字节 ≈ 703 KB              │
│                                                         │
│  全局字典开销（RedisObject 本身）：                    │
│    dictEntry: 24 + key SDS(4+10) = 38 字节            │
│    redisObject: 16 字节                                │
│    ziplist/hashtable: 10000 字段的 ziplist/hashtable  │
│                                                         │
│  粗略总计：≈ 1 MB（不含主表扩容器）                   │
└─────────────────────────────────────────────────────────┘
```

### 2.4 每个 field 的额外 overhead 精确对比

```
1 个 field vs 10000 个 field 的 overhead 分析：

┌─────────────────────────────────────────────────────────┐
│                                                         │
│  类型        │ 1 个 field      │ 10000 个 field        │
│  ────────────┼─────────────────┼─────────────────────── │
│  编码        │ ziplist         │ hashtable             │
│  ────────────┼─────────────────┼─────────────────────── │
│  每 field    │                │                        │
│  平均 overhead│                │                        │
│  （估算）     │  ziplist entry │  dictEntry + 2个SDS   │
│              │  ~14-16 字节   │  ≈ 56-72 字节        │
│              │                │                        │
│  1个field总: │  dictEntry+    │  不适用（不用此编码）  │
│              │  redisObject+  │                        │
│              │  ziplist      │                        │
│              │  ≈ 50-80 字节 │                        │
│  ────────────┼─────────────────┼─────────────────────── │
│  10000field │  不适用        │  总计 ≈ 0.8-1.2 MB    │
│  总 overhead │                │  每field ≈ 56-72B    │
│              │                │                        │
│  每 field    │                │                        │
│  overhead    │  ≈ 50-80 字节 │  ≈ 56-72 字节        │
│  差异        │  (ziplist紧凑) │  (hashtable 有链表)   │
└─────────────────────────────────────────────────────────┘

关键发现：
- ziplist（少量 field）：每个 entry 有 prevrawlen + encoding 开销
  → 数据量少时 overhead 占比高
- hashtable（大量 field）：每个 field = dictEntry + 2个SDS + robj
  → 数据量大时 overhead 更稳定（≈ 56-72 字节/field）
- hashtable 扩容器：2x 桶数组，额外 2x bucket 指针数组
```

## 3. MEMORY USAGE 的边界情况

### 3.1 MEMORY USAGE 统计边界

```
MEMORY USAGE 统计 vs 不统计的内容：

┌─────────────────────────────────────────────────────────┐
│  ✅ 统计在内：                                          │
│  - dictEntry（key 的哈希表节点）                       │
│  - redisObject（类型头，16字节）                        │
│  - ptr 指向的实际数据（SDS / ziplist / dict 等）       │
│  - 内部编码的元数据（ziplist header, dict header 等） │
│                                                         │
│  ❌ 不统计在内：                                        │
│  - 全局哈希表桶数组（db->dict 的桶）                   │
│    └─ 因为这是全局共享结构，不是 key 私有               │
│  - 共享对象池（0-9999 的小整数、常见字符串等）         │
│    └─ refcount 引用，不单独计数                         │
│  - Lua 脚本虚拟机内存                                   │
│  - AOF/RDB 缓冲区                                       │
│  - 客户端输入/输出缓冲区（client buffer）              │
│  - Pub/Sub 订阅者列表                                    │
└─────────────────────────────────────────────────────────┘
```

### 3.2 Redis 7.0 的内存优化（Hash 编码改进）

```
Redis 7.0 引入了 Listpack（替代 ziplist）：

┌─────────────────────────────────────────────────────────┐
│  ziplist vs listpack 对比：                             │
│                                                         │
│  ziplist 问题：                                         │
│  - entry 的 prevrawlen 变长编码，在中部插入/删除时    │
│    可能导致后续所有 entry 的重新编码                     │
│  - 级联更新，O(n²) 最坏情况                             │
│                                                         │
│  listpack 改进：                                        │
│  - 移除 prevrawlen，改为 total_bytes 反向遍历          │
│  - 每次只修改自身，不影响邻居                           │
│  - 每个 entry 更紧凑                                    │
│  Redis 7.0 Hash/Zset/SortSet 默认用 listpack          │
└─────────────────────────────────────────────────────────┘

Redis 7.0 Hash 编码阈值变化：
- hash-max-listpack-entries: 512
- hash-max-listpack-value: 128  ← 从 64 提升到 128

额外优化：
- Redis 7.0 Hash 可以用 zset-min-listpack-with-value 优化
- 内存节省约 10-20%（listpack 更紧凑）
```

## 🔥 扩展追问 1：Redis 的 dictEntry 什么时候不计入内存？

> **A:** 特殊情况下 dictEntry 由全局表分摊，不单独计入：

```
MEMORY USAGE 不包含 dictEntry 的情况：

1. 内部临时表（如 LRU 淘汰的候选队列）：
   Redis 内部维护一个全局 dict 用于 key 索引
   这个 dict 的桶数组是全局共享的
   MEMORY USAGE 不统计这个全局 dict 的桶数组

2. 共享对象池的对象：
   SET k 0 → redisObject.ptr 指向共享对象池的 0
   MEMORY USAGE k → 只统计 dictEntry + redisObject
   不统计 ptr 指向的共享对象（共享池总内存不在单个 key 中）

3. pipeline / pubsub 临时消息：
   不走 key 索引，临时分配后释放
   MEMORY USAGE 无法追踪
```

## 🔥 扩展追问 2：MEMORY USAGE 有性能问题吗？SAMPLES 参数的作用？

> **A:** 会的，当 hash/zset 很大时：

```bash
# MEMORY USAGE key SAMPLES N

# 默认 SAMPLES = 5
# 表示：对于 hashtable/zset，只抽样 N 个元素估算

# SAMPLES=5 时（默认，快速但有误差）：
MEMORY USAGE big_hash
  --> 抽样 5 个 field，估算平均大小 × field 总数
  --> 返回估算值，不是精确值

# SAMPLES=0 时（全量扫描，精确但慢）：
MEMORY USAGE big_hash SAMPLES 0
  --> 遍历所有 field，准确计算总内存
  --> 大 Hash 可能卡顿数秒

# 生产环境建议：
# - 对小 key（< 1000 field）：SAMPLES 0（误差无意义）
# - 对大 key（> 10000 field）：用 SAMPLES 100（估算足够准）
#   误差估算：±5% 以内

# 查看大 key：
redis-cli --bigkeys           # 扫描找出最大的 key
redis-cli --scan | head -100 | xargs -I{} redis-cli MEMORY USAGE {}
```

## 🔥 扩展追问 3：Redis 的整数共享池是如何节省内存的？

> **A:** Redis 预分配 0-9999 的整数对象供共享使用：

```c
// Redis 源码：共享整数对象池
#define OBJ_SHARED_INTEGERS 10000

robj *shared.integers[OBJ_SHARED_INTEGERS];

// 初始化时：
for (int i = 0; i < OBJ_SHARED_INTEGERS; i++) {
    shared.integers[i] = makeObjectShared(createObject(REDIS_STRING, (void*)(long)i));
    shared.integers[i]->encoding = OBJ_ENCODING_INT;
}

// SET k 42
// redisObject.ptr → shared.integers[42]（复用，不新建）
// 节省：16(robj) + 8(SDS) ≈ 24 字节

// MEMORY USAGE k
// --> dictEntry + redisObject + shared 指针 ≈ 40 字节
// 不再额外计算共享对象本身的内存（已预分配，全局共享）
```

```
MEMORY USAGE 在共享整数时的计算：

┌─────────────────────────────────────────────────────────┐
│  SET small_int "1234"    -- 1234 < 10000，共享池       │
│  MEMORY USAGE small_int  -- ≈ 48 字节                 │
│      dictEntry: 24 字节                                 │
│      redisObject: 16 字节                              │
│      ptr: 指向共享整数对象（已预分配，不另计）         │
│                                                         │
│  SET large_int "99999"  -- 99999 > 9999，需新建        │
│  MEMORY USAGE large_int -- ≈ 56 字节                   │
│      dictEntry: 24 字节                                 │
│      redisObject: 16 字节                              │
│      ptr → 实际 int 对象: 16 字节                      │
└─────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 4：MEMORY STATS 命令比 MEMORY USAGE 多了什么？

> **A:** MEMORY STATS 提供全局内存视角：

```bash
MEMORY STATS

# 输出结构：
{
  "used.memory": 12345678,          # 总使用内存（字节）
  "used.memory.human": "11.77M",   # 人类可读
  "used.memory_peak": 23456789,    # 峰值内存
  "used.memory_rss": 15000000,     # RSS（实际物理内存）
  "used.memory_overhead": 1234567, # 元数据开销（dictEntry 等）
  "used.memory_startup": 1048576,  # Redis 启动时固定开销
  "memory_fragmentation_ratio": 1.23,  # RSS/used.memory
  "allocator_resident": ...,       # 分配器占用
  "allocator_frag_ratio": ...,     # 碎片率
  "keys.per.database": {
     "0": 1000,                     # db0 有 1000 个 key
     "1": 0
  },
  "keyspace.hits": 100000,
  "keyspace.misses": 1000,
  "db.0": {                         # db0 各类型的 key 统计
     "keys": 1000,
     "hashtable.size": 1024,
     "listpack.size": ...
  }
}

# 关键洞察：
# - memory_fragmentation_ratio > 1.5 → 需要 BGREWRITEAOF 压缩
# - used.memory_rss >> used.memory → 内存碎片严重
# - allocator_frag_ratio > 1.2 → jemalloc 碎片
```

## 🔥 扩展追问 5：如何快速定位 Redis 中的内存杀手（大 key）？

> **A:** 几种方法的对比：

```bash
# 方法1：redis-cli --bigkeys（最常用）
redis-cli --bigkeys

# 输出：
# Scanning 100000 keys
# Big string: k1 → 10MB
# Big list: k2 → 50000 elements
# Big hash: k3 → 10000 fields
# Big set: k4 → 50000 members

# 优点：单命令，快速
# 缺点：只找每种类型的最大 key，不是全部大 key

# 方法2：SCAN + MEMORY USAGE 脚本
redis-cli --scan | while read key; do
  size=$(redis-cli MEMORY USAGE "$key" 2>/dev/null)
  if [ "$size" -gt 1048576 ]; then  # > 1MB
    echo "$key: $size bytes"
  fi
done

# 方法3：MEMORY DOCTOR（Redis 4.0+）
redis-cli MEMORY DOCTOR
# 返回内存诊断报告和建议

# 方法4：Redis 7.0+ MEMORY USAGE WITH-SAMPLES
# 对大 Hash 用采样估算，快速筛选
redis-cli --scan | while read key; do
  size=$(redis-cli MEMORY USAGE "$key" SAMPLES 100 2>/dev/null)
  if [ "$size" -gt 10485760 ]; then  # > 10MB
    echo "$key: $size bytes (sampled)"
  fi
done
```

---

> **一句话总结**

`MEMORY USAGE key` 精确统计该 key 的 **dictEntry + redisObject + 实际数据结构** 全部内存，但**不统计**全局共享的哈希表桶数组和共享对象池；空 String 约 43-56 字节（含 dictEntry 24B + robj 16B + SDS 3B）；1 个 field 的 Hash 用 ziplist 编码每 field overhead 约 14-16 字节（紧凑），10000 个 field 切换为 hashtable 后每 field overhead 约 **56-72 字节**（含 dictEntry 24B + field SDS + value robj+SDS），两者差距主要来自 hashtable 的链表指针和扩容器开销。

> **面试加分话术**

> "我在实际排查中遇到过一个大 Hash 占用 500MB 内存的问题——`MEMORY USAGE key` 显示每个 field 约 68 字节，10000 个 field 理论上应该只有 680KB，但实际占用远超预期。分析下来发现是 Redis 4.0 之前 `hash-max-ziplist-entries` 默认只有 512，7.0 已升到 512，`hash-max-listpack-value` 从 64 升到 128——这个改动让 ziplist/listpack 编码能覆盖更多场景。我们的解决方案是：对热度高但 field 数量多的 Hash，改用 ZSET 存储（MEMBERS 作为 field，定期用 HINCRBY 合并冷数据），内存从 500MB 降到 80MB。另外 MEMORY STATS 的 `memory_fragmentation_ratio` 长期超过 1.5 时，`BGREWRITEAOF` 压缩能释放 30-40% 内存。"
