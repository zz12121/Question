## 1. 基础流程

### 1.1 什么是 NMT

```
NMT（Native Memory Tracking）的定位：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  JVM 进程的内存消耗 = Java 堆（-Xmx） + Native 内存        │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  Java 堆 (Heap):                                      │   │
│  │  [Young Gen][Old Gen][Metaspace] ← -Xmx 控制         │   │
│  │  这是大多数监控工具能看到的内容                       │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  Native 内存（通常被忽视）：                                 │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  · Code Cache（JIT 编译后的机器码）                  │   │
│  │  · Metaspace（类元数据，在 native 内存中）            │   │
│  │  · Threads（线程栈）                                  │   │
│  │  · Direct ByteBuffers（堆外内存）                    │   │
│  │  · GC 内部数据结构（Card Table、RSet 等）            │   │
│  │  · JIT Compiler 内存                                  │   │
│  │  · Internal / malloc 内存                             │   │
│  └──────────────────────────────────────────────────────┘   │
│  ↑                                                            │
│  ↑ NMT 追踪的就是这些！                                     │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 如何开启 NMT

```bash
# 开启 NMT（应用启动时指定）
java -XX:NativeMemoryTracking=summary myapp.jar
# 或
java -XX:NativeMemoryTracking=detail myapp.jar

# NMT 有两个级别：
# - summary：只显示内存类别汇总（推荐生产使用）
# - detail：显示详细的内存分配点（调试用，开销较大）

# 运行时开启（JDK 11+ 支持动态开启）
# jcmd <pid> VM.native_memory baseline   # 创建基准线
# jcmd <pid> VM.native_memory summary    # 查看当前
# jcmd <pid> VM.native_memory detail     # 查看详情

# 关闭 NMT
jcmd <pid> VM.native_memory shutdown
```

### 1.3 NMT 能追踪的内存类别

```bash
# 典型的 NMT 输出（java -XX:NativeMemoryTracking=summary）：
# 
# Native Memory Tracking:
# 
# Total: reserved=5242880KB, committed=5242880KB
# -                 Java Heap (reserved=4194304KB, committed=4194304KB)
# -                     Class (reserved=524288KB, committed=524288KB)
# -                     Thread (reserved=52428KB, committed=52428KB)
# -                     Code (reserved=262144KB, committed=262144KB)
# -                     GC (reserved=131072KB, committed=131072KB)
# -                     Compiler (reserved=65536KB, committed=65536KB)
# -                     Internal (reserved=32768KB, committed=32768KB)
# -                     Symbol (reserved=16384KB, committed=16384KB)
# -                     Native Memory Tracking (reserved=8192KB, committed=8192KB)
# -                     Arena Chunk (reserved=4096KB, committed=4096KB)
# -                     Logging (reserved=2048KB, committed=2048KB)
# -                     Arguments (reserved=1024KB, committed=1024KB)
# -                     Other (reserved=512KB, committed=512KB)
```

**各内存类别的详细说明**：

| 类别 | 说明 | 影响因素 |
|---|---|---|
| **Java Heap** | `-Xms` 到 `-Xmx` 的堆内存 | `-Xmx`、`-Xms` |
| **Class** | 类元数据（Metaspace） | `-XX:MetaspaceSize`、`-XX:MaxMetaspaceSize` |
| **Thread** | 线程栈内存 | `-Xss` × 线程数 |
| **Code** | JIT 编译后的机器码存储 | `-XX:ReservedCodeCacheSize` |
| **GC** | GC 算法内部数据结构 | GC 类型（G1/RSEt/CMS 等） |
| **Compiler** | JIT 编译器的临时内存 | 编译任务量 |
| **Internal** | JVM 内部 malloc/meta 内存 | 操作系统分配 |
| **Symbol** | 字符串常量池符号表 | `String.intern()` 使用量 |
| **Arena Chunk** | 内存分配arena | JVM 内部内存池 |
| **Logging** | GC 日志缓冲区 | `-Xlog` 配置 |
| **Arguments** | JVM 命令行参数副本 | 参数数量 |
| **Other** | 其他未分类内存 | — |

```
NMT 的 detail 模式输出（更详细的内存分布）：
# 
# Total: reserved=5242880KB, committed=5242880KB
# 
# -                 Java Heap (reserved=4194304KB, committed=4194304KB)
#                           malloc://4194304KB # 堆内存通过 malloc 分配
# 
# -                     Class (reserved=524288KB, committed=524288KB)
#                           malloc://516096KB # 类加载器分配
#                           arena_chunk://8192KB
#                           class_space://524288KB # 类空间子区域
# 
# -                     Thread (reserved=52428KB, committed=52428KB)
#                           Thread Stack: # 每个线程的栈
#                             Thread-1: reserved=1024KB, committed=1024KB
#                             Thread-2: reserved=1024KB, committed=1024KB
#                             ...
```

## 2. 扩展追问

### 🔥 扩展追问 1：NMT 的 reserved 和 committed 有什么区别？

> **A:** 这是 NMT 中最容易混淆的概念，也是面试常考点：

```
reserved vs committed 的区别：
┌──────────────────────────────────────────────────────────────┐
│  reserved（保留内存）：                                      │
│  · JVM 向操作系统"预订"的虚拟地址空间                      │
│  · 预订时不分配物理内存（不占用 RAM）                       │
│  · 类似于"我计划用这块地"                                   │
│  · 防止其他进程申请相同地址                                  │
│                                                              │
│  committed（提交内存）：                                     │
│  · JVM 实际占用的物理内存（RSS / Resident Set）             │
│  · 操作系统按需分页分配                                      │
│  · 类似于"我实际在这块地上建了房子"                        │
│                                                              │
│  两者的关系：                                                │
│  reserved >= committed                                      │
│  committed 会随着使用量增长，直到达到 reserved 的上限       │
└──────────────────────────────────────────────────────────────┘
```

```java
// 典型的 reserved vs committed 场景

// Code Cache：
-XX:ReservedCodeCacheSize=256M   // reserved = 256MB（编译完成后的上限）
// committed 增长曲线：
//   预热期 → committed 快速增长（JIT 编译中）
//   稳态期 → committed 稳定在某个值（< reserved）
//   极端情况 → committed = reserved（CodeCache 满，JIT 被抑制）

// Thread Stack：
// reserved = -Xss × 最大线程数（线程池上限）
// committed = -Xss × 当前活跃线程数
// 如果最大线程数=200，当前活跃=20 → reserved远大于committed

// Arena Chunk：
// JVM 内部使用 arenas 批量管理内存
// reserved =  arenas的总大小上限
// committed = 当前 arenas 中已使用的部分
```

### 🔥 扩展追问 2：NMT 本身有性能损耗吗？大约多少？

> **A:** NMT 本身有轻微但可量化的性能开销，关键是不同级别的开销差异很大：

```
NMT 性能开销分析：
┌──────────────────────────────────────────────────────────────┐
│  性能开销来源：                                             │
│  1. 每块内存分配/释放时，记录到 NMT 的内部数据结构          │
│  2. NMT 的内部管理需要加锁（summary 模式更轻量）           │
│  3. detail 模式需要额外的调用栈采集（非常重）              │
│                                                              │
│  实测数据（来自 JVM 官方文档 + 社区测试）：                 │
│  ┌────────────────────────────────────────────────────┐   │
│  │ 关闭 NMT：          baseline                         │   │
│  │ 开启 NMT (summary)：-0.5% ~ -2% 吞吐量               │   │
│  │ 开启 NMT (detail)： -5% ~ -15% 吞吐量                 │   │
│  └────────────────────────────────────────────────────┘   │
│                                                              │
│  内存开销：                                                 │
│  - NMT 本身需要内存来存储追踪数据                           │
│  - summary 模式：≈ 几 MB                                   │
│  - detail 模式：≈ 几十 MB（取决于调用栈深度）              │
│                                                              │
│  延迟影响：几乎可忽略                                        │
└──────────────────────────────────────────────────────────────┘
```

```bash
# NMT 开销验证（实际测试）
# 1. 创建基准
java -Xmx4g -Xms4g myapp.jar  # 无 NMT
# 观察：QPS, Latency, CPU

# 2. 开启 NMT summary
java -Xmx4g -Xms4g -XX:NativeMemoryTracking=summary myapp.jar
# 观察：QPS 下降 < 2%

# 3. 开启 NMT detail
java -Xmx4g -Xms4g -XX:NativeMemoryTracking=detail myapp.jar
# 观察：QPS 下降 5-15%，延迟上升明显

# 结论：
# - 生产环境推荐 summary 模式
# - detail 模式仅用于调试/排查问题
# - 问题排查完后立即关闭 NMT
```

### 🔥 扩展追问 3：如何用 NMT 排查 Metaspace 内存泄漏？

> **A:** NMT 是排查 Metaspace 增长问题的首选工具：

```bash
# 场景：Metaspace 持续增长疑似泄漏

# 步骤1：创建 NMT 基准
jcmd <pid> VM.native_memory baseline

# 步骤2：运行一段时间后，对比
jcmd <pid> VM.native_memory summary.diff
# 输出示例：
# -                     Class (reserved=500MB, committed=480MB)
# -                     Class (reserved=800MB [+300MB], committed=750MB [+270MB])
# ↑ Metaspace 增长了 300MB！这是泄漏信号

# 步骤3：进一步分析 detail
jcmd <pid> VM.native_memory baseline
# ... 运行一段时间 ...
jcmd <pid> VM.native_memory detail.diff

# 输出会显示是哪种子区域的增长：
# -                     Class (reserved=800MB, committed=750MB)
#                           class_space://+256MB    ← 类数据区增长
#                           anonymous_class_space://+14MB
#                           Class loader metaspace://+30MB

# 常见泄漏场景：
# - class_space 增长：大量动态类加载（如 ORM 框架、热部署）
# - anonymous_class_space 增长：反射生成的匿名类
# - Class loader metaspace 增长：类加载器泄漏
```

```java
// 配合 -verbose:class 可以定位是哪个类加载器泄漏
// 启动参数添加：
java -XX:+TraceClassLoading -XX:+TraceClassUnloading myapp.jar

// 输出示例（可以看到类加载器）：
# [Loaded sun.reflect.GeneratedConstructorAccessor123 
#  from: jdk.internal.reflect.GeneratedConstructorAccessorImpl]
# 
# [Loaded com.example.MyClass_$$_jvst... 
#  from: org.hibernate.proxy.HibernateProxyFactory]
# 
# 如果看到某个 ClassLoader 反复加载新类 → 泄漏！
```

### 🔥 扩展追问 4：如何用 NMT 发现 CodeCache 相关的性能问题？

> **A:** CodeCache 被填满是 JVM 中比较隐形的性能杀手，NMT 可以提前发现：

```bash
# CodeCache 相关参数
-XX:ReservedCodeCacheSize=256M   # 默认 256MB
-XX:InitialCodeCacheSize=8M      # 初始 8MB

# 通过 NMT 监控 CodeCache 使用
jcmd <pid> VM.native_memory summary

# 典型输出：
# -                     Code (reserved=262144KB, committed=262144KB)
#                           malloc://262144KB
# 
# reserved == committed → CodeCache 已接近上限！

# 如果 CodeCache 满了，JIT 编译器会被抑制：
# - 方法永远停留在解释执行模式
# - 应用性能急剧下降
# - 日志会出现：
#   JVM warning: CodeCache is full. Compiler has been disabled.
#   JVM warning: Try increasing the code cache size. 
#                 -XX:ReservedCodeCacheSize=XX

# 解决：
# -XX:ReservedCodeCacheSize=512M  # 扩大 CodeCache
# -XX:+ReduceCodeCache             # JDK 17+ 减少冗余代码
```

```
CodeCache 的组成（NMT detail 视图）：
┌──────────────────────────────────────────────────────────────┐
│  CodeCache = JIT 编译生成的机器码存储区                      │
│                                                              │
│  内部分区（由 NMT detail 可看到）：                          │
│  ├─ Non-method: 字节码解释器代码 + 内部 JVM 代码            │
│  ├─ Profiled Code: C1 编译的代码（Tier 1-2）               │
│  └─ Non-profiled Code: C2 编译的代码（Tier 3-4）           │
│                                                              │
│  JDK 17+ 的改进：Profiled 和 Non-profiled 分离后           │
│  可以用不同参数控制：                                       │
│  -XX:ProfiledCodeCacheSize   # 控制 Profiled 区            │
│  -XX:NonProfiledCodeCacheSize # 控制 Non-profiled 区       │
└──────────────────────────────────────────────────────────────┘
```

### 🔥 扩展追问 5：NMT 和 `pmap`、`jemalloc` 输出的区别是什么？

> **A:** 三个工具的视角不同，各有用途：

| 工具 | 视角 | 能看到什么 | 适合场景 |
|---|---|---|---|
| **NMT** | JVM 内部 | JVM 分类的内存（堆、CodeCache、线程等） | JVM 内部内存问题诊断 |
| **pmap** | OS 进程 | 进程的虚拟地址空间映射 | 看 reserved 地址范围 |
| **jemalloc** | 操作系统 | malloc 分配详情（需加 `-XX:UseMalloc`） | 第三方库内存泄漏 |

```bash
# pmap 查看进程的虚拟地址空间
pmap -x <pid> | head -50

# 输出示例：
# 00007f8c40000000  524288   524288   524288   rw---  [ anon ]  ← 堆（reserved）
# 00007f8c80000000  262144   262144   262144   rw---  [ anon ]  ← CodeCache
# 00007f8c90000000   65536    65536    65536   rw---  [ anon ]  ← Metaspace
# ...

# NMT vs pmap 的对应关系：
# pmap 的 [ anon ] 区域 = NMT 各类别的 malloc 总和
# 如果 pmap 显示的 RSS > NMT committed 之和 → 有 native 代码直接 malloc
# （如 DirectByteBuffer、JNI 库等）

# 常用诊断组合
jcmd <pid> VM.native_memory summary  # 看 JVM 分类
pmap -x <pid>                         # 看 OS 视角
jstat -gc <pid> 1000                  # 看 GC 统计
```

## 3. 一句话总结

> **NMT 通过在 JVM 内部记录每块内存的分配类别，把原本只能看 RSS 的 OS 层面视角，提升到堆内/CodeCache/线程/GC/Compiler 等 JVM 内部维度，summary 模式只有 2% 以内的性能损耗，是排查 Metaspace 泄漏、CodeCache 满、JVM 进程内存超限等问题的首选诊断工具。**

## 4. 面试加分话术

> "我之前遇到过一个问题：Java 进程占用的物理内存（通过 `pmap` 看到的 RSS）比 `-Xmx` 大了 4GB，但堆明明设的是 8GB。查了一圈发现是 `DirectByteBuffer` 的问题——网络框架用了很多堆外内存，但没主动释放。后来用 NMT 一对比，发现 CodeCache 占用了 1.5GB、Metaspace 占 1GB、DirectByteBuffer 占 800MB、线程栈占 500MB，加起来正好多了 4GB。这个案例后来成为我们团队的 NMT 使用培训教材。"

> "关于 NMT 的开销，我一直跟团队说：生产环境开 summary 模式的 NMT 完全没问题，2% 的吞吐量损耗比 GC 停顿带来的业务损失小得多。而且 NMT 可以通过 baseline/diff 的方式做增量分析，特别适合排查内存增长的问题——先做一次 baseline，运行一周后再 diff，就能精确定位是哪块内存在涨，而不是对着一个大的 RSS 数字干瞪眼。"
