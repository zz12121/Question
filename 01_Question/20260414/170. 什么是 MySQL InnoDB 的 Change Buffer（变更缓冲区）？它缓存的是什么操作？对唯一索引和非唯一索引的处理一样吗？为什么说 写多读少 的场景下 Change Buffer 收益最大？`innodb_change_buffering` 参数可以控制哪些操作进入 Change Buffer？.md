# 170. MySQL InnoDB 的 Change Buffer（变更缓冲区）

## 1. 基础流程

### 1.1 Change Buffer 的定位

```
Change Buffer 的设计背景——为什么需要它？
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  传统写入流程（无 Change Buffer）：                          │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  UPDATE orders SET status='paid' WHERE user_id=123  │   │
│  │       ↓                                               │   │
│  │  1. 读取 user_id=123 对应的 B+Tree 索引页          │   │
│  │     → 磁盘 I/O！（随机读）                         │   │
│  │  │                                                   │   │
│  │  ← 2. 修改索引页（把 user_id=123 标记为更新）      │   │
│  │  │                                                   │   │
│  │  ← 3. 写回磁盘                                       │   │
│  └──────────────────────────────────────────────────────┘   │
│  问题：每次 UPDATE 都要随机读取索引页 → I/O 开销巨大      │
│                                                              │
│  Change Buffer 后的写入流程：                               │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  UPDATE orders SET status='paid' WHERE user_id=123  │   │
│  │       ↓                                               │   │
│  │  1. 将变更操作记录到 Change Buffer（内存 + 磁盘）   │   │
│  │     → 无需读取磁盘！写入 Buffer Pool 的日志结构      │   │
│  │  2. 立刻返回客户端                                  │   │
│  │  3. 后台 Master Thread 合并：将 Change Buffer       │   │
│  │     中的变更应用到实际的 B+Tree 索引页              │   │
│  └──────────────────────────────────────────────────────┘   │
│  优势：写操作完全无 I/O 等待，合并操作在后台批量完成       │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 Change Buffer 缓存的是什么

```
Change Buffer 的核心能力——它不是简单的写缓存：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  Change Buffer = 记录对"二级索引"修改操作的缓冲区           │
│                                                              │
│  记录的修改操作类型：                                        │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  INSERT：     插入一条索引记录                       │   │
│  │  DELETE-MARK：标记一条索引记录为已删除               │   │
│  │  UPDATE：     更新一条索引记录                       │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  存储的核心信息：                                           │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  Change Buffer Entry = {                             │   │
│  │      space_id: 表空间ID（哪个表/索引）               │   │
│  │      page_no:  页号（二级索引页）                   │   │
│  │      op_type: 操作类型（INSERT/DELETE/UPDATE）       │   │
│  │      data:    被修改的索引键值                       │   │
│  │  }                                                    │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  合并时机（Merge）：                                        │
│  1. Master Thread 后台定时合并（每 1 秒/10 秒）             │
│  2. 读取二级索引页时（读缓存 + 合并）                     │
│  3. 数据库空闲时（innodb_idle_flush_pct）                  │
└──────────────────────────────────────────────────────────────┘
```

### 1.3 唯一索引 vs 非唯一索引

```
Change Buffer 对不同索引类型的不同策略：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  非唯一二级索引（✅ 可以使用 Change Buffer）：               │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  CREATE TABLE orders (                              │   │
│  │      id BIGINT PRIMARY KEY,                         │   │
│  │      user_id BIGINT,  ← 非唯一二级索引              │   │
│  │      INDEX idx_user_id (user_id)                    │   │
│  │  );                                                 │   │
│  │                                                       │   │
│  │  UPDATE orders SET status='paid' WHERE user_id=123 │   │
│  │  → 需要修改 idx_user_id 索引                       │   │
│  │  → user_id=123 是否存在 → 未知！                   │   │
│  │  → 可以安全地先缓存，后续再合并检查                 │   │
│  │  → ✅ Change Buffer 生效                            │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  唯一二级索引（❌ 不能使用 Change Buffer）：                │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  CREATE TABLE orders (                              │   │
│  │      id BIGINT PRIMARY KEY,                         │   │
│  │      order_no VARCHAR(64) UNIQUE, ← 唯一索引       │   │
│  │      INDEX idx_order_no (order_no)                  │   │
│  │  );                                                 │   │
│  │                                                       │   │
│  │  INSERT INTO orders (order_no) VALUES ('A001')     │   │
│  │  → 必须立即检查 order_no='A001' 是否已存在        │   │
│  │  → 唯一性约束必须在操作返回前确认                  │   │
│  │  → ❌ 无法延迟，必须立即读取磁盘加载索引页       │   │
│  │  → Change Buffer 不适用！                         │   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

## 2. 扩展追问

### 🔥 扩展追问 1：为什么说"写多读少"的场景下 Change Buffer 收益最大？

> **A:** Change Buffer 的本质是用内存换 I/O，适合的场景是写入量大但读取量小的场景：

```
Change Buffer 的收益模型：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  场景1：写多读少（Change Buffer 收益最大化）               │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  写入：10万次/秒 二级索引更新                       │   │
│  │  读取：100次/秒  二级索引查询                      │   │
│  │                                                       │   │
│  │  无 Change Buffer：                                  │   │
│  │  → 每次写入都要随机读磁盘加载索引页                │   │
│  │  → 10万次 × 1次磁盘读 = 10万次 I/O/秒            │   │
│  │                                                       │   │
│  │  有 Change Buffer：                                  │   │
│  │  → 写入全部写入 Change Buffer（0次磁盘读）        │   │
│  │  → 读取时触发合并：最多 100次/秒 合并              │   │
│  │  → 10万次写入 → 0次 I/O；100次读取 → 100次 I/O   │   │
│  │  → 总计：100次 I/O/秒（vs 10万次）               │   │
│  │                                                       │   │
│  │  收益：I/O 降低 1000 倍！                          │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  场景2：写多读也多（Change Buffer 反而有副作用）           │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  写入：10万次/秒 二级索引更新                       │   │
│  │  读取：10万次/秒 二级索引查询                      │   │
│  │                                                       │   │
│  │  写入全部写入 Change Buffer                         │   │
│  │  读取每次都要触发合并                              │   │
│  │  → 合并操作本身带来额外开销                        │   │
│  │  → 10万次/秒 合并 → CPU 开销 + 内存开销           │   │
│  │  → Change Buffer 变成了负担                         │   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

```sql
-- 典型适合 Change Buffer 的场景：
-- 1. 日志型应用：大量 INSERT，很少 SELECT
-- 2. 数据采集/IoT：每秒写入几万条，读很少
-- 3. 报表系统：白天写入，晚上批量读
-- 4. 订单流水表：大量 INSERT 记录，查询往往走主键

-- 不适合 Change Buffer 的场景：
-- 1. 主从读写分离：写少读多，且从库是主要读库
-- 2. 实时报表系统：写入即读取（OLTP 混合负载）
-- 3. 配置表：频繁读，很少写
```

### 🔥 扩展追问 2：`innodb_change_buffering` 参数可以控制哪些操作？

> **A:** MySQL 提供了细粒度的参数来控制哪些类型的操作使用 Change Buffer：

```sql
-- 查看当前配置
SHOW VARIABLES LIKE 'innodb_change_buffering';

-- 参数可选值（MySQL 5.6+）：
-- ┌──────────────────────────────────────────────────────┐
-- │  innodb_change_buffering = 'all'    ← 默认，全开     │
-- │    → INSERT + DELETE_MARKING + DELETE + PURGE      │
-- │                                                        │
-- │  innodb_change_buffering = 'none'   ← 全关            │
-- │    → Change Buffer 完全禁用                          │
-- │                                                        │
-- │  innodb_change_buffering = 'inserts' ← 仅插入         │
-- │    → 只缓存 INSERT 操作                              │
-- │                                                        │
-- │  innodb_change_buffering = 'deletes' ← 仅删除标记     │
-- │    → 只缓存 DELETE_MARKING                          │
-- │                                                        │
-- │  innodb_change_buffering = 'changes' ← 插入+删除标记  │
-- │    → INSERT + DELETE_MARKING                        │
-- │                                                        │
-- │  innodb_change_buffering = 'purges'  ← 仅 purge      │
-- │    → 只缓存后台 purge 线程的物理删除                 │
-- │                                                        │
-- │  innodb_change_buffering = 'read_only' ← 只读模式     │
-- │    → 从库配置：不生成新的 Change Buffer 条目        │
-- └──────────────────────────────────────────────────────┘

-- my.cnf 配置示例（电商场景，写多读少）
[mysqld]
innodb_change_buffering = all  # 默认全开即可

-- 如果读多写少，可以关闭 Change Buffer
SET GLOBAL innodb_change_buffering = 'none';

-- 监控 Change Buffer 的状态
SHOW ENGINE INNODB STATUS\G

-- 关键输出：
-- -------------------------------------
-- INSERT BUFFER AND ADAPTIVE HASH INDEX
-- -------------------------------------
-- Ibuf: size 1, free list len 0, seg size 2, 0 merges
-- merged operations:
--  insert 120000, delete mark 30000, delete 5000    ← 已合并的操作数
-- discards: 0                                                 ← 丢弃的操作数
-- Ibuf size: 10000   ← 当前 Change Buffer 大小（内存中的部分）
-- 0.00 hash searches/s, 98.00 non-hash searches/s
```

### 🔥 扩展追问 3：Change Buffer 的内存结构和大小限制是什么？

> **A:** Change Buffer 有独立的内存区域，它的大小受 `innodb_change_buffer_max_size` 参数控制：

```
Change Buffer 的内存布局：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  Change Buffer 的物理存储（内存 + 磁盘）：                   │
│                                                              │
│  内存部分：ibuf 的 btree（系统表空间中）                    │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  ibuf btree = Change Buffer 的 B+Tree 结构           │   │
│  │  └─ key = (space_id, page_no)                       │   │
│  │  └─ value = {op_type, data}                        │   │
│  │  └─ 存在系统表空间（system tablespace）的 ibdata 文件│   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  持久化：                                                   │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  写入 Change Buffer 时 → 同时写入 redo log           │   │
│  │  → 即使 MySQL 崩溃，Change Buffer 记录不丢失       │   │
│  │  → 恢复时从 redo log 重做                           │   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

```sql
-- 控制 Change Buffer 的最大内存占用
SHOW VARIABLES LIKE 'innodb_change_buffer_max_size';
-- 默认值：25（即占 Buffer Pool 的 25%，默认）

-- 计算方式：
-- 如果 innodb_buffer_pool_size = 8GB
-- innodb_change_buffer_max_size = 25（默认）
-- → Change Buffer 最大 = 8GB × 25% = 2GB

-- 实际使用量：
SHOW ENGINE INNODB STATUS\G
-- Ibuf: size 1, seg size 2
-- seg size = 实际的 Change Buffer 大小（以 buffer pool 页为单位）
-- 每个 buffer pool 页 = innodb_page_size（默认 16KB）
-- seg size 2 = 2 × 16KB = 32KB 当前使用的内存

-- 调优建议：
-- 写多场景：增大到 50（最大）
SET GLOBAL innodb_change_buffer_max_size = 50;

-- 读多写少场景：减小到 10 或 5
SET GLOBAL innodb_change_buffer_max_size = 10;

-- 查看 Change Buffer 的合并状态
SHOW GLOBAL STATUS LIKE 'Innodb_ibuf%';
-- Innodb_ibuf_size: 10000   ← 当前 Change Buffer 大小
-- Innodb_ibuf_count_insert: 1000  ← 插入操作计数
-- Innodb_ibuf_merge_count: 500    ← 已合并的操作数
```

### 🔥 扩展追问 4：Change Buffer 和 Insert Buffer 是什么关系？

> **A:** 这是 MySQL 发展过程中的历史遗留问题，Insert Buffer 是 Change Buffer 的前身：

```
Insert Buffer → Change Buffer 的演进：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  MySQL 5.5 及之前：                                         │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  Insert Buffer（只处理 INSERT）                       │   │
│  │  → 只能缓存 INSERT 操作                             │   │
│  │  → DELETE 和 UPDATE 的二级索引变更无法缓存           │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  MySQL 5.6 及之后（InnoDB 1.2.x）：                        │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  Change Buffer = Insert Buffer 的扩展                │   │
│  │  → 缓存 INSERT + DELETE_MARK + DELETE + UPDATE      │   │
│  │  → 向后兼容：'inserts' 参数值等价于旧版 insert buffer │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  为什么改名？                                               │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  Insert Buffer 的名字具有误导性：                     │   │
│  │  1. 它不只处理 INSERT，也处理 DELETE 和 UPDATE        │   │
│  │  2. 它是"二级索引变更"的缓冲区，非仅"插入"缓冲区  │   │
│  │  → MySQL 团队将其重命名为 Change Buffer              │   │
│  │  → 更准确地反映其真实功能                            │   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

### 🔥 扩展追问 5：Change Buffer 合并时如果索引页已被淘汰出 Buffer Pool 怎么办？

> **A:** 这是 Change Buffer 合并机制的核心处理逻辑——如果目标页不在 Buffer Pool 中，合并会被推迟：

```
Change Buffer 合并的完整流程：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  Master Thread 触发合并时：                                 │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  1. 从 ibuf btree 中取出一个 Change Buffer Entry      │   │
│  │  │   Entry = {space_id=5, page_no=128, op=INSERT}   │   │
│  │  │                                                      │   │
│  │  2. 尝试从 Buffer Pool 中读取目标页                   │   │
│  │  │   页(space_id=5, page_no=128) 在 Buffer Pool 吗？ │   │
│  │  │                                                      │   │
│  │  ├─ 如果在 Buffer Pool 中：                           │   │
│  │  │    → 直接应用 Change Buffer Entry                  │   │
│  │  │    → 更新 Buffer Pool 中的索引页                   │   │
│  │  │    → 标记页为 dirty（后续由 flush 线程写盘）      │   │
│  │  │                                                      │   │
│  │  ├─ 如果不在 Buffer Pool 中（已被淘汰）：             │   │
│  │  │    → 不做任何事                                    │   │
│  │  │    → Change Buffer Entry 保留在 ibuf btree 中    │   │
│  │  │    → 下次读取该页时再触发合并                     │   │
│  │  │    → 直到被读取前，Change Buffer Entry 一直存在   │   │
│  │  │                                                      │   │
│  │  └─ 如果页已被purge（物理删除）：                    │   │
│  │       → 直接丢弃该 Change Buffer Entry               │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  读触发合并的场景（Lazy Merge）：                           │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  SELECT * FROM orders WHERE user_id=123             │   │
│  │       ↓                                              │   │
│  │  读取 user_id=123 的索引页（page 128）              │   │
│  │       ↓                                              │   │
│  │  检测到 Change Buffer 中有 page 128 的待合并操作   │   │
│  │       ↓                                              │   │
│  │  读取旧索引页 → 合并 Change Buffer → 返回最新数据  │   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

## 3. 一句话总结

> **Change Buffer 是 MySQL 用来缓存对二级索引修改操作的内存结构（唯一索引因唯一性约束必须立即读取磁盘而无法使用），它把随机的索引页读取变成顺序的 Change Buffer 写入，让"写多读少"场景下的 I/O 开销降低几个数量级，通过 Master Thread 后台合并或读取时触发合并将变更应用到实际的 B+Tree 索引页，并通过 redo log 确保持久化安全。**

## 4. 面试加分话术

> "我之前帮一个 IoT 平台做 MySQL 性能优化，他们的设备状态表每秒写入 5 万条记录，但查询主要是按设备 ID 查主键，状态变化几乎不查。用 `SHOW ENGINE INNODB STATUS` 一看，`Ibuf: size` 只有 200，说明 Change Buffer 根本没被用上——后来排查发现所有的二级索引都是唯一索引（设备编号 unique），唯一索引无法使用 Change Buffer，这可能就是设计时的一个失误。后面他们加了一个非唯一的时间戳索引做分页查询，那部分查询的延迟直接从 15ms 降到了 2ms，就是因为 Change Buffer 生效了。"

> "关于 Change Buffer 的监控，我一直推荐团队关注两个指标：`Innodb_ibuf_size`（当前 Change Buffer 大小）和 `Innodb_ibuf_merges`（合并次数）。如果 Change Buffer 满了（`innodb_change_buffer_max_size` 达到上限），新的变更只能直接写磁盘，Change Buffer 形同虚设。这时候要么调大 `innodb_change_buffer_max_size`，要么就得检查为什么写入量突然这么大。"
