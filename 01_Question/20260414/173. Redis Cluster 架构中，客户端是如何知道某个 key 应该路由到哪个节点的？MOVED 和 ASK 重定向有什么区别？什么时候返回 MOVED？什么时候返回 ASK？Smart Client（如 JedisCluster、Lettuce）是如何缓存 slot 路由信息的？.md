# Q173: Redis Cluster 路由机制与重定向策略

## 1. 基础流程

### 1.1 Redis Cluster 分片架构

```
Redis Cluster 将 16384 个 Slot 分布到 N 个节点

┌─────────────────────────────────────────────────────────┐
│  Redis Cluster 拓扑（6 节点，3 主 3 从）               │
│                                                         │
│         Slot 范围分配：                                 │
│                                                         │
│   Node-A (Master)  ──→ Slot 0-5460                     │
│   Node-B (Master)  ──→ Slot 5461-10922                 │
│   Node-C (Master)  ──→ Slot 10923-16383                │
│                                                         │
│   Node-A1 (Slave of A) ←── 主从同步                    │
│   Node-B1 (Slave of B)                                 │
│   Node-C1 (Slave of C)                                 │
│                                                         │
│   客户端路由：                                          │
│   key → CRC16(key) % 16384 → Slot → 节点 IP:PORT       │
└─────────────────────────────────────────────────────────┘
```

```
CRC16 散列算法（Redis Cluster 专用）：

公式：slot = CRC16(key) % 16384

CRC16 原始算法产生的值范围：0-65535
取模 16384 后映射到 0-16383 的 Slot 范围

Redis 源码：
unsigned int crc16(const char *key, unsigned int len) {
    // CRC16-CCITT 多项式
    // x^16 + x^12 + x^5 + 1
}

// Slot 计算
unsigned int keyHashSlot(const char *key, int keylen) {
    unsigned int hash = crc16(key, keylen);
    return hash % HASH_SLOTS;  // HASH_SLOTS = 16384
}

注意：
- 同一个 key 永远映射到同一个 Slot
- Slot 不等于 Node，Slot 到 Node 的映射是动态的（集群扩缩容时）
```

### 1.2 客户端路由 key 到节点的过程

```
完整路由决策流程：

┌─────────────────────────────────────────────────────────┐
│  客户端发起请求: GET mykey                              │
│                                                         │
│  Step 1: 计算 Slot                                      │
│    CRC16("mykey") = 12543                               │
│    12543 % 16384 = 12543                                │
│    → Slot = 12543                                       │
│                                                         │
│  Step 2: 查找 Slot → Node 映射                          │
│    查本地缓存路由表:                                     │
│      Slot 12543 → Node-B (192.168.1.2:6379)            │
│                                                         │
│  Step 3: 直接连接目标节点                               │
│    直接向 Node-B 发送 GET mykey                          │
└─────────────────────────────────────────────────────────┘

第一次连接（或缓存失效）时：
┌─────────────────────────────────────────────────────────┐
│  客户端发起: GET mykey                                  │
│                                                         │
│  本地无 Slot 缓存 → 随机选一个节点询问                   │
│  发送: CLUSTER SLOTS（获取全量槽映射）                  │
│                                                         │
│  服务端返回:                                             │
│    [0, 5460, ["192.168.1.1", "6379"], ["192.168.1.4", "6379"]] │
│    [5461, 10922, ["192.168.1.2", "6379"], ...]          │
│    [10923, 16383, ["192.168.1.3", "6379"], ...]         │
│                                                         │
│  客户端构建本地 Slot → Node 路由表                      │
│  然后按表路由请求                                        │
└─────────────────────────────────────────────────────────┘
```

## 2. MOVED vs ASK 重定向

### 2.1 两者对比表

| 维度 | MOVED | ASK |
|------|-------|-----|
| 触发时机 | Slot 永久迁移完成 | Slot 迁移进行中（中间状态） |
| Slot 所有权 | 已切换到新节点 | 仍在旧节点（迁移中） |
| 客户端行为 | 更新本地缓存 | 临时去新节点（不更新缓存） |
| 是否重新发送请求 | 是（用新节点重试） | 是（本次去新节点） |
| 后续请求受影响 | 否（缓存已更新） | 否（下次仍走旧节点） |
| 错误码 | -MOVED | -ASK |

### 2.2 MOVED 重定向（Slot 已迁移完成）

```
场景：Slot 12543 已完全从 Node-B 迁移到 Node-D

客户端行为：

客户端:  GET mykey        ──→ Node-B（假设缓存未更新）
Node-B:  -MOVED 12543 192.168.1.4:6379
         │
         │ 客户端收到 MOVED 后：
         ↓
         1. 更新本地 Slot 缓存：Slot 12543 → Node-D
         2. 立即用新节点重试
         3. 后续所有请求直接走 Node-D（无需再重定向）

通信过程：
Client ──→ Node-B:  GET mykey
Client ←── Node-B:  -MOVED 12543 192.168.1.4:6379

Client ──→ Node-D:  GET mykey   ← 重试
Client ←── Node-D:  value
```

### 2.3 ASK 重定向（Slot 正在迁移中）

```
场景：Slot 12543 正在从 Node-B 迁移到 Node-D（迁移未完成）

┌─────────────────────────────────────────────────────────┐
│  迁移中间状态：                                          │
│                                                         │
│  Slot 12543                                             │
│  ┌──────────────┐      ┌──────────────┐               │
│  │  Node-B      │      │  Node-D      │               │
│  │  (源节点)     │ ──→  │  (目标节点)   │               │
│  │              │      │              │               │
│  │  迁移中...   │      │  已收到部分   │               │
│  │  key 还在这  │      │  key 已在这  │               │
│  └──────────────┘      └──────────────┘               │
│                                                         │
│  Slot 12543 的归属权：仍在 Node-B                       │
│  新增的 key：写入 Node-D（新值在 D）                    │
│  旧的 key：读 Node-B（旧值在 B）                        │
└─────────────────────────────────────────────────────────┘

ASK 重定向通信过程：

客户端:  GET mykey        ──→ Node-B（按缓存路由）
Node-B:  -ASK 12543 192.168.1.4:6379
         │
         │ 客户端收到 ASK 后（不更新缓存）：
         ↓
         1. 先发 ASKING 命令（通知目标节点"允许访问迁移中槽"）
         2. 再发原请求 GET mykey 到 Node-D
         3. 本地缓存不变（下次仍走 Node-B）

通信细节：
Client ──→ Node-B:  GET mykey
Client ←── Node-B:  -ASK 12543 192.168.1.4:6379

Client ──→ Node-D:  ASKING
Client ←── Node-D:  +OK

Client ──→ Node-D:  GET mykey
Client ←── Node-D:  value

下次请求（缓存未更新）：
Client ──→ Node-B:  GET mykey   ← 仍然走旧节点
```

### 2.3 ASK vs MOVED 的设计意图

```
为什么需要两种重定向？

MOVED = "Slot 已搬家，永久更新缓存"
ASK   = "Slot 正在搬家，临时去邻居看一眼"

ASK 的设计哲学：
- 避免 MOVED 风暴（迁移中大量 MOVED 导致客户端缓存频繁失效）
- 迁移期间：新 key 写新节点，旧 key 读旧节点
- 迁移完成后：旧节点通知客户端 → 变成 MOVED → 永久切换

┌─────────────────────────────────────────┐
│  MOVED 重定向次数：N（N = key 数量）     │
│  ASK   重定向次数：2N（迁移期间每次都 ASK）│
│                                         │
│  但 ASK 不触发缓存更新，开销更小         │
└─────────────────────────────────────────┘
```

## 3. Smart Client（Jedis / Lettuce）如何缓存 Slot 路由

### 3.1 JedisCluster 路由缓存机制

```java
// JedisCluster 核心结构
public class JedisCluster {
    
    // Slot → Node 映射缓存
    private final JedisClusterConnectionHandler connectionHandler;
    
    // 获取连接时自动刷新路由
    public String get(String key) {
        // 1. 计算 slot
        int slot = JedisClusterHashCRC16.getSlot(key);
        
        // 2. 从缓存获取 node
        JedisPooled connection = cache.get(slot);
        
        // 3. 发送命令
        try {
            return connection.get(key);
        } catch (JedisMovedDataException e) {
            // MOVED 重定向 → 更新缓存
            renewSlotCache(e);
            // 重试
            return get(key);
        } catch (JedisAskDataException e) {
            // ASK 重定向 → 不更新缓存，临时访问
            return askToSlot(e, key);
        }
    }
}
```

```
JedisCluster 缓存更新流程：

┌─────────────────────────────────────────────────────────┐
│  MOVED 触发缓存更新                                      │
│                                                         │
│  1. 收到 -MOVED 12543 192.168.1.4:6379                 │
│  2. 解析出新节点地址                                     │
│  3. 更新本地 Slot 缓存: Slot 12543 → Node-D            │
│  4. 后续所有请求直接走 Node-D（无需重定向）             │
│                                                         │
│  ASK 不触发缓存更新                                     │
│                                                         │
│  1. 收到 -ASK 12543 192.168.1.4:6379                   │
│  2. 解析出新节点地址                                     │
│  3. 发送 ASKING + 原命令到 Node-D                       │
│  4. 本地 Slot 缓存不变                                  │
│  5. 下次请求仍走原节点                                  │
└─────────────────────────────────────────────────────────┘
```

### 3.2 Lettuce 的路由缓存机制

```java
// Lettuce（更现代，支持响应式）
RedisClusterClient client = RedisClusterClient.create(RedisURI.create("redis://192.168.1.1:6379"));

// Lettuce 自动维护 Partitions（类似 Slot 映射表）
StatefulRedisClusterConnection<String, String> connection = 
    client.connect();

// Partitions 自动刷新策略：
// 1. 启动时：CLUSTER SLOTS 获取全量映射
// 2. MOVED 重定向时：自动更新 Partitions
// 3. 定期后台刷新：PARTITIONRefreshMode.ADVANCED（默认）
// 4. 手动刷新：connection.setAutoFlushCommands(false)

// ASK 处理：Lettuce 自动处理，无需手动
// - 检测到 ASK 重定向
// - 自动发送 ASKING
// - 临时路由到目标节点
```

### 3.3 缓存失效与重新加载

```
Smart Client vs Dumb Client 对比：

┌─────────────────────────────────────────────────────────┐
│  Smart Client（JedisCluster / Lettuce / Redisson）     │
│                                                         │
│  ✓ 本地缓存 Slot → Node 映射                           │
│  ✓ MOVED 自动更新缓存                                   │

│                                                         │

│    MOVED ──→ 收到后立即更新缓存（对应 Slot）           │

│  redis-cli 处理重定向：                                  │
```

### 3.4 重定向次数优化

```
JedisCluster 重定向次数实测：

场景1：冷启动（缓存为空）
  GET mykey → Node-B → MOVED → Node-D → value
  = 2 次请求（1 次 MOVED + 1 次实际）

场景2：缓存已更新
  GET mykey → Node-D → value
  = 1 次请求（零重定向）

场景3：迁移进行中
  GET mykey → Node-B → ASK → Node-D → value
  = 3 次请求（1 次 ASK + 1 次 ASKING + 1 次实际）

优化建议：
1. 启动时预热：连接集群后执行 CLUSTER SLOTS 预加载缓存
2. 开启拓扑刷新：后台定期执行 CLUSTER NODES 同步变化
3. 避免热点 key 迁移：迁移前先复制，迁移后切换
4. Pipeline 谨慎使用：MOVED 时整个 Pipeline 需重试
```

## 🔥 扩展追问 1：为什么是 16384 个 Slot？不是 65536 或其他数字？

> **A:** 16384 是 Redis 作者 Antirez 在性能和内存之间的权衡：

```
设计考量：

1. Slot 数量决定网络包大小
   每个节点需要告诉客户端："我的 Slot 是哪些"
   用 bitmap 表示：16384 bit = 2 KB
   如果是 65536 slot = 8 KB（网络包更大）
   Cluster 节点之间每 100ms 交换 PONG 包，包含自己的 Slot 信息

2. Slot 足够多吗？
   16384 >> 集群节点数（一般最多 1000+ 节点）
   平均每个节点负责 ~16384/N 个 Slot
   足够细粒度分布

3. 为什么不是更多（如 2^16=65536）？
   - 每个 PING/PONG 包更大
   - 节点间 Gossip 通信带宽增加
   - 内存中 Slot 数组占用更多

实际场景：
100 节点集群，每个节点平均 ~164 个 Slot
迁移时影响范围小，节点间协调成本低
```

## 🔥 扩展追问 2：JedisCluster Pipeline 遇到 MOVED 重定向怎么办？

> **A:** 这是 JedisCluster Pipeline 的经典陷阱：

```java
// 问题代码
JedisCluster cluster = new JedisCluster(...);
try (JedisPooled client = cluster.getConnectionHandler().getConnection()) {
    // Pipeline 内 10 个 key，可能分布在不同 Slot
    Pipeline p = client.pipelined();
    for (int i = 0; i < 10; i++) {
        p.set("key" + i, "value" + i);
    }
    p.sync();  // 如果任何一个 key MOVED，整个 Pipeline 可能失败
}

// 解决方案1：使用 JedisSlotBasedPipeline（Jedis 2.10+）
// 自动按 Slot 分组，分开执行
JedisClusterPipeline pipeline = new JedisClusterPipeline(...);
pipeline.refreshClusterInfo();
pipeline.set("key1", "value1");
pipeline.set("key2", "value2");
// 内部按 Slot 分组，不同 Slot 的命令发给不同节点
pipeline.sync();

// 解决方案2：手动按 Slot 分组
Map<Integer, List<String>> slotToKeys = new HashMap<>();
for (int i = 0; i < 100; i++) {
    int slot = JedisClusterHashCRC16.getSlot("key" + i);
    slotToKeys.computeIfAbsent(slot, k -> new ArrayList<>()).add("key" + i);
}
// 按 Slot 分组后，每组单独 Pipeline 执行
```

## 🔥 扩展追问 3：Redis Cluster 的主从切换期间，客户端会受影响吗？

> **A:** 会，但 Smart Client 有 failover 感知机制：

```
故障转移流程与客户端感知：

┌─────────────────────────────────────────────────────────┐
│  场景：Node-B（Master）宕机                              │
│                                                         │
│  Step 1: Node-B1（从节点） 感知到主节点失联              │
│  Step 2: Node-B1 发起选举（投票）                                               │
│  Step 3: Node-B1 升级为 Master                          │
│  Step 4: 更新 Slot 12543 → Node-B1                     │
│                                                         │
│  客户端受影响时间窗口：                                   │
│  ① 宕机检测延迟（默认 500ms）                          │
│  ② 选举 + 升级（秒级）                                  │
│  ③ MOVED 重定向（客户端收到后更新缓存）                 │
└─────────────────────────────────────────────────────────┘

Smart Client 处理主从切换：
1. MOVED 重定向 → 缓存更新 → 自动切到新 Master
2. 连接池重建 → JedisCluster 自动重建与新 Master 的连接

问题：
- 切换期间请求失败（failover 未完成时）
- 建议：客户端配置重试（ JedisCluster 支持 maxTotal/maxAttempts）
```

## 🔥 扩展追问 5：MULTI/EXEC 事务在 Redis Cluster 中有什么限制？

> **A:** Redis Cluster 不支持跨 Slot 事务，这是核心限制：

```
Redis Cluster 事务限制：

┌─────────────────────────────────────────────────────────┐
│  MULTI/EXEC 事务要求：所有 key 必须在同一个 Slot         │
│  且同一个节点上                                          │
└─────────────────────────────────────────────────────────┘

错误示例：
  WATCH key1      -- Slot 100
  WATCH key2      -- Slot 200  ← 不同 Slot！
  MULTI
  GET key1
  GET key2
  EXEC
  --> 报错：CROSSSLOT Keys in request

正确做法：使用 Hash Tag 强制同 Slot

  用户想跨节点操作两个 key：
  user:100:profile  (user_id=100)   → Slot A
  user:200:profile  (user_id=200)   → Slot B

  用 Hash Tag 强制同 Slot：
  user:{100}:profile   → Slot X
  user:{100}:cache     → Slot X  ← 同 Slot，事务可用

Hash Tag 规则：
  key 中的 {xxx} 部分参与 CRC16 计算
  如果 key 中有 {}，则只用 {} 内的内容计算 slot
  如果 key 中无 {}，则用完整 key 计算 slot

其他限制：
  - 不支持 SELECT（切换 db，集群只有 db0）
  - 不支持 MOVE（跨节点移动 key）
```

---

> **一句话总结**

Redis Cluster 通过 `CRC16(key) % 16384` 将 key 映射到 Slot，每个 Slot 永久属于某个 Master 节点，客户端本地缓存 Slot→Node 映射表后直接路由；**MOVED** 表示 Slot 已完全迁移到新节点，客户端需更新本地缓存永久切换；**ASK** 表示 Slot 正在迁移中，客户端仅发 `ASKING` 命令临时访问新节点但不更新缓存；Smart Client（JedisCluster/Lettuce）通过启动时 `CLUSTER SLOTS` 预加载路由表、MOVED 时实时更新缓存、ASK 时自动处理的三层策略最小化重定向次数。

> **面试加分话术**

> "我在生产环境遇到过 Redis Cluster 迁移时的重定向风暴问题——当时我们从 3 主 3 从扩到 6 主 6 从，数据迁移期间大量请求收到 MOVED，导致客户端缓存频繁失效，QPS 抖动了 30%。分析下来是因为我们的 JedisCluster 没有配置拓扑刷新，后台线程没开，导致缓存一直是最旧的。后来我们做了三件事：1）开启 `JedisClusterConnectionHandler` 的定时 `CLUSTER NODES` 刷新；2）迁移前先 `SCAN` 出热点 key，优先迁移；3）用 Hash Tag 确保业务事务的多个 key 落在同一 Slot，这样 `MULTI/EXEC` 事务才能正常工作。切流量时采用灰度切 5%→20%→100%，把 MOVED 重定向控制在 1% 以内。"
