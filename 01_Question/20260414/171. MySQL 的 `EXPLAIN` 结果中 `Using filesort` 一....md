## 1. 基础流程

### 1.1 `Using filesort` 的真实含义

```
⚠️ 关键误解澄清：
"filesort" ≠ "file-based sort"（磁盘排序）
"filesort" = "sorting not using index order"（无法走索引的有序排序）
```

```
MySQL 执行流程：

┌─────────────────────────────────────────────────────────┐
│  EXPLAIN 分析阶段                                       │
│                                                         │
│  能否用索引直接满足 ORDER BY？                           │
│      │                                                 │
│      ├─ YES ─→ 不需要 filesort，索引本身就是有序的      │
│      │                                                 │
│      └─ NO  ─→ 需要额外排序阶段 ─→ Using filesort       │
└─────────────────────────────────────────────────────────┘

  示例数据：
  ┌─────┬──────┐
  │ id  │ name │  假设查询: ORDER BY name
  ├─────┼──────┤
  │  3  │  Alice │  如果 name 上有索引 → 不需要 filesort
  │  1  │  Bob   │  如果无索引 或 使用了 WHERE 条件         │
  │  2  │  Carol │  → MySQL 会先全表扫描，                   │
  └─────┴──────┘    再执行内存/外部排序 → Using filesort
```

### 1.2 MySQL 内部排序算法演化

```
MySQL 排序算法演进时间线：

MySQL 5.6 及之前：     快速排序（Quick Sort）
        ↓
MySQL 8.0+：           TimSort（Python/Java 相同算法）
        ↓
MySQL 8.0.32+：       沈队优化 —— 提升大结果集排序性能
```

```
┌─────────────────────────────────────────────────────────┐
│  Quick Sort（MySQL 5.x 默认）                           │
│                                                         │
│  时间复杂度：O(n log n)（平均/最好）                     │
│  最坏情况：O(n²)（退化为桩化序列）                       │
│  空间复杂度：O(log n) 递归栈                              │
│  特点：非稳定排序                                       │
│                                                         │
│  两阶段策略：                                           │
│  Phase 1: 分批排序（每次处理 sort_buffer_size 大小）     │
│  Phase 2: 外部归并（多路归并写临时文件）                 │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  TimSort（MySQL 8.0+ 默认）                             │
│                                                         │
│  时间复杂度：O(n log n)（最坏）                         │
│  空间复杂度：O(n)                                        │
│  特点：稳定排序（相等元素保持原始顺序）                  │
│                                                         │
│  核心思想：                                             │
│  将数组分成多个 RUN（天然有序段，初始 size=1）          │
│  小 RUN 用 insertion sort 构建                          │
│  大 RUN 直接识别（升序/降序段 ≥ 32）                   │
│  用二进制归并栈（merge stack）合并相邻 RUN             │
│                                                         │
│  优势：数组局部有序时性能极佳                           │
│  为什么 MySQL 选它：减少最坏情况 O(n²) 概率             │
└─────────────────────────────────────────────────────────┘
```

### 1.3 何时触发"外部归并排序"（磁盘临时文件）

```
触发条件：排序数据量 > sort_buffer_size

外部排序执行流程（两阶段归并）：

Phase 1: 分批生成有序 RUN（内存排序）
┌──────────────────────────────────────┐
│  数据流                               │
│  [全表扫描行] ──→ sort_buffer ──→ 排序 RUN 1（块）──┐
│                      ↑  (sort_buffer_size)            │
│                                              ↓        │
│  当 sort_buffer 满了时，输出一个已排序的 RUN 块      │
│  写为临时文件，继续处理下一批                        │
└──────────────────────────────────────┘

Phase 2: 多路归并（外部归并）
┌──────────────────────────────────────┐
│  假设有 3 个 RUN 文件块               │
│                                         │
│  RUN_1: [3, 7, 12, 25, 31, ...]        │
│  RUN_2: [5, 9, 18, 22, 41, ...]        │
│  RUN_3: [1, 6, 14, 28, 33, ...]        │
│                 ↓                      │
│         优先队列（堆）多路归并          │
│                 ↓                      │
│  最终有序输出: [1, 3, 5, 6, 7, 9, ...]  │
└──────────────────────────────────────┘

关键公式：
  所需临时文件数量 N：
    N = ceil(排序数据总量 / sort_buffer_size)

  归并路数 M（受 max_length_for_sort_data 限制）：
    M = sort_buffer_size / 每行平均大小

  归并趟数 = ceil(log_M(N))
```

```
触发外部排序的阈值判断：

┌────────────────────────────────────────────────────────┐
│  max_length_for_sort_data (默认: 1024 字节)            │
│                                                        │
│  每行大小 ≤ max_length_for_sort_data？                 │
│     │                                                 │
│     ├─ YES：使用"原始行指针"模式（仅存指针+排序键）    │
│     │        节省 sort_buffer 空间                    │
│     │                                                 │
│     └─ NO：使用"完整行"模式（存完整列值）             │
│            消耗更多 sort_buffer，可能更早触发外部排序   │
└────────────────────────────────────────────────────────┘
```

### 1.4 `sort_buffer_size` vs `tmp_table_size`

```sql
-- 查看当前配置
SHOW VARIABLES LIKE '%sort%';
SHOW VARIABLES LIKE '%tmp_table%';

-- 查看排序相关状态
SHOW STATUS LIKE 'Sort_merge_passes';   -- 归并趟数（越大说明外部排序越多）
SHOW STATUS LIKE 'Sort_range';          -- 索引范围扫描触发的排序次数
SHOW STATUS LIKE 'Sort_rows';           -- 排序的行数
SHOW STATUS LIKE 'Sort_scan';           -- 全表扫描触发的排序次数
```

```sql
-- 核心配置参数对比
-- sort_buffer_size: 每个会话线程独立分配
-- tmp_table_size:  内存临时表大小上限
-- tmpdir:          外部排序临时文件存放目录

-- 排序过程中两个参数的协作关系：
SELECT 
    name, 
    age, 
    dept
FROM employee
WHERE dept = 'IT'
ORDER BY age;

-- MySQL 执行路径：
-- 1. WHERE 条件过滤 ──→ 内存临时表（tmp_table_size 限制）
--    └─ 如果中间结果 > tmp_table_size ──→ 转换为磁盘 MyISAM 临时表
-- 2. ORDER BY 排序 ──→ sort_buffer（sort_buffer_size 限制）
--    └─ 如果排序数据 > sort_buffer_size ──→ 外部归并排序
```

```
内存排序 vs 外部排序触发边界：

sort_buffer_size = 256KB（默认 256KB = 262144 字节）

情况1：排序数据量 ≤ 256KB
  ──────────────────────────────
  ✓ 全在 sort_buffer 内完成
  ✓ 仅一次排序（快速排序 / TimSort）
  ✓ 归并趟数 Sort_merge_passes = 0

情况2：排序数据量 > 256KB
  ──────────────────────────────
  ✗ 触发外部归并排序
  ✗ 需要读写 tmpdir 临时文件
  ✗ Sort_merge_passes > 0

tmp_table_size = 16MB（默认）

情况3：中间结果 ≤ 16MB
  ──────────────────────────────
  ✓ 使用内存临时表（HEAP 引擎）

情况4：中间结果 > 16MB
  ──────────────────────────────
  ✗ 转换为磁盘 MyISAM 临时表
  ✗ 性能显著下降（磁盘 IO）
```

### 1.5 生产环境调优建议

```sql
-- 推荐配置（根据数据量调整）
SET GLOBAL sort_buffer_size = 4 * 1024 * 1024;   -- 4MB
SET GLOBAL tmp_table_size = 64 * 1024 * 1024;    -- 64MB
SET GLOBAL max_length_for_sort_data = 2048;       -- 2KB

-- 在 ORDER BY/GROUP BY 中合理使用索引
-- 创建复合索引时，将排序列放在最后
CREATE INDEX idx_dept_age ON employee(dept, age);

-- 这样查询可以直接使用索引顺序，避免 filesort
EXPLAIN SELECT * FROM employee WHERE dept = 'IT' ORDER BY age;
-- Extra: Using index condition  ✓ 无 filesort
```

## 🔥 扩展追问 1：`Using filesort` 一定慢吗？什么情况下可接受？

> **A:** 不一定慢。关键看排序数据量：

| 场景 | 排序量 | 性能影响 |
|------|--------|----------|
| 排序 10-100 行 | 很小 | 几乎无感，内存排序很快 |
| 排序 1000 行 | 中等 | 可能需外部排序，延迟上升 |
| 排序 10 万+ 行 | 大量 | 明显慢，需优化索引 |
| 分页 `LIMIT 10` 大结果集 | 特殊情况 | 优先队列算法优化（MySQL 6.0+） |

```
MySQL 6.0+ 优化：分页 LIMIT 场景使用优先队列

原始行为：排序全部数据，再取前 10 条
优化行为：用堆（Priority Queue）维护 Top-10，无需全量排序

适用于：ORDER BY score DESC LIMIT 10（百万行取前 10）
```

## 🔥 扩展追问 2：什么是 `max_length_for_sort_data`？调大它有什么影响？

> **A:** 它控制 MySQL 选择哪种排序模式：

```sql
-- 模式1：全字段模式（rowid 排序）
-- 每行大小 > max_length_for_sort_data 时触发
-- sort_buffer 存：排序键 + 主键
-- 排序完成后，用主键回表取其他字段
-- 缺点：多一次回表 IO

-- 模式2：全量排序模式
-- 每行大小 ≤ max_length_for_sort_data 时触发
-- sort_buffer 存：所有 SELECT 的列
-- 排序完成直接返回，无需回表

-- 调大 max_length_for_sort_data 的影响：
--   PRO: 减少回表次数（减少 IO）
--   CON: 每行消耗 sort_buffer 更多
--        → 更早触发外部排序（sort_buffer 更容易满）
--   CON: 排序内存占用翻倍
```

## 🔥 扩展追问 3：MySQL 8.0 TimSort 的具体实现细节是什么？

> **A:** MySQL 8.0 的 TimSort 有以下关键实现：

```sql
-- MySQL 使用 TimSort 的场景：
-- 1. 排序数据量 ≤ sort_buffer_size（内存排序）
-- 2. 排序数据量 > sort_buffer_size（外部排序 RUN 内部）

-- TimSort 核心参数：
-- MIN_MERGE = 32（最小归并段大小）
-- RUN: 连续升序或严格降序的片段

-- 二进制归并栈（merge stack）：
-- 当 RUN 数量满足特定条件时自动触发归并
-- 避免最坏情况 O(n log²n)

-- 实战技巧：利用 TimSort 的"天然有序检测"
-- 如果你的数据本身有某种局部有序性，
-- TimSort 性能远优于 Quick Sort
-- 这是 MySQL 8.0 排序性能普遍提升的原因之一
```

## 🔥 扩展追问 4：如何判断生产环境的排序是否在用磁盘临时文件？

> **A:** 通过以下方法诊断：

```sql
-- Step 1: 查看归并趟数
SHOW STATUS LIKE 'Sort_merge_passes';
-- 如果 > 0，说明发生外部归并排序

-- Step 2: 查看排序相关所有状态
SHOW STATUS LIKE 'Sort%';

-- 解读：
-- Sort_merge_passes: 归并趟数（外部排序发生次数）
-- Sort_range:       索引范围扫描触发的排序
-- Sort_scan:        全表扫描触发的排序
-- Sort_rows:        已排序的总行数

-- Step 3: 分析 EXPLAIN 的扫描行数
EXPLAIN SELECT ... ORDER BY ...;
-- rows 列越大，外部排序可能性越高

-- Step 4: 查看 tmpdir 磁盘使用
-- Linux: watch -n 1 'ls -lh /tmp/mysql_sort*'
-- 如果临时文件很大，说明外部排序频繁

-- Step 5: 实时监控（performance_schema）
SELECT * FROM performance_schema.events_statements_summary_by_digest
WHERE DIGEST_TEXT LIKE '%ORDER BY%';
```

> **一句话总结**

`Using filesort` 意味着 MySQL 需要额外排序阶段（不依赖索引顺序），**不一定触发磁盘排序**——当数据能放入 `sort_buffer_size` 时走内存排序（MySQL 8.0+ 用 TimSort，MySQL 5.x 用快速排序），只有数据量超过阈值才会触发外部归并排序（写临时文件、多趟归并），此时 `Sort_merge_passes` 状态值 > 0，`sort_buffer_size` 和 `tmp_table_size` 分别控制排序阶段和中间结果表的内存上限，调大它们可减少磁盘排序，但过大会挤占连接内存。
