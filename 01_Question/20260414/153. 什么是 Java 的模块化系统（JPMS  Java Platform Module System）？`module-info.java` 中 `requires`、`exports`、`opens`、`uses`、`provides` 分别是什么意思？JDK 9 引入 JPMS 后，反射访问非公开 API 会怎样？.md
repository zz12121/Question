## 1. 基础流程

### 模块化系统的核心概念

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Java 模块化系统（JPMS）                            │
│                                                                     │
│  传统 JAR 的问题：                                                  │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  jar1.jar ──┐                                              │   │
│  │  jar2.jar ──┼──→ 类路径（ClassPath）──→ 全部公开，无边界    │   │
│  │  jar3.jar ──┘                                              │   │
│  │                                                             │   │
│  │  • 无法明确声明依赖关系                                     │   │
│  │  • 无法封装内部 API                                          │   │
│  │  • JAR Hell（版本冲突）                                      │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  模块化后：                                                        │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │   ┌──────────┐          ┌──────────┐                       │   │
│  │   │ module A │ requires ──→│ module B │                       │   │
│  │   │ exports  │          │ exports  │                       │   │
│  │   │   pkg1   │          │   pkg1   │                       │   │
│  │   └──────────┘          └──────────┘                       │   │
│  │        ↑                       ↑                             │   │
│  │        │                       │                             │   │
│  │   只看见 exports 的包      内部包完全隐藏                      │   │
│  │                                                             │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### module-info.java 的结构

```java
// module-info.java 示例
module com.example.myapp {
    
    // 依赖其他模块
    requires com.example.utils;        // 必需依赖
    requires static com.example.logging; // 可选依赖（编译时需要，运行时可能不存在）
    
    // 暴露哪些包给其他模块
    exports com.example.api;           // 导出包（完全公开）
    exports com.example.internal to com.example.web;  // 精确导出（只给指定模块）
    
    // 开放哪些包给反射访问（运行时）
    opens com.example.config;          // 开放包（运行时可反射）
    opens com.example.config to com.example.spring;  // 只给指定模块开放
    
    // 声明服务接口（SPI）
    uses com.example.spi.CacheService;  // 使用服务（消费方）
    
    // 提供服务实现
    provides com.example.spi.CacheService 
        with com.example.impl.RedisCache;  // 提供实现
}
```

### module-info.java 关键字速查

| 关键字 | 含义 | 作用对象 |
|--------|------|---------|
| `requires` | 声明依赖的模块 | 模块级别 |
| `exports` | 导出包（供其他模块使用） | 包级别 |
| `exports ... to` | 精确导出（限制可见性） | 包级别 |
| `opens` | 开放包给反射访问（运行时） | 包级别 |
| `opens ... to` | 精确开放（限制反射来源） | 包级别 |
| `uses` | 声明使用 SPI 服务接口 | 模块级别 |
| `provides ... with` | 提供 SPI 接口实现 | 模块级别 |
| `transitive` | 传递依赖（requires transitive） | 模块级别 |

## 2. 扩展追问

### ## 🔥 扩展追问 1：exports 和 opens 有什么区别？

> **A:** `exports` 是**编译时和运行时的公开声明**，其他模块在编译时就能使用；`opens` 只在**运行时允许反射访问**，编译时仍然看不到。

```
┌─────────────────────────────────────────────────────────────────────┐
│                      exports vs opens 的区别                         │
│                                                                     │
│  exports（编译时+运行时公开）：                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  module A {                                                  │   │
│  │      exports com.example.api;                               │   │
│  │  }                                                           │   │
│  │                           │                                  │   │
│  │  module B {                 │                                  │   │
│  │      requires A;            │                                  │   │
│  │                           ↓                                  │   │
│  │  // 编译时可见，代码直接引用                                  │   │
│  │  com.example.api.Class obj = new ...();                       │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  opens（仅运行时反射可见）：                                         │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  module A {                                                  │   │
│  │      opens com.example.config;                               │   │
│  │  }                                                           │   │
│  │                           │                                  │   │
│  │  module B {                 │                                  │   │
│  │      // 编译时不可见！无法直接引用                             │   │
│  │      // 只能通过反射访问                                      │   │
│  │      Class<?> clazz = Class.forName("com.example.config.X"); │   │
│  │      Field f = clazz.getDeclaredField("field");               │   │
│  │      f.setAccessible(true);  // 允许访问私有成员              │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

| 维度 | exports | opens |
|------|---------|-------|
| **编译时可见** | ✅ 是 | ❌ 否 |
| **运行时反射** | ❌ 不可反射 private | ✅ 可反射 private |
| **使用场景** | 公开 API | 框架配置注入（Spring/Hibernate） |
| **典型用户** | 其他业务模块 | 字节码框架、ORM、容器 |

### ## 🔥 扩展追问 2：JDK 9 后反射访问非公开 API 会怎样？

> **A:** JDK 9 后如果模块不允许访问，调用 `setAccessible(true)` 会抛出 `InaccessibleObjectException`。

```java
// JDK 8 及之前：反射几乎无限制
public class Jdk8Reflection {
    public static void main(String[] args) throws Exception {
        // 反射访问任意类的任意成员
        Field f = String.class.getDeclaredField("value");
        f.setAccessible(true);  // ✅ 成功
    }
}

// JDK 9+：受模块系统限制
public class Jdk9Reflection {
    public static void main(String[] args) throws Exception {
        Field f = String.class.getDeclaredField("value");
        f.setAccessible(true);  // ⚠️ 可能抛出 InaccessibleObjectException
        
        // 错误信息：
        // java.lang.reflect.InaccessibleObjectException: 
        // Unable to make field private final char[] accessible: 
        // module java.base does not "opens java.lang" to unnamed module
    }
}
```

```
┌─────────────────────────────────────────────────────────────────────┐
│                    JDK 9+ 反射限制机制                               │
│                                                                     │
│  JDK 内部模块（java.base 等）：                                      │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  java.base                                                │   │
│  │      exports java.lang;    ← 导出包，编译时可见            │   │
│  │      opens java.lang;       ← 不开放，无法反射 private      │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  问题：Spring/Hibernate 需要反射 private 字段来注入依赖              │
│                                                                     │
│  解决方案：                                                         │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  1. JVM 参数打开权限                                        │   │
│  │     --add-opens java.base/java.lang=ALL-UNNAMED           │   │
│  │                                                             │   │
│  │  2. 在 module-info.java 中声明                              │   │
│  │     opens com.example.config to ALL-UNNAMED;               │   │
│  │                                                             │   │
│  │  3. 使用 --add-opens 脚本启动                                │   │
│  │     java --add-opens java.base/java.lang=ALL-UNNAMED ...   │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### ## 🔥 扩展追问 3：uses 和 provides 是什么？和 SPI 有什么关系？

> **A:** `uses` 和 `provides` 是 JPMS 对 **SPI（Service Provider Interface）** 的原生支持，和 `ServiceLoader` 配合使用。

```java
// ========== 服务接口定义 ==========
// module com.example.spi
module com.example.spi {
    exports com.example.spi;  // 导出 SPI 接口
}

// SPI 接口
package com.example.spi;
public interface Cache {
    void put(String key, Object value);
    Object get(String key);
}

// ========== 服务提供方 ==========
// module com.example.redis
module com.example.redis {
    requires com.example.spi;
    exports com.example.impl;
    
    // 声明提供 Cache 的实现
    provides com.example.spi.Cache 
        with com.example.impl.RedisCache;
}

// 配置文件（可选，和 provides 二选一）
// META-INF/services/com.example.spi.Cache
// 内容：com.example.impl.RedisCache

// ========== 服务使用方 ==========
// module com.example.app
module com.example.app {
    requires com.example.spi;
    requires com.example.redis;
    
    // 声明使用 Cache 服务
    uses com.example.spi.Cache;
    
    // 代码中获取实现
    ServiceLoader<Cache> loader = ServiceLoader.load(Cache.class);
    Cache cache = loader.findFirst().orElseThrow();
}
```

> **uses vs requires：**
> - `requires` 是强依赖，必须有才能编译
> - `uses` 是声明性依赖，可以没有实现（运行时决定）

### ## 🔥 扩展追问 4：模块化系统有什么优缺点？

> **A:** JPMS 的核心优势是**明确依赖边界**和**封装内部 API**，但也带来了**迁移成本高**和**学习曲线陡峭**的问题。

| 优点 | 说明 |
|------|------|
| **显式依赖** | `requires` 明确声明依赖，JAR Hell 更容易排查 |
| **封装内部 API** | `exports` 控制哪些包对外可见 |
| **安全增强** | 运行时反射受限，减少内部 API 滥用 |
| **优化打包** | jlink 可以创建最小化 JRE |
| **原生支持 SPI** | `provides/uses` 替代部分 META-INF 配置 |

| 缺点 | 说明 |
|------|------|
| **迁移成本** | 大量第三方库不支持模块化 |
| **反射限制** | 框架需要额外配置才能工作 |
| **复杂性** | 概念多，学习曲线陡 |
| **不是真正的模块化** | 模块间仍有传递依赖问题 |

```bash
# 查看模块信息
java --list-modules

# 查看某个模块的依赖
java --describe-module java.sql

# 打包自定义 JRE（只包含需要的模块）
jlink --add-modules java.base,java.sql --output myjre
```

### ## 🔥 扩展追问 5：Spring/Hibernate 等框架是如何适配模块化的？

> **A:** 主流框架通过 **Automatic Module** 和 **JVM 参数** 两种方式适配模块化系统。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    框架适配模块化的方式                               │
│                                                                     │
│  方式1：Automatic Module（自动模块）                                 │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  • 不带 module-info.java 的 JAR                              │   │
│  │  • JAR 名称自动成为模块名                                     │   │
│  │    spring-core-5.3.jar → module name: spring.core           │   │
│  │  • 自动 exports 所有包                                        │   │
│  │  • 可以 requires 所有显式模块                                  │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  方式2：JVM 参数打开模块权限                                         │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  # Spring Boot 启动脚本                                       │   │
│  │  java \                                                    │   │
│  │      --add-opens java.base/java.lang=ALL-UNNAMED \          │   │
│  │      --add-opens java.base/java.util=ALL-UNNAMED \          │   │
│  │      --add-opens java.base/java.lang.reflect=ALL-UNNAMED \  │   │
│  │      -jar app.jar                                           │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  方式3：opens 声明（框架自己声明）                                    │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  module com.example.framework {                              │   │
│  │      opens com.example.internal to ALL-UNNAMED;              │   │
│  │  }                                                           │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

```bash
# Spring Boot 2.6+ 的兼容配置
# application.properties
spring.main.allow-circular-references=true
spring.main.allow-bean-definition-overriding=true

# 需要的 JVM 参数（典型配置）
# --add-opens java.base/java.lang=ALL-UNNAMED
# --add-opens java.base/java.lang.reflect=ALL-UNNAMED
# --add-opens java.base/java.util=ALL-UNNAMED
# --add-opens java.base/java.nio=ALL-UNNAMED
# --add-opens java.base/sun.nio.ch=ALL-UNNAMED
```

## 3. 一句话总结

> **JPMS 通过 module-info.java 实现显式依赖和封装，exports 导出包供其他模块编译时使用，opens 仅运行时允许反射访问，uses/provides 实现 SPI；JDK 9 后反射非导出包的 private 成员会抛出 InaccessibleObjectException，需要通过 JVM 参数 --add-opens 或 opens 声明来解除限制。**

> "我在项目里做过模块化改造——把一个大工程拆成多个模块，用 requires 和 exports 明确边界后，依赖关系清晰多了。还有个经验：Spring Boot 2.6 之后默认禁止了循环依赖，部分原因也是模块化后需要更明确的边界。还有个坑：如果用 jlink 打包自定义 JRE，一定要记得 `java --list-modules` 检查需要的模块有没有被包含进去。另外反射限制这个事，生产环境用 Spring/Hibernate 基本都会遇到，建议在启动脚本里加 `--add-opens`，不然框架初始化时会报错。"
