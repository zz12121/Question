## 1. 基础流程

### 1.1 CPU 缓存层级架构

```
CPU 核心架构（简化模型）:
┌──────────────────────────────────────────────────────────┐
│                        CPU Core 0                        │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐               │
│  │  Regs   │    │  L1 d   │    │  L1 i   │  ← ns 级     │
│  │  (B)    │←──→│(64B/核) │←──→│(64B/核) │               │
│  └─────────┘    └────┬────┘    └─────────┘               │
│                      │                                   │
│              ┌───────┴───────┐                          │
│              │  L2 (Cache)   │  ← ~ns 级               │
│              │  (256KB/核)   │                          │
│              └───────┬───────┘                          │
│                      │                                   │
│              ┌───────┴───────┐                          │
│              │  L3 (Shared)  │  ← ~10ns 级              │
│              │  (MB 级)      │                          │
│              └───────┬───────┘                          │
└──────────────────────┼──────────────────────────────────┘
                       │
              ┌────────┴────────┐
              │      RAM       │  ← ~100ns 级
              │   (GB 级)      │
              └────────────────┘

内存访问延迟对比（大约）:
  L1 Cache:    1 ~ 2 ns
  L2 Cache:    3 ~ 5 ns
  L3 Cache:    10 ~ 20 ns
  RAM:         80 ~ 120 ns
  SSD:         0.1 ~ 1 ms
```

### 1.2 什么是伪共享？

**核心概念**：两个线程分别修改**同一缓存行**上的不同变量，但因为缓存一致性协议（MESI），导致本不该互相影响的两个线程产生了**不必要的缓存同步开销**。

```
伪共享的本质：
┌────────────────────────────────────────────────────────┐
│            一个 Cache Line = 64 字节（通常）            │
│                                                        │
│  线程A (Core 0)              线程B (Core 1)             │
│  ┌─────────────────────┐     ┌─────────────────────────┐│
│  │ @Snippet-1 padding=0 │     │ @Snippet-2 padding=0   ││
│  │  long counter1;      │     │  long counter2;        ││
│  │  ← 线程A 只改这里     │     │  ← 线程B 只改这里      ││
│  │  (8字节)             │     │  (8字节)               ││
│  │  (但 counter1 和     │     │  (但和 counter1 在     ││
│  │   counter2 在同      │     │   同一个 cache line)   ││
│  │   一个缓存行里)      │     │                        ││
│  └─────────────────────┘     └─────────────────────────┘│
│           ↑                               ↑             │
│           └───────── MESI 协议 ──────────┘             │
│                                                        │
│  问题：线程A 修改 counter1 → 触发 Cache Line 失效       │
│        线程B 的 counter2 虽然没改，也要被迫刷新！       │
│        两个线程相互等待，效率反而不如单线程。           │
└────────────────────────────────────────────────────────┘
```

### 1.4 伪共享的性能影响实测

```java
import java.util.concurrent.*;

public class FalseSharingBenchmark {
    // 方案A：两个变量紧挨着（伪共享）
    static class NoPadding {
        volatile long counter1 = 0;
        volatile long counter2 = 0;
    }

    // 方案B：缓存行填充（手工 padding）
    static class WithPadding {
        volatile long counter1 = 0;
        long p1, p2, p3, p4, p5, p6, p7;  // 7 * 8 = 56 字节填充
        volatile long counter2 = 0;
    }

    public static void main(String[] args) throws Exception {
        int iterations = 100_000_000;

        // 测试无 padding 版本
        NoPadding noPadding = new NoPadding();
        long t1 = runTest(noPadding, iterations);
        System.out.println("无 padding (伪共享): " + t1 + " ms");

        // 测试有 padding 版本
        WithPadding withPadding = new WithPadding();
        long t2 = runTest(withPadding, iterations);
        System.out.println("有 padding (隔离): " + t2 + " ms");

        System.out.println("性能差距: " + (t1 * 1.0 / t2) + "x");
    }

    static long runTest(Object obj, int iterations) throws Exception {
        CountDownLatch latch = new CountDownLatch(2);

        Thread t1 = new Thread(() -> {
            for (long i = 0; i < iterations; i++) {
                if (obj instanceof NoPadding) ((NoPadding) obj).counter1++;
                else ((WithPadding) obj).counter1++;
            }
            latch.countDown();
        });
        Thread t2 = new Thread(() -> {
            for (long i = 0; i < iterations; i++) {
                if (obj instanceof NoPadding) ((NoPadding) obj).counter2++;
                else ((WithPadding) obj).counter2++;
            }
            latch.countDown();
        });

        long start = System.currentTimeMillis();
        t1.start(); t2.start();
        latch.await();
        return System.currentTimeMillis() - start;
    }
}

// 典型输出（4核机器）：
// 无 padding (伪共享): ~8500 ms
// 有 padding (隔离):   ~1200 ms
// 性能差距: ~7x
```

## 2. 扩展追问

### 🔥 扩展追问 1：`@Contended` 注解是如何解决伪共享的？

> **A:** `@Contended` 是 Java 8 引入的注解（JEP 142），用于告诉 JVM 在对象布局时，将被注解的字段**与其他字段隔离到不同的缓存行**。

```java
// JDK 源码中的应用示例（ConcurrentHashMap.Node）
@sun.misc.Contended
static final class CounterCell {
    volatile long value;
    CounterCell(long x) { value = x; }
}

// 使用时：多个线程的 CounterCell 分布在不同缓存行
// 避免 count 累加时互相触发伪共享
```

```java
// @Contended 的工作原理
public class CounterCellArray {
    // @Contended 确保这个数组的元素彼此间隔至少一个 cache line
    @sun.misc.Contended
    private volatile CounterCell[] cells;

    // 等价于编译器在内存中插入了 padding：
    // 每个 CounterCell 之间至少隔了 128 字节（默认间距）
    // 使得每个元素的修改不会触发相邻元素的缓存行失效
}
```

```
@Contended 内存布局（默认 128 字节间隔）:
┌──────────┐ ┌───────────────────┐ ┌──────────┐ ┌───────────────────┐
│ Cell[0]  │ │ Padding (120B)   │ │ Cell[1]  │ │ Padding (120B)   │  ...
│ (8B)     │ │ + CacheLine边界  │ │ (8B)     │ │ + CacheLine边界  │    ← 64B
└──────────┘ └───────────────────┘ └──────────┘ └───────────────────┘
           ↑                          ↑
           同一个缓存行（64B）           同一个缓存行（64B）
           → 彼此独立，不会互相影响
```

**注意**：`@Contended` 需要 **JVM 参数** `-XX:-RestrictContended` 开启才生效（默认开启但会忽略用户代码的 `@Contended`）：

```bash
# 开启用户代码 @Contended
-XX:-RestrictContended
# 或者（等效）
-XX:+UnlockDiagnosticVMOptions -XX:+AlwaysPreTouch
```

### 🔥 扩展追问 2：Disruptor 高性能队列中用到了哪些消除伪共享的技术？

> **A:** Disruptor（LMAX 出品）是伪共享消除的教科书级实现，用了三层防护：

```
Disruptor 的三层伪共享防护：
┌────────────────────────────────────────────────────────────┐
│ 第一层：RingBuffer 的 Sequence（指针）本身隔离               │
│                                                            │
│  public class Sequence extends RhsVariable {                │
│      // extends RhsVariable { long value; }                │
│      // 只有一个 value 字段 + @Contended                    │
│      // 确保每个 Sequence 独占一个缓存行                    │
│  }                                                         │
│                                                            │
│  Sequence[] sequences;  // 消费者共享这些 sequence          │
│  // 每个 sequence 之间自动隔离                              │
└────────────────────────────────────────────────────────────┘
                              ↓
┌────────────────────────────────────────────────────────────┐
│ 第二层：关键字段使用 @Contended 注解                        │
│                                                            │
│  // JUMP 缓存行填充（Disruptor 自己实现的简化版）           │
│  abstract class Sequence {                                 │
│      protected long value;                                │
│  }                                                         │
│                                                            │
│  // Jdk8 中 Sequence 继承自 PadCacheLine（填充父类）        │
│  abstract class PadCacheLine {                             │
│      long p1, p2, p3, p4, p5, p6, p7, p8;  // 8 * 8 = 64 │
│  }                                                         │
│                                                            │
│  abstract class Sequence extends PadCacheLine {            │
│      volatile long value;                                 │
│  }                                                         │
└────────────────────────────────────────────────────────────┘
                              ↓
┌────────────────────────────────────────────────────────────┐
│ 第三层：RingBuffer 的数组索引访问利用 JNI + CPU缓存预取     │
│                                                            │
│  // 环形数组的设计：相邻槽位之间隔开，避免顺序访问伪共享   │
│  // 但这个主要是为了优化缓存命中率                          │
└────────────────────────────────────────────────────────────┘
```

```java

// 注意：Disruptor 在 Java 8 之前使用纯手工 padding，Java 8+ 使用 @Contended

// 方式1：纯手工 padding（JDK 8 之前的做法）
public class Sequence extends Padding {
    private volatile long value = INITIAL_VALUE;
    // ...
}

// 父类负责填充缓存行
abstract class Padding {
    protected long p1, p2, p3, p4, p5, p6, p7;  // 前填充
    // 后填充在另一个子类中
}

// 方式2：Java 8+ @Contended
public class Sequence {
    @sun.misc.Contended
    private volatile long value;
}

// 使用对比
public class DisruptorDemo {
    public static void main(String[] args) {
        // 多个消费者各自持有一个 Sequence
        // 这些 Sequence 彼此隔离在不同的缓存行
        // 生产者 Publish 时写入自己的 Sequence
        // 消费者通过读取自己的 Sequence 判断可以消费到哪

        // 核心公式：sequence - array[sequence % array.length]
        // → 多个消费者序列互相独立，不存在伪共享
    }
}
```

```
Disruptor 的 RingBuffer + Sequence 架构：
┌────────────────────────────────────────────────────────────┐
│                    RingBuffer (数组)                       │
│  [0] [1] [2] [3] [4] [5] ... [n-1]                         │
│   ↑                                              ↑         │
│   └──────────────────── 环形数组 ─────────────────────┘   │
│                                                            │
│  生产者 Sequence:  12 ──────────────────────────────────   │
│  消费者1 Sequence:       8 ─── (独立缓存行)                │
│  消费者2 Sequence:           10 ── (独立缓存行)           │
│                                                            │
│  每个 Sequence 独占一个缓存行，彼此修改互不影响！          │
└────────────────────────────────────────────────────────────┘
```

### 🔥 扩展追问 3：为什么 Java 的数组和对象字段布局要避免伪共享，而不是让 CPU 解决？

> **A:** 这是一个很好的系统设计问题，核心在于 **MESI 协议的开销**：

```
MESI 协议中"Modified"状态的核心问题：
┌────────────────────────────────────────────────────────────┐
│  假设 Core 0 修改了 counter1（在同一缓存行的 counter2 未变）│
│                                                            │
│  1. Core 0 将 cache line 标记为 Modified                   │
│  2. Core 1 的同一 cache line 收到 Invalidate 信号           │
│  3. Core 1 必须将该行写回 RAM（或转发给 Core 0）            │
│  4. Core 0 后续如果 counter2 被修改... 重复上述流程          │
│                                                            │
│  问题：Cache Line 大小 = 64B，即使只改了 1 个字节          │
│        其他 63 字节的变化也要同步给所有核心                │
│                                                            │
│  解决思路：                                               │
│  → 操作系统：无能为力（一致性协议在内核实现）              │
│  → CPU 硬件：有"细粒度锁"但实现复杂                    │
│  → 软件层面：在字段间插入 padding，让每个变量独占缓存行   │
└────────────────────────────────────────────────────────────┘
```

**为什么 CPU 不用软件视角的细粒度锁？**
- MESI 是硬件级别的**缓存行粒度**一致性协议
- 如果要做字节级一致性，硬件复杂度爆炸（电路面积、功耗大幅增加）
- 所以业界选择了**软件侧填充**的方案

### 🔥 扩展追问 4：除了 `@Contended` 和手动 padding，还有哪些消除伪共享的方法？

> **A:** 有三种主要方法，从低到高：

| 方法 | 原理 | 适用场景 |
|---|---|---|
| **手动 padding** | 插入 dummy 字段 | JDK 8 之前、无 @Contended 支持时 |
| **`@Contended` 注解** | JVM 自动插入 padding | JDK 8+，标准库内部使用 |
| **Thread-Local + 批量归并** | 每个线程独立副本，最后合并 | 高并发计数器（LongAdder）|

```java
// LongAdder 的设计（伪共享消除的经典案例）
public class LongAdder extends Striped64 {
    // Striped64 内部维护了一个 Cell[]
    // 每个 Cell 独占一个缓存行（通过 @Contended）
    // 线程累加时 hash 到不同的 Cell
    // 最后 sum() 时把所有 Cell 加起来

    transient volatile Cell[] cells;

    // Cell 类（核心）
    @sun.misc.Contended
    static final class Cell {
        volatile long value;
        Cell(long x) { value = x; }
        // ...
    }
}

// 伪代码：线程如何累加
public void increment() {
    add(1);
}

final void add(long x) {
    Cell[] cs = cells;
    if (cs == null) {
        // 尝试累加 base
        boolean collided = false;
        if (casBase(...)) { /* success */ }
    }
    int h = getProbe();  // 线程的 hash 值
    int index = h & (cs.length - 1);
    Cell c = cs[index];
    c.value += x;  // 每个线程累加不同的 cell，无伪共享
}
```

## 3. 一句话总结

> **伪共享是 CPU 缓存一致性协议的副作用：两个毫不相干的变量碰巧落在同一缓存行，导致修改任意一个都会让另一个线程的缓存行失效。解决方案是填充缓存行（padding）或用 `@Contended` 注解隔离，Disruptor 正是靠这个在超高并发下跑出百万 TPS。**

## 4. 面试加分话术

> "我之前分析过生产环境的 `LongAdder` vs `AtomicLong` 性能问题。用 `AtomicLong` 做计数器时，两个线程在高并发下会因为伪共享导致缓存行抖动，`LongAdder` 把累加操作分散到不同缓存行的 Cell 里，彻底规避了这个问题。后来在压测时发现，8 核机器上 `LongAdder` 的 QPS 能到 1200 万，`AtomicLong` 只有 400 万，差了将近 3 倍。"

> "关于 Disruptor，我读源码时印象最深的是它手工 padding 的写法——JDK 8 之前没有 `@Contended`，它就通过继承 `Padding` 类在父类里塞 8 个 long（64 字节）做前后填充。这种设计理念就是**让数据布局对 CPU 友好**，而不是等出了问题再调优。"
