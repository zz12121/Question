## 1. 基础流程

### 1.1 `KEYS *` 的致命问题

```
KEYS 命令的底层实现：

┌─────────────────────────────────────────────────────────┐
│  Redis 单线程执行模型                                    │
│                                                         │
│  Redis 6.0 之前：完全单线程                            │
│  Redis 6.0+： I/O 线程并行读写，命令执行单线程         │
│                                                         │
│  命令执行是串行的！                                     │
│  所有客户端请求排队，单线程依次执行                     │
│  一个慢命令会阻塞所有其他命令                          │
└─────────────────────────────────────────────────────────┘

Redis 6.0+ 多线程架构（命令执行仍单线程）：
┌─────────────────────────────────────────────────────────┐
│  I/O Thread Pool    Main Thread (命令执行)            │
│  ───────────────    ────────────────────              │
│  并行读取          ←→   串行执行命令  ←→  并行写回    │
│  并行解析                              （所有请求排队） │
└─────────────────────────────────────────────────────────┘
```

```
KEYS * 的执行过程（源码级分析）：

// Redis 源码（server.c）
void KEYSCommand(client *c) {
    robj *pattern = c->argv[1];  // KEYS 命令只有 2 个参数
    
    // 全局数据库扫描
    // db->dict = 全局哈希表，存储所有 key
    // 遍历所有桶的所有节点
    // 时间复杂度 = O(N)，N = 数据库中所有 key 的总数
    
    dictEntry *de;
    long long numkeys = 0;
    addReplyMultiBulkLen(c, numkeys);
    
    // 遍历全局 dict
    de = dictNext(c->scan_ctx);
    while(de) {
        sds key = dictGetKey(de);
        // glob 模式匹配
        if (stringmatchlen(pattern, ..., key, ...) == 1) {
            addReplyBulk(c, key);  // 每匹配一个就发一次
            numkeys++;
        }
        de = dictNext(c->scan_ctx);
    }
}

问题：
1. O(N) 遍历全库所有 key（百万级 = 几秒）
2. 返回所有匹配 key（百万 key = 几百 MB 网络传输）
3. 主线程阻塞 = 所有客户端在这几秒内完全卡死
```

## 2. 替代方案

### 2.1 SCAN 核心原理与使用

```
SCAN 的设计哲学：游标式迭代，不阻塞

┌─────────────────────────────────────────────────────────┐
│  KEYS:   O(N) 一次性全量返回（阻塞）                   │
│  SCAN:   O(1) 每次只返回一个批次（不阻塞）             │
│                                                         │
│  SCAN 是一次完整的遍历，被分成多次返回                  │
│  每次返回一个小批次（COUNT 控制）                      │
└─────────────────────────────────────────────────────────┘

SCAN 命令格式：
SCAN cursor [MATCH pattern] [COUNT count] [TYPE type]

工作原理：
┌─────────────────────────────────────────────────────────┐
│  cursor = 0（第 1 次调用，返回 0 才算结束）            │
│                                                         │
│  第一次: SCAN 0 MATCH user:* COUNT 100                │
│  返回:                                                 │
│    1) "125"      ← 下次游标（不是 0，需继续）         │
│    2) ["user:1", "user:2", ..., "user:100"]           │
│                                                         │
│  第二次: SCAN 125 MATCH user:* COUNT 100              │
│  返回:                                                 │
│    1) "0"        ← 游标为 0，遍历完成                  │
│    2) ["user:101", ..., "user:200"]                   │
│                                                         │
│  重复直到游标为 0                                       │
└─────────────────────────────────────────────────────────┘
```

```python
# Python 实现 SCAN 遍历
import redis

r = redis.Redis(host='localhost', port=6379, decode_responses=True)

def scan_keys(pattern, count=1000):
    """安全遍历所有匹配 key"""
    cursor = 0
    total = 0
    while True:
        cursor, keys = r.scan(cursor, match=pattern, count=count)
        # 每次只处理一小批，不阻塞
        for key in keys:
            # 在此处理每个 key
            yield key
            total += 1
        
        if cursor == 0:
            break  # 遍历完成
    
    print(f"遍历完成，共 {total} 个 key")

# 使用示例：删除所有过期 key
for key in scan_keys("cache:*"):
    ttl = r.ttl(key)
    if ttl > 0 and ttl < 60:  # 剩余 TTL < 60 秒
        r.delete(key)
```

### 2.2 SCAN vs 其他方案的对比

| 方案 | 复杂度 | 阻塞 | 适用场景 | 缺点 |
|------|--------|------|----------|------|
| `KEYS *` | O(N) | 严重阻塞 | **绝对禁止生产** | 全库扫描，返回数据量大 |
| `SCAN` | O(N)/分片 | 几乎无阻塞 | 按模式遍历 | 需循环调用，不保证顺序 |
| `SSCAN` | O(N) | 无阻塞 | Set 遍历 | 只适用 Set 类型 |
| `HSCAN` | O(N) | 无阻塞 | Hash field 遍历 | 只适用 Hash |
| `ZSCAN` | O(N) | 无阻塞 | ZSet 遍历 | 只适用 ZSet |
| `TYPE` | O(1) | 无阻塞 | 判断单 key 类型 | 仅单 key |
| `INFO keys` | O(1) | 无阻塞 | 查看 key 总数 | 无 pattern |

### 2.4 SCAN 的陷阱与调优

```bash
# COUNT 参数不是返回数量上限，是每次遍历的桶数量
# Redis 内部遍历 bucket，不是遍历 key

SCAN 0 COUNT 1000
# 返回可能只有 100 个 key（bucket 中 key 稀疏）
# 也可能返回 2000 个 key（bucket 中 key 密集）

# 正确理解：
# COUNT 控制的是"每次迭代检查的桶数"
# 不是"每次返回的 key 数"

# 实际调优：
# 大数据量：用 COUNT 1000（减少迭代次数）
# 小数据量：用 COUNT 100（减少单次开销）

# TYPE 过滤（Redis 6.2+）
SCAN 0 MATCH user:* COUNT 100 TYPE string
# 遍历时只返回 string 类型的 key（过滤在服务端）
# 减少无效 key 的传输
```

## 🔥 扩展追问 1：SCAN 在 Redis Cluster 中如何工作？

> **A:** Cluster 中 SCAN 需要对每个节点分别执行：

```python
# Cluster 中 SCAN 的正确做法
def cluster_scan(cluster_client, pattern, count=1000):
    """遍历 Redis Cluster 所有节点"""
    all_keys = []
    
    # 获取所有节点
    for node in cluster_client.get_nodes():
        # 对每个主节点执行 SCAN
        if node.is_master():
            cursor = 0
            while True:
                cursor, keys = node.scan(
                    cursor=cursor,
                    match=pattern,
                    count=count
                )
                all_keys.extend(keys)
                
                if cursor == 0:
                    break  # 该节点扫描完成
    
    return all_keys

# 注意：
# 1. 只需要扫描主节点（从节点自动同步）
# 2. 每个节点独立 cursor，不共享
# 3. Cluster 不保证 key 的全局顺序
# 4. 复杂度和单机相同：O(各节点 key 数之和)
```

## 🔥 扩展追问 2：LRANGE 0 -1 为什么也很危险？

> **A:** LRANGE 的复杂度与 SCAN 相同，但更容易被忽略：

```bash
# LRANGE key 0 -1 的实现：
# 返回列表从 0 到最后一个元素的所有成员
# 复杂度 = O(N)，N = 列表长度

# 问题场景：
# 微博 Timeline：LRANGE timeline:user:100 0 -1
# 如果用户关注 10000 人，Timeline = 10000 条
# 返回 10000 条记录 = 阻塞主线程

# 对比 SCAN 和 LRANGE 的危险程度：
┌─────────────────────────────────────────────────────────┐
│  命令            │ 复杂度 │ 触发条件                    │
│  ────────────────┼────────┼────────────────────────────│
│  KEYS *          │ O(N)   │ key 总数                    │

│  HGETALL        │ O(N)   │ Hash field 总数              │

│  ZRANGE 0 -1   │ O(N)   │ ZSet 成员数                  │
└─────────────────────────────────────────────────────────┘
```

```python
# Python 实现危险命令监控
import subprocess
import re

def monitor_dangerous_commands():
    """监控 Redis 危险命令"""
    process = subprocess.Popen(
        ['redis-cli', 'MONITOR'],
        stdout=subprocess.PIPE,
        stderr=subprocess.PIPE
    )
    
    dangerous_pattern = re.compile(r'"(KEYS|SCAN|FLUSHDB|FLUSHALL|MGET|HGETALL|SMEMBERS)"')
    
    try:
        for line in iter(process.stdout.readline, b''):
            if dangerous_pattern.search(line.decode()):
                print(f"[警告] 危险命令: {line.decode().strip()}")
    except KeyboardInterrupt:
        process.terminate()
```

## 🔥 扩展追问 5：Redis 7.0 的 ACL + 危险命令阻断实战配置

> **A:** 完整配置示例：

```bash

redis-cli -u redis://admin:password@localhost:6379 AUTH admin
```

> **一句话总结**

`KEYS *` 是 O(N) 全库扫描，遍历所有桶返回所有 key，在百万级数据下会阻塞主线程数秒导致整个 Redis 不可用；生产环境必须用 `SCAN` 游标迭代（每次 O(1) 返回一批，永不阻塞），或重构业务逻辑用 Set/ZSet 做 key 索引；Redis 7.0 的 `FUNCTION LIST`/`COMMAND DOCS` 虽然也是 O(N)，但 N 是固定小值（函数数量/命令数量），风险远低于 `KEYS *` 的 O(N·全库key数)。

> **面试加分话术**

> "我在上一家公司真踩过 KEYS 的坑——半夜有个需求要清理 `order:timeout:*` 的 key，手动执行了 `KEYS order:timeout:*`，当时数据库有 300 万 key，结果整个 Redis 卡了 8 秒，线上大量 `Connection timeout`，SRE 直接发 P0 告警。事后分析：生产环境的 Redis Cluster 有 6 个节点，每台 50 万 key，KEYS 命令只扫描了当前节点的 50 万 key 还好，但凌晨有运维脚本在另一个节点跑了 KEYS *，直接导致该节点 8 秒不可用，其他节点的请求全部超时。后来我们做了三件事：1）所有代码强制用 SCAN；2）Redis ACL 禁止 KEYS 命令；3）引入了 Redisson 的 RKeys API 内部强制走 SCAN，从源头规避。另外 SCAN 在 Redis Cluster 中还有个坑——必须遍历所有主节点，每个节点的 cursor 独立，第一次做集群 key 清理时漏掉了两个节点，后来写了单元测试专门校验 scan 的完整性。"
