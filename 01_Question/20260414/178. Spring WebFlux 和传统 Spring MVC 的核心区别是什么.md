# Q178: Spring WebFlux vs Spring MVC 深度对比

## 1. 基础流程

### 1.1 架构对比：同步阻塞 vs 异步非阻塞

```
┌─────────────────────────────────────────────────────────┐
│  Spring MVC（传统模式）                                  │
│                                                         │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐         │
│  │  Thread  │   │  Servlet │   │  Handler │         │
│  │  Pool    │──→│  Thread  │──→│  (同步)   │──→ DB │
│  │ (200)    │   │  (1:1)   │   │  BLOCKED  │        │
│  └──────────┘   └──────────┘   └──────────┘         │
│                                                         │
│  每个请求占用一个线程                                   │
│  线程等待 I/O（DB/远程调用）时阻塞                     │
│  线程池耗尽 → 请求等待 → 接口超时                      │
│                                                         │
│  同步模型：请求进来 → 处理完 → 返回 → 线程释放         │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  Spring WebFlux（响应式模式）                           │
│                                                         │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐         │
│  │  Event   │   │  Netty   │   │  Handler │         │
│  │  Loop    │──→│  NIO     │──→│  (异步)   │──→ DB │
│  │  Group   │   │  (少量)  │   │  ON-DEMAND│        │
│  └──────────┘   └──────────┘   └──────────┘         │
│                                                         │
│  少量线程处理大量并发连接                               │
│  I/O 操作注册回调，数据就绪时触发                       │
│  线程不阻塞，数据就绪才处理                             │
│                                                         │
│  异步模型：请求进来 → 注册回调 → 立即释放 → 数据就绪时处理│
└─────────────────────────────────────────────────────────┘
```

### 1.2 Spring MVC vs WebFlux 底层线程模型

```java
// Spring MVC 线程模型（Tomcat 默认 200 线程）
// 假设并发 1000 请求：
// Thread-1: 处理请求1（等待DB 10ms，线程阻塞10ms）
// Thread-2: 处理请求2（等待DB 10ms，线程阻塞10ms）
// ...
// Thread-200: 处理请求200（全部阻塞中）
// 请求 201-1000: 等待线程池（排队/拒绝）

// Spring WebFlux 线程模型（Netty 少量线程）
// EventLoopGroup: 2-4 个线程处理所有连接
// 假设并发 1000 请求：
// EventLoop-1: 处理请求1（注册DB查询回调，立即处理请求2）
// EventLoop-2: 处理请求2（注册DB查询回调，立即处理请求3）
// ...
// 数据库查询完成 → 回调触发 → 继续处理

// 关键指标对比：
// Spring MVC:  200 线程 × 10000 连接 = 50 连接/线程
// Spring WebFlux: 4 线程 × 10000 连接 = 2500 连接/线程
```

## 2. Mono / Flux / Publisher 区别

### 2.1 Reactive Streams 规范核心接口

```java
// Reactive Streams 规范（JVM 标准，Java 9+ 内置）
// 包：org.reactivestreams

public interface Publisher<T> {
    void subscribe(Subscriber<? super T> s);  // 发布者，发布 T 类型数据
}

public interface Subscriber<T> {
    void onSubscribe(Subscription s);   // 订阅时触发
    void onNext(T t);                   // 收到数据
    void onError(Throwable t);          // 出错
    void onComplete();                  // 完成
}

public interface Subscription {
    void request(long n);   // 请求多少个数据
    void cancel();          // 取消订阅
}

public interface Processor<T, R> extends Subscriber<T>, Publisher<R> {}
// Processor = Subscriber + Publisher（处理+转发）
```

```
Publisher 契约（背压机制）：

Subscriber ──→ request(3) ──→ Publisher
    │                            │
    │←─── onSubscribe ───────────┘
    │
    │←─── onNext(1) ─────────────┐
    │←─── onNext(2) ─────────────┼── 最多 3 个（按需发送）
    │←─── onNext(3) ─────────────┘
    │
    │←─── onComplete ──────────────── 正常结束
    或
    │←─── onError ──────────────────── 出错结束
```

### 2.2 Mono vs Flux vs Publisher 关系

```
┌─────────────────────────────────────────────────────────┐
│  Reactive Streams 类型体系                              │
│                                                         │
│           ┌─────────────────────┐                      │
│           │   Publisher<T>      │  ← 标准接口，1:N 发布者│
│           │  ( Reactive Streams )│                      │
│           └──────────┬──────────┘                      │
│                      │                                  │
│         ┌───────────┴───────────┐                     │
│         │                       │                      │
│  ┌──────┴──────┐        ┌───────┴───────┐            │
│  │    Mono<T>   │        │   Flux<T>     │            │
│  │   (0 或 1)   │        │   (0 到 N)    │            │
│  └─────────────┘        └───────────────┘            │
│                                                         │
│  Mono: 最多发出 1 个元素，然后完成（或 1 个错误）     │
│  Flux: 发出 0-N 个元素，然后完成（或 1 个错误）       │
│                                                         │
│  都是 Publisher 的具体实现                              │
│  Publisher 本身是接口，直接实现无意义                   │
└─────────────────────────────────────────────────────────┘

常见误区澄清：
- Publisher 是规范接口，不能直接实例化
- Mono 和 Flux 是 Project Reactor 的实现类
- Java 标准库只定义了接口，具体实现在 RxJava/Reactor 中
```

### 2.3 Mono 和 Flux 常用操作符对比

```java
// Mono 常用操作（0 或 1 个元素）

Mono<String> mono = Mono.just("hello");           // 创建
mono.map(String::toUpperCase);                      // 转换
mono.flatMap(this::fetchData);                      // 异步链式调用
mono.filter(s -> s.length() > 3);                   // 过滤
mono.defaultIfEmpty("default");                     // 空时默认值
mono.switchIfEmpty(Mono.error(...));                // 空时切换错误
mono.delayElement(Duration.ofSeconds(1));           // 延迟
mono.timeout(Duration.ofSeconds(5));                // 超时
mono.retry(3);                                     // 重试 3 次
mono.onErrorReturn("fallback");                     // 错误时返回
mono.doOnSuccess(System.out::println);              // 副作用回调

// Flux 常用操作（0 到 N 个元素）

Flux<String> flux = Flux.just("a", "b", "c");      // 创建
flux.map(String::toUpperCase);                       // 转换
flux.flatMap(this::fetchBatch);                     // 异步批处理
flux.filter(s -> s.startsWith("a"));                // 过滤
flux.take(5);                                       // 取前 5 个
flux.skip(10);                                      // 跳过前 10 个
flux.buffer(100);                                   // 按 100 个分批
flux.window(5);                                     // 按 5 个分窗
flux.collectList();                                 // 收集为 List（变成 Mono）
flux.collectMap();                                  // 收集为 Map
flux.reduce((a, b) -> a + b);                      // 聚合
flux.mergeWith(otherFlux);                          // 合并
flux.zipWith(otherFlux);                            // 压缩合并
flux.distinct();                                    // 去重
flux.sort();                                        // 排序
flux.doOnNext(System.out::println);                 // 副作用
flux.doOnError(Throwable::printStackTrace);         // 错误回调

// 冷热信号
Flux<Long> cold = Flux.interval(Duration.ofSeconds(1));  // 冷信号：每次订阅重新开始
Flux<Long> hot = hotPublisher.cache();                    // 热信号：订阅时从当前开始
```

### 2.4 Mono / Flux 订阅时机

```java
// 核心原则：没有 subscribe()，数据流永远不会执行

// ❌ 不订阅 = 不执行
Mono.just("data")
    .map(String::toUpperCase);  // 只是描述处理逻辑，不执行

// ✅ 订阅才执行
Mono.just("data")
    .map(String::toUpperCase)
    .subscribe(System.out::println);  // 输出 "DATA"

// Spring WebFlux 内部自动订阅：
// @GetMapping 返回 Mono/Flux → Spring 自动订阅

// 四种订阅方式：
mono.subscribe();                                   // 1. 只关心完成/错误
mono.subscribe(consumer);                          // 2. 关心数据和完成
mono.subscribe(consumer, errorHandler);            // 3. 关心数据、错误
mono.subscribe(consumer, errorHandler, completeHandler); // 4. 全回调

// 实际使用建议：
mono
    .map(...)
    .flatMap(...)
    .doOnError(...)          // 日志记录
    .onErrorReturn(...)      // 错误时返回默认值
    .subscribe();
```

```java
// 场景1: Fan-out 并行调用（WebFlux 优势明显）
public Mono<UserDetailVO> getUserDetail(Long userId) {
    // Spring MVC: 串行调用，总耗时 = t1 + t2 + t3 + t4
    // WebFlux: 并行调用，总耗时 = max(t1, t2, t3, t4)
    
    Mono<User> userMono = userClient.getUser(userId);
    Mono<List<Order>> ordersMono = orderClient.getOrders(userId);
    Mono<List<Comment>> commentsMono = commentClient.getComments(userId);
    Mono<UserStats> statsMono = statsClient.getStats(userId);
    
    return Mono.zip(userMono, ordersMono, commentsMono, statsMono)
        .map(tuple -> {
            User u = tuple.getT1();
            List<Order> o = tuple.getT2();
            List<Comment> c = tuple.getT3();
            UserStats s = tuple.getT4();
            return new UserDetailVO(u, o, c, s);
        });
}

// 场景2: SSE 流式推送（WebFlux 原生支持）
@GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<WeatherData> streamWeather(@RequestParam String city) {
    return Flux.interval(Duration.ofSeconds(1))
        .map(tick -> fetchWeather(city))  // 每秒推送一次
        .take(60);                        // 推送 60 次
}

// 场景3: WebSocket 实时通信
@Configuration
@EnableWebSocket
public class WebSocketConfig {
    @Bean
    public WebSocketHandler webSocketHandler() {
        return (session -> {
            Flux<String> flux = session.receive()
                .map(WebSocketMessage::getPayloadAsText)
                .map(msg -> "Echo: " + msg);
            return session.send(flux.map(session::textMessage));
        });
    }
}
```

```java
// ❌ 错误示例：在 WebFlux 中使用阻塞代码
@Service
public class BadService {
    
    @Autowired
    private UserMapper userMapper;  // MyBatis，同步阻塞！
    
    public Mono<User> getUser(Long id) {
        // 这里阻塞了 EventLoop 线程！
        User user = userMapper.selectById(id);  // JDBC 阻塞
        return Mono.just(user);
    }
    
    // 正确做法：用阻塞数据库的适配器包装
    public Mono<User> getUserRight(Long id) {
        return Mono.fromCallable(() -> userMapper.selectById(id))
            .subscribeOn(Schedulers.boundedElastic())  // 移到专用线程池
            .flatMap(this::enrichUser);
    }
}

// ❌ 错误示例：CPU 密集型阻塞 EventLoop
@GetMapping("/compute")
public Mono<BigDecimal> compute(@RequestParam int n) {
    // 复杂计算，阻塞 EventLoop！
    BigDecimal result = fibonacci(n);  // CPU 密集，阻塞 EventLoop
    return Mono.just(result);
}

// ✅ 正确做法
@GetMapping("/compute")
public Mono<BigDecimal> compute(@RequestParam int n) {
    return Mono.fromCallable(() -> fibonacci(n))
        .subscribeOn(Schedulers.boundedElastic())  // 线程池执行
        .subscribeOn(Schedulers.parallel());      // 或并行 CPU 池
}
```

## 🔥 扩展追问 1：R2DBC 是什么？和 JDBC 有什么区别？

> **A:** R2DBC 是 Reactive Relational Database Connectivity：

```java
// JDBC（同步阻塞）
Connection conn = dataSource.getConnection();
PreparedStatement ps = conn.prepareStatement("SELECT * FROM user WHERE id = ?");
ps.setInt(1, id);
ResultSet rs = ps.executeQuery();  // 阻塞等待
while (rs.next()) {
    User u = new User(rs.getLong("id"), rs.getString("name"));
}

// R2DBC（异步非阻塞）
return connectionFactory.create()
    .flatMap(conn -> conn.createStatement("SELECT * FROM user WHERE id = $1")
        .bind("$1", id)
        .execute())
    .flatMap(result -> result.map((row, metadata) -> 
        new User(row.get("id", Long.class), row.get("name", String.class))
    ));

// 支持 R2DBC 的数据库：
// - PostgreSQL: r2dbc-postgresql
// - MySQL: r2dbc-mysql（异步驱动）
// - SQL Server: r2dbc-mssql
// - H2: r2dbc-h2

// Spring Data R2DBC 示例
public interface UserRepository extends ReactiveCrudRepository<User, Long> {
    Mono<User> findByName(String name);
    Flux<User> findByAgeGreaterThan(int age);
}
```

## 🔥 扩展追问 2：WebFlux 如何处理数据库事务？

> **A:** WebFlux 的事务处理比 MVC 复杂：

```java
// Spring WebFlux 事务处理

// 方式1: TransactionalOperator（响应式事务）
@Service
public class ReactiveUserService {
    
    @Autowired
    private R2dbcEntityTemplate template;
    
    public Mono<User> createUser(User user) {
        TransactionalOperator operator = TransactionalOperator.create(template.getDatabaseClient());
        
        return template.insert(User.class)
            .using(user)
            .as(operator.transactional())  // 注册事务
            .then(template.execute())
            .thenReturn(user);
    }
}

// 方式2: @Transactional + AspectJ（编译时织入）
@Service
@Transactional  // 需要配合 spring-tx
public class ReactiveTransactionService {
    
    @Autowired
    private UserRepository userRepository;
    
    public Mono<User> createWithRelated(User user) {
        return userRepository.save(user)
            .flatMap(u -> createRelatedEntity(u.getId()))
            .as(transactionalOperator::transactional);
    }
}

// ⚠️ 注意：
// - WebFlux 事务不支持声明式 @Transactional 直接生效
// - 需要显式使用 TransactionalOperator
// - 或者使用 MongoDB Reactive 的事务（MongoDB 4.0+ 支持）
// - R2DBC 事务需要数据库本身支持（如 PostgreSQL 的 MVCC）
```

## 🔥 扩展追问 3：WebFlux 中如何正确处理错误和超时？

> **A:** 错误处理是 WebFlux 最复杂的部分之一：

```java
// WebFlux 错误处理策略

// 1. 全局异常处理器（类似 @ControllerAdvice）
@Configuration
@Order(-2)
public class WebExceptionHandler implements ErrorWebExceptionHandler {
    
    @Override
    public Mono<Void> handle(ServerWebExchange exchange, Throwable ex) {
        ServerHttpResponse response = exchange.getResponse();
        response.setStatusCode(HttpStatus.INTERNAL_SERVER_ERROR);
        
        String body = switch (ex) {
            case UserNotFoundException e -> "{\"error\":\"User not found\"}";
            case BusinessException e -> "{\"error\":\"Business error\"}";
            default -> "{\"error\":\"Internal error\"}";
        };
        
        return response.writeWith(
            Mono.just(response.bufferFactory().wrap(body.getBytes()))
        );
    }
}

// 2. 短路兜底（onErrorReturn）
public Mono<User> getUser(Long id) {
    return userRepository.findById(id)
        .onErrorReturn(User.DEFAULT);  // 出错返回默认值
}

// 3. 切换到另一个 Mono（onErrorResume）
public Mono<User> getUser(Long id) {
    return userRepository.findById(id)
        .onErrorResume(ex -> fallbackRepository.findById(id));  // 降级
}

// 4. 忽略错误继续（onErrorResume with empty）
public Mono<User> getUser(Long id) {
    return userRepository.findById(id)
        .filter(user -> user.isActive())
        .switchIfEmpty(Mono.error(new UserInactiveException()));
}

// 5. 超时处理
public Mono<User> getUser(Long id) {
    return userRepository.findById(id)
        .timeout(Duration.ofSeconds(3))                  // 3 秒超时
        .onErrorReturn(TimeoutException.class, User.DEFAULT)  // 超时兜底
        .doOnError(TimeoutException.class, 
            e -> metricsService.recordTimeout("getUser"));
}

// 6. 重试机制
public Mono<User> getUser(Long id) {
    return userRepository.findById(id)
        .retryWhen(Retry.backoff(3, Duration.ofMillis(100))  // 指数退避
            .filter(ex -> ex instanceof TransientDataAccessException)  // 只重试瞬时错误
            .doBeforeRetry(signal -> log.warn("Retrying...", signal.failure())));
}
```

> **一句话总结**

Spring WebFlux 基于 **Reactor**（Mono 0-1 元素 / Flux 0-N 元素）实现异步非阻塞 IO，通过 Netty EventLoop 少量线程处理大量并发，**必须搭配非阻塞数据源**（R2DBC、MongoDB Reactive、Redis Reactive）才能发挥优势，**MyBatis/JDBC 同步数据库在其中反而成为瓶颈**；适合高并发 I/O 密集型（>5000 并发、外部 API 调用、流式推送），不适合 CPU 密集型、低并发简单 CRUD、团队不熟悉响应式编程的场景。

> **面试加分话术**

> "我在上一个项目里同时用过 Spring MVC 和 WebFlux，可以说说实际踩坑经验：当时我们把一个高并发查询服务从 Spring MVC 迁移到 WebFlux，用 JMeter 压测 5000 并发，MVC 线程池 200 线程直接打满，延迟从 20ms 飙升到 500ms，QPS 断崖下跌；但切到 WebFlux 后延迟稳定在 25ms，QPS 从 8000 升到 25000，效果非常明显——前提是我们的数据层用了 R2DBC + PostgreSQL异步驱动，不是 MyBatis。后来新来的同事不懂响应式，在 WebFlux 服务里偷偷用了 `userMapper.selectById()`（MyBatis），导致那个接口的 EventLoop 线程被阻塞 100ms，结果整个服务的并发能力从 25000 跌回 12000，排查了 2 天才发现是 MyBatis 同步驱动的问题。所以现在我们团队的规范是：Spring MVC 配 MyBatis，WebFlux 配 R2DBC + Spring Data R2DBC，泾渭分明，不允许混用。"
