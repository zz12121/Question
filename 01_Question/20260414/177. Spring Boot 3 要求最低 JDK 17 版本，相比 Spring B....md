## 1. 基础流程

### 1.1 Spring Boot 3 vs 2.x 重大变化总览

```
┌─────────────────────────────────────────────────────────┐
│  Spring Boot 版本演进时间线                              │
│                                                         │
│  Spring Boot 2.0    →  2018.01  (JDK 8 最低要求)        │
│  Spring Boot 2.7    →  2022.05  (JDK 8/17)            │
│  Spring Boot 3.0    →  2022.11  (JDK 17 最低要求)     │
│  Spring Boot 3.2    →  2023.11  (JDK 17/21)          │
│                                                         │
│  关键分界线：2022.11 Spring Boot 3.0 发布               │
└─────────────────────────────────────────────────────────┘
```

```
Spring Boot 3 重大变化矩阵：

┌────────────────────────────────────────────────────────┐
│  分类         │ 变化项              │ 影响范围          │
│  ────────────┼────────────────────┼───────────────────│
│  JDK 版本     │ JDK 8 → JDK 17    │ ⚠️ 破坏性升级    │
│  命名空间     │ javax → jakarta   │ ⚠️ 破坏性迁移    │
│  底层框架    │ Spring 6          │ ✅ 内部改进       │
│  AutoConfig  │ 条件注解新增      │ ✅ 向后兼容       │
│  依赖版本    │ 全面升级          │ ⚠️ 需要验证      │
│  GraalVM     │ 一等公民支持      │ ✅ 新能力         │
└────────────────────────────────────────────────────────┘

JDK 17 带来的新能力（Spring Boot 3 可直接使用）：

├─ Records (JDK 16 正式)       → 直接作为 DTO
├─ Sealed Classes (JDK 17)      → 接口更精确的访问控制
├─ Pattern Matching (JDK 16+)   → switch 表达式增强
├─ Text Blocks (JDK 15+)       → SQL/JSON 多行字符串
├─ Switch Expressions (JDK 14+) → 更简洁的 switch
└─ Foreign Function API (JDK 17) → 调用 native 代码
```

### 1.2 `javax.*` → `jakarta.*` 迁移深度解析

```
迁移背景：

┌─────────────────────────────────────────────────────────┐
│  Jakarta EE 9 命名空间变更                              │
│                                                         │
│  Java EE 8:           javax.servlet.*                  │
│  Jakarta EE 9:        jakarta.servlet.*               │
│                                                         │
│  为什么改？                                             │
│  - javax.* 包名归 Oracle 所有                          │
│  - Eclipse Foundation 接手 Jakarta EE 后改名           │
│  - 避免与 Oracle 商标冲突                              │
│                                                         │
│  影响的生态：                                           │
│  - Servlet API（Jakarta Servlet）                     │
│  - JPA（Jakarta Persistence）                          │
│  - JAXB（Jakarta XML Binding）                        │
│  - JSON-P / JSON-B（Jakarta JSON Processing）        │
│  - Bean Validation（Jakarta Validation）              │
│  - Dependency Injection（Jakarta Injection）           │
│  - 等等...                                            │
└─────────────────────────────────────────────────────────┘
```

```
<dependency>
    <groupId>jakarta.persistence</groupId>
```

### 2.2 新增 @Conditional 系列注解

```

│                                                         │
│  @Configuration
│  @ConditionalOnNotWarDeployment                        │

│      // 只有 JAR 部署才加载                            │
│  }                                                      │

┌─────────────────────────────────────────────────────────┐
│  3. @ConditionalOnWebApplication (已存在，增强)        │

┌─────────────────────────────────────────────────────────┐

│  @Configuration

│  @ConditionalOnMissingClass("org.postgresql.Driver")   │
│  public class R2dbcConfig {                            │
│      // 通过 class name 字符串检测，不需 import       │
```

### 3.2 Spring Security 6.x 变化

```java
// Spring Security 6.x 重大变化

// 1. WebSecurityConfigurerAdapter 已废弃
// Spring Boot 3 中完全移除

// Spring Boot 2.x（废弃写法）：
@Configuration
public class SecurityConfig extends WebSecurityConfigurerAdapter {
    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http.authorizeRequests().anyRequest().authenticated();
    }
}

// Spring Boot 3.x（推荐写法）：
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http.authorizeHttpRequests(auth -> auth
            .anyRequest().authenticated()
        );
        return http.build();
    }
}

// 2. CSRF 默认开启（Spring Security 6.0+）
// Spring Security 5.x 默认关闭，Spring Security 6.x 默认开启

// 3. Session Management 变化
http.sessionManagement(session -> session
    .sessionCreationPolicy(SessionCreationPolicy.STATELESS)
);

// 4. Lambda DSL（必须使用）
// 不再支持 .antMatchers()，用 .requestMatchers() 或 Lambda DSL
```

## 🔥 扩展追问 2：`@ConfigurationProperties` 在 Spring Boot 3 有变化吗？

> **A:** 有增强，但基本用法不变：

```java
// Spring Boot 3 对 @ConfigurationProperties 的改进：

// 1. 类型安全的配置属性（已存在，增强）
// 完全支持 Java 17 Records
@ConfigurationProperties(prefix = "app.user")
public record UserProperties(
    String name,
    int age,
    List<String> roles
) {}

// Spring Boot 3 之前需要 class
// Spring Boot 3 推荐用 record（不可变，简洁）

// 2. @ConfigurationPropertiesScan（3.0 新增）
@SpringBootApplication
@ConfigurationPropertiesScan  // 自动扫描所有 @ConfigurationProperties
public class MyApplication {}

// 3. 嵌套配置更简洁
@ConfigurationProperties(prefix = "app")
public record AppProperties(
    Database database,
    Cache cache
) {
    public record Database(String url, String username) {}
    public record Cache(int ttl, boolean enabled) {}
}

// application.yml:
// app:
//   database:
//     url: jdbc:mysql://localhost:3306/test
//     username: root
//   cache:
//     ttl: 3600
//     enabled: true
```

## 🔥 扩展追问 4：Spring Boot 3 的配置文件（application.yml/properties）有变化吗？

> **A:** 基本兼容，但有一些新特性：

```yaml
# Spring Boot 3 新配置文件特性：

# 1. 占位符嵌套（3.0+）
app:
  db:
    url: jdbc:mysql://${DB_HOST:localhost}:${DB_PORT:3306}
  # 嵌套占位符完美支持

# 2. 多文档 YAML（Multi-document YAML）增强
# File: application.yml
spring:
  config:
    activate:
      on-profile: dev
---
spring:
  config:
    activate:
      on-profile: prod
# 与 2.x 兼容

# 3. 配置服务器增强
spring:
  cloud:
    config:
      uri: https://config-server:8888
      fail-fast: true  # 3.0 新增：配置获取失败时快速失败
      retry:
        max-attempts: 6  # 重试次数 3.x 默认为 3，2.x 为 6

# 4. 新增配置属性
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s  # 3.0 新增：优雅关闭超时
  main:
    lazy-initialization: true          # 3.0 新增：懒初始化 Bean
    web-application-type: servlet     # 3.0 新增：显式指定类型

# 5. Relaxed Binding 规则（更严格但更一致）
# Spring Boot 3 对绑定更严格，建议使用标准的 kebab-case
```

## 🔥 扩展追问 5：Spring Boot 2.7 到 3.x 的迁移实战路径是什么？

> **A:** 完整的迁移检查清单：

```bash
# 迁移步骤清单

# Step 1: JDK 升级
java -version  # 确认 JDK 17+
# 下载安装 JDK 17+
# JAVA_HOME = JDK 17

# Step 2: Spring Boot 版本跳跃
# 先升到 2.7.x（最后一个 2.x 版本），确保所有测试通过
# 再升到 3.0.x
# 最后升到 3.2.x（当前稳定版）

# Step 3: 依赖扫描
# 使用 Spring Boot 官方迁移工具
# https://github.com/spring-projects/spring-boot/wiki/Spring-Boot-3.0-Migration-Guide

# Maven 依赖检查：
./mvnw dependency:analyze | grep -i javax
# 找出所有 javax 依赖

# Step 4: import 批量替换
# IntelliJ: Replace in Files
# javax.annotation → jakarta.annotation
# javax.persistence → jakarta.persistence
# javax.validation → jakarta.validation
# javax.servlet → jakarta.servlet
# javax.transaction → jakarta.transaction
# javax.jms → jakarta.jms

# Step 5: 配置类迁移（WebSecurityConfigurerAdapter）
# 见前面的 Security 迁移示例

# Step 6: 测试
./mvnw test

# Step 7: 运行时验证
# 逐个功能测试，特别关注：
# - 数据库连接（JPA/Hibernate 6）
# - 文件上传/下载（Servlet API 变化）
# - 定时任务（@Scheduled）
# - 消息队列（JMS → Jakarta JMS）
# - 缓存（如果用了特定 API）

# Step 8: GraalVM Native Image 验证（可选）
./mvnw -Pnative spring-boot:compile-no-fork
```

> **一句话总结**

Spring Boot 3 要求 JDK 17 最低，核心是 `javax.*` → `jakarta.*` 命名空间全面迁移（影响所有 Servlet/JPA/Validation/WebSocket 等组件），`WebSecurityConfigurerAdapter` 被 Lambda DSL 替代，`@AutoConfiguration` 位置从 `spring.factories` 迁移到 `AutoConfiguration.imports` 文件，新增 `@ConditionalOnWarDeployment`、`@ConditionalOnNotWarDeployment`、支持 `Type.SERVLET/REACTIVE` 的 `@ConditionalOnWebApplication`，以及 `@ConfigurationPropertiesScan` 和 Record 作为配置属性类型。

> **面试加分话术**

> "我完整经历过一次 Spring Boot 2.7 → 3.0 的生产迁移，踩了几个大坑：第一个是 `javax.persistence` → `jakarta.persistence`，我们项目用了 Hibernate 5.6 + Spring Data JPA，必须同时升级到 Hibernate 6.2，但 Hibernate 6 的 Entity 映射规则有变化——`@Column(length = 100)` 的 length 语义变了，varchar 映射默认变成 `varchar(255)`，导致老数据截断，后来改成 `@Column(columnDefinition = "VARCHAR(100)")` 才解决；第二个坑是 Spring Security 6，`WebSecurityConfigurerAdapter` 直接没了，所有配置必须改成 `SecurityFilterChain` Bean 加 Lambda DSL，我们有 5 个配置类花了 3 天才改完；第三个坑是部分第三方 SDK（如旧版 Elasticsearch 7.x）还没迁移到 jakarta，临时用了 `jakarta.jakartaee-bom` 兼容层过渡，但这个方案不推荐长期使用。迁移完成后，用 Spring Boot 3 的 GraalVM Native Image 部署，启动时间从 8 秒降到 300ms，对我们这种 Kubernetes 缩容频繁的服务价值很大。"
