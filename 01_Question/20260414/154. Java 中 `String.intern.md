## 1. 基础流程

### intern() 的核心原理

```
┌─────────────────────────────────────────────────────────────────────┐
│                      String.intern() 原理                           │
│                                                                     │
│  字符串常量池（String Table）：                                      │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  JDK 6: PermGen（永久代）                                    │   │
│  │  JDK 7+: Heap（堆内存）                                      │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  intern() 的作用：                                                  │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  1. 如果常量池中已经存在该字符串 → 返回常量池中的引用        │   │
│  │  2. 如果常量池中不存在 → 将字符串复制到常量池 → 返回引用    │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### intern() 代码示例

```java
// ========== 示例1：基本用法 ==========
String s1 = new String("hello");
String s2 = s1.intern();  // intern 后获取常量池引用

System.out.println(s1 == s2);  // false：s1 在堆，s2 在常量池

String s3 = "hello";       // 字面量，直接从常量池获取
System.out.println(s2 == s3);  // true：都指向常量池的同一个对象！

// ========== 示例2：new String() 的问题 ==========
String a = new String("hello") + new String("world");
String b = "helloworld";

System.out.println(a == b);  // JDK 6: false  JDK 7+: true!
// 原因：拼接的字符串在 JDK 7+ 会被优化为 intern()

// ========== 示例3：常量折叠 ==========
String c = "a" + "b";  // 编译时优化为 "ab"
String d = "ab";
System.out.println(c == d);  // true：都是常量池的 "ab"
```

## 2. 扩展追问

### ## 🔥 扩展追问 1：JDK 6 和 JDK 7+ 的 intern 行为有什么区别？

> **A:** JDK 6 时字符串常量池在 PermGen（永久代），`intern()` 会把字符串复制到 PermGen；JDK 7+ 常量池移到堆中，`intern()` 只在常量池中记录引用，不再复制，大字符串也不会占满 PermGen。

```
┌─────────────────────────────────────────────────────────────────────┐
│                 JDK 6 vs JDK 7+ intern() 行为对比                     │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                       JDK 6                                │   │
│  │                                                             │   │
│  │   堆内存                  PermGen                          │   │
│  │  ┌──────────────┐       ┌──────────────────┐               │   │
│  │  │ "hello"      │       │ String Table      │               │   │
│  │  │ (new String) │       │ "hello" ← 复制！  │               │   │
│  │  │              │       │ (和堆里的是不同对象)│              │   │
│  │  └──────────────┘       └──────────────────┘               │   │
│  │                                                             │   │
│  │   问题：PermGen 空间有限，大字符串 intern() 可能 OOM         │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                      JDK 7+                                │   │
│  │                                                             │   │
│  │                          堆内存                            │   │
│  │  ┌──────────────────────────────────────────────────┐      │   │
│  │  │                  String Table                     │      │   │
│  │  │  "hello" ←──────────┐                            │      │   │
│  │  │      (和堆对象是同一个引用！)  │                   │      │   │
│  │  └──────────────────────────────────────────────────┘      │   │
│  │                                                             │   │
│  │   改进：不再复制，只记录引用，节省内存                        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

```java
// ========== 关键区别示例 ==========

// 场景：大量动态字符串
String dynamic = new String("verylongstring".toCharArray());

// JDK 6：
// 1. 堆中创建 "verylongstring"
// 2. intern() 复制一份到 PermGen 的 String Table
// 3. 返回 PermGen 中的引用
// 4. 问题：PermGen 空间有限，大量 intern() 会 OOM

// JDK 7+：
// 1. 堆中创建 "verylongstring"
// 2. intern() 检查常量池，发现没有
// 3. 在常量池（堆）中记录一个指针，指向堆中的对象
// 4. 返回堆中的引用
// 5. 改进：不再复制，节省内存，也没有 PermGen OOM 问题
```

### ## 🔥 扩展追问 2：字符串常量池在 JDK 8 中位于哪里？

> **A:** JDK 8 把字符串常量池从 PermGen 移到了**堆内存**中，成为 StringTable（一个 HashMap），这是 Metaspace 溢出问题不再影响字符串常量池。

| JDK 版本 | 字符串常量池位置 | 大小限制 | OOM 类型 |
|---------|---------------|---------|---------|
| JDK 6 | PermGen（永久代） | `-XX:MaxPermSize` | PermGen OOM |
| JDK 7 | 堆（实验性） | `-XX:MaxPermSize` | - |
| JDK 8+ | 堆（StringTable） | `-XX:StringTableSize` | Heap OOM |

```bash
# StringTable 大小配置
-XX:StringTableSize=6000000  # 默认约 60 万桶，可调大

# 查看 StringTable 统计信息（JDK 8u60+）
jinfo -flag StringTableSize <pid>
jcmd <pid> VM.stringtable
```

### ## 🔥 扩展追问 3：大量调用 intern() 可能导致什么问题？

> **A:** 大量调用 `intern()` 会导致 **StringTable 膨胀**，占用大量堆内存，可能引发 OOM；且 HashMap 结构会拖慢查询性能。

```java
// ========== 危险场景：大量动态字符串 intern ==========

// ❌ 错误用法：把所有数据都 intern
public class BadExample {
    public static void main(String[] args) {
        // 模拟处理 1000 万条数据
        for (int i = 0; i < 10000000; i++) {
            // 每个不同的字符串都会被 intern
            String s = generateUniqueString(i);
            s.intern();  // 危险！StringTable 不断膨胀
        }
    }
}

// ========== 正确用法：只 intern 确实会重复的字符串 ==========

// ✅ 场景1：常量复用
String constant = "配置项_xxx".intern();  // 字符串固定，复用有意义

// ✅ 场景2：存在大量重复的字符串
// 例如：日志级别 ["DEBUG", "INFO", "WARN", "ERROR"]
// 100 万条日志，重复使用这 4 个字符串，intern() 有价值

// ❌ 场景3：每条数据都不同
// 例如：用户 ID、商品 ID
// intern() 毫无意义，只会浪费内存
```

```
┌─────────────────────────────────────────────────────────────────────┐
│                    intern() 的性能考量                              │
│                                                                     │
│  StringTable 原理：                                                 │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  HashMap<String, StringTableEntry>                          │   │
│  │                                                             │   │
│  │  intern() 过程：                                            │   │
│  │  1. 计算字符串 hashCode                                      │   │
│  │  2. 查找 HashMap 桶位置（hash % tableSize）                   │   │
│  │  3. 遍历桶内链表，比较 equals()                              │   │
│  │  4. 找到 → 返回引用；没找到 → 插入新条目                      │   │
│  │                                                             │   │
│  │  时间复杂度：O(n)，n = 桶内链表长度                          │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  StringTableSize 的影响：                                           │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  StringTableSize = 1000 → 10000 条字符串，链表长 → 查询慢   │   │
│  │  StringTableSize = 100000 → 分散存储 → 查询快               │   │
│  │                                                             │   │
│  │  建议：预估字符串数量 × 2，减少 hash 冲突                     │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

| intern() 适用场景 | intern() 不适用场景 |
|-----------------|-------------------|
| 字符串常量池化 | 大量唯一字符串（如 UUID、用户 ID） |
| 字符串重复率高 | 对性能敏感的短生命周期代码 |
| 需要比较 == 的场景 | 字符串长度很大（每个都占大量堆内存） |

### ## 🔥 扩展追问 4：intern() 和 == 比较有什么关系？

> **A:** `intern()` 的核心价值是让字符串可以用 `==` 比较而不是 `equals()`，前提是字符串确实会在常量池中重复。

```java
// ========== intern() 实现 == 比较 ==========

// ❌ 不使用 intern：必须用 equals()
String a = new String("hello");
String b = new String("hello");
System.out.println(a == b);           // false
System.out.println(a.equals(b));     // true

// ✅ 使用 intern：可以用 ==
String a = new String("hello").intern();
String b = new String("hello").intern();
System.out.println(a == b);           // true！

// ========== 实际应用：减少内存 + == 比较 ==========

// 场景：处理大量重复字符串（如消息队列消息）
class Message {
    private final String type;  // 消息类型有限：ORDER/CANCEL/REFUND
    
    public Message(byte[] data) {
        this.type = new String(data, 0, 10).intern();  // intern 后复用
    }
    
    // 可以用 == 比较了
    public boolean isOrder() {
        return this.type == "ORDER";  // 快速比较！
    }
}
```

### ## 🔥 扩展追问 5：String 的 equals() 和 == 有什么区别？

> **A:** `equals()` 比较字符串内容，`==` 比较对象引用；intern() 后的字符串可以让 `==` 也比较内容。

```java
// String equals() 源码
public boolean equals(Object anObject) {
    if (this == anObject) {
        return true;  // 先比较引用
    }
    if (anObject instanceof String) {
        String anotherString = (String) anObject;
        int n = value.length;  // value 是 char[]
        if (n == anotherString.value.length) {
            byte v1[] = hashCode();      // 缓存了 hashCode
            byte v2[] = anotherString.hashCode();
            if (n == hashCode) {
                return true;  // hashCode 不同直接返回 false
            }
            // 逐字符比较
        }
    }
    return false;
}

// intern() 让 == 可用的原因：
// 同一个字符串 intern() 后，指向 StringTable 中的同一个对象
// 所以 == 比较的是同一个引用，结果等价于 equals()
```

## 3. 一句话总结

> **String.intern() 让字符串进入常量池避免重复创建，JDK 6 把字符串复制到 PermGen 常量池（空间有限），JDK 7+ 只在堆中常量池记录引用不再复制；JDK 8 常量池位于堆内的 StringTable；大量 intern() 会导致 StringTable 膨胀和 OOM，只适合有大量重复值的字符串。**

> "我在项目里踩过 intern() 的坑——当时用 intern() 做消息类型去重，误把所有用户 ID 都 intern 了，导致 StringTable 从几十 MB 涨到 2G。后来查了一圈才发现，intern() 只适合那种真正会重复的字符串，像用户 ID 这种每次都不同的完全没必要。还有个经验：JDK 8 把常量池移到堆是个好事，但 StringTable 大小要调，`-XX:StringTableSize=6000000` 设大一些避免 hash 冲突。另外用 jcmd 可以直接看 StringTable 的统计信息，很方便排查问题。还有个细节：`-XX:+UseStringDeduplication`（G1 GC 支持）可以自动识别重复字符串并去重，比手动 intern 更安全。"
