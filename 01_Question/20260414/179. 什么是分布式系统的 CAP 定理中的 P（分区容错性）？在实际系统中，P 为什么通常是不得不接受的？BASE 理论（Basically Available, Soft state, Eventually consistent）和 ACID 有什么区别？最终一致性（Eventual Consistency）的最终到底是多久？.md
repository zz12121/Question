## 1. 基础流程

### 1.1 CAP 定理的三元悖论

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│            CAP 定理（Brewer's Theorem）                  │
│                                                         │
│                    ┌─────── C ───────┐                  │
│                    │  Consistency   │                  │
│                    │   一致性        │                  │
│                    └───────┬─────────┘                  │
│                            │                            │
│              ┌─────────────┼─────────────┐             │
│              │             │             │             │
│              ▼             │             ▼             │
│         ┌─────────┐        │       ┌─────────┐         │
│         │    A    │        │       │    P    │         │
│         │Availability│      │       │  分区    │         │
│         │ 可用性   │        │       │ 容错性   │         │
│         └─────────┘        │       └─────────┘         │
│                            │                            │
│         只能同时满足 2 个        │                        │
│         网络分区时必须牺牲 1 个   │                        │
└─────────────────────────────────────────────────────────┘

三个特性：
- C（Consistency）：所有节点看到相同的数据，写入后立即读取得到最新值
- A（Availability）：每个请求都能收到响应（非错误）
- P（Partition Tolerance）：分区（网络故障/节点宕机）时系统仍能运行

核心结论：CA = 不存在，CP 和 AP 是真实的选择
```

### 1.2 分区容错性 P 的真实含义

```
┌─────────────────────────────────────────────────────────┐
│  为什么 P 是"不得不接受"的？                            │
│                                                         │
│  网络分区（Network Partition）在现实中无法完全避免        │
│                                                         │
│  什么是分区？                                          │
│  ┌─────────────────────────────────────────────────┐   │
│  │  节点 A                            节点 B         │   │
│  │  ┌────────┐                      ┌────────┐    │   │
│  │  │  Master │    断网/延迟         │ Slave  │    │   │
│  │  └────────┘    ────────────────  └────────┘    │   │
│  │                 Network Partition                 │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  故障类型：                                            │
│  - 机房级别的网络分区（光缆被挖断）                    │
│  - 云服务商的可用区之间网络隔离                        │
│  - 交换机/路由器故障                                  │
│  - 极端网络延迟（> 秒级）                            │
│  - 节点 Full GC 导致的假死                            │
│                                                         │
│  在没有分区时：                                        │
│  C + A 可以同时满足                                    │
│  就像单机数据库（MySQL 单实例）                      │
│                                                         │
│  一旦发生分区：                                        │
│  CP 系统：停止响应，保证数据一致                      │
│  AP 系统：继续响应，返回可能不一致的数据              │
└─────────────────────────────────────────────────────────┘
```

```
P 不是"可选项"，而是"必然项"：

┌────────────────────────────────────────────────────────┐
│  问题：CAP 定理说只能三选二，为什么 P 是必然的？       │
│                                                        │
│  回答：CAP 定理的 P，指的是"系统是否在分区时运行"    │
│                                                        │
│  - 如果放弃 P：系统遇到分区就停止服务                  │
│  - 这意味着：任何网络故障都导致系统不可用             │
│  - 在真实分布式系统中，这是不可接受的                  │
│                                                        │
│  因此：                                                │
│  ✓ 真实系统必须选 P                                   │
│  ✓ 然后在 C 和 A 之间做选择                            │
│  ✓ CA 系统只存在于单机/单网络环境                     │
│                                                        │
│  实际上分布式系统的选择只有：                          │
│  - CP（如 Zookeeper、HBase、MongoDB 多数派）        │
│  - AP（如 Cassandra、DynamoDB、Redis Cluster）        │
└────────────────────────────────────────────────────────┘
```

### 1.3 CP vs AP 系统的典型代表

```
CP 系统（一致性优先）：
┌─────────────────────────────────────────────────────────┐
│  系统         │ 分区时行为       │ 典型场景             │
│  ────────────┼─────────────────┼─────────────────────│
│  Zookeeper   │ 停止写入         │ 配置中心、选举       │
│  HBase       │ 停止写入         │ 大数据分析           │
│  MongoDB     │ 降级为单机       │ 主从切换             │
│  etcd        │ 停止写入         │ 服务发现             │
│  RocketMQ    │ Master 切换      │ 消息队列             │
│                                                         │
│  共同特点：分区时牺牲可用性，保证数据强一致           │
└─────────────────────────────────────────────────────────┘

AP 系统（可用性优先）：
┌─────────────────────────────────────────────────────────┐
│  系统           │ 分区时行为           │ 典型场景         │
│  ──────────────┼─────────────────────┼─────────────────│
│  Cassandra     │ 继续读写，最终同步   │ 写入密集型      │
│  DynamoDB      │ 继续读写，读协调     │ 全球化部署      │
│  Redis Cluster │ 从节点继续读         │ 缓存层          │
│  DNS           │ 继续服务             │ 域名解析         │
│  Kafka（默认）  │ ISR 副本外继续写入  │ 日志系统        │
│                                                         │
│  共同特点：分区时继续服务，数据最终一致               │
└─────────────────────────────────────────────────────────┘
```

## 2. BASE 理论

### 2.1 BASE 理论详解

```
┌─────────────────────────────────────────────────────────┐
│  BASE = Basically Available + Soft state + Eventually consistent
│                                                         │
│  诞生背景：                                             │
│  - CAP 定理证明了 C 和 A 在分区时不可兼得              │
│  - BASE 提出了在分布式系统中实现可用性的实用方案      │
│  - 来源于 eBay 的实践经验（Dan Pritzker, 2008）       │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  B - Basically Available（基本可用）                    │
│  ─────────────────────────────────────────────────     │
│                                                         │
│  含义：系统保证可用性，但允许降级                      │
│                                                         │
│  降级方式：                                             │
│  1. 数量降级：部分节点故障时，QPS 降低但服务仍在      │
│  2. 功能降级：购物车只读时返回缓存数据，不支持下单    │
│  3. 时限降级：返回过期数据，但标记"仅供参考"         │
│                                                         │
│  例子：                                                 │
│  - 微博热搜降级：显示缓存的昨天热搜                    │
│  - 电商大促：关闭某些非核心功能（评论、积分）        │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  S - Soft State（软状态 / 中间状态）                   │
│  ─────────────────────────────────────────────────     │
│                                                         │
│  含义：系统状态可以在一段时间内不一致                  │
│                                                         │
│  三种软状态：                                          │
│  1. 主从数据不一致：Master 已写入，Slave 还未同步      │
│  2. 缓存数据不一致：Cache 和 DB 数据不同步             │
│  3. 多副本数据不一致：3 副本中有 1 副本是旧数据       │
│                                                         │
│  与硬状态对比：                                         │
│  - 硬状态：状态机状态必须严格一致（ACID）             │
│  - 软状态：允许中间状态存在，不影响系统可用性         │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  E - Eventually Consistent（最终一致性）                │
│  ─────────────────────────────────────────────────     │
│                                                         │
│  含义：系统在一段时间后，数据最终达到一致              │
│                                                         │
│  保证最终一致性的机制：                                 │
│  1. 异步复制（从 Master 同步到 Slave）                │
│  2. 冲突解决策略（Last-Write-Wins / Vector Clock）    │
│  3. 读修复（Read Repair：读取时顺便修复过期数据）     │
│  4. 写修复（Write Repair：写入时补偿更新其他副本）    │
│  5. Anti-Entropy（后台定期校验副本一致性）            │
└─────────────────────────────────────────────────────────┘
```

### 2.2 BASE vs ACID 对比

```
┌──────────────────────────────────────────────────────────────────┐
│                        BASE vs ACID                              │
│                                                                  │
│  ACID（传统关系型数据库）         BASE（分布式系统）              │
│  ──────────────────────────        ──────────────────────         │
│                                                                  │
│  Atomic（原子性）              Basically Available              │
│  全有或全无                    允许部分可用                      │
│                                 ↓                               │
│  Consistent（一致性）           Soft State                      │
│  每个事务前后数据库都一致      允许中间状态                      │
│                                 ↓                               │
│  Isolated（隔离性）            Eventually Consistent             │
│  并发事务互不干扰              最终数据一致                      │
│                                 ↓                               │
│  Durable（持久性）             （隐含保证）                      │
│  事务提交后不丢失              数据最终会持久化                  │
│                                                                  │
│  目标：强一致性                 目标：高可用 + 最终一致          │
│  牺牲：性能和可用性             牺牲：实时一致性                │
│  适用：单机、单节点             适用：分布式、多节点            │
└──────────────────────────────────────────────────────────────────┘
```

```
业务场景选择指南：

┌─────────────────────────────────────────────────────────┐
│  选 ACID（MySQL/PostgreSQL）：                         │
│  - 金融交易、转账                                      │
│  - 库存扣减                                            │
│  - 订单状态流转                                        │
│  - 任何不允许数据冲突的业务                            │
│                                                         │
│  选 BASE（Cassandra/MongoDB/Redis）：                  │
│  - 用户行为日志收集                                    │
│  - 社交媒体的点赞、阅读数                              │
│  - 实时排行榜                                          │
│  - 购物车（允许短暂数据不一致）                       │
│  - 物联网时序数据                                      │
└─────────────────────────────────────────────────────────┘
```

## 3. 最终一致性的"最终"是多久？

### 3.1 "最终"的时间量化

```
┌─────────────────────────────────────────────────────────┐
│  "最终"不是无限等待，是有条件的收敛                     │
│                                                         │
│  影响因素：                                           │
│  1. 副本数量（N）                                      │
│  2. 同步成功需要的副本数（W）                          │
│  3. 读取成功需要的副本数（R）                          │
│  4. 网络延迟                                          │
│  5. 故障检测时间                                       │
│  6. 冲突解决机制                                       │
└─────────────────────────────────────────────────────────┘

一致性模型光谱（从强到弱）：

┌─────────────────────────────────────────────────────────┐
│  强一致性（Linearizability）                            │
│  写入后立即可读，延迟 < 毫秒级                         │
│  代表：Zookeeper、etcd                                  │
│  代价：写入延迟高，分区时不可用                        │
├─────────────────────────────────────────────────────────┤
│  顺序一致性（Sequential Consistency）                    │
│  所有节点看到相同顺序的写入                            │
│  代表：Flink checkpoint                                 │
│  代价：比线性一致性稍宽松                              │
├─────────────────────────────────────────────────────────┤
│  因果一致性（Causal Consistency）                        │
│  有因果关系的操作有序，无因果关系的可乱序              │
│  代表：DynamoDB                                         │
│  代价：实现复杂                                         │
├─────────────────────────────────────────────────────────┤
│  最终一致性（Eventual Consistency）                     │
│  无更新时，所有副本最终一致                            │
│  "最终"：毫秒到秒级（取决于配置）                      │
│  代表：Cassandra、DynamoDB、Redis Cluster              │
│  代价：读取可能返回旧数据                             │
├─────────────────────────────────────────────────────────┤
│  读己之所写（Read-Your-Writes Consistency）             │
│  自己写入的一定能读到                                   │
│  代表：带 session 的 Cassandra                          │
│  代价：需要客户端 session 追踪                         │
├─────────────────────────────────────────────────────────┤
│  单调读（Monotonic Read）                              │
│  每次读到的数据不比上次旧                              │
│  代表：DynamoDB + consistent read                      │
│  代价：可能限制负载均衡                               │
└─────────────────────────────────────────────────────────┘
```

### 3.2 不同系统的"最终"时间实测

```
┌─────────────────────────────────────────────────────────┐
│  典型系统的收敛时间                                     │
│                                                         │
│  Redis Cluster：                                        │
│  - 异步复制，Slave 延迟一般 < 1 秒                     │
│  - 分区恢复后，复制差距在几秒内追赶完成                │
│  - Cluster 故障切换：秒级（10-30 秒含故障检测）        │
│                                                         │
│  Cassandra：                                            │
│  - 默认：写入 3 节点（W=1），读取 R=1                  │
│  - 一致性修复（Read Repair）：读取时主动修复           │
│  - 典型收敛时间：毫秒到秒                              │
│  - Hinted Handoff：故障节点恢复后补偿写入              │
│                                                         │
│  DynamoDB：                                             │
│  - 多区域部署，全球复制延迟 几十毫秒到几百毫秒         │
│  - 最终一致性读取：最终 < 1 秒                        │
│  - 强一致性读取：读请求路由到最新副本                 │
│                                                         │
│  MongoDB 副本集：                                       │
│  - 异步复制，Secondary 延迟 一般 < 1 秒                 │
│  - Write Concern w:1，"最终"几分钟内                  │
│  - Write Concern w:majority，约数百毫秒               │
│                                                         │
│  Kafka（分区副本）：                                   │
│  - ISR 机制：同步副本之间延迟 < 毫秒级                │
│  - 异步副本：消息积压时延迟 可能达到分钟级              │
│  - 消费者 Lag = 数据积压量                              │
└─────────────────────────────────────────────────────────┘
```

### 3.3 业务层面的"最终"可接受性判断

```
问：用户能感知到"最终不一致"吗？

┌─────────────────────────────────────────────────────────┐
│  可接受场景（用户无感知）：                             │
│                                                         │
│  ✓ 点赞数：少几十个赞，用户根本不知道                  │
│  ✓ 阅读量：少几百次阅读，无人关注                      │
│  ✓ 评论数：稍有延迟，不影响核心功能                    │
│  ✓ 排行榜：每小时更新一次，用户不觉得慢               │
│  ✓ 已读标记：稍有时延，可接受                          │
│                                                         │
│  不可接受场景（用户会投诉）：                          │
│                                                         │
│  ✗ 余额查询：余额已扣但显示未扣，用户恐慌             │
│  ✗ 库存扣减：超卖，用户下单成功但无货                 │
│  ✗ 订单状态：已支付但仍显示待支付，用户重复支付       │
│  ✗ 优惠券：已使用但系统未感知，重复核销               │
│  ✗ 转账金额：已转出但显示未转，恐慌投诉               │
└─────────────────────────────────────────────────────────┘

实战策略：

场景1: 点赞（可接受最终一致性）
- 直接写本地/主节点
- 异步扩散到其他副本
- 展示时用本地缓存
- 后台补偿同步

场景2: 账户余额（强一致性）
- 使用分布式事务（如 Seata AT 模式）
- 或使用乐观锁版本号
- 扣减时 CAS 重试

场景3: 订单状态（混合策略）
- 核心状态（已支付）用强一致性
- 非核心状态（已发货通知）用最终一致性
- 用户可见状态：强一致性
- 内部状态：最终一致性
```

## 🔥 扩展追问 1：Zookeeper 为什么选择 CP 而不是 AP？

> **A:** Zookeeper 的定位决定它必须选 CP：

```
Zookeeper 的核心使用场景：

┌─────────────────────────────────────────────────────────┐
│  Zookeeper 的典型应用：                                 │
│                                                         │
│  1. 分布式锁（抢锁）                                    │
│     ├─ 锁的获取必须是强一致的                          │
│     ├─ 两个客户端不能同时获得同一把锁                   │
│     └─ 必须有唯一正确的答案                            │
│                                                         │
│  2. 服务注册与发现                                      │
│     ├─ 某个服务只能注册在一个 Master 上               │
│     └─ 客户端看到的配置必须一致                        │
│                                                         │
│  3. Leader 选举                                        │
│     ├─ 集群必须选出一个 Leader                         │
│     └─ 所有节点必须认同同一个 Leader                   │
│                                                         │
│  4. 配置管理                                            │
│     ├─ 所有节点必须读取到相同的配置                    │
│     └─ 配置变更必须原子通知                           │
└─────────────────────────────────────────────────────────┘

为什么不能选 AP？

Zookeeper 分区时的行为（CP）：
- 分区后，少数派节点停止服务
- 多数派继续服务，保证一致性
- 分区恢复后，状态自动同步

如果 Zookeeper 选 AP（假想）：
- 分区后两边都响应
- 锁在两个分区各被一个客户端获得
- 分布式锁完全失效
- 服务注册在两边各有一个 Master
- 整个分布式协调系统崩溃

结论：
- Zookeeper 的存在价值就是提供强一致性协调
- 如果它选 AP，就失去了存在的意义
- 所以 Zookeeper/etcd 永远选 CP
```

## 🔥 扩展追问 2：如何实现"读写都满足一致性"的系统？

> **A:** 实际是 CAP 的不同组合策略：

```
方案1: 读写都走 Leader（强一致）
┌─────────────────────────────────────────────────────────┐
│  任何节点收到读请求 → 转发到 Leader → Leader 返回      │
│  写入 → 同步到多数派节点 → 返回成功                   │
│                                                         │
│  优点：读写都是强一致的                                 │
│  缺点：读取延迟高（多一次网络跳转）                    │
│  代表：Zookeeper（ZAB 协议）                           │
└─────────────────────────────────────────────────────────┘

方案2: Leader 读 + 法定人数写（折中）
┌─────────────────────────────────────────────────────────┐
│  写入：同步到 N/2+1 节点（W）→ 返回成功               │
│  读取：读任意节点（R=1）                              │
│  同步修复：后台 anti-entropy 校验                      │
│                                                         │
│  读写延迟：                                             │
│  - 写延迟：高（N/2 节点同步）                         │
│  - 读延迟：低（单节点）                               │
│  代表：DynamoDB（可调一致性）                          │
│                                                         │
│  配置公式（Quorum）：                                   │
│  W + R > N → 读写必然重叠 → 强一致性                 │
│  例如：N=3, W=2, R=2 → 2+2 > 3，读写必重叠           │
└─────────────────────────────────────────────────────────┘

方案3: 同步复制 + 延迟读取
┌─────────────────────────────────────────────────────────┐
│  写入：同步到所有节点（同步复制）                      │
│  读取：等待数据同步到本地后再响应                     │
│                                                         │
│  问题：延迟极高，跨机房时不可用                        │
│  代表：传统 MySQL 主从半同步复制（lossless）          │
└─────────────────────────────────────────────────────────┘

实际推荐配置（Cassandra 为例）：
┌─────────────────────────────────────────────────────────┐
│  场景1: 强一致性要求                                   │
│  CONSISTENCY_LEVEL = ALL                               │
│  写入：等待所有节点确认                                │
│  读取：读取所有节点返回最新                            │
│  延迟：极高，分区时不可用                               │
│                                                         │
│  场景2: 读写均衡                                        │
│  CONSISTENCY_LEVEL = QUORUM                            │
│  N=3, W=2, R=2                                         │
│  延迟：中等，分区时仍可写（R+W>N 保证）                │
│                                                         │
│  场景3: 写入优先                                        │
│  CONSISTENCY_LEVEL = ONE                               │
│  写入：写入 1 个节点就返回                             │
│  读取：读取 1 个节点                                   │
│  延迟：极低，最终一致性                                │
└─────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 3：Sagas 模式是最终一致性的解决方案吗？

> **A:** Sagas 是处理分布式事务最终一致性的经典模式：

```
Sagas vs 传统分布式事务（2PC）：

┌─────────────────────────────────────────────────────────┐
│  2PC（两阶段提交）：                                    │
│  - 准备阶段：所有参与者 Prepare                         │
│  - 提交阶段：所有参与者 Commit                         │
│  - 问题：阻塞、单点故障、数据不一致风险               │
│  - 代表：X/Open DTP                                    │
│                                                         │
│  Sagas（长事务模式）：                                  │
│  - 把大事务拆成多个本地事务                            │
│  - 每个本地事务有对应的补偿事务                        │
│  - 某步失败时，逆向执行已完成的补偿事务                │
│  - 不阻塞，异步执行                                     │
└─────────────────────────────────────────────────────────┘

Sagas 实现示例（订单 → 支付 → 库存）：

┌─────────────────────────────────────────────────────────┐
│  正向流程：                                             │
│  Step 1: 创建订单（本地事务）                         │
│  Step 2: 预扣库存（本地事务）                         │
│  Step 3: 预占积分（本地事务）                         │
│  Step 4: 调用支付（外部服务）                          │
│  Step 5: 确认订单                                      │
│                                                         │
│  失败补偿（如果 Step 4 失败）：                        │
│  Compensate Step 3: 返还积分                          │
│  Compensate Step 2: 返还库存                          │
│  Compensate Step 1: 取消订单                          │
└─────────────────────────────────────────────────────────┘

Sagas 的两个变种：

┌─────────────────────────────────────────────────────────┐
│  编排式 Sagas（Choreography）：                         │
│  - 每个服务自己订阅事件，执行后发布事件               │
│  - 事件驱动，无中心协调者                              │
│  - 适合简单场景                                        │
│                                                         │
│  协调式 Sagas（Orchestration）：                        │
│  - 有一个中心协调者（Saga Orchestrator）              │
│  - 协调者决定下一步执行/补偿                           │
│  - 适合复杂场景，更可控                               │
│  - 代表：Seata Sagas                                   │
└─────────────────────────────────────────────────────────┘

Sagas 的局限性：
- 补偿不是 100% 可靠（外部系统调用可能失败）
- 无法保证隔离性（并发时有中间状态可见）
- 需要幂等设计（补偿可能重复执行）
```

## 🔥 扩展追问 4：为什么 Redis Cluster 是 AP 而不是 CP？

> **A:** Redis Cluster 的设计哲学与 Redis 的定位一致：

```
Redis 的定位：高性能内存缓存/数据结构服务

┌─────────────────────────────────────────────────────────┐
│  Redis Cluster 的 AP 选择（可用性优先）：                │
│                                                         │
│  分区发生时：                                          │
│  - 从节点（Slave）仍然响应读请求                        │
│  - 新写入可以写入主节点（Master）                       │
│  - 分区恢复后，自动同步缺失的数据                       │
│                                                         │
│  为什么这样设计？                                       │
│  1. Redis 的价值在于"快"                               │
│     - 作为缓存，数据丢失可接受                        │
│     - 不可用比数据不一致更严重                        │
│                                                         │
│  2. 数据最终来自数据库                                  │
│     - 缓存雪崩 → 从 DB 回源                            │
│     - Redis 短暂不可用 → 系统降级到 DB               │
│                                                         │
│  3. 分区时间通常很短                                   │
│     - 机房内网络故障：秒级恢复                        │
│     - 数据不一致时间：秒级                           │
│     - 对缓存场景完全可接受                            │
└─────────────────────────────────────────────────────────┘

Redis Cluster 分区的实际行为：

分区前（3 主 3 从）：
  Master-A (Slot 0-5460)
  Master-B (Slot 5461-10922)
  Master-C (Slot 10923-16383)
  各有 1 个 Slave

分区后（Master-B 和 Slave-B 分开）：
  分区侧 A+C（多数派）：
    继续接受所有 Slot 的写入
    服务正常
  
  分区侧 B（少数派）：
    Master-B 停止写入（CP 行为）
    Slave-B 升为 Master（故障转移）
    新 Master-B' 继续服务

问题：分区时 Master-B 的数据可能丢失！
  如果 Master-B 在分区期间写入新数据
  而 Slave-B 还没同步
  分区恢复后，这部分数据永久丢失

这是 AP 的代价：可能丢失数据，但服务持续可用
```

## 🔥 扩展追问 5：如何在业务代码中量化"最终一致性窗口"？

> **A:** 监控和度量最终一致性收敛时间：

```
衡量指标：

┌─────────────────────────────────────────────────────────┐
│  指标1: Replication Lag（复制延迟）                     │
│                                                         │
│  副本延迟 = Master 最新 LSN - Slave 已同步 LSN          │
│                                                         │
│  Redis:                                                 │

│  master_repl_offset: 12345678                          │
│  slave_repl_offset: 12345678                           │
│  lag: 0  ← 延迟（秒）                                 │
│                                                         │
│  MySQL:                                                 │
│  SHOW SLAVE STATUS\G                                   │
│  Seconds_Behind_Master: 0                              │
│  Read_Master_Log_Pos: xxx                             │
│                                                         │
│  Cassandra:                                             │
│  nodetool tpstats                                      │
│  # 查看 Replication Stage 队列长度                      │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  指标2: Conflict Rate（冲突率）                         │
│                                                         │
│  冲突数 / 总写入数 = 冲突率                            │
│                                                         │
│  Cassandra:                                             │
│  nodetool proxyhistograms                             │
│  # 查看读写冲突统计                                    │
│                                                         │
│  DynamoDB:                                              │

│  ConditionalCheckFailedException → 冲突发生           │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  业务层面的可接受阈值：                                  │
│                                                         │
│  对于延迟敏感业务（点赞/阅读数）：                      │
│  - 接受范围：P99 < 5 秒                                │
│  - 告警阈值：P99 > 30 秒                              │
│                                                         │
│  对于一致性敏感业务（订单）：                          │
│  - 接受范围：P99 < 1 秒                               │

│                                                         │
│  对于金融业务：                                         │
│  - 必须强一致性，不接受最终一致性                      │
│  - 使用分布式事务（Seata/TCC）                        │
└─────────────────────────────────────────────────────────┘
```

> **一句话总结**

**P（分区容错性）是分布式系统无法回避的必然**，真实系统只能在 CP（保证强一致但分区时部分不可用，如 Zookeeper/etcd）和 AP（保证可用但可能返回旧数据，如 Redis Cluster/Cassandra）之间二选一；**BASE 理论是对 AP 系统的工程化实践总结**，通过"基本可用 + 软状态 + 最终一致性"换取高可用性；**"最终"通常在毫秒到秒级收敛**（Redis 复制延迟 <1s、Cassandra 读修复毫秒级、DynamoDB 全球复制 <1s），但对余额/库存/订单等强一致场景必须用分布式事务或强一致协议，不能依赖最终一致性。

> **面试加分话术**

> "我之前负责过一套分布式营销积分系统，对 CAP 和 BASE 的取舍有很深的实战感受。最初用的是 MySQL 主从，读写都在主库，一致性没问题但单机 QPS 只有 8000，大促时完全扛不住；后来引入了 Redis Cluster 做缓存层，把积分查询和部分更新走 Redis，数据最终从 MySQL 回源。结果踩了一个坑：积分扣减时 Redis 写成功但 MySQL 同步失败了（网络抖动），导致用户积分扣了但 DB 里没扣，用 Redis 的余额比 MySQL 大，用户刷出了负数积分。后来我们做了两件事：一是积分扣减走分布式事务（Seata AT 模式），Redis 只做缓存加速，核心写链路保证强一致；二是加了一套后台对账系统，每分钟扫描 Redis 和 MySQL 的积分差值，超过阈值立即告警并自动补偿。这套机制上线后，最终一致性窗口控制在 5 秒内，分区时也不会出现数据不一致。核心教训就是：**BASE 只适用于可以容忍短暂不一致的业务，核心余额/库存类业务必须强一致，不要为了性能牺牲正确性**。"
