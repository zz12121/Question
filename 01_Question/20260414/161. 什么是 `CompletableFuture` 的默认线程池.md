## 1. 基础流程

### 1.1 ForkJoinPool.commonPool() 是什么

```
ForkJoinPool.commonPool() 的定位：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│   CompletableFuture.supplyAsync(() -> xxx)                 │
│        ↓                                                     │
│   默认使用 ForkJoinPool.commonPool()                       │
│        ↓                                                     │

│   CompletableFuture.supplyAsync(() -> xxx, customPool)     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

```java
// commonPool 的获取方式
ForkJoinPool common = ForkJoinPool.commonPool();

// 线程数计算规则
public static int commonPoolParallelism() {
    return ForkJoinPool.getCommonPoolParallelism();
}

// 默认值 = max(1, Runtime.availableProcessors() - 1)
```

### 1.3 commonPool 的线程复用机制

```
ForkJoinPool 的 Work-Stealing 工作窃取算法：

普通线程池（阻塞模型）：
┌─────────────────────────────────────────────────────────────┐
│  线程1: [任务A][任务B(阻塞)][任务C(等待)]  ← 线程B被阻塞后  │
│          线程1空闲，但只能等任务B完成                         │
│  线程2: [任务D] ← 完全空闲，无法"偷"任务A                   │
│                                                              │
│  问题：阻塞任务导致线程资源浪费                              │
└─────────────────────────────────────────────────────────────┘

ForkJoinPool（工作窃取模型）：
┌─────────────────────────────────────────────────────────────┐
│  线程1的队列: [任务A][任务B]                                 │
│                          ↓                                  │
│              B阻塞后，线程1继续处理队列尾部                  │
│                          ↓                                  │
│  线程2空闲时: "偷"任务A ← 从线程1队列尾部偷取（快！）      │
│                                                              │
│  线程3空闲时: "偷"任务A ← 从线程1队列尾部偷取              │
│                                                              │
│  核心思想：LIFO（尾部偷取）> FIFO（头部偷取）               │
│            → 减少 cache line 竞争                           │
└─────────────────────────────────────────────────────────────┘
```

```java
// commonPool 中的 ForkJoinTask
// ForkJoinTask 是轻量级任务（比 Thread 更轻量）：
//   - 内部有 workQueue（双端队列）
//   - 任务可以"分叉"(fork)出子任务
//   - 子任务可以被其他线程"窃取"(join)

// 示例：RecursiveTask（带返回值的分治任务）
public class SumTask extends RecursiveTask<Long> {
    private final long[] array;
    private final int start, end;
    private static final int THRESHOLD = 10000;

    public SumTask(long[] array, int start, int end) {
        this.array = array;
        this.start = start;
        this.end = end;
    }

    @Override
    protected Long compute() {
        if (end - start <= THRESHOLD) {
            // 顺序计算
            long sum = 0;
            for (int i = start; i < end; i++) sum += array[i];
            return sum;
        }

        // 分治：拆成两个子任务
        int mid = (start + end) / 2;
        SumTask left = new SumTask(array, start, mid);
        SumTask right = new SumTask(array, mid, end);

        left.fork();   // 异步提交到队列
        return right.compute() + left.join();  // 窃取 + 合并
    }
}

// 使用 commonPool 执行
ForkJoinPool.commonPool().invoke(new SumTask(data, 0, data.length));
```

## 2. 扩展追问

### 🔥 扩展追问 1：如果 commonPool 中提交了阻塞任务，会发生什么？

> **A:** 后果很严重——**commonPool 是全局共享的，阻塞任务会饿死其他无辜代码**：

```
阻塞任务的灾难场景：
┌──────────────────────────────────────────────────────────────┐
│  commonPool 默认线程数 = CPU - 1 = 7（8核机器）             │
│                                                               │
│  某模块：CF.supplyAsync(() -> blockingIO())                  │
│  线程1: [blockingIO 任务] ← 阻塞在网络/磁盘 I/O 上          │
│  线程2: [blockingIO 任务] ← 阻塞在网络/磁盘 I/O 上          │
│  线程3: [blockingIO 任务] ← 阻塞在网络/磁盘 I/O 上          │
│  ...                                                          │
│  线程7: [blockingIO 任务] ← 阻塞在网络/磁盘 I/O 上          │
│                                                               │
│  所有 7 个线程都被阻塞！                                      │
│                                                               │
│  其他代码：                                                   │
│    parallelStream().forEach(...) ← 找不到可用线程，排队     │
│    CompletableFuture.allOf(...)      ← 找不到可用线程，排队 │
│    Stream.parallel().sum()           ← 找不到可用线程，排队 │
│                                                               │
│  结果：整个 JVM 的并行流和 CompletableFuture 全部瘫痪       │
└──────────────────────────────────────────────────────────────┘
```

```java
// 演示：阻塞任务饿死 commonPool
public class BlockedCommonPool {
    public static void main(String[] args) throws Exception {
        ForkJoinPool pool = ForkJoinPool.commonPool();
        System.out.println("初始活跃线程: " + pool.getActiveThreadCount());  // 0

        // 提交 7 个阻塞任务（占满 commonPool）
        List<CompletableFuture<Void>> tasks = new ArrayList<>();
        for (int i = 0; i < 7; i++) {
            CompletableFuture<Void> f = CompletableFuture.runAsync(() -> {
                try {
                    // 模拟阻塞 I/O（如 HTTP 调用超时 30 秒）
                    Thread.sleep(30_000);
                } catch (InterruptedException e) {}
            });
            tasks.add(f);
        }

        System.out.println("7个阻塞任务提交后活跃线程: " + pool.getActiveThreadCount());  // 7
        // 所有线程都被阻塞了

        // 再提交一个普通任务
        CompletableFuture<String> innocent = CompletableFuture.supplyAsync(() -> {
            System.out.println("我被执行了吗？");
            return "done";
        });
        // → 被阻塞在队列中，直到有线程释放！
        // 如果前面的阻塞任务不结束，这个任务永远不会被执行

        // 验证：parallelStream 也会受影响
        IntStream.range(0, 1000).parallel()
            .forEach(i -> {
                // 这个并行流找不到线程，被迫串行执行或阻塞
            });
    }
}
```

**解决方案：永远不要在 commonPool 中提交阻塞任务**

```java
// ✅ 正确做法：使用专用线程池处理阻塞 I/O
ExecutorService ioPool = new ThreadPoolExecutor(
    20, 20,
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000),
    new ThreadPoolExecutor.CallerRunsPolicy()
);

CompletableFuture.supplyAsync(() -> blockingHttpCall(), ioPool)
    .thenApply(result -> process(result));

// ✅ 或者使用虚拟线程（Java 21+）处理阻塞 I/O
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    CompletableFuture.supplyAsync(() -> blockingHttpCall(), executor)
        .thenApply(result -> process(result));
}
```

### 🔥 扩展追问 2：parallelStream 和 commonPool 的关系是什么？

> **A:** `parallelStream()` 默认也使用 `ForkJoinPool.commonPool()`，两者完全共享同一个池：

```java
// 底层都指向同一个 commonPool
public class Streams {
    public static <T> Stream<T> parallelStream(Supplier<T> supplier) {
        return new ReferencePipeline<>(..., 
            ForkJoinPool.commonPool()  // ← 完全相同！
        );
    }
}

// 示例：parallelStream 和 CompletableFuture 争抢同一池子
public class StreamVsCF {
    public static void main(String[] args) {
        // 线程1：parallelStream 占用 3 个线程
        List<String> result1 = IntStream.range(0, 1000)
            .parallel()
            .mapToObj(i -> heavyComputation(i))
            .collect(Collectors.toList());

        // 线程2：CompletableFuture 同时提交，也用 commonPool
        CompletableFuture.supplyAsync(() -> heavyComputation2())
            .thenApply(x -> heavyComputation3());

        // 两者都在 7 个线程的池子里抢资源
        // → 实际并行度下降
    }
}

// ✅ 解决方案：为 parallelStream 配置独立池
System.setProperty("java.util.concurrent.ForkJoinPool.common.parallelism", "100");
// 警告：全局生效，可能影响其他模块

// ✅ 更好的方案：用独立 ForkJoinPool
ForkJoinPool myPool = new ForkJoinPool(8);
myPool.submit(() ->
    IntStream.range(0, 1000)
        .parallel()
        .mapToObj(i -> heavyComputation(i))
        .collect(Collectors.toList())
);
myPool.shutdown();

// ✅ CompletableFuture 也用独立池
ExecutorService myExec = Executors.newFixedThreadPool(8);
CompletableFuture.supplyAsync(() -> xxx, myExec);
```

### 🔥 扩展追问 3：ForkJoinPool 和普通线程池（ThreadPoolExecutor）的核心区别是什么？

> **A:** 区别在于**任务模型**和**线程利用率**：

| 维度 | `ForkJoinPool` | `ThreadPoolExecutor` |
|---|---|---|
| **核心算法** | Work-Stealing（工作窃取） | Work-Chaining（固定线程分配） |
| **任务类型** | ForkJoinTask（可分叉/合并） | Runnable / Callable |
| **线程饥饿** | 空闲线程主动窃取其他队列的任务 | 阻塞线程直接浪费 |
| **适用场景** | 分治递归、数据并行 | 独立阻塞任务、I/O 操作 |
| **阻塞容忍** | 阻塞任务会破坏池子 | 每个线程独立，互不影响 |
| **任务粒度** | 细粒度子任务 | 粗粒度独立任务 |
| **并行度控制** | 通过 parallelism 参数 | 通过核心/最大线程数 |

```java
// ThreadPoolExecutor 的问题（线程饥饿）：
ExecutorService exec = Executors.newFixedThreadPool(4);
exec.submit(() -> {
    Thread.sleep(10_000);  // 线程1被阻塞
});
exec.submit(() -> {
    Thread.sleep(10_000);  // 线程2被阻塞
});
exec.submit(() -> {
    Thread.sleep(10_000);  // 线程3被阻塞
});
exec.submit(() -> {
    Thread.sleep(10_000);  // 线程4被阻塞
});
exec.submit(() -> {
    System.out.println("我永远不会被执行！");  // 队列排队
});
// 线程5-... 都在队列中等待，前4个线程永久阻塞

// ForkJoinPool 的问题（阻塞任务）：
// 如果提交 ForkJoinTask.compute() 内部有阻塞...
RecursiveTask<Long> badTask = new RecursiveTask<>() {
    protected Long compute() {
        try {
            Thread.sleep(10_000);  // ❌ 阻塞！
        } catch (Exception e) {}
        return 1L;
    }
};
// 7 个线程都被阻塞 → 整个池子瘫痪
// → ForkJoinPool 设计假设任务都是"计算密集型"
```

### 🔥 扩展追问 4：如何监控 commonPool 的健康状态？

> **A:** 有几种实用的监控手段：

```java
public class CommonPoolMonitor {
    public static void main(String[] args) {
        ForkJoinPool pool = ForkJoinPool.commonPool();

        // 定期打印池子状态
        ScheduledExecutorService monitor = 
            Executors.newSingleThreadScheduledExecutor();
        
        monitor.scheduleAtFixedRate(() -> {
            System.out.println("=== ForkJoinPool 状态 ===");
            System.out.println("parallelism:    " + pool.getParallelism());
            System.out.println("poolSize:       " + pool.getPoolSize());
            System.out.println("activeCount:    " + pool.getActiveThreadCount());
            System.out.println("runningCount:   " + pool.getRunningThreadCount());
            System.out.println("queuedCount:    " + pool.getQueuedSubmissionCount());
            System.out.println("stealCount:     " + pool.getStealCount());  // 被窃取的任务数
            System.out.println("=============");
        }, 0, 3, TimeUnit.SECONDS);

        // 触发一个长任务
        CompletableFuture.supplyAsync(() -> {
            try { Thread.sleep(5000); } catch (Exception e) {}
            return "done";
        });
    }
}
```

**关键监控指标解释**：

| 指标 | 含义 | 异常信号 |
|---|---|---|
| `activeThreadCount` | 正在执行任务的线程数 | ≈ parallelism = 线程全忙 |
| `runningThreadCount` | 运行中（未阻塞）的线程数 | < activeThreadCount = 有阻塞 |
| `queuedSubmissionCount` | 队列中等待的任务数 | 持续增长 = 吞吐量不足 |
| `stealCount` | 全局任务窃取总次数 | 增长缓慢 = 任务分布不均 |
| `poolSize` | 当前线程数 | 等于 parallelism = 无空闲 |

### 🔥 扩展追问 5：为什么 ForkJoinPool 默认 parallelism = CPU - 1？

> **A:** 这是 J.U.Executor 设计者的经验值，背后的逻辑：

```
CPU 核数假设：N = Runtime.availableProcessors()

设计考量：
┌─────────────────────────────────────────────────────────────┐
│  场景1：N = 8，如果 pool 用了 8 个线程                      │
│         → 如果有 1 个线程被 GC 暂停（Stop-The-World）      │
│         → 剩下 7 个线程等 1 个 → 并行度降为 7/8           │
│         → GC 线程和业务线程争抢 CPU                        │
│                                                               │
│  场景2：N = 8，如果 pool 用 7 个线程                       │
│         → GC 有 1 个完整核心可用                           │
│         → 业务线程不受 GC 暂停影响                          │
│         → 并行度 = 7/8                                      │
│                                                               │
│  结论：留 1 个核心给 GC/OS 是最佳实践                       │
│                                                               │
│  ⚠️ 但对于计算密集型任务，parallelism = N 效果更好         │
│  （因为 GC 可以用更短的 STW，任务之间无 I/O 等待）         │
└─────────────────────────────────────────────────────────────┘
```

```bash
# 生产环境建议：
# 计算密集型任务（JAVA-9+ 并行 GC）：
#   -XX:ParallelGCThreads=N  # 设置 GC 线程数 = N
#   -Djava.util.concurrent.ForkJoinPool.common.parallelism=N

# 这样 GC 和业务线程各用各的核心，互不干扰
```

## 3. 一句话总结

> **`ForkJoinPool.commonPool()` 是 JVM 全局共享的并行计算池，默认线程数 = CPU 核数 - 1，使用工作窃取算法但**假设任务都是计算密集型**，一旦提交阻塞 I/O 任务占满所有线程，就会饿死 parallelStream、CompletableFuture 和所有其他共用该池的代码——所以处理 I/O 类任务必须用 ThreadPoolExecutor 专用池。**

## 4. 面试加分话术

> "我之前踩过一个坑：有个接口用 `CompletableFuture.supplyAsync()` 调用下游接口超时时间是 30 秒，结果并发量一上来，7 个线程全部被阻塞，`parallelStream` 的数据处理全部卡死。最后用 `arthas` 的 `thread` 命令看到 commonPool 的 7 个线程状态全是 `WAITING`，才定位到问题。后面把所有 I/O 型 `supplyAsync` 全部改成了自定义的 `ThreadPoolExecutor`，用监控看到 commonPool 的活跃线程数稳定在 0-1，彻底解决。"

> "关于选型，我的经验是：**I/O 密集型用 ThreadPoolExecutor，计算密集型用 ForkJoinPool**，commonPool 只用来跑一些轻量的、确定不会阻塞的并行计算，比如 List 的并行流处理 `IntStream.range(0, 1000).parallel().sum()` 这种。如果你不确定任务会不会阻塞，永远不要用 commonPool。"
