### 基础流程

```
引用强度从强到弱：

① 强引用（Strong Reference）
   └─ Object obj = new Object()
   └─ GC 从不回收，宁可 OOM

② 软引用（Soft Reference）
   └─ SoftReference<T> ref = new SoftReference<>(obj)
   └─ GC 在内存不足时回收

③ 弱引用（Weak Reference）
   └─ WeakReference<T> ref = new WeakReference<>(obj)
   └─ GC 每次回收都会清理

④ 虚引用（Phantom Reference）
   └─ PhantomReference<T> ref = new PhantomReference<>(obj, queue)
   └─ 对象没实际用途，GC 时收到通知
```

**简单示例：**

```java
// 强引用：普通写法，GC 不回收
Object strong = new Object();

// 软引用：内存不足时才回收
SoftReference<Object> soft = new SoftReference<>(new Object());

// 弱引用：下一次 GC 就回收
WeakReference<Object> weak = new WeakReference<>(new Object());

// 虚引用：配合引用队列，追踪回收时机
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantom = new PhantomReference<>(new Object(), queue);
```

## 🔥 扩展追问 1：四种引用的回收时机分别是什么？

**GC 回收规则：**

```
┌──────────────────────────────────────────────────────────────┐
│  强引用（Strong）                                            │
│  └─ 永不回收！哪怕OOM也不回收                                 │
│  └─ 只要 GC Roots 能到达，就一定存活                          │
├──────────────────────────────────────────────────────────────┤
│  软引用（Soft）                                              │
│  └─ 内存足够时：不回收                                        │
│  └─ 内存不足时：回收                                          │
│  └─ 回收前还会尝试最后一次，不会直接OOM                        │
├──────────────────────────────────────────────────────────────┤
│  弱引用（Weak）                                              │
│  └─ 每次 GC 都回收                                           │
│  └─ 不管内存够不够                                           │
│  └─ 比软引用更激进                                           │
├──────────────────────────────────────────────────────────────┤
│  虚引用（Phantom）                                           │
│  └─ 形同虚设，无法通过 get() 获取对象                         │
│  └─ 对象回收时，reference 会被放入引用队列                    │
│  └─ 唯一作用：追踪对象何时被回收                              │
└──────────────────────────────────────────────────────────────┘
```

**代码验证：**

```java
import java.lang.ref.*;

public class ReferenceTest {
    public static void main(String[] args) throws InterruptedException {
        // 软引用
        SoftReference<byte[]> softRef = new SoftReference<>(new byte[1024 * 1024 * 10]);
        System.out.println("软引用：" + softRef.get());  // 有值
        
        // 手动 GC（不一定触发，取决于 GC 策略）
        System.gc();
        Thread.sleep(100);
        System.out.println("GC 后软引用：" + softRef.get());  // 可能还有
        
        // 弱引用
        WeakReference<byte[]> weakRef = new WeakReference<>(new byte[1024 * 1024 * 10]);
        System.out.println("弱引用：" + weakRef.get());  // 有值
        
        System.gc();
        Thread.sleep(100);
        System.out.println("GC 后弱引用：" + weakRef.get());  // null！
        
        // 虚引用
        ReferenceQueue<byte[]> queue = new ReferenceQueue<>();
        PhantomReference<byte[]> phantomRef = new PhantomReference<>(
            new byte[1024 * 1024 * 10], queue);
        System.out.println("虚引用 get()：" + phantomRef.get());  // 永远是 null
        
        System.gc();
        Thread.sleep(100);
        System.out.println("引用队列是否有记录：" + queue.poll());  // PhantomReference
    }
}
```

**面试加分话术：**

> "记法很简单：**强引用不死，软引用等内存不足，弱引用每次都死，虚引用只是个通知**。软引用适合做缓存，弱引用适合做容器（Map 的 key），虚引用适合做对象生命周期追踪。面试官可能追问：`System.gc()` 能不能保证触发 GC？——不能，gc() 只是建议 JVM 去 GC，真正什么时候回收是 JVM 决定的。"

## 🔥 扩展追问 2：WeakHashMap 的原理是什么？用在哪些场景？

**WeakHashMap vs 普通 HashMap：**

```java
// 普通 HashMap：即使没有外部引用，key 也不回收
Map<String, Integer> map = new HashMap<>();
String key = new String("hello");  // key 对象
map.put(key, 1);
key = null;  // key 没有强引用了
System.gc(); // 但 map 里还引用着，key 不会回收
System.out.println(map.size());  // 仍然是 1

// WeakHashMap：没有外部引用时，下一次 GC 回收
Map<String, Integer> weakMap = new WeakHashMap<>();
String weakKey = new String("world");
weakMap.put(weakKey, 2);
weakKey = null;  // 没有外部引用了
System.gc();     // 回收！
System.out.println(weakMap.size());  // 0
```

**WeakHashMap 内部原理：**

```java
// 核心：Entry 继承 WeakReference
private static class Entry<K,V> extends WeakReference<K> implements Map.Entry<K,V> {
    V value;
    Entry<K,V> next;
    
    Entry(K key, V value, ReferenceQueue<K> queue) {
        super(key, queue);  // key 被 WeakReference 持有
        this.value = value;
    }
}

// expungeStaleEntries()：GC 后清理已被回收的 key
private void expungeStaleEntries() {
    for (Object e; (e = queue.poll()) != null; ) {
        // 从 HashMap 中移除对应的 Entry
        removeEntry((Entry<K,V>) e);
    }
}
```

**典型使用场景：**

```java
// 场景1：缓存 Class 对象（类加载器的内存泄漏）
// 避免卸载后的 Class 还被 Map 持有
Map<Class<?>, ClassInfo> classCache = new WeakHashMap<>();

// 场景2：简单缓存（不需要长期保存）
Map<String, ExpensiveObject> cache = new WeakHashMap<>();
public ExpensiveObject get(String key) {
    return cache.computeIfAbsent(key, k -> createExpensive(k));
}

// 场景3：ThreadLocal 的 Map 就是用 WeakReference
// ThreadLocalMap.Entry 继承 WeakReference<ThreadLocal<?>>
// 这样 ThreadLocal 对象没有外部引用时，可以被回收
```

**面试加分话术：**

> "WeakHashMap 的关键设计是：**key 用弱引用持有，value 用强引用持有**。所以如果 key 没有外部引用了，GC 就会把 key 回收，Entry 也跟着清理。典型的坑是：如果 value 也持有 key 的引用，那就形成强引用链，key 回收不了了。最常见的应用是 ThreadLocal——ThreadLocalMap 用弱引用持有 ThreadLocal 对象，保证 ThreadLocal 用完即回收。"

## 🔥 扩展追问 3：ThreadLocal 为什么用弱引用？内存泄漏是怎么回事？

**ThreadLocal 的结构：**

```
Thread 对象
    ↓ 持有
ThreadLocalMap（类似于 HashMap）
    ↓ 数组元素是 Entry
Entry extends WeakReference<ThreadLocal<?>>
    ↓ 强引用
value（Object）
```

**为什么 ThreadLocal 用弱引用：**

```java
// 如果 ThreadLocal 用强引用持有 key：
ThreadLocal<User> tl = new ThreadLocal<>();
tl.set(user);
tl = null;  // tl 置空了

// 问题：user 还在！ThreadLocalMap 还持有强引用
// → 内存泄漏：ThreadLocal 对象和 user 对象都无法回收

// 用弱引用后：
// tl = null
// → 只有一个弱引用指向 ThreadLocal 对象
// → 下次 GC，ThreadLocal 对象被回收（只剩下虚引用了）
// → Entry.key = null
// → 但 value 还是强引用！
```

**真正的内存泄漏场景：**

```java
// 线程池复用场景：
public class ThreadPoolDemo {
    private static ThreadLocal<User> tl = new ThreadLocal<>();
    
    public void process(User user) {
        tl.set(user);
        // 业务处理...
        // 如果忘记 tl.remove()，就泄漏了！
    }
}

// 线程池复用线程 → Thread 一直活着 → ThreadLocalMap 一直活着
// → 如果不用 remove，value 对象（user）永远不会被回收
```

**正确的 ThreadLocal 使用方式：**

```java
public class SafeThreadLocalDemo {
    // 用 try-finally 确保 remove
    private static ThreadLocal<User> tl = new ThreadLocal<>();
    
    public void process(Long userId) {
        try {
            User user = userService.findById(userId);
            tl.set(user);
            
            doBusiness();  // 业务方法里都能通过 tl.get() 获取
            
        } finally {
            tl.remove();  // 必须清理！
        }
    }
    
    // Spring 等框架都是这么用的
}

// Spring MVC 中，RequestContextHolder 的 ThreadLocal
// 每次请求结束都会清理
```

**泄漏原因图解：**

```
线程存活时：
┌─────────────────────────────────────────────────┐
│ ThreadLocalMap                                 │
│ ┌─────────────┐                                │
│ │ Entry {     │                                │
│ │   key (弱) →│────→ ThreadLocal 对象           │
│ │   value (强)│←──── "user对象" 强引用          │
│ │ }           │                                │
│ └─────────────┘                                │
│                                                 │
│ 强引用链：Thread → ThreadLocalMap → Entry → value│
│           完全绑定了，除非 remove，否则泄漏       │
└─────────────────────────────────────────────────┘

如果用强引用持有 ThreadLocal：
┌─────────────────────────────────────────────────┐
│ ThreadLocal tl = new ThreadLocal<>();  ←─── 这个强引用断了
│ ThreadLocalMap → Entry → key → ???  // ThreadLocal 对象也没了？
└─────────────────────────────────────────────────┘

用弱引用：
┌─────────────────────────────────────────────────┐
│ ThreadLocal tl = null;  ←─ tl 断了              │
│ ThreadLocalMap → Entry → key (弱) ──→ GC 回收 ThreadLocal
│ ThreadLocalMap → Entry → value (强) ──→ user 还在！← 泄漏！
└─────────────────────────────────────────────────┘

正确做法：
try {
    tl.set(user);
    doSomething();
} finally {
    tl.remove();  // 断开 value 的引用
}
```

**面试加分话术：**

> "ThreadLocal 用弱引用是为了解决**ThreadLocal 对象本身泄漏**的问题——如果用强引用，`tl = null` 后 Map 还持有它，内存永远不释放。但弱引用只解决了 key 的问题，**value 仍然需要手动 remove**，否则在线程池场景下就会泄漏。我之前排查过一个生产问题：用户 session 数据莫名其妙变大，最后定位到是 Filter 里往 ThreadLocal 塞了数据但没清理，线程复用导致数据越积越多。所以现在团队规范是：**只要用 ThreadLocal，必须配 finally.remove()**，写代码时直接写好，不要依赖后来者记得清理。"

## 🔥 扩展追问 4：软引用在缓存中的应用？LRU 缓存怎么实现？

**软引用缓存原理：**

```java
// 简单的软引用缓存
public class SoftRefCache<K, V> {
    private final Map<K, SoftReference<V>> cache = new HashMap<>();
    
    public V get(K key) {
        SoftReference<V> ref = cache.get(key);
        if (ref == null) return null;
        
        V value = ref.get();  // 可能已经被 GC 回收
        if (value == null) {
            cache.remove(key);  // 清理空引用
        }
        return value;
    }
    
    public void put(K key, V value) {
        cache.put(key, new SoftReference<>(value));
    }
}
```

**高级 LRU 缓存实现（WeakReference + 双向链表）：**

```java
public class LRUCache<K, V> {
    private final int capacity;
    private final Map<K, Node<K, V>> map;
    private final Node<K, V> head, tail;
    
    public LRUCache(int capacity) {
        this.capacity = capacity;
        this.map = new HashMap<>();
        this.head = new Node<>();
        this.tail = new Node<>();
        head.next = tail;
        tail.prev = head;
    }
    
    public V get(K key) {
        Node<K, V> node = map.get(key);
        if (node == null) return null;
        
        // 移到链表头部（最近使用）
        moveToHead(node);
        return node.value;
    }
    
    public void put(K key, V value) {
        Node<K, V> node = map.get(key);
        if (node != null) {
            node.value = value;
            moveToHead(node);
        } else {
            Node<K, V> newNode = new Node<>(key, value);
            map.put(key, newNode);
            addToHead(newNode);
            
            if (map.size() > capacity) {
                Node<K, V> removed = removeTail();
                map.remove(removed.key);
            }
        }
    }
    
    // 内部类 Node...
}
```

**Guava Cache 帮我们做好的方案：**

```java
// 用 Guava Cache，不用自己写
LoadingCache<Key, Graph> cache = CacheBuilder.newBuilder()
    .maximumSize(1000)              // 超过 1000 条后 LRU 淘汰
    .weakKeys()                     // key 用弱引用
    .weakValues()                   // value 用弱引用
    .softValues()                  // value 用软引用（内存不足时淘汰）
    .expireAfterWrite(10, TimeUnit.MINUTES)  // 写后 10 分钟过期
    .recordStats()                  // 开启统计
    .build(new CacheLoader<Key, Graph>() {
        @Override
        public Graph load(Key key) {
            return createExpensiveGraph(key);
        }
    });

// 使用
Graph graph = cache.get(key);  // 自动加载、缓存、淘汰
```

**面试加分话术：**

> "软引用做缓存的核心思路是：**让 JVM 来决定什么时候回收**。内存充足时缓存命中率高，内存不足时自动淘汰，保护应用不会 OOM。但如果完全依赖软引用淘汰，GC 频率会变高，实际生产中建议结合 `maximumSize` 限制条数。另外 Spring 内核的 `CachingHelper`、MyBatis 的 `TransactionalCache` 都是用软引用思想实现的缓存——底层都是 `SoftReference` 配合引用队列做清理。"

## 🔥 扩展追问 5：虚引用的真正用途是什么？引用队列怎么用？

**虚引用的两个特点：**

```java
// 1. get() 永远返回 null
ReferenceQueue<byte[]> queue = new ReferenceQueue<>();
PhantomReference<byte[]> phantom = new PhantomReference<>(new byte[1024], queue);

System.out.println(phantom.get());  // 永远是 null！

// 2. 对象回收后，PhantomReference 会被放入队列
```

**引用队列的工作流程：**

```
对象 A 被 PhantomReference 引用
        ↓
GC 决定回收对象 A
        ↓
GC 将 PhantomReference<A> 放入 ReferenceQueue
        ↓
应用线程从 queue.poll() 取到通知
        ↓
执行清理逻辑（释放堆外内存、关闭连接等）
```

**典型场景 2：追踪对象回收（用于调试/监控）：**

```java
public class ObjectLifeTracker {
    private static final ReferenceQueue<Object> QUEUE = new ReferenceQueue<>();
    private static final Map<PhantomReference<?>, String> TRACKING_MAP = new ConcurrentHashMap<>();
    
    public static <T> PhantomReference<T> track(T obj, String name) {
        PhantomReference<T> ref = new PhantomReference<>(obj, QUEUE);
        TRACKING_MAP.put(ref, name);
        return ref;
    }
    
    // 启动监控线程
    public static void startMonitoring() {
        Thread monitor = new Thread(() -> {
            while (!Thread.interrupted()) {
                try {
                    Reference<?> ref = QUEUE.remove(1000); // 最多等 1 秒
                    if (ref != null) {
                        System.out.println(TRACKING_MAP.get(ref) + " 被回收了！");
                        TRACKING_MAP.remove(ref);
                    }
                } catch (InterruptedException e) {
                    break;
                }
            }
        });
        monitor.setDaemon(true);
        monitor.start();
    }
}
```

**面试加分话术：**

> "虚引用是四种引用里最"虚"的一个——`get()` 永远返回 null，单独用毫无意义。它的价值在于**引用队列**：当对象被回收时，JVM 把虚引用放进队列，你就能收到通知。这在**堆外内存**管理中特别重要——DirectByteBuffer 不受 GC 直接管理，靠虚引用 + Cleaner 机制实现自动释放。另外 NIO 的 ByteBuffer.allocateDirect() 底层就是这么干的。面试时可以加一句：虚引用不是用来缓存的，它是**资源回收通知机制**。"

## 🔥 扩展追问 6：WeakReference 在实际开发中的典型使用场景？

**场景 1：Class 对象的缓存（解决类加载器泄漏）：**

```java
// 场景：热部署、插件化框架
// 问题：ClassLoader 卸载不掉，因为 Class 对象被 Map 持有

public class ClassMetadataCache {
    // WeakHashMap 的 key 是弱引用，ClassLoader 没有外部引用时就回收
    private Map<ClassLoader, Map<String, Class<?>>> cache = new WeakHashMap<>();
    
    public Class<?> loadClass(ClassLoader loader, String name) {
        Map<String, Class<?>> classMap = cache.computeIfAbsent(loader, 
            k -> new HashMap<>());
        return classMap.computeIfAbsent(name, n -> {
            try {
                return Class.forName(n, true, loader);
            } catch (ClassNotFoundException e) {
                throw new RuntimeException(e);
            }
        });
    }
}
```

**场景 2：监听器容器（防止监听器对象泄漏）：**

```java
// 场景：GUI 编程、事件总线
// 问题：监听器注册后忘了 unregister，对象就泄漏了

public class WeakEventBus {
    // 用 WeakReference 持有监听器，不用手动 unregister
    private Set<WeakReference<EventListener>> listeners = ConcurrentHashMap.newKeySet();
    
    public void register(EventListener listener) {
        listeners.add(new WeakReference<>(listener));
    }
    
    public void post(Event event) {
        // GC 后自动清理死掉的监听器
        listeners.removeIf(ref -> {
            EventListener listener = ref.get();
            if (listener == null) return true;  // 已经被回收，移除
            listener.onEvent(event);
            return false;
        });
    }
}
```

**场景 3：缓存 Memoization（函数记忆化）：**

```java
// 场景：耗时的计算，缓存但不能无限缓存
public class Memoizer<K, V> {
    // 弱引用 Map：没有外部引用时自动清理
    private Map<K, WeakReference<V>> cache = new ConcurrentHashMap<>();
    
    public V compute(K key, Function<K, V> expensive) {
        WeakReference<V> ref = cache.get(key);
        V value = ref != null ? ref.get() : null;
        
        if (value != null) {
            return value;
        }
        
        value = expensive.apply(key);
        cache.put(key, new WeakReference<>(value));
        return value;
    }
}
```

**面试加分话术：**

> "WeakReference 最常见的三个场景：**WeakHashMap 做容器、监听器自动清理、Memoization 缓存**。实际开发中我用过 WeakEventBus——注册事件监听器不需要手动注销，监听器对象没被外部引用时自动从总线移除，很省心。另外 WeakReference 在**类元数据缓存**、**动态类加载器**场景也很常见，Spring 的 `CachingMetadataReaderFactory` 就用了 WeakReference 来避免 ClassLoader 泄漏。"

## 🔥 扩展追问 7：四种引用和 GC Root 的关系？可达性分析？

**GC Root 可达性分析：**

```
┌─────────────────────────────────────────────────────────┐
│  GC Root（gcRoots）：                                     │
│  ├─ 虚拟机栈中引用的对象（方法里的局部变量）               │
│  ├─ 方法区中类静态属性引用的对象                          │
│  ├─ 方法区中常量引用的对象（final 常量）                  │
│  └─ 本地方法栈中 JNI 引用的对象                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  GC Root → 强引用链 → 对象存活（不会被 GC）               │
│                                                         │
│  GC Root → 软引用 → 对象存活（内存不足时回收）            │
│                                                         │
│  GC Root → 弱引用 → 对象不存活（下次 GC 回收）            │
│                                                         │
│  无 GC Root 链 → 对象不可达 → 等待 GC 回收               │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**四种引用的可达性层次：**

```
强引用（StrongRef）
  └─ GC Root 直连，最强

软引用（SoftRef）
  └─ GC Root → SoftRef → Object
  └─ 中间强度，OOM 前回收

弱引用（WeakRef）
  └─ GC Root → WeakRef → Object
  └─ 弱强度，下一轮 GC 就清

虚引用（PhantomRef）
  └─ GC Root → PhantomRef → Object（但 get() 返回 null）
  └─ 虚设，仅用于回收通知
```

**面试加分话术：**

> "GC 回收对象时，先从 GC Root 出发做可达性分析。**强引用**只要 GC Root 能到达就存活；**软引用**在内存不足时第一次回收；**弱引用**不管内存够不够，下次 GC 一律回收；**虚引用**只用来追踪回收时机。需要注意的是：软引用和弱引用本身作为 GC Root 的一部分，它们持有的对象如果外部没有引用，**同样会被回收**——所以 `WeakReference.get() == null` 不一定是坏事，只是告诉你对象没了。"

## 🔥 扩展追问 8：总结对比和应用场景速记表？

**四者对比：**

| 引用类型 | 回收时机 | get() 返回 | 典型用途 |
|---|---|---|---|
| **强引用** | 从不回收 | 对象本身 | 普通变量 |
| **软引用** | 内存不足时 | 对象或 null | 图片缓存、LRU 缓存 |
| **弱引用** | 每次 GC | 对象或 null | ThreadLocal、WeakHashMap、监听器容器 |
| **虚引用** | 回收时放入队列 | 永远 null | 堆外内存释放、对象生命周期追踪 |

**一句话速记：**

```
强引用：除非手动=null，否则不死
软引用：内存不够了再死
弱引用：下次 GC 就死
虚引用：死了通知我一声
```

**常见面试追问：**

```java
// 追问 1：WeakReference 和 SoftReference 哪个更"弱"？
// 答：WeakReference 更弱。WeakReference 在每次 GC 都回收，SoftReference 只在内存不足时才回收。

// 追问 2：有了 WeakHashMap，还需要 WeakReference 吗？
// 答：WeakHashMap 是容器，WeakReference 是基础引用类型。
//     WeakHashMap 适合作为 value 不重要、key 需要自动清理的缓存；
//     WeakReference 更灵活，可以单独持有引用，不依赖 Map 结构。

// 追问 3：ThreadLocal 为什么用 WeakReference 而不是强引用？
// 答：如果 Thread 持有 ThreadLocalMap，强引用会让 ThreadLocal 对象无法被回收，
//     导致 ThreadLocal 本身泄漏（即使外部代码已经不用它了）。
```

**面试加分话术：**

> "四种引用本质上是在给 GC 回收"开绿灯"。强引用是红灯（永不回收），软引用是黄灯（内存不足才绿），弱引用是短绿（每次 GC 都亮），虚引用是闪光灯（只是通知你灯灭了）。在 JDK 源码里，WeakHashMap 用弱引用做 key、Spring 用 WeakReference 做类缓存、Netty 用虚引用 + Cleaner 释放堆外内存——理解这四种引用，源码看起来会更通透。"

## 一句话总结（面试收尾用）

> "Java 四种引用代表了对象存活的不同宽松程度：**强引用永不回收、软引用等内存不足、弱引用每次 GC 都回收、虚引用只是回收通知**。实际开发中记住三个黄金组合：`ThreadLocal + 弱引用`防止 key 泄漏、`WeakHashMap`做容器缓存、`SoftReference` + LRU做内存敏感缓存。如果面试问到 JVM 调优或者排查 OOM，说清楚这四种引用怎么配合，能体现出你对 GC 机制的深层理解。"
