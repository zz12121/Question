
Lambda 的关键点：
- 函数式接口：只有一个抽象方法的接口
- Lambda 实现原理：invokedynamic 指令 + 动态方法区（Lambda 元factory）
- 编译时生成 xxx$$Lambda$1 内部类
- 捕获机制：捕获变量必须是 effectively final
- 类型推断
## Q62 Java 8 的 Lambda 表达式是如何实现的？函数式接口是什么？

### 基础流程

```
Lambda 表达式写代码时
        ↓
编译阶段： invokedynamic 指令 + 生成 BootstrapMethods
        ↓
第一次执行时：LambdaMetafactory 创建动态代理对象
        ↓
生成 xxx$$Lambda$1/2/3... 内部类（不在源码中）
        ↓
调用时执行内部类的逻辑
```

**函数式接口定义：**

```java
// 标识注解（可选，但建议加）
@FunctionalInterface
public interface Function<T, R> {
    R apply(T t);  // 只能有一个抽象方法
    
    // default 方法可以有多个
    default Function<T, T> identity() {
        return t -> t;
    }
}
```

**Lambda 三种写法：**

```java
// 1. 表达式风格
List<Integer> list = Arrays.asList(1, 2, 3);
list.forEach(item -> System.out.println(item));

// 2. 方法引用（更简洁）
list.forEach(System.out::println);  // 等价于 item -> System.out.println(item)

// 3. 构造方法引用
Function<String, User> factory = User::new;  // 等价于 name -> new User(name)
```

## 🔥 扩展追问 1：Lambda 底层是怎么实现的？为什么不用匿名内部类？

**编译后的真相：**

```java
// 源代码
list.forEach(item -> System.out.println(item));

// 编译后生成的内部类（反编译看到的是这样的）
class ForEach$$Lambda$1 implements Consumer {
    public void accept(Object item) {
        System.out.println(item);  // 逻辑直接写在 accept 里
    }
}
```

**为什么不直接用匿名内部类？**

| 方面 | 匿名内部类 | Lambda |
|---|---|---|
| **文件数** | 每个表达式生成一个 .class | 多个 Lambda 共享一个 $$Lambda$N |
| **this 指向** | 指向匿名类实例 | 指向包围它的类（更符合直觉） |
| **编译产物** | 单独的文件 | 依附于调用方的类文件 |
| **变量捕获** | 必须是 final | 必须是 final 或 effectively final |

```java
public class Test {
    public void run() {
        String x = "hello";
        
        // 匿名内部类：this = 匿名内部类实例
        Runnable r1 = new Runnable() {
            public void run() {
                System.out.println(this);  // 输出 Test$1
            }
        };
        
        // Lambda：this = Test 实例
        Runnable r2 = () -> {
            System.out.println(this);  // 输出 Test
        };
    }
}
```

**面试加分话术：**

> "Lambda 底层靠 invokedynamic 指令实现，编译时生成的不是真正的匿名内部类，而是一个**动态生成的 lambda 表达式对象**。这样设计的好处是：延迟绑定 + 按需加载，只有第一次执行时才真正创建对象，性能更好。另外匿名内部类会生成独立 .class 文件，Lambda 不会，这减少了类加载开销。"

## 🔥 扩展追问 2：invokedynamic 是什么？和普通方法调用有什么区别？

**普通方法调用 vs invokedynamic：**

```
普通方法调用（invokevirtual）：
    编译器硬编码了方法签名
    运行时直接跳到方法地址
    编译时就决定了是谁的方法

invokedynamic：
    编译器只生成一个"调用点"
    运行时第一次才确定要调什么
    通过 BootstrapMethods 动态绑定
```

**字节码对比：**

```java
// 源代码
Runnable r = () -> System.out.println("hello");

// 普通编译（匿名内部类风格）
new #2 = new Runnable() {...}
invokevirtual #2.run()

// Lambda 编译（invokedynamic 风格）
invokedynamic #run() [BootstrapMethods...]
// 第一次执行时才生成 lambda 对象
```

**BootstrapMethods 的工作原理：**

```java
// 反编译看到的 BootstrapMethods 结构
0: invokestatic LambdaMetafactory.metafactory(
        // 方法类型描述
        "(Ljava/lang/Object;)V",
        // 实际调用的方法
        "accept(Ljava/lang/Object;)V",
        // 生成的内部类
        "ForEach$$Lambda$1")
```

**面试加分话术：**

> "invokedynamic 是 JDK 7 引入的指令，本来是为了给 JRuby、Groovy 这些 JVM 上的动态语言用的。Lambda 表达式借用了这个机制，实现了**延迟绑定**——编译时不知道具体要执行什么代码，运行时第一次调用时才通过 LambdaMetafactory 工厂创建真正的实现类。这也是为什么 Lambda 能做到按需加载、不会增加静态编译的复杂度。"

## 🔥 扩展追问 3：Lambda 捕获变量有什么限制？为什么？

**核心规则：只能捕获 final 或 effectively final 的变量**

```java
public void test() {
    int a = 1;        // effectively final
    // a++;            // 加上这行，下面的 Lambda 就报错了
    
    Runnable r = () -> {
        // 能捕获 a
        System.out.println(a);
        // 不能修改 a
        // a = 2;  // 编译报错
    };
}
```

**原因解释：**

```
Lambda 底层是单独的对象
        ↓
捕获的变量实际上是通过构造方法传进去的
        ↓
如果变量能被修改 → 多线程下不一致
        ↓
所以强制要求 final（内部类同理）
```

**被捕获的变量怎么传进去？**

```java
// 反编译看到的生成类
class Test$$Lambda$1 implements Runnable {
    private final int captured_a;  // final，不能改
    
    Test$$Lambda$1(int captured_a) {
        this.captured_a = captured_a;
    }
    
    public void run() {
        System.out.println(captured_a);  // 读取构造时传入的值
    }
}

// 调用处
Test$$Lambda$1 r = new Test$$Lambda$1(1);  // 把 a 的值复制进去
```

**面试加分话术：**

> "这个限制是出于线程安全的考虑。Lambda 捕获的变量实际上是**值拷贝**传进去的，不是引用。如果允许修改，那多线程下你改、他读，各自看到不同的值，逻辑就乱了。所以 JVM 强制要求 captured 变量必须是 final 或者 effectively final。如果业务上真的需要修改，可以用 AtomicInteger、数组封装，或者换成普通的局部变量操作。"

## 🔥 扩展追问 4：方法引用和 Lambda 有什么关系？有哪些类型？

**方法引用就是 Lambda 的简写形式：**

```java
// Lambda 写法
str -> str.toUpperCase()
str -> System.out.println(str)
(name) -> new User(name)

// 方法引用写法
String::toUpperCase              // 实例方法引用
System.out::println              // 静态方法引用
User::new                        // 构造方法引用
```

**四种方法引用：**

| 类型 | 语法 | 等价 Lambda | 示例 |
|---|---|---|---|
| **静态方法** | `类名::静态方法` | `x -> 类名.静态方法(x)` | `Integer::parseInt` |
| **实例方法（对象）** | `对象::实例方法` | `x -> 对象.实例方法(x)` | `System.out::println` |
| **实例方法（类型）** | `类名::实例方法` | `(x, y) -> x.实例方法(y)` | `String::equals` |
| **构造方法** | `类名::new` | `x -> new 类名(x)` | `User::new` |

**特别注意：第三种 `类名::实例方法`**

```java
// 这个最特殊，隐藏了一个参数
List<String> names = Arrays.asList("Tom", "Bob");

// String::compareTo 等价于 (a, b) -> a.compareTo(b)
names.sort(String::compareTo);  // 第一个参数是调用者，后面是方法参数

// 等价于
names.sort((a, b) -> a.compareTo(b));
```

**面试加分话术：**

> "方法引用是 Lambda 的语法糖，写起来更简洁。记不住的时候想想：**冒号前面是对象或类，后面是方法名**。需要注意的是 `String::equals` 这种形式，它会把第一个参数当调用者，所以 `sort(String::compareTo)` 实际等价于 `(a, b) -> a.compareTo(b)`。方法引用不是任何时候都好——当方法名不直观时，用 Lambda 写清楚逻辑反而更好。"

## 🔥 扩展追问 5：常用的函数式接口有哪些？怎么选？

**Java 8 内置的四大函数式接口：**

| 接口 | 方法签名 | 用途 | 示例 |
|---|---|---|---|
| **Consumer<T>** | `void accept(T t)` | 消费，无返回值 | `System.out::println` |
| **Supplier<T>** | `T get()` | 生产，无输入 | `User::new` |
| **Function<T,R>** | `R apply(T t)` | 转换，输入→输出 | `String::toUpperCase` |
| **Predicate<T>** | `boolean test(T t)` | 判断，返回 boolean | `String::isEmpty` |

**衍生接口（带基本类型优化）：**

```java
// 避免装箱/拆箱开销
IntFunction<R>     // 输入 int
IntToLongFunction // int → long
ToIntFunction<T>  // 返回 int
BooleanSupplier    // 无输入，返回 boolean
```

**实际选择建议：**

```java
// 转换：Function
Function<String, User> parser = User::new;

// 判断：Predicate
Predicate<String> notEmpty = s -> !s.isEmpty();
List<String> filtered = list.stream().filter(notEmpty).toList();

// 消费：Consumer
Consumer<String> printer = System.out::println;
printer.accept("hello");

// 生产：Supplier
Supplier<LocalDateTime> now = LocalDateTime::now;
LocalDateTime timestamp = now.get();
```

**面试加分话术：**

> "这四个接口我建议记住，因为 Stream 里的 `map`、`filter`、`forEach`、`reduce` 底层就是用的这些。最容易搞混的是 `Consumer` 和 `Function`——前者消费后扔掉，后者消费后返回新东西。另外如果涉及基本类型（int、long），尽量用专门的 `IntFunction`、`LongPredicate` 之类的，避免自动装箱，这是性能优化的细节。"

## 🔥 扩展追问 6：Lambda 在 Stream 中的作用是什么？Pipeline 是怎么工作的？

**Stream 三部曲：**

```
创建 → 中间操作（Lazy）→ 终端操作（触发执行）
        ↓
   filter/map/flatMap...
        ↓
   收集（forEach/collect/toList）
```

**Lazy 特性：**

```java
List<String> list = Arrays.asList("a", "b", "c");

Stream<String> stream = list.stream()
    .filter(s -> {                     // 不会立即执行
        System.out.println("filter: " + s);
        return true;
    })
    .map(s -> {                        // 也不会立即执行
        System.out.println("map: " + s);
        return s.toUpperCase();
    });

System.out.println("还没执行...");      // 只打印这句

stream.forEach(System.out::println);   // 终端操作！从这里才开始执行
```

**输出顺序（串行）：**

```
还没执行...
filter: a
map: a
a
filter: b
map: b
b
filter: c
map: c
c
```

**背后的原理：**

```java
// filter 返回的是一个 Lazy pipeline 对象
// 实际代码是链式构建了一个 Runnable 链
// 只有终端操作 forEach 才会真正触发执行

// 类似这样的结构（简化）：
class Pipeline {
    Runnable head;
    
    void forEach(Consumer action) {
        // 从后往前触发动态链接
        action.run();  // 最终执行
    }
}
```

**面试加分话术：**

> "Stream 的精髓在于**惰性求值**——中间操作不会真正执行，只有遇到终端操作时才会触发执行。这种设计使得 Stream 可以做**短路优化**，比如 `filter().findFirst()` 找到第一个就停了，不会全部遍历。另外 Pipeline 内部用的是责任链模式，每个操作包装成一个 stage，最后一个 stage 触发时，从后往前逐级 pull 数据，这和 Reactor 响应式编程的思路是一致的。"

## 🔥 扩展追问 7：@FunctionalInterface 注解是必须的吗？不标会怎样？

**验证：**

```java
// 标了 @FunctionalInterface
@FunctionalInterface
interface MyFunc {
    void run();
    default void log() {}  // default 不算抽象方法
}

// 编译通过

// 如果多加一个抽象方法
@FunctionalInterface
interface MyFunc2 {
    void run();
    void execute();  // 加了这个 → 编译报错！
}

// 报错：不是有效的函数式接口，因为有两个抽象方法
```

**不标的话：**

```java
// 完全可以不标，Lambda 照样能用
interface MyFunc {
    void run();
}

// 正常使用
Runnable r = () -> System.out.println("ok");
```

**那为什么要标？**

| 作用 | 说明 |
|---|---|
| **编译期检查** | 多加一个抽象方法立刻报错，早发现 |
| **文档作用** | 告诉其他开发者这个接口是设计给 Lambda 用的 |
| **团队协作** | IDE 会提示，可以被 Stream 识别 |

**面试加分话术：**

> "@FunctionalInterface 不是必须的，Lambda 能不能用只看接口里是不是只有一个抽象方法。但加上是好的实践——它是**声明式的契约**，相当于告诉团队：'这个接口是函数式的，请用 Lambda 的方式调用'。而且编译器会帮你把关，不小心多加一个抽象方法就报错，避免线上事故。另外 JDK 里很多注解如 `@Override`，本身也不是强制的，但加上就是最佳实践。"

## 🔥 扩展追问 8：Lambda 和 Stream 的性能怎么样？什么时候不该用？

**Lambda 的性能开销：**

```
调用一次 Lambda：
    invokedynamic → 查缓存 → 调用

如果循环里用 Lambda：
    循环 1000 次 → 每次查缓存（很快，但还是有点开销）
```

**什么时候用 Lambda：**

```java
// ✅ 推荐：一次性操作，代码简洁
list.stream()
    .filter(User::isActive)
    .map(User::getName)
    .collect(Collectors.toList());

// ✅ 推荐：可读性确实更好
map.forEach((k, v) -> System.out.println(k + ":" + v));
```

**什么时候不用 Lambda：**

```java
// ❌ 不推荐：性能敏感的热点循环
for (int i = 0; i < 10000000; i++) {
    list.add(process(i));  // 不用 stream，直接 for 循环
}

// ❌ 不推荐：调试困难
list.stream()
    .map(x -> {
        int result = complex(x);  // 断点打这里很难受
        return result;
    })
    .filter(x -> x > 0)
    .collect(toList());

// ❌ 不推荐：多线程场景下需要明确控制
ExecutorService es = Executors.newFixedThreadPool(4);
// 普通 for 循环更容易控制线程
```

**性能对比（大数据量）：**

```java
// 单核 CPU，小数据量（<1000）：Lambda 几乎没区别
// 多核 CPU，大数据量：parallelStream() 可以快
// 单核 CPU，大数据量：普通 for 循环更快

// parallelStream 的使用前提：
// 1. 数据量足够大（拆分的开销才值得）
// 2. 没有共享状态
// 3. 不是 IO 密集型（网络请求场景下多线程帮助不大）
```

**面试加分话术：**

> "Lambda 本身的性能损耗很小，主要开销是**调用链路多了几步**（invokedynamic → 查缓存 → invoke）。但这个开销在业务代码里通常可以忽略，除非是在高频交易系统里写算法。真正要避免的是：在热点循环里用 Lambda（可以用普通 for 循环替代）、调试困难的链式调用（拆成多行或者干脆不用 Stream）。另外强调一点——**并行 Stream**（parallelStream）不是银弹，数据量小、有共享状态、IO 密集的场景用并行反而更慢。"

---

## 一句话总结（面试收尾用）

> "Lambda 表达式底层靠 invokedynamic 实现，编译时生成的不是真正的匿名内部类，而是运行时动态创建的 lambda 对象，这样可以减少类加载开销。函数式接口只有一个抽象方法，配合 Lambda 可以写出简洁的函数式代码。日常开发中结合 Stream 使用效果更好，但要记住 Stream 不是万能的——热点循环、性能敏感场景还是老老实实用 for 循环。"
