## Q67 什么是死锁？死锁的四个必要条件？如何排查和避免死锁？

### 基础流程

```
死锁定义：
  两个或多个线程互相持有对方需要的资源
  等待对方释放，形成循环依赖
  所有人都在等待，谁都无法前进
        ↓
四个必要条件（Coffman）：
  ① 互斥条件：资源只能被一个线程持有
  ② 持有并等待：线程持有资源，同时等待其他资源
  ③ 不可抢占：资源不能被强制剥夺
  ④ 循环等待：线程之间形成循环等待链
        ↓
打破任意一个条件 → 死锁不再发生
```

**死锁示意图：**

```
线程 A                         线程 B
   │                              │
   ├─ 持有锁 X                    ├─ 持有锁 Y
   │                              │
   └─ 等待锁 Y  ←──────────────┐   └─ 等待锁 X  ←──┐
                               │                    │
                               └────────────────────┘
                          循环等待，无法打破
```

**最简单的死锁代码：**

```java
// 两个线程互相持有对方需要的锁
public class DeadLockDemo {
    private final Object lockA = new Object();
    private final Object lockB = new Object();
    
    public void method1() {
        synchronized (lockA) {
            System.out.println("线程A：持有锁A，等待锁B...");
            synchronized (lockB) {
                System.out.println("线程A：拿到锁B，执行逻辑");
            }
        }
    }
    
    public void method2() {
        synchronized (lockB) {
            System.out.println("线程B：持有锁B，等待锁A...");
            synchronized (lockA) {
                System.out.println("线程B：拿到锁A，执行逻辑");
            }
        }
    }
}

public static void main(String[] args) {
    DeadLockDemo demo = new DeadLockDemo();
    
    Thread t1 = new Thread(() -> demo.method1());
    Thread t2 = new Thread(() -> demo.method2());
    
    t1.start();
    t2.start();
    // 程序卡住，无法退出
}
```

## 🔥 扩展追问 1：死锁和活锁、饥饿的区别？

**三种"卡住"的对比：**

```
┌─────────────────────────────────────────────────────────┐
│ 死锁（Deadlock）                                         │
│ ├─ 状态：线程都在阻塞、什么都不做                         │
│ ├─ 原因：互相等待对方持有的锁                            │
│ └─ 需要外部干预才能打破                                  │
├─────────────────────────────────────────────────────────┤
│ 活锁（Livelock）                                         │
│ ├─ 状态：线程都在执行，但都在做无用功（不断重试）         │
│ ├─ 原因：检测到冲突后主动让步，但让步后又被对方抢走       │
│ └─ 程序在运行，但没有任何进展                            │
├─────────────────────────────────────────────────────────┤
│ 饥饿（Starvation）                                        │
│ ├─ 状态：某个线程始终得不到资源                          │
│ ├─ 原因：优先级太高 or 锁竞争不公平                      │
│ └─ 其他线程正常运行，只有"倒霉"的线程饿死                │
└─────────────────────────────────────────────────────────┘
```

**活锁示例（数据库场景）：**

```java
// 活锁：两个事务不断互相让步，永不提交
Transaction t1 = new Transaction();
Transaction t2 = new Transaction();

while (!t1.isCommitted()) {
    t1.begin();
    t1.lock(row1);        // 拿到 row1
    if (t2.hasLock(row2)) { // 检测到冲突
        t1.rollback();      // 主动回滚，让步
        sleep(1);
        continue;
    }
    t1.lock(row2);
    t1.commit();
}

// t1 回滚后，t2 又遇到同样问题，也回滚
// 两人不断让步，永不前进 = 活锁
```

**饥饿示例（线程优先级）：**

```java
// 高优先级线程一直抢占 CPU，低优先级线程饿死
Thread low = new Thread(() -> {
    while (!stopped) {
        System.out.println("低优先级：好久没机会执行了...");
    }
});
low.setPriority(Thread.MIN_PRIORITY);  // 优先级 1

Thread high = new Thread(() -> {
    while (true) {
        // 高优先级线程一直运行
    }
});
high.setPriority(Thread.MAX_PRIORITY); // 优先级 10

low.start();
high.start();
// low 几乎得不到执行机会 = 饥饿
```

**三者对比表：**

| 类型 | 线程状态 | 需要干预 | 例子 |
|---|---|---|---|
| **死锁** | 阻塞（Block） | ✅ 需要 | 互相等锁 |
| **活锁** | 运行（Runnable） | ✅ 可能需要 | 不断让步 |
| **饥饿** | 阻塞或低优先级运行 | ❌ 自行恢复 | 优先级太低 |

**面试加分话术：**

> "死锁、活锁、饥饿虽然都表现为'卡住'，但本质不同：**死锁是大家都不动**，**活锁是大家都在动但做无用功**，**饥饿是有人始终没机会动**。记住一个区别技巧：死锁的线程是 Blocked 状态，活锁的线程是 Runnable 状态（在 CPU 上跑），饥饿的线程可能是 Blocked 也可能是 Runnable 但没进展。看 `jstack` 输出时，BLOCKED 是死锁候选人，RUNNABLE 可能是活锁。"

## 🔥 扩展追问 3：如何避免死锁？四种破坏条件的策略？

**破坏死锁四个条件的策略：**

```
死锁四条件，打破任意一个即可：
        ↓
┌─────────────────────────────────────────────────────────┐
│ ① 互斥条件                                              │
│   ❌ 无法打破（锁本身就是互斥的）                        │
├─────────────────────────────────────────────────────────┤
│ ② 持有并等待 → 一次性获取所有需要的锁                    │
│   ✅ 策略：要么全拿到，要么一个都不拿                    │
├─────────────────────────────────────────────────────────┤
│ ③ 不可抢占 → 设置锁获取超时                             │
│   ✅ 策略：等不到就放弃，释放已持有的                    │
├─────────────────────────────────────────────────────────┤
│ ④ 循环等待 → 锁的获取顺序固定                           │
│   ✅ 策略：所有线程按相同顺序获取锁                      │
└─────────────────────────────────────────────────────────┘
```

**策略一：固定锁顺序（最常用）**

```java
// ❌ 容易死锁：两个线程按不同顺序拿锁
class BankTransfer {
    private final Object lockA = new Object();
    private final Object lockB = new Object();
    
    public void transferAtoB() {
        synchronized (lockA) {        // 先锁A
            synchronized (lockB) {      // 再锁B
                // 转账
            }
        }
    }
    
    public void transferBtoA() {
        synchronized (lockB) {        // 先锁B ← 顺序不同！
            synchronized (lockA) {    // 再锁A ← 可能死锁
                // 转账
            }
        }
    }
}

// ✅ 不会死锁：所有地方都用相同顺序
class SafeBankTransfer {
    private final Object lockA = new Object();
    private final Object lockB = new Object();
    
    // 统一按 ID 大小顺序获取锁
    public void transfer(Account from, Account to, double amount) {
        Account first = from.getId() < to.getId() ? from : to;
        Account second = from.getId() < to.getId() ? to : from;
        
        synchronized (getLock(first)) {      // 始终先锁 ID 小的
            synchronized (getLock(second)) { // 再锁 ID 大的
                // 转账逻辑
            }
        }
    }
    
    private Object getLock(Account a) {
        return a.getId() == 1 ? lockA : lockB;
    }
}
```

**策略二：加锁前检查（tryLock + 回退）**

```java
// 如果 tryLock 失败，释放已持有的锁，稍后重试
public boolean safeTransfer(Account a, Account b) {
    while (true) {
        if (a.lock.tryLock()) {
            try {
                if (b.lock.tryLock()) {
                    try {
                        // 执行转账
                        return doTransfer(a, b);
                    } finally {
                        b.lock.unlock();
                    }
                }
            } finally {
                a.lock.unlock();
            }
        }
        // 等一下再试，避免活锁
        Thread.sleep(10);
    }
}
```

**策略三：使用开放调用（Open Call）**

```java
// ❌ 在持有锁的情况下调用外部方法（危险）
class BadExample {
    private final Object lock = new Object();
    
    public void method() {
        synchronized (lock) {
            // ❌ 如果外部方法回调本类方法，可能死锁
            externalService.call(data);
        }
    }
}

// ✅ 把外部调用放在锁外面
class GoodExample {
    private final Object lock = new Object();
    
    public void method() {
        // 先在锁外准备好数据
        Data prepared = prepareData();
        
        synchronized (lock) {
            // 只在锁内做必须同步的操作
            updateState(prepared);
        }
        
        // 外部调用放在锁外
        externalService.call(prepared);
    }
}
```

**策略四：死锁检测（银行家算法思想）**

```java
// 定期检测锁依赖图，发现死锁就干预
public class DeadLockDetector {
    private final ScheduledExecutorService scheduler = 
        Executors.newScheduledThreadPool(1);
    
    public void start() {
        scheduler.scheduleAtFixedRate(() -> {
            Map<Thread, Set<Object>> threadLocks = getThreadLockSnapshot();
            // 检测环：如果线程 A 持有 X 等待 Y，线程 B 持有 Y 等待 X → 死锁
            
            if (detectCycle(threadLocks)) {
                System.out.println("检测到死锁！执行干预...");
                // 干预策略：让优先级低的线程放弃锁
                intervention();
            }
        }, 5, 5, TimeUnit.SECONDS);
    }
}
```

**面试加分话术：**

> "避免死锁有四种策略，但最实用的就两条：**固定顺序获取锁** 和 **用 tryLock 加超时**。固定顺序是最彻底的——只要所有代码都按同一个顺序拿锁，死锁的条件④（循环等待）就被打破了。tryLock 加超时是兜底——如果等不到锁，就放弃这次竞争，记录日志后走降级逻辑，比程序卡死强一百倍。我之前做过一个转账系统，就是用账户 ID 排序 + tryLock(2s) 做双重保障，上线两年没出过死锁问题。"

## 🔥 扩展追问 4：synchronized 死锁 vs ReentrantLock 死锁？有区别吗？

**synchronized 的死锁问题：**

```java
// synchronized 死锁：无法干预，只能等或重启
synchronized (A) {
    synchronized (B) {
        // 业务逻辑
    }
}
// 如果另一个线程反过来锁 B 再锁 A → 死锁

// 问题：synchronized 没有超时、没有中断
// 一旦进入死锁状态，只能：
// 1. 等其他线程释放（永远等不到）
// 2. kill -9 杀掉进程
```

**ReentrantLock 的死锁处理：**

```java
// ReentrantLock 可以超时 + 中断
public class SafeLockDemo {
    private final ReentrantLock lock = new ReentrantLock();
    
    public void tryWithTimeout() {
        try {
            // 尝试获取锁，最多等 3 秒
            boolean acquired = lock.tryLock(3, TimeUnit.SECONDS);
            
            if (acquired) {
                try {
                    // 业务逻辑
                } finally {
                    lock.unlock();
                }
            } else {
                System.out.println("等锁超时，执行降级逻辑");
                fallback();
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            System.out.println("等待被中断，优雅退出");
        }
    }
}
```

**两者的对比：**

| 特性 | synchronized | ReentrantLock |
|---|---|---|
| **超时等待** | ❌ 不支持 | ✅ `tryLock(time)` |
| **中断等待** | ❌ 不支持 | ✅ `lockInterruptibly()` |
| **公平锁** | ❌ 非公平 | ✅ 可选公平/非公平 |
| **多个条件** | ❌ 单一 wait/notify | ✅ 多个 Condition |
| **锁是否可重入** | ✅ | ✅ |
| **死锁恢复** | ❌ 无法恢复 | ✅ 可超时放弃 |

**公平锁 vs 非公平锁：**

```java
// 非公平锁（默认）：有可能插队，吞吐量大
ReentrantLock unfairLock = new ReentrantLock();

// 公平锁：按等待顺序，FIFO，可能吞吐量低
ReentrantLock fairLock = new ReentrantLock(true);

// 场景选择：
// - 高并发、无饥饿风险 → 非公平（默认）
// - 关键业务、不容许饿 → 公平锁
```

**面试加分话术：**

> "synchronized 和 ReentrantLock 都会产生死锁，但 ReentrantLock 多了**超时**和**中断**两个救生通道。synchronized 一旦卡死，只能等或者 kill，而 ReentrantLock 可以 `tryLock(timeout)` 主动放弃，给了业务代码处理的机会——记录日志、告警、或者回退到降级逻辑。代价是写法更繁琐，要 try-finally 解锁。生产环境里，对于可能死锁的关键路径（如跨账户转账），建议用 ReentrantLock 加超时；一般场景用 synchronized 足够，而且 JVM 也在不断优化 synchronized 的性能。"

## 🔥 扩展追问 5：数据库死锁是怎么发生的？如何避免？

**数据库死锁的典型场景：**

```sql
-- 场景1：两个事务互相等待行锁
-- 事务1
BEGIN;
UPDATE orders SET status = 'paid' WHERE id = 100;  -- 锁住 id=100
UPDATE orders SET status = 'paid' WHERE id = 200;  -- 等待 id=200

-- 事务2（同时执行）
BEGIN;
UPDATE orders SET status = 'paid' WHERE id = 200;  -- 锁住 id=200
UPDATE orders SET id = 100;  -- 等待 id=100 → 死锁！

-- 场景2：索引顺序不一致导致死锁
UPDATE orders SET status = 'paid' WHERE user_id = 1 AND id > 100;  -- 锁 id=101,102...
UPDATE orders SET status = 'paid' WHERE user_id = 1 AND id > 200;  -- 锁 id=201,202...
-- 如果两个范围有重叠，可能死锁
```

**MySQL 死锁日志查看：**

```sql
-- 查看最近死锁
SHOW ENGINE INNODB STATUS;

-- 输出中的关键信息：
-- TRANSACTION A：等待 lock
-- TRANSACTION B：持有锁
-- HOLDS THE LOCK
-- WAITING FOR THIS LOCK TO BE GRANTED
```

**数据库死锁的避免策略：**

```java
// 策略1：按固定顺序访问资源
@Modifying
@Transactional
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    // 按 ID 大小排序，先锁小的
    Long firstId = fromId < toId ? fromId : toId;
    Long secondId = fromId < toId ? toId : fromId;
    
    accountMapper.updateBalance(firstId, -amount);  // 先锁 ID 小的
    accountMapper.updateBalance(secondId, amount);  // 再锁 ID 大的
}

// 策略2：减小锁范围（只锁必要的行）
// ❌ 大范围锁
UPDATE orders SET status = 'paid' WHERE status = 'pending';
// 锁住了所有 pending 行

// ✅ 小范围锁：批量少锁
List<Order> orders = orderMapper.selectPending(100);
for (Order order : orders) {
    orderMapper.updateStatus(order.getId(), "paid");  // 每次只锁一行
}

// 策略3：加超时 + 重试
@Retryable(maxAttempts = 3, value = DataAccessException.class)
public void batchUpdate() {
    // MyBatis-Plus 的乐观锁、或者手动 retry
}

// 策略4：用乐观锁替代悲观锁
// 乐观锁：用版本号 CAS，不阻塞
@Version
private Integer version;

UPDATE orders 
SET status = 'paid', version = version + 1 
WHERE id = ? AND version = ?;  -- 版本不对就更新 0 行，抛异常
```

**面试加分话术：**

> "数据库死锁和 Java 死锁本质一样——都是**循环等待资源**。MySQL InnoDB 的行锁是逐行加的，如果两个事务锁的顺序相反，就会死锁。避免方法也很像 Java 的固定顺序策略：**按主键/索引的固定顺序更新**。另外要区分**悲观锁（SELECT FOR UPDATE）**和**乐观锁（版本号）**——悲观锁锁得狠但容易死锁，乐观锁不阻塞但需要重试。高并发写入场景优先用乐观锁（如库存扣减），跨表批量操作才用悲观锁但要按顺序。"

## 🔥 扩展追问 6：什么是线程池导致的死锁？如何避免？

**线程池内的死锁（Thread Starvation Deadlock）：**

```java
// 经典场景：线程池任务互相等待
ExecutorService executor = Executors.newFixedThreadPool(2);

// 任务 A：需要等任务 B 的结果
Future<String> taskA = executor.submit(() -> {
    Future<String> taskB = executor.submit(() -> {
        return "B的结果";  // 任务 B
    });
    return "A依赖B: " + taskB.get();  // A 等 B
});

// 任务 B：需要等任务 A 的结果（假设还有空闲线程）
Future<String> taskB = executor.submit(() -> {
    return "B的结果";  // 但如果线程池只有 2 个线程
    // 且都被 A 和另一个 B 占用了...
});
```

**更隐蔽的线程池死锁：**

```java
// 使用 ForkJoinPool 也可能死锁
ForkJoinPool pool = new ForkJoinPool(2);

pool.invoke(new RecursiveTask<Integer>() {
    @Override
    protected Integer compute() {
        if (需要等待子任务) {
            // ForkJoinPool 默认是工作窃取，理论上不会死锁
            // 但如果任务有外部依赖...
            return 子任务.join();
        }
        return result;
    }
});
```

**解决线程池死锁的方法：**

```java
// 方法1：扩大线程池（不根本解决）
ExecutorService executor = Executors.newFixedThreadPool(10);  // 线程数 > 任务依赖深度

// 方法2：用 submit 而不是 submit + get 嵌套
// ❌ 嵌套 get 可能死锁
executor.submit(() -> {
    Future<String> f = executor.submit(() -> compute());
    return f.get();  // 嵌套调用
});

// ✅ 用 CompletableFuture 链
CompletableFuture.supplyAsync(this::step1, executor)
    .thenCompose(result -> 
        CompletableFuture.supplyAsync(() -> step2(result), executor))
    .thenCompose(result -> 
        CompletableFuture.supplyAsync(() -> step3(result), executor));

// 方法3：确保线程池足够大
// 线程池大小公式：
// N = CPU 核心数
// W = 等待时间 / 计算时间
// 线程数 = N × (1 + W)
// 如果任务都是 IO 密集型（等待时间长），线程数可以设大一些

// 方法4：用无依赖的任务链
// A → B → C，提交成一个链，不嵌套
CompletableFuture.runAsync(() -> step1(), executor)
    .thenRun(() -> step2())  // 不用 executor
    .thenRun(() -> step3());
```

**线程池配置建议：**

```java
// CPU 密集型：线程数 = CPU 核心数 + 1
int cpuCores = Runtime.getRuntime().availableProcessors();
ExecutorService cpuBound = new ThreadPoolExecutor(
    cpuCores + 1, cpuCores + 1,
    0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<>()
);

// IO 密集型：线程数 = CPU 核心数 × 2 ~ CPU 核心数 × (1 + 等待时间/计算时间)
ExecutorService ioBound = new ThreadPoolExecutor(
    cpuCores * 2, cpuCores * 4,
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000)
);
```

**面试加分话术：**

> "线程池死锁是个容易被忽视的问题。核心原因是一个任务提交到线程池后，又在任务里调用另一个任务并等待结果——如果线程池线程已满，两个任务互相等待，就死锁了。解决方法有两个：**扩大线程池**（最简单，但浪费资源）和**不嵌套提交**（用 CompletableFuture 链式调用替代 submit + get）。面试时可以加一个经验：线程池大小和任务类型有关，CPU 密集型线程数等于核心数，IO 密集型可以设到核心数的 2~4 倍。"

## 🔥 扩展追问 7：哲学家就餐问题？死锁模型的经典案例？

**问题描述：**

```
5 个哲学家围坐一桌，5 根筷子，每人左右各一根
        ↓
思考 → 饿了 → 拿筷子 → 吃饭 → 放筷子 → 思考
        ↓
问题：如果每个人都先拿左边的筷子，再拿右边的
      等右边的筷子（左邻的左筷子），
      每个人都拿左边、都在等右边 → 死锁！
```

**代码实现（死锁版）：**

```java
class Philosopher implements Runnable {
    private final int id;
    private final Object leftFork;
    private final Object rightFork;
    
    public Philosopher(int id, Object leftFork, Object rightFork) {
        this.id = id;
        this.leftFork = leftFork;
        this.rightFork = rightFork;
    }
    
    public void run() {
        while (true) {
            think();
            synchronized (leftFork) {      // 先拿左边
                System.out.println(id + " 拿起左边筷子");
                synchronized (rightFork) {  // 再拿右边 → 可能死锁
                    eat();
                }
                // 放下右边
            }
            // 放下左边
        }
    }
}

// 启动（死锁版本）
public class DiningPhilosophersDeadlock {
    public static void main(String[] args) {
        Object[] forks = new Object[5];
        for (int i = 0; i < 5; i++) forks[i] = new Object();
        
        for (int i = 0; i < 5; i++) {
            // ❌ 每个人先左后右，顺序相同 → 死锁
            new Thread(new Philosopher(i, forks[i], forks[(i+1)%5])).start();
        }
    }
}
```

**解决方案一：改变最后一个人的取筷顺序**

```java
class SafePhilosopher implements Runnable {
    private final int id;
    private final Object leftFork;
    private final Object rightFork;
    
    public SafePhilosopher(int id, Object leftFork, Object rightFork) {
        this.id = id;
        this.leftFork = leftFork;
        this.rightFork = rightFork;
    }
    
    public void run() {
        while (true) {
            think();
            
            // 核心：最后一个人先右后左，打破循环等待
            if (id == 4) {
                synchronized (rightFork) {
                    synchronized (leftFork) {
                        eat();
                    }
                }
            } else {
                synchronized (leftFork) {
                    synchronized (rightFork) {
                        eat();
                    }
                }
            }
        }
    }
}
```

**解决方案二：限制同时拿筷子的人数（服务员检查）**

```java
class WaiterPhilosopher implements Runnable {
    private static final Semaphore table = new Semaphore(4); // 最多4人同时拿
    
    public void run() {
        while (true) {
            think();
            
            try {
                table.acquire();  // 服务员许可：保证不超过4人同时拿
                
                leftFork.acquire();
                rightFork.acquire();
                
                eat();
                
                leftFork.release();
                rightFork.release();
                
            } finally {
                table.release();
            }
        }
    }
}
```

**解决方案三：分奇偶号处理**

```java
class OddEvenPhilosopher implements Runnable {
    public void run() {
        while (true) {
            think();
            
            if (id % 2 == 0) {
                // 偶数哲学家：先左后右
                left.acquire();
                right.acquire();
            } else {
                // 奇数哲学家：先右后左
                right.acquire();
                left.acquire();
            }
            
            eat();
            
            left.release();
            right.release();
        }
    }
}
```

**面试加分话术：**

> "哲学家就餐问题是分布式系统里死锁模型的经典原型——它完美展示了死锁的四个必要条件：**互斥**（筷子独占）、**持有并等待**（拿着左边等右边）、**不可抢占**（不能抢别人的筷子）、**循环等待**（A等B、B等C...回到A）。面试时讲这个问题，能体现出你对死锁理论的理解深度。实际工程中的解决方法本质都一样：**打破循环等待**（奇偶顺序不同）或**限制竞争数量**（信号量控制并发）。"

```

│  死锁代码审查 Checklist                                    │
├─────────────────────────────────────────────────────────────┤
│ 1. 多把锁是否按固定顺序获取？                               │

└─────────────────────────────────────────────────────────────┘
```

**团队死锁规范示例：**

```java
// 规范1：统一锁顺序工具类
public class LockOrder {
    // 按对象 ID 排序，返回 [先锁, 后锁]
    public static Object[] order(Object a, Object b) {
        return System.identityHashCode(a) < System.identityHashCode(b)
            ? new Object[]{a, b}
            : new Object[]{b, a};
    }
}

// 使用
public void transfer(Account a, Account b) {
    Object[] locks = LockOrder.order(a, b);
    synchronized (locks[0]) {
        synchronized (locks[1]) {
            // 业务逻辑
        }
    }
}

// 规范2：强制使用 tryLock 超时
public class SafeLockTemplate {
    public static void executeWithLock(Lock lock, Runnable task, long timeoutMs) {
        try {
            boolean acquired = lock.tryLock(timeoutMs, TimeUnit.MILLISECONDS);
            if (!acquired) {
                throw new LockTimeoutException("获取锁超时: " + lock);
            }
            try {
                task.run();
            } finally {
                lock.unlock();
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("等待锁被中断", e);
        }
    }
}

// 规范3：禁止在持锁状态下调用外部方法
// 用 AOP/注解切面检查
@MustNotHoldLock
public void callExternalService() {  // 如果有 @MustNotHoldLock 注解会报错
    externalClient.invoke();
}
```

**死锁监控告警：**

```java
// 监控线程等待锁的时间，超过阈值告警
public class LockMonitor {
    private static final Map<Thread, LockInfo> LOCKING_THREADS = new ConcurrentHashMap<>();
    
    public static void onLockAcquired(Thread thread, Object lock, long startTime) {
        LOCKING_THREADS.put(thread, new LockInfo(lock, startTime));
    }
    
    public static void onLockReleased(Thread thread) {
        LockInfo info = LOCKING_THREADS.remove(thread);
        if (info != null && info.getDuration() > 5000) {
            // 超过 5 秒没释放，发告警
            AlertUtil.send("线程持有锁超过5秒，可能死锁: " + thread.getName());
        }
    }
}
```

## 一句话总结（面试收尾用）

> "死锁的四个必要条件是**互斥、持有并等待、不可抢占、循环等待**，打破任意一个就不会死锁。工程实践中**固定锁顺序**是最彻底的方案，**tryLock 加超时**是最实用的兜底。排查用 `jstack`，数据库死锁看 `SHOW ENGINE INNODB STATUS`。面试时能结合哲学家就餐问题或者实际踩过的坑讲的，面试官会高看你一眼——说明你不只懂理论，还真正处理过生产环境的死锁问题。"
