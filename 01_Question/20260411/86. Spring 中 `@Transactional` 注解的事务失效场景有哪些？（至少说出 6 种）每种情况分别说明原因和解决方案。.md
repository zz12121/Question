让我列出常见的事务失效场景：
1. 方法内部调用（this调用）- 不走代理
2. 非public方法 - Spring AOP限制
3. 异常被catch吞掉 - 不会回滚
4. 异常类型不匹配 - 默认只回滚RuntimeException
5. rollbackFor配置错误
6. 同一个类内部调用 - 不走代理
7. 多线程调用 - 事务上下文不传递
8. 数据库不支持事务（如MyISAM）
9. 传播行为配置错误 - PROPAGATION_SUPPORTS等
10. 事务超时配置导致提前回滚
11. 使用了不正确的事务管理器
12. 方法被final/static修饰 - 无法被代理
### 基础流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                    @Transactional 事务原理                          │

│                                                                     │

│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │

│  │  public void transfer(String from, String to, int amount)  │   │
│  │      │                                                      │   │
│  │      ▼                                                      │   │
│  │  Spring AOP 代理拦截                                        │   │
│  │      │                                                      │   │
│  │      ▼                                                      │   │

│  │      │                                                      │   │
│  │      ▼                                                      │   │
│  │  无异常 → commit                                            │   │
│  │  有异常 → rollback                                          │   │
│  │                                                             │   │
│  └─────────────────────────────────────────────────────────────┘   │

│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │  ❌ this.method()  → 不走代理，事务不生效                  │   │

│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

```java
// 正确的事务配置
@Service
public class TransferService {
    
    @Autowired
    private AccountMapper accountMapper;
    
    // ✅ 正确：public 方法 + 异常正确抛出
    @Transactional(rollbackFor = Exception.class)
    public void transfer(String from, String to, int amount) {
        accountMapper.decrease(from, amount);
        // 模拟异常
        if (amount > 10000) {
            throw new RuntimeException("单笔转账超限");
        }
        accountMapper.increase(to, amount);
    }
    
    // ✅ 正确：调用同类方法走代理
    @Autowired
    private TransferService self;  // 注入自身代理
    
    public void outerTransfer() {
        self.transfer("A", "B", 100);  // 通过代理调用
    }
    
    // ✅ 正确：捕获后重新抛出
    @Transactional(rollbackFor = Exception.class)
    public void transferWithRetry(String from, String to, int amount) {
        try {
            doTransfer(from, to, amount);
        } catch (Exception e) {
            // 处理后重新抛出，让事务生效
            throw new RuntimeException("转账失败", e);
        }
    }
}
```

## 🔥 扩展追问 1：方法内部调用（this调用）导致事务失效

**原因：绕过代理对象，直接调用目标方法**

```
┌─────────────────────────────────────────────────────────────────┐
│                    this 调用 vs 代理调用                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【this 调用（失效）】                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @Service                                                │   │
│  │  public class OrderService {                            │   │
│  │                                                         │   │
│  │      @Transactional                                     │   │
│  │      public void createOrder() {                        │   │
│  │          insertOrder();   // ❌ this.insertOrder()      │   │
│  │          updateStock();   // ← 事务在这里失效           │   │
│  │      }                                                  │   │
│  │                                                         │   │
│  │      @Transactional                                     │   │
│  │      public void updateStock() { }                     │   │
│  │  }                                                      │   │
│  │                                                         │   │
│  │  原因：this 是目标对象，不是 Spring 代理对象             │   │
│  │       绕过了 AOP，事务注解不生效                        │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  【代理调用（生效）】                                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @Service                                                │   │
│  │  public class OrderService {                            │   │
│  │                                                         │   │
│  │      @Autowired                                         │   │
│  │      private OrderService self;  // 注入自身代理        │   │
│  │                                                         │   │
│  │      public void createOrder() {                        │   │
│  │          self.updateStock();  // ✅ 通过代理调用        │   │
│  │      }                                                  │   │
│  │                                                         │   │
│  │      @Transactional                                     │   │
│  │      public void updateStock() { }                     │   │
│  │  }                                                      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> A: 方法内部调用（`this.xxx()`）会绕过 Spring 代理对象，直接调用目标方法，AOP 拦截不到，所以事务不生效。解决方案有三种：
> 第一，注入自身代理（`@Autowired private XxxService self;`），通过 `self.method()` 调用；
> 第二，使用 `ApplicationContext` 获取代理对象；
> 第三，重构代码，把需要事务的方法单独放到另一个 Service，通过注入调用。

## 🔥 扩展追问 2：异常被 catch 吞掉导致事务失效

**原因：异常没有传播到代理层，无法触发回滚**

```
┌─────────────────────────────────────────────────────────────────┐
│                    异常被吞没的情况                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【catch 吞掉异常（失效）】                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @Transactional                                        │   │
│  │  public void transfer() {                               │   │
│  │      try {                                             │   │
│  │          // 业务操作                                    │   │
│  │          doSomething();                                 │   │
│  │      } catch (Exception e) {                           │   │
│  │          // ❌ 异常被吞掉                               │   │
│  │          log.error("失败", e);                         │   │
│  │      }                                                 │   │
│  │      // ← 代理认为方法正常执行完成，提交事务            │   │
│  │  }                                                     │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  【正确做法：捕获后重新抛出】                                     │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @Transactional(rollbackFor = Exception.class)          │   │
│  │  public void transfer() {                               │   │
│  │      try {                                             │   │
│  │          doSomething();                                 │   │
│  │      } catch (Exception e) {                           │   │
│  │          // ✅ 重新抛出，让代理感知到异常               │   │
│  │          throw new BusinessException("业务失败", e);    │   │
│  │      }                                                 │   │
│  │  }                                                     │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  【不推荐：编程式事务】                                           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @Autowired                                           │   │
│  │  private PlatformTransactionManager txManager;          │   │
│  │                                                         │   │
│  │  public void transfer() {                              │   │
│  │      TransactionStatus status = txManager.getTransaction(null); │
│  │      try {                                             │   │
│  │          doSomething();                                 │   │
│  │          txManager.commit(status);                      │   │
│  │      } catch (Exception e) {                           │   │
│  │          txManager.rollback(status);                    │   │
│  │      }                                                 │   │
│  │  }                                                     │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> A: 事务基于异常判断是否回滚，如果 catch 吞掉异常，代理层感知不到异常，就认为方法正常执行完成，直接提交事务。解决方案是 catch 后重新抛出异常，或者使用 `TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()` 标记回滚。实际项目中，如果确实需要捕获异常做处理但又要回滚，推荐用自定义异常 + `rollbackFor` 配置。

## 🔥 扩展追问 3：非 RuntimeException 默认不回滚

**原因：Spring 默认只对运行时异常回滚**

```
┌─────────────────────────────────────────────────────────────────┐
│                    rollbackFor 配置决定回滚范围                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Spring 默认回滚规则：                                           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  RuntimeException → ✅ 回滚                            │   │
│  │  Error → ✅ 回滚                                        │   │
│  │  受检异常（IOException/SQLException）→ ❌ 不回滚      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  【抛出受检异常（失效）】                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @Transactional  // 默认只回滚 RuntimeException        │   │
│  │  public void transfer() throws IOException {           │   │
│  │      // ...                                            │   │
│  │      throw new IOException("网络异常");  // ❌ 不会回滚│   │
│  │  }                                                     │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  【正确配置】                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @Transactional(rollbackFor = Exception.class)          │   │
│  │  // ✅ 继承自 Exception，包括 IOException              │   │
│  │  public void transfer() throws IOException { }         │   │
│  │                                                         │   │
│  │  @Transactional(rollbackFor = {IOException.class, ...})│   │
│  │  // ✅ 指定多个异常类型                                 │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> A: Spring 默认只对 `RuntimeException` 和 `Error` 回滚，如果抛出受检异常（如 `IOException`、`SQLException`）默认不会回滚。解决方案是显式配置 `rollbackFor = Exception.class`，或者指定具体异常类型。实际项目建议统一配置 `rollbackFor = Exception.class`，避免漏配。

## 🔥 扩展追问 4：非 public 方法事务不生效

**原因：Spring AOP 基于 JDK 动态代理，public 方法才能被代理**

```
┌─────────────────────────────────────────────────────────────────┐
│                    Spring AOP 限制                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Spring 默认使用 JDK 动态代理：                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  代理类实现目标接口                                      │   │
│  │                                                         │   │
│  │  public class $Proxy28 implements IOrderService {       │   │
│  │      private IOrderService target;                      │   │
│  │                                                         │   │
│  │      public void createOrder() {  // ✅ public 能代理  │   │
│  │          MethodBeforeAdvice...                          │   │
│  │          target.createOrder();                         │   │
│  │      }                                                  │   │
│  │                                                         │   │
│  │      void createPrivate() {  // ❌ private 无法代理     │   │
│  │          target.createPrivate();                        │   │
│  │      }                                                  │   │
│  │  }                                                      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  【失效场景】                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @Transactional                                        │   │
│  │  private void transfer() {  // ❌ private 不生效      │   │
│  │      // ...                                             │   │
│  │  }                                                     │   │
│  │                                                         │   │
│  │  @Transactional                                        │   │
│  │  protected void transfer() {  // ❌ protected 也不生效 │   │
│  │      // ...                                             │   │
│  │  }                                                     │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  【解决方案】                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. 改为 public 方法                                    │   │
│  │                                                         │   │
│  │  2. 开启 CGLIB 代理（可代理非 public）                  │   │
│  │     spring.aop.proxy-target-class=true                  │   │
│  │                                                         │   │
│  │  3. 但 CGLIB 仍然不支持 private 方法                   │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> A: Spring AOP 默认基于 JDK 动态代理，只能代理接口方法，且必须是 public 方法。如果是 private/protected 方法，代理无法增强，事务不生效。解决方案：方法改为 public，或者开启 `spring.aop.proxy-target-class=true` 强制使用 CGLIB 代理。但即使是 CGLIB 也不支持 private 方法，所以 private 方法不能加 `@Transactional`。

## 🔥 扩展追问 5：多线程调用导致事务失效

**原因：事务上下文绑定在当前线程，不传递给子线程**

```
┌─────────────────────────────────────────────────────────────────┐
│                    线程隔离导致事务不传递                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  线程隔离原理：                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  ThreadLocal<Map<DataSource, Connection>>              │   │
│  │                                                         │   │
│  │  事务信息存在 ThreadLocal 中                            │   │
│  │  每个线程有独立的 ThreadLocal 副本                       │   │
│  │  子线程看不到父线程的事务上下文                          │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  【多线程调用（失效）】                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @Transactional                                        │   │
│  │  public void transfer() {                               │   │
│  │      accountMapper.decrease(from, amount);              │   │
│  │                                                         │   │
│  │      new Thread(() -> {                                │   │
│  │          accountMapper.increase(to, amount);  // ❌    │   │
│  │          // 这个操作不在父事务中                        │   │
│  │      }).start();                                        │   │
│  │                                                         │   │
│  │      // 主线程事务提交                                  │   │
│  │      // 子线程操作可能独立提交或回滚                     │   │
│  │  }                                                     │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  【正确做法】                                                    │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  1. 不用多线程，所有操作在主线程                        │   │
│  │                                                         │   │
│  │  2. 使用异步框架 + 事务同步器                           │   │
│  │     @Transactional                                     │   │
│  │     public void transfer() {                            │   │
│  │         accountMapper.decrease(from, amount);          │   │
│  │         TransactionSynchronizationManager               │   │
│  │             .registerSynchronization(new TransactionSync(() -> {  │
│  │                 asyncService.doAsync(to, amount);      │   │
│  │             }));                                        │   │
│  │     }                                                   │   │
│  │                                                         │   │
│  │  3. 使用 @Async + @TransactionalInherited              │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> A: Spring 事务上下文存在 `ThreadLocal` 中，每个线程有独立的副本，子线程看不到父线程的事务。解决方案有三种：第一，不使用多线程，所有操作串行执行；第二，使用 `TransactionSynchronizationManager` 在事务提交后异步执行；第三，使用线程池方案需要特别注意事务边界。实际项目中，如果必须用多线程，建议把需要事务的操作放到一个线程内执行，异步操作单独起事务。

## 🔥 扩展追问 6：传播行为配置错误导致事务失效

**原因：不当的传播行为导致不开事务或不开新事务**

```
┌─────────────────────────────────────────────────────────────────┐
│                    事务传播行为详解                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  常见传播行为：                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  REQUIRED（默认）：加入当前事务，没有则创建新事务       │   │
│  │  REQUIRES_NEW：每次都创建新事务，挂起当前事务         │   │
│  │  SUPPORTS：支持当前事务，没有则以非事务执行            │   │
│  │  NOT_SUPPORTED：以非事务执行，挂起当前事务             │   │
│  │  MANDATORY：必须在事务中执行，否则抛异常               │   │
│  │  NEVER：以非事务执行，有事务则抛异常                   │   │
│  │  NESTED：嵌套事务（Savepoint）                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  【SUPPORTS 失效场景】                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @Transactional(propagation = Propagation.SUPPORTS)     │   │
│  │  public void transfer() {                               │   │
│  │      // 如果外层没有事务，这里以非事务执行             │   │
│  │      accountMapper.decrease(from, amount);              │   │
│  │      accountMapper.increase(to, amount);                │   │
│  │  }                                                      │   │
│  │                                                         │   │
│  │  // 调用方没有 @Transactional                           │   │
│  │  public void outer() {                                  │   │
│  │      transfer();  // ❌ SUPPORTS 不开事务              │   │
│  │  }                                                      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  【REQUIRES_NEW 场景】                                           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  @Transactional(propagation = Propagation.REQUIRES_NEW) │   │
│  │  public void log() {  // 独立事务，不受主事务回滚影响  │   │
│  │      // 记录日志，即使业务回滚日志也保留               │   │
│  │  }                                                      │   │
│  │                                                         │   │
│  │  @Transactional                                        │   │
│  │  public void transfer() {                               │   │
│  │      accountMapper.decrease(from, amount);              │   │
│  │      logService.log("转账");  // REQUIRES_NEW          │   │
│  │      throw new RuntimeException("失败");               │   │
│  │      // 事务回滚，但 logService 的事务已提交            │   │
│  │  }                                                      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> A: `Propagation.SUPPORTS` 表示"支持当前事务，没有则以非事务执行"，如果调用方没有事务，这里就不会开事务。`Propagation.REQUIRES_NEW` 每次都创建新事务，可以隔离部分操作不受主事务回滚影响。常见错误是内部方法用 `SUPPORTS`，但调用方没有事务，导致事务失效。实际项目中，如果方法需要事务保障，推荐用默认的 `REQUIRED`，确保一定在事务中执行。

## 🔥 扩展追问 7：数据库不支持事务（如 MyISAM 引擎）

**原因：数据库层面的限制**

```sql
-- 查看表引擎
SHOW CREATE TABLE account;

-- Account 表使用 MyISAM（不支持事务）
CREATE TABLE `account` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `balance` decimal(10,2) DEFAULT NULL,
  PRIMARY KEY (`id`)
) ENGINE=MyISAM DEFAULT CHARSET=utf8;
```

> A: Spring 事务依赖数据库事务支持，如果表使用 MyISAM 引擎（不支持事务），即使代码加了 `@Transactional`，也不会有事务效果。解决方案是改为 InnoDB 引擎。另外，分布式场景下如果跨多个数据库，Spring 事务也无法处理，需要用分布式事务方案（如 Seata）。

## 一句话总结（面试收尾用）

> **@Transactional 失效场景主要有：方法内部 this 调用绕过代理、异常被 catch 吞掉导致代理感知不到、非 RuntimeException 默认不回滚、非 public 方法 Spring AOP 无法代理、多线程调用事务上下文在 ThreadLocal 中不传递、传播行为 SUPPORTS 在无外层事务时不开事务、MySQL MyISAM 引擎不支持事务。解决方案包括注入自身代理调用、捕获异常后重新抛出、配置 rollbackFor、避免 private 方法、事务操作不放多线程、用 REQUIRED 传播行为、切换 InnoDB 引擎。**

**面试加分话术：**

> "我之前遇到过 @Transactional 失效导致的数据不一致问题。当时业务是转账操作，扣款成功但收款失败，按理说应该回滚，但钱扣了没加回来。排查发现是项目里统一配置了 `rollbackFor = Exception.class`，但有个同事在 service 里单独配了 `@Transactional` 没配 rollbackFor，他代码里抛的是 `SQLException`（受检异常），所以没有回滚。还有一次是用多线程处理批量转账，用 `CompletableFuture.supplyAsync` 异步执行，每个子线程的操作都没在事务里，有几条成功了有几条失败了。后来改成同步执行或者用 `TransactionSynchronizationManager` 在主事务提交后异步执行才解决。还有个坑是 `this` 调用不走代理，我们在项目里统一要求内部调用必须注入自身代理，通过 `self.method()` 调用，避免了这个陷阱。"
