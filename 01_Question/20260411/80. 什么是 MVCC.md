## 🔥 扩展追问 1：undo log 版本链是如何组织的？

**undo log 的读写流程：**

```
┌─────────────────────────────────────────────────────────────────┐
│                     undo log 版本链读取流程                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  当前数据行（最新版本）：                                        │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  id=1, amount=200, trx_id=20, roll_ptr=0x88             │  │
│  │                                                         │  │
│  │  读取时：                                               │  │
│  │  1. 比较 trx_id=20 和 ReadView                          │  │
│  │  2. 如果不可见，循 roll_ptr 找 undo log                │  │
│  │                                                         │  │
│  └───────────────────────────────────────────────────────────┘  │
│                         │                                       │
│                         │ roll_ptr = 0x88                      │
│                         ▼                                       │
│  undo log (版本2):                                           │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  type=UPDATE, old_value={amount=100}, roll_ptr=0x66    │  │
│  │                                                         │  │
│  │  读取时：                                               │  │
│  │  1. 比较 trx_id=10 和 ReadView                          │  │
│  │  2. 如果可见，返回 old_value                            │  │
│  │                                                         │  │
│  └───────────────────────────────────────────────────────────┘  │
│                         │                                       │
│                         │ roll_ptr = 0x66                      │
│                         ▼                                       │
│  undo log (版本1，最早):                                       │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  type=INSERT, pk=1                                     │  │
│  │  roll_ptr=NULL（链表尾部）                             │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**undo log 的类型和清理时机：**

| undo log 类型 | 包含内容 | 清理时机 |
|--------------|---------|---------|
| **INSERT undo** | 主键值（用于回滚插入） | 事务提交后立即删除 |
| **UPDATE undo** | 旧值（用于 MVCC + 回滚） | 无活跃事务引用时 purge |

```sql
-- INSERT undo 示例
INSERT INTO orders VALUES (1, 'A001', 100);
-- undo log: {type: INSERT, id=1}
-- 事务提交后立即删除

-- UPDATE undo 示例
UPDATE orders SET amount = 200 WHERE id = 1;
-- undo log: {type: UPDATE, old_amount=100}
-- 需要保留到没有事务需要读取旧版本

-- DELETE 的特殊处理
DELETE FROM orders WHERE id = 1;
-- 实际上：UPDATE orders SET deleted=1（标记删除）
-- undo log 保留记录，支持回滚
```

**版本链的 GC（purge）：**

```sql
-- purge 线程清理不再需要的 undo log
-- 条件：无活跃事务的 ReadView 引用该 undo log

SHOW ENGINE INNODB STATUS;
-- History list length: 50  ← undo log 数量，过多说明 purge 跟不上
-- 原因：长事务 + 大量更新
```

## 🔥 扩展追问 2：ReadView 的可见性判断规则？

**ReadView 核心四字段：**

```java
class ReadView {
    // 活跃事务 ID 列表（未提交的事务）
    private final List<Long>    m_ids;
    
    // ReadView 创建时的最小活跃事务 ID
    private final long          min_trx_id;
    
    // ReadView 创建时的最大事务 ID（已分配的最大值+1）
    private final long          max_trx_id;
    
    // 当前事务 ID
    private final long          creator_trx_id;
}
```

**可见性判断算法：**

```
可见性判断 = 事务 ID 对比

规则 1️⃣：trx_id == creator_trx_id
┌────────────────────────────────────────────────────────────┐
│                                                            │
│  当前事务自己修改的 ──▶ 可见 ✓                             │
│                                                            │
│  SELECT * FROM orders WHERE id = 1;                       │
│  UPDATE orders SET amount = 300 WHERE id = 1;             │
│  SELECT * FROM orders WHERE id = 1;                       │
│  ──▶ 读到 amount=300（自己改的，当然可见）                │
│                                                            │
└────────────────────────────────────────────────────────────┘

规则 2️⃣：trx_id < min_trx_id
┌────────────────────────────────────────────────────────────┐
│                                                            │
│  旧事务提交的 ──▶ 可见 ✓                                   │
│                                                            │
│  ReadView 创建时 min_trx_id=15                            │
│  trx_id=10 ──▶ 在 ReadView 创建前已提交 ──▶ 可见         │
│                                                            │
└────────────────────────────────────────────────────────────┘

规则 3️⃣：trx_id >= max_trx_id
┌────────────────────────────────────────────────────────────┐
│                                                            │
│  未来事务的 ──▶ 不可见（事务号按顺序分配）                 │
│                                                            │
│  ReadView 创建时 max_trx_id=20                           │
│  trx_id=25 ──▶ 在 ReadView 创建后才开始 ──▶ 不可见       │
│                                                            │
│  这是 MVCC 的基础：只读"历史快照"                         │
│                                                            │
└────────────────────────────────────────────────────────────┘

规则 4️⃣：trx_id 在 m_ids 中
┌────────────────────────────────────────────────────────────┐
│                                                            │
│  活跃事务（未提交）的 ──▶ 不可见 ✗                        │
│                                                            │
│  ReadView.m_ids = [20, 25]                               │
│  trx_id=20 ──▶ 在 m_ids 中 ──▶ 不可见                    │
│  trx_id=25 ──▶ 在 m_ids 中 ──▶ 不可见                    │
│                                                            │
│  这是 RR 的关键：看不到其他未提交事务的修改                │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**ReadView 创建时机（隔离级别决定）：**

| 隔离级别 | ReadView 何时创建 | 说明 |
|----------|-----------------|------|
| **RU** | 每个 SELECT 都创建新 ReadView | 每次都读到最新 |
| **RC** | 每个 SELECT 都创建新 ReadView | 每次都读到最新（可能幻读） |
| **RR** | 事务第一次 SELECT 时创建 | 事务内始终用同一个快照 |
| **SERIALIZABLE** | 不需要 MVCC | 锁定所有读取的行 |

## 🔥 扩展追问 3：RC 和 RR 级别下 MVCC 的行为差异？

**RC 级别：每次 SELECT 都生成新 ReadView**

```
时间线：

T1: 事务 A（ID=10）开始
    │
T2: 事务 B（ID=20）开始
    │
T3: 事务 A 读取 ──▶ ReadView_A₁
    │                  m_ids=[20], min=20, max=21
    │                  读到 amount=100
    │
T4: 事务 B UPDATE amount=200 ──▶ 已提交
    │
T5: 事务 A 再次读取 ──▶ ReadView_A₂（每次都创建新 ReadView）
    │                  m_ids=[], min=21, max=21（事务B已提交）
    │                  读到 amount=200 ❗（两次读取结果不同）
    │
T6: 事务 A 读取 amount=200
    │                  与 T3 不同：不可重复读
    │
T7: 事务 A 提交
```

**RR 级别：事务第一次 SELECT 时创建 ReadView**

```
时间线：

T1: 事务 A（ID=10）开始
    │
T2: 事务 B（ID=20）开始
    │
T3: 事务 A 第一次读取 ──▶ ReadView_A（整个事务复用）
    │                        m_ids=[20], min=20, max=21
    │                        读到 amount=100
    │
T4: 事务 B UPDATE amount=200 ──▶ 已提交
    │
T5: 事务 A 再次读取 ──▶ ReadView_A（复用！不复建！）
    │                        m_ids=[20], min=20, max=21
    │                        读到 amount=100 ✓（事务B不可见）
    │
T6: 事务 A 提交
```

**对比表：**

| 对比项 | RC | RR |
|--------|-----|-----|
| ReadView 创建频率 | 每次 SELECT | 事务第一次 SELECT |
| 快照时间 | 每次 SELECT 时 | 事务开始时 |
| 同一事务内多次读取 | 可能不同 | 始终相同 ✓ |
| 幻读 | 可能发生 | 可能发生* |
| 并发度 | 更高（更少阻塞） | 较低 |

```sql
-- RC 演示
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
START TRANSACTION;
SELECT * FROM orders WHERE user_id = 100;  -- ReadView₁
-- 事务 B 提交了新插入
SELECT * FROM orders WHERE user_id = 100;  -- ReadView₂（重新创建）
-- 可能看到新插入的行 ❗

-- RR 演示
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
START TRANSACTION;
SELECT * FROM orders WHERE user_id = 100;  -- ReadView₁（固定快照）
-- 事务 B 提交了新插入
SELECT * FROM orders WHERE user_id = 100;  -- 复用 ReadView₁
-- 看不到新插入的行 ✓（如果使用覆盖索引）
-- 但：如果有其他事务插入了满足条件的行，下次 SELECT 可能看到（Next-Key Lock 导致）
```

## 🔥 扩展追问 4：为什么 RR 能解决不可重复读？

**不可重复读 = 同一事务内，两次读取同一行，结果不同**

```
不可重复读场景（RC 下）：

事务 A                          事务 B
  │                               │
  │  SELECT amount FROM orders   │
  │  WHERE id = 1                │
  │──────────────────────────────▶ amount=100
  │                               │
  │                               │ UPDATE amount=200
  │                               │ COMMIT
  │                               │
  │  SELECT amount FROM orders   │
  │  WHERE id = 1                │
  │──────────────────────────────▶ amount=200 ❌（不可重复读）
  │                               │
  │  COMMIT                       │
```

```
RR 解决不可重复读：

事务 A                          事务 B
  │                               │
  │  START TRANSACTION          │
  │                               │
  │  SELECT amount FROM orders   │  ← 创建 ReadView
  │  WHERE id = 1               │    m_ids=[事务B的ID]
  │──────────────────────────────▶ amount=100
  │                               │
  │                               │ UPDATE amount=200
  │                               │ COMMIT
  │                               │
  │  SELECT amount FROM orders   │
  │  WHERE id = 1               │  ← 复用同一个 ReadView
  │──────────────────────────────▶ amount=100 ✓（不可重复读解决了）
  │                               │
  │  COMMIT                       │
```

**核心原因：ReadView 复用**

```sql
-- MySQL 内部实现
-- RR 级别：每次执行 SELECT，使用相同的 ReadView

// 伪代码
if (isolation_level == RR) {
    if (trx->read_view == NULL) {
        // 第一次 SELECT，创建 ReadView
        trx->read_view = create_read_view();
    }
    // 后续 SELECT 复用同一个 ReadView
    use_read_view(trx->read_view);
}

if (isolation_level == RC) {
    // 每次 SELECT 都创建新 ReadView
    read_view = create_read_view();
    use_read_view(read_view);
}
```

## 🔥 扩展追问 5：幻读是什么？MVCC 能完全解决吗？

**幻读 = 两次读取，结果集行数不同（新增了行）**

```
幻读场景：

事务 A                          事务 B
  │                               │
  │  SELECT * FROM orders       │
  │  WHERE user_id = 100        │
  │──────────────────────────────▶ 返回 10 行
  │                               │
  │                               │ INSERT INTO orders VALUES(...)
  │                               │   (user_id=100, 新插入)
  │                               │ COMMIT
  │                               │
  │  SELECT * FROM orders       │
  │  WHERE user_id = 100        │
  │──────────────────────────────▶ 返回 11 行 ❗（幻读）
```

**MVCC 解决幻读的局限性：**

| 场景 | MVCC 能解决？ | 解决方式 |
|------|--------------|---------|
| 读已提交但未写入的行 | ✅ | ReadView 过滤未提交 |
| 读其他事务已提交的插入 | ✅ RC/✅ RR | ReadView 过滤 |
| **读+写（当前读）** | ❌ | 需要锁 |
| **范围查询 + 插入** | ❌ | 需要 Next-Key Lock |

**MVCC 的边界：**

```sql
-- MVCC 能解决：快照读（SELECT 不带锁）
SELECT * FROM orders WHERE id = 1;  -- 快照读，用 MVCC

-- MVCC 不能解决：当前读（加锁读）
SELECT * FROM orders WHERE id = 1 FOR UPDATE;  -- 当前读，用锁

-- MVCC 不能完全解决幻读的场景：
-- 事务 A                    事务 B
--   │                         │
--   │ SELECT * FROM orders    │
--   │ WHERE amount > 100     │
--   │ FOR UPDATE              │ ← 当前读，加锁
--   │────────────────────────▶│
--   │                         │ INSERT INTO orders(amount=150)
--   │                         │ ← 被 Next-Key Lock 阻塞 ✓
```

**Next-Key Lock（RR 级别的增强）：**

```sql
-- RR 级别：快照读 + Next-Key Lock
-- 既利用 MVCC，又防止幻读

SELECT * FROM orders WHERE id BETWEEN 1 AND 10 FOR UPDATE;
-- 锁定区间 [1, 10] + 区间内可能插入的位置
-- 其他事务无法插入 id 在 [1,10] 范围内的记录
```

## 🔥 扩展追问 6：MVCC + 锁的完整并发控制模型？

**快照读 vs 当前读：**

```
┌─────────────────────────────────────────────────────────────────┐
│                     读写并发控制模型                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  MVCC（乐观）              Locking（悲观）                       │
│  ┌────────────────┐        ┌────────────────┐                  │
│  │                │        │                │                  │
│  │  快照读         │        │  当前读         │                  │
│  │  SELECT（无锁） │        │  SELECT FOR    │                  │
│  │                │        │  UPDATE        │                  │
│  │  利用 ReadView │        │  INSERT        │                  │
│  │  读取历史版本   │        │  DELETE        │                  │
│  │                │        │                │                  │
│  │  不加锁         │        │  加锁          │                  │
│  │  不阻塞写       │        │  可能阻塞其他写 │                  │
│  │                │        │                │                  │
│  └────────────────┘        └────────────────┘                  │
│                                                                 │
│  组合使用：                                                      │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │                                                           │ │
│  │  读：MVCC（不加锁）                                        │ │
│  │  写：先读（MVCC）再写（加锁）                              │ │
│  │                                                           │ │
│  │  SELECT amount FROM orders WHERE id=1;  -- MVCC 快照读   │ │
│  │  (基于 ReadView 读到旧值)                                  │ │
│  │                                                           │ │
│  │  UPDATE orders SET amount=200 WHERE id=1;  -- 当前读+写  │ │
│  │  (加排他锁，读取最新值再更新)                              │ │
│  │                                                           │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**MVCC 与锁的配合：**

```sql
-- 读：MVCC
SELECT * FROM orders WHERE id = 1;  -- 不阻塞其他事务的写入

-- 写：锁
UPDATE orders SET amount = 200 WHERE id = 1;  -- 加 X 锁

-- RC + MVCC：每次读都创建新快照，可能读到最新数据
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
START TRANSACTION;
SELECT amount FROM orders WHERE id = 1;  -- ReadView₁
-- 此时其他事务提交了 amount=200
SELECT amount FROM orders WHERE id = 1;  -- ReadView₂
-- 读到 amount=200

-- RR + MVCC：事务内快照固定，不会读到其他事务的提交
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
START TRANSACTION;
SELECT amount FROM orders WHERE id = 1;  -- ReadView（固定）
-- 此时其他事务提交了 amount=200
SELECT amount FROM orders WHERE id = 1;  -- 复用 ReadView
-- 还是读到 amount=100
```

## 🔥 扩展追问 7：ReadView 与 purge 的协作？

**undo log 清理时机 = 无活跃事务需要它**

```
┌─────────────────────────────────────────────────────────────────┐
│                     purge 决策流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  purge 线程需要判断：                                             │
│  "这个 undo log 是否还有事务需要读取？"                          │
│                                                                 │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                                                           │  │
│  │  undo log 记录：                                          │  │
│  │  {trx_id=10, old_value=amount=100}                       │  │
│  │                                                           │  │
│  │  判断条件：                                                │  │
│  │                                                           │  │
│  │  ┌─────────────────────────────────────────────────────┐ │  │
│  │  │  当前所有活跃事务的 ReadView:                        │ │  │
│  │  │                                                     │ │  │
│  │  │  事务 15: ReadView₁(trx_id=[20])                  │ │  │
│  │  │  事务 18: ReadView₂(trx_id=[20])                  │ │  │
│  │  │  事务 22: ReadView₃(trx_id=[])  ← 事务22已提交   │ │  │
│  │  │                                                     │ │  │
│  │  │  所有 ReadView 的 max_trx_id = 23                  │ │  │
│  │  │  undo log.trx_id = 10 < 23 ──▶ 理论上可以 purge   │ │  │
│  │  │                                                     │ │  │
│  │  │  但是：ReadView₁/₂ 的 m_ids=[20]                  │ │  │
│  │  │  trx_id=10 不在 [20] 中 ──▶ 可以 purge ✓          │ │  │
│  │  │                                                     │ │  │
│  │  └─────────────────────────────────────────────────────┘ │  │
│  │                                                           │  │
│  │  结论：这个 undo log 可以被 purge                        │  │
│  │                                                           │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**长事务的危害：**

```sql
-- 长事务会阻止 undo log 清理

-- 事务 A：开始后不提交
START TRANSACTION;
SELECT * FROM orders;  -- 创建 ReadView，m_ids=[]
-- ❌ 不提交，ReadView 一直存在

-- 事务 B/C/D/...：大量更新
START TRANSACTION;
UPDATE orders SET amount = 200 WHERE id = 1;
COMMIT;
-- 每条 UPDATE 产生 undo log

-- 问题：
-- 1. undo log 无法清理（ReadView 引用旧 undo）
-- 2. undo 表空间膨胀
-- 3. History list length 持续增长

SHOW ENGINE INNODB STATUS;
-- History list length: 50000  ← 异常偏高
```

## 🔥 扩展追问 8：MVCC 在实际开发中的应用？

**应用一：乐观锁**

```java
// 利用 MVCC 的版本号特性
public boolean updateAmountOptimistic(Long id, BigDecimal newAmount) {
    // 读取当前版本
    Order order = jdbcTemplate.queryForObject(
        "SELECT * FROM orders WHERE id = ?", id);
    
    // 更新时检查版本
    int rows = jdbcTemplate.update(
        "UPDATE orders SET amount = ?, version = version + 1 " +
        "WHERE id = ? AND version = ?",
        newAmount, id, order.getVersion());
    
    return rows == 1;
}
```

**应用二：读多写少场景**

```java
// 利用 MVCC 减少锁竞争
@Service
public class OrderService {
    
    @Transactional(isolation = Isolation.REPEATABLE_READ)
    public OrderVO getOrder(Long id) {
        // 快照读，不加锁
        // 高并发下，多个请求可以同时读取
        // 只有写才会加锁
        Order order = orderMapper.selectById(id);
        return convertToVO(order);
    }
    
    @Transactional
    public void updateOrder(Long id, OrderUpdateDTO dto) {
        // 当前读，加排他锁
        // 保证更新的准确性
        Order order = orderMapper.selectForUpdate(id);
        // ... 更新逻辑
    }
}
```

**应用三：防止脏读（但应用层需要处理）**

```java
// RC 级别 + 应用层重试
@Transactional(isolation = Isolation.READ_COMMITTED)
public void processOrder(Long orderId) {
    Order order = orderMapper.selectById(orderId);
    if (order.getStatus() != OrderStatus.PENDING) {
        throw new BusinessException("订单状态不对");
    }
    // 问题：读和写之间，其他事务可能已修改状态
    // 解决方案：UPDATE 时检查状态
    int rows = orderMapper.updateStatus(orderId, 
        OrderStatus.PENDING, OrderStatus.PROCESSING);
    if (rows == 0) {
        throw new BusinessException("订单状态已被修改，请重试");
    }
}
```

## 一句话总结（面试收尾用）

> **MVCC 通过为每行数据保存多个版本（undo log 版本链）+ 每次查询生成 ReadView（快照）实现，ReadView 的可见性规则是：自己改的可见、旧快照（trx_id < min_trx_id）可见、未提交事务的修改不可见、未来的事务不可见；RR 级别在事务第一次 SELECT 时创建 ReadView 并复用，所以同一事务内多次 SELECT 结果相同，解决不可重复读，而 RC 每次 SELECT 都创建新 ReadView，可能读到其他事务新提交的修改。**

---

**面试加分话术：**
> "我之前遇到过一个问题：线上 RR 隔离级别，但报表查询有时候会多出几行数据，排查发现是查询用到了覆盖索引，而另一个事务在事务内插入了满足条件的新数据，但因为是快照读所以没被锁住（因为没 FOR UPDATE），这是 RR 下的幻读场景。后来在报表查询里加了一把写锁或者改用 RC 隔离级别解决了。我的经验是：MVCC 能解决大部分读写冲突，但写完再读的当前读场景一定要加锁，否则容易出现先读后写导致的更新丢失问题，这个在分布式系统里也叫 write skew。"
