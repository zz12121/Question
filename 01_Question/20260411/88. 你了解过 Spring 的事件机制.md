### 基础流程
1. Spring事件三剑客：ApplicationEventPublisher、ApplicationEvent、ApplicationListener
2. 同步机制：默认是同步的，监听器在发布线程执行
3. 异步机制：通过@Async或ApplicationEventMulticaster配置实现异步
4. 执行顺序：通过@Order注解、Ordered接口或@Priority控制
5. TransactionalEventListener：事务提交后/回滚后执行


6. 还需要考虑多个监听器同时处理事件时的协调问题，以及如何在事务边界处确保事件处理的正确性。事务监听器的使用场景和配置方式也很关键，需要展示具体的代码示例和执行流程图。

## 2. 核心代码示例

```java
// ========== 事件定义 ==========
// 方式一：普通事件（PayloadApplicationEvent 通用包装）
applicationEventPublisher.publishEvent("消息内容");
applicationEventPublisher.publishEvent(new OrderCreatedEvent(this, orderId));

// 方式二：自定义事件类
@Getter
public class OrderCreatedEvent extends ApplicationEvent {
    private final String orderId;
    private final BigDecimal amount;
    
    public OrderCreatedEvent(Object source, String orderId, BigDecimal amount) {
        super(source);
        this.orderId = orderId;
        this.amount = amount;
    }
}

// ========== 监听器实现 ==========

// 方式一：接口方式（同步）
@Component
public class OrderEventListener implements ApplicationListener<OrderCreatedEvent> {
    @Override
    public void onApplicationEvent(OrderCreatedEvent event) {
        // 同步执行，在发布线程中运行
        log.info("收到订单创建事件: orderId={}", event.getOrderId());
    }
}

// 方式二：注解方式（同步）
@Component
public class OrderAnnotationListener {
    @EventListener
    public void handleOrderEvent(OrderCreatedEvent event) {
        log.info("注解方式监听: {}", event.getOrderId());
    }
}

// 方式三：异步监听（独立线程池）
@Component
public class AsyncEventListener {
    @Async  // 需要 @EnableAsync
    @EventListener
    public void handleAsync(OrderCreatedEvent event) {
        log.info("异步处理: {}", event.getOrderId());
    }
}

// 方式四：带条件的监听
@Component
public class ConditionalListener {
    @EventListener(condition = "#event.amount > 1000")
    public void handleHighValue(OrderCreatedEvent event) {
        log.info("大额订单: {}", event.getAmount());
    }
}

// ========== 事务监听器 ==========
@Component
public class TransactionalListener {
    
    // 事务提交成功后执行
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void afterCommit(OrderCreatedEvent event) {
        log.info("事务已提交，发送通知");
        // 发短信、发MQ消息等
    }
    
    // 事务回滚后执行
    @TransactionalEventListener(phase = TransactionPhase.AFTER_ROLLBACK)
    public void afterRollback(OrderCreatedEvent event) {
        log.info("事务已回滚，记录日志");
    }
    
    // 无论成功失败都执行（最终一致）
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMPLETION)
    public void afterCompletion(OrderCreatedEvent event) {
        log.info("事务结束（成功或回滚）");
    }
}
```

## 3. 执行顺序控制

```java
// 三种方式控制监听器执行顺序

// 方式一：@Order 注解（数字越小越先执行）
@Component
@Order(1)  // 最先
public class FirstListener {
    @EventListener
    public void handle(OrderCreatedEvent event) { }
}

@Component
@Order(2)  // 其次
public class SecondListener {
    @EventListener
    public void handle(OrderCreatedEvent event) { }
}

// 方式二：实现 Ordered 接口
@Component
public class OrderedListener implements Ordered {
    @Override
    public int getOrder() {
        return 0;  // 返回值越小优先级越高
    }
    
    @EventListener
    public void handle(OrderCreatedEvent event) { }
}

// 方式三：@Priority 注解（用于同一个类有多个监听器方法）
@EventListener
@Priority(1)
public void handleFirst(OrderCreatedEvent event) { }

@EventListener
@Priority(2)
public void handleSecond(OrderCreatedEvent event) { }
```

## 4. 异步配置

```java
// ========== 全局异步配置 ==========
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {
    
    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(10);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("async-event-");
        executor.initialize();
        return executor;
    }
}

// ========== 自定义事件广播器（控制异步行为） ==========
@Bean
public ApplicationEventMulticaster applicationEventMulticaster() {
    SimpleApplicationEventMulticaster multicaster = new SimpleApplicationEventMulticaster();
    multicaster.setTaskExecutor(new SimpleAsyncTaskExecutor());  // 每个事件新线程
    return multicaster;
}
```

## 🔥 扩展追问 1：同步 vs 异步怎么选？

> **A:** 看场景。同步适合需要事务一致性的场景，比如下单成功后同步扣库存，失败一起回滚。异步适合耗时操作，比如发短信、发MQ消息、记录日志这些不影响主流程的。

**对比表格：**

| 特性 | 同步模式 | 异步模式 |
|------|---------|---------|
| 执行线程 | 发布线程 | 线程池 |
| 事务一致性 | ✅ 天然支持 | ❌ 需要额外处理 |
| 响应延迟 | 累积所有监听器时间 | 立即返回 |
| 失败处理 | 事务内可回滚 | 需手动补偿 |
| 适用场景 | 数据一致性要求高 | 非核心流程解耦 |

## 🔥 扩展追问 2：@Async 和 @EventListener 混用的坑？

> **A:** 有个大坑！如果在同一个监听器方法上同时加 @Async 和 @TransactionalEventListener，@TransactionalEventListener 的事务检测会失效，因为方法已经在另一个线程执行了，ThreadLocal 事务上下文传不过去。

**正确做法：**

```java
// ❌ 错误：@Async 会导致事务上下文丢失
@Async
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void handle(OrderCreatedEvent event) {
    // 这里拿不到原事务的 Connection
}

// ✅ 正确：拆成两个监听器
@EventListener
public void handleSync(OrderCreatedEvent event) {
    // 同步阶段：此时在事务内，可以做数据库操作
}

@Async
public void handleAsync(OrderCreatedEvent event) {
    // 异步阶段：通过 MQ 等方式触发
}
```

## 🔥 扩展追问 3：Spring Boot 自动配置有什么默认行为？

> **A:** Spring Boot 默认帮你配好了 SimpleApplicationEventMulticaster，如果没有自定义，就是同步执行。要开启异步有两种方式：

```java
// 方式一：application.yml 配置
spring:
  task:
    execution:
      pool:
        core-size: 5
      thread-name-prefix: async-

# 然后在监听器上加 @Async

// 方式二：自定义 Multicaster
@Bean
public ApplicationEventMulticaster eventMulticaster() {
    SimpleApplicationEventMulticaster m = new SimpleApplicationEventMulticaster();
    m.setTaskExecutor(Executors.newFixedThreadPool(10));
    return m;
}
```

## 🔥 扩展追问 4：多个监听器报错会影响其他监听器吗？

> **A:** 默认情况下会的。某个监听器抛异常，后续监听器就不会执行了。

**想让每个监听器独立运行，有两种方案：**

```java
// 方案一：每个监听器内部 try-catch
@Component
public class SafeListener {
    @EventListener
    public void handle(OrderCreatedEvent event) {
        try {
            // 业务逻辑
        } catch (Exception e) {
            log.error("监听器执行失败", e);
        }
    }
}

// 方案二：自定义 Multicaster，遍历调用而非直接调用
@Bean
public ApplicationEventMulticaster eventMulticaster(
        BeanFactory beanFactory) {
    return new SimpleApplicationEventMulticaster() {
        @Override
        public void multicastEvent(ApplicationEvent event, 
                                   @Nullable ResolvableType type) {
            for (ApplicationListener<?> listener : getApplicationListeners(event, type)) {
                try {
                    invokeListener(listener, event);
                } catch (Exception e) {
                    log.error("监听器 {} 执行失败", listener, e);
                }
            }
        }
    };
}
```

## 🔥 扩展追问 5：@TransactionalEventListener 的执行时机具体是什么时候？

> **A:** 它依赖于 Spring 的事务同步机制。四个阶段：

```
事务开始
    ↓
【BEFORE_COMMIT】 事务提交前（很少用，因为此时还没真正提交）
    ↓
事务真正提交 ← AFTER_COMMIT 在这里执行（最常用）
    ↓
【AFTER_ROLLBACK】 事务回滚后（如果回滚了）
    ↓
【AFTER_COMPLETION】 事务结束（无论成功还是回滚）
```

**实际应用场景：**

```java
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void afterOrderCommit(OrderCreatedEvent event) {
    // ✅ 最常用：订单已真正入库，可以发MQ通知下游系统
    mqProducer.send("order-topic", event.getOrderId());
}

@TransactionalEventListener(phase = TransactionPhase.AFTER_COMPLETION)
public void cleanup(OrderCreatedEvent event) {
    // 无论成功失败都清理资源，比如释放锁、清理缓存
    redisTemplate.delete("order:lock:" + event.getOrderId());
}
```

## 一句话总结

> **Spring 事件默认同步执行，通过 @EventListener + @Async 实现异步，用 @Order/Ordered 控制顺序，@TransactionalEventListener 实现事务绑定的事件监听，确保业务解耦的同时不破坏事务一致性。**
