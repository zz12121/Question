这是 Java 并发编程中的知识点，属于 JUC 包里的一个工具类。
**Exchanger 基础：**
- 用于两个线程之间交换数据
- 当一个线程调用 exchange() 方法后会阻塞，直到另一个线程也调用 exchange()
- 场景：遗传算法、数据结构填充、生产者-消费者数据传输

**核心原理：**
- 底层基于 CAS + 内部定义的槽（slot）
- 槽用来存放要交换的数据
- 使用槽的两种状态：EMPTY 和 FULL
- 还涉及超时机制
### 基础流程

```
线程 A                          线程 B
  │                              │
  │──── exchange(dataA) ─────────▶│  阻塞等待
  │                              │
  │         ◀─── exchange(dataB) ──│  阻塞等待
  │                              │
  ▼                              ▼
 dataA ←──────────────────────── dataB
  │                              │
  ▼                              ▼
 线程 A 拿到 dataB              线程 B 拿到 dataA
 线程 A 继续执行                线程 B 继续执行
```

**核心代码示例：**

```java
Exchanger<Integer> exchanger = new Exchanger<>();

// 线程 A
new Thread(() -> {
    Integer dataA = 100;
    System.out.println("线程A 想交换: " + dataA);
    try {
        Integer received = exchanger.exchange(dataA);
        System.out.println("线程A 收到: " + received);
    } catch (InterruptedException e) {
        e.printStackTrace();
    }
}).start();

// 线程 B
new Thread(() -> {
    Integer dataB = 200;
    System.out.println("线程B 想交换: " + dataB);
    try {
        Integer received = exchanger.exchange(dataB);
        System.out.println("线程B 收到: " + received);
    } catch (InterruptedException e) {
        e.printStackTrace();
    }
}).start();

// 输出:
// 线程A 想交换: 100
// 线程B 想交换: 200
// 线程A 收到: 200
// 线程B 收到: 100
```

**带超时的 exchange：**

```java
Integer received = exchanger.exchange(dataA, 5, TimeUnit.SECONDS);
// 5秒内没配对成功就抛 TimeoutException
```

## 🔥 扩展追问 1：Exchanger 的底层实现原理？

Exchanger 内部维护了一个 **槽（Slot）**，基于 **CAS** 实现：

```
核心数据结构：
┌─────────────────────────────────────────────────┐
│  Exchanger                                  │
│  ┌─────────────────────────────────────────┐  │
│  │  private volatile Node[] arena;         │  │  ← 参与者数组
│  │  private final Participant participant; │  │
│  │  private volatile Node slot;            │  │  ← 单槽（简化版）
│  └─────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘

Node 结构：
┌─────────────────────────────────────────────────┐
│  static final class Node {                    │
│      final int index;      // 槽位索引         │
│      final int bound;      // 界限值            │
│      final int collides;  // 冲突计数         │
│      final boolean isSecond; // 是否第二个线程  │
│      Object item;          // 携带的数据        │
│      volatile Object fill; // 交换结果         │
│      volatile Node next;   // 链表指针          │
│  }                                            │
└─────────────────────────────────────────────────┘
```

**交换流程：**
1. 线程到达时，用 CAS 把自己的数据放到 slot 中
2. 检查对面是否已经放了数据
3. 如果放了，取走对方数据，完成交换
4. 如果没放，阻塞等待（通过 Unsafe.park）
5. 对端线程被唤醒后取走数据

## 🔥 扩展追问 2：Exchanger 只能两个线程交换吗？

**只能两个线程！** 这是 Exchanger 和其他并发工具最大的区别：

| 特性 | Exchanger | SynchronousQueue | BlockingQueue |
|------|------------|-------------------|---------------|
| 配对数量 | **2** | 2 | 多 |
| 方向 | 双向交换 | 单向传输 | 单向传输 |
| 数据量 | 1对1交换 | 1对1传递 | 1对N传递 |
| API | exchange() | put()/take() | put()/take() |

**为什么限制两个？**
- 实现更简单高效
- 满足"数据核对"场景（一收一发数量必须相等）
- 超过两个线程会出现三方死锁问题

## 🔥 扩展追问 3：Exchanger 的典型使用场景？

**场景一：遗传算法中的种群交换**
```java
// 两个个体交换基因片段
Exchanger<Individual> exchanger = new Exchanger<>();
Individual a = population.get(0);
Individual b = exchanger.exchange(a);
// a 拿到 b 的基因，b 拿到 a 的基因
```

**场景二：双缓冲数据结构填充**
```java
Exchanger<Buffer> exchanger = new Exchanger<>();
Buffer filled = fillBuffer();  // 填满缓冲区

while (true) {
    Buffer empty = exchanger.exchange(filled);  // 换出已满的
    process(empty);                               // 处理数据
    filled = fillBuffer(empty);                   // 重新填充
}
```

**场景三：日志双缓冲区交替**
```
线程A（生产者）：写满缓冲区A → 交换 → 继续写
线程B（消费者）：拿到缓冲区A → 消费 → 换回空缓冲区

永远保持两个缓冲区循环使用，避免锁竞争
```

**场景四：数据核对（金融/电商常见）**
```java
// 发送方和接收方核对流水
Exchanger<RecordList> exchanger = new Exchanger<>();
RecordList sent = sendRecords();
RecordList received = exchanger.exchange(sent);
// 核对差异
```

## 🔥 扩展追问 4：Exchanger 与 SynchronousQueue 的区别？

这是高频面试题！

| 对比项 | Exchanger | SynchronousQueue |
|--------|-----------|-------------------|
| **作用** | 交换数据（双向） | 传递数据（单向） |
| **数据归属** | 双方数据互换 | 数据从一方转移到另一方 |
| **容量** | 1（隐式槽） | 0（不存储） |
| **配对后** | 线程A拿线程B的，线程B拿线程A的 | 线程A把数据给线程B |
| **类比** | 交换名片 | 交接棒 |

```java
// SynchronousQueue - 单向传递
SynchronousQueue<Integer> sq = new SynchronousQueue<>();
sq.put(100);  // 阻塞，等有人 take()
sq.take();    // 阻塞，等有人 put()

// Exchanger - 双向交换
Exchanger<Integer> ex = new Exchanger<>();
ex.exchange(100);  // 阻塞，等有人也调用 exchange()
```

## 🔥 扩展追问 5：Exchanger 的超时场景怎么处理？

```java
Exchanger<Bundle> exchanger = new Exchanger<>();

public Bundle safeExchange(Bundle data, long timeoutMs) {
    try {
        return exchanger.exchange(data, timeoutMs, TimeUnit.MILLISECONDS);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        return null;  // 被中断，返回 null
    } catch (TimeoutException e) {
        log.warn("数据交换超时，可能对端线程已退出");
        return null;  // 超时，返回 null
    }
}
```

**超时后的降级策略：**
- 重试 N 次后放弃
- 记录未交换数据，后续补偿
- 通知监控告警

## 🔥 扩展追问 6：Exchanger 怎么应对线程不匹配的情况？

如果只有一个线程调用 exchange()，会一直阻塞。

**解决方案：**

```java
// 方案一：使用超时
Future<Bundle> future = executor.submit(() -> 
    exchanger.exchange(data, 10, TimeUnit.SECONDS)
);

// 方案二：配合 CountDownLatch 检测
CountDownLatch latch = new CountDownLatch(1);
Future<?> future = executor.submit(() -> {
    latch.countDown();
    exchanger.exchange(data);
});
latch.await(1, TimeUnit.SECONDS);  // 1秒内没配对就放弃

// 方案三：双 Exchanger 兜底
Exchanger<Bundle> ex1 = new Exchanger<>();
Exchanger<Bundle> ex2 = new Exchanger<>();  // 备用
```

## 🔥 扩展追问 7：Exchanger 在高并发下会失效吗？

**可能失效！** 当超过 2 个线程同时竞争时：

```
线程 A ──▶ exchange() ──▶ slot
线程 B ──▶ exchange() ──▶ slot  ← 冲突！

线程 C ──▶ 看到 slot 满了 ──▶ 创建新槽位 arena[1]
线程 D ──▶ 看到 arena[1] 满了 ──▶ 创建 arena[2]
```

**问题：**
- arena 数组扩容有开销
- 大量线程竞争会导致 CPU 空转
- 极端情况退化为串行

**优化建议：**
```java
// 用信号量控制同时参与的线程数
Semaphore semaphore = new Semaphore(2);
Exchanger<Bundle> exchanger = new Exchanger<>();

Runnable task = () -> {
    semaphore.acquire();
    try {
        exchanger.exchange(data);
    } finally {
        semaphore.release();
    }
};
```

## 🔥 扩展追问 8：Exchanger 的内存可见性保证？

exchange() 前后有 **happens-before** 保证：

```
线程 A                          线程 B
   │                               │
   │  write shared data             │  write shared data  
   │  happens-before               │  happens-before
   ▼                               ▼
   exchange(A) ────────────────▶  exchange(B)
   │                               │
   ◀─────────── exchange ──────────┤
   │                               │
   │  happens-before               │  happens-before
   ▼                               ▼
   read shared data                read shared data
```

**底层原理：**
- 交换过程依赖 `Unsafe.park/unpark` 和 CAS
- 这些操作本身带 volatile 语义
- JMM 保证配对线程之间数据可见
