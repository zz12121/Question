## 🔥 扩展追问 1：Mixed GC 和 Full GC 的核心区别？

| 对比项 | Young GC | Mixed GC | Full GC |
|--------|----------|----------|---------|
| **触发时机** | Eden 区满 | Old 区占比 > IHOP | Mixed GC 失败 |
| **收集范围** | 新生代（Eden + Survivor） | 新生代 + 部分老年代 | 整个堆 |
| **回收器类型** | STW，年轻代收集器 | G1 老年代并发收集 | 单线程/并行 Serial/Parallel GC |
| **并发程度** | 完全 STW | 部分并发（并发标记） | 完全 STW |
| **停顿时间** | 短（< IHOP 参数控制） | 中等（可配置） | 长（尽量避免） |
| **目标** | 保留 Survivor 区 | 降低老年代占用 | 最终保障 |
| **频率** | 高 | 中 | 低（尽量避免） |

**G1 的设计目标：**
- Young GC 保持停顿短
- Mixed GC 替代传统 Full GC 的大部分场景
- Full GC 视为"失败恢复机制"，不应频繁发生

## 🔥 扩展追问 3：什么情况下会触发 Full GC？

**Full GC 触发条件清单：**

```
┌─────────────────────────────────────────────────────────────────┐
│                    Full GC 触发条件                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  条件1️⃣：Mixed GC 来不及回收                                    │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ 老年代空间不足以容纳要晋升的对象                      │       │
│  │ → 担保失败 → Full GC                                │       │
│  │                                                      │       │
│  │ -XX:-G1Ergonomics 会禁用自动调整                    │       │
│  │ -XX:G1ReservePercent=10  增加预留空间               │       │
│  └─────────────────────────────────────────────────────┘       │
│                            │                                    │
│  条件2️⃣：Humongous 对象分配失败                                │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ Humongous = 单个分区 > Region 50% 的对象             │       │
│  │ 直接分配到 Old 区                                   │       │
│  │ 如果没有连续的大分区 → Full GC                       │       │
│  └─────────────────────────────────────────────────────┘       │
│                            │                                    │
│  条件3️⃣：显式调用 System.gc()                                 │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ -XX:+ExplicitGCInvokesConcurrent                   │       │
│  │   → G1 用并发方式处理（减少停顿）                    │       │
│  │ -XX:-ExplicitGCInvokesConcurrent                   │       │
│  │   → 直接 Full GC                                    │       │
│  └─────────────────────────────────────────────────────┘       │
│                            │                                    │
│  4️⃣：元空间（Metaspace）不足                                   │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ JDK 8 使用 Metaspace 替代 PermGen                   │       │
│  │ 类加载过多 → Metaspace 满 → Full GC                 │       │
│  └─────────────────────────────────────────────────────┘       │
│                            │                                    │
│  5️⃣：堆膨胀导致分配失败                                         │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ G1 堆弹性伸缩                                        │       │
│  │ 如果无法收缩到目标大小 → Full GC                      │       │
│  └─────────────────────────────────────────────────────┘       │
│                            │                                    │
│  6️⃣：并发模式失败（Concurrent Mode Failure）                    │
│  ┌─────────────────────────────────────────────────────┐       │
│  │ Mixed GC 期间，用户线程产生太多垃圾                   │       │
│  │ Old 区满了                                           │       │
│  │ → 降级为 Full GC                                     │       │
│  └─────────────────────────────────────────────────────┘       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**典型场景代码：**

```java
public class FullGCEscape {

    // 场景1：Humongous 对象导致 Full GC
    public void humongousObject() {
        // 单个数组超过 Region 大小的 50%
        // G1 会分配多个连续的 Humongous 区域
        // 如果空间不足 → Full GC
        byte[] bigData = new byte[4 * 1024 * 1024];  // 4MB，假设 Region=1MB
    }

    // 场景2：快速晋升导致老年代满
    public void rapidPromotion() {
        List<Object> list = new ArrayList<>();
        for (int i = 0; i < 100000; i++) {
            // 每个对象约 1KB
            // Survivor 区装不下，快速晋升 Old 区
            list.add(new byte[900]);  // 实际占用 ~1KB（含对象头）
        }
    }

    // 场景3：显式 GC
    public void explicitGC() {
        System.gc();  // 可能触发 Full GC，除非开启 -XX:+ExplicitGCInvokesConcurrent
    }
}
```

## 🔥 扩展追问 4：如何避免 Full GC？

**配置优化：**

```bash
# 调整 IHOP（老年代启动混合收集的阈值）
# 默认 45%，太低会频繁 Mixed GC，太高会来不及回收
-XX:InitiatingHeapOccupancyPercent=50  # 推荐范围 40-60

# 增加 G1 预留空间（应对晋升高峰）
-XX:G1ReservePercent=15  # 默认 10%，增加预留减少 Full GC

# 调整 Mixed GC 的分区数量
-XX:G1MixedGCCountTarget=8  # 一次 Mixed GC 收集的老年代分区数

# 调整最大 GC 停顿目标（平衡吞吐量和停顿）
-XX:MaxGCPauseMillis=200  # G1 会尽量满足，但不一定能达成

# 处理 Humongous 对象的参数
-XX:G1HeapRegionSize=4m  # 增大 Region，减少 Humongous 对象
```

**代码层面优化：**

```java
// 避免大对象
public class AvoidHumongous {
    // ❌ 大数组触发 Humongous，频繁 Full GC
    byte[] bad = new byte[10 * 1024 * 1024];

    // ✅ 拆分成多个小对象
    List<byte[]> chunks = IntStream.range(0, 10)
        .mapToObj(i -> new byte[1024 * 1024])
        .collect(Collectors.toList());
}

// 减少内存分配速率
public class ReduceAllocationRate {
    // ❌ 频繁创建临时对象
    public String badMethod() {
        String s = "";
        for (int i = 0; i < 100; i++) {
            s += i;  // 每次循环创建新 String
        }
        return s;
    }

    // ✅ 使用 StringBuilder
    public String goodMethod() {
        StringBuilder sb = new StringBuilder();
        for (int i = 0; i < 100; i++) {
            sb.append(i);
        }
        return sb.toString();
    }
}
```

## 🔥 扩展追问 5：G1 的分区回收价值如何计算？

**G1 的分区选择算法（Evacuation Failure 预防）：**

```
回收价值公式：

  回收价值 = 垃圾数量 / 分区大小 × 回收成本系数

示例：
  分区 A：垃圾 75%，分区大小 1MB → 价值 0.75
  分区 B：垃圾 50%，分区大小 1MB → 价值 0.50
  分区 C：垃圾 90%，分区大小 2MB → 价值 0.45

  G1 优先选择价值高的分区 → 用最少时间回收最多垃圾
```

**G1MixedGCLiveThresholdPercent：**

```bash
# 活对象比例超过这个值的分区不回收
-XX:G1MixedGCLiveThresholdPercent=85  # 默认 85%

# 解释：
# 活对象 15% 的分区 → 回收价值高
# 活对象 90% 的分区 → 回收价值低，不值得回收
```

**GC 失败前的预防机制：**

```
Evacuation Failure（转移失败）= Full GC 的前兆

触发条件：
  1. 选中的分区复制时，发现没有空闲分区
  2. 转移指针失败

G1 的应对：
  1. 记录所有转移失败的分区
  2. 增加 Mixed GC 频率
  3. 增加目标老年代分区数
  4. 如果还是失败 → Full GC
```

## 🔥 扩展追问 6：G1 的并发标记和 CMS 有何不同？

| 特性 | CMS | G1 |
|------|-----|-----|
| **代际划分** | 物理分区（New/Old） | 逻辑分区（Region） |
| **收集方式** | 标记-清除（内存碎片） | 标记-复制（无碎片） |
| **并发阶段** | 并发标记、并发预清理 | 并发标记 + 最终标记 + 筛选 |
| **停顿类型** | STW（初始/最终标记） | STW（初始/最终/筛选） |
| **Mixed GC** | 无（只有 CMS 老年代收集） | 有（新生代+部分老年代） |
| **内存碎片** | 有（需定期 Full GC 整理） | 无（复制到空闲分区） |
| **大对象处理** | 直接进老年代（可能触发 Full GC） | Humongous 独立分配 |

```
CMS 标记-清除：
  ┌────────────┐       ┌────────────┐
  │  ██  ██    │  ──▶  │           │    ← 内存碎片
  │    ██  ██ │       │     ██    │
  └────────────┘       └────────────┘

G1 标记-复制：
  ┌────────────┐       ┌────────────┐
  │  ██  ██    │  ──▶  │ ██████████│    ← 无碎片
  │    ██  ██ │       │           │
  └────────────┘       └────────────┘
       源分区                目标分区
```

## 一句话总结（面试收尾用）

> **G1 的 Young GC 只收集新生代分区，Mixed GC 在老年代占比超过 IHOP 阈值时启动，会收集新生代加上垃圾最多的部分老年代分区（标记-复制，无内存碎片），而 Full GC 是 G1 的"兜底机制"，当 Mixed GC 来不及回收、Humongous 对象分配失败或显式 GC 时触发，应尽量避免，配置上通过调高 IHOP、增加预留空间和减少大对象来预防。**
