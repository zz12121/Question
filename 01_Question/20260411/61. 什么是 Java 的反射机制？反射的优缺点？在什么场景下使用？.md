### 基础流程

```
获取 Class 对象
    ↓
① Class.forName("全限定类名")
② 对象.getClass()
③ 类名.class
    ↓
② 通过 Class 对象操作成员
┌──────────────────────────────────────┐
│  Constructor → newInstance()        │  创建对象
│  Method → invoke(obj, args)         │  调用方法
│  Field → get/set(obj, value)        │  操作属性
└──────────────────────────────────────┘
```

**核心代码示例：**

```java
// 1. 获取 Class
Class<?> clazz = Class.forName("com.example.User");

// 2. 创建实例
Object obj = clazz.getDeclaredConstructor().newInstance();

// 3. 获取方法并调用
Method setName = clazz.getDeclaredMethod("setName", String.class);
setName.invoke(obj, "张三");

// 4. 获取属性并赋值（绕过权限检查）
Field nameField = clazz.getDeclaredField("name");
nameField.setAccessible(true);  // 关键！否则 private 属性能被访问
nameField.set(obj, "李四");
```

## 🔥 扩展追问 1：反射为什么能破坏封装？setAccessible 有什么用？

**原理：**

- Java 有访问权限修饰符（`private`、`protected`、`public`）
- 正常代码受 JVM 安全管理器检查
- 反射通过 `setAccessible(true)` 可以**绕过编译期检查**，直接操作 `private` 成员

```java
public class PrivateClass {
    private String secret = "机密信息";
}

// 反射可以这样访问：
Field field = PrivateClass.class.getDeclaredField("secret");
field.setAccessible(true);  // 绕过 private 检查
String value = (String) field.get(new PrivateClass());  // 成功拿到
```

**面试加分话术：**

> "setAccessible 底层是 Unsafe 类直接操作对象内存，修改了 JVM 的访问标志位。它是框架的基石——Spring 依赖注入、MyBatis 映射、JUnit 测试都用它。不过在 Java 9+ 模块化后，需要加 `--add-opens java.base/java.lang=ALL-UNNAMED` 才能使用。"

## 🔥 扩展追问 2：反射的性能差在哪？怎么优化？

**性能损耗的三个来源：**

| 损耗点 | 说明 | 影响 |
|---|---|---|
| **Method.invoke** | 每次调用要检查参数类型、权限 | 每次多 3~10μs |
| **权限检查** | setAccessible(false) 时每次都检查 | 可用 setAccessible(true) 跳过 |
| **泛型擦除** | getParameterTypes() 返回擦除后的类型 | 缓存 Class 对象 |
| **即时编译失效** | JIT 无法优化反射调用 | 热点反射可先生成 MethodHandle |

**优化方案：**

```java
// 方案1：缓存 Method 对象（最常用）
public class ReflectionCache {
    private static final Map<String, Method> CACHE = new ConcurrentHashMap<>();
    
    public static Object invoke(String className, String methodName, 
                                 Object target, Object... args) throws Exception {
        String key = className + "#" + methodName;
        Method method = CACHE.computeIfAbsent(key, k -> {
            try {
                Method m = Class.forName(className).getDeclaredMethod(methodName);
                m.setAccessible(true);  // 一次设置，后续直接用
                return m;
            } catch (Exception e) {
                throw new RuntimeException(e);
            }
        });
        return method.invoke(target, args);
    }
}

// 方案2：MethodHandle（JDK 7+，性能更好）
MethodHandles.Lookup lookup = MethodHandles.lookup();
MethodHandle mh = lookup.findVirtual(User.class, "setName", 
    MethodType.methodType(void.class, String.class));
mh.invokeExact(user, "张三");
```

**面试加分话术：**

> "反射本身不慢，**慢的是每次查找和权限检查**。优化思路就三条：缓存 Method 对象避免重复查找、`setAccessible(true)` 跳过权限检查、用 MethodHandle 在热点场景替代反射。Spring Boot 的 Spring-core 包里就有专门的 `ReflectionUtils` 做反射优化。"

## 🔥 扩展追问 3：反射在框架中是怎么用的？举几个例子？

**典型应用场景：**

```
┌─────────────────────────────────────────────────────────┐
│                    反射在框架中的应用                     │
├─────────────────────────────────────────────────────────┤
│ Spring      │ 依赖注入：扫描 @Autowired，setAccessible  │
│             │  Bean 创建：Class.newInstance()           │
├─────────────────────────────────────────────────────────┤
│ MyBatis     │ 结果映射：把 ResultSet 映射到对象          │
│             │ SQL 参数：对象属性拼接到 SQL               │
├─────────────────────────────────────────────────────────┤
│ JDBC        │ DriverManager 用 Class.forName 加载驱动  │
│             │ RowMapper 用反射设值                      │
├─────────────────────────────────────────────────────────┤
│ Spring MVC  │ 参数绑定：请求参数反射设到 Controller 参数  │
├─────────────────────────────────────────────────────────┤
│ 注解处理器   │ 编译期/运行时读取注解做逻辑处理            │
└─────────────────────────────────────────────────────────┘
```

**Spring 依赖注入简化实现：**

```java
public class Simple DI Container {
    public void inject(Object bean) {
        for (Field field : bean.getClass().getDeclaredFields()) {
            if (field.isAnnotationPresent(Autowired.class)) {
                Object dependency = createOrGetBean(field.getType());
                field.setAccessible(true);
                field.set(bean, dependency);  // 反射赋值
            }
        }
    }
}
```

**面试加分话术：**

> "可以说反射是**框架的灵魂**。没有反射，Spring 就没法自动注入，MyBatis 没法映射结果集。你用的 @Autowired、@Value、@Resource 这些注解，底层全是反射在工作。我之前做过一个简化版 IOC 容器，就是为了加深理解——核心就是三个步骤：扫描类 → 解析注解 → 反射赋值。"

## 🔥 扩展追问 4：Class.forName 和 ClassLoader.loadClass 的区别？

| 方面 | Class.forName | ClassLoader.loadClass |
|---|---|---|
| **初始化时机** | **立刻执行 static 代码块** | **延迟到首次 newInstance** |
| **参数** | 支持初始化器参数 | 不支持 |
| **典型场景** | JDBC 驱动加载 | 类懒加载、模块化加载 |
| **异常** | checked exception | unchecked exception |

**代码对比：**

```java
// Class.forName：JDBC 驱动必须用它
Class.forName("com.mysql.cj.jdbc.Driver");  // 触发 static{}，注册驱动

// ClassLoader.loadClass：适合插件化、模块化场景
ClassLoader loader = new URLClassLoader(urls);
Class<?> clazz = loader.loadClass("com.plugin.MyPlugin");  // 不初始化，等需要时再动
```

**面试加分话术：**

> "区别在于**初始化时机**。Class.forName 会触发 static 块，所以 JDBC 驱动必须用它——驱动里的 static 块负责向 DriverManager 注册。而 ClassLoader.loadClass 是懒加载，常用于插件化框架或者需要动态卸载类的场景，比如 OSGi、Tomcat 的热部署。"

## 🔥 扩展追问 5：反射是否线程安全？

**结论：Class 对象本身是线程安全的，但操作的对象不是。**

```java
// Class 对象：JVM 保证每个类只有一个，由 Bootstrap ClassLoader 管理
// 多个线程获取同一个 Class.class 是安全的

// 但是 Field/Method/Constructor 是可变对象，且不是线程安全的！
public class UnsafeReflect {
    static Method method;  // 危险！
    
    public static void init() {
        method = User.class.getDeclaredMethod("setName", String.class);
        method.setAccessible(true);
        // 如果 A 线程在调用，B 线程改了 setAccessible...
    }
}
```

**安全使用建议：**

```java
// 建议：用 ThreadLocal 缓存，或者每次 copy
public class SafeReflect {
    private static final ThreadLocal<Method[]> CACHE = ThreadLocal.withInitial(() -> {
        // 每个线程独立缓存
        return new Method[16];  // 预分配
    });
}
```

## 🔥 扩展追问 6：既然反射有性能问题，为什么不用字节码生成替代？

**三种方案的对比：**

| 方案 | 代表 | 性能 | 复杂度 | 适用场景 |
|---|---|---|---|---|
| **反射** | JDK 自带 | 慢（但可优化） | 低 | 通用框架、一次性的动态调用 |
| **字节码生成** | cglib、ASM、Javassist | 极快 | 高 | 批量代理（AOP）、ORM |
| **动态代理** | JDK Proxy | 快 | 中 | 接口代理 |

**Spring 为什么选 cglib 而不是 JDK 动态代理？**

```java
// JDK 动态代理：必须接口
public class JDKProxy implements InvocationHandler {
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        return method.invoke(target, args);  // 还是要反射！
    }
}

// cglib：基于 ASM 生成子类字节码，直接调用
// 子类重写方法，方法体直接调原始逻辑，无反射开销
public class CglibProxy extends UserService {
    @Override
    public void save() {
        // 生成的字节码直接 call super.save()
        // 比反射快 10 倍以上
    }
}
```

**面试加分话术：**

> "这是一个很好的问题。反射、字节码生成、动态代理是递进关系。反射最灵活但最慢；字节码生成（如 cglib/ASM）直接生成 .class 文件，零反射开销，但写起来复杂；JDK 动态代理是折中方案，但只能代理接口。Spring 5.x 默认用 cglib，就是用复杂度换性能。但如果你的业务只是偶尔调一次反射，缓存优化就够了，没必要上字节码生成。"

## 🔥 扩展追问 7：反射在 RPC 框架（Dubbo/Feign）里是怎么用的？

**核心用途：反序列化**

```
网络传输：JSON/Protobuf 二进制
        ↓
RPC 框架收到字节流
        ↓
知道类型：序列化时带了类名（如 "com.example.User"）
        ↓
Class.forName("com.example.User") 获取 Class
        ↓
反射调用 setter / 构造函数 创建对象
        ↓
填充属性
```

**简化版实现：**

```java
public class FastjsonDeserializer<T> {
    public T deserialize(String json, Class<T> clazz) {
        // 1. 创建实例（跳过构造函数，用反射）
        T obj = clazz.getDeclaredConstructor().newInstance();
        
        // 2. 获取所有 setter
        for (Field field : clazz.getDeclaredFields()) {
            String setterName = "set" + capitalize(field.getName());
            try {
                Method setter = clazz.getDeclaredMethod(setterName, field.getType());
                setter.setAccessible(true);
                
                // 3. 从 JSON 拿值，类型转换后反射设进去
                Object value = parseValue(json, field.getName(), field.getType());
                setter.invoke(obj, value);
            } catch (NoSuchMethodException ignored) {}
        }
        return obj;
    }
}
```

**面试加分话术：**

> "RPC 框架里反射最核心的应用就是**运行时动态创建对象并填充属性**。Dubbo 用 Hessian/Protostuff 序列化，Feign 用 Gson/Jackson，底层都是：拿到 Class → 反射创建实例 → 反射设值。如果不用反射，框架根本不知道要创建什么类型的对象。另外，Feign 的参数绑定也是用反射——Controller 方法的参数名、注解值，都是运行时反射解析的。"

## 一句话总结（面试收尾用）

> "Java 反射通过 Class 对象在**运行时动态获取类信息并操作成员**，是 Spring、MyBatis、JDBC 等框架的基石。优点是灵活性高、解耦做得好；缺点是性能差（可优化）和破坏封装（需要权限控制）。面试时能结合 Spring IOC/MyBatis 原理讲的，基本就是加分项。"
