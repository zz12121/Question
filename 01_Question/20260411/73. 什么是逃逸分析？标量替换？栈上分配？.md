  │ 不逃逸 (NoEscape)     ─▶ 栈上分配 / 标量替换  │
  │ 参数逃逸 (ArgEscape)  ─▶ 栈上分配（谨慎优化） │

  └──────────────────────────────────────────────┘

**核心代码示例：**

```java
public class EscapeAnalysisDemo {

    // ❌ 逃逸分析关闭时：对象在堆上分配，触发 GC
    public void badMethod() {
        for (int i = 0; i < 100000; i++) {
            Point p = new Point(i, i + 1);  // 堆上分配
            p.distance();
        }
    }

    // ✅ 逃逸分析开启时：标量替换，无需分配对象
    public void goodMethod() {
        for (int i = 0; i < 100000; i++) {
            // JIT 识别 p 不逃逸，直接用 x=i, y=i+1 替代
            Point p = new Point(i, i + 1);  // 实际编译后：int x=i; int y=i+1
            p.distance();
        }
    }

    // ❌ 强制逃逸：方法返回对象引用
    public Point factory() {
        Point p = new Point(0, 0);
        return p;  // 全局逃逸，必须堆分配
    }

    static double distance(Point p) {
        return Math.sqrt(p.x * p.x + p.y * p.y);  // 参数逃逸
    }
}
```

## 🔥 扩展追问 1：逃逸分析的具体判断逻辑？

**HotSpot 的逃逸分析时机：**

```
逃逸分析 = Early Escape Analysis（激进优化）

编译阶段：
  字节码 ──▶ C1 编译（轻量）──▶ C2 编译（激进）
                            │
                            ▼
                    逃逸分析 + 标量替换 + 栈上分配
                            │
                            ▼
                      生成优化后的机器码

关键：逃逸分析在 C2 阶段进行（-server 模式默认开启）
```

**逃逸的判断规则：**

| 逃逸类型 | 条件 | 分配位置 |
|---------|------|---------|
| **NoEscape** | 对象只在方法内使用 | 栈上分配 / 标量替换 |
| **ArgEscape** | 作为参数传递给其他方法，但方法不会存储引用 | 栈上分配（分析调用者） |
| **GlobalEscape** | 返回给方法外部、存入静态变量、放入集合 | 必须堆分配 |

```java
class EscapeRules {
    Point global;  // 静态变量 = GlobalEscape

    // 情况1：返回引用 = 全局逃逸
    public Point create1() {
        return new Point(1, 2);  // ❌ 逃逸
    }

    // 情况2：作为参数传递，但内部未保存
    public void use2(Point p) {
        p.distance();  // ✅ NoEscape（JIT 能分析到）
    }

    // 情况3：存入成员变量
    public void set3(Point p) {
        this.global = p;  // ❌ 逃逸
    }

    // 情况4：存入局部集合但不传出
    public void use4(Point p) {
        List<Point> list = new ArrayList<>();
        list.add(p);  // ⚠️ 参数逃逸
        p.distance();
    }
}
```

## 🔥 扩展追问 2：标量替换是什么？

**标量（Scalar）** = 不可再分的基本类型（如 int、long、double）  
**聚合量（Aggregate）** = 对象（可以拆解成多个标量）

**标量替换 = 把对象拆解成它的成员变量：**

```java
// 原始代码
public double calc() {
    Point p = new Point(3, 4);
    return p.x + p.y;
}

// 标量替换后（JIT 优化）
public double calc() {
    int x = 3;  // 用标量替代对象
    int y = 4;
    return x + y;
}

// 完全不需要 new Point()！
```

**触发标量替换的完整示例：**

```java
public class ScalarReplacement {
    static class Node {
        int x;
        int y;
        Node next;
    }

    // 不开启标量替换：new Node() 10万次，GC压力大
    public int badSum() {
        int sum = 0;
        for (int i = 0; i < 100000; i++) {
            Node n = new Node();
            n.x = i;
            n.y = i * 2;
            sum += n.x + n.y;
        }
        return sum;
    }

    // 开启标量替换：new Node() 编译后消失
    // 实际效果相当于：
    // for (int i = 0; i < 100000; i++) {
    //     int x = i;
    //     int y = i * 2;
    //     sum += x + y;
    // }
}
```

## 🔥 扩展追问 3：栈上分配是怎么回事？

**对象分配位置对比：**

```
┌─────────────────────────────────────────────────────────┐
│                    线程私有栈                            │
│  ┌─────────────────────────────────────────────────┐   │
│  │  Stack Frame (foo)                              │   │
│  │  ┌─────────────┐                                │   │
│  │  │ local_vars  │  ← 标量替换后的变量在这里       │   │
│  │  │ x = 3       │                                │   │
│  │  │ y = 4       │                                │   │
│  │  └─────────────┘                                │   │
│  │  ┌─────────────┐                                │   │
│  │  │ operands   │                                │   │
│  │  └─────────────┘                                │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │  Stack Frame (bar)                              │   │
│  │  ...                                            │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘

                        ▼

┌─────────────────────────────────────────────────────────┐
│                    Java 堆                             │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐    │
│  │  Object A   │  │  Object B   │  │  Object C   │    │
│  │  (逃逸对象)  │  │ (逃逸对象)  │  │  (逃逸对象)  │    │
│  └─────────────┘  └─────────────┘  └─────────────┘    │
│                        ▲                               │
│                   GC 管理区域                           │
└─────────────────────────────────────────────────────────┘
```

**栈上分配 vs 堆分配：**

| 特性 | 栈上分配 | 堆分配 |
|------|----------|--------|
| 分配速度 | ✅ 快（指针碰撞） | 较慢（TLAB + 指针碰撞） |
| 回收速度 | ✅ 自动释放（出栈即销毁） | 慢（需要 GC） |
| 生命周期 | 方法级别 | 对象级别 |
| 线程安全 | ✅ 线程私有，无竞争 | 需要 GC 保证 |
| GC 压力 | ✅ 无 | 大 |

## 🔥 扩展追问 5：逃逸分析的局限性？

**局限性一：动态分析困难**

```java
public void tricky() {
    List<String> list = new ArrayList<>();
    if (Math.random() > 0.5) {
        // 动态判断，可能无法静态分析
        if (list.size() > 0) {
            return list.get(0);  // 传入外部
        }
    }
    // 保守策略：不优化
}
```

**局限性二：继承和多态**

```java
class Base { int x; }
class Derived extends Base { int y; }

public Base factory(boolean flag) {
    Base obj;
    if (flag) {
        obj = new Base();
    } else {
        obj = new Derived();  // 运行时才知道具体类型
    }
    return obj;  // 可能保守，不做优化
}
```

**局限性三：循环中的对象**

```java
// HotSpot 对简单循环优化较好，但复杂情况会保守
public void loopTricky(int n) {
    for (int i = 0; i < n; i++) {
        Node node = new Node(i);  // 循环依赖分析
        process(node);
    }
}
```

## 🔥 扩展追问 6：逃逸分析 + 锁消除的关系？

**锁消除基于逃逸分析：**

```java
// 原始代码
public synchronized void safeMethod() {
    StringBuilder sb = new StringBuilder();
    sb.append("hello");
    sb.append("world");
}

// 逃逸分析：sb 不逃逸 → 不会有多线程竞争
// 锁消除：synchronized 自动移除
// 实际效果：
public void safeMethod() {
    StringBuilder sb = new StringBuilder();  // 栈上分配
    sb.append("hello");
    sb.append("world");
}

// 参数开启
-XX:+DoEscapeAnalysis       # 逃逸分析
-XX:+EliminateLocks         # 锁消除（默认开启）
```

**StringBuilder vs StringBuffer：**

| 类 | 线程安全 | 逃逸分析优化 |
|----|----------|--------------|
| StringBuffer | synchronized | 锁消除（性能好） |
| StringBuilder | 非 synchronized | 标量替换（性能更好） |

## 🔥 扩展追问 8：实际开发中如何利用逃逸分析优化？

**原则一：控制对象作用域**

```java
// ❌ 扩大作用域，对象逃逸
public List<Point> badScope(List<Point> result) {
    Point p = new Point(0, 0);  // 作用域扩大到整个方法
    for (int i = 0; i < 1000; i++) {
        p.set(i, i);
        result.add(p);  // 逃逸
    }
    return result;
}

// ✅ 缩小作用域
public List<Point> goodScope(int n) {
    List<Point> result = new ArrayList<>();
    for (int i = 0; i < n; i++) {
        Point p = new Point(i, i);  // 作用域仅限循环内
        result.add(p);
    }
    return result;
}
```

**原则二：返回对象属性而非对象**

```java
// ❌ 返回对象（逃逸）
public Point getPosition() {
    return this.position;  // 全局逃逸
}

// ✅ 返回基本类型（不逃逸）
public double getX() { return this.x; }
public double getY() { return this.y; }
```

**原则三：构造器内避免 this 逸出**

```java
// ❌ 构造器 this 逸出
public class ThisEscape {
    public ThisEscape(EventSource source) {
        source.register(this);  // 在构造完成前就暴露 this
    }
}

// ✅ 使用工厂方法
public class SafeConstruction {
    private final int x;
    private final int y;

    private SafeConstruction(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public static SafeConstruction create(EventSource source) {
        SafeConstruction sc = new SafeConstruction(1, 2);
        source.register(sc);  // 构造完成后注册
        return sc;
    }
}
```

## 一句话总结（面试收尾用）

> **逃逸分析是 JIT 编译期的静态分析技术，判断对象是否逃逸出方法作用域，若不逃逸则可通过标量替换（拆解为基本类型）和栈上分配优化，从而减少堆分配、降低 GC 压力、提升性能，是 JVM 性能优化的重要手段。**
