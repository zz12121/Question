## 🔥 扩展追问 1：为什么是 16384 个 slot？能否修改？

**16384 vs 其他数字的选择：**

| 设计考量 | 说明 |
|---------|------|
| 心跳包大小 | 每个 slot 占 2 字节，16384 × 2 = 32KB |
| 节点扩展性 | 可支持数千个节点，每个节点平均 5461 个 slot |
| 迁移压力 | 节点间迁移 slot 时，压力可控 |
| 通信效率 | 节点间每秒多次心跳，过大包会造成网络阻塞 |

```
为什么不选更多 slot（比如 2^16 = 65536）？
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  如果用 65536 个 slot：                                         │
│  • 每个 slot 占 2 字节 → 心跳包 = 65536 × 2 = 128KB            │
│  • 每秒多次心跳 → 网络开销翻 4 倍                               │
│  • 节点越多，心跳包越大 → 集群规模受限                          │
│                                                                 │
│  16384 的权衡：                                                 │
│  • 32KB 心跳包 → 在节点数量和包大小间取得平衡                   │
│  • 16384 可支持上千个节点 → 实际够用                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**能否修改？**

> A: 不能，16384 是 Redis Cluster 协议硬编码的常量。如果需要更多分片，有两个方案：
> 1. **Codis**：代理分片方案，slot 数量可自定义（默认 1024）
> 2. **客户端分片**：Jedis Sharding/ShardingSphere，按一致性哈希分配数据
> 
> 但这两个方案都失去了 Redis Cluster 的官方支持和部分特性（如内置故障转移）。

## 🔥 扩展追问 2：MOVED 和 ASK 重定向的区别？

**两种重定向的对比：**

| 类型 | 含义 | 客户端动作 | 适用场景 |
|------|------|-----------|---------|
| MOVED | slot 已永久迁移 | 更新本地映射，下次直接访问新节点 | 迁移完成后的正常更新 |
| ASK | slot 正在迁移中 | 先发 ASKING 命令，再访问目标节点 | 迁移进行中的临时处理 |

```
┌─────────────────────────────────────────────────────────────────┐
│                      MOVED vs ASK 原理                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  MOVED（永久迁移）：                                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  节点 A ──▶ slot 100 已迁移到节点 B                    │   │
│  │                                                         │   │
│  │  返回: MOVED 100 192.168.1.102:6379                     │   │
│  │                                                         │   │
│  │  客户端处理:                                            │   │
│  │  1. 更新本地 slot → node 映射                          │   │
│  │  2. 之后所有 slot 100 的请求直接打节点 B               │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ASK（临时迁移）：                                              │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  slot 100 正在从节点 A 迁移到节点 B                     │   │
│  │                                                         │   │
│  │  返回: ASK 100 192.168.1.102:6379                       │   │
│  │                                                         │   │
│  │  客户端处理:                                            │   │
│  │  1. ASKING 命令（告诉目标节点"我已申请"）              │   │
│  │  2. 去节点 B 执行请求                                    │   │
│  │  3. 不更新本地映射（下一次可能还在 A）                   │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  简单记忆：MOVED = 永久搬家，ASK = 临时借住                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> A: MOVED 表示 slot 已永久迁移到新节点，客户端应更新本地映射并直接访问新节点。ASK 表示 slot 正在迁移中，客户端需要先发 ASKING 命令再去目标节点执行。ASK 不会更新本地映射，因为迁移还没完成，下次可能还要打回原节点。实际业务中 ASK 重定向比较少见，通常只在 `redis-cli --cluster reshard` 手动迁移 slot 时触发。

## 🔥 扩展追问 3：Sentinel 和 Cluster 的本质区别？

**核心对比：**

| 维度 | Redis Sentinel | Redis Cluster |
|------|---------------|---------------|
| **设计目标** | 高可用 (HA) | 高可用 + 数据分片 |
| **数据存储** | 主从同步，每节点全量数据 | 分片存储，每节点部分数据 |
| **客户端感知** | 连接 Sentinel 代理，自动转发 | 直连任意节点，自动 MOVED 重定向 |
| **故障转移** | Sentinel 监控 + 投票决定新主节点 | 从节点自动竞选 + 投票 |
| **垂直扩展** | 受单机内存限制 | 通过分片突破单机内存限制 |
| **跨 slot 操作** | ✅ 支持 | ❌ 不支持跨 slot MGET/MSET |
| **运维复杂度** | ⭐ 较低 | ⭐ 较高 |

```
┌─────────────────────────────────────────────────────────────────┐
│                      选型决策树                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  数据量 < 单机内存？                                              │
│       │                                                        │
│       ├── YES ──▶ 追求高可用？                                   │
│       │               │                                         │
│       │               ├── YES ──▶ Sentinel                      │
│       │               │                                         │
│       │               └── NO ──▶ 单机部署够用                    │
│       │                                                        │
│       └── NO ──▶ 数据量 > 单机内存？                             │
│                      │                                          │
│                      └── YES ──▶ Cluster                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> A: Sentinel 和 Cluster 的本质区别是"是否做数据分片"。Sentinel 只解决高可用问题，数据不分区，所有节点存储相同内容，写操作只在主节点。Cluster 同时解决高可用和数据分片，通过 16384 个 slot 把数据分散到多个主节点，突破单机内存限制，但代价是跨 slot 操作不支持、运维复杂度更高。

## 🔥 扩展追问 4：集群节点数量有限制吗？

> A: 理论上支持数千个节点，但节点越多 Gossip 通信量越大（每秒多次心跳，每个节点携带所有槽位信息）。建议生产环境控制在 100-200 个节点以内。另外有两个硬性要求：
> 1. **至少 3 个主节点**才能正常工作
> 2. **超过半数主节点可用**集群才能接收写请求（类似 Raft 多数派）
>
> 如果节点数太少（比如只有 2 主），一个节点挂了会导致集群不可写，因为无法达到 N/2+1 的多数派要求。

## 🔥 扩展追问 5：Hash Tag 解决跨 Slot 问题？

**问题场景：**

```java
// 假设需要批量获取用户的多条数据
String key1 = "user:10086:profile";
String key2 = "user:10086:cart";
String key3 = "user:10086:wishlist";

// 这三个 key 可能分布在不同的 slot
// MGET 需要所有 key 在同一 slot，否则报错
cluster.mget(key1, key2, key3);  // ❌ 可能失败
```

**解决：使用 Hash Tag**

```java
// Hash Tag：大括号内的部分决定 slot
String key1 = "user:{10086}:profile";   // slot = hash("10086")
String key2 = "user:{10086}:cart";      // slot = hash("10086")
String key3 = "user:{10086}:wishlist";  // slot = hash("10086")

// 现在三个 key 一定在同一 slot
cluster.mget(key1, key2, key3);  // ✅ 成功
```

```
┌─────────────────────────────────────────────────────────────────┐
│                      Hash Tag 原理                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  原始 key: "user:{10086}:profile"                              │
│                                                                 │
│  Redis 计算 slot 时：                                           │
│  1. 查找是否有 {}                                               │
│  2. 有 {}，取 {} 内的内容: "10086"                             │
│  3. CRC16("10086") % 16384 = slot                              │
│                                                                 │
│  没有 Hash Tag 时：                                             │
│  CRC16("user:10086:profile") % 16384 = slot                   │
│                                                                 │
│  ✅ Hash Tag 保证同一用户的所有 key 在同一 slot                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> A: Hash Tag 通过 `{}` 强制指定 slot 计算的内容。Redis 只对 `{}` 内的字符串计算 CRC16，适用于需要批量操作同一用户数据的场景。缺点是如果 Hash Tag 设计不当，可能导致 slot 分布不均匀（某些 slot 数据特别多）。

## 一句话总结（面试收尾用）

> **Redis Cluster 通过 CRC16(key) % 16384 实现数据分片，16384 个 slot 平均分配给各主节点；故障转移靠 Gossip 协议检测节点存活，超过半数主节点确认后从节点发起竞选获得 N/2+1 票升级为主节点；Sentinel 只解决高可用不分区，Cluster 同时解决高可用+数据分片但不支持跨 slot 操作；建议用 Hash Tag `{}` 解决跨 slot 批量操作问题。**
