## 🔥 扩展追问 1：synchronized 锁住的是什么？Monitor 是什么？

**synchronized 锁住的是对象的 Monitor（监视器锁）：**

```
每个 Java 对象都关联一个 Monitor：
        ↓
┌─────────────────────────────────────────────────────────┐
│  Monitor（对象监视器）                                   │
├─────────────────────────────────────────────────────────┤
│  Owner                                                  │
│  └─ 当前持有锁的线程 ID                                  │
├─────────────────────────────────────────────────────────┤
│  EntryList                                              │
│  └─ 等待获取锁的线程队列（BLOCKED 状态）                 │
├─────────────────────────────────────────────────────────┤
│  WaitSet                                                │
│  └─ 调用过 wait() 的线程队列（WAITING 状态）             │
├─────────────────────────────────────────────────────────┤
│  Recursion（锁重入计数）                                │
│  └─ 同一个线程重入次数                                   │
└─────────────────────────────────────────────────────────┘
```

**Monitor 的工作流程：**

```java
// 线程 A 进入 synchronized(this)
// Monitorenter 指令执行：

synchronized (this) {
    // ...
}

// 字节码层面：
// 0: monitorenter   ← 把 this 的 Monitor 的 Owner 设为当前线程
// ... 临界区代码 ...
// 1: monitorexit   ← 释放锁，Owner 设为 null，唤醒 EntryList 中的线程

// 执行流程：
// 线程 A：Monitor.Enter → Owner = 线程A → 执行临界区 → Monitor.Exit
// 线程 B：Monitor.Enter → 发现 Owner ≠ 自己 → 进入 EntryList 阻塞
// 线程 B：等线程 A 执行完 monitorexit → 被唤醒 → 尝试获取锁
```

**对象头的 Mark Word（32位/64位 JVM）：**

```
64位 JVM 的 Mark Word（对象头的一部分）：

┌──────────────────────────────────────────────────────────────────────┐
│ 无锁状态（NEW）                                                      │
│   hashcode(25bit) + age(4bit) + biased(1bit) + lock(2bit) = 64bit  │
│   lock: 01, biased: 0                                               │
├──────────────────────────────────────────────────────────────────────┤
│ 偏向锁                                                               │
│   thread(54bit) + epoch(2bit) + age(4bit) + biased(1bit) + lock(2bit)│
│   lock: 01, biased: 1                                               │
├──────────────────────────────────────────────────────────────────────┤
│ 轻量级锁                                                             │
│   ptr to LockRecord(62bit) + lock(2bit)                             │
│   lock: 00                                                          │
├──────────────────────────────────────────────────────────────────────┤
│ 重量级锁                                                             │
│   ptr to Monitor(62bit) + lock(2bit)                               │
│   lock: 10                                                          │
├──────────────────────────────────────────────────────────────────────┤
│ GC 标记                                                              │
│   empty(62bit) + lock(2bit)                                        │
│   lock: 11                                                          │
└──────────────────────────────────────────────────────────────────────┘
```

**面试加分话术：**

> "synchronized 锁的不是代码块本身，而是**对象的 Monitor 监视器**。每个 Java 对象都自带一个 Monitor（在对象头的 Mark Word 里），Monitor 有三个关键组件：**Owner** 记录谁在持锁、**EntryList** 排着等锁的线程、**WaitSet** 是调用 wait() 后释放锁的线程池。线程执行 `synchronized` 块时，JVM 底层执行 `monitorenter` 指令，把当前线程设为 Owner；出了块执行 `monitorexit`，Owner 归零，EntryList 里排队的线程被唤醒。"

## 🔥 扩展追问 2：synchronized 的锁升级过程？偏向锁→轻量级锁→重量级锁？

**锁升级是不可逆的：**

```
对象刚 new 出来：无锁状态
        ↓ 第一次获取锁
偏向锁（Biased Lock）
        ↓ 有其他线程来抢
轻量级锁（Lightweight Lock）
        ↓ 自旋超过阈值（默认 10 次）
重量级锁（Heavyweight Lock）

⚠️ 只能升级，不能降级
```

**偏向锁详解：**

```java
// 第一次被线程 A 获取时：
// Mark Word 里记录线程 A 的 ID（54bit）

Mark Word:
  [thread: 线程A的ID(54bit)] [epoch:0] [age:0] [biased:1] [lock:01]

// 之后线程 A 重入：
// 只比较线程 ID，相同就进入
// 不需要 CAS，直接比对 → 开销极小

// 线程 B 来抢：
// 比较 Mark Word 中的线程 ID，发现不是自己
// → 撤销偏向锁，膨胀为轻量级锁
```

**轻量级锁详解：**

```java
// 线程 A 获取轻量级锁：
// 1. 在线程栈帧中创建 Lock Record
// 2. 把 Mark Word 复制到 Lock Record（displaced header）
// 3. CAS 把 Mark Word 指向 Lock Record

// CAS 成功 → 线程 A 获取轻量级锁
// CAS 失败 → 自旋重试（默认自旋 10 次）

Lock Record 结构：
┌─────────────────────────────────────────┐
│ Owner: null                             │
│ Displaced Header: 原 Mark Word 的值       │
└─────────────────────────────────────────┘

Mark Word (轻量级锁状态)：
┌─────────────────────────────────────────┐
│ ptr to Lock Record (62bit) + lock: 00   │
└─────────────────────────────────────────┘
```

**自旋锁的问题：**

```java
// 自旋次数默认 10 次（-XX:PreBlockSpin）
// JDK 1.8 引入自适应自旋：
// 如果之前自旋成功率高 → 多自旋几次
// 如果之前自旋失败率高 → 减少自旋次数

// 自旋超过阈值 → 膨胀为重量级锁
// 线程被 OS 挂起（不消耗 CPU）
```

**重量级锁：**

```java
// 重量级锁 = Monitor = OS 互斥锁（Mutex）
// 线程不是忙等（自旋），而是阻塞（sleep）

// 线程 B 尝试获取重量级锁：
// 1. 进入 EntryList（BLOCKED 状态）
// 2. OS 将线程 B 挂起（不消耗 CPU）
// 3. 线程 A 释放锁，唤醒 EntryList 中的线程
// 4. 线程 B 被唤醒，尝试获取锁

// ⚠️ 重量级锁开销大：
//   - 用户态 → 内核态切换（一次约 1-5μs）
//   - 线程挂起 + 唤醒（一次约 10-100μs）
//   - 不适合临界区很长的场景
```

**锁升级流程图：**

```
┌─────────────────────────────────────────────────────────────────┐
│  synchronized(obj)                                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  无锁 → 检查 Mark Word                                          │
│     ↓                                                           │
│  偏向锁（第一次）→ 记录线程 ID                                   │
│     ↓ 有竞争                                                     │
│  轻量级锁（CAS 自旋）→ 每次重入 CAS                             │
│     ↓ 自旋超过阈值                                              │
│  重量级锁（OS 互斥）→ 线程阻塞                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**面试加分话术：**

> "synchronized 的锁升级是 JVM 的自适应优化：先给锁加**偏向**（一个人用不加锁开销），有人来抢就**轻量级**（CAS 自旋，不用 OS 挂起），自旋等不到就**膨胀**为重量级（真正的互斥锁，线程阻塞）。三个阶段各有优势：偏向锁几乎零开销、单线程场景最优；轻量级锁避免了线程切换、适合短时竞争；重量级锁在长竞争时稳得住。锁升级是**单向不可逆**的，所以线上不要轻易开偏向锁（JDK 1.6 默认开，如果并发度高可以关掉 `-XX:-UseBiasedLocking`）。"

## 🔥 扩展追问 3：synchronized 和 Lock（ReentrantLock）的区别？

**核心区别表：**

| 特性 | synchronized | ReentrantLock |
|---|---|---|
| **加锁方式** | 隐式（自动获取/释放） | 显式（手动 lock/unlock） |
| **超时等待** | ❌ 不支持 | ✅ `tryLock(timeout)` |
| **可中断** | ❌ 不支持 | ✅ `lockInterruptibly()` |
| **公平/非公平** | ❌ 非公平 | ✅ 可选两种 |
| **多条件** | ❌ 单一 wait/notify | ✅ 多个 Condition |
| **锁粒度** | 粗（整个方法/代码块） | 细（可以精确控制） |
| **性能** | JDK 6+ 已大量优化 | 好 |

**代码对比：**

```java
// synchronized：隐式锁，自动释放
public synchronized void method() {
    // 临界区
}  // 自动释放锁

// ✅ synchronized 的优势：代码简洁，不会忘记释放

// ReentrantLock：显式锁，必须 try-finally
private final ReentrantLock lock = new ReentrantLock();

public void method() {
    lock.lock();
    try {
        // 临界区
    } finally {
        lock.unlock();  // 必须手动释放，容易漏
    }
}

// ✅ ReentrantLock 的优势：可超时、可中断
public void tryWithTimeout() {
    try {
        if (lock.tryLock(3, TimeUnit.SECONDS)) {  // 超时
            try {
                // 业务逻辑
            } finally {
                lock.unlock();
            }
        } else {
            // 超时后的降级逻辑
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
}
```

**Condition（多条件等待）：**

```java
// synchronized 只能用 wait/notifyAll（单一条件）
synchronized (lock) {
    while (condition != desired) {
        lock.wait();  // 只有一个等待队列
    }
}

// ReentrantLock 可以有多个 Condition
private final ReentrantLock lock = new ReentrantLock();
private final Condition conditionA = lock.newCondition();
private final Condition conditionB = lock.newCondition();

lock.lock();
try {
    // 等待条件 A
    while (!readyA) {
        conditionA.await();  // 只唤醒等待 conditionA 的线程
    }
    
    // 或者等待条件 B
    while (!readyB) {
        conditionB.await();  // 只唤醒等待 conditionB 的线程
    }
    
    // 业务逻辑...
    
    // 唤醒特定条件的线程
    conditionA.signal();  // 比 notifyAll 更精确
    conditionB.signal();
} finally {
    lock.unlock();
}
```

**公平锁 vs 非公平锁：**

```java
// ReentrantLock 可以选公平锁（按等待顺序）
ReentrantLock fairLock = new ReentrantLock(true);  // 公平

// synchronized 永远是非公平：
// 新来的线程可能直接抢锁，不排队

// 公平锁适用场景：
// - 低并发、无饥饿问题
// - 关键业务不容许插队

// 非公平锁适用场景（synchronized 和默认 ReentrantLock）：
// - 高并发、吞吐量优先
// - 临界区短（减少线程切换开销）
```

**面试加分话术：**

> "synchronized 和 ReentrantLock 的选择，我的经验是：**能用 synchronized 就不用 Lock**，因为 synchronized 是 JVM 内置的，编译器会做优化（锁消除、锁粗化），而 ReentrantLock 是 SDK 层面的，要手动 unlock，容易漏。但如果需要**超时、中断、公平锁、多条件**这些功能，就用 ReentrantLock。比如批量订单处理，需要可中断的锁+超时降级，就用 `tryLock(timeout)`；比如需要精确控制 wait/notify 的场景，用 `Condition` 比 `wait/notify` 更清晰。JDK 1.6 之后 synchronized 做了大量优化，在低并发场景下性能已经和 ReentrantLock 持平。"

## 🔥 扩展追问 4：什么是锁消除和锁粗化？JIT 编译器的优化？

**锁消除（Lock Elision）：**

```java
// JIT 编译器会分析：锁对象是否逃逸出方法？
public void append(StringBuilder sb) {
    sb.append("hello");  // sb 是局部变量，不会逃逸
}

public void test() {
    StringBuilder sb = new StringBuilder();
    append(sb);  // sb 不会逃逸到其他线程
}

// 编译优化后：
// JIT 发现 sb 是局部变量，不可能被其他线程访问
// → 自动消除 synchronized
// 原来：synchronized(sb) { sb.append(...); }
// 优化后：无锁直接执行 append()

// 逃逸分析（Escape Analysis）：
// 如果锁对象不会逃逸出线程，锁就是多余的
// - 局部变量（不逃逸）
// - 返回值（逃逸）
// - 传入其他线程（逃逸）
```

**锁粗化（Lock Coarsening）：**

```java
// ❌ 频繁加锁解锁
public String badConcat(String[] parts) {
    StringBuilder sb = new StringBuilder();
    for (String part : parts) {
        synchronized (sb) {    // 每次循环都要加锁解锁
            sb.append(part);
        }
    }
    return sb.toString();
}

// ✅ 锁粗化优化后
public String goodConcat(String[] parts) {
    StringBuilder sb = new StringBuilder();
    synchronized (sb) {        // JIT 把锁合并，只加一次
        for (String part : parts) {
            sb.append(part);
        }
    }
    return sb.toString();
}

// 合并的临界区，减少加锁/解锁次数
```

**JIT 编译器的 synchronized 优化总结：**

```
┌──────────────────────────────────────────────────────────────┐
│  JIT 编译优化                                                  │
├──────────────────────────────────────────────────────────────┤
│  逃逸分析（Escape Analysis）                                  │
│  └─ 锁对象不逃逸 → 锁消除（无锁执行）                         │
├──────────────────────────────────────────────────────────────┤
│  锁粗化（Lock Coarsening）                                    │
│  └─ 多次相邻加锁 → 合并成一次                                 │
├──────────────────────────────────────────────────────────────┤
│  锁升级（Lock Escalation）                                    │
│  └─ 同一个对象频繁加锁 → 升级为重量级锁（不可逆）            │
├──────────────────────────────────────────────────────────────┤
│  偏向锁优化（Biased Locking）                                 │
│  └─ 单线程重复加锁 → 只比较线程 ID，不 CAS                     │
└──────────────────────────────────────────────────────────────┘
```

**逃逸分析代码验证：**

```java
// 场景1：不逃逸 → 锁消除
public void test1() {
    Object obj = new Object();  // 局部变量
    synchronized (obj) {
        // obj 不逃逸，锁可以消除
    }
}

// 场景2：逃逸 → 不能消除
public Object test2() {
    Object obj = new Object();
    synchronized (obj) {
        return obj;  // obj 作为返回值，逃逸了
    }
}

// 场景3：传入其他线程 → 不能消除
public void test3(Object param) {
    synchronized (param) {
        // param 可能是其他线程传入，逃逸了
    }
}
```

**面试加分话术：**

> "JIT 编译器对 synchronized 的优化主要有两条：**锁消除**和**锁粗化**。锁消除基于逃逸分析——如果锁对象不会逃出方法（比如只是局部变量），那锁就是多余的，直接去掉。锁粗化是反过来的优化——如果一段代码里频繁加锁解锁（比如循环里加锁），JIT 会把它们合并成一个大锁，减少加解锁开销。JDK 1.8 默认开启这些优化。另外说一个面试常问的：**StringBuffer 用了 synchronized，StringBuilder 没用，性能上 StringBuilder 更快**——就是因为 synchronized 多了锁开销（在 JIT 优化之前），但 JDK 1.6 之后差距已经很小了。"

## 🔥 扩展追问 5：synchronized 同步方法 vs 同步代码块，哪个更好？

**两者的区别：**

```java
// 同步方法（方法级锁）
public synchronized void method() {
    // 等价于 synchronized(this)
}

// 同步代码块（对象级锁）
public void method() {
    synchronized (this) {
        // 只锁这一块代码
    }
}

// 同步代码块（类级锁）
public static synchronized void staticMethod() {
    // 等价于 synchronized(类.class)
}

public void method() {
    synchronized (类.class) {
        // 锁住整个类
    }
}
```

**为什么说同步代码块更好：**

```java
// ❌ 同步方法：锁粒度大，整个方法都锁住
public synchronized void processOrder(Order order) {
    validate(order);        // 验证 → 不需要锁
    calculatePrice(order);  // 计算 → 不需要锁
    saveToDb(order);        // 保存 → 才需要锁
    sendNotification(order); // 通知 → 不需要锁
    // 整个方法持有锁，其他线程全等着
}

// ✅ 同步代码块：锁粒度小，只锁必要的部分
public void processOrder(Order order) {
    validate(order);        // 无锁
    calculatePrice(order);  // 无锁
    
    synchronized (this) {   // 只锁这一块
        saveToDb(order);    // 关键操作加锁
    }                       // 其他线程可以并行做前面的步骤
    
    sendNotification(order); // 无锁
}

// ✅ 锁 this 的问题：其他同步方法也会被阻塞
// 如果 processOrder 和另一个同步方法都锁 this
// → 两个方法互斥，即使逻辑上不冲突
```

**不同锁对象的对比：**

```java
public class Counter {
    private final Object countLock = new Object();  // 专用锁对象
    private int count = 0;
    
    // ✅ 最佳实践：用单独的锁对象
    public void increment() {
        synchronized (countLock) {  // 只锁 count 相关操作
            count++;
        }
    }
    
    // ✅ 其他方法完全不受影响
    public void print() {
        System.out.println(count);  // 无锁读取
    }
}

// ❌ 锁 this：阻塞所有同步方法
class BadCounter {
    public synchronized void method1() { }
    public synchronized void method2() { }  // method1 在执行时，method2 也要等
}
```

**锁粒度选择原则：**

```
锁粒度越细 → 并发度越高 → 复杂度越高
        ↓
┌─────────────────────────────────────────────────────────┐
│ 同步方法（锁 this / 类.class）                           │
│   ├─ 优点：代码简洁                                      │
│   ├─ 缺点：粒度粗，影响并发                              │
│   └─ 适用：整个方法都需要同步、逻辑简单                   │
├─────────────────────────────────────────────────────────┤
│ 同步代码块（锁特定对象）                                  │
│   ├─ 优点：粒度细，并发度高                              │
│   ├─ 缺点：需要找合适的锁对象                            │
│   └─ 适用：只有部分代码需要同步、并发性能敏感             │
├─────────────────────────────────────────────────────────┤
│ 静态同步方法（锁类.class）                               │
│   ├─ 特点：所有实例共享同一把锁                          │
│   └─ 适用：静态变量需要同步                              │
└─────────────────────────────────────────────────────────┘
```

**面试加分话术：**

> "**同步代码块优于同步方法**，因为锁粒度更可控。同步方法锁的是 `this`（实例方法）或 `类.class`（静态方法），粒度太大。同步代码块可以锁特定对象，比如锁 `countLock` 只保护计数器，锁 `orderMap` 只保护订单缓存，其他方法完全不受影响。最佳实践是**尽量用同步代码块，锁专门的对象**，不要锁 `this`（会阻塞其他正常方法）。Spring 框架里的 `StringRedisTemplate` 就是用专门的 `ReentrantLock` 保护 Redis 连接，粒度控制得很细。"

## 🔥 扩展追问 6：synchronized 和 volatile 的区别？两者能替代吗？

**volatile 的作用：**

```java
// volatile 保证两点：
// 1. 可见性：写操作立刻刷新到主内存，读操作从主内存读取
// 2. 禁止指令重排序

volatile boolean flag = false;

// 线程 A 写
flag = true;  // 立刻刷新到主内存，其他线程立刻可见

// 线程 B 读
while (!flag) {  // 每次都从主内存读
    // ...
}
```

**synchronized vs volatile 对比：**

| 维度 | synchronized | volatile |
|---|---|---|
| **原子性** | ✅ 保证（整个代码块原子） | ❌ 不保证（只能保证单个读/写） |
| **可见性** | ✅ 保证（unlock 前刷新主内存） | ✅ 保证（写屏障+读屏障） |
| **有序性** | ✅ 保证（as-if-serial） | ⚠️ 部分保证（禁止指令重排） |
| **性能** | 较重（阻塞线程） | 极轻（无锁） |
| **适用场景** | 复合操作 | 单个变量读写 |

**volatile 不能替代 synchronized 的例子：**

```java
// ❌ volatile 无法保证原子性
volatile int count = 0;

// 线程 A 和线程 B 同时执行
count++;  // 不是原子操作！

// 实际执行顺序（可能）：
// 线程A: 读取 count=0 → 寄存器+1 → 写回 count=1
// 线程B: 读取 count=0 → 寄存器+1 → 写回 count=1  ← 线程B读到的是旧值！
// 最终结果：count=1（期望是2）→ 数据丢失！
```

**必须用 synchronized 的场景：**

```java
// ✅ 复合操作：必须 synchronized
private int count = 0;

public synchronized void increment() {  // 或者用 AtomicInteger
    count++;  // 读-改-写三部曲
}

// ✅ 多步操作：必须 synchronized
public synchronized void transfer(Account from, Account to, double amount) {
    from.withdraw(amount);  // 步骤1
    to.deposit(amount);     // 步骤2
    // 两步必须原子完成
}
```

**volatile 适合的场景：**

```java
// ✅ 单一标志位：volatile 足够
volatile boolean shutdown = false;

public void shutdown() {
    shutdown = true;
}

public void run() {
    while (!shutdown) {
        // 处理任务
    }
}

// ✅ 双重检查锁定（DCL）
private volatile Singleton instance;  // 必须加 volatile！

public Singleton getInstance() {
    if (instance == null) {           // 第一次检查
        synchronized (Singleton.class) {
            if (instance == null) {   // 第二次检查
                instance = new Singleton();  // volatile 防止指令重排
            }
        }
    }
    return instance;
}

// ✅ 配合 CAS 使用
private volatile AtomicInteger ai = new AtomicInteger(0);
```

**指令重排的问题（为什么 volatile 能禁止）：**

```java
// 问题：new Singleton() 可能被重排
instance = new Singleton();

// 重排后：
// 1. 分配内存
// 2. 调用构造
// 3. 赋值引用（这个最重要！）

// 如果重排为 1 → 3 → 2：
// 其他线程在第 3 步后认为 instance != null
// 但实际上对象还没构造完成！
// → 读取到半成品对象（未初始化字段有默认值）

// volatile 禁止这个重排：
// 写 volatile：写屏障 =storeload（后续的读写必须在这之后）
// 读 volatile：读屏障 =loadacquire（之前的读写必须在这之前）
```

**面试加分话术：**

> "synchronized 和 volatile 常被混用，但它们解决的是不同问题：**volatile 只保证可见性和禁止重排，不保证原子性**；**synchronized 三样都保证**。简单记法：**单一变量的读或写用 volatile，复合操作必须 synchronized**。最经典的坑是 `volatile int count; count++`——这个不是原子操作，必须用 `synchronized` 或 `AtomicInteger`。另外 volatile 的开销极低（一次内存屏障），比 synchronized 轻 20~100 倍，所以标志位、状态位用 volatile 足够。"

## 🔥 扩展追问 7：synchronized 的可重入性是怎么实现的？

**可重入锁的概念：**

```java
// 同一个线程可以多次获取同一把锁
synchronized (this) {
    synchronized (this) {  // 同一个线程，再次进入
        synchronized (this) {  // 还可以再进
            // 业务逻辑
        }
    }
}

// 如果不可重入，这段代码会死锁
// synchronized 天然可重入，JVM 维护了锁的持有计数
```

**可重入的实现原理：**

```
线程 A 获取锁 → Owner = 线程A，Recursion = 1
        ↓ 再次获取同一把锁
线程 A 再次获取 → Recursion = 2
        ↓ 第 N 次
线程 A 再次获取 → Recursion = N
        ↓ 释放一次
线程 A 释放锁 → Recursion = N-1（还不是 0，不真正释放）
        ↓ 释放到 0
线程 A 完全释放 → Owner = null，唤醒 EntryList 中的线程
```

**锁记录（Lock Record）和可重入：**

```java
// 可重入的关键：Lock Record 和 displaced header
// 
// 第一次获取锁（偏向锁）：
// Mark Word → 记录线程 ID

// 第二次获取（轻量级锁）：
// 线程栈帧创建 Lock Record
// displaced header = 第一次的 Mark Word（包含线程ID）
// Mark Word → 指向新的 Lock Record
// 每次重入，创建一个新的 Lock Record

// 释放时：
// 每次 monitorexit，弹出最新的 Lock Record
// 直到所有 Lock Record 都弹出，才真正释放锁

// 线程栈帧（可重入时的结构）：
// Lock Record 1: displaced_header = 原 Mark Word
// Lock Record 2: displaced_header = (偏向锁状态)
// Lock Record 3: displaced_header = ...
// → Recursion = 3
```

**ReentrantLock 的可重入：**

```java
// ReentrantLock 也是可重入的
ReentrantLock lock = new ReentrantLock();

lock.lock();           // 第 1 次
try {
    lock.lock();       // 第 2 次（可重入）
    try {
        lock.lock();   // 第 3 次
        try {
            // 业务逻辑
        } finally {
            lock.unlock();  // 第 3 次释放
        }
    } finally {
        lock.unlock();  // 第 2 次释放
    }
} finally {
    lock.unlock();     // 第 1 次释放
}

// ReentrantLock 内部维护了持有计数
// sync.exclusiveOwnerThread = 当前线程
// getHoldCount() = 当前线程持有锁的次数
```

**不可重入锁的例子（容易搞混）：**

```java
// 不可重入锁：同一个线程不能重复获取
public class NonReentrantLock {
    private boolean locked = false;
    
    public synchronized void lock() {
        if (locked) {
            // 如果是同一个线程，也要等 → 死锁！
            throw new RuntimeException("Lock already held!");
        }
        locked = true;
    }
    
    public synchronized void unlock() {
        locked = false;
    }
}

// ✅ synchronized 是可重入的，不会死锁
// ❌ 手动实现的互斥锁如果处理不当，可能不可重入
```

**可重入的意义：**

```
可重入的好处：
        ↓
1. 避免死锁：递归调用时不会卡死
2. 简化代码：同一个方法可以递归调用
3. 支持父子方法都加锁

不可重入的代价：
        ↓
synchronized methodA() {
    methodB();  // 如果 B 也用 synchronized，会死锁
}
```

**面试加分话术：**

> "synchronized 是**可重入锁**，同一个线程可以多次获取同一把锁，JVM 靠**锁记录（Lock Record）**实现：每次重入在栈帧创建一个 Lock Record，displaced header 保存之前的 Mark Word，持有计数 +1。释放时弹出一个 Lock Record，计数 -1，直到归零才真正释放。ReentrantLock 也是可重入的，内部用 `state` 字段计数。这和不可重入锁（如某些操作系统 mutex）形成对比——不可重入的锁如果同一个线程两次加锁，会死锁。所以如果自己写互斥锁，要注意实现可重入逻辑。"

## 🔥 扩展追问 8：wait/notify/notifyAll 和 synchronized 的关系？

**wait/notify 必须配合 synchronized 使用：**

```java
// ❌ 错误：不在 synchronized 里调用 wait/notify
// 编译报错：IllegalMonitorStateException
obj.wait();   // 必须先 synchronized(obj)

// ✅ 正确：
synchronized (obj) {
    while (condition != true) {
        obj.wait();  // 释放 obj 的锁，进入 WaitSet
    }
    // ...
}

// 另一个线程：
synchronized (obj) {
    condition = true;
    obj.notify();  // 随机唤醒 WaitSet 中的一个线程
    // 或者 obj.notifyAll() 唤醒所有
}
```

**wait/notify 的状态流转：**

```
线程 T 执行 synchronized(obj)：
        ↓
获取锁，进入临界区
        ↓
调用 obj.wait()：
  ① 加入 WaitSet
  ② 释放 obj 的锁（关键！其他线程可以进来了）
  ③ 阻塞（WAITING 状态）
        ↓
线程 T 等待被唤醒：
  · notify() → 从 WaitSet 取出，放入 EntryList（BLOCKED）
  · notifyAll() → 所有 WaitSet 线程都移到 EntryList
        ↓
线程 T 抢到锁（某个时刻）：
  继续从 wait() 下一行执行
```

**生产者-消费者模式：**

```java
// 仓库
public class Warehouse {
    private final int MAX = 100;
    private int count = 0;
    
    public synchronized void produce(int n) {
        while (count + n > MAX) {
            try {
                wait();  // 仓库满了，等消费
            } catch (InterruptedException e) {}
        }
        count += n;
        notifyAll();  // 通知消费者
    }
    
    public synchronized int consume(int n) {
        while (count < n) {
            try {
                wait();  // 库存不够，等生产
            } catch (InterruptedException e) {}
        }
        count -= n;
        notifyAll();  // 通知生产者
        return n;
    }
}
```

**notify vs notifyAll 的区别：**

```java
// notify()：随机唤醒一个等待者
// 问题：可能唤醒错的人（唤醒同类）→ 所有线程都醒着等

synchronized (lock) {
    while (condition == 0) {
        lock.wait();  // 所有消费者都等
    }
    // ...
    lock.notify();  // 唤醒一个消费者
}

// 如果生产者调用 notify：
// 可能唤醒另一个生产者（而不是消费者）
// → 所有消费者还在等，生产者已经走了 → 死锁！

// ✅ notifyAll()：唤醒所有等待者
// 确保至少有一个能处理
synchronized (lock) {
    // ...
    lock.notifyAll();  // 唤醒所有，消费者和生产者都醒
}

// 建议：优先用 notifyAll，简单不容易出错
// notify 适用于：等待条件互斥、唤醒者和等待者是同质的
```

**虚假唤醒（Spurious Wakeup）：**

```java
// 虚假唤醒：wait() 可能无原因被唤醒
// JDK 文档明确说了，wait() 必须在循环里调用

// ❌ 错误写法
if (count > 0) {  // 如果被虚假唤醒，可能 count 还是 0
    count--;
    return;
}

// ✅ 正确写法
while (count == 0) {  // 醒来后重新检查条件
    lock.wait();
}
count--;
```

**完整模式：标准模板：**

```java
public synchronized void doWhenReady() {
    while (!ready) {           // 循环检查（防止虚假唤醒）
        try {
            wait();           // 释放锁，等待
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
    // 条件满足，执行逻辑
}

public synchronized void setReady() {
    ready = true;
    notifyAll();               // 通知所有等待者（优先用 notifyAll）
}
```

**面试加分话术：**

> "wait/notify 是配合 synchronized 的线程通信机制，**必须在 synchronized 块里调用**，否则抛 `IllegalMonitorStateException`。wait() 会**释放锁**并进入 WaitSet，notify() 只唤醒一个，notifyAll() 唤醒所有。面试常问：**为什么 wait 要放在 while 循环里？** ——因为存在**虚假唤醒**（Spurious Wakeup），JDK 文档明确说了 wait 可能无原因返回，必须重新检查条件。另外记住：**notify() 比 notifyAll() 风险大**，除非你确定等待条件是互斥的，否则优先用 notifyAll()。实际开发中更推荐用 `Condition` 的 `await()/signal()`，更灵活，可以多个条件队列，不像 wait/notify 只有单一队列。"

## 一句话总结（面试收尾用）

> "synchronized 底层靠对象头的 **Mark Word** 和 **Monitor 监视器** 实现：Monitor 有 Owner、EntryList、WaitSet 三个组件，分别记录持锁线程、等待线程、wait 线程。锁升级分三个阶段：**偏向锁**（单线程极轻量）、**轻量级锁**（CAS 自旋）、**重量级锁**（OS 互斥）。JDK 1.6 之后做了大量优化（锁消除、锁粗化、适应性自旋），日常开发中 synchronized 性能已经不输 ReentrantLock，能用 synchronized 的优先用它，需要超时/中断/多条件时才用 Lock。"
