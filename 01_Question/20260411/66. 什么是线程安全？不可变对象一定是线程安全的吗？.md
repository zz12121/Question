### 基础流程

```
什么是线程安全？
        ↓
定义：多个线程同时访问某个类/对象/方法
      不会导致数据破坏、状态错误
        ↓
线程不安全的原因：
      ├─ 竞态条件（Race Condition）
      ├─ 共享变量可见性
      └─ 原子性问题
        ↓
线程安全的实现：
      ├─ 不可变对象（Immutable）
      ├─ 同步（synchronized）
      ├─ 并发工具（Lock、Atomic*）
      └─ 无共享（ThreadLocal）
```

**最简单的线程安全例子：**

```java
// ❌ 线程不安全
class Counter {
    private int count = 0;
    public void increment() { count++; }  // count++ 不是原子的
}

// ✅ 线程安全：不可变
class ImmutableCounter {
    private final int count;  // final，不可变
    public ImmutableCounter(int count) { this.count = count; }
    public ImmutableCounter increment() {
        return new ImmutableCounter(count + 1);  // 返回新对象
    }
}

// ✅ 线程安全：同步
class SafeCounter {
    private int count = 0;
    public synchronized void increment() { count++; }
}
```

## 🔥 扩展追问 1：什么是竞态条件？举几个常见的例子？

**竞态条件的本质：**

```
竞态条件（Race Condition）：
        ↓
多个线程对共享资源的访问顺序
依赖 Timing（时序）
        ↓
结果正确性不可预测
```

**常见竞态条件类型：**

```java
// 类型1：Check-Then-Act（先检查再行动）
if (obj == null) {    // 线程 A、B 同时检查，都发现是 null
    obj = new Obj();  // 线程 A、B 都创建了对象
}                      // obj 最终可能只有一个被使用，或者两个都被覆盖

// 类型2：Read-Modify-Write（读-改-写）
count++;  // 不是原子操作！
// 实际分解：1. 读取 count  2. count+1  3. 写回 count
// 两个线程同时读到一个旧值，结果就不对了

// 类型3：先增后检查
sequence++;
if (sequence == 100) {  // 可能多个线程同时超过 100
    // 发奖、关门等操作被触发多次
}
```

**实际业务场景中的竞态：**

```java
// 场景1：库存扣减
public boolean deductStock(Long productId, int quantity) {
    Product p = productMapper.selectById(productId);
    if (p.getStock() >= quantity) {       // 检查
        p.setStock(p.getStock() - quantity);
        productMapper.updateById(p);        // 改写
        return true;
    }
    return false;
}
// ❌ 两个请求同时来，都查到库存够，都扣了 → 超卖！

// 场景2：订单重复提交
public void createOrder(Order order) {
    if (orderMapper.exists(order.getId())) {  // 检查是否存在
        return;  // 认为重复了
    }
    orderMapper.insert(order);  // 但两个请求可能同时通过检查
}
// ❌ 重复订单！

// 场景3：用户登录次数限制
if (loginFailCount < 5) {
    loginFailCount++;  // 非原子！
}
// ❌ 多个并发请求可能绕过限制
```

**面试加分话术：**

> "竞态条件就是**执行结果依赖线程执行的先后顺序**。最经典的例子就是 i++——看似一行代码，实际上是"读取→修改→写回"三步。如果两个线程同时读到旧值，就会有一次自增丢失。排查竞态条件有个技巧：**找到所有对共享变量的 read 和 write，然后问自己：如果线程 A 在线程 B 读写之间穿插进来会怎样？** 能回答出来的地方，就是潜在的竞态条件。"

## 🔥 扩展追问 2：可见性和原子性有什么区别？happens-before 原则是什么？

**可见性 vs 原子性：**

```
┌─────────────────────────────────────────────────────────┐
│ 可见性：线程 A 修改了共享变量，线程 B 能看到最新值吗？     │
├─────────────────────────────────────────────────────────┤
│ 原子性：某个操作要么全部完成，要么完全不执行              │
│         中间不能被其他线程打断                           │
└─────────────────────────────────────────────────────────┘
```

**可见性问题演示：**

```java
// 可见性问题
class VisibilityDemo {
    private boolean flag = false;
    private int x = 0;
    
    // 线程 A 执行
    public void writer() {
        x = 1;          // 1. 先写 x
        flag = true;    // 2. 再写 flag
    }
    
    // 线程 B 执行
    public void reader() {
        if (flag) {      // 看到 flag=true
            System.out.println(x);  // x 一定是 1 吗？
        }
    }
}

// 问题：JIT 编译器可能优化重排
// 线程 B 可能看到 flag=true，但 x 还是 0
// 因为 flag 和 x 写入顺序可能被重排
```

**happens-before 原则（8条核心规则）：**

```java
// 规则1：程序顺序规则
// 同一个线程中，前面的操作 happens-before 后面的
x = 1;
y = 2;  // y = 2 happens-before y 读不到 x=1 的情况吗？是的

// 规则2：监视器锁规则
// 锁的解锁 happens-before 后续的加锁
synchronized (lock) {
    x = 1;  // happens-before
}
synchronized (lock) {
    int v = x;  // 一定能读到 x=1
}

// 规则3：volatile 变量规则
// volatile 写 happens-before 后续的 volatile 读
volatile boolean ready = true;  // writer

while (ready) {  // reader - 一定能读到最新值
    int x = data;
}

// 规则4：线程启动规则
// Thread.start() happens-before 被启动线程中的任何操作
Thread t = new Thread(() -> x = 1);
x = 100;    // A：主线程
t.start(); // B：启动子线程
// B 一定能读到 x=100（因为 start happens-before 线程内操作）

// 规则5：线程终止规则
// 线程中的所有操作 happens-before 其他线程 join 返回
t.join();
// 主线程从这里继续执行时，能看到 t 线程的所有操作结果

// 规则6：传递性规则
// A happens-before B，B happens-before C → A happens-before C
```

**面试加分话术：**

> "happens-before 是 JVM 规定的**可见性保障**——满足这个关系的操作对，后面的操作一定能"看到"前面的结果。面试常问的是 volatile 的 happens-before 语义：**volatile 写 happens-before volatile 读**，所以读写不会被重排到临界区外面，这就是它保证可见性的底层原理。理解这个规则后，像双重检查锁（DCL）为什么要加 volatile、`Thread.sleep()` 为什么不能保证可见性这些问题就都清楚了。"

## 🔥 扩展追问 3：synchronized 关键字的原理？锁升级过程？

**synchronized 的原理：**

```
synchronized(obj)
        ↓
检查 obj 的 Mark Word（对象头）
        ↓
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
        ↓
加锁成功：记录线程 ID / 指向栈帧的锁记录
加锁失败：进入 EntryList 等待，阻塞线程
```

**锁升级过程（不可逆）：**

```
无锁状态（对象刚 new）
        ↓ 第一次获取锁
偏向锁（Mark Word 记录线程 ID）
        ↓ 有竞争（其他线程尝试获取）
轻量级锁（CAS 抢锁，自旋等待）
        ↓ 自旋超过阈值（默认 10 次）
重量级锁（操作系统 Mutex，线程阻塞）
```

**代码对应原理：**

```java
// 偏向锁：第一次被持有时，记录线程 ID
synchronized (this) {
    // 如果没有竞争，Mark Word 里记录了当前线程 ID
    // 同一个线程重入，只需要比较线程 ID，不用 CAS
}

// 轻量级锁：CAS 自旋
// 线程在栈帧创建 Lock Record
// CAS 把 Mark Word 复制到 Lock Record
// CAS 把 Mark Word 指向栈帧的 Lock Record
// 如果 CAS 失败，说明有竞争，开始自旋

// 重量级锁：系统调用
// 自旋超过阈值，膨胀为重量级锁
// 线程被操作系统阻塞（挂起）
// 持有锁的线程释放时，唤醒等待的线程
```

**不同加锁方式的升级：**

```java
// 锁对象（对象头里的 Mark Word）
synchronized (this) { }  // 偏向锁开始

// 锁 Class 对象（类级别的锁）
synchronized (Singleton.class) { }  // 直接是类锁，类对象上的锁

// 锁静态方法
public static synchronized void method() { }
// 等价于 synchronized (ClassName.class)
```

**面试加分话术：**

> "synchronized 锁升级是**逐步升级、不可逆**的过程：先偏向（一个人用，减少 CAS 开销），有竞争就膨胀到轻量级（自旋等，不用挂起线程），自旋耗尽就变成重量级（OS 互斥锁，线程挂起）。这个设计是为了**锁开销和竞争程度的匹配**——没有竞争就用最轻量的偏向锁，有短时竞争就用自旋，避免线程切换开销，只有长时竞争才用重量级锁。面试官可能追问：为什么 synchronized 不能指定锁升级策略？——因为是 JVM 自动优化的，Java 6 开始加入的偏向锁和轻量级锁，代码写法不变，性能自动提升。"

## 🔥 扩展追问 4：不可变对象一定是线程安全的吗？为什么？

**结论：是的，不可变对象一定是线程安全的。**

**为什么？**

```
不可变对象的特点：
        ↓
1. 所有字段都是 final（构造后就不可变）
2. 没有提供 setter 或其他修改方法
3. 对象创建后状态不可变
        ↓
既然不可变：
  ├─ 多个线程同时读 → 结果相同 ✅
  ├─ 没有线程能修改 → 没有竞态条件 ✅
  └─ 没有可见性问题 → 读到的都是同一个对象 ✅
```

**不可变类的正确写法：**

```java
// ✅ 正确的不可变类
public final class ImmutableUser {
    private final String name;     // final 字段
    private final int age;         // 基本类型天然不可变
    private final List<String> tags; // 引用类型：需要防内部修改
    
    public ImmutableUser(String name, int age, List<String> tags) {
        this.name = name;
        this.age = age;
        // 防御性拷贝：复制一份，防止调用者后续修改原 list
        this.tags = new ArrayList<>(tags);
    }
    
    // 只提供 getter，不提供 setter
    public String getName() { return name; }
    public int getAge() { return age; }
    
    // 返回防御性拷贝，不直接返回内部引用
    public List<String> getTags() {
        return new ArrayList<>(tags);
    }
}
```

**容易犯的错误：**

```java
// ❌ 错误1：字段没有用 final
class NotImmutable {
    private String name;  // 没有 final，可变！
    // ...
}

// ❌ 错误2：可变类型字段没有防御性拷贝
class BadImmutable {
    private final List<String> items = new ArrayList<>();
    
    public List<String> getItems() {
        return items;  // ❌ 直接返回内部引用！
    }
    
    // 调用者可以修改
    // immutable.getItems().add("hacked!");  ← 破坏了不可变性
}

// ❌ 错误3：可变对象作为参数传入
class BadImmutable2 {
    private final List<String> list;
    
    public BadImmutable2(List<String> list) {
        this.list = list;  // ❌ 没有拷贝！
    }
}

// 外部修改
List<String> original = new ArrayList<>();
original.add("a");
BadImmutable2 obj = new BadImmutable2(original);
original.add("b");  // obj 的 list 也被改了！
```

**不可变对象 + final 的真正含义：**

```
┌─────────────────────────────────────────────────────────┐
│ final 在多线程中的作用：                                 │
├─────────────────────────────────────────────────────────┤
│ final 字段：构造后不可变                                │
│     └─ 构造器中如果有 this 逸出，final 可能失效          │
│                                                          │
│ 引用不可变 ≠ 对象内容不可变！                            │
│     └─ List<String> list = Collections.unmodifiableList(inner);│
│     └─ list 引用不可变，但内容需要额外保证                │
└─────────────────────────────────────────────────────────┘
```

**this 逸出（构造器中的坑）：**

```java
// ❌ 构造器中 this 逸出
public class UnsafeEscape {
    private final List<String> list;
    
    public UnsafeEscape() {
        list = new ArrayList<>();
        // 错误：在构造器里把 this 传给另一个线程
        new Thread(() -> System.out.println(list.size())).start();
        // 另一个线程可能看到未构造完成的对象！
    }
}

// ✅ 正确做法：等构造完成后再启动
public class SafeEscape {
    private final List<String> list;
    
    public SafeEscape() {
        list = new ArrayList<>();
    }
    
    // 显式创建静态工厂方法
    public static SafeEscape create() {
        SafeEscape instance = new SafeEscape();
        // 构造完成后，再发布给其他线程
        instance.initialize();
        return instance;
    }
}
```

**面试加分话术：**

> "不可变对象**一定是线程安全的**，因为它的状态创建后就固定了，没有修改路径，自然没有竞态条件和可见性问题。这也是为什么 JDK 新增的类（如 String、BigInteger、LocalDateTime）基本都是不可变的。但是！**不可变不等于安全发布**——如果 `this` 在构造器里逸出了，JVM 重排序可能导致其他线程看到部分构造的对象。所以不可变类必须遵守三条：**所有字段加 final、引用类型做防御性拷贝、构造器里不要 this 逸出**。"

## 🔥 扩展追问 5：哪些类是天然不可变的？不可变类的实现方式？

**JDK 常见不可变类：**

| 类 | 类型 | 说明 |
|---|---|---|
| String | 不可变 | 最典型的不可变类 |
| BigInteger / BigDecimal | 不可变 | 任意精度 |
| Integer / Long 等包装类 | 不可变 | 包装了基本类型 |
| LocalDateTime | 不可变 | Java 8 日期时间 |
| Optional | 不可变 | 值容器 |
| UUID | 不可变 | 唯一标识 |

**如何让一个普通类变成不可变类：**

```java
// 步骤1：类加 final（防止被继承后重写）
public final class ImmutablePoint {
    
    // 步骤2：所有字段加 final
    private final double x;
    private final double y;
    private final String label;
    
    // 步骤3：构造器初始化所有字段
    public ImmutablePoint(double x, double y, String label) {
        this.x = x;
        this.y = y;
        this.label = label;
    }
    
    // 步骤4：只提供 getter，不提供 setter
    
    // 步骤5：可变引用类型要做防御性拷贝
    public String getLabel() {
        return label;  // String 不可变，直接返回即可
    }
}
```

**可变依赖的不可变类：**

```java
// 如果类持有可变对象，如何保持不可变性？
public final class ImmutableOrder {
    private final String orderId;
    private final BigDecimal amount;
    private final List<OrderItem> items;  // 可变对象
    
    public ImmutableOrder(String orderId, BigDecimal amount, List<OrderItem> items) {
        this.orderId = orderId;
        this.amount = amount;
        // 防御性拷贝：复制一份
        this.items = new ArrayList<>(items);
    }
    
    // 返回拷贝，不返回原引用
    public List<OrderItem> getItems() {
        return new ArrayList<>(items);
    }
}
```

**不可变对象的"修改"技巧——返回新对象：**

```java
// String 的方法都是返回新 String，自己不改
String s = "hello";
String s2 = s.toUpperCase();  // 返回 "HELLO"，s 还是 "hello"

// 自定义不可变类也应该这样
public final class ImmutableUser {
    private final String name;
    private final int age;
    
    // 想"改" name？返回新的对象
    public ImmutableUser withName(String newName) {
        return new ImmutableUser(newName, this.age);
    }
    
    // 想"改" age？返回新的对象
    public ImmutableUser withAge(int newAge) {
        return new ImmutableUser(this.name, newAge);
    }
}

ImmutableUser u1 = new ImmutableUser("张三", 25);
ImmutableUser u2 = u1.withAge(26);  // u1 不变，u2 是新的
```

**面试加分话术：**

> "不可变类的实现规范可以记成**五字诀**：**final 类**（防继承）、**final 字段**（防修改）、**构造器初始化**、**无 setter**、**引用拷贝**。JDK 的 String 就是完美范本——所有方法返回新 String、自己不动、final class + private char[]。实际项目中，如果一个对象创建后不会变（配置对象、DTO、值对象），就做成不可变类，**线程安全零成本**，GC 压力也小（不可变对象频繁创建，但 JVM 会优化）。"

## 🔥 扩展追问 6：线程安全的实现方式有哪些？各有什么优缺点？

**五种实现方式对比：**

| 方式 | 原理 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| **不可变对象** | 状态不可变 | 最简单、无锁、无竞态 | 需要复制对象、内存开销 | 值对象、配置 |
| **synchronized** | 互斥锁 | JDK 内置、简单 | 锁粒度大、阻塞线程 | 代码块、方法 |
| **Lock** | 可重入锁 + CAS | 可中断、可超时、多条件 | 需要手动 unlock | 复杂同步 |
| **Atomic*** | CAS | 无锁、高性能 | 只适合单一变量 | 计数器、flag |
| **ThreadLocal** | 线程隔离 | 无锁、无竞争 | 不能共享数据 | 线程上下文 |

**代码示例：**

```java
// 方式1：不可变
public final class ImmutableConfig {
    private final String url;
    private final int timeout;
    // 无 setter，线程安全
}

// 方式2：synchronized
public class SafeList<T> {
    private final List<T> list = new ArrayList<>();
    public synchronized void add(T item) { list.add(item); }
    public synchronized T get(int idx) { return list.get(idx); }
}

// 方式3：Lock
public class SafeList2<T> {
    private final List<T> list = new ArrayList<>();
    private final Lock lock = new ReentrantLock();
    
    public void add(T item) {
        lock.lock();
        try {
            list.add(item);
        } finally {
            lock.unlock();
        }
    }
}

// 方式4：Atomic
public class AtomicCounter {
    private final AtomicInteger count = new AtomicInteger(0);
    public void increment() { count.incrementAndGet(); }
    public int get() { return count.get(); }
}

// 方式5：ThreadLocal
public class UserContext {
    private static final ThreadLocal<User> CURRENT_USER = new ThreadLocal<>();
    public static User getCurrent() { return CURRENT_USER.get(); }
    public static void set(User u) { CURRENT_USER.set(u); }
    public static void clear() { CURRENT_USER.remove(); }
}
```

**并发容器替代同步容器：**

```java
// ❌ 老写法：Collections.synchronizedMap
Map<String, User> syncMap = Collections.synchronizedMap(new HashMap<>());
// 整个 Map 一把锁，效率低

// ✅ 新写法：ConcurrentHashMap
Map<String, User> concurrentMap = new ConcurrentHashMap<>();
// 分段锁（ JDK 8后改成 CAS + synchronized），并发度高

// ❌ 老写法：Vector（所有方法 synchronized）
Vector<User> syncList = new Vector<>();

// ✅ 新写法：CopyOnWriteArrayList
CopyOnWriteArrayList<User> copyOnWrite = new CopyOnWriteArrayList<>();
// 读不加锁、写复制，适用于读多写少场景
```

**面试加分话术：**

> "选线程安全实现方式有个原则：**能不用锁就不用锁，锁越少越好**。首选不可变对象（最干净）；如果必须可变，看变量类型——计数器用 Atomic 类、Map 用 ConcurrentHashMap、List 用 CopyOnWriteArrayList 或者 ConcurrentLinkedQueue。如果业务逻辑复杂，需要可重入、可中断、可超时，才用显式 Lock。我之前优化过一个接口，把 `synchronized(this)` 换成 `ConcurrentHashMap` 的 `computeIfAbsent`，QPS 从 200 提升到 2000，锁竞争少了，性能就上来了。"

## 🔥 扩展追问 7：什么是线程安全的复合操作？long/double 的线程安全？

**复合操作：先读、再算、后写**

```java
// ❌ 线程不安全的复合操作
public class Counter {
    private long count = 0;  // long 是 64 位，可能被分两次读写！
    
    public void increment() {
        count++;  // count++ 是复合操作
    }
    
    public long get() {
        return count;
    }
}

// ✅ 解决方案1：synchronized
public class SafeCounter1 {
    private long count = 0;
    
    public synchronized void increment() {
        count++;
    }
    
    public synchronized long get() {
        return count;
    }
}

// ✅ 解决方案2：AtomicLong
public class SafeCounter2 {
    private final AtomicLong count = new AtomicLong(0);
    
    public void increment() {
        count.incrementAndGet();
    }
    
    public long get() {
        return count.get();
    }
}
```

**long/double 的特殊问题：**

```
普通变量：int（32位） → 一次读/写完成，天然原子
long/double（64位） → 可能分两次读/两次写，不保证原子！

32位 JVM 读取 long：
        ↓
高 32 位 ←─── 一次读
低 32 位 ←─── 另一次读
        ↓
如果两个线程同时读，可能读到撕裂值（Half-Written）

64位 JVM（JDK 8+ 默认）：
  - 开启了压缩指针（+UseCompressedOops）
  - long/double 仍然不保证原子，除非加 volatile
```

**volatile long/double：**

```java
// volatile 保证可见性和读写原子性（64位写入原子）
public class VolatileCounter {
    private volatile long count = 0;
    
    public void increment() {
        count++;  // 不是原子的！volatile 只保证读写原子
    }
    
    public long get() {
        return count;  // 读是原子的
    }
}

// ✅ 正确做法：AtomicLong
private final AtomicLong count = new AtomicLong(0);

// ✅ 或者：synchronized（整个操作原子）
private long count = 0;
public synchronized void increment() {
    count++;
}
```

**面试加分话术：**

> "复合操作的线程安全性要看**整个操作是否原子**。`count++` 不是原子，因为包含了读、改、写三步；`count.get()` 是原子（但如果不用 volatile，可能读不到最新值）。另外 long/double 要单独注意——**非 volatile 的 long/double 在 32 位 JVM 上有撕裂读的问题**，虽然现在基本都用 64 位 JVM 了，但加上 volatile 是好习惯。另外强调一点：`AtomicLong.incrementAndGet()` 底层是 CAS + 自旋，比 synchronized 在低竞争下快很多，但在高竞争下反而更慢——没有银弹，要看实际场景。"

## 🔥 扩展追问 8：如何写出线程安全的代码？有什么最佳实践？

**最佳实践总结：**

```
┌─────────────────────────────────────────────────────────┐
│  线程安全编码 Checklist                                 │
├─────────────────────────────────────────────────────────┤
│ 1. 能用不可变对象，就不要可变                            │
│ 2. 限制共享：能ThreadLocal就ThreadLocal                  │
│ 3. 限制修改：能 final 就 final，能 volatile 就 volatile  │
│ 4. 缩小锁范围：synchronized(目标对象)，不是 this        │
│ 5. 优先用并发工具：ConcurrentHashMap、Atomic*            │
│ 6. 原子类不能包治百病：复合操作要整体加锁                 │
│ 7. 死锁风险：加锁顺序要一致                              │
│ 8. 避免在锁内调用外部方法（可能死锁/性能差）             │
└─────────────────────────────────────────────────────────┘
```

**最佳实践代码示例：**

```java
// ✅ 实践1：优先不可变
public record User(Long id, String name, List<String> roles) { }
// Java 14+ 的 record 天生不可变，简洁

// ✅ 实践2：缩小锁范围
private final Object lock = new Object();
private int count;

// ❌ 大范围
synchronized(this) {
    count++;
    doSomethingElse();
}

// ✅ 小范围
synchronized(lock) {
    count++;
}
doSomethingElse();  // 锁外执行

// ✅ 实践3：避免在锁内调用外部方法
public void process() {
    synchronized(lock) {
        // ❌ 可能在持锁状态下调用外部服务，外部也可能回调持锁方法
        externalService.send(message);
    }
}

// ✅ 实践4：死锁避免（锁顺序一致）
// 线程 A：先锁 lock1 再锁 lock2
synchronized(lock1) {
    synchronized(lock2) {
        doA();
    }
}

// 线程 B：也先锁 lock1 再锁 lock2（顺序一致，不会死锁！）
synchronized(lock1) {
    synchronized(lock2) {
        doB();
    }
}

// ✅ 实践5：用并发容器
Map<String, User> users = new ConcurrentHashMap<>();
users.computeIfAbsent("key", k -> loadUser(k));
```

**检查线程安全问题的工具：**

```java
// IDEA 插件：IntelliJ Thread Sanitizer (TSan)
// 运行时检测数据竞争

// 代码审查 checklist：
// 1. 所有共享可变字段是否加了同步？
// 2. 锁的粒度是否太大？
// 3. 是否有隐藏的共享状态（static 字段、缓存）？
// 4. ThreadLocal 是否在用完时清理了？
// 5. 是否有先检查后行动的竞态？
// 6. 是否有不安全的 lazy 初始化？
```

**不安全的 lazy 初始化 vs 安全的：**

```java
// ❌ 线程不安全：双重检查锁定
class UnsafeLazy {
    private static Resource resource;
    
    public static Resource getInstance() {
        if (resource == null) {           // 第一次检查
            synchronized (UnsafeLazy.class) {
                if (resource == null) {   // 第二次检查
                    resource = new Resource();  // 指令重排！
                }
            }
        }
        return resource;
    }
    // 问题：new Resource() 可能被 JIT 重排
    // 1. 分配内存  2. 调用构造  3. 赋值给引用
    // 如果 3 在 2 之前，resource != null 但对象未初始化完
}

// ✅ 正确做法1：加 volatile
class SafeLazy1 {
    private static volatile Resource resource;
    
    public static Resource getInstance() {
        if (resource == null) {
            synchronized (SafeLazy1.class) {
                if (resource == null) {
                    resource = new Resource();
                }
            }
        }
        return resource;
    }
}

// ✅ 正确做法2：静态内部类（推荐）
class SafeLazy2 {
    private static class Holder {
        static final Resource INSTANCE = new Resource();
    }
    
    public static Resource getInstance() {
        return Holder.INSTANCE;  // JVM 保证线程安全 + 懒加载
    }
}

// ✅ 正确做法3：直接用枚举（最简洁）
enum ResourceFactory {
    INSTANCE;
    private final Resource resource = new Resource();
    public Resource get() { return resource; }
}
```

**面试加分话术：**

> "写出线程安全代码的核心是**最小化共享、最小化可变**。能 threadlocal 就 threadlocal，能不可变就不变。如果必须同步，记住三个原则：**锁范围要小（只包必要的代码）、锁持有时间要短（不要在锁里调外部方法）、加锁顺序要一致（防止死锁）**。关于 DCL（Dubble-Checked Locking），我建议直接用**静态内部类Holder 模式**或者**枚举单例**，不用 volatile 也不会有问题，而且更简洁——JVM 在类加载阶段就保证了线程安全，比自己写 DCL 可靠多了。"

## 一句话总结（面试收尾用）

> "线程安全的本质是**控制共享可变状态的访问**——要么不动（不可变）、要么隔离（ThreadLocal）、要么同步（synchronized/Lock/Atomic）。不可变对象一定是线程安全的，但要注意**防御性拷贝**和**构造器 this 逸出**。实际写代码时，优先用不可变对象和并发容器，实在需要同步才加锁，注意**锁范围要小、持有时间要短、加锁顺序要一致**。做到了这三点，大多数线程安全问题都能规避。"
