```java
// Jedis SCAN 完整示例
public class RedisScanDemo {
    
    private JedisPool jedisPool;
    
    /**
     * 安全遍历所有匹配的 key（不阻塞 Redis）
     */
    public Set<String> scanKeys(String pattern, int count) {
        Set<String> result = new HashSet<>();
        
        try (Jedis jedis = jedisPool.getResource()) {
            ScanParams params = new ScanParams();
            params.match(pattern);
            params.count(count);
            
            String cursor = "0";
            
            do {
                // 执行 SCAN
                ScanResult<String> scanResult = jedis.scan(cursor, params);
                
                // 获取本批次结果
                List<String> keys = scanResult.getResult();
                result.addAll(keys);
                
                // 获取下次游标
                cursor = scanResult.getCursor();
                
                // cursor = "0" 表示遍历完成
                
            } while (!"0".equals(cursor));
        }
        
        return result;
    }
    
    /**
     * 安全批量删除（SCAN + DEL）
     */
    public long safeDelete(String pattern) {
        long deletedCount = 0;
        
        try (Jedis jedis = jedisPool.getResource()) {
            ScanParams params = new ScanParams();
            params.match(pattern);
            params.count(100);
            
            String cursor = "0";
            
            do {
                ScanResult<String> scanResult = jedis.scan(cursor, params);
                List<String> keys = scanResult.getResult();
                
                if (!keys.isEmpty()) {
                    // 批量删除
                    long n = jedis.del(keys.toArray(new String[0]));
                    deletedCount += n;
                }
                
                cursor = scanResult.getCursor();
                
            } while (!"0".equals(cursor));
        }
        
        return deletedCount;
    }
    
    /**
     * 扫描大 key（同时获取大小）
     */
    public void scanBigKeys() {
        try (Jedis jedis = jedisPool.getResource()) {
            String cursor = "0";
            
            do {
                ScanResult<Tuple> scanResult = jedis.sscan(
                    "my-big-zset", cursor, 
                    new ScanParams().count(100)
                );
                
                for (Tuple tuple : scanResult.getResult()) {
                    String member = tuple.getElement();
                    double score = tuple.getScore();
                    // 处理...
                }
                
                cursor = scanResult.getCursor();
                
            } while (!"0".equals(cursor));
        }
    }
}
```

## 🔥 扩展追问 1：为什么 SCAN 会返回重复元素？

**核心原因：Redis 字典的渐进式 rehash**

```
┌─────────────────────────────────────────────────────────────────┐
│                  渐进式 rehash 导致重复                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Redis 字典在扩容时，新旧两个 bucket 数组同时存在：             │
│                                                                 │
│  old_table[0] ──▶ [key1]                                       │
│  old_table[1] ──▶ [key2]                                       │
│  old_table[2] ──▶ [key3]                                       │
│                                                                 │
│  new_table[0] ──▶ [key1]   ← key1 已迁移到这里                │
│  new_table[2] ──▶ [key3]   ← key3 已迁移到这里                │
│  new_table[4] ──▶ [key2]   ← key2 正在迁移                    │
│                                                                 │
│  SCAN 在新旧数组间跳跃遍历：                                     │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  Step1: 访问 old_table[0] → 返回 key1                   │   │
│  │  Step2: 访问 old_table[2] → 返回 key3                   │   │
│  │  Step3: 访问 new_table[0] → 返回 key1  ← 重复！         │   │
│  │  Step4: 访问 new_table[4] → 返回 key2                   │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ✅ 解决方案：客户端对结果去重（用 HashSet）                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> A: 重复是因为 Redis 字典在扩容/缩容时使用渐进式 rehash，新旧两个 bucket 数组同时存在。SCAN 在两个数组间跳跃遍历，同一个 key 可能在迁移前后被访问两次。缩容时同理。客户端需要对结果去重，比如用 HashSet 接收。

## 🔥 扩展追问 2：为什么 SCAN 会漏掉元素？

**核心原因：遍历期间有新 key 被写入**

```
┌─────────────────────────────────────────────────────────────────┐
│                    新写入导致漏元素                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  时间线：                                                         │
│                                                                 │
│  T0: SCAN cursor=0，开始遍历 bucket 数组                        │
│  T1: 遍历到 bucket[5]，返回一批 keys                            │
│  T2: 新 key 被写入 bucket[6]                                     │
│  T3: cursor 翻倍，溢出重置为 0                                   │
│  T4: 遍历结束                                                    │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  如果新 key 落在 bucket[8]（高位区域）                  │   │
│  │  而 cursor 已经在 T3 溢出重置                           │   │
│  │  bucket[8] 永远不会被访问                               │   │
│  │  → 新 key 漏掉了                                        │   │
│  │                                                         │   │
│  │  如果新 key 落在当前游标之后的位置                       │   │
│  │  下一轮就能访问到                                        │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ✅ 理解：SCAN 返回的是"遍历开始时的数据 + 遍历期间写入的数据"    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> A: 漏掉元素是因为 SCAN 遍历期间有新 key 被写入。如果新 key 直接写入 new_table 的高位 bucket，而 SCAN 还没来得及访问到就溢出了，这个 key 就会漏掉。SCAN 不保证强一致性，它只保证所有在遍历开始时存活的 key 都会被访问至少一次。实际业务中漏元素的概率很低，但客户端需要理解这是一个"最终一致"的遍历。

## 🔥 扩展追问 3：COUNT 参数到底是什么意思？

**COUNT 只是提示，不是精确值**

```
┌─────────────────────────────────────────────────────────────────┐
│                    COUNT 参数的真实含义                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  COUNT = 100 ≠ 返回 100 个 key                                   │
│                                                                 │
│  Redis 的行为：                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  COUNT 只是一个"提示"（hint）                           │   │
│  │                                                         │   │
│  │  Redis 遍历 bucket：                                     │   │
│  │  • 如果 bucket[0] 有 50 个 key → 返回 50 个             │   │
│  │  • 如果 bucket[1] 有 80 个 key → 返回 80 个             │   │
│  │  • 如果 bucket[2] 有 5 个 key → 返回 5 个               │   │
│  │                                                         │   │
│  │  实际返回数量 = bucket 深度决定的 key 数量               │   │
│  │  跟 COUNT 无关！                                        │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  COUNT 的作用：                                                 │
│  • 影响遍历轮数（COUNT 越大，轮数越少）                        │
│  • 但不影响每次返回的数量                                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> A: COUNT 不是"每次返回 N 个 key"，而是一个提示值。Redis 实际上是按 bucket 遍历的，每次返回的是当前 bucket 内所有匹配的 key。如果某个 bucket 里有 1000 个匹配的 key，即使 COUNT=10，也会一次性全部返回。COUNT 的作用是影响需要遍历多少轮 bucket，值越大遍历轮数越少，但单次返回的数据量不可控。

## 🔥 扩展追问 4：生产环境如何使用 SCAN？

**实战技巧**

```java
// 场景1：安全清理过期 key
public class SafeKeyCleanup {
    
    private JedisPool jedisPool;
    
    /**
     * 安全删除匹配的所有 key（不阻塞）
     */
    public void safeCleanup(String pattern) {
        try (Jedis jedis = jedisPool.getResource()) {
            ScanParams params = new ScanParams();
            params.match(pattern);
            params.count(100);  // 每批处理 100 个
            
            String cursor = "0";
            long totalDeleted = 0;
            
            while (true) {
                ScanResult<String> result = jedis.scan(cursor, params);
                List<String> keys = result.getResult();
                
                if (!keys.isEmpty()) {
                    // 分批删除，避免阻塞
                    long deleted = jedis.del(keys.toArray(new String[0]));
                    totalDeleted += deleted;
                    
                    System.out.println("已删除 " + totalDeleted + " 个 key");
                }
                
                cursor = result.getCursor();
                if ("0".equals(cursor)) {
                    break;
                }
                
                // 可选：避免删除太快
                try {
                    Thread.sleep(10);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    break;
                }
            }
            
            System.out.println("清理完成，共删除 " + totalDeleted + " 个 key");
        }
    }
}

// 场景2：统计 key 分布
public Map<String, Long> analyzeKeyDistribution() {
    Map<String, Long> patternCount = new HashMap<>();
    
    try (Jedis jedis = jedisPool.getResource()) {
        // 统计不同前缀的 key 数量
        String[] patterns = {"user:*", "order:*", "product:*", "cache:*"};
        
        for (String pattern : patterns) {
            long count = 0;
            ScanParams params = new ScanParams();
            params.match(pattern);
            params.count(1000);
            
            String cursor = "0";
            do {
                ScanResult<String> result = jedis.scan(cursor, params);
                count += result.getResult().size();
                cursor = result.getCursor();
            } while (!"0".equals(cursor));
            
            patternCount.put(pattern, count);
        }
    }
    
    return patternCount;
}

// 场景3：SCAN 期间监控进度
public void monitorScanProgress() {
    long startTime = System.currentTimeMillis();
    long totalKeys = 0;
    
    try (Jedis jedis = jedisPool.getResource()) {
        ScanParams params = new ScanParams();
        params.match("*");
        params.count(1000);
        
        String cursor = "0";
        while (!"0".equals(cursor)) {
            ScanResult<String> result = jedis.scan(cursor, params);
            List<String> keys = result.getResult();
            
            totalKeys += keys.size();
            cursor = result.getCursor();
            
            // 每秒打印进度
            long elapsed = System.currentTimeMillis() - startTime;
            if (elapsed > 1000) {
                System.out.println("已扫描 " + totalKeys + " 个 key，游标: " + cursor);
                startTime = System.currentTimeMillis();
            }
        }
    }
    
    System.out.println("扫描完成，共 " + totalKeys + " 个 key");
}
```

```bash
# Shell 脚本版 SCAN 清理
#!/bin/bash
PATTERN="temp:*"
COUNT=100
CURSOR=0
DELETED=0

while true; do
    result=$(redis-cli SCAN $CURSOR MATCH "$PATTERN" COUNT $COUNT)
    CURSOR=$(echo "$result" | head -1)
    KEYS=$(echo "$result" | tail -n +2)
    
    if [ -n "$KEYS" ]; then
        n=$(echo "$KEYS" | wc -l)
        echo "$KEYS" | redis-cli DEL $(cat) > /dev/null
        DELETED=$((DELETED + n))
        echo "已删除 $DELETED 个 key"
    fi
    
    if [ "$CURSOR" = "0" ]; then
        break
    fi
done

echo "清理完成，共删除 $DELETED 个 key"
```

> A: 生产环境使用 SCAN 有几个技巧：第一，每次 COUNT 不要太大（100-1000 即可），避免单次返回过多数据；第二，批量操作后适当 sleep，避免 Redis CPU 被打满；第三，客户端必须对结果去重，因为 rehash 会导致重复；第四，可以用 SCAN 监控进度，打印已扫描数量和游标位置便于排查。

## 🔥 扩展追问 5：SCAN 能否替代 KEYS？

**结论：SCAN 适合遍历，KEYS 适合调试**

| 场景 | 推荐命令 | 原因 |
|------|---------|------|
| 生产环境遍历 key | SCAN | 不阻塞，可安全使用 |
| 开发调试查找 key | KEYS | 结果直接，但注意阻塞 |
| 定时清理任务 | SCAN + DEL | 必须用 SCAN |
| 统计 key 数量 | SCAN | 用 SCAN 累加 |
| 查找特定 key | KEYS user:10086 | 单次、小数据量可用 |

> A: SCAN 不能完全替代 KEYS。KEYS 的优点是返回结果简单直接，不需要客户端处理游标迭代。但 KEYS 会阻塞 Redis，在数据量大时很危险。生产环境遍历、清理、统计必须用 SCAN；开发调试时查找特定 key，KEYS 可以用但要注意数据量。

## 一句话总结（面试收尾用）

> **KEYS 会阻塞 Redis 执行全量扫描，数据量大时可能阻塞数十秒导致服务不可用；SCAN 是增量迭代式遍历，用游标分批次返回结果，每次只访问少量 bucket，O(1) 复杂度不阻塞；游标基于 Redis 内部 bucket 数组实现，按 bucket 索引迭代而非 key 数量迭代；重复是因为 rehash 时新旧 bucket 数组同时存在，漏元素是因为遍历期间新 key 写入落在游标未访问的区域，客户端需要对结果去重并理解 SCAN 是"最终一致"的遍历。**

**面试加分话术：**

> "我之前踩过一个 SCAN 的坑：线上需要清理一批临时缓存 key，用 SCAN 遍历删除，本来以为安全，结果发现删了 3 天都没删完。后来排查发现两个问题：一是 COUNT 设置太大，每次返回几十万个 key，DEL 操作把 Redis 打满了；二是没有去重，同一个 key 被返回了两次，导致重复删除白耗资源。改了三个地方：一是 COUNT 从 10000 改成 100，每次少处理；二是加了 HashSet 去重；三是加了进度监控每秒打印进度。还有个经验：SCAN 遍历期间不要执行 blocking 命令（BLPOP、BRPOP 这些），会互相影响。如果数据量很大又很紧急，用 SCAN + pipeline 批量删除比直接 DEL 更稳定。"
