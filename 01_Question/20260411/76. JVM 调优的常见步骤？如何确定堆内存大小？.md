## 🔥 扩展追问 1：如何确定堆内存大小？

**公式法：**

```
堆大小 = 正常运行时的堆占用 × 1.5 ~ 2

示例：
  正常运行时堆占用 = 2GB
  → 建议 -Xmx = 3GB ~ 4GB
  
  为什么留余量？
  → 流量高峰
  → 内存抖动（GC 期间对象临时增长）
  → 代码 Bug（内存泄漏前兆）
```

**容量规划四象限：**

```
                    流量波动大
                        │
         ┌──────────────┼──────────────┐
         │              │              │
    堆    │    激进方案  │   保守方案   │    堆
    大    │              │              │    小
    │    │   固定 -Xmx   │  弹性扩容    │    │
    │    │   留 30% 余量 │  云原生 K8s  │    │
    │    │              │              │    │
         ├──────────────┼──────────────┤
         │              │              │
    堆    │   保守方案   │   激进方案   │
    小    │  固定 -Xmx   │  固定 -Xmx   │
         │  留 50% 余量  │  留 20% 余量 │
         └──────────────┴──────────────┘
                        │
                    流量波动小
```

**推荐配置：**

| 场景 | 推荐配置 | 原因 |
|------|----------|------|
| 2-4 核服务器 | -Xmx2g | 够用，GC 停顿可控 |
| 8-16 核服务器 | -Xmx4g ~ 8g | 并发量大，堆大 GC 反而快 |
| 32+ 核服务器 | -Xmx8g ~ 16g | 配合大堆用 G1/ZGC |
| 容器/K8s | 容器内存 75% | 留 25% 给 OS + 直接内存 |

## 🔥 扩展追问 2：新生代大小如何配置？

**经验公式：**

```
新生代大小 = 平均存活对象 × GC 频率 × 期望停顿占比

或者简化版：
  高并发、对象生命周期短 → 新生代大（减少 Minor GC）
  低并发、对象生命周期长 → 新生代小（减少对象晋升）
```

**参数对比：**

| 参数 | 说明 | 推荐场景 |
|------|------|---------|
| `-Xmn1g` | 固定新生代大小 | 生产环境（避免动态伸缩开销） |
| `-XX:NewRatio=2` | 老年代:新生代=2:1 | 通用场景 |
| `-XX:SurvivorRatio=8` | Eden:Survivor=8:1 | 正常情况 |
| `-XX:SurvivorRatio=4` | Eden:Survivor=4:1 | Survivor 区经常满 |

**实战配置：**

```bash
# 4核8G机器，跑电商秒杀服务
# 瞬时流量大，对象生命周期短
-Xms6g -Xmx6g           # 堆固定
-Xmn3g                  # 新生代50%（比默认1/3大）
-XX:SurvivorRatio=6     # Eden:Survivor = 6:1:1
-XX:MaxTenuringThreshold=15  # 延迟晋升

# 4核8G机器，跑离线计算服务
# 批量处理，对象生命周期长
-Xms8g -Xmx8g           # 堆固定
-Xmn1g                  # 新生代12.5%（比默认小）
-XX:NewRatio=7          # 老年代:新生代 = 7:1
```

## 🔥 扩展追问 3：如何选择垃圾收集器？

**选型决策树：**

```
                          JVM 调优选收集器
                                │
            ┌───────────────────┼───────────────────┐
            │                   │                   │
      吞吐量优先？          延迟优先？           超大堆？
            │                   │                   │
            ▼                   ▼                   ▼
      Parallel GC         P99 < 200ms ──▶ G1
            │                   │           P99 < 10ms ──▶ ZGC
            ▼                   ▼                   ▼
      -XX:+UseParallelGC    -XX:+UseG1GC       -XX:+UseZGC
            │                   │                   
      CMS (JDK 11 前)      JDK 8+               JDK 11+
            │                   │                   
            ▼                   ▼                   
      吞吐量 优先           平衡型               
```

**收集器对比表：**

| 收集器 | 停顿时间 | 吞吐量 | 内存碎片 | JDK 版本 |
|--------|----------|--------|----------|----------|
| Serial | 长 | 低 | 无 | 全部 |
| Parallel | 中 | **高** | 无 | 全部 |
| CMS | 短 | 中 | 有 | JDK 7-14 |
| G1 | 可控 | 高 | 无 | **JDK 9+ 默认** |
| ZGC | **极短 <10ms** | 高 | 无 | JDK 11+ |
| Shenandoah | **极短 <10ms** | 高 | 无 | JDK 8u4+ |

**实战推荐：**

```bash
# 通用场景（JDK 8+）
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200

# 高吞吐批处理（JDK 8）
-XX:+UseParallelGC
-XX:+UseParallelOldGC
-XX:ParallelGCThreads=8

# 低延迟金融系统（JDK 11+）
-XX:+UseZGC
-XX:MaxGCPauseMillis=5

# 云原生微服务（JDK 11+）
-XX:+UseG1GC
# 配合容器资源限制
```

**五种 OOM 类型：**

| OOM 类型 | 原因 | 解决方案 |
|----------|------|----------|
| `java.lang.OutOfMemoryError: Java heap space` | 堆内存不足/内存泄漏 | 扩大堆、修复泄漏 |
| `java.lang.OutOfMemoryError: GC overhead limit exceeded` | GC 用时 > 98%，回收不了 | 同上 |
| `java.lang.OutOfMemoryError: Metaspace` | 类加载过多 | 扩大 Metaspace、检查 ClassLoader |
| `java.lang.OutOfMemoryError: Direct buffer memory` | NIO 堆外内存泄漏 | 检查 Netty/ByteBuffer 泄漏 |
| `java.lang.OutOfMemoryError: unable to create new native thread` | 线程数过多 | 减少线程数、检查线程泄漏 |

## 🔥 扩展追问 8：实际调优案例？

**案例：电商秒杀系统 GC 优化**

```
背景：
- 4 核 8G 服务器
- TPS 5000，促销时 50000
- 问题：GC 停顿 3-5 秒，用户体验极差

排查过程：
┌────────────────────────────────────────────────────────┐
│ Step 1: GC 日志分析                                    │
│   - Young GC: 20次/秒，每次 30ms → OK                  │
│   - Mixed GC: 2次/分钟，每次 3s → ❌ 问题！             │
│                                                        │
│ Step 2: 原因定位                                       │
│   - IHOP=45%，老年代占用 80% 才触发                     │
│   - 促销流量大，存活对象多，Survivor 装不下             │
│   → 对象大量晋升老年代                                  │
│                                                        │
│ Step 3: 调优                                          │
│   # 方案 A：提前触发 Mixed GC                           │
│   -XX:InitiatingHeapOccupancyPercent=35  # 45→35      │
│                                                        │
│   # 方案 B：增加 G1 预留空间                            │
│   -XX:G1ReservePercent=20  # 10→20                    │
│                                                        │
│   # 方案 C：限制每次 Mixed GC 收集量                    │
│   -XX:G1MixedGCCountTarget=16  # 8→16                 │
```

```bash
#!/bin/bash
```

## 一句话总结（面试收尾用）

> **JVM 调优遵循"确立目标 → 收集数据 → 分析瓶颈 → 参数调优 → 验证效果"的五步法，堆大小通常设为正常运行内存的 1.5-2 倍，新生代建议固定大小（G1 用 Region 不需单独设），通过 GC 日志分析频率和停顿时间，优先选择 G1（通用）或 ZGC（低延迟）收集器，最终用压测验证调优效果。**
