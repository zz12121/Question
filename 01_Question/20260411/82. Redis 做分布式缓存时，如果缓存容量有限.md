## 🔥 扩展追问 1：LRU vs LFU vs Random 各自适用场景？

**三种策略对比：**

| 策略         | 原理        | 优点       | 缺点                       | 适用场景            |
| ---------- | --------- | -------- | ------------------------ | --------------- |
| **LRU**    | 淘汰最久未访问的  | 保留热点数据   | 批量扫描后recently-visited会干扰 | **读多写少**，热点明显   |
| **LFU**    | 淘汰访问频率最低的 | 保留高频数据   | 对突发流量敏感度低                | **持续热点**，访问分布稳定 |
| **Random** | 随机淘汰      | 无状态，开销极低 | 命中率不稳定                   | 内存充足/对命中率要求不高   |

**LRU 适用场景：**

```java
// 场景1：热点新闻列表
// 用户集中访问最新新闻，热点明显
// LRU 能很好地保留 recent 热点

// 场景2：用户 Session 缓存
// 用户登录后频繁访问自己的数据
// LRU 适合"最近用过"的模式

// 场景3：商品详情缓存
// 热门商品被频繁访问，冷门商品很少被访问
```

```python
# Python 实现简单 LRU
from collections import OrderedDict

class LRUCache:
    def __init__(self, capacity):
        self.cache = OrderedDict()
        self.capacity = capacity
    
    def get(self, key):
        if key in self.cache:
            self.cache.move_to_end(key)  # 最近访问
            return self.cache[key]
        return -1
    
    def put(self, key, value):
        if key in self.cache:
            self.cache.move_to_end(key)
        self.cache[key] = value
        if len(self.cache) > self.capacity:
            self.cache.popitem(last=False)  # 淘汰最老的
```

**LFU 适用场景：**

```java
// 场景1：排行榜/热搜
// 访问频率稳定的场景

// 场景2：API 限流
// 基于访问频率的限流策略

// 场景3：搜索引擎结果缓存
// 搜索词有稳定热度分布
```

```python
# Python 实现简单 LFU
import heapq

class LFUCache:
    def __init__(self, capacity):
        self.capacity = capacity
        self.freq = {}  # key -> frequency
        self.min_freq = 0
        self.cache = {}  # key -> value
    
    def get(self, key):
        if key not in self.cache:
            return -1
        freq = self.freq[key]
        self.freq[key] = freq + 1
        # 移动到新频率的桶
        return self.cache[key]
    
    def put(self, key, value):
        if key in self.cache:
            self.cache[key] = value
            self.freq[key] += 1
        else:
            if len(self.cache) >= self.capacity:
                # 淘汰最小频率的
                min_freq_key = min(self.freq, key=self.freq.get)
                del self.cache[min_freq_key]
                del self.freq[min_freq_key]
            self.cache[key] = value
            self.freq[key] = 1
            self.min_freq = 1
```

**Random 适用场景：**

```java
// 场景1：内存充足，缓存容量大
// 随机淘汰的影响被稀释

// 场景2：临时缓存（不追求高命中率）
// 如：分布式锁的临时持有者列表

// 场景3：尝鲜式缓存（快速失败降级）
// 缓存满了就随机踢掉，凉了再热

// 场景4：内存警告时的紧急释放
// maxmemory-emergency-eviction-policy: random
```

**如何选择：**

```
选择决策树：

┌─────────────────────────────────────────────────────────┐
│                                                         │
│  数据有明显的冷热分层？                                  │
│       │                                                │
│       ├── 是 ──▶ 热点数据很重要？                        │
│       │         │                                        │
│       │         ├── 是 ──▶ LRU（最近访问）              │
│       │         │                                        │
│       │         └── 否 ──▶ LFU（累计频率）              │
│       │                                                │
│       └── 否 ──▶ Random（无状态，开销最低）             │
│                                                         │
│  内存紧张，追求极致命中率？ ──▶ LFU                     │
│  读多写多，热点飘移？    ──▶ LRU                       │
│  数据无明显规律？        ──▶ Random                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 2：Redis 的近似 LRU 算法原理？

**精确 LRU vs 近似 LRU：**

```
┌─────────────────────────────────────────────────────────────────┐
│                     精确 LRU vs 近似 LRU                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  精确 LRU（理论实现）：                                          │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                                                           │  │
│  │  每条数据记录"上次访问时间"                               │  │
│  │                                                           │  │
│  │  ┌─────────────────────────────────────────────────────┐ │  │
│  │  │  数据1: 上次访问=10:00:00                           │ │  │
│  │  │  数据2: 上次访问=10:05:00  ← LRU 候选（最老）       │ │  │
│  │  │  数据3: 上次访问=10:03:00                           │ │  │
│  │  │  数据4: 上次访问=10:01:00                           │ │  │
│  │  └─────────────────────────────────────────────────────┘ │  │
│  │                                                           │  │
│  │  淘汰：数据2（访问时间最早）                             │  │
│  │                                                           │  │
│  │  问题：需要维护有序链表，O(N) 复杂度                     │  │
│  │  Redis 内存受限，无法承受这个开销                        │  │
│  │                                                           │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                 │
│  近似 LRU（Redis 实现）：                                        │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                                                           │  │
│  │  Redis Object 头部有 24 bits 存储 LRU 时钟              │  │
│  │                                                           │  │
│  │  ┌─────────────────────────────────────────────────────┐ │  │
│  │  │  typedef struct redisObject {                       │ │  │
│  │  │      unsigned type:4;      // 数据类型              │ │  │
│  │  │      unsigned encoding:4;  // 编码方式              │ │  │
│  │  │      unsigned lru:LRU_BITS; // LRU 时钟（24bits）   │ │  │
│  │  │      int refcount;         // 引用计数              │ │  │
│  │  │      void *ptr;            // 数据指针              │ │  │
│  │  │  } robj;                                           │ │  │
│  │  └─────────────────────────────────────────────────────┘ │  │
│  │                                                           │  │
│  │  LRU 时钟 = server.unixtime % LRU_CLOCK_MAX             │  │
│  │           = 当前时间秒 % 2^24 (约 194 天循环)            │  │
│  │                                                           │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**近似 LRU 算法流程：**

```
maxmemory-samples = 5（默认）

淘汰时：

1. 随机抽样 5 个 key
┌────────────────────────────────────────────────────────────┐
│  随机选中的 key：                                            │
│                                                             │
│  key1: lru=100  │  key2: lru=150  │  key3: lru=80        │
│  key4: lru=200  │  key5: lru=120  │                      │
│                                                             │
│  当前 lru clock = 180                                        │
│                                                             │
│  差值 = clock - lru:                                        │
│  key1: 80    │  key2: 30    │  key3: 100 ← 最大（最老）   │
│  key4: -20   │  key5: 60   │                              │
│  （负数说明  │  （负数说明  │                              │
│   已循环）   │   已循环）   │                              │
│                                                             │
│  淘汰 key3（差值最大，最久未访问）                          │
└────────────────────────────────────────────────────────────┘

2. 如果抽样不够，再抽样（最多 16 次）
   maxmemory-samples 越大，越接近精确 LRU
   但 CPU 消耗也越大
```

**maxmemory-samples 参数对比：**

```bash
# maxmemory-samples = 5（默认）
# 速度快，但不够精确

# maxmemory-samples = 16
# 更接近精确 LRU，但淘汰操作更慢

# maxmemory-samples = 10（推荐平衡值）
CONFIG SET maxmemory-samples 10
```

**近似 LRU vs 精确 LRU 差距有多大？**

```
实验数据（Redis 官方）：

100% LRU（精确）：
  样本数 = 数据集大小，每次遍历找最老的

10% LRU（近似，maxmemory-samples=10）：
  命中率 = 精确 LRU 的 96%

50% LRU（近似，maxmemory-samples=10）：
  命中率 = 精确 LRU 的 99%

结论：近似 LRU 的命中率损失很小（1-4%）
      但换来了 O(1) 的淘汰复杂度
```

## 🔥 扩展追问 4：LFU 的实现原理？

**Redis 4.0+ 支持 LFU：**

```
┌─────────────────────────────────────────────────────────────────┐
│                     LFU 实现原理                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  LFU = Most Frequently Used（访问频率最高）                     │
│                                                                 │
│  Redis Object 中的 LFU 信息：                                   │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                                                           │  │
│  │  在 LRU 24 bits 的空间内复用了 LFU 信息                   │  │
│  │                                                           │  │
│  │  高 16 bits: 计数器（LFU counter，0-255）                 │  │
│  │  低 8 bits:  上次衰减时间（decay time）                   │  │
│  │                                                           │  │
│  │  为什么不直接存累计次数？                                 │  │
│  │  → 因为一个 long 需要 64 bits，内存开销大                  │  │
│  │  → 用 0-255 的对数计数，够用且省空间                      │  │
│  │                                                           │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                 │
│  计数器衰减（Decay）：                                          │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                                                           │  │
│  │  每隔一段时间（可配置），counter 会衰减                    │  │
│  │                                                           │  │
│  │  lfu-decay-time = 1（默认，单位：分钟）                   │  │
│  │                                                           │  │
│  │  示例：                                                    │  │
│  │  - counter = 100                                          │  │
│  │  - 1 分钟没访问：counter = counter / 2 = 50              │  │
│  │  - 再过 1 分钟没访问：counter = 50 / 2 = 25              │  │
│  │  - ...                                                    │  │
│  │                                                           │  │
│  │  效果：冷门数据慢慢被衰减，热门数据保留高计数             │  │
│  │                                                           │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**LFU 参数配置：**

```bash
# 衰减时间（多久没访问就衰减一次）
CONFIG SET lfu-decay-time 1  # 默认 1 分钟

# 查看 LFU 信息
OBJECT FREQ <key>
# 返回：counter 值（0-255）

# 设置 counter（用于测试）
OBJECT SETFREQ <key> 100
```

**LFU vs LRU 对比实验：**

```
场景：秒杀系统

时间线：
T1: 1000 个用户各自访问自己的商品（每个商品被访问 1 次）
T2: 商品 A 被 10000 个用户访问（商品 A 变成真正热点）
T3: 商品 A 停止访问，冷了下来

LRU 结果：
- 商品 A 被淘汰（因为最近没人访问）
- 问题：商品 A 其实是高热度的！

LFU 结果：
- 商品 A counter 很高（被访问 10000 次）
- 即使停止访问，衰减需要时间
- 短时间内商品 A 仍被保留
- 问题：对突发热点的响应较慢

结论：
- LRU：适合"最近热点"的场景
- LFU：适合"持续热点"的场景
```

## 🔥 扩展追问 5：如何选择淘汰策略？

**策略选择决策表：**

| 场景 | 推荐策略 | 原因 |
|------|----------|------|
| **缓存通用场景** | allkeys-lru | 通用性强，命中率稳定 |
| **明确有 TTL 的缓存** | volatile-lru | 只淘汰会过期的数据 |
| **排行榜/热搜** | volatile-lfu / allkeys-lfu | 持续热度更重要 |
| **Session 缓存** | allkeys-lru | 最近访问很重要 |
| **限流/计数** | volatile-lfu | 访问频率决定是否淘汰 |
| **只写不读的缓存** | allkeys-random | 无所谓热点 |
| **内存接近上限** | volatile-lru | 优先淘汰有过期的 |
| **防止数据丢失** | noeviction | 拒绝写入，但查询正常 |

**组合策略推荐：**

```bash
# 推荐配置 1：通用缓存（4GB）
maxmemory 4gb
maxmemory-policy allkeys-lru
maxmemory-samples 10

# 推荐配置 2：有过期数据的缓存
maxmemory 4gb
maxmemory-policy volatile-lru
maxmemory-samples 10

# 推荐配置 3：追求高命中率
maxmemory 4gb
maxmemory-policy allkeys-lfu
lfu-decay-time 1

# 推荐配置 4：稳定热点的缓存
maxmemory 4gb
maxmemory-policy allkeys-lru
maxmemory-samples 16
```

**volatile vs allkeys 的选择逻辑：**

```
┌─────────────────────────────────────────────────────────────────┐
│                     什么时候用 volatile-xxx？                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  条件：缓存的数据都有 TTL                                       │
│                                                                 │
│  场景：                                                         │
│  1. 临时缓存（如用户 Session，过期自动清除）                     │
│  2. 热点数据缓存（如商品详情，设 1 小时 TTL）                   │
│  3. 计算结果缓存（如分页结果，设 5 分钟 TTL）                  │
│                                                                 │
│  优点：                                                         │
│  ✅ 永久数据不被淘汰（如配置字典）                               │
│  ✅ 过期数据自动清理                                             │
│                                                                 │
│  缺点：                                                         │
│  ❌ 如果所有数据都永不过期，会退化成 allkeys-xxx                │
│  ❌ 需要额外维护 TTL                                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 7：4GB 缓存容量下的具体策略设计？

**场景分析：**

```
4GB 缓存 vs 数据量 >> 4GB

常见场景：
- 商品数据 1000 万条（每条 1KB = 10GB）
- 用户数据 500 万条（每条 2KB = 10GB）
- 订单数据 1 亿条（每条 0.5KB = 50GB）

问题：Redis 只能存 4GB，如何最大化价值？
```

**分层缓存策略：**

```java
// 分层缓存设计

public class TieredCacheService {
    
    // L1: 本地缓存（Caffeine/Guava Cache）
    // 容量：100MB，淘汰策略：LRU
    // 命中率预计：80%
    private Cache<String, Object> l1Cache = Caffeine.newBuilder()
            .maximumSize(100_000)
            .expireAfterWrite(1, TimeUnit.MINUTES)
            .build();
    
    // L2: Redis 缓存
    // 容量：4GB，淘汰策略：allkeys-lru
    // 命中率预计：95%（含 L1 未命中）
    
    // L3: 数据库（MySQL）
    // 兜底
    
    public Object get(String key) {
        // L1 查找
        Object l1 = l1Cache.getIfPresent(key);
        if (l1 != null) {
            return l1;
        }
        
        // L2 查找
        Object l2 = redisTemplate.opsForValue().get(key);
        if (l2 != null) {
            // 回填 L1
            l1Cache.put(key, l2);
            return l2;
        }
        
        // L3 查询
        Object l3 = database.query(key);
        
        // 回填 L2
        redisTemplate.opsForValue().set(key, l3, 1, TimeUnit.HOURS);
        l1Cache.put(key, l3);
        
        return l3;
    }
}
```

**缓存容量分配策略：**

```bash
# 分层容量分配

# L1: 本地缓存（不占 Redis 内存）
# - 使用进程内存
# - 可用 100-200MB
# - Caffeine/Guava

# L2: Redis 缓存
# maxmemory = 4gb
# maxmemory-policy = allkeys-lru

# 分层容量预估：
# - 热点数据（前 10%）：1000 万 × 10% × 1KB = 1GB
# - 温数据（前 30%）：1000 万 × 30% × 1KB = 3GB
# - 冷数据（后 70%）：1000 万 × 70% × 1KB = 7GB（存不下）

# 结论：4GB 只能缓存前 30-40% 的热点数据
# 策略：热点缓存 L2，温数据缓存 L1 兜底，冷数据走数据库
```

**TTL 分层设计：**

```java
}
```

## 🔥 扩展追问 8：LFU 突发流量问题如何解决？

**LFU 的"突发流量"问题：**

```
问题场景：

某商品平时每天访问 100 次
突然有一天被热搜带火，1 小时内访问 10 万次
热搜过后，LFU counter 很高，需要很久才能衰减

┌─────────────────────────────────────────────────────────────┐
│  时间线：                                                     │
│                                                             │
│  T0: 正常期，counter = 100                                  │
│  T1: 热搜来，1 小时内访问 10 万次                           │
│      → counter = 100 + log(100000) ≈ 100 + 17 = 117        │
│      → 因为是对数增长，counter 涨得没想象中快                │
│                                                             │
│  T2: 热搜结束                                                │
│  T3: 衰减开始（lfu-decay-time = 1 分钟）                   │
│      → 每分钟 counter = counter / 2                        │
│      → 衰减到 100 需要多久？ ln(117)/ln(2) ≈ 7 次 = 7 分钟  │
│                                                             │
│  结论：LFU 能较快适应热点变化                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**为什么用对数计数？**

```c
// Redis LFU 计数器增量逻辑
uint8_t LFULogIncr(uint8_t counter) {
    if (counter == 255) return 255;  // 最大值
    
    // 概率计算
    double r = (double)rand() / RAND_MAX;
    double threshold = 1.0 / (pow(server.lfu_log_factor, 256 - counter));
    
    if (r < threshold) {
        counter++;
    }
    return counter;
}

// server.lfu_log_factor = 10（默认）
// counter 越小，越容易增加
// counter 越大，越难增加
// counter 接近 255 时，基本不会再增加
```

**LFU 调优参数：**

```bash
# lfu-decay-time：衰减间隔（单位：分钟）
# 值越小：越快衰减，冷数据快速被淘汰
# 值越大：保持热度更久

# 推荐：
# - 热点飘移快：lfu-decay-time = 1
# - 热点稳定：lfu-decay-time = 10

# lfu-log-factor：计数器增量难度
# 值越大：counter 越难增长（适合超大数据量）
# 值越小：counter 越容易增长（适合小数据量）

CONFIG SET lfu-decay-time 5
CONFIG SET lfu-log-factor 10
```

**突发流量的最佳实践：**

```java
// 方案：突发流量用 LRU，持续热度用 LFU

// 实现：双缓存
public class AdaptiveCacheService {
    
    // LRU 缓存：处理突发流量
    private Cache<String, Object> lruCache = Caffeine.newBuilder()
            .maximumSize(10_000)
            .expireAfterWrite(1, TimeUnit.HOURS)
            .recordStats()
            .build();
    
    // LFU 缓存：处理持续热点
    private RedisTemplate<String, Object> lfuRedis;
    
    public Object get(String key) {
        // 先查 LRU（响应突发流量）
        Object result = lruCache.getIfPresent(key);
        if (result != null) {
            // 同时回填 Redis
            lfuRedis.opsForValue().set(key, result, 24, TimeUnit.HOURS);
            return result;
        }
        
        // 查 LFU
        result = lfuRedis.opsForValue().get(key);
        if (result != null) {
            lruCache.put(key, result);
            return result;
        }
        
        // 查数据库
        return database.query(key);
    }
}
```

## 一句话总结（面试收尾用）

> **Redis 8 种淘汰策略分两类：volatile-xxx 只淘汰有过期时间的键，allkeys-xxx 淘汰所有键；LRU 淘汰最久未访问的，适合热点飘移场景但会被批量扫描污染；LFU 淘汰访问频率最低的，适合持续热点场景但对突发流量响应慢；Redis 使用近似 LRU，通过随机抽样 5 个 key 并比较 idle time 实现，O(1) 复杂度但命中率接近精确 LRU；4GB 有限容量建议用分层缓存（L1 本地 + L2 Redis allkeys-lru）最大化命中率。**

**面试加分话术：**
> "我之前遇到过一个问题：Redis 4GB 内存，存了 5000 万个用户 Session 详情，每次运营批量导出会触发大量淘汰，命中率从 95% 跌到 30%。排查发现是 LRU 被批量扫描污染了，所有被扫描过的 Session 都变成'最近访问'，把真正的热点挤掉了。后来改了两招：一是加了一层本地缓存（Caffeine 100MB），批量扫描直接打本地缓存，不进 Redis；二是换成 allkeys-lfu，把 maxmemory-samples 从 5 调到 10，同时 lfu-decay-time 设为 1 分钟。优化后命中率稳定在 98%，批量导出不再影响 Redis。还有个经验：数据有明显 TTL 的优先用 volatile-lru，这样永久数据不会被误淘汰。"
