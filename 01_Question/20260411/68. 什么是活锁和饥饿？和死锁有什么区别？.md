## Q68 什么是活锁和饥饿？和死锁有什么区别？

> 💡 这道题和 Q67 的追问 1 高度相关，这里给出更完整的独立回答。

### 基础流程

```
三种"卡住"的状态：

┌─────────────────────────────────────────────────────────────┐
│  死锁（Deadlock）                                           │
│  ├─ 线程都在等待，什么都不做（BLOCKED 状态）                 │
│  ├─ 互相持有对方需要的资源                                  │
│  └─ 只能外部干预打破                                       │
├─────────────────────────────────────────────────────────────┤
│  活锁（Livelock）                                           │
│  ├─ 线程都在运行，但不断让步做无用功（RUNNABLE 状态）        │
│  ├─ 检测到冲突后主动让步，让步后又冲突，一直循环            │
│  └─ 程序在跑，但没有任何进展                                │
├─────────────────────────────────────────────────────────────┤
│  饥饿（Starvation）                                         │
│  ├─ 某个/某些线程始终得不到资源                            │
│  ├─ 其他线程正常运行                                        │
│  └─ 可能自行恢复，也可能一直饿死                            │
└─────────────────────────────────────────────────────────────┘
```

**三者的关键区别：**

| 对比维度 | 死锁 | 活锁 | 饥饿 |
|---|---|---|---|
| **线程状态** | BLOCKED（阻塞） | RUNNABLE（在 CPU 上跑） | BLOCKED 或低优先级运行 |
| **是否占用 CPU** | ❌ 不占用 | ✅ 占用（不断重试） | ❌ 可能不占用 |
| **进展** | ❌ 没有任何进展 | ⚠️ 有进展但做无用功 | ❌ 目标线程无进展 |
| **打破方式** | 外部干预（kill/jstack） | 可能自行恢复 | 可能自行恢复 |
| **CPU 消耗** | 无 | 高（不断重试） | 低 |

## 🔥 扩展追问 1：活锁具体是怎么发生的？举几个实际场景？

**活锁的本质：**

```
活锁 = 死锁 + 主动让步
        ↓
线程 A 持有资源 R1，等待 R2
线程 B 持有资源 R2，等待 R1
        ↓
检测到死锁风险（或者已经死锁）
        ↓
主动让步：线程 A 释放 R1，线程 B 释放 R2
        ↓
重新抢 → 又冲突 → 又让步 → 永远循环
```

**场景一：转账死锁后回退变成活锁**

```java
// 加了超时回退，但回退策略导致活锁
public class LivelockTransfer {
    private final ReentrantLock lock1 = new ReentrantLock();
    private final ReentrantLock lock2 = new ReentrantLock();
    
    public boolean transfer(Account a, Account b, double amount) {
        while (true) {
            if (a.lock.tryLock(100, TimeUnit.MILLISECONDS)) {
                try {
                    if (b.lock.tryLock(100, TimeUnit.MILLISECONDS)) {
                        try {
                            doTransfer(a, b, amount);
                            return true;
                        } finally {
                            b.lock.unlock();
                        }
                    }
                } finally {
                    a.lock.unlock();
                }
            }
            // ⚠️ 活锁陷阱：两个账户同时高频转账
            // 线程 A 等 A→B，线程 B 等 B→A
            // 都超时都回退，然后立刻重试 → 永远在重试
            
            // 没有退避间隔！CPU 空转
            Thread.sleep(1);  // ✅ 加个退避就好了
        }
    }
}
```

**场景二：两个人在走廊相遇互相让路**

```
走廊宽度只容一人通过：

正常情况：
  ← A →
  ← B →   → 两人擦肩而过 ✓

活锁情况：
  A 靠左走 ←─── B 靠右走
  A 靠右让 ←─── B 靠左让
  A 靠左 ←───── B 靠右
  ... 永远互相让，没人过去 ❌

程序中就是：检测冲突 → 双方让步 → 又冲突 → 又让步
```

**场景三：分布式锁的活锁**

```java
// Redis 分布式锁的竞争
// 多个客户端同时抢锁，都抢不到就都退避重试
// 如果锁持有时间短、退避时间短 → 大量并发重试 → 活锁

// 客户端 A：尝试 100 次
for (int i = 0; i < 100; i++) {
    if (redis.setnx(lockKey, value, "PX", 100)) {  // 锁 100ms
        // 拿到锁
    }
    // 没拿到，10ms 后重试
    Thread.sleep(10);
}

// 客户端 B：同样逻辑
// 如果 100ms 内锁释放了，两个客户端都会在 10ms 后重试
// 高并发时，大量客户端同时醒来，同时抢 → 又同时失败 → 活锁
```

**场景四：数据库乐观锁重试风暴**

```java
// 乐观锁失败重试
public void updateWithRetry(Long id, UpdateRequest req) {
    int attempts = 0;
    while (attempts < 10) {
        User user = userMapper.selectById(id);
        user.setName(req.getName());
        
        int affected = userMapper.updateById(user);  // version++
        if (affected == 1) {
            return;  // 成功
        }
        // 失败，重试 ⚠️
        attempts++;
    }
    throw new OptimisticLockException("重试超过次数");
}

// 问题：如果热点行（如商品库存）被大量请求同时更新
// 版本号冲突 → 大量请求都重试 → 活锁
// 所有请求都在重试，没人能前进
```

**面试加分话术：**

> "活锁最形象的比喻就是**两个人在走廊互相让路**：你左我也左，你右我也右，永远过不去。和死锁的区别在于：死锁是大家都在**等**（BLOCKED），活锁是大家都在**动**（RUNNABLE），但动的都是无用功。实际业务中活锁比死锁更难发现——程序在跑、CPU 在烧，但业务没有任何进展。预防方法是**加随机退避**（`Thread.sleep(random())`），让重试时间错开，避免所有线程同一时刻醒来。"

## 🔥 扩展追问 2：饥饿具体是怎么发生的？举几个实际场景？

**饥饿的本质：**

```
饥饿 = 某些线程始终得不到资源
        ↓
其他线程正常运行
        ↓
受害线程可能永远得不到执行机会
```

**场景一：低优先级线程饿死**

```java
// 高并发系统里，低优先级任务永远没机会
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(1);

scheduler.scheduleAtFixedRate(() -> {
    // 定时任务，优先级 1（最低）
}, 0, 1, TimeUnit.SECONDS);

// 如果高优先级线程占满 CPU，低优先级线程几乎不会运行
Thread lowPriority = new Thread(() -> {
    while (!done) {
        System.out.println("低优先级任务执行");
    }
});
lowPriority.setPriority(Thread.MIN_PRIORITY);  // 优先级 1

Thread highPriority = new Thread(() -> {
    while (true) {
        // 密集计算，占满 CPU
    }
});
highPriority.setPriority(Thread.MAX_PRIORITY); // 优先级 10

lowPriority.start();
highPriority.start();
// lowPriority 几乎得不到 CPU 时间片 = 饥饿
```

**场景二：synchronized 非公平锁导致饥饿**

```java
// synchronized 是非公平锁
// 新来的线程可能插队，长期占着 CPU

synchronized (lock) {
    while (true) {
        // 长时间占着锁
        // 其他所有等待的线程都饿着
    }
}

// 饥饿线程的状态：BLOCKED（Waiting for monitor entry）
// 持有锁的线程一直在跑，其他线程阻塞等待
```

**场景三：线程池饥饿（任务分配不均）**

```java
// FixedThreadPool，单一队列，所有任务排队
ExecutorService executor = Executors.newFixedThreadPool(2);

// 任务队列：10 个耗时任务
for (int i = 0; i < 10; i++) {
    executor.submit(() -> {
        // 每个任务耗时 1 小时
        Thread.sleep(Duration.ofHours(1));
    });
}

// 问题：
// 线程 1 跑了任务 1, 3, 5, 7, 9
// 线程 2 跑了任务 2, 4, 6, 8, 10
// 看起来公平？

// 但如果任务类型不同：
// 线程 1 一直跑 IO 密集任务（等网络）
// 线程 2 一直跑 CPU 密集任务
// CPU 密集任务抢了大部分 CPU，IO 任务一直等 CPU = 线程 2 相对饥饿
```

**场景四：公平锁 vs 非公平锁的饥饿**

```java
// 非公平锁：可能插队，高并发下吞吐量高，但可能饥饿
ReentrantLock unfair = new ReentrantLock();

// 公平锁：严格 FIFO，高并发吞吐量低，但不会饥饿
ReentrantLock fair = new ReentrantLock(true);

// 公平锁饥饿场景：
// 线程 A 刚释放锁，等着的线程 B 被唤醒
// 新来的线程 C 插队，直接拿到锁
// B 又得等 → 如果 C 一直来，B 可能永远得不到 → 饥饿

// 但 ReentrantLock 公平锁实际用的是 CLH 队列
// 新线程不会真的插队到队列头部
// 所以公平锁本身不造成饥饿，只是响应慢
```

**饥饿 vs 活锁的对比：**

```
饥饿：一个或部分线程饿死，其他正常
        ↓
      ┌─ 持有者一直占着资源
      └─ 受害线程长时间等待

活锁：所有线程都在动，但都在做无用功
        ↓
      ┌─ 没有线程持有资源
      └─ 都在让步、重试、再让步
```

**面试加分话术：**

> "饥饿的典型特征是**有人吃不上饭**：高优先级线程抢光资源、低优先级任务永远排不上、持有锁的线程长时间不释放。饥饿和活锁的区分点是：**饥饿是单方面的**（只有受害方没进展），**活锁是大家都在跑但都没进展**。在 Java 并发里，饥饿通常和线程优先级、锁公平性有关——`Thread.setPriority()` 改变线程优先级，`ReentrantLock(true)` 是公平锁保证 FIFO。用 `jstack` 看饥饿线程的状态，会发现是 `BLOCKED` 或 `WAITING`，而活锁线程是 `RUNNABLE`（在跑但做无用功）。"

## 🔥 扩展追问 3：死锁、活锁、饥饿如何用代码检测和模拟？

**线程状态对照表（jstack 输出）：**

| jstack 状态 | 对应情况 |
|---|---|
| `BLOCKED` | 等待 monitor 锁，死锁候选人 |
| `WAITING` | 等待某个条件，Object.wait() |
| `TIMED_WAITING` | 限时等待，Thread.sleep() |
| `RUNNABLE` | 正常运行，活锁可能在这里 |
| `TERMINATED` | 线程结束 |

**死锁检测代码：**

```java
// 手动检测死锁（基于有向图）
public class DeadLockDetector {
    public static boolean detect() {
        Map<Thread, Set<Object>> threadLocks = new HashMap<>();
        
        // 遍历所有线程，找到它们持有的锁
        for (Thread t : Thread.getAllStackTraces().keySet()) {
            if (t == null) continue;
            Set<Object> held = getHeldLocks(t);
            threadLocks.put(t, held);
        }
        
        // 检测环
        for (Thread t : threadLocks.keySet()) {
            if (hasCycleFrom(t, threadLocks, new HashSet<>())) {
                return true;  // 发现死锁
            }
        }
        return false;
    }
}

// Arthas 一键检测
// thread -b  → 直接输出死锁信息
```

**活锁检测代码：**

```java
// 监控线程是否在"空转"
public class LivelockDetector {
    public static void detect() {
        for (Thread t : Thread.getAllStackTraces().keySet()) {
            StackTraceElement[] stack = t.getStackTrace();
            
            // 如果线程一直在同一个方法里跑
            // 可能是活锁
            if (isSpinning(t, stack)) {
                System.out.println("检测到活锁线程: " + t.getName());
            }
        }
    }
    
    private static boolean isSpinning(Thread t, StackTraceElement[] stack) {
        if (t.getState() != Thread.State.RUNNABLE) return false;
        
        // 检测是否在 while 循环里空转
        for (StackTraceElement frame : stack) {
            if (frame.getMethodName().equals("run") 
                && frame.getClassName().contains("Lambda")) {
                return true;  // 可能在自旋
            }
        }
        return false;
    }
}
```

**饥饿检测：**

```java
// 监控各线程的 CPU 占用
public class StarvationDetector {
    public static void detect() {
        long[] cpuBefore = new long[Thread.activeCount()];
        Thread[] threads = new Thread[Thread.activeCount()];
        int count = Thread.enumerate(threads);
        
        for (int i = 0; i < count; i++) {
            if (threads[i] == null) continue;
            cpuBefore[i] = getThreadCpuTime(threads[i]);
        }
        
        try {
            Thread.sleep(1000);  // 采样 1 秒
        } catch (InterruptedException e) {}
        
        for (int i = 0; i < count; i++) {
            if (threads[i] == null) continue;
            long cpuAfter = getThreadCpuTime(threads[i]);
            long delta = cpuAfter - cpuBefore[i];
            
            // 如果线程 CPU 占用极低或为 0，可能饥饿
            if (delta < 10_000_000) { // < 10ms CPU 时间
                System.out.println("可能饥饿的线程: " + threads[i].getName() 
                    + "，状态: " + threads[i].getState());
            }
        }
    }
}
```

```java
public class AllThreeDeadlocks {
    
    // 死锁模拟
    public static void deadlock() throws InterruptedException {
        Object A = new Object();
        Object B = new Object();
        
        Thread t1 = new Thread(() -> {
            synchronized (A) {

                synchronized (B) {  // 等 B

        Thread t2 = new Thread(() -> {

                }
            }

    public static void livelock() throws InterruptedException {
        AtomicBoolean t1HasMoved = new AtomicBoolean(false);
        AtomicBoolean t2HasMoved = new AtomicBoolean(false);
        
        Thread t1 = new Thread(() -> {
            while (true) {
                if (!t1HasMoved.get()) {
                    // 移动一步
                    t1HasMoved.set(true);
                    t2HasMoved.set(false);
                }
                sleep(1);
            }
        });
        
        Thread t2 = new Thread(() -> {
            while (true) {
                if (!t2HasMoved.get()) {
                    // 检测冲突，回退
                    t2HasMoved.set(true);
                    t1HasMoved.set(false);  // 让 t1 动
                }
                sleep(1);
            }
        });
        
        t1.start(); t2.start();
    }
    
    // 饥饿模拟
    public static void starvation() {
        Thread high = new Thread(() -> {
            while (true) { /* 空转 */ }
        });
        high.setPriority(Thread.MAX_PRIORITY);
        
        Thread low = new Thread(() -> {
            while (true) {
                System.out.println("低优先级执行了一次");
                sleep(1000);  // 等 1 秒才有机会
            }
        });
        low.setPriority(Thread.MIN_PRIORITY);
        
        high.start();
        low.start();
    }
}
```

> "排查三种问题用 `jstack` 是万能的：**死锁**直接看 `BLOCKED` 线程，`jstack` 会打印 'Found N deadlock'；**活锁**看 `RUNNABLE` 但 CPU 高，线程在 while 循环里空转；**饥饿**看 `BLOCKED` 或 `WAITING` 线程中是否有某个一直饿着。生产环境建议用 **Arthas 的 `thread -b`** 自动检测死锁，**Pinpoint / SkyWalking** 做线程监控。理论要结合实际——如果能讲清楚你排查过的线上活锁问题（某接口超时但 CPU 不高、线程堆栈全在重试），面试官会非常感兴趣。"

## 🔥 扩展追问 4：如何解决活锁？退避策略的原理和实现？

**活锁的根本原因是"太积极"：**

```
冲突 → 双方让步 → 又冲突 → 又让步 → 无限循环
        ↓
解决方案：让步之后，退避一段时间再重试
        ↓
让冲突方有时间错开执行
```

**随机退避策略（最常用）：**

```java
// ❌ 无退避 → 活锁
while (attempts < MAX) {
    if (tryLock()) return SUCCESS;
    // 立刻重试，所有线程同一时刻醒来，同时失败
}

// ✅ 随机退避 → 打破活锁
Random random = new Random();
while (attempts < MAX) {
    if (tryLock()) return SUCCESS;
    
    // 退避：0~100ms 随机等待
    long backoff = random.nextInt(100);
    Thread.sleep(backoff);  // 时间错开，避免再次冲突
    
    attempts++;
}
```

**指数退避策略（更适合分布式）：**

```java
// 重试次数越多，等待时间越长
public class ExponentialBackoff {
    private static final long INITIAL_BACKOFF_MS = 10;
    private static final long MAX_BACKOFF_MS = 1000;
    
    public static <T> T execute(Supplier<T> action, int maxAttempts) {
        long backoff = INITIAL_BACKOFF_MS;
        
        for (int i = 0; i < maxAttempts; i++) {
            try {
                return action.get();
            } catch (Exception e) {
                if (i == maxAttempts - 1) throw e;
                
                // 指数退避：10ms → 20ms → 40ms → 80ms → ... → 1000ms
                Thread.sleep(backoff);
                backoff = Math.min(backoff * 2, MAX_BACKOFF_MS);
                
                // 加随机抖动（jitter），避免多线程同时重试
                Thread.sleep(random.nextInt((int) backoff));
            }
        }
        throw new RuntimeException("重试耗尽");
    }
}

// Redis / 分布式锁推荐使用：公式
// delay = min(cap, base * 2^attempt + random_jitter)
```

**CAS 自旋 + 退避：**

```java
// 原子变量自旋 + 退避
public class BackoffCAS {
    private final AtomicInteger counter = new AtomicInteger(0);
    private final Random random = new Random();
    
    public void increment() {
        int attempts = 0;
        while (true) {
            int current = counter.get();
            if (counter.compareAndSet(current, current + 1)) {
                return;
            }
            
            // 自旋次数越多，退避越长
            if (attempts > 10) {
                // 重试 10 次还不成功，sleep 一下
                LockSupport.parkNanos(random.nextInt(100_000)); // 0~100μs
            }
            attempts++;
        }
    }
}
```

**线程池 + 信号量解决活锁：**

```java
// 限制并发数量，避免活锁
Semaphore permits = new Semaphore(2);  // 最多 2 个并发

public void process(Task task) {
    try {
        permits.acquire();
        try {
            // 业务逻辑
        } finally {
            permits.release();
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
}

// 配合队列优先级
PriorityBlockingQueue<Task> queue = new PriorityBlockingQueue<>();
// 优先级高的先处理，避免低优先级永远饿死
```

**面试加分话术：**

> "活锁的解决方案说白了就是一句话：**退避 + 随机**。无论是 Redis 锁的 `random sleep`、HTTP 重试的指数退避 + jitter，还是 CAS 自旋的 `parkNanos`，核心思想都一样——让冲突方错开执行时间。实际代码里特别要注意加 **Jitter（抖动）**，不然所有线程退避 10ms 后同时醒来，还是会冲突。指数退避适合分布式场景（网络请求），固定退避适合单机（线程池竞争）。如果面试官追问 jitter，可以补充：**Full Jitter**（`random(0, base)`）比 **Equal Jitter**（`base + random(0, base/2)`）在分布式场景下更能分散压力，是 AWS 的工程师推荐的。"

## 🔥 扩展追问 5：饥饿如何解决？线程优先级、锁公平性、公平队列？

**解决饥饿的策略：**

```
饥饿的根因：某个/某些线程得不到资源
        ↓
解决方案：
  ① 提高优先级（慎用，不推荐）
  ② 用公平锁（FIFO）
  ③ 用公平队列（真正解决）
  ④ 减少持有时间
```

**策略一：公平锁（ReentrantLock）**

```java
// ❌ 非公平锁：可能插队，高吞吐但可能饥饿
ReentrantLock unfair = new ReentrantLock();

// ✅ 公平锁：按等待顺序获取（FIFO）
ReentrantLock fair = new ReentrantLock(true);

// 公平锁原理：CLH 队列严格排队
// 新来的线程如果队列非空，必须排到队尾
// 不会插队 → 不会饥饿
```

**公平锁 vs 非公平锁的选择：**

| 场景 | 推荐锁 | 原因 |
|---|---|---|
| 高并发网关 | 非公平 | 吞吐量大，允许插队 |
| 金融交易 | 公平 | 不容许任何饥饿 |
| 短临界区 | 非公平 | 持有时间短，不容易饿 |
| 长临界区 | 公平 | 长任务会让其他线程等太久 |

**策略二：多队列 + 负载均衡**

```java
// 多队列负载均衡：避免单队列饥饿
// 比如线程池 + 多个队列
public class MultiQueueExecutor {
    private final List<BlockingQueue<Runnable>> queues;
    private final List<Thread> workers;
    
    public MultiQueueExecutor(int queueCount, int workersPerQueue) {
        queues = new ArrayList<>();
        workers = new ArrayList<>();
        
        for (int i = 0; i < queueCount; i++) {
            BlockingQueue<Runnable> queue = new LinkedBlockingQueue<>();
            queues.add(queue);
            
            for (int j = 0; j < workersPerQueue; j++) {
                Thread worker = new Thread(() -> {
                    while (!Thread.interrupted()) {
                        try {
                            // 阻塞获取
                            Runnable task = queue.take();
                            task.run();
                        } catch (InterruptedException e) {
                            break;
                        }
                    }
                });
                workers.add(worker);
                worker.start();
            }
        }
    }
    
    public void submit(Runnable task) {
        // 轮询提交到不同队列
        int queueIdx = nextQueue.getAndIncrement() % queues.size();
        queues.get(queueIdx).offer(task);
    }
}
```

**策略三：减少锁持有时间**

```java
// ❌ 长临界区 → 其他线程饿着
public void badMethod() {
    synchronized (lock) {
        doStep1();  // 耗时 1 秒
        callExternalService();  // 耗时 5 秒
        doStep3();  // 耗时 1 秒
    }
}

// ✅ 缩小临界区
public void goodMethod() {
    // 只在锁内做必须同步的操作
    synchronized (lock) {
        prepareData();
    }
    
    // 耗时的放锁外
    Result result = callExternalService();  // 5 秒，锁外
    
    synchronized (lock) {
        updateState(result);  // 很快
    }
}
```

**策略四：使用 StampedLock 的乐观读**

```java
// StampedLock：读多写少场景，减少写锁饥饿
StampedLock lock = new StampedLock();

public Object read() {
    // 乐观读：不加锁，先试试
    long stamp = lock.tryOptimisticRead();
    Object data = this.data;  // 读取
    
    // 检查读期间是否有写操作
    if (!lock.validate(stamp)) {
        // 有写发生，升级为悲观读
        stamp = lock.readLock();
        try {
            data = this.data;
        } finally {
            lock.unlockRead(stamp);
        }
    }
    return data;
}

public void write(Object newData) {
    long stamp = lock.writeLock();
    try {
        this.data = newData;
    } finally {
        lock.unlockWrite(stamp);
    }
}
```

**面试加分话术：**

> "饥饿的解决思路有三条：**给饿的线程开绿灯**（提高优先级，不推荐）、**强制公平排队**（公平锁，吞吐量低）、**减少饥饿根源**（缩小临界区、负载均衡）。Java 里的公平锁（`ReentrantLock(true)`）用的是 CLH 队列，严格 FIFO，不会插队，但吞吐量比非公平锁低 3~5 倍，因为线程切换开销大。所以高并发系统通常用非公平锁靠高吞吐弥补饥饿，但如果业务上**某个操作不容许饿**（比如定时任务扣库存），就用公平锁或者单独建一个高优先级队列。StampedLock 的乐观读也是减少饥饿的好办法——读不加锁，只有检测到冲突才升级成悲观读。"

## 🔥 扩展追问 6：实际业务中如何区分这三种情况？

**排查流程图：**

```
收到告警：某接口超时/无响应
        ↓
jstack 抓取线程堆栈
        ↓
┌───────────────────────────────────────────────┐
│ 查看线程状态：                                 │
│                                               │
│ RUNNABLE → 可能是活锁（CPU 高）               │
│ BLOCKED  → 可能是死锁或饥饿（CPU 低）          │
│ WAITING  → 等待某些条件                       │
└───────────────────────────────────────────────┘
        ↓
CPU 使用率判断
        ↓
┌───────────────────────────────────────────────┐
│ CPU 高 + RUNNABLE → 活锁                      │
│   └─ 找 while/for 循环                        │
│   └─ 检测到冲突就重试                          │
│                                               │
│ CPU 低 + BLOCKED → 死锁                        │
│   └─ jstack 提示 Found N deadlock              │
│   └─ 或者线程互相等锁                          │
│                                               │
│ CPU 正常 + 部分线程长期 BLOCKED → 饥饿         │
│   └─ 某些线程始终等不到资源                    │
└───────────────────────────────────────────────┘
```

**实际案例分析：**

```java
// 案例1：支付接口超时（活锁）
// 监控发现 CPU 占用 80%，接口响应时间 > 30s
// jstack 看到多个线程在：
while (true) {
    try {
        lock.tryLock(1, TimeUnit.MILLISECONDS);
        if (acquired) break;
    } catch (...) {}
}
// 所有线程都在 tryLock → 超时 → 重试 → tryLock → ...（活锁）

// 案例2：订单创建卡死（死锁）
// 监控发现 CPU 占用 0%，接口完全无响应
// jstack 看到：
// "pool-1-thread-1" waiting for monitor entry
//   - locked <0x000000076ab00038>
// "pool-1-thread-2" waiting for monitor entry
//   - locked <0x000000076ab00040>
// Found 1 deadlock!
// 线程 A 持有 X 等待 Y，线程 B 持有 Y 等待 X → 死锁

// 案例3：定时任务一直不执行（饥饿）
// 监控发现某个定时任务近 1 小时没执行
// 其他定时任务正常
// jstack 看到该线程状态 RUNNABLE，但始终得不到 CPU
// 原因：更高优先级的线程占满了 CPU
```

**快速判断口诀：**

```
CPU 高 → RUNNABLE → 大概率活锁
CPU 低 → BLOCKED → 看 jstack 是否有 deadlock 提示
       ↓
       有 deadlock → 死锁
       无 deadlock → 可能是饥饿（部分线程长期 BLOCKED）
```

**面试加分话术：**

> "区分三种情况有一个万能口诀：**CPU 高是活锁，CPU 低是死锁，部分饿是饥饿**。生产环境我通常这样做：先 `top -Hp` 看 CPU，`jstack` 抓堆栈。如果 CPU 很高 + 线程都是 RUNNABLE 在跑，那活锁概率大——活锁的线程不占锁，但占 CPU。如果是 CPU 很低 + 线程都 BLOCKED，jstack 看一下有没有 'Found N deadlock'，有就是死锁，没有就是饥饿（某些线程等不到资源）。我之前踩过一个坑：接口响应 500ms，CPU 不高，以为是数据库慢，加上索引还是不行，最后 jstack 发现两个线程死锁了——A 等 B 的结果，B 等 A 的锁。所以遇到超时问题，**第一反应就是 jstack**，比加日志快 10 倍。"

## 🔥 扩展追问 7：synchronized 会导致饥饿吗？ReentrantLock 公平 vs 非公平？

**synchronized 是非公平锁：**

```java
// synchronized → 非公平锁
// 可能新线程插队，老线程长期等待 → 饥饿

synchronized (this) {
    // 长时间持有锁
    while (true) {
        // 其他所有线程都饿着
    }
}

// synchronized 不保证 FIFO
// 新来的线程可能直接抢到锁
// 等待队列中的线程可能永远轮不到
```

**ReentrantLock 的公平 vs 非公平：**

```java
// 非公平（默认）：吞吐量高，可能饥饿
ReentrantLock unfairLock = new ReentrantLock();

// 公平：严格 FIFO，不会饥饿，但吞吐量低
ReentrantLock fairLock = new ReentrantLock(true);

// 公平锁的 hasQueuedThreads() 可以查有没有等锁的线程
if (fairLock.hasQueuedThreads()) {
    System.out.println("有人在排队等锁");
}
```

**ReentrantLock 公平锁的实现（CLH 队列）：**

```
新线程来获取锁：
        ↓
查看队列是否为空
        ↓
空 → 直接 CAS 抢锁（不需要排队）
        ↓
非空 → 构造 Node，加入队列尾部
        ↓
自旋等待前驱节点释放锁
        ↓
收到前驱的 unpark()，轮到自己了
```

**饥饿 vs 公平锁的关系：**

```
┌─────────────────────────────────────────────────────┐
│ 非公平锁：                                           │
│ ├─ 新线程可能插队                                   │
│ ├─ 吞吐量大                                        │
│ └─ 可能导致等待线程饥饿                              │
├─────────────────────────────────────────────────────┤
│ 公平锁：                                             │
│ ├─ 严格 FIFO，队列顺序                              │
│ ├─ 不会导致饥饿                                     │
│ └─ 吞吐量低（线程切换多）                           │
├─────────────────────────────────────────────────────┤
│ 自旋锁（Spin Lock）：                                 │
│ ├─ CAS + 自旋                                       │
│ ├─ 极端情况下某些线程可能一直自旋抢不到               │
│ └─ 适合临界区极短的情况                              │
└─────────────────────────────────────────────────────┘
```

**实战选择建议：**

```java
// 高并发短临界区：非公平（默认），吞吐量优先
ReentrantLock lock = new ReentrantLock();

// 低并发长临界区：公平，保证响应时间
ReentrantLock fairLock = new ReentrantLock(true);

// 写多读少：StampedLock 乐观读，减少写锁饥饿
StampedLock sl = new StampedLock();
```

**面试加分话术：**

> "synchronized 是非公平锁，ReentrantLock 可以选公平或非公平。非公平锁吞吐量大是因为允许插队——持有锁的线程释放后，新来的线程可以直接抢，不用进队列等。但代价是队列里的线程可能饿死。公平锁用 CLH 队列严格排队，不会饿，但每次都要入队出队，线程切换开销大。一般来说：**高并发网关用非公平**（吞量优先），**金融交易用公平**（不容许任何饥饿），**读多写少用 StampedLock**（乐观读减少写锁饥饿）。面试还能加一句：JDK 1.8 之后 synchronized 做了大量优化（偏向锁、轻量级锁、锁消除），在大多数场景下性能已经和 ReentrantLock 持平甚至更好。"

## 🔥 扩展追问 8：总结对比 + 面试口诀

**三者终极对比表：**

| 维度 | 死锁 | 活锁 | 饥饿 |
|---|---|---|---|
| **线程状态** | BLOCKED | RUNNABLE | BLOCKED/WAITING |
| **CPU 占用** | 无（0%） | 高（100%） | 正常或低 |
| **是否等锁** | ✅（互相等） | ❌（都在抢） | ⚠️（等不到） |
| **是否在动** | ❌（不动） | ✅（空转） | ❌（等不到） |
| **自动恢复** | ❌ 不能 | ⚠️ 可能 | ⚠️ 可能 |
| **JVM 检测** | `jstack Found N deadlock` | ❌ 无 | ❌ 无 |
| **解决方案** | 打破循环等待/加超时 | 退避+随机 | 公平锁/提高优先级 |

**面试口诀：**

```
死锁：大家都在等，死都不动（CPU 0%，jstack 直接报）
活锁：大家都在动，做无用功（CPU 满%，while 循环空转）
饥饿：有人吃不上饭，其他人正常（CPU 正常，部分线程饿死）

死锁 → 打破循环（固定顺序/加超时）
活锁 → 退避随机（Thread.sleep(random)）
饥饿 → 公平排队（ReentrantLock(true)）
```

**面试加分话术：**

> "这道题的核心是理解三种不同的'卡住'状态：**死锁是大家都不动互相等、死锁是 CPU 0%；活锁是大家都在动但做无用功，CPU 100%；饥饿是有人始终没机会，CPU 正常但某些线程饿死**。记不住的话就想生活场景：死锁 = 两个人互相等对方先开口，都不说话；活锁 = 两个人互相让路，你左我也左，永远过不去；饥饿 = VIP 插队，普通人等了半天没轮到。面试时如果能画出三个图示，面试官会觉得你理解得很透彻。"

## 一句话总结（面试收尾用）

> "死锁、活锁、饥饿都是多线程的'卡住'现象，但本质不同：**死锁**是大家都在等（BLOCKED，CPU 0%），**活锁**是大家都在跑但没进展（RUNNABLE，CPU 100%，while 空转），**饥饿**是有人始终吃不上饭（部分线程长期 BLOCKED）。区分方法是看 `jstack` 输出 + CPU 占用率。解决方案分别是：死锁加超时或固定顺序，活锁加退避随机，饥饿用公平锁或减少锁持有时间。面试能讲清楚这三个的区别和各自应对策略，基本就是满分答案了。"
