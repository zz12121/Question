### 基础流程

```
序列化：对象 → 字节序列（存储/传输）
        ↓
实现方式：
    1. 实现 Serializable 接口（最常用）
    2. 实现 Externalizable 接口（自定义序列化逻辑）
    3. 实现 java.io.Serializable（标记接口，无方法）
        ↓
反序列化：字节序列 → 对象
        ↓
关键点：serialVersionUID 版本校验
```

**最简单的序列化实现：**

```java
// 步骤1：实现 Serializable
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    
    private String name;
    private int age;
    private transient String password;  // transient：不序列化
    
    // 不需要实现任何方法！
}

// 序列化
User user = new User("张三", 25, "secret");
ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("user.txt"));
oos.writeObject(user);

// 反序列化
ObjectInputStream ois = new ObjectInputStream(new FileInputStream("user.txt"));
User u = (User) ois.readObject();
```

## 🔥 扩展追问 1：serialVersionUID 是什么？不声明会怎样？

**作用：版本校验**

```
序列化时：
    对象 → 计算 serialVersionUID → 写入字节流
        ↓
反序列化时：
    字节流里的 UID ←→ 类的 UID
        ↓
  匹配 ✓ → 正常反序列化
  不匹配 ✗ → InvalidClassException！
```

**两种声明方式：**

```java
// 方式1：显式声明（推荐）
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    // ...
}

// 方式2：不声明（不推荐！）
// JVM 会自动生成，但规则复杂：
// - 根据类名、接口、成员计算哈希
// - 改任何代码都可能导致 UID 变化
// - 不同 JDK 版本、不同编译器实现可能不同
```

**自动生成 UID 的问题：**

```java
public class User implements Serializable {
    private String name;
}

// 序列化后保存到文件

// 不小心加了新字段
public class User implements Serializable {
    private String name;
    private int age;  // 字段变了，UID 也变了！
}

// 反序列化 → InvalidClassException
// 因为 age 变了，JVM 重新计算的 UID 和文件里的对不上
```

**面试加分话术：**

> "serialVersionUID 是序列化的版本契约。**强烈建议每个 Serializable 类都显式声明**，因为自动生成不可控——你改一个字段、加一个方法、甚至换个 JDK 版本，UID 可能就变了。另外要注意：UID 只和**签名相关**的成员有关，`static` 和 `transient` 字段不影响 UID 计算，但加 `private static final` UID 本身就是签名的一部分。"

## 🔥 扩展追问 2：transient 关键字和 static 字段会被序列化吗？

**三类字段的命运：**

| 字段类型 | 序列化 | 说明 |
|---|---|---|
| 普通字段 | ✅ | 正常序列化 |
| `transient` | ❌ | 被跳过 |
| `static` | ❌ | 属于类，不属于对象 |

**transient 的典型用法：**

```java
public class User implements Serializable {
    private String name;
    
    // 密码不想存，太敏感
    private transient String password;
    
    // 缓存数据，不需要序列化（重新查就行）
    private transient List<Order> cachedOrders;
    
    // 敏感字段也可以用自定义序列化处理（下面讲）
}

User user = new User("张三", "123456");
ObjectOutputStream oos = ...;
oos.writeObject(user);

// 反序列化后：
// user.name = "张三"
// user.password = null  ← transient，被跳过了
// user.cachedOrders = null
```

**static 字段的特殊情况：**

```java
public class User implements Serializable {
    private String name;
    
    // static 属于类，不属于对象实例
    private static String company = "某公司";  
    
    // 普通 static 也不会被序列化
    private static int count;  
}

// 反序列化后：
// user.name = "张三"（恢复了）
// User.company = "某公司"（取决于类加载时的值，不受序列化影响）
// User.count = 0（重新初始化）
```

**面试加分话术：**

> "记住原则：**序列化的是对象的状态，不是类的状态**。static 属于类，所以不序列化；transient 是显式声明跳过的字段。需要强调的是：transient 不是用来保护敏感数据的——它只是不序列化，字节流里根本没有这个字段的值，如果有人拿到字节流也能猜出格式。更安全的方式是**自定义序列化**，对敏感数据加密后再存。"

## 🔥 扩展追问 3：自定义序列化怎么做？writeObject 和 readObject

**为什么需要自定义？**

```
默认序列化问题：
    1. transient 字段变成 null，无法恢复有用状态
    2. 想要加密/压缩/做业务校验
    3. 序列化某些派生字段而不是原始字段
```

**自定义序列化的写法：**

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    
    private String name;
    private transient String password;  // 密码序列化时加密
    private transient int loginCount;   // 不需要序列化
    
    // 序列化时会调用这个
    private void writeObject(ObjectOutputStream oos) throws IOException {
        oos.defaultWriteObject();  // 先走默认序列化
        oos.writeObject(encrypt(password));  // 再写入加密后的密码
    }
    
    // 反序列化时会调用这个
    private void readObject(ObjectInputStream ois) throws IOException, ClassNotFoundException {
        ois.defaultReadObject();  // 先走默认反序列化
        this.password = decrypt((String) ois.readObject());  // 再解密
        this.loginCount = 0;  // 重置为 0
    }
    
    private String encrypt(String pwd) {
        // 加密逻辑
        return Base64.getEncoder().encodeToString(pwd.getBytes());
    }
    
    private String decrypt(String encrypted) {
        // 解密逻辑
        return new String(Base64.getDecoder().decode(encrypted));
    }
}
```

**Externalizable vs Serializable：**

```java
// Externalizable：完全控制序列化逻辑
public class User implements Externalizable {
    public String name;
    public int age;
    
    @Override
    public void writeExternal(ObjectOutput out) throws IOException {
        out.writeObject(name);  // 手动写每个字段
        out.writeInt(age);
    }
    
    @Override
    public void readExternal(ObjectInput in) throws IOException, ClassNotFoundException {
        this.name = (String) in.readObject();
        this.age = in.readInt();
    }
}
```

**两者的区别：**

| 方面 | Serializable | Externalizable |
|---|---|---|
| 实现方式 | 标记接口 | 需要实现两个方法 |
| 定制程度 | 低（默认 + 自定义 hook） | 完全控制 |
| 性能 | 一般（反射检测） | 好（自己写，不用反射） |
| 默认行为 | 自动序列化所有非 transient | 完全不自动序列化 |
| 使用场景 | 绝大多数场景 | 追求极致性能或特殊格式 |

**面试加分话术：**

> "99% 的场景用 `Serializable + writeObject/readObject` 就够了。Externalizable 适合对序列化性能有极致要求、或者需要控制老数据兼容性的场景（比如你发的版本里要能读旧格式）。但要注意：Externalizable 反序列化时会调用**无参构造方法**创建对象，所以必须保证有无参构造，否则抛异常。Serializable 不需要无参构造，因为它是用对象内存直接恢复的。"

## 🔥 扩展追问 4：Java 原生序列化有什么性能问题？为什么 RPC 框架不用它？

**性能问题：**

```
原生序列化的三大缺点：
        ↓
① 字节流臃肿
    └─ 包含类名、字段名、类型信息
    └─ 一个 User 对象序列化后可能是 JSON 的 3~5 倍

② 速度慢
    └─ 反射 + 递归遍历字段
    └─ 比 JSON/Protobuf 慢 10~100 倍

③ 安全性问题
    └─ readObject 可以执行任意代码
    └─ 反序列化漏洞（Apache Commons Collections、Jackson）
```

**字节流大小对比：**

```java
User user = new User("张三", 25, "男", "北京");

// Java 序列化字节流（约 100+ 字节）
// 包含：类全限定名、serialVersionUID、各字段类型+名称+值...

// JSON（约 40 字节）
// {"name":"张三","age":25,"gender":"男","city":"北京"}

// Protobuf 二进制（约 15 字节）
// 紧凑二进制，无字段名，只有编号
```

**反序列化漏洞示例：**

```java
// 恶意构造的字节流
// 当 ObjectInputStream.readObject() 时
// 会执行类里的 readObject 逻辑
// 如果类里写了危险代码...

public class Evil implements Serializable {
    private void readObject(ObjectInputStream ois) throws Exception {
        ois.defaultReadObject();
        Runtime.getRuntime().exec("rm -rf /");  // 执行任意命令！
    }
}

// 只要有人反序列化这个对象，命令就执行了
// 这就是著名的反序列化漏洞链（ysoserial）
```

**为什么 RPC 框架不用 Java 序列化：**

| 框架 | 序列化方案 | 原因 |
|---|---|---|
| Dubbo | Hessian / Protostuff / Kryo | 速度快，字节小，跨语言支持 |
| gRPC | Protocol Buffers | 跨语言、IDL 定义、向后兼容 |
| Feign | Jackson / Gson | JSON 可读性好，前端兼容 |
| Spring Cloud | JSON | 通用性好，调试方便 |

**面试加分话术：**

> "RPC 框架不用 Java 原生序列化主要三个原因：**大、慢、危险**。字节流里带着类名和字段名，数据量膨胀好几倍；反射检测开销大，比 JSON/Protobuf 慢几十倍；最致命的是反序列化漏洞——攻击者可以构造恶意字节流让你执行任意代码，像 Apache Commons Collections、Jackson 都爆过这种漏洞。所以生产环境 Dubbo 一般配 Hessian 或 Fastjson2，Spring Cloud 默认用 Jackson，都是成熟方案。"

## 🔥 扩展追问 5：序列化版本的兼容性问题怎么处理？

**场景：线上跑着 v1.0，升级到 v2.0 要兼容旧数据**

**加字段（常见）：**

```java
// v1.0
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
}

// v2.0：加了 age 字段
public class User implements Serializable {
    private static final long serialVersionUID = 1L;  // UID 不变！
    private String name;
    private int age;  // 新字段，反序列化时用默认值 0
}

// 结果：旧数据能读，新字段是默认值
```

**删字段：**

```java
// v2.0：删了 phone 字段
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
    // phone 字段删了
    
    // 新加的字段
    private String email;
}

// 结果：旧数据能读，phone 数据丢失（本来就不要了）
```

**改字段（危险）：**

```java
// v1.0
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private String phone;  // String 类型
}

// v2.0：改成 int 类型 ❌ 危险！
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private int phone;  // int 类型，UID 相同
}

// 反序列化旧数据：phone = 0（因为旧 String 的 hash 值不是数字）
// 或者抛 InvalidClassException（取决于字段类型兼容性）
```

**正确的字段变更策略：**

```
┌─────────────────────────────────────────────────────────┐
│ 安全的变更                                               │
├─────────────────────────────────────────────────────────┤
│ ✅ 加字段（serialVersionUID 不变）                       │
│ ✅ 删字段（serialVersionUID 不变）                       │
│ ✅ 改字段名（serialVersionUID 不变）                     │
│ ✅ 加 transient 字段                                     │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ 危险的变更                                               │
├─────────────────────────────────────────────────────────┤
│ ❌ 改字段类型                                            │
│ ❌ 改 serialVersionUID                                   │
│ ❌ 改 static 字段（不影响，但没意义）                    │
│ ❌ 改 final 字段的默认值（会丢失）                        │
└─────────────────────────────────────────────────────────┘
```

**版本兼容最佳实践：**

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    
    // 始终保持这个 UID 不变
    private String name;
    
    // 新版本加字段：用默认值
    private int version = 1;  // 或加版本标识
    
    // 如果必须改类型：用两个字段过渡
    // 旧字段：标记为 @Deprecated
    // 新字段：用于新版本
    @Deprecated
    private transient String oldPhone;  // 旧格式，用完就删
    private String phone;  // 新格式
}
```

**面试加分话术：**

> "序列化兼容性核心原则是：**UID 不变，加删字段相对安全，改类型危险**。实战中我建议：每个版本升级前，先想好这个类未来会不会改；如果可能改，就用 ProtoBuf 这种用 IDL 定义的方案，字段用编号而不是名字，删字段也安全。另外 `defaultWriteObject()` 后记得 `defaultReadObject()` 配对，否则可能漏掉某些字段。"

## 🔥 扩展追问 6：单例类和序列化怎么配合？readResolve 是干什么的？

**问题：序列化会破坏单例**

```java
// 普通单例
public class Singleton implements Serializable {
    private static final Singleton INSTANCE = new Singleton();
    
    private Singleton() {}  // 私有构造，外部不能 new
    
    public static Singleton getInstance() {
        return INSTANCE;
    }
}

// ❌ 序列化破坏单例！
Singleton s1 = Singleton.getInstance();
Singleton s2 = (Singleton) ois.readObject();
// s1 == s2？ ❌ false！
// 因为 readObject 重新 new 了一个对象！
```

**readResolve 修复：**

```java
public class Singleton implements Serializable {
    private static final Singleton INSTANCE = new Singleton();
    
    private Singleton() {}
    
    public static Singleton getInstance() {
        return INSTANCE;
    }
    
    // 关键：反序列化完成后，JVM 会调用这个方法
    // 返回值会替换反序列化生成的对象
    private Object readResolve() {
        return INSTANCE;  // 返回唯一的实例
    }
}

// ✅ 修复后
Singleton s1 = Singleton.getInstance();
Singleton s2 = (Singleton) ois.readObject();
// s1 == s2？ ✅ true！
```

**readResolve 原理：**

```
ObjectInputStream.readObject()
        ↓
创建新对象（用无参构造）
        ↓
从字节流读数据，填充字段
        ↓
检查是否有 readResolve()
        ↓
调用 readResolve()，返回值替换原对象
        ↓
返回替换后的对象
```

**面试加分话术：**

> "readResolve 是反序列化的"后处理钩子"，除了修复单例外，还常用于**枚举类型**——JDK 官方推荐用枚举实现单例，就是因为枚举的序列化有特殊处理，不需要 readResolve。但注意：readObject 返回的对象会被 JVM 直接丢弃，如果有代价很高的初始化逻辑（比如建连接池），建议在 readResolve 里也做清理，避免浪费。"

## 🔥 扩展追问 7：序列化和反序列化有哪些替代方案？各有什么优缺点？

**主流序列化方案对比：**

| 方案 | 速度 | 字节大小 | 跨语言 | 可读性 | 场景 |
|---|---|---|---|---|---|
| **Java 原生** | 慢 | 大 | ❌ | 差 | 内部临时存储 |
| **JSON (Jackson)** | 中 | 中 | ✅ | ✅ | Web API |
| **Fastjson2** | 快 | 中 | ✅ | ✅ | 高并发 API |
| **XML** | 慢 | 很大 | ✅ | ✅ | 老系统集成 |
| **Hessian** | 快 | 小 | ✅ | ❌ | Dubbo RPC |
| **Protobuf** | 极快 | 极小 | ✅ | ❌ | 追求性能 |
| **Kryo** | 极快 | 小 | ❌ | ❌ | 内部高性能 |
| **Avro** | 快 | 小 | ✅ | ❌ | 大数据 |

**JSON vs Protobuf 对比：**

```java
// Protobuf 定义文件
message User {
    string name = 1;  // 编号代替字段名
    int32 age = 2;
}

// 序列化后字节流：
// [08 12 10 E5 BC A0 E4 B8 89]
// 08 = name 字段编号+类型
// 12 = 长度 18 字节
// 10 = age 字段编号+类型
// 12 = 值 18

// 相比 JSON 少了字段名，大幅节省空间
```

**Dubbo 如何配置序列化：**

```xml
<!-- dubbo.properties -->
dubbo.protocol.serialization=hessian2
dubbo.protocol.compact=true
```

```java
// Spring Boot 配置
@DubboReference(serialization = "fastjson2")
private UserService userService;
```

**面试加分话术：**

> "选序列化方案要看场景：**对外 API 用 JSON**（可读性好、前后端通用），**内部 RPC 用 Protobuf 或 Hessian**（速度快、字节小），**大数据场景用 Avro**（Schema 演进做得好）。有一点要注意：JSON 虽然可读，但有**反序列化漏洞**，Fastjson 1.x 之前爆过好几次，建议升级到 2.x 或者换成 Gson。另外 Protobuf 需要预定义 IDL，字段编号不能改，改了要升级版本号，否则新老数据不兼容。"

## 🔥 扩展追问 8：序列化在分布式系统中是怎么用的？ Session 复制、Redis 缓存等场景？

**分布式 session 复制：**

```
浏览器请求 → 负载均衡 → Server A
                              ↓
                    Session.setAttribute("user", user)
                              ↓
                    序列化 user → 发送给 Server B
                              ↓
                    Server B 反序列化 user → Session 中可用
```

```java
// Spring Session + Redis 序列化配置
@Configuration
public class SessionConfig {
    @Bean
    public CookieSerializer cookieSerializer() {
        DefaultCookieSerializer serializer = new DefaultCookieSerializer();
        serializer.setCookieName("SESSION");
        serializer.setCookiePath("/");
        serializer.setDomainNamePattern("^.+?\\.(\\w+\\.[a-z]+)$");
        return serializer;
    }
    
    @Bean
    public RedisSerializer<Object> redisSerializer() {
        // FastJson2 序列化比 JDK 原生快 5 倍
        return new GenericFastJson2RedisSerializer();
    }
}
```

**Redis 缓存序列化：**

```java
// RedisTemplate 默认用 JDK 序列化
// 问题：字节大、可读性差、跨语言不通用

// 改成 JSON 序列化
@Configuration
public class RedisConfig {
    @Bean
    public RedisTemplate<String, Object> redisTemplate(
            RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);
        
        // 用 Jackson 序列化 value
        Jackson2JsonRedisSerializer<Object> serializer = 
            new Jackson2JsonRedisSerializer<>(Object.class);
        template.setValueSerializer(serializer);
        
        // 用 String 序列化 key
        template.setKeySerializer(new StringRedisSerializer());
        template.setHashKeySerializer(new StringRedisSerializer());
        
        template.afterPropertiesSet();
        return template;
    }
}
```

**分布式对象传输（Dubbo/Feign）：**

```java
// Dubbo 传输 User 对象
// 发送端：序列化 User → 字节数组 → 网络传输
// 接收端：字节数组 → 反序列化 → User 对象

// User 必须实现 Serializable
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    
    private Long id;
    private String name;
    private List<String> roles;  // 嵌套对象也必须可序列化
}

// 如果用了自定义类做字段
public class Order implements Serializable {
    private static final long serialVersionUID = 1L;
    
    private User user;  // 嵌套的 User 也必须可序列化！
}
```

**常见坑：**

```java
// ❌ 坑1：Lambda 和内部类没有 serialVersionUID
// 序列化到 Redis 后，反序列化时找不到原始类
Supplier<User> factory = User::new;  // 不能序列化！

// ❌ 坑2：LocalDateTime 等 Java 8 时间类型（某些序列化器需要配置）
// Fastjson 1.x 要加配置才能序列化 LocalDateTime
// Fastjson2 / Jackson 默认支持

// ❌ 坑3：循环引用
public class A implements Serializable {
    public B b;
}
public class B implements Serializable {
    public A a;  // A 里有 B，B 里有 A → StackOverflowError！
}
```

**面试加分话术：**

> "分布式系统里序列化无处不在：Session 共享、Redis 缓存、RPC 调用、消息队列传输。踩过的坑主要是三个：**内部类/Lambda 不能序列化**、**循环引用 StackOverflow**、**Java 8 时间类型不兼容**。我建议团队统一用 JSON 序列化（Fastjson2 或 Jackson），配置好模块支持 Java 8 时间类型、`Optional`、`BigDecimal`，这样大多数场景都能覆盖。"

## 一句话总结（面试收尾用）

> "Java 序列化通过 `Serializable` 接口把对象转成字节流，核心是 `serialVersionUID` 版本校验。生产环境有三个注意事项：transient 不等于安全（只是跳过序列化）、反序列化有安全漏洞（别信外部输入）、原生序列化太慢（用 Hessian/Protobuf/Fastjson2）。分布式系统中 Session 共享和 Redis 缓存都会用到序列化，选型时要兼顾速度、字节大小和跨语言兼容性。"
