## 🔥 扩展追问 1：HandlerAdapter 的作用是什么？为什么要适配？

**核心：统一不同类型的 Handler 调用方式**

```
┌─────────────────────────────────────────────────────────────────┐
│                    HandlerAdapter 适配器模式                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  问题：Handler 类型多种多样，调用方式不同                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  Controller 接口：                                       │   │
│  │      ModelAndView handleRequest(request, response);    │   │
│  │                                                         │   │
│  │  @RequestMapping 方法：                                  │   │
│  │      @ResponseBody User getUser(Long id);              │   │
│  │                                                         │   │
│  │  HttpRequestHandler：                                   │   │
│  │      void handleRequest(HttpServletRequest,            │   │
│  │                           HttpServletResponse);         │   │
│  │                                                         │   │
│  │  Servlet：                                              │   │
│  │      void service(ServletRequest, ServletResponse);    │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  解决方案：HandlerAdapter 统一接口                               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  public interface HandlerAdapter {                      │   │
│  │      boolean supports(Object handler);  // 是否支持     │   │
│  │      ModelAndView handle(HttpServletRequest,          │   │
│  │                          HttpServletResponse,          │   │
│  │                          Object handler);               │   │
│  │  }                                                      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  supports(handler) 职责：                               │   │
│  │  • 告诉 DispatcherServlet "我能处理这个 Handler"         │   │
│  │  • 遍历所有 HandlerAdapter，找到第一个 supports=true   │   │
│  │                                                         │   │
│  │  handle() 职责：                                       │   │
│  │  • 统一调用不同类型的 Handler                           │   │
│  │  • 返回统一的 ModelAndView                              │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

| Handler 类型 | HandlerAdapter | 调用方式 |
|-------------|----------------|---------|
| @RequestMapping | RequestMappingHandlerAdapter | `handler.invoke(req, rsp, handlerMethod)` |
| Controller 接口 | SimpleControllerHandlerAdapter | `handler.handleRequest(req, rsp)` |
| HttpRequestHandler | HttpRequestHandlerAdapter | `handler.handleRequest(req, rsp)` |
| Servlet | SimpleServletHandlerAdapter | `handler.service(req, rsp)` |

> A: HandlerAdapter 的作用是"统一不同 Handler 的调用方式"。Spring 支持多种 Handler（@RequestMapping、Controller 接口、HttpRequestHandler、Servlet），调用方式各不相同，HandlerAdapter 通过适配器模式屏蔽了这些差异。`supports()` 方法判断能否处理这个 Handler，`handle()` 方法统一调用并返回 ModelAndView。DispatcherServlet 不需要知道 Handler 的具体类型，只需要通过 HandlerAdapter 统一操作。

**两种 HandlerMapping 及其优先级**

```
┌─────────────────────────────────────────────────────────────────┐
│                    HandlerMapping 查找流程                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  DispatcherServlet.doDispatch():                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  HandlerExecutionChain handler = handlerMappings        │   │
│  │      .getHandler(request);                             │   │
│  │                                                         │   │

│  │                                                         │   │

│                                                                 │
│  Spring 内置的 HandlerMapping：                                  │
│  ┌─────────────────────────────────────────────────────────┐   │

│  │     • 从所有 @Controller/@RestController 扫描           │   │
│  │     • 构建 URL → HandlerMethod 映射                     │   │
│  │                                                         │   │
│  │  2. BeanNameUrlHandlerMapping                            │   │
│  │     • Bean 名称作为 URL                                 │   │
│  │     • <bean name="/user" class="...UserController"/>   │   │
│  │                                                         │   │
│  │  3. SimpleUrlHandlerMapping                              │   │
│  │     • 手动配置 URL → Handler 映射                        │   │

│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  查找过程：                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  请求: GET /api/user/100                                │   │
│  │                                                         │   │

│                                                                 │

// 自定义参数解析器示例
@Component
public class CurrentUserArgumentResolver implements HandlerMethodArgumentResolver {
    
    @Override
    public boolean supportsParameter(MethodParameter parameter) {
        // 只有标注 @CurrentUser 的 User 类型参数才使用此解析器
        return parameter.hasParameterAnnotation(CurrentUser.class) 
            && parameter.getParameterType() == User.class;
    }
    
    @Override
    public Object resolveArgument(MethodParameter parameter,
                                  ModelAndViewContainer mavContainer,
                                  NativeWebRequest webRequest,
                                  WebDataBinderFactory binderFactory) {
        // 从 SecurityContext 或 Session 中获取当前用户
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        if (auth != null && auth.getPrincipal() instanceof User) {
            return (User) auth.getPrincipal();
        }
        return null;
    }
}

// 注册自定义解析器
@Configuration
public class WebMvcConfig implements WebMvcConfigurer {
    @Autowired
    private CurrentUserArgumentResolver currentUserArgumentResolver;
    
    @Override
    public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
        resolvers.add(currentUserArgumentResolver);
    }
}

// 使用
@RestController
public class UserController {
    
    @GetMapping("/profile")
    public User profile(@CurrentUser User user) {
        return user;  // 自动注入当前登录用户
    }
}
```

> A: HandlerMethodArgumentResolver 负责解析 Controller 方法参数。Spring 内置了几十个解析器，按顺序遍历找到第一个 supports() 返回 true 的来处理。比如 @PathVariable 对应 PathVariableMethodArgumentResolver，@RequestBody 对应 RequestResponseBodyMethodProcessor（内部调用 HttpMessageConverter 做反序列化）。自定义参数解析器需要实现 `supportsParameter()` 判断是否支持，然后 `resolveArgument()` 从请求中获取值注入。

## 🔥 扩展追问 4：拦截器（HandlerInterceptor）和 Filter 的区别？

**三层拦截器对比**

```
┌─────────────────────────────────────────────────────────────────┐
│                    拦截器三层对比                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │           ┌─────────────────────┐                      │   │
│  │           │     Filter          │  Servlet 规范       │   │
│  │           │  (Tomcat/容器层)     │  → 最早执行         │   │
│  │           └──────────┬──────────┘                      │   │
│  │                      │                                   │   │
│  │           ┌──────────▼──────────┐                      │   │
│  │           │  DispatcherServlet  │  Spring MVC 入口    │   │
│  │           └──────────┬──────────┘                      │   │
│  │                      │                                   │   │
│  │           ┌──────────▼──────────┐                      │   │
│  │           │   Interceptor      │  Spring MVC 层       │   │
│  │           │  (HandlerInterceptor)│ → 精确到 Handler   │   │
│  │           └──────────┬──────────┘                      │   │
│  │                      │                                   │   │
│  │           ┌──────────▼──────────┐                      │   │
│  │           │   Controller       │  业务逻辑             │   │
│  │           └─────────────────────┘                      │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  Filter vs Interceptor vs AOP：                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Filter：    Servlet 规范，Spring 外，任何请求都经过    │   │
│  │              适合：字符编码、跨域、安全过滤（Shiro/JWT）│   │
│  │                                                         │   │
│  │  Interceptor：Spring MVC 组件，只拦截 Controller       │   │
│  │              适合：登录检查、权限验证、日志记录        │   │
│  │                                                         │   │
│  │  AOP：      Spring Bean 层面拦截                        │   │
│  │              适合：业务层方法拦截、事务管理            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  Interceptor 三个方法：                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  preHandle(request, response, handler)                  │   │
│  │  • 在 Handler 执行前调用                                │   │
│  │  • 返回 true → 继续执行，返回 false → 拦截请求         │   │
│  │                                                         │   │
│  │  postHandle(request, response, handler, modelAndView)   │   │
│  │  • 在 Handler 执行后、视图渲染前调用                    │   │
│  │  • 可修改 ModelAndView                                 │   │
│  │                                                         │   │
│  │  afterCompletion(request, response, handler, ex)        │   │
│  │  • 在视图渲染完成后、始终执行                          │   │
│  │  • 适合：清理资源、记录日志                            │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```java
// 自定义拦截器
public class LoginInterceptor implements HandlerInterceptor {
    
    @Override
    public boolean preHandle(HttpServletRequest request, 
                             HttpServletResponse response,
                             Object handler) throws Exception {
        // 1. 登录检查
        HttpSession session = request.getSession();
        User user = (User) session.getAttribute("user");
        
        if (user == null) {
            // 未登录，返回 401
            response.setStatus(401);
            response.getWriter().write("{\"code\":401,\"msg\":\"未登录\"}");
            return false;  // 拦截请求
        }
        
        // 2. 权限检查（如果是HandlerMethod类型）
        if (handler instanceof HandlerMethod) {
            HandlerMethod hm = (HandlerMethod) handler;
            RequiredRole role = hm.getMethodAnnotation(RequiredRole.class);
            if (role != null && !user.hasRole(role.value())) {
                response.setStatus(403);
                response.getWriter().write("{\"code\":403,\"msg\":\"权限不足\"}");
                return false;
            }
        }
        
        return true;  // 放行
    }
    
    @Override
    public void postHandle(HttpServletRequest request,
                           HttpServletResponse response,
                           Object handler,
                           ModelAndView modelAndView) throws Exception {
        // 可修改视图数据
        if (modelAndView != null) {
            modelAndView.addObject("currentUser", 
                request.getSession().getAttribute("user"));
        }
    }
    
    @Override
    public void afterCompletion(HttpServletRequest request,
                                HttpServletResponse response,
                                Object handler,
                                Exception ex) throws Exception {
        // 清理资源
        // 记录请求日志
        log.info("请求处理完成: {} - {}", request.getRequestURI(), ex);
    }
}

// 注册拦截器
@Configuration
public class WebMvcConfig implements WebMvcConfigurer {
    
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new LoginInterceptor())
                .addPathPatterns("/**")                    // 拦截所有路径
                .excludePathPatterns(                       // 排除这些路径
                    "/login", 
                    "/static/**",
                    "/error"
                );
    }
}
```

> A: Filter 是 Servlet 规范，最早执行，任何请求都经过，适合字符编码、跨域、Spring Security 的 URL 过滤；Interceptor 是 Spring MVC 组件，只拦截 Controller Handler，有 preHandle/postHandle/afterCompletion 三个切点，适合登录检查、权限验证；AOP 在业务层拦截，适合方法级别的切面。Filter 和 Interceptor 的核心区别是执行顺序：Filter 在 DispatcherServlet 之前，Interceptor 在之后。

```java
// 全局异常处理器
@ControllerAdvice
public class GlobalExceptionHandler {
    
    @ExceptionHandler(BusinessException.class)
    @ResponseBody
    public Result handleBusiness(BusinessException e) {
        return Result.error(e.getCode(), e.getMessage());
    }
    
    @ExceptionHandler(NullPointerException.class)
    @ResponseBody
    public Result handleNPE(NullPointerException e) {
        log.error("空指针异常", e);
        return Result.error(500, "系统异常");
    }
    
    @ExceptionHandler(Exception.class)
    public ModelAndView handleException(Exception e) {
        // 返回错误页面
        ModelAndView mav = new ModelAndView("error");
        mav.addObject("message", e.getMessage());
        return mav;
    }
}

// REST 风格统一响应
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    @ExceptionHandler(Exception.class)
    public Result handleException(Exception e) {
        log.error("系统异常", e);
        return Result.error(500, "系统繁忙，请稍后重试");
    }
}
```

> A: Spring MVC 异常处理流程是：Controller 抛出异常后，DispatcherServlet 调用 HandlerExceptionResolver 处理。@ExceptionHandler 注解的方法会匹配异常类型并处理，可以返回 JSON（@ResponseBody）或视图（ModelAndView）。@ControllerAdvice 定义的 @ExceptionHandler 对所有 Controller 生效，实现全局异常处理。实际项目建议统一用 @ControllerAdvice + @ResponseBody 返回 JSON 格式的错误信息。

## 一句话总结（面试收尾用）

> **Spring MVC 请求流程是：请求 → DispatcherServlet → HandlerMapping 找到 Handler → HandlerAdapter 适配调用 → Interceptors.preHandle → 参数解析 → 执行目标方法 → Interceptors.postHandle → ViewResolver 解析视图 → View.render → Interceptors.afterCompletion → 返回响应。HandlerAdapter 的核心作用是适配不同类型的 Handler（@RequestMapping、Controller、HttpRequestHandler），通过 supports() 判断是否支持，handle() 统一调用并返回 ModelAndView。**
