### 基础流程

```
ReentrantReadWriteLock：读写锁分离
        ↓
同一资源：
  ├─ 读操作：加读锁（共享，多个线程同时读）
  └─ 写操作：加写锁（排他，独占写）
        ↓
锁的互斥规则：
  ├─ 读 ←→ 读：不互斥（可以同时读）
  ├─ 读 ←→ 写：互斥（读等写、写等读）
  └─ 写 ←→ 写：互斥（排队写）
```

**读锁和写锁的关系：**

| 对比 | 读锁（共享锁） | 写锁（排他锁） |
|---|---|---|
| **获取条件** | 无其他线程持有写锁 | 无其他线程持有写锁 |
| **并发读** | ✅ 多个线程同时持有读锁 | ❌ 不行 |
| **与写锁互斥** | ✅ 持有读锁时不能写 | ✅ 持有写锁时可以读 |
| **重入** | ✅ 可重入（多次获取） | ✅ 可重入（多次获取） |
| **锁降级** | ❌ 不能（读→写 需要重新获取） | ✅ 可以（写→读 自动降级） |

**基本使用示例：**

```java
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
ReadLock readLock = rwLock.readLock();
WriteLock writeLock = rwLock.writeLock();

// 读操作：加读锁
public void read() {
    readLock.lock();
    try {
        // 读数据
        return data;
    } finally {
        readLock.unlock();
    }
}

// 写操作：加写锁
public void write(Object value) {
    writeLock.lock();
    try {
        // 写数据
        this.data = value;
    } finally {
        writeLock.unlock();
    }
}
```

## 🔥 扩展追问 1：读锁为什么可以多个线程同时持有？

**读锁的共享特性：**

```
读锁的本质：
        ↓
当没有线程持有写锁时
多个读线程可以同时进入临界区
        ↓
因为：读操作不修改数据
多个线程同时读，不会产生竞态条件
```

**读锁的获取逻辑：**

```java
// AQS 内部状态 state 拆分为两个部分：
// 高 16 位：读锁计数
// 低 16 位：写锁计数

// 读锁获取：
if (写锁未被持有) {
    读锁计数 +1  // CAS 修改高 16 位
    允许进入  // 多个线程可以同时进入
}

// 读锁释放：
读锁计数 -1  // CAS 修改高 16 位
// 直到读锁计数归零，才真正释放

// 写锁获取：
if (state == 0) {  // 没有任何锁时才能获取
    CAS 写锁计数 +1
}
// 如果有读锁或写锁持有 → 等待
```

**为什么读读不互斥：**

```
线程 A 获取读锁 → state 高16位 = 1
线程 B 获取读锁 → state 高16位 = 2
线程 C 获取读锁 → state 高16位 = 3

三个线程同时在临界区里读数据
        ↓
没有数据修改 → 不会有竞态条件
        ↓
✅ 读读可以并发
```

**什么时候不适合用读写锁：**

```java
// ❌ 读多写少，但读很慢：读锁持有时间长 → 写线程饿着
public class BadReadWriteLock {
    private final ReentrantReadWriteLock lock = new ReentrantReadWriteLock();
    private String data;
    
    // 读：持有锁时间长
    public String read() {
        lock.readLock().lock();
        try {
            Thread.sleep(5000);  // 模拟耗时读操作
            return data;
        } finally {
            lock.readLock().unlock();
        }
    }
    
    // 写：需要等所有读完成
    public void write(String value) {
        lock.writeLock().lock();
        try {
            this.data = value;
        } finally {
            lock.writeLock().unlock();
        }
    }
}

// 问题：100 个线程在读，1 个线程要写
// → 写线程要等 100 个读线程全部释放
// → 读线程持续来，写线程可能永远等不到
```

**StampedLock 作为替代：**

```java
// JDK 1.8 引入的 StampedLock
// 乐观读：完全不阻塞，允许写线程插队
StampedLock sl = new StampedLock();

public String read() {
    // 1. 先乐观读（不阻塞）
    long stamp = sl.tryOptimisticRead();
    String data = this.data;
    
    // 2. 检查乐观读之后是否有写操作
    if (!sl.validate(stamp)) {
        // 有写操作，升级为悲观读
        stamp = sl.readLock();
        try {
            data = this.data;
        } finally {
            sl.unlockRead(stamp);
        }
    }
    return data;
}

public void write(String value) {
    long stamp = sl.writeLock();
    try {
        this.data = value;
    } finally {
        sl.unlockWrite(stamp);
    }
}
```

**面试加分话术：**

> "读锁之所以能多个线程同时持有，是因为读操作**不会修改数据**，不会产生竞态条件。读写锁用 AQS 的 state 字段，高 16 位存读锁计数、低 16 位存写锁计数，读锁获取时 CAS 加读锁计数，写锁获取时必须等读锁计数归零。但要注意：读写锁适合**读多写少、写锁持有时间短**的场景。如果读操作本身很慢或者需要长时间持有锁，读写锁反而会导致写线程饥饿——这时用 StampedLock 的乐观读更好，写线程可以随时插队。"

## 🔥 扩展追问 2：锁降级（Write-Down）的原理？写锁如何降级为读锁？

**锁降级的定义：**

```
锁降级 = 持有写锁 → 降级为读锁
        ↓
为什么需要降级？
        ↓
持有写锁时做了写操作
但后续需要读取数据
        ↓
方案1：先释放写锁，再加读锁（中间有缝隙）
        ↓ 可能其他线程在这期间写了数据 → 不可见

方案2：直接降级为读锁（无缝隙）
        ↓ 写锁→读锁，内部原子完成，不会有其他线程写入
```

**锁降级的代码：**

```java
// ❌ 错误做法：释放写锁后再加读锁（有缝隙）
public Object wrongUpgrade() {
    writeLock.lock();
    try {
        // 写入数据
        data = computeData();
    } finally {
        writeLock.unlock();  // ⚠️ 释放了，其他线程可以进来了
    }
    
    // 这里有缝隙！其他线程可能已经改了 data
    readLock.lock();
    try {
        return data;
    } finally {
        readLock.unlock();
    }
}

// ✅ 正确做法：锁降级
public Object correctDowngrade() {
    writeLock.lock();
    try {
        // 写入数据
        data = computeData();
        
        // 锁降级：先获取读锁，再释放写锁
        readLock.lock();  // 获取读锁
        try {
            writeLock.unlock();  // 释放写锁（现在只剩读锁了）
            return data;
        } catch (Exception e) {
            writeLock.unlock();
            throw e;
        }
    } finally {
        // 如果上面的 try 块里已经降级，这里只剩下读锁
        // 需要再释放读锁
        if (writeLock.isHeldByCurrentThread()) {
            // 还没降级
        } else {
            readLock.unlock();
        }
    }
}
```

**更简洁的降级写法：**

```java
// ✅ 标准锁降级模式
public String safeDowngrade() {
    writeLock.lock();
    try {
        data = computeData();  // 写操作
        
        // 降级为读锁：先持有读锁，再释放写锁
        readLock.lock();
        try {
            writeLock.unlock();  // 释放写锁
            return data;         // 此时只有读锁
        } catch (Exception e) {
            writeLock.unlock();
            throw e;
        }
    } finally {
        // 兜底：确保读锁被释放
        if (readLock.tryLock()) {
            readLock.unlock();
        }
    }
}
```

**降级的内部原理：**

```
持有写锁状态：
        ↓
Thread A: writeLock = 1, readLock = 0
        ↓
获取读锁（降级）：
readLock.tryAcquireShared(1)
        ↓
检查 state：
  state = 0x00010000 (写锁=1, 读锁=0)
  tryAcquireShared 返回 -1（获取失败？）→ 不！
        ↓
ReentrantReadWriteLock 特殊处理：
  如果当前线程持有写锁 → 允许获取读锁
  state = 0x00010001 (写锁=1, 读锁=1)
        ↓
释放写锁：
writeLock.tryRelease(0)
        ↓
state = 0x00000001 (写锁=0, 读锁=1)
        ↓
✅ 成功降级为读锁
```

**ReentrantReadWriteLock 源码关键逻辑：**

```java
// tryAcquireShared（读锁获取）
protected final int tryAcquireShared(int unused) {
    Thread current = Thread.currentThread();
    int c = getState();
    
    // 关键：如果有其他线程持有写锁，返回 -1（获取失败）
    if (exclusiveCount(c) != 0) {
        if (getExclusiveOwnerThread() != current)
            return -1;  // 有写锁但不是自己持有 → 失败
        // 如果是自己持有写锁 → 允许获取读锁（降级场景）
    }
    
    // 读锁计数 +1
    int r = sharedCount(c) + 1;
    if (r > count)
        return -1;
    if (compareAndSetState(c, c + 1)) {
        firstReader = current;
        firstReaderHoldCount = 1;
        return 1;
    }
    return -1;
}
```

**面试加分话术：**

> "**写锁可以降级为读锁**，这是 ReentrantReadWriteLock 的一个重要特性。降级的正确写法是：**先获取读锁，再释放写锁**，两步都不能省。很多人容易犯的错误是先 unlock 再 lock——中间有个时间窗口，其他线程可能进来修改数据。JDK 源码里 tryAcquireShared 专门处理了这个场景：如果当前线程已经持有写锁，允许它再获取读锁（其他线程的写锁则不允许）。降级在**缓存更新**场景很常见：先写缓存（持写锁），再读缓存（降级读锁）。"

## 🔥 扩展追问 3：读锁可以升级为写锁吗？为什么？

**锁升级（Read-to-Write）：**

```
答案：❌ 不支持
        ↓
为什么？
        ↓
如果读锁可以升级为写锁：
        ↓
线程 A、B 同时获取读锁
        ↓
线程 A 想升级为写锁
        ↓
线程 B 还在持有读锁
        ↓
线程 A 等 B 释放读锁
        ↓
线程 B 等线程 A 释放读锁
        ↓
死锁！
```

**代码验证：**

```java
// ❌ ReentrantReadWriteLock 不支持锁升级
public void cannotUpgrade() {
    readLock.lock();
    try {
        // 在这里想升级为写锁？
        // 没有办法直接升级！
        
        // 只能先释放读锁，再获取写锁
        readLock.unlock();  // 释放读锁
        writeLock.lock();   // 获取写锁
        try {
            // 业务逻辑
        } finally {
            writeLock.unlock();
        }
    } finally {
        // 如果上面已经释放读锁了，这里就不需要了
    }
}

// ⚠️ 这个做法有缝隙：释放读锁后，获取写锁前
// 其他线程可能已经修改了数据
```

**死锁场景演示：**

```
线程 A：持有读锁
线程 B：持有读锁
        ↓
线程 A 想升级为写锁：
  → 等待线程 B 释放读锁
        ↓
线程 B 还没决定要做什么，继续持有读锁
        ↓
线程 A 永远等不到 → 死锁？

实际上不会死锁，因为 ReentrantReadWriteLock 直接不支持这个操作
→ tryAcquireShared 返回 -1（失败）
→ 直接返回，不会阻塞等待
```

**为什么需要锁升级？**

```
场景：先读后写，读完再决定要不要写
        ↓
读锁：获取读锁
读数据：根据数据决定是否修改
写锁：修改数据
        ↓
问题：如果释放读锁后、获取写锁前，有其他线程修改了数据
→ 自己读到的数据已经过时了
→ 修改时基于旧数据 → 错误！
```

**解决锁升级需求的其他方案：**

```java
// 方案1：直接用写锁（不用读锁）
public void method() {
    writeLock.lock();  // 直接用写锁，不需要升级
    try {
        // 读数据
        // 决定是否写
    } finally {
        writeLock.unlock();
    }
}
// 问题：没有并发读的性能优势

// 方案2：StampedLock（支持乐观读 + 升级）
StampedLock sl = new StampedLock();

public void upgradeExample() {
    // 1. 乐观读
    long stamp = sl.tryOptimisticRead();
    data = this.data;
    
    // 2. 验证，发现需要写
    if (needsWrite(data)) {
        // 3. 升级为写锁（注意：stamp 可能已失效）
        long writeStamp = sl.tryConvertToWriteLock(stamp);
        if (writeStamp != 0) {
            stamp = writeStamp;  // 升级成功
            this.data = computeNewData(data);
        } else {
            // 升级失败（有其他读锁），重新走完整流程
            sl.unlockRead(stamp);
            stamp = sl.writeLock();
            try {
                this.data = computeNewData(data);
            } finally {
                sl.unlockWrite(stamp);
            }
        }
    } else {
        sl.unlockRead(stamp);  // 释放乐观读
    }
}
```

**StampedLock 的锁转换：**

```
StampedLock 的锁转换 API：
        ↓
tryConvertToWriteLock(stamp)
  └─ 尝试将乐观读/读锁升级为写锁
  └─ 如果当前只有自己持有读锁 → 直接升级
  └─ 否则返回 0（升级失败）

tryConvertToReadLock(stamp)
  └─ 尝试将写锁降级为读锁
  └─ 几乎总会成功
```

**面试加分话术：**

> "**读锁不能升级为写锁**，这是读写锁的设计选择——因为如果允许，同一时间可能有两个或更多读锁持有者，都想升级为写锁，就会互相等待对方释放读锁，形成死锁。JDK 直接不提供这个 API，想要类似功能有两种方案：**方案一**是直接用写锁（但失去读读并发优势），**方案二**是用 **StampedLock 的 tryConvertToWriteLock()**，它会尝试乐观读升级为写锁，失败了就返回 0，让代码走降级流程。实际业务中，如果需要先读后写的场景，我建议直接用写锁，或者用 StampedLock 的乐观读+升级模式。"

## 🔥 扩展追问 4：ReentrantReadWriteLock 的公平 vs 非公平模式？

**公平模式 vs 非公平模式：**

```java
// 非公平模式（默认）：吞吐量高，可能插队
ReentrantReadWriteLock unfair = new ReentrantReadWriteLock();

// 公平模式：严格 FIFO，不会插队
ReentrantReadWriteLock fair = new ReentrantReadWriteLock(true);

// 公平模式原理：
// 所有等待线程按顺序排队（FIFO）
// 新来的线程如果队列非空，必须排队
// 不会出现写线程插队的情况
```

**公平 vs 非公平的差异：**

```
非公平模式（默认）：
        ↓
写线程释放锁 → 唤醒一个等待的读线程
        ↓
如果有新读线程来（还没进队列）
        ↓
可能直接抢到锁 → 读线程插队
        ↓
写线程继续等 → 可能饥饿

公平模式：
        ↓
写线程释放锁 → 按队列顺序唤醒
        ↓
新来的读线程必须排在队列后面
        ↓
先排队的读线程先执行
        ↓
不会出现插队，没有饥饿
```

**公平模式下的写饥饿问题：**

```java
// 公平模式：读线程持续来，写线程可能还是饿
// 队列：R1, R2, R3, R4, R5...
// 写线程 W 等在队列后面
// 每来一个新读线程 R，都排在 W 前面
// W 可能永远等不到 → 写线程饥饿

// 非公平模式下更严重：
// 释放锁 → 新读线程直接抢 → W 一直等
```

**非公平模式的吞吐优势：**

```java
// 非公平模式为什么吞吐量大：
// 因为允许读线程插队

// 时间线：
// 时刻0：写锁释放
// 时刻1：读线程 R1 来 → 立刻拿到读锁
// 时刻2：读线程 R2 来 → 也拿到读锁（读写不互斥）
// 时刻3：读线程 R3 来 → 继续拿
// 时刻4：写线程 W 来 → 等所有读完成
// 时刻5-∞：更多读线程来 → 继续拿

// 如果是公平模式：
// W 在时刻4 排在 R3 后面
// R3 之后的读线程都在 W 后面
// → W 之后的读线程也要等 W 完成 → 读线程也慢

// 非公平让读读可以插队，吞吐量大
```

**选择建议：**

```java
// 高并发读场景 → 非公平（默认）
// 优势：读线程可以插队，吞吐量大

// 写优先场景 → 公平模式
// 优势：写线程不会饿

// 但公平模式本身也不能完全解决写饥饿：
// 如果读线程持续来，新读线程插队在写线程后面
// 但写线程前面的读线程可能持续增多

// 更强的写优先：StampedLock
StampedLock sl = new StampedLock();
sl.writeLock();  // 写锁独占，强制等待所有读完成
```

**面试加分话术：**

> "ReentrantReadWriteLock 默认是非公平的，吞吐量高但可能导致**写线程饥饿**。公平模式用 CLH 队列严格 FIFO，但写饥饿问题仍然存在——因为读线程可以不断来，新的读线程插在写线程前面，写线程还是等很久。更强的写优先方案是 **StampedLock**：写锁持有时强制所有读线程让路（通过 readLock 失败重试）。选型建议：**读多写少、无饥饿风险 → 默认非公平**；**需要写优先 → 公平模式或 StampedLock**。另外注意：公平模式吞吐量比非公平低 3~5 倍，因为线程切换开销大。"

## 🔥 扩展追问 5：StampedLock 和 ReentrantReadWriteLock 的区别？

**两者的核心区别：**

| 特性 | ReentrantReadWriteLock | StampedLock |
|---|---|---|
| **读锁模式** | 悲观读（阻塞写） | 乐观读（不阻塞） |
| **锁转换** | 写锁可降级读锁 | 写锁可降级读锁，读锁可升级写锁 |
| **重入** | ✅ 完全可重入 | ✅ 完全可重入 |
| **公平性** | 可选公平/非公平 | 非公平（不可选） |
| **Condition** | ✅ 支持写锁 Condition | ❌ 不支持 |
| **性能** | 较好 | 更好（乐观读开销极低） |

**StampedLock 三种锁模式：**

```java
StampedLock sl = new StampedLock();

// 1. 写锁（排他）
long stamp = sl.writeLock();
try {
    // 写操作
} finally {
    sl.unlockWrite(stamp);
}

// 2. 悲观读锁（和 ReadWriteLock 类似）
long stamp = sl.readLock();
try {
    // 读操作
} finally {
    sl.unlockRead(stamp);
}

// 3. 乐观读（核心优势）
long stamp = sl.tryOptimisticRead();  // 不阻塞，立刻返回戳
String data = this.data;               // 读数据（可能读到不一致）
if (!sl.validate(stamp)) {            // 验证：这段时间是否有写？
    // 有写！重新读
    stamp = sl.readLock();
    try {
        data = this.data;
    } finally {
        sl.unlockRead(stamp);
    }
}
```

**StampedLock 的乐观读原理：**

```
乐观读的执行流程：
        ↓
① tryOptimisticRead()：获取当前 stamp（类似版本号）
        ↓
② 读取数据（不阻塞）
        ↓
③ validate(stamp)：检查 stamp 是否过期
        ↓
├─ stamp 未过期 → 数据一致 → 直接使用
└─ stamp 过期 → 有写发生 → 升级为悲观读
```

**StampedLock 的坑：**

```java
// ⚠️ 坑1：不支持 Condition
StampedLock sl = new StampedLock();
sl.newCondition();  // ❌ UnsupportedOperationException！

// ⚠️ 坑2：乐观读可能读到过期数据
// validate() 之后到读取之间可能又有写发生 → ABA 问题

// ⚠️ 坑3：锁升级失败需要手动重试
long stamp = sl.writeLock();
try {
    // 写操作
} finally {
    sl.unlockWrite(stamp);
}

// ⚠️ 坑4：不可重入
// StampedLock 不支持重入，多次 writeLock() 会死锁
```

**实际选择建议：**

```java
// ✅ 用 ReentrantReadWriteLock 的场景：
// - 需要 Condition
// - 读锁持有时间长
// - 需要公平模式

// ✅ 用 StampedLock 的场景：
// - 读多写少，锁持有时间短
// - 需要乐观读优化
// - 需要读锁升级为写锁

// ✅ 用 synchronized 的场景：
// - 简单的互斥需求
// - 代码简洁优先
```

**StampedLock 完整示例：**

```java
public class StampedLockCache {
    private final StampedLock sl = new StampedLock();
    private Object data;
    
    // 乐观读（性能最佳）
    public Object get() {
        long stamp = sl.tryOptimisticRead();
        Object d = this.data;
        
        if (!sl.validate(stamp)) {  // 验证戳
            stamp = sl.readLock();
            try {
                d = this.data;
            } finally {
                sl.unlockRead(stamp);
            }
        }
        return d;
    }
    
    // 悲观读（需要读取后做判断）
    public Object getAndCheck() {
        long stamp = sl.readLock();
        try {
            if (this.data == null) {
                return null;
            }
            // 读完发现需要写 → 升级
            long ws = sl.tryConvertToWriteLock(stamp);
            if (ws != 0) {
                stamp = ws;  // 升级成功
                this.data = loadData();  // 写
            }
            return this.data;
        } finally {
            sl.unlock(stamp);
        }
    }
    
    // 写
    public void write(Object value) {
        long stamp = sl.writeLock();
        try {
            this.data = value;
        } finally {
            sl.unlockWrite(stamp);
        }
    }
}
```

**面试加分话术：**

> "StampedLock 是 ReentrantReadWriteLock 的增强版，核心区别是**多了一个乐观读模式**：读时不加锁，直接读，验证戳有效就说明没写冲突，比悲观读快 5~10 倍。另外 StampedLock 支持**读锁升级为写锁**（tryConvertToWriteLock），而 ReentrantReadWriteLock 不支持。但 StampedLock 有两个限制：**不支持 Condition**（不能做 wait/notify）和**不支持重入**（同一个线程重复加写锁会死锁）。实际选型：高性能缓存用 StampedLock，复杂同步场景用 ReentrantReadWriteLock。"

// 缓存场景：大量读，偶尔写
public class Cache<K, V> {
    private final ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Map<K, V> cache = new ConcurrentHashMap<>();
    
    // 读缓存：读锁，多线程并发
    public V get(K key) {
        rwLock.readLock().lock();
        try {
            return cache.get(key);
        } finally {
            rwLock.readLock().unlock();
        }
    }
    
    // 写缓存：写锁，独占
    public void put(K key, V value) {
        rwLock.writeLock().lock();
        try {
            cache.put(key, value);
        } finally {
            rwLock.writeLock().unlock();
        }
    }
    
    // 批量写：一次性加写锁，避免多次加解锁
    public void putAll(Map<K, V> entries) {
        rwLock.writeLock().lock();
        try {
            cache.putAll(entries);  // 一次putAll
        } finally {
            rwLock.writeLock().unlock();
        }
    }
}

```java
// 库存：读写都要精准
public class StockManager {
    private final ReentrantReadWriteLock lock = new ReentrantReadWriteLock();
    private final Map<Long, Integer> stock = new HashMap<>();
    
    // 查询库存：多线程读
    public Integer getStock(Long productId) {
        lock.readLock().lock();
        try {
            return stock.getOrDefault(productId, 0);
        } finally {
            lock.readLock().unlock();
        }
    }
    
    // 扣减库存：需要原子性，用写锁
    public boolean deduct(Long productId, int quantity) {
        lock.writeLock().lock();
        try {
            Integer current = stock.get(productId);
            if (current == null || current < quantity) {
                return false;  // 库存不足
            }
            stock.put(productId, current - quantity);
            return true;
        } finally {
            lock.writeLock().unlock();
        }
    }
    
    // 批量扣减：复杂原子操作
    public boolean deductMultiple(Map<Long, Integer> deductions) {
        lock.writeLock().lock();
        try {
            for (Map.Entry<Long, Integer> entry : deductions.entrySet()) {
                Integer current = stock.get(entry.getKey());
                if (current == null || current < entry.getValue()) {
                    return false;
                }
            }
            // 全部检查通过，再扣减
            for (Map.Entry<Long, Integer> entry : deductions.entrySet()) {
                stock.compute(entry.getKey(), (k, v) -> v - entry.getValue());
            }
            return true;
        } finally {
            lock.writeLock().unlock();
        }
    }
}
```

**并发性能对比：**

```
场景对比：

synchronized(this)：
  ├─ 读操作 → 串行
  └─ 写操作 → 串行
  └─ 性能：最差

ReentrantLock：
  ├─ 读操作 → 串行
  └─ 写操作 → 串行
  └─ 性能：比 synchronized 好

ReentrantReadWriteLock：
  ├─ 读操作 → 并发（多个线程同时读）
  └─ 写操作 → 串行
  └─ 性能：读多写少时最优

ConcurrentHashMap：
  ├─ 读操作 → 无锁（volatile/CAS）
  └─ 写操作 → 桶级锁并发
  └─ 性能：Map 场景最佳
```

**面试加分话术：**

> "读写锁在**读多写少**的业务里能极大提升并发性能。典型场景有：**缓存**（大量读、偶尔写）、**配置中心**（读远多于写，写完需要立刻生效）、**商品库存**（查询多、扣减少）。实现时注意：批量写要一次性加写锁，不要循环里加解锁；如果读锁持有时间长，StampedLock 的乐观读更好；另外**不能用读写锁实现分布式锁**——它是 JVM 内的锁，多 JVM 实例之间不生效，分布式锁要用 Redis 或 Zookeeper。"

## 🔥 扩展追问 7：ReentrantReadWriteLock 的实现原理？AQS 如何拆分读锁和写锁？

**AQS state 的拆分设计：**

```
AQS 的 state 是 int（32位）
        ↓
ReentrantReadWriteLock 拆分使用：
        ↓
┌──────────────────────────────────────────────┐
│ 高 16 位：读锁计数（共享锁计数）              │
│ 低 16 位：写锁计数（排他锁计数）              │
└──────────────────────────────────────────────┘

获取锁状态：
  exclusiveCount(c) = c & 0x0000FFFF  → 写锁计数
  sharedCount(c) = c >>> 16           → 读锁计数
```

**写锁的获取（排他锁）：**

```java
// tryAcquire（写锁获取）
protected final boolean tryAcquire(int acquires) {
    Thread current = Thread.currentThread();
    int c = getState();
    int w = exclusiveCount(c);  // 获取写锁计数
    
    if (c != 0) {
        // 已经有锁了
        if (w == 0 || current != getExclusiveOwnerThread()) {
            // 有读锁持有 OR 有其他线程持有写锁 → 获取失败
            return false;
        }
        // 是同一个线程持有写锁 → 可重入
    }
    
    // CAS 增加写锁计数
    if (compareAndSetState(c, c + acquires)) {
        setExclusiveOwnerThread(current);
        return true;
    }
    return false;
}
```

**读锁的获取（共享锁）：**

```java
// tryAcquireShared（读锁获取）
protected final int tryAcquireShared(int unused) {
    Thread current = Thread.currentThread();
    int c = getState();
    int w = exclusiveCount(c);
    
    // 关键判断：有写锁，且不是自己持有 → 获取失败
    if (w != 0 && getExclusiveOwnerThread() != current) {
        return -1;  // 获取失败
    }
    
    // 有自己持有的写锁 → 允许获取读锁（降级场景）
    
    int r = sharedCount(c) + 1;  // 读锁计数 +1
    if (r > count) {
        return -1;  // 读锁超过上限
    }
    
    // CAS 增加读锁计数
    if (compareAndSetState(c, c + SHARED_UNIT)) {
        // 记录当前读线程的信息
        return 1;  // 获取成功
    }
    return -1;  // CAS 失败，重试
}

// SHARED_UNIT = 1 << 16 = 65536
// 每次加读锁，state + 65536（不影响低16位）
```

**CLH 队列在读写锁中的特殊处理：**

```
读写锁的等待队列：
        ↓
每个节点有 node.waitStatus
        ↓
读锁获取失败 → 创建节点，加入队列尾部
        ↓
设置节点为 SHARED（共享模式）
        ↓
前驱节点的 nextWaiter = 当前节点
        ↓
自旋等待前驱节点释放
        ↓
写锁释放 → 唤醒队列中的下一个写锁节点
        ↓
如果是读锁节点 → 唤醒队列中所有连续的读锁节点
```

**读锁和写锁的唤醒链：**

```java
// 写锁释放时，唤醒策略：
// 1. 如果队列中下一个是写锁节点 → 只唤醒那个写锁
// 2. 如果队列中下一个是读锁节点 → 唤醒所有连续的读锁节点

// 原因：读锁是共享的，一次唤醒可以服务多个读线程
```

**HoldCounter（读锁重入计数）：**

```java
// 每个读锁持有线程有一个 HoldCounter
// 用于记录当前线程持有读锁的次数（支持重入）

// ThreadLocal 存储当前线程的 HoldCounter
private transient ThreadLocalHoldCounter readHolds;

// HoldCounter 结构：
static final class HoldCounter {
    int count = 0;
    // 记录持有读锁的线程 ID（不是线程对象，防止内存泄漏）
    long tid = Thread.currentThread().getId();
}

// 第一个获取读锁的线程：firstReader
// 第一个读线程的重入计数：firstReaderHoldCount
// 其他读线程：readHolds ThreadLocal
```

**面试加分话术：**

> "ReentrantReadWriteLock 底层靠 AQS 实现，state 的高 16 位存读锁计数、低 16 位存写锁计数。写锁获取和 ReentrantLock 差不多，但多了一个判断：如果是读锁持有状态就直接失败。读锁获取比较特殊，它会检查是否有其他线程持有写锁（如果有就失败），但**如果当前线程自己持有写锁，则允许获取读锁**（这是支持降级的原因）。读锁的重入计数用 HoldCounter 记录，第一个读线程用 firstReader，后续读线程用 ThreadLocal 分开存。另外，写锁释放时会特殊处理：如果是连续的读锁节点，一次性全部唤醒（因为读锁是共享的）。"

## 🔥 扩展追问 8：常见错误用法和避坑指南？

**坑一：读写不加锁或锁错对象**

public class BadExample {
    private final ReentrantReadWriteLock lock = new ReentrantReadWriteLock();
    private Object data;
    
    public void wrongWrite() {
        lock.writeLock().lock();
        // ... 做了一堆操作 ...

**坑二：先释放读锁再获取写锁（无缝隙保证）**

public Object badUpgrade() {
    lock.readLock().lock();
    try {
        Object data = this.data;
        // 基于 data 做判断...
        lock.readLock().unlock();  // ⚠️ 释放读锁
        
        lock.writeLock().lock();   // ⚠️ 有缝隙！

            // 此时 data 可能已经过期了
            if (needsUpdate(data)) {
                this.data = computeUpdate(data);

        // 这里也需要处理
    }
}

// ✅ 正确：直接用写锁

    lock.writeLock().lock();
    try {
        // 一次性读写
        if (needsUpdate(this.data)) {
            this.data = computeUpdate(this.data);

        return this.data;
    } finally {

    }
}

**避坑总结表：**

```
┌──────────────────────────────────────────────────────────────┐
│  读写锁避坑 Checklist                                         │
├──────────────────────────────────────────────────────────────┤
│ 1. 读写锁要配套：读用 readLock，写用 writeLock               │
│ 2. 复合操作（check-then-act）必须用写锁                      │
│ 3. 锁降级：先获取读锁，再释放写锁                             │
│ 4. 读锁内不要调用外部方法（可能有嵌套锁）                    │
│ 5. finally 释放锁：重入要匹配 unlock 次数                     │
│ 6. 锁粒度要小：锁外做耗时操作，锁内只做必要操作              │
│ 7. 读锁持有时间长 → 考虑 StampedLock 乐观读                  │
│ 8. 需要 Condition → 必须用 ReentrantReadWriteLock（StampedLock不支持）│
└──────────────────────────────────────────────────────────────┘
```

**面试加分话术：**

> "读写锁的坑主要在三个方面：**复合操作不加锁**（check-then-act 看似读实际要写）、**读锁里调外部方法**（可能嵌套死锁）、**锁粒度太大**（把整个方法都锁住，吞吐量差）。最常见的问题是 `stock--` 这种复合操作——读和写都在里面，但只加了读锁，两个线程同时读到一个旧值都通过了检查，然后都去扣减，超卖。解决方法是**读和写都要根据业务语义决定用读锁还是写锁**，不要以为不写数据就能用读锁。实际项目里，如果缓存场景锁粒度控制不好，并发性能可能还不如直接用 `synchronized`——所以要配合压测看效果。"

## 一句话总结（面试收尾用）

> "ReentrantReadWriteLock 通过拆分读锁（共享）和写锁（排他），实现**读读并发、读写互斥、写写互斥**。读锁之所以能多线程持有，是因为读不修改数据不会产生竞态；写锁可以降级为读锁（先获取读锁再释放写锁），但读锁不能升级为写锁（会死锁）。适合**读多写少、写锁持有时间短**的场景；读锁持有时间长或需要乐观读时用 **StampedLock**；需要 Condition 时只能选 **ReentrantReadWriteLock**。使用时要避免：复合操作不加锁、读锁内调外部方法、锁粒度太大。"
