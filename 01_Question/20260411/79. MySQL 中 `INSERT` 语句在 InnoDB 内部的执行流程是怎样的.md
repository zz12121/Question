### 基础流程

```
┌─────────────────────────────────────────────────────────────────┐
│                  MySQL INSERT 完整流程                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  客户端                                                         │
│     │                                                           │
│     │  1. 发送 SQL                                              │
│     ▼                                                           │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                     MySQL Server 层                        │  │
│  │                                                           │  │
│  │  2. 解析器（Parser）                                       │  │
│  │     SQL ──▶ 解析树                                         │  │
│  │                                                           │  │
│  │  3. 优化器（Optimizer）                                    │  │
│  │     解析树 ──▶ 执行计划                                    │  │
│  │                                                           │  │
│  │  4. 执行器（Executor）                                     │  │

│     │  5. 调用 InnoDB 接口                                     │
│     ▼                                                           │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                    InnoDB 存储引擎                        │  │
│  │                                                           │  │
│  │  6. 事务开启（autocommit=0 时）                            │  │
│  │     │                                                      │  │
│  │     │ 7. 写入 undo log（MVCC 快照）                        │  │
│  │     │                                                      │  │
│  │     │ 8. 定位数据页（Buffer Pool）                          │  │
│  │     │                                                      │  │
│  │     │ 9. 修改数据页（标记为脏页）                          │  │
│  │     │                                                      │  │
│  │     │ 10. 写入 redo log（prepare）                        │  │
│  │     │                                                      │  │
│  │     │ 11. 事务提交                                          │  │
│  │     │                                                      │  │
│  │     │ 12. 写入 binlog                                       │  │
│  │     │                                                      │  │
│  │     │ 13. redo log 状态改为 commit                         │  │
│  │                                                           │  │
│  └──────────────────────────────────────────────────────────┘  │
│     │                                                           │
│     │  14. 返回结果给客户端                                     │
│     ▼                                                           │
│  后台线程（异步）                                                │
│     │  15. 脏页刷新（Checkpoint）                              │
│     │  16. redo log 归档/删除                                 │
│     │                                                           │
└─────────────────────────────────────────────────────────────────┘
```

**核心代码示例：**

```sql
-- 最简单的 INSERT
INSERT INTO orders (order_no, user_id, amount) VALUES ('A001', 100, 99.9);

-- 执行计划（简化版）
EXPLAIN INSERT INTO orders (order_no, user_id, amount) VALUES ('A001', 100, 99.9);
-- type: const（主键/唯一键查找）
-- key: PRIMARY
-- rows: 1

-- 事务控制
START TRANSACTION;  -- 显式开启事务
INSERT INTO orders VALUES (...);
COMMIT;  -- 触发两阶段提交

-- autocommit 模式
SET autocommit = 1;  -- 默认开启，每条 SQL 自动提交
INSERT INTO orders VALUES (...);  -- 自动 COMMIT
```

## 🔥 扩展追问 1：redo log 的写入时机和原理？

**redo log = 物理日志，恢复已提交事务的修改**

```
┌─────────────────────────────────────────────────────────────────┐
│                     redo log 写入流程                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  redo log 结构：                                                │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  redo log buffer（内存）                                   │ │
│  │  ┌─────────┬─────────┬─────────┬─────────┐              │ │
│  │  │ log 1  │ log 2  │ log 3  │ log 4  │  ← redo log 记录 │ │
│  │  └─────────┴─────────┴─────────┴─────────┘              │ │
│  │                                                           │ │
│  │  参数: innodb_log_buffer_size（默认 16MB）               │ │
│  └───────────────────────────────────────────────────────────┘ │
│         │                                                       │
│         │  write() 到 os buffer                               │
│         ▼                                                       │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  redo log file（磁盘）                                    │ │
│  │  ib_logfile0, ib_logfile1（循环写入）                    │ │
│  │                                                           │ │
│  │  ┌─────────────────────────────────────────────────────┐│ │
│  │  │  [LSN 1000] [LSN 1001] [LSN 1002] ...  [LSN 9999]  ││ │
│  │  │  checkpoint                                         ││ │
│  │  └─────────────────────────────────────────────────────┘│ │
│  │                                                           │ │
│  │  参数: innodb_log_file_size（默认 48MB）                │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**redo log 何时写入？**

```sql
-- redo log 写入策略（innodb_flush_log_at_trx_commit）
-- ┌──────────────────────────────────────────────────────────┐
-- │ 参数值   │ write()时机        │ flush()时机   │ 安全性   │
-- ├──────────────────────────────────────────────────────────┤
-- │    0     │ 每秒一次            │ 每秒一次       │ 低 ❌     │
-- │    1     │ 每次提交时          │ 每次提交时     │ 高 ✓     │
-- │    2     │ 每次提交时          │ 每秒一次        │ 中 ⚠️    │
-- └──────────────────────────────────────────────────────────┘

-- 推荐配置（数据安全优先）
SET GLOBAL innodb_flush_log_at_trx_commit = 1;
```

**两阶段提交：**

```
┌─────────────────────────────────────────────────────────────────┐
│                     两阶段提交流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  INSERT 语句执行：                                              │
│                                                                 │
│  时刻 T1: 执行器调用 InnoDB 接口                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  // Step 1: 写数据页（Buffer Pool）                    │   │
│  │  change_buffer.insert(记录);                           │   │
│  │  │                                                     │   │
│  │  │                                                     │   │
│  │  // Step 2: 写 redo log（prepare 状态）                │   │
│  │  redo_log.write(记录, LSN, prepare);  ← redo log 写入  │   │
│  │  │                                                     │   │
│  │  │                                                     │   │
│  │  // Step 3: 写 binlog                                   │   │
│  │  binlog.write(SQL);              ← binlog 写入         │   │
│  │  │                                                     │   │
│  │  │                                                     │   │
│  │  // Step 4: redo log 状态改为 commit                   │   │
│  │  redo_log.set_commit(LSN);       ← redo log commit    │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  时刻 T2: 返回客户端"执行成功"                                  │
│                                                                 │
│  时刻 T3: 后台线程异步刷盘                                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  checkpoint 线程 ──▶ 脏页刷新到磁盘                      │   │
│  │  redo log 循环覆盖 ──▶ 旧的 redo log 可重用              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 2：undo log 的写入时机和作用？

**undo log = 撤销日志，用于 MVCC 和回滚**

```
┌─────────────────────────────────────────────────────────────────┐
│                     undo log 结构                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  undo log 链（每条记录多个版本）                                │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │                                                           │ │
│  │  trx_id=100 的更新                                        │ │
│  │       │                                                   │ │
│  │       │  undo ptr                                         │ │
│  │       │  (指向上一版本)                                    │ │
│  │       ▼                                                   │ │
│  │  ┌─────────────────────────────────────────────────────┐ │ │
│  │  │ undo record: {                                      │ │ │
│  │  │   type: UPDATE                                     │ │ │
│  │  │   old_value: amount=100                           │ │ │
│  │  │   undo_ptr: trx_id=90 的 undo record              │ │ │
│  │  │ }                                                   │ │ │
│  │  └─────────────────────────────────────────────────────┘ │ │
│  │       │                                                   │ │
│  │       │  undo ptr                                         │ │
│  │       ▼                                                   │ │
│  │  ┌─────────────────────────────────────────────────────┐ │ │
│  │  │ undo record: {                                      │ │ │
│  │  │   type: INSERT（初始插入）                         │ │ │
│  │  │   主键: id=1                                       │ │ │
│  │  │ }                                                   │ │ │
│  │  └─────────────────────────────────────────────────────┘ │ │
│  │                                                           │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**undo log 写入时机：**

```sql
-- undo log 在事务开始时就要写入

-- INSERT 的 undo log（最简单）
BEGIN;
INSERT INTO orders VALUES (1, 'A001', 100);
-- undo log: 记录"删除主键=1的记录"

-- UPDATE 的 undo log
UPDATE orders SET amount = 200 WHERE id = 1;
-- undo log: 记录"amount=100（旧值）"

-- DELETE 的 undo log（标记删除）
DELETE FROM orders WHERE id = 1;
-- undo log: 记录"插入主键=1的记录"

COMMIT;  -- INSERT 的 undo log 可立即删除
          -- UPDATE/DELETE 的 undo log 保留（MVCC）
```

**undo log 与 MVCC：**

```
┌─────────────────────────────────────────────────────────────────┐
│                     Read Committed 隔离级别                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  事务 A（READ）                  事务 B（WRITE）                  │
│     │                              │                            │
│     │  SELECT amount              │                            │
│     │  FROM orders               │                            │
│     │  WHERE id = 1              │                            │
│     │─────────────────────────────▶ 读到 amount=100             │
│     │                              │                            │
│     │                              │ UPDATE amount=200         │
│     │                              │ 1. 写 undo log（记录100）  │
│     │                              │ 2. 改数据页 amount=200    │
│     │                              │ 3. 事务未提交              │
│     │                              │                            │
│     │  SELECT amount              │                            │
│     │  FROM orders               │                            │
│     │  WHERE id = 1              │                            │
│     │─────────────────────────────▶ 读 undo log，找到旧值100    │
│     │  结果: amount=100 ✓        │                            │
│     │                              │                            │
│     │                              │ COMMIT                    │
│     │                              │ 1. binlog 写入             │
│     │                              │ 2. 释放锁                  │
│     │                              │                            │
│     │  SELECT amount              │                            │
│     │─────────────────────────────▶ 读到 amount=200（最新值）  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 3：binlog 的写入时机和作用？

**binlog = 逻辑日志，用于主从复制和数据恢复**

```
┌─────────────────────────────────────────────────────────────────┐
│                     binlog 写入流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  binlog 结构：                                                  │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  binlog cache（每个会话独立）                             │ │
│  │  ┌─────────────────────────────────────────────────────┐ │ │
│  │  │  [Event: BEGIN]                                    │ │ │
│  │  │  [Event: INSERT]  ← SQL 或 行变更（row格式）       │ │ │
│  │  │  [Event: COMMIT]                                   │ │ │
│  │  └─────────────────────────────────────────────────────┘ │ │
│  │  参数: binlog_cache_size（默认 32KB）                   │ │
│  └───────────────────────────────────────────────────────────┘ │
│         │                                                       │
│         │ 事务提交时写入                                        │
│         ▼                                                       │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  binlog file（磁盘）                                      │ │
│  │  mysql-bin.000001, mysql-bin.000002                      │ │
│  │                                                           │ │
│  │  ┌─────────────────────────────────────────────────────┐ │ │
│  │  │  Format_desc (文件头)                              │ │ │
│  │  │  Query_event (BEGIN)                               │ │ │
│  │  │  Table_map_event (表结构)                          │ │ │
│  │  │  Write_rows_event (INSERT 数据)                    │ │ │
│  │  │  Xid_event (COMMIT)                                │ │ │
│  │  └─────────────────────────────────────────────────────┘ │ │
│  │                                                           │ │
│  │  参数: sync_binlog=1（每次提交强制刷盘）                  │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**binlog 何时写入？**

```sql
-- binlog 写入时机：事务提交时

-- 写入策略（sync_binlog）
-- ┌──────────────────────────────────────────────────────────┐
-- │ 参数值   │ 刷盘时机              │ 性能     │ 安全性     │
-- ├──────────────────────────────────────────────────────────┤
-- │    0     │ 由 OS 决定            │ 最高 ⚡  │ 低 ❌      │
-- │    1     │ 每次 COMMIT 时刷盘    │ 中等     │ 高 ✓ 推荐  │
-- │    N>1   │ 每 N 次 COMMIT 刷盘   │ 较高     │ 中等 ⚠️   │
-- └──────────────────────────────────────────────────────────┘

-- 推荐配置
SET GLOBAL sync_binlog = 1;
```

**binlog 三种格式：**

```sql
-- 1. STATEMENT（SQL 语句）
SET binlog_format = STATEMENT;
-- binlog: UPDATE orders SET amount = amount + 1 WHERE id = 1
-- 优点：日志量小
-- 缺点：主从可能不一致（NOW(), RAND() 等函数）

-- 2. ROW（行变更）
SET binlog_format = ROW;
-- binlog: UPDATE orders SET amount=101 WHERE id=1
-- 优点：主从完全一致
-- 缺点：日志量大（MySQL 5.7+ 可用 binlog_row_image=minimal 优化）

-- 3. MIXED（混合模式）
SET binlog_format = MIXED;
-- MySQL 自动选择：普通 SQL 用 STATEMENT，有风险用 ROW
```

## 🔥 扩展追问 4：redo log、undo log、binlog 的区别总结？

| 维度 | redo log | undo log | binlog |
|------|----------|----------|--------|
| **类型** | 物理日志 | 逻辑日志 | 逻辑日志 |
| **内容** | 页的修改（"哪个页的哪个偏移改了什么"） | 逆向操作（"如何撤销这个修改"） | SQL 语句或行变更 |
| **作用** | 崩溃恢复（已提交事务） | 回滚 + MVCC | 主从复制 + 数据恢复 |
| **写入时机** | 随时写（事务中） | 事务开始时写 | 事务提交时写 |
| **持久化** | innodb_flush_log_at_trx_commit | 自动持久化 | sync_binlog |
| **空间** | 循环使用（固定大小） | Undo 表空间（可回收） | 追加写入（可配置） |
| **crash-safe** | ✅ 支持 | ✅ MVCC 支持 | ❌ 需配合 redo |
| **从库有吗？** | ❌ 只有主库有 | ❌ 只有主库有 | ✅ 主从都有 |


## 🔥 扩展追问 5：两阶段提交为什么能保证数据一致性？

**两阶段提交 = 分布式事务思想在 MySQL 内部的应用**

```
┌─────────────────────────────────────────────────────────────────┐
│                     两阶段提交原理                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  正常提交流程：                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                                                         │   │
│  │  T1: 写 redo log（prepare）                            │   │
│  │      redo: [prepare, LSN=1000, data=100]               │   │
│  │      ✓ 成功                                             │   │
│  │                                                         │   │
│  │  T2: 写 binlog                                          │   │
│  │      binlog: [INSERT ...]                               │   │
│  │      ✓ 成功                                             │   │
│  │                                                         │   │
│  │  T3: 写 redo log（commit）                             │   │
│  │      redo: [commit, LSN=1000]                          │   │
│  │      ✓ 成功                                             │   │
│  │                                                         │   │
│  │  事务提交完成                                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  异常场景 1：redo log prepare 成功，binlog 失败                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  T1: redo prepare ✓                                    │   │
│  │  T2: binlog ✗  ← 写失败                                 │   │
│  │                                                         │   │
│  │  恢复：扫描 redo log，发现 prepare 未 commit            │   │
│  │       → 执行 undo log 回滚                             │   │
│  │       → 事务回滚，数据一致 ✓                            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  异常场景 2：binlog 成功，redo log commit 失败                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  T1: redo prepare ✓                                    │   │
│  │  T2: binlog ✓                                          │   │
│  │  T3: redo commit ✗  ← 写失败                          │   │
│  │                                                         │   │
│  │  恢复：扫描 redo log，发现 prepare 已写 binlog          │   │
│  │       → redo log 状态设为 commit                       │   │
│  │       → 事务提交，数据一致 ✓                            │   │
│  │                                                         │   │
│  │  解释：binlog 已写入说明业务已确认                      │   │
│  │       必须保证数据库与 binlog 一致                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  异常场景 3：T1/T2/T3 之前宕机                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  T1: redo prepare ✗                                    │   │
│  │       → 事务未开始，无需处理 ✓                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**关键保证：binlog 已写入 = 事务必须提交**

```sql
-- MySQL 的核心原则：
-- "binlog 是 MySQL 的事实标准，binlog 写了就要保证提交"

-- 因此两阶段提交的顺序是：
-- 1. redo log (prepare)  ← 可以随时写
-- 2. binlog              ← 核心，必须先写
-- 3. redo log (commit)   ← 确保 redo 与 binlog 一致
```

## 🔥 扩展追问 6：Buffer Pool 在 INSERT 中扮演什么角色？

**Buffer Pool = InnoDB 的缓存池，减少磁盘 IO**

```
┌─────────────────────────────────────────────────────────────────┐
│                     Buffer Pool 结构                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Buffer Pool（默认 128MB，可配置）                               │

│  │                                                           │ │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐        │ │
│  │  │ Page 1  │ │ Page 2  │ │ Page 3  │ │ Page 4  │        │ │
│  │  │ (数据)   │ │ (数据)   │ │ (空闲)   │ │ (空闲)   │        │ │

│  │                                                           │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**INSERT 在 Buffer Pool 中的流程：**

```sql
-- INSERT 实际操作流程

-- 1. 检查数据页是否在 Buffer Pool
--    - 在 → 直接修改
--    - 不在 → 从磁盘加载到 Buffer Pool

-- 2. 修改数据页
--    - 写入新记录
--    - 标记页为脏页（加入 Flush List）

-- 3. 如果是唯一索引
--    - 检查唯一约束（可能触发随机 IO）

-- 4. 后台线程异步刷盘
--    - checkpoint 线程定期刷脏页
--    - 保证 redo log 可以覆盖之前的内容

-- Buffer Pool 配置
SHOW VARIABLES LIKE 'innodb_buffer_pool_size';
-- +-----------------------+----------+
-- | Variable_name         | Value    |
-- +-----------------------+----------+
-- | innodb_buffer_pool_size | 134217728 |  -- 128MB
-- +-----------------------+----------+

-- 配置建议：物理内存的 60-80%
SET GLOBAL innodb_buffer_pool_size = 8 * 1024 * 1024 * 1024;  -- 8GB
```

## 🔥 扩展追问 7：change buffer 是什么？INSERT 会用到吗？

**change buffer = 对非唯一辅助索引的变更缓存**

```
┌─────────────────────────────────────────────────────────────────┐
│                     change buffer 原理                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  问题：                                                          │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │                                                           │ │
│  │  辅助索引（普通索引）不是按主键排序的                       │ │
│  │  插入一条记录后，辅助索引页可能不在 Buffer Pool 中          │ │
│  │                                                           │ │
│  │  如果每次插入都读磁盘 → IO 暴增                            │ │
│  │                                                           │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
│  change buffer 解决方案：                                         │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │                                                           │ │
│  │  记录变更到 change buffer（内存）                          │ │
│  │  而不是立即更新索引页                                      │ │
│  │                                                           │ │
│  │  ┌─────────────────────────────────────────────────────┐ │ │
│  │  │  change buffer:                                     │ │ │
│  │  │  {                                                   │ │ │
│  │  │    "orders/order_no_index": [                       │ │ │
│  │  │      {op: INSERT, key: "A001", pk: 1},              │ │ │
│  │  │      {op: INSERT, key: "A002", pk: 2}              │ │ │
│  │  │    ]                                                 │ │ │
│  │  │  }                                                   │ │ │
│  │  └─────────────────────────────────────────────────────┘ │ │
│  │                                                           │ │
│  │  后续读索引页时，合并 change buffer                        │ │
│  │                                                           │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**INSERT 何时使用 change buffer：**

```sql
-- 唯一索引：不使用 change buffer（必须立即检查唯一性）
-- 普通索引：使用 change buffer（延迟合并）

-- 示例：
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,           -- 聚簇索引，无 change buffer
    order_no VARCHAR(32) UNIQUE,     -- 唯一索引，不使用 change buffer
    user_id BIGINT,                  -- 普通索引，使用 change buffer ✓
    INDEX idx_user_id (user_id)      -- 普通索引，使用 change buffer ✓
);

INSERT INTO orders VALUES (1, 'A001', 100);
-- order_no: 唯一索引 → 必须立即检查 → 无 change buffer
-- user_id: 普通索引 → 可以缓存 → 使用 change buffer

-- change buffer 参数
SHOW VARIABLES LIKE 'innodb_change_buffer%';
-- +-------------------------------+----------+
-- | Variable_name                 | Value    |
-- +-------------------------------+----------+
-- | innodb_change_buffer_max_size | 25       |  -- 占 Buffer Pool 的 25%
-- | innodb_change_buffering       | all      |  -- all=全部缓存
-- +-------------------------------+----------+
```

## 一句话总结（面试收尾用）

> **INSERT 执行时先写 undo log（用于回滚和 MVCC），再修改 Buffer Pool 中的数据页并标记脏页，同时写 redo log（prepare 状态），事务提交时先写 binlog 再将 redo log 置为 commit 状态，这就是两阶段提交，保证 binlog 和 InnoDB 数据一致；binlog 用于主从复制，redo log 用于崩溃恢复，undo log 用于回滚和 MVCC，三者缺一不可。**

**面试加分话术：**
> "我之前排查过一个主从延迟问题，从库比主库慢了好几秒，排查发现从库配置了 sync_binlog=1000（不是1），而且大事务一次插入10万条数据，导致 binlog 刷盘间隔太长。还有个经验：普通索引的 INSERT 比唯一索引快，因为唯一索引要用 change buffer，需要立即检查唯一约束，而普通索引可以把变更缓存起来异步合并。另外 innodb_flush_log_at_trx_commit 设置为1是数据安全的基本要求，设置为0或2在高并发场景下性能更好，但一旦服务器断电可能丢失1秒的数据，线上必须配成1。"
