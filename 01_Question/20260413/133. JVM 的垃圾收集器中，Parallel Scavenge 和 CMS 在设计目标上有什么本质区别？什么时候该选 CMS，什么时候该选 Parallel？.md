## 1. 基础流程

### 设计目标的本质区别

```
┌─────────────────────────────────────────────────────────────────────┐
│              Parallel Scavenge vs CMS 设计目标                       │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    Parallel Scavenge                        │   │
│  │  目标：吞吐量优先（Throughput First）                        │   │
│  │                                                             │   │
│  │  • 吞吐量 = 运行用户代码时间 / (运行用户代码时间 + GC时间)     │   │
│  │  • 追求：单位时间内完成更多任务                                │   │
│  │  • 适用：后台批处理、离线计算、离线任务                       │   │
│  │  • 关键字：吞吐量、CPU利用率、最高效率                       │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                         CMS                                 │
│  │  目标：停顿时间优先（Latency First）                          │   │
│  │                                                             │   │
│  │  • 追求：减少 STW（Stop-The-World）停顿时间                   │   │
│  │  • 适用：Web 服务、API服务、交互式应用                        │   │
│  │  • 关键字：低延迟、响应时间、用户体验                         │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Parallel Scavenge 收集流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                   Parallel Scavenge 收集流程                         │
│                                                                     │
│  年轻代：Parallel Scavenge（复制算法，多线程并发）                   │
│  老年代：Parallel Old（标记-整理算法，多线程并发）                   │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                        STW（全局停顿）                        │   │
│  │                                                             │   │
│  │   Phase 1: 初始标记（Initial Mark） ──→ 停顿                  │   │
│  │      ↓                                                      │   │
│  │   Phase 2: 并发标记（Concurrent Mark） ──→ 不停顿             │   │
│  │      ↓                                                      │   │
│  │   Phase 3: 重新标记（Remark） ──→ 停顿                        │   │
│  │      ↓                                                      │   │
│  │   Phase 4: 并发清理（Concurrent Sweep） ──→ 不停顿            │   │
│  │      ↓                                                      │   │
│  │   Phase 5: 并发重整（Concurrent Reset） ──→ 不停顿            │   │
│  │                                                             │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  ⚠️ 注意：CMS 的 STW 停顿时间短，但总 GC 时间更长                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 两者的核心对比

| 维度 | Parallel Scavenge | CMS |
|------|------------------|-----|
| **设计目标** | 吞吐量优先 | 停顿时间优先 |
| **GC 方式** | 独占式（STW 时暂停所有线程） | 并发式（大部分时间不暂停） |
| **吞吐量** | 高 | 中等 |
| **停顿时间** | 长 | 短 |
| **CPU 资源** | 尽量用满所有 CPU | 与用户线程共享 CPU |
| **内存碎片** | 无碎片（标记-整理） | 有碎片（标记-清除） |
| **适用场景** | 后台批处理、离线任务 | Web 服务、API、交互应用 |

## 2. 扩展追问

### ## 🔥 扩展追问 1：CMS 为什么要设计成并发收集？代价是什么？

> **A:** CMS 设计成并发收集是为了**减少停顿时间**，让用户线程和 GC 线程同时运行。但代价是 CPU 资源被 GC 和用户线程共享，另外因为并发执行会产生新的垃圾（浮动垃圾），且标记-清除算法会产生内存碎片。

```
┌─────────────────────────────────────────────────────────────────────┐
│                      CMS 并发收集的代价                              │
│                                                                     │
│  1. CPU 资源竞争                                                    │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  默认：CMS  GC 线程数 = (CPU核心数 + 3) / 4                   │   │
│  │                                                             │   │
│  │  4核 CPU → 1个 GC 线程                                        │   │
│  │  8核 CPU → 2个 GC 线程                                        │   │
│  │  16核 CPU → 4个 GC 线程                                       │   │
│  │                                                             │   │
│  │  CPU 资源被 GC 占用，吞吐量下降                                │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  2. 浮动垃圾（Floating Garbage）                                    │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  并发标记期间用户线程新产生的对象：                            │   │
│  │                                                             │   │
│  │  T1: 标记阶段开始                                             │   │
│  │  T2: 用户线程创建对象 A ──→ A 没被标记                         │   │
│  │  T3: 清理阶段 ──→ A 被当成垃圾清理掉！                         │   │
│  │                                                             │   │
│  │  只能留到下一次 GC 清理                                       │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  3. 内存碎片                                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  标记-清除算法 → 内存不连续 → 大对象无法分配 → Full GC         │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### ## 🔥 扩展追问 2：CMS 为什么会产生 Concurrent Mode Failure？如何应对？

> **A:** Concurrent Mode Failure 发生在**并发标记阶段老年代空间不够用**时，CMS 不得不退化成 Serial Old（单线程标记-整理），造成长时间的 STW。

```java
/**
 * Concurrent Mode Failure 发生条件
 * 
 * 1. 并发标记期间，老年代对象增长超过预留空间
 * 2. 用户线程在标记期间创建了大对象
 * 3. 浮动垃圾积累导致实际可用空间不足
 */
public class CMSFailureDemo {
    
    // JVM 参数：-XX:CMSInitiatingOccupancyFraction=70
    // 含义：老年代使用率达到 70% 时触发 CMS
    // 
    // 问题：浮动垃圾 + 对象增长 > 30% 预留空间 → 失败
}
```

| 应对策略 | JVM 参数 | 说明 |
|---------|---------|------|
| 调低触发阈值 | `-XX:CMSInitiatingOccupancyFraction=60` | 更早触发，留更多空间 |
| 启用增量模式 | `-XX:+UseCMSInitiatingOccupancyOnly` | 强制使用设定值 |
| 增大堆内存 | `-Xmx4g` | 空间更大，预留更多 |
| 切换收集器 | `-XX:+UseG1GC` | G1 用 Region 方式更好 |

### ## 🔥 扩展追问 3：Parallel Scavenge 的自适应调节策略是什么？

> **A:** Parallel Scavenge 提供了一个**自适应调节**参数 `-XX:+UseAdaptiveSizePolicy`，开启后 JVM 会根据运行时数据自动调节年轻代大小、Eden/Survivor 比例等参数，让吞吐量始终保持在目标范围内。

```java
/**
 * Parallel Scavenge 自适应策略
 * 
 * 关键参数：
 * -XX:GCTimeRatio=19        // GC 时间占比 = 1/(1+19) = 5%
 * -XX:MaxGCPauseMillis=200  // 最大停顿时间目标（软目标，不保证）
 * -XX:+UseAdaptiveSizePolicy // 开启自适应调节
 * 
 * JVM 会自动调整：
 * • 新生代大小（-Xmn）
 * • Eden/Survivor 比例
 * • 对象晋升年龄
 * 
 * 目的：在满足停顿时间目标的前提下，最大化吞吐量
 */
public class AdaptivePolicyDemo {
    // 如果停顿时间太长 → 自动增大年轻代
    // 如果 GC 太频繁 → 自动减小年轻代
    // 自动平衡
}
```

> **注意：** 自适应策略可能与预期不符，生产环境常用 `-XX:+UseAdaptiveSizePolicy=false` 手动指定参数。

### ## 🔥 扩展追问 4：CMS 和 G1 的区别是什么？什么时候选 G1？

> **A:** G1 可以看作是 CMS 的**进化版**，解决了 CMS 的内存碎片问题和停顿时间不可控问题。G1 把堆划分成多个大小相等的 Region，采用标记-整理算法，可以控制最大停顿时间。

| 维度 | CMS | G1 |
|------|-----|-----|
| 内存管理 | 传统分代（Eden/Survivor/Old） | Region 划分（2048 个 Region） |
| 算法 | 标记-清除（有碎片） | 标记-整理（无碎片） |
| 停顿时间 | 不可控（依赖对象分配率） | 可控（-XX:MaxGCPauseMillis=200） |
| 内存碎片 | 有碎片，需要 Full GC 整理 | 无碎片 |
| 大对象 | 直接进老年代 | 可以跨 Region（Humongous 区） |
| 适用场景 | 低延迟场景（替代方案） | JDK 11+ 默认，低延迟+低碎片 |

```java
// G1 调优参数示例
-XX:+UseG1GC                    // 启用 G1
-XX:MaxGCPauseMillis=200       // 最大停顿时间 200ms（软目标）
-XX:G1HeapRegionSize=4m         // Region 大小
-XX:InitiatingHeapOccupancyPercent=45  // 触发并发 GC 的堆使用率
```

### ## 🔥 扩展追问 5：JDK 11+ 该怎么选收集器？

> **A:** JDK 11+ 推荐**默认使用 G1**，如果对延迟要求极高（99.99% SLA），可以考虑 ZGC 或 Shenandoah。

| 收集器 | JDK 版本 | 特点 | 适用场景 |
|--------|---------|------|----------|
| **G1** | 9+ 默认 | 平衡型，可控停顿，无碎片 | 通用场景，推荐首选 |
| **ZGC** | 11+ | 亚毫秒级停顿，染色指针 | 超低延迟，TB 级堆 |
| **Shenandoah** | 12+ | 类似 ZGC，开源 | 低延迟，OpenJDK |
| **Epsilon** | 11+ | 无 GC，内存耗尽直接 OOM | 短命程序、测试 |

```java
// JDK 11+ 推荐配置
// 通用 Web 服务
-XX:+UseG1GC
-XX:MaxGCPauseMillis=100

// 超低延迟（内存 < 100GB）
-XX:+UseZGC
-XX:MaxGCPauseMillis=10
```

## 3. 一句话总结

> **Parallel Scavenge 追求吞吐量优先、牺牲停顿时间，采用独占式 STW 收集；CMS 追求低延迟优先、牺牲吞吐量，采用并发收集但有内存碎片和浮动垃圾问题；吞吐量优先选 Parallel Scavenge，低延迟优先选 CMS，JDK 11+ 推荐默认使用 G1。**

## 4. 面试加分话术

> "我在项目里用过 CMS 和 G1，对比很明显——当时接手一个老系统用的是 CMS+ParNew，年轻代 GC 很频繁还好，但老年代 CMS 经常触发 Concurrent Mode Failure，后来改成了 G1，调了 MaxGCPauseMillis=200，停顿时间稳定多了。还有个经验：CMS 对 CPU 敏感，核心数少（4核以下）的时候用 CMS 可能反而更慢，因为 GC 线程抢了 CPU。这种场景我建议用 Parallel Scavenge 拼吞吐量，或者直接升级 JDK 用 G1。另外 CMS 的 `-XX:CMSInitiatingOccupancyFraction` 要根据实际对象分配率调，我们当时调到了 65 才稳定。"
