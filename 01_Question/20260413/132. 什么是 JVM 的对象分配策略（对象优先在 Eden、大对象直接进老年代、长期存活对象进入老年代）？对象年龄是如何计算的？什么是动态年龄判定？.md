 */
public class AgeCalculator {
    public static void main(String[] args) {
        // 默认参数：-XX:MaxTenuringThreshold=15
        // Survivor 区比例：-XX:TargetSurvivorRatio=50
        
        // 对象生命周期示例：
        // 第1次Minor GC：Eden → Survivor0，年龄=1
        // 第2次Minor GC：Survivor0 → Survivor1，年龄=2
        // 第3次Minor GC：Survivor1 → Survivor0，年龄=3
        // ...

## 2. 扩展追问

### ## 🔥 扩展追问 1：为什么要分区？Eden 和 Survivor 的意义是什么？

> **A:** 分代的目的是**根据对象生命周期合理分配内存**，减少 GC 开销。年轻代的对象大多是"朝生夕死"的，Eden 区满了触发 Minor GC 时，大部分对象直接被回收，只有少量存活的对象会被复制到 Survivor 区。这个设计让大部分垃圾在 Eden 区就被清理掉，不用扫描整个堆。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    分代收集的必要性                                  │
│                                                                     │
│  经验数据：                                                         │
│  • 80%~90% 的对象朝生夕死，活不过第一次 Minor GC                     │
│  • 只有少量对象会长期存活                                             │
│                                                                     │
│  如果不分代：                                                        │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  全堆扫描 ──→ 扫描所有对象 ──→ 大量无用对象被反复检查            │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  分代后：                                                           │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Minor GC: 只扫描年轻代 ──→ 大量对象直接回收                    │   │
│  │  Full GC:  扫描整个堆 ──→ 对象少，效率高                        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### ## 🔥 扩展追问 2：为什么需要两个 Survivor 区？

> **A:** Survivor 区的设计借鉴了**复制算法**的思想。两个 Survivor 区（S0 和 S1）交替使用，解决了一个关键问题：**对象去向**。

```java
/**
 * 复制算法 + Survivor 区的配合
 * 
 * 核心思想：每次 Minor GC 后，存活的对象复制到另一个 Survivor 区
 * 原来的 Survivor 区直接清空，避免内存碎片
 */
public class SurvivorMechanism {
    
    // Minor GC 发生前的状态
    // ┌─────────────────┬─────────────────┐
    // │      Eden       │      S0         │  ← S1 为空
    // │   (对象存活)     │   (可能有对象)    │
    // └─────────────────┴─────────────────┘
    
    // Minor GC 后
    // 1. Eden + S0 中存活的对象复制到 S1
    // 2. Eden 和 S0 清空
    // ┌─────────────────┬─────────────────┐
    // │      Eden       │      S1         │  ← S0 为空
    // │    (空了)        │   (存活对象)     │
    // └─────────────────┴─────────────────┘
    
    // 下一次 Minor GC 后，S0 和 S1 互换
}
```

> **好处：**
> - 不需要额外的大块空闲内存
> - 避免内存碎片的产生
> - 存活对象通过复制"整理"了内存

### ## 🔥 扩展追问 3：对象年龄为什么默认是 15？不是 16 或 8？

> **A:** 对象年龄存储在 **对象头（Mark Word）** 的 **4 bits** 中，4 bits 最大能表示的值是 15（1111），所以年龄最大就是 15。

```java
/**
 * 对象头结构（32位虚拟机）
 * 
 * ┌───────────────────────────────────────┐
 * │         Mark Word (32 bits)           │
 * ├──────────┬────────────────────────────┤
 * │  age(4)  │        其他标记位           │  ← 年龄只有4位，最大15
 * └──────────┴────────────────────────────┘
 * 
 * 64位虚拟机同样只有 4 bits 用于对象年龄
 */
public class AgeBitsDemo {
    // -XX:MaxTenuringThreshold 可以设置年龄阈值
    // 但最大只能是 15
    // JVM 参数：-XX:MaxTenuringThreshold=15
}
```

> **可配置性：** 可以通过 `-XX:MaxTenuringThreshold=N` 设置，但最大只能是 15。

### ## 🔥 扩展追问 4：动态年龄判定在什么场景下有用？

> **A:** 动态年龄判定是**一个保护机制**，防止 Survivor 区空间不足导致对象提前进入老年代。

```java
/**
 * 动态年龄判定触发场景
 * 
 * 场景：大量短期存活对象
 * 
 * 例如：循环创建临时对象
 */
public class DynamicAgeDemo {
    public static void main(String[] args) {
        // 假设 Survivor 区只有 10MB
        // TargetSurvivorRatio = 50% → 阈值 5MB
        
        // 不断创建对象，这些对象活不过几次 GC
        // 但每次 Minor GC 后，Survivor 区可能容纳不下
        
        for (int i = 0; i < 10000; i++) {
            byte[] data = new byte[1024];  // 1KB 临时对象
            // 处理...
        }
        
        // 动态年龄判定介入：
        // 同年龄对象总大小 > 50%，则进入老年代
        // 避免 Survivor 区溢出
    }
}
```

> **实际影响：**
> - 如果 Survivor 区太小，对象可能还没到 15 岁就被"动态"送入老年代
> - 可能导致老年代频繁 Full GC

### ## 🔥 扩展追问 5：大对象为什么直接进老年代？如何配置？

> **A:** 大对象直接进老年代是为了**避免大对象在年轻代来回复制**。大对象（长数组、大字符串等）如果放在 Eden 区，Minor GC 时复制到 Survivor 区会非常耗时；另外大对象容易导致 Survivor 区空间频繁不足。

```java
// JVM 参数配置
public class LargeObjectConfig {
    public static void main(String[] args) {
        // -XX:PretenureSizeThreshold=1024
        // 含义：大于 1KB 的对象直接在老年代分配
        
        // 常见配置：
        // -XX:PretenureSizeThreshold=1048576  // 1MB
        
        // 示例：
        byte[] bigArray = new byte[1024 * 1024];  // 1MB 大数组
        // 分配到老年代，不占用年轻代空间
    }
}
```

| 参数 | 说明 | 示例 |
|------|------|------|
| `-XX:PretenureSizeThreshold=N` | 大于该值的对象直接进老年代 | `-XX:PretenureSizeThreshold=1048576` (1MB) |
| `-XX:MaxTenuringThreshold=N` | 对象年龄阈值 | `-XX:MaxTenuringThreshold=15` |
| `-XX:TargetSurvivorRatio=N` | Survivor 区填充比例 | `-XX:TargetSurvivorRatio=50` |

---

## 3. 一句话总结

> **JVM 对象分配遵循 Eden 优先、大对象直接进老年代、长期存活对象年龄达 15 后进老年代的策略；对象年龄在 Minor GC 时从 Survivor 区复制一次增加 1；动态年龄判定会在 Survivor 区同龄对象总大小超过 50% 时提前将大于等于该年龄的对象送入老年代，防止 Survivor 区溢出。**
