## 1. 基础流程

### newCachedThreadPool 的线程模型

```
┌─────────────────────────────────────────────────────────────────────┐
│                 newCachedThreadPool 线程池模型                       │
│                                                                     │
│  corePoolSize = 0          maximumPoolSize = Integer.MAX_VALUE      │
│  keepAliveTime = 60s       queue = SynchronousQueue                │
│                                                                     │
│   任务进来 ──→ 有空闲线程？ ──→ 有 ──→ 复用该线程执行                 │
│                    │                                               │
│                    ├── 无 ──→ 创建新线程执行 ──→ 60秒无任务销毁      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### newFixedThreadPool 的线程模型

```
┌─────────────────────────────────────────────────────────────────────┐
│                 newFixedThreadPool 线程池模型                        │
│                                                                     │
│  corePoolSize = n              maximumPoolSize = n                  │
│  keepAliveTime = 0s            queue = LinkedBlockingQueue(无界)    │
│                                                                     │
│   任务进来 ──→ 队列 ──→ 线程慢慢消费                                 │
│                                                                     │
│  ⚠️ 问题：队列无界，任务堆积 → OOM                                   │
└─────────────────────────────────────────────────────────────────────┘
```

### 问题一：无界队列导致 OOM

```java
// newFixedThreadPool 实际等价于:
new ThreadPoolExecutor(
    10,                                    // corePoolSize = 10
    10,                                    // maximumPoolSize = 10
    0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<Runnable>()   // ⚠️ 默认容量 Integer.MAX_VALUE
);

// 场景：高峰期 1000 QPS 涌入，10个线程处理不过来
// 结果：LinkedBlockingQueue 堆积 → OOM
```

### 问题二：无界线程数导致 OOM

```java
// newCachedThreadPool 实际等价于:
new ThreadPoolExecutor(
    0,                                     // corePoolSize = 0
    Integer.MAX_VALUE,                     // ⚠️ 线程数无上限！
    60L, TimeUnit.SECONDS,
    new SynchronousQueue<Runnable>()       // 队列容量为1，必须有线程立刻拿走
);

// 场景：1000个请求同时超时，创建1000个线程
// 结果：内存暴涨 → OOM
```

### 问题现场还原

```java
public class ThreadPoolOOMDemo {
    // newFixedThreadPool 炸弹
    static ExecutorService pool = Executors.newFixedThreadPool(10);
    
    public static void main(String[] args) {
        for (int i = 0; i < 1_000_000; i++) {
            pool.submit(() -> {
                try {
                    Thread.sleep(1000);
                } catch (InterruptedException e) {}
            });
        }
    }
    // 运行结果:
    // Exception in thread "main" java.lang.OutOfMemoryError: GC overhead limit exceeded
    // 或: java.lang.OutOfMemoryError: Unable to create new native thread
}
```

## 2. 扩展追问

### ## 🔥 扩展追问 1：阿里规约具体说了什么？

> **《阿里巴巴 Java 开发手册》强制要求：**
> 【强制】线程池不允许使用 Executors 去创建，而是通过 `ThreadPoolExecutor` 的方式
> 
> 理由：避免资源被耗尽的风险

**Executors 创建线程池问题汇总：**

| 方法 | corePool | maxPool | 队列 | 问题 |
|------|----------|---------|------|------|
| `newFixedThreadPool` | n | n | LinkedBlockingQueue(无界) | 队列堆积 → OOM |
| `newCachedThreadPool` | 0 | Integer.MAX_VALUE | SynchronousQueue | 线程数无界 → OOM |
| `newSingleThreadExecutor` | 1 | 1 | LinkedBlockingQueue(无界) | 单线程 + 无界队列 = 堵死 |
| `newScheduledThreadPool` | core | Integer.MAX_VALUE | DelayedWorkQueue | 最大线程无界 |

### ## 🔥 扩展追问 2：正确的线程池创建方式是什么？

```java
// ✅ 推荐：显式指定核心参数，有界队列
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10,        // corePoolSize：核心线程数
    50,        // maximumPoolSize：最大线程数（通常是 core 的 2-5 倍）
    60L,       // keepAliveTime：多余线程存活时间
    TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000),  // ✅ 有界队列，设置合理容量
    new ThreadFactory() {
        private AtomicInteger count = new AtomicInteger(1);
        @Override
        public Thread newThread(Runnable r) {
            return new Thread(r, "biz-pool-" + count.getAndIncrement());
        }
    },
    new ThreadPoolExecutor.CallerRunsPolicy()  // ✅ 拒绝策略
);
```

### ## 🔥 扩展追问 3：核心参数怎么定？

```
┌─────────────────────────────────────────────────────────────────┐
│                    线程池参数计算公式                            │
│                                                                 │
│  N_threads = N_cpu × U_cpu × (1 + W/C)                          │
│                                                                 │
│  • N_cpu = Runtime.getRuntime().availableProcessors()          │
│  • U_cpu = 目标CPU利用率 (通常 0.7-0.9)                          │
│  • W = 等待时间 (等待IO/等待锁/等待网络)                         │
│  • C = 计算时间 (CPU实际运算)                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**不同场景参数建议：**

| 场景 | 任务类型 | corePool | maxPool | 队列容量 | keepAlive |
|------|----------|----------|---------|----------|-----------|
| **CPU密集型** | 计算/加密/压缩 | CPU+1 | CPU+1 | 200-500 | 60s |
| **IO密集型** | DB/Redis/外部API | CPU×2~8 | CPU×2~4 | 500-2000 | 30-60s |
| **混合型** | 两者混合 | CPU×(1+W/C) | core×2~5 | 1000+ | 60s |

```java
// CPU密集型：避免CPU完全饱和
int cpuCount = Runtime.getRuntime().availableProcessors();
ThreadPoolExecutor cpuPool = new ThreadPoolExecutor(
    cpuCount + 1, cpuCount + 1, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(200), new CallerRunsPolicy()
);

// IO密集型：等待时可处理更多任务
ThreadPoolExecutor ioPool = new ThreadPoolExecutor(
    cpuCount * 2, cpuCount * 4, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000), new CallerRunsPolicy()
);
```

### ## 🔥 扩展追问 4：拒绝策略怎么选？

| 策略 | 行为 | 适用场景 | 风险 |
|------|------|----------|------|
| `AbortPolicy` | 抛RejectedExecutionException | 业务可降级 | 任务直接丢失 |
| `CallerRunsPolicy` | 由调用方线程执行 | 需要保证任务不丢失 | 调用线程阻塞，可能拖慢上游 |
| `DiscardPolicy` | 静默丢弃 | 无关紧要的任务 | 任务静默丢失 |
| `DiscardOldestPolicy` | 丢弃队列最旧的任务 | 优先处理新任务 | 旧任务丢失 |

```java
// 场景1：支付订单处理
// ✅ CallerRunsPolicy：保证任务不丢失，调用方自动限流

// 场景2：日志收集
// ✅ DiscardOldestPolicy：新日志比旧日志重要

// 场景3：营销活动（可丢弃）
// ✅ DiscardPolicy：非核心任务
```

### ## 🔥 扩展追问 5：生产环境怎么监控？

```java
public class ThreadPoolMonitor {
    public static void printStatus(ThreadPoolExecutor executor) {
        System.out.println("========== ThreadPool Status ==========");
        System.out.println("Active Threads:   " + executor.getActiveCount());
        System.out.println("Pool Size:        " + executor.getPoolSize());
        System.out.println("Queue Size:       " + executor.getQueue().size());
        System.out.println("Completed Tasks:  " + executor.getCompletedTaskCount());
        System.out.println("========================================");
    }
}

// 告警规则
if (executor.getQueue().size() > 800) {
    alert("队列堆积超过80%，当前容量: " + executor.getQueue().size());
}
if (executor.getActiveCount() >= executor.getMaximumPoolSize()) {
    alert("线程池已饱和，所有线程都在工作");
}
```

## 3. 一句话总结

> **Executors 创建线程池的问题是 newFixedThreadPool 用无界队列和 newCachedThreadPool 用无界线程数都可能导致 OOM，应通过 ThreadPoolExecutor 显式设置核心线程数、最大线程数、有界队列和 CallerRunsPolicy 拒绝策略，CPU 密集型设为核心 CPU+1，IO 密集型设为核心 CPU×2~8。**

## 4. 面试加分话术

> "我在项目里真实遇到过这个问题——当时有个定时任务用 newFixedThreadPool 处理订单同步，队列堆积到 30 万直接 OOM 了。后来改成有界队列 + CallerRunsPolicy + 监控告警，队列超过 1000 就触发扩容逻辑。还有个经验：CPU 密集型用 Runtime.getRuntime().availableProcessors() + 1，IO 密集型乘以 2 到 8 倍，混合型按等待时间和计算时间的比例来估算。另外线上一定要加监控，我们加了队列大小和活跃线程数的告警，提前发现问题。"
