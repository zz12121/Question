## 1. 基础流程

### 加锁类型与索引的关系

```
┌─────────────────────────────────────────────────────────────────────┐
│                 SELECT ... FOR UPDATE 加锁原理                        │
│                                                                     │
│  条件字段有索引？ ──→ 是 ──→ 走索引 ──→ 行锁/间隙锁                    │
│        │                                                            │
│        ├── 无索引 ──→ 全表扫描 ──→ 表锁（所有记录+间隙）             │
│        │                                                            │
│        └── 主键/唯一索引 ──→ 行锁（精确命中）                         │
│                                                                     │
│  锁住的是：                                                         │
│  • 索引记录本身（行锁）                                             │
│  • 索引之间的间隙（间隙锁）                                          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 三种锁的示意

```
┌─────────────────────────────────────────────────────────────────────┐
│                      InnoDB 行锁类型                                  │
│                                                                     │
│  id:    1       5       10      15      20      30                   │
│       ┌───┐   ┌───┐   ┌───┐   ┌───┐   ┌───┐   ┌───┐              │
│       │ 1 │   │ 5 │   │10 │   │15 │   │20 │   │30 │              │
│       └───┘   └───┘   └───┘   └───┘   └───┘   └───┘              │
│         │       │       │       │       │       │                  │
│       行锁   间隙锁   间隙锁   间隙锁   间隙锁   间隙锁              │
│      (REC)   (GAP)   (GAP)   (GAP)   (GAP)   (GAP)                  │
│                                                                     │
│  锁定的范围：                                                        │
│  • 行锁：锁住 id=10 的记录                                          │
│  • 间隙锁：锁住 (5,10)、(10,15) 之间的间隙                          │
│  • Next-Key Lock：行锁+间隙锁 = (5,15)                              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 核心代码示例

```sql
-- 场景1：主键/唯一索引 → 行锁
SELECT * FROM user WHERE id = 10 FOR UPDATE;
-- 锁住：id=10 的行

-- 场景2：普通索引 → 间隙锁
SELECT * FROM user WHERE age = 25 FOR UPDATE;
-- 假设 age 有索引，age=25 的记录有 3 条
-- 锁住：age=25 的行 + 前后间隙

-- 场景3：无索引 → 表锁
SELECT * FROM user WHERE name = '张三' FOR UPDATE;
-- 锁住：整张表！

-- 场景4：范围查询 → Next-Key Lock
SELECT * FROM user WHERE id BETWEEN 10 AND 20 FOR UPDATE;
-- 锁住：[10,20] 区间内所有记录 + 前后间隙
```

## 2. 扩展追问

### ## 🔥 扩展追问 1：RC 和 RR 隔离级别下加锁范围有什么不同？

> **A:** **RC（读已提交）** 级别下，`SELECT ... FOR UPDATE` 只锁住**扫描到的行**，不加间隙锁；**RR（可重复读）** 级别下会加 **Next-Key Lock**，锁定扫描范围及所有间隙，防止幻读。

| 隔离级别 | 加锁范围 | 防止幻读 | 说明 |
|---------|---------|---------|------|
| **RC** | 只锁扫描到的行 | ❌ 不防止 | 可能有幻读（其他事务插入的新行） |
| **RR** | 行锁 + 间隙锁（Next-Key Lock） | ✅ 防止 | 锁定一个范围，无幻读 |
| **SERIALIZABLE** | 隐式转换为 FOR UPDATE | ✅ 防止 | 完全串行化 |

```sql
-- RR 级别下的 Next-Key Lock 示例
-- 表: user (id 主键, age 普通索引)
-- 数据: id = 1, 5, 10, 15

-- 执行：
SELECT * FROM user WHERE age = 25 FOR UPDATE;

-- RR 级别加锁：
-- 1. age=25 可能存在的位置有 2 个区间：
--    (-∞, 10) 和 (10, +∞)
-- 2. 锁定这两个区间（间隙锁）
-- 3. 如果有 id=25 的记录，锁住该行（行锁）

-- 实际效果：
-- 其他事务无法插入 age 在 10~+∞ 之间的记录（除非提交）
-- 其他事务无法插入 age 在 -∞~10 之间的记录（除非提交）
```

```sql
-- RC 级别：只锁扫描到的记录
SELECT * FROM user WHERE id > 10 FOR UPDATE;
-- 假设 id > 10 有 100 条记录
-- RC：只锁这 100 条记录，不锁间隙
-- 其他事务可以插入 id=11, 12, 13...（只要不在锁住的行里）

-- RR 级别：锁扫描范围 + 间隙
SELECT * FROM user WHERE id > 10 FOR UPDATE;
-- 锁定 id > 10 的所有记录
-- 锁定 (最后一条锁住的记录, +∞) 的间隙
-- 其他事务无法插入 id > 10 的记录
```

### ## 🔥 扩展追问 2：什么情况下会退化为表锁？

> **A:** 主要有三种情况会退化为表锁：**无索引/索引失效**、**锁等待超时**、**显式加表锁**。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    退化为表锁的几种情况                                │
│                                                                     │
│  情况1：无索引或索引失效                                             │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  SELECT * FROM orders WHERE status = 'paid' FOR UPDATE;     │   │
│  │  如果 status 字段没有索引 → 全表扫描 → 表锁                  │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  情况2：索引失效                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  • WHERE 子句使用了函数/运算：WHERE YEAR(create_time) = 2024  │   │
│  │  • 类型转换：WHERE phone = 123456  (phone 是 varchar)        │   │
│  │  • LIKE 开头是通配符：WHERE name LIKE '%张%'                  │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  情况3：记录不存在时的 RR 级别                                        │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  SELECT * FROM user WHERE id = 999 FOR UPDATE;              │   │
│  │  -- id=999 不存在                                            │   │
│  │  -- RR 级别下仍会加间隙锁，锁定 (998, +∞) 或 (前一个值, +∞)   │   │
│  │  -- 大范围查询不存在的记录 → 大范围间隙锁 → 类似表锁          │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  情况4：显式表锁                                                    │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  LOCK TABLES orders READ/WRITE;                             │   │
│  │  -- 直接锁定整张表                                            │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

```sql
-- 索引失效示例
-- phone 字段是 varchar(20) 索引

-- ❌ 索引失效（隐式类型转换）
SELECT * FROM user WHERE phone = 13800138000 FOR UPDATE;

-- ✅ 索引生效
SELECT * FROM user WHERE phone = '13800138000' FOR UPDATE;

-- ❌ 索引失效（函数）
SELECT * FROM user WHERE YEAR(birthday) = 2000 FOR UPDATE;

-- ✅ 索引生效
SELECT * FROM user WHERE birthday BETWEEN '2000-01-01' AND '2000-12-31' FOR UPDATE;
```

### ## 🔥 扩展追问 3：间隙锁（Gap Lock）和 Next-Key Lock 有什么区别？

> **A:** **间隙锁**只锁定索引记录之间的缝隙，**Next-Key Lock** 是行锁+间隙锁的组合（`记录锁 + Gap Lock`），是 RR 级别防止幻读的核心机制。

| 锁类型 | 锁定内容 | 作用 |
|-------|---------|------|
| **Record Lock（行锁）** | 单条索引记录 | 阻止其他事务修改/删除该记录 |
| **Gap Lock（间隙锁）** | 索引记录之间的间隙 | 阻止其他事务在间隙中插入新记录 |
| **Next-Key Lock** | 行锁 + 间隙锁 | 阻止修改/删除/插入，完整防止幻读 |

```sql
-- Next-Key Lock 示例
-- 表: product (id 主键, category_id 普通索引, name)
-- 数据: id=1(category=1), id=5(category=1), id=10(category=2)

SELECT * FROM product WHERE category_id = 1 FOR UPDATE;

-- Next-Key Lock 加锁范围：
-- ┌─────────────────────────────────────────────────────────────┐
-- │  category_id = 1 的记录 × 2                                  │
-- │  category_id 索引的间隙：                                      │
-- │    (-∞, 1)      → 锁定                                       │
-- │    (1, 1)       → 锁定（id=1和id=5之间）                       │
-- │    (1, +∞)      → 锁定（id=5之后到2之前）                       │
-- │                                                             │
-- │  最终效果：锁定 category_id=1 的所有区间                      │
-- │  其他事务无法插入 category_id=1 或 category_id=2 的记录！    │
-- └─────────────────────────────────────────────────────────────┘
```

### ## 🔥 扩展追问 4：死锁是怎么产生的？如何避免？

> **A:** 死锁产生于**互相等待对方持有的锁**，常见于并发事务以不同顺序访问相同资源。

```
┌─────────────────────────────────────────────────────────────────────┐
│                      死锁产生示意                                    │
│                                                                     │
│  事务A：                            事务B：                          │
│  ───────                           ───────                          │
│  SELECT * WHERE id=1 FOR UPDATE;    SELECT * WHERE id=2 FOR UPDATE; │
│  -- 锁定 id=1                       -- 锁定 id=2                    │
│       ↓                                   ↓                          │
│  UPDATE ... WHERE id=2;             UPDATE ... WHERE id=1;           │
│  -- 等待 id=2（事务B持有）           -- 等待 id=1（事务A持有）        │
│       ↓                                   ↓                          │
│     死锁！                               死锁！                        │
│                                                                     │
│  解决：InnoDB 会选择回滚 UNDO 量最小的事务                           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

```sql
-- 避免死锁的策略

-- 1. 统一访问顺序（所有事务按相同顺序访问资源）
BEGIN;
SELECT * FROM account WHERE id = 1 FOR UPDATE;  -- 先锁 id=1
SELECT * FROM account WHERE id = 2 FOR UPDATE;  -- 再锁 id=2
COMMIT;

BEGIN;
SELECT * FROM account WHERE id = 2 FOR UPDATE;  -- 先锁 id=2
SELECT * FROM account WHERE id = 1 FOR UPDATE;  -- 再锁 id=1
COMMIT;

-- 2. 减小锁范围（只锁必要的记录）
-- ❌ 锁范围过大
SELECT * FROM orders WHERE status = 'pending' FOR UPDATE;

-- ✅ 精确锁定
SELECT * FROM orders WHERE id = ? FOR UPDATE;

-- 3. 设置合理的锁等待超时
-- innodb_lock_wait_timeout = 5  (默认50秒太长)

-- 4. 使用低隔离级别（降低锁冲突概率）
-- RC 级别只锁行，不锁间隙，死锁概率更低
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

### ## 🔥 扩展追问 5：UPDATE 和 DELETE 的加锁和 SELECT ... FOR UPDATE 有什么区别？

> **A:** `UPDATE` 和 `DELETE` 的加锁逻辑和 `SELECT ... FOR UPDATE` 类似，但有一个关键区别：**UPDATE 会先定位到记录再锁定**，而 `SELECT ... FOR UPDATE` 是显式声明要加排他锁。

```sql
-- SELECT ... FOR UPDATE：先加锁，再读取
SELECT * FROM user WHERE id = 10 FOR UPDATE;
-- 立即锁定 id=10 的行

-- UPDATE：先读取判断，再加锁
UPDATE user SET balance = balance - 100 WHERE id = 10;
-- 执行过程：
-- 1. InnoDB 先定位到 id=10 的记录
-- 2. 加 X 锁（排他锁）
-- 3. 执行更新

-- DELETE：和 UPDATE 类似
DELETE FROM user WHERE id = 10;
-- 定位 → 加 X 锁 → 删除

-- 关键区别：
-- • UPDATE/DELETE 在记录不存在时不加锁（RC）或加间隙锁（RR）
-- • SELECT ... FOR UPDATE 在记录不存在时也会加间隙锁（RR）

-- 示例：RR 级别下
SELECT * FROM user WHERE id = 999 FOR UPDATE;  -- id=999 不存在 → 加间隙锁
UPDATE user SET name = 'test' WHERE id = 999;   -- id=999 不存在 → 不加锁！
```

| 操作 | 记录存在 | 记录不存在（RR） | 记录不存在（RC） |
|------|---------|-----------------|-----------------|
| SELECT ... FOR UPDATE | 行锁 | 间隙锁 | 无锁 |
| UPDATE WHERE 主键 | 行锁 | 无锁 | 无锁 |
| UPDATE WHERE 普通索引 | 行锁+间隙锁 | 间隙锁 | 无锁 |
| DELETE WHERE 主键 | 行锁 | 无锁 | 无锁 |

---

## 3. 一句话总结

> **SELECT ... FOR UPDATE 在有索引时加行锁或间隙锁（RR），无索引时退化为表锁；RR 级别加 Next-Key Lock（行锁+间隙锁）防止幻读，RC 级别只锁扫描到的行不加间隙锁；UPDATE/DELETE 定位到记录才加锁，记录不存在时不加锁，但 SELECT FOR UPDATE 在 RR 下仍会加间隙锁。**

## 4. 面试加分话术

> "我在项目里遇到过死锁问题——当时做库存扣减，并发下单的时候两个事务分别锁住了不同的订单，然后去更新对方的库存，产生了死锁。后来统一了 SQL 的执行顺序，先按主键排序再逐条锁定，避免了死锁。还有个经验：RC 级别下 `SELECT FOR UPDATE` 只锁扫描到的行，适合读多写少的场景，但要注意幻读问题；RR 级别会锁间隙，锁冲突更严重，但不会有幻读。另外有个坑：普通索引的间隙锁会锁定相邻区间的数据，比如锁定 age=25 会把 (20,30) 整个区间都锁了，如果数据量大可能导致大范围阻塞。"
