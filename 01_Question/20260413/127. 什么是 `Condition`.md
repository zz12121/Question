### 1.2 Condition 与 Object 等待队列对比

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                    synchronized + Object.wait()  vs  Lock + Condition          │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                 synchronized + Object.wait() (内置锁)                    │   │
│  │                                                                          │   │
│  │   Object obj = new Object();                                            │   │
│  │                                                                          │   │
│  │   synchronized(obj) {          // 1. 获取锁                              │   │
│  │       while (condition != 期望) {                                       │   │
│  │           obj.wait();          // 2. 释放锁，进入 obj 的等待队列         │   │
│  │       }                                                                │   │
│  │       // 3. 条件满足，处理                                              │   │
│  │   }                                      // 4. 释放锁                    │   │
│  │                                                                          │   │
│  │   synchronized(obj) {                                                  │   │
│  │       condition = 满足;                                                │   │
│  │       obj.notify();             // 随机唤醒一个等待线程                   │   │
│  │       // 或 obj.notifyAll()     // 唤醒所有等待线程                       │   │
│  │   }                                      // 必须持有锁才能 notify        │   │
│  │                                                                          │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                    ReentrantLock + Condition (显式锁)                     │   │
│  │                                                                          │   │
│  │   ReentrantLock lock = new ReentrantLock();                              │   │
│  │   Condition notFull = lock.newCondition();   // 生产者条件                │   │
│  │   Condition notEmpty = lock.newCondition();   // 消费者条件                │   │
│  │                                                                          │   │
│  │   lock.lock();                         // 1. 获取锁                        │   │
│  │   try {                                                                    │   │
│  │       while (队列已满) {                                                   │   │
│  │           notFull.await();          // 2. 释放锁，进入 notFull 队列        │   │
│  │       }                                                                    │   │
│  │       // 入队                                                              │   │
│  │       notEmpty.signal();           // 3. 唤醒 notEmpty 上的消费者          │   │
│  │   } finally {                                                                    │   │
│  │       lock.unlock();                   // 4. 释放锁                        │   │
│  │   }                                                                            │   │
│  │                                                                          │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 1.3 核心代码示例

#### 1. Condition 基础用法

```java
import java.util.concurrent.locks.*;

public class ConditionDemo {
    
    private final ReentrantLock lock = new ReentrantLock();
    
    // 创建两个 Condition
    private final Condition notFull = lock.newCondition();   // 队列不满
    private final Condition notEmpty = lock.newCondition();  // 队列不空
    
    private final Object[] items = new Object[100];
    private int count = 0;
    
    // ====== 生产者：put() ======
    public void put(Object x) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length) {
                // 队列满了，等待消费者消费
                notFull.await();  // 释放锁，进入 notFull 等待队列
            }
            items[count++] = x;
            System.out.println("生产: " + x + ", 当前数量: " + count);
            // 唤醒等待的消费者
            notEmpty.signal();  // 只唤醒 notEmpty 队列中的线程
        } finally {
            lock.unlock();
        }
    }
    
    // ====== 消费者：take() ======
    public Object take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) {
                // 队列空了，等待生产者生产
                notEmpty.await();  // 释放锁，进入 notEmpty 等待队列
            }
            Object x = items[--count];
            System.out.println("消费: " + x + ", 当前数量: " + count);
            // 唤醒等待的生产者
            notFull.signal();  // 只唤醒 notFull 队列中的线程
            return x;
        } finally {
            lock.unlock();
        }
    }
}
```

#### 2. 有界阻塞队列完整实现

```java
import java.util.concurrent.locks.*;
import java.util.*;

public class BoundedBlockingQueue<E> {
    
    private final int capacity;
    private final Object[] items;
    
    private int count = 0;          // 当前元素数量
    private int putIndex = 0;      // 下一个写入位置
    private int takeIndex = 0;     // 下一个读取位置
    
    private final ReentrantLock lock = new ReentrantLock();
    
    // 两个独立条件
    private final Condition notFull = lock.newCondition();    // not full condition
    private final Condition notEmpty = lock.newCondition();   // not empty condition
    
    public BoundedBlockingQueue(int capacity) {
        this.capacity = capacity;
        this.items = new Object[capacity];
    }
    
    /**
     * 阻塞直到队列不满
     */
    public void put(E e) throws InterruptedException {
        if (e == null) throw new NullPointerException();
        
        lock.lockInterruptibly();  // 可中断获取锁
        try {
            // 使用 while 而非 if（防止虚假唤醒）
            while (count == capacity) {
                notFull.await();  // 等待队列不满
            }
            
            items[putIndex] = e;
            putIndex = (putIndex + 1) % capacity;  // 环形队列
            count++;
            
            // 唤醒一个等待获取的线程
            notEmpty.signal();
            
        } finally {
            lock.unlock();
        }
    }
    
    /**
     * 阻塞直到队列不空
     */
    public E take() throws InterruptedException {
        lock.lockInterruptibly();
        try {
            while (count == 0) {
                notEmpty.await();  // 等待队列不空
            }
            
            @SuppressWarnings("unchecked")
            E e = (E) items[takeIndex];
            items[takeIndex] = null;  // 帮助 GC
            takeIndex = (takeIndex + 1) % capacity;
            count--;
            
            // 唤醒一个等待放入的线程
            notFull.signal();
            
            return e;
        } finally {
            lock.unlock();
        }
    }
    
    /**
     * 带超时获取
     */
    public E poll(long timeout, TimeUnit unit) throws InterruptedException {
        long nanos = unit.toNanos(timeout);
        
        lock.lockInterruptibly();
        try {
            while (count == 0) {
                if (nanos <= 0) {
                    return null;  // 超时
                }
                // 等待指定时间，并返回剩余时间
                nanos = notEmpty.awaitNanos(nanos);
            }
            
            @SuppressWarnings("unchecked")
            E e = (E) items[takeIndex];
            items[takeIndex] = null;
            takeIndex = (takeIndex + 1) % capacity;
            count--;
            
            notFull.signal();
            return e;
        } finally {
            lock.unlock();
        }
    }
    
    /**
     * 非阻塞获取
     */
    public E poll() {
        lock.lock();
        try {
            if (count == 0) {
                return null;
            }
            @SuppressWarnings("unchecked")
            E e = (E) items[takeIndex];
            items[takeIndex] = null;
            takeIndex = (takeIndex + 1) % capacity;
            count--;
            notFull.signal();
            return e;
        } finally {
            lock.unlock();
        }
    }
    
    /**
     * 返回队列大小
     */
    public int size() {
        lock.lock();
        try {
            return count;
        } finally {
            lock.unlock();
        }
    }
}
```

#### 3. 生产者-消费者测试

```java
public class ProducerConsumerTest {
    
    public static void main(String[] args) {
        BoundedBlockingQueue<Integer> queue = new BoundedBlockingQueue<>(5);
        
        // 生产者线程
        Thread producer = new Thread(() -> {
            for (int i = 0; i < 20; i++) {
                try {
                    queue.put(i);
                    Thread.sleep(100);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            }
        }, "Producer");
        
        // 消费者线程
        Thread consumer = new Thread(() -> {
            while (true) {
                try {
                    Integer item = queue.take();
                    System.out.println("消费: " + item);
                    Thread.sleep(200);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    break;
                }
            }
        }, "Consumer");
        
        producer.start();
        consumer.start();
        
        try {
            producer.join();
            Thread.sleep(5000);
            consumer.interrupt();
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }
}

### 2.1 🔥 追问 1：Condition 和 Object.wait/notify() 的核心区别？

> **A:**

**全面对比表：**

| 维度 | Condition | Object.wait/notify |
|------|-----------|-------------------|
| **所属** | `Lock.newCondition()` 创建 | `Object` 类内置方法 |
| **需要锁** | 必须先获取显式 Lock | 必须先获取 synchronized 锁 |
| **等待队列** | 每个 Condition 独立队列 | 只有一个隐式等待队列 |
| **唤醒方式** | signal() 精确唤醒指定 Condition | notify() 随机唤醒一个 |
| **唤醒个数** | signal() / signalAll() | notify() / notifyAll() |
| **支持超时** | await(long time) / awaitNanos() | wait(long timeout) |
| **支持中断** | awaitInterruptibly() | wait() 会抛 InterruptedException |
| **公平性** | 可配合公平锁/非公平锁 | 无法控制公平性 |
| **调试** | 可关联同步器对象 | 无调试信息 |

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                        等待队列数量对比                                         │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  Object（单一等待队列）：                                                          │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                                                                          │   │
│  │   Object queue                                                              │   │
│  │   ┌──────────────────────────────────────────────────────────┐           │   │
│  │   │  Thread-1 (等条件A)                                     │           │   │
│  │   │  Thread-2 (等条件B)  ← 唤醒时无法区分！                   │           │   │
│  │   │  Thread-3 (等条件C)                                     │           │   │
│  │   └──────────────────────────────────────────────────────────┘           │   │
│  │                                                                          │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
│  Condition（多等待队列）：                                                         │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                                                                          │   │
│  │   Condition notFull                    Condition notEmpty               │   │

### 2.2 🔥 追问 2：为什么 signal() 可能导致"假唤醒"？必须用 while 循环？

> **A:**

**虚假唤醒（Spurious Wakeup）问题：**

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                          虚假唤醒原理与防御                                       │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                      什么是虚假唤醒？                                     │   │
│  │                                                                          │   │
│  │  虚假唤醒是 pthread（JVM 底层依赖）的合法行为：                            │   │
│  │                                                                          │   │
│  │  - wait() 可能因为系统原因（非 notify）返回                               │   │
│  │  - 原因：信号中断、用户线程唤醒等                                          │   │
│  │  - JVM 不保证每次 await() 返回都是由 signal() 唤醒                        │   │
│  │                                                                          │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                        while vs if 的区别                                │   │
│  │                                                                          │   │
│  │  // ❌ 错误：用 if（可能导致队列越界）                                     │   │
│  │  if (count == capacity) {        // 被虚假唤醒时，这里判断为 false        │   │
│  │      notFull.await();                                                      │   │
│  │  }                                                                 │   │
│  │  queue.add(item);                  // ⚠️ 可能超出容量！                   │   │
│  │                                                                          │   │
│  │                                                                          │   │
│  │  // ✅ 正确：用 while（即使虚假唤醒也会重新检查）                           │   │
│  │  while (count == capacity) {      // 虚假唤醒后再次检查                    │   │
│  │      notFull.await();                                                          │   │
│  │  }                                                                 │   │
│  │  queue.add(item);                  // ✅ 安全                            │   │
│  │                                                                          │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘
```

**代码示例：**

```java
public class WhyWhileLoop {
    
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull = lock.newCondition();
    private int count = 0;
    private final int capacity = 10;
    
    public void put(Object item) throws InterruptedException {
        lock.lock();
        try {
            // ⚠️ 必须用 while，不能用 if！
            while (count == capacity) {
                notFull.await();  // 等待队列不满
            }
            
            // 此时队列一定不满（因为有 while 循环保证）
            items[count++] = item;
            notEmpty.signal();  // 唤醒消费者
            
        } finally {
            lock.unlock();
        }
    }
}
```

### 2.3 🔥 追问 3：Condition.await() 等待时会释放锁吗？

> **A:**

**是的，await() 会释放锁！**

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                          await() 的锁释放机制                                   │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                          await() 调用过程                                │   │
│  │                                                                          │   │
│  │   lock.lock();                    // 1. 获取锁                           │   │
│  │   try {                                                                   │   │
│  │       while (条件不满足) {                                                 │   │
│  │           // ⚠️ await() 执行时：                                         │   │
│  │           // 1. 将当前线程加入 Condition 等待队列                         │   │
│  │           // 2. 原子地释放持有的锁                                         │   │
│  │           // 3. 线程状态变为 WAITING                                       │   │
│  │           await();                                                        │   │
│  │           // 4. 被 signal() 唤醒后重新获取锁                               │   │
│  │           // 5. 等待重新获取锁成功后返回                                    │   │
│  │       }                                                                   │   │
│  │       // 业务逻辑                                                          │   │
│  │   } finally {                                                             │   │
│  │       lock.unlock();                  // 释放锁                          │   │
│  │   }                                                                       │   │
│  │                                                                          │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                        与 LockSupport.park() 对比                        │   │
│  │                                                                          │   │
│  │   LockSupport.park()  → 不会释放锁！                                      │   │
│  │   Condition.await()   → 会释放锁！                                        │   │
│  │                                                                          │   │
│  │   这是为什么 AQS 内部 park 前要先释放锁：                                  │   │
│  │                                                                          │   │
│  │   lock.lock();                                                           │   │
│  │   try {                                                                  │   │
│  │       if (!tryAcquire()) {                                               │   │
│  │           // park 前必须释放锁                                             │   │
│  │           release();                                                     │   │
│  │           LockSupport.park();                                           │   │
│  │           // 被唤醒后重新获取锁                                            │   │
│  │           acquire();                                                      │   │
│  │       }                                                                   │   │
│  │   } finally { lock.unlock(); }                                           │   │
│  │                                                                          │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 2.4 🔥 追问 4：ArrayBlockingQueue 和 LinkedBlockingQueue 有什么区别？

> **A:**

**JDK 实现对比：**

| 维度 | ArrayBlockingQueue | LinkedBlockingQueue |
|------|-------------------|---------------------|
| **存储结构** | 有界数组（Object[]） | 无界链表节点（Node） |
| **默认容量** | 必须指定容量 | Integer.MAX_VALUE（实际无界） |
| **内存分配** | 预分配，大小固定 | 按需分配，动态增长 |
| **吞吐量** | 较低（有界，固定大小） | 较高（可选 capacity） |
| **内存占用** | 固定，不额外开销 | 每个节点多一个 Node 对象 |
| **GC 压力** | 低（数组） | 高（大量 Node 对象） |
| **插入** | 需计算 index % capacity | 只需操作链表头尾 |
| **实现复杂度** | 较简单 | 较复杂（两把锁） |

**LinkedBlockingQueue 特点：**

```java
public class LinkedBlockingQueueDemo {
    
    // LinkedBlockingQueue 使用两把锁实现入队/出队并行
    private final ReentrantLock takeLock = new ReentrantLock();
    private final ReentrantLock putLock = new ReentrantLock();
    
    private final Condition notEmpty = takeLock.newCondition();
    private final Condition notFull = putLock.newCondition();
    
    // put() 使用 putLock
    public void put(E e) {
        if (e == null) throw new NullPointerException();
        int c = -1;
        Node<E> node = new Node<E>(e);
        
        putLock.lock();
        try {
            while (count == capacity) {
                notFull.await();  // 队列满等 notFull
            }
            enqueue(node);
            c = count.getAndIncrement();
            if (c + 1 < capacity) {
                notFull.signal();  // 还有空间，唤醒其他生产者
            }
        } finally {
            putLock.unlock();
        }
        if (c == 0) {
            // 只有 c==0 时才需要唤醒消费者（从空变非空）
            signalNotEmpty();
        }
    }
    
    // take() 使用 takeLock
    public E take() {
        E x;
        int c = -1;
        takeLock.lock();
        try {
            while (count == 0) {
                notEmpty.await();  // 队列空等 notEmpty
            }
            x = dequeue();
            c = count.getAndDecrement();
            if (c > 1) {
                notEmpty.signal();  // 还有元素，唤醒其他消费者
            }
        } finally {
            takeLock.unlock();
        }
        if (c == capacity) {
            // 只有 c==capacity 时才需要唤醒生产者（从满变非满）
            signalNotFull();
        }
    }
}
```

### 2.5 🔥 追问 5：如何在不使用 Lock 和 Condition 的情况下实现阻塞队列？

> **A:**

**无锁 + wait/notify 版本：**

```java
/**
 * 使用 synchronized + Object.wait/notify 实现（简化版）
 * 
 * 注意：这种方式不如 Lock + Condition 高效
 */
public class SimpleBlockingQueueSync<E> {
    
    private final int capacity;
    private final Object[] items;
    private int count = 0;
    private int putIndex = 0;
    private int takeIndex = 0;
    
    public SimpleBlockingQueueSync(int capacity) {
        this.capacity = capacity;
        this.items = new Object[capacity];
    }
    
    // ⚠️ synchronized 方式：只能有一个等待队列，无法区分生产者/消费者
    public synchronized void put(Object x) throws InterruptedException {
        while (count == capacity) {
            wait();  // ⚠️ 所有线程在同一个队列等待
        }
        items[putIndex] = x;
        putIndex = (putIndex + 1) % capacity;
        count++;
        notify();  // ⚠️ 随机唤醒，可能唤醒生产者而非消费者
    }
    
    public synchronized Object take() throws InterruptedException {
        while (count == 0) {
            wait();
        }
        Object x = items[takeIndex];
        items[takeIndex] = null;
        takeIndex = (takeIndex + 1) % capacity;
        count--;
        notify();  // 随机唤醒
        return x;
    }
}
```

## 3. 一句话总结

> **Condition 是 Lock 绑定的等待/通知机制，每个 Condition 都有独立的等待队列，可精确区分不同条件的等待线程（如生产者等 notFull、消费者等 notEmpty），避免了 synchronized + Object.wait() 的单一队列和惊群效应。实现有界阻塞队列时必须用 while 循环防止虚假唤醒，并在入队/出队后精确 signal 对应的 Condition。**

## 4. 面试加分话术

> "Condition 我之前用它实现过一个**多线程文件下载器**。
> 
> 需求是：下载任务放到队列里，多个消费者线程并发下载，但队列满时要阻塞生产者。当时用 `synchronized + wait/notify` 实现了一个版本，但线上发现当下载量大时，消费者经常收到 notify 但发现队列还是空的——就是 wait/notify 随机唤醒导致的。
> 
> 后来改成 `ReentrantLock + 两个 Condition`：
> ```java
> // notFull：生产者等这个
> // notEmpty：消费者等这个
> while (count == capacity) notFull.await();  // 队列满等
> while (count == 0) notEmpty.await();         // 队列空等
> ```
> 
> 改成之后，**假唤醒问题就消失了**，CPU 使用率也降下来了，因为不会无意义地唤醒一堆线程。
> 
> 关于 while 而非 if，这是 Java 并发编程的经典规范——**所有 wait/await 都必须放在 while 循环里**，因为虚假唤醒是 JLS/JVM 允许的行为，不在代码里防御就会出问题。"
