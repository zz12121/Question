### 1.2 Enum 类核心结构

```
java.lang.Object
        │
        └── java.lang.Enum<E extends Enum<E>>
                │
                ├── private final String name;        // 枚举名称
                ├── private final int ordinal;        // 枚举顺序（0,1,2...）
                ├── protected Enum(String name, int ordinal)
                ├── public final String name()
                ├── public final int ordinal()
                ├── public static T[] values()         // 所有枚举值
                ├── public static T valueOf(String)  // 按名称查找
                ├── protected final Object clone()   // 禁止克隆
                ├── private void readObject(ObjectInputStream)  // 禁止反序列化
                └── private void readObject0(ObjectInputStream) // 防止破坏单例
```

### 1.3 核心代码示例

#### 1. 枚举基础用法

```java
public enum Status {
    SUCCESS("200", "成功"),
    ERROR("500", "系统错误"),
    LOADING("0", "加载中");
    
    private final String code;
    private final String desc;
    
    // 私有构造方法
    Status(String code, String desc) {
        this.code = code;
        this.desc = desc;
    }
    
    public String getCode() { return code; }
    public String getDesc() { return desc; }
    
    // 根据 code 查找枚举
    public static Status fromCode(String code) {
        for (Status s : values()) {
            if (s.code.equals(code)) {
                return s;
            }
        }
        return null;
    }
}

// 使用
Status s = Status.SUCCESS;
System.out.println(s.name());    // SUCCESS
System.out.println(s.ordinal());  // 0
System.out.println(s.getCode()); // 200
System.out.println(s.getDesc()); // 成功
```

#### 2. 枚举实现接口

```java
/**
 * 定义接口
 */
public interface BusinessCode {
    int getCode();
    String getMessage();
}

/**
 * 枚举实现接口：订单状态
 */
public enum OrderStatus implements BusinessCode {
    PENDING(1001, "待支付") {
        @Override
        public String getMessage() {
            return "订单待支付，请在30分钟内完成";
        }
    },
    PAID(1002, "已支付") {
        @Override
        public String getMessage() {
            return "支付成功，订单正在处理中";
        }
    },
    SHIPPED(1003, "已发货") {
        @Override
        public String getMessage() {
            return "商品已发货，请注意查收";
        }
    },
    COMPLETED(1004, "已完成") {
        @Override
        public String getMessage() {
            return "交易完成，感谢您的购买";
        }
    };
    
    private final int code;
    private final String defaultMsg;
    
    OrderStatus(int code, String defaultMsg) {
        this.code = code;
        this.defaultMsg = defaultMsg;
    }
    
    @Override
    public int getCode() {
        return code;
    }
    
    // 每个枚举值可以覆写方法，实现差异化行为
    @Override
    public abstract String getMessage();
}

// 使用接口统一处理
public class ResultBuilder {
    public static String build(BusinessCode status) {
        return String.format("code=%d, message=%s", status.getCode(), status.getMessage());
    }
}

// 调用
ResultBuilder.build(OrderStatus.PAID);  // code=1002, message=支付成功...
```

#### 3. 枚举单例模式

```java
/**
 * 枚举单例：线程安全 + 防止反射/反序列化破坏
 */
public enum SingletonEnum {
    INSTANCE;
    
    private final Resource resource;
    
    SingletonEnum() {
        resource = new Resource();
    }
    
    public Resource getResource() {
        return resource;
    }
    
    // 单例的业务方法
    public void doSomething() {
        System.out.println("单例方法执行");
    }
}

// 使用
SingletonEnum singleton = SingletonEnum.INSTANCE;
singleton.doSomething();
```

### 2.1 🔥 追问 1：枚举为什么可以防止反射和反序列化破坏单例？

> **A:**

**五层防护机制：**

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                     枚举单例的五大安全保障                                         │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────────┐ │
│  │  防护1: 构造方法私有 + JVM 层面保证                                           │ │
│  │                                                                              │ │
│  │  Enum 的构造方法是私有的，且由 JVM 在类加载阶段直接创建枚举实例。              │ │
│  │  即使通过反射调用 setAccessible(true) 也会在第一行就抛异常。                  │ │
│  └─────────────────────────────────────────────────────────────────────────────┘ │
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────────┐ │
│  │  防护2: 反射防御（源码层面）                                                  │ │
│  │                                                                              │ │
│  │  Constructor<T> 的 newInstance() 方法第一行就检查：                        │ │
│  │                                                                              │ │
│  │  if ((clazz.getModifiers() & Modifier.ENUM) != 0)                          │ │
│  │      throw new IllegalArgumentException("Cannot reflect enums");            │ │
│  └─────────────────────────────────────────────────────────────────────────────┘ │
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────────┐ │
│  │  防护3: 反序列化防御（自动调用 valueOf）                                      │ │
│  │                                                                              │ │
│  │  序列化时只保存 name 字段                                                   │ │
│  │  反序列化时自动调用 valueOf(name)，返回 JVM 缓存的枚举实例                    │ │
│  │  不会创建新对象！                                                           │ │
│  │                                                                              │ │
│  │  // Enum 的 readResolve 方法                                                │ │
│  │  protected Object readResolve() throws ObjectStreamException {             │ │
│  │      return Enum.valueOf(declaringClass, name);  // 返回已有实例           │ │
│  │  }                                                                          │ │
│  └─────────────────────────────────────────────────────────────────────────────┘ │
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────────┐ │
│  │  防护4: JVM 类加载器的固有安全性                                              │ │
│  │                                                                              │ │
│  │  枚举类由 Bootstrap ClassLoader 加载，枚举实例在类的静态初始化期间创建。       │ │
│  │  这由 JVM 保证是线程安全的，且在应用程序的整个生命周期内只有一个实例。         │ │
│  └─────────────────────────────────────────────────────────────────────────────┘ │
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────────┐ │
│  │  防护5: 编译期检查                                                           │ │
│  │                                                                              │ │
│  │  javac 编译器会检查枚举类型，确保 switch 语句穷举所有枚举值。                  │ │
│  │  后续添加新枚举值会触发编译错误，强制开发者更新所有相关代码。                 │ │
│  └─────────────────────────────────────────────────────────────────────────────┘ │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘
```

**源码验证：**

```java
// java.lang.reflect.Constructor#newInstance()
public T newInstance(Object... initargs) throws InstantiationException, ... {
    // 第一行就检查是否是枚举类型
    if ((clazz.getModifiers() & Modifier.ENUM) != 0)
        throw new IllegalArgumentException("Cannot reflect enums.");
    // ...后面的代码不会执行到
}

// java.lang.Enum#readResolve() - 反序列化保护
protected Object readResolve() throws ObjectStreamException {
    // 反序列化时自动调用，返回 JVM 缓存的枚举实例
    return Enum.valueOf(declaringClass, name);
}
```

### 2.2 🔥 追问 2：枚举 vs 双重检查锁单例，对比分析

> **A:**

**全面对比表格：**

| 维度 | 枚举单例 | 双重检查锁（DCL）单例 |
|------|---------|---------------------|
| **线程安全** | ✅ JVM 固有保证 | ✅ volatile + synchronized |
| **延迟加载** | ❌ 否（类加载即创建） | ✅ 是（第一次调用时创建） |
| **反射防御** | ✅ 天然防御 | ❌ 需额外代码防护 |
| **反序列化防御** | ✅ 天然防御 | ❌ 需额外代码防护 |
| **实现复杂度** | ⭐⭐⭐ 极简 | ⭐⭐⭐⭐⭐ 复杂 |
| **可读性** | ⭐⭐⭐⭐⭐ 一目了然 | ⭐⭐⭐ 需要理解 volatile/同步语义 |
| **能否实现懒加载** | ❌ 不能 | ✅ 可以 |
| **是否需要额外导入** | ❌ 无需 | ❌ 无需 |
| **是否需要序列化管理** | ❌ 无需 | ❌ 需要 readResolve |
| **创建时机** | 类加载时（类初始化阶段） | 首次使用时（主动调用时） |
| **能否被破坏** | ❌ 不能 | ⚠️ 反射可以 |

**双重检查锁完整实现（对比）：**

```java
public class SingletonDCL {
    
    // volatile 防止指令重排
    // instance = new SingletonDCL() 分为三步：
    // 1. 分配内存
    // 2. 调用构造方法
    // 3. 将引用赋值给 instance
    // 没有 volatile，步骤 3 可能被重排到步骤 2 之前，导致返回半初始化的对象
    private volatile static SingletonDCL instance;
    
    // 私有构造
    private SingletonDCL() {
        // 防御反射：第二次调用时抛异常
        if (instance != null) {
            throw new IllegalStateException("Already initialized");
        }
    }
    
    public static SingletonDCL getInstance() {
        if (instance == null) {  // 第一次检查：避免不必要的同步
            synchronized (SingletonDCL.class) {  // 加锁
                if (instance == null) {  // 第二次检查
                    instance = new SingletonDCL();
                }
            }
        }
        return instance;
    }
    
    // 防御反序列化：必须添加
    private Object readResolve() {
        return instance;  // 返回已有实例
    }
}
```

**枚举单例实现（对比）：**

```java
public enum SingletonEnum {
    INSTANCE;  // 就这一行！
    
    private final Resource resource;
    
    SingletonEnum() {
        resource = new Resource();
    }
    
    public Resource getResource() {
        return resource;
    }
}
// 无需任何额外代码！天然线程安全、天然防止反射和反序列化破坏
```

### 2.3 🔥 追问 3：枚举的 ordinal() 和 name() 有什么区别？有什么坑？

> **A:**

**对比分析：**

| 方法 | 返回值 | 稳定性 | 用途 |
|-----|-------|-------|-----|
| `name()` | 枚举声明时的名字（String） | ✅ 稳定 | 持久化、序列化、日志 |
| `ordinal()` | 枚举的顺序位置（int） | ⚠️ 不稳定 | 仅用于 EnumSet/EnumMap 内部 |

**隐藏的坑：**

```java
public enum Season {
    SPRING, SUMMER, AUTUMN, WINTER;
}

// ✅ name() - 稳定
Season.SPRING.name();  // "SPRING"

// ⚠️ ordinal() - 危险！
Season.SPRING.ordinal();  // 0
Season.WINTER.ordinal();   // 3

// ❌ 坑1: 新增枚举值会改变其他枚举的 ordinal
enum Season {
    SPRING, SUMMER, AUTUMN, WINTER, RAINY;  // 如果在中间插入
}
// SPRING.ordinal() 仍然是 0
// 但 AUTUMN 变成了 2 而不是原来的 2

// ❌ 坑2: 数据库存储 ordinal 导致数据错乱
// 场景：数据库存的是 ordinal=2，但你改了枚举顺序
// 之前 ordinal=2 是 AUTUMN，改后 ordinal=2 是 WINTER！
```

**正确做法：使用自定义字段而不是 ordinal：**

```java
public enum Season {
    SPRING(1, "春天"),
    SUMMER(2, "夏天"),
    AUTUMN(3, "秋天"),
    WINTER(4, "冬天");
    
    private final int code;    // 稳定的业务编码
    private final String desc;
    
    Season(int code, String desc) {
        this.code = code;
        this.desc = desc;
    }
    
    public int getCode() { return code; }
    public String getDesc() { return desc; }
    
    // 根据 code 查找（稳定的映射关系）
    public static Season fromCode(int code) {
        for (Season s : values()) {
            if (s.code == code) return s;
        }
        throw new IllegalArgumentException("Invalid code: " + code);
    }
}

// 数据库存储 code，不存储 ordinal
@EnumMapping(nameProperty = "code")
private Season season;
```

### 2.4 🔥 追问 4：枚举在数据库中的最佳实践

> **A:**

**三种方案对比：**

| 方案 | 存储值 | 优点 | 缺点 |
|-----|-------|------|------|
| **存 ordinal** | 整数（0,1,2...） | 存储小、查询快 | 枚举顺序变化会数据错乱 ❌ |
| **存 name** | 字符串（SPRING...） | 可读性好 | 存储大、改名风险 |
| **存业务 code** | 自定义整数/字符串 | 稳定、解耦 | 需要额外映射代码 ✅ |

**推荐方案：存储业务 code + MyBatis 枚举类型处理器：**

```java
// 1. 枚举定义
public enum OrderStatus implements BaseEnum<Integer> {
    PENDING(1001, "待支付"),
    PAID(1002, "已支付"),
    SHIPPED(1003, "已发货");
    
    private final Integer code;
    private final String desc;
    
    OrderStatus(Integer code, String desc) {
        this.code = code;
        this.desc = desc;
    }
    
    @Override
    public Integer getCode() { return code; }
    public String getDesc() { return desc; }
    
    @Override
    public Object deserialize(Integer code) {
        return fromCode(code);
    }
    
    public static OrderStatus fromCode(Integer code) {
        for (OrderStatus s : values()) {
            if (s.code.equals(code)) return s;
        }
        return null;
    }
}

// 2. MyBatis 类型处理器
@MappedTypes(OrderStatus.class)
public class OrderStatusTypeHandler extends BaseTypeHandler<OrderStatus> {
    
    @Override
    public void setNonNullParameter(PreparedStatement ps, int i, 
                                     OrderStatus parameter, JdbcType jdbcType) 
                                     throws SQLException {
        ps.setInt(i, parameter.getCode());  // 存 code
    }
    
    @Override
    public OrderStatus getNullableResult(ResultSet rs, String columnName) 
                                         throws SQLException {
        return OrderStatus.fromCode(rs.getInt(columnName));
    }
    
    @Override
    public OrderStatus getNullableResult(ResultSet rs, int columnIndex) 
                                         throws SQLException {
        return OrderStatus.fromCode(rs.getInt(columnIndex));
    }
    
    @Override
    public OrderStatus getNullableResult(CallableStatement cs, int columnIndex) 
                                         throws SQLException {
        return OrderStatus.fromCode(cs.getInt(columnIndex));
    }
}
```

### 2.5 🔥 追问 5：枚举在 switch 中的用法和注意事项

> **A:**

**标准用法：**

```java
public class SwitchEnumExample {
    
    public String handleStatus(OrderStatus status) {
        switch (status) {
            case PENDING:
                return "待处理";
            case PAID:
                return "已支付";
            case SHIPPED:
                return "已发货";
            case COMPLETED:
                return "已完成";
            default:
                throw new IllegalArgumentException("未知状态: " + status);
        }
    }
}
```

**⚠️ 注意事项：**

```java
// ❌ 坑1: switch 使用 ordinal 而非 name
switch (status.ordinal()) {  // 不推荐！ordinal 可能变化
    case 0: // SPRING?
    case 1: // SUMMER?
}

// ✅ 推荐：使用 name()
switch (status.name()) {
    case "PENDING":
}

// ✅ 最佳：直接使用枚举值（编译器会自动检查穷举）
switch (status) {
    // 如果后续添加新枚举值但忘记更新 switch，编译时报错！
}


// ❌ 坑2: 忘记 default 分支
// 添加新枚举值后，旧代码可能导致 NullPointerException

// ✅ 推荐：使用 JDK 12+ 的 switch 表达式
public String handleStatus(OrderStatus status) {
    return switch (status) {
        case PENDING -> "待处理";
        case PAID -> "已支付";
        case SHIPPED -> "已发货";
        case COMPLETED -> "已完成";
        // 编译器强制要求穷举所有枚举值
    };
}
```

## 3. 一句话总结

> **Java 枚举本质是一个继承 `java.lang.Enum` 的 final 类，每个枚举值是 static final 字段。枚举可以实现接口来定义不同枚举值的差异化行为。枚举单例天然线程安全且防止反射/反序列化破坏，比双重检查锁单例更简洁可靠，但无法懒加载。实际开发中应避免使用 `ordinal()`，而是用自定义业务 code 来确保稳定性。**

## 4. 面试加分话术

> "枚举是我日常写业务代码用得很多的特性。
> 
> 举几个实际场景：订单状态、支付状态、用户角色这些，**必须用枚举**，不用枚举的后果就是满天飞的魔法值（0、1、2），后期根本没法维护。
> 
> 关于枚举单例，我之前重构过一个配置中心客户端，原来用的是 DCL 单例，代码写得很长还要处理反射和反序列化。后来改成枚举单例，代码从 50 行压缩到 10 行，**既简洁又安全**。
> 
> 但枚举单例有个局限：**不能懒加载**。如果单例创建很耗时且不一定会被用到，就得用 DCL 或者**静态内部类单例**（也是 JVM 固有安全的）。这是两者的核心区别。
> 
> 还有个小技巧：**枚举可以实现接口**，这样可以用同一个接口引用不同的枚举实现，比如不同支付渠道用不同的策略类，代码会更清晰。"
