### 1.2 park/unpark 原理（UNSAFE 层面）

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                           park/unpark 底层原理                                    │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                    每个线程都有一个"许可"(permit)                        │   │
│  │                                                                          │   │
│  │       Thread A          Thread B          Thread C                      │   │
│  │    ┌──────────┐      ┌──────────┐      ┌──────────┐                   │   │
│  │    │ permit: 1│      │ permit: 0│      │ permit: 0│                   │   │
│  │    │   ✅可用  │      │   ⛔阻塞  │      │   ⛔阻塞  │                   │   │
│  │    └──────────┘      └──────────┘      └──────────┘                   │   │
│  │                                                                          │   │
│  │   permit 是 0 或 1 的boolean标记，类比"通行证"                           │   │
│  │                                                                          │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                         park() 行为                                        │   │
│  │                                                                          │   │
│  │   park() {                                                               │   │
│  │       if (permit available) {      // 有许可？                           │   │
│  │           permit--;                  // 消费许可                          │   │
│  │           return;                    // 立即返回，不阻塞                  │   │
│  │       }                                                                    │   │
│  │       // 无许可，阻塞当前线程，等待 unpark 或 interrupt                   │   │
│  │       block();                                                            │   │
│  │   }                                                                       │   │
│  │                                                                          │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                        unpark() 行为                                      │   │
│  │                                                                          │   │
│  │   unpark(thread) {                                                       │   │
│  │       if (permit < 1) {              // 许可 < 1？                       │   │
│  │           permit++;                  // 发放许可                          │   │
│  │       }                                                                    │   │
│  │       // 唤醒指定线程（如果正在阻塞）                                      │   │
│  │       if (thread is blocked) {                                           │   │
│  │           wakeup(thread);                                                │   │
│  │       }                                                                   │   │
│  │   }                                                                       │   │
│  │                                                                          │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 1.3 核心代码示例

#### 1. LockSupport 基础用法

```java
import java.util.concurrent.locks.LockSupport;

public class LockSupportDemo {
    
    public static void main(String[] args) throws InterruptedException {
        
        Thread consumer = new Thread(() -> {
            System.out.println("消费者线程启动，等待产品...");
            
            // 阻塞，等待生产者
            LockSupport.park();  
            
            System.out.println("消费者被唤醒，继续执行！");
        }, "Consumer");
        
        consumer.start();
        Thread.sleep(1000);
        
        // 主线程唤醒消费者
        System.out.println("主线程唤醒消费者...");
        LockSupport.unpark(consumer);
        
        consumer.join();
    }
}

// ====== 带 blocker 的 park（便于调试）======
class BlockerDemo {
    
    private final Object blocker = new Object();
    
    public void process() {
        System.out.println("开始处理...");
        
        // blocker 用于调试和诊断（JVM 工具能看到阻塞原因）
        LockSupport.park(blocker);
        
        System.out.println("被唤醒");
    }
    
    public void wakeup() {
        LockSupport.unpark(waitingThread);
    }
}
```

#### 2. park/unpark 实现简单的阻塞队列

```java
import java.util.concurrent.locks.LockSupport;

public class SimpleBlockingQueue<T> {
    
    private Node<T> head;
    private Node<T> tail;
    
    // 等待队列
    private Thread waitingTaker;
    
    private static class Node<T> {
        T value;
        Node<T> next;
    }
    
    public T take() throws InterruptedException {
        while (head == null) {
            // 等待前先设置 blocker（用于调试）
            LockSupport.park(this);
            
            // 被中断时退出
            if (Thread.interrupted()) {
                throw new InterruptedException();
            }
        }
        
        T value = head.value;
        head = head.next;
        return value;
    }
    
    public void put(T value) {
        Node<T> node = new Node<>();
        node.value = value;
        
        if (tail == null) {
            head = tail = node;
            // 唤醒等待的消费者
            if (waitingTaker != null) {
                LockSupport.unpark(waitingTaker);
            }
        } else {
            tail.next = node;
            tail = node;
        }
    }
}
```

#### 3. park/unpark vs wait/notify 对比

```java
public class ComparisonDemo {
    
    // ====== wait/notify 方式 ======
    public synchronized void waitStyle() throws InterruptedException {
        while (conditionNotMet) {
            wait();  // 必须持有锁才能调用
        }
        // 处理逻辑
    }
    
    public synchronized void notifyStyle() {
        conditionMet = true;
        notify();  // 随机唤醒一个 wait 的线程
        // 或者 notifyAll() 唤醒所有
    }
    
    // ====== park/unpark 方式 ======
    public void parkStyle() {
        while (conditionNotMet) {
            LockSupport.park();  // 不需要持有锁！
        }
        // 处理逻辑
    }
    
    public void unparkStyle() {
        conditionMet = true;
        LockSupport.unpark(targetThread);  // 精确唤醒指定线程
    }
}
```

### 2.1 🔥 追问 1：park/unpark 和 wait/notify 的核心区别？

> **A:**

**全面对比：**

| 维度 | LockSupport.park/unpark | Object.wait/notify |
|------|------------------------|-------------------|
| **需要持有锁** | ❌ 不需要 | ✅ 必须持有对象锁 |
| **调用位置** | 任意位置 | 必须在 synchronized 内 |
| **唤醒方式** | 精确唤醒指定线程 | 随机唤醒一个 or 唤醒所有 |
| **中断响应** | ✅ 支持，`park()` 立即返回 | ⚠️ 需要处理 `InterruptedException` |
| **permit 机制** | ✅ 有（0/1），可先 unpark 再 park | ❌ 无 |
| **死锁风险** | ⚠️ 较低（permit 可累积） | ⚠️ 较高（notify 可能唤醒错误线程） |
| **调试友好** | ✅ park(Object) 可传 blocker | ❌ 无 blocker 信息 |
| **底层实现** | UNSAFE.park/unpark (C++) | native 方法，Monitor |
| **属于** | JUC (java.util.concurrent) | Object 类 |
| **推荐场景** | AQS、线程池、无锁队列 | 简单 synchronized 场景 |

**为什么选择 LockSupport 而不是 wait/notify：**

| 原因 | 说明 |
|-----|------|
| **1. 无需获取锁** | AQS 的 CLH 队列线程在 park 前可能没持有锁，wait/notify 无法使用 |
| **2. 精确唤醒** | unpark(Thread) 可以精确唤醒队列中的指定节点，避免惊群效应 |
| **3. permit 机制** | 支持先 unpark 再 park，避免通知丢失 |
| **4. 中断语义清晰** | park 被中断时直接返回，调用方自行判断中断状态 |
| **5. 调试友好** | park(blocker) 可以传入同步器对象，便于问题排查 |

**AQS 唤醒流程：**

```java
// AQS.unparkSuccessor() 源码
private void unparkSuccessor(Node node) {
    int ws = node.waitStatus;
    if (ws < 0) {
        // CAS 修改状态
        compareAndSetWaitStatus(node, ws, 0);
    }
    
    Node s = node.next;  // 获取后继节点
    if (s == null || s.waitStatus > 0) {
        // 找到第一个非取消的节点
        s = null;
        for (Node t = tail; t != null && t != node; t = t.prev) {
            if (t.waitStatus <= 0) {
                s = t;
            }
        }
    }
    
    if (s != null) {
        // ⚠️ 精确唤醒指定线程
        LockSupport.unpark(s.thread);
    }
}
```

```java
public class BlockerUsage {
    
    private final ReentrantLock lock = new ReentrantLock();
    
    public void doSomething() {
        lock.lock();
        try {
            while (conditionNotMet) {
                // ✅ 正确：传入 this 或同步器对象
                LockSupport.park(this);
                
                // ✅ 也可传入具体的同步器
                // LockSupport.park(lock);
            }
        } finally {
            lock.unlock();
        }
    }
}
```

**交替打印奇偶实现：**

```java
public class PrintOddEven {
    
    private static int count = 1;
    private static final int MAX = 100;
    private static Thread oddThread;
    private static Thread evenThread;
    
    public static void main(String[] args) {
        
        oddThread = new Thread(() -> {
            while (count <= MAX) {
                if (count % 2 == 1) {
                    System.out.println("奇数: " + count);
                    count++;
                    LockSupport.unpark(evenThread);  // 唤醒偶数线程
                    LockSupport.park();              // 自己阻塞
                } else {
                    LockSupport.park();  // 等待偶数线程
                }
            }
        }, "Odd");
        
        evenThread = new Thread(() -> {
            while (count <= MAX) {
                if (count % 2 == 0) {
                    System.out.println("偶数: " + count);
                    count++;
                    LockSupport.unpark(oddThread);  // 唤醒奇数线程
                    LockSupport.park();              // 自己阻塞
                } else {
                    LockSupport.park();  // 等待奇数线程
                }
            }
        }, "Even");
        
        oddThread.start();
        evenThread.start();
    }
}
```

## 3. 一句话总结

> **LockSupport 是 JUC 底层的线程阻塞原语，通过 permit 机制（0/1）控制线程阻塞与唤醒。与 wait/notify 的核心区别是不需要持有锁、支持精确唤醒、支持先 unpark 再 park（permit 累积）、中断语义清晰。AQS 使用 LockSupport 是因为队列节点线程在 park 前可能未持有锁，且需要精确唤醒队列中的指定节点。**

## 4. 面试加分话术

> "LockSupport 我之前用来解决过一个线程池的假死问题。
> 
> 场景是这样的：线上反馈有个定时任务线程池里的任务偶尔会卡死，jstack 看到所有线程都处于 `park` 状态，但没有死锁迹象。后来加了 `park(Object blocker)` 参数后，再出问题 jstack 能直接看到是哪个同步器导致的——原来是业务逻辑里的 `Condition.await()` 一直不 signal。
> 
> 关于 AQS 为什么用 LockSupport 而不是 wait/notify，我的理解是：**AQS 的 CLH 队列里，线程在入队之前就已经开始尝试获取锁了，这时候线程可能还没持有 AQS 的锁，根本没法调 wait()。** 而且 wait/notify 是随机唤醒，AQS 需要精确唤醒队列里的后继节点。
> 
> permit 机制是个很巧妙的设计。之前有个坑：**生产者先生产了一个数据但还没调用 signal()，消费者就已经 wait() 了**。用 wait/notify 会导致通知丢失，但 park/unpark 的 permit 机制保证只要之前有 unpark，park 就不会阻塞。
