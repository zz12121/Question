### 三种框架代码对比

```java
// ========== ASM（底层，最难）==========
// ASM 需要手动拼接字节码指令
public class AsmExample {
    public static void main(String[] args) throws Exception {
        ClassWriter cw = new ClassWriter(ClassWriter.COMPUTE_FRAMES);
        cw.visit(Opcodes.V1_8, Opcodes.ACC_PUBLIC, "Example", null,
                 "java/lang/Object", null);
        
        // 新增方法
        MethodVisitor mv = cw.visitMethod(
            Opcodes.ACC_PUBLIC, "sayHello", "()V", null, null);
        mv.visitCode();
        mv.visitFieldInsn(Opcodes.GETSTATIC, "java/lang/System", 
                          "out", "Ljava/io/PrintStream;");
        mv.visitLdcInsn("Hello ASM");
        mv.visitMethodInsn(Opcodes.INVOKEVIRTUAL, "java/io/PrintStream",
                            "println", "(Ljava/lang/String;)V", false);
        mv.visitInsn(Opcodes.RETURN);
        mv.visitMaxs(0, 0);
        mv.visitEnd();
        
        byte[] bytecode = cw.toByteArray();
    }
}

// ========== Javassist（源码风格）==========
public class JavassistExample {
    public static void main(String[] args) throws Exception {
        ClassPool pool = ClassPool.getDefault();
        CtClass clazz = pool.makeClass("com.example.JavassistExample");
        
        // 新增方法，源码风格
        CtMethod sayHello = CtNewMethod.make(
            "public void sayHello() { System.out.println(\"Hello Javassist\"); }",
            clazz);
        clazz.addMethod(sayHello);
        
        byte[] bytecode = clazz.toBytecode();
    }
}

// ========== ByteBuddy（DSL，最优雅）==========
public class ByteBuddyExample {
    public static void main(String[] args) throws Exception {
        Class<?> dynamicType = new ByteBuddy()
            .subclass(Object.class)
            .name("com.example.DynamicClass")
            .method(named("sayHello"))
                .intercept(MethodDelegation.to(new GreetingInterceptor()))
            .make()
            .load(getClass().getClassLoader())
            .getLoaded();
        
        // 使用动态生成的类
        dynamicType.getMethod("sayHello").invoke(dynamicType.newInstance());
    }
    
    static class GreetingInterceptor {
        public String greet() {
            return "Hello ByteBuddy";
        }
    }
}
```

## 2. 扩展追问

### ## 🔥 扩展追问 1：Spring AOP 用的是哪种方式？

> **A:** Spring AOP **默认使用 JDK 动态代理**（`java.lang.reflect.Proxy`），如果目标类没有实现接口则使用 **CGLIB**（底层基于 ASM）。

```
┌─────────────────────────────────────────────────────────────────────┐
│                      Spring AOP 代理选择策略                          │
│                                                                     │
│                    目标类是否实现接口？                               │
│                          ↓                                          │
│              ┌───────────┴───────────┐                              │
│              ↓                       ↓                               │
│             是                       否                              │
│              ↓                       ↓                               │
│  ┌───────────────────┐     ┌───────────────────┐                    │
│  │   JDK 动态代理    │     │     CGLIB 代理     │                    │
│  │                   │     │                   │                    │
│  │ Proxy.newInstance │     │ 基于 ASM 字节码    │                    │
│  │ 生成 $Proxy 类    │     │ 生成子类           │                    │
│  └───────────────────┘     └───────────────────┘                    │
│                                                                     │
│  配置方式：                                                         │
│  • proxyTargetClass=true → 强制使用 CGLIB                            │
│  • <aop:aspectj-autoproxy proxy-target-class="true"/>               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

| 维度 | JDK 动态代理 | CGLIB |
|------|-------------|--------|
| **原理** | 实现接口，生成 `$Proxy` 实现类 | 继承目标类，生成子类 |
| **限制** | 必须有接口 | 无法代理 final 方法/类 |
| **性能** | 反射调用，略慢 | 快速调用 |
| **生成方式** | `java.lang.reflect.Proxy` | ASM |
| **Spring 默认** | 接口时使用 | 无接口或配置时使用 |

### ## 🔥 扩展追问 2：JDK 动态代理为什么必须有接口？

> **A:** JDK 动态代理**只能增强接口方法**，原因是它通过**接口约束**来生成代理类。

```java
// JDK 动态代理的限制
public class JdkProxyLimit {
    
    // ❌ 错误：UserService 没有实现任何接口
    // 原因：JDK 动态代理需要用接口来约束方法签名
    public class UserService {  // 没有 implements Xxx
        public void save() {}
    }
    
    // ✅ 正确：必须有接口
    public interface IUserService {
        void save();
    }
    
    public class UserServiceImpl implements IUserService {
        public void save() {}
    }
    
    // JDK 动态代理生成：
    // $Proxy extends Proxy implements IUserService
    //                            ↑
    //                     必须有接口！
}
```

```java
// JDK 动态代理完整示例
public class JdkProxyDemo {
    public static void main(String[] args) {
        IUserService target = new UserServiceImpl();
        
        // 生成代理对象
        IUserService proxy = (IUserService) Proxy.newProxyInstance(
            target.getClass().getClassLoader(),
            target.getClass().getInterfaces(),  // ← 必须传接口！
            (proxy, method, args) -> {
                System.out.println("前置逻辑");
                Object result = method.invoke(target, args);
                System.out.println("后置逻辑");
                return result;
            }
        );
        
        proxy.save();  // 调用走代理
    }
}
```

### ## 🔥 扩展追问 3：CGlib 生成的代理类是什么样的？

> **A:** CGLIB 通过**继承目标类**生成子类，重写目标方法，在方法前后添加增强逻辑。

```
┌─────────────────────────────────────────────────────────────────────┐
│                      CGLIB 代理类结构                                 │
│                                                                     │
│  原始类：                                                           │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │ public class UserService {                                     │  │
│  │     public void save() { /* 原始逻辑 */ }                      │  │
│  │ }                                                              │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                                                                     │
│  CGLIB 生成的子类：                                                 │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │ public class UserService$$EnhancerByCGLIB$$abc123              │  │
│  │     extends UserService {  // 继承                                │  │
│  │                                                             │  │
│  │     public void save() {                                     │  │
│  │         // 拦截器逻辑                                         │  │
│  │         MethodInterceptor.invoke(method);  // 调用父类原始方法   │  │
│  │     }                                                        │  │
│  │ }                                                              │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                                                                     │
│  ⚠️ 限制：final 方法无法被子类重写，代理失效                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

```java
// CGLIB 代理示例
public class CglibProxyDemo {
    public static void main(String[] args) {
        // 创建增强器
        Enhancer enhancer = new Enhancer();
        enhancer.setSuperclass(UserService.class);  // 设置父类
        enhancer.setCallback((MethodInterceptor) (obj, method, args, proxy) -> {
            System.out.println("前置逻辑");
            Object result = proxy.invokeSuper(obj, args);  // 调用父类方法
            System.out.println("后置逻辑");
            return result;
        });
        
        // 生成代理对象
        UserService proxy = (UserService) enhancer.create();
        proxy.save();
    }
}
```

### ## 🔥 扩展追问 4：ByteBuddy 相比其他框架有什么优势？

> **A:** ByteBuddy 的核心优势是**API 优雅**和**自动处理复杂情况**（桥接方法、泛型、注解等），而且性能接近原生 ASM。

```java
// ByteBuddy 的 DSL 风格示例
public class ByteBuddyAdvantage {
    public static void main(String[] args) throws Exception {
        
        // 1. 方法匹配 + 增强
        new ByteBuddy()
            .subclass(TargetClass.class)
            .method(named("save").and(returns(void.class)))
            .intercept(MethodDelegation.to(new SaveInterceptor()))
            .make()
            .load(getClass().getClassLoader())
            .getLoaded();
        
        // 2. 自动生成 constructor
        // 3. 自动生成 getter/setter
        // 4. 自动处理 @generated 注解
        // 5. 自动处理 @Inherited 注解
        
        // 6. 运行时重定义已加载的类
        new ByteBuddy()
            .redefine(AlreadyLoadedClass.class)
            .method(named("update"))
            .intercept(MethodDelegation.to(new UpdateInterceptor()))
            .make()
            .load(getClass().getClassLoader(), INJECTION);
    }
}
```

| 特性 | ByteBuddy | Javassist | ASM |
|------|-----------|-----------|-----|
| API 风格 | DSL 链式 | 源码风格 | 指令拼接 |
| 学习成本 | 低 | 中 | 高 |
| 运行时重定义类 | ✅ | ✅ | ✅ |
| 自动桥接方法 | ✅ | ❌ | ❌ |
| 泛型感知 | ✅ | 部分 | ❌ |
| 注解处理 | ✅ | ✅ | 手动 |
| 性能 | 高 | 中 | 最高 |

### ## 🔥 扩展追问 5：字节码增强在框架中的实际应用有哪些？

> **A:** 字节码增强是现代 Java 框架的**基础设施**，几乎无处不在。

```java
/**
 * 字节码增强的应用场景
 */

// 场景1：Spring 声明式事务
@Transactional
public void transfer() { /* 转账逻辑 */ }
// → Spring AOP 生成代理，方法前后开启/提交/回滚事务

// 场景2：MyBatis Mapper 接口
UserMapper mapper = sqlSession.getMapper(UserMapper.class);
mapper.selectById(1);
// → MyBatis 用 JDK 动态代理生成 Mapper 代理，执行 SQL

// 场景3：Hibernate 实体加载
@Entity
public class User { /* 实体类 */ }
User user = session.get(User.class, 1L);
// → Hibernate 用 Javassist/ASM 生成子类，
//   拦截 get/set 方法实现懒加载、脏检查

// 场景4：性能监控
@Monitor
public void heavyMethod() { /* 重操作 */ }
// → 框架自动统计方法调用时间、次数、异常率

// 场景5：Mock 测试
UserService mock = Mockito.mock(UserService.class);
when(mock.findById(1L)).thenReturn(user);
// → Mockito 用 ByteBuddy/ASM 生成 Mock 对象

// 场景6：全链路追踪
// SkyWalking、Pinpoint 等 APM 工具
// → 字节码插桩，追踪每次调用的耗时和调用链
```

## 3. 一句话总结

> **字节码增强是在类加载前或运行时修改字节码的技术，ASM 最底层最快但 API 复杂，Javassist 支持源码风格 API 易用性好，ByteBuddy 用 DSL 风格 API 最优雅且自动处理桥接方法等复杂情况；Spring AOP 默认对有接口的类用 JDK 动态代理，对无接口的类用基于 ASM 的 CGLIB 生成子类代理。**
