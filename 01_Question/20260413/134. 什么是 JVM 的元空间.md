# 134. 什么是 JVM 的元空间（Metaspace）溢出？

## 1. 基础流程

### Metaspace 的位置

```
┌─────────────────────────────────────────────────────────────────────┐
│                      JVM 内存布局对比                                │
│                                                                     │
│  JDK 7 及之前：                                                      │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │ Heap        │         PermGen (永久代)       │ Native Memory │   │
│  │ (Eden/Old)  │ (类信息/方法信息/字符串常量池)   │               │   │
│  └───────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  JDK 8+：                                                           │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │ Heap                 │         Metaspace (元空间)            │   │
│  │ (Eden/Old)           │  • 类结构（klass）                     │   │
│  │                      │  • 方法信息                             │   │
│  │                      │  • 静态变量 → 移到堆                   │   │
│  │                      │  • 运行时常量池 → 移到堆               │   │
│  └───────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  注意：Metaspace 使用本地内存（Native Memory），不受 -Xmx 限制！      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Metaspace 溢出的原理

```
┌─────────────────────────────────────────────────────────────────────┐
│                     Metaspace OOM 发生机制                           │
│                                                                     │
│  类加载流程：                                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  1. 类加载器加载 .class 文件                                  │   │
│  │  2. JVM 在 Metaspace 分配空间存储类元数据：                    │   │
│  │     • Klass 结构体（类名/父类/接口/字段/方法）                │   │
│  │     • 方法字节码                                              │   │
│  │     • 注解                                                    │   │
│  │     • 编译器生成的内部结构（泛型签名、匿名类信息）              │   │
│  │  3. 类卸载：只有当没有对象引用且没有类加载器引用时才卸载        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  OOM 触发：                                                         │
│  不断加载新类 → Metaspace 占用增长 → 超过 MaxMetaspaceSize → OOM   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Metaspace OOM 代码演示

```java
/**
 * Metaspace OOM 演示
 * 
 * 触发方式：不断动态生成类
 */
public class MetaspaceOOMDemo {
    
    public static void main(String[] args) {
        // 使用 ClassLoader 不断加载新类
        // 默认 MetaspaceSize 很小，约 21MB
        
        int count = 0;
        while (true) {
            // 每轮循环创建并加载一个新类
            // 类会被 Metaspace 永久持有
            generateClass();
            count++;
            System.out.println("已加载类数量: " + count);
        }
    }
    
    static void generateClass() throws Exception {
        // 动态生成类
        // 这里简化处理，实际用 ASM 或 Javassist 生成
    }
}

// 错误信息：
// java.lang.OutOfMemoryError: Metaspace
//   (而不是 java.lang.OutOfMemoryError: PermGen space)
```

### Metaspace 和 PermGen 的核心区别

| 维度 | PermGen（jdk 7 及之前） | Metaspace（jdk 8+） |
|------|------------------------|---------------------|
| **位置** | JVM 堆内存 | 本地内存（Native Memory） |
| **大小限制** | 受 -XX:MaxPermSize 限制 | 受 -XX:MaxMetaspaceSize 限制（默认无上限！） |
| **溢出处理** | 固定大小，容易 OOM | 可以动态扩展，也可以设置上限 |
| **字符串常量池** | 在 PermGen 中 | 移到堆内存（StringTable） |
| **静态变量** | 在 PermGen 中 | 移到堆内存 |
| **OOM 错误信息** | `PermGen space` | `Metaspace` |
| **GC 回收** | Full GC 时回收 | 类加载器卸载时回收 |

## 2. 扩展追问

### ## 🔥 扩展追问 1：哪些场景会导致 Metaspace OOM？

> **A:** Metaspace OOM 主要发生在**大量动态类生成**或**类加载器泄漏**的场景。

```java
/**
 * 常见导致 Metaspace OOM 的场景
 */
public class MetaspaceOOMScenarios {
    
    // 场景1：大量使用反射/动态代理
    // Spring、Hibernate 等框架大量创建代理类
    // 每次方法调用可能生成新的代理类
    public void springScenario() {
        // Spring AOP 默认使用 JDK 动态代理
        // 每个被代理的类都会生成一个 $Proxy 子类
        
        // 如果 Bean 配置了 CGLIB 代理，问题更严重
        // 每个类会生成两个子类：
        // • 代理子类
        // • EnhancerByCGLIB 子类
    }
    
    // 场景2：JSP 页面（老项目常见）
    // 每个 JSP 第一次访问时编译成 Servlet 类
    // 热部署时旧类不卸载，不断累积
    
    // 场景3：OSGi 容器
    // BundleClassLoader 动态加载/卸载类
    
    // 场景4：动态字节码生成
    // • ASM / Javassist 运行时生成类
    // • RPC 框架（Dubbo）生成服务接口实现类
    // • ORM 框架（Hibernate/MyBatis）生成实体类代理
    
    // 场景5：类加载器泄漏（最隐蔽！）
    // 自定义 ClassLoader 没有正确释放
    // ThreadLocal 持有类加载器引用
    // 静态集合持有类加载器引用
}
```

### ## 🔥 扩展追问 2：类加载器泄漏是怎么回事？

> **A:** 类加载器泄漏是 Metaspace OOM 的**最隐蔽原因**。即使 Metaspace 空间足够，如果类加载器无法被卸载，Metaspace 就会持续增长。

```
┌─────────────────────────────────────────────────────────────────────┐
│                     类加载器泄漏原理                                  │
│                                                                     │
│  类卸载条件（必须同时满足）：                                         │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  1. 堆中不存在该类加载器的实例                                 │   │
│  │  2. 堆中不存在该类加载器加载的任何类                            │   │
│  │  3. 该类加载器不再被其他存活对象引用                            │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  泄漏场景：                                                         │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  ThreadLocal<UserClassLoader> 持有引用                        │   │
│  │  ↓                                                          │   │
│  │  UserClassLoader 无法被卸载                                   │   │
│  │  ↓                                                          │   │
│  │  UserClassLoader 加载的所有类无法被卸载                        │   │
│  │  ↓                                                          │   │
│  │  Metaspace 持续增长 → OOM                                     │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

```java
// 常见的泄漏代码模式
public class ClassLoaderLeakExample {
    
    // ❌ 泄漏：ThreadLocal 持有类加载器
    private static ThreadLocal<ClassLoader> loaderHolder = 
        new ThreadLocal<>();
    
    // ❌ 泄漏：静态集合持有类加载器
    private static List<ClassLoader> loaders = new ArrayList<>();
    
    // ❌ 泄漏：静态集合持有 Class 对象
    private static List<Class<?>> loadedClasses = new ArrayList<>();
    
    // ✅ 正确：在 finally 中清理
    public void loadClass(String name) {
        try {
            ClassLoader loader = createClassLoader();
            loaderHolder.set(loader);
            // 加载类...
        } finally {
            loaderHolder.remove();  // 关键！
        }
    }
}
```

### ## 🔥 扩展追问 3：如何设置 Metaspace 大小？

> **A:** Metaspace 大小通过 JVM 参数控制，有初始值和最大值两个概念。

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `-XX:MetaspaceSize=N` | 初始 Metaspace 大小，触发第一次 GC 的阈值 | 约 21MB（JDK 11+） |
| `-XX:MaxMetaspaceSize=N` | 最大 Metaspace 大小，**默认无上限！** | 无限制 |
| `-XX:MinMetaspaceFreeRatio=N` | GC 后空闲 Metaspace 最小比例 | 40% |
| `-XX:MaxMetaspaceFreeRatio=N` | GC 后空闲 Metaspace 最大比例 | 70% |

```java
// 推荐的生产环境配置
// JVM 参数：
// -XX:MetaspaceSize=256m      // 初始大小
// -XX:MaxMetaspaceSize=512m    // 最大限制，防止无限增长
// -XX:+UseGCOverheadLimit     // 防止 GC 开销过大

/**
 * 配置思路：
 * 1. MetaspaceSize 设为较大值，减少扩容停顿
 * 2. MaxMetaspaceSize 必须设置，防止本地内存被耗尽
 * 3. 监控 Metaspace 使用量，调整上限
 */

// 查看 Metaspace 使用情况
// jstat -gc <pid>
// Metaspace:
```

```bash
# 监控 Metaspace
jstat -gc <pid> | grep Metaspace

# 输出示例：
# MCMN       MCMX        MC       CCSMN      CCSMX       CCSC     YGC   FGC
# 0.0    1234567.0    45678.0       0.0   1234567.0    5678.0      10     2

# MCMX = Metaspace max capacity（最大容量）
```

### ## 🔥 扩展追问 4：字符串常量池移到堆后有什么影响？

> **A:** JDK 8 把字符串常量池从 PermGen 移到了堆内存，这是个**好事**，带来了几个改变。

```
┌─────────────────────────────────────────────────────────────────────┐
│                字符串常量池迁移的影响                                 │
│                                                                     │
│  优点：                                                             │
│  1. 不再受 Metaspace 大小限制，字符串常量池可以弹性增长              │
│  2. 受 GC 管理，不会因为 Metaspace OOM 导致字符串常量池问题          │
│  3. PermGen 相关问题彻底解决                                        │
│                                                                     │
│  变化：                                                             │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  JDK 7 之前：                                                  │   │
│  │  String s = "hello";                                          │   │
│  │  → 先检查 intern() 方法，常量池在 PermGen                     │   │
│  │                                                             │   │
│  │  JDK 8+：                                                    │   │
│  │  String s = "hello";                                         │   │
│  │  → 先检查 StringTable（Hash表），在堆中                       │   │
│  │  → intern() 把字符串放入 StringTable                         │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

| 操作 | JDK 7 之前 | JDK 8+ |
|------|-----------|--------|
| String.intern() 存储位置 | PermGen | 堆（StringTable） |
| intern() OOM 风险 | 有（PermGen 满） | 无（堆空间充足） |
| 字符串常量占用内存 | PermGen | 堆 |

### ## 🔥 扩展追问 5：如何排查 Metaspace OOM？

> **A:** Metaspace OOM 的排查需要结合工具分析哪些类占用了 Metaspace。

```bash
# 1. 添加启动参数，保留 OOM 时的堆转储
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/dump.hprof

# 2. 使用 MAT 分析堆转储文件
# 重点关注：
# • Histogram → Group by Class Loader
# • Dominator Tree → 查找持有 ClassLoader 的 GC Root

# 3. 使用 jcmd 分析 Metaspace
jcmd <pid> VM.native_memory summary

# 4. 使用 Arthas（阿里诊断工具）
arthas> classloader -t
# 列出所有类加载器及加载的类数量

arthas> sc -d *Mapper | grep classLoader
# 查看哪个类加载器加载了最多的类
```

## 3. 一句话总结

> **Metaspace 是 JDK 8 替代 PermGen 的本地内存区域，存储类元数据，默认无上限（最大可占用整个本地内存），OOM 主要发生在大量动态类生成或类加载器泄漏的场景；通过设置 `-XX:MaxMetaspaceSize` 限制上限，通过 `-XX:MetaspaceSize` 设置初始阈值，生产环境必须设置最大值并监控使用量。**

## 4. 面试加分话术

> "我在项目里排查过一次 Metaspace OOM——线上服务跑了三天就开始 OOM，当时以为是内存泄漏，用 MAT 分析了堆转储才发现是 Dubbo 服务每次调用都生成新的服务接口代理类，类加载器没释放。后来升级了 Dubbo 版本，同时加了 `-XX:MaxMetaspaceSize=512m` 限制，还在 Arthas 里加了类加载器的监控。还有个经验：生产环境一定要给 Metaspace 设上限，我见过没设上限的服务本地内存被 Metaspace 吃满导致系统都卡死了。另外 Spring Boot 应用默认 MetaspaceSize 会大一些，因为 CGLIB 代理用得多。"
