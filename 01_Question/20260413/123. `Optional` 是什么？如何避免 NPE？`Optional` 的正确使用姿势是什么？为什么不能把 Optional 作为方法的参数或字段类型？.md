### 1.3 核心代码示例

#### 1. Optional 创建与基础使用

```java
public class OptionalBasics {
    
    public static void main(String[] args) {
        
        // ===== 创建 Optional =====
        
        // 1. 空 Optional
        Optional<String> empty = Optional.empty();
        
        // 2. of() - 值不能为 null，否则 NPE
        Optional<String> notNull = Optional.of("Hello");
        // Optional.of(null); // 抛出 NullPointerException!
        
        // 3. ofNullable() - 接受 null，返回 empty
        Optional<String> nullable = Optional.ofNullable(null);  // empty
        Optional<String> nullable2 = Optional.ofNullable("World");  // 有值
        
        
        // ===== 常见错误用法 =====
        
        // ❌ 错误 1：isPresent() + get() 跟 null 检查没区别
        if (nullable.isPresent()) {
            System.out.println(nullable.get());  // 还是可能 NPE（竞态）
        }
        
        // ❌ 错误 2：用 Optional 包装基本类型
        Optional<int> badIdea = Optional.of(1);  // ❌ 不能用基本类型！
        OptionalInt goodIdea = OptionalInt.of(1);  // ✅ 用 OptionalInt
        
        // ❌ 错误 3：过度使用 Optional
        // Optional<Optional<Optional<String>>> nested = ...;  // 过度嵌套
    }
}
```

#### 2. 正确避免 NPE

```java
@Service
public class SafeUserService {
    
    // ====== 场景1：方法返回值 ======
    
    /**
     * 返回 Optional 而不是 null
     */
    public Optional<User> findById(Long id) {
        User user = userRepository.findById(id);
        return Optional.ofNullable(user);  // 包装而不是直接返回 null
    }
    
    // 调用方必须处理
    public void handleUser(Long id) {
        Optional<User> userOpt = findById(id);
        
        // ✅ 方式1：orElse 使用默认值
        User user = userOpt.orElse(User.DEFAULT);
        
        // ✅ 方式2：orElseGet 懒加载默认值（更高效）
        User user2 = userOpt.orElseGet(() -> createDefaultUser());
        
        // ✅ 方式3：orElseThrow 有意义异常
        User user3 = userOpt.orElseThrow(
            () -> new UserNotFoundException("用户不存在: " + id)
        );
    }
    
    
    // ====== 场景2：链式调用替代多层 null 检查 ======
    
    // ❌ 传统写法：多层嵌套 null 检查
    public String getCityNameOld(User user) {
        if (user != null) {
            Address address = user.getAddress();
            if (address != null) {
                City city = address.getCity();
                if (city != null) {
                    return city.getName();
                }
            }
        }
        return "Unknown";
    }
    
    // ✅ Optional 写法：链式调用
    public String getCityNameNew(User user) {
        return Optional.ofNullable(user)
            .map(User::getAddress)
            .map(Address::getCity)
            .map(City::getName)
            .orElse("Unknown");
    }
    
    
    // ====== 场景3：Optional 作为方法参数 ======
    // ⚠️ 不推荐！详见追问4
    public void processUser(Optional<User> userOpt) {
        userOpt.ifPresent(user -> {
            // 业务逻辑
        });
    }
    
    
    // ====== 场景4：集合安全处理 ======
    
    public String getFirstElement(List<String> list) {
        return Optional.ofNullable(list)
            .filter(l -> !l.isEmpty())
            .map(l -> l.get(0))
            .orElse("empty");
    }
}
```

#### 3. 高级用法：map / flatMap / filter

```java
public class OptionalAdvanced {
    
    public static void main(String[] args) {
        
        // ===== map：值的转换 =====
        Optional<String> name = Optional.of("Alice");
        Optional<Integer> length = name.map(String::length);
        System.out.println(length.orElse(0));  // 5
        
        
        // ===== flatMap：值本身是 Optional 的情况 =====
        Optional<Person> person = Optional.of(new Person());
        
        // ❌ 错误：map 会产生 Optional<Optional<String>>
        Optional<Optional<String>> wrong = person.map(Person::getCarName);
        
        // ✅ 正确：flatMap 扁平化
        Optional<String> correct = person.flatMap(p -> p.getCar().map(Car::getName));
        
        
        // ===== filter：条件过滤 =====
        Optional<Integer> age = Optional.of(25);
        Optional<Integer> adultAge = age.filter(a -> a >= 18);
        Optional<Integer> invalidAge = age.filter(a -> a < 0);  // empty
        
        
        // ===== stream()：Java 9+ 转换 =====
        List<Optional<String>> list = Arrays.asList(
            Optional.of("a"),
            Optional.empty(),
            Optional.of("b")
        );
        
        // 把所有 Optional 转为 Stream，合并处理
        String result = list.stream()
            .flatMap(Optional::stream)  // 空 Optional 变成空 Stream
            .collect(Collectors.joining(","));
        // 结果: "a,b"
    }
}

// 模拟实体类
class Person {
    private String name;
    private Optional<Car> car = Optional.empty();
    
    public Optional<Car> getCar() { return car; }
}

class Car {
    private String name;
    public Optional<String> getName() { return Optional.ofNullable(name); }
}
```

### 2.2 🔥 追问 2：Optional 的正确使用姿势

> **A:**

**✅ 正确用法 vs ❌ 错误用法：**

| 场景 | ❌ 错误用法 | ✅ 正确用法 | 说明 |
|------|-----------|-----------|------|
| **方法返回值** | `User findById()` 返回 null | `Optional<User> findById()` | 明确告诉调用者可能无值 |
| **链式调用** | 多层 if (obj != null) | `obj.map().map().orElse()` | 声明式，更简洁 |
| **默认值** | `user != null ? user : new User()` | `user.orElse(new User())` | 更语义化 |
| **懒计算默认值** | `user != null ? user : createDefault()` | `user.orElseGet(() -> createDefault())` | 避免不必要的创建 |
| **条件执行** | `if (opt.isPresent()) opt.get()` | `opt.ifPresent(System.out::println)` | 更安全 |
| **抛异常** | `opt.isPresent() ? opt.get() : throw` | `opt.orElseThrow(() -> new ...)` | 更简洁 |

**实际项目推荐用法：**

```java
@Service
public class RecommendedUsage {
    
    // ====== 1. 作为返回值（强烈推荐） ======
    public Optional<User> findUser(Long id) {
        return userRepository.findById(id)
            .map(this::toUser);
    }
    
    // ====== 2. 链式数据转换 ======
    public String getUserCityName(Long userId) {
        return findUser(userId)
            .flatMap(User::getAddress)           // Address 本身是 Optional
            .flatMap(Address::getCity)
            .map(City::getName)
            .orElse("未知城市");
    }
    
    // ====== 3. 条件执行 ======
    public void sendNotification(Long userId) {
        findUser(userId)
            .filter(User::hasEmailVerified)
            .ifPresent(user -> notificationService.send(user.getEmail()));
    }
    
    // ====== 4. 多条件组合 ======
    public ValidationResult validateOrder(Order order) {
        Optional<User> userOpt = findUser(order.getUserId());
        Optional<Product> productOpt = findProduct(order.getProductId());
        
        if (userOpt.isEmpty()) {
            return ValidationResult.error("用户不存在");
        }
        if (productOpt.isEmpty()) {
            return ValidationResult.error("商品不存在");
        }
        
        // 组合逻辑
        return validate(userOpt.get(), productOpt.get());
    }
    
    // ====== 5. 集合批量处理（Java 9+）======
    public List<String> getValidUserNames(List<Optional<User>> userList) {
        return userList.stream()
            .flatMap(Optional::stream)  // 过滤掉 empty，保留值
            .filter(User::isActive)
            .map(User::getName)
            .collect(Collectors.toList());
    }
}
```

### 2.3 🔥 追问 3：为什么不能把 Optional 作为字段类型？

> **A:**

**Optional 作为字段的五大问题：**

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                     Optional 作为字段类型的五大问题                               │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│  问题1: 序列化不支持 ❌                                                          │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │  Optional<String> cannot be serialized                                   │   │
│  │  // Jackson、GWT、大多数 ORM 都不支持 Optional 序列化                      │   │
│  │                                                                             │   │
│  │  @Entity                                                                          │   │
│  │  public class User {                                                          │   │
│  │      private Optional<String> nickname;  // ❌ 序列化失败！                 │   │
│  │  }                                                                              │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
│  问题2: 表征意义不清晰 ❌                                                         │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │  Optional 本质是"方法返回值语义"，表示"可能无值"                            │   │
│  │  字段本身就是可选的，不需要再包装一层                                        │   │
│  │                                                                             │   │
│  │  private String nickname;  // 本身就是可空的，不需要 Optional              │   │
│  │  private @Nullable String nickname;  // ✅ 用 @Nullable 注解               │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
│  问题3: 增加内存开销 ❌                                                           │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │  Optional 本身是个对象，有额外 16 字节头 + 引用 8 字节                       │   │
│  │  而 null 引用只需 8 字节                                                   │   │
│  │  如果一个类有 10 个字段，全用 Optional = 多占用 ~200 字节                   │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
│  问题4: 增加 API 复杂度 ❌                                                       │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │  // 不用 Optional                                                             │   │
│  │  user.getNickname()  // 返回 String 或 null                                 │   │
│  │                                                                             │   │
│  │  // 用 Optional                                                                 │   │
│  │  user.getNickname().orElse("匿名")  // 太啰嗦                               │   │
│  │  // 每次访问都要处理，累死调用方                                              │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
│  问题5: ORM 集成问题 ❌                                                         │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │  MyBatis / JPA 映射 Optional 字段时：                                     │   │
│  │  - SELECT nickname FROM user → Optional<String>  // 自动映射？              │   │
│  │  - UPDATE 时 Optional.empty() 怎么处理？                                   │   │
│  │  - null vs empty 语义混淆                                                   │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘
```

**正确做法：@Nullable 注解替代**

```java
@Entity
@Table(name = "user")
public class User {
    
    // ✅ 正确：字段保持原类型，用 @Nullable 表示可空
    @Column(nullable = true)
    @Nullable
    private String nickname;
    
    // ✅ 正确：必填字段用 @NonNull
    @Column(nullable = false)
    @NonNull
    private String username;
    
    // ✅ 正确：Java 17+ 可用 @NotNull 替代
    private String email;
    
    // ===== Optional 用于方法返回值 ======
    
    // ✅ 正确：Service 层返回 Optional
    public Optional<User> findById(Long id) {
        return userRepository.findById(id);
    }
    
    // ✅ 正确：Repository 可返回 Optional（Spring Data 支持）
    Optional<User> findByUsername(String username);
}
```

### 2.4 🔥 追问 4：Optional 作为方法参数是否合适？

> **A:**

**Optional 作为参数的争议：**

| 立场 | 观点 | 适用场景 |
|-----|------|---------|
| **不推荐 ❌** | 语义混乱，API 难看 | 大多数场景 |
| **可以接受 ✅** | 需要区分"不传"和"传 null"时 | 极特殊场景 |

**为什么不推荐：**

```java
// ❌ 不好的设计：Optional 参数
public void sendEmail(Optional<String> address, Optional<String> subject) {
    address.ifPresent(addr -> {
        // 业务逻辑
    });
}

// 调用时：
sendEmail(Optional.of("test@xxx.com"), Optional.empty());  // 丑！
sendEmail(Optional.of("test@xxx.com"), null);  // 混乱！


// ✅ 更好的设计：重载方法
public void sendEmail(String address) {
    sendEmail(address, "默认主题");
}

public void sendEmail(String address, String subject) {
    // 业务逻辑
}

// 调用时：
sendEmail("test@xxx.com");
sendEmail("test@xxx.com", "自定义主题");


// ✅ 更好的设计：Builder 模式
public EmailRequest {
    private String address;
    private String subject;
    private String content;
    private Optional<String> cc;  // Optional 只在内部使用
    
    public EmailRequest address(String address) { this.address = address; return this; }
    public EmailRequest subject(String subject) { this.subject = subject; return this; }
    public EmailRequest cc(String cc) { this.cc = Optional.ofNullable(cc); return this; }
}
```

**唯一可接受的场景：**

```java
// 需要区分"不提供值"和"提供 null"
@PostMapping("/search")
public List<User> search(
    @RequestParam Optional<String> name,
    @RequestParam Optional<Integer> age) {
    
    // name.isEmpty() → 不按 name 过滤
    // name = Optional.of(null) → 按 name=null 过滤
    // name = Optional.of("张三") → 按 name="张三" 过滤
    
    return userService.search(
        name.filter(s -> !s.isEmpty()),  // 过滤空字符串
        age
    );
}
```

**最佳实践总结：**

```
Optional 作为参数：❌ 尽量避免
Optional 作为返回值：✅ 强烈推荐
Optional 作为字段：❌ 避免，用 @Nullable
Optional 作为集合元素：❌ 避免
```

## 3. 一句话总结

> **Optional 本质是一个值容器，用于替代 null 表示"可能有值也可能无值"。正确用法是作为方法返回值，让调用者必须处理无值情况。但 Optional **不能**作为字段类型（不支持序列化、语义混乱、增加内存开销），应该用 `@Nullable` 注解替代。Optional 作为方法参数也尽量避免，会让 API 变得丑陋。**

## 4. 面试加分话术

> "Optional 是 Java 8 引入的特性，我用得比较多的场景是**替代多层 null 判断**。
> 
> 举个小例子：之前有段老代码 `user.getAddress().getCity().getName()`，用了 5 层判空，每次改需求都要改这坨代码。后来用 Optional 改成了链式调用：
> ```java
> return Optional.ofNullable(user)
>     .map(User::getAddress)
>     .map(Address::getCity)
>     .map(City::getName)
>     .orElse("未知城市");
> ```
> 代码清爽多了，而且后续加字段只需要加一行 `.map()`。
> 
> 但 Optional 也被滥用过。团队里有人把 Optional 当字段类型用，结果 JSON 序列化失败，生产事故。所以我们后来定了规范：**Optional 只用于返回值，字段用 `@Nullable` 注解**，团队成员一看就明白什么意思。
> 
> 还有个小技巧：**`orElseGet()` 比 `orElse()` 性能好**。`orElse()` 无论有没有值都会执行参数表达式，而 `orElseGet()` 只在无值时才执行 Supplier。所以如果创建默认值开销大，用 `orElseGet()`。"
