让我组织一下这道题的内容：

**Online DDL 的核心概念**：
- DDL（Data Definition Language）操作如 ALTER TABLE 在执行时会锁表
- Online DDL 允许 DDL 和 DML 并发执行
- MySQL 5.6 和 8.0 的实现方式不同

**MySQL 5.6 的实现**：
- 引入了 ALGORITHM=INPLACE（但有限制）
- ALGORITHM=COPY（拷贝表方式）
- 在线DDL主要通过"影子拷贝"实现

**MySQL 8.0 的改进**：
- 默认 INPLACE
- 改进的DDL元数据锁管理
- 更多的DDL操作支持在线执行

**ALGORITHM=INPLACE vs ALGORITHM=COPY**：
- INPLACE：直接在原表上修改，不拷贝数据
- COPY：创建临时表，拷贝数据
- INPLACE 更快，但有些操作不支持
# 138. 什么是 MySQL 的 Online DDL？

### ALGORITHM=INPLACE vs COPY 的区别

```
┌─────────────────────────────────────────────────────────────────────┐
│                  ALGORITHM=COPY 工作流程                             │
│                                                                     │
│  原表：t_order                                                     │
│                                                                     │
│  1. 创建临时表：t_order_gho123                                     │
│                                                                     │
│  2. 对原表加全局写锁（阻塞所有读写）                                 │
│                                                                     │
│  3. 拷贝数据：原表 → 临时表                                         │
│     ┌─────────┐        ┌─────────────────┐                        │
│     │ t_order │  ──→   │ t_order_gho123  │                        │
│     │ 1000万  │  COPY  │ (带新结构)       │                        │
│     └─────────┘        └─────────────────┘                        │
│                                                                     │
│  4. 删除原表，重命名临时表                                          │
│                                                                     │
│  5. 释放锁                                                          │
│                                                                     │
│  问题：                                                             │
│  • 磁盘空间翻倍（需要同时存储原表和临时表）                        │
│  • 锁表时间长                                                      │
│  • 大表操作可能需要数小时                                          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│                  ALGORITHM=INPLACE 工作流程                         │
│                                                                     │
│  1. 准备阶段（短暂 MDL 锁）                                         │
│     • 分析DDL操作是否支持 inplace                                  │
│     • 创建必要的元数据记录                                          │
│                                                                     │
│  2. 执行阶段（无锁）                                               │
│     • DML 操作继续执行                                              │
│     • DDL 在后台悄然修改表结构                                       │
│                                                                     │
│  3. 提交阶段（短暂 MDL 锁）                                         │
│     • 应用 DDL 期间的变更                                          │
│     • 完成元数据更新                                                │
│                                                                     │
│  优势：                                                             │
│  • 不拷贝数据，磁盘空间需求小                                       │
│  • 锁表时间极短                                                    │
│  • 允许 DML 并发执行                                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 核心代码示例

```sql
-- ========== ALGORITHM=COPY（5.5 方式）==========
ALTER TABLE orders ADD COLUMN source VARCHAR(50), ALGORITHM=COPY;

-- ========== ALGORITHM=INPLACE（5.6+ 方式）==========
ALTER TABLE orders ADD COLUMN source VARCHAR(50), ALGORITHM=INPLACE;

-- ========== MySQL 8.0 默认就是 INPLACE ==========
ALTER TABLE orders ADD COLUMN source VARCHAR(50);
-- 等价于 ALGORITHM=INPLACE, LOCK=NONE

-- ========== 显式指定锁策略 ==========
ALTER TABLE orders ADD COLUMN source VARCHAR(50), 
    ALGORITHM=INPLACE, 
    LOCK=SHARED;  -- 允许读，阻塞写

ALTER TABLE orders ADD COLUMN source VARCHAR(50), 
    ALGORITHM=INPLACE, 
    LOCK=EXCLUSIVE;  -- 不允许任何并发

-- ========== 查看 DDL 执行情况 ==========
-- 通过 performance_schema 监控
SELECT * FROM performance_schema.events_statements_history 
WHERE SQL_TEXT LIKE '%ALTER TABLE%';
```

## 2. 扩展追问

### ## 🔥 扩展追问 1：MySQL 5.6 和 8.0 在 Online DDL 上有什么区别？

> **A:** MySQL 5.6 的 Online DDL 支持有限，很多操作仍需要 COPY 方式；MySQL 8.0 大幅改进了 Online DDL，支持的操作更多，锁冲突更少。

| DDL 操作 | MySQL 5.6 ALGORITHM | MySQL 8.0 ALGORITHM | 说明 |
|---------|--------------------|--------------------|------|
| ADD COLUMN | INPLACE（部分支持） | INPLACE | 8.0 完全支持 |
| DROP COLUMN | INPLACE | INPLACE | 8.0 更快 |
| ADD INDEX | INPLACE | INPLACE | 都支持 |
| CHANGE COLUMN | COPY | INPLACE（部分） | 8.0 改进 |
| MODIFY COLUMN | COPY | INPLACE（部分） | 8.0 改进 |
| ADD PRIMARY KEY | COPY | INPLACE | 仍需拷贝 |
| DROP PRIMARY KEY | 不支持 | COPY | 8.0 也不支持 |
| 字符集修改 | COPY | INPLACE | 都需重建 |

```sql
-- MySQL 8.0 的改进：
-- 1. Instant ADD COLUMN（瞬间加列）
ALTER TABLE orders ADD COLUMN remark TEXT, ALGORITHM=INSTANT;
-- 列被添加到表定义中，但数据文件中不存储
-- 读取时动态拼接默认值

-- 2. 更智能的锁管理
-- 8.0 把 DDL 分成多个阶段，MDL 锁只在关键阶段持有

-- 3. 原子 DDL
CREATE TABLE t1 (id INT) ALGORITHM=COPY;
-- 8.0 支持原子性：要么成功，要么回滚（不像之前可能留下半成品表）
```

### ## 🔥 扩展追问 2：为什么有时候 ALTER TABLE 会自动退化为 COPY？

> **A:** 当 DDL 操作涉及**行格式变更**、**列位置移动**或**主键变更**时，即使指定了 `ALGORITHM=INPLACE`，MySQL 也会退化为 COPY。

```sql
-- 会退化为 COPY 的情况

-- 1. 更改列的数据类型（可能数据不兼容）
ALTER TABLE orders MODIFY COLUMN amount DECIMAL(10,4), ALGORITHM=INPLACE;
-- ERROR: Cannot change column type in online DDL

-- 2. 添加/删除主键
ALTER TABLE orders DROP PRIMARY KEY, ALGORITHM=INPLACE;
-- 退化为 COPY

-- 3. 修改 ROW_FORMAT
ALTER TABLE orders ROW_FORMAT=COMPRESSED, ALGORITHM=INPLACE;
-- 可能退化为 COPY

-- 4. 修改字符集
ALTER TABLE orders CONVERT TO CHARACTER SET utf8mb4, ALGORITHM=INPLACE;
-- 退化为 COPY（需要重新编码所有文本数据）

-- 5. 显式要求
ALTER TABLE orders ADD COLUMN source VARCHAR(50), ALGORITHM=COPY;
-- 强制 COPY

-- 查看实际的执行算法
ALTER TABLE orders ADD COLUMN source VARCHAR(50), ALGORITHM=INPLACE;
-- 查看警告：
SHOW WARNINGS;
-- Message: Cannot use inplace algorithm due to ...
```

### ## 🔥 扩展追问 3：pt-online-schema-change（pt-osc）和 gh-ost 是什么？

> **A:** pt-osc 和 gh-ost 是第三方工具，通过**触发器+影子表**的方式实现更安全的在线 DDL，绕过 MySQL 原生 Online DDL 的限制。

```
┌─────────────────────────────────────────────────────────────────────┐
│                      pt-osc 工作原理                                 │
│                                                                     │
│  原始表：orders                                                     │
│                                                                     │
│  1. 创建影子表：_orders_new（目标结构）                              │
│                                                                     │
│  2. 在原表上创建三个触发器：                                        │
│     • AFTER INSERT → 同时插入 _orders_new                           │
│     • AFTER UPDATE → 同时更新 _orders_new                           │
│     • AFTER DELETE → 同时删除 _orders_new                           │
│                                                                     │
│  3. 分批次拷贝数据（每次 1000 行）                                  │
│     ┌─────────┐                                                   │
│     │ orders  │  ──→  1000行/批 ──→  _orders_new                   │
│     └─────────┘                                                   │
│                                                                     │
│  4. 数据同步完成后，交换表名                                        │
│     RENAME TABLE orders TO _orders_old, _orders_new TO orders;      │
│                                                                     │
│  5. 删除原表和触发器                                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│                      gh-ost 工作原理（GitHub 出品）                  │
│                                                                     │
│  与 pt-osc 的区别：                                                 │
│  • 不使用触发器（避免锁竞争）                                        │
│  • 基于 binlog 同步增量数据                                         │
│  • 可暂停、可回滚                                                   │
│  • 对主从复制更友好                                                 │
│                                                                     │
│  流程：                                                             │
│  1. 创建影子表                                                      │
│  2. 读取原表 binlog position                                       │
│  3. 分批拷贝数据（类似 pt-osc）                                      │
│  4. 监听 binlog，应用增量变更                                       │
│  5. 校验数据一致性                                                  │
│  6. 切换表名                                                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

```bash
# pt-osc 示例
pt-online-schema-change \
    --alter "ADD COLUMN source VARCHAR(50)" \
    --user=root \
    --password=xxx \
    D=test,t=orders \
    --execute

# gh-ost 示例
gh-ost \
    --database="test" \
    --table="orders" \
    --alter="ADD COLUMN source VARCHAR(50)" \
    --user="root" \
    --password="xxx" \
    --execute
```

| 工具 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **pt-osc** | 触发器同步 | 成熟稳定 | 触发器有开销，锁竞争 |
| **gh-ost** | binlog 同步 | 无触发器，可暂停 | 需要 binlog 支持 |
| **原生 Online DDL** | inplace 修改 | 最简单，开销最小 | 支持的操作有限 |

### ## 🔥 扩展追问 4：Online DDL 期间的 binlog 是怎么保证一致的？

> **A:** Online DDL 期间通过**临时日志（row log）**记录 DDL 期间的变更，DDL 完成后应用这些变更，确保 binlog 记录的是最终一致的表结构。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Online DDL 的数据一致性保证                        │
│                                                                     │
│  DDL 执行阶段：                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │    UPDATE orders SET amount = 100 WHERE id = 1             │   │
│  │                          ↓                                   │   │
│  │              DML 操作写入原表                                │   │
│  │                          ↓                                   │   │
│  │              同时写入 Row Log（临时日志）                    │   │
│  │              （记录了变更的前镜像和后镜像）                    │   │
│  │                                                             │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  DDL 提交阶段：                                                    │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  将 Row Log 中的变更应用到新表结构                            │   │
│  │                                                             │   │
│  │  binlog 记录：                                               │   │
│  │  • DDL 开始位置                                              │   │
│  │  • DDL 操作本身                                             │   │
│  │  • Row Log 中的变更                                          │   │
│  │  • DDL 提交位置                                              │   │
│  │                                                             │   │
│  │  从库应用时：                                                │   │
│  │  • 按 binlog 顺序执行                                        │   │
│  │  • 确保和主库一致的最终结果                                   │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### ## 🔥 扩展追问 5：MySQL 8.0 的 Instant ADD COLUMN 是什么原理？

> **A:** MySQL 8.0.12 引入了 **Instant ADD COLUMN**，新增列只在表定义中添加，不修改数据文件，实现真正的"瞬间"加列。

```sql
-- Instant ADD COLUMN（8.0.12+）
ALTER TABLE orders ADD COLUMN remark TEXT, ALGORITHM=INSTANT;

-- 普通加列 vs Instant 加列
┌─────────────────────────────────────────────────────────────────────┐
│                        普通加列（INPLACE）                           │
│                                                                     │
│  1. 修改表定义（立即完成）                                          │
│  2. 修改每行数据（在后台）                                          │
│  3. 表越大，变更时间越长                                            │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│                      Instant 加列（8.0+）                           │
│                                                                     │
│  1. 修改表定义（立即完成）→ 完成！                                  │
│  2. 数据文件完全不修改！                                            │
│                                                                     │
│  读取时：                                                           │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  行数据（不含新列）                                          │   │
│  │      +                                                         │   │
│  │  记录头部的列计数（自动增加）                                 │   │
│  │      ↓                                                         │   │
│  │  新列的值 = 默认值（NULL 或 DEFAULT）                        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  限制：                                                             │
│  • 只支持 ADD COLUMN（新增列）                                      │
│  • 不支持添加 FIRST COLUMN / AFTER COLUMN（指定位置）               │
│  • 不支持新增 NOT NULL 列（除非有显式 DEFAULT）                     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 3. 一句话总结

> **MySQL Online DDL 通过 INPLACE 算法避免拷贝表实现原地修改，COPY 算法通过临时表拷贝数据；5.6 开始支持部分 Online DDL 但限制较多，8.0 改进为默认 INPLACE 并引入 Instant ADD COLUMN；指定 ALGORITHM=INPLACE 时若涉及行格式、主键等变更会自动退化为 COPY，大表建议用 pt-osc 或 gh-ost 工具更安全。**

## 4. 面试加分话术

> "我在项目里做过大表加索引的优化——当时订单表 5000 万数据，用原生 ALTER TABLE ADD INDEX 直接卡死了。后来用 pt-osc 工具，先在从库执行验证，确认没问题再切主库，加索引过程中读写完全不受影响。还有个经验：MySQL 8.0 的 Instant ADD COLUMN 很好用，加字段几乎是瞬间完成的，但要注意新增的列不能有 NOT NULL 约束（除非有默认值）。另外 DDL 之前一定要备份，gh-ost 比 pt-osc 更适合大表，因为它用 binlog 同步增量数据，不会因为触发器竞争导致延迟。还有个坑：执行 DDL 之前先 `SELECT * FROM information_schema.innodb_trx` 检查有没有长事务，DDL 开始时会等待所有事务提交，如果有大事务卡着，DDL 会一直等待甚至超时。"
