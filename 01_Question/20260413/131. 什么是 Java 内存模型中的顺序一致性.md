## 1. 基础流程

### 顺序一致性的定义

```
┌─────────────────────────────────────────────────────────────────────┐
│                    顺序一致性（Sequential Consistency）              │
│                                                                     │
│  定义：所有线程看到的操作执行顺序 = 程序代码的编写顺序                  │
│                                                                     │
│  理想状态：                                                         │
│                                                                     │
│  Thread A                      Thread B                             │
│  ─────────                     ─────────                            │
│  write x = 1 ─────────────────→ read x = 1（一定看到）               │
│  write y = 2 ─────────────────→ read y = 2（一定看到）               │
│                                                                     │
│  实际上（没有同步的情况下）：                                         │
│                                                                     │
│  Thread A                      Thread B                             │
│  ─────────                     ─────────                            │
│  write x = 1 ─────────┐                                             │
│                       ↓                                             │
│  write y = 2          → read x 可能看到 0（指令重排/缓存可见性）      │
│                       → read y 可能看到 0                           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 为什么加了 volatile/synchronized 也不是"严格按代码顺序执行"？

```java
public class SequentialDemo {
    private int x = 0;
    private volatile int y = 0;
    
    public void writer() {
        x = 1;           // 语句1
        y = 2;           // 语句2（volatile）
        z = 3;           // 语句3
    }
    
    public void reader() {
        // volatile 保证 y 的写操作对所有线程可见
        // 但不保证 x=1 一定在 y=2 之前执行完
    }
}
```

```
┌─────────────────────────────────────────────────────────────────────┐
│                      指令重排与内存屏障                              │
│                                                                     │
│  编译器优化（编译时）：                                              │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  源代码：x=1; y=2; z=3;                                       │   │
│  │                    ↓                                         │   │
│  │  处理器可能重排（单线程结果不变的前提下）：                      │   │
│  │  y=2; x=1; z=3;    ← 改变了执行顺序                            │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  volatile 的作用（以 volatile 写为例）：                            │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  StoreStore 屏障 ──→ y=2 ──→ StoreLoad 屏障                  │   │
│  │                      ↑                                       │   │
│  │  禁止上面的写和下面的读写重排到 y=2 之后                        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  但！x=1 仍然可能被重排到 y=2 之前或之后（只要不在 y=2 之前）          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### volatile 的内存屏障（以 x86 为例）

```
┌─────────────────────────────────────────────────────────────────────┐
│                      volatile 写的内存屏障                          │
│                                                                     │
│  StoreStore 屏障 ──→ volatile写 ──→ StoreLoad 屏障                  │
│                                                                     │
│  • StoreStore：禁止 volatile写之前的普通写 重排到 volatile写之后      │
│  • StoreLoad： 禁止 volatile写之后的所有读写 重排到 volatile写之前    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│                      volatile 读的内存屏障                          │
│                                                                     │
│  LoadLoad 屏障 ──→ volatile读 ──→ LoadStore 屏障                   │
│                                                                     │
│  • LoadLoad： 禁止 volatile读之前的普通读 重排到 volatile读之后       │
│  • LoadStore：禁止 volatile读之前的普通写 重排到 volatile读之后       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 经典案例：单例模式的"双重检查锁"

```java
public class Singleton {
    // volatile 解决的是：instance 变量的可见性问题
    // 而不是：构造函数内指令的重排问题
    private static volatile Singleton instance;
    
    public static Singleton getInstance() {
        if (instance == null) {                 // T1 检查
            synchronized (Singleton.class) {
                if (instance == null) {         // T2 检查
                    instance = new Singleton(); // 问题在这里！
                }
            }
        }
        return instance;
    }
}
```

```
┌─────────────────────────────────────────────────────────────────────┐
│          new Singleton() 的字节码层面（非 volatile）                  │
│                                                                     │
│  1. 分配内存                                                      │
│  2. 调用构造函数 ──→ instance 引用的对象初始化                       │
│  3. 将引用写入 instance 变量                                        │
│                                                                     │
│  处理器可能重排为：                                                  │
│  1. 分配内存                                                        │
│  2. 将引用写入 instance 变量 ──→ 其他线程看到 instance != null      │
│  3. 调用构造函数 ──→ 但对象还没初始化完！                            │
│                                                                     │
│  volatile 阻止了 2-3 的重排，强制先初始化再赋值                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 2. 扩展追问

### ## 🔥 扩展追问 1：JMM 的三大特性是什么？volatile 保证了多少？

> **A:** JMM（Java Memory Model）要求保证三大特性：**原子性**、**可见性**、**有序性**。

| 特性 | volatile 保证 | synchronized 保证 | 说明 |
|------|--------------|-----------------|------|
| 原子性 | 仅 volatile 读写本身 | ✅ 整个同步块 | volatile 不保证复合操作的原子性 |
| 可见性 | ✅ 写后立即刷新到主存，读立即从主存读取 | ✅ 释放锁时刷新缓存 | 两者的可见性语义等价 |
| 有序性 | ✅ 禁止指令重排（单线程内） | ✅ 同一锁的获取/释放形成屏障 | synchronized 只保证单个锁的顺序 |

```java
// ❌ volatile 不能保证原子性！
private volatile int count = 0;

void increment() {  // 不是原子操作
    count++;  // 读取-修改-写入，三步操作
}

// 场景：两个线程同时执行
// Thread A: 读取 count=0
// Thread B: 读取 count=0
// Thread A: 计算 0+1=1，写入
// Thread B: 计算 0+1=1，写入
// 结果: count=1（而不是2！）

// ✅ 需要 synchronized 保证原子性
private int count = 0;
synchronized void increment() {
    count++;
}
```

### ## 🔥 扩展追问 2：as-if-serial 语义是什么？

> **A:** as-if-serial 的意思是：**在单线程中，编译器和处理器可以任意重排指令，只要保证重排后程序执行结果不变**。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    as-if-serial 示例                                 │
│                                                                     │
│  double pi = 3.14;         // 语句1                                 │
│  double r   = 1.0;          // 语句2                                 │
│  double area = pi * r * r; // 语句3（依赖1和2的结果）                │
│                                                                     │
│  在单线程中，1和2没有依赖关系，可以重排：                             │
│  double r   = 1.0;          // 先执行                                │
│  double pi = 3.14;         // 后执行                                │
│  double area = pi * r * r; // 一定最后执行                           │
│                                                                     │
│  结果不变，但指令执行顺序 ≠ 代码编写顺序                              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

> **关键点：** as-if-serial 只约束**单线程**，不约束**多线程**。多线程下，其他线程可能看到被重排后的执行顺序。

### ## 🔥 扩展追问 3：happens-before 原则是什么？为什么重要？

> **A:** happens-before 是 JMM 定义的一种**偏序关系**：如果 A happens-before B，那么 A 的执行结果对 B 可见，且 A 的执行顺序在 B 之前。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    happens-before 核心规则                           │
│                                                                     │
│  1. 程序顺序规则：同一线程中，代码书写顺序 happens-before 执行顺序    │
│                                                                     │
│  2. volatile 规则：volatile 写 happens-before 后续的 volatile 读    │
│                                                                     │
│  3. 锁规则：synchronized 解锁 happens-before 后续加锁                 │
│                                                                     │
│  4. 线程启动规则：Thread.start() happens-before 被启动线程的任何操作  │
│                                                                     │
│  5. 线程终止规则：线程所有操作 happens-before 其他线程检测到终止      │
│                                                                     │
│  6. 传递性：A happens-before B，B happens-before C → A happens-before C │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

```java
// 示例：volatile 的 happens-before 保证
int x = 0;
volatile boolean ready = false;

void writer() {
    x = 1;             // ①
    ready = true;      // ② volatile写
}

void reader() {
    if (ready) {       // ③ volatile读
        System.out.println(x);  // 一定看到 x=1
    }
}

// 因为：① happens-before ② happens-before ③
```

### ## 🔥 扩展追问 4：synchronized 的有序性保证和 volatile 有什么区别？

> **A:** synchronized 只保证**同一个锁**的获取和释放之间的代码有序；volatile 保证的是**所有线程**对该变量的读写有序。

| 维度 | synchronized | volatile |
|------|-------------|----------|
| 作用范围 | 代码块/方法 | 单个变量 |
| 有序性 | 同一把锁内部有序 | 写线程和后续读线程之间有序 |
| 可见性 | 释放锁时刷新缓存 | 写后立即刷新，读前立即读取 |
| 原子性 | ✅ 保证 | ❌ 不保证复合操作 |

```java
// synchronized 的有序性：只对同一把锁有效
synchronized (lock) {
    x = 1;      // 这些指令不会重排到 synchronized 块外部
    y = 2;
    z = x + y;  // 也不会被重排
}
// 释放锁时刷新缓存

// 两个 synchronized(不同锁) 之间，指令可以自由重排
synchronized (lock1) { a = 1; }   // 和下面无关
synchronized (lock2) { b = 2; }   // 可以重排
```

### ## 🔥 扩展追问 5：CPU 缓存和内存屏障是什么关系？

> **A:** 内存屏障是 CPU 层面的指令，用来**禁止特定类型的指令重排**；volatile 和 synchronized 的语义最终要靠内存屏障来落地。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    CPU 缓存结构                                       │
│                                                                     │
│        CPU ──→ L1 Cache ──→ L2 Cache ──→ L3 Cache ──→ 主存           │
│       寄存器                        (所有核共享)                     │
│                                                                     │
│  问题：                                                             │
│  • 每个核有自己的缓存，修改后不会立即同步到主存                        │
│  • 写缓冲（Store Buffer）导致写操作延迟刷新                           │
│  • 无效队列（Invalid Queue）导致读取延迟更新                          │
│                                                                     │
│  内存屏障的作用：                                                    │
│  • SFENCE/MFENCE/LFENCE：强制刷新缓存/等待刷新完成                     │
│  • 确保屏障前的指令先于屏障后的指令执行                               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

## 3. 一句话总结

> **顺序一致性是理想状态，volatile 和 synchronized 保证的是 happens-before 关系（可见性和有序性），而不是"代码严格按顺序逐行执行"；volatile 通过内存屏障禁止指令重排但只针对单个变量，synchronized 保证同一锁的原子性和有序性，两者都不保证复合操作的原子性。**
