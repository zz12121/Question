## 1. 基础流程

### 1.1 Record 是什么？

```
┌─────────────────────────────────────────────────────────┐
│  Record（JDK 16 正式特性）                              │
│                                                         │
│  一种全新的类声明形式，专门用于"纯数据载体"             │
│                                                         │
│  核心理念：                                            │
│  "Data classes should be expressively data-centric,    │
│   not merely a carrier for scattered fields"             │
│                                                         │
│  数据类应该表达性地以数据为中心                        │
│  而不是零散字段的搬运工                                │
└─────────────────────────────────────────────────────────┘

为什么需要 Record？

┌─────────────────────────────────────────────────────────┐
│  传统 DTO（贫血模型）的冗余代码：                      │
│                                                         │
│  public class User {                                   │
│      private final Long id;                            │
│      private final String name;                        │
│      private final int age;                            │
│                                                         │
│      public User(Long id, String name, int age) {      │
│          this.id = id;                                │
│          this.name = name;                            │
│          this.age = age;                              │
│      }                                                │
│                                                         │
│      public Long getId() { return id; }               │
│      public String getName() { return name; }         │
│      public int getAge() { return age; }              │
│                                                         │
│      public boolean equals(Object o) { ... }           │
│      public int hashCode() { ... }                    │
│      public String toString() { ... }                 │
│  }                                                    │
│  60 行代码！                                          │
└─────────────────────────────────────────────────────────┘

Record 一行搞定：
┌─────────────────────────────────────────────────────────┐
│  public record User(Long id, String name, int age) {} │
│                                                         │
│  等价于上面的全部 60 行代码                             │
│  自动生成：构造方法 + getter + equals + hashCode + toString│
└─────────────────────────────────────────────────────────┘
```

### 1.2 Record 生成的代码一览

```java
// 源码
public record User(Long id, String name, int age) {}

// 编译器自动生成的内容（等价的 Java 代码）

public final class User extends java.lang.Record {
    
    // 1. private final 字段（与参数一一对应）
    private final Long id;
    private final String name;
    private final int age;
    
    // 2. 标准的 canonical 构造方法
    public User(Long id, String name, int age) {
        this.id = id;
        this.name = name;
        this.age = age;
    }
    
    // 3. getter 方法（不是 getXxx，而是直接字段名）
    public Long id() { return this.id; }    // 注意：不是 getId()
    public String name() { return this.name; }
    public int age() { return this.age; }
    
    // 4. equals()（基于所有字段的值比较）
    public boolean equals(java.lang.Object o) {
        if (this == o) return true;
        if (o == null || !(o instanceof User)) return false;
        User that = (User) o;
        return this.id.equals(that.id) 
            && this.name.equals(that.name) 
            && this.age == that.age;
    }
    
    // 5. hashCode()（基于所有字段）
    public int hashCode() {
        return java.util.Objects.hash(this.id, this.name, this.age);
    }
    
    // 6. toString()（所有字段的字符串表示）
    public java.lang.String toString() {
        return "User[id=" + this.id 
            + ", name=" + this.name 
            + ", age=" + this.age + "]";
    }
}
```

### 1.4 Record 本质解析

```
┌─────────────────────────────────────────────────────────┐
│  Record 的本质：语法糖（SYNTAX SUGAR）                  │
│                                                         │
│  Record 不是运行时特性，是编译时语法糖                   │
│  编译后生成的是标准 .class 文件                         │
│  字节码中就是一个普通的 final 类继承自 java.lang.Record │
│                                                         │
│  Record 的类修饰符：                                    │
│  - 自动 final 类（不可被继承）                         │
│  - 自动继承 java.lang.Record                           │
│  - 自动实现 java.io.Serializable（隐式）               │
└─────────────────────────────────────────────────────────┘

// 反编译 record 后的字节码结构
$ javap -p User.class

public final class com.example.User extends java.lang.Record 
    implements java.io.Serializable {
    
    private final java.lang.Long id;
    private final java.lang.String name;
    private final int age;
    
    // Canonical 构造方法
    public com.example.User(java.lang.Long, java.lang.String, int);
    
    // getter 方法（字段名直接作为方法名）
    public final java.lang.Long id();
    public final java.lang.String name();
    public final int age();
    
    // Object 方法
    public boolean equals(java.lang.Object);
    public int hashCode();
    public java.lang.String toString();
    
    // 组件名称（Record 特有）
    private static java.util.List $fields();
    
    // Record 的组件（字段描述）
    private static java.lang.RecordComponent[] $recordComponents();
}
```

## 2. Record 的构造方法详解

### 2.1 构造方法的 3 种形式

```java
// 形式1：自动生成的 canonical 构造方法（最常用）
public record User(Long id, String name, int age) {}

// 等价于：
public record User(Long id, String name, int age) {
    public User(Long id, String name, int age) {
        this.id = id;
        this.name = name;
        this.age = age;
    }
}

// 形式2：紧凑构造方法（Compact Constructor）
// 参数列表为空，只做校验
public record User(Long id, String name, int age) {
    
    // 紧凑构造方法：无参数列表，自动委托 canonical
    // 用途：做参数校验，而不是赋值
    public User {
        if (id == null) {
            throw new IllegalArgumentException("id cannot be null");
        }
        if (name == null || name.isBlank()) {
            throw new IllegalArgumentException("name cannot be blank");
        }
        if (age < 0 || age > 150) {
            throw new IllegalArgumentException("age out of range");
        }
        
        // 可以在这里做一些转换
        name = name.strip();
    }
}

// 形式3：完整构造方法（Full Canonical Constructor）
// 完全自定义的构造方法
public record User(Long id, String name, int age) {
    
    public User(Long id, String name, int age) {
        // 必须显式给所有字段赋值
        if (id == null) {
            throw new IllegalArgumentException("id cannot be null");
        }
        this.id = id;
        this.name = (name != null) ? name.strip() : "";
        this.age = Math.max(0, Math.min(age, 150));
    }
    
    // 额外构造方法（短路构造）
    public User(String name) {
        this(null, name, 0);  // 调用 canonical 构造
    }
}
```

### 2.2 Record 构造方法的限制

```java
// Record 构造方法规则：

public record Point(int x, int y) {
    
    // ✅ 合法的紧凑构造方法
    public Point {
        // 只能做字段的初始化和校验
        // 无需声明参数，编译器自动推导
        // 隐式调用 super()
    }
    
    // ✅ 额外的构造方法（委托）
    public Point(int x) {
        this(x, 0);  // 必须委托给 canonical 构造
    }
    
    // ✅ 额外的构造方法（String 解析）
    public Point(String s) {
        this(parse(s));  // parse 返回 int[]
    }
    
    // ❌ 非法的构造方法形式：
    
    // ❌ 不能声明与字段同名的参数：
    // public Point(int x, int y) {  // x 和 y 与字段名冲突！
    //     this.x = x;
    //     this.y = y;
    // }
    
    // ❌ 不能直接 this() 和 super() 同时调用
    // super() 由编译器隐式处理
    
    // ❌ 不能在 full constructor 中省略字段赋值
    // public Point(int x, int y) {  // 编译错误！
    //     this.x = x;
    //     // 缺少 this.y = y;
    // }
}
```

## 3. Record 的继承与接口实现

### 3.1 Record 不能继承类（extends）

```java
// ❌ Record 不能继承任何类
public record User(Long id) extends Person {}  // 编译错误！

// ❌ Record 不能被继承（自动 final）
public record Admin(Long id, String role) {}  // 自动 final

public record SuperAdmin(Long id) extends Admin {}  // 编译错误！
// Cannot inherit from final class 'Admin'
```

```
为什么 Record 不能继承？

┌─────────────────────────────────────────────────────────┐
│  Record 的设计哲学：                                      │
│                                                         │
│  1. Record 是"纯数据载体"，不是"行为的载体"           │
│     - 如果可以继承，父类的状态就不透明了                │
│     - equals/hashCode/toString 的语义会变复杂           │
│                                                         │
│  2. Record 已经隐式继承了 java.lang.Record              │
│     - Java 不支持多继承                                 │
│     - 所以 Record 不能再 extends 其他类                 │
│                                                         │
│  3. Record 自动 final                                   │
│     - 不能被继承，防止子类破坏父类的数据不变性          │
│     - 符合"不可变数据类"的设计原则                      │
└─────────────────────────────────────────────────────────┘

Record 的隐式继承链：
java.lang.Object
        ↑
        │ (隐式继承)
java.lang.Record  ←  getClass(), toString(), equals(), hashCode()
        ↑
        │ (显式声明)
User(Long id, String name)  ←  自定义数据字段和方法
```

### 3.2 Record 可以实现接口

```java
// ✅ Record 可以实现接口（implements）

// 接口定义
public interface Identifiable {
    Long getId();
    default String getIdString() {
        return String.valueOf(getId());
    }
}

// 校验接口
public interface Validatable {
    void validate();
}

// Record 实现多个接口
public record User(Long id, String name, int age) 
    implements Identifiable, Validatable {
    
    // 需要实现接口的抽象方法
    @Override
    public Long getId() {
        return id;  // 等价于 id()
    }
    
    @Override
    public void validate() {
        if (id == null || id < 0) {
            throw new IllegalStateException("Invalid id");
        }
        if (name == null || name.isBlank()) {
            throw new IllegalStateException("Invalid name");
        }
    }
}

// 使用
User user = new User(1L, "ZhangSan", 25);
user.validate();
System.out.println(user.getIdString());  // "1"
```

### 3.3 Record 实现接口的最佳实践

```java
// 模式：Record + 接口分离

// 1. 定义纯数据 Record
public record User(Long id, String name, int age) {
    
    // 紧凑构造方法（做校验）
    public User {
        Objects.requireNonNull(id, "id cannot be null");
        Objects.requireNonNullElseGet(name, () -> "Unknown");
    }
}

// 2. 定义业务行为接口
public interface UserService {
    void createUser(User user);
    void updateUser(User user);
    void deleteUser(User user);
}

// 3. Record 可以在 lambda 中使用
public class UserProcessor {
    
    // Record 可以作为函数式接口的实现
    private final Function<User, String> formatter = User::toString;
    
    // Record 可以作为策略模式的数据载体
    public void processUsers(List<User> users, Consumer<User> action) {
        users.forEach(action);
    }
}

// 4. Record 与序列化
public record TransferEvent(
    String fromAccount,
    String toAccount,
    BigDecimal amount,
    Instant timestamp
) implements Serializable {
    
    // Record 隐式实现 Serializable
    // 所有字段必须可序列化
    // 如果字段是自定义类型，需要也实现 Serializable
}
```

## 4. Record 字段的特殊规则

### 4.1 字段修饰符限制

```java
// Record 字段的修饰符规则：

public record User(Long id, String name, int age) {}

// 等价于：
public final class User extends java.lang.Record {
    private final Long id;
    private final String name;
    private final int age;
    // ...
}

// Record 的字段天然是：
// private final（编译器强制）

// ❌ 不能加其他修饰符：
public record User(
    volatile Long id,       // ❌ volatile 不允许
    static String name,      // ❌ static 不允许
    public int age,          // ❌ public 不允许（字段）
    private String address  // ❌ private 不允许
) {}

// Record 可以有 static 字段（在 record 体内部）
public record User(Long id, String name) {
    public static final User UNKNOWN = new User(-1L, "Unknown");
    public static int MAX_NAME_LENGTH = 100;
}

// ❌ Record 不能有实例字段（只能在参数列表中）
public record User(Long id) {
    private String cache;  // ❌ 编译错误！不能有额外实例字段
    public static String globalCache;  // ✅ 可以有 static 字段
}
```

### 4.2 Record 与 volatile 字段

```
为什么 Record 字段不能用 volatile？

┌─────────────────────────────────────────────────────────┐
│  Record 的核心设计原则：不可变性（Immutability）        │
│                                                         │
│  Record 字段 = private final                           │
│  不可变 = 所有字段在构造后永不改变                     │
│                                                         │
│  volatile 的作用：                                      │
│  volatile = 保证多线程间的可见性                      │
│  可见性 = 一个线程修改后，其他线程立即可见            │
│                                                         │
│  关键逻辑：                                             │
│  如果字段是 final 的 → 构造后永不改变                │
│  如果永不改变 → 就不需要 volatile 保证可见性         │
│  所以 volatile 在 Record 字段上是多余的              │
│                                                         │
│  总结：                                                │
│  volatile ≠ Record 需要                               │
│  Record = 天然不可变 = 不需要 volatile               │
└─────────────────────────────────────────────────────────┘

但 Record 的组件（字段值）可以被其他线程安全地读取：
User user = new User(1L, "ZhangSan");
user.id();  // 线程 A 写
user.id();  // 线程 B 读 → 安全（final 保证了可见性）
```

## 5. Record 的高级用法

### 5.1 Record 的组件（Component）反射 API

```java
import java.lang.reflect.RecordComponent;

public record User(Long id, String name, int age) {}

public class RecordReflectionDemo {
    public static void main(String[] args) {
        // 获取 Record 的组件信息
        RecordComponent[] components = User.class.getRecordComponents();
        
        for (RecordComponent rc : components) {
            System.out.println("Component: " + rc.getName());
            System.out.println("  Type: " + rc.getType());
            System.out.println("  Getter: " + rc.getAccessor());
            System.out.println("  GenericType: " + rc.getGenericType());

// 输出：
// Component: id
//   Type: class java.lang.Long
//   Getter: public java.lang.Long User.id()
//   GenericType: class java.lang.Long
// Component: name

//   Getter: public java.lang.String User.name()
```

### 5.2 Record 与模式匹配（Pattern Matching）

```java
// Java 16+ Record 与 instanceof 模式匹配

// 传统写法
public void printUser(Object obj) {
    if (obj instanceof User) {
        User user = (User) obj;  // 需要强制转换
        System.out.println(user.name());
    }
}

// Java 16+ 模式匹配
public void printUser(Object obj) {
    if (obj instanceof User(Long id, String name, int age)) {
        // 直接解构，自动类型转换
        System.out.println(name + " (age: " + age + ")");
    }
}

// Java 21+ switch 模式匹配 + Record
public String describe(Object obj) {
    return switch (obj) {
        case User(Long id, String name, int age) 
            when age < 18 -> "未成年用户: " + name;
        case User(Long id, String name, int age) 
            when age < 60 -> "在职用户: " + name;
        case User(Long id, String name, int age) 
            -> "退休用户: " + name;
        case null -> "空对象";
        case default -> "未知类型";
    };
}
```

## 🔥 扩展追问 1：Record 和 Lombok @Data 一起用会冲突吗？

> **A:** 可以同时用，但大多数注解会冲突或多余：

```java
// ⚠️ Record + Lombok 共用的注意事项

// ❌ 冲突：Lombok 的 @Getter 会多余
// Record 已经自动生成所有 getter，不需要再写 @Getter
@Data  // 会尝试生成 setter（Record 没有 setter！）
public record User(Long id, String name) {}
// Lombok 可能会干扰 Record 的自动生成

// ✅ 合法的 Lombok + Record 组合
// 只用 Lombok 的 @Builder（生成 Builder 模式）
public record User(Long id, String name, int age) {
    
    @Builder.Default
    private static final int MAX_RETRY = 3;  // static 字段
}

// ✅ 用 Lombok 生成 Builder
public record Order(Long orderId, String product, int quantity) {
    
    // Lombok @Builder 可以在 record 上使用
    // 生成 OrderBuilder.builder() 工厂方法
}

// Record 的 Builder 模式
// 推荐使用三方库：https://github.com/Randgul/record-builder
import com.github.fwberriman.RecordBuilder;

@RecordBuilder
public record Point(int x, int y) {}

// 生成 PointBuilder，可以这样用：
Point p = PointBuilder.builder()
    .x(10)
    .y(20)
    .build();

// ✅ 最佳实践：纯 Record，不混用 Lombok
// Record 已经覆盖了 @Data 80% 的功能
// 需要其他功能时单独引入
```

## 🔥 扩展追问 2：Record 的 equals/hashCode/toString 是基于值还是引用？

> **A:** Record 的 equals/hashCode 是基于组件的**值比较**：

```java
public record User(Long id, String name) {}

// 值语义 vs 引用语义

User u1 = new User(1L, "ZhangSan");
User u2 = new User(1L, "ZhangSan");

// equals：值比较（内容相同 → true）
System.out.println(u1.equals(u2));  // true ✓

// == 引用比较（是否同一对象）
System.out.println(u1 == u2);  // false ✓

// hashCode：基于值
System.out.println(u1.hashCode() == u2.hashCode());  // true ✓

// toString：包含所有组件值
System.out.println(u1.toString());  // User[id=1, name=ZhangSan]
System.out.println(u2.toString());  // User[id=1, name=ZhangSan]

// ⚠️ 注意：引用类型的 equals
public record Broken(Long id, StringBuilder name) {}

StringBuilder sb1 = new StringBuilder("A");
StringBuilder sb2 = new StringBuilder("A");

Broken b1 = new Broken(1L, sb1);
Broken b2 = new Broken(1L, sb2);

b1.equals(b2);  // ❌ false！
// 因为 StringBuilder 没有重写 equals
// Record 的 equals 调用的是 StringBuilder 的 equals
// StringBuilder 继承了 Object 的 equals = 引用比较

// ✅ 正确做法：使用不可变对象作为字段类型
public record Good(Long id, String name) {}
// String 重写了 equals = 值比较
// 所以 Good 的 equals 正常工作
```

## 🔥 扩展追问 3：Record 的序列化有什么特殊之处？

> **A:** Record 的序列化是特殊处理过的：

```java
// Record 序列化特点

    private static final long serialVersionUID = 1L;

oos.writeObject(user);

    new ByteArrayInputStream(baos.toByteArray())
);
User restored = (User) ois.readObject();
```

```java
@GetMapping("/user/{id}")
public User getUser(@PathVariable Long id) {

@PostMapping("/user")
public void createUser(@RequestBody User user) {
    userService.create(user);

    return userService.update(id, req.id(), req.name());  // 传新 Record
```

> **一句话总结**

Java Record（JDK 16）是**不可变数据类**的语法糖，编译后生成继承自 `java.lang.Record` 的 `final` 类，自动生成 `constructor` + 组件名 getter + 值语义 `equals/hashCode/toString`；与 Lombok `@Data` 的本质区别在于 **Record 是语言内置（无需依赖）且天然不可变**（无 setter、字段强制 `private final`），而 Lombok `@Data` 是编译时注解（需要依赖、可生成可变字段）；Record **可以自定义构造方法**（紧凑形式或完整形式）且可 **implements 多个接口**，但 **不能 extends 其他类**（已隐式继承 Record 且自动 `final`），字段也 **不能用 `volatile` 修饰**（`final` 字段天然线程安全，不需要 volatile）。

---

> **面试加分话术**

> "我最近在项目里全面推行了 Record 替代 Lombok@Data 做 DTO，效果非常好。我们有个订单查询接口的 DTO 原来是 Lombok `@Data`，团队里有两个人习惯用 setter 改值，导致并发场景下数据错乱；改成 Record 后，编译期就禁止了 setter，数据不可变，线上这个 bug 彻底消失。另外，Spring Boot 3 配 Record 做配置类非常丝滑——只需要 `@ConfigurationProperties(prefix = 'app.user')` 加 `@ConfigurationPropertiesScan`，完全不需要 `@EnableConfigurationProperties`，Record 自动暴露构造函数给 Spring 绑定，代码从 30 行降到 1 行。但 Record 也有坑：Jackson 反序列化需要 2.14+ 版本，老项目升级时 Jackson 版本不够就会报错；另外 Record 不能有 `transient` 字段，如果某个 DTO 字段需要排除序列化就得用 `@JsonIgnore` 注解在 getter 上。Lombok 和 Record 混用也有问题——同一个字段 Lombok 尝试生成 getter，Record 也自动生成 getter，可能导致编译警告甚至冲突。我的经验是：**DTO/配置类/返回值用 Record，Domain 实体用 Lombok（需要 @Data），泾渭分明**。"
