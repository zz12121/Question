```lua
-- ========== EVAL 基本用法 ==========
EVAL "return redis.call('GET', KEYS[1])" 1 "user:100"
-- 返回: "value"

-- ========== 多参数传递 ==========
EVAL "
    local key = KEYS[1]
    local amount = tonumber(ARGV[1])
    local current = redis.call('GET', key) or '0'
    local new = tonumber(current) + amount
    redis.call('SET', key, new)
    return new

-- ========== 批量操作（原子性） ==========
-- Lua 脚本保证原子性，无需 MULTI/EXEC

    redis.call('EXPIRE', KEYS[1], ARGV[1])
    return n
" 1 "request:counter" 3600

-- 预加载
SCRIPT LOAD "return redis.call('GET', KEYS[1])"
```

### 🔥 扩展追问 2：Functions 的命令和使用方式？

> A: **FUNCTION LOAD/FCALL/FCALLRO 等命令管理函数。**

```lua
-- ========== 定义函数库 ==========
FUNCTION LOAD "
    #!lua name=my_library
    
    local function get_and_incr(key, ttl)
        local val = redis.call('GET', key)
        if val then
            redis.call('INCR', key)
            return val
        else
            redis.call('SET', key, 1, 'EX', ttl or 3600)
            return nil
        end
    end
    
    local function batch_get(keys)
        local result = {}
        for i, key in ipairs(keys) do
            result[i] = redis.call('GET', key)
        end
        return result
    end
    
    -- 注册函数
    redis.register_function('get_and_incr', get_and_incr)
    redis.register_function('batch_get', batch_get)
"
-- 返回: 1) "my_library"  2) ["get_and_incr", "batch_get"]

-- ========== 调用函数 ==========
-- FCALL: 调用可写函数
FCALL get_and_incr 1 "counter:test" 7200
-- 参数: 函数名, key数量, keys..., args...

-- FCALLRO: 调用只读函数（更快，无副本复制）
FCALL get_and_incr 1 "counter:test"  -- 错误，get_and_incr 写操作

-- 只读函数示例
FUNCTION LOAD "
    #!lua name=read_lib
    
    local function safe_get(key)
        return redis.call('GET', key)
    end
    
    local function check_exists(key)
        return redis.call('EXISTS', key)
    end
    
    redis.register_function{
        function_name='safe_get',
        callback=safe_get,
        flags={'no-writes'}
    }
    redis.register_function{
        function_name='check_exists',
        callback=check_exists,
        flags={'no-writes'}
    }
"

-- 只读调用（更快）
FCALLRO safe_get 1 "user:100"

-- ========== 函数管理命令 ==========
FUNCTION LIST
-- 返回所有已加载的函数库

FUNCTION LIST LIBRARY_NAME my_library
-- 返回指定库中的函数

FUNCTION LIST WITHCODE
-- 包含源码

FUNCTION DELETE my_library
-- 删除函数库

FUNCTION FLUSH
-- 清空所有函数

-- ========== 查看函数信息 ==========
FCALL get_and_incr 1 "counter:test" 7200
-- 如果函数不存在，返回错误
-- (error) ERR Function not found
```

### 🔥 扩展追问 3：Functions 的 flags 和权限控制？

> A: **通过 flags 控制函数的读写权限，以及超时和内存限制。**

```lua
-- ========== 注册只读函数 ==========
redis.register_function({
    callback = function(keys, args)
        return redis.call('GET', keys[1])
    end,
    function_name = 'get_value',
    flags = {'no-writes'}  -- 只读标记
})

-- ========== 注册允许写入函数 ==========
redis.register_function({
    callback = function(keys, args)
        return redis.call('SET', keys[1], args[1])
    end,
    function_name = 'set_value',
    flags = {}  -- 可写（默认）
})

-- ========== 全局变量限制 ==========
-- Functions 中不允许访问全局变量
-- 错误示例：
-- global_var = 1  -- 不允许

-- 正确做法：使用 local 变量
local function my_func()
    local temp = 1  -- OK
    -- ...
end

-- ========== 无 random 模式 ==========
-- Functions 中使用 math.random() 结果是确定性的
-- 用于保证复制一致性
```

### 🔥 扩展追问 4：EVALSHA vs Functions 性能对比和选择建议？

## 3. 一句话总结

> **EVALSHA 比 EVAL 高效在跳过脚本解析编译、网络传输更少；Lua 脚本有限制包括禁止 RANDOMKEY 等非确定性命令、超时控制、浮点数精度问题；Redis 7.0 Functions 是可持久化的函数库，通过 FUNCTION LOAD/FCALL 管理，支持库组织、flags 权限控制，比传统脚本更易复用和管理。**

## 4. 面试加分话术

> "我们项目在 Redis 5.6 升级到 7.0 时，把所有 Lua 脚本都迁移成了 Functions。主要收益：1）脚本定义在 Redis 端，不用每个语言的 SDK 都维护一份脚本代码了；2）用 `FCALLRO` 处理只读场景，性能提升 15% 左右；3）函数库按业务模块组织，比如 `order_lib`、`inventory_lib`，代码更清晰。踩过的坑：1）Functions 里不能访问全局变量，调试时想打印日志发现 `print()` 被禁了；2）`no-writes` flag 要仔细设置，不小心设错了在高并发下会触发奇怪的错误；3）Redis Cluster 模式下，所有节点都要加载相同的函数，用 `FCALL` 前先用 `FUNCTION LOAD` 确保函数存在。还有个经验：如果是 Redis Cluster，`EVALSHA` 需要用 `EVALSHA_RO` 只读版本，不然会报 CROSSSLOT 错误。"

---

**追问预判：**
- "Lua 脚本超时了怎么办？" → 可以用 `SCRIPT KILL` 杀死脚本，但如果是写操作会导致数据部分更新
- "Redis Cluster 下 Lua 脚本有什么限制？" → 所有 key 必须在同一个 slot（或者用 hash tags），否则报 CROSSSLOT 错误
