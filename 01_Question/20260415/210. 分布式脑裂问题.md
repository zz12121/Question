## 1. 基础流程

### 1.1 什么是脑裂？

```
脑裂（Split-Brain）定义：

网络分区后，集群被分割成两个或多个独立的子集群
每个子集群认为自己是"主"，继续提供服务
导致数据不一致，甚至数据损坏

形象比喻：
┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│   正常状态：                                                               │
│                                                                            │
│        ┌─────────────────────────────────────────────────────────┐         │
│        │                    集群 (3节点)                         │         │
│        │                                                         │         │
│        │   Node1 ──选举──▶ Leader                               │         │
│        │   Node2              ↑                                 │         │
│        │   Node3 ────────────┼───────── 追随 Leader               │         │
│        │                                                         │         │
│        └─────────────────────────────────────────────────────────┘         │
│                                                                            │
│   脑裂发生：                                                               │
│                                                                            │
│        ┌────────────────────┐      ┌────────────────────┐                │
│        │   分区 A (2节点)    │      │   分区 B (1节点)    │                │
│        │                    │      │                    │                │
│        │ Node1 ──选举──▶ A-Leader  │                    │                │
│        │ Node2 ───────────▶ A-Leader│ Node3 ──选举──▶ B-Leader          │
│        │                    │      │                    │                │
│        │ [继续服务，可写]    │      │ [继续服务，可写]    │                │
│        │                    │      │                    │                │
│        │ 数据版本: V1       │      │ 数据版本: V2       │                │
│        │ 客户余额: 1000     │      │ 客户余额: 800      │                │
│        └────────────────────┘      └────────────────────┘                │
│                    ⬆                               ⬆                       │
│                    └─────────────┬─────────────────┘                       │
│                                  │                                         │
│                         网络中断 / 脑裂发生！                              │
└────────────────────────────────────────────────────────────────────────────┘
```

### 1.3 ZooKeeper 过半数投票机制

```
ZooKeeper 的 Quorum 机制：

┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│  核心原则：只有获得多数（Majority/Najor）票数的节点才能成为 Leader          │
│                                                                            │
│  ┌────────────────────────────────────────────────────────────────────┐   │
│  │                                                                    │   │
│  │   ZooKeeper 集群：                                                 │   │
│  │                                                                    │   │
│  │   节点数    Quorum(过半数)   容错能力                               │   │
│  │   ┌────────┬─────────────────┬─────────────┐                        │   │
│  │   │   1    │       1        │   0 节点    │ ← 不可用               │   │
│  │   ├────────┼─────────────────┼─────────────┤                        │   │
│  │   │   2    │       2        │   0 节点    │ ← 不可用               │   │
│  │   ├────────┼─────────────────┼─────────────┤                        │   │
│  │   │   3    │       2        │   1 节点    │ ← 最多容1节点故障     │   │
│  │   ├────────┼─────────────────┼─────────────┤                        │   │
│  │   │   4    │       3        │   1 节点    │ ← 最多容1节点故障     │   │
│  │   ├────────┼─────────────────┼─────────────┤                        │   │
│  │   │   5    │       3        │   2 节点    │ ← 最多容2节点故障     │   │
│  │   ├────────┼─────────────────┼─────────────┤                        │   │
│  │   │   7    │       4        │   3 节点    │ ← 最多容3节点故障     │   │
│  │   └────────┴─────────────────┴─────────────┘                        │   │
│  │                                                                    │   │
│  └────────────────────────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────────────────────────┘

选举过程（ZAB 协议）：

┌────────────────────────────────────────────────────────────────────────────┐
│  假设 5 节点集群，Node1 是 Leader，Node1 挂了                                 │
│                                                                            │
│  Phase 1: 候选阶段（Looking）                                              │
│  ──────────────────────────────────────────────────────────────────────── │
│  所有节点状态变为 LOOKING，开始选举                                         │
│  每个节点投票：(myid, zxid, epoch)                                         │
│      myid   = 节点编号（越大越优先）                                        │
│      zxid   = 最新事务 ID（越大数据越新）                                    │
│      epoch  = 投票轮次                                                      │
│                                                                            │
│  Node3 投票: (3, 100, 5)                                                   │
│  Node4 投票: (4, 99, 5)    ← myid 更大，更优先                             │
│  Node5 投票: (5, 98, 5)    ← myid 最大，最优先                            │
│                                                                            │
│  Phase 2: 收集阶段                                                         │
│  ──────────────────────────────────────────────────────────────────────── │
│  各节点收集所有投票，比较优先级                                            │
│  Node3: 收到 (4,99,5) 和 (5,98,5)，发现不如自己的(3,100,5)                 │
│         保持投自己 (3, 100, 5)                                            │
│                                                                            │
│  Phase 3: 统计阶段                                                         │
│  ──────────────────────────────────────────────────────────────────────── │
│  Node3 收到 2 票（自己+Node4），共 3 票                                    │
│  Node4 收到 2 票（自己+Node3），共 3 票                                   │
│  Node5 收到 1 票（只有自己）                                                │
│                                                                            │
│  结果：Node3 和 Node4 各 3 票，达到 quorum=3，Node3 成为 Leader          │
└────────────────────────────────────────────────────────────────────────────┘
```

### 1.4 Redis Sentinel 参数详解

```
Redis Sentinel 核心参数：

┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│  sentinel monitor <master-name> <ip> <port> <quorum>                      │
│                                                                            │
│  <quorum>:                                                                │
│    含义：认定 Master 主观下线（SDOWN）所需的 Sentinel 最少票数                │
│    默认：2                                                                │
│    建议：>= (Sentinel总数 / 2 + 1)                                         │
│                                                                            │
│    例如：3个 Sentinel                                                      │
│    quorum=2：至少2个 Sentinel 认为 Master 下线才触发客观下线（ODOWN）       │
│    quorum=1：只要1个 Sentinel 认为 Master 下线就触发 ODOWN（危险！）       │
│                                                                            │
├────────────────────────────────────────────────────────────────────────────┤
│                                                                            │
│  sentinel down-after-milliseconds <master-name> <milliseconds>              │
│                                                                            │
│  含义：Master 被认为"主观下线"的超时时间                                    │
│  默认：30000ms (30秒)                                                      │
│                                                                            │
│  配合 quorum 使用：                                                        │
│                                                                            │
│  ┌────────────────────────────────────────────────────────────────────┐  │
│  │  timeline:                                                        │  │
│  │                                                                    │  │
│  │  t=0ms:     Master 无响应                                          │  │
│  │  t=30000ms: Sentinel1 认为 Master SDOWN（等待 quorum）              │  │
│  │  t=35000ms: Sentinel2 认为 Master SDOWN（凑齐2票）                  │  │
│  │  t=35000ms: 触发 ODOWN（客观下线），开始选举                        │  │
│  │                                                                    │  │
│  │  注意：如果 Sentinel1 和 Sentinel2 在不同机房                        │  │
│  │        可能是网络分区，不是真的 Master 挂了                         │  │
│  └────────────────────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────────────────────┘

Sentinel 完整配置示例：

# sentinel.conf

# 监控主节点
sentinel monitor mymaster 127.0.0.1 6379 2

# 主观下线时间（30秒无响应）
sentinel down-after-milliseconds mymaster 30000

# 故障转移超时时间
sentinel failover-timeout mymaster 180000

# 故障转移时，同时同步新主服务器的从服务器数量
# 建议：不要太多，否则从服务器短时间内流量激增
sentinel parallel-syncs mymaster 1

# 认证密码
sentinel auth-pass mymaster mystrongpassword

# 通知脚本（可选）
sentinel notification-script mymaster /path/to/notify.sh
sentinel client-reconfig-script mymaster /path/to/reconfig.sh
```

## 🔥 扩展追问 1：Redis Sentinel 的 `quorum` 和 `parallel-syncs` 怎么配合设置？

> A: **quorum 决定"谁说了算"，parallel-syncs 决定"故障转移多快"。quorum 太小会放大脑裂风险，parallel-syncs 太大导致从节点雪崩。**

```
quorum 设置策略：

┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│  Sentinel 数 = 3 时：                                                      │
│  ┌───────────────┬────────────────────┬────────────────────────────────┐  │
│  │  quorum=1    │  极危险 ⚠️          │  任何1个Sentinel都能触发切换   │  │
│  │              │  容易产生双主       │  网络抖动就切换                │  │
│  ├───────────────┼────────────────────┼────────────────────────────────┤  │
│  │  quorum=2    │  推荐 ✅            │  至少2票才触发切换             │  │
│  │              │  平衡安全性和可用性  │  容忍1个Sentinel故障          │  │
│  ├───────────────┼────────────────────┼────────────────────────────────┤  │
│  │  quorum=3    │  过度安全 ⚠️        │  必须所有Sentinel同意         │  │
│  │              │  任何1个Sentinel挂了 │  无法触发切换                  │  │
│  │              │  集群不可用          │                               │  │
│  └───────────────┴────────────────────┴────────────────────────────────┘  │
│                                                                            │
│  Sentinel 数 = 5 时：                                                      │
│  ┌───────────────┬────────────────────────────────────────────────────┐  │
│  │  quorum=2     │  危险 ⚠️  容忍3个节点故障，但容易双主               │  │
│  │  quorum=3     │  推荐 ✅  平衡安全性和可用性                         │  │
│  │  quorum=4     │  安全但可用性低                                      │  │
│  └───────────────┴────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────────────────────┘

parallel-syncs 设置策略：

┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│  sentinel parallel-syncs mymaster 1                                        │
│                                                                            │
│  含义：故障转移时，同时有多少个从服务器可以重新配置新主                      │
│                                                                            │
│  parallel-syncs=1（推荐）：                                                │
│  ┌────────────────────────────────────────────────────────────────────┐   │
│  │  Master 挂了                                                          │   │
│  │      │                                                               │   │
│  │      ├── 从1 ──同步新主──▶ 完成 ──上线 ✅                           │   │
│  │      ├── 从2 ──等待 ──同步新主──▶ 完成 ──上线 ✅                   │   │
│  │      └── 从3 ──等待 ──等待 ──同步新主──▶ 完成 ──上线 ✅           │   │
│  │                                                                      │   │
│  │  优点：逐个同步，Master 压力小                                       │   │
│  │  缺点：所有从节点恢复时间长                                         │   │
│  └────────────────────────────────────────────────────────────────────┘   │
│                                                                            │
│  parallel-syncs=3（不推荐）：                                             │
│  ┌────────────────────────────────────────────────────────────────────┐   │
│  │  Master 挂了                                                          │   │
│  │      │                                                               │   │
│  │      ├── 从1 ──同步新主──▶ Master 压力: 3倍！ ❌                   │   │
│  │      ├── 从2 ──同步新主──▶ Master 可能 OOM                         │   │
│  │      └── 从3 ──同步新主──▶ 雪崩                                     │   │
│  │                                                                      │   │
│  │  优点：恢复快                                                         │   │
│  │  缺点：Master 被打死                                                  │   │
│  └────────────────────────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 2：双主（双写双活）场景下脑裂的后果是什么？

> A: **双主会导致数据双向写入，最终数据不一致、版本冲突、甚至数据损坏，是最严重的脑裂后果。**

```
双主脑裂场景：

┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│  正常状态：                                                                │
│                                                                            │
│     应用1 ──▶ Master-A (写 key:counter=100)                               │
│     应用2 ──▶ Master-A (读 key:counter=100) ✅                            │
│                                                                            │
│  脑裂发生：                                                                │
│                                                                            │
│     应用1 ──▶ Master-A (写 key:counter=110)                               │
│     应用2 ──▶ Master-B (写 key:counter=120)  ← 网络分区，对端不可达     │
│                                                                            │
│     分区合并后：                                                           │
│                                                                            │
│     Master-A: counter=110                                                 │
│     Master-B: counter=120                                                 │
│     ← 数据冲突！Redis 使用异步复制，Master-B 不知道有 110 这个版本        │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

数据不一致的几种情况：

┌────────────────────────────────────────────────────────────────────────────┐
│  情况1：计数器类数据（最严重）                                             │
│                                                                            │
│     脑裂期间：                                                             │
│     Master-A: counter = 110 (+10笔订单)                                    │
│     Master-B: counter = 120 (+20笔订单)                                   │
│                                                                            │
│     合并后：                                                               │
│     Master-A 成为主，counter=110                                            │
│     Master-B 成为从，同步后 counter=110（丢失20笔！）                      │
│                                                                            │
│     结果：用户付款了 20 笔，账户只扣了 10 笔的款 ❌                        │
│                                                                            │
├────────────────────────────────────────────────────────────────────────────┤
│  情况2：库存扣减（超卖）                                                   │
│                                                                            │
│     库存=100，100人下单                                                    │
│     Master-A 卖了 50 单 → 库存=50                                         │
│     Master-B 卖了 60 单 → 库存=40                                         │
│                                                                            │
│     合并后：                                                               │
│     库存=40，50人拿到订单，60人拿到订单，共110单！                         │
│     实际只有100件货 ❌ 超卖 10 件                                         │
│                                                                            │
├────────────────────────────────────────────────────────────────────────────┤
│  情况3：幂等性破坏                                                         │
│                                                                            │
│     订单号生成器：                                                         │
│     Master-A 生成了 order:1001, order:1002                               │
│     Master-B 也生成了 order:1001, order:1002（相同！）                    │
│                                                                            │
│     合并后：                                                               │
│     两边都有 order:1001，但内容不同 ❌                                     │
│     主键冲突或数据覆盖                                                     │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 3：ZooKeeper 的 Leader 选举如何防止脑裂？为什么奇数节点比偶数节点好？

> A: **ZooKeeper 通过"过半数才能工作"的机制确保任意时刻最多只有一个分区能工作；奇数节点比偶数节点有更好的容错性。**

```
为什么过半数能防止脑裂：

┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│  场景：5节点集群，网络分区                                                  │
│                                                                            │
│  ┌──────────────────────┐  ┌──────────────────────┐                      │
│  │  分区A: 3节点        │  │  分区B: 2节点         │                      │
│  │                      │  │                       │                      │
│  │  3 >= 5/2+1=3 ✅    │  │  2 < 3 ❌             │                      │
│  │  可以选举，可以服务  │  │  无法选举，无法服务   │                      │
│  └──────────────────────┘  └──────────────────────┘                      │
│                                                                            │
│  结果：只有分区A继续服务，不会产生双主 ✅                                  │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘
```

> A: **业务层的脑裂应对策略包括：数据版本化 + CAS 乐观锁、幂等设计、降级、数据校验与修复。**

```java
// 策略1：数据版本化 + CAS 乐观锁
public boolean transfer(String accountId, BigDecimal amount) {
    String key = "account:" + accountId;
    
    while (true) {
        // 1. 获取当前值和版本
        Account account = redisTemplate.opsForValue().get(key);
        BigDecimal currentBalance = account.getBalance();
        Long version = account.getVersion();
        
        // 2. 计算新值
        BigDecimal newBalance = currentBalance.add(amount);
        
        // 3. CAS 更新（版本匹配才更新）
        Boolean success = redisTemplate.opsForValue().setIfAbsent(
            key,
            new Account(newBalance, version + 1),
            Duration.ofHours(1),
            RedisSetOptions.setOptions(true)  // onlyIfPresent
        );
        
        if (Boolean.TRUE.equals(success)) {
            return true;  // 更新成功
        }
        
        // 版本冲突，重试
        try {
            Thread.sleep(10);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}
```

```java
// 策略2：幂等设计（防重复操作）
public String createOrder(OrderRequest request) {
    // 生成幂等 key（基于业务唯一键）
    String idempotentKey = "idempotent:order:" + 
        request.getUserId() + ":" + request.getProductId() + ":" + request.getRequestId();
    
    // 1. 检查是否已处理
    String existingOrderId = redisTemplate.opsForValue().get(idempotentKey);
    if (existingOrderId != null) {
        log.warn("重复请求，直接返回已有订单: {}", existingOrderId);
        return existingOrderId;
    }
    
    // 2. SETNX 幂等锁（10秒过期）
    Boolean acquired = redisTemplate.opsForValue().setIfAbsent(
        idempotentKey, "PROCESSING", Duration.ofSeconds(10));
    
    if (!Boolean.TRUE.equals(acquired)) {
        throw new BusinessException("请求正在处理中");
    }
    
    try {
        // 3. 执行业务
        String orderId = doCreateOrder(request);
        
        // 4. 存储结果（保留30天）
        redisTemplate.opsForValue().set(idempotentKey, orderId, Duration.ofDays(30));
        
        return orderId;
    } finally {
        // 不删除 idempotentKey，保留用于幂等查询
    }
}
```

```java
// 策略3：降级方案（脑裂期间只读）
public class SplitBrainSafeService {
    
    // 脑裂检测：最近一次主从同步时间
    private volatile long lastSyncTime = 0;
    
    @Autowired
    private RedisConnectionFactory factory;
    
    public boolean isDataFresh() {
        // 如果超过 30 秒没同步，认为数据可能过期
        return (System.currentTimeMillis() - lastSyncTime) < 30_000;
    }
    
    public Result queryData(String key) {
        if (!isDataFresh()) {
            // 数据可能过期，尝试降级
            log.warn("数据可能过期，降级为只读模式");
            return queryFromBackup(key);
        }
        return queryFromPrimary(key);
    }
    
    public void writeData(String key, Object value) {
        if (!isDataFresh()) {
            // 拒绝写入，防止双主导致的数据不一致
            throw new BusinessException("系统处于降级模式，暂停写入");
        }
        writeToPrimary(key, value);
    }
    
    // 定期更新同步时间（主从复制成功时触发）
    @Scheduled(fixedRate = 1000)
    public void updateSyncTime() {
        // 通过 INFO replication 获取主从延迟
        Connection conn = factory.getConnection();
        String info = conn.serverCommands().info("replication");
        // 解析 offset，计算同步时间
        lastSyncTime = System.currentTimeMillis();
        conn.close();
    }
}
```

## 🔥 扩展追问 5：Nacos 的 CP+AP 双模式是怎么处理脑裂的？

> A: **Nacos 默认 AP 模式允许注册表短暂不一致以保证可用性；CP 模式使用 Raft 协议，通过 Leader 选举保证强一致性，但网络分区时无法服务。**


## 3. 一句话总结

> **脑裂是网络分区导致的"多主并立"现象，ZooKeeper 通过过半数投票（quorum >= N/2+1）保证最多一个分区能工作，Redis Sentinel 通过 quorum+down-after-milliseconds 组合减少误判，生产环境应避免双主双活，业务层通过版本化 CAS、幂等设计、数据校验兜底应对。**

## 4. 面试加分话术

> "我之前踩过一次 Redis Sentinel 脑裂的坑：线上 3 节点 Sentinel，quorum=2，down-after-milliseconds=30000，Master 在凌晨 3 点抖动了一下，一个 Sentinel 认为它 down 了，触发了故障转移。等网络恢复时，原 Master 还在写数据，同时新主也在写，等运维发现的时候已经有 10 万条数据不一致了。后来复盘，我们的解法是：quorum 改成 2 但要求 down-after-milliseconds=60000，减少网络抖动的误判；同时加了运维流程——故障转移后运维必须立即确认原主是否真的挂了，没确认前不允许业务写原主。业务层我们也加了兜底：所有写操作都带版本号，Redis SET 时用 `SET key value XX NX` 防止覆盖，库存扣减全部走 Lua 脚本保证原子性。配置中心的选型也改了，服务发现用 Nacos AP 保证可用，配置中心用 Nacos CP 保证一致性，数据校验定时任务每 5 分钟跑一次，一不一致就能看到。"

```java
// 业务层防脑裂代码
@Aspect
@Component
public class SplitBrainProtectionAspect {
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    @Around("@annotation(SplitBrainSafe)")
    public Object around(ProceedingJoinPoint point) throws Throwable {
        // 1. 检查主从延迟
        ReplicationInfo info = getReplicationInfo();
        if (info.getLag() > 1000) {
            log.warn("主从延迟过大: {}ms，拒绝写操作", info.getLag());
            throw new ServiceUnavailableException("数据同步延迟过高");
        }
        
        // 2. 检查是否有故障转移正在进行
        if (isFailoverInProgress()) {
            throw new ServiceUnavailableException("故障转移中，暂停写入");
        }
        
        return point.proceed();
    }
}
```
