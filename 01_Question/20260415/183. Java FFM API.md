## 基础流程

### 1. 什么是 FFM API？

FFM API 是 **JDK 22 正式特性**（历经 JEP 412→416→424→426→429→434→454 的演进），用于 **Java 程序直接调用 native 代码和操作堆外内存**，无需 JNI 胶水代码。

### 2. FFM API 取代了什么？

| API | 时代 | 能力 | 痛点 |
|-----|------|------|------|
| **JNI** | JDK 1.1+ | 调用 C/C++ 库 | 需写 C Header、编译器、脱离 JVM 内存管理 |
| **Unsafe** | JDK 9-21 | 堆外内存 + 原子操作 | 私有 API（`--add-opens`）、无安全检查 |
| **JOL** | 工具库 | 内存布局分析 | 仅分析，不能操作 |
| **DirectByteBuffer** | JDK 1.4+ | 堆外 Buffer | GC 清理慢，无自定义分配器 |
| **FFM API** | JDK 22+ | **统一接口**：调用 native + 操作堆外内存 | 学习曲线 |

### 3. MemorySegment（内存段）

```java
// MemorySegment 是堆外内存的统一抽象
public interface MemorySegment {

    // 1. 堆内数组 → 内存段（零拷贝视图）
    byte[] heapArray = new byte[1024];
    MemorySegment fromArray = MemorySegment.ofArray(heapArray);
    // 等价于：ByteBuffer.wrap(heapArray).asReadOnlyBuffer()

    // 2. 本地内存分配（Arena 管理）
    Arena arena = Arena.ofAuto();  // 自动管理
    MemorySegment nativeSegment = arena.allocate(1024);

    // 3. 内存映射文件
    MemorySegment mapped = MemorySegment.mapFile(
        Path.of("data.bin"), 0, 1024,
        FileChannel.MapMode.READ_WRITE, arena
    );

    // 4. 读写操作（类型安全）
    VarHandle intHandle = MemoryLayout.sequenceElement()
        .withBitAlignment(32)
        .withOrder(ByteOrder.BIG_ENDIAN);

    // 设置值（无锁 CAS）
    intHandle.setVolatile(nativeSegment, 0L, 42);

    // 获取值
    int value = (int) intHandle.getVolatile(nativeSegment, 0L);
}
```

### 4. Arena（内存分配器）

```java
// Arena 是 FFM API 的核心创新：托管式堆外内存分配

public class ArenaDemo {

    public static void main(String[] args) {
        // 1. 自动 Arena（JVM 退出时释放，最安全）
        try (Arena autoArena = Arena.ofAuto()) {
            MemorySegment autoSeg = autoArena.allocate(1024);
            System.out.println("自动管理，无需手动释放");
        }

        // 2. 受限 Arena（当前线程独占，性能最优）
        try (Arena confined = Arena.ofConfined()) {
            MemorySegment confinedSeg = confined.allocate(1024);
            // 仅限当前线程使用
            fillSegment(confinedSeg);
        }

        // 3. 共享 Arena（多线程共享，需显式 close()）
        Arena shared = Arena.ofShared();
        ExecutorService executor = Executors.newFixedThreadPool(2);
        
        try {
            Future<?> f1 = executor.submit(() -> {
                MemorySegment s1 = shared.allocate(256);
                // 线程1使用
            });
            Future<?> f2 = executor.submit(() -> {
                MemorySegment s2 = shared.allocate(256);
                // 线程2使用
            });
            f1.get();
            f2.get();
        } finally {
            shared.close(); // 显式释放
            executor.shutdown();
        }

        // 4. Native Arena（直接调用系统 malloc/free）
        try (Arena nativeArena = Arena.ofConfined()) {
            // 获取系统 malloc
            SymbolLookup libc = SymbolLookup.loaderLookup()
                .or(SymbolLookup.libraryLookup("msvcrt.dll")); // Windows
            
            MemorySegment malloc = libc.find("malloc")
                .orElseThrow(() -> new UnsatisfiedLinkError("malloc not found"));
            
            // 调用 malloc(1024)
            long ptr = (long) C_LONG
                .outLoader()
                .invoke(unsafe_alloc, 1024L);
        }
    }

    private static void fillSegment(MemorySegment seg) {
        // 使用 MemoryAccess 便捷类
        for (long i = 0; i < 1024; i++) {
            MemoryAccess.setByteAtOffset(seg, i, (byte) (i % 256));
        }
    }
}
```

### 5. 直接调用 C 函数

```java
// 完整示例：调用 C 的 malloc / memcpy / free

public class CFunctionInvoke {

    // Step 1: 定义 C 函数签名（使用 C ABI 布局）
    private static final SystemABI ABI = System.getSystemModule()
        .map(mod -> mod.getLayer().findModule("java.base")
            .flatMap(m -> m.getLayer().findModule("java.base")))
        .flatMap(m -> m.flatMap(SystemABI::of))
        .orElseThrow();

    // 或者更简洁的方式：
    static {
        System.loadLibrary("native"); // 加载 native 库
    }

    public static void main(String[] args) throws Throwable {
        // 获取 libc 的符号查找器
        SymbolLookup stdlib = SymbolLookup.libraryLookup(
            Platform.isWindows() ? "msvcrt.dll" : "libc.so.6",
            Arena.ofAuto()
        );

        // 查找 malloc / free / memcpy
        MemorySegment mallocAddr = stdlib.find("malloc")
            .orElseThrow();
        MemorySegment freeAddr = stdlib.find("free")
            .orElseThrow();
        MemorySegment memcpyAddr = stdlib.find("memcpy")
            .orElseThrow();

        // 定义方法描述符
        // malloc(size_t) -> void*
        FunctionDescriptor mallocDesc = FunctionDescriptor.of(
            ValueLayout.ADDRESS,  // 返回值：void*
            ValueLayout.JAVA_LONG   // 参数：size_t
        );
        
        // free(void*)
        FunctionDescriptor freeDesc = FunctionDescriptor.of(
            ValueLayout.JAVA_VOID, // 返回值：void
            ValueLayout.ADDRESS    // 参数：void*
        );

        // memcpy(dst, src, n)
        FunctionDescriptor memcpyDesc = FunctionDescriptor.of(
            ValueLayout.ADDRESS,
            ValueLayout.ADDRESS,   // dst
            ValueLayout.ADDRESS,    // src
            ValueLayout.JAVA_LONG   // n
        );

        // 获取可调用方法句柄
        MethodHandle malloc = Linker.nativeLinker()
            .downcallHandle(mallocAddr, mallocDesc);
        
        MethodHandle free = Linker.nativeLinker()
            .downcallHandle(freeAddr, freeDesc);

        // 调用 malloc(1024)
        try (Arena arena = Arena.ofAuto()) {
            // 分配本地内存
            MemorySegment mallocedMem = arena.allocate(1024);

            // 写入数据
            MemorySegment src = arena.allocateFrom("Hello, FFM!".getBytes());
            
            // 调用 memcpy
            MethodHandle memcpy = Linker.nativeLinker()
                .downcallHandle(memcpyAddr, memcpyDesc);
            
            memcpy.invoke(mallocedMem, src, 11L);

            // 读取结果
            byte[] result = mallocedMem.toArray(ValueLayout.JAVA_BYTE);
            System.out.println(new String(result)); // "Hello, FFM!"
        }
    }
}
```

```java
public class CStructLayout {

    // C: struct Point { int x; int y; double z; }
    static final GroupLayout POINT = MemoryLayout.structLayout(
        ValueLayout.JAVA_INT.withName("x"),
        ValueLayout.JAVA_INT.withName("y"),
        ValueLayout.JAVA_DOUBLE.withName("z")
```

## 🔥 扩展追问 1：FFM API vs JNI 性能对比？

```java
// 性能对比测试框架
public class PerfBenchmark {

    // JNI 方式：Java 调用 native 方法（已有 C 桥接代码）
    public native long jniMalloc(long size);
    static { System.loadLibrary("perf"); }

    public static void main(String[] args) throws Exception {
        int iterations = 1_000_000;
        Arena arena = Arena.ofAuto();

        // JNI 路径
        PerfBenchmark jni = new PerfBenchmark();
        long start = System.nanoTime();
        for (int i = 0; i < iterations; i++) {
            long ptr = jni.jniMalloc(64);
            // JNI 侧需手动 free
        }
        long jniTime = System.nanoTime() - start;

        // FFM 路径
        MemorySegment segment = arena.allocate(64);
        long ffmStart = System.nanoTime();
        for (int i = 0; i < iterations; i++) {
            // 直接操作，无需跨 JVM/native 边界
            MemoryAccess.setByteAtOffset(segment, 0, (byte) i);
        }
        long ffmTime = System.nanoTime() - ffmStart;

        System.out.println("JNI: " + jniTime / 1_000_000 + " ms");
        System.out.println("FFM: " + ffmTime / 1_000_000 + " ms");
    }
}
```

> **A:** FFM 性能优势来自：
> 1. **消除胶水代码**：JNI 需要 `javac -h` 生成 Header + 手写 `.c` 实现；FFM 纯 Java 声明，运行时动态生成桩代码
> 2. **减少边界穿越**：JNI 每次调用需 `JavaVM->CallNativeMethod()` 上下文切换；FFM 在**首次调用后缓存方法句柄**，后续直接跳转
> 3. **内存布局优化**：`MemoryLayout` 让 JVM 直接操作 native 内存格式，零拷贝
> 4. **逃逸分析友好**：`Arena` 分配的内存不在 GC 堆，避免 GC 暂停对 native 代码的干扰
> 5. **实测数据**：简单 malloc/free 场景，FFM 比 JNI 快 **30-50%**（无 C++ 业务逻辑时）

## 🔥 扩展追问 2：FFM API 如何调用 Linux 系统调用？

```java
// 调用 Linux read(int fd, void* buf, size_t count)
public class LinuxSyscall {

    static final SystemABI ABI = CLinker.systemCLinker();

    public static void main(String[] args) throws Throwable {
        // 加载 libc
        SymbolLookup libc = SymbolLookup.libraryLookup(
            "libc.so.6", Arena.ofAuto()
        );

        // 定义 read 系统调用签名
        // ssize_t read(int fd, void* buf, size_t count)
        FunctionDescriptor readDesc = FunctionDescriptor.of(
            ValueLayout.JAVA_LONG,    // ssize_t
            ValueLayout.JAVA_INT,      // int fd
            ValueLayout.ADDRESS,       // void* buf
            ValueLayout.JAVA_LONG      // size_t count
        );

        // 定义 write 系统调用签名
        // ssize_t write(int fd, const void* buf, size_t count)
        FunctionDescriptor writeDesc = FunctionDescriptor.of(
            ValueLayout.JAVA_LONG,
            ValueLayout.JAVA_INT,
            ValueLayout.ADDRESS,
            ValueLayout.JAVA_LONG
        );

        // 获取方法句柄
        MethodHandle read = ABI.downcallHandle(
            libc.find("read").get(), readDesc
        );
        MethodHandle write = ABI.downcallHandle(
            libc.find("write").get(), writeDesc
        );

        // 使用 try-with-resources 自动管理内存
        try (Arena arena = Arena.ofAuto()) {
            MemorySegment buf = arena.allocate(1024);

            // read(STDIN_FILENO=0, buf, 1024)
            long bytesRead = (long) read.invoke(0, buf, 1024L);
            if (bytesRead > 0) {
                // write(STDOUT_FILENO=1, buf, bytesRead)
                write.invoke(1, buf, bytesRead);
            }
        }
    }
}
```

> **A:** 
> - FFM 可以直接调用**任何动态链接库**中的函数，包括 glibc、libc、libpthread 等
> - `SymbolLookup.libraryLookup(path, arena)` 加载 `.so`/`.dll`/`.dylib`
> - `SymbolLookup.loaderLookup()` 查找 `System.loadLibrary()` 加载的库
> - 系统调用本质也是 C 库函数（`read/write/open/close`），通过 libc 间接调用

## 🔥 扩展追问 3：FFM API 与 Unsafe 的关系？Unsafe 会被移除吗？

```java
// Unsafe 的典型用法（即将废弃）
public class UnsafeDemo {
    private static final Unsafe UNSAFE = getUnsafe();

    public long getFieldOffset(Class<?> clazz, String fieldName) {
        return UNSAFE.objectFieldOffset(
            clazz.getDeclaredField(fieldName)
        );
    }

    // FFM 替代 Unsafe 的能力映射
    public class FFMReplacement {
        // Unsafe: 获取字段偏移量
        // FFM: 使用 MemoryLayout.pathsGroupElement("fieldName")
        
        // Unsafe: compareAndSwapObject
        // FFM: VarHandle.withInvokeExact() + MemorySegment
        
        // Unsafe: allocateInstance（绕过构造方法）
        // FFM: 需要通过 JNI 或其他方式（暂无直接替代）
        
        // Unsafe: putLong / getLong（堆外内存）
        // FFM: MemorySegment + ValueLayout.JAVA_LONG
    }
}
```

> **A:** 
> - **Unsafe 不会被立即移除**，但会逐步限制使用（`--illegal-access` 警告）
> - FFM 是 Unsafe 的**官方替代方案**：同等的堆外内存操作能力，但**类型安全、可审计、编译器支持**
> - FFM **不提供** `allocateInstance()`（绕过构造方法），这是设计决定
> - **迁移策略**：新代码用 FFM，老代码中的 Unsafe 在 JDK 21+ 会收到弃用警告

## 🔥 扩展追问 4：FFM API 的安全限制？

```java
// FFM API 安全模型
public class FFMSecurity {

    public static void main(String[] args) {
        // 1. 默认情况下，FFM 需要 --enable-native-access
        // java --enable-native-access=ALL-UNNAMED MyApp
        
        // 2. Module-info.java 中声明
        // module my.app {
        //     opens my.app to jdk.incubator.foreign;
        //     requires jdk.incubator.foreign;
        // }

        // 3. 权限检查点
        // - 调用 native 代码：需要 JNIHandle
        // - 分配 native 内存：通过 Arena（受限于配置的内存上限）
        // - 内存映射文件：受操作系统权限约束
    }

    // FFM 的安全边界
    static final class SecurityBoundaries {
        // ✅ 安全操作
        // - 通过 Linker.downcallHandle() 调用已声明的 C 函数
        // - 通过 Arena.allocate() 分配受控的堆外内存
        
        // ⚠️ 需要显式授权
        // - --enable-native-access
        // - 模块权限声明
        
        // ❌ 禁止操作
        // - 访问任意内存地址（Unsafe 可以，FFM 不行）
        // - 调用未在 SymbolLookup 中声明的符号
    }
}
```

> **A:** 
> - FFM 需要 `--enable-native-access=ALL-UNNAMED` 或模块声明才能使用
> - **相比 Unsafe**：Unsafe 可以访问任意内存地址，FFM 只能访问 `MemorySegment` 分配的内存
> - **相比 JNI**：JNI 可以调用任何已加载的 native 函数，FFM 需要显式 `SymbolLookup`
> - **安全优势**：类型安全 + 范围检查 + 编译器校验，而非 Unsafe 的"万能钥匙"

```java
    );

    public void serialize(Person p, MemorySegment buf) {
        PERSON_LAYOUT.set(ValueLayout.JAVA_LONG, buf, 0L, "id", p.id());
        // ...直接写入二进制，无 GC 开销
    }
}
```

> **A:** FFM 的核心应用场景：
> 1. **机器学习推理**：直接调用 TensorFlow/PyTorch ONNX Runtime 的 C API，绕过 JNI 开销
> 2. **高性能网络**：零拷贝数据包处理（配合 io_uring DPDK）
> 3. **图形/游戏引擎**：调用 OpenGL/Vulkan/Unity C 接口
> 4. **序列化框架**：直接二进制序列化，无 GC 对象分配
> 5. **FFI 胶水层**：替代 Python ctypes/Ruby FFI，作为 Java 的官方 FFI 标准

---

## 一句话总结

> **FFM API = 纯 Java 调用 native 代码 + 托管式堆外内存 + 类型安全内存布局，三者合一终结 JNI 的"黑暗时代"。**

## 面试加分话术

> "FFM API 是我非常看好的 JDK 22 新特性。我在研究 RocketMQ 的高并发网络模型时发现，它大量使用 DirectByteBuffer 做零拷贝，但 ` Cleaner.create()` 的回收依赖 GC，无法精确控制释放时机。FFM 的 `Arena` 模式完美解决这个问题——`Arena.ofConfined()` 在当前线程结束时自动释放，无需 GC 介入。另外像 TensorFlow Java API 底层就是 JNI 封装，调用开销不小，用 FFM 重写后理论上可以有 30-50% 的性能提升。这也是为什么很多做基础设施的团队在等 JDK 23 的 FFM 稳定版。"
