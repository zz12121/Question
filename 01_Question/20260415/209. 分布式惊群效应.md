## 1. 基础流程

### 1.1 什么是惊群效应？

```
惊群效应（Thundering Herd）：

一只狼嚎叫 → 所有狼同时扑向猎物
一个请求击穿 → N个请求同时击穿

定义：当多个消费者同时感知到某个资源可用或某个条件满足时，
     它们会同时竞争同一个资源，导致瞬时过载、系统雪崩。
```

```
正常情况 vs 惊群情况：

┌────────────────────────────────────────────────────────────────────────────┐
│  正常情况（N个请求，缓存命中）                                              │
│                                                                             │
│   请求1 ──▶ [查缓存] ──▶ 命中 ──▶ 返回 ✅                                  │
│   请求2 ──▶ [查缓存] ──▶ 命中 ──▶ 返回 ✅                                  │
│   请求3 ──▶ [查缓存] ──▶ 命中 ──▶ 返回 ✅                                  │
│   ...                                                                    │
│                                                                             │
│   负载：每个请求 O(1)，数据库无感知                                        │
└────────────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────────────┐
│  惊群情况（缓存失效瞬间）                                                   │
│                                                                             │
│   请求1 ──┐                                                               │
│   请求2 ──┼──▶ [缓存 MISS] ──▶ 同时查 DB ──▶ DB 被打死 ❌                 │
```

### 1.3 惊群 vs 缓存雪崩 vs 缓存击穿 区分

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        三者对比                                              │
├───────────────┬─────────────────────┬─────────────────────┬─────────────────┤
│               │     惊群效应         │     缓存雪崩         │     缓存击穿     │
│               │  Thundering Herd     │    Cache Avalanche  │   Cache Pierce │
├───────────────┼─────────────────────┼─────────────────────┼─────────────────┤
│ 触发时机       │ 某个资源突然可用时   │ 大量缓存同时过期     │ 单个热点key过期 │
│               │ N个消费者同时争抢     │ 或缓存服务宕机       │ N个请求同时穿透 │
├───────────────┼─────────────────────┼─────────────────────┼─────────────────┤
│ 发生位置       │ 缓存/队列/服务发现   │ 缓存层整体失效       │ 单一key失效    │
├───────────────┼─────────────────────┼─────────────────────┼─────────────────┤
│ 请求行为       │ 主动同时发起请求     │ 被动等待缓存恢复     │ 主动同时穿透    │
├───────────────┼─────────────────────┼─────────────────────┼─────────────────┤
│ 影响范围       │ 单类资源/N个请求     │ 整个缓存层/大量key   │ 单个key/相关请求│
├───────────────┼─────────────────────┼─────────────────────┼─────────────────┤
│ 典型场景       │ ①批量缓存过期重建   │ ①Redis宕机           │ ①热点key过期   │
│               │ ②服务重启后注册发现  │ ②统一TTL失效        │ ②大促热点商品   │
│               │ ③限流恢复后重试     │ ③缓存初始化失败     │                 │
├───────────────┼─────────────────────┼─────────────────────┼─────────────────┤
│ 共同点         │ 都可能导致 DB 过载、雪崩                                      │
└───────────────┴─────────────────────┴─────────────────────┴─────────────────┘

视觉化对比：

惊群效应：
  某个时刻T → 大量请求同时从缓存miss → 同时查DB → DB过载
  ▲ 时间线：所有请求集中在T时刻

缓存雪崩：
  时间线T1 ─────────────────────────▶ T2 ─────────────────────────▶
       ↑                                  ↑
    缓存批量过期                       大量请求积压
       │                                    │
    缓慢恢复？                      可能持续到缓存重建

缓存击穿：
  1个热点key → 过期 → N个请求同时穿透 → DB过载
  ▲ 范围小，但并发量大
```

## 2. 解决方案详解

### 2.1 方案一：分布式锁 + 单次查询（最常用）

```
架构：
┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│   请求1 ──▶ [尝试获取锁] ──▶ 获取成功 ──▶ 查DB ──▶ 写缓存 ──▶ 返回        │
│   请求2 ──▶ [尝试获取锁] ──▶ 获取失败 ──▶ 等待100ms ──▶ 读缓存 ──▶ 返回  │
│   请求3 ──▶ [尝试获取锁] ──▶ 获取失败 ──▶ 等待200ms ──▶ 读缓存 ──▶ 返回  │
│   ...                                                                    │
│                                                                            │
│   结果：只有1个线程查DB，其他线程等待后读缓存                                │
└────────────────────────────────────────────────────────────────────────────┘
```

```java
// 实现示例
public User getUserById(Long id) {
    String cacheKey = "user:" + id;
    String lockKey = "lock:user:" + id;
    
    // 1. 先查缓存
    User cached = redisTemplate.opsForValue().get(cacheKey);
    if (cached != null) {
        return cached;
    }
    
    // 2. 尝试获取分布式锁（SETNX）
    String token = UUID.randomUUID().toString();
    Boolean acquired = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, token, Duration.ofSeconds(10));
    
    if (Boolean.TRUE.equals(acquired)) {
        try {
            // 3. 获取锁成功：查DB → 写缓存
            User user = userRepository.findById(id);
            redisTemplate.opsForValue().set(cacheKey, user, Duration.ofHours(1));
            return user;
        } finally {
            // 4. 释放锁（只能释放自己的锁）
            if (token.equals(redisTemplate.opsForValue().get(lockKey))) {
                redisTemplate.delete(lockKey);
            }
        }
    } else {
        // 5. 未获取锁：等待后重试读缓存
        try {
            Thread.sleep(100 + new Random().nextInt(100));
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        // 递归重试
        return getUserById(id);
    }
}
```

### 2.2 方案二：Jitter 随机抖动（最简单有效）

```
原理：将同一个TTL加上随机偏移量，让过期时间分散开

┌────────────────────────────────────────────────────────────────────────────┐
│  无 Jitter（所有key同时过期）                                                │
│                                                                            │
│  时间 ───────────────────────────────────────────────────────────────▶    │
│  缓存key: [key1 expires] [key2 expires] [key3 expires] ... [keyN expires]│
│             ↑               ↑               ↑                    ↑       │
│             └───────────────┴───────────────┴────────────────────┘       │
│                       集中在 T=3600 时刻                                   │
└────────────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────────────┐
│  有 Jitter（过期时间分散）                                                  │
│                                                                            │
│  时间 ───────────────────────────────────────────────────────────────▶    │
│  缓存key: [key1 expires] [key3 expires] [key2 expires] [keyN expires]... │
│             ↑              ↑               ↑                ↑             │
│             └──────────────┴────────────────┴────────────────┘            │
│                    分散在 T=3000~4200 范围内                               │
└────────────────────────────────────────────────────────────────────────────┘
```

```java
// 方案1：设置 TTL 时加 Jitter
public void setWithJitter(String key, Object value, long baseTtlSeconds) {
    // baseTtl 的 ±10% 作为 jitter 范围
    double jitter = baseTtlSeconds * 0.1;
    long actualTtl = (long) (baseTtlSeconds + (Math.random() * jitter * 2 - jitter));
    
    redisTemplate.opsForValue().set(key, value, Duration.ofSeconds(actualTtl));
}

// 方案2：主动提前过期（提前 TTL 的 10% 就开始重建）
public User getUserById(Long id) {
    String cacheKey = "user:" + id;
    CacheEntry entry = redisTemplate.opsForValue().get(cacheKey);
    
    if (entry == null) {
        return loadAndCache(id);  // 缓存未命中，直接加载
    }
    
    // 检查是否即将过期（提前10%就触发重建）
    long ttl = redisTemplate.getExpire(cacheKey, TimeUnit.SECONDS);
    long originalTtl = 3600L;
    if (ttl < originalTtl * 0.1) {
        // 即将过期，异步触发重建（单线程）
        CompletableFuture.runAsync(() -> loadAndCache(id));
    }
    
    return entry.getUser();
}
```

### 2.4 方案四：队列化 + Worker 模式

```
场景：Worker 争抢任务时的惊群

Redis BRPOP 惊群：
  Worker1 ──▶ BRPOP ──▶ 唤醒 ──▶ 抢到任务 ✅
  Worker2 ──▶ BRPOP ──▶ 唤醒 ──▶ 没抢到 ❌（白醒）
  Worker3 ──▶ BRPOP ──▶ 唤醒 ──▶ 没抢到 ❌（白醒）

解决方案：使用 BLMOVE 替代 BRPOP

BLMOVE 惊群缓解：
  Worker1 ──▶ BLMOVE left-pop ──▶ 原子操作 ──▶ 1次只出1个 ✅
  Worker2 ──▶ BLMOVE left-pop ──▶ 原子操作 ──▶ 1次只出1个 ✅
  Worker3 ──▶ BLMOVE left-pop ──▶ 原子操作 ──▶ 1次只出1个 ✅
  
  每次只唤醒1个Worker，没有惊群
```

```java
// 解决方案：使用 BLMOVE 替代 BRPOP
public String popTask() {
    // BLMOVE: 从 source 左边弹出一个元素，阻塞等待
    // timeout = 0 表示无限等待，直到有元素
    // 每次只返回1个元素，无惊群
    return redisTemplate.opsForList()
            .leftPop("task_queue", 
                     new Duration(0, java.time.temporal.ChronoUnit.SECONDS));
}

// 或者使用 Redis Streams（推荐）
public Map<Object, Object> consumeTask() {
    // XREADGROUP: 消费组模式
    // 消息只会被一个 Consumer 消费，不会惊群
    List<Map.Entry<String, Map<Object, Object>>> result = 
        redisTemplate.opsForStream()
            .read(StreamReadOptions.empty().count(1).block(Duration.ofSeconds(5)),
                  StreamOffset.create("task_stream", ReadOffset.last()));
    return result.isEmpty() ? null : result.get(0).getValue();
}
```

## 🔥 扩展追问 2：Jitter 策略的数学原理？为什么 Jitter 能有效分散压力？

> A: **Jitter 将集中分布变为均匀分布，通过统计学原理把"同步失效"变成"异步分散"。**

```
数学推导：

假设：10000个 key，TTL = 3600秒，无 Jitter

每个 key 在 3600s 内的任意时刻失效概率相等
→ 大量 key 的失效时间集中在 3600s 附近
→ 失效密度 = 10000 / 3600 ≈ 3个/秒

┌────────────────────────────────────────────────────────────────────────────┐
│  无 Jitter：失效分布（集中在 TTL 时刻）                                    │
│                                                                            │
│  密度                                                                       
│    ▲                                                                   ▲   │
│    │                    ████████                                        │
│    │               ████████████████                                     │
│    │          ████████████████████████                                  │
│    │       ██████████████████████████████                              │
│    │    ████████████████████████████████████                             │
│    │ ██████████████████████████████████████████                          │
│    └──────────────────────────────────────────────────────────────▶ 时间  │
│                                                             TTL=3600       │
│                       集中在 t=3600 时刻                                  │
└────────────────────────────────────────────────────────────────────────────┘

假设：10000个 key，TTL = 3600秒，加 ±10% Jitter

Jitter 范围：[3240, 3960]，均匀分布
→ 失效时间分散在 3240~3960 秒
→ 失效密度 = 10000 / 720 ≈ 14个/秒（仍然集中，但范围扩大了 5倍）

真正有效的是"提前过期 + 异步重建"策略：
→ 不等 TTL 到期，提前触发重建
→ 多个 key 的重建时间自然分散
→ 数据库负载被平滑化
```

```java
// 实际有效的 Jitter 实现（提前过期 + 随机延迟）
public class JitterCacheService {
    
    // 推荐公式：baseTtl * (0.5 ~ 1.0) 随机
    // 实际过期时间 = baseTtl + random(0, baseTtl * 0.5)
    // 优点：保证最小缓存时间，同时最大延长到 1.5 倍
    
    public long calculateActualTtl(long baseTtl) {
        // 50%~150% 随机范围
        double jitterRatio = 0.5 + Math.random();
        return (long) (baseTtl * jitterRatio);
    }
    
    // 或者使用"固定百分比 + 随机秒数"
    public long calculateWithFixedJitter(long baseTtl) {
        // 固定在 baseTtl ± 10% 范围内
        int jitterSeconds = new Random().nextInt(720) - 360; // -360~+360
        return Math.max(1, baseTtl + jitterSeconds);
    }
}
```

## 🔥 扩展追问 3：分布式锁方案有什么坑？SETNX 获取锁后服务崩溃了怎么办？

> A: **SETNX + TTL 是经典方案，但有"锁续期"、"锁释放误删"、"锁粒度"三个坑。**

```
坑1：锁续期问题（锁自动过期了但业务还没执行完）

┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│   线程A: SETNX lock:123 → 获取锁成功，TTL=10s                              │
│   线程A: 业务执行中... (需要30秒)                                          │
│                                                                            │
│   10秒后: TTL到期 → 锁自动释放 ← 线程B获取锁成功！                         │
│   线程A: 业务执行完成 → 释放锁 → 删除线程B的锁 ❌                          │
│                                                                            │
│   结果：线程B的锁被删除，线程C又可以获取锁                                  │
│   并发：A 和 B 同时持有锁 → 并发执行 → 数据不一致                          │
└────────────────────────────────────────────────────────────────────────────┘

解决方案：锁续期（watchdog）
```

```java
// Redisson 锁续期机制
RLock lock = redissonClient.getLock("myLock");
try {
    // watchDog 自动续期：每 10 秒续期一次，总共 30 秒
    // 如果不指定 leaseTime，启用 watchdog
    lock.lock(30, TimeUnit.SECONDS);  // 业务执行时间上限
    
    // Redisson 内部会每 10 秒检查：是否还持有锁？
    // 如果持有，自动续期 +10 秒
} finally {
    lock.unlock();
}

// 手动续期（如果业务执行时间不固定）
public void executeWithLock(String lockKey, Runnable task) {
    RLock lock = redissonClient.getLock(lockKey);
    lock.lock();
    try {
        // 启动一个定时器，每 10 秒续期一次
        scheduledExecutor.scheduleAtFixedRate(() -> {
            if (lock.isHeldByCurrentThread()) {
                lock.extendLeaseTime(10, TimeUnit.SECONDS);
            }
        }, 0, 10, TimeUnit.SECONDS);
        
        task.run();
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

```
坑2：锁释放误删问题

线程A: 业务执行完毕，检查 token... → token=UUID_A ✅
线程B: SETNX lock → 获取锁成功（因为 A 的锁已过期）→ token=UUID_B
线程A: DELETE lock → 删除了 B 的锁 ❌

解决方案：比较 token 再删除（Lua 脚本保证原子性）
```

```java
// Lua 脚本：保证原子性比较 + 删除
String luaScript = 
    "if redis.call('get', KEYS[1]) == ARGV[1] then " +
    "    return redis.call('del', KEYS[1]) " +
    "else " +
    "    return 0 " +
    "end";

// Java 代码
public boolean releaseLock(String lockKey, String token) {
    Long result = redisTemplate.execute(
        new DefaultRedisScript<>(luaScript, Long.class),
        Collections.singletonList(lockKey),
        token
    );
    return result != null && result == 1;
}
```

```
坑3：锁粒度问题

所有请求抢同一把大锁 → 串行化 → 性能差

解决方案：分段锁 / 按 key 分桶

// 错误：所有用户抢一把锁
@Cacheable(key = "#id")
→ @Cacheable(key = "'user:' + #id")  // 每个用户一把锁
```

## 🔥 扩展追问 5：Nginx 层的惊群问题？如何解决 Worker 进程间的惊群？

> A: **Nginx 使用 `accept_mutex` 和 `multi_accept` 配置解决惊群，Redis 6.0+ 使用 `io-threads` 解决 IO 惊群。**

```
Nginx Worker 惊群问题：

┌────────────────────────────────────────────────────────────────────────────┐
│  旧版 Nginx（accept_mutex off）：                                          │
│                                                                            │
│   连接1 ──▶ 全部 Worker 同时唤醒 ──▶ 争抢连接 ──▶ 惊群 ❌                 │
│   连接2 ──▶ 全部 Worker 同时唤醒 ──▶ 争抢连接 ──▶ 惊群 ❌                 │
└────────────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────────────┐
│  新版 Nginx（accept_mutex on）：                                           │
│                                                                            │
│   ┌────────────────────────────────────────────────────────────────────┐  │
│   │ Worker0 获取 accept_mutex 锁                                        │  │
│   │ 连接1 ──▶ Worker0 独占处理 ✅                                       │  │
│   │ Worker0 释放锁                                                       │  │
│   │                                                                      │  │
│   │ Worker1 获取 accept_mutex 锁                                        │  │
│   │ 连接2 ──▶ Worker1 独占处理 ✅                                       │  │
│   └────────────────────────────────────────────────────────────────────┘  │
│                                                                            │
│   优点：无惊群                                                             │
│   缺点：请求处理有一定延迟（等锁）                                         │
└────────────────────────────────────────────────────────────────────────────┘
```

```nginx
# nginx.conf
worker_processes auto;

events {
    # 开启 accept 锁（解决 Worker 间惊群）
    accept_mutex on;
    accept_mutex_delay 500ms;  # 获取锁失败后，500ms 后重试
    
    # 一次接受多个连接
    multi_accept on;
    
    # 使用 epoll（Linux）
    use epoll;
}

http {
    # Redis 6.0+ io-threads（解决 Redis IO 惊群）
    # 在 redis.conf 中配置
    io-threads 4;
    io-threads-do-reads yes;  # 读操作也用多线程
}
```

## 4. 一句话总结

> **惊群效应是"多个消费者同时感知并竞争同一资源"的瞬时过载现象，区分它的关键是"是否由单个资源触发"；Jitter 通过将集中分布变为均匀分布来缓解缓存雪崩，分布式锁通过串行化查询来防止击穿，两者组合是生产环境的标准解法。**

## 5. 面试加分话术

> "我之前踩过一个典型的惊群坑：线上 Redis 缓存了用户 Session，过期时间是固定的 1 小时，结果每天晚上 8 点整（上线 1 小时后）都有一次 DB 毛刺。排查发现是那批 Session 批量过期导致的雪崩，后来改成 `baseTtl + random(0, baseTtl * 0.2)` 加 Jitter，每秒钟只有几个 Session 重建，DB 毛刺直接消失了。另外一个场景是优惠券系统，热点优惠券有 10 万人抢，同一时间抢同一个 key，缓存一过期直接击穿。我们当时的解法是：缓存命中后主动检查 TTL，提前 10% 就触发异步重建；同时用 Redisson 分布式锁，只允许 1 个线程查 DB，其他 99999 个等待。这个组合拳打下来，P99 从 3000ms 降到了 80ms。Grafana 上看惊群发生时 Redis OPS 从 10 万直接跌到 0，说明请求全堵在 DB 上了，锁上线后 OPS 稳定在 5000 左右波动。"

```java
// Jitter + 锁 组合方案
@Configuration
public class CacheConfig {

    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {
        // Jitter TTL 计算
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofSeconds(calculateJitterTtl(3600)))
                .serializeKeysWith(...)
                .serializeValuesWith(...);

        return RedisCacheManager.builder(factory)
                .cacheDefaults(config)
                .withInitialCacheConfigurations(Map.of(
                    "hotData", config.entryTtl(Duration.ofSeconds(calculateJitterTtl(1800))),
                    "coldData", config.entryTtl(Duration.ofSeconds(calculateJitterTtl(86400)))
                ))
                .build();
    }

    // TTL = baseTtl + random(0, baseTtl * 0.2)
    private long calculateJitterTtl(long baseTtl) {
        return baseTtl + (long) (Math.random() * baseTtl * 0.2);
    }
}

// 分布式锁防击穿
@Bean
public RedissonClient redissonClient() {
    Config config = new Config();
    config.useSingleServer()
          .setAddress("redis://127.0.0.1:6379")
          .setConnectionMinimumIdleSize(10)
          .setConnectionPoolSize(100);
    return Redisson.create(config);
}
```
