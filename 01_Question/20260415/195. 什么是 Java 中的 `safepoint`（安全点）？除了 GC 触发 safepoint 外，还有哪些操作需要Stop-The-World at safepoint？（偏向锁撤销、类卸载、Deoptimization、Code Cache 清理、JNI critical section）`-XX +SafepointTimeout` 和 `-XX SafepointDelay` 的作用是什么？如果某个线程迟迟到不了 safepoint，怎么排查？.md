# 195. JVM Safepoint（安全点）

## 1. 基础流程

### 什么是 Safepoint？

Safepoint 是 JVM 执行过程中**所有线程都必须停顿的同步点**，用于执行需要"全局一致性"的操作。

### 为什么需要 Safepoint？

```
┌─────────────────────────────────────────────────────────────┐
│                 为什么需要 Safepoint？                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  问题：JVM 需要执行全局操作时，不知道其他线程在做什么           │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  GC 需要：                                          │   │

│  │  - 此时可以安全地读取线程状态                       │   │
│  │  - 执行全局操作                                     │   │
│  │  - 恢复所有线程                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  安全点位置（编译时插入）：                                   │
│  - 方法返回前                                               │

│  - Lock.inflate / wait 调用前                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 核心配置参数

```bash
# ========== Safepoint 超时配置 ==========
# 启用 safepoint 超时检测
-XX:+SafepointTimeout

# 超时阈值（毫秒），默认 10000ms
-XX:SafepointTimeoutDelay=10000

# 超时时打印所有线程的栈信息
-XX:+SafepointTimeoutAbrtExit

# ========== Safepoint 调试 ==========
# 打印 safepoint 相关统计
-XX:+PrintSafepointStatistics
-XX:PrintSafepointStatisticsCount=1

# 输出示例：
# vmop [threads: total initially_running wait_to_block]
# RevokeBias [245 1 0] [0 threads tooks 0 ms]
# No Safepoint [1023 0 0] [0 threads tooks 0 ms]

# ========== 偏向锁相关 ==========
# 禁用偏向锁（高并发场景可考虑）
-XX:-UseBiasedLocking

# 批量重偏向阈值
-XX:BiasedLockingBulkRebiasThreshold=20
# 批量撤销阈值
-XX:BiasedLockingBulkRevokeThreshold=40
```

## 2. 扩展追问

### 🔥 扩展追问 1：JNI Critical Section 和 Safepoint 的关系？

> A: **JNI Critical Section 进入和离开时会触发 safepoint poll。**

```java
// JNI Critical Section 示例
public class JNICriticalDemo {
    
    // JNI 方法
    private static native int[] getIntArray(int size);
    private static native void releaseIntArray(int[] arr);
    
    public static void main(String[] args) {
        // 进入 critical section
        int[] arr = getIntArray(100);  // ← safepoint poll
        
        // 访问数组（native 代码持有裸指针）
        for (int i = 0; i < arr.length; i++) {
            arr[i] = i;
        }
        
        // 离开 critical section
        releaseIntArray(arr);  // ← safepoint poll
        
        // 这里才能安全 GC
    }
}

// 原理：
// ┌─────────────────────────────────────────────────────────┐
// │  JNI Critical Section 的约束                             │
// │                                                          │
// │  1. 不能调用 Java 方法                                   │
// │  2. 不能分配 Java 对象                                    │
// │  3. 不能抛出异常                                         │
// │  4. 不能获取 Java 对象锁                                 │
// │                                                          │
// │  原因：                                                   │
// │  Native 代码可能持有裸指针，GC 移动对象会导致指针失效    │
// └─────────────────────────────────────────────────────────┘
```

### 🔥 扩展追问 2：如果某个线程迟迟到不了 safepoint，怎么排查？

> A: **通过日志、诊断命令、Profiler 三步定位问题。**

```bash
# ========== 1. 启用 Safepoint 日志 ==========
java -XX:+PrintSafepointStatistics \
     -XX:PrintSafepointStatisticsCount=1 \
     -jar app.jar

# 日志分析：
# vmop                    [threads: total initially_running wait_to_block]
# UpdateForAllocationRate  [245   1              0]            [0 threads tooks 0 ms]
# no vm operation         [245   0              0]            [0 threads tooks 2 ms]
# EnableBiasedLocking      [245   0              0]            [0 threads tooks 0 ms]
# 
# ← 第一个数字大 = 线程数多
# ← 第二个数字大 = 响应慢（有问题）

# ========== 2. Safepoint Timeout ==========
# 启用超时检测（默认 10 秒）
-XX:+SafepointTimeout \
-XX:SafepointTimeoutDelay=5000 \
-XX:+SafepointTimeoutAbort

# 触发后会打印：
# # SafepointSynchronize::begin: timeout detected, did all threads stop?
# # SafepointSynchronize::begin: threads: 245 total, 1 with delayed start
# # SafepointSynchronize::begin: timeout waiting for thread: MyThread @ tid=12345

# ========== 3. Arthas 排查 ==========
# 查看线程状态
thread

# 找到处于 BLOCKED 或 TIMED_WAITING 状态的线程
# 特别关注那些在 native 方法中的线程

# 查看特定线程的栈
thread <tid>

# 找到"卡住"的线程

# ========== 4. jstack 抓取 ==========
jstack <pid> > thread_dump.log

# 分析：
# - 是否有线程在执行无限循环的 native 代码
# - 是否有线程在 JNI Critical Section 中
# - 是否有线程卡在外部系统调用（数据库、HTTP）

# ========== 5. 常见"卡住"原因 ==========
# 原因1: 无限循环的 native 代码
// C 代码：
while (!done) {
    // busy waiting - 永远不会到 safepoint
}

// 解决：定期调用 CheckForSafepoints()
```

```java
// 原因2: 不调用任何 Java 方法
// JIT 编译后的代码可能完全没有 safepoint
public class NoSafepointLoop {
    // JIT 编译后这个循环没有任何 safepoint
    // 如果 GC 需要 stop world，会一直等待
    public static void hotLoop() {

        for (int i = 0; i < 1_000_000_000; i++) {  // ← 无 safepoint

// 解决：JIT 会识别这种情况并插入 safepoint
// 但如果是 native 代码则无法控制

// 原因3: Thread.sleep 在 native 方法中
public class JNICode {
    private native void blockingOperation();
    
    // JNIEXPORT void JNICALL Java_com_xxx_blockingOperation(JNIEnv* env, ...) {
    //     while (!done) {
    //         Sleep(1000);  // ← native sleep，不触发 safepoint
    //     }
    // }
}
```

### 🔥 扩展追问 4：虚拟线程中的 Safepoint 有什么变化？

> A: **JDK 21 虚拟线程大大减少了 safepoint 阻塞的影响。**

```java
// ========== 平台线程 vs 虚拟线程 ==========

// 平台线程：需要 OS 线程，safepoint 阻塞整个线程
public class PlatformThreadSafepoint {
    public static void main(String[] args) throws Exception {
        Thread t = new Thread(() -> {
            try {
                // 这会触发 safepoint
                Thread.sleep(1000);  // ← safepoint
                
                // 这个循环可能没有 safepoint
                while (running) {
                    // busy work
                }
            } catch (InterruptedException e) {}
        });
    }
}

// 虚拟线程：挂起时不占用 OS 线程，safepoint 影响更小
public class VirtualThreadSafepoint {
    public static void main(String[] args) throws Exception {
        Thread.ofVirtual().start(() -> {
            try {
                Thread.sleep(1000);  // ← 虚拟线程挂起，不影响 carrier
                
                while (running) {
                    // 虚拟线程循环
                    Thread.yield();  // ← 让出 carrier
                }
            } catch (InterruptedException e) {}
        });
    }
}

// 关键区别：
// ┌─────────────────────────────────────────────────────────┐
// │  平台线程：                                              │
// │  - 线程在 safepoint 阻塞                                │
// │  - GC 必须等待所有线程到达                               │
// │                                                          │
// │  虚拟线程：                                              │
// │  - 挂起的虚拟线程不占用 carrier 线程                     │
// │  - carrier 线程到达 safepoint                           │
// │  - 虚拟线程的状态保存在栈中                              │
// │  - GC 可以正常进行                                       │
// │                                                          │
// │  优点：虚拟线程大量减少 STW 对业务的影响                 │
// └─────────────────────────────────────────────────────────┘
```

## 3. 一句话总结

> **Safepoint 是 JVM 实现全局一致性操作的"全局停顿点"，GC、偏向锁撤销、类卸载、Deoptimization 等都需要在 safepoint 完成；虚拟线程通过挂起机制大幅降低了 safepoint 对业务的影响；通过 PrintSafepointStatistics 和 SafepointTimeout 参数可以监控和排查线程到达 safepoint 的问题。**

## 4. 面试加分话术

> "之前排查过一个 GC 停顿时间突然变长的问题：GC 日志显示 Stop-The-World 停了 5 秒，但堆内存只有 2GB，不应该这么慢。后来加上 `-XX:+PrintSafepointStatistics -XX:SafepointTimeout` 参数，发现是某个线程卡在 JNI 调用外部 DLL 的无限循环里，导致所有线程等它到达 safepoint。解决方案是：1）用 Arthas `thread` 命令定位卡住的线程，2）修改 native 代码，增加周期性调用 `env->CheckSafepoint()` 的逻辑，3）如果确实无法修改，就把这个线程设为 JVM 独立线程，不参与 GC 的 safepoint。这个经历让我深刻理解到 safepoint 的设计初衷——它是 JVM 实现全局安全的必要机制，但必须确保所有代码路径都能及时到达。"
