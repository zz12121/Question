### BGSAVE 和 BGREWRITEAOF 的互斥关系

```
┌─────────────────────────────────────────────────────────────┐
│              BGSAVE vs BGREWRITEAOF 互斥关系                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【Redis 3.x 之前】                                         │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                       │   │
│  │   BGSAVE     ←─────────────────────────→     BGREWRITEAOF  │
│  │   执行中                                        执行中      │
│  │       ↓                                              ↓      │
│  │   设置 aof_child_pid = -1                   设置 rdb_child_pid = -1  │
│  │       ↓                                              ↓      │
│  │   BGREWRITEAOF 被拒绝                          BGSAVE 被拒绝  │
│  │   (Already fork another child)               (Already fork another child)  │
│  │                                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  【Redis 4.0+ 改进】                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                       │   │
│  │   BGSAVE 执行中 + BGREWRITEAOF 同时执行？              │   │
│  │                                                       │   │
│  │   答案：可以！但有条件                               │   │
│  │                                                       │   │
│  │   场景:                                              │   │
│  │   1. BGSAVE 后台保存 RDB                            │   │
│  │   2. 同时 AOF 重写也在进行                           │   │
│  │                                                       │   │
│  │   原理:                                              │   │
│  │   ┌─────────────────────────────────────────────┐   │   │
│  │   │   rdb_child_pid != -1  AND                   │   │   │
│  │   │   aof_child_pid != -1                        │   │   │
│  │   │   = 两者可以同时执行                           │   │   │
│  │   └─────────────────────────────────────────────┘   │   │
│  │                                                       │   │
│  │   限制:                                              │   │
│  │   - 如果手动执行 BGSAVE，必须等它完成                │   │
│  │   - 之后的 BGREWRITEAOF 才能执行                    │   │
│  │                                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  【server 结构体中的状态】                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                       │   │
│  │  struct redisServer {                                 │   │
│  │      // RDB 子进程状态                              │   │
│  │      pid_t rdb_child_pid;         // -1 = 无子进程   │   │
│  │      int rdb_child_type;           // RDB_CHILD_xxx │   │
│  │                                                       │   │
│  │      // AOF 子进程状态                              │   │
│  │      pid_t aof_child_pid;         // -1 = 无子进程   │   │
│  │      int aof_child_type;           // AOF_CHILD_xxx │   │
│  │  };                                                  │   │
│  │                                                       │   │
│  │  // 互斥逻辑                                         │   │
│  │  if (server.rdb_child_pid != -1) {                   │   │

### 🔥 扩展追问 1：BGREWRITEAOF 过程中，新的 AOF 操作如何合并？

```
┌─────────────────────────────────────────────────────────────┐
│              AOF 重写期间的增量合并机制                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【重写前】                                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                       │   │
│  │  AOF 文件:                                            │   │
│  │  SET a 1                                              │   │
│  │  SET b 2                                              │   │
│  │  INCR a        → 变成 SET a 2                       │   │
│  │  SET c 3                                              │   │
│  │                                                       │   │
│  │  重写后:                                              │   │
│  │  SET a 2                                              │   │
│  │  SET b 2                                              │   │
│  │  SET c 3                                              │   │
│  │                                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  【重写过程中有新写入】                                     │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                       │   │
│  │  t1: BGREWRITEAOF 启动                               │   │
│  │       子进程开始读取 Redis 数据                       │   │
│  │       ↓                                              │   │
│  │  t2: 客户端执行 SET d 4                              │   │
│  │       ↓                                              │   │
│  │       写入 AOF buffer (主进程缓冲区)                 │   │
│  │       同时写入 AOF 重写缓冲区 (子进程缓冲区)          │   │
│  │       ↓                                              │   │
│  │  t3: 子进程完成，发送信号给父进程                     │   │
│  │       ↓                                              │   │
│  │  t4: 父进程:                                         │   │
│  │       1. 收到子进程信号                              │   │
│  │       2. 将 AOF 重写缓冲区的内容追加到新的 AOF       │   │

│  │                                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │

│  │  auto-aof-rewrite-percentage 100  # AOF文件比上次大100%  │   │
│  │  auto-aof-rewrite-min-size 64mb # 最小64MB才触发    │   │
│  │                                                       │   │
│  │  # AOF 持久化策略                                    │   │
│  │  appendfsync everysec    # 每秒同步（推荐）          │   │
│  │  appendfsync always     # 每次写入同步（最安全）    │   │
│  │  appendfsync no         # 依赖 OS（最快）            │   │
│  │                                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘

### 🔥 扩展追问 2：Redis 6.0 的多线程 IO 对持久化有影响吗？

> A: **多线程 IO 只处理客户端命令读写，不影响持久化线程。**

```bash
# Redis 6.0 多线程 IO 架构
# ┌─────────────────────────────────────────────────────────┐
# │                                                          │
# │                    Main Thread                           │
# │                          │                               │
# │   ┌─────────────────────┼─────────────────────┐       │
# │   │                     │                     │       │
# │   ↓                     ↓                     ↓       │
# │ IO Thread 1         IO Thread 2         IO Thread 3      │
# │ (客户端读写)        (客户端读写)        (客户端读写)      │
# │                                                          │
# │   ⚠️ 注意：这些 IO 线程只处理客户端命令                  │
# │   ⚠️ 不处理：BGSAVE、BGREWRITEAOF、复制等后台操作        │
# │                                                          │
# └─────────────────────────────────────────────────────────┘

# 配置多线程 IO
io-threads 4        # 线程数（包括主线程 = 4 个线程）
io-threads-do-reads yes  # 启用多线程读取

# 验证
redis-cli INFO stats | grep -i thread
# thread_funnel: Global thread funnel count: 1
```

### 🔥 扩展追问 3：COW (Copy-On-Write) 在持久化中的表现？

> A: **fork 后父进程内存的修改会触发 COW，是持久化期间内存增长的主因。**

```bash
# COW 原理图解
# ┌─────────────────────────────────────────────────────────┐
# │                                                          │
# │  fork() 瞬间:                                           │
# │  ┌─────────────────────────────────────────────────┐  │
# │  │  父进程内存页 ──→ 子进程共享（只读）              │  │
# #  └─────────────────────────────────────────────────┘  │
# │                                                          │
# │  写操作触发 COW:                                        │
# │  ┌─────────────────────────────────────────────────┐  │
# │  │  Redis 执行 SET user:1 "new_value"              │  │
# │  │         ↓                                         │  │
# │  │  触发 COW:                                        │  │
# │  │  ┌─────────────────────────────────────────────┐ │ │
# │  │  │  页 A (被修改)                               │ │ │
# │  │  │  复制一份页 A' 给子进程                      │ │ │
# │  │  │  父进程使用新的页 A                          │ │ │
# │  │  └─────────────────────────────────────────────┘ │ │
# │  └─────────────────────────────────────────────────┘  │
# │                                                          │
# │  结果:                                                   │
# │  - 父进程内存增加 (复制的页)                           │
# │  - 子进程看到的数据保持 fork 时的状态                   │
# │                                                          │
# └─────────────────────────────────────────────────────────┘

# 监控 COW 消耗
redis-cli INFO commandstats | grep -i cow
# cmdstat_bgrewriteaof:calls=5,usec=500,usec_per_call=100
# cmdstat_bgsave:calls=10,usec=1000,usec_per_call=100

# 查看 Linux 进程的 COW 信息
cat /proc/<redis_pid>/status | grep -i vmpeak
# VmPeak: 52428800 kB    # 进程使用的峰值内存
# VmSize: 52428800 kB    # 当前虚拟内存
# VmRSS: 52000000 kB     # 实际物理内存
# VmSwap: 0 kB          # 交换空间

# 查看 COW 页数量
cat /proc/<redis_pid>/pagemap | wc -l
# 或者
pmap -x <redis_pid> | tail -1
```

### 🔥 扩展追问 4：大内存 Redis 的持久化优化建议？

> A: **通过配置、架构、运维三方面优化大内存场景下的持久化。**

```bash
# ========== 1. 配置优化 ==========
# 推荐配置
save 900 1      # 关闭自动 RDB（如果不需要）
save 300 10000  # 30分钟超过10000次写才 RDB
save 60 1000000 # 5分钟超过100万次写才 RDB

# AOF 配置
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec    # 不建议 always
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 1gb    # 超过1GB才重写

# COW 优化
# 无需额外配置，Redis 自动使用 COW

# ========== 2. THP 必须关闭 ==========
# /etc/rc.local 或启动脚本
if test -f /sys/kernel/mm/transparent_hugepage/enabled; then
    echo never > /sys/kernel/mm/transparent_hugepage/enabled
fi
if test -f /sys/kernel/mm/transparent_hugepage/defrag; then
    echo never > /sys/kernel/mm/transparent_hugepage/defrag
fi

# ========== 3. 架构优化 ==========
# 方案A: Redis Cluster + 小实例
# - 每个实例 20-30GB，避免大实例 fork 卡顿
# - 缺点: 管理复杂度增加

# 方案B: Redis + AOF Only
# - 关闭 RDB，只用 AOF
# - appendfsync everysec + aof-load-truncated yes
# - 缺点: AOF 文件可能很大

# 方案C: 混合持久化 (Redis 5.0+)
# - 开启混合持久化
aof-use-rdb-preamble yes
# - AOF 重写时先写 RDB，再追加 AOF
# - 兼具 RDB 快速恢复 + AOF 完整持久化

# ========== 4. 监控脚本 ==========
#!/bin/bash
REDIS_PID=$(pgrep redis-server)
if [ -z "$REDIS_PID" ]; then
    echo "Redis not running"
    exit 1
fi

# 检查 fork 耗时
fork_time=$(redis-cli INFO stats | grep -i "latest_fork" | awk -F: '{print $2}')
echo "Latest fork time: ${fork_time}ms"

# 检查子进程状态
rdb_pid=$(redis-cli INFO persistence | grep -i "rdb_child_pid" | awk -F: '{print $2}')
aof_pid=$(redis-cli INFO persistence | grep -i "aof_child_pid" | awk -F: '{print $2}')
echo "RDB child: $rdb_pid, AOF child: $aof_pid"

# 检查内存使用
used_memory=$(redis-cli INFO memory | grep -i "used_memory_human" | awk -F: '{print $2}')
echo "Memory used: $used_memory"

# 检查 THP 状态
thp_enabled=$(cat /sys/kernel/mm/transparent_hugepage/enabled 2>/dev/null | grep -c "\[always\]")
if [ "$thp_enabled" -gt 0 ]; then
    echo "⚠️ WARNING: THP is enabled!"
fi
```

## 3. 一句话总结

> **Redis 通过 fork() COW 机制实现 RDB 和 AOF 重写，两者通过 server.rdb_child_pid 和 server.aof_child_pid 互斥（Redis 4.0+ 可同时执行）；大内存 fork 耗时主要在页表复制（50GB 约 250ms），THP 必须关闭因为会按 2MB 粒度 COW 导致内存暴涨；可通过混合持久化、关闭自动 RDB、优化架构来缓解大实例的持久化压力。**

## 4. 面试加分话术

> "之前运维过一个大内存 Redis 集群（单实例 200GB），踩过不少坑。第一个坑是 THP：当时没注意到服务器默认开启 THP，BGSAVE 时内存直接飙到 250GB+，差点触发 OOM。后来在启动脚本里加了一句 `echo never > /sys/kernel/mm/transparent_hugepage/enabled` 才解决。第二个坑是 fork 耗时：200GB 实例 fork 一次要 1.5 秒左右，导致主线程在这期间无法响应请求，用 `redis-cli INFO LATENCY` 监控发现延迟抖动明显。后来改成 Redis Cluster 方案，每个实例控制在 30GB 以内，fork 控制在 150ms 以内。另外建议生产环境开启 `aof-use-rdb-preamble yes` 混合持久化，恢复时先加载 RDB 部分，再用 AOF 追增量，兼顾恢复速度和完整性。"

---

**追问预判：**
- "RDB 和 AOF 怎么选？" → 推荐混合持久化，Redis 5.0+ 默认开启，兼具两者优点
- "Redis Cluster 下持久化有什么注意？" → 每个节点独立持久化，建议关闭自动触发改用定时任务，控制时机避免多个节点同时 fork
