## 基础流程

### 1. ThreadGroup 的设计初衷

`ThreadGroup` 诞生于 **JDK 1.0**，设计理念是**批量管理线程**——将相关线程归组、统一设置优先级、统一中断、统一异常处理。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    ThreadGroup 原始设计                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ThreadGroup 结构:                                                  │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │   main (ThreadGroup)                                        │   │
│  │   ├─ system (ThreadGroup)                                  │   │
│  │   │   └─ Reference Handler                                  │   │
│  │   │   └─ Finalizer                                         │   │
│  │   │   └─ Signal Dispatcher                                 │   │
│  │   │                                                        │   │
│  │   └─ user (ThreadGroup) ← 业务线程组                        │   │
│  │       ├─ http-nio-1                                        │   │
│  │       ├─ http-nio-2                                        │   │
│  │       └─ task-executor-1                                   │   │
│  │                                                             │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  设计的核心能力:                                                     │
│  ├─ interrupt()    → 批量中断组内所有线程                           │
│  ├─ setMaxPriority() → 设置组内线程最大优先级                       │
│  ├─ uncaughtException() → 组内线程未捕获异常的默认处理              │
│  └─ activeCount() / activeGroupCount() → 统计活跃线程/组数          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**原始设计 API**：

```java
public class ThreadGroupOriginalDesign {

    public static void main(String[] args) {
        
        // ============ 创建线程组 ============
        ThreadGroup group = new ThreadGroup("MyGroup");
        
        // 设置组内线程的最大优先级（子线程不能超过此值）
        group.setMaxPriority(Thread.NORM_PRIORITY + 2);
        
        // ============ 在组内创建线程 ============
        Thread t1 = new Thread(group, () -> {
            System.out.println("Task 1 running");
        }, "Worker-1");
        
        Thread t2 = new Thread(group, () -> {
            System.out.println("Task 2 running");
        }, "Worker-2");
        
        t1.start();
        t2.start();
        
        // ============ 批量操作 ============
        // 中断组内所有线程
        // group.interrupt();
        
        // 等待组内所有线程结束
        group.destroy();  // 前提：组内无活跃线程
        
        // ============ 获取线程组信息 ============
        System.out.println("Group name: " + group.getName());
        System.out.println("Active threads: " + group.activeCount());
        System.out.println("Active groups: " + group.activeGroupCount());
        System.out.println("Max priority: " + group.getMaxPriority());
    }
}
```

### 2. 为什么被废弃（JDK 19）

```java
// JEP 425: Virtual Threads 明确废弃 ThreadGroup

public class WhyDeprecated {

    public static void main(String[] args) {
        
        /*
         * ThreadGroup 被废弃的原因：
         * 
         * ┌───────────────────────────────────────────────────────┐
         * │ 1. 虚拟线程时代的设计冲突                               │
         * ├───────────────────────────────────────────────────────┤
         * │                                                       │
         * │  虚拟线程不归属于任何 ThreadGroup！                      │
         * │  Thread.currentThread().getThreadGroup() 返回           │
         * │  的是父级平台线程所在的组，而非虚拟线程自己的组           │
         * │                                                       │
         * │  这导致 ThreadGroup 的所有批量操作对虚拟线程无效         │
         * └───────────────────────────────────────────────────────┘
         * 
         * ┌───────────────────────────────────────────────────────┐
         * │ 2. API 设计缺陷                                       │
         * ├───────────────────────────────────────────────────────┤
         * │                                                       │
         * │  activeCount() 不准确                                 │
         * │  - 统计结果依赖瞬时快照                                │
         * │  - 不包括尚未启动的线程                                │
         * │  - 僵尸线程可能被计入                                 │
         * │                                                       │
         * │  继承模型问题                                          │
         * │  - ThreadGroup 可以嵌套                               │
         * │  - 但很少有人真正用嵌套的 ThreadGroup                  │
         * └───────────────────────────────────────────────────────┘
         * 
         * ┌───────────────────────────────────────────────────────┐
         * │ 3. 优先级控制形同虚设                                  │
         * ├───────────────────────────────────────────────────────┤
         * │                                                       │
         * │  setMaxPriority() 不能真正限制子线程的优先级           │
         * │  - 线程创建后可以单独修改优先级                        │
         * │  - 平台线程优先级本身在 OS 层面就不精确                │
         * └───────────────────────────────────────────────────────┘
         * 
         * ┌───────────────────────────────────────────────────────┐
         * │ 4. 异常处理机制已被替代                                 │
         * ├───────────────────────────────────────────────────────┤
         * │                                                       │
         * │  Thread.setDefaultUncaughtExceptionHandler()          │
         * │  提供了更细粒度的全局异常处理                           │
         * │  比 ThreadGroup.uncaughtException() 更灵活             │
         * └───────────────────────────────────────────────────────┘
         */
    }
}
```

**废弃注解**：

### 3. 活跃线程数统计不准确

```java
public class ActiveCountInaccuracy {

    public static void main(String[] args) throws InterruptedException {
        
        ThreadGroup group = new ThreadGroup("TestGroup");
        
        // 创建 3 个线程
        for (int i = 0; i < 3; i++) {
            Thread t = new Thread(group, () -> {
                try {
                    Thread.sleep(100);
                } catch (InterruptedException e) { }
            });
            t.start();
        }
        
        // ============ 陷阱1：递归统计 ============
        int count = group.activeCount();  // 仅统计直接子线程
        
        Thread[] threads = new Thread[count * 2];  // 扩大数组
        int actual = group.enumerate(threads, false);  // false=不递归子组
        
        System.out.println("activeCount(): " + count);
        System.out.println("enumerate(false): " + actual);
        
        // ============ 陷阱2：动态变化 ============
        Thread newThread = new Thread(group, () -> { });
        System.out.println("Before start: " + group.activeCount());  // 可能是 3
        
        newThread.start();
        System.out.println("After start: " + group.activeCount());   // 可能是 4
        
        newThread.join();
        System.out.println("After join: " + group.activeCount());   // 恢复 3
        
        // ============ 陷阱3：虚拟线程不归组 ============
        Thread virtual = Thread.ofVirtual().name("virtual-1").start(() -> {
            ThreadGroup vg = Thread.currentThread().getThreadGroup();
            System.out.println("Virtual thread's group: " + vg.getName());
            // 输出: "main" 或其他系统组，不是业务组！
        });
        virtual.join();
        
        // ============ 陷阱4：结果过期 ============
        int staleCount = group.activeCount();  // 获取后立即过期
        // 在你使用 staleCount 时，活跃线程可能已变化
        
        group.destroy();
    }
}
```

```
┌────────────────────────────────────────────────────────────────┐
│              activeCount() 不准确的场景                            │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  ┌──────────────────────────────────────────────────────────┐ │
│  │ 1. 快照过期                                               │ │
│  │    activeCount() 返回后，线程可能立即结束/启动             │ │
│  │    结果永远是一个"可能不准确"的数字                        │ │
│  └──────────────────────────────────────────────────────────┘ │
│                                                                │
│  ┌──────────────────────────────────────────────────────────┐ │
│  │ 2. 虚拟线程不参与统计                                      │ │
│  │    Virtual threads are not members of any ThreadGroup    │ │
│  │    activeCount() 对虚拟线程返回 0                          │ │
│  └──────────────────────────────────────────────────────────┘ │
│                                                                │
│  ┌──────────────────────────────────────────────────────────┐ │
│  │ 3. 不包含未启动线程                                        │ │
│  │    Thread t = new Thread(group, ...);                   │ │
│  │    group.activeCount(); // 不包含 t                      │ │
│  │    t.start();                                            │ │
│  │    group.activeCount(); // 才包含                         │ │
│  └──────────────────────────────────────────────────────────┘ │
│                                                                │
│  ┌──────────────────────────────────────────────────────────┐ │
│  │ 4. 线程池中的线程                                          │ │
│  │    ExecutorService.submit() 创建的是池线程                 │ │
│  │    池线程不属于任何 ThreadGroup                            │ │
│  └──────────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────────┘
```

### 4. 替代方案

```java
import java.util.concurrent.*;

public class ThreadGroupAlternatives {

    public static void main(String[] args) {
        
        // ============ 方案1：ExecutorService + ThreadFactory ============
        
        // 替代：使用自定义 ThreadFactory 创建"有标识"的线程
        ThreadFactory factory = new ThreadFactory() {
            private int count = 0;
            
            @Override
            public Thread newThread(Runnable r) {
                Thread t = new Thread(r, "Worker-" + count++);
                t.setPriority(Thread.NORM_PRIORITY);
                t.setDaemon(false);
                return t;
            }
        };
        
        ExecutorService executor = Executors.newFixedThreadPool(
            4, factory
        );
        
        // 优雅关闭
        executor.shutdown();  // 停止接收新任务
        executor.awaitTermination(60, TimeUnit.SECONDS);  // 等待完成
        // 或
        executor.shutdownNow();  // 立即中断
        
        // ============ 方案2：CompletableFuture 聚合 ============
        CompletableFuture.allOf(
            CompletableFuture.runAsync(() -> task1()),
            CompletableFuture.runAsync(() -> task2()),
            CompletableFuture.runAsync(() -> task3())
        ).join();  // 等待所有完成
        
        // ============ 方案3：结构化并发（JDK 21）===========
        // 见 Q187
        
        // ============ 方案4：自定义线程管理 ============
        ThreadManager manager = new ThreadManager("App");
        manager.start("http-worker-1", () -> runHttpServer());
        manager.start("http-worker-2", () -> runHttpServer());
        
        // 批量中断
        manager.shutdown();  // 中断所有线程
        
        // 等待完成
        manager.awaitTermination(30, TimeUnit.SECONDS);
        
        // 获取状态
        System.out.println("Active threads: " + manager.getActiveCount());
    }

    // ============ 自定义线程管理器 ============
    static class ThreadManager {
        private final String name;
        private final Map<String, Thread> threads = new ConcurrentHashMap<>();
        private volatile boolean shutdown = false;
        
        public ThreadManager(String name) {
            this.name = name;
        }
        
        public Thread start(String threadName, Runnable task) {
            Thread t = new Thread(() -> {
                while (!shutdown) {
                    try {
                        task.run();
                    } catch (InterruptedException e) {
                        break;
                    }
                }
            }, threadName);
            
            threads.put(threadName, t);
            t.start();
            return t;
        }
        
        public void shutdown() {
            shutdown = true;
            threads.values().forEach(Thread::interrupt);
        }
        
        public void awaitTermination(long timeout, TimeUnit unit) 
            throws InterruptedException {
            long deadline = System.nanoTime() + unit.toNanos(timeout);
            for (Thread t : threads.values()) {
                long remaining = deadline - System.nanoTime();
                if (remaining <= 0) break;
                t.join(remaining, 0);
            }
        }
        
        public int getActiveCount() {
            return (int) threads.values().stream()
                .filter(Thread::isAlive)
                .count();
        }
    }

    private static void task1() { }
    private static void task2() { }
    private static void task3() { }
    private static void runHttpServer() throws InterruptedException {
        Thread.sleep(1000);
    }
}
```

## 🔥 扩展追问 1：ThreadGroup 的 uncaughtException 有什么用？

```java
public class UncaughtExceptionHandling {

    public static void main(String[] args) {
        
        // ============ ThreadGroup 的异常处理 ============
        ThreadGroup group = new ThreadGroup("Worker") {
            @Override
            public void uncaughtException(Thread t, Throwable e) {
                System.err.println("Group caught: " + e.getMessage());
                // 记录日志、发送告警等
            }
        };
        
        Thread t = new Thread(group, () -> {
            throw new RuntimeException("Task failed!");
        });
        t.start();
        
        // ============ 现代替代：全局 UncaughtExceptionHandler ============
        Thread.setDefaultUncaughtExceptionHandler((t2, e) -> {
            System.err.println("Global handler: " + e.getMessage());
            // 发送告警到监控系统
        });
        
        // ============ 线程级别覆盖 ============
        Thread t2 = new Thread(() -> {
            throw new RuntimeException("Thread-specific error");
        });
        t2.setUncaughtExceptionHandler((t3, e) -> {
            System.err.println("Thread handler: " + e.getMessage());
        });
        t2.start();
        
        // 优先级：Thread.uncaughtExceptionHandler > ThreadGroup > Thread.default
    }
}
```

> **A:** 
> - `ThreadGroup.uncaughtException()` 是**默认兜底**处理
> - **优先级**：线程自身 Handler > ThreadGroup > Thread.default
> - `Thread.setDefaultUncaughtExceptionHandler()` 是全局设置
> - 两者都被 `ExecutorService` 的 `afterExecute()` 覆盖——JUC 框架完全绕过了 ThreadGroup

## 🔥 扩展追问 2：JUC 框架如何替代 ThreadGroup 的批量管理？

```java
public class JUC替代方案 {

    public static void main(String[] args) throws Exception {
        
        // ============ 场景1：批量关闭线程池 ============
        ExecutorService executor = Executors.newFixedThreadPool(10);
        
        // 提交任务
        Future<?> f1 = executor.submit(() -> sleep(1000));
        Future<?> f2 = executor.submit(() -> sleep(2000));
        Future<?> f3 = executor.submit(() -> sleep(500));
        
        // 批量取消（替代 group.interrupt()）
        executor.shutdownNow();  // 发送 interrupt 到所有池线程
        
        // ============ 场景2：线程池状态监控 ============
        ThreadPoolExecutor tp = (ThreadPoolExecutor) executor;
        
        System.out.println("Active count: " + tp.getActiveCount());
        System.out.println("Pool size: " + tp.getPoolSize());
        System.out.println("Completed tasks: " + tp.getCompletedTaskCount());
        System.out.println("Queue size: " + tp.getQueue().size());
        
        // 动态调整（替代 group.setMaxPriority()）
        tp.setCorePoolSize(5);
        tp.setMaximumPoolSize(20);
        tp.setKeepAliveTime(60, TimeUnit.SECONDS);
        
        executor.shutdown();
        
        // ============ 场景3：ScheduledExecutorService 定时任务 ============
        ScheduledExecutorService scheduler = 
            Executors.newScheduledThreadPool(2);
        
        // 定时任务
        ScheduledFuture<?> schedule = scheduler.scheduleAtFixedRate(
            () -> System.out.println("Heartbeat"),
            0, 1, TimeUnit.SECONDS
        );
        
        // 取消定时任务
        schedule.cancel(false);
        
        scheduler.shutdown();
    }
    
    static void sleep(long ms) throws InterruptedException {
        Thread.sleep(ms);
    }
}
```

> **A:** JUC 替代 ThreadGroup 的方式：
> 1. **批量关闭**：`ExecutorService.shutdownNow()` 替代 `group.interrupt()`
> 2. **状态监控**：`ThreadPoolExecutor.getActiveCount()` 替代 `group.activeCount()`
> 3. **动态调整**：`setCorePoolSize()` 替代 `setMaxPriority()`
> 4. **ThreadFactory**：创建时设置线程属性，替代线程组继承

## 🔥 扩展追问 3：ThreadGroup 的 destroy() 有什么坑？

```java
public class DestroyPitfalls {

    public static void main(String[] args) throws InterruptedException {
        
        ThreadGroup group = new ThreadGroup("Test");
        
        // ============ 坑1：必须无活跃线程才能 destroy ============
        Thread t = new Thread(group, () -> {
            try {
                Thread.sleep(1000);
            } catch (InterruptedException e) { }
        });
        t.start();
        
        // ⚠️ 如果此时调用 destroy()，抛出 IllegalThreadStateException
        // group.destroy();  // ❌ 失败
        
        t.join();  // 等待线程结束
        
        // ✅ 现在可以 destroy
        group.destroy();
        System.out.println("Group destroyed successfully");
        
        // ============ 坑2：destroy 后不能复用 ============
        ThreadGroup group2 = new ThreadGroup("Test2");
        group2.destroy();
        
        // ⚠️ 销毁后的 group 无法再创建线程
        // new Thread(group2, ...);  // ❌ IllegalThreadStateException
        
        // ============ 坑3：子 ThreadGroup 必须先销毁 ============
        ThreadGroup parent = new ThreadGroup("Parent");
        ThreadGroup child = new ThreadGroup(parent, "Child");
        
        // ⚠️ 必须先销毁 child
        // parent.destroy();  // ❌ 如果 child 未销毁
        
        child.destroy();
        parent.destroy();
    }
}
```

> **A:** `destroy()` 的问题：
> 1. **必须无活跃线程**：有线程运行时会抛异常
> 2. **不可逆**：销毁后 ThreadGroup 无法使用
> 3. **父子依赖**：父组销毁前，子组必须先销毁
> 4. **无自动清理**：线程结束时不会自动从组中移除

## 🔥 扩展追问 4：如何监控生产环境的线程状态？

```java
public class ProductionMonitoring {

    public static void main(String[] args) {
        
        // ============ 方法1：JMX 监控 ============
        /*
         * 启动时添加 JMX：
         * -Dcom.sun.management.jmxremote
         * -Dcom.sun.management.jmxremote.port=9999
         * 
         * 使用 JConsole 或 JMX Client 连接查看：
         * - 线程列表
         * - 线程状态
         * - CPU 使用率
         */
        
        // ============ 方法2：JFR（Java Flight Recorder）===========
        /*
         * jmc -XX:StartFlightRecording=duration=60s,filename=recording.jfr
         * 
         * 关注事件：
         * - jdk.ThreadStart
         * - jdk.ThreadEnd
         * - jdk.JavaThreadStart
         * - jdk.JavaThreadEnd
         */
        
        // ============ 方法3：代码级监控 ============
        ThreadMXBean threadMXBean = ManagementFactory.getThreadMXBean();
        
        System.out.println("Thread count: " + threadMXBean.getThreadCount());
        System.out.println("Peak thread count: " + threadMXBean.getPeakThreadCount());
        System.out.println("Daemon thread count: " + threadMXBean.getDaemonThreadCount());
        System.out.println("Total started: " + threadMXBean.getTotalStartedThreadCount());
        
        // 获取死锁线程
        long[] deadlocks = threadMXBean.findDeadlockedThreads();
        if (deadlocks != null) {
            System.out.println("Deadlock detected!");
            for (ThreadInfo info : threadMXBean.getThreadInfo(deadlocks)) {
                System.out.println("Deadlocked: " + info.getThreadName());
            }
        }
        
        // ============ 方法4：定期 dump ============
        ThreadDump.dump();  // 类似 jstack 输出
    }

    // ============ 线程Dump工具 ============
    static class ThreadDump {
        public static void dump() {
            StringBuilder sb = new StringBuilder();
            sb.append("Full Thread Dump:\n\n");
            
            Thread.getAllStackTraces().forEach((thread, stack) -> {
                sb.append(String.format('"' + thread.getName() + '" id=%d state=%s\n',
                    thread.getId(), thread.getState()));
                
                for (StackTraceElement ste : stack) {
                    sb.append("  at " + ste + "\n");
                }
                sb.append("\n");
            });
            
            System.out.println(sb);
        }
    }
}
```

> **A:** 生产环境监控方案：
> 1. **JMX + JConsole**：实时查看线程状态
> 2. **JFR**：低开销录制线程事件
> 3. **ThreadMXBean API**：代码级获取线程信息
> 4. **jstack**：进程 dump 查看阻塞状态
> 5. **Arthas**：阿里诊断工具，`thread` 命令查看线程

## 🔥 扩展追问 5：ThreadGroup 在 JDK 内部还有使用吗？

```java
public class InternalUsage {

    public static void main(String[] args) {
        
        // ============ JDK 内部仍使用 ThreadGroup ============
        
        // 1. 系统线程组（system/main）
        ThreadGroup system = Thread.currentThread().getThreadGroup();
        System.out.println("System group: " + system.getName());
        
        // 2. AppletClassLoader 使用 ThreadGroup
        // Applet 运行时创建自己的 ThreadGroup
        
        // 3. SecurityManager（已废弃 JDK 17）
        // SecurityManager.getThreadGroup()
        
        // ============ 关键点：虚拟线程不参与 ============
        Thread virtual = Thread.startVirtualThread(() -> {
            ThreadGroup vg = Thread.currentThread().getThreadGroup();
            System.out.println("Virtual thread's group: " + vg.getName());
            // 永远不是业务组！
        });
    }
}
```

> **A:** ThreadGroup 仍在 JDK 内部使用：
> 1. **main 线程组**：系统根组
> 2. **Applet 运行时**：隔离 applet 线程
> 3. **SecurityManager**（已废弃）
> 4. **但虚拟线程不参与**：虚拟线程永远属于系统组，业务 ThreadGroup 无法管理

## 一句话总结

> **ThreadGroup 被废弃是因为虚拟线程不参与分组 + activeCount() 快照不准确 + API 设计过时；替代方案是 `ExecutorService` 批量管理 + `ThreadFactory` 自定义线程 + JMX/JFR 监控。**

## 面试加分话术

> "ThreadGroup 被废弃的核心原因是**虚拟线程时代的设计冲突**。我在排查一个性能问题时发现：项目里用 ThreadGroup 统计活跃线程，`activeCount()` 一直显示 0，但服务明明在正常处理请求。后来发现是因为用了虚拟线程，虚拟线程不归属于任何 ThreadGroup，`activeCount()` 统计的永远是平台线程，虚拟线程的活跃度根本看不了。所以我们改用了 `ThreadPoolExecutor` + Micrometer 指标暴露线程池状态，配合 `jfr` 录制 `jdk.ThreadStart` 事件，通过 Grafana 大盘实时监控。如果现在还有人面试问 ThreadGroup，可以直接说：这是一个历史包袱，虚拟线程时代已被淘汰，生产环境用 ExecutorService 管理线程。"
