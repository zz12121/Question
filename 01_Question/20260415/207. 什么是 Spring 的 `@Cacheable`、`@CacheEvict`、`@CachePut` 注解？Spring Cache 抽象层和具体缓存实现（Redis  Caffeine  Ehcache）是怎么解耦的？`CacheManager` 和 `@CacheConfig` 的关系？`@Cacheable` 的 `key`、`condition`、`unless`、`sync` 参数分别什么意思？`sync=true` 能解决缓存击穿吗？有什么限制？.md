### 1.3 核心配置代码

```java
// 方式1：Redis + Spring Cache 配置
@Configuration
@EnableCaching
public class RedisCacheConfig {

    @Bean
    public RedisConnectionFactory redisConnectionFactory() {
        RedisStandaloneConfiguration config = 
            new RedisStandaloneConfiguration("localhost", 6379);
        return new LettuceConnectionFactory(config);
    }

    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {
        // 默认配置
        RedisCacheConfiguration defaultConfig = RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofHours(1))           // 默认过期时间
                .serializeKeysWith(                      // key 序列化
                    SerializationPair.fromSerializer(
                        new StringRedisSerializer()))
                .serializeValuesWith(                    // value 序列化
                    SerializationPair.fromSerializer(
                        new GenericJackson2JsonRedisSerializer()))
                .disableCachingNullValues();             // 禁止缓存 null

        // 按 cache 名称单独配置
        Map<String, RedisCacheConfiguration> cacheConfigs = new HashMap<>();
        cacheConfigs.put("users", defaultConfig.entryTtl(Duration.ofMinutes(30)));
        cacheConfigs.put("products", defaultConfig.entryTtl(Duration.ofHours(2)));
        cacheConfigs.put("sessions", defaultConfig.entryTtl(Duration.ofDays(7)));

        return RedisCacheManager.builder(factory)
                .cacheDefaults(defaultConfig)
                .withInitialCacheConfigurations(cacheConfigs)
                .transactionAware()  // 支持事务
                .build();
    }
}
```

```java
// 方式2：Caffeine 本地缓存配置（高性能场景）
@Configuration
@EnableCaching
public class CaffeineCacheConfig {

    @Bean
    public CacheManager cacheManager() {
        CaffeineCacheManager cacheManager = new CaffeineCacheManager();
        
        cacheManager.setCaffeine(Caffeine.newBuilder()
                .initialCapacity(1000)         // 初始容量
                .maximumSize(10000)            // 最大容量，超出按 leastRecentlyUsed 淘汰
                .expireAfterWrite(1, TimeUnit.HOURS)   // 写入后过期
                .expireAfterAccess(10, TimeUnit.MINUTES) // 访问后过期
                .recordStats());               // 开启统计
        
        return cacheManager;
    }
}
```

```java
// 方式3：多级缓存（本地 + Redis）
@Configuration
@EnableCaching
public class MultiLevelCacheConfig {

    @Bean
    public CacheManager cacheManager(
            CaffeineCacheManager caffeineManager,
            RedisCacheManager redisManager) {
        
        // 使用 Spring 的 CompositeCacheManager
        CompositeCacheManager cacheManager = new CompositeCacheManager();
        cacheManager.setCacheManagers(Arrays.asList(
                caffeineManager,   // 第一级：本地缓存（最快）
                redisManager        // 第二级：Redis（分布式）
        ));
        return cacheManager;
    }
}
```

### 1.4 @CacheConfig 类级别配置

```java
// @CacheConfig 定义类级别的公共缓存配置
@CacheConfig(cacheNames = "users", cacheManager = "redisCacheManager")
public class UserService {

    // 此方法继承类级别的 cacheNames = "users"
    @Cacheable(key = "#id")
    public User getUserById(Long id) {
        return userRepository.findById(id);
    }

    // 覆盖类级别的 cacheNames
    @Cacheable(value = "adminUsers", key = "#id")
    public User getAdminUser(Long id) {
        return userRepository.findAdminById(id);
    }

    // 缓存更新
    @CachePut(key = "#result.id")
    public User updateUser(User user) {
        return userRepository.save(user);
    }

    // 单条删除
    @CacheEvict(key = "#id")
    public void deleteUser(Long id) {}

    // 清空整个缓存
    @CacheEvict(allEntries = true)
    public void deleteAllUsers() {}
}
```

### 1.5 核心注解参数详解

```java
@Cacheable(
    value = "users",                    // 缓存名称（对应 CacheManager 中的 cache）
    key = "'user:' + #id",             // SpEL 表达式计算缓存 key
    keyGenerator = "myKeyGenerator",    // 自定义 KeyGenerator（优先级高于 key）
    cacheManager = "redisCacheManager", // 指定 CacheManager Bean 名称
    cacheResolver = "myCacheResolver", // 自定义 CacheResolver
    condition = "#id > 100",           // SpEL：条件为 true 才缓存
    unless = "#result == null",        // SpEL：条件为 true 则不缓存
    sync = false                        // 是否同步（解决缓存击穿）
)
public User getUser(Long id) { ... }
```

```java
@CacheEvict(
    value = "users",
    key = "#id",               // 删除指定 key
    allEntries = false,        // true = 清空整个 users 缓存
    beforeInvocation = false   // true = 方法执行前删除，false = 方法执行后删除
)
public void deleteUser(Long id) { ... }
```

```java
@CachePut(
    value = "users",
    key = "#result.id",        // #result 指方法返回值
    condition = "#user.active",
    unless = "#result.vip == false"
)
```

## 🔥 扩展追问 1：`key`、`condition`、`unless` 三个参数的区别和使用场景？

> A: **三者都是 SpEL 表达式，但作用时机和用途完全不同。**

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        三参数对比                                           │
├──────────────────┬──────────────────────────────────────────────────────────┤
│ 参数              │ 作用时机 & 用途                                           │
├──────────────────┼──────────────────────────────────────────────────────────┤
│ key              │ 计算缓存的键（缓存放在哪个 slot）                          │
│                  │ key = "'user:' + #id" → 缓存 key = "user:123"           │
│                  │ 默认 key = methodName + methodParams                      │
├──────────────────┼──────────────────────────────────────────────────────────┤
│ condition        │ 方法执行前判断，满足条件才进入缓存逻辑                      │
│                  │ condition = "#id > 100" → id≤100 不走缓存，直接查 DB     │
│                  │ 用于：过滤不需要缓存的小数据/冷数据                        │
├──────────────────┼──────────────────────────────────────────────────────────┤
│ unless           │ 方法执行后判断，满足条件则跳过缓存操作                      │
│                  │ unless = "#result == null" → 结果为 null 不缓存          │
│                  │ unless = "#result.vip == false" → 非 VIP 不缓存          │
│                  │ 用于：过滤异常结果、空值、特殊标记的数据                    │
└──────────────────┴──────────────────────────────────────────────────────────┘

常用 SpEL 变量：

┌────────────────────────────────────────────────────────────────────────────┐
│  变量              │ 含义                      │ 示例                       │
├───────────────────┼──────────────────────────┼───────────────────────────┤
│ #root.methodName  │ 当前方法名                │ #root.method.name          │
│ #root.targetClass │ 当前目标类                │ #root.targetClass.name     │
│ #root.args[0]     │ 当前方法参数（按顺序）    │ #root.args[0]             │
│ #p0 / #a0         │ 第 0 个参数（简写）        │ #p0 = #id                 │
│ #result           │ 方法返回值（仅 @CachePut  │ #result.id                │
│                   │ 和 @Cacheable 的 unless） │                            │
│ #exception        │ 方法抛出的异常            │ #exception.message        │
└───────────────────┴──────────────────────────┴───────────────────────────┘
```

```java
// 实战示例
@Cacheable(
    value = "product",
    key = "'product:' + #category + ':' + #page",  // 复合 key
    condition = "#page <= 100",    // 前 100 页才缓存
    unless = "#result.total <= 0"   // 空结果不缓存
)
public Page<Product> searchProducts(String category, int page) {
    // ...
}

// key 表达式写法对比
key = "#id"                          // 直接用参数
key = "'user:' + #user.id"           // 嵌套属性
key = "#user.name + ':' + #type"     // 字符串拼接
key = "#root.methodName + '(' + #root.args[0] + ')'"  // 方法签名式
```

## 🔥 扩展追问 2：`sync=true` 是什么？能解决缓存击穿吗？有什么限制？

> A: **`sync=true` 通过分布式锁保证只有一个线程去查数据库，可以缓解但不能完全解决缓存击穿。**

```
缓存击穿场景：

┌────────────────────────────────────────────────────────────────────────────┐
│  sync = false (默认)                                                         │
│                                                                              │
│  1000个并发请求同时访问 user:123                                              │
│                                                                              │
│  ┌────────────────────────────────────────────────────────────────────────┐ │
│  │ t1: 1000个请求同时查缓存 → 全部 miss                                   │ │
│  │ t2: 1000个请求同时查 DB ← 击穿！1000个 DB 查询并发                      │ │
│  │ t3: 1000个请求同时写缓存                                               │ │
│  └────────────────────────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────────────┐
│  sync = true                                                                 │
│                                                                              │
│  ┌────────────────────────────────────────────────────────────────────────┐ │
│  │ t1: 第1个请求获取锁，查 DB，写缓存                                     │ │
│  │ t2: 其他 999 个请求等待锁                                              │ │
│  │ t3: 锁释放后，其他请求直接读缓存 ← 避免击穿                           │ │
│  └────────────────────────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────────────────────┘
```

```java
// sync=true 的使用
@Cacheable(value = "users", key = "#id", sync = true)
public User getUserById(Long id) {
    // 只有获取到锁的线程会执行这里
    // 其他线程等待锁释放后直接读缓存
    return userRepository.findById(id);
}
```

```
sync=true 的限制：

┌────────────────────────────────────────────────────────────────────────────┐
│  限制1：只解决单节点击穿，分布式环境下多节点依然可能击穿                      │
│                                                                              │
│  Node1: 请求A 获取锁 → 查 DB                                                │
│  Node2: 请求B 获取锁（不同节点，无锁） → 查 DB  ← 击穿                      │
│                                                                              │
│  解决方案：必须配合 Redis 分布式锁（SETNX / Redisson）                       │
├────────────────────────────────────────────────────────────────────────────┤
│  限制2：只有一个线程等待，其他等待线程会抛出异常                             │
│                                                                              │
│  Spring Cache sync 内部用的是 JUC 的 ReentrantLock                         │
│  如果等待超时（默认10秒），会抛出 InterruptedException                      │
├────────────────────────────────────────────────────────────────────────────┤
│  限制3：不能和 unless 一起用（unless 需要执行完方法才能判断）                │
│                                                                              │
│  @Cacheable(sync = true, unless = "#result == null")  ← 编译报错          │
├────────────────────────────────────────────────────────────────────────────┤
│  限制4：本地缓存（如 Caffeine）使用 sync 无意义                             │
│                                                                              │
│  Caffeine 是单节点缓存，sync 锁只在本机生效                                  │
│  分布式缓存才需要 sync（Redis + sync 有意义）                                │
└────────────────────────────────────────────────────────────────────────────┘
```

```java
// 生产环境推荐：分布式锁 + 缓存击穿防护
@Cacheable(value = "users", key = "#id")
public User getUserById(Long id) {
    return getUserByIdWithLock(id);
}

private User getUserByIdWithLock(Long id) {
    String lockKey = "lock:user:" + id;
    String lockValue = UUID.randomUUID().toString();
    
    try {
        // 1. 尝试获取分布式锁
        Boolean acquired = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, lockValue, Duration.ofSeconds(10));
        
        if (Boolean.TRUE.equals(acquired)) {
            // 2. 获取锁成功：查 DB → 写缓存
            User user = userRepository.findById(id);
            if (user != null) {
                redisTemplate.opsForValue().set(
                    "user:" + id, user, Duration.ofHours(1));
            }
            return user;
        } else {
            // 3. 未获取锁：等待后重试读缓存
            Thread.sleep(100);
            Object cached = redisTemplate.opsForValue().get("user:" + id);
            if (cached != null) {
                return (User) cached;
            }
            // 兜底：递归重试
            return getUserByIdWithLock(id);
        }
    } finally {
        // 4. 释放锁（只能释放自己的锁）
        String currentValue = (String) redisTemplate.opsForValue().get(lockKey);
        if (lockValue.equals(currentValue)) {
            redisTemplate.delete(lockKey);
        }
    }
}
```

## 🔥 扩展追问 3：@CacheEvict 的 `beforeInvocation=true` vs `false` 的区别？

> A: **`beforeInvocation=true` 在方法执行前清缓存，`false`（默认）在方法执行成功后才清缓存。**

```
beforeInvocation = false（默认）：

┌────────────────────────────────────────────────────────────────────────────┐
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │  步骤1: 执行 deleteUser(123) 方法体                                 │  │
│  │  步骤2: 捕获异常                                                     │  │
│  │  步骤3: 方法抛异常 → 不执行删除缓存 ← 缓存未清理！                    │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                              │
│  场景：如果 deleteUser 失败，缓存里还有旧数据，下次查到的还是脏数据         │
└────────────────────────────────────────────────────────────────────────────┘

beforeInvocation = true：

┌────────────────────────────────────────────────────────────────────────────┐
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │  步骤1: 先删除缓存                                                   │  │
│  │  步骤2: 执行 deleteUser(123) 方法体                                  │  │
│  │  步骤3: 方法抛异常 → 缓存已清理，下次重新查 DB                       │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                              │
│  场景：无论方法成功失败，缓存都会被清理（安全策略）                          │
└────────────────────────────────────────────────────────────────────────────┘

实战建议：

// 删除操作：建议 beforeInvocation = true
@CacheEvict(value = "users", key = "#id", beforeInvocation = true)
public void deleteUser(Long id) {
    // 即使抛异常，缓存也会被清理
}

// 更新操作：建议 beforeInvocation = false（方法成功才清缓存）
@CacheEvict(value = "users", key = "#result.id", beforeInvocation = false)
public User updateUser(User user) {
    // 只有方法成功返回才清理缓存
}
```

## 🔥 扩展追问 4：Spring Cache 和 MyBatis 二级缓存能共存吗？有什么坑？

> A: **可以共存，但有严重的数据一致性问题，必须谨慎使用。**

```
共存场景的问题：

┌────────────────────────────────────────────────────────────────────────────┐
│  写操作流程：                                                              │
│                                                                              │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │  updateUser(user)                                                   │  │
│  │      │                                                              │  │
│  │      ├── MyBatis 一级缓存（SqlSession 级别） ← 清理                │  │
│  │      ├── MyBatis 二级缓存（Mapper 级别）   ← 清理                 │  │
│  │      └── Spring Cache (@CacheEvict)       ← 清理                 │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
│                                                                              │
│  读操作流程（出现问题）：                                                   │
│                                                                              │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │  getUser(id)                                                        │  │
```

## 🔥 扩展追问 5：如何自定义 KeyGenerator？

> A: **实现 `KeyGenerator` 接口，配合 `@Cacheable(keyGenerator)` 使用。**

```java
// 1. 定义 KeyGenerator
@Component
public class MyKeyGenerator implements KeyGenerator {

    @Override
    public Object generate(Object target, Method method, Object... params) {
        // 生成格式：类名.方法名:参数1:参数2:参数3
        StringBuilder key = new StringBuilder();
        key.append(target.getClass().getSimpleName())
           .append(".")
           .append(method.getName())
           .append(":");
        
        for (int i = 0; i < params.length; i++) {
            if (i > 0) key.append(":");
            Object param = params[i];
            if (param == null) {
                key.append("NULL");
            } else if (param instanceof String) {
                key.append((String) param);
            } else if (param instanceof Number) {
                key.append(param);
            } else {
                // 对于复杂对象，尝试调用 toString()
                key.append(param.toString());
            }
        }
        
        return key.toString();
    }
}

// 2. 使用自定义 KeyGenerator
@Cacheable(value = "users", keyGenerator = "myKeyGenerator")
public User getUserById(Long id) { ... }

// 3. 全局配置（替代默认的 SimpleKeyGenerator）
@Configuration
@EnableCaching
public class CacheConfig {
    
    @Bean
    @Override
    public KeyGenerator keyGenerator() {
        return new MyKeyGenerator();
    }
}
```

## 3. 一句话总结

> **`@Cacheable` 负责查时缓存、`@CachePut` 负责写时更新、`@CacheEvict` 负责删除缓存；Spring Cache 通过 `CacheManager` 抽象层解耦具体实现（Redis/Caffeine/Ehcache），`key` 计算缓存 slot、`condition` 在执行前过滤、`unless` 在执行后过滤、`sync=true` 用本地锁缓解击穿但需配合分布式锁才能完全解决。**

## 4. 面试加分话术

> "我们项目里 Spring Cache 落地得比较细。拿用户中心举例：热点用户数据走 `@Cacheable(value = 'users', key = '#id', sync = true)`，但 sync=true 在多节点下只能锁本地，所以我们还配了 Redis SETNX 分布式锁，双保险防击穿。缓存序列化用的 `GenericJackson2JsonRedisSerializer`，配置了 `cacheNullValues = false`，避免 null 值被缓存导致业务空指针。更新和删除都用 `@CacheEvict(beforeInvocation = true)`，确保即使方法抛异常缓存也被清理。分页查询没走 Spring Cache，因为分页参数组合太多容易击穿，用的是手动 `redisTemplate.opsForHash` 自己做热点 key 缓存。踩过一个坑：MyBatis 二级缓存和 Spring Cache 共存时出现数据不一致，后来直接把 MyBatis 二级缓存禁了，全部走 Spring Cache + Redis。"

```java
// 推荐 CacheManager 配置模板
@Bean
public CacheManager cacheManager(RedisConnectionFactory factory) {
    RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofHours(1))
            .serializeKeysWith(SerializationPair.fromSerializer(new StringRedisSerializer()))
            .serializeValuesWith(SerializationPair.fromSerializer(new GenericJackson2JsonRedisSerializer()))
            .disableCachingNullValues()           // 禁止缓存 null
            .computePrefixWith(CacheNamePrefix.INSTANCE);
    
    return RedisCacheManager.builder(factory)
            .cacheDefaults(config)
            .withInitialCacheConfigurations(Map.of(
                "hotData", config.entryTtl(Duration.ofMinutes(30)),
                "coldData", config.entryTtl(Duration.ofDays(7))
            ))
            .transactionAware()
            .build();
}
```
