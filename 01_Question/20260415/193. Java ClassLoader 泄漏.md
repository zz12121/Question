# 193. Java ClassLoader 泄漏

## 1. 基础流程

### 什么是 ClassLoader 泄漏？

ClassLoader 泄漏是指 **ClassLoader 无法被 GC 回收**，导致其加载的所有类（包括静态变量、Class 对象本身）都常驻内存，最终引发 OOM。

```
┌─────────────────────────────────────────────────────────────┐
│                   正常 ClassLoader vs 泄漏                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【正常情况】                                                │
│  ┌──────────┐    释放后     ┌──────────┐                   │
│  │ClassLoader│ ─────────→  │  卸载成功  │                   │
│  │ + ClassA  │             │  GC 回收  │                   │
│  │ + ClassB  │             └──────────┘                   │
│  └──────────┘                                               │
│       ↑                                                     │
│   无引用                                                    │
│                                                             │
│  【泄漏情况】                                                │
│  ┌──────────┐    隐藏引用链                                 │
│  │ClassLoader│ ─────────→ ⚠️ 无法释放                       │
│  │ + ClassA  │      ↑                                       │
│  │ + ClassB  │      │                                       │
│  └──────────┘      │                                       │
│       ↑            │                                       │
│       │            │                                       │
│  ┌────┴────────────┐                                      │
│  │  ThreadLocal    │                                      │
│  │  静态集合        │                                      │

```

## 2. 扩展追问

### 🔥 扩展追问 1：WeakHashMap 能解决 ClassLoader 泄漏吗？

> A: **WeakHashMap 可以缓解，但并不能完全解决所有泄漏场景。**

```java
// WeakHashMap 的工作原理
// Key（ClassLoader）没有强引用时，Entry 会被 GC 回收

Map<ClassLoader, Map<String, Class<?>>> cache = 
    new WeakHashMap<>();

// ✓ 优点：ClassLoader 无其他引用时，Entry 自动清理
// ✗ 缺点：

// 1. Value 仍然持有 Key 的引用（除非用 WeakReference）
// 如果 value 被静态集合持有，即使 key 可回收也无效

// 2. ThreadLocal 的 Key 是弱引用，但 Value 是强引用
// 正确用法：
ThreadLocal<Connection> tl = new ThreadLocal<>();
try {
    Connection conn = getConnection();
    tl.set(conn);
    // 业务逻辑
} finally {
    tl.remove();  // 必须手动 remove！
}
```

### 🔥 扩展追问 2：如何预防 Tomcat 热部署泄漏？

> A: **通过配置、代码规范和监控三管齐下。**

```java
// ========== 1. Tomcat 配置优化 ==========
// conf/context.xml
<Context reloadable="true">
    <!-- 启用内存泄漏检测 -->
    <Listener className="org.apache.catalina.core.JreMemoryLeakPreventionListener"/>
    <Listener className="org.apache.catalina.core.ThreadLocalLeakPreventionListener"/>
</Context>

// ========== 2. 代码规范 ==========
public class SafeServlet {
    
    // ❌ 错误：使用线程池 + ThreadLocal
    private static ThreadLocal<User> userHolder = new ThreadLocal<>();
    
    // ✓ 正确：使用 RequestContextHolder (Spring)
    public void process() {
        // Spring 的 RequestContextHolder 基于 ThreadLocal
        // 但通过 Listener 在请求结束时清理
        User user = RequestContextHolder.currentRequestAttributes()
            .getAttribute("user", RequestAttributes.SCOPE_REQUEST);
    }
    
    // ✓ 正确：使用后必ず remove
    public void safeProcess() {
        Connection conn = null;
        try {
            conn = ds.getConnection();
            connectionHolder.set(conn);
            // 业务逻辑
        } finally {
            connectionHolder.remove();  // 必须！
        }
    }
}

// ========== 3. 启动检查 ==========
// catalina.sh
JAVA_OPTS="$JAVA_OPTS -Djava.security.manager \
  -XX:+EnableObjectAllocationInEdenSpace \
  -XX:+TraceClassLoading \
  -XX:+TraceClassUnloading"
```

### 🔥 扩展追问 3：OSGi 的 ClassLoader 泄漏有什么特殊之处？

> A: **OSGi 有独特的类加载器委托模型，泄漏更难排查。**

```
┌─────────────────────────────────────────────────────────────┐
│                   OSGi ClassLoader 模型                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Bundle A ──── imports ────→ Bundle B (导出 Package)         │
│      ↓                                                      │
│  BundleClassLoader (A)                                      │
│      ↓                                                      │
│  Delegation Chain:                                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Bundle A's ClassLoader                              │   │
│  │       ↓                                              │   │
│  │  Parent ClassLoader (AppClassLoader)                 │   │
│  │       ↓                                              │   │
│  │  Bootstrap ClassLoader                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  泄漏场景：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Bundle A 引用了 Bundle B 导出的类 X                   │   │
│  │  Bundle B 卸载时，类 X 的 Class 对象仍被 A 持有        │   │
│  │  A 的 ClassLoader 持有 → B 的 ClassLoader 无法卸载    │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  解决：BundleContext.listeners + BundleListener            │
│  卸载前必须注销所有监听器                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 🔥 扩展追问 4：Spring 如何避免 ClassLoader 泄漏？

> A: **Spring 通过合理的生命周期管理和清理机制避免泄漏。**

```java
// ========== Spring 的防护机制 ==========

// 1. DisposableBean + @PreDestroy
@Component
public class MyService implements DisposableBean {
    
    private Connection connection;
    
    @PreDestroy
    public void cleanup() {
        // 关闭连接、清理资源
        if (connection != null) {
            connection.close();
        }
    }
}

// 2. RequestContextListener 清理 ThreadLocal
// web.xml
<listener>
    <listener-class>
        org.springframework.web.context.request.RequestContextListener
    </listener-class>
</listener>

// 3. OpenEntityManagerInViewInterceptor
// 防止 EntityManager 泄漏

// 4. 自定义 Scope 清理
@Configuration
public class MyConfig {
    
    @Bean
    @Scope("session")
    public MyBean myBean() {
        return new MyBean();
    }
    
    // 自定义 scope 的清理回调
    @Bean
    public CustomScopeConfigurer customScopeConfigurer() {
        CustomScopeConfigurer configurer = new CustomScopeConfigurer();
        configurer.addScope("myScope", new CustomScope() {
            private final Map<String, Object> beans = new ConcurrentHashMap<>();
            
            @Override
            public Object get(String name, ObjectFactory<?> factory) {
                return beans.computeIfAbsent(name, k -> factory.getObject());
            }
            
            @Override
            public Object remove(String name) {
                return beans.remove(name);
            }
            
            @Override
            public void registerDestructionCallback(String name, Runnable callback) {
                // 清理回调
            }
            
            @Override
            public Object resolveContextualObject(String key) {
                return null;
            }
            
            @Override
            public String getConversationId() {
                return null;
            }
        });
        return configurer;
    }
}
```

## 3. 一句话总结

> **ClassLoader 泄漏的本质是"隐藏的引用链"阻止了 ClassLoader 卸载，导致其加载的所有类永久驻留内存；排查靠 jmap/Arthas/Profiler 追踪引用链，解决靠正确清理 ThreadLocal、静态集合、监听器等引用源。**

## 4. 面试加分话术

> "我之前排查过一个 Tomcat 热部署导致内存泄漏的生产问题：应用每2天 OOM一次，最后定位到是 DAO 层用了 ThreadLocal 持有数据库连接，而连接是从 WebappClassLoader 加载的 DataSource 获取的。线程池复用线程时，ThreadLocal 中的连接一直没 remove，导致 ClassLoader 无法卸载。后来用 Arthas 的 `classloader` 命令发现 WebappClassLoader 实例数持续增长，配合 `jmap -histo` 确认是 Class 对象累积。解决方案是封装 ThreadLocal 工具类，在 finally 块强制清理。从那以后我给团队定了规范：所有 ThreadLocal 使用必须配对 remove，并加入 Code Review 检查项。"

**追问预判：**
- "WeakReference 和 SoftReference 的区别？" → WeakReference 每次 GC 都回收，SoftReference 只有内存不足时才回收
- "如何验证 ClassLoader 卸载成功？" → 用 `-verbose:class` 或 `-XX:+TraceClassUnloading` 观察日志
