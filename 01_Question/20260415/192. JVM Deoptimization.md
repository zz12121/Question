## 1. 基础流程

### 什么是 Deoptimization？

Deoptimization 是 JVM 将 JIT 编译后的优化代码**撤销**，回退到解释执行或重新编译的过程。

```
┌─────────────────────────────────────────────────────────────┐
│                      JVM 执行引擎                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [解释执行]  ←─────── Deoptimization ──────── [JIT 编译]    │
│       ↑                                            │        │
│       │                                            │        │
│   OopMap 安全点                                激进优化      │
│   无优化代码                               (Speculative)     │
│                                                             │
└─────────────────────────────────────────────────────────────┘

优化类型：
┌────────────────────┐     ┌────────────────────┐
│   逃逸分析优化       │     │   内联优化          │
│   - 对象栈上分配     │     │   - 方法直接展开    │
│   - 锁粗化/锁消除    │     │   - 虚调用→直调用   │
│   - 标量替换         │     │                    │
└────────────────────┘     └────────────────────┘
         ↓                           ↓
    这些优化会被 Deopt 撤销      这些优化会被 Deopt 撤销
```

### 去优化的触发场景

```
┌─────────────────────────────────────────────────────────────┐
│                    Deoptimization 触发条件                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1️⃣ 类型不匹配 (Type Check Failure)                         │
│     - 假设 A 是 A 类型 → 内联为直调用                        │
│     - 运行时发现是 B 子类 → 撤销内联                         │
│                                                             │
│  2️⃣ 类继承关系变化 (Class Hierarchy Change)                   │
│     - 加载了新的子类                                          │
│     - CHA (Class Hierarchy Analysis) 假设失效               │
│                                                             │
│  3️⃣ 反优化点 (Monotonic Counter)                             │
│     - 热点计数超过阈值触发重编译                              │
│     - 条件再次不满足时回退                                    │
│                                                             │
│  4️⃣ 栈上替换失败 (OSR Failure)                               │
│     - 正在执行的方法被替换失败                                │
│                                                             │
│  5️⃣ 假设被打破 (Speculation Failure)                         │
│     - 分支预测失败                                           │
│     - 类型预测失败                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 核心配置参数

```bash
# 启用/禁用 Deoptimization 相关
-XX:+UnlockDiagnosticVMOptions
-XX:+DoEscapeAnalysis          # 开启逃逸分析（默认开启）
-XX:+AlwaysPreTouch            # 预热时分配所有内存
-XX:+TraceDeoptimization       # 追踪去优化过程（诊断用）
-XX:+PrintCompilation         # 打印编译日志

# OSR 配置
-XX:CompileThreshold=1000      # 触发编译的调用次数
-XX:OnStackReplacePercentage=140  # OSR 触发阈值百分比

# 类卸载相关
-XX:+ClassUnloading            # 启用类卸载（默认开启）
-XX:CMSClassUnloadingEnabled   # CMS GC 时类卸载
```

### Deoptimization 过程详解

```
┌─────────────────────────────────────────────────────────────┐
│                    Deoptimization 执行流程                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 检测到 Deopt 条件                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 例：虚表检查发现类继承关系已变化                        │   │
│  └─────────────────────────────────────────────────────┘   │
│                          ↓                                  │
│  Step 2: 设置 Deopt 标记                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ nmethod 被标记为 "not_entrant"                        │   │
│  │ 阻止新调用进入，但当前执行可以完成                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                          ↓                                  │
│  Step 3: 安全点挂起                                         │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 所有线程到达安全点，停止执行                           │   │
│  │ 保存寄存器状态到栈帧                                   │   │
│  └─────────────────────────────────────────────────────┘   │
│                          ↓                                  │
│  Step 4: 生成 Uncommon Trap                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 在反优化点插入 trap 帧                                │   │
│  │ 替换原编译代码                                        │   │
│  └─────────────────────────────────────────────────────┘   │
│                          ↓                                  │
│  Step 5: 返回解释执行                                       │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 恢复解释器状态，继续执行                               │   │
│  │ 计数器累积，条件满足时重新触发 JIT                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 代码示例：逃逸分析被 Deopt 的场景

```java
public class DeoptDemo {
    
    // ❌ 内联 + 锁消除优化
    // 假设 JIT 认定 lock 对象不逃逸 → 消除 synchronized
    // 如果反射修改了对象，触发 Deopt
    public synchronized void process() {
        // ... 业务逻辑
    }
    
    // ❌ 标量替换被 Deopt
    // JIT 将 point.x, point.y 替换为寄存器变量
    // 如果对象被逸出到其他线程，触发 Deopt
    public double distance(Point p) {
        return Math.sqrt(p.x * p.x + p.y * p.y);
    }
    
    // ❌ 虚调用被内联后被 Deopt
    public void print(List<Integer> list) {
        // 假设只有 ArrayList，JIT 内联
        // 如果运行时传入 LinkedList，触发 Deopt
        for (Integer i : list) {
            System.out.println(i);
        }
    }
}

// 触发 Deopt 的常见场景
public class DeoptTrigger {
    
    // 场景1：Class Hierarchy Analysis 失效
    public static void triggerCHA() {
        List<String> list = new ArrayList<>();
        printList(list);  // JIT: 只有 ArrayList，内联
        
        // 模拟：动态加载 LinkedList 子类
        // 反射触发类加载
        Class.forName("java.util.LinkedList");
        
        printList(list);  // ← Deopt: CHA 假设失效
    }
    
    // 场景2：类型不匹配
    public static void triggerTypeMismatch() {
        Object obj = "initial";  // 编译时认为是 String
        // JIT 内联 String.length()
        
        obj = new Object();  // 运行时类型变化
        
        // obj.length() ← Deopt: 类型不匹配
    }
    
    private static void printList(List<?> list) {
        list.forEach(System.out::println);
    }
}
```

## 2. 扩展追问

### 🔥 扩展追问 1：OSR 也会被 Deopt 吗？

> A: **是的，OSR 同样会被 Deoptimize。**

```
┌─────────────────────────────────────────────────────────────┐
│                 On-Stack Replacement (OSR)                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  解释执行中 ────── 触发阈值 ──────→ 编译中                   │
│      ↑                                  │                   │
│      │                                  ↓                   │
│      │                           OSR 替换到编译版本          │
│      │                                  │                   │
│      │         ┌───────────────────────┘                   │
│      │         ↓                                           │
│      │    可能触发 Deopt ←─── 执行中条件变化                 │
│      │         ↓                                           │
│      └──── 返回解释执行 ←───────────────────────────────────┘
│                                                             │
│  OSR 触发时机：                                              │
│  - 循环计数达到 OnStackReplacePercentage * CompileThreshold │
│  - 解释执行"太慢"，不等方法返回就切换                         │
│                                                             │
│  OSR Deopt 场景：                                           │
│  1. 编译后的代码发现新的反优化点                             │
│  2. 类继承关系在 OSR 执行期间发生变化                         │
│  3. GC 导致栈布局变化                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```java
public class OSRExample {
    public static void main(String[] args) {
        long sum = 0;
        // 解释执行一段时间，循环计数达到 OSR 阈值
        // JIT 编译完成后，在循环中间进行 OSR
        for (int i = 0; i < 10_000_000; i++) {
            sum += i;  // OSR 发生在这里
        }
        
        // OSR 后，如果类被卸载或条件变化
        // 可能触发 Deopt，回退到解释执行
    }
}
```

### 🔥 扩展追问 2：Deoptimization 和类卸载的关系？

> A: **类卸载会触发依赖该类的 JIT 代码被 Deoptimize。**

```
┌─────────────────────────────────────────────────────────────┐
│                 类卸载触发 Deoptimization                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  类卸载条件 (满足全部)：                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 1. 类实例全部死亡                                    │   │
│  │ 2. 加载类的 ClassLoader 已死亡                       │   │
│  │ 3. 没有 reflection 引用                              │   │
│  │ 4. 无编译代码持有该类引用                             │   │
│  └─────────────────────────────────────────────────────┘   │
│                          ↓                                  │
│  类卸载执行：                                                │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ JVM 移除类的 Metadata                                │   │
│  │ 卸载前：先 Deopt 所有使用该类的 nmethod               │   │
│  │ 因为类结构（虚表、方法表）已不存在                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                          ↓                                  │
│  后果：                                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ - 依赖该类的编译代码必须撤销                          │   │
│  │ - 后续调用走解释执行                                  │   │
│  │ - 条件满足时可重新编译                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 🔥 扩展追问 3：常见去优化类型有哪些？

> A: **主要有三种去优化类型。**

| Deopt 类型 | 触发条件 | 处理方式 |
|------------|----------|----------|
| **Uncommon Trap** | 假设不常发生的分支被执行 | 跳到解释器执行 |
| **Method Handle Trap** | MethodHandle 调用点未绑定 | 重新解析 |
| **Branch Trap** | 分支预测失败 | 返回解释执行 |

```java
// Uncommon Trap 示例
public class UncommonTrapDemo {
    
    // JIT 假设 x > 0 是常见路径，内联优化
    // 如果 x <= 0 被执行，触发 uncommon trap
    public int optimize(int x) {
        if (x > 0) {
            return fastPath(x);  // 内联优化
        } else {
            return slowPath(x);  // uncommon trap
        }
    }
    
    // 触发 Deopt 的日志
    // [bci:10]Uncommon trap occurred in method:optimize
    // Reason: null_check
}
```

### 🔥 扩展追问 4：如何诊断 Deoptimization 问题？

> A: **使用 JVM 诊断工具和日志。**

```bash
# 1. 开启 Deoptimization 日志
-XX:+UnlockDiagnosticVMOptions
-XX:+TraceDeoptimization
-XX:+LogCompilation  # 编译日志

# 2. 查看日志输出
# Deoptimization events 会记录类似：
# <deoptimization compiled_id='42' method='process' bci='15' reason='type_check_divergence'/>

# 3. 使用 JMH 基准测试验证性能
public class DeoptBenchmark {
    @Benchmark
    public void measureDeopt(Blackhole bh) {
        // Warmup 后触发 OSR
        // 如果经常 Deopt，性能反而下降
    }
}

# 4. 使用 -XX:+PrintCompilation 观察
# 每行编译日志的 flags 含义：
# % = OSR编译
# ! = 有异常处理器
# s = 同步编译
```

## 3. 一句话总结

> **JVM Deoptimization 是激进优化的"安全网"——当 JIT 编译的假设被打破时，将代码回退到解释执行，确保程序正确性；与类卸载配合，确保无引用类可被回收；OSR 本质上也是一种特殊的 Deopt 机制。**

## 4. 面试加分话术

> "在实际项目中，我们遇到过一个问题：生产环境突然大量 Deoptimization 导致性能骤降。后来定位到是因为代码中用了策略模式 + 反射加载实现类，JIT 优化后触发了类继承关系变化的去优化。通过添加 `-XX:+PrintCompilation` 日志和分析 `hs_err*.log`，定位到了问题根因。最终通过将反射缓存静态化、减少动态类加载解决了这个问题。这个经历让我深刻理解到 JIT 优化是把双刃剑——必须了解它的边界条件才能写出高性能代码。"

**追问预判：**
- "Deoptimization 会影响 GC 吗？" → 会，因为 nmethod 会被标记为弱引用可回收
- "如何避免不必要的 Deoptimization？" → 避免反射修改对象类型、减少动态类加载
