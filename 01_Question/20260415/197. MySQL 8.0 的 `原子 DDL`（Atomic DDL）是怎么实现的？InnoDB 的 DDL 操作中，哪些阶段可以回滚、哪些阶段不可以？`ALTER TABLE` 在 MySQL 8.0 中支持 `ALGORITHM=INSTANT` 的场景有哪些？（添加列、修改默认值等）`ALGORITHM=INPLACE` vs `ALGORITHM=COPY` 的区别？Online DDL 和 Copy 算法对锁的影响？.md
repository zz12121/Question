# 197. MySQL 8.0 原子 DDL 与 ALTER TABLE 算法

## 1. 基础流程

### 什么是原子 DDL？

MySQL 8.0 引入的原子 DDL 保证 DDL 操作要么**完全成功**，要么**完全回滚**，不会留下半成品表或损坏的元数据。

```

│              MySQL 8.0 原子 DDL 执行流程                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【DDL 执行前】                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  mysql.tables:                                        │   │
│  │  - name='t1', normal                                │   │
│  └─────────────────────────────────────────────────────┘   │

│  │                                                       │   │
│  │  BEGIN;                                               │   │
```

### 核心配置与 SQL 示例

```sql
-- ========== 1. 原子 DDL 配置 ==========
-- MySQL 8.0 默认启用，无需配置
SHOW VARIABLES LIKE 'ddl_log';
-- +---------------+-------+
-- | Variable_name | Value |
-- | ddl_log.general | ON   |
-- +---------------+-------+

-- 查看 DDL 日志
SELECT * FROM performance_schema.innodb_ddl_log;

-- ========== 2. ALGORITHM 指定 ==========
-- 强制使用 INSTANT（不支持则报错）
ALTER TABLE t1 ADD COLUMN c1 INT, ALGORITHM=INSTANT;

-- 强制使用 INPLACE（不支持则报错）
ALTER TABLE t1 ADD INDEX idx_c1(c1), ALGORITHM=INPLACE;

-- 强制使用 COPY（不支持则报错）
ALTER TABLE t1 MODIFY c1 VARCHAR(100), ALGORITHM=COPY;

-- 让 MySQL 自动选择最佳算法（默认）
ALTER TABLE t1 ADD COLUMN c2 INT, ALGORITHM=DEFAULT;

-- ========== 3. LOCK 选项 ==========
-- 最小化锁（不支持则报错）
ALTER TABLE t1 ADD COLUMN c1 INT, LOCK=NONE;

-- 允许排他锁
ALTER TABLE t1 ADD COLUMN c1 INT, LOCK=SHARED;

-- 默认：必要时升级为排他锁
ALTER TABLE t1 ADD COLUMN c1 INT, LOCK=DEFAULT;

-- ========== 4. Online DDL 组合示例 ==========
-- 添加列（INSTANT，无需排他锁）
ALTER TABLE orders 
    ADD COLUMN shipping_status VARCHAR(20) DEFAULT 'pending',
    ALGORITHM=INSTANT, LOCK=NONE;

-- 添加索引（INPLACE，轻量锁）
ALTER TABLE orders 
    ADD INDEX idx_status(shipping_status),
    ALGORITHM=INPLACE, LOCK=NONE;

-- 修改列类型（COPY，全程排他锁）
ALTER TABLE orders 
    MODIFY COLUMN total_amount DECIMAL(12,2),
    ALGORITHM=COPY, LOCK=SHARED;

-- ========== 5. 查看 DDL 执行情况 ==========
-- 长时间 DDL 可以中断
KILL QUERY <connection_id>;

-- 查看正在执行的 DDL
SELECT * FROM information_schema.INNODB_TRX 
WHERE trx_query LIKE 'ALTER TABLE%';

-- ========== 6. Instant ADD COLUMN 限制 ==========
-- 支持的操作
ALTER TABLE t1 ADD COLUMN c1 INT AFTER col_last;        -- 添加列（末尾）
ALTER TABLE t1 ADD COLUMN c2 INT, ALGORITHM=INSTANT;     -- 添加列
ALTER TABLE t1 ALTER COLUMN col1 SET DEFAULT 100;        -- 修改默认值
ALTER TABLE t1 ALTER COLUMN col1 DROP DEFAULT;           -- 删除默认值
ALTER TABLE t1 ADD INDEX idx(c1);                         -- 添加索引
ALTER TABLE t1 DROP INDEX idx;                            -- 删除索引

-- 不支持 INSTANT 的操作（会退化为 COPY）
ALTER TABLE t1 ADD COLUMN c1 INT FIRST;                   -- 添加到首位 ✗
ALTER TABLE t1 MODIFY COLUMN c1 VARCHAR(100);             -- 修改列类型 ✗
ALTER TABLE t1 DROP COLUMN c1;                             -- 删除列 ✗
ALTER TABLE t1 ADD COLUMN c1 INT, ADD INDEX idx(c1);       -- 同时添加列和索引 ✗
```

## 2. 扩展追问

### 🔥 扩展追问 1：DDL LOG 是如何实现原子性的？

> A: **DDL LOG 记录操作轨迹，支持失败回滚和恢复。**

```sql
-- DDL LOG 表结构（内部）
-- 位于 mysql.innodb_ddl_log（隐藏表）

-- DDL 操作流程：
-- ┌─────────────────────────────────────────────────────────┐
-- │  1. DDL 开始时：                                        │
-- │     INSERT INTO ddl_log (操作类型, 目标, 备份信息)     │
-- │                                                       │
-- │  2. DDL 执行中：                                       │
-- │     UPDATE ddl_log SET status='executing'            │
-- │                                                       │
-- │  3. DDL 成功：                                          │
-- │     UPDATE ddl_log SET status='committed'            │
-- │     -- 异步删除 ddl_log 记录                           │
-- │                                                       │
-- │  4. DDL 失败：                                          │
-- │     -- 读取 ddl_log 中未提交记录                       │
-- │     -- 反向执行回滚操作                                │
-- │     -- 删除 ddl_log 记录                              │
-- └─────────────────────────────────────────────────────────┘

-- 查看 DDL 重建日志
SHOW VARIABLES LIKE 'innodb_print_ddl_logs';
SET GLOBAL innodb_print_ddl_logs = ON;

-- 失败恢复示例：
-- 假设 ADD COLUMN 中途失败
-- 1. DDL LOG 记录：需要删除临时添加的列定义
-- 2. 读取 LOG：发现有未完成的 ADD COLUMN
-- 3. 回滚：删除列定义，恢复原表结构
```

### 🔥 扩展追问 2：INSTANT ADD COLUMN 的内部实现原理？

> A: **不修改数据文件，只更新数据字典，在读取时动态拼接。**

```sql
-- INSTANT ADD COLUMN 原理图
-- ┌─────────────────────────────────────────────────────────┐
-- │  添加列前的数据文件（row format）                         │
-- │  ┌──────────────────────────────────────────────────┐  │
-- │  │  DB_ROW_ID | TXN_ID | ROLL_PTR | col1 | col2     │  │
-- │  │  [固定长度字段，物理存储]                          │  │
-- │  └──────────────────────────────────────────────────┘  │
-- │                                                       │
-- │  添加列后的数据文件：                                    │
-- │  ┌──────────────────────────────────────────────────┐  │
-- │  │  DB_ROW_ID | TXN_ID | ROLL_PTR | col1 | col2     │  │
-- │  │  [原有数据不变]                                   │  │
-- │  ├──────────────────────────────────────────────────┤  │
-- │  │  [新增列的值存储在 行溢出区域/额外区域]            │  │
-- │  │  - 不在主记录中，节省空间                         │  │
-- │  │  - 读取时从记录末尾动态读取                        │  │
-- │  └──────────────────────────────────────────────────┘  │
-- │                                                       │
-- │  数据字典更新：                                          │
-- │  ┌──────────────────────────────────────────────────┐  │
-- │  │  Table Def:                                      │  │
-- │  │  - 原始列数: 2                                   │  │
-- │  │  - 当前列数: 3 (metadata stored separately)      │  │
-- │  │  - Instant Column Count: 2 (原始)               │  │
-- │  └──────────────────────────────────────────────────┘  │
-- └─────────────────────────────────────────────────────────┘

-- 查看 instant 列信息
SELECT * FROM information_schema.innodb_tables 
WHERE name LIKE '%t1%';

-- 查看表当前列数
SHOW CREATE TABLE t1;

-- 限制：
-- - 最多支持 64 次 INSTANT ADD COLUMN
-- - 删除列后不能再次添加同名列
-- - 不支持在 FIRST 位置添加（必须末尾）
```

### 🔥 扩展追问 3：在线 DDL 对主从复制有什么影响？

> A: **从库需要同步执行相同的 DDL，需要注意复制延迟。**

```sql
-- 主从场景下的 DDL 注意事项

-- 1. pt-osc (pt-online-schema-change)
-- 第三方工具，通过触发器同步数据
-- ┌─────────────────────────────────────────────────────────┐
-- │  主库:                                                   │  │
-- │    - 创建新表结构                                         │  │
-- │    - 创建 DELETE/INSERT/UPDATE 触发器                   │  │
-- │    - 批量迁移数据                                         │  │
-- │    - 交换表名                                             │  │
-- │  从库:                                                   │  │
-- │    - 接收同样的 DDL（正常复制）                          │  │
-- │    - 无需触发器                                           │  │
-- │  问题:                                                   │  │
-- │    - DDL 期间 binlog 格式 必须是 STATEMENT              │  │
-- │    - 触发器复制到从库，可能导致执行顺序问题               │  │
-- └─────────────────────────────────────────────────────────┘

-- 2. gh-ost (GitHub Online Schema Change)
-- 基于 binlog 解析，不使用触发器
-- ┌─────────────────────────────────────────────────────────┐
-- │  优势:                                                   │  │
-- │    - 从库通过 binlog 同步，无触发器问题                  │  │
-- │    - 可暂停、可限流                                       │  │
-- │    - 对主库影响更小                                       │  │
-- └─────────────────────────────────────────────────────────┘

-- 3. MySQL 原生 Online DDL (MySQL 5.6+)
-- 建议在从库先执行，验证后再主库执行
-- 从库配置:
SET @@session.sql_log_bin = 0;  -- 禁止记录 binlog
ALTER TABLE t1 ADD COLUMN c1 INT, ALGORITHM=INSTANT;
SET @@session.sql_log_bin = 1;

-- 然后在主库执行同样的 DDL
```

### 🔥 扩展追问 4：如何判断 DDL 应该使用哪种算法？

> A: **通过 EXPLAIN 和 information_schema 判断。**

```sql
-- ========== 1. EXPLAIN 查看预估 ==========
EXPLAIN ALTER TABLE t1 ADD COLUMN c1 INT;

-- 输出中可以看到 algorithm
-- mysql> EXPLAIN ALTER TABLE t1 ADD COLUMN c1 INT;
-- +----+-------------+-------+------------+-------------+
-- | id | select_type | table | partitions | type        |
-- +----+-------------+-------+------------+-------------+
-- |  1 | ALTER       | t1    | NULL       | ALL         |  <- COPY
-- +----+-------------+-------+------------+-------------+

-- ========== 2. 查看 ALGORITHM 支持情况 ==========
-- 查看所有支持的操作
SELECT 
    operation_name,
    supported_algorithms,
    instant_support
FROM information_schema.optimizer_trace;

-- 或者通过 performance_schema
SELECT 
    VARIABLE_NAME,
    VARIABLE_VALUE 
FROM performance_schema.variables_info
WHERE VARIABLE_NAME LIKE '%algorithm%';

-- ========== 3. 监控 DDL 进度 ==========
SHOW PROCESSLIST;
-- State 列显示: "copy to tmp table" = COPY 算法
-- State 列显示: "alter in progress" = INPLACE 算法

-- ========== 4. DDL 算法选择决策树 ==========
-- ┌─────────────────────────────────────────────────────────┐
-- │                   DDL 操作类型判断                       │
-- │                    │                                    │
-- │         ┌──────────┴──────────┐                        │
-- │         │                     │                        │
-- │         ▼                     ▼                        │
-- │    是 INSTANT               检查 INPLACE                 │
-- │    适用场景?                  │                          │
-- │         │              ┌─────┴─────┐                   │
-- │         │              │           │                   │
-- │         ▼              ▼           ▼                   │
-- │     ALGORITHM=      是 INPLACE    需要 COPY            │
-- │       INSTANT        适用场景?      │                   │
-- │                       │           │                    │
-- │                       ▼           ▼                    │
-- │                   ALGORITHM=    ALGORITHM=             │
-- │                     INPLACE       COPY                 │
-- └─────────────────────────────────────────────────────────┘

-- ========== 5. 实用 SQL：检查大表 DDL 风险 ==========
-- 查看表大小
SELECT 
    TABLE_NAME,
    ROUND(DATA_LENGTH / 1024 / 1024, 2) AS '数据大小(MB)',
    ROUND(INDEX_LENGTH / 1024 / 1024, 2) AS '索引大小(MB)',
    TABLE_ROWS
FROM information_schema.tables
WHERE TABLE_SCHEMA = 'your_database'
ORDER BY DATA_LENGTH DESC;
```

## 3. 一句话总结

> **MySQL 8.0 原子 DDL 通过 DDL LOG 实现操作原子性，支持失败回滚；ALTER TABLE 有 INSTANT（仅改元数据，最快）、INPLACE（增量修改）、COPY（全表重建）三种算法，INSTANT 支持添加列/索引/改默认值等场景；锁影响从 COPY 全程排他到 INSTANT 毫秒级不等，生产环境优先使用 INSTANT 或 INPLACE。**

## 4. 面试加分话术

> "我们生产环境 MySQL 8.0 遇到过大表加索引导致从库延迟飙升的问题。排查发现是主库用 ALGORITHM=INPLACE 添加索引，binlog 复制到从库后从库用 COPY 算法重建表。后来做了两个优化：1）所有大表 DDL 强制指定 ALGORITHM=INSTANT（业务允许的情况下），2）主从都设置 `sql_log_bin=0` 在从库先验证算法，验证通过再主库执行。另外，MySQL 8.0 的原子 DDL 还解决了一个历史问题——以前 DDL 中途失败会产生僵尸表（表名存在但结构损坏），现在有 DDL LOG 自动清理。实际测试过：给 5000 万行的订单表加索引，INSTANT 算法 3 秒完成，COPY 算法需要 45 分钟而且期间主库完全不可写。"
