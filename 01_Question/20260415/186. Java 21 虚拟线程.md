## 基础流程

### 1. 什么是虚拟线程？

虚拟线程是 **JDK 21 正式特性**（历经 JEP 425→428→436→443→444 的演进），是 JVM 层面的**轻量级用户态线程**，由 Go 的 Goroutine 和 Erlang 的 Actor 模型启发。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    平台线程 vs 虚拟线程对比                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  平台线程模型（1:1 映射）:                                          │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Thread-1  ──────────────────────────────────────────────►  │   │
│  │  (OS Thread)   内核线程（1:1）        用户代码                 │   │
│  │                上下文切换开销 ~2000ns                        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  虚拟线程模型（M:N 映射）:                                          │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │  Virtual-1 ─┐                                              │   │
│  │  Virtual-2 ─┼──►  Carrier Thread (OS Thread)  ────► CPU  │   │
│  │  Virtual-3 ─┤     上下文切换开销 ~100ns                     │   │
│  │  ...        │                                              │   │
│  │  Virtual-N ─┘                                              │   │
│  │                                                             │   │
│  │  虚拟线程 = 挂载在载体线程上的任务                            │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  关键差异:                                                          │
│  ├─ 内存占用: 平台线程 ~1MB（栈） vs 虚拟线程 ~200-400 bytes       │
│  ├─ 创建速度: 平台线程 ~1ms vs 虚拟线程 ~1 microsecond            │
│  └─ 上下文切换: 平台线程 ~2000ns vs 虚拟线程 ~100ns               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**核心概念**：

```java
public class VirtualThreadBasics {

    public static void main(String[] args) throws InterruptedException {
        
        // ============ 创建虚拟线程的 3 种方式 ============
        
        // 1. Thread.ofVirtual() — JDK 21 新 API
        Thread vThread1 = Thread.ofVirtual()
            .name("my-virtual-1")
            .start(() -> {
                System.out.println("Hello from Virtual Thread: " 
                    + Thread.currentThread());
            });

        // 2. Thread.startVirtualThread() — 便捷方法
        Thread vThread2 = Thread.startVirtualThread(() -> {
            System.out.println("Virtual thread running");
        });

        // 3. Executors.newVirtualThreadPerTaskExecutor() — 线程池方式
        try (ExecutorService executor = 
                Executors.newVirtualThreadPerTaskExecutor()) {
            Future<String> future = executor.submit(() -> {
                return "Result from virtual thread";
            });
            System.out.println(future.get());
        }

        // ============ 虚拟线程的特征 ============
        vThread1.join();
        
        Thread virtual = Thread.currentThread();
        System.out.println("Virtual thread: " + virtual.isVirtual()); // true
        System.out.println("Thread ID: " + virtual.threadId());       // 可获取
        System.out.println("Carrier: " + virtual.toString());        // 包含载体线程
    }
}
```

### 2. 虚拟线程 vs 平台线程的本质区别

```java
// 本质区别：线程栈的存储位置和管理方式

public class ThreadNature {

    public static void main(String[] args) throws Exception {
        
        // ============ 平台线程栈 ============
        // - 在 Heap 中分配，大小固定（默认 1MB，可调 -Xss）
        // - 线程销毁时，栈内存被 GC 回收
        // - 栈帧：JVM 规范固定结构
        
        // ============ 虚拟线程栈 ============
        // - 仍然是 Java 对象，但栈内容存在 Heap 的连续内存块中
        // - 大小动态增长（按需分配），初始很小（约 1KB）
        // - 不再是 OS 线程的栈，而是"延续对象"（Continutation）

        // 验证虚拟线程栈在 Heap 中
        System.out.println("===== 线程信息 =====");
        Thread platformThread = Thread.ofPlatform().start(() -> {
            System.out.println("Platform: " + Thread.currentThread().getName());
        });
        platformThread.join();

        Thread virtualThread = Thread.ofVirtual().start(() -> {
            Thread current = Thread.currentThread();
            System.out.println("Virtual: " + current.getName());
            System.out.println("  - isVirtual: " + current.isVirtual());
            System.out.println("  - Carrier: " + getCarrierThread(current));
        });
        virtualThread.join();
    }

    private static String getCarrierThread(Thread t) {
        try {
            Field field = t.getClass().getDeclaredField("carrierThread");
            field.setAccessible(true);
            Thread carrier = (Thread) field.get(t);
            return carrier.getName();
        } catch (Exception e) {
            return "unknown";
        }
    }
}
```

**本质区别对照表**：

| 维度 | 平台线程 | 虚拟线程 |
|------|---------|---------|
| **实现** | OS 原生线程（JNI/native） | JVM 管理的用户态线程 |
| **栈存储** | 堆外或堆内（1MB 固定） | 堆内连续内存块（按需增长） |
| **阻塞行为** | 阻塞 OS 线程 | 挂起虚拟线程，释放载体线程 |
| **调度** | OS 内核调度 | JVM ForkJoinPool（工作窃取） |
| **生命周期** | 与 OS 线程绑定 | 独立，可随时挂载/卸载 |
| **创建开销** | ~1ms（syscall） | ~1μs（对象分配） |
| **内存占用** | ~1MB/线程 | ~200-400 bytes/线程 |

### 3. 载体线程（Carrier Thread）

```java
public class CarrierThreadDemo {

    public static void main(String[] args) throws Exception {
        
        // 虚拟线程挂载在载体线程上运行
        try (ExecutorService executor = 
                Executors.newVirtualThreadPerTaskExecutor()) {
            
            // 提交 10 个虚拟线程任务
            IntStream.range(0, 10).forEach(i -> {
                executor.submit(() -> {
                    Thread current = Thread.currentThread();
                    System.out.println("Task " + i + " on " + current.getName() 
                        + " (Carrier: " + getCarrierName(current) + ")");
                    
                    // 模拟 I/O 操作
                    try {
                        Thread.sleep(100);
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }
                });
            });
        }

        // 观察输出：同一个 Carrier 可能执行多个 Virtual 任务
        // 关键：虚拟线程阻塞时，载体线程被释放去执行其他虚拟任务！
    }

    private static String getCarrierName(Thread t) {
        try {
            // JDK 内部字段：carrierThread
            Thread carrier = (Thread) t.getClass()
                .getDeclaredField("carrierThread")
                .apply(t, true);
            return carrier.getName();
        } catch (Exception e) {
            return "N/A";
        }
    }
}
```

### 4. Pinning（钉住）问题

```java
// 什么是 Pinning？虚拟线程被"钉"在载体线程上，无法卸载

public class PinningDemo {

    // ============ 场景1：synchronized 块/方法 ============
    // 这是 JDK 21 虚拟线程最常见的 Pinning 原因！
    
    static class SynchronizedCounter {
        private int count = 0;
        
        // ❌ 钉住！进入 synchronized 会让虚拟线程 Pin 到载体
        public synchronized void increment() {
            // 在这个方法执行期间，虚拟线程无法卸载
            // 其他虚拟线程也无法使用这个载体
            count++;
            try {
                Thread.sleep(10); // 模拟 I/O（此时应该释放载体！）
            } catch (InterruptedException e) { }
        }
        
        // ✅ 不钉住！使用 ReentrantLock
        private final ReentrantLock lock = new ReentrantLock();
        
        public void incrementWithLock() throws InterruptedException {
            lock.lock();
            try {
                count++;
                Thread.sleep(10); // 可以安全地释放载体
            } finally {
                lock.unlock();
            }
        }
    }

    // ============ 场景2：本地方法 / native 代码 ============
    // 如果调用的方法包含 native 代码，虚拟线程无法挂起

    // ============ 场景3：JNI 调用 ============
    // JNI 方法会钉住虚拟线程
}
```

```

│  │  synchronized (obj) { ... }                             │ │

│  │  任何 synchronized 代码块                                 │ │
│  │                                                          │ │

│  │  VarHandle / AtomicInteger 等原子类                      │ │

│                                                                │
│  原因:                                                         │

│  - ReentrantLock 等 JUC 锁，内部已正确处理虚拟线程挂起           │
└────────────────────────────────────────────────────────────────┘
```

### 5. 如何避免 Pinning？

```java
public class AvoidPinning {

    // ============ 方案1：使用 ReentrantLock ============
    private final ReentrantLock lock = new ReentrantLock();
    
    public void doWork() throws InterruptedException {
        lock.lock();
        try {
            // 业务逻辑
            Thread.sleep(100); // ✅ 虚拟线程可以安全挂起
        } finally {
            lock.unlock();
        }
    }

    // ============ 方案2：使用 @Contended 或扩展 JUC 锁 ============
    // JDK 的 j.u.c 锁都已支持虚拟线程（内部使用 park/unpark）
    
    // ============ 方案3：启用 JEP 423 的 Pinning 监控 ============
    // JDK 21: -XX:+UseVirtualThreads
    // JDK 21: -XX:+TraceVirtualThreadPinning (JFR 事件)
    
    // ============ 方案4：使用 Volatile 替代部分 synchronized ============
    private volatile boolean running = true;
    
    // 对于简单的标志位，volatile 足够且不触发 Pinning

    // ============ 方案5：拆分 synchronized 块 ============
    // 将 I/O 操作移出 synchronized 块
    
    public void goodDesign() throws InterruptedException {
        Object lockObj = new Object();
        
        // 仅在必要时持有锁
        int value;
        synchronized (lockObj) {
            value = this.value; // 快速操作
        }
        
        // I/O 在锁外执行
        Thread.sleep(100); // ✅ 不影响其他虚拟线程
        
        synchronized (lockObj) {
            this.value = value + 1;
        }
    }
}
```

## 🔥 扩展追问 1：虚拟线程的内存模型和 GC 影响？

```java
public class VirtualThreadMemory {

    public static void main(String[] args) throws Exception {
        
        // ============ 虚拟线程栈内存分配 ============
        
        // 1. 初始栈大小：约 512 字节（可配置）
        // 2. 最大栈大小：可配置，但增长方式不同
        
        // 虚拟线程栈是按需增长的：
        // - 初始：1 个页面（约 4KB）的一部分
        // - 增长：按 4KB 页面递增
        // - 最大：默认 1MB（与平台线程相同，可通过 -Xss2m 调整）
        
        // ============ 内存占用对比 ============
        System.out.println("===== 内存对比 =====");
        
        // 平台线程：1000 个线程 × 1MB = ~1GB
        // 虚拟线程：1000 个线程 × ~2KB = ~2MB
        
        // ============ 虚拟线程栈在 GC 中的表现 ============
        // 虚拟线程栈是普通的 Java 对象（Continuation），在堆中分配
        // GC 会像管理其他对象一样管理它
        // 不需要特殊处理
        
        // ============ 溢出检测 ============
        Thread virtual = Thread.ofVirtual().start(() -> {
            try {
                recursive(1);
            } catch (StackOverflowError e) {
                System.out.println("Stack overflow at depth: " + 
                    e.getStackTrace().length);
            }
        });
        virtual.join();
    }

    static void recursive(int depth) {
        recursive(depth + 1);
    }
}
```

> **A:** 虚拟线程栈内存特点：
> 1. **按需分配**：初始仅几百字节，最大 1MB
> 2. **Heap 中分配**：作为普通对象被 GC 管理
> 3. **零 GC 压力**：不像平台线程栈在堆外，GC 无需特殊处理
> 4. **1000 线程场景**：平台线程 ~1GB vs 虚拟线程 ~2MB

## 🔥 扩展追问 2：虚拟线程与 ThreadLocal 的关系？

```java
public class VirtualThreadAndThreadLocal {

    // ============ ThreadLocal 在虚拟线程中的行为 ============
    
    static ThreadLocal<String> context = ThreadLocal.withInitial(() -> 
        Thread.currentThread().getName()
    );

    public static void main(String[] args) throws Exception {
        
        // 平台线程：每个线程独立的值
        Thread t1 = Thread.ofPlatform().name("platform-1").start(() -> {
            System.out.println("Platform-1: " + context.get());
            context.set("modified by platform-1");
        });
        t1.join();
        
        Thread t2 = Thread.ofPlatform().name("platform-2").start(() -> {
            System.out.println("Platform-2: " + context.get()); // 独立值
        });
        t2.join();
        
        // 虚拟线程：行为相同，但值不共享
        Thread v1 = Thread.ofVirtual().name("virtual-1").start(() -> {
            System.out.println("Virtual-1: " + context.get());
            context.set("modified by virtual-1");
        });
        v1.join();
        
        Thread v2 = Thread.ofVirtual().name("virtual-2").start(() -> {
            System.out.println("Virtual-2: " + context.get()); // 独立值
        });
        v2.join();
        
        // ============ ScopedLocal（新增）===========
        // JDK 21 引入了 ScopedLocal（预览），更适合虚拟线程
        // 允许在作用域边界共享值，但不会泄漏
        
        // ScopedLocal.withValue(ctx, () -> { ... });
        // 退出作用域后，ctx 自动清理
    }
}
```

> **A:** 
> - `ThreadLocal` 在虚拟线程中**正常工作**，每个虚拟线程有独立的值
> - **潜在问题**：如果用 `Thread.ofVirtual()` 创建大量短生命周期虚拟线程，`ThreadLocal` 不会被及时清理（GC 仍会回收）
> - **JDK 21 新增 `ScopedLocal`**：更安全的值传递方式，支持作用域边界清理
> - **经验**：避免在虚拟线程中存储大对象到 ThreadLocal，用完即清理

## 🔥 扩展追问 3：虚拟线程的调度算法？

```java
public class VirtualThreadScheduler {

    public static void main(String[] args) {
        
        // 虚拟线程使用 ForkJoinPool 作为调度器
        // 调度算法：工作窃取（Work Stealing）
        
        // 工作窃取原理：
        // 1. 每个 Carrier 有自己的任务队列
        // 2. Carrier 空闲时，从其他 Carrier 队列尾部"偷"任务
        // 3. 保证所有 Carrier 负载均衡
        
        // ============ 调度器配置 ============
        
        // 默认调度器（ForkJoinPool.common）
        try (ExecutorService executor = 
                Executors.newVirtualThreadPerTaskExecutor()) {
            // 使用默认调度器
        }
        
        // 自定义调度器
        ExecutorService customScheduler = new ForkJoinPool(
            4,                          // 并行度
            ForkJoinPool.defaultForkJoinWorkerThreadFactory,
            null,
            true                        // asyncMode
        );
        
        Thread.ofVirtual()
            .scheduler(customScheduler)  // 指定调度器
            .start(() -> System.out.println("Custom scheduler"));
    }
}
```

> **A:** 虚拟线程调度特点：
> 1. **默认调度器**：`ForkJoinPool.common`（CPU 核心数个 Carrier）
> 2. **调度算法**：工作窃取，保证负载均衡
> 3. **可自定义**：`Thread.ofVirtual().scheduler(executor)` 指定调度器
> 4. **阻塞友好**：虚拟线程阻塞时，Carrier 立即调度其他任务

## 🔥 扩展追问 5：虚拟线程的最佳实践？

```java
public class BestPractices {

    public static void main(String[] args) {
        
        // ============ ✅ 推荐做法 ============
        
        // 1. 使用 newVirtualThreadPerTaskExecutor 而非固定池
        try (ExecutorService executor = 
                Executors.newVirtualThreadPerTaskExecutor()) {
            // 每个任务一个虚拟线程，不复用
        }
        
        // 2. 避免在 synchronized 中执行长时间 I/O
        // 用 ReentrantLock 替代
        
        // 3. 避免在虚拟线程中使用 ThreadLocal 存储大对象
        // 用 ScopedLocal 或请求作用域对象
        
        // 4. 谨慎使用 Object.wait() / Thread.sleep()
        // 虚拟线程中两者都正确工作
        
        // ============ ❌ 不推荐做法 ============
        
        // 1. 不要用虚拟线程做 CPU 密集计算
        // → 虚拟线程的调度开销不适合 CPU-bound
        
        // 2. 不要混用平台线程和虚拟线程做互斥
        // → 可能导致死锁
        
        // 3. 不要在虚拟线程中使用 Thread.interrupt()
        // → 行为复杂，建议用 CancellationToken
        
        // 4. 不要依赖 Thread 的 identity（ID/Priority 等）
        // → 虚拟线程的这些属性可能不准确
    }
}
```

## 一句话总结

> **虚拟线程 = JVM 管理的用户态轻量线程（M:N 映射），阻塞时自动释放载体线程，1M 内存可跑 5000+ 线程；但 synchronized 会触发 Pinning 导致无法卸载，应用需迁移到 ReentrantLock 或 JUC 锁。**
