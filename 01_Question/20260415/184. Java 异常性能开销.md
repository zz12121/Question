### 2. 异常对象的内存开销

```java
public class ExceptionMemoryCost {

    public static void main(String[] args) throws Throwable {
        // 测量单个异常的内存开销
        long before = Runtime.getRuntime().totalMemory() 
                    - Runtime.getRuntime().freeMemory();
        
        // 创建 10000 个异常
        Exception[] exceptions = new Exception[10000];
        for (int i = 0; i < 10000; i++) {
            try {
                throw new RuntimeException("test");
            } catch (Exception e) {
                exceptions[i] = e;
            }
        }
        
        // 强制 GC 后测量（不推荐生产环境）
        System.gc();
        Thread.sleep(100);
        
        long after = Runtime.getRuntime().totalMemory() 
                   - Runtime.getRuntime().freeMemory();
        long used = after - before;
        
        System.out.println("10000 异常对象内存开销: " + used / 1024 / 1024 + " MB");
        System.out.println("平均每个异常: " + used / 10000 + " bytes");
        
        // 打印一个异常的 stackTrace
        System.out.println("\n=== StackTrace 示例 ===");
        for (StackTraceElement ste : exceptions[0].getStackTrace()) {
            System.out.println(ste.getClassName() + "." + 
                             ste.getMethodName() + "(" + 
                             ste.getFileName() + ":" + 
                             ste.getLineNumber() + ")");
        }
    }
}
```

**典型开销估算**：

| 组件 | 大小 | 说明 |
|------|------|------|
| Throwable 对象头 | 12-16 bytes | mark word + klass pointer + flags |
| String message | 40-56 bytes | 空消息时仍约 40 bytes |
| cause (引用) | 4/8 bytes | 无 cause 时为 null |
| **stackTrace[]** | **可变** | **核心开销** |
| 1 帧 StackTraceElement | 24-40 bytes | 3 字符串引用 + int |
| 字符串对象 (每个) | 24-48 bytes | 字符串内容本身 |
| **总计（5 帧深度）** | **≈ 300-500 bytes** | |

### 3. 为什么说"频繁创建异常对象"是性能问题？

```java
// 典型反模式：使用异常做流程控制
public class ExceptionAntiPattern {

    // ❌ 反模式：频繁抛出捕获
    public Integer parseWithException(String num) {
        try {
            return Integer.parseInt(num); // NumberFormatException
        } catch (NumberFormatException e) {
            return null; // 异常用于控制流
        }
    }

    // ✅ 正解：先检查后使用
    public Integer parseWithCheck(String num) {
        if (num == null || num.isEmpty()) {
            return null;
        }
        try {
            return Integer.parseInt(num);
        } catch (NumberFormatException e) {
            return null; // 真正的解析错误
        }
    }

    // ❌ 反模式：循环中异常处理
    public void processLoop() {
        for (String item : items) {
            try {
                doSomething(item);
            } catch (ItemException e) {
                log.warn("处理失败", e); // 每项都可能创建异常
            }
        }
    }
}

// 性能对比测试
public class ExceptionPerfTest {
    
    public static void main(String[] args) {
        int iterations = 1_000_000;
        
        // 方案1：异常控制流
        long start = System.nanoTime();
        int count1 = 0;
        for (int i = 0; i < iterations; i++) {
            try {
                throw new RuntimeException();
            } catch (Exception e) {
                count1++;
            }
        }
        long exceptionTime = System.nanoTime() - start;
        
        // 方案2：正常判断
        long start2 = System.nanoTime();
        int count2 = 0;
        boolean flag = false;
        for (int i = 0; i < iterations; i++) {
            if (flag) {
                count2++;
            }
        }
        long normalTime = System.nanoTime() - start2;
        
        System.out.println("异常方式: " + exceptionTime/1_000_000 + " ms");
        System.out.println("判断方式: " + normalTime/1_000_000 + " ms");
        System.out.println("差距: " + exceptionTime/normalTime + "x");
    }
}
```

### 4. `OmitStackTraceInFastThrow` 参数详解

```java
// JVM 参数: -XX:-OmitStackTraceInFastThrow
//           -XX:+OmitStackTraceInFastThrow (默认开启)

// 这个参数的含义：

/*
 * JVM 优化的核心逻辑：
 * 
 * 对于某些"热门的已知异常"，JVM 会预判这是高频抛出的异常
 * （如 NullPointerException、ArrayIndexOutOfBoundsException）
 * 
 * 优化策略：不填充完整的 StackTrace，改为预分配的轻量异常
 * 
 * 效果：
 * - 开启(OmitStackTraceInFastThrow=true):  异常抛出 ≈ 快速返回
 * - 关闭(OmitStackTraceInFastThrow=false): 每次都填充完整栈
 */
```

**JVM 预分配异常的源码逻辑**（hotspot/src/share/vm/runtime/sharedRuntime.cpp）：

```cpp
// 简化伪代码
JVM_ENTRY(_jthrowable, JVM_FillInStackTrace(_jthrowable receiver, TRAPS))
    // OmitStackTraceInFastThrow 优化的核心位置
    
    if (EnableOmitStackTraceInFastThrow) {
        // 检查异常类型是否在白名单中
        if (is_frequent_exception_type(receiver)) {
            // 已知异常类型：NullPointerException、ArithmeticException等
            // 跳过填充 StackTrace！
            return preallocated_out_of_memory_error; // 或预分配实例
        }
    }
    
    // 非优化路径：正常填充完整栈
    fillInStackTrace(receiver);
JVM_END
```

**白名单中的异常类型**（HotSpot 行为）：

```

├────────────────────────────────────────────────────────────────┤
│                                                                │
│  ┌────────────────────┬───────────────────────────────────┐   │
│  │ 异常类型            │ 常见场景                            │   │
│  ├────────────────────┼───────────────────────────────────┤   │
│  │ NullPointerException│ obj.field / obj.method()         │   │
```

## 🔥 扩展追问 1：什么情况下 StackTrace 会真的被填充？

```java
// StackTrace 填充的触发条件

public class StackTraceFillConditions {

    public static void main(String[] args) {
        RuntimeException e = new RuntimeException("test");
        
        // ❌ 不触发填充（OmitStackTraceInFastThrow=true 时）
        throw e; // JVM 可能直接用预分配异常
        
        // ✅ 触发填充的操作：
        
        // 1. e.getStackTrace()
        StackTraceElement[] trace = e.getStackTrace();
        
        // 2. e.fillInStackTrace() 显式调用
        e.fillInStackTrace();
        
        // 3. e.printStackTrace() / e.printStackTrace(PrintStream)
        e.printStackTrace();
        
        // 4. Throwable.initCause() — 某些路径
        e.initCause(new RuntimeException("cause"));
        
        // 5. 序列化时（如果配置了）
        // ObjectOutputStream 会写入 StackTrace
    }

    // 实际场景中的陷阱
    static class Logger {
        static void log(Throwable t) {
            // ❌ 反模式：log.info("error", t) 
            // 触发 StackTrace 填充！
            
            // ✅ 正解：只 log message
            // log.info("error: " + t.getMessage());
        }
    }
}
```

> **A:** StackTrace 填充发生在以下情况：
> 1. **显式获取**：`getStackTrace()` / `printStackTrace()`
> 2. **JIT 优化关闭时**：`OmitStackTraceInFastThrow=false`
> 3. **非白名单异常**：`extends Exception`/`extends Throwable` 的自定义异常
> 4. **序列化**：通过 `ObjectOutputStream` 序列化 Throwable 时
> 5. **远程传输**：微服务间传递异常时（如 gRPC 错误）

## 🔥 扩展追问 2：`getStackTrace()` vs `setStackTrace()` 的区别？

```java
public class StackTraceManipulation {

    public static void main(String[] args) {
        // 场景：重新抛出异常时，手动设置 StackTrace
        try {
            try {
                methodA();
            } catch (Exception e) {
                // 包装异常，保留原始栈
                throw new BusinessException("业务错误", e);
            }
        } catch (Exception e) {
            // 如果没有 initCause + setStackTrace，
            // 新异常的 StackTrace 会被覆盖为当前位置
        }
    }

    // 正确的异常链包装
    static class BusinessException extends Exception {
        private static final StackTraceElement[] EMPTY_TRACE = 
            new StackTraceElement[0];
        
        public BusinessException(String msg, Throwable cause) {
            super(msg, cause);
            // 保留原始异常的 StackTrace
            setStackTrace(cause.getStackTrace());
            // 将栈顶指向原始抛出位置，而非包装位置
            fillInStackTrace();
        }
    }
}
```

> **A:** 
> - `getStackTrace()`：读取栈信息，**触发 fillInStackTrace**（如未填充）
> - `setStackTrace(StackTraceElement[])`：**批量替换**栈帧，常用于异常包装
> - `fillInStackTrace()`：**显式填充**当前线程的栈帧，覆盖已有栈
> - **常见场景**：异常跨线程传递后，栈信息失效，需要 `setStackTrace` 手动修复

## 🔥 扩展追问 3：异步线程中的异常 StackTrace 特点？

```java
public class AsyncStackTrace {

    public static void main(String[] args) throws InterruptedException {
        // 场景：主线程等待子线程异常
        Thread t = new Thread(() -> {
            throw new RuntimeException("子线程异常");
        });
        t.start();
        t.join(); // 主线程等待
        
        // 此时主线程看到的 StackTrace 是：
        // java.lang.RuntimeException: 子线程异常
        //     at AsyncStackTrace.lambda$main$0(AsyncStackTrace.java:4)
        //     ... 正确的子线程栈
        
        // 但如果用 Future.get() 捕获：
        ExecutorService executor = Executors.newSingleThreadExecutor();
        Future<?> future = executor.submit(() -> {
            throw new RuntimeException("子线程异常 via Future");
        });
        
        try {
            future.get(); // 阻塞获取结果
        } catch (ExecutionException e) {
            // e.getCause() 包含原始异常
            // e.getCause().getStackTrace() 是子线程栈！
            e.getCause().printStackTrace();
        }
    }
}

// 异步框架的 StackTrace 处理
class AsyncFramework {
    static class WrappedException extends RuntimeException {
        private final String originalThreadName;
        private final StackTraceElement[] originalTrace;
        
        public WrappedException(Throwable t) {
            super(t.getMessage());
            this.originalThreadName = Thread.currentThread().getName();
            this.originalTrace = t.getStackTrace();
            // 在新线程中，栈顶是包装层，需要调整
        }
        
        @Override
        public StackTraceElement[] getStackTrace() {
            return originalTrace;
        }
    }
}
```

> **A:** 
> - 异步线程的异常 StackTrace **默认是子线程的栈**，主线程不可见
> - `Future.get()` + `ExecutionException`：异常被包装，`getCause()` 保存原始异常
> - `Thread.join()`：子线程异常传播到主线程，StackTrace 保持正确
> - **CompletableFuture 陷阱**：supplyAsync 抛出的异常在被 `join()` 前，StackTrace 可能未填充

## 🔥 扩展追问 4：Spring 框架中的异常处理优化？

```java
// Spring MVC 全局异常处理的性能考量

@RestControllerAdvice
public class GlobalExceptionHandler {

    // ❌ 反模式：每次都序列化完整 StackTrace
    @ExceptionHandler(Exception.class)
    public ErrorResponse handleException(Exception e) {
        return new ErrorResponse(
            e.getClass().getName(),
            e.getMessage(),
            e.getStackTrace() // ⚠️ 强制填充 + 序列化开销
        );
    }

    // ✅ 正解1：只取必要信息
    @ExceptionHandler(Exception.class)
    public ErrorResponse handleException2(Exception e) {
        return new ErrorResponse(
            e.getClass().getSimpleName(), // 不需要全限定名
            e.getMessage(),
            null // 不返回 StackTrace
        );
    }

    // ✅ 正解2：限制 StackTrace 深度
    @ExceptionHandler(Exception.class)
    public ErrorResponse handleException3(Exception e) {
        StackTraceElement[] fullTrace = e.getStackTrace();
        StackTraceElement[] limitedTrace = 
            Arrays.copyOfRange(fullTrace, 0, Math.min(10, fullTrace.length));
        
        return new ErrorResponse(..., limitedTrace);
    }

    // ✅ 正解3：异步日志 + 快速返回
    @Async
    @ExceptionHandler(Exception.class)
    public void handleExceptionAsync(Exception e) {
        // 异步记录完整 StackTrace
        logger.error("Unexpected error", e);
        // 快速返回 HTTP 500，不等待日志写入
    }
}

// Spring Boot 配置：关闭详细错误页面的 StackTrace
// server.error.include-stacktrace=never
// server.error.include-exception=false
```

> **A:** Spring 框架中的异常优化策略：
> 1. **不返回 StackTrace 给前端**：API 只需 error code + message
> 2. **日志异步化**：`@Async` 异常日志，不阻塞 HTTP 响应
> 3. **按需填充**：`getMessage()` 不触发填充，只有 `getStackTrace()` 才触发
> 4. **配置关闭**：`server.error.include-stacktrace=never` 防止微服务间传递完整栈

## 一句话总结

> **`OmitStackTraceInFastThrow=true` 时，JVM 对已知异常类型（NPE/AIOOBE 等）跳过栈填充，将抛出耗时从 150-600ns 降至 2-4ns；但自定义异常不在白名单，频繁创建仍会触发完整 StackTrace 填充。**

## 面试加分话术

> "我在优化公司监控系统时发现，告警模块每秒创建 10 万+ 个自定义 `AlertException`，CPU profiler 显示异常创建占总 CPU 的 23%。分析后发现：JVM 的 `OmitStackTraceInFastThrow` 只对 `NullPointerException` 等白名单类型生效，自定义异常每次都 `fillInStackTrace`。解决方案是改用**返回码枚举**替代异常做业务流控，CPU 降到 1.2%。另一个经验是：用 `Future.get()` 捕获异步异常时，`ExecutionException.getCause().getStackTrace()` 会包含完整子线程栈，这在高并发异步框架中很容易打爆日志量，务必记得设置 StackTrace 长度上限。"
