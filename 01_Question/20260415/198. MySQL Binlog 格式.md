## 1. 基础流程

### binlog 三种格式对比

```
┌─────────────────────────────────────────────────────────────┐
│              MySQL binlog 三种格式                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【STATEMENT 格式】                                         │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  # at 1234                                          │   │
│  │  #210816 10:00:00 server id 1                        │   │
│  │  Query   thread_id=10    exec_time=0    error_code=0 │   │
│  │  SET TIMESTAMP=1629074400                           │   │
│  │  INSERT INTO orders (user_id, amount) VALUES (1, 100) │   │
│  │                                                      │   │
│  │  ✓ 只记录 SQL 语句                                   │   │
│  │  ✓ binlog 文件小                                     │   │
│  │  ✗ 某些语句结果不确定（NOW(), UUID()等）              │   │
│  │  ✗ DELETE/UPDATE 可能丢数据                          │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  【ROW 格式】                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  # at 1234                                          │   │
│  │  #210816 10:00:00 server id 1                        │   │
│  │  Table_map: table_id=123 (db.orders)                │   │
│  │  Write_rows: table_id=123                           │   │
│  │  ### INSERT INTO `db`.`orders`                      │   │
│  │  ### SET                                           │   │
│  │  ###   @1=1  (id)                                   │   │
│  │  ###   @2=100  (amount)                             │   │
│  │  ###   @3='2021-08-16 10:00:00'                    │   │
│  │                                                      │   │
│  │  ✓ 记录每行变更的完整数据                             │   │
│  │  ✓ 结果确定，不会丢数据                              │   │
│  │  ✗ binlog 文件大（可能 5-10x STATEMENT）             │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  【MIXED 格式】                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  默认使用 STATEMENT                                  │   │
│  │  遇到以下情况自动切换到 ROW：                         │   │
│  │  - 使用 NOW(), UUID(), RAND() 等不确定函数          │   │
│  │  - 使用触发器或存储过程                              │   │
│  │  - 多表关联更新                                      │   │
│  │  - 使用 AUTO_INCREMENT 列                           │   │
│  │                                                      │   │
│  │  ✓ 平衡：文件大小 + 数据安全                          │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### ROW 格式为什么最安全但最占空间？

```
┌─────────────────────────────────────────────────────────────┐
│         ROW 格式"最安全"的原理                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【STATEMENT 格式的坑】                                     │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  问题：NOW() 在主从执行时值不同                       │   │
│  │                                                      │   │
│  │  主库:                                              │   │
│  │    INSERT INTO log (event, time) VALUES ('login', NOW()) │   │
│  │    -- NOW() = 10:00:00                              │   │
│  │                                                      │   │
│  │  从库（延迟 1 秒）:                                  │   │
│  │    -- NOW() = 10:00:01  ← 数据不一致！              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  问题：LIMIT 导致数据丢失                            │   │
│  │                                                      │   │
│  │  DELETE FROM orders WHERE status='canceled' LIMIT 1000  │   │
│  │                                                      │   │
│  │  主库删除: 订单 1, 3, 5, 7, 9...                     │   │
│  │  从库执行: 订单可能是 2, 4, 6, 8, 10...（同一时间   │   │
│  │           另一连接插入）                             │   │
│  │  结果: 删除了不同的数据！                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  【ROW 格式的解决方案】                                     │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  ROW 记录的是每行的"before image"和"after image"    │   │
│  │                                                      │   │
│  │  DELETE: 记录被删除的整行数据                        │   │
│  │  UPDATE: 记录修改前和修改后的完整行                  │   │
│  │  INSERT: 记录插入的完整行                            │   │
│  │                                                      │   │
│  │  从库执行: 直接应用这行数据，不依赖任何函数           │   │
│  │  结果: 100% 一致                                    │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  【空间占用对比】                                           │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  UPDATE orders SET status='completed' WHERE id=1    │   │
│  │                                                      │   │
│  │  STATEMENT:  ~100 bytes                             │   │
│  │    - SQL 语句本身                                    │   │
│  │                                                      │   │
│  │  ROW (FULL):  ~500 bytes                            │   │
│  │    - before: id=1, user_id=100, status='pending'... │   │
│  │    - after:  id=1, user_id=100, status='completed'..│   │
│  │    - 包含所有列的完整数据                            │   │
│  │                                                      │   │
│  │  ROW (MINIMAL):  ~150 bytes                         │   │
│  │    - 只记录主键和变更的列                           │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### binlog_row_image 参数

```sql
-- ========== binlog_row_image 参数 ==========
-- FULL (默认): 记录所有列
-- MINIMAL: 只记录主键和变更的列
-- NOBLOB: 记录所有列，但跳过未修改的 BLOB/TEXT

-- FULL 模式
UPDATE orders SET status='completed' WHERE id=1;
-- binlog 记录: 
--   before: 全部 20 列
--   after: 全部 20 列

-- MINIMAL 模式
UPDATE orders SET status='completed' WHERE id=1;
-- binlog 记录:
--   before: id=1 + status 列
--   after:  id=1 + status 列
-- 节省: ~60-80% 空间

-- NOBLOB 模式
UPDATE orders SET remark='new' WHERE id=1;
-- binlog 记录:
--   before: 所有列（但 BLOB 列如果没改就跳过）
--   after:  所有列（但 BLOB 列如果没改就跳过）
```


### 丢数据和重复消费的场景

```
┌─────────────────────────────────────────────────────────────┐
│         数据同步丢数据/重复消费场景分析                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【可能丢数据的场景】                                        │
│                                                             │
│  1️⃣ Canal 消费位点未提交                                     │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Consumer                                              │   │
│  │    1. 读取 binlog event #100                         │   │
│  │    2. 处理数据（写入 ES）                            │   │
│  │    3. ⏰ 此时宕机，位点未提交                       │   │
│  │    4. 重启后，从上次提交的位点继续                   │   │
│  │    5. event #100 被跳过 → 丢数据！                  │   │
│  │                                                      │   │
│  │  解决方案:                                           │   │
│  │  - 先写入 MQ（Kafka），再异步消费                    │   │
│  │  - 业务表和消息表用同一事务                           │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  2️⃣ DDL 语句未处理                                         │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  -- MySQL 执行                                        │   │
│  │  ALTER TABLE orders ADD COLUMN new_col VARCHAR(50); │   │
│  │                                                      │   │
│  │  Canal 默认过滤 DDL（保护消费者）                     │   │
│  │  但如果消费者需要结构同步：                           │   │
│  │  - 旧表结构 + 新数据 → 字段丢失                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  3️⃣ ROW 格式 + MINIMAL 模式 + 无主键表                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  UPDATE orders SET status='ok' WHERE user_id=100     │   │
│  │                                                      │   │
│  │  MINIMAL 记录: user_id + status                     │   │
│  │  但如果 user_id 不是唯一索引：                        │   │
│  │  - 可能匹配多行                                      │   │
│  │  - 从库执行后数据不一致                              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  【可能重复消费的场景】                                      │
│                                                             │
│  4️⃣ At-Least-Once 语义（最常见）                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Consumer                                              │   │
│  │    1. 读取 event #100                                │   │
│  │    2. 处理成功                                       │   │
│  │    3. ⏰ 此时宕机，ack 未发送                        │   │
│  │    4. 重启后重新消费 event #100 → 重复！            │   │
│  │                                                      │   │
│  │  解决方案:                                           │   │
│  │  - 消费端做幂等处理（唯一键 upsert）                 │   │
│  │  - 使用 event 的 GTID 或事务 ID 去重                 │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  5️⃣ Canal/Debezium 重启导致位点回退                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Consumer                                              │   │
│  │    1. 消费到 binlog position 1000                   │   │
│  │    2. Canal 重启，配置丢失                           │   │
│  │    3. 从 position 0 重新消费                        │   │
│  │    4. 大量重复数据                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 幂等处理示例

```java
// Canal 消费者幂等处理
public class OrderSyncConsumer {
    
    // 方案1：基于主键 upsert
    public void processInsert(Order order) {
        String sql = "INSERT INTO es_orders (id, user_id, amount, ...) " +
                     "VALUES (?, ?, ?, ...) " +
                     "ON DUPLICATE KEY UPDATE " +
                     "user_id = VALUES(user_id), " +
                     "amount = VALUES(amount), ...";
        jdbcTemplate.update(sql, order.getId(), order.getUserId(), ...);
    }
    
    // 方案2：基于唯一键幂等
    public void processEvent(BinlogEvent event) {
        // 使用 event 的唯一标识
        String uniqueKey = event.getGtid() + "-" + event.getPosition();
        
        if (processedCache.contains(uniqueKey)) {
            return;  // 已处理，跳过
        }
        
        // 执行业务逻辑
        doProcess(event);
        
        // 标记已处理
        processedCache.add(uniqueKey);
    }
    
    // 方案3：基于业务时间戳 + 版本号
    public void processUpdate(OrderUpdate update) {
        // 使用 UPDATE 的 WHERE 条件确保幂等
        String sql = "UPDATE es_orders SET status = ?, update_time = ? " +
                     "WHERE id = ? AND update_time < ?";
        int affected = jdbcTemplate.update(sql, 
            update.getStatus(), 
            update.getUpdateTime(),
            update.getId(),
            update.getUpdateTime());
        
        if (affected == 0) {
            // 可能是重复消息，忽略
            log.info("Duplicate update ignored: {}", update);
        }
    }
}
```

## 3. GTID（全局事务标识）

### GTID 工作原理

```
┌─────────────────────────────────────────────────────────────┐
│                    GTID 工作原理                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【GTID 格式】                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  GTID = source_id : transaction_id                  │   │
│  │                                                      │   │
│  │  例如: 3E11FA47-71CA-11E1-9E33-C80AA9429562:12345    │   │
│  │         ↑                                        ↑    │   │
│  │         MySQL Server UUID                       事务号  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  【GTID 自动赋值】                                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                       │   │
│  │  事务提交时:                                          │   │
│  │    1. 生成 GTID: server_uuid:next_transaction_id    │   │
│  │    2. 写入 binlog                                   │   │
│  │    3. 更新 gtid_executed 全局变量                   │   │
│  │                                                       │   │
│  │  gtid_executed 示例:                                  │   │
│  │    3E11FA47-71CA-11E1-9E33-C80AA9429562:1-12345     │   │
│  │    A9B3F8C1-71CB-11E1-9E44-D00AA9421234:1-100       │   │
│  │                                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  【GTID 复制流程】                                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                       │   │
│  │  Master:                                              │   │
│  │    BEGIN;                                            │   │
│  │    INSERT INTO t1 VALUES (1);  ← GTID: uuid:100     │   │
│  │    COMMIT;                                            │   │
│  │                                                       │   │
│  │    ┌─────────────────────────────────────────────┐  │   │
│  │    │  binlog: GTID_EVENT                          │  │   │
│  │    │    { uuid: 100 }                             │  │   │
│  │    │  TABLE_MAP_EVENT                             │  │   │
│  │    │  WRITE_ROWS_EVENT                            │  │   │
│  │    └─────────────────────────────────────────────┘  │   │
│  │                                                       │   │
│  │  Slave:                                              │   │
│  │    1. 接收 GTID: uuid:100                           │   │
│  │    2. 检查 gtid_executed 是否已包含                 │   │
│  │       - 已包含: 跳过，不执行                         │   │
│  │       - 未包含: 执行，并加入 gtid_executed          │   │
│  │                                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### GTID vs 传统 position 复制

```
┌─────────────────────────────────────────────────────────────┐
│              GTID vs 传统 Position 复制对比                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                  传统 Position 复制                    │  │
│  ├──────────────────────────────────────────────────────┤  │
│  │  CHANGE MASTER TO                                     │  │
│  │    MASTER_HOST='master',                             │  │
│  │    MASTER_LOG_FILE='mysql-bin.001234',               │  │
│  │    MASTER_LOG_POS=5678;                              │  │
│  │                                                       │  │
│  │  问题:                                                │  │
│  │  - 位置需要手动计算和记录                              │  │
│  │  - 主从切换时容易出错                                  │  │
│  │  - 无法判断某个事务是否已执行                          │  │
│  │  - 误切换到错误从库导致数据不一致                      │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                    GTID 复制                          │  │
│  ├──────────────────────────────────────────────────────┤  │
│  │  CHANGE MASTER TO                                     │  │
│  │    MASTER_HOST='master',                             │  │
│  │    MASTER_AUTO_POSITION=1;                           │  │
│  │                                                       │  │
│  │  ✓ 自动:                                             │  │
│  │    - 只需指定 MASTER_AUTO_POSITION=1                 │  │
│  │    - MySQL 自动计算需要同步的事务                     │  │
│  │  ✓ 安全:                                             │  │
│  │    - 自动跳过已执行的事务                             │  │
│  │    - 不可能出现重复执行                               │  │
│  │    - 主从切换时自动对齐                               │  │
```

## 4. 扩展追问

### 🔥 扩展追问 1：GTID 的限制和坑有哪些？

> A: **GTID 有一些使用限制，不注意会导致复制中断。**

```sql
-- ========== GTID 限制 ==========

-- 1. 不允许跨库事务引用同一个表
CREATE TABLE t1 (id INT PRIMARY KEY);
CREATE DATABASE db2;
-- ✗ 以下语句在 GTID 模式下可能出错
-- UPDATE db1.t1 a, db2.t1 b SET a.id = b.id;

-- 2. 不允许事务中使用临时表
BEGIN;
CREATE TEMPORARY TABLE temp_order (...);  -- ✗ 禁止
INSERT INTO temp_order VALUES (1);
COMMIT;

-- 3. 不允许在事务中使用自动提交
BEGIN;
INSERT INTO t1 VALUES (1);
COMMIT;

-- 自动提交会创建独立事务，可能与 GROUP REPLICATION 冲突

-- 4. gtid_next 必须明确指定
SET GTID_NEXT = 'uuid:100';  -- 必须
BEGIN;
INSERT INTO t1 VALUES (1);
COMMIT;
SET GTID_NEXT = 'AUTOMATIC';

-- ========== GTID 常见问题 ==========

-- 问题1: 跳过错误事务
-- 场景: 从库执行出错，希望跳过
SET GTID_NEXT = 'uuid:101';  -- 要跳过的事务
BEGIN;
COMMIT;  -- 空提交，标记为已执行
SET GTID_NEXT = 'AUTOMATIC';

-- 问题2: 清空 GTID 执行历史
-- ⚠️ 危险操作，仅在特殊情况下使用
RESET SLAVE ALL;
RESET MASTER;  -- 清空 gtid_executed
```

### 🔥 扩展追问 2：binlog 同步到 Kafka 如何保证顺序？

> A: **binlog 本身是顺序的，但 Kafka 分区会打乱顺序。**

```java
// 问题分析
// ┌─────────────────────────────────────────────────────────┐
// │  MySQL binlog:                                           │
// │    T1: INSERT id=1                                       │
// │    T2: UPDATE id=1 SET name='A'                         │
// │    T3: UPDATE id=1 SET name='B'                         │
// │    按顺序: 1 → 'A' → 'B'                                 │
// │                                                         │
// │  Kafka 多分区:                                           │
// │    Partition 0: T1, T3                                   │
// │    Partition 1: T2                                       │
// │    Consumer 可能先处理 T3 再处理 T2                      │
// │    结果: id=1 的最终状态可能是 'A' 而不是 'B'             │
// └─────────────────────────────────────────────────────────┘

// 解决方案1: 单分区顺序
// Canal 配置
canal:
  mq:
    partitions: 1  # 单分区保证顺序

// 解决方案2: 按主键哈希分区
// Canal 配置
canal:
  mq:
    partitionHash: db.table.*:pk  # 按表+主键哈希

// 解决方案3: 消费端本地排序
public class OrderedConsumer {
    
    private Map<String, List<BinlogEvent>> buffer = new ConcurrentHashMap<>();
    
    public void consume(BinlogEvent event) {
        String key = event.getTable() + ":" + event.getPrimaryKey();
        
        // 先缓存
        buffer.computeIfAbsent(key, k -> new CopyOnWriteArrayList<>())
              .add(event);
        
        // 检查是否可以按顺序处理
        List<BinlogEvent> events = buffer.get(key);
        if (isOrdered(events)) {
            events.sort(Comparator.comparingLong(BinlogEvent::getSeq));
            events.forEach(this::process);
            buffer.remove(key);
        }
    }
}
```

## 5. 一句话总结

> **binlog 有 STATEMENT/ROW/MIXED 三种格式，ROW 记录每行完整数据最安全但空间最大，STATEMENT 只记 SQL 可能丢数据；`binlog_row_image` 控制 ROW 格式记录哪些列；Canal/Debezium 可能丢数据（位点未提交）或重复消费（at-least-once），需要消费端幂等；GTID 是全局事务标识，解决了传统 position 复制的主从切换难题，但有临时表、跨库事务等限制。**

## 6. 面试加分话术

> "我们做过 Canal 到 Elasticsearch 的数据同步方案，踩过一个坑：ROW 格式的 binlog 有 100G+，Canal 解析速度跟不上，延迟从秒级飙升到小时级。最后优化了三点：1）开启 `binlog_row_image=MINIMAL`，减少 60% binlog 体积，2）Canal 加了消息缓冲，攒批写入 Kafka 再消费，3）ES 端用 bulk API 批量索引。另外关于重复消费问题：Canal 官方说 at-least-once，所以我们用 MySQL 的主键做 ES 的 `_id`，这样即使重复消费也只是覆盖，不会产生脏数据。GTID 方案我们也在用，最大的好处是主从切换时不用再算 position 了，`MASTER_AUTO_POSITION=1` 一切搞定，从库自动对齐。"
