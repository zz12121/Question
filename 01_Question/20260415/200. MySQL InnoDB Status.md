### PURGE 线程详解

```

│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                       │   │

│  │                                                       │   │

SHOW ENGINE INNODB STATUS\G

SELECT 
    NAME, 
    COMMENT 
FROM performance_schema.setup_instruments 
WHERE NAME LIKE '%purge%';

-- ========== 3. 监控 purge 进度 ==========
SHOW GLOBAL STATUS LIKE 'Innodb_purge%';

-- 输出:
-- Innodb_purge_rseg_push_size        0
-- Innodb_purge_queue_length          0
-- Innodb_purge_batch_size            300
-- Innodb_system_trx_id               123456789
-- Innodb_trx_immutable               0

-- ========== 4. 查看长事务 ==========
SELECT 
    trx_id,
    trx_state,
    trx_started,
    trx_tables_locked,
    trx_rows_locked,
    trx_mysql_thread_id,
    trx_query,
    trx_isolation_level
FROM information_schema.INNODB_TRX
ORDER BY trx_started;

-- 找出超过 1 小时的长事务
SELECT 
    TIMESTAMPDIFF(HOUR, trx_started, NOW()) AS hours,
    trx_id,
    trx_query
FROM information_schema.INNODB_TRX
WHERE TIMESTAMPDIFF(HOUR, trx_started, NOW()) > 1;

-- ========== 5. UNDO 相关配置 ==========
SHOW VARIABLES LIKE 'innodb_undo%';

-- innodb_undo_tablespaces: Undo 表空间数量
-- innodb_undo_log_truncate: 是否截断 undo 表空间
-- innodb_undo_directory: Undo 文件存放目录
```

## 2. 扩展追问

### 🔥 扩展追问 1：如何解决 History List 过大的问题？

> A: **通过调优 purge 线程、处理长事务、配置 undo 自动回收来解决。**

```sql
-- ========== 方案1：增加 Purge 线程 ==========
SET GLOBAL innodb_purge_threads = 8;

-- 动态调整后生效，但需要重启才能持久化
-- 需要同时确保 innodb_undo_tablespaces >= innodb_purge_threads

-- ========== 方案2：处理长事务 ==========
-- 找出长事务并通知相关方提交或回滚
SELECT 
    trx_mysql_thread_id AS thread_id,
    trx_started,
    TIMESTAMPDIFF(MINUTE, trx_started, NOW()) AS minutes,
    trx_query
FROM information_schema.INNODB_TRX
ORDER BY minutes DESC;

-- 如果是测试环境，可以直接 kill
-- KILL <thread_id>;

-- ========== 方案3：开启 Undo 表空间自动截断 ==========
-- MySQL 5.7+ 支持
SET GLOBAL innodb_undo_log_truncate = ON;
SET GLOBAL innodb_undo_tablespaces = 4;  -- 至少 2 个
SET GLOBAL innodb_undo_log_encrypt = OFF;

-- 当某个 undo 表空间超过阈值时，会被截断回收

-- ========== 方案4：调整 innodb_max_purge_lag ==========
-- History List 超过此值时，延迟 DML 操作
SET GLOBAL innodb_max_purge_lag = 10000;  -- 默认 0，不延迟

-- 注意：延迟 DML 会影响性能，但比 undo 空间耗尽好

-- ========== 方案5：手动触发 Purge ==========
-- 设置 innodb_purge_stop_now = ON 可以暂停 purge
SET GLOBAL innodb_purge_stop_now = ON;

-- 处理完后恢复
SET GLOBAL innodb_purge_stop_now = OFF;

-- ========== 方案6：清理旧的 Undo 页面 ==========
-- 查看 undo 表空间使用情况
SELECT 
    NAME,
    SPACE,
    SIZE,
    STATE
FROM information_schema.INNODB_TABLESPACES
WHERE NAME LIKE '%undo%';
```

### 🔥 扩展追问 2：innodb_max_purge_lag 延迟机制原理？

> A: **通过延迟 DML 操作来减缓 History List 增长速度。**

```
┌─────────────────────────────────────────────────────────────┐
│              innodb_max_purge_lag 延迟机制                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【工作原理】                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                       │   │
│  │  History List Length = 5000                         │   │
│  │  innodb_max_purge_lag = 10000                        │   │
│  │                                                       │   │
│  │  条件: History List > innodb_max_purge_lag          │   │
│  │        5000 > 10000? 否 → 不延迟                      │   │
│  │                                                       │   │
│  │  当 History List = 15000:                          │   │
│  │        15000 > 10000? 是 → 延迟 DML                  │   │
│  │                                                       │   │
│  │  延迟时间计算:                                        │   │
│  │  delay = (History List - max_purge_lag) * 10us     │   │
│  │  = (15000 - 10000) * 10 = 50000us = 50ms           │   │
│  │                                                       │   │
│  │  每个 DML 操作额外等待 50ms                          │   │
│  │                                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  【innodb_max_purge_lag_delay】                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                       │   │
│  │  最大延迟时间上限                                     │   │
│  │  innodb_max_purge_lag_delay = 0 (默认)              │   │
│  │                                                       │   │
│  │  设置后:                                              │   │
│  │  delay = min((History - max) * 10us, max_delay)    │   │
│  │                                                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 🔥 扩展追问 3：DELETE 和物理删除的关系？

> A: **DELETE 是逻辑删除，Purge 线程负责物理删除。**

```sql
-- DELETE 语句的执行过程
BEGIN;
DELETE FROM orders WHERE id = 100;  -- 逻辑删除
COMMIT;

-- 执行过程：
-- ┌─────────────────────────────────────────────────────────┐
-- │  1. 在记录头信息中标记 deleted_flag = 1                  │  │
-- │                                                         │  │
-- │  2. 如果有辅助索引，更新辅助索引                          │  │
-- │                                                         │  │
-- │  3. 记录 undo (用于回滚)                               │  │
-- │                                                         │  │
-- │  4. COMMIT 后，这条记录:                                │  │
-- │     - 在 MVCC 视角仍然"存在"                          │  │
-- │     - 但不可见                                          │  │
-- │                                                         │  │
-- │  5. Purge 线程在适当时候:                               │  │
-- │     - 物理删除记录                                      │  │
-- │     - 释放页面空间                                      │  │
-- │     - 更新辅助索引                                      │  │
-- └─────────────────────────────────────────────────────────┘

-- 为什么不立即物理删除？
-- ┌─────────────────────────────────────────────────────────┐
-- │                                                         │
-- │  问题: 如果其他事务正在读取旧版本怎么办？                │  │
-- │                                                         │  │
│  │  事务 A: SELECT * FROM orders WHERE id = 100;         │  │
│  │    → 应该看到这条记录（事务A started before DELETE）  │  │
│  │                                                         │  │
│  │  事务 B (id 更大): DELETE orders WHERE id = 100;      │  │
│  │    → 提交                                              │  │
│  │                                                         │  │
│  │  如果立即物理删除:                                      │  │
│  │    事务 A 无法读取历史版本 → 数据不一致！               │  │
│  │                                                         │  │
│  │  正确做法:                                              │  │
│  │    1. 标记为 deleted                                   │  │
│  │    2. 等 Purge Thread 确认 无人需要后再删除             │  │
│  │                                                         │  │
│  └─────────────────────────────────────────────────────────┘
```

### 🔥 扩展追问 4：MySQL 8.0 的 Undo Log 独立表空间有什么改进？

> A: **MySQL 8.0 支持独立的 Undo 表空间，可以在线截断回收空间。**

```sql
-- ========== MySQL 8.0 Undo 表空间配置 ==========

-- 创建独立的 Undo 表空间
CREATE UNDO TABLESPACE undo_003 ADD DATAFILE 'undo_003.ibu';

-- 查看当前 undo 表空间
SELECT 
    NAME,
    SPACE,
    FILE_TYPE,
    STATE
FROM information_schema.FILES
WHERE FILE_TYPE = 'UNDO LOG';

-- 输出示例:
-- +----------+-------+------------+-------+
-- | NAME     | SPACE | FILE_TYPE | STATE |
-- +----------+-------+------------+-------+
-- | innodb_0 | 0     |UNDO LOG   | active|
-- | innodb_1 | 1     |UNDO LOG   | active|
-- | innodb_2 | 2     |UNDO LOG   | active|
-- | undo_003 | 3     |UNDO LOG   | active|
-- +----------+-------+------------+-------+

-- 开启自动截断
SET GLOBAL innodb_undo_log_truncate = ON;

-- 截断阈值 (表空间使用率超过此值会触发截断)
SET GLOBAL innodb_undo_log_encrypt = ON;  -- 加密

-- 查看截断状态
SHOW ENGINE INNODB STATUS\G
-- Trx purge vector looks like:
--  undo 0 on nid with 0 uses, 0 pages truncated
--  undo 1 on nid with 0 uses, 0 pages truncated
```

## 3. 一句话总结

> **SHOW ENGINE INNODB STATUS 的 TRANSACTIONS 部分显示 History List Length（未清理的 undo 记录数），过长会导致 undo 空间膨胀和 MVCC 变慢；Purge 线程负责删除不再需要的 undo log 和已标记删除的记录；通过增加 `innodb_purge_threads`、处理长事务、开启 Undo 表空间自动截断来解决 History List 过长问题。**

## 4. 面试加分话术

> "之前排查过一个 MySQL 从库延迟问题：监控发现 History List Length 从几千飙升到几十万，延迟从秒级变成几小时。根因是应用有个定时任务，每 5 分钟批量更新 10 万条订单状态，产生了大量 undo，而应用启用了 autocommit=1 导致每个 UPDATE 都是独立事务。MySQL 的 Purge 线程是批量处理的，跟不上产生速度。解决方案：1）把这个定时任务改成大事务，`BEGIN; UPDATE ... WHERE ...; COMMIT;`，2）把 autocommit 改成 OFF 并显式事务管理，3）增加 `innodb_purge_threads` 到 8，4）开启 `innodb_undo_log_truncate` 自动回收空间。调优后 History List 稳定在几百以内。另外还有个经验：如果 History List 超过百万，优先找长事务，因为一个运行 2 小时的事务可能阻塞所有 purge。"
