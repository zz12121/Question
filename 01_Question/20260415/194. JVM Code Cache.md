## 1. 基础流程

### 什么是 Code Cache？

Code Cache 是 JVM 的一块**非堆内存区域**，用于存储 JIT 编译器生成的 native 机器码。

### 核心配置参数

```bash
# ========== Code Cache 大小配置 ==========
# 初始大小 (默认 2496KB ≈ 2.4MB)
-XX:InitialCodeCacheSize=2496K

# 保留大小 (默认 48MB，32位 JVM 是 32MB)
-XX:ReservedCodeCacheSize=48M

# JDK 8 之前 Code Cache 是 PermGen 的一部分
# JDK 8+ Code Cache 是独立的 Native Memory 区域

# ========== 分段 Code Cache (JDK 9+) ==========
# 启用分段
-XX:+SegmentedCodeCache

# 各段大小 (JDK 9+ 支持)
-XX:NonNMethodCodeHeapSize=5M      # Non-Nmethod 段
-XX:ProfiledCodeHeapSize=20M       # Profiled 段  
-XX:NonProfiledCodeHeapSize=20M    # Non-Profiled 段

# ========== 其他相关参数 ==========
# JIT 编译线程数
-XX:CICompilerCount=4

# 编译阈值
-XX:CompileThreshold=10000

# Tiered Compilation 开关
-XX:+TieredCompilation
```

### 监控 Code Cache 使用率

```bash
# ========== 1. jstat 监控 ==========
# JDK 8 及之前
jstat -printcompilation <pid> 1000

# JDK 9+ 
jstat -class <pid>
jstat -compiler <pid>

# ========== 2. jcmd 查看 ==========
jcmd <pid> VM.native_memory summary

# 输出中的 "Code" 部分就是 Code Cache 使用情况
# Total: reserved=524288KB, committed=524288KB
#   - Code: reserved=48MB, committed=48MB

# ========== 3. NMT (Native Memory Tracking) ==========
# 启动时开启
java -XX:NativeMemoryTracking=summary -jar app.jar

# 运行中分析
jcmd <pid> VM.native_memory summary

# 跟踪 Code Cache
jcmd <pid> VM.native_memory baseline
# ... 运行一段时间 ...
jcmd <pid> VM.native_memory summary.diff

# ========== 4. -XX:+PrintCodeCache ==========
# 每次 JIT 编译时打印
java -XX:+PrintCodeCache -jar app.jar

# 输出示例：
# CodeCache: size=2457600 Used=2457600 avail=0 max=2457600
# # WARNING: This output is likely truncated.

# ========== 5. JMX / JConsole ==========
# 使用 JConsole 连接
# MBean: java.lang.MemoryPool -> Code Cache

# ========== 6. Arthas ==========
dashboard  # 内存区域信息

# ========== 7. 代码获取 ==========
import java.lang.management.*;

public class CodeCacheMonitor {
    public static void main(String[] args) {
        MemoryPoolMXBean codeCache = ManagementFactory.getMemoryPoolMXBeans()
            .stream()
            .filter(p -> p.getName().contains("Code Cache"))
            .findFirst()
            .orElseThrow();
        
        System.out.println("Code Cache Info:");
        System.out.println("  Name: " + codeCache.getName());
        System.out.println("  Type: " + codeCache.getType());
        System.out.println("  Usage: " + codeCache.getUsage());
        System.out.println("  Peak: " + codeCache.getPeakUsage());
        
        MemoryUsage usage = codeCache.getUsage();
        long used = usage.getUsed();
        long max = usage.getMax();
        
        System.out.println("使用率: " + (used * 100.0 / max) + "%");
        
        // 告警阈值
        if (used > max * 0.9) {
            System.out.println("⚠️ Code Cache 使用率超过 90%!");
        }
    }
}
```

## 2. 扩展追问

### 🔥 扩展追问 1：ReservedCodeCacheSize 和 InitialCodeCacheSize 的关系？

> A: **ReservedCodeCacheSize 是 Code Cache 的最大上限，InitialCodeCacheSize 是启动时预留的起始大小。**

```java
// 关系图示
// InitialCodeCacheSize: JVM 启动时立即向 OS 申请的内存
// ReservedCodeCacheSize: Code Cache 可以扩展到的最大大小

memory layout:
// Initial 时
┌─────────────────────────────────────────────┐
│  Reserved (max) = 48MB                       │
│  ┌──────────────────────────────────────┐  │
│  │  Committed (init) = 2.4MB             │  │  ← 实际分配
│  │  [ InitialCodeCacheSize ]            │  │
│  └──────────────────────────────────────┘  │
└─────────────────────────────────────────────┘

// 运行时扩展
┌─────────────────────────────────────────────┐
│  Reserved (max) = 48MB                       │
│  ┌──────────────────────────────────────┐  │
│  │  Committed = 30MB (动态增长)          │  │  ← 按需增长
│  │  [ Used: 28MB ]                      │  │
│  │                                      │  │
│  │  [ Free: 2MB ]                       │  │
│  └──────────────────────────────────────┘  │
└─────────────────────────────────────────────┘

// 满了
┌─────────────────────────────────────────────┐
│  Reserved (max) = 48MB                       │
│  ┌──────────────────────────────────────┐  │
│  │  Committed = 48MB (MAX!)             │  │  ← 无法再增长
│  │  [ Used: 48MB ]                      │  │
│  │  [ Free: 0 ] ← 无法再分配            │  │
│  └──────────────────────────────────────┘  │
│                                              │
│  ⚠️ JIT 编译停止！                          │
└─────────────────────────────────────────────┘

// 配置建议：
// - InitialCodeCacheSize: 通常保持默认，2.4MB 足够初期使用
// - ReservedCodeCacheSize: 根据应用复杂度调整
//   • 简单应用：48MB 足够
//   • 复杂应用/分层编译：考虑 96MB-256MB
```

### 🔥 扩展追问 2：Code Cache 和类卸载的关系？

> A: **Code Cache 中的 nmethod（native code）必须随 ClassLoader 一起被卸载和清理。**

### 🔥 扩展追问 3：如何优化 Code Cache 使用？

> A: **通过合理配置、代码优化、监控告警三方面入手。**

```java
// ========== 1. 合理设置大小 ==========
// JVM 参数
-XX:ReservedCodeCacheSize=96M    # 大应用
-XX:InitialCodeCacheSize=2M
-XX:+UseCodeCacheFlushing        # 尝试回收未使用代码缓存（默认开启）

// ========== 2. 分段代码缓存 (JDK 9+) ==========
// 分段后更容易管理，也更容易定位问题
-XX:+SegmentedCodeCache

// ========== 3. 减少 JIT 编译量 ==========
// 避免无意义的热点
public class BadCodeExample {
    // ❌ 每次调用都创建新对象 → 逃逸分析失效 → 额外编译
    public String concatBad(String a, String b) {
        return new StringBuilder().append(a).append(b).toString();
    }
    
    // ✓ 复用 StringBuilder
    public String concatGood(String a, String b) {
        return a + b;  // JIT 会优化
    }
}

// ========== 4. 监控脚本示例 ==========
import java.lang.management.*;

public class CodeCacheWatcher {
    public static void main(String[] args) throws Exception {
        while (true) {
            MemoryPoolMXBean codeCache = ManagementFactory.getMemoryPoolMXBeans()
                .stream()
                .filter(p -> p.getName().contains("Code Cache"))
                .findFirst()
                .orElseThrow();
            
            MemoryUsage usage = codeCache.getUsage();
            long used = usage.getUsed();
            long max = usage.getMax();
            double ratio = used * 100.0 / max;
            
            System.out.printf("[%tT] CodeCache: %d/%d (%.1f%%)%n", 
                System.currentTimeMillis(), used, max, ratio);
            
            if (ratio > 95) {
                System.err.println("🚨 CRITICAL: CodeCache 即将满！");
                // 触发告警：发送邮件/钉钉/短信
            }
            
            Thread.sleep(30000);  // 每30秒检查一次
        }
    }
}
```

### 🔥 扩展追问 4：分层编译各层代码大小差异有多大？

> A: **C2 生成的代码通常比 C1 大 5-10 倍，因为优化程度更高。**

| 特性 | C1 (Tier 1/2/3) | C2 (Tier 4) |
|------|-----------------|--------------|
| **优化程度** | 轻量优化 | 激进优化 |
| **代码大小** | 基准 | +5~10x |
| **编译时间** | 快 (<10ms) | 慢 (50-500ms) |
| **执行速度** | 基准 | +2~5x |
| **典型优化** | 基本内联、寄存器分配 | 虚调用内联、逃逸分析、锁消除 |

```java
// 实际案例对比
public class OptimizedCode {
    // 原始代码
    public int sum(List<Integer> list) {
        int sum = 0;
        for (Integer i : list) {
            sum += i;
        }
        return sum;
    }
    
    // C1 编译：基础内联
    // 生成的机器码约 100 字节
    
    // C2 编译：可能完全内联 + 向量化
    // 生成的机器码约 800 字节
    // 循环展开、SIMD 指令
    
    // 内存占用估算：
    // 1000 个热点方法 × 800 字节 = 800KB
    // 如果每个方法有多个编译版本 → 2-4MB
}
```

## 3. 一句话总结

> **Code Cache 是 JVM 存储 JIT 编译 native 代码的非堆内存区域，默认 48MB 满则 JIT 停止导致性能骤降；C2 代码比 C1 大 5-10 倍，分层编译通过 C1→C2 渐进优化平衡编译速度和代码质量；通过 NMT、jstat、PrintCodeCache 监控，SegmentedCodeCache 分段管理。**

## 4. 面试加分话术

> "生产环境我们遇到过 Code Cache 满的问题：应用启动后 3 天开始频繁 Full GC，且响应时间从 50ms 飙升到 500ms。一开始以为是堆内存问题，用 NMT 分析才发现是 Code Cache 使用率飙到 99%。根因是业务代码中有大量动态生成的类（脚本引擎、规则引擎），每个类都触发 JIT 编译。最终通过两个方案解决：1）增加 `ReservedCodeCacheSize` 到 128MB，2）限制动态类生成的上限。从那以后我在监控平台上加了 Code Cache 使用率的告警，阈值设到 80% 就预警。"

---

**追问预判：**
- "Code Cache 是堆外内存吗？" → 是，属于 Native Memory，受 OS 物理内存约束
- "Code Cache flushing 是什么？" → JVM 尝试回收未使用代码空间（`-XX:+UseCodeCacheFlushing`），但效果有限
