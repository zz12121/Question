### 1.1 传统 requirepass 的痛点

├─────────────────┼────────────────────────────────────────────┤

### 1.5 核心配置示例

```bash
# redis.conf

# 启用 ACL（Redis 6.0+ 默认开启）
aclfile /etc/redis/redis-users.acl

# ========== 用户定义 ==========

# 超级管理员（所有权限）
user admin on >"Str0ngAdm1nP@ss!" ~* +@admin

# 业务读写用户（只能操作 cache:* 和 session:* 前缀的 key）
user app_rw on >"AppR3adWr1t3K3y!" ~cache:*|session:* +@read +@write -@dangerous

# 业务只读用户（只能读取 cache:* 前缀的 key）
user app_ro on >"AppR34d0nlyK3y!" ~cache:* -@write +@read

# 监控用户（只允许执行监控相关命令）
user monitor_user on >"M0n1t0rK3y!" ~* +@read +MONITOR +CLIENT LIST +INFO

# 只允许 pub/sub 的用户
user pubsub_user on >"PubSubK3y!" ~* +@pubsub -@write -@dangerous

# 禁用默认用户（必须显式创建用户才能连接）
user default on nopass ~* +@read -@dangerous

# ========== 全局开关 ==========
# 是否在 ACL 规则错误时关闭连接（安全优先）
acl-sanitize-dump yes

# 是否在命令执行前检查 ACL（默认开启）
acl-check-permissions yes
```

## 🔥 扩展追问 1：USER 权限和 KEY 权限的区别是什么？两者如何配合？

> A: **USER 权限控制"能执行什么命令"，KEY 权限控制"能操作哪些 key"。**

```
USER 权限（命令级别）：

┌─────────────────────────────────────────────────────────────┐
│  +@read      允许所有读命令 (GET, MGET, HGET, LRANGE...)     │
│  -@write     禁止所有写命令 (SET, DEL, HSET...)             │
│  +@admin     允许管理命令 (CONFIG, SHUTDOWN...)              │
│  -@dangerous 禁止危险命令 (FLUSHDB, KEYS, MONITOR...)        │
└─────────────────────────────────────────────────────────────┘
```

```
KEY 权限（数据级别）：

┌─────────────────────────────────────────────────────────────┐
│  ~cache:*        允许访问 cache: 开头的所有 key             │
│  ~session:*      允许访问 session: 开头的所有 key           │
│  ~order:*        允许访问 order: 开头的所有 key             │
│  ~pub:*|sub:*    允许访问 pub:* 或 sub:* 前缀              │
│  ~*              允许访问所有 key（慎用）                   │
│  %w~news:*       只允许写 news: 开头的 key（Redis 7.0+）    │
│  %r~msg:*        只允许读 msg: 开头的 key（Redis 7.0+）    │
└─────────────────────────────────────────────────────────────┘
```

```
两者配合示例：

场景：只读用户，但只能看自己的数据

# 用户 alice 只能读取 user:alice:* 前缀的 key
ACL SETUSER alice on >password ~user:alice:* +@read -@write -@dangerous

# 验证
AUTH alice password
GET user:alice:profile     # ✅ OK
GET user:bob:profile       # ❌ (error) NOGROUP Permission denied
GET cache:global:config    # ❌ (error) NOGROUP Permission denied
SET user:alice:profile x   # ❌ (error) NOGROUP Permission denied
```

## 🔥 扩展追问 2：Redis 7.0 的 ACL 有什么新特性？`%R` 和 `%W` 模式是什么？

> A: **Redis 7.0 引入了 key 的读写分离权限控制。**

```
Redis 7.0 新增特性：

┌─────────────────────────────────────────────────────────────┐
│  %R~<pattern>    只允许读（Read）符合模式的 key              │
│  %W~<pattern>   只允许写（Write）符合模式的 key             │
│  %RW~<pattern>  允许读写符合模式的 key（等价于 ~pattern）   │
└─────────────────────────────────────────────────────────────┘

使用场景对比：

# 传统方式（Redis 6.x）
# 用户只能访问 cache:* 前缀，但这个配置允许读写
user app_rw on >password ~cache:* +@read +@write -@dangerous

# Redis 7.0 方式（更细粒度）
# 用户可以读所有 cache:* 前缀，但只能写 cache:temp:* 前缀
user app_rw on >password %R~cache:* %W~cache:temp:* +@read +@write -@dangerous

# 示例：
AUTH app_rw password
GET cache:user:1          # ✅ OK（读 cache:*）
GET cache:product:2        # ✅ OK（读 cache:*）
SET cache:user:1 "new"     # ❌ Permission denied（不能写 cache:user:1）
SET cache:temp:session "x" # ✅ OK（写 cache:temp:*）
```

## 🔥 扩展追问 3：如何实现"某个客户端只能访问以 `user:` 为前缀的 key"？

> A: **使用 `~user:*` key 模式 + 最小命令权限。**

```bash
# 方法1：命令行创建
ACL SETUSER user_reader on >"UserR34dK3y!" ~user:* +@read -@write -@dangerous -@admin

# 方法2：redis.conf 配置
user user_reader on >"UserR34dK3y!" ~user:* +@read -@write -@dangerous -@admin

# 验证流程
redis-cli -u redis://user_reader:UserR34dK3y!@127.0.0.1:6379

GET user:123:profile   # ✅ OK
GET user:456:token     # ✅ OK
GET order:789:info     # ❌ NOGROUP Permission denied
GET cache:global       # ❌ NOGROUP Permission denied
SET user:123:profile x # ❌ NOGROUP Permission denied
DEL user:789           # ❌ NOGROUP Permission denied
```

```python
# Python 连接示例（使用 redis-py）
import redis

# 只读用户连接
r = redis.Redis(
    host='127.0.0.1',
    port=6379,
    password='UserR34dK3y!',
    username='user_reader'
)

r.get('user:123:profile')   # ✅ 正常读取
r.get('order:789:info')     # ❌ 权限拒绝
r.set('user:123:profile', 'x')  # ❌ 权限拒绝
```

## 🔥 扩展追问 4：ACL 的 `RESET` 和 `ON`/`OFF` 是什么？用户状态机是怎么工作的？

> A: **ACL 用户有状态机，`RESET` 会重置到初始状态，`ON`/`OFF` 控制启用/禁用。**

```
ACL 用户状态机：

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   ┌─────────┐                                               │
│   │  RESET  │ ◀── 默认状态，重置所有权限                    │
│   └────┬────┘                                               │
│        │                                                    │
│        ▼                                                    │
│   ┌─────────────────────────────────────────┐               │
│   │ enabled: off                           │               │
│   │ - 无法登录                              │               │
│   │ - AUTH 该用户会被拒绝                   │               │
│   └────────────────────┬────────────────────┘               │
│                        │ ON                                  │
│                        ▼                                    │
│   ┌─────────────────────────────────────────┐               │
│   │ enabled: on, nopass                     │               │
│   │ - 可以登录，无需密码                      │               │
│   │ - 无 key 权限                            │               │
│   │ - 无命令权限                            │               │
│   └────────────────────┬────────────────────┘               │
│                        │ >password                           │
│                        ▼                                    │
│   ┌─────────────────────────────────────────┐               │
│   │ enabled: on, password set              │               │
│   │ - 需要密码认证                           │               │
│   │ - 根据规则允许/拒绝命令                   │               │
│   └─────────────────────────────────────────┘               │
│                                                             │
└─────────────────────────────────────────────────────────────┘

常用命令：

# 重置用户到初始状态（清空所有规则）
ACL SETUSER reader RESET

# 启用用户 + 设置密码
ACL SETUSER reader on >password

# 禁用用户
ACL SETUSER reader off

# 无密码用户（危险！）
ACL SETUSER reader on nopass ~* +@admin
```

## 🔥 扩展追问 5：ACL 和主从复制、哨兵、集群的关系？

> A: **主从复制的认证和 ACL 是独立的，但可以通过 ACL 控制从节点权限。**

```
主从复制 + ACL 架构：

┌──────────────────┐                    ┌──────────────────┐
│     Master       │◀── REPLICAOF ────│     Replica      │
│                  │                    │                  │
│  用户配置:        │                    │  用户配置:        │
│  - admin (master)│                    │  - replica_user  │
│  - app_rw        │                    │                  │
│                  │                    │  权限要求:       │
│  权限: +@admin   │                    │  - REPLICAOF     │
│  权限: +@read    │                    │  - +@read        │
│        +@write  │                    │  - 只能读        │
└──────────────────┘                    └──────────────────┘

Replica 配置：
masteruser replica_user
masterauth replica_pass

# Replica 用户权限配置（在 Master 上）
ACL SETUSER replica_user on >replica_pass ~* +@read +REPLICAOF +CONFIGURE
```

```
哨兵 + ACL：

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   哨兵配置文件 sentinel.conf:                               │
│                                                             │
│   sentinel auth-user replica_user                          │
│   sentinel auth-pass replica_pass                          │
│                                                             │
│   这样哨兵在 failover 时可以正确认证到新 master             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 3. 一句话总结

> **Redis ACL 通过 `USER`（命令权限）+ `KEY`（数据权限）两层模型实现最小权限原则，核心规则是 `+@category / -@category` 控制命令，`~pattern` 控制 key 访问范围，比 requirepass 的全局密码安全 100 倍。**

## 4. 面试加分话术

> "我们线上 Redis 6.0 升级后全面落地了 ACL。拿一个具体场景举例：原来是所有微服务共用一个密码，监控用一套、缓存读写用一套、运维工具又用一套，特别混乱。升级后我们拆成了 4 个用户——`app_ro` 只有 `+@read` 权限，只能读 `cache:*` 前缀；`app_rw` 有 `+@read +@write` 权限，只能操作 `cache:*` 和 `session:*`；`monitor_user` 只允许 `+MONITOR` 和 `+CLIENT LIST`；`admin` 是 DBOps 专用，`+@admin` 全权限。每次配置变更走 ACL 文件发布，`SENTINEL` 配置里也配了 `auth-user`，failover 时不会认证失败。生产验证下来，ACL 拒绝日志里 90% 是开发测试环境的误操作，真正被攻击的请求一次都没有——最小权限原则真的管用。"
