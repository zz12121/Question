# 196. Java Compact Strings（JDK 9）

## 1. 基础流程

### 什么是 Compact Strings？

JDK 9 引入的优化，将 String 内部存储从 `char[]`（UTF-16 固定）改为 `byte[]` + `coder`（动态编码）。

### coder 切换机制

```java
// String 内部关键代码（JDK 9+）
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {
    
    // 编码器常量
    static final byte LATIN1 = 0;
    static final byte UTF16  = 1;
    
    // 内部存储
    @Stable
    private final byte[] value;
    
    private final byte coder;  // LATIN1 或 UTF16
    
    // 判断是否需要 LATIN1
    private static boolean isLatin1(String value) {
        return COMPACT_STRINGS && coder == LATIN1;
    }
    
    // 判断字符串是否全部是 Latin1 字符
    public static boolean isLatin1(char[] value, int off, int len) {
        // 遍历检查每个字符是否 <= 255
        for (int i = 0; i < len; i++) {
            if (value[off + i] > 0xFF) {  // 超出 LATIN1 范围
                return false;
            }
        }
        return true;
    }
    
    // 构造器：根据内容选择编码
    public String(char[] value, int off, int len, Void nullCheck) {
        if (len > 0) {
            // 尝试使用 LATIN1
            if (COMPACT_STRINGS && isLatin1(value, off, len)) {
                this.coder = LATIN1;
                this.value = StringUTF16.compress(value, off, len);
            } else {
                this.coder = UTF16;
                this.value = StringUTF16.toBytes(value, off, len);
            }
        } else {
            this.coder = LATIN1;
            this.value = ByteArray.EMPTY_BYTES;
        }
    }
    
    // 静态编译开关
    private static final boolean COMPACT_STRINGS;
    static {
        COMPACT_STRINGS = VersionSpecific.isSupported(9);
    }
}
```

## 2. 扩展追问

### 🔥 扩展追问 1：对 `String.compareTo()` 有什么影响？

> A: **JDK 9+ 实现了 LATIN1 和 UTF16 两套 compareTo，LATIN1 路径更快。**

```java
// JDK 9+ StringLatin1 / StringUTF16 compareTo

// StringLatin1.compareTo (LATIN1 路径)
public static int compareTo(byte[] value, byte[] other) {
    int len1 = value.length;
    int len2 = other.length;
    int lim = Math.min(len1, len2);
    
    for (int k = 0; k < lim; k++) {
        // byte 直接比较，无需 char→byte 转换
        int c1 = value[k] & 0xFF;  // unsigned
        int c2 = other[k] & 0xFF;
        if (c1 != c2) {
            return c1 - c2;
        }
    }
    return len1 - len2;
}

// StringUTF16.compareTo (UTF16 路径)
public static int compareTo(byte[] value, byte[] other) {
    int len1 = value.length >> 1;  // byte → char 转换
    int len2 = other.length >> 1;
    int lim = Math.min(len1, len2);
    
    for (int k = 0; k < lim; k++) {
        int c1 = getChar(value, k);
        int c2 = getChar(other, k);
        if (c1 != c2) {
            return c1 - c2;
        }
    }
    return len1 - len2;
}

// 性能对比：
// ┌─────────────────────────────────────────────────────────┐
// │  操作              │  LATIN1 (JDK9+)  │  UTF16 (JDK9+)   │
// │────────────────────┼──────────────────┼──────────────────│
// │  char比较           │  byte直接比       │  需要getChar    │
// │  内存访问           │  1字节/字符       │  2字节/字符      │
// │  CPU缓存命中率      │  更高             │  更低           │
// │  性能提升           │  ~2x             │  基本持平       │
// └─────────────────────────────────────────────────────────┘
```

### 🔥 扩展追问 2：对 `indexOf()` 有什么影响？

> A: **同样实现了两套 indexOf，LATIN1 使用单字节搜索，效率更高。**

```java
// StringLatin1.indexOf
public static int indexOf(byte[] value, int ch, int fromIndex) {
    if (!canEncode(ch)) {  // 超范围直接返回-1
        return -1;
    }
    byte c = (byte) ch;
    int max = value.length;
    
    // 单字节搜索
    for (int i = fromIndex; i < max; i++) {
        if (value[i] == c) {
            return i;
        }
    }
    return -1;
}

// StringUTF16.indexOf  
public static int indexOf(byte[] value, int ch, int fromIndex) {
    if (ch >= Character.MIN_SUPPLEMENTARY_CODE_VALUE) {
        // 处理增补字符（2个char）
        int[] helper = new int[] { ch };
        return indexOfSupplementary(value, helper, fromIndex);
    }
    char c = (char) ch;
    int max = value.length >> 1;  // byte → char
    
    // 双字节搜索
    for (int i = fromIndex; i < max; i++) {
        if (getChar(value, i) == c) {
            return i;
        }
    }
    return -1;
}

// 性能对比：
// ┌─────────────────────────────────────────────────────────┐
// │  场景              │  LATIN1 vs UTF16 性能差距            │
// │────────────────────┼────────────────────────────────────│
// │  短字符串          │  LATIN1 快 ~30%                     │
// │  长字符串          │  LATIN1 快 ~50%（缓存友好）          │
// │  子字符串搜索      │  差异更大                           │
// └─────────────────────────────────────────────────────────┘
```

### 🔥 扩展追问 4：Compact Strings 对其他方法的影响？

> A: **几乎所有字符串操作都实现了 LATIN1/UTF16 两套实现。**

```java
// 典型方法的两套实现

// toUpperCase / toLowerCase
StringLatin1.toUpperCase(value);   // LATIN1
StringUTF16.toUpperCase(value);     // UTF16

// codePointCount
StringLatin1.codePointCount(value, offset, count);   // 1:1
StringUTF16.codePointCount(value, offset, count);    // 需要处理增补字符

// substring
StringLatin1.newString(value, offset, subLen);   // 返回 LATIN1
StringUTF16.newString(value, offset, subLen);    // 返回 UTF16

// getBytes
StringCoding.encode(value, coder, enc);  // 根据 coder 选择编码器

// 特殊处理：charAt
public char charAt(int index) {
    if (isLatin1()) {
        // LATIN1: byte → char 零扩展
        return (char) (value[index] & 0xFF);
    } else {
        // UTF16: 两个字节组合
        return StringUTF16.getChar(value, index);
    }
}

// 特殊处理：codePointAt
public int codePointAt(int index) {
    if (isLatin1()) {
        return value[index] & 0xFF;
    } else {
        return StringUTF16.getChar(value, index);
    }
}
```

## 3. 一句话总结

> **JDK 9 引入的 Compact Strings 将 String 内部从固定 UTF-16 char[] 改为 byte[] + coder，ASCII 字符串内存节省 50%，中文/混合字符串因编码固定为 UTF16 反而多占 20%；JVM 通过 LATIN1/UTF16 两套实现优化所有字符串操作，LATIN1 路径更高效；可通过 `-XX:+/-CompactStrings` 控制开关。**

## 4. 面试加分话术

> "在优化电商系统时，我们分析过 String 的内存占用：热点数据中 80% 是纯 ASCII（SKU、订单号、用户ID等），引入 JDK 9 后仅 String 优化就减少了 30% 的堆内存占用。具体验证方法：用 `jmap -histo` 看 String 实例数，`jcmd <pid> GC.class_stats` 加上 Instrumentation 算实际内存，配合 `-XX:+PrintStringStatistics` 可以看到 String 的内存节省量。另外注意一个问题：混合字符串（如日志中的中文+英文）会触发 LATIN1→UTF16 切换，这个切换是不可逆的，所以在高并发场景下如果频繁拼接中英文，建议初始化时统一使用 UTF16 编码的 StringBuilder。"
