## 1. 基础流程

### 1.1 为什么需要 Lazy Free？

```
传统 DEL 的问题：
┌─────────────────────────────────────────────────────────────┐
│                        主线程                               │
│  ┌─────────┐                                               │
│  │  DEL    │───▶ 同步释放 10GB 的 Hash ───▶ 阻塞 2-3 秒     │
│  │  key    │                                               │
│  └─────────┘                                               │
└─────────────────────────────────────────────────────────────┘

Lazy Free 的解决方案：
┌─────────────────────────────────────────────────────────────┐
│                        主线程                               │
│  ┌─────────┐      ┌─────────────────┐      ┌─────────────┐  │
│  │ UNLINK  │───▶  │ 引用计数-1      │───▶  │ 后台线程    │  │
│  │  key    │      │ 立即返回         │      │ 异步释放内存 │  │
│  └─────────┘      └─────────────────┘      └─────────────┘  │
│                      (主线程)             (bio_free_crypt)  │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 UNLINK vs DEL 核心区别

```
DEL (同步删除)：
t1: DEL myhash     ──▶ 主线程开始释放 100万个 field
t2: [阻塞 500ms]   ──▶ 只能干等，其他命令全部积压
t3: 返回 OK        ──▶ 其他命令才能继续执行

UNLINK (异步删除)：
t1: UNLINK myhash  ──▶ 主线程：引用计数-1，立即返回
t2: 返回 1         ──▶ 主线程立即响应其他命令
t3: [后台线程]     ──▶ bio_free_crypt 线程开始释放内存
t4: 内存实际释放    ──▶ 与主线程完全隔离
```

### 1.3 核心代码原理（伪代码）

```c
// UNLINK 命令核心逻辑
def unlinkCommand(client):
    key = client.argv[1]
    
    // 1. 主线程操作：引用计数 -1
    robj *o = lookupKey(key, LOOKUP_NOTOUCH)
    if o and o->refcount == 1:
        // 2. 立即从 dict 中摘除（主线程，极快）
        dbDelete(key)  // 只是从哈希表删除
        o->refcount = 0
        
        // 3. 放入待释放队列（主线程，极快）
        bioCreateJob(BIO_FREE_OBJ, o, NULL)  // 后台线程接管
        
        return 1
    else:
        // 引用计数 > 1，还有其他地方引用，改为 DEL
        return deleteKey(key)
```

### 1.5 相关参数配置

```bash
# redis.conf

# UNLINK/FLUSHDB 等异步删除的后台线程数
# 默认: bio_num_ops = 2 (2个后台线程)
# 大数据量场景建议: 4-8
bio_num_ops 4

# 控制 UNLINK/FLUSHDB 异步删除的阈值
# 当单个对象占用的内存超过此值时，才使用后台线程释放
# 默认: 0 (所有删除都异步)
# 如果设置为 64mb，则只有 >64mb 的对象才异步删除
lazyfree-lazy-user-size 64mb

# 防止删除太快（限制每次迭代释放的对象数）
# 默认: 1 (每次事件循环最多释放1个对象)
# 增大可加速释放，但会增加 CPU 开销
lazyfree-lazy-user-flush no

# ========== 以下是自动触发的 lazy free ==========

# 1. 内存淘汰时是否异步删除
lazyfree-lazy-eviction no
# 场景: maxmemory-policy = allkeys-lru 时，淘汰的旧对象异步删除

# 2. 过期 key 被扫描到时是否异步删除
lazyfree-lazy-expire no
# 场景: 过期 key 躺在 dict 里，被 redisActiveDefragSdsDict 扫描到

# 3. 主从复制时，从节点本地删除 RDB 文件是否异步
lazyfree-lazy-server-del no
# 场景: replicaof 切换时，旧数据清理

# 4. 显式调用 DEL 时是否走异步（测试用）
replica-lazy-flush no
```

## 🔥 扩展追问 1：UNLINK 一定比 DEL 快吗？什么时候用 DEL 更合适？

> A: **不一定。UNLINK 的"快"是指主线程响应快，不是整体回收快。**

```
场景对比：

┌─────────────────┬──────────────────────┬──────────────────────┐
│     场景         │        DEL           │       UNLINK         │
├─────────────────┼──────────────────────┼──────────────────────┤
│ 小 Key (1KB)    │ 微秒级，主线程直删    │ 反而多一次入队开销   │
│ 大 Key (1GB)    │ 阻塞 2-5 秒          │ 立即返回，异步释放   │
│ 内存紧张        │ 主线程卡顿+后台线程争抢│ 推荐，减少 OOM 风险  │
│ 内存充裕        │ DEL 更简单直接        │ UNLINK 无副作用      │
└─────────────────┴──────────────────────┴──────────────────────┘

结论：
• 小 Key (< lazyfree-lazy-user-size): DEL 和 UNLINK 性能差异可忽略
• 大 Key: 必须用 UNLINK
• 内存紧张: 必须用 UNLINK（防止主线程阻塞导致OOM检测超时）
```

## 🔥 扩展追问 2：FLUSHDB ASYNC 的执行过程？和 FLUSHDB 的区别？

> A: **FLUSHDB ASYNC 不会真的"清空"数据库，而是把数据库对象标记为待删除。**

```
FLUSHDB (同步阻塞)：
t1: FLUSHDB      ──▶ 主线程遍历 10万个 key，逐个 free
t2: [阻塞 1秒]   ──▶ 所有客户端超时
t3: 返回 OK      ──▶ 数据库清空

FLUSHDB ASYNC (非阻塞)：
t1: FLUSHDB ASYNC
    │
    ├── ① 主线程: 创建一个新的空 dict (新数据库)
    ├── ② 主线程: 将 dbptr 指向新数据库
    ├── ③ 主线程: 旧数据库对象入队 bio_free_crypt
    └── ④ 立即返回 OK (主线程无阻塞)

    ├── [后台线程处理中...]
    ├── bio_free_crypt: 遍历旧 dict，释放 10万个 key
    └── 旧数据库对象完全释放
```

**代码核心：**

```c
// FLUSHDB ASYNC 核心逻辑
def flushdbAsyncCommand(client):
    // 1. 创建新数据库（空 dict）
    dbDict = dictCreate(&dbDictType, NULL)
    
    // 2. 原子替换（主线程，极快）
    client->db->dict = dbDict
    client->db->expires = dictCreate(&keyptrDictType, NULL)
    
    // 3. 旧数据库放入 lazy free 队列
    oldDict = old_db->dict
    oldExpires = old_db->expires
    
    // 4. 释放旧数据库（后台线程）
    bioCreateJob(BIO_FREE_DB, oldDict, oldExpires, 
                 lazyfree-lazy-user-flush ? 1 : 0)
    
    return 1  // 立即返回
```

## 🔥 扩展追问 4：后台线程池 BIO 是什么？UNLINK 的数据什么时候真的被释放？

> A: **BIO (Background I/O) 是 Redis 的后台任务线程池，专门处理耗时操作。**

```
BIO 线程池架构：

┌─────────────────────────────────────────────────────────────┐
│                    BIO 后台线程池                           │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ BIO 线程 0                                            │   │
│  │   处理: BIO_CLOSE_FILE (AOF 刷盘)                    │   │
│  │   处理: BIO_AOF_FSYNC (fsync 调用)                    │   │
│  └─────────────────────────────────────────────────────┘   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ BIO 线程 1                                            │   │
│  │   处理: BIO_FREE_OBJ (大对象释放)                     │   │
│  └─────────────────────────────────────────────────────┘   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ BIO 线程 2                                            │   │
│  │   处理: BIO_FREE_OBJ (大对象释放)                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  队列: pthread_cond_signal + mutex 保证线程安全             │
└─────────────────────────────────────────────────────────────┘

UNLINK 对象的生命周期：

t1: UNLINK mykey
    └── 主线程: obj->refcount = 0
    └── 主线程: 放入 bio_pending[BIO_FREE_OBJ] 队列
    └── 主线程: 返回 1

t2-tN: [后台线程陆续取走并释放]
    └── BIO 线程 1: o = queue.pop()
    └── BIO 线程 1: zfree(o)  // 真正释放内存
    └── BIO 线程 1: signal 主线程: lazyfree_pending_objects--

⚠️ 注意：UNLINK 后内存不一定立即释放给 OS
  • free() 可能不会立即归还给 OS（glibc 的 ptmalloc 行为）
  • 可用 INFO memory 查看 used_memory 和 used_memory_rss
  • 真正归还需要调用 MALLOC_TRIM 或重启
```

## 🔥 扩展追问 6：Redis 6.0 的多线程 IO 和 Lazy Free 的关系？有没有冲突？

> A: **没有冲突，多线程 IO 只处理网络读写，Lazy Free 依然是独立的 BIO 线程。**

```
Redis 6.0 多线程 IO 架构：

┌─────────────────────────────────────────────────────────────────┐
│                        Redis 进程                               │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                    主线程 (main)                        │    │
│  │   处理命令: SET GET UNLINK FLUSHDB ...                  │    │
│  └─────────────────────────────────────────────────────────┘    │
│                            │                                    │
│                            ▼                                    │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │              IO Threads (io_threads_num = 4)            │    │
│  │                                                         │    │
│  │  Thread 1: 读取客户端 A 请求 ──▶ 写回客户端 A 响应       │    │
│  │  Thread 2: 读取客户端 B 请求 ──▶ 写回客户端 B 响应       │    │
│  │  Thread 3: 读取客户端 C 请求 ──▶ 写回客户端 C 响应       │    │
│  │  Thread 4: 读取客户端 D 请求 ──▶ 写回客户端 D 响应       │    │
│  └─────────────────────────────────────────────────────────┘    │
│                            │                                    │
│                            ▼                                    │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │              BIO 后台线程 (lazy free)                    │    │
│  │                                                         │    │
│  │  Thread 1: 释放大对象内存 (UNLINK 的真正工作)            │    │
│  │  Thread 2: 释放大对象内存                                │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  关键点：                                                       │
│  • IO Threads 只做 socket 读写，不处理命令逻辑                   │
│  • UNLINK 命令本身在主线程执行，只是引用计数-1，立即返回          │
│  • 真正的内存释放在 BIO 线程，完全独立于 IO Threads              │
└─────────────────────────────────────────────────────────────────┘
```

## 3. 一句话总结

> **UNLINK 通过引用计数把"删除"动作从主线程转移到后台 BIO 线程，实现真正的非阻塞删除，代价是内存暂时不归还 OS。设置 `lazyfree-lazy-eviction=yes` 是生产环境的标配，防止大 Key 淘汰时雪崩。**

## 4. 面试加分话术

> "我之前踩过一个坑——线上 Redis 内存打满，触发 allkeys-lru 淘汰，结果因为没开 `lazyfree-lazy-eviction`，淘汰一个大 Hash（几百万 field）时主线程阻塞了 3 秒，所有请求超时，客户端疯狂重试，直接把 Redis 打死了。后来紧急重启、调参才恢复。从那以后我把 `lazyfree-lazy-eviction`、`lazyfree-lazy-expire`、`lazyfree-lazy-server-del` 全都开了，`maxmemory` 也留了 20% buffer 给后台线程。UNLINK 和 DEL 的选择也定了：代码里统一用 UNLINK，小 Key 性能损失微乎其微，大 Key 不会阻塞主线程。"
