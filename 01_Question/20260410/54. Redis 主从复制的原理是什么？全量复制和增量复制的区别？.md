## 先说结论

Redis 主从复制通过 **RDB 全量复制** 或 **复制缓冲区增量复制** 实现数据同步；**全量复制**是首次连接或复制偏移量丢失时传输完整 RDB 文件，**增量复制**是断线重连后在复制积压缓冲区中找到偏移量续传命令，避免全量同步开销。

### 2. 全量复制（Full Resynchronization）

```bash
# 触发条件
1. 从节点首次连接主节点
2. 主节点 runid 变更（重启）
3. 复制偏移量不在积压缓冲区内

# 执行流程
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    全量复制详细流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Master 节点：                                                    │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  1. 收到 PSYNC ? -1 或 offset 不在缓冲区内                │   │
│  │       ↓                                                  │   │
│  │  2. 执行 bgsave，生成 RDB 快照文件                        │   │
│  │       ↓                                                  │   │
│  │  3. fork 子进程进行持久化，父进程继续处理请求              │   │
│  │       ↓                                                  │   │
│  │  4. 将新写入命令存入复制积压缓冲区（replication backlog）   │   │
│  │       ↓                                                  │   │
│  │  5. RDB 生成完成，发送给 Slave                            │   │
│  │       ↓                                                  │   │
│  │  6. 发送积压缓冲区中的命令                                  │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  Slave 节点：                                                     │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  1. 连接 Master，发送 PSYNC                                │   │
│  │       ↓                                                  │   │
│  │  2. 收到 +FULLRESYNC，清空本地数据                         │   │
│  │       ↓                                                  │   │
│  │  3. 接收 RDB 文件，加载到内存                              │   │
│  │       ↓                                                  │   │
│  │  4. 接收并执行积压缓冲区命令                                │   │
│  │       ↓                                                  │   │
│  │  5. 进入命令传播阶段，持续同步                              │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  开销：                                                           │
│  • CPU：fork 子进程，bgsave 消耗 CPU                            │
│  • 内存：COW（写时复制）期间内存占用翻倍                          │
│  • 网络：传输整个 RDB 文件                                        │
│  • 时间：RDB 生成 + 传输 + 加载，从节点在此期间不可用              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**全量复制的开销问题：**

```bash
# 问题场景：频繁的全量复制
# 1. 主节点内存 10GB，RDB 文件 5GB
# 2. 10 个从节点同时断开重连
# 3. 主节点需要 fork 10 次，内存压力巨大
# 4. 网络带宽被占满，影响正常请求

# 解决方案：无磁盘复制（Redis 2.8.18+）
repl-diskless-sync yes
# 主节点直接将 RDB 通过网络发送给从节点，不写入磁盘
```

### 3. 增量复制（Partial Resynchronization）

```bash
# 触发条件
1. 主从连接断开，在 repl-backlog-size 时间内重连
2. 从节点记录的复制偏移量还在主节点的积压缓冲区内

# 核心机制：复制积压缓冲区（Replication Backlog）
```

```
┌─────────────────────────────────────────────────────────────────┐
│                  复制积压缓冲区（Backlog）                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  固定长度的 FIFO 队列（默认 1MB，可配置）                          │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  偏移量: 1000    1001    1002    1003    ...    1100      │   │
│  │         ↓       ↓       ↓       ↓              ↓         │   │
│  │  命令:  SET     GET     DEL     HSET   ...    SET        │   │
│  │  1000   k1      k2      k3      k4          k101         │   │
│  │                                                          │   │
│  │  master_repl_offset = 1100 (当前最新偏移)                  │   │
│  │  backlog 大小 = 100 (存储 1000~1100 的命令)                │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  Slave 重连场景：                                                 │
│                                                                  │
│  场景 A：offset = 1050 在缓冲区内 ✅                              │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  backlog: [1000 ... 1050 ... 1100]                       │   │
│  │                   ↑                                      │   │
│  │               slave_offset                               │   │
│  │  只需发送 1051~1100 的命令即可                              │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  场景 B：offset = 900 不在缓冲区内 ❌                              │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  backlog: [1000 ... 1100]                                │   │
│  │   ↑                                                      │   │
│  │  slave_offset (已过期)                                    │   │
│  │  必须进行全量复制！                                         │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

```bash
# 配置复制积压缓冲区大小
# redis.conf
repl-backlog-size 64mb    # 根据写入量调整
repl-backlog-ttl 3600     # 没有从节点连接时，保留缓冲区的秒数

# 估算公式：
# backlog_size = 每秒写入量 * 允许断开时间 * 2
# 例如：10MB/s 写入，允许断开 60 秒
# backlog_size = 10 * 60 * 2 = 1200MB
```

### 4. PSYNC 命令详解

```bash
# Redis 2.8+ 使用 PSYNC 替代 SYNC

# 格式
PSYNC <runid> <offset>

# 参数
# runid: 主节点运行 ID，首次连接时为 ?
# offset: 从节点已复制的偏移量，首次连接时为 -1

# 响应
# +FULLRESYNC <runid> <offset>  # 全量复制
# +CONTINUE                     # 增量复制
# -ERR                          # 错误
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    PSYNC 执行流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Slave 发送: PSYNC ? -1                                         │
│       ↓                                                          │
│  Master 判断:                                                    │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  1. runid = ?  → 首次连接 → 全量复制                      │    │
│  │                                                         │    │
│  │  2. runid ≠ master.runid → 主节点重启 → 全量复制          │    │
│  │                                                         │    │
│  │  3. offset < backlog[0] → 偏移过期 → 全量复制             │    │
│  │                                                         │    │
│  │  4. 其他情况 → 偏移在缓冲区内 → 增量复制                   │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 3：主从复制的常见问题

### 问题 1：复制风暴

```
场景：主节点重启，所有从节点同时发起全量复制
后果：主节点 CPU/内存/网络压力剧增

解决方案：
1. 树状复制结构（主 → 从 → 从）
2. 调整 repl-diskless-sync-delay，错峰复制
3. 使用 Redis Cluster 分散压力
```

### 问题 2：主从延迟

```bash
# 监控延迟
redis-cli> INFO replication | grep offset
# 比较 master_repl_offset 和 slave 的 offset

# 解决方案：
# 1. 避免大 Key 操作（大 Hash、大 List）
# 2. 增加从节点带宽
# 3. 使用读写分离时，强制走主节点查询
redis-cli> GET key  # 可能读到旧数据
redis-cli> GET key  # 延迟几秒后读到新数据

# 业务层处理：延迟敏感操作直接读主节点
```

### 问题 3：复制中断

```bash
# 查看复制中断原因
redis-cli> INFO replication
# master_link_status:down
# master_last_io_seconds_ago:123

# 常见原因：
# 1. 网络问题
# 2. 主节点内存不足，kill 复制连接
# 3. 复制缓冲区溢出

# 解决方案：
# 1. 增加 repl-timeout
# 2. 增大 repl-backlog-size
# 3. 使用专线或 VPN
```

## 🔥 扩展追问 4：Redis 4.0+ 的 PSYNC2

```bash
# Redis 4.0 引入 PSYNC2，解决切换主节点后的部分复制问题

# 场景：
# 原主节点 A，从节点 B
# A 故障，B 提升为主节点
# A 恢复后作为 B 的从节点

# Redis 2.8：
# A 的 runid 改变，B 无法识别，必须全量复制

# Redis 4.0+：
# B 保存了 A 的 replid2 和 second_repl_offset
# A 重连时提供原 replid，B 可以部分复制
```

## 一句话收尾

Redis 主从复制通过 **PSYNC 命令** 协调，**全量复制**传输 RDB 文件适合首次连接，**增量复制**从积压缓冲区续传命令适合断线重连；合理配置 `repl-backlog-size` 避免频繁全量复制是运维关键。
