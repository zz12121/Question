## 先说结论

线程池 7 个核心参数：**corePoolSize**（核心线程数）、**maximumPoolSize**（最大线程数）、**keepAliveTime**（空闲存活时间）、**workQueue**（任务队列）、**threadFactory**（线程工厂）、**handler**（拒绝策略）、**unit**（时间单位）。合理配置需根据任务类型（CPU 密集/IO 密集/混合）选择线程数，队列要有界防 OOM，拒绝策略根据业务选择（CallerRunsPolicy 适合降级，DiscardPolicy 适合非关键任务）。

## 🔥 核心概念

**解决什么问题：**
- 减少线程创建销毁开销
- 控制并发线程数量
- 提高资源利用率和响应速度

**设计思想：**
- 池化复用：线程复用，减少创建成本
- 流量削峰：队列缓冲突发流量

## 🔥 详细展开

### 1. 7 个核心参数详解

```java
public ThreadPoolExecutor(
    int corePoolSize,           // 1. 核心线程数
    int maximumPoolSize,        // 2. 最大线程数
    long keepAliveTime,         // 3. 空闲线程存活时间
    TimeUnit unit,              // 4. 时间单位
    BlockingQueue<Runnable> workQueue,  // 5. 任务队列
    ThreadFactory threadFactory,        // 6. 线程工厂
    RejectedExecutionHandler handler    // 7. 拒绝策略
)
```

| 参数 | 作用 | 注意事项 |
|------|------|----------|
| **corePoolSize** | 常驻线程数，即使空闲也保留 | 除非设置 `allowCoreThreadTimeOut` |
| **maximumPoolSize** | 最大线程数，队列满后扩容上限 | 包含核心线程数 |
| **keepAliveTime** | 非核心线程空闲存活时间 | 核心线程默认永久存活 |
| **unit** | 时间单位（秒、毫秒等） | `TimeUnit.SECONDS` 等 |
| **workQueue** | 存放待执行任务的队列 | **必须有界**，防 OOM |
| **threadFactory** | 创建线程的工厂 | 可设置线程名、守护状态、异常处理器 |
| **handler** | 队列满且线程数达最大时的处理策略 | 4 种内置 + 自定义 |

### 2. 线程池执行流程

```
提交任务
    ↓
核心线程数 < corePoolSize？
    ├── 是 → 创建新线程执行任务
    └── 否 → 任务入队列
                ↓
                队列已满？
                    ├── 否 → 等待执行
                    └── 是 → 线程数 < maximumPoolSize？
                                    ├── 是 → 创建非核心线程执行任务
                                    └── 否 → 执行拒绝策略
```

**关键理解：**
- 先扩容到核心线程数
- 然后任务入队列（不是继续创建线程！）
- 队列满后才扩容到最大线程数
- 最后才拒绝

### 3. 任务队列选择

| 队列类型 | 特点 | 适用场景 |
|----------|------|----------|
| **ArrayBlockingQueue** | 有界数组，FIFO | 固定容量，内存可控 |
| **LinkedBlockingQueue** | 无界链表（默认 `Integer.MAX_VALUE`） | ⚠️ 慎用，易 OOM |
| **SynchronousQueue** | 不存储元素，直接移交 | 高吞吐，线程数可动态伸缩 |
| **PriorityBlockingQueue** | 优先级排序 | 任务有优先级 |
| **DelayQueue** | 延迟执行 | 定时任务 |

**生产环境建议：**

```java
// ❌ 危险：无界队列，内存可能耗尽
new LinkedBlockingQueue<>()

// ✅ 安全：有界队列
new ArrayBlockingQueue<>(1000)
new LinkedBlockingQueue<>(1000)  // 指定容量
```

### 4. 拒绝策略

| 策略 | 行为 | 适用场景 |
|------|------|----------|
| **AbortPolicy**（默认） | 抛 `RejectedExecutionException` | 需要感知拒绝 |
| **CallerRunsPolicy** | 由调用线程执行任务 | 降级处理，降低提交速度 |
| **DiscardPolicy** | 静默丢弃任务 | 非关键任务 |
| **DiscardOldestPolicy** | 丢弃最老任务，重试提交 | 新任务更重要 |

**自定义拒绝策略：**

```java
new RejectedExecutionHandler() {
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        // 记录日志
        log.error("Task rejected: {}", r);
        
        // 保存到数据库或 MQ，后续补偿
        saveToDeadLetterQueue(r);
        
        // 或抛出自定义异常
        throw new BizException("系统繁忙，请稍后重试");
    }
}
```

### 6. 合理配置线程池

#### 6.1 线程数计算公式

**CPU 密集型任务**（计算、数据处理）：

```java
// 线程数 = CPU 核心数 + 1
int poolSize = Runtime.getRuntime().availableProcessors() + 1;
```

**IO 密集型任务**（网络、磁盘）：

```java
// 线程数 = CPU 核心数 × (1 + 等待时间 / 计算时间)
// 粗略估算：CPU 核心数 × 2 或 × 4
int poolSize = Runtime.getRuntime().availableProcessors() * 2;

// 更精确：根据压测调整
```

**混合任务：**

```java
// 拆分成两个线程池，避免互相影响
ExecutorService cpuPool = Executors.newFixedThreadPool(cpuCore + 1);
ExecutorService ioPool = Executors.newFixedThreadPool(cpuCore * 4);
```

#### 6.2 配置示例

**CPU 密集型（订单计算）：**

```java
new ThreadPoolExecutor(
    4,                      // 4 核 CPU
    4,                      // 最大也是 4（不需要更多）
    0L,                     // 无意义，不扩容
    TimeUnit.MILLISECONDS,
    new ArrayBlockingQueue<>(100),  // 有界队列
    new CustomThreadFactory("compute"),
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

**IO 密集型（HTTP 调用）：**

```java
new ThreadPoolExecutor(
    8,                      // 核心 8 个
    32,                     // 最大 32 个
    60L,                    // 空闲 60 秒回收
    TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(1000),
    new CustomThreadFactory("http"),
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

**异步通知（低延迟）：**

```java
new ThreadPoolExecutor(
    0,                      // 核心 0，任务来再创建
    Integer.MAX_VALUE,      // 理论上无限
    60L,
    TimeUnit.SECONDS,
    new SynchronousQueue<>(),  // 直接移交，不排队
    new CustomThreadFactory("notify"),
    new ThreadPoolExecutor.DiscardPolicy()  // 非关键，可丢弃
);
```

## 🔥 扩展追问 1：Executors 工厂方法的陷阱

```java
// ❌ 禁止使用！无界队列，易 OOM
Executors.newFixedThreadPool(10);
// 等价于：new ThreadPoolExecutor(10, 10, 0L, LinkedBlockingQueue<Runnable>())

// ❌ 禁止使用！允许创建无限线程
Executors.newCachedThreadPool();
// 等价于：new ThreadPoolExecutor(0, Integer.MAX_VALUE, 60L, SynchronousQueue<Runnable>())

// ❌ 禁止使用！单线程 + 无界队列
Executors.newSingleThreadExecutor();
// 等价于：new ThreadPoolExecutor(1, 1, 0L, LinkedBlockingQueue<Runnable>())

// ❌ 禁止使用！DelayedWorkQueue 虽无界但可控，但仍有风险
Executors.newScheduledThreadPool(10);
```

**为什么阿里巴巴 Java 开发手册禁止使用？**

| 方法 | 问题 | 后果 |
|------|------|------|
| `newFixedThreadPool` | `LinkedBlockingQueue` 无界 | 任务堆积，OOM |
| `newCachedThreadPool` | `maxPoolSize = Integer.MAX_VALUE` | 线程数爆炸，CPU 耗尽 |
| `newSingleThreadExecutor` | 无界队列 + 单线程 | 任务堆积，OOM |

**正确做法：手动创建，指定有界队列**

```java
// ✅ 推荐
new ThreadPoolExecutor(
    10, 10, 0L, TimeUnit.MILLISECONDS,
    new ArrayBlockingQueue<>(100),  // 有界！
    new ThreadFactoryBuilder().setNameFormat("pool-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

## 🔥 扩展追问 2：Tomcat/Jetty 的线程池为什么不一样？

```java
// Tomcat 线程池（扩展了 ThreadPoolExecutor）
public class ThreadPoolExecutor extends java.util.concurrent.ThreadPoolExecutor {
    
    // 关键区别：核心线程数满了后，优先创建线程（而不是先入队）
    @Override
    public void execute(Runnable command) {
        if (executor != null) {
            try {
                // 尝试直接提交（创建新线程）
                executor.execute(command);
            } catch (RejectedExecutionException rx) {
                // 满了才入队
                if (!((TaskQueue) getQueue()).force(command)) {
                    throw new RejectedExecutionException("...");
                }
            }
        }
    }
}

// Tomcat 配置
server:
  tomcat:
    threads:
      max: 200          # 最大线程数
      min-spare: 10     # 核心线程数
    max-connections: 10000  # 最大连接数
    accept-count: 100   # 队列长度
```

**Tomcat 与标准线程池的区别：**

| 特性 | 标准 ThreadPoolExecutor | Tomcat ThreadPoolExecutor |
|------|------------------------|---------------------------|
| 核心线程满后 | 先入队 | 先创建线程到最大 |
| 适用场景 | 后台任务 | Web 请求（低延迟优先） |

## 🔥 扩展追问 3：ForkJoinPool 是什么？与普通线程池的区别？

```java
// Fork/Join 框架（Java 7+）
// 适用于：分治算法、递归任务、工作窃取

ForkJoinPool forkJoinPool = new ForkJoinPool(4);

// 提交 RecursiveTask（有返回值）
int result = forkJoinPool.invoke(new FibonacciTask(10));

// RecursiveAction（无返回值）
forkJoinPool.invoke(new SortTask(array));

// 递归任务示例
class FibonacciTask extends RecursiveTask<Integer> {
    final int n;
    
    FibonacciTask(int n) { this.n = n; }
    
    @Override
    protected Integer compute() {
        if (n <= 1) return n;
        
        // 拆分任务
        FibonacciTask f1 = new FibonacciTask(n - 1);
        FibonacciTask f2 = new FibonacciTask(n - 2);
        
        f1.fork();  // 异步执行
        f2.fork();
        
        return f1.join() + f2.join();  // 等待结果
    }
}
```

**与普通线程池的区别：**

| 特性 | ThreadPoolExecutor | ForkJoinPool |
|------|-------------------|--------------|
| 任务类型 | Runnable/Callable | RecursiveTask/RecursiveAction |
| 工作模式 | 提交-执行 | 分治-递归 |
| 线程空闲 | 等待任务 | 窃取其他线程任务 |
| 适用场景 | 独立任务 | 可拆分的大任务 |

**工作窃取（Work Stealing）：**

```
线程 1 的队列：[大任务A] → [子任务A1] → [子任务A2]
线程 2 的队列：[大任务B] → [子任务B1]

线程 2 执行完自己的任务后：
    ↓
从线程 1 队列尾部窃取 [子任务A2]
    ↓
减少线程空闲，提高利用率
```

**并行流底层：**

```java
List<Integer> numbers = Arrays.asList(1, 2, 3, 4, 5);

// 底层使用 ForkJoinPool.commonPool()
numbers.parallelStream()
       .map(n -> n * n)
       .collect(Collectors.toList());

// commonPool 线程数 = CPU 核心数 - 1
```

## 一句话收尾

线程池 7 参数要熟记，队列必须有界防 OOM，CPU 密集配核数、IO 密集配 2-4 倍核数；Executors 工厂方法有陷阱（无界队列/无限线程），生产环境必须手动创建；Tomcat 线程池优先扩容而非入队，ForkJoinPool 用工作窃取适合分治任务！
