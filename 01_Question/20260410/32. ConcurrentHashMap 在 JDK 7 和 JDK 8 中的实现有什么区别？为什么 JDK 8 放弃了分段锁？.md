## 先说结论

JDK 7 使用 **Segment 分段锁**（默认 16 段，每段独立加锁），细粒度控制并发；JDK 8 改为 **CAS + synchronized + 红黑树**，锁粒度细化到链表头节点，并发度更高。放弃分段锁的原因是：分段锁粒度仍较粗、扩容复杂、内存占用大，而 CAS 操作在 JDK 8 性能大幅提升，配合 synchronized 优化（偏向锁/轻量级锁）能实现更好的并发性能。

**解决什么问题：**
- HashMap 非线程安全，并发操作导致数据丢失或死循环

**核心思想演进：**
- JDK 7：分段锁，减少锁竞争范围
- JDK 8：无锁读 + 细粒度锁写，最大化并行度

### 2. JDK 8：CAS + synchronized + 红黑树

```java
// ConcurrentHashMap 结构（JDK 8）
public class ConcurrentHashMap<K,V> {
    // 整个哈希表
    transient volatile Node<K,V>[] table;
    
    // 基础节点
    static class Node<K,V> implements Map.Entry<K,V> {
        final int hash;
        final K key;
        volatile V val;           // volatile 保证可见性
        volatile Node<K,V> next;  // 链表下一个节点
    }
    
    // 红黑树节点
    static final class TreeNode<K,V> extends Node<K,V> {
        TreeNode<K,V> parent;
        TreeNode<K,V> left;
        TreeNode<K,V> right;
        TreeNode<K,V> prev;
        boolean red;
    }
    
    // 转发节点（扩容时使用）
    static final class ForwardingNode<K,V> extends Node<K,V> {
        final Node<K,V>[] nextTable;
    }
}
```

**结构图解：**

```
ConcurrentHashMap
└── Node[] table ──┬──→ 链表 Node → Node → Node
                   ├──→ 链表 Node → Node
                   ├──→ TreeBin（红黑树根节点）
                   └──→ null
                   
锁粒度：每个链表/树根节点（synchronized 锁住头节点）
```

**put 操作（JDK 8）：**

```java
final V putVal(K key, V value, boolean onlyIfAbsent) {
    if (key == null || value == null) throw new NullPointerException();
    int hash = spread(key.hashCode());
    int binCount = 0;
    
    for (Node<K,V>[] tab = table;;) {
        Node<K,V> f; int n, i, fh;
        
        // 1. 表未初始化，先初始化
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();
        
        // 2. 目标桶为空，CAS 直接插入（无锁！）
        else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            if (casTabAt(tab, i, null, new Node<K,V>(hash, key, value, null)))
                break;
        }
        
        // 3. 发现转发节点，说明正在扩容，帮助扩容
        else if ((fh = f.hash) == MOVED)
            tab = helpTransfer(tab, f);
        
        // 4. 桶非空，synchronized 锁住头节点
        else {
            V oldVal = null;
            synchronized (f) {  // 只锁当前桶！
                if (tabAt(tab, i) == f) {
                    if (fh >= 0) {  // 链表
                        binCount = 1;
                        for (Node<K,V> e = f;; ++binCount) {
                            K ek;
                            if (e.hash == hash &&
                                ((ek = e.key) == key || (ek != null && key.equals(ek)))) {
                                oldVal = e.val;
                                if (!onlyIfAbsent)
                                    e.val = value;
                                break;
                            }
                            Node<K,V> pred = e;
                            if ((e = e.next) == null) {
                                pred.next = new Node<K,V>(hash, key, value, null);
                                break;
                            }
                        }
                    }
                    else if (f instanceof TreeBin) {  // 红黑树
                        Node<K,V> p;
                        binCount = 2;
                        if ((p = ((TreeBin<K,V>)f).putTreeVal(hash, key, value)) != null) {
                            oldVal = p.val;
                            if (!onlyIfAbsent)
                                p.val = value;
                        }
                    }
                }
            }
            
            // 链表长度 >= 8，转红黑树
            if (binCount != 0) {
                if (binCount >= TREEIFY_THRESHOLD)
                    treeifyBin(tab, i);
                if (oldVal != null)
                    return oldVal;
                break;
            }
        }
    }
    addCount(1L, binCount);
    return null;
}
```

### 3. 核心区别对比

| 维度 | JDK 7 | JDK 8 |
|------|-------|-------|
| **数据结构** | Segment[] + HashEntry[] | Node[] + 链表/红黑树 |
| **锁机制** | ReentrantLock（Segment 级别） | synchronized（桶级别）+ CAS |
| **锁粒度** | 16 个 Segment（默认） | 每个桶的头节点 |
| **并发度** | 默认 16 | 理论无上限（桶数量） |
| **读操作** | 无锁（volatile） | 无锁（volatile） |
| **写操作** | 锁 Segment | 锁单个桶 |
| **扩容** | Segment 独立扩容 | 多线程协作扩容 |
| **哈希冲突** | 链表 | 链表 → 红黑树（>=8） |

### 4. 为什么放弃分段锁？

#### 原因 1：锁粒度仍较粗

```java
// JDK 7：16 个 Segment，高并发下仍有竞争
// 1000 个线程访问同一个 Segment，只能串行

// JDK 8：1000 个线程访问不同桶，完全并行
// 锁粒度 = 桶数量（默认 16，扩容后更多）
```

#### 原因 2：扩容复杂

```java
// JDK 7：每个 Segment 独立扩容
// 问题：Segment 内数组大小不一致，难以全局优化

// JDK 8：整个表一起扩容，多线程协作
// 每个线程负责一部分桶的迁移，效率更高
```

#### 原因 3：内存占用大

```java
// JDK 7：每个 Segment 是一个独立对象
// Segment 数量 × Segment 对象头 = 额外内存开销

// JDK 8：直接用 Node 数组，结构更紧凑
// 去掉 Segment 中间层，减少对象数量
```

#### 原因 4：JDK 8 synchronized 优化

```
JDK 6 之前：synchronized 是重量级锁，性能差
JDK 6 之后：引入锁升级机制

无锁 → 偏向锁 → 轻量级锁（CAS）→ 重量级锁

ConcurrentHashMap 场景：
- 大部分桶无竞争：无锁或偏向锁
- 少量竞争：轻量级锁（CAS）
- 激烈竞争：才升级为重量级锁

实际测试：JDK 8 synchronized 在大多数场景下性能优于 ReentrantLock
```

#### 原因 5：红黑树优化极端情况

```java
// JDK 7：链表过长时，查询退化为 O(n)

// JDK 8：链表长度 >= 8 转红黑树，查询 O(log n)
// 树化阈值 8，退化阈值 6（防止频繁转换）
```

private final void transfer(Node<K,V>[] tab, Node<K,V>[] nextTab) {
    int n = tab.length, stride;
    
    // 计算每个线程负责的桶数量
    if ((stride = (NCPU > 1) ? (n >>> 3) / NCPU : n) < MIN_TRANSFER_STRIDE)
        stride = MIN_TRANSFER_STRIDE;
    
    // 初始化新表
    if (nextTab == null) {
        try {
            @SuppressWarnings("unchecked")
            Node<K,V>[] nt = (Node<K,V>[])new Node<?,?>[n << 1];
            nextTab = nt;
        } catch (Throwable ex) {
            sizeCtl = Integer.MAX_VALUE;
            return;
        }
        nextTable = nextTab;
        transferIndex = n;
    }
    
    int nextn = nextTab.length;
    ForwardingNode<K,V> fwd = new ForwardingNode<K,V>(nextTab);
    
    boolean advance = true;
    boolean finishing = false;
    
    // 自旋迁移桶
    for (int i = 0, bound = 0;;) {
        Node<K,V> f; int fh;
        
        // 分配下一个要迁移的桶
        while (advance) {
            int nextIndex, nextBound;
            if (--i >= bound || finishing)
                advance = false;
            else if ((nextIndex = transferIndex) <= 0) {
                i = -1;
                advance = false;
            }
            else if (U.compareAndSwapInt(this, TRANSFERINDEX, nextIndex,
                      nextBound = (nextIndex > stride ? nextIndex - stride : 0))) {
                bound = nextBound;
                i = nextIndex - 1;
                advance = false;
            }
        }
        
        // 迁移完成
        if (i < 0 || i >= n || i + n >= nextn) {
            if (finishing) {
                nextTable = null;
                table = nextTab;
                sizeCtl = (n << 1) - (n >>> 1);  // 新阈值 0.75n
                return;
            }
            // ...
        }
        
        // 迁移具体桶...
    }
}

## 🔥 扩展追问 1：size() 方法怎么保证准确性？

```java
// JDK 7：锁定所有 Segment 后统计
public int size() {
    final Segment<K,V>[] segments = this.segments;
    long sum = 0;
    long check = 0;
    int[] mc = new int[segments.length];
    
    // 重试 2 次，避免锁定
    for (int k = 0; k < RETRIES_BEFORE_LOCK; ++k) {
        check = 0;
        for (int i = 0; i < segments.length; ++i) {
            sum += segments[i].count;
            mc[i] = segments[i].modCount;
        }
        for (int i = 0; i < segments.length; ++i)
            check += mc[i];
        if (check == sum)
            return (int)sum;
    }
    
    // 重试失败，全表锁定
    lockAllSegments();
    try {
        return (int)sum;
    } finally {
        unlockAllSegments();
    }
}
```

## 🔥 扩展追问 3：为什么 JDK 8 用 synchronized 而不是 ReentrantLock？

| 对比项 | synchronized | ReentrantLock |
|--------|--------------|---------------|
| 实现 | JVM 内置 | AQS 框架 |
| 锁升级 | 支持（无锁→偏向→轻量→重量） | 不支持，直接 AQS |
| 条件队列 | 一个（wait/notify） | 多个（Condition） |
| 中断响应 | 不支持 | 支持 |
| 公平性 | 非公平 | 可选公平/非公平 |
| 性能（JDK 8+） | 更优（JVM 优化好） | 略逊 |

**ConcurrentHashMap 场景：**
- 不需要多个 Condition
- 不需要中断响应
- 不需要公平锁
- synchronized 的锁升级机制更适合（大部分桶无竞争）

## 一句话收尾

JDK 7 分段锁（16 把锁）→ JDK 8 CAS + 细粒度 synchronized（每个桶一把锁），放弃分段锁是因为锁粒度仍粗、扩容复杂、内存占用大，而 JDK 8 的 synchronized 优化后性能更好，配合红黑树解决了链表过长问题！
