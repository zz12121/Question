## 先说结论

Redis Hash 底层使用 **ziplist（压缩列表）** 和 **hashtable（哈希表）** 两种编码；当 **field-value 数量少且单个元素短** 时用 ziplist 节省内存，当 **数据量大或元素长** 时自动转为 hashtable 保证 O(1) 访问性能。

#### entry 结构详解

```c
// entry 存储单个 field 或 value
// 采用变长编码，根据内容大小选择不同存储方式

// entry 结构
<prevlen> <encoding> <content>
  变长      变长      变长

// prevlen：前一个 entry 的长度，方便反向遍历
//   - 如果 < 254 字节，用 1 字节存储
//   - 如果 >= 254 字节，用 5 字节存储（第1字节为0xFE）

// encoding：编码方式，同时表示 content 长度
//   - |00xxxxxx|：6 位表示长度，最大 63 字节
//   - |01xxxxxx| xxxxxxxx：14 位表示长度，最大 16383 字节
//   - |10000000| xxxxxxxx xxxxxxxx xxxxxxxx：32 位表示长度
//   - |11000000|：int16_t 类型
//   - |11010000|：int32_t 类型
//   - |11100000|：int64_t 类型
//   - |11110000|：24 位有符号整数
//   - |11111110|：8 位有符号整数
```

### 2. hashtable（哈希表）结构

```c
// Redis 使用 dict（字典）实现 hashtable
// 采用渐进式 rehash，避免一次性迁移导致阻塞

typedef struct dict {
    dictType *type;      // 类型特定函数
    void *privdata;      // 私有数据
    dictht ht[2];        // 两个哈希表，用于渐进式 rehash
    long rehashidx;      // rehash 进度，-1 表示未在进行
    int16_t pauserehash; // 安全迭代器计数
} dict;

typedef struct dictht {
    dictEntry **table;   // 哈希表数组
    unsigned long size;  // 哈希表大小
    unsigned long sizemask;  // 掩码，用于计算索引
    unsigned long used;  // 已有节点数量
} dictht;

typedef struct dictEntry {
    void *key;           // field（SDS）
    union {
        void *val;       // value（SDS）
        uint64_t u64;
        int64_t s64;
        double d;
    } v;
    struct dictEntry *next;  // 拉链法解决冲突
} dictEntry;
```

```
┌─────────────────────────────────────────────────────────────────┐

│  ┌──────────────────────────────────────────────────────────┐   │
│  │  ht[0] ─────────→  dictht (主哈希表)                      │   │
│  │  ht[1] ─────────→  dictht (rehash 时用)                   │   │
```

```
│       ↓                                                          │
```

## 🔥 扩展追问 1：ziplist 的优缺点

| 特性 | ziplist | hashtable |
|------|---------|-----------|
| **内存占用** | 极低（紧凑存储） | 较高（指针开销） |
| **访问速度** | O(n) 顺序查找 | O(1) 哈希查找 |
| **插入/删除** | 可能需要整体内存移动 | 只需操作链表 |
| **适用场景** | 少量小数据 | 大量数据 |
| **CPU 缓存** | 友好（连续内存） | 一般（指针跳转） |

**ziplist 的连锁更新问题：**

```c
// ziplist 的 prevlen 用变长编码
// 当 entry 长度变化时，可能导致连锁更新

// 场景：entry1 长度从 253 字节变为 255 字节
// entry1.prevlen 从 1 字节变为 5 字节
// entry1 整体长度增加 4 字节
// entry2.prevlen 需要更新...

// 最坏情况：O(n) 的更新复杂度
// 但概率极低，实际影响有限
```

## 🔥 扩展追问 2：渐进式 rehash

```c
// 当 hashtable 负载因子过高时，需要扩容并 rehash
// Redis 采用渐进式 rehash，避免阻塞

// rehash 过程：
// 1. 为 ht[1] 分配空间（ht[0].size * 2）
// 2. 将 rehashidx 设为 0，表示开始 rehash
// 3. 每次对 dict 进行增删改查时，顺带迁移 ht[0][rehashidx] 的桶
// 4. 所有桶迁移完成后，ht[1] 变为 ht[0]，释放原 ht[0]

// 查询逻辑：
dictEntry *dictFind(dict *d, const void *key) {
    // 如果在 rehash，先尝试在 ht[0] 找，找不到再去 ht[1]
    if (dictIsRehashing(d)) {
        // 顺带迁移一个桶
        _dictRehashStep(d);
    }
    // 先查 ht[0]，再查 ht[1]
    // ...
}
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    渐进式 rehash 过程                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  初始状态：                                                       │
│  ht[0]: size=4, used=4  (负载因子=1，需要扩容)                    │
│  ht[1]: null                                                     │
│                                                                  │
│  开始 rehash：                                                    │
│  ht[0]: size=4, used=4                                           │
│  ht[1]: size=8, used=0  (新分配，2倍大小)                         │
│  rehashidx = 0                                                   │
│                                                                  │
│  每次操作顺带迁移：                                                │
│  GET/SET/DELETE 时：                                              │
│  1. 迁移 ht[0].table[rehashidx] 的所有 entry 到 ht[1]             │
│  2. rehashidx++                                                  │
│                                                                  │
│  完成 rehash：                                                    │
│  rehashidx = 4 (等于 ht[0].size)                                  │
│  ht[0] ← ht[1]  (交换指针)                                        │
│  ht[1] ← null                                                    │
│  rehashidx = -1                                                  │
│                                                                  │
│  优势：将一次性 O(n) 操作分散到多次 O(1) 操作，避免阻塞             │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 一句话收尾

Redis Hash 底层用 **ziplist** 存储少量小数据（省内存）和 **hashtable** 存储大量数据（O(1)访问），当 **field-value 对超过 512 或单个元素超过 64 字节** 时自动从 ziplist 转为 hashtable，且转换是单向不可逆的。
