## 先说结论

分布式事务是指**跨多个服务/数据库的事务操作**，需要保证 ACID 特性。常见方案：

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|---------|
| **2PC** | 强一致 | 低 | 中 | 传统数据库XA，很少用 |
| **3PC** | 强一致 | 低 | 高 | 理论改进，实际很少用 |
| **TCC** | 最终一致 | 高 | 高 | 金融支付（Seata）|
| **Saga** | 最终一致 | 高 | 中 | 长事务流程（Seata/Axon）|
| **本地消息表** | 最终一致 | 高 | 中 | 跨服务异步通知 |

**选型建议**：短事务用 TCC，长流程用 Saga，简单异步用本地消息表，2PC/3PC 谨慎使用。

## 🔥 详细展开：五种分布式事务方案

### 1. 2PC（Two-Phase Commit）两阶段提交

```
┌─────────────────────────────────────────────────────────┐
│                    2PC 执行流程                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   协调者（Coordinator）        参与者（Participants）      │
│   ┌─────────┐                  ┌─────────┐              │
│   │  事务管理 │                  │  各服务  │              │
│   │  器 TM  │                  │  资源管理 │              │
│   └────┬────┘                  │  器 RM  │              │
│        │                       └────┬────┘              │
│        │                            │                   │
│        │────── Phase 1: 准备阶段 ───→│                   │
│        │   发送 Prepare 请求         │                   │
│        │                            │                   │
│        │←──── 执行本地事务，写 Undo/Redo ──               │
│        │    返回 Yes/No              │                   │
│        │                            │                   │
│        │  [所有返回 Yes？]           │                   │
│        │      ├── Yes → 继续        │                   │
│        │      └── No  → 回滚        │                   │
│        │                            │                   │
│        │────── Phase 2: 提交阶段 ───→│                   │
│        │   发送 Commit/Rollback      │                   │
│        │                            │                   │
│        │←──── 执行提交/回滚 ──────────│                   │
│        │    返回 ACK                 │                   │
│        │                            │                   │
│        │──── 事务结束 ─────────────→│                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**优缺点：**
- ✅ 强一致性
- ❌ 同步阻塞，性能差
- ❌ 单点故障（协调者挂了）
- ❌ 数据不一致风险（第二阶段部分参与者没收到 Commit）

**Java 实现：**
```java

        return new UserTransactionManager();
    }
    
    @Bean

        return new JtaTransactionManager(userTransactionManager(), userTransactionManager());
    }

// 使用
@Transactional
public void createOrder() {
    inventoryService.deduct();  // DB1
    orderService.create();      // DB2
    accountService.debit();     // DB3
    // 任一失败，全部回滚
}
```

### 2. 3PC（Three-Phase Commit）三阶段提交

```
┌─────────────────────────────────────────────────────────┐
│                    3PC 执行流程                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   Phase 1: CanCommit（询问阶段）                          │
│   ─────────────────────────────                         │
│   协调者询问参与者是否可以执行事务                          │
│   参与者检查自身状态（不执行事务）                          │
│   目的：提前发现无法执行的情况，减少阻塞                    │
│                                                         │
│   Phase 2: PreCommit（预提交阶段）                        │
│   ───────────────────────────────                       │
│   参与者预执行事务，写 Undo/Redo                          │
│   参与者锁定资源                                          │
│                                                         │
│   Phase 3: DoCommit（提交阶段）                           │
│   ────────────────────────────                          │
│   协调者发送 Commit 或 Abort                              │
│   参与者正式提交或回滚                                     │
│                                                         │
│   改进点：                                                │
│   - 引入超时机制（协调者和参与者都有超时处理）               │
│   - CanCommit 提前过滤，减少资源锁定时间                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**2PC vs 3PC：**

| 特性 | 2PC | 3PC |
|------|-----|-----|
| 阶段数 | 2 | 3 |
| 阻塞 | 严重 | 较轻 |
| 超时处理 | 无 | 有 |
| 复杂度 | 中 | 高 |
| 实际使用 | 极少 | 几乎不用 |

> 3PC 理论上解决了 2PC 的阻塞问题，但实现复杂，网络分区时仍有数据不一致风险，实际很少使用。

### 3. TCC（Try-Confirm-Cancel）

```
┌─────────────────────────────────────────────────────────┐
│                    TCC 执行流程                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   业务层实现三个接口：                                     │
│                                                         │
│   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐ │
│   │    Try      │ →  │  Confirm    │    │   Cancel    │ │
│   │   预留资源   │    │   确认提交   │    │   取消回滚   │ │
│   └─────────────┘    └─────────────┘    └─────────────┘ │
│        │                  ↑                  ↑          │
│        │                  │                  │          │
│        └──────────────────┴──────────────────┘          │
│                    根据执行结果选择                       │
│                                                         │
│   示例：扣减库存                                          │
│   ─────────────────                                     │
│   Try：检查库存，冻结待扣减的库存（预留）                   │
│   Confirm：真正扣减冻结的库存                              │
│   Cancel：释放冻结的库存                                   │
│                                                         │
│   特点：                                                  │
│   - 业务侵入性强（需实现3个方法）                          │
│   - 无全局锁，性能高                                       │
│   - Confirm/Cancel 需保证幂等                             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**TCC 完整示例：**

```java
// 1. 定义 TCC 接口
@LocalTCC
public interface InventoryTccAction {
    
    @TwoPhaseBusinessAction(name = "deductInventory", 
        commitMethod = "commit", rollbackMethod = "rollback")
    boolean tryDeduct(@BusinessActionContextParameter(paramName = "skuId") Long skuId,
                      @BusinessActionContextParameter(paramName = "count") Integer count);
    
    boolean commit(BusinessActionContext context);
    
    boolean rollback(BusinessActionContext context);
}

// 2. 实现 TCC 接口
@Component
public class InventoryTccActionImpl implements InventoryTccAction {
    
    @Autowired
    private InventoryMapper inventoryMapper;
    
    @Autowired
    private InventoryFreezeMapper freezeMapper;
    
    // Try：预留资源
    @Override
    public boolean tryDeduct(Long skuId, Integer count) {
        // 1. 检查库存
        Integer stock = inventoryMapper.getStock(skuId);
        if (stock < count) {
            return false;  // 库存不足
        }
        
        // 2. 冻结库存（预留）
        freezeMapper.freeze(skuId, count);
        return true;
    }
    
    // Confirm：确认提交
    @Override
    public boolean commit(BusinessActionContext context) {
        Long skuId = Long.valueOf(context.getActionContext("skuId"));
        Integer count = Integer.valueOf(context.getActionContext("count"));
        
        // 1. 真正扣减库存
        inventoryMapper.decrement(skuId, count);
        
        // 2. 释放冻结记录
        freezeMapper.delete(skuId, count);
        
        return true;
    }
    
    // Cancel：取消回滚
    @Override
    public boolean rollback(BusinessActionContext context) {
        Long skuId = Long.valueOf(context.getActionContext("skuId"));
        Integer count = Integer.valueOf(context.getActionContext("count"));
        
        // 释放冻结的库存
        freezeMapper.delete(skuId, count);
        
        return true;
    }
}

// 3. 业务使用（Seata 注解）
@Service
public class OrderService {
    
    @Autowired
    private InventoryTccAction inventoryTccAction;
    
    @Autowired
    private OrderTccAction orderTccAction;
    
    @Autowired
    private AccountTccAction accountTccAction;
    
    @GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
    public void createOrder(CreateOrderRequest request) {
        // Try 阶段：预留资源
        inventoryTccAction.tryDeduct(request.getSkuId(), request.getCount());
        orderTccAction.tryCreate(request);
        accountTccAction.tryDebit(request.getUserId(), request.getAmount());
        
        // 所有 Try 成功，Seata 自动调用 Confirm
        // 任一 Try 失败，Seata 自动调用 Cancel
    }
}
```

**TCC 优缺点：**

| 优点 | 缺点 |
|------|------|
| 性能高（无全局锁） | 业务侵入性强 |
| 最终一致性 | 开发成本高（3倍代码量） |
| 适合短事务 | Confirm/Cancel 需幂等 |
| 可异步执行 Confirm | 空回滚、悬挂问题需处理 |

### 4. Saga 模式

```
┌─────────────────────────────────────────────────────────┐
│                    Saga 执行流程                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   长事务拆分为多个本地事务，每个有对应的补偿操作               │
│                                                         │
│   正向流程（T1 → T2 → T3 → ...）                          │
│   ┌─────┐   ┌─────┐   ┌─────┐   ┌─────┐                │
│   │ T1  │ → │ T2  │ → │ T3  │ → │ T4  │                │
│   └─────┘   └─────┘   └─────┘   └─────┘                │
│                                                         │
│   任一失败，执行补偿（回滚已执行的）                         │
│   ┌─────┐   ┌─────┐   ┌─────┐                          │
│   │ T1  │ → │ T2  │ → │ T3  │  X 失败                   │
│   └──┬──┘   └──┬──┘   └──┬──┘                          │
│      │         │         │                              │
│      ↓         ↓         ↓                              │
│   ┌─────┐   ┌─────┐   ┌─────┐                          │
│   │ C1  │ ← │ C2  │ ← │ C3  │  补偿操作                  │
│   └─────┘   └─────┘   └─────┘                          │
│                                                         │
│   两种实现方式：                                          │
│   1. 编排式（Choreography）：各服务监听事件，自主决策        │
│   2. 协调式（Orchestration）：中央协调器统一调度            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Saga 示例（Seata 实现）：**

```java

            "catch": [{ "error": "Failed", "next": "CompensateAccount" }]
        },
        "CompensateAccount": {
            "type": "serviceTask",
            "serviceName": "accountServiceImpl",
            "methodName": "credit",
            "next": "CompensateOrder"
        },
        "CompensateOrder": {
            "type": "serviceTask",
            "serviceName": "orderServiceImpl",
            "methodName": "cancel",
            "next": "CompensateInventory"

@Service
public class OrderSagaService {
    
    @Autowired
    private StateMachineEngine stateMachineEngine;
    
    public void createOrder(OrderRequest request) {

            "CreateOrderSaga",  // 状态机名称
            null,               // 当前登录用户

        );
        
        if (!ExecutionStatus.SU.equals(inst.getStatus())) {
            throw new BusinessException("订单创建失败");
        }
    }
}
```

**Saga vs TCC：**

| 特性 | TCC | Saga |
|------|-----|------|
| 粒度 | 方法级 | 服务级 |
| 侵入性 | 高（3个方法） | 中（补偿方法） |
| 隔离性 | 好（预留资源） | 差（已提交可见） |
| 适用场景 | 短事务 | 长事务 |
| 回滚 | 直接回滚 | 补偿回滚 |

### 5. 本地消息表

```
┌─────────────────────────────────────────────────────────┐
│                  本地消息表方案                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   核心思想：将分布式事务转为本地事务 + 异步通知              │
│                                                         │
│   ┌─────────────┐        ┌─────────────┐               │
│   │   业务服务   │        │   消息服务   │               │
│   │  (DB1)      │        │  (DB2)      │               │
│   │             │        │             │               │
│   │  业务表      │        │  消息表      │               │
│   │  + 消息表    │        │             │               │
│   │  (同一事务)  │        │             │               │
│   └──────┬──────┘        └──────┬──────┘               │
│          │                      │                       │
│          │  1. 业务操作 + 写入消息（同一本地事务）         │
│          │                      │                       │
│          │  2. 定时扫描消息表，发送到 MQ                   │
│          └─────────────────────→│                       │
│                                 │                       │
│                                 │  3. 消费方消费消息       │
│                                 │                       │
│                                 │  4. 消费成功，删除消息   │
│                                                         │
│   消息表结构：                                            │
│   ─────────────────                                      │
│   id | topic | body | status | retry_count | create_time │
│                                                         │
│   status: 0-待发送 1-已发送 2-消费成功 3-消费失败           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**本地消息表实现：**

```java
@Component
public class LocalMessageService {
    
    @Autowired
    private MessageRepository messageRepository;
    
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    
    // 发送消息（业务方调用）
    @Transactional
    public void sendMessage(String topic, String body) {
        // 1. 保存业务数据（省略）
        
        // 2. 同一事务中保存消息
        Message message = new Message();
        message.setTopic(topic);
        message.setBody(body);
        message.setStatus(0);  // 待发送
        message.setRetryCount(0);
        messageRepository.save(message);
    }
    
    // 定时任务：扫描并发送消息
    @Scheduled(fixedRate = 5000)
    public void scanAndSend() {
        List<Message> pendingMessages = messageRepository
            .findByStatusAndCreateTimeBefore(0, LocalDateTime.now().minusMinutes(1));
        
        for (Message msg : pendingMessages) {
            try {
                kafkaTemplate.send(msg.getTopic(), msg.getBody())
                    .addCallback(
                        success -> {
                            msg.setStatus(1);  // 已发送
                            messageRepository.save(msg);
                        },
                        failure -> {
                            msg.setRetryCount(msg.getRetryCount() + 1);
                            messageRepository.save(msg);
                        }
                    );
            } catch (Exception e) {
                // 重试次数+1，超过阈值告警
                msg.setRetryCount(msg.getRetryCount() + 1);
                messageRepository.save(msg);
            }
        }
    }
}

// 消费方：幂等消费
@Component
public class OrderMessageConsumer {
    
    @Autowired
    private IdempotencyService idempotencyService;
    
    @KafkaListener(topics = "order-created")
    public void onMessage(String message, Acknowledgment ack) {
        OrderEvent event = JSON.parseObject(message, OrderEvent.class);
        
        // 幂等检查
        if (idempotencyService.isProcessed(event.getOrderId())) {
            ack.acknowledge();
            return;
        }
        
        // 处理业务
        processOrder(event);
        
        // 标记已处理
        idempotencyService.markProcessed(event.getOrderId());
        
        ack.acknowledge();
    }
}
```

```yaml
      async-commit-buffer-limit: 10000
    tm:
      commit-retry-count: 5
      rollback-retry-count: 5

# 模式选择
# AT 模式：自动代理数据源，无侵入（类似2PC）
# TCC 模式：需实现 Try/Confirm/Cancel
# Saga 模式：状态机引擎
# XA 模式：传统XA协议
```

---

## 一句话收尾

分布式事务没有银弹：**2PC/3PC** 强一致但性能差，**TCC** 适合短事务高并发，**Saga** 适合长流程，**本地消息表** 适合异步场景；生产环境推荐 **Seata 的 AT 或 TCC 模式**，根据业务特性选择，最终一致性是主流实践。
