## 先说结论

MySQL 主流存储引擎包括 **InnoDB、MyISAM、Memory、Archive** 等；**InnoDB 支持事务、行级锁、崩溃恢复、MVCC**，而 **MyISAM 只支持表级锁、不支持事务**；生产环境选择 InnoDB 是因为**数据安全性和并发性能**是核心诉求。

## 🔥 详细展开

### 1. InnoDB vs MyISAM 详细对比

| 特性 | InnoDB | MyISAM |
|------|--------|--------|
| **事务支持** | ✅ ACID 完整支持 | ❌ 不支持 |
| **锁粒度** | 行级锁（并发高） | 表级锁（并发低） |
| **崩溃恢复** | ✅ Redo/Undo Log | ❌ 可能损坏 |
| **外键约束** | ✅ 支持 | ❌ 不支持 |
| **MVCC** | ✅ 支持（读不阻塞写） | ❌ 不支持 |
| **索引类型** | 聚簇索引（数据即索引） | 非聚簇索引（指针指向数据） |
| **全文索引** | 5.6+ 支持 | ✅ 原生支持（曾为其优势） |
| **表空间** | 共享/独立表空间 | 数据、索引、结构分离 |
| **存储限制** | 64TB | 256TB |
| **适用场景** | 高并发 OLTP | 读多写少、日志分析 |

### 2. 核心差异深度解析

#### 差异 1：事务支持

```sql
-- InnoDB：事务安全
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE account SET balance = balance + 100 WHERE id = 2;
COMMIT;  -- 要么全成功，要么全回滚

-- MyISAM：无事务，中途崩溃可能导致数据不一致
-- 第一条执行成功，第二条失败，钱就丢了
```

#### 差异 2：锁机制

```sql
-- InnoDB：行级锁，只锁需要的行
-- Session A
BEGIN;
SELECT * FROM orders WHERE user_id = 100 FOR UPDATE;  -- 只锁 user_id=100 的行

-- Session B
SELECT * FROM orders WHERE user_id = 200 FOR UPDATE;  -- ✅ 不阻塞，锁不同行


-- MyISAM：表级锁，整个表被锁
-- Session A
LOCK TABLES orders READ;  -- 整个表被读锁

-- Session B
UPDATE orders SET status = 1 WHERE id = 1;  -- ❌ 阻塞！等待表锁释放
```

#### 差异 3：索引结构

```
┌─────────────────────────────────────────────────────────────────┐
│                    InnoDB 聚簇索引                               │
├─────────────────────────────────────────────────────────────────┤

│  ┌──────────┬──────────────────────────────┐                    │

│  ├──────────┼──────────────────────────────┤                    │
│  │    1     │  {name: 'Alice', age: 25...} │                    │
│  │    2     │  {name: 'Bob', age: 30...}   │                    │

│  │  name    │  PK(id)  │  ← 叶子节点存主键值，需要回表            │
```

### 3. 为什么生产环境都用 InnoDB？

```java
// 生产环境的核心诉求：
// 1. 数据不能丢（ACID）
// 2. 并发要高（行级锁）
// 3. 故障可恢复（崩溃恢复）
// 4. 读不阻塞写（MVCC）

// 这些 MyISAM 都无法满足
```

| 生产需求 | InnoDB | MyISAM | 说明 |
|---------|--------|--------|------|
| **金融交易** | ✅ | ❌ | 必须事务保证资金安全 |
| **电商订单** | ✅ | ❌ | 高并发下单需要行锁 |
| **用户系统** | ✅ | ❌ | 读写混合，MVCC 避免阻塞 |
| **日志分析** | ✅ | ⚠️ | MyISAM 曾适合，现可用 Archive |

**MyISAM 的唯一适用场景（现在也少用了）：**
- 纯读场景（如数据仓库，但 ClickHouse 更好）
- 需要表级压缩且只读
- 全文检索（但 Elasticsearch 更好）

## 🔥 扩展追问 1：InnoDB 的崩溃恢复机制

```sql
-- InnoDB 通过 Redo Log 保证崩溃恢复
-- 写入流程：Buffer Pool → Redo Log Buffer → OS Cache → Redo Log File

-- 查看 Redo Log 配置
SHOW VARIABLES LIKE 'innodb_log%';

/*
innodb_log_file_size    = 1G      -- 单个 Redo Log 文件大小
innodb_log_files_in_group = 2     -- Redo Log 文件数量
innodb_flush_log_at_trx_commit = 1 -- 每次事务提交都刷盘
*/

-- 崩溃恢复过程：
-- 1. 启动时检查数据页 LSN 和 Redo Log LSN
-- 2. 如果数据页 LSN < Redo Log LSN，说明页未刷盘
-- 3. 重放 Redo Log，恢复数据
```

```
崩溃场景：
┌─────────────────────────────────────────────────────────────┐
│  事务提交 → Redo Log 刷盘 → 数据页在 Buffer Pool              │
│       ↓                                                    │
│  💥 突然断电                                                │
│       ↓                                                    │
│  重启后：Redo Log 已记录，但数据页未刷盘                      │
│       ↓                                                    │
│  InnoDB 自动重放 Redo Log，恢复数据 ✅                        │
└─────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 2：InnoDB 的 MVCC 实现

```sql
-- MVCC：多版本并发控制，实现读不阻塞写

-- 表结构
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    age INT,
    DB_TRX_ID BIGINT,      -- 隐藏列：最后修改的事务ID
    DB_ROLL_PTR POINTER    -- 隐藏列：回滚指针
) ENGINE=InnoDB;

-- 事务 100 修改
BEGIN;
UPDATE user SET age = 26 WHERE id = 1;  -- 创建新版本
COMMIT;

-- 事务 200 读取（快照读，不受事务100影响）
SELECT * FROM user WHERE id = 1;  -- 读到修改前的版本

-- 通过 Undo Log 构建历史版本链
```

```
MVCC 版本链：
┌─────────────────────────────────────────────────────────────┐
│  当前版本（事务100提交）                                       │
│  ┌─────────┬──────┬─────┬───────────┬────────────┐          │
│  │  id=1   │ Alice│ 26  │ DB_TRX_ID │ DB_ROLL_PTR│          │
│  │         │      │     │   = 100   │  → 指向旧版 │          │
│  └─────────┴──────┴─────┴───────────┴────────────┘          │
│                           ↓                                  │
│  历史版本（Undo Log）                                         │
│  ┌─────────┬──────┬─────┬───────────┬────────────┐          │
│  │  id=1   │ Alice│ 25  │ DB_TRX_ID │ DB_ROLL_PTR│          │
│  │         │      │     │   = 50    │  → 指向更旧 │          │
│  └─────────┴──────┴─────┴───────────┴────────────┘          │
│                                                              │
│  ReadView 决定事务能看到哪个版本                              │
└─────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 4：MyISAM 迁移到 InnoDB 的注意事项

```sql
-- 1. 索引差异：InnoDB 主键是聚簇索引，选择合适的 PK 很重要
-- ❌ 避免：使用 UUID 作为主键（随机插入导致页分裂）
-- ✅ 推荐：自增 ID 或趋势递增的 ID

-- 2. 行数统计差异
-- MyISAM：SELECT COUNT(*) FROM table; -- O(1)，直接读元数据
-- InnoDB：SELECT COUNT(*) FROM table; -- O(n)，需要扫描

-- 解决方案：用计数器表或近似值
SELECT TABLE_ROWS FROM INFORMATION_SCHEMA.TABLES 
WHERE TABLE_NAME = 'orders';

-- 3. 全文检索替代方案
-- MyISAM 原生支持 FULLTEXT
-- InnoDB 5.6+ 也支持，但生产环境推荐 Elasticsearch

-- 4. 内存使用差异
-- MyISAM：索引缓存、数据缓存分开
-- InnoDB：Buffer Pool 统一管理，需要调整大小
SET GLOBAL innodb_buffer_pool_size = 4 * 1024 * 1024 * 1024;  -- 4GB
```

## 🔥 扩展追问 5：其他存储引擎适用场景

| 引擎 | 特点 | 适用场景 |
|------|------|---------|
| **Memory** | 数据存内存，重启丢失 | 临时表、缓存、会话数据 |
| **Archive** | 高压缩，只支持 INSERT/SELECT | 日志归档、历史数据 |
| **CSV** | 纯文本 CSV 格式 | 数据交换、简单导入导出 |
| **Blackhole** | 写入即丢弃 | 主从复制中继、测试 |
| **NDB** | 分布式集群 | MySQL Cluster 高可用 |

```sql
-- Archive 引擎示例（日志归档）
CREATE TABLE access_logs_archive (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    log_data TEXT,
    created_at TIMESTAMP
) ENGINE=Archive;

-- 压缩比通常可达 10:1
INSERT INTO access_logs_archive SELECT * FROM access_logs 
WHERE created_at < DATE_SUB(NOW(), INTERVAL 1 MONTH);
```

## 一句话收尾

**InnoDB 以事务、行锁、MVCC、崩溃恢复**全面碾压 MyISAM，满足生产环境对**数据安全和并发性能**的核心诉求；MyISAM 因**表锁、无事务、易损坏**已被淘汰，仅在特定历史系统或纯读分析场景偶见。
