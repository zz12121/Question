## 先说结论

MySQL 锁按粒度分为**表锁、页锁、行锁**，按功能分为**读锁（共享）、写锁（排他）**；InnoDB 行锁包括**记录锁（Record Lock）、间隙锁（Gap Lock）、临键锁（Next-Key Lock）**，其中临键锁是记录锁+间隙锁的组合，用于解决幻读问题。

## 🔥 详细展开

### 1. 表锁 vs 行锁

#### 表锁（Table Lock）

```sql
-- MyISAM 默认使用表锁
-- 显式加表锁
LOCK TABLES user READ;    -- 表读锁（共享）
LOCK TABLES user WRITE;   -- 表写锁（排他）

-- 解锁
UNLOCK TABLES;
```

**表锁的并发问题：**

```
┌─────────────────────────────────────────────────────────────────┐
│  Session A              │  Session B                            │
├─────────────────────────┼───────────────────────────────────────┤
│  LOCK TABLES user WRITE │                                       │
│  （获得写锁）             │                                       │
│                         │  SELECT * FROM user                   │
│                         │  （阻塞，等待表锁释放）                  │
│                         │                                       │
│  INSERT INTO user...    │                                       │
│  （正常执行）             │                                       │
│                         │  （继续阻塞）                          │
│                         │                                       │
│  UNLOCK TABLES          │                                       │
│  （释放锁）               │  （立即执行查询）                      │
└─────────────────────────┴───────────────────────────────────────┘
```

#### 行锁（Row Lock）

```sql
-- InnoDB 默认使用行锁
-- 显式加行锁
SELECT * FROM user WHERE id = 1 FOR UPDATE;  -- 排他锁（X锁）
SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE;  -- 共享锁（S锁）

-- 隐式加锁（自动）
UPDATE user SET age = 26 WHERE id = 1;  -- 自动加 X 锁
```

**行锁的优势：**

```
┌─────────────────────────────────────────────────────────────────┐
│  Session A              │  Session B                            │
├─────────────────────────┼───────────────────────────────────────┤
│  BEGIN;                 │                                       │
│  UPDATE user SET        │                                       │
│    age = 26             │                                       │
│  WHERE id = 1;          │                                       │
│  （给 id=1 加 X 锁）      │                                       │
│                         │  UPDATE user SET                      │
│                         │    age = 27                           │
│                         │  WHERE id = 2;                        │
│                         │  ✅ 正常执行，只锁 id=1，id=2 没锁      │
│                         │                                       │
│  COMMIT;                │                                       │
│  （释放锁）               │                                       │
└─────────────────────────┴───────────────────────────────────────┘
```

### 2. InnoDB 行锁的三种类型

#### 记录锁（Record Lock）

```sql
-- 锁定索引中的具体记录
SELECT * FROM user WHERE id = 1 FOR UPDATE;
-- 只锁住 id = 1 这一行
```

```
┌─────────────────────────────────────────────────────────────────┐
│  记录锁：锁定具体的索引记录                                       │
│                                                                  │
│  数据：id = [1, 5, 10, 15, 20]                                   │
│                                                                  │
│  锁定 id = 10：                                                  │
│  ┌─────┐  ┌─────┐  ┌─────────┐  ┌─────┐  ┌─────┐               │
│  │  1  │  │  5  │  │   10    │  │  15 │  │  20 │               │
│  │     │  │     │  │  🔒X    │  │     │  │     │               │
│  └─────┘  └─────┘  └─────────┘  └─────┘  └─────┘               │
│                       ↑                                          │
│                   只锁这一行                                     │
│                                                                  │
│  其他行可以正常读写 ✅                                            │
└─────────────────────────────────────────────────────────────────┘
```

#### 间隙锁（Gap Lock）

```sql
-- 锁定索引记录之间的间隙，防止幻读
-- 在 RR（可重复读）隔离级别下生效

SELECT * FROM user WHERE id > 5 AND id < 15 FOR UPDATE;
-- 锁住 (5, 15) 之间的间隙，不包含 5 和 15
```

```
┌─────────────────────────────────────────────────────────────────┐
│  间隙锁：锁定索引记录之间的"间隙"                                 │
│                                                                  │
│  数据：id = [1, 5, 10, 15, 20]                                   │
│                                                                  │
│  锁定范围 (5, 15)：                                              │
│  ┌─────┐  ┌─────┐  ┌─────────┐  ┌─────┐  ┌─────┐               │
│  │  1  │  │  5  │  │   10    │  │  15 │  │  20 │               │
│  │     │  │     │  │         │  │     │  │     │               │
│  └─────┘  └─────┘  └─────────┘  └─────┘  └─────┘               │
│              ↑──────────────────↑                                │
│                   间隙锁范围                                     │
│                   阻止插入 id = 6, 7, 8...14                     │
│                                                                  │
│  注意：间隙锁之间不冲突，可以共存                                  │
│  Session A 锁 (5, 15)，Session B 也可以锁 (5, 15)                │
└─────────────────────────────────────────────────────────────────┘
```

#### 临键锁（Next-Key Lock）

```sql
-- 记录锁 + 间隙锁的组合
-- 锁定记录本身 + 记录前面的间隙

SELECT * FROM user WHERE id <= 10 FOR UPDATE;
-- 锁住 (-∞, 1], (1, 5], (5, 10]
```

```
┌─────────────────────────────────────────────────────────────────┐
│  临键锁：左开右闭区间 [前间隙, 当前记录]                          │
│                                                                  │
│  数据：id = [1, 5, 10, 15, 20]                                   │
│                                                                  │
│  锁定 id <= 10：                                                 │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────┐  ┌─────┐       │
│  │ (-∞, 1] │  │  (1, 5] │  │  (5, 10]│  │  15 │  │  20 │       │
│  │   🔒    │  │    🔒   │  │    🔒   │  │     │  │     │       │
│  └─────────┘  └─────────┘  └─────────┘  └─────┘  └─────┘       │
│       ↑           ↑           ↑                                  │
│    间隙+记录    间隙+记录    间隙+记录                            │
│                                                                  │
│  阻止：                                                          │
│  • 插入 id = 0, 2, 6, 8, 11（在间隙内）                          │
│  • 修改/删除 id = 1, 5, 10（记录锁）                             │
│                                                                  │
│  这是 InnoDB 解决幻读的核心机制！                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3. 锁的兼容矩阵

```
┌─────────────────────────────────────────────────────────────────┐
│                      锁兼容性矩阵                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│              │   IS    │   IX    │    S    │    X    │         │
│  ────────────┼─────────┼─────────┼─────────┼─────────┤         │
│      IS      │   ✅    │   ✅    │   ✅    │   ❌    │         │
│      IX      │   ✅    │   ✅    │   ❌    │   ❌    │         │
│      S       │   ✅    │   ❌    │   ✅    │   ❌    │         │
│      X       │   ❌    │   ❌    │   ❌    │   ❌    │         │
│                                                                  │
│  IS = 意向共享锁，IX = 意向排他锁                                  │
│  S = 共享锁（读锁），X = 排他锁（写锁）                            │
│                                                                  │
│  规则：意向锁之间兼容，意向锁与表级 S/X 锁判断                     │
│       行级 S/X 锁与表级 IS/IX 锁不直接冲突                        │
└─────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 1：意向锁的作用

```sql
-- 意向锁是表级锁，表示事务有意向对表中的某些行加锁

-- Session A：给某行加 X 锁
BEGIN;
SELECT * FROM user WHERE id = 1 FOR UPDATE;  -- 自动加 IX 锁（表级）+ X 锁（行级）

-- Session B：想加表级 X 锁
LOCK TABLES user WRITE;  -- 需要等待 Session A 的 IX 锁释放
-- 因为 IX 和 X 不兼容
```

**为什么需要意向锁？**

```
没有意向锁时：
┌─────────────────────────────────────────────────────────────────┐
│  Session A 给 id=1 加行锁                                        │
│       ↓                                                          │
│  Session B 想加表锁，需要检查表中每一行是否被锁定                  │
│       ↓                                                          │
│  遍历全表，性能 O(n)                                              │
└─────────────────────────────────────────────────────────────────┘

有意向锁时：
┌─────────────────────────────────────────────────────────────────┐
│  Session A 给 id=1 加行锁，同时加 IX 意向锁（表级）               │
│       ↓                                                          │
│  Session B 想加表锁，只需检查 IX 锁是否存在                       │
│       ↓                                                          │
│  直接发现冲突，性能 O(1)                                          │
└─────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 2：锁的查看和排查

```sql
-- 查看当前锁情况（MySQL 8.0）
SELECT * FROM performance_schema.data_locks;
SELECT * FROM performance_schema.data_lock_waits;

-- 查看锁等待（经典方法）
SHOW ENGINE INNODB STATUS;

-- 查看事务和锁
SELECT 
    r.trx_id waiting_trx_id,
    r.trx_mysql_thread_id waiting_thread,
    r.trx_query waiting_query,
    b.trx_id blocking_trx_id,
    b.trx_mysql_thread_id blocking_thread,
    b.trx_query blocking_query
FROM information_schema.innodb_lock_waits w
INNER JOIN information_schema.innodb_trx b ON b.trx_id = w.blocking_trx_id
INNER JOIN information_schema.innodb_trx r ON r.trx_id = w.requesting_trx_id;
```

**死锁示例：**

```sql
-- Session A
BEGIN;
UPDATE account SET balance = balance - 100 WHERE id = 1;  -- 锁 id=1
-- 等待一会儿
UPDATE account SET balance = balance + 100 WHERE id = 2;  -- 需要 id=2 的锁

-- Session B
BEGIN;
UPDATE account SET balance = balance - 100 WHERE id = 2;  -- 锁 id=2
-- 等待一会儿
UPDATE account SET balance = balance + 100 WHERE id = 1;  -- 需要 id=1 的锁

-- 结果：死锁！MySQL 会自动检测并回滚其中一个事务
-- ERROR 1213 (40001): Deadlock found when trying to get lock
```

## 🔥 扩展追问 3：不同隔离级别下的锁行为

```sql
-- 读未提交（RU）：不加锁，直接读最新数据（可能脏读）

-- 读已提交（RC）：
-- • 普通 SELECT：不加锁，读快照（MVCC）
-- • SELECT FOR UPDATE：只加记录锁，不加间隙锁

-- 可重复读（RR，默认）：
-- • 普通 SELECT：不加锁，读快照（MVCC）
-- • SELECT FOR UPDATE：加临键锁（记录锁 + 间隙锁）

-- 串行化（Serializable）：
-- • 所有 SELECT 都加共享锁，完全串行
```

**幻读问题的锁解决：**

```sql
-- 事务隔离级别：RR
BEGIN;
SELECT * FROM user WHERE age > 20 FOR UPDATE;
-- 锁住 age > 20 的所有记录 + 间隙
-- 其他事务无法插入 age > 20 的新记录
-- 解决了幻读问题！

-- 注意：普通 SELECT 不加锁，可能幻读
SELECT * FROM user WHERE age > 20;  -- 快照读，可能幻读
```

## 🔥 扩展追问 4：优化锁竞争

```sql
-- 1. 尽量使用索引，避免升级为表锁
-- 没走索引时，InnoDB 会退化为表锁！
UPDATE user SET age = 26 WHERE name = 'Alice';  
-- 如果 name 没索引，会锁整个表！

-- 2. 控制事务粒度，减少锁持有时间
BEGIN;
-- 只放必要的操作
UPDATE order SET status = 1 WHERE id = 1;
COMMIT;  -- 尽快提交

-- 3. 按固定顺序访问资源，避免死锁
-- 所有事务都先操作 A 表，再操作 B 表

-- 4. 使用乐观锁替代悲观锁
UPDATE product SET stock = stock - 1, version = version + 1 
WHERE id = 1 AND version = 5;  -- 乐观锁，无锁等待

-- 5. 降低隔离级别（如果业务允许）
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- RC 级别下，间隙锁减少，并发性能提升
```

## 一句话收尾

MySQL 锁按粒度分**表锁、行锁**，按功能分**读锁、写锁**；InnoDB 行锁细分为**记录锁（锁行）、间隙锁（锁范围）、临键锁（记录+间隙）**，其中临键锁通过**左开右闭区间**解决幻读，是 RR 隔离级别实现可重复读的核心机制。
