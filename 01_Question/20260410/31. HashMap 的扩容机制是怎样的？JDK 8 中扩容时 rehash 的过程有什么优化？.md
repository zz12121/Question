## 先说结论

HashMap 在元素超过 **容量 × 负载因子（默认 0.75）** 时触发扩容，容量翻倍为 **2 的幂次**。JDK 8 对 rehash 的核心优化是：**元素在新数组中的位置只可能是原位置或原位置 + 旧容量**，无需重新计算 hash，通过 `(hash & oldCap) == 0` 快速判断，且扩容时采用 **尾插法** 避免了 JDK 7 的并发死循环问题。

---

## 🔥 核心概念

**解决什么问题：**
- 哈希冲突导致链表过长，查询退化为 O(n)
- 保持负载均衡，维持 O(1) 平均时间复杂度

**基本原理：**
- 数组容量保持 2 的幂次，方便位运算
- 扩容时迁移元素，重新计算存储位置
- 负载因子平衡空间与时间的 trade-off

## 🔥 详细展开

### 1. 扩容触发条件

```java
// HashMap.put() 中的扩容判断
final V putVal(int hash, K key, V value, boolean onlyIfAbsent, boolean evict) {
    Node<K,V>[] tab; Node<K,V> p; int n, i;
    
    // 首次初始化
    if ((tab = table) == null || (n = tab.length) == 0)
        n = (tab = resize()).length;
    
    // 无冲突，直接插入
    if ((p = tab[i = (n - 1) & hash]) == null)
        tab[i] = newNode(hash, key, value, null);
    
    else {
        // 处理冲突...
        
        // 插入后检查是否需要扩容
        if (++size > threshold)  // threshold = capacity * loadFactor
            resize();
    }
}
```

**关键参数：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `DEFAULT_INITIAL_CAPACITY` | 16 | 初始容量 |
| `DEFAULT_LOAD_FACTOR` | 0.75 | 负载因子 |
| `threshold` | 12 (16×0.75) | 扩容阈值 |
| `MAXIMUM_CAPACITY` | 1 << 30 | 最大容量 |

### 2. 扩容过程（resize 方法）

```java
final Node<K,V>[] resize() {
    Node<K,V>[] oldTab = table;
    int oldCap = (oldTab == null) ? 0 : oldTab.length;
    int oldThr = threshold;
    int newCap, newThr = 0;
    
    if (oldCap > 0) {
        // 超过最大容量，不再扩容
        if (oldCap >= MAXIMUM_CAPACITY) {
            threshold = Integer.MAX_VALUE;
            return oldTab;
        }
        // 容量翻倍
        else if ((newCap = oldCap << 1) < MAXIMUM_CAPACITY &&
                 oldCap >= DEFAULT_INITIAL_CAPACITY)
            newThr = oldThr << 1;  // 阈值也翻倍
    }
    
    // 创建新数组
    @SuppressWarnings({"rawtypes","unchecked"})
    Node<K,V>[] newTab = (Node<K,V>[])new Node[newCap];
    table = newTab;
    
    // 迁移旧数据
    if (oldTab != null) {
        for (int j = 0; j < oldCap; ++j) {
            Node<K,V> e;
            if ((e = oldTab[j]) != null) {
                oldTab[j] = null;  // 帮助 GC
                
                if (e.next == null)
                    // 单个节点，直接计算新位置
                    newTab[e.hash & (newCap - 1)] = e;
                else if (e instanceof TreeNode)
                    // 红黑树拆分
                    ((TreeNode<K,V>)e).split(this, newTab, j, oldCap);
                else {
                    // 链表拆分 - JDK 8 优化核心！
                    preserve order
                    Node<K,V> loHead = null, loTail = null;
                    Node<K,V> hiHead = null, hiTail = null;
                    Node<K,V> next;
                    
                    do {
                        next = e.next;
                        // 关键判断：hash & oldCap == 0
                        if ((e.hash & oldCap) == 0) {
                            // 位置不变：仍在原索引 j
                            if (loTail == null)
                                loHead = e;
                            else
                                loTail.next = e;
                            loTail = e;
                        }
                        else {
                            // 位置变化：新索引 = j + oldCap
                            if (hiTail == null)
                                hiHead = e;
                            else
                                hiTail.next = e;
                            hiTail = e;
                        }
                    } while ((e = next) != null);
                    
                    if (loTail != null) {
                        loTail.next = null;
                        newTab[j] = loHead;
                    }
                    if (hiTail != null) {
                        hiTail.next = null;
                        newTab[j + oldCap] = hiHead;
                    }
                }
            }
        }
    }
    
    return newTab;
}
```

### 3. JDK 8 的 rehash 优化

#### 优化 1：位置只可能是原位置或原位置 + 旧容量

```java
// 旧容量为 16 时，newCap = 32
// 元素 hash 决定新位置：

hash:     0101 0101  (假设)
oldCap:   0001 0000  (16)
newCap-1: 0001 1111  (31)

// 旧位置计算：
index = hash & (16-1) = hash & 15 = hash & 0000 1111

// 新位置计算：
index = hash & (32-1) = hash & 31 = hash & 0001 1111

// 区别只在第 5 位（oldCap 的最高位）
// 如果 hash 的第 5 位是 0，位置不变
// 如果 hash 的第 5 位是 1，位置 = 原位置 + 16
```

**图解：**

```
旧数组（容量16）          新数组（容量32）
索引 0: [A]              索引 0: [A]  ← hash & 16 == 0
索引 1: [B]              索引 1: [B]  ← hash & 16 == 0
...                      ...
索引 15: [C,D]           索引 15: [C] ← hash & 16 == 0
                         索引 31: [D] ← hash & 16 != 0 (15+16)
```

#### 优化 2：无需重新计算 hash

```java
// JDK 7：需要重新计算
int newIndex = hash(key) & (newCapacity - 1);

// JDK 8：只需判断一位
if ((hash & oldCap) == 0) {
    // 位置不变
} else {
    // 位置 = 原位置 + oldCap
}
```

#### 优化 3：尾插法避免死循环

```java
// JDK 7：头插法（并发扩容时可能形成环）
void transfer(Entry[] newTable) {
    Entry[] src = table;
    int newCapacity = newTable.length;
    for (int j = 0; j < src.length; j++) {
        Entry<K,V> e = src[j];
        if (e != null) {
            src[j] = null;
            do {
                Entry<K,V> next = e.next;
                int i = indexFor(e.hash, newCapacity);
                e.next = newTable[i];  // 头插：新元素指向旧头
                newTable[i] = e;       // 新元素成为新头
                e = next;
            } while (e != null);
        }
    }
}

// JDK 8：尾插法（保持链表顺序）
Node<K,V> loHead = null, loTail = null;
do {
    next = e.next;
    if ((e.hash & oldCap) == 0) {
        if (loTail == null)
            loHead = e;
        else
            loTail.next = e;  // 尾插：追加到链表尾部
        loTail = e;
    }
    // ...
} while ((e = next) != null);
loTail.next = null;  // 断开旧引用
```

**JDK 7 并发死循环原因：**

```
线程 A 和 B 同时扩容，都读到 A→B→C 链表

线程 A 执行到：e=A, next=B，被挂起

线程 B 完成扩容（头插法），链表变成 C→B→A

线程 A 恢复：
  - e=A, next=B（但 B 现在指向 A！）
  - 插入 A
  - e=B, next=A（B.next 被改成 A）
  - 插入 B
  - e=A, next=null（A.next 是 null）
  - 插入 A（A 已经在链表里了！）
  
结果：B→A→B→A... 形成环！
```

### 4. 红黑树拆分（JDK 8）

```java
// TreeNode.split()
final void split(HashMap<K,V> map, Node<K,V>[] tab, int index, int bit) {
    TreeNode<K,V> b = this;
    
    // 同样拆成两组
    TreeNode<K,V> loHead = null, loTail = null;
    TreeNode<K,V> hiHead = null, hiTail = null;
    int lc = 0, hc = 0;
    
    for (TreeNode<K,V> e = b, next; e != null; e = next) {
        next = (TreeNode<K,V>)e.next;
        e.next = null;
        
        if ((e.hash & bit) == 0) {
            if ((e.prev = loTail) == null)
                loHead = e;
            else
                loTail.next = e;
            loTail = e;
            ++lc;
        }
        else {
            if ((e.prev = hiTail) == null)
                hiHead = e;
            else
                hiTail.next = e;
            hiTail = e;
            ++hc;
        }
    }
    
    // 如果拆分后节点数 <= 6，退化为链表
    if (loHead != null) {
        if (lc <= UNTREEIFY_THRESHOLD)  // 6
            tab[index] = loHead.untreeify(map);
        else {
            tab[index] = loHead;
            if (hiHead != null)
                loHead.treeify(tab);  // 重新树化
        }
    }
    
    if (hiHead != null) {
        if (hc <= UNTREEIFY_THRESHOLD)
            tab[index + bit] = hiHead.untreeify(map);
        else {
            tab[index + bit] = hiHead;
            if (loHead != null)
                hiHead.treeify(tab);
        }
    }
}
```

## 🔥 扩展追问 1：为什么容量必须是 2 的幂次？

```java
// 2^n - 1 的二进制是全 1
// n=4:  16-1 = 15 = 0000 1111
// n=5:  32-1 = 31 = 0001 1111

// 取模运算可以用位运算替代
index = hash % 16;     // 慢
index = hash & 15;     // 快！hash & (16-1)

// 同时保证 hash 值均匀分布到数组各位置
```

**如果不是 2 的幂次：**
- `hash & (capacity-1)` 不能替代取模
- 某些位永远为 0，导致冲突增加

## 🔥 扩展追问 2：hash() 方法为什么要右移 16 位？

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

**原因：** 让高 16 位也参与运算

```
hashCode:     1111 0000 1111 0000 1111 0000 1111 0000
              ↓ 无符号右移 16 位
>>> 16:       0000 0000 0000 0000 1111 0000 1111 0000
              ↓ 异或
hash:         1111 0000 1111 0000 0000 0000 0000 0000

// 如果不做扰动，只用低几位：
index = hash & (16-1) = hash & 0x0F
// 高位的信息被浪费了，冲突概率增加
```

## 🔥 扩展追问 3：为什么负载因子是 0.75？

| 负载因子 | 空间利用率 | 冲突概率 | 适用场景 |
|----------|-----------|----------|----------|
| 0.5 | 低（浪费空间） | 低 | 内存充足，追求极致性能 |
| **0.75** | **平衡** | **适中** | **默认，通用场景** |
| 1.0 | 高 | 高 | 内存紧张，可接受更多冲突 |

**数学依据：** 泊松分布

```
// 当负载因子为 0.75 时，链表平均长度为 0.5
// 链表长度达到 8 的概率已经很低（0.00000606）
// 所以树化阈值设为 8
```

```java

* 2:    0.07581633

* 5:    0.00015795

* 7:    0.00000094
* 8:    0.00000006  ← 链表长度达到 8 的概率
```

## 一句话收尾

HashMap 扩容容量翻倍，JDK 8 用 `(hash & oldCap)` 快速判断新位置（原位置或原位置+旧容量），尾插法避免并发死循环；容量保持 2 的幂次是为了位运算取模，负载因子 0.75 是空间与时间的最佳平衡点！
