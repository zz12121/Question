## 先说结论

**CAS**（Compare And Swap）是一种**无锁原子操作**，通过比较内存值与预期值，相等则更新，否则重试。它基于 CPU 原子指令（如 `cmpxchg`）实现，性能优于锁但存在 **ABA 问题**（值从 A→B→A，CAS 无法感知中间变化）。解决 ABA 问题可用 **版本号/时间戳**（`AtomicStampedReference`）或 **标记位**（`AtomicMarkableReference`）。

---

## 🔥 核心概念

**解决什么问题：**
- 多线程并发修改共享变量时的数据竞争
- 避免传统锁的开销（上下文切换、阻塞唤醒）

**基本原理：**
- 乐观锁思想：先操作，冲突时重试
- 依赖硬件原子指令保证操作的原子性

## 🔥 详细展开

### 1. CAS 的操作流程

```java
// 伪代码
function cas(memoryLocation, expectedValue, newValue):
    if *memoryLocation == expectedValue:
        *memoryLocation = newValue
        return true
    else:
        return false
```

**Java 中的 CAS：**

```java
// Unsafe 类提供 CAS 操作
public final native boolean compareAndSwapInt(
    Object o,       // 对象
    long offset,    // 字段偏移量
    int expected,   // 预期值
    int x           // 新值
);

// AtomicInteger 使用示例
AtomicInteger atomicInt = new AtomicInteger(0);

// 原子递增
int newValue = atomicInt.incrementAndGet();

// 底层调用
public final int incrementAndGet() {
    return unsafe.getAndAddInt(this, valueOffset, 1) + 1;
}
```

**实际执行流程：**

```
线程 A 读取值：0
    ↓
准备更新为：1
    ↓
CAS(0, 1)：检查内存值是否为 0
    ↓
是 → 更新为 1，成功
否 → 说明被其他线程修改，重试
```

### 2. Java CAS 源码分析

```java
// AtomicInteger.getAndAddInt() 自旋实现
public final int getAndAddInt(Object o, long offset, int delta) {
    int v;
    do {
        // 获取当前值（volatile 读）
        v = getIntVolatile(o, offset);
    } while (!compareAndSwapInt(o, offset, v, v + delta));
    // 如果 CAS 失败，循环重试（自旋）
    return v;
}
```

**CPU 指令层面（x86）：**

```asm
; cmpxchg 指令（Compare and Exchange）
lock cmpxchg dword ptr [eax], ecx
; lock 前缀保证多核原子性
; cmpxchg 比较 eax 和 [eax]，相等则 ecx 写入 [eax]
```

### 3. ABA 问题详解

**问题场景：**

```
时间线：
  t1: 线程 A 读取值为 A
  t2: 线程 B 将值改为 B
  t3: 线程 C 将值改回 A
  t4: 线程 A CAS 检查：值还是 A，更新成功！

问题：线程 A 不知道中间发生过 B 的变化
      如果 A 是链表头节点，可能指向已被删除的节点！
```

**代码演示：**

```java
public class ABAProblem {
    
    static AtomicInteger atomicInt = new AtomicInteger(100);
    
    public static void main(String[] args) throws InterruptedException {
        Thread t1 = new Thread(() -> {
            int expected = 100;
            // 期望 100，改为 101
            boolean success = atomicInt.compareAndSet(expected, 101);
            System.out.println("T1 CAS: " + success + ", value: " + atomicInt.get());
        });
        
        Thread t2 = new Thread(() -> {
            // 100 → 200 → 100（ABA 变化）
            atomicInt.set(200);
            System.out.println("T2 changed to 200");
            atomicInt.set(100);
            System.out.println("T2 changed back to 100");
        });
        
        t2.start();
        t2.join();  // 确保 T2 先完成 ABA
        
        t1.start();
        t1.join();
        
        // T1 的 CAS 成功了，但它没感知到中间的变化！
    }
}
```

**实际危害（链表操作）：**

```java
// 链表头节点：A → B → C

// 线程 1：想删除 A，使头节点变为 B
// 预期：head = A，新值 = B
cas(head, A, B);

// 线程 2：先删除 A，再删除 B，再插入 A
// head: A → B → C
//      ↓
// head: B → C
//      ↓
// head: C
//      ↓
// head: A → C  （A 被重新插入，但 next 指向 C，B 已丢失！）

// 线程 1 执行 CAS：head 还是 A，成功改为 B
// 结果：B 已经被删除，head 指向一个已回收的节点！
```

### 4. ABA 问题的解决方案

#### 方案 1：版本号（AtomicStampedReference）

```java
public class AtomicStampedReference<V> {
    
    private static class Pair<T> {
        final T reference;    // 值
        final int stamp;      // 版本号
        // ...
    }
    
    // CAS 同时检查值和版本号
    public boolean compareAndSet(V expectedRef, V newRef,
                                  int expectedStamp, int newStamp) {
        Pair<V> current = pair;
        return expectedRef == current.reference &&
               expectedStamp == current.stamp &&
               ((newRef == current.reference && newStamp == current.stamp) ||
                casPair(current, Pair.of(newRef, newStamp)));
    }
}
```

**使用示例：**

```java
AtomicStampedReference<Integer> stampedRef = 
    new AtomicStampedReference<>(100, 0);  // 初始值 100，版本 0

// 获取值和版本
int[] stampHolder = new int[1];
int value = stampedRef.get(stampHolder);
int stamp = stampHolder[0];

// CAS：同时检查值和版本
boolean success = stampedRef.compareAndSet(100, 101, stamp, stamp + 1);

// ABA 场景：
// 线程 B：100 → 200（版本 0→1）→ 100（版本 1→2）
// 线程 A：预期 (100, 0)，实际 (100, 2)，CAS 失败！
```

#### 方案 2：标记位（AtomicMarkableReference）

```java
// 用 boolean 标记是否被修改过
AtomicMarkableReference<Integer> markableRef = 
    new AtomicMarkedReference<>(100, false);

// 标记删除（逻辑删除）
markableRef.attemptMark(100, true);

// CAS 检查值和标记
markableRef.compareAndSet(100, 101, false, false);  // 如果已标记，失败
```

**适用场景：**
- `AtomicStampedReference`：需要知道修改次数
- `AtomicMarkableReference`：只需要知道是否被修改过

#### 方案 3：Java 8 的 LongAdder/DoubleAdder

```java
// 分段思想，避免 CAS 竞争
LongAdder adder = new LongAdder();

// 高并发下，不同线程操作不同 cell，减少 CAS 失败
adder.increment();
adder.add(10);

// 获取总和
long sum = adder.sum();
```

### 5. CAS 的优缺点

| 优点 | 缺点 |
|------|------|
| 无锁，性能高（无上下文切换） | ABA 问题 |
| 无死锁风险 | 自旋消耗 CPU（高竞争时） |
| 实现简单 | 只能保证单个变量原子性 |
| 适用于读多写少 | 无法解决多变量一致性问题 |

**性能对比：**

```java
// 测试：1000 万递增操作
// synchronized:     ~300ms
// AtomicInteger:    ~150ms（CAS）
// LongAdder:        ~50ms（分段）
```

## 🔥 扩展追问 2：Unsafe 类是什么？为什么 CAS 依赖它？

```java
// sun.misc.Unsafe：直接操作内存、执行 CAS
public final class Unsafe {
    
    // 获取字段偏移量
    public native long objectFieldOffset(Field f);
    
    // CAS 操作
    public final native boolean compareAndSwapInt(Object o, long offset, int expected, int x);
    
    // 直接内存操作
    public native long allocateMemory(long bytes);
    public native void freeMemory(long address);
    
    // 线程调度
    public native void park(boolean isAbsolute, long time);
    public native void unpark(Object thread);
}
```

**为什么叫 Unsafe：**
- 直接操作内存，破坏 Java 内存安全
- 绕过访问控制（可访问 private 字段）
- 可能导致 JVM 崩溃

**获取方式（不建议）：**

```java
// 反射获取（Java 9+ 被限制）
Field f = Unsafe.class.getDeclaredField("theUnsafe");
f.setAccessible(true);
Unsafe unsafe = (Unsafe) f.get(null);
```

**Java 9+ 替代方案：**

```java
// java.lang.invoke.VarHandle（官方推荐）
VarHandle handle = MethodHandles.lookup()
    .findVarHandle(AtomicInteger.class, "value", int.class);

handle.compareAndSet(atomicInt, 0, 1);
```

## 🔥 扩展追问 3：乐观锁 vs 悲观锁

| 维度 | 乐观锁（CAS） | 悲观锁（synchronized） |
|------|--------------|------------------------|
| 思想 | 先操作，冲突再重试 | 先加锁，确保独占 |
| 适用 | 读多写少 | 写多读少 |
| 性能 | 无竞争时极好 | 稳定 |
| 缺点 | 高竞争时自旋浪费 CPU | 线程阻塞开销 |
| 实现 | CAS 指令 | 操作系统互斥量 |

**数据库中的乐观锁：**

```sql
-- 版本号控制
UPDATE account 
SET balance = balance - 100, version = version + 1
WHERE id = 1 AND version = 5;  -- 版本号检查

-- 影响行数为 0 表示冲突，业务层重试
```

## 一句话收尾

CAS 是无锁原子操作，基于 CPU 指令实现，性能高但有 ABA 问题；ABA 通过版本号（AtomicStampedReference）或标记位（AtomicMarkableReference）解决；高并发计数用 LongAdder 分段优化，多变量一致性还是得用锁！
