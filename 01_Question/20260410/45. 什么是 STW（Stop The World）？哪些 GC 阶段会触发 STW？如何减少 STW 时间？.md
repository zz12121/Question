## 先说结论

STW（Stop The World）是 **JVM 暂停所有应用线程进行 GC 的阶段**，标记、整理等关键步骤需要 STW；减少 STW 的核心思路是**选择低延迟收集器（ZGC/G1）、调小堆内存、减少对象分配、优化代码结构**。

---

## 🔥 核心概念

### 为什么需要 STW？

```
┌─────────────────────────────────────────────────────────────────┐
│                     没有 STW 的问题                              │
├─────────────────────────────────────────────────────────────────┤
│  时间轴 →                                                        │
│                                                                  │
│  线程A:  标记对象X存活 ──→ 修改引用 ──→ 对象X被回收（误删！）      │
│              ↑                           ↓                       │
│  GC线程:     └──────── 认为X是垃圾 ──────┘                       │
│                                                                  │
│  问题：并发标记时，应用线程修改了对象引用关系                      │
│  导致：存活对象被误回收，程序崩溃                                  │
└─────────────────────────────────────────────────────────────────┘
```

**STW 的本质：** 暂停所有 Mutator（应用线程），保证 GC 期间堆内存的一致性视图。

## 🔥 详细展开

### 1. 各收集器的 STW 阶段

#### Serial/Parallel（全阶段 STW）

```
┌─────────────────────────────────────────────────────────────┐
│  应用程序 │  STW  │ GC线程 │  STW  │ GC线程 │  STW  │ 应用程序 │
│    ▓▓    │ ████ │  ▓▓▓  │ ████ │  ▓▓▓  │ ████ │   ▓▓    │
│          │ 标记  │       │ 清理  │       │ 整理  │          │
│          │      │       │      │       │      │          │
│          └──────┴───────┴──────┴───────┴──────┘          │
│                    所有阶段都 STW                            │
└─────────────────────────────────────────────────────────────┘
```

#### CMS（部分并发，但仍有 STW）

```
┌─────────────────────────────────────────────────────────────┐
│  初始标记 │ 并发标记 │ 重新标记 │ 并发清除 │ 并发重置        │
│   STW   │  并发   │   STW   │  并发   │  并发            │
│   ███   │   ▓▓▓  │   ███   │   ▓▓▓  │   ▓▓▓           │
│  (很快)  │        │ (较长)   │        │                 │
│          │        │         │        │                 │
│  根节点   │ 遍历   │ 修正并发 │ 清理   │ 重置标记        │
│  直接引用 │ 对象图 │ 期间的   │ 垃圾   │ 数据结构        │
│          │        │ 引用变更 │        │                 │
└─────────────────────────────────────────────────────────────┘
```

#### G1（目标可控的 STW）

```
Young GC:
┌─────────────────────────────────────────────────────────────┐
│  STW │  根扫描  │  更新RS  │  标记存活  │  复制对象  │  清理  │
│  ███ │   ▓▓    │   ▓▓    │    ▓▓     │    ▓▓     │   ▓▓  │
│      │         │         │           │           │       │
│      └──────── 全部在 STW 中并行执行 ──────────────────────┘
│      耗时取决于存活对象数量和复制时间                          │
└─────────────────────────────────────────────────────────────┘

Mixed GC:
┌─────────────────────────────────────────────────────────────┐
│  初始标记 │ 根区域扫描 │ 并发标记 │ 最终标记 │ 筛选回收      │
│   STW   │   并发    │   并发   │   STW   │   STW        │
│   ███   │    ▓▓▓   │   ▓▓▓   │   ███   │   ███        │
│  (很快)  │          │         │ (较长)   │ (复制算法)    │
└─────────────────────────────────────────────────────────────┘
```

#### ZGC（几乎无 STW）

```
┌─────────────────────────────────────────────────────────────┐
│  并发标记  │  并发预备  │  并发重分配  │  并发重映射         │
│   ▓▓▓    │    ▓▓▓   │     ▓▓▓     │     ▓▓▓           │
│  (并发)   │   (并发)   │    (并发)    │    (并发)          │
│                                                              │
│  只有极短的停顿（< 1ms）：                                     │
│  - 初始标记：标记根节点                                        │
│  - 最终标记：处理 SATB 队列                                    │
└─────────────────────────────────────────────────────────────┘
```

### 2. 触发 STW 的具体阶段详解

| 阶段 | 作用 | 为什么必须 STW |
|------|------|---------------|
| **根节点枚举** | 标记 GC Roots | 必须获得一致的根集合快照 |
| **初始标记** | 标记根直接引用 | 避免并发修改导致漏标 |
| **最终标记** | 处理 SATB/增量更新 | 修正并发期间的引用变更 |
| **对象复制** | 复制存活对象到新区域 | 必须保证对象完整性 |
| **内存整理** | 压缩内存消除碎片 | 移动对象时不能访问 |

```java
// GC Roots 包括：
// 1. 虚拟机栈中的本地变量
// 2. 方法区中类静态属性
// 3. 方法区中常量
// 4. 本地方法栈中的 JNI 引用
// 5. 所有被同步锁持有的对象

public class GCRootsDemo {
    private static Object staticObj = new Object();  // GC Root
    
    public void method() {
        Object localVar = new Object();  // GC Root（栈中）
        // ...
    }  // localVar 不再是 GC Root，对象可回收
}
```

## 🔥 扩展追问 1：如何减少 STW 时间？

### 方案 1：选择合适的收集器

```bash
# 小堆/低延迟要求 → ZGC
-XX:+UseZGC
-XX:ZCollectionInterval=5

# 中等堆/平衡需求 → G1
-XX:+UseG1GC
-XX:MaxGCPauseMillis=100  # 设置目标停顿时间

# 避免使用 Serial/Parallel（全阶段 STW）
```

### 方案 2：调整堆内存大小

```bash
# 原则：堆越小，GC 越快，但频率越高
#       堆越大，GC 越慢，但频率越低

-Xms4g -Xmx4g              # 固定堆大小，避免动态调整
-Xmn1g                     # 新生代大小（G1 不需要）
-XX:MetaspaceSize=256m     # 元空间初始大小

# G1 特定调优
-XX:G1HeapRegionSize=16m   # Region 大小
-XX:InitiatingHeapOccupancyPercent=35  # 提前触发并发标记
```

### 方案 3：优化对象分配

```java
// ❌ 避免：大量短生命周期对象
public void bad() {
    for (int i = 0; i < 1000000; i++) {
        byte[] data = new byte[1024];  // 每次循环都分配
        process(data);
    }  // 产生大量垃圾，频繁 Young GC
}

// ✅ 推荐：对象复用
public void good() {
    byte[] buffer = new byte[1024];  // 复用缓冲区
    for (int i = 0; i < 1000000; i++) {
        Arrays.fill(buffer, (byte) 0);
        process(buffer);
    }
}
```

### 方案 4：避免大对象

```java
// ❌ 避免：大对象直接进入老年代
byte[] hugeData = new byte[100 * 1024 * 1024];  // 100MB

// ✅ 推荐：分块处理
public void processLargeData(InputStream in) throws IOException {
    byte[] buffer = new byte[8192];  // 8KB 缓冲区
    int bytesRead;
    while ((bytesRead = in.read(buffer)) != -1) {
        processChunk(buffer, bytesRead);
    }
}
```

### 方案 5：使用对象池

```java
// 高并发场景下，使用对象池减少分配
public class ObjectPool<T> {
    private final Queue<T> pool;
    private final Supplier<T> factory;
    
    public T borrow() {
        T obj = pool.poll();
        return obj != null ? obj : factory.get();
    }
    
    public void release(T obj) {
        pool.offer(obj);
    }
}

// 使用
ObjectPool<byte[]> bufferPool = new ObjectPool<>(
    () -> new byte[8192],
    100  // 池大小
);
```

## 🔥 扩展追问 2：并发标记的算法（三色标记）

```
┌─────────────────────────────────────────────────────────────┐
│                     三色标记法                               │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  白色：未访问的对象（潜在的垃圾）                             │
│  灰色：已访问，但引用未扫描完的对象                           │
│  黑色：已访问，引用已扫描完的对象（存活）                     │
│                                                              │
│  初始状态：根节点为灰色，其他为白色                           │
│  结束状态：黑色为存活，白色为垃圾                             │
│                                                              │
└─────────────────────────────────────────────────────────────┘

并发标记的问题：漏标（对象丢失）
┌─────────────────────────────────────────────────────────────┐
│  情况：黑色对象引用白色对象，灰色对象失去对白色对象的引用       │
│                                                              │
│  标记前：      标记中（并发）：      结果：                   │
│                                                              │
│    灰色          黑色              黑色                      │
│    /  \          /                /                        │
│  白色  B        白色              B（白色被漏标！）           │
│                                                              │
│  灰色断开对白色的引用，黑色新增对白色的引用                    │
│  但黑色不会再扫描，白色被误认为是垃圾                          │
└─────────────────────────────────────────────────────────────┘
```

**解决方案：**

```java
// 1. CMS：增量更新（Incremental Update）
// 当黑色对象新增白色引用时，将黑色重新标记为灰色
// 缺点：需要重新扫描，增加开销

// 2. G1：SATB（Snapshot At The Beginning）
// 在并发标记开始时建立快照
// 当引用被修改时，记录原始引用关系
// 优点：不需要重新扫描黑色对象

// 3. ZGC：染色指针 + 读屏障
// 访问对象时检查染色位
// 自动完成指针的自愈（remap）
```

// 代码中检测长停顿
public class STWDetector {
    private static final Thread detector = new Thread(() -> {
        long lastTime = System.currentTimeMillis();
        while (!Thread.interrupted()) {
            long current = System.currentTimeMillis();
            long gap = current - lastTime;
            if (gap > 1000) {  // 超过 1 秒认为有 STW
                System.err.println("Detected STW: " + gap + "ms");
            }
            lastTime = current;
            try {
                Thread.sleep(100);
            } catch (InterruptedException e) {
                break;
            }
        }
    });
    
    static {
        detector.setDaemon(true);
        detector.start();
    }
}

## 🔥 扩展追问 4：安全点（Safepoint）与安全区域

```java
// STW 不是立即停止所有线程，而是等待线程到达安全点

// 安全点：线程可以暂停的位置
// - 方法调用返回前
// - 循环迭代末尾
// - 抛出异常时

// 问题：长时间运行的循环可能不进入安全点
public class LongLoop {
    public void count() {
        long sum = 0;
        // ❌ 大计数循环，可能长时间不进入安全点
        for (int i = 0; i < Integer.MAX_VALUE; i++) {
            sum += i;
        }
    }
    
    public void countSafe() {
        long sum = 0;
        // ✅ 使用 long，JVM 会在每次迭代检查安全点
        for (long i = 0; i < Integer.MAX_VALUE; i++) {
            sum += i;
        }
    }
}
```

## 一句话收尾

STW 是 GC 为保证内存一致性而暂停应用线程的机制，**新生代收集、根节点枚举、对象复制**等阶段需要 STW；减少 STW 的核心是**选择 ZGC/G1 低延迟收集器、控制堆大小、优化对象分配、避免大对象和长生命期对象**。
