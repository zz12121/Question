## 先说结论

方法区是 JVM 规范定义的**运行时数据区域**，存储类元数据、常量池、静态变量等；JDK 8 用 **Metaspace（本地内存）** 替换 **PermGen（永久代，堆内存）**，核心原因是**解决 PermGen 大小固定导致的 OOM 问题**，同时简化 GC 复杂度。

## 🔥 详细展开

### 1. 方法区存储的数据

| 数据类型 | 具体说明 | 示例 |
|---------|---------|------|
| **类型信息** | 类/接口/枚举/注解的完整描述 | 全限定名、修饰符、父类、接口列表 |
| **常量池** | 编译期生成的各种字面量和符号引用 | `String s = "abc"` 中的 `"abc"` |
| **字段信息** | 字段名、类型、修饰符 | `private int age` |
| **方法信息** | 方法名、返回类型、参数、修饰符 | `public void sayHello()` |
| **静态变量** | JDK 7 后移至堆中 | `static int count = 0` |
| **JIT 编译代码** | 热点代码的本地机器码缓存 | 被频繁调用的方法编译后的代码 |

```java
public class MethodAreaDemo {
    // 1. 类型信息: MethodAreaDemo 类的元数据
    // 2. 常量池: "Hello", "World" 等字符串常量
    // 3. 字段信息: message 字段的描述
    // 4. 方法信息: main(), sayHello() 的描述
    
    private String message = "Hello";  // 实例变量在堆中
    
    public static void main(String[] args) {
        // "World" 进入字符串常量池
        String world = "World";
        System.out.println(world);
    }
    
    public void sayHello() {
        System.out.println(message);
    }
}
```

### 2. PermGen vs Metaspace 对比

| 特性 | PermGen (JDK 7-) | Metaspace (JDK 8+) |
|------|------------------|-------------------|
| **内存位置** | JVM 堆内存 | 本地内存（Native Memory） |
| **默认大小** | 64MB (Server) / 32MB (Client) | 无上限（受限于系统内存） |
| **调整参数** | `-XX:PermSize` / `-XX:MaxPermSize` | `-XX:MetaspaceSize` / `-XX:MaxMetaspaceSize` |
| **垃圾回收** | Full GC 回收 | 单独的 GC 策略 |
| **存储内容** | 类元数据 + 字符串常量池 + 静态变量 | 仅类元数据 |
| **OOM 类型** | `java.lang.OutOfMemoryError: PermGen space` | `java.lang.OutOfMemoryError: Metaspace` |

### 3. JDK 8 为什么要替换 PermGen？

#### 原因 1：固定大小导致 OOM

```java
// 典型 PermGen OOM 场景：动态生成大量类
// 如 CGLIB 动态代理、反射、Groovy/Scala 等动态语言

// CGLIB 动态生成代理类
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(Target.class);
enhancer.setCallback((MethodInterceptor) (obj, method, args, proxy) -> {
    return proxy.invokeSuper(obj, args);
});

// 循环创建大量代理类
for (int i = 0; i < 100_000; i++) {
    enhancer.setCallbackFilter(new CallbackFilter() {
        @Override
        public int accept(Method method) {
            return 0;
        }
    });
    Class<?> proxyClass = enhancer.createClass();  // 每个类都占用 PermGen
}
// 结果: java.lang.OutOfMemoryError: PermGen space
```

#### 原因 2：Full GC 性能问题

```
PermGen 的问题：
- 和老年代一起进行 Full GC，停顿时间长
- 类卸载条件苛刻（类加载器、所有实例、Class对象都需可回收）

Metaspace 的改进：
- 使用独立的垃圾回收策略
- 更高效的类元数据管理
- 延迟分配，按需增长
```

#### 原因 3：与 JRockit/HotSpot 融合

```
Oracle 收购 BEA（JRockit）后，需要统一 JVM 代码库：
- JRockit 没有 PermGen 概念
- 移除 PermGen 可以简化代码，减少维护成本
- 为未来的模块化（JPMS）做准备
```

#### 原因 4：字符串常量池和静态变量迁移

```java
// JDK 6: 字符串常量池在 PermGen
// JDK 7: 字符串常量池移至堆中（-XX:StringTableSize 调整）
// JDK 8: 静态变量也移至堆中

public class StringPoolDemo {
    public static void main(String[] args) {
        String s1 = new String("hello") + new String("world");
        s1.intern();  // JDK 6: 复制到 PermGen 常量池
                      // JDK 7+: 在堆中维护常量池引用
        
        String s2 = "helloworld";
        System.out.println(s1 == s2);  // JDK 6: false, JDK 7+: true
    }
}
```

### 4. Metaspace 的配置与调优

```bash
# 常用 JVM 参数
-XX:MetaspaceSize=128m          # 初始空间大小

-XX:MaxMetaspaceFreeRatio=70    # GC后最大空闲比例
```

```java
// 监控 Metaspace 使用
public class MetaspaceMonitor {
    public static void main(String[] args) {
        // 获取内存池信息
        for (MemoryPoolMXBean pool : ManagementFactory.getMemoryPoolMXBeans()) {
            if (pool.getName().contains("Metaspace")) {
                MemoryUsage usage = pool.getUsage();
                System.out.println("Metaspace 使用: " + usage.getUsed() / 1024 / 1024 + " MB");
                System.out.println("Metaspace 提交: " + usage.getCommitted() / 1024 / 1024 + " MB");
                System.out.println("Metaspace 最大: " + usage.getMax() / 1024 / 1024 + " MB");
            }
        }
    }
}
```

## 🔥 扩展追问 2：运行时常量池 vs 字符串常量池

| 常量池 | 位置 | 存储内容 |
|-------|------|---------|
| **Class 文件常量池** | `.class` 文件中 | 字面量、符号引用（编译期） |
| **运行时常量池** | 方法区/Metaspace | Class 文件常量池的运行时表示 |
| **字符串常量池** | 堆中（JDK 7+） | 字符串字面量实例 |

```java
public class ConstantPoolDemo {
    public static void main(String[] args) {
        String s1 = "abc";           // 进入字符串常量池（堆）
        String s2 = new String("abc"); // 堆中新建对象
        String s3 = s2.intern();      // 返回常量池中的引用
        
        System.out.println(s1 == s3);  // true
        System.out.println(s1 == s2);  // false
        
        // 运行时常量池中的符号引用
        // #1 = Methodref          #6.#15         // java/lang/Object."<init>":()V
        // #2 = String             #16            // abc
    }
}
```

## 🔥 扩展追问 4：类卸载的条件

```java
// JVM 卸载类的条件（非常严格）：
// 1. 该类所有实例已被回收
// 2. 加载该类的 ClassLoader 已被回收
// 3. 该类的 Class 对象没有在任何地方被引用

// 典型场景：OSGi 热部署、Tomcat 热加载
public class ClassUnloadDemo {
    public static void main(String[] args) throws Exception {
        URLClassLoader loader = new URLClassLoader(
            new URL[]{new File("target/classes").toURI().toURL()},
            null  // 父加载器设为 null，隔离
        );
        
        Class<?> clazz = loader.loadClass("com.example.TempClass");
        Object instance = clazz.getDeclaredConstructor().newInstance();
        
        // 释放引用
        instance = null;
        clazz = null;
        loader.close();  // JDK 7+ 需要显式关闭
        loader = null;
        
        System.gc();  // 建议 GC，但不保证立即卸载
    }
}
```

## 一句话收尾

方法区存储类元数据，JDK 8 用 **Metaspace（本地内存）** 替换 **PermGen（堆内存）** 解决了固定大小导致的 OOM 问题，同时将字符串常量池和静态变量移至堆中，让 GC 管理更高效、更灵活。
