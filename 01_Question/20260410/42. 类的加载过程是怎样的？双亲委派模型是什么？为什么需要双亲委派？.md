## 先说结论

类的加载过程是**加载 → 验证 → 准备 → 解析 → 初始化**五个阶段；双亲委派模型是指类加载器优先把请求委派给父加载器，只有当父加载器无法完成时才自己加载。它的核心作用是**保证类的唯一性和安全性**，防止核心类库被篡改。

---

## 🔥 核心概念

### 类的生命周期

```
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
     ↑_________________________|
           (连接阶段)
```

### 双亲委派模型层级

```
      ┌─────────────────┐
      │   Bootstrap     │  ← 启动类加载器 (C++实现，加载<JAVA_HOME>/lib)
      │   ClassLoader   │     加载: java.lang.*, java.util.* 等核心类
      └────────┬────────┘
               │
      ┌────────▼────────┐
      │   Extension     │  ← 扩展类加载器 (Java实现，加载<JAVA_HOME>/lib/ext)
      │   ClassLoader   │     加载: javax.*, 扩展类库
      └────────┬────────┘
               │
      ┌────────▼────────┐
      │   Application   │  ← 应用类加载器 (加载classpath下的类)
      │   ClassLoader   │     加载: 我们自己写的类、第三方依赖
      └────────┬────────┘
               │
      ┌────────▼────────┐
      │    Custom       │  ← 自定义类加载器 (Tomcat、Spring Boot等)
      │   ClassLoader   │     加载: 热部署、模块隔离等场景
      └─────────────────┘
```

## 🔥 详细展开

### 1. 类加载的五个阶段

#### ① 加载 (Loading)
- 通过**全限定名**获取二进制字节流
- 将字节流转化为方法区的**运行时数据结构**
- 在堆中生成 `Class` 对象，作为方法区数据的访问入口

```java
// 触发加载的时机
Class<?> clazz = Class.forName("com.example.User");  // 显式加载
User user = new User();  // 隐式加载（new 关键字）
```

#### ② 验证 (Verification)
确保字节码是**安全、合法、符合JVM规范**的：
- **文件格式验证**：魔数、版本号、常量池等
- **元数据验证**：类的继承关系、抽象方法实现等
- **字节码验证**：控制流分析、类型转换合法性
- **符号引用验证**：能否找到对应的类/方法/字段

#### ③ 准备 (Preparation)
- 为**类变量（static）分配内存**并设置**默认值**
- **注意**：`final static` 常量在此阶段直接赋初始值

```java
public class Demo {
    private static int a = 10;      // 准备阶段: a = 0; 初始化阶段: a = 10
    private static final int B = 20; // 准备阶段: B = 20 (直接赋初值)
}
```

#### ④ 解析 (Resolution)
将**符号引用**转换为**直接引用**：
- 符号引用：字符串形式的描述（如 `java/lang/Object`）
- 直接引用：内存地址指针或偏移量

```java
// 符号引用示例
invokevirtual #4  // #4 是常量池中的符号引用，指向 java/io/PrintStream.println

// 解析后变为直接引用（内存地址）
invokevirtual 0x7f8a2b3c  // 指向 PrintStream.println 的实际方法地址
```

#### ⑤ 初始化 (Initialization)
执行 **`<clinit>()` 方法**：
- 编译器自动收集所有 `static` 变量赋值和 `static{}` 块
- 按**源码顺序**执行
- **父类的 `<clinit>()` 先执行**

```java
public class InitDemo {
    static {
        System.out.println("static block 1");
    }
    
    private static int a = 10;
    
    static {
        System.out.println("static block 2, a=" + a);
    }
    
    // 编译后生成 <clinit>() 方法，执行顺序:
    // 1. static block 1
    // 2. a = 10
    // 3. static block 2
}
```

### 2. 双亲委派模型详解

#### 核心代码 (ClassLoader.loadClass)

```java
protected Class<?> loadClass(String name, boolean resolve) 
    throws ClassNotFoundException {
    
    synchronized (getClassLoadingLock(name)) {
        // 1. 检查类是否已加载
        Class<?> c = findLoadedClass(name);
        
        if (c == null) {
            // 2. 委派给父加载器
            if (parent != null) {
                c = parent.loadClass(name, false);
            } else {
                c = findBootstrapClassOrNull(name);  // 委派到启动类加载器
            }
            
            // 3. 父加载器无法加载，自己加载
            if (c == null) {
                c = findClass(name);  // 子类实现，如 URLClassLoader
            }
        }
        
        if (resolve) {
            resolveClass(c);
        }
        return c;
    }
}
```

#### 为什么需要双亲委派？

| 优势 | 说明 |
|------|------|
| **避免重复加载** | 先委托父加载器，确保同一个类只被加载一次 |
| **保证安全性** | 核心类库（`java.lang.*`）只能由启动类加载器加载，防止恶意篡改 |
| **保证扩展性** | 父加载器加载的类对子加载器可见，反之不行 |

#### 破坏双亲委派的场景

```java
// 场景1: JDBC (SPI机制)
// DriverManager 由 BootstrapClassLoader 加载
// 但具体驱动实现由 AppClassLoader 加载
// 需要 Thread.currentThread().getContextClassLoader() 逆向委派

// 场景2: Tomcat 热部署
// 每个 Web 应用有自己的类加载器，优先加载自己的类
// 破坏双亲委派实现应用隔离

// 场景3: OSGi 模块化
// 每个 Bundle 有自己的类加载器，形成网状结构
```

## 🔥 扩展追问 1：如何自定义类加载器？

```java
public class CustomClassLoader extends ClassLoader {
    
    private String classPath;
    
    public CustomClassLoader(String classPath) {
        this.classPath = classPath;
    }
    
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        // 1. 加载字节码
        byte[] classData = loadClassData(name);
        
        // 2. 定义类
        return defineClass(name, classData, 0, classData.length);
    }
    
    private byte[] loadClassData(String name) {
        // 从自定义路径加载 .class 文件
        String path = classPath + File.separatorChar 
                     + name.replace('.', File.separatorChar) + ".class";
        
        try (InputStream is = new FileInputStream(path);
             ByteArrayOutputStream baos = new ByteArrayOutputStream()) {
            
            int bufferSize = 1024;
            byte[] buffer = new byte[bufferSize];
            int length;
            while ((length = is.read(buffer)) != -1) {
                baos.write(buffer, 0, length);
            }
            return baos.toByteArray();
            
        } catch (IOException e) {
            throw new RuntimeException(e);
        }
    }
}

// 使用
CustomClassLoader loader = new CustomClassLoader("/path/to/classes");
Class<?> clazz = loader.loadClass("com.example.MyClass");
```

## 🔥 扩展追问 2：打破双亲委派的应用场景

### Tomcat 类加载器架构

```
        Bootstrap
           │
       CommonLoader  ← 共享类库 (Servlet API等)
       /        \
  Catalina     Shared
  (Tomcat内核)    │
              Webapp1  Webapp2  Webapp3
              (应用隔离，优先加载自己的类)
```

```java
// Tomcat 的 WebAppClassLoader 核心逻辑
@Override
public Class<?> loadClass(String name) throws ClassNotFoundException {
    // 1. 先自己加载（破坏双亲委派）
    Class<?> clazz = findLoadedClass(name);
    if (clazz == null) {
        clazz = findClass(name);  // 先尝试自己加载
    }
    
    // 2. 自己加载不到，再委托父加载器
    if (clazz == null) {
        clazz = super.loadClass(name);
    }
    return clazz;
}
```

**为什么打破？**
- 每个 Web 应用可能依赖不同版本的 Spring、Hibernate
- 需要**应用级别隔离**，避免类冲突
- 支持**热部署**（卸载旧的类加载器，创建新的）

## 🔥 扩展追问 3：类加载的时机（主动引用 vs 被动引用）

```java
public class SuperClass {
    static {
        System.out.println("SuperClass init!");
    }
    public static int value = 123;
    public static final int CONST = 456;  // 编译期常量
}

public class SubClass extends SuperClass {
    static {
        System.out.println("SubClass init!");
    }
}

// ========== 主动引用（触发初始化）==========
new SubClass();           // 输出: SuperClass init! → SubClass init!
SubClass.value;           // 输出: SuperClass init!
Class.forName("SubClass"); // 输出: SuperClass init! → SubClass init!

// ========== 被动引用（不触发初始化）==========
SubClass.CONST;           // 不输出！编译期常量直接替换
SubClass[] array = new SubClass[10];  // 不输出！数组类型由JVM生成
System.out.println(SuperClass.value); // 只输出 SuperClass init!
```

## 一句话收尾

类的加载是**加载→验证→准备→解析→初始化**五部曲，双亲委派通过**向上委托、向下加载**保证类唯一性和安全性，但在需要**热部署、版本隔离**的场景下会主动打破这一模型。
