## 先说结论

Redis 哨兵（Sentinel）提供**监控、通知、自动故障转移、配置提供者**四大功能；故障转移流程包括**主观下线 → 客观下线 → 选举 Leader → 选择新主节点 → 故障迁移**，通过 Raft 算法保证多哨兵一致性。

## 🔥 详细展开

### 1. 哨兵的四大功能

| 功能 | 说明 | 实现方式 |
|------|------|---------|
| **监控** | 监控主从节点健康状态 | 定期 PING 检测 |
| **通知** | 故障时通知管理员或其他应用 | 通过 API 发送通知 |
| **自动故障转移** | 主节点故障时自动切换 | 选举 + 切换流程 |
| **配置提供者** | 为客户端提供主节点地址 | 查询 Sentinel 获取 |

### 2. 故障检测机制

#### 主观下线（SDOWN, Subjective Down）

```bash
# 单个 Sentinel 认为节点下线
# 条件：在 down-after-milliseconds 内无响应

# 配置
sentinel down-after-milliseconds mymaster 5000
# 5 秒内无响应，标记为主观下线
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    主观下线判定                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Sentinel ──PING──► Master                                       │
│       │         ◄──NO PONG──                                     │
│       │         (超时 5s)                                        │
│       ▼                                                          │
│  ┌─────────────────┐                                             │
│  │ 标记 SDOWN      │                                             │
│  │ +sdown master   │                                             │
│  │  mymaster       │                                             │
│  │  192.168.1.10   │                                             │
│  └─────────────────┘                                             │
│                                                                  │
│  注意：这只是单个 Sentinel 的主观判断，                           │
│        可能是网络抖动或 Sentinel 自身问题                         │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

#### 客观下线（ODOWN, Objective Down）

```bash
# 多个 Sentinel 达成一致，确认节点真正下线
# 需要满足法定人数（quorum）

# 配置
sentinel monitor mymaster 192.168.1.10 6379 2
# 最后参数 2 表示需要 2 个 Sentinel 同意才判定客观下线
```

```
┌─────────────────────────────────────────────────────────────────┐
│                    客观下线判定                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Sentinel 1 发现 SDOWN                                           │
│       │                                                          │
│       ▼                                                          │
│  向其他 Sentinel 询问：                                          │
│  "你们也认为 master 下线了吗？"                                    │
│       │                                                          │
│       ▼                                                          │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐                          │
│  │Sentinel2│  │Sentinel3│  │Sentinel4│                          │
│  │  SDOWN  │  │  OK     │  │  SDOWN  │                          │
│  └────┬────┘  └────┬────┘  └────┬────┘                          │
│       │            │            │                                │
│       └────────────┴────────────┘                                │
│                  ↓                                               │
│  2 个 Sentinel 同意（>= quorum=2）                                │
│       ↓                                                          │
│  ┌─────────────────┐                                             │
│  │ 标记 ODOWN      │                                             │
│  │ +odown master   │                                             │
│  │  mymaster       │                                             │
│  │  #quorum 2/3    │                                             │
│  └─────────────────┘                                             │
│       ↓                                                          │
│  开始故障转移流程                                                  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3. 故障转移完整流程

```
┌─────────────────────────────────────────────────────────────────┐
│                  故障转移完整流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Step 1: 选举 Sentinel Leader                                    │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  使用 Raft 算法选举 Leader Sentinel                        │   │
│  │  • 每个发现 ODOWN 的 Sentinel 都想成为 Leader              │   │
│  │  • 向其他 Sentinel 发送 "我要当 Leader" 请求               │   │
│  │  • 获得多数票（超过半数）的 Sentinel 成为 Leader           │   │
│  │  • 如果平票，等待随机时间后重试                            │   │
│  └──────────────────────────────────────────────────────────┘   │
│       ↓                                                          │
│  Step 2: 选择新主节点                                            │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  从 Slave 中选择最优的一个作为新主节点：                    │   │
│  │                                                          │   │
│  │  筛选条件：                                               │   │
│  │  1. 排除已经下线的 Slave                                  │   │
│  │  2. 排除最近 5 秒内没有回复 PING 的 Slave                  │   │
│  │  3. 排除与旧主节点断开连接超过 down-after-milliseconds     │   │
│  │     的 Slave（数据太旧）                                   │   │
│  │                                                          │   │
│  │  排序规则：                                               │   │
│  │  1. 优先级（replica-priority）小的优先                     │   │
│  │  2. 复制偏移量大的优先（数据新）                            │   │
│  │  3. Run ID 小的优先                                        │   │
│  └──────────────────────────────────────────────────────────┘   │
│       ↓                                                          │
│  Step 3: 提升新主节点                                            │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  1. 向选中的 Slave 发送 SLAVEOF NO ONE                    │   │
│  │     使其变为主节点                                         │   │
│  │  2. 等待新主节点确认晋升成功                                │   │
│  └──────────────────────────────────────────────────────────┘   │
│       ↓                                                          │
│  Step 4: 重新配置从节点                                          │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  向其他 Slave 发送 SLAVEOF <new_master_ip> <port>         │   │
│  │  让它们成为新主节点的从节点                                  │   │
│  └──────────────────────────────────────────────────────────┘   │
│       ↓                                                          │
│  Step 5: 更新配置                                                │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  1. 更新 Sentinel 内部的 master 配置                       │   │
│  │  2. 通过 pub/sub 通知客户端主节点已变更                     │   │
│  │  3. 旧主节点恢复后，将其配置为新主节点的从节点               │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 4. 哨兵配置示例

```bash
# sentinel.conf

# 监控主节点
# sentinel monitor <master-name> <ip> <port> <quorum>
sentinel monitor mymaster 192.168.1.10 6379 2

# 主观下线时间（毫秒）
sentinel down-after-milliseconds mymaster 5000

# 故障转移超时时间
sentinel failover-timeout mymaster 60000

# 并行同步的从节点数量
sentinel parallel-syncs mymaster 1

# 从节点优先级（数字小的优先成为主节点）
# 0 表示永远不会成为主节点
sentinel replica-priority mymaster 100

# 认证信息
sentinel auth-pass mymaster mypassword

# 通知脚本
sentinel notification-script mymaster /var/redis/notify.sh

# 客户端重新配置脚本
sentinel client-reconfig-script mymaster /var/redis/reconfig.sh
```

## 🔥 扩展追问 3：脑裂问题及解决

```
脑裂场景：
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  网络分区前：                                                     │
│  ┌─────────┐         ┌─────────┐         ┌─────────┐            │
│  │ Sentinel│◄───────►│ Sentinel│◄───────►│ Sentinel│            │
│  │    A    │         │    B    │         │    C    │            │
│  └────┬────┘         └────┬────┘         └────┬────┘            │
│       │                   │                   │                  │
│       └───────────────────┴───────────────────┘                  │
│                           │                                      │
│                      ┌────┴────┐                                 │
│                      │  Master │                                 │
│                      └────┬────┘                                 │
│                           │                                      │
│                      ┌────┴────┐                                 │
│                      │  Slave  │                                 │
│                      └─────────┘                                 │
│                                                                  │
│  网络分区后：                                                     │
│  Zone 1:              Zone 2:                                    │
│  ┌─────────┐         ┌─────────┐                                 │
│  │ Sentinel│         │ Sentinel│                                 │
│  │    A    │         │    B,C  │                                 │
│  └────┬────┘         └────┬────┘                                 │
│       │                   │                                      │
│  ┌────┴────┐         ┌────┴────┐                                 │
│  │  Master │         │  Slave  │  ← B,C 选举 Slave 为新主节点     │
│  │ (旧主)  │         │ (新主)  │                                 │
│  └─────────┘         └─────────┘                                 │
│                                                                  │
│  结果：两个主节点同时存在，数据不一致！                             │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**解决方案：**

```bash
# 配置最小从节点数
min-replicas-to-write 1      # 至少 1 个从节点连接才允许写
min-replicas-max-lag 10      # 从节点延迟不超过 10 秒

# 这样旧主节点在分区后没有从节点连接，拒绝写入
# 避免脑裂期间的数据丢失
```

## 🔥 扩展追问 4：哨兵 vs Cluster 模式选择

| 特性 | Sentinel | Cluster |
|------|----------|---------|
| **架构** | 主从复制 + 哨兵监控 | 分片集群 |
| **数据分片** | 不支持 | 支持（16384 槽） |
| **自动故障转移** | 支持 | 支持 |
| **水平扩展** | 不支持（只能垂直扩展） | 支持 |
| **读写分离** | 支持 | 支持 |
| **部署复杂度** | 较低 | 较高 |
| **适用场景** | 数据量小、高可用需求 | 大数据量、高并发 |

---

## 一句话收尾

Redis 哨兵通过**多节点投票**实现客观下线判定，使用 **Raft 算法选举 Leader** 协调故障转移，流程包括**选主、晋升、重配从节点**；部署时至少**3 个哨兵节点**并配置 **min-replicas** 防止脑裂。
