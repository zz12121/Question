## 先说结论

**慢查询**是执行时间超过阈值的 SQL 语句，通过 `slow_query_log` 开启日志记录；分析慢查询的核心工具是 **`EXPLAIN` 分析执行计划**、**`mysqldumpslow`/`pt-query-digest` 聚合分析**，优化方向包括**索引优化、SQL 改写、架构调整**。

## 🔥 核心概念

### 慢查询判定标准

```
┌─────────────────────────────────────────────────────────────────┐
│                    慢查询判定条件                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  满足以下任一条件即记录为慢查询：                                 │
│                                                                  │
│  1. 查询时间 > long_query_time（默认 10 秒）                     │
│     SET GLOBAL long_query_time = 1;  -- 超过 1 秒即慢查询         │
│                                                                  │
│  2. 未使用索引且记录到 log_queries_not_using_indexes             │
│     即使查询很快，没走索引也记录                                  │
│                                                                  │
│  3. 管理语句（ALTER/ANALYZE 等）超过阈值                          │
│     log_slow_admin_statements = ON                               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 🔥 详细展开

### 1. 开启慢查询日志

#### 临时开启（运行时）

```sql
-- 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';

-- 设置慢查询时间阈值（秒）
SET GLOBAL long_query_time = 1;

-- 设置日志文件路径
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';

-- 记录未使用索引的查询
SET GLOBAL log_queries_not_using_indexes = 'ON';

-- 查看配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query%';
SHOW VARIABLES LIKE 'log_queries_not_using_indexes';
```

```ini
# /etc/mysql/my.cnf 或 /etc/my.cnf

[mysqld]

# 慢查询日志文件路径

# 记录未使用索引的查询
log_queries_not_using_indexes = 1

# 记录慢管理语句（如 ALTER、ANALYZE）
```

### 2. 慢查询日志格式

```sql
# Time: 2024-01-15T10:30:45.123456Z
# User@Host: root[root] @ localhost []  Id:   123
# Query_time: 5.234567  Lock_time: 0.001234  Rows_sent: 100  Rows_examined: 1000000
SET timestamp=1705317045;
SELECT o.order_id, o.amount, u.username 
FROM orders o 
JOIN user u ON o.user_id = u.id 
WHERE o.create_time > '2024-01-01' 
ORDER BY o.amount DESC 
LIMIT 100;
```

**字段解读：**

| 字段 | 含义 | 分析要点 |
|------|------|---------|
| `Time` | 查询发生时间 | 定位问题时段 |
| `User@Host` | 执行用户和来源 | 识别异常来源 |
| `Query_time` | 查询总耗时 | 核心指标 |
| `Lock_time` | 等待锁的时间 | 锁竞争问题 |
| `Rows_sent` | 返回客户端的行数 | 应远小于 Rows_examined |
| `Rows_examined` | 扫描的行数 | 越大说明索引效果越差 |

### 3. 慢查询分析方法

#### 方法 1：直接查看日志

```bash
# 查看最新慢查询
tail -f /var/log/mysql/slow.log

# 统计慢查询数量
grep -c "Query_time" /var/log/mysql/slow.log

# 找出最慢的 10 条
grep -A 1 "Query_time" /var/log/mysql/slow.log | grep -v "^--$" | sort -t: -k2 -nr | head -20
```

#### 方法 2：mysqldumpslow（官方工具）

```bash
# 按出现次数排序，显示前 10 条
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log

# 按平均执行时间排序
mysqldumpslow -s at -t 10 /var/log/mysql/slow.log

# 按总执行时间排序
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log

# 只显示特定表的查询
mysqldumpslow -g "orders" /var/log/mysql/slow.log

# 常用参数说明：
# -s: 排序方式 (c=次数, t=总时间, at=平均时间, l=锁定时间, r=返回行数)
# -t: 显示前 N 条
# -g: 正则过滤
```

**输出示例：**

```bash
Reading mysql slow query log from /var/log/mysql/slow.log
Count: 100  Time=5.23s (523s total)  Lock=0.01s (1s total)  Rows=100.0 (10000), root[root]@localhost
  SELECT * FROM orders WHERE user_id = N ORDER BY create_time DESC LIMIT N;
```

#### 方法 3：pt-query-digest（Percona Toolkit）

```bash
# 安装 Percona Toolkit
sudo apt-get install percona-toolkit

pt-query-digest --since 1h /var/log/mysql/slow.log
```

### 4. 使用 EXPLAIN 分析执行计划

```sql
-- 分析慢查询的执行计划
EXPLAIN SELECT * FROM orders WHERE user_id = 100 AND status = 1;

-- 详细格式（MySQL 5.6+）
EXPLAIN FORMAT=JSON SELECT * FROM orders WHERE user_id = 100;

-- 实际执行（MySQL 5.6+）
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 100;
```

**EXPLAIN 关键字段：**

| 字段 | 含义 | 优化建议 |
|------|------|---------|
| `type` | 访问类型 | 从优到差：system > const > eq_ref > ref > range > index > ALL |
| `possible_keys` | 可能使用的索引 | 为空说明无可用索引 |
| `key` | 实际使用的索引 | 为空说明没走索引 |
| `rows` | 预估扫描行数 | 越小越好 |
| `Extra` | 额外信息 | `Using filesort`、`Using temporary` 需优化 |

```
type 类型详解：
┌──────────┬────────────────────────────────────────────────┐
│ 类型      │ 说明                                           │
├──────────┼────────────────────────────────────────────────┤
│ system   │ 表只有一行（系统表）                            │
│ const    │ 主键或唯一索引等值查询，最多一行                  │
│ eq_ref   │ 主键/唯一索引作为 JOIN 连接条件                  │
│ ref      │ 非唯一索引等值查询                              │
│ range    │ 索引范围查询（BETWEEN、>、<、IN）               │
│ index    │ 全索引扫描（比 ALL 好，但仍需优化）              │
│ ALL      │ 全表扫描（最慢，必须优化）                       │
└──────────┴────────────────────────────────────────────────┘
```

## 🔥 扩展追问 1：常见慢查询优化方案

### 场景 1：无索引导致全表扫描

```sql
-- 慢查询
SELECT * FROM orders WHERE user_id = 100;
-- type: ALL, rows: 1000000, Extra: Using where

-- 优化：添加索引
CREATE INDEX idx_user_id ON orders(user_id);
-- type: ref, rows: 10, key: idx_user_id
```

### 场景 2：索引未生效

```sql
-- 索引失效的情况
SELECT * FROM orders WHERE YEAR(create_time) = 2024;  -- 函数导致失效
SELECT * FROM orders WHERE user_id = '100';           -- 隐式类型转换
SELECT * FROM orders WHERE status = 1 ORDER BY create_time;  -- filesort

-- 优化
SELECT * FROM orders WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';
SELECT * FROM orders WHERE user_id = 100;  -- 确保类型一致
CREATE INDEX idx_status_time ON orders(status, create_time);  -- 联合索引避免排序
```

### 场景 3：深分页问题

```sql
-- 慢查询（深分页）
SELECT * FROM orders ORDER BY id DESC LIMIT 1000000, 10;
-- 需要扫描 1000010 行

-- 优化：延迟关联
SELECT o.* 
FROM orders o
JOIN (SELECT id FROM orders ORDER BY id DESC LIMIT 1000000, 10) tmp 
ON o.id = tmp.id;
-- 子查询只查 id（覆盖索引），再关联查完整数据
```

### 场景 4：大表 JOIN

```sql
-- 慢查询
SELECT o.*, u.username 
FROM orders o
JOIN user u ON o.user_id = u.id
WHERE o.create_time > '2024-01-01';
-- 两张大表 JOIN，产生笛卡尔积

-- 优化 1：确保 JOIN 字段有索引
CREATE INDEX idx_user_id ON orders(user_id);

-- 优化 2：减少数据量后再 JOIN
SELECT o.*, u.username 
FROM (
    SELECT * FROM orders 
    WHERE create_time > '2024-01-01'
) o
JOIN user u ON o.user_id = u.id;
```

## 🔥 扩展追问 2：Performance Schema 分析

```sql
-- MySQL 5.6+ 推荐使用 Performance Schema
-- 开启语句监控
UPDATE performance_schema.setup_consumers 
SET ENABLED = 'YES' 
WHERE NAME LIKE '%events_statements%';

UPDATE performance_schema.setup_instruments 
SET ENABLED = 'YES', TIMED = 'YES' 
WHERE NAME LIKE '%statement/%';

-- 查询执行时间最长的 SQL
SELECT 
    DIGEST_TEXT as query,
    SCHEMA_NAME as db,
    COUNT_STAR as exec_count,
    SUM_TIMER_WAIT/1000000000000 as total_time_sec,
    AVG_TIMER_WAIT/1000000000000 as avg_time_sec,
    SUM_ROWS_SENT as total_rows_sent,
    SUM_ROWS_EXAMINED as total_rows_examined
FROM performance_schema.events_statements_summary_by_digest
ORDER BY SUM_TIMER_WAIT DESC
LIMIT 10;
```

## 🔥 扩展追问 4：慢查询优化 checklist

```
□ 1. 是否走了索引？（EXPLAIN 看 key 字段）
□ 2. 索引是否生效？（避免函数、隐式转换、like '%xxx'）
□ 3. 是否全表扫描？（type = ALL 需优化）
□ 4. 是否需要排序？（Extra 有 filesort 需优化）
□ 5. 是否需要临时表？（Extra 有 temporary 需优化）
□ 6. 扫描行数是否过多？（rows 应远小于表总行数）
□ 7. 返回行数是否合理？（Rows_sent vs Rows_examined）
□ 8. 锁等待时间是否过长？（Lock_time 高说明锁竞争）
□ 9. 是否需要分页优化？（深分页用延迟关联）
□ 10. 是否需要分库分表？（数据量过大考虑架构调整）
```

## 一句话收尾

慢查询是**执行超阈值或未走索引**的 SQL，通过 `slow_query_log` 开启日志，用 **`mysqldumpslow`/`pt-query-digest` 聚合分析** + **`EXPLAIN` 分析执行计划**，优化方向是**加索引、改 SQL、调架构**，核心目标是**降低扫描行数、避免全表扫描、减少额外排序**。
