## 先说结论

Java 异常体系以 `Throwable` 为根，分为 **Error**（系统级错误，不可恢复）和 **Exception**（程序异常，可处理）。Exception 又分 **Checked Exception**（编译期检查，必须处理）和 **Unchecked Exception**（运行时异常，不强制处理）。项目中采用 **分层异常处理策略**：DAO 层抛原始异常、Service 层转业务异常、Controller 层统一拦截，配合 **全局异常处理器** 和 **错误码规范**，避免 try-catch 泛滥和异常吞没。

---

## 🔥 核心概念

**解决什么问题：**
- 程序运行中的意外情况需要被捕获和处理
- 区分可恢复的业务异常和不可恢复的系统错误

**设计哲学：**
- Checked：强制调用方处理，提高健壮性
- Unchecked：简化代码，避免过度防御

## 🔥 详细展开

### 1. Java 异常体系结构

```
Throwable
├── Error（系统级错误，不应捕获）
│   ├── OutOfMemoryError
│   ├── StackOverflowError
│   ├── NoClassDefFoundError
│   └── ...
│
└── Exception（程序异常，可捕获处理）
    ├── Checked Exception（编译期检查）
    │   ├── IOException
    │   ├── SQLException
    │   ├── ClassNotFoundException
    │   └── ...
    │
    └── RuntimeException（Unchecked，运行时）
        ├── NullPointerException
        ├── IllegalArgumentException
        ├── IllegalStateException
        ├── IndexOutOfBoundsException
        └── ...
```

### 2. Checked vs Unchecked 对比

| 特性 | Checked Exception | Unchecked Exception |
|------|-------------------|---------------------|
| **继承** | 直接继承 Exception | 继承 RuntimeException |
| **编译检查** | ✅ 必须处理或声明 | ❌ 不强制 |
| **使用场景** | 可恢复的业务异常 | 编程错误/不可恢复 |
| **处理方法** | try-catch / throws | 可选处理 |
| **代码侵入** | 高（改变方法签名） | 低 |
| **示例** | IOException、SQLException | NPE、IllegalArgument |

**Checked Exception 示例：**

```java
// 必须处理或声明 throws
public void readFile(String path) throws IOException {
    FileReader reader = new FileReader(path);  // 可能抛 FileNotFoundException
    reader.read();                              // 可能抛 IOException
}

// 调用方必须处理
try {
    readFile("test.txt");
} catch (IOException e) {
    // 处理异常
}
```

**Unchecked Exception 示例：**

```java
// 不强制处理
public void divide(int a, int b) {
    if (b == 0) {
        throw new IllegalArgumentException("除数不能为0");
    }
    return a / b;
}

// 调用方可以选择处理，也可以不处理
divide(10, 0);  // 编译通过，运行时才报错
```

### 3. 如何选择异常类型？

**使用 Checked Exception：**
- 调用方可以合理恢复（如重试、使用默认值）
- 异常是方法契约的一部分（如文件不存在）

**使用 Unchecked Exception：**
- 编程错误（空指针、数组越界）
- 无法恢复（配置错误、资源耗尽）
- 简化 API 设计（避免 throws 污染）

**现代 Java 趋势：** 倾向于多用 Unchecked，配合 Optional 和错误码

```java
// 传统 Checked
public User findUser(long id) throws UserNotFoundException;

// 现代 Unchecked + Optional
public Optional<User> findUser(long id);

// 或明确返回结果对象
public Result<User> findUser(long id) {
```

#### 全局异常处理器

```java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {
    
    // 业务异常
    @ExceptionHandler(BizException.class)
    public Result<Void> handleBizException(BizException e) {
        log.warn("业务异常：{}", e.getMessage());
        return Result.fail(e.getErrorCode());
    }
    
    // 参数校验异常
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<Void> handleValidationException(MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getFieldErrors().stream()
            .map(FieldError::getDefaultMessage)
            .collect(Collectors.joining(", "));
        return Result.fail(ErrorCode.PARAM_ERROR, message);
    }
    
    // 其他运行时异常
    @ExceptionHandler(RuntimeException.class)
    public Result<Void> handleRuntimeException(RuntimeException e) {
        log.error("系统异常", e);
        return Result.fail(ErrorCode.SYSTEM_ERROR);
    }
    
    // 兜底
    @ExceptionHandler(Exception.class)
    public Result<Void> handleException(Exception e) {
        log.error("未知异常", e);
        return Result.fail(ErrorCode.SYSTEM_ERROR);
    }
}
```

### 5. 异常处理最佳实践

#### ✅ 应该做的

```java
// 1. 捕获具体异常，不要捕获 Throwable/Exception
try {
    // ...
} catch (IOException e) {  // ✅ 具体异常
    // 处理
}

// 2. 不要吞没异常
try {
    // ...
} catch (IOException e) {
    log.error("操作失败", e);  // ✅ 至少记录日志
    throw new BizException(ErrorCode.SYSTEM_ERROR, e);  // 或包装后抛出
}

// 3. 使用 try-with-resources
try (InputStream is = new FileInputStream(file)) {
    // 自动关闭资源
}  // ✅ 自动调用 close()

// 4. 异常信息要有用
throw new IllegalArgumentException("用户ID不能为空");  // ✅ 清晰描述
throw new IllegalArgumentException("param error");     // ❌ 太模糊

// 5. 异常链传递
try {
    // ...
} catch (SQLException e) {
    throw new BizException(ErrorCode.DB_ERROR, e);  // ✅ 保留原始异常
}
```

#### ❌ 不应该做的

```java
// 1. 空 catch 块（吞异常）
try {
    // ...
} catch (Exception e) {
    // ❌ 什么都不做！程序"静默失败"
}

// 2. 捕获 Throwable/Error
try {
    // ...
} catch (Throwable e) {  // ❌ 会捕获 OutOfMemoryError 等致命错误
    // ...
}

// 3. 用异常控制流程
try {
    return stack.pop();
} catch (EmptyStackException e) {  // ❌ 应该用 isEmpty() 判断
    return null;
}

// 4. 忽略 InterruptedException
try {
    Thread.sleep(1000);
} catch (InterruptedException e) {
    // ❌ 应该恢复中断状态
    Thread.currentThread().interrupt();
}

// 5. 异常信息包含敏感数据
throw new BizException("用户 " + password + " 登录失败");  // ❌ 泄露密码！
```

### 6. 异步/线程池中的异常处理

```java
// 问题：ExecutorService 中抛出的异常被吞没
executor.submit(() -> {
    // 抛出的异常不会打印！
    throw new RuntimeException("error");
});

// 解决 1：使用 Future 获取异常
Future<?> future = executor.submit(() -> {
    throw new RuntimeException("error");
});
try {
    future.get();  // 这里会抛出 ExecutionException
} catch (ExecutionException e) {
    log.error("任务执行失败", e.getCause());
}

// 解决 2：包装 Runnable/Callable
executor.submit(new ExceptionCatcher(() -> {
    // 业务逻辑
}));

public class ExceptionCatcher implements Runnable {
    private final Runnable task;
    
    @Override
    public void run() {
        try {
            task.run();
        } catch (Exception e) {
            log.error("任务执行异常", e);
            // 或发送告警
        }
    }
}

// 解决 3：自定义 ThreadFactory 设置 UncaughtExceptionHandler
ThreadFactory threadFactory = r -> {
    Thread t = new Thread(r);
    t.setUncaughtExceptionHandler((thread, e) -> {
        log.error("未捕获异常 in thread {}", thread.getName(), e);
    });
    return t;
};
ExecutorService executor = Executors.newFixedThreadPool(4, threadFactory);
```

### 7. CompletableFuture 异常处理

```java
CompletableFuture.supplyAsync(() -> {
    // 可能抛异常
    return riskyOperation();
})
.thenApply(result -> {
    // 处理结果
    return transform(result);
})
.exceptionally(ex -> {  // 捕获异常，提供默认值
    log.error("操作失败", ex);
    return defaultValue;
})
.handle((result, ex) -> {  // 统一处理正常结果和异常
    if (ex != null) {
        log.error("操作失败", ex);
        return defaultValue;
    }
    return result;
});
```

## 🔥 扩展追问 1：try-catch-finally 和 try-with-resources 的区别？

```java
// 传统写法（繁琐，易出错）
InputStream is = null;
try {
    is = new FileInputStream(file);
    // 使用 is
} catch (IOException e) {
    // 处理异常
} finally {
    if (is != null) {
        try {
            is.close();  // 可能抛异常！
        } catch (IOException e) {
            // 又要处理...
        }
    }
}

// try-with-resources（Java 7+，简洁安全）
try (InputStream is = new FileInputStream(file);
     OutputStream os = new FileOutputStream(outFile)) {
    // 使用 is 和 os
} catch (IOException e) {
    // 处理异常
}
// 自动调用 close()，且会抑制 close() 抛出的异常
```

**实现原理：**

```java
// 编译后等价于
try {
    final InputStream is = new FileInputStream(file);
    Throwable $ex = null;
    try {
        // 使用 is
    } catch (Throwable $t) {
        $ex = $t;
        throw $t;
    } finally {
        if (is != null) {
            if ($ex != null) {
                try {
                    is.close();
                } catch (Throwable $t2) {
                    $ex.addSuppressed($t2);  // 抑制异常
                }
            } else {
                is.close();
            }
        }
    }
}
```

## 🔥 扩展追问 2：异常对性能的影响？

| 场景 | 开销 | 建议 |
|------|------|------|
| 创建异常对象 | 高（需填充堆栈） | 避免在热点代码创建异常 |
| 抛出和捕获 | 中等 | 不要用异常控制流程 |
| 堆栈深度 | 越深越慢 | 及时捕获，避免深层传播 |

**优化：预创建异常（谨慎使用）**

```java
// 某些框架（如 Netty）使用预创建异常避免堆栈填充
public class StacklessException extends RuntimeException {
    
    @Override
    public Throwable fillInStackTrace() {
        return this;  // 不填充堆栈
    }
}
```

```java
public abstract class AppException extends RuntimeException {

    protected final int httpStatus;

public class BizException extends AppException { }

// 2. 异常转换器（不同层转换）
public interface ExceptionTranslator<T extends Throwable> {
    AppException translate(T ex);
}

@Component
public class DataAccessExceptionTranslator implements ExceptionTranslator<DataAccessException> {
    public AppException translate(DataAccessException ex) {
        if (ex instanceof DuplicateKeyException) {
            return new BizException("DATA_DUPLICATE", "数据已存在");
        }
        return new SysException("DB_ERROR", "数据库操作失败", ex);
    }
}

@Component
public class ExceptionAlerter {

@Component
public class ExceptionMetrics {
```

## 一句话收尾

Java 异常分 Error（系统级）和 Exception（程序级），Exception 又分 Checked（强制处理）和 Unchecked（运行时）；项目用分层策略（DAO 抛、Service 转、Controller 拦）+ 全局处理器 + 错误码规范，避免异常吞没和 try-catch 泛滥，异步场景要特别注意异常捕获！
