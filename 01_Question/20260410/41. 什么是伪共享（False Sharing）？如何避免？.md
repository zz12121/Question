## 先说结论

**伪共享（False Sharing）**是多核 CPU 缓存系统中，**不同线程修改同一缓存行（Cache Line，通常 64 字节）的不同变量**，导致缓存频繁失效的性能问题。虽然线程修改的是不同变量，但因在同一缓存行，一个线程的修改会触发其他核的缓存失效，强制从内存重新加载，大幅降低性能。解决方法包括：**字节填充（Padding）**、**@Contended 注解（Java 8）**、**避免热点字段集中**。

---

## 🔥 核心概念

**解决什么问题：**
- 多线程高并发场景下的缓存性能优化
- 消除无竞争情况下的缓存同步开销

**核心思想：**
- 缓存行是 CPU 缓存的最小单位（64 字节）
- 伪共享让无竞争的变量产生"假竞争"

## 🔥 详细展开

### 1. CPU 缓存架构回顾

```
CPU 核心 1          CPU 核心 2
┌─────────┐        ┌─────────┐
│ L1 缓存 │        │ L1 缓存 │
│ 32KB    │        │ 32KB    │
├─────────┤        ├─────────┤
│ L2 缓存 │        │ L2 缓存 │
│ 256KB   │        │ 256KB   │
└────┬────┘        └────┬────┘
     │                  │
     └────────┬─────────┘
              │
         ┌────┴────┐
         │ L3 缓存 │        ← 多核共享
         │ 8MB     │
         └────┬────┘
              │
         ┌────┴────┐
         │  主内存  │
         │  16GB   │
         └─────────┘
```

**缓存行（Cache Line）：**
- 缓存的最小传输单位，通常 **64 字节**
- 即使只读 1 字节，也会加载整个缓存行

### 2. 伪共享的产生

```java
public class FalseSharingExample {
    
    // volatile 保证可见性，但引发伪共享
    private volatile long value1 = 0;  // 8 字节
    private volatile long value2 = 0;  // 8 字节
    // value1 和 value2 可能在同一缓存行（共 16 字节 < 64 字节）
    
    public static void main(String[] args) throws InterruptedException {
        FalseSharingExample example = new FalseSharingExample();
        
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 100000000; i++) {
                example.value1++;  // 修改 value1
            }
        });
        
        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 100000000; i++) {
                example.value2++;  // 修改 value2
            }
        });
        
        long start = System.currentTimeMillis();
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        System.out.println("耗时: " + (System.currentTimeMillis() - start) + "ms");
        // 伪共享时：~3000ms
        // 消除伪共享后：~300ms（10 倍提升！）
    }
}
```

**伪共享过程图解：**

```
初始状态：
CPU 核心 1 缓存行：[value1][value2][...]（64 字节）状态：Exclusive
CPU 核心 2 缓存行：[value1][value2][...]（64 字节）状态：Exclusive

时间线：
t1: 核心 1 修改 value1
    ↓
    缓存行状态变为 Modified
    核心 2 的缓存行失效（Invalid）
    
t2: 核心 2 要修改 value2
    ↓
    发现缓存行失效，从核心 1 或内存重新加载
    加载完整 64 字节缓存行
    
t3: 核心 2 修改 value2
    ↓
    核心 1 的缓存行又失效了！
    
t4: 核心 1 要修改 value1
    ↓
    又要重新加载缓存行...
    
结果：两个线程无竞争地修改不同变量，却因缓存行共享导致频繁缓存失效！
```

### 3. 伪共享 vs 真共享

| 特性 | 真共享（True Sharing） | 伪共享（False Sharing） |
|------|----------------------|------------------------|
| **竞争对象** | 同一变量 | 不同变量 |
| **是否需要同步** | 需要（互斥访问） | 不需要（逻辑上无竞争） |
| **问题原因** | 数据竞争 | 缓存行布局 |
| **解决方法** | 锁、原子变量 | 缓存行填充 |
| **性能影响** | 严重（串行化） | 严重（缓存失效） |

### 4. 解决方法

#### 方法 1：手动填充（Padding）

```java
// Java 7 及之前：手动填充
public class PaddedLong {
    
    // 前置填充：56 字节（7 个 long）
    private long p1, p2, p3, p4, p5, p6, p7;
    
    // 实际数据：8 字节
    private volatile long value;
    
    // 后置填充：56 字节（7 个 long）
    private long p8, p9, p10, p11, p12, p13, p14;
    
    // 总计：56 + 8 + 56 = 120 字节 > 64 字节
    // 确保 value 独占一个缓存行
}

// 使用
public class NoFalseSharing {
    private PaddedLong value1 = new PaddedLong();
    private PaddedLong value2 = new PaddedLong();
    // value1 和 value2 一定在不同缓存行
}
```

**填充计算：**
```
缓存行大小：64 字节
long 大小：8 字节

需要填充：64 - 8 = 56 字节
需要 long 数量：56 / 8 = 7 个

前置 7 个 long + 数据 1 个 long = 64 字节（一个缓存行）
后置 7 个 long 防止与其他数据共享
```

#### 方法 2：@Contended 注解（Java 8+）

```java
// Java 8 引入，JVM 自动填充
import sun.misc.Contended;

public class ContendedExample {
    
    @Contended
    private volatile long value1;
    
    @Contended
    private volatile long value2;
    
    // JVM 自动在 value1 和 value2 周围填充
    // 确保各自独占缓存行
}

// 启动参数开启（JDK 9+ 需要）
// -XX:-RestrictContended
```

**@Contended 原理：**
```java
// 编译后，JVM 在字段前后插入填充
// 相当于：
class ContendedExample {
    [128 bytes padding]  // JVM 根据 -XX:ContendedPaddingWidth 设置（默认 128）
    volatile long value1;
    [128 bytes padding]
    
    [128 bytes padding]
    volatile long value2;
    [128 bytes padding]
}
```

#### 方法 3：继承方式（Java 7 常用）

```java
// 继承抽象类，利用父类字段填充
abstract class RingBufferPad {
    // 56 字节填充
    protected long p1, p2, p3, p4, p5, p6, p7;
}

abstract class RingBufferFields extends RingBufferPad {
    // 实际数据开始
    protected long cursor;
    // ...
}

public class RingBuffer extends RingBufferFields {
    // 后置填充
    private long p8, p9, p10, p11, p12, p13, p14;
}
```

### 5. 实际案例：Disruptor 框架

```java
// Disruptor 的 Sequence 类（高性能队列）
class LhsPadding {
    protected long p1, p2, p3, p4, p5, p6, p7;
}

class Value extends LhsPadding {
    protected volatile long value;
}

class RhsPadding extends Value {
    protected long p8, p9, p10, p11, p12, p13, p14;
}

public class Sequence extends RhsPadding {
    // 确保 value 独占缓存行
    // 这是 Disruptor 高性能的关键之一
}
```

// 1. 只在 JDK 内部使用（sun.misc 包）

## 🔥 扩展追问 3：现代 CPU 和 JVM 对伪共享的优化

**硬件层面：**
- Intel Broadwell+ 引入 **CAT（Cache Allocation Technology）**
- 部分 CPU 支持缓存行级别的隔离

**JVM 层面：**
- `-XX:+UseCondCardMark`：减少卡表更新带来的伪共享
- `-XX:ContendedPaddingWidth`：调整填充宽度（默认 128，可设为 64）

**但硬件优化有限，关键路径仍需手动处理：**

```java
// 高并发计数器，必须使用填充或 @Contended
public class HighPerformanceCounter {
    
    @Contended
    private volatile long count;
    
    public void increment() {
        // 使用 Unsafe 或 VarHandle 实现 CAS
        UNSAFE.getAndAddLong(this, COUNT_OFFSET, 1L);
    }
}
```

## 一句话收尾

伪共享是多核 CPU 缓存行共享导致的性能陷阱，不同线程修改同一缓存行的不同变量会触发频繁缓存失效；解决用字节填充（7 个 long）或 @Contended 注解让变量独占缓存行，JDK 8 的 ConcurrentHashMap、LongAdder 都用了这个技巧，高并发场景性能可提升 5-10 倍！
