## 先说结论

**CAP 理论**：分布式系统中，一致性（Consistency）、可用性（Availability）、分区容错性（Partition Tolerance）三者不可兼得，最多只能同时满足两项，**分区容错性是必选项**，所以实际是在 CP 和 AP 之间做选择。

**BASE 理论**：对 CAP 中一致性和可用性权衡的结果，核心思想是**基本可用（Basically Available）、软状态（Soft State）、最终一致（Eventually Consistent）**，是分布式系统设计的实践指导。

## 🔥 详细展开：CAP 实际案例

### 1. CP 系统（牺牲可用性，保证一致性）

```
┌─────────────────────────────────────────────────────────┐
│                    ZooKeeper / etcd                      │
│                    （分布式协调服务）                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  写请求流程：                                             │
│                                                         │
│  Client → Leader → 同步到多数节点(Follower) → 返回成功    │
│              ↓                                          │
│         如果同步失败（分区）                               │
│              ↓                                          │
│         拒绝写请求（牺牲可用性）                           │
│                                                         │
│  特点：                                                  │
│  - 选举期间不可用（几十秒）                                │
│  - 数据强一致                                            │
│  - 适合配置中心、分布式锁                                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2. AP 系统（牺牲一致性，保证可用性）

```
┌─────────────────────────────────────────────────────────┐
│              Cassandra / DynamoDB / Eureka               │
│              （分布式数据库 / 服务注册中心）                │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  写请求流程：                                             │
│                                                         │
│  Client → 任意节点 → 本地写入成功 → 立即返回               │
│              ↓                                          │
│         后台异步同步到其他节点                              │
│              ↓                                          │
│         可能读到旧数据（牺牲一致性）                        │
│                                                         │
│  特点：                                                  │
│  - 始终可用                                              │
│  - 最终一致                                              │
│  - 适合用户会话、商品浏览、服务发现                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3. 常见系统的 CAP 选择

| 系统 | 类型 | CAP 选择 | 说明 |
|------|------|---------|------|
| **MySQL** | 单机/主从 | CA | 无分区时强一致+可用 |
| **ZooKeeper** | 分布式协调 | CP | 选举期间不可用 |
| **etcd** | 分布式 KV | CP | Raft 算法保证一致 |

| **Cassandra** | 分布式 DB | AP | 可调一致性级别 |
| **MongoDB** | 分布式 DB | CP | 默认主从强一致 |
| **Redis Cluster** | 分布式缓存 | AP | 异步复制，可能丢数据 |

## 🔥 扩展追问 1：BASE 理论详解

BASE 是对 CAP 理论的延伸和工程实践，**放弃强一致性，追求最终一致性**。

```
┌─────────────────────────────────────────────────────────┐
│                    BASE 理论                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   ┌─────────────────────────────────────────────────┐   │
│   │              Basically Available                 │   │
│   │                 基本可用                          │   │
│   │  - 响应时间损失：正常 0.5s → 降级 2s               │   │
│   │  - 功能损失：非核心功能关闭，核心功能可用           │   │
│   └─────────────────────────────────────────────────┘   │
│                         ↓                               │
│   ┌─────────────────────────────────────────────────┐   │
│   │                Soft State                        │   │
│   │                 软状态                            │   │
│   │  - 允许中间状态，数据暂时不一致                    │   │
│   │  - 不同节点数据副本可以暂时不同                    │   │
│   └─────────────────────────────────────────────────┘   │
│                         ↓                               │
│   ┌─────────────────────────────────────────────────┐   │
│   │           Eventually Consistent                  │   │
│   │              最终一致性                           │   │
│   │  - 不保证实时一致，但保证最终达到一致              │   │
│   │  - 时间窗口内不一致可接受                          │   │
│   └─────────────────────────────────────────────────┘   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### BASE vs ACID

| 特性 | ACID（传统数据库） | BASE（分布式系统） |
|------|------------------|------------------|
| **一致性** | 强一致性 | 最终一致性 |
| **事务** | 支持 | 不支持或柔性事务 |
| **隔离性** | 严格隔离 | 无隔离或宽松隔离 |
| **性能** | 相对较低 | 高吞吐、高性能 |
| **适用** | 金融交易 | 互联网高并发场景 |

### 2. 库存扣减的 CAP 权衡实战

```java
/**
 * 方案1：CP 方案（强一致，性能较低）
 * 适用：秒杀库存（不能超卖）
 */
@Service
public class StrictStockService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    // Redis + Lua 保证原子性（CP）
    public boolean deductStock(Long skuId, Integer count) {
        String script = 
            "if (redis.call('get', KEYS[1]) - ARGV[1] >= 0) then " +
            "    return redis.call('decrby', KEYS[1], ARGV[1]) " +
            "else " +
            "    return -1 " +
            "end";
        
        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(script, Long.class),
            Collections.singletonList("stock:" + skuId),
            count.toString()
        );
        
        return result != null && result >= 0;
    }
}

/**
 * 方案2：AP + 最终一致（高性能，可接受短暂不一致）
 * 适用：普通商品库存
 */
@Service
public class EventuallyStockService {
    
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    
    // 本地扣减后立即返回（AP）
    public boolean deductStock(Long skuId, Integer count) {
        // 1. 本地缓存扣减
        boolean success = localCache.deduct(skuId, count);
        if (!success) {
            return false;
        }
        
        // 2. 异步同步到数据库（最终一致）
        kafkaTemplate.send("stock-deduct", skuId + ":" + count);
        
        return true;
    }
    
    // 消费者异步落库
    @KafkaListener(topics = "stock-deduct")
    public void onMessage(String message) {
        // 更新数据库库存
        stockMapper.decrement(parseSkuId(message), parseCount(message));
    }
}
```

## 🔥 扩展追问 3：面试高频考点

### 1. CAP 的常见误区

```
❌ 误区1：CAP 是三选二，必须放弃一个
✅ 正确：P 是必须选的，实际是在 C 和 A 之间权衡

❌ 误区2：选择了 AP 就完全不要一致性
✅ 正确：AP 是牺牲强一致，追求最终一致

❌ 误区3：一个系统只能选 CP 或 AP
✅ 正确：不同子系统可以不同选择（如订单CP，搜索AP）
```

### 2. 经典面试题

**Q：ZooKeeper 是 CP 还是 AP？**
```
答：ZooKeeper 是 CP 系统。
- 选举期间集群不可用（牺牲可用性）
- 保证数据强一致性
- 适合配置管理、分布式锁等场景
```

**Q：Eureka 为什么设计为 AP？**
```
答：服务注册中心必须保证高可用。
- 服务发现不能因为网络分区而拒绝服务
- 自我保护机制防止误剔除健康实例
- 可接受短暂的不一致，不能容忍不可用
```

**Q：如何实现最终一致性？**
```
答：常用手段：
1. 异步消息（MQ）+ 本地消息表
2. 定时对账补偿
3. Saga 模式（长事务拆分）
4. TCC 模式（预留-确认-取消）
```

## 一句话收尾

CAP 理论揭示了分布式系统的**本质约束**（分区时 C 和 A 不可兼得），BASE 理论提供了**工程实践路径**（基本可用+最终一致）；实际项目中应根据业务特性**分层权衡**——核心交易选 CP，用户体验选 AP，通过异步补偿机制实现最终一致性。
