## 先说结论

**volatile** 保证**可见性**（一个线程修改立即可见）和**有序性**（禁止指令重排），但**不保证原子性**；**synchronized** 同时保证**可见性、原子性和有序性**。volatile 比 synchronized 更轻量，适合**单写多读**（如状态标志、双重检查锁）的场景，不适合**多写**或**复合操作**（如 i++）。

---

## 🔥 核心概念

**解决什么问题：**
- 多线程环境下的数据一致性问题
- CPU 缓存导致的可见性问题
- 编译器/CPU 指令重排导致的有序性问题

**本质区别：**
- volatile：内存屏障，无锁，轻量
- synchronized：互斥锁，阻塞，重量

## 🔥 详细展开

### 1. 三个特性的对比

| 特性 | volatile | synchronized |
|------|----------|--------------|
| **可见性** | ✅ 保证 | ✅ 保证 |
| **原子性** | ❌ 不保证 | ✅ 保证 |
| **有序性** | ✅ 保证（禁止重排） | ✅ 保证（串行执行） |
| **阻塞** | ❌ 不会 | ✅ 会 |
| **适用场景** | 单写多读、状态标志 | 多写、复合操作 |

### 2. volatile 的内存语义

```java
public class VolatileExample {
    
    // volatile 变量
    private volatile boolean flag = false;
    private volatile int value = 0;
    
    public void writer() {
        value = 42;           // ① 写普通变量
        flag = true;          // ② 写 volatile 变量（内存屏障）
    }
    
    public void reader() {
        if (flag) {           // ③ 读 volatile 变量（内存屏障）
            System.out.println(value);  // ④ 保证看到 42
        }
    }
}
```

**内存屏障（Memory Barrier）：**

```
写 volatile：StoreStoreBarrier + StoreLoadBarrier
读 volatile：LoadLoadBarrier + LoadStoreBarrier

效果：
- 写 volatile 之前的代码不能重排到后面
- 读 volatile 之后的代码不能重排到前面
```

**可见性实现：**

```
线程 A 修改 volatile 变量
    ↓
立即刷新到主内存（JMM 规定）
    ↓
使其他线程的缓存行失效（MESI 协议）
    ↓
线程 B 读取时从主内存重新加载
```

### 3. 为什么 volatile 不保证原子性？

```java
public class VolatileNotAtomic {
    
    private volatile int count = 0;
    
    // i++ 不是原子操作！
    public void increment() {
        count++;  // 分解为：读取 → 修改 → 写入
    }
}
```

**i++ 的字节码：**

```java
// count++ 实际执行：
getfield      #2    // ① 读取 count 到操作数栈
iconst_1            // ② 准备常量 1
iadd                // ③ 相加
putfield      #2    // ④ 写回 count

// 问题：4 步操作可能被其他线程打断
// 线程 A 读取 0 → 线程 B 读取 0 → A 写入 1 → B 写入 1
// 结果：1（期望 2），丢失更新！
```

**验证：**

```java
public class Test {
    private volatile int count = 0;
    
    public static void main(String[] args) throws InterruptedException {
        Test test = new Test();
        
        // 1000 个线程各加 1000 次
        ExecutorService executor = Executors.newFixedThreadPool(100);
        for (int i = 0; i < 1000; i++) {
            executor.submit(() -> {
                for (int j = 0; j < 1000; j++) {
                    test.count++;  // 不保证原子性！
                }
            });
        }
        
        executor.shutdown();
        executor.awaitTermination(1, TimeUnit.MINUTES);
        
        System.out.println(test.count);  // 结果 < 1,000,000！
    }
}
```

### 5. volatile 的适用场景

#### 场景 1：状态标志（最典型）

```java
public class VolatileFlag {
    
    private volatile boolean running = true;
    
    public void shutdown() {
        running = false;  // 所有线程立即可见
    }
    
    public void doWork() {
        while (running) {  // 读取 volatile，不会死循环
            // 执行任务
        }
    }
}
```

**为什么不用 synchronized？**
- 不需要原子性，只需要可见性
- volatile 无锁，性能更好

#### 场景 2：双重检查锁（DCL）

```java
public class Singleton {
    
    private volatile static Singleton instance;  // 必须 volatile！
    
    public static Singleton getInstance() {
        if (instance == null) {                    // ① 第一次检查（无锁）
            synchronized (Singleton.class) {
                if (instance == null) {            // ② 第二次检查（有锁）
                    instance = new Singleton();    // ③ 创建对象
                }
            }
        }
        return instance;
    }
}
```

**为什么必须 volatile？**

```java
// instance = new Singleton() 分解为：
1. 分配内存空间
2. 初始化对象（构造方法）
3. 将引用指向内存地址

// 指令重排后可能变成 1-3-2：
1. 分配内存空间
2. 将引用指向内存地址（此时 instance != null，但未初始化！）
3. 初始化对象

// 其他线程可能拿到未初始化的对象！
```

volatile 禁止 2 和 3 重排，保证安全。

#### 场景 3：读写锁的读优化

```java
public class ReadMostly {
    
    private volatile int value;
    
    // 读操作（无锁，高并发）
    public int getValue() {
        return value;
    }
    
    // 写操作（synchronized 保证原子性）
    public synchronized void setValue(int value) {
        this.value = value;
    }
}
```

### 6. synchronized 的适用场景

#### 场景 1：复合操作

```java
public class SafeIncrement {
    
    private int count = 0;
    
    // 必须 synchronized 或 AtomicInteger
    public synchronized void increment() {
        count++;  // 读-改-写，需要原子性
    }
    
    public synchronized int getAndIncrement() {
        return count++;  // 返回旧值并递增
    }
}
```

#### 场景 2：多变量一致性

```java
public class BankAccount {
    
    private int balance;
    private int transactionCount;
    
    // 两个变量必须同时修改，保证一致性
    public synchronized void deposit(int amount) {
        balance += amount;
        transactionCount++;
    }
}
```

#### 场景 3：阻塞等待

```java
public class BlockingQueue<T> {
    
    private final List<T> list = new ArrayList<>();
    
    public synchronized void put(T item) throws InterruptedException {
        while (list.size() == capacity) {
            wait();  // 需要 synchronized + wait/notify
        }
        list.add(item);
        notifyAll();
    }
}
```

## 🔥 扩展追问 1：volatile 的 happens-before 规则

```java
// happens-before 规则保证：
// 写 volatile → 读 volatile 之间有顺序关系

public class HappensBefore {
    
    private int a = 0;
    private volatile int b = 0;
    
    public void writer() {
        a = 1;        // ① 在写 volatile 之前
        b = 2;        // ② 写 volatile
    }
    
    public void reader() {
        int x = b;    // ③ 读 volatile
        int y = a;    // ④ 保证看到 a = 1
        
        // 如果 x == 2，则 y 一定 == 1
    }
}
```

**规则：**
- 程序顺序规则：同一线程中，前面的操作 happens-before 后面的
- volatile 规则：写 volatile happens-before 读 volatile
- 传递性：A happens-before B，B happens-before C → A happens-before C

## 🔥 扩展追问 2：Java 8 中的 LongAdder 比 volatile + synchronized 更好？

```java
// 高并发计数，不要用 volatile + synchronized
public class Counter {
    private volatile long count;
    
    public synchronized void increment() {  // 竞争激烈时性能差
        count++;
    }
}

// 用 LongAdder（分段思想，类似 ConcurrentHashMap）
public class BetterCounter {
    private final LongAdder count = new LongAdder();
    
    public void increment() {
        count.increment();  // 无锁，高并发性能更好
    }
    
    public long sum() {
        return count.sum();
    }
}
```

**LongAdder 原理：**

```
base 变量（无竞争时直接用）
+ cells[] 数组（竞争激烈时分散到不同 cell）
= 总和

每个线程 hash 到不同 cell，减少竞争
```

## 🔥 扩展追问 3：volatile 和 Atomic 类的区别

| 对比 | volatile | Atomic 类（如 AtomicInteger） |
|------|----------|------------------------------|
| 可见性 | ✅ | ✅ |
| 原子性 | ❌ | ✅（CAS 实现） |
| 实现 | 内存屏障 | Unsafe + CAS |
| 适用 | 单写多读、标志位 | 计数、累加、CAS 操作 |
| ABA 问题 | 无 | 有（AtomicStampedReference 解决） |

```java
// AtomicInteger 的 CAS 操作
public final int incrementAndGet() {
    return unsafe.getAndAddInt(this, valueOffset, 1) + 1;
}

// 底层：CAS 自旋
public final int getAndAddInt(Object o, long offset, int delta) {
    int v;
    do {
        v = getIntVolatile(o, offset);  // 读取
    } while (!compareAndSwapInt(o, offset, v, v + delta));  // CAS
    return v;
}
```
