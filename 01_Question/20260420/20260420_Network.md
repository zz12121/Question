## TCP协议原理（6题）

296. TCP三次握手的详细过程是什么？为什么是三次而不是两次？（客户端发SYN→服务端回SYN+ACK→客户端发ACK / 两次握手无法确认客户端接收能力 / 三次握手确保双方都确认了对方的发送和接收能力 / 每次握手的状态变化：CLOSED→SYN_SENT→SYN_RCVD→ESTABLISHED / 如果是两次握手：服务端发完SYN+ACK就建立连接，但客户端可能不想连接，浪费服务端资源）三次握手过程中可以携带数据吗？（第三次ACK可以携带数据，前两次不行 / RFC793允许第三次握手携带数据 / TCP Fast Open可以在SYN中携带数据）

297. TCP四次挥手的详细过程是什么？为什么需要TIME_WAIT状态？持续多久？（主动关闭方发FIN→被动方回ACK→被动方发FIN→主动方回ACK / TIME_WAIT持续2MSL（Maximum Segment Lifetime，通常60秒）/ 为什么需要TIME_WAIT：1）确保最后一个ACK能到达对端（如果丢失，对端会重发FIN）/ 2）确保旧连接的延迟报文在新连接建立前消失 / 如果没有TIME_WAIT会怎样？旧连接的延迟报文可能被新连接误收 / TIME_WAIT过多怎么办？`tcp_tw_reuse` 允许复用TIME_WAIT连接 / 增大端口范围 / 长连接保活避免频繁关闭）

298. TCP的滑动窗口机制是怎么工作的？它和流量控制是什么关系？（滑动窗口是TCP流量控制的核心机制 / 接收方在ACK中通告自己的接收窗口大小（rwnd）/ 发送方根据rwnd控制发送速率 / 窗口滑动：收到ACK后窗口左沿右移，新数据发送后窗口右沿右移 / 零窗口：接收方通告窗口为0时发送方暂停发送 / 零窗口探测：发送方定期发1字节探测窗口是否恢复 / 糊涂窗口综合征：接收方每次只通告小窗口导致效率低 / Nagle算法和Delayed ACK的交互问题）

299. TCP的拥塞控制有哪些阶段？各自的核心算法是什么？（慢启动：cwnd从1开始指数增长（每RTT翻倍）/ 拥塞避免：cwnd线性增长（每RTT加1）/ 快速重传：收到3个重复ACK立即重传丢失报文，不等超时 / 快速恢复：不回到慢启动，cwnd减半后继续线性增长 / 超时重传：RTO超时后回到慢启动，ssthresh=cwnd/2 / BBR算法（Google）：基于带宽和RTT的拥塞控制，不依赖丢包判断 / CUBIC是Linux默认拥塞控制算法 / 区别：传统算法基于丢包 / BBR基于带宽探测）

300. TCP的保活机制（KeepAlive）和HTTP的Keep-Alive有什么区别？（TCP KeepAlive：操作系统层面，检测连接是否存活 / 默认参数：`tcp_keepalive_time=7200s` / `tcp_keepalive_intvl=75s` / `tcp_keepalive_probes=9` / 即2小时无数据后开始探测 / HTTP Keep-Alive：应用层面，复用TCP连接发送多个HTTP请求 / `Connection: keep-alive` 头 / 避免每次请求都三次握手 / 两者关系：HTTP Keep-Alive依赖TCP连接保持 / TCP KeepAlive确保底层连接存活 / 生产环境建议：调小TCP KeepAlive参数 / HTTP层也要设置超时和最大请求数）

301. TCP的延迟确认（Delayed ACK）和Nagle算法是怎么交互的？会导致什么问题？（Delayed ACK：接收方不等立即回ACK，等200ms或攒够2个满尺寸报文再回 / Nagle算法：发送方将小包攒成大包再发 / 交互问题：发送方发了小包等ACK→接收方等200ms再回ACK→发送方收到ACK才发下一个小包 / 导致每个小包延迟200ms / 典型场景：Redis客户端禁用Nagle / MySQL客户端禁用Nagle / HTTP请求禁用Nagle / 解决方案：设置TCP_NODELAY禁用Nagle / 或发送大包避免小包交互 / TCP_QUICKACK禁用Delayed ACK）

## UDP与传输层（4题）

302. UDP和TCP的核心区别是什么？UDP适用于什么场景？（TCP：面向连接、可靠、有序、流量控制、拥塞控制 / UDP：无连接、不可靠、无序、无流量控制、头部8字节 vs TCP 20字节 / UDP适用场景：实时音视频（丢帧可接受）/ DNS查询（简单快速）/ 游戏同步（低延迟优先）/ QUIC（在UDP上实现可靠传输）/ IoT设备上报（小包效率高）/ UDP比TCP快的原因：无连接建立开销、无拥塞控制、无重传等待）

303. QUIC协议是什么？HTTP/3为什么选择基于UDP的QUIC而不是TCP？（QUIC是Google设计的传输层协议，基于UDP / 核心特性：0-RTT连接建立 / 连接迁移（支持IP变化，移动网络切换不断连）/ 多路复用无队头阻塞 / 内置TLS 1.3 / 为什么不用TCP：TCP的队头阻塞在HTTP/2中仍存在（一个TCP包丢失阻塞所有流）/ TCP连接建立需要1-RTT+TLS需要1-2RTT / TCP连接不支持IP迁移 / QUIC在用户态实现拥塞控制，不需要修改操作系统内核 / QUIC的挑战：UDP可能被中间设备拦截 / UDP的缓冲区默认较小 / 运维工具不成熟）

304. TCP的粘包和半包问题是什么？为什么UDP没有这个问题？（粘包：发送方发送的两个包被接收方一次读出 / 半包：发送方的一个包被接收方分两次读出 / TCP是流式协议，没有消息边界 / 应用层必须自己定义消息边界 / 解决方案：固定长度 / 分隔符 / 长度字段前缀（最常用）/ UDP没有这个问题：UDP是数据报协议，每个sendto对应一个recvfrom / 保留消息边界 / 但UDP有分片问题：超过MTU（1500字节）会在IP层分片 / 分片丢失导致整个数据报丢弃）

305. TCP的RST报文是什么？什么情况下会产生RST？（RST是TCP的复位报文，立即终止连接 / 场景1：连接不存在时收到报文（端口未监听）/ 场景2：请求连接时对方拒绝 / 场景3：已建立的连接收到不合法的报文（序号超出窗口）/ 场景4：程序崩溃后重启，收到旧连接的报文 / 场景5：设置了SO_LINGER=0主动发RST关闭 / RST的影响：不经过四次挥手，连接立即断开 / 数据丢失：发送缓冲区的数据不会发送 / 接收方会收到 `Connection reset by peer` / 和正常FIN关闭的区别：FIN是优雅关闭 / RST是强制关闭）

## HTTP协议（6题）

306. HTTP/1.1、HTTP/2、HTTP/3的核心区别是什么？（HTTP/1.1：文本协议、每个请求一个TCP连接（Keep-Alive复用但串行）/ 队头阻塞：前一个请求未完成时后一个请求阻塞 / HTTP/2：二进制帧、多路复用（一个连接并行多个流）、头部压缩HPACK、服务器推送 / 仍存在TCP层队头阻塞 / HTTP/3：基于QUIC（UDP）、0-RTT连接、连接迁移、多路复用无队头阻塞 / 内置TLS 1.3 / 对比：HTTP/1.1→HTTP/2性能提升主要靠多路复用和头部压缩 / HTTP/2→HTTP/3提升主要靠解决TCP队头阻塞和连接建立延迟）

307. HTTP/2的多路复用是怎么实现的？它解决了什么问题？还有什么遗留问题？（核心实现：二进制帧（Frame）/ 每个请求响应是一个Stream / Stream内部用Stream ID标识 / 每个Stream可以分成多个Frame / Frame可以交错发送 / 接收方按Stream ID重组 / 解决了HTTP/1.1的队头阻塞：一个连接可以并行多个请求 / 但仍有TCP层队头阻塞：一个TCP包丢失会阻塞所有Stream / HTTP/2的流量控制：每个Stream有独立的流量控制窗口 / Stream优先级：可以设置权重和依赖关系 / HPACK头部压缩：静态表+动态表+哈夫曼编码）

308. HTTPS的TLS握手过程是怎样的？TLS 1.2和TLS 1.3有什么区别？（TLS 1.2握手：ClientHello→ServerHello+证书+ServerKeyExchange→ClientKeyExchange+ChangeCipherSpec+Finished→ChangeCipherSpec+Finished / 2-RTT完成 / TLS 1.3握手：ClientHello（带支持的keyshare）→ServerHello+证书+Finished / 1-RTT完成 / TLS 1.3优化：去除了不必要的密码套件 / 只保留ECDHE密钥交换（前向安全）/ 去除了RSA密钥交换 / 0-RTT：如果之前连接过，可以在ClientHello中携带应用数据 / 前向安全（Forward Secrecy）：即使私钥泄露也无法解密历史流量 / 依赖ECDHE临时密钥）

309. HTTP缓存机制是怎么工作的？强缓存和协商缓存有什么区别？（强缓存：浏览器直接使用本地缓存，不发送请求 / Cache-Control: max-age=3600（1小时内有效）/ Expires: 绝对时间（HTTP/1.0）/ 协商缓存：浏览器向服务器验证缓存是否有效 / If-Modified-Since / Last-Modified（基于修改时间）/ If-None-Match / ETag（基于内容哈希）/ 服务器返回304 Not Modified表示缓存有效 / 返回200表示缓存失效 / 优先级：Cache-Control > Expires / ETag > Last-Modified / no-cache：不使用强缓存，但可以使用协商缓存 / no-store：完全不缓存 / 必须同时设置Cache-Control和ETag才能保证缓存行为正确）

310. HTTP状态码的分类和常见状态码的含义是什么？哪些状态码容易混淆？（1xx信息类：100 Continue / 2xx成功：200 OK、201 Created、204 No Content / 3xx重定向：301永久重定向（浏览器缓存）、302临时重定向（不缓存）、304 Not Modified（缓存有效）/ 4xx客户端错误：400 Bad Request、401 Unauthorized（未认证）、403 Forbidden（已认证但无权限）、404 Not Found、429 Too Many Requests / 5xx服务端错误：500 Internal Server Error、502 Bad Gateway（网关收到无效响应）、503 Service Unavailable（服务不可用）、504 Gateway Timeout（网关超时）/ 容易混淆：301 vs 302 / 401 vs 403 / 502 vs 504 / 502是上游返回无效响应 / 504是上游超时无响应）

311. Cookie、Session、Token三种认证方式的原理和区别是什么？（Cookie：服务端Set-Cookie→浏览器自动携带 / 有SameSite/HttpOnly/Secure属性 / 跨域限制 / Session：服务端存储会话数据，SessionID通过Cookie传递 / 服务端有状态 / 集群需要Session共享 / Token：JWT（JSON Web Token）自包含 / 无状态 / 包含Header.Payload.Signature / 不依赖Cookie / 跨域友好 / 对比：Cookie是传输机制 / Session是服务端存储方案 / Token是无状态认证方案 / Cookie+Session：传统Web应用 / Token：前后端分离/移动端/微服务 / JWT的缺点：无法主动失效（除非加黑名单）/ Token较大（包含Claims）/ 续期方案：Refresh Token）

## DNS与CDN（4题）

312. DNS解析的完整流程是什么？递归查询和迭代查询有什么区别？（完整流程：浏览器DNS缓存→操作系统DNS缓存→hosts文件→本地DNS服务器→根域名服务器→顶级域名服务器→权威域名服务器 / 递归查询：客户端只问本地DNS，本地DNS负责返回最终结果 / 迭代查询：本地DNS逐级问根→顶级→权威 / DNS记录类型：A（IPv4地址）/ AAAA（IPv6地址）/ CNAME（别名）/ MX（邮件）/ NS（域名服务器）/ TXT（文本记录）/ TTL：缓存过期时间 / DNS污染：中间人篡改DNS响应 / DNS劫持：ISP篡改DNS结果 / 防御：DNSSEC / HTTPDNS / DoH / DoT）

313. CDN的工作原理是什么？回源的条件有哪些？（CDN = 内容分发网络 / 边缘节点缓存静态资源 / 工作流程：用户请求→DNS解析到最近的CDN节点→CDN节点有缓存则直接返回→无缓存则回源站获取 / 回源条件：缓存过期（TTL到期）/ 缓存未命中（首次访问）/ 源站内容更新（Purge）/ CDN缓存策略：按文件类型/路径/状态码 / CDN回源率：衡量CDN效率 / 降低回源率：增大TTL / 预热（主动推送）/ 合理设置缓存层级 / CDN不能缓存的：个性化内容 / 实时数据 / POST请求 / CDN vs 代理：CDN是缓存加速 / 反向代理是统一入口 / 动态加速：TCP优化/路由优化/压缩）

314. HTTPDNS是什么？为什么要用HTTPDNS替代传统DNS？（HTTPDNS：用HTTP协议向DNS服务器查询域名 / 不走传统UDP 53端口的DNS / 解决的问题：1）Local DNS劫持：ISP篡改DNS结果 / 2）调度不精准：Local DNS的IP不代表用户真实位置 / 3）DNS缓存刷新慢：TTL内无法更新 / HTTPDNS的优势：绕过Local DNS直接查询权威DNS / 客户端带上IP信息，精准调度 / 实时生效：客户端直接获取最新IP / 客户端SDK：阿里HTTPDNS / 腾讯HTTPDNS / 适用场景：移动App / 对延迟敏感的业务 / 国内DNS环境差 / 注意：需要SDK集成 / Web端无法使用（浏览器不支持）/ 回退机制：HTTPDNS失败时降级传统DNS）

315. DNS的TTL对系统有什么影响？TTL设置多大合适？（TTL = Time To Live / DNS记录在缓存中的存活时间 / TTL小的好处：DNS变更快速生效 / IP切换后用户快速切换 / TTL大的好处：减少DNS查询 / 降低延迟 / 减轻DNS服务器压力 / 设置建议：稳定服务TTL=3600秒 / 可能变更的服务TTL=60-300秒 / 割接前降低TTL到60秒 / 割接完成后再调高 / JVM的DNS缓存：`networkaddress.cache.ttl` 默认30秒 / `networkaddress.cache.negative.ttl` 默认10秒 / 问题：TTL过小→DNS查询频繁→延迟增加 / TTL过大→IP变更后用户长时间连不上 / CDN的TTL：静态资源TTL很长（86400秒）/ 动态内容TTL=0或不缓存）

## WebSocket与实时通信（4题）

316. WebSocket协议的握手过程是怎样的？和HTTP长轮询有什么区别？（WebSocket握手：客户端发HTTP请求，带 `Upgrade: websocket` 头 / 服务端返回101 Switching Protocols / 握手后协议从HTTP切换为WebSocket / 全双工通信 / HTTP长轮询：客户端发请求→服务端有数据才返回→客户端立即再发请求 / 伪双向 / 对比：延迟：WebSocket毫秒级 / 长轮询秒级 / 连接开销：WebSocket一次握手 / 长轮询每次请求HTTP头 / 服务器资源：WebSocket连接占用少 / 长轮询每次请求占用线程 / 适用场景：WebSocket→实时聊天/推送/协作 / 长轮询→兼容性要求高/低频推送）

317. SSE（Server-Sent Events）和WebSocket的区别是什么？各自适用什么场景？（SSE：服务端向客户端单向推送 / 基于HTTP / 文本协议 / 自动重连 / 浏览器原生支持EventSource / WebSocket：全双工 / 独立协议 / 二进制/文本 / 需要手动重连 / 区别：方向：SSE单向（服务端→客户端）/ WebSocket双向 / 协议：SSE基于HTTP / WebSocket独立协议 / 数据类型：SSE只支持文本 / WebSocket支持二进制 / 重连：SSE自动 / WebSocket手动 / 兼容性：SSE不支持IE / WebSocket不支持老浏览器 / SSE适用：通知推送/新闻流/股票行情 / WebSocket适用：聊天/游戏/协作编辑）

318. WebSocket的心跳机制怎么实现？为什么需要心跳？（WebSocket Ping/Pong帧：协议内置心跳机制 / 一方发Ping→另一方回Pong / 也可以用应用层心跳：定时发送自定义消息 / 为什么需要心跳：1）检测连接是否存活 / 2）防止中间设备（代理/防火墙/NAT）因空闲超时断开连接 / 3）NAT映射超时：不同系统NAT超时时间不同（1-30分钟）/ 心跳间隔建议：30-60秒 / Netty的IdleStateHandler：读空闲/写空闲/全空闲 / WebSocket关闭帧：Close帧（code 1000正常关闭）/ 异常断连检测：心跳超时后主动关闭 / 重连策略：指数退避重连）

319. WebSocket在Nginx反向代理下需要注意什么配置？（Nginx默认不支持WebSocket / 需要配置 `proxy_set_header Upgrade $http_upgrade` / `proxy_set_header Connection "upgrade"` / 超时配置：`proxy_read_timeout 3600s` / 默认60秒会断开 / 完整配置：

```nginx
location /ws {
    proxy_pass http://backend;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_read_timeout 3600s;
    proxy_send_timeout 3600s;
}
```

注意事项：`proxy_read_timeout` 必须大于心跳间隔 / 多个WebSocket连接需要sticky session / 负载均衡用IP Hash或自定义Header / WebSocket和HTTP混用同一个端口时Nginx配置 / SSL终端：WSS需要Nginx配置SSL证书）

## 网络排查与安全（6题）

320. TCP的TIME_WAIT状态堆积怎么排查和解决？（`netstat -an | grep TIME_WAIT | wc -l` 统计数量 / 大量TIME_WAIT的原因：短连接过多（HTTP/1.0无Keep-Alive）/ 客户端频繁创建关闭连接 / 解决方案1：`tcp_tw_reuse=1` 允许复用TIME_WAIT连接（仅客户端）/ 解决方案2：`tcp_tw_recycle=1`（NAT环境下可能导致丢包，不推荐）/ 解决方案3：长连接Keep-Alive避免频繁关闭 / 解决方案4：增大端口范围 `net.ipv4.ip_local_port_range` / 解决方案5：服务端由客户端主动关闭（让TIME_WAIT出现在客户端）/ HTTP/1.1默认Keep-Alive / 连接池复用 / 监控：`ss -s` 查看连接状态统计）

321. 如何用tcpdump和Wireshark抓包分析网络问题？（tcpdump基本用法：`tcpdump -i eth0 -nn port 80` / `-i` 指定网卡 / `-nn` 不解析域名和端口名 / `port` 过滤端口 / `host` 过滤IP / `tcp[tcpflags] & tcp-syn != 0` 过滤SYN包 / `-w file.pcap` 保存为pcap文件 / `-s 0` 抓完整包 / Wireshark分析：打开pcap文件 / 过滤器：`tcp.stream eq 1` / `http.request` / `tcp.analysis.retransmission` / 分析TCP握手：看SYN/SYN+ACK/ACK / 分析延迟：看TCP RTT / 分析重传：看重复ACK和快速重传 / 分析关闭：看FIN/RST / 统计：Wireshark的Statistics→Flow Graph / 注意：生产环境抓包可能影响性能）

322. 323. 什么是跨域问题？CORS是怎么解决跨域的？（同源策略：协议+域名+端口必须完全相同 / 跨域限制：XMLHttpRequest/Fetch不能请求不同源的资源 / CORS（Cross-Origin Resource Sharing）：服务端通过HTTP头声明允许的跨域规则 / 简单请求：直接发请求，服务端通过 `Access-Control-Allow-Origin` 响应 / 预检请求（Preflight）：非简单请求先发OPTIONS / 服务端返回允许的方法和头 / `Access-Control-Allow-Origin` / `Access-Control-Allow-Methods` / `Access-Control-Allow-Headers` / `Access-Control-Allow-Credentials` / `Access-Control-Max-Age` / 凭证：Cookie跨域需要 `withCredentials=true` + `Allow-Credentials=true` + `Allow-Origin` 不能是 `*` / 注意：CORS是浏览器行为，Postman等工具不受限制）

325. 什么是TCP的SYN Cookie？它怎么防御SYN Flood攻击？（SYN Flood攻击原理：攻击者发送大量SYN但不完成第三次握手 / 服务端为每个SYN分配资源并等待ACK / 半连接队列满后无法服务正常请求 / 传统防御：增大半连接队列 / 减小SYN+ACK重试次数 / SYN Cookie原理：不在半连接队列中保存状态 / 而是将状态编码在SYN+ACK的初始序号中 / 收到ACK时从序号反算验证 / 不需要分配内存保存半连接 / 验证通过才分配资源 / 缺点：无法携带TCP选项（MSS等）/ 不支持窗口缩放 / 影响性能 / Linux配置：`tcp_syncookies=1` / 正常情况关闭 / 攻击时自动启用 / 更好的方案：前端防火墙清洗 / CDN防护 / 分布式防御）
