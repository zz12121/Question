## 先说结论

| 数据类型 | 底层结构 | 一句话概括 | 典型场景 |
|---------|---------|-----------|---------|
| **String** | SDS（简单动态字符串） | KV 缓存之王 | 缓存、计数器、分布式锁、Session |
| **Hash** | Dict（字典） | 对象存储，字段级操作 | 用户信息、购物车、商品详情 |
| **List** | Quicklist（压缩链表+双向链表） | 有序列表，可做队列/栈 | 消息队列、最新消息 Feed、任务队列 |
| **Set** | Dict（无序去重） | 去重集合，支持交并差 | 标签、关注列表、共同好友 |
| **Zset** | Dict + 跳表（SkipList） | 带权重的有序集合 | 排行榜、延迟队列、带权重的优先级队列 |

## 逐个详解

### String — KV 缓存之王

#### 底层结构：SDS（Simple Dynamic String）

```c
// Redis 源码中的 SDS 结构（不是 C 字符串！）
struct sdshdr {
    int len;        // 已用长度
    int free;       // 剩余空间
    char buf[];     // 字节数组
};
```

**SDS 相比 C 字符串的优势**：

| 特性 | C 字符串 | SDS |
|-----|---------|-----|
| 获取长度 | O(n) 遍历 | O(1)，有 `len` 字段 |
| 缓冲区溢出 | 可能发生 | 自动扩容，不会溢出 |
| 内存重分配 | 每次增长都要 realloc | 惰性空间释放 + 预分配 |
| 二进制安全 | 遇到 `\0` 就截断 | `len` 记录长度，支持二进制 |

#### 常用命令

```sql
SET name "Tom"              -- 设置值
GET name                     -- 获取值
SETNX name "Jerry"          -- key 不存在才设（分布式锁核心）
SETEX name 300 "Tom"        -- 设置值 + 过期时间（原子操作）
SET name "Tom" EX 300       -- 同上，等价写法
SET name "Tom" NX EX 300    -- 不存在才设置，且有过期时间（SETNX + EX 合体）
MSET a 1 b 2 c 3            -- 批量设置
MGET a b c                   -- 批量获取
INCR order_count             -- 原子 +1（计数器）
INCRBY order_count 100       -- 原子 +100
DECR stock                   -- 原子 -1
INCRBYFLOAT price 0.5        -- 原子 +0.5（浮点）
STRLEN name                  -- 获取长度
APPEND name " Junior"       -- 拼接，返回新长度
```

#### 核心使用场景

```java
// 1. 缓存（最常见）
String key = "user:100";
redis.set(key, JSON.toJSONString(user));
String val = redis.get(key);
User user = JSON.parseObject(val, User.class);

// 2. 分布式锁（SET NX EX 组合）
String lockKey = "lock:order:" + orderId;
String result = redis.set(lockKey, Thread.currentThread().getName(),
    SetParams.setParams().nx().ex(30));  // 原子设值 + 过期30秒
if ("OK".equals(result)) {
    try {
        // 业务逻辑
    } finally {
        redis.del(lockKey);  // 释放锁
    }
}
// 缺陷：锁续期问题 → Redisson 用看门狗自动续期解决

// 3. 计数器
redis.incr("page:view:" + pageId);
redis.incr("user:login:" + userId + ":" + today);  // 当天登录次数

// 4. Session 共享
// 多节点部署时，Session 存 Redis，Nginx 轮询到哪台都能拿到同一 Session

// 5. 分布式 ID 生成（雪花算法 Redis 版）
redis.incr("snowflake:" + workerId);  // 高位 + Redis自增，低位时间戳
```

### Hash — 对象存储

#### 底层结构：Dict（字典，渐进式 rehash）

```c
// Redis Hash 的内部结构
struct dict {
    dictType *type;       // 哈希函数
    dictEntry **ht_table[2];  // 两张哈希表，rehash 时交替
    long rehashidx;       // rehash 进度，-1 表示未在进行
    long size;            // 哈希表大小
    long used;            // 已使用的槽位数
};
```

#### 常用命令

```sql
HSET user:100 name "Tom" age "25" city "Beijing"  -- 设置 hash 字段
HGET user:100 name                                   -- 获取单个字段
HMGET user:100 name age city                         -- 批量获取字段
HGETALL user:100                                     -- 获取所有字段值
HINCRBY user:100 age 1                               -- 字段原子 +1
HINCRBYFLOAT user:100 score 0.5                      -- 原子 +0.5
HEXISTS user:100 name                                -- 字段是否存在
HDEL user:100 city                                   -- 删除字段
HLEN user:100                                        -- 字段数量
HKEYS user:100                                        -- 所有字段名
HVALS user:100                                        -- 所有字段值
```

#### 核心使用场景

```java
// 1. 对象存储（比 String 更细粒度）
// ❌ String 方式：改年龄要整个对象序列化/反序列化
redis.set("user:100", JSON.toJSONString(user));  // 改一个字段要全量操作

// ✅ Hash 方式：改单个字段
redis.hset("user:100", "age", "26");
redis.hincrBy("user:100", "age", 1);  // 原子 +1

// 2. 购物车
HSET cart:100:888 sku:001 '{"name":"iPhone","num":1,"price":6999}'
HINCRBY cart:100:888 sku:001 1       -- 商品数量 +1
HGET cart:100:888 sku:001             -- 查看某商品数量
ZREM cart:100:888 sku:001             -- 删除某商品
HLEN cart:100:888                      -- 购物车商品种类数

// 3. 商品详情（字段对应数据库列）
HSET product:001 name "iPhone16" price 6999 stock 100 category "phone"
HMGET product:001 price stock  -- 只查两个字段，无需读全表
```

### List — 有序列表

#### 底层结构：Quicklist（JDK8 之后）

```
Quicklist = 多个压缩链表（ziplist）+ 双向指针串联
    ↓
压缩链表：元素少时（< 某阈值）用连续内存存储，节省指针开销
双向链表：元素多时自动切换，平衡内存占用和插入效率
    ↓
本质是 LinkedList<ziplist>
```

**为什么不用单向链表或数组？**

| 结构 | 头插 | 尾插 | 随机访问 | 内存 |
|-----|-----|------|---------|------|
| 数组 | O(n) | O(1) | O(1) | 连续，可能扩容 |
| 双向链表 | O(1) | O(1) | O(n) | 每节点 2 指针，开销大 |
| **Quicklist** | O(1) | O(1) | O(n) | 内存紧凑，性能均衡 ✅ |

#### 常用命令

```sql
LPUSH msg:queue "task1"           -- 头部插入（栈）
RPUSH msg:queue "task3"           -- 尾部插入（队列）
LPOP msg:queue                     -- 头部弹出
RPOP msg:queue                     -- 尾部弹出
LRANGE msg:queue 0 -1              -- 获取所有元素（0 -1 = 全部）
LINDEX msg:queue 2                 -- 按索引获取（0 = 头）
LINSERT msg:queue BEFORE "task2" "task1.5"  -- 插入
LLEN msg:queue                     -- 长度
LTRIM msg:queue 0 99               -- 裁剪，保留 [0, 99]
BRPOP msg:queue 0                  -- 阻塞弹出，超时 0 = 永久阻塞
```

#### 核心使用场景

```java
// 1. 消息队列（轻量版，不适合高可靠场景）
// 生产者
redis.lpush("mq:orders", orderId);
// 消费者
String orderId = redis.brpop("mq:orders", 0);  // 阻塞等待，有新消息立即返回

// ⚠️ Redis 消息队列的缺陷：
// - 没有消费确认机制（可能丢消息）
// - 没有消息持久化（默认内存）
// - 适合解耦场景，不适合金融级可靠性
// 解决方案：用专业 MQ：RabbitMQ / RocketMQ / Kafka

// 2. 最新消息 Feed（类似微博首页）
redis.lpush("feed:user:100", messageId);
redis.ltrim("feed:user:100", 0, 49);  // 只保留最近 50 条
List<String> feed = redis.lrange("feed:user:100", 0, 9);  // 获取前 10 条

// 3. 任务队列
// LPUSH + BRPOP = 先进先出队列
redis.lpush("worker:tasks", taskJson);
// 后台线程
while (true) {
    String task = redis.brpop("worker:tasks", 0);
    process(task);
}

// 4. 栈（后进先出）
redis.lpush("stack:undo", action);
// 撤销时弹出最近的操作
String lastAction = redis.lpop("stack:undo");
```

### Set — 无序去重集合

#### 底层结构：Dict（字典，值为 NULL）

```c
// Set = Dict，key 是元素，value 恒为 NULL（只关注 key 的存在性）
struct intset {        // 小整数集合优化（当所有元素都是整数且数量少时）
    int32_t encoding;  // 编码方式：INTSET_ENC_INT16/32/64
    int32_t length;    // 元素数量
    int8_t  contents[]; // 元素数组，从小到大排序存储
};
```

**转换规则**：`Set` 元素全为整数 + 数量少（< 512 默认）时用 `intset`；否则自动转成 `dict`。

#### 常用命令

```sql
SADD tags:article:100 "Java" "Redis" "MySQL"      -- 添加元素（自动去重）
SREM tags:article:100 "MySQL"                       -- 删除元素
SMEMBERS tags:article:100                           -- 获取所有标签
SISMEMBER tags:article:100 "Java"                   -- 判断是否存在
SCARD tags:article:100                              -- 集合基数（标签数量）
SRANDMEMBER tags:article:100 2                      -- 随机取 2 个（不删除）
SPOP tags:article:100 1                              -- 随机弹出 1 个（会删除）
SINTER tags:article:100 tags:article:200            -- 交集：共同标签
SUNION tags:article:100 tags:article:200            -- 并集：所有标签
SDIFF tags:article:100 tags:article:200             -- 差集：A有B没有的
SINTERSTORE tags:common tags:article:100 tags:article:200  -- 存交集结果
```

#### 核心使用场景

```java
// 1. 标签系统（去重 + 交集查询）
redis.sadd("tags:user:100", "Java", "Python", "AI");
redis.sadd("tags:user:200", "Java", "Golang", "AI");
// 共同标签
Set<String> common = redis.sinter("tags:user:100", "tags:user:200");  // [Java, AI]

// 2. 关注/粉丝列表（去重）
redis.sadd("follow:100", "201", "202", "203");
redis.sadd("follow:201", "100");  // 互相关注 = 互粉
// 共同关注（你关注的人也关注了他）
Set<String> mutuals = redis.sinter("follow:100", "follow:201");

// 3. UV 统计（去重计数）
redis.sadd("page:uv:20250410", "user:001", "user:002", "user:003");
long uv = redis.scard("page:uv:20250410");  // 去重后的访问人数

// 4. 黑名单/白名单
redis.sadd("black:ips", "192.168.1.100", "192.168.1.101");
boolean isBlocked = redis.sismember("black:ips", "192.168.1.100");  // true = 拦截
```

### Zset — 带权重排序集合

#### 底层结构：Dict + 跳表（SkipList）

```
Redis Zset 的核心数据结构：

Dict（字典）     +    SkipList（跳表）
   ↓                      ↓
 O(1) 按元素查     O(log n) 按分数范围查 + 有序遍历
```

**为什么 Zset 同时需要 Dict 和 SkipList？**

```c
// Redis Zset 源码结构
typedef struct zset {
    dict *dict;          // {member: score}，O(1) 查某成员分数
    skiplist *zsl;       // 按分数排序的跳表，O(log n) 按排名/范围查
} zset;

// 跳表层数：每层概率 1/2，最高 32 层（1亿数据也只需 32 层跳）
// 层数计算：随机数生成器决定，新节点晋升到第 k 层的概率 = 1/2^k
```

**跳表演示（查找 score=85 的节点）**：

```
Level 3:  [head] --------→ [score=90] --------→ [tail]
Level 2:  [head] → [60] → [85] → [90] --------→ [tail]
Level 1:  [head] → [60] → [70] → [85] → [88] → [90] → [tail]
          ↓
从 head 开始，Level 3 发现 90 > 85 → 降到 Level 2
找到 85 ✅（只需遍历 3 个节点，而不是链表的 6 个）
```

#### 常用命令

```sql
ZADD leaderboard 100 "Tom" 90 "Jerry" 85 "Alice"  -- 添加（分数+成员）
ZSCORE leaderboard "Tom"                              -- 获取某成员分数
ZINCRBY leaderboard 10 "Tom"                         -- 分数原子 +10
ZRANGE leaderboard 0 9 WITHSCORES                    -- 获取 TOP 10（升序）
ZREVRANGE leaderboard 0 9 WITHSCORES                  -- 获取 TOP 10（降序）
ZRANK leaderboard "Tom"                               -- 获取排名（升序，0=最低）
ZREVRANK leaderboard "Tom"                            -- 获取排名（降序，0=最高）
ZCOUNT leaderboard 80 100                            -- 80~100 分之间人数
ZRANGEBYSCORE leaderboard 80 100 WITHSCORES LIMIT 0 10  -- 分數段查询
ZREM leaderboard "Alice"                              -- 删除成员
ZCARD leaderboard                                     -- 总人数
```

#### 核心使用场景

```java
// 1. 排行榜（实时 TOP N）
// 游戏积分榜 / 直播间热度榜 / 文章热度榜
redis.zadd("game:leaderboard", 1500, "player:001");
redis.zadd("game:leaderboard", 2000, "player:002");
redis.zadd("game:leaderboard", 1800, "player:003");
// 获取 TOP 3
Set<Tuple> top3 = redis.zrevrangeWithScores("game:leaderboard", 0, 2);
// 每秒更新热度：用户每点赞一次
redis.zincrby("article:rank", 1, "article:101");

// 2. 带权重的延迟队列
// 分数 = 执行时间戳，任务按时间顺序执行
redis.zadd("delay:queue", System.currentTimeMillis() + 5000, taskId);  // 5秒后执行
// 定时任务每 100ms 拉取到期的任务
Set<String> dueTasks = redis.zrangebyscore("delay:queue",
    0, System.currentTimeMillis(), limit, 0, 10);
for (String taskId : dueTasks) {
    redis.zrem("delay:queue", taskId);  // 取出后删除
    processTask(taskId);
}

// 3. 滑动窗口限流（Zset 实现）
public boolean isAllowed(String userId, int maxQps) {
    String key = "rate:" + userId;
    long now = System.currentTimeMillis();
    long windowStart = now - 1000;
    
    // 删除窗口外的旧记录
    redis.zremrangeByScore(key, 0, windowStart);
    
    // 统计当前窗口内请求数
    long count = redis.zcard(key);
    if (count >= maxQps) return false;  // 限流
    
    // 记录本次请求
    redis.zadd(key, now, String.valueOf(now));
    redis.expire(key, 2);  // 2秒过期
    return true;
}

// 4. 权重型消息队列（主播按热度排序）
redis.zadd("live:queue", 10000, "room:001");  // 热度 10000
redis.zadd("live:queue", 5000, "room:002");   // 热度 5000
// 优先推送给热度高的直播间
String topRoom = redis.zrevrange("live:queue", 0, 0).iterator().next();
```

### 追问 1：5 种数据类型的内部编码（底层实现）？

```sql
-- 查看某 key 的内部编码
OBJECT ENCODING "user:100";        -- "embstr" / "int" / "ziplist" / "hashtable" ...

-- String 的编码类型：
--   int：整数值（能用 long 存储的）
--   embstr：短字符串（<= 39 字节），SDS + RedisObject 连续内存分配
--   raw：长字符串，SDS 和 RedisObject 分离分配

-- Hash 的编码：
--   ziplist：field+value 对少时用（field<512 且 value<64B），节省内存
--   hashtable：元素多了自动转 dict

-- List 的编码：
--   ziplist：元素少时用（< 某阈值）
--   quicklist：JDK8 后统一用 quicklist = 多个 ziplist 串联
--   quicklist 节点数配置：list-max-ziplist-size -2 = 每个 ziplist 最大 8KB

-- Set 的编码：
--   intset：小整数集合优化（全是整数 + 数量少）
--   hashtable：dict 实现

-- Zset 的编码：
--   ziplist：元素少时用（zset-max-ziplist-entries 128）
--   skiplist + dict：元素多了用（平衡 O(log n) 范围查和 O(1) 点查）
```

### 追问 2：Redis 内存满了会怎样？如何淘汰？

```sql
-- 查看当前内存使用
INFO memory
MEMORY USED: 5.23GB
MEMORY LIMIT: 8GB

-- 查看淘汰策略
CONFIG GET maxmemory-policy
-- maxmemory-policy: allkeys-lru

-- 8 种淘汰策略：
--   noeviction：满了直接拒绝写入（默认）
--   volatile-lru：设置了过期时间的 key 中，淘汰最近最少使用的 ← 最常用
--   allkeys-lru：所有 key 中，淘汰 LRU ← 全局缓存推荐
--   volatile-lfu：设置了过期时间的 key 中，淘汰最少使用的
--   allkeys-lfu：所有 key 中，淘汰 LFU
--   volatile-random：设置了过期时间的 key 中，随机淘汰
--   allkeys-random：所有 key 中，随机淘汰
--   volatile-ttl：设置了过期时间的 key 中，淘汰 TTL 最短的（快过期的）
```

> **面试加分**：生产环境推荐 `allkeys-lru`（全局缓存）或 `volatile-lru`（有过期时间的缓存），LRU 比 LFU 更常用因为实现简单且效果够好。

### 追问 3：Redis 的过期键删除策略？

```
Redis 有三种删除策略：

1. 惰性删除（被动）：访问 key 时检查是否过期，过期则删除
   - 优点：CPU 友好，不主动检查
   - 缺点：过期键大量堆积时占内存（无访问就永远不删）

2. 定期删除（主动）：每隔一段时间随机抽检一批 key，删过期的
   - Redis 每秒随机 20 次（hz=10，每次20个抽样）
   - 平衡了 CPU 和内存

3. 定时删除（主动）：为每个 key 创建定时器，到期立刻删除
   - 优点：内存友好，到期即删
   - 缺点：CPU 开销大，不推荐
   - Redis 没有用这种策略

实际 Redis = 惰性删除 + 定期删除 组合使用
```

### 追问 5：Redis 分布式锁的完整实现（Redisson 原理）？

```java
// 简单版（SETNX，有缺陷）
String lockKey = "order:lock:" + orderId;
String result = redis.set(lockKey, threadId, SetParams.nx().ex(30));
// 缺陷1：释放锁时，可能误删别人的锁（任务执行超时，自动过期，但锁刚好被别人拿了）
// 缺陷2：锁没有续期，任务执行超过30秒，锁被自动释放，另一个任务进来了

// 完善版（Redisson 看门狗机制）
RLock lock = redisson.getLock("order:lock:" + orderId);
lock.lock();  // 内部自动开启看门狗（默认10秒续期一次，锁自动延期）
try {
    // 业务逻辑
} finally {
    lock.unlock();  // 释放锁，自动停止看门狗
}

// Redisson 看门狗源码逻辑：
// lock() → scheduleExpirationRenewal(threadId)
// → 每 internalLockLeaseTime/3 = 10/3 = 3.3秒 执行一次续期
// → 续期：PEXPIRE lockKey 30000
// → unlock() 时 cancelExpirationRenewal(threadId) 停止续期
```

### 追问 6：Zset 的 ZRANGE 和 ZRANGEBYSCORE 有什么区别？

```sql
-- ZRANGE：按排名范围查
ZRANGE leaderboard 0 9 WITHSCORES     -- 取第 1~10 名（按分数升序）
ZREVRANGE leaderboard 0 9 WITHSCORES  -- 取第 1~10 名（按分数降序）

-- ZRANGEBYSCORE：按分数范围查
ZRANGEBYSCORE leaderboard 80 100       -- 取 80~100 分的所有人
ZREVRANGEBYSCORE leaderboard 100 80   -- 取 100~80 分的所有人（降序写法）

-- ZRANGEBYLEX：按字典序范围查（适用于分数相同的场景）
-- 分值相同的情况下，按 member 字典序排序（应用：活动排行榜）
```
