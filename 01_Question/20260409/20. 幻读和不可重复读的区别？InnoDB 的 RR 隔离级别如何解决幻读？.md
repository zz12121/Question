Phantom read and non-repeatable read are fundamentally different problems—one involves rows appearing for the first time in a query, while the other involves the same row changing between reads. InnoDB's Repeatable Read isolation level uses MVCC to solve non-repeatable reads through consistent snapshots, but still needs additional mechanisms like Next-Key Locking to fully address phantom reads. The distinction between current reads and snapshot reads is key here—MVCC handles the snapshot reads, while gap locks handle the write scenarios that could cause phantom data to appear.
## 先说结论

| | 不可重复读（Non-Repeatable Read） | 幻读（Phantom Read） |
|---|---|---|
| **问题本质** | 同一行数据被其他事务**修改**了 | 其他事务**新增/删除了行**，导致两次查询结果集**行数变了** |
| **发生场景** | 读-改-读：两次读取同一行，内容不一致 | 读-写-读：两次读取整个结果集，**多了或少了行** |
| **侧重点** | 已有行的数据内容变化 | 结果集中**出现全新的行** |
| **RR 隔离级别** | ✅ MVCC 快照读完全解决 | ⚠️ 快照读基本解决，当前读靠 Next-Key Lock 彻底解决 |

## 逐个详解

### 不可重复读：MVCC 快照读 ✅ 已解决

```sql
-- 事务A（RR 隔离级别）
START TRANSACTION;

-- 第一次读：生成 ReadView，此时事务B还没提交
SELECT * FROM users WHERE id = 1;
-- 结果：age=25

-- 事务B（另一会话）：UPDATE users SET age=30 WHERE id=1; COMMIT;
-- 事务A 中间时刻插一句：事务B提交了

-- 第二次读：复用同一个 ReadView，age=25 永远不变 ✅
SELECT * FROM users WHERE id = 1;
-- 结果：age=25（看不到事务B的修改）
```

**原理**：RR 级别下，ReadView 只在事务**第一次 SELECT 时生成**，之后复用。同一个事务内多次读同一行，读到的永远是快照版本。

---

### 幻读：快照读 + 当前读 分别处理

#### 快照读（普通 SELECT）：RR 靠 MVCC 基本解决

```sql
-- 事务A（RR）
START TRANSACTION;
SELECT * FROM orders WHERE status='pending';  -- 读到 5 条，快照版本
-- 事务B此时插入了一条新订单，status='pending'
COMMIT;
SELECT * FROM orders WHERE status='pending';  -- 仍然读到 5 条 ✅
-- 快照读，靠 MVCC，事务B的新行不在 ReadView 可见范围内
```

#### 当前读（SELECT FOR UPDATE / INSERT / UPDATE / DELETE）：靠 Next-Key Lock 彻底解决

```sql
-- 事务A（RR）当前读，对索引加 Next-Key Lock
SELECT * FROM orders WHERE status='pending' FOR UPDATE;
-- 锁住所有 status='pending' 的行 + 这些行之间的间隙
-- 其他事务无法在这个范围内插入新行

-- 事务B：INSERT INTO orders(id, status) VALUES(99, 'pending');
-- BLOCKED！等待锁，或直接报死锁
```

**Next-Key Lock 的锁区**：
```
假设 status='pending' 的行有：id=1, id=5, id=10
Next-Key Lock 锁住：
- (负无穷, 1]           ← 记录锁 + 间隙
- (1, 5]               ← 记录锁 + 间隙
- (5, 10]              ← 记录锁 + 间隙
- (10, 正无穷)         ← 记录锁 + 间隙
-- 其他事务无法在 (负无穷, 正无穷) 的任何位置插入 status='pending' 的新行
```

---

## RR 隔离级别的完整防幻读策略

```
事务开启
    ↓
第一次 SELECT → 生成 ReadView（MVCC 快照读）
    ↓
后续普通 SELECT → 复用 ReadView（快照读，解决不可重复读 + 基本幻读）
    ↓
当前读（FOR UPDATE / INSERT / UPDATE / DELETE）
    ↓
Next-Key Lock 锁住相关索引区间（彻底防止幻写）
```

**两个机制配合**：
- MVCC 解决 **快照读** 的幻读问题
- 锁机制解决 **当前读** 的幻写问题

## 🔥 扩展追问

### 追问 1：MVCC + Next-Key Lock 为什么不能完全杜绝所有幻读？

**理论上确实不能100%杜绝**，但实际场景已经极少了。

```sql
-- 场景：先快照读，后当前读，中间有缝隙

事务A：SELECT * FROM orders WHERE id > 10;  -- 快照读，读到 id=15
                                                 -- 幻读窗口！
事务B：INSERT INTO orders(id) VALUES(12);  -- id=12 在 (10, 正无穷) 区间
事务B：COMMIT;
事务A：SELECT * FROM orders WHERE id > 10 FOR UPDATE;  -- 当前读，锁区间时发现 id=12 已存在
-- 读到 id=12 的行，但这个行是 B 插入的，行为上表现为"幻读"
```

> **面试加分话术**：这种边界情况非常极端，实际工程中99.9%的幻读场景被 RR + Next-Key Lock 完全覆盖了。

### 追问 2：MVCC 快照读能读到其他事务的 DELETE 吗？

**能读到_deleted 标记的行**，因为 DELETE 在 InnoDB 内部是**标记删除**（更新 DELETE_FLAG=1），不是真正删除：

```sql
-- 事务B
DELETE FROM users WHERE id = 1;  -- 内部变成 UPDATE，设置删除标记

-- 事务A 快照读时：
-- 版本链中该行 DB_ROLL_PTR 指向的 Undo 记录还存在
-- 可见性判断：DELETE_FLAG 未被当前 ReadView 确认 → 该行对事务A不可见
-- 结果：事务A 读不到 id=1 这行（看似被删了，但其实只是标记删除）

-- 但如果版本链中有旧版本记录，且当前版本被标记删除：
-- 顺着 ROLL_PTR 读上一条 Undo 版本即可
```

### 追问 3：为什么 RC 级别也会有幻读？

```sql
-- 事务A（RC）
START TRANSACTION;

-- 第一次 SELECT：生成 ReadView-1
SELECT * FROM orders WHERE status='pending';  -- 读到 5 条

-- 事务B 插入一条新订单并提交
INSERT INTO orders VALUES(6, 'pending'); COMMIT;

-- 第二次 SELECT：重新生成 ReadView-2（RC 每次都重新生成！）
SELECT * FROM orders WHERE status='pending';  -- 读到 6 条 ← 幻读！
```

**RC 幻读原因**：每次 SELECT 都重新生成 ReadView，事务B提交后，新 ReadView 就能看到那条新插入的行。

### 追问 4：ReadView 可见性判断的代码级逻辑

```sql
-- 伪代码：MySQL 源码层可见性判断
bool is_visible(trx_id) {
    // 1. 如果是自己改的，可以读
    if (trx_id == creator_trx_id) return true;

    // 2. 如果比当前最小活跃事务ID还小，一定已提交
    if (trx_id < min_active_trx_id) return true;

    // 3. 如果在自己之后开启，一定未提交
    if (trx_id >= max_trx_id) return false;

    // 4. 如果在活跃列表中，说明未提交，看不见
    if (trx_id in active_trx_ids) return false;

    // 5. 否则，已提交，可以读
    return true;
}
```

### 追问 5：RC 和 RR 隔离级别的实际选择建议

```sql
-- 读已提交（RC）适用场景：
-- 报表查询、数据同步、主从复制场景
-- 优点：并发性能好，减少锁冲突
-- 缺点：同一事务内查询结果可能不同，需要业务侧容忍

-- 可重复读（RR）适用场景：
-- 金融交易、订单处理、库存扣减等强一致性业务
-- 优点：事务内数据快照稳定
-- 缺点：间隙锁可能造成更严重的锁等待
```

> **阿里巴巴实战建议**：金融类业务强烈建议 RR；读多写少的数据分析用 RC；用 RR 时注意设计好索引范围，避免大范围间隙锁影响并发。

### 追问 6：串行化（Serializable）如何解决幻读？

```sql
-- Serializable 级别
-- InnoDB 自动将所有普通 SELECT 转换为 SELECT ... LOCK IN SHARE MODE
-- 即所有读都加共享锁，写全部串行

-- 等价于：
-- 快照读直接被禁止了，读全部变成当前读
-- 幻读理论上彻底不存在（因为所有读写都加锁）

-- 性能代价极大，互联网项目几乎不用
```

### 追问 7：MySQL 8.0 新增的隔离级别 SNAPSHOT 是什么？

```sql
-- MySQL 8.0 新增 READ COMMITTED SNAPSHOT（默认开启）
-- 其实是将 RC 级别的实现改为基于 MVCC（之前 RC 是用锁实现）
SET GLOBAL TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- 行为不变，但实现更高效（读不加锁）

-- MySQL 8.0 新增 NOWAIT 和 SKIP LOCKED
SELECT * FROM orders FOR UPDATE NOWAIT;  -- 有锁直接报错，不等待
SELECT * FROM orders FOR UPDATE SKIP LOCKED;  -- 跳过有锁的行
-- 高并发抢单/扣库存场景非常有用
```

## 一句话收尾

> **不可重复读是"同一行的内容变了"，幻读是"结果集里多了或少了行"；InnoDB 的 RR 级别用 MVCC 快照读解决不可重复读和快照读场景的幻读，用 Next-Key Lock 当前读锁住索引区间彻底解决幻写，两者配合让 RR 成为业务场景中最实用的隔离级别。**
