Key points to cover:
1. RDB - Redis Database snapshot, point-in-time copy
2. AOF - Append Only File, write-ahead log
3. Differences between them
4. Production configuration recommendations
5. Hybrid persistence (RDB + AOF in Redis 7.0+)
6. Extension questions about fork, COW, data recovery, etc.
## 先说结论

| | RDB | AOF |
|---|---|---|
| **原理** | 定时生成数据快照，存成 `.rdb` 文件 | 每个写操作追加到日志，存成 `.aof` 文件 |
| **文件大小** | 小（二进制，压缩后） | 大（文本，重写后变小） |
| **恢复速度** | 快（直接加载内存） | 慢（重放所有命令） |
| **数据完整性** | 差（两次快照之间可能丢数据） | 好（可配置每秒/每命令同步） |
| **IO 性能影响** | 定期 fork，主进程阻塞 | 持续写入，无 fork 开销 |
| **生产推荐** | 不单独用 | **AOF + RDB 混合持久化** |

## RDB — 定时快照

### 工作原理

```
主进程
    ↓
BGSAVE / 自动触发
    ↓
fork() 子进程 → COW（Copy-On-Write）
    ↓
子进程遍历内存，将数据序列化写入 RDB 文件
    ↓
写完后替换旧 RDB 文件
```

### 触发方式

```bash
# 1. 手动触发
redis-cli BGSAVE        # 后台异步，不阻塞主进程
redis-cli SAVE          # 同步执行，阻塞主进程（生产禁用）

# 2. 自动触发（redis.conf 配置）
save 900 1      # 900秒内≥1次写操作 → 触发 BGSAVE
save 300 10     # 300秒内≥10次写操作 → 触发 BGSAVE
save 60 10000   # 60秒内≥10000次写操作 → 触发 BGSAVE

# 3. SHUTDOWN 时自动触发 SAVE（默认开启）
shutdown save   # 关闭时顺便保存一次
```

### 优缺点分析

```
✅ RDB 优点：
- 文件紧凑（经 gzip 压缩），适合备份、灾备
- 恢复速度极快（直接加载 .rdb）
- fork 在后台执行，对主进程影响小
- 适合做全量冷备份（比如每晚 3 点备份一次）

❌ RDB 缺点：
- 两次快照之间的数据会丢失（如果 Redis 崩溃在快照之间）
  例：每5分钟 BGSAVE，4分59秒时崩溃 → 丢失近5分钟数据
- fork() 瞬间如果数据量大，可能内存翻倍（COW 机制撑住）
- 频繁 fork() 会影响响应延迟
```

## AOF — 命令日志

### 工作原理

```
客户端发送写命令
    ↓
主进程将命令追加到 AOF Buffer
    ↓
根据配置的同步策略，刷盘到 .aof 文件
    ↓
AOF 文件越来越大 → BGREWRITEAOF 重写瘦身
```

### 三种同步策略

```bash
# redis.conf 配置
appendonly yes
appendfsync everysec   # 推荐！兼顾性能和数据安全

# 三种策略对比：

# 1. always（每条命令都刷盘）
appendfsync always
# 数据最完整，但 QPS 大降（每次写都要等磁盘IO）
# 每秒只能处理 ~1万次写入

# 2. everysec（每秒刷一次，默认推荐）
appendfsync everysec
# 每秒后台线程刷一次，最多丢1秒数据
# 性能和数据安全的最佳平衡点 ✅

# 3. no（操作系统决定）
appendfsync no
# 交给 OS，Linux 默认 30 秒左右刷一次
# 性能最好，但最多丢 30 秒数据，最不推荐
```

### AOF 重写（BGREWRITEAOF）

```bash
# 为什么需要重写？
# AOF 记录每条写命令，100 次 INCR counter = 100 条记录
# 重写后变成：SET counter 100 = 1 条记录

# 触发条件（redis.conf）
auto-aof-rewrite-percentage 100  # AOF 文件比上次重写后增长 100% → 触发
auto-aof-rewrite-min-size 64mb   # AOF 文件至少 64MB 才触发重写（防止小文件频繁重写）

# 手动触发
redis-cli BGREWRITEAOF  # 后台异步执行，不阻塞
```

**重写原理**：
```
主进程
    ↓
fork() 子进程
    ↓
子进程遍历内存，对每个 key 只保留最新一条命令
    ↓
同时主进程还在接收新命令，追加到 AOF 重写缓冲区
    ↓
子进程写完新 AOF → 通知主进程
    ↓
主进程把 AOF 重写缓冲区的命令追加到新 AOF 文件
    ↓
用新 AOF 替换旧 AOF
```

## 生产环境最佳配置

### Redis 7.0 之前：RDB + AOF 混合（手动）

```bash
# redis.conf — 持久化配置
save 900 1
save 300 10
save 60 10000
# ↑ RDB 定时快照作为冷备份

appendonly yes
appendfsync everysec
# ↑ AOF 作为实时数据安全保证

auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
# ↑ AOF 重写，避免文件无限膨胀

aof-load-truncated yes
# ↑ AOF 文件被截断时自动恢复（避免重写一半的 AOF 无法加载）

# AOF 开启后 RDB 功能仍然生效
# 恢复时：AOF 重放恢复 → 但 AOF 太慢
# 所以建议：恢复时用 RDB + 最新 AOF 做增量
```

### Redis 7.0+：混合持久化（推荐！）

```bash
# redis.conf — Redis 7.0+
aof-use-rdb-preamble yes   # 默认开启！混合持久化

# 工作原理：
# - AOF 文件开头是一份 RDB 快照（binary 格式）
# - 后面追加的是 AOF 命令（text 格式）
# 
# 恢复时：
# 1. 先识别 RDB 头 → 快速加载 RDB 部分
# 2. 再重放后面的 AOF 命令
# → 恢复速度接近 RDB，数据完整性接近 AOF ✅
```

```
AOF 文件结构（Redis 7.0+）：
┌──────────────┐
│  RDB Binary  │  ← 内存快照压缩存储（快速加载）
├──────────────┤
│  AOF CMD 1   │  ← RDB 生成后的增量写命令
│  AOF CMD 2   │
│  AOF CMD 3   │
└──────────────┘
```

### 完整生产配置示例

```bash
# ========== 持久化配置 ==========
# 混合持久化（Redis 7.0+ 默认开启）
aof-use-rdb-preamble yes
appendonly yes
appendfsync everysec
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
aof-load-truncated yes

# RDB 快照（冷备份）
save 900 1
save 300 10
save 60 10000
stop-writes-on-bgsave-error yes   # BGSAVE 失败时停止写入（防数据丢失）
rdbcompression yes               # RDB 压缩（CPU换空间）
rdbchecksum yes                  # RDB 校验（数据完整性）

# ========== 内存配置 ==========
maxmemory 8gb                    # 最大内存（根据服务器配置调整）
maxmemory-policy allkeys-lru     # 内存满时淘汰策略
maxmemory-samples 5              # LRU 采样数（越多越精确，越耗CPU）

# ========== 子进程配置 ==========
client-output-buffer-limit normal 256mb 64mb 60
client-output-buffer-limit replica 256mb 64mb 60
client-output-buffer-limit pubsub 32mb 8mb 60
# ↑ 防止复制缓冲区、订阅缓冲区撑爆内存

# ========== 备份相关 ==========
dir /data/redis                 # 持久化文件存放目录
dbfilename dump.rdb             # RDB 文件名
appendfilename "appendonly.aof"  # AOF 文件名
```

## 🔥 扩展追问

### 追问 1：RDB 的 fork() 和 COW 机制详解

```
fork() 是类 Unix 系统的进程复制操作：
    ↓
fork() 后，父子进程共享同一份物理内存（共享内存段）
    ↓
当任意一方要对某内存页写操作时：
    → 触发 COW（Copy-On-Write）
    → 复制出一份新内存页给写操作方
    → 另一方继续共享原来的内存页
    ↓
整个过程：
- fork() 是瞬间完成的（不复制物理内存）
- fork() 后物理内存增长 ≈ 0
- 真正复制的只有被修改的那些页
- 写操作越多，COW 消耗的内存越大
```

```
RDB BGSAVE 内存翻倍场景：
Redis 内存 8GB
    ↓
fork() 后，父子进程共享 8GB 物理内存（COW）
    ↓
子进程在写 .rdb 文件时，遍历内存：
    → COW 触发，复制被访问过的内存页
    → 最大复制量 ≈ 8GB（最坏情况：遍历了所有内存页）
    ↓
此时最大内存占用：8GB（共享）+ 8GB（COW）= 16GB
    ↓
这就是为什么 Redis 服务器内存必须 ≥ 2倍 Redis 实际使用量
```

> **面试加分话术**：我之前碰到过一个案例，Redis 8GB 内存，服务器只有 10GB，结果 BGSAVE 时 OOM 了。后面加了 swap 才恢复数据，根本原因就是没预估 COW 的内存开销。

### 追问 2：Redis 崩溃后如何恢复数据？

```bash
# 恢复优先级（Redis 会自动按这个顺序处理）

# 情况1：有 AOF + RDB（混合持久化）
# Redis 自动检测 AOF 文件是否包含 RDB 头
# → 先加载 RDB 部分（快速恢复）
# → 再重放 AOF 部分（补充增量）

# 情况2：只有 AOF
redis-server --appendonly yes --dir /data/redis
# → 自动加载 appendonly.aof 并重放所有命令
# → 如果 AOF 文件损坏（结尾不完整）
redis-check-aof --fix /data/redis/appendonly.aof
# → 会询问是否 truncate 损坏的部分
# → 修复后数据可能少量丢失

# 情况3：只有 RDB
redis-server --dbfilename dump.rdb --dir /data/redis
# → 自动加载 dump.rdb

# 情况4：同时有 AOF 和 RDB，但 AOF 更新
# → Redis 优先使用 AOF（RDB 可能已经过期）
# → 如果 AOF 损坏，Redis 会拒绝启动（除非设置 aof-load-truncated yes）
```

### 追问 3：AOF 文件越来越大的问题怎么解决？

```
问题：AOF 记录所有写命令，1亿次写操作 = 1亿条命令
     文件从最初的 100MB 膨胀到 10GB

原因：
- 每条命令都是文本（SET key value = ~20 字节）
- DEL key 后又 SET → 中间多了一条无效命令
- 计数器 INCR 1000次 → 1000 条命令，实际只需 SET key 1000

解决：BGREWRITEAOF 重写
    ↓
原理：遍历内存，只保留每个 key 的最终状态
    ↓
SET key A → SET key B → SET key C = 最终只保留 SET key C
INCR key 1000次 = 最终只保留 SET key 1000
    ↓
10GB 文件重写后可能变成 500MB
```

### 追问 4：RDB vs AOF 性能对比

```bash
# 测试脚本：100万次写操作
测试场景：Redis 4核 8GB 内存

RDB（每分钟 BGSAVE）：
- BGSAVE 耗时：约 3 秒
- 期间主进程：几乎无影响（COW 机制）
- QPS：正常 ~10万/秒
- 丢数据风险：最多丢1分钟

AOF（everysec）：
- 每次刷盘：约 0.5 秒
- 期间主进程：可以继续处理请求
- QPS：~8万/秒（刷盘期间略降）
- 丢数据风险：最多丢1秒

RDB + AOF 混合（Redis 7.0+）：
- 恢复速度：接近纯 RDB（因为开头是 RDB 二进制）
- 数据完整性：和 AOF 一样（因为后面是增量命令）
- 实际生产推荐这个方案 ✅
```

### 追问 5：主从复制中，RDB 和 AOF 的配合

```bash
# 主从架构下的持久化策略：
# 
# 从节点（replica）：
#   → 默认接收主节点的 RDB 传输
#   → 从节点可以用 AOF 做本地持久化（但一般不建议开启）
# 
# 主节点：
#   → 必须开启持久化（AOF 或 RDB）
#   → 否则主节点宕机重启后数据为空 → 从节点同步空数据 → 全库数据丢失！
# 
# 正确配置：
# 主节点：RDB + AOF 混合持久化
# 从节点：RDB 持久化（从主节点 RDB 同步即可）
```

### 追问 6：如何验证持久化是否正常工作？

# 1. 检查 RDB 文件是否存在

# 2. 检查最近一次 BGSAVE 时间

# 3. 检查 AOF 重写状态

## 一句话收尾

> **RDB 是定时拍快照，适合做冷备份和快速恢复；AOF 是实时写日志，适合保证数据不丢；生产环境推荐 Redis 7.0+ 混合持久化（`aof-use-rdb-preamble yes`），兼顾 RDB 的恢复速度和 AOF 的数据完整性，同时记得主节点必须开持久化，否则主从架构下一旦主节点重启就是一场数据灾难。**
