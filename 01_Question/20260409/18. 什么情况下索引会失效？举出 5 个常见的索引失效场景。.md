让我整理5个常见的索引失效场景：
1. 函数/计算 - 在索引列上使用函数或表达式
2. 类型转换 - 隐式类型转换导致不走索引
3. 最左前缀违反 - 跳过联合索引的前缀列
4. LIKE前缀匹配 - LIKE '%xxx' 开头


5. OR连接 - OR两边不是同列或一边不满足索引
6. NOT操作 - !=、<>、NOT IN、IS NOT NULL
7. 范围列之后 - 范围条件后的列无法使用索引
8. 使用SELECT * - 回表开销大，MySQL可能选择全表扫描
9. 数据量过小 - 全表扫描比索引查询更快时
---
## 先说结论

索引失效就是 **MySQL 优化器判断"走索引还不如全表扫描快"**，然后偷偷给你换成 `type=ALL`。常见的坑有：**函数/计算、隐式类型转换、跳过最左前缀、LIKE 前缀通配、OR 条件不均匀**。下面逐个讲。

---

## 5 个常见索引失效场景

### 场景一：在索引列上使用函数或表达式

```sql
-- 有索引 idx_create_time(create_time)
SELECT * FROM orders WHERE YEAR(create_time) = 2025;           -- ❌ 失效
SELECT * FROM orders WHERE create_time + INTERVAL 1 DAY > NOW(); -- ❌ 失效

-- 正确写法：把计算移到右边常量上
SELECT * FROM orders WHERE create_time >= '2025-01-01';          -- ✅ 有效
SELECT * FROM orders WHERE create_time >= DATE_SUB(NOW(), INTERVAL 1 DAY); -- ✅ 有效
```

**原理**：索引树存的是 `create_time` 的原始值，你一加函数，MySQL 就不知道拿哪个值去比较了，必须把索引列套上函数再比较，等于每行都要算一遍，优化器干脆放弃索引。

---

### 场景二：隐式类型转换

```sql
-- user_id 是 VARCHAR(20)，存的是数字字符串
SELECT * FROM users WHERE user_id = 12345;         -- ❌ 隐式转成数字再比较
SELECT * FROM users WHERE user_id = '12345';       -- ✅ 加引号显式指定

-- 实际执行：
SELECT * FROM users WHERE CAST(user_id AS unsigned) = 12345;
-- 索引列被强制 CAST，索引失效
```

**原理**：MySQL 会把字符串列和数字比较时，把字符串转成数字，这就相当于在索引列上套了一层隐式函数。

> **面试加分话术**：我之前有个线上 bug，就是 UUID 存成了 VARCHAR，查的时候没加引号，导致 QPS 翻了几倍。

---

### 场景三：违反最左前缀原则

```sql
-- 联合索引 idx_a_b_c(a, b, c)
SELECT * FROM t WHERE a = 1 AND b = 2 AND c = 3;    -- ✅ 三列都用上了
SELECT * FROM t WHERE a = 1 AND c = 3;              -- ✅ 用到 a，跳过 b 到 c
SELECT * FROM t WHERE b = 2 AND c = 3;              -- ❌ 没有 a，从头跳过，失效
SELECT * FROM t WHERE a = 1 AND b > 2 AND c = 3;    -- ⚠️ c 失效（b 是范围）
```

```
索引结构：[a=1, b=2] → [a=1, b=3] → [a=2, b=1] → ...

a=1 时：能定位到一个连续的区间 ✅
b=2 时：不知道 a 是多少，无法定位 ❌
```

> **小技巧**：范围列（`>`、`<`、`BETWEEN`）之后的列都用不上索引。

---

### 场景四：LIKE 以通配符开头

```sql
-- title 有索引
SELECT * FROM articles WHERE title LIKE 'Java面试%';   -- ✅ 前缀确定，走索引
SELECT * FROM articles WHERE title LIKE '%Java面试%';  -- ❌ 不知道前缀，全表扫
SELECT * FROM articles WHERE title LIKE '%Java面试';   -- ❌ 同上
```

**原理**：B+ 树靠前缀有序排布，`%` 开头意味着你不知道从哪个节点开始找，只能从根节点遍历全部。

> **延伸问题**：如果真的需要模糊匹配怎么办？
> - 全文索引：`MATCH(title) AGAINST('Java面试')`
> - ES/MySQL 8 支持 `ngram` 分词
> - 前缀长度可控时，可用 `LIKE 'Java%'`

---

### 场景五：OR 连接的条件不满足索引

```sql
-- user_id 有索引，status 没有索引
SELECT * FROM users WHERE user_id = 1 OR status = 'active';     -- ❌ 失效
SELECT * FROM users WHERE user_id = 1 OR email = 'a@b.com';      -- ❌ 失效

-- 改成 UNION
SELECT * FROM users WHERE user_id = 1
UNION ALL
SELECT * FROM users WHERE status = 'active' AND user_id != 1;   -- ✅ 两个都能走索引
```

**原理**：OR 要求两边条件都能用索引，只要有一边不满足，MySQL 就认为不如全表扫。

> **小技巧**：如果 OR 两边都是同列的索引，`OR` 不一定会失效。比如 `id > 1 OR id < 10` 这种。

## 🔥 扩展追问

### 追问 1：还有哪些容易忽略的索引失效场景？

```sql
-- 6. 使用 NOT、!=、<> 操作符
SELECT * FROM orders WHERE status != 'completed';    -- ❌ 可能失效
SELECT * FROM orders WHERE status NOT IN ('a','b'); -- ❌ 可能失效

-- 7. IS NULL / IS NOT NULL
SELECT * FROM users WHERE phone IS NULL;            -- ⚠️ 取决于数据分布
SELECT * FROM users WHERE phone IS NOT NULL;        -- ⚠️ 同上

-- 8. 数据量过小时走全表扫描更优
-- MySQL 优化器基于统计信息判断，估算行数少时直接全表扫描
```

### 追问 2：MySQL 优化器是如何判断走索引还是全表扫描的？

关键看 **成本（Cost）**：

| 成本因素 | 说明 |
|---------|------|
| 估算行数 | 通过统计信息（`ANALYZE TABLE` 更新） |
| 随机 IO 次数 | 索引深度的磁盘读取次数 |
| 顺序 IO 效率 | 全表扫描的顺序读 vs 索引的随机读 |
| 回表成本 | 索引查到主键后，再回表查完整行数据 |

```
EXPLAIN 输出中的 cost 字段可以看到估算成本：
cost=1000 vs cost=50 → MySQL 选 cost 低的
```

> **面试加分**：可以用 `EXPLAIN ANALYZE`（MySQL 8.0）看到实际执行成本和估算成本的对比。

### 追问 3：什么是回表？如何减少回表？

```sql
-- t 表：id(主键), name, age, email
-- 索引：idx_age(age)

-- 查 age=25 的 name（name 不在索引里）
SELECT name FROM t WHERE age = 25;
-- 1. 先用 idx_age 找到 age=25 对应的主键 id
-- 2. 再用主键索引回表查完整的 name 字段  ← 回表

-- 优化：使用覆盖索引，把要查的字段也放进索引
-- 索引：idx_age_name(age, name)
SELECT name FROM t WHERE age = 25;  -- 直接在索引树拿到 name，无需回表 ✅
```

**回表次数 = 匹配的行数**，如果回表次数很多，MySQL 觉得不如直接全表扫，就放弃索引了。

### 追问 4：如何排查索引失效？

```sql
-- 1. EXPLAIN 查看执行计划
EXPLAIN SELECT * FROM orders WHERE create_time >= '2025-01-01';

-- 2. 关键字段：
--    type: ALL = 全表扫描, range = 范围扫描, ref = 索引查找
--    key: 实际用的索引名
--    rows: 估算扫描行数
--    extra: Using filesort / Using temporary / Using index = 覆盖索引

-- 3. MySQL 8.0+: EXPLAIN ANALYZE 看实际执行
EXPLAIN ANALYZE SELECT ...

-- 4. 检查统计信息是否过期
SHOW TABLE STATUS FROM db LIKE 'orders';
-- Rows 列就是估算值，和实际差太多就该 ANALYZE 了

ANALYZE TABLE orders;  -- 更新统计信息
```

### 追问 5：索引列上有函数就一定失效吗？

**不一定！** MySQL 8.0 引入了**函数索引**：

```sql
-- 普通写法（失效）
SELECT * FROM t WHERE YEAR(create_time) = 2025;

-- MySQL 8.0+ 函数索引（生效）
ALTER TABLE t ADD INDEX idx_year ((YEAR(create_time)));
SELECT * FROM t WHERE YEAR(create_time) = 2025;  -- ✅ 走 idx_year
```

> **面试加分**：这也是为什么阿里巴巴《Java开发手册》要求禁止在索引列上做计算——因为老版本 MySQL 没有函数索引。

### 追问 6：JOIN 查询的索引失效

```sql
-- t1 join t2，t2 的外键列没有索引
SELECT * FROM t1 JOIN t2 ON t1.id = t2.t1_id;
-- t2.t1_id 没有索引 → t2 全表扫，JOIN 效率极差

-- 驱动表选择错误
EXPLAIN SELECT * FROM t1 JOIN t2 ON t1.id = t2.t1_id;
-- 看谁是小表（被驱动表有索引更优）
```

### 追问 7：ORDER BY 和 GROUP BY 与索引的关系

```sql
-- 联合索引 idx(a, b, c)
SELECT * FROM t WHERE a = 1 ORDER BY b, c;  -- ✅ 利用索引天然有序
SELECT * FROM t WHERE a = 1 ORDER BY c, b;  -- ❌ 文件内排序（Using filesort）
-- ORDER BY 的顺序必须和索引列顺序一致，且方向一致
```

> **GROUP BY 也是同样的原理**：如果能在索引上直接完成分组，就不需要临时表。

---

## 一句话收尾

> **索引失效的本质是"索引列上动了手脚"——加函数、类型转换、跳过列顺序，都会让 MySQL 优化器觉得索引不如全表扫描经济，果断放弃。排查用 `EXPLAIN`，治本靠 `ANALYZE TABLE` 更新统计信息 + 合理设计联合索引。**
