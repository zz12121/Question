## 先说结论

|方面|ArrayList|LinkedList|
|---|---|---|
|底层结构|**动态数组**|**双向链表**|
|随机访问|O(1) ✅ 快|O(n) ❌ 慢|
|头部操作|O(n) ❌ 慢|O(1) ✅ 快|
|尾部操作|O(1) 均摊|O(1) ✅ 快|
|中间插入/删除|O(n)|O(1) ✅（如果已知位置）|
|内存占用|小（只需数组空间）|大（要存前后指针）|
|缓存友好|✅ CPU 缓存友好|❌ 内存分散|

---

## 核心原理

### ArrayList 内存布局

```
内存连续：[元素0][元素1][元素2][元素3][元素4]...
          ↑
        首地址，CPU 一读一片（缓存友好）
```

### LinkedList 内存布局

```
内存分散：[元素0] ↔ [元素1] ↔ [元素2] ↔ [元素3] ↔ [元素4]
          ↓         ↓         ↓         ↓         ↓
         内存1     内存2     内存3     内存4     内存5
（任意位置，物理不连续）
```

## 🔥 扩展追问 1：ArrayList 尾部操作是 O(1) 吗？

**是，但不是严格意义上的 O(1)。**

```java
// ArrayList add 到尾部
public boolean add(E e) {
    ensureCapacityInternal(size + 1);  // 可能触发扩容！O(n)
    elementData[size++] = e;
    return true;
}
```

**情况分析：**

- **不扩容**：直接放元素 → O(1)
- **需要扩容**：新建数组 + 拷贝所有元素 → O(n)，但扩容是 **均摊 O(1)**

> 均摊的意思：虽然偶尔要 O(n)，但 N 次操作总共 O(N)，平均每次还是 O(1)。

## 🔥 扩展追问 2：ArrayList 扩容机制？


```java
// 扩容公式
newCapacity = oldCapacity + (oldCapacity >> 1)
// 即：新容量 = 旧容量 + 旧容量的一半
// 16 → 24 → 36 → 54 → ...
```

**流程：**

```
add 第 10 个元素 → 容量 16，不用扩
add 第 17 个元素 → 超出容量，触发扩容
    ↓
创建新数组（容量 24）
    ↓
System.arraycopy 拷贝原数组所有元素
    ↓
放入新元素
```

**为什么不是 2 倍？**

- 1.5 倍更保守，避免一次性占用太多内存
- 其实 JDK 6 之前是 2 倍，后来改成 1.5 了

## 🔥 扩展追问 3：什么场景下 LinkedList 比 ArrayList 慢？

### 场景一：按索引访问（最常见！）

```java
// ArrayList：直接算地址，O(1)
E get(int index) {
    return elementData[index];  // 一步到位
}

// LinkedList：要从头或从尾遍历，O(n)
E get(int index) {
    // JDK 优化了：index < size/2 从头找，否则从尾找
    // 但还是 O(n)
    Node<E> x = first;
    for (int i = 0; i < index; i++) {
        x = x.next;
    }
    return x.item;
}
```

**实测对比（10万次随机访问）：**

```
ArrayList:   5ms
LinkedList:  1200ms  ← 差了 200 倍！
```

### 场景二：尾部追加（ArrayList 反而更快！）

很多人以为链表增删快，这是**最大的误解**。

```java
// LinkedList 尾插
linkedList.add("a");  // 确实是 O(1)

// 但！
// 实际上 LinkedList 每次 add 都要 new Node 对象
// 而且要修改前后指针
// 现代 JVM 优化后，new 对象开销不小

// ArrayList 尾插
arrayList.add("a");   // 大部分情况也是 O(1)
// 即使扩容，平均下来也是均摊 O(1)
// 而且不需要 new 节点，就是个数组赋值
```

**实测（100万次 add）：**

```
ArrayList:   80ms
LinkedList:  200ms  ← ArrayList 反而快！
```

### 场景三：范围遍历

```java
// 遍历 10 万元素
for (int i = 0; i < list.size(); i++) {
    list.get(i);  // ArrayList: O(1) × N = O(n)
                  // LinkedList: O(n) × N = O(n²) ← 灾难！
}
```

**如果真要遍历 LinkedList，正确姿势：**

```java
// 用迭代器，LinkedList 才能发挥优势
for (String s : linkedList) { ... }  // 迭代器内部是顺着指针走的
// 但 fori + get(i) 就是灾难
```

## 🔥 扩展追问 4：为什么链表遍历反而要用迭代器？

```java
// Java 增强 for 循环（编译后实际是迭代器）
for (String item : linkedList) {
    // 编译后等价于：
    Iterator<String> it = linkedList.iterator();
    while (it.hasNext()) {
        String item = it.next();
    }
}
```

**迭代器遍历的优势：**

```
LinkedList 迭代器：next() 就是 x = x.next → O(1)
LinkedList get(i)：每次都要从头/尾重新走 → O(n)
```

**所以：**

- **遍历用迭代器**：LinkedList 也可以很优雅
- **按索引随机访问**：LinkedList 就是垃圾

## 🔥 扩展追问 5：LinkedList 为什么内存占用大？

```java
// ArrayList 存一个元素
Object[] elementData;  // 每个位置只存元素本身

// LinkedList 存一个元素
class Node<E> {
    E item;        // 存元素
    Node<E> prev;  // 指向前一个 ← 多了 8 字节（JVM 64位）
    Node<E> next;  // 指向后一个 ← 多了 8 字节
}
// 每个元素多 16 字节，100万个元素多 ~16MB！
```

**另外：**

- 数组元素在内存中是连续的，CPU 缓存一行能加载多个
- 链表节点分散在堆里，CPU 缓存命中极低

## 🔥 扩展追问 6：HashMap 里的链表为什么要转红黑树？

因为链表的查询是 **O(n)**，太慢。

```
链表长度 = 8 时查询要遍历 8 次
红黑树 O(logn) ≈ 3 次

而且红黑树在增删时通过旋转保持平衡
查找、插入、删除都是 O(logn)
```

**为什么不直接用红黑树替代链表？**

- 红黑树节点更大（要存颜色、父节点等）
- 节点少的时候，链表的 O(n) 其实也很快
- 链表简单，省内存

## 🔥 扩展追问 7：工作中怎么选？

**用 ArrayList：**

- 90% 的场景
- 按索引随机访问
- 尾部追加（这才是日常高频操作）
- 需要缓存友好

**用 LinkedList：**

- 需要频繁头插/头删
- 队列、栈的实现
- 真正的中间插入/删除（但得配合迭代器）
- 数据量超大且内存充足

**实际建议：**

> "除非有明确的性能瓶颈证明需要用 LinkedList，否则无脑 ArrayList。因为开发中随机访问和遍历远比中间增删多，而且 ArrayList 缓存友好 + 内存紧凑，实际性能往往更好。"

---

## 一句话收尾

> "ArrayList 是连续数组，查快插删慢；LinkedList 是双向链表，插删快但查慢。按索引随机访问时 LinkedList 能慢上百倍，所以日常开发 ArrayList 是首选。"
