# 21. 一条 SQL 执行很慢，你的排查思路是什么？EXPLAIN 的字段你都关注哪些？

---

## 先说结论

**排查链路**：慢查询日志 → EXPLAIN 分析 → 索引审查 → 表统计信息 → 锁分析 → 改写 SQL / 加索引。

**EXPLAIN 重点看**：`type`（连接类型）、`key`（实际用到的索引）、`rows`（扫描行数）、`extra`（Using filesort / Using temporary / Using index）四个字段。

---

## 完整排查思路

### 第一步：定位慢 SQL

```sql
-- 1. 查看是否开启慢查询日志
SHOW VARIABLES LIKE 'slow_query_log%';
SHOW VARIABLES LIKE 'long_query_time';  -- 超过这个秒数才记录

-- 2. 查看慢查询日志内容（文件路径）
SHOW VARIABLES LIKE 'slow_query_log_file';
-- 读出来找到慢 SQL

-- 3. MySQL 8.0+：用 sys.schema_table_statistics_with_buffer
SELECT * FROM sys.statement_analysis ORDER BY exec_count DESC LIMIT 10;
SELECT * FROM sys.statements_with_sorting;
SELECT * FROM sys.statements_with_temp_tables;

-- 4. 实时抓当前执行的慢 SQL
SHOW PROCESSLIST;  -- 找 Time 很长、Command=Query 的行
```

### 第二步：EXPLAIN 分析执行计划

```sql
-- 基本用法
EXPLAIN SELECT * FROM orders WHERE user_id = 100 AND status = 'pending';

-- MySQL 8.0+：EXPLAIN ANALYZE（会实际执行并显示真实成本）
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 100 AND status = 'pending';
```

**输出示例**：
```
+----+-------------+--------+------+---------------+------+---------+------------+------+----------+-------+
| id | select_type| table  | type | key           | rows | filtered| Extra      | cost | actual   | rRows |
+----+-------------+--------+------+---------------+------+---------+------------+------+----------+-------+
|  1 | SIMPLE     | orders | ref  | idx_user_stat |  120 |  100.00 | Using where| 138  | 0.023ms  |    12 |
+----+-------------+--------+------+---------------+------+---------+------+----------+-------+-------+
```

### 第三步：逐字段解读

#### 1. type（连接类型）—— 最重要！

| type 值 | 含义 | 好坏 |
|--------|------|------|
| **system** | 表只有一行（系统表） | 最好 |
| **const** | 主键/唯一索引等值查询，最多匹配一行 | ✅ 好 |
| **eq_ref** | 多表 JOIN，对于每个驱动表行，只匹配一行 | ✅ 好 |
| **ref** | 非唯一索引等值匹配，返回所有匹配行 | ✅ 可接受 |
| **range** | 索引范围扫描（>, <, BETWEEN, IN, LIKE 前缀） | ✅ 可接受 |
| **index** | 全索引扫描（比 ALL 好，但仍然要遍历整个索引） | ⚠️ 差 |
| **ALL** | **全表扫描** | ❌ 最差！ |

```
❌ type=ALL = 全表扫描，一定要优化
⚠️ type=index = 扫描整个索引树，也要注意 rows
✅ type=const/eq_ref/ref/range = 正常
```

#### 2. key（实际使用的索引）

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 100;
-- key 列显示：idx_user_id
-- 说明真的用到了 idx_user_id 索引

-- 如果 key=NULL，说明没走索引
EXPLAIN SELECT * FROM orders WHERE status = 'pending';  -- status 无索引 → key=NULL
```

#### 3. rows（估算扫描行数）

```sql
-- 订单表 1000 万行，但查询只扫描了 12 行
EXPLAIN SELECT * FROM orders WHERE user_id = 100;
-- rows = 12  ✅ 索引非常有效

-- 但如果 rows = 5000000，说明走了全表扫描或范围太大
EXPLAIN SELECT * FROM orders WHERE create_time > '2024-01-01';
-- rows = 9865432  ⚠️ 扫描了接近全表，需要优化
```

#### 4. Extra（额外信息）—— 最容易被忽略！

```sql
-- ❌ Using filesort：无法用索引排序，需要额外文件排序
EXPLAIN SELECT * FROM orders WHERE user_id = 100 ORDER BY create_time DESC;
-- Extra: Using filesort ← create_time 排序无法利用索引

-- ❌ Using temporary：需要创建临时表
EXPLAIN SELECT * FROM t GROUP BY category;
-- Extra: Using temporary; Using filesort ← GROUP BY 既要临时表又要排序

-- ✅ Using index：覆盖索引，不需要回表
EXPLAIN SELECT user_id, status FROM orders WHERE user_id = 100;
-- Extra: Using index ← 查的字段全在索引里，直接返回

-- ✅ Using index condition：索引下推（ICP）
EXPLAIN SELECT * FROM orders WHERE user_id = 100 AND status LIKE 'pen%';
-- Extra: Using index condition ← ICP 优化，把条件下推给存储引擎层

-- ✅ Using where：需要回表后用 WHERE 条件过滤
EXPLAIN SELECT * FROM orders WHERE user_id = 100 AND remark = 'vip';
-- Extra: Using where; Using index ← 先用索引找到 user_id=100，再过滤 remark

-- ⚠️ Using join buffer (Block Nested Loop)：JOIN 缓冲，大表关联极慢
EXPLAIN SELECT * FROM t1 JOIN t2 ON t1.id = t2.t1_id;
-- Extra: Using join buffer (Block Nested Loop) ← t2 没索引，需要暴力嵌套循环
```

#### 5. key_len（索引使用的字节数）

```sql
-- 联合索引 idx_user_status(user_id, status)
-- user_id 是 INT（4字节），status 是 VARCHAR(20) 带长度前缀
EXPLAIN SELECT * FROM orders WHERE user_id = 100;
-- key_len = 4  ← 只用了第一列 user_id

EXPLAIN SELECT * FROM orders WHERE user_id = 100 AND status = 'pending';
-- key_len = 4 + 20*3 + 2 = 68 ← 用了两列
```

#### 6. possible_keys vs key

```sql
-- possible_keys：可供选择的索引有哪些
-- key：优化器最终选中的索引
EXPLAIN SELECT * FROM orders WHERE user_id = 100 AND order_no = 'A001';
-- possible_keys: idx_user_id, idx_order_no
-- key: idx_user_id  ← 优化器选了 user_id 索引（可能因为基数更高）
```

## 第四步：索引专项审查

```sql
-- 1. 查看表有哪些索引
SHOW INDEX FROM orders;

-- 2. 检查索引选择性（Cardinality）
SHOW INDEX FROM orders;
-- Cardinalinality/行数 = 选择性，越接近 1 越好
-- 性别字段（0/1）选择性极差，不适合单独建索引

-- 3. 检查索引统计信息是否过期
SHOW TABLE STATUS LIKE 'orders';
-- Rows: 估算值，可能和实际差很多

-- 4. 强制使用某个索引（诊断用）
EXPLAIN SELECT * FROM orders USE INDEX(idx_user_id) WHERE user_id = 100;

-- 5. 强制忽略某个索引（诊断用）
EXPLAIN SELECT * FROM orders IGNORE INDEX(idx_user_id) WHERE user_id = 100;
```

## 第五步：锁分析和连接数

```sql
-- 1. 查看当前锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;

-- 2. 查看当前锁
SELECT * FROM information_schema.INNODB_LOCKS;

-- 3. 查看事务等待状态
SELECT * FROM information_schema.INNODB_TRX;

-- 4. 查看连接数
SHOW STATUS LIKE 'Threads_connected';
SHOW VARIABLES LIKE 'max_connections';

-- 5. 查看最大连接数使用情况
SHOW STATUS LIKE 'Max_used_connections';
```

## 第六步：改写优化

```sql
-- ❌ 慢 SQL 示例
SELECT * FROM orders
WHERE YEAR(create_time) = 2025 AND MONTH(create_time) = 3;
-- 问题：索引列上用函数，全表扫描

-- ✅ 优化：范围查询
SELECT * FROM orders
WHERE create_time >= '2025-03-01' AND create_time < '2025-04-01';


-- ❌ 慢 SQL：OR 导致索引失效
SELECT * FROM users WHERE user_id = 1 OR age = 25;

-- ✅ 优化：拆成 UNION
SELECT * FROM users WHERE user_id = 1
UNION ALL
SELECT * FROM users WHERE age = 25 AND user_id != 1;


-- ❌ 慢 SQL：LIKE %开头
SELECT * FROM orders WHERE remark LIKE '%退款%';

-- ✅ 优化：全文索引（数据量大时）
SELECT * FROM orders WHERE MATCH(remark) AGAINST('退款');


-- ❌ 慢 SQL：大 OFFSET
SELECT * FROM orders ORDER BY id LIMIT 1000000, 10;
-- 问题：MySQL 先扫描前 100 万行再返回 10 行

-- ✅ 优化：延迟关联
SELECT * FROM orders o
INNER JOIN (SELECT id FROM orders ORDER BY id LIMIT 1000000, 10) t
ON o.id = t.id;


-- ❌ 慢 SQL：分页总数过多
SELECT SQL_CALC_FOUND_ROWS * FROM orders LIMIT 1000000, 10;
-- 问题：SQL_CALC_FOUND_ROWS 会强制全表扫描

-- ✅ 优化：游标分页（基于上一页最大 ID）
SELECT * FROM orders WHERE id > #{last_id} ORDER BY id LIMIT 10;
```

## 🔥 扩展追问

### 追问 1：EXPLAIN 和 PROFILING 的区别？

```sql
-- EXPLAIN：估算执行计划（不真正执行）
EXPLAIN SELECT * FROM orders WHERE user_id = 100;

-- PROFILING：真实执行后分析每一步耗时（MySQL 5.6 已废弃）
SET profiling = 1;
SELECT * FROM orders WHERE user_id = 100;
SHOW PROFILES;                    -- 列出所有查询的耗时
SHOW PROFILE FOR QUERY 1;         -- 某条查询各阶段耗时

-- MySQL 8.0+ 替代方案：performance_schema
SELECT * FROM performance_schema.events_statements_history;
SELECT * FROM performance_schema.events_statements_summary_by_digest;
```

### 追问 2：MySQL 优化器是如何选择索引的？

```
MySQL 优化器选择索引的核心依据：成本估算
    ↓
成本 = CPU成本 + IO成本
    ↓
CPU成本：估算处理行数（rows_filtered）、排序成本
IO成本：估算磁盘页读取次数
    ↓
统计信息（innodb_stats_sample_pages）决定估算精度
    ↓
成本最低的方案胜出
```

```sql
-- 查看优化器的成本估算（MySQL 8.0）
EXPLAIN FORMAT=JSON SELECT * FROM orders WHERE user_id = 100;
-- 输出中可以看到 cost_details

-- 如果优化器选错了索引，可以强制指定
SELECT * FROM orders FORCE INDEX(idx_user_id) WHERE user_id = 100;
```

### 追问 3：如何判断是否需要加索引？

```sql
-- 基准原则：
-- 1. WHERE 条件列 → 加索引
-- 2. JOIN 的 ON 条件列 → 加索引
-- 3. ORDER BY / GROUP BY 列 → 加索引（最好和 WHERE 共用同一索引）

-- 判断索引覆盖度：
EXPLAIN SELECT user_id, status, create_time FROM orders WHERE user_id = 100;
-- user_id 有索引，但查了 3 个字段
-- 如果只有 idx_user_id(user_id)：需要回表查 status 和 create_time
-- 如果有 idx_user_status_time(user_id, status, create_time)：覆盖索引，无需回表 ✅

-- 监控索引使用情况（MySQL 8.0）
SELECT * FROM performance_schema.table_io_waits_summary_by_index_usage
WHERE object_name = 'orders';
-- 查看索引是否被使用过，长期未使用的索引可以考虑删除
```

### 追问 4：MySQL 8.0 的 EXPLAIN 有哪些改进？

```sql
-- 1. EXPLAIN ANALYZE（最重磅！）
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 100;
-- 输出：
-- -> Filter: (orders.user_id = 100)  (cost=138 rows=12)
--   -> Using index condition          (actual time=0.023..0.031 rows=12 loops=1)
--     -> Index range scan on orders using idx_user_id  (cost=138 rows=120)
--                                              ↑ 估算成本   ↑ 实际成本

-- 2. EXPLAIN FORMAT=JSON 更详细
EXPLAIN FORMAT=JSON SELECT * FROM orders WHERE user_id = 100\G

-- 3. 可视化执行计划（MySQL Workbench / IDE 直接看）
-- 树形结构，节点上显示成本，一目了然
```

### 追问 5：慢 SQL 日志如何和生产告警联动？

```sql
-- my.cnf 配置
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 1        -- 超过 1 秒记录
log_queries_not_using_indexes = 1  -- 没走索引的也记录

-- 定时分析慢查询日志的工具
-- 1. mysqldumpslow（MySQL 自带）
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log
# 找出最慢的 10 条 SQL

-- 2. pt-query-digest（Percona Toolkit，最强大）
pt-query-digest /var/log/mysql/slow.log
# 按执行次数、总耗时、平均耗时分组排名

-- 3. Prometheus + Grafana 接入 MySQL Exporter
# 实时监控 queries_per_second、slow_query_rate
```

### 追问 6：如何用 ANALYZE TABLE 和 OPTIMIZE TABLE 优化？

```sql
-- ANALYZE TABLE：更新索引统计信息（让优化器估算更准确）
ANALYZE TABLE orders;
-- 场景：表数据量变化超过 10% 后，或发现 EXPLAIN rows 和实际差太多时

-- OPTIMIZE TABLE：重建表 + 重建索引（回收碎片）
OPTIMIZE TABLE orders;
-- 场景：大量 DELETE/UPDATE 后，数据页有大量碎片

-- ALTER TABLE 重建（效果更好，MySQL 5.6+ 支持在线 DDL）
ALTER TABLE orders ENGINE = InnoDB;
-- 或
ALTER TABLE orders FORCE;

-- 定期执行计划：
-- 每周日凌晨低峰期：
-- ANALYZE TABLE t; OPTIMIZE TABLE t;
```

### 追问 7：连接池和 SQL 性能的关系

```sql
-- 慢 SQL 不一定是 SQL 本身的问题，可能是连接等待

-- 查看当前连接状态
SHOW PROCESSLIST;
-- State: " Sending data" = 真正在查
-- State: "Waiting for table metadata lock" = 等锁
-- State: "Copying to tmp table" = 临时表排序
-- State: "Sending cached result to client" = 结果在发送中

-- 应用侧常见问题：
-- 1. 连接池太小 → 请求排队等待
-- 2. 查询未关闭连接 → 连接泄漏
-- 3. N+1 查询问题 → 1 次查询查出 N 条，再查 N 次详情
SELECT * FROM orders WHERE user_id = 100; -- 查出 20 个订单
-- 代码中循环查详情：
for order_id in [1,2,3...]: SELECT * FROM orders WHERE id = order_id;  ← 20次！
-- ✅ 优化：JOIN 一次查出
SELECT o.*, u.name FROM orders o LEFT JOIN users u ON o.user_id = u.id WHERE o.user_id = 100;
```

## 一句话收尾

> SQL 慢的本质是"扫描了不该扫的行"——全表扫描、索引选错、回表过多、排序/分组无索引、锁冲突都是诱因；EXPLAIN 的 type + key + rows + extra 四件套是排查标配，治本是建好联合索引 + 用好覆盖索引 + 定期 `ANALYZE TABLE` 更新统计信息。
