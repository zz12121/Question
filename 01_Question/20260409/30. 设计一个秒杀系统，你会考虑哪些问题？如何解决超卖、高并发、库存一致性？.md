这是一个综合性很强的系统设计题，秒杀系统涉及：
- 超卖问题
- 高并发处理
- 库存一致性
- 前端/后端优化
- 缓存策略
- 消息队列
- 数据库优化
- 限流熔断
- 幂等性
## 先说结论

秒杀系统的核心挑战是 **高并发 + 库存有限**，必须解决三个问题：

| 核心问题 | 本质 | 解决思路 |
|---------|------|---------|
| **超卖** | 多个请求同时读到库存=1，都去减库存 | **数据库唯一索引 + 乐观锁 + 限流** |
| **高并发** | 瞬时流量远超系统承受能力 | **分层拦截 + 缓存 + 异步** |
| **库存一致性** | 扣减后数据要一致，不能多扣也不能少扣 | **Redis预减 + MQ异步 + 最终一致性** |

**一句话答案**：通过 Redis 缓存拦截 90% 流量 → 消息队列削峰 → 数据库乐观锁保证库存不超卖，配合限流熔断保护系统。

## 二、核心问题及解决方案

### 1. 超卖问题

**问题本质**：多个请求同时读到 `stock = 1`，都执行 `stock - 1`，导致 stock 变成 -N。

**解决方案**：

```sql
-- 方案一：数据库唯一索引（最简单粗暴）
CREATE TABLE stock (
    id BIGINT PRIMARY KEY,
    product_id BIGINT,
    stock INT NOT NULL,
    version INT DEFAULT 0,  -- 乐观锁版本号
    UNIQUE KEY uk_product (product_id)
);

-- 方案二：乐观锁更新（推荐）
UPDATE stock 
SET stock = stock - 1, version = version + 1 
WHERE product_id = #{productId} 
  AND stock > 0 
  AND version = #{currentVersion};

-- 方案三：SQL 限制（最安全）
UPDATE seckill_stock 
SET available_stock = available_stock - 1 
WHERE seckill_id = #{seckillId} 
  AND available_stock > 0 
  AND orders_count < max_orders;
```

```java

### 2. 高并发处理

**分层拦截策略**：

```
请求量: 1000000 ──→ 限流后 10000 ──→ Redis 拦截后 1000 ──→ 数据库 100
        ↓             ↓                ↓                  ↓
      100%          1%                0.1%               0.01%
```

// ② Redis 分布式限流
public boolean tryAcquire(String key, int limit, int expire) {
    String lua = """
        local count = redis.call('incr', KEYS[1])
        if count == 1 then
            redis.call('expire', KEYS[1], ARGV[1])
        end
        return count <= ARGV[2] and 1 or 0
        """;
    return "1".equals(scriptExecutor.execute(lua, 
        List.of(key), List.of("1", String.valueOf(limit))));
}

// ③ Redis 预扣库存（Lua 原子操作）
public String seckillWithLua(Long userId, Long productId) {
    String lua = """
        -- 1. 检查用户是否已购买
        local hasBought = redis.call('sismember', 'seckill:bought:' .. KEYS[1], ARGV[1])
        if hasBought == 1 then
            return '-2'  -- 已购买
        end
        
        -- 2. 检查库存
        local stock = tonumber(redis.call('get', KEYS[2]))
        if stock <= 0 then
            return '-1'  -- 库存不足
        end
        
        -- 3. 扣减库存
        redis.call('decr', KEYS[2])
        
        -- 4. 标记用户已购买
        redis.call('sadd', 'seckill:bought:' .. KEYS[1], ARGV[1])
        
        -- 5. 发送消息到 MQ
        redis.call('lpush', KEYS[3], ARGV[2])
        
        return '1'
        """;  // 1=成功
}
```

### 3. 库存一致性

**方案：Redis 缓存 + 可靠消息 + 数据库兜底**

```java
// 完整秒杀流程
@Service
public class SeckillService {
    
    @Autowired private RedisTemplate<String, Object> redis;
    @Autowired private RabbitTemplate rabbitTemplate;
    
    public Result seckill(Long userId, Long productId) {
        // Step 1: 限流检查
        if (!rateLimiter.tryAcquire("seckill:" + productId, 1000)) {
            return Result.error("抢购太火爆，请稍后");
        }
        
        // Step 2: 检查用户是否已购买（Redis Set）
        Boolean bought = redis.opsForSet()
            .isMember("seckill:bought:" + productId, userId);
        if (Boolean.TRUE.equals(bought)) {
            return Result.error("您已抢购过该商品");
        }
        
        // Step 3: Redis 原子扣减库存
        String result = redis.execute(seckillScript, 
            List.of("stock:" + productId, "seckill:bought:" + productId),
            userId.toString(), buildOrderMessage(userId, productId));
        
        if (!"1".equals(result)) {
            return Result.error(result.equals("-1") ? "库存不足" : "已购买");
        }
        
        // Step 4: 异步创建订单
        return Result.success("抢购成功");
    }
}

// MQ 消费者：真正扣减数据库库存
@RabbitListener(queues = "seckill.order")
public void handleOrder(String message) {
    OrderMessage msg = JSON.parseObject(message, OrderMessage.class);
    
    // 乐观锁扣减数据库库存
    int retry = 3;
    while (retry-- > 0) {
        int affected = stockMapper.decreaseStockWithLock(
            msg.getProductId(), msg.getQuantity());
        if (affected > 0) {
            orderMapper.createOrder(msg);
            return;
        }
    }
    
    // 扣减失败，补偿 Redis 库存
    redis.opsForValue().increment("stock:" + msg.getProductId());
    log.error("库存扣减失败，productId={}", msg.getProductId());
}
```

## 三、关键技术对比

| 维度 | 纯数据库方案 | Redis + MQ 方案 | 最优方案 |
|-----|-------------|----------------|---------|
| **并发能力** | 低（100-500 QPS） | 中（5000-10000 QPS） | 高（50000+ QPS） |
| **超卖风险** | 存在（需加锁） | 低 | 极低（原子操作） |
| **一致性** | 强一致 | 最终一致 | 最终一致+补偿 |
| **复杂度** | 低 | 中 | 中高 |
| **适用场景** | 小规模秒杀 | 中等规模 | 电商大促 |

> "秒杀系统的核心矛盾是 **库存有限 vs 请求无限**，我的设计思路是 **分层拦截、逐级过滤**：
> 
> 1. **限流层**：网关层用令牌桶/漏桶算法，把无效请求拦截在门外
> 2. **缓存层**：Redis 拦截 99% 的库存查询请求，库存预热到缓存
> 3. **异步层**：MQ 削峰填谷，避免数据库被打爆
> 4. **数据库层**：乐观锁 + 唯一索引兜底，保证数据最终一致
>
> 关于超卖，我会在 SQL 层面加 `stock > 0` 条件，配合版本号乐观锁，双重保障。"

### Q30-1: Redis 库存扣减和数据库库存如何保持最终一致？

```java
// 定时对账 + 补偿机制
@Scheduled(fixedRate = 60000)
public void reconcileStock() {
    List<Product> products = productMapper.selectSeckillProducts();
    for (Product p : products) {
        Integer redisStock = (Integer) redis.opsForValue().get("stock:" + p.getId());
        Integer dbStock = stockMapper.selectStock(p.getId());
        
        if (!redisStock.equals(dbStock)) {
            // 以数据库为准，同步 Redis
            redis.opsForValue().set("stock:" + p.getId(), dbStock);
            log.warn("库存不一致，已同步，productId={}，redis={}，db={}", 
                p.getId(), redisStock, dbStock);
        }
    }
}
```

// 方案：Redis Set + 数据库唯一约束
public boolean checkUserBought(Long userId, Long productId) {
    // 1. Redis 检查（高性能）
    Boolean exists = redis.opsForSet()
        .isMember("seckill:bought:" + productId, userId);
    if (Boolean.TRUE.equals(exists)) {
        return true;
    }
    
    // 2. 数据库兜底（防止 Redis 宕机）
    int count = orderMapper.countByUserAndProduct(userId, productId);
    if (count > 0) {
        redis.opsForSet().add("seckill:bought:" + productId, userId);
        return true;
    }
    return false;
}
