## 先说结论

**Java 对象在堆内存中的布局 = 对象头（Header）+ 实例数据（Instance Data）+ 对齐填充（Padding）**

```
┌─────────────────────────────────────┐
│           对象头 (Object Header)     │  ← 12字节（64位压缩指针）或16字节
│  ┌─────────────────────────────┐    │
│  │    Mark Word (8字节)        │    │  ← 哈希码、GC年龄、锁状态
│  ├─────────────────────────────┤    │
│  │  Class Pointer (4/8字节)    │    │  ← 指向类元数据的指针
│  └─────────────────────────────┘    │
├─────────────────────────────────────┤
│        实例数据 (Instance Data)      │  ← 字段值，按类型占用不同字节
├─────────────────────────────────────┤
│        对齐填充 (Padding)            │  ← 8字节对齐，提高存取效率
└─────────────────────────────────────┘
```

## 🔥 对象内存布局详解

### 1. 对象头（Object Header）

#### Mark Word（标记字段）

**64 位 JVM 下占 8 字节（64 bit）**

```
不同锁状态下的 Mark Word 布局：

无锁状态：
├─  unused:25 ─┬─ hashCode:31 ─┬─ unused:1 ─┬─ age:4 ─┬─ biased_lock:1 ─┬─ lock:2 ─┤
│    未使用      │   哈希码       │   未使用    │ GC年龄  │   偏向锁标志     │ 锁标志   │
└───────────────┴───────────────┴───────────┴─────────┴─────────────────┴─────────┘
   0-24          25-55           56          57-60     61                62-63

偏向锁状态：
├─ thread:54 ─┬─ epoch:2 ─┬─ unused:1 ─┬─ age:4 ─┬─ biased_lock:1 ─┬─ lock:2 ─┤
│   线程ID     │   时间戳   │   未使用    │ GC年龄  │      1           │   01    │
└─────────────┴───────────┴───────────┴─────────┴─────────────────┴─────────┘

轻量级锁状态：
├─                ptr_to_lock_record:62                ─┬─ lock:2 ─┤
│              指向栈中 Lock Record 的指针               │   00    │
└───────────────────────────────────────────────────────┴─────────┘

重量级锁状态：
├─                ptr_to_monitor:62                     ─┬─ lock:2 ─┤
│              指向 ObjectMonitor 的指针                 │   10    │
└───────────────────────────────────────────────────────┴─────────┘

GC标记状态：
├─                       空:62                          ─┬─ lock:2 ─┤
│                      标记位用于GC                     │   11    │
└───────────────────────────────────────────────────────┴─────────┘
```

**锁标志位含义：**

| lock:2 | biased_lock:1 | 状态 |
|--------|---------------|------|
| 01 | 0 | 无锁 |
| 01 | 1 | 偏向锁 |
| 00 | - | 轻量级锁 |
| 10 | - | 重量级锁 |
| 11 | - | GC标记 |

#### Class Pointer（类型指针）

```
作用：指向方法区（元空间）中该对象的类元数据

64位 JVM：
- 开启压缩指针（-XX:+UseCompressedOops）：4字节
- 关闭压缩指针：8字节

32位 JVM：4字节
```

**为什么需要压缩指针？**

```
64位地址空间，指针占8字节，对象头就16字节了
开启压缩指针后，4字节可以表示 2^35 = 32GB 堆内存
超过32GB，压缩指针自动失效
```

#### 数组长度（仅数组对象有）

```
如果是数组对象，对象头还要额外加 4 字节存储数组长度

数组对象布局：
┌──────────┬──────────────┬────────────┬──────────────┬─────────┐
│ Mark Word │ Class Pointer │ Array Length │ Instance Data │ Padding │
│  8字节   │    4字节      │    4字节     │              │         │
└──────────┴──────────────┴────────────┴──────────────┴─────────┘
```

---

### 2. 实例数据（Instance Data）

```
存储对象的字段值，按声明顺序排列

字段类型占用：
┌─────────────┬─────────────┐
│    类型     │   字节数    │
├─────────────┼─────────────┤
│   boolean   │     1       │
│    byte     │     1       │
│    short    │     2       │
│    char     │     2       │
│    int      │     4       │
│   float     │     4       │
│    long     │     8       │
│   double    │     8       │
│   引用类型   │   4/8字节   │
└─────────────┴─────────────┘
```

**字段排列顺序（HotSpot）：**

```java
// 按照字段大小，从大到小排列，节省空间
class Example {
    long l;    // 8字节
    int i;     // 4字节
    short s;   // 2字节
    byte b;    // 1字节
    boolean flag; // 1字节
    Object ref;   // 4字节（压缩指针）
}
// 总大小：8 + 4 + 2 + 1 + 1 + 4 = 20字节
// 对齐后：24字节
```

**父类字段在前：**

```java
class Parent {
    int parentField;  // 排在前面
}

class Child extends Parent {
    long childField;  // 排在后面
}
```

---

### 3. 对齐填充（Padding）

```
目的：让对象大小是 8 字节的整数倍（64位 JVM）

原因：CPU 按字长（64位=8字节）访问内存，对齐后效率最高

示例：
对象实际占 20 字节 → 填充 4 字节 → 变成 24 字节
对象实际占 28 字节 → 填充 4 字节 → 变成 32 字节
```

## 🔥 扩展追问 2：不同对象的大小计算

### 空对象

```java
class Empty {}

// 对象头：12字节（Mark Word 8 + Class Pointer 4）
// 实例数据：0字节
// 对齐填充：4字节
// 总大小：16字节
```

### 带字段的对象

```java
class Person {
    int age;        // 4字节
    boolean flag;   // 1字节
    long id;        // 8字节
    Object ref;     // 4字节（压缩指针）
}

// 对象头：12字节
// 实例数据：4 + 1 + 8 + 4 = 17字节（实际按8+4+4+1=17，但会重排）
// 实际排列：long(8) + int(4) + Object(4) + boolean(1) = 17
// 对齐填充：7字节
// 总大小：12 + 17 + 7 = 36 → 对齐到 40？不对，重新算：

// 正确计算：
// 对象头：12字节
// 字段按大小降序：long(8) + int(4) + Object ref(4) + boolean(1) = 17
// 但 12 + 17 = 29，对齐到 32
// 所以总大小：32字节
```

## 🔥 扩展追问 3：指针压缩（Compressed Oops）

### 什么是 Oop？

```
Oop = Ordinary Object Pointer（普通对象指针）
就是指向对象的引用
```

### 压缩指针原理

```
64位 JVM，不压缩时：
- 对象引用占 8 字节
- 最大可寻址 2^64 = 天文数字，实际用不到

开启压缩指针（-XX:+UseCompressedOops，默认开启）：
- 对象引用占 4 字节
- 通过偏移计算，可寻址 2^35 = 32GB

计算公式：
实际地址 = 堆基址 + (压缩指针 << 3)
```

### 压缩指针的边界

```
堆内存 < 4GB：  压缩指针效果最佳
堆内存 4-32GB： 压缩指针可用
堆内存 > 32GB： 压缩指针失效，自动退回 8 字节
```

**建议：** 堆内存设置要么 < 32GB，要么 > 48GB（留出压缩指针失效的缓冲区）

## 🔥 扩展追问 4：对象头与 synchronized 的关系

```
synchronized 的锁信息就存在 Mark Word 里！

无锁：
├─ hashCode:31 ─┬─ age:4 ─┬─ biased_lock:0 ─┬─ lock:01 ─┤
│   存哈希码     │ GC年龄  │    非偏向锁      │   无锁    │
└───────────────┴─────────┴─────────────────┴───────────┘

偏向锁：
├─ thread:54 ─┬─ epoch:2 ─┬─ age:4 ─┬─ biased_lock:1 ─┬─ lock:01 ─┤
│   线程ID     │   时间戳   │ GC年龄  │     偏向锁       │   无锁    │
└─────────────┴───────────┴─────────┴─────────────────┴───────────┘

轻量级锁：
├─              ptr_to_lock_record:62              ─┬─ lock:00 ─┤
│            指向线程栈中的 Lock Record              │  轻量级锁  │
└───────────────────────────────────────────────────┴───────────┘

重量级锁：
├─                ptr_to_monitor:62                 ─┬─ lock:10 ─┤
│              指向 ObjectMonitor                    │  重量级锁  │
└───────────────────────────────────────────────────┴───────────┘
```

**为什么 hashCode 和偏向锁冲突？**

```
无锁时 Mark Word 存 hashCode
偏向锁时 Mark Word 存线程ID

没有足够空间同时存两个！
所以调用 hashCode() 后，偏向锁会被撤销
```

## 🔥 扩展追问 5：缓存行对齐与伪共享

### 什么是伪共享（False Sharing）？

```
CPU 缓存以缓存行（Cache Line，64字节）为单位加载数据

两个变量在同一个缓存行：
┌────────────────────────────────────────────────────────────────┐
│  变量A (8字节)  │  变量B (8字节)  │  ... 48字节 ...  │  缓存行  │
└────────────────────────────────────────────────────────────────┘
        ↑                   ↑
    Thread-1 修改      Thread-2 读取

Thread-1 修改变量A → 整个缓存行失效 → Thread-2 的变量B也失效
```

### 解决方案：@Contended（JDK 8）

```java
import sun.misc.Contended;

public class PaddedObject {
    @Contended
    private volatile long value1;  // 独占一个缓存行
    
    @Contended
    private volatile long value2;  // 独占一个缓存行
}

// 会在对象布局中插入 128 字节填充（前后各64字节）
// 确保 value1 和 value2 不在同一个缓存行
```

**需要开启：**

```
-XX:-RestrictContended  // 关闭限制，启用 @Contended
```

## 🔥 扩展追问 6：对象头的 GC 信息

```
Mark Word 里存储了 GC 需要的信息：

1. GC 年龄（age:4）
   - 最多 15 岁（4位二进制）
   - 超过阈值进入老年代（-XX:MaxTenuringThreshold=15）

2. GC 标记位
   - 标记-清除算法时用到
   - 锁标志位 = 11 时表示 GC 标记状态
```

## 一句话收尾

> "Java 对象在堆内存中的布局分为三部分：对象头（Mark Word + Class Pointer + 数组长度）、实例数据（字段值）、对齐填充（8字节对齐）。对象头的 Mark Word（64位占8字节）存储了哈希码、GC年龄、锁状态等关键信息，synchronized 的锁升级过程就是修改 Mark Word 的内容。64位 JVM 默认开启压缩指针，Class Pointer 占4字节，对象头总共12字节，加上对齐填充后对象大小是8的倍数。"
