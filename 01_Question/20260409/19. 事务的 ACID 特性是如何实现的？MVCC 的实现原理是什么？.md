## 先说结论

ACID 靠四把锁和一套日志机制撑起来：
- **A 原子性** → Undo Log（回滚）
- **C 一致性** → 前面三个保证后自动达到一致状态
- **I 隔离性** → 锁 + MVCC
- **D 持久性** → Redo Log（崩溃恢复）

MVCC 就是 InnoDB 的"无锁读"黑科技，靠 **ReadView + 隐藏列** 让读写不打架。

## ACID 逐个拆解

### A — 原子性（Atomicity）

**实现机制：Undo Log（回滚日志）**

```
事务开始
   ↓
执行 UPDATE users SET age = 30 WHERE id = 1;  -- 旧值 age=25
   ↓
Undo Log 记录：{id=1, age=25}  ←  rollback 时用这个恢复
   ↓
事务提交 / rollback
```

**核心原理**：每条修改都会生成一条反向的 Undo Log，事务失败或回滚时，顺着 Undo Log 物理反操作回去。

```sql
-- 模拟回滚过程
BEGIN;
UPDATE users SET age = 30 WHERE id = 1;  -- 假设 age 原来是 25
-- 此时 Undo Log 已有 {id=1, age=25}
ROLLBACK;
-- InnoDB 读取 Undo Log，把 age 改回 25
```

---

### C — 一致性（Consistency）

**一致性的本质**：原子性 + 隔离性 + 持久性全部做到位，数据库状态自然是一致的。

> **面试加分话术**：一致性和完整性约束（主键、外键、CHECK 约束、唯一索引）也密切相关，MySQL 会通过触发器、约束检查来保证应用层的一致性。

---

### I — 隔离性（Isolation）

**实现机制：锁机制 + MVCC**

#### 1. 锁机制（四种隔离级别）

```
读未提交（Read Uncommitted）   →  无锁        → 脏读
读已提交（Read Committed）     →  MVCC        → 不可重复读
可重复读（Repeatable Read）    →  MVCC + 间隙锁 → 幻读
串行化（Serializable）          →  锁表        → 无并发
```

#### 2. 锁的类型

```sql
-- 1. 共享锁（S锁）：读读可以并发
SELECT * FROM users LOCK IN SHARE MODE;  -- 加 S 锁

-- 2. 排他锁（X锁）：读写互斥
SELECT * FROM users FOR UPDATE;          -- 加 X 锁
UPDATE users SET age = 30 WHERE id = 1; -- 自动加 X 锁

-- 3. 记录锁（Record Lock）：锁单行
SELECT * FROM users WHERE id = 1 FOR UPDATE;  -- 锁 id=1 这一行

-- 4. 间隙锁（Gap Lock）：锁范围（防止幻读）
SELECT * FROM users WHERE age BETWEEN 20 AND 30 FOR UPDATE;
-- 锁 (20, 30) 这个区间，其他事务无法插入 age=25 的新行

-- 5. 临键锁（Next-Key Lock）：记录锁 + 间隙锁
-- InnoDB RR 级别默认用临键锁，既锁行也锁间隙
```

#### 3. MVCC（见下文专门讲解）

---

### D — 持久性（Durability）

**实现机制：Redo Log + 双写缓冲（Double Write Buffer）**

```
用户写入数据
    ↓
写入 Buffer Pool（内存）         ← 数据还在内存
写入 Redo Log Buffer（内存）    ← 日志先落盘准备
    ↓
事务提交时：
Redo Log Buffer → Redo Log File（磁盘）  ← 持久化成功
    ↓
后台线程异步把 Buffer Pool → 数据文件（磁盘）
```

**为什么先写 Redo Log？**
- 数据文件有 16KB 页，Redo Log 是追加写的顺序 IO，比随机 IO 快 100 倍
- 崩溃后从 Redo Log 恢复，而不是重做所有操作

**崩溃恢复流程**：
```
数据库崩溃
    ↓
重启 → 读取 Redo Log
    ↓
找到所有未刷盘的已提交事务 → 重做（Redo）
找到所有未提交事务         → 回滚（Undo）
    ↓
恢复完成，数据一致
```

**双写缓冲（防止页撕裂）**：
```
写数据页（16KB）
    ↓
先完整写 Double Write Buffer（磁盘，顺序写）
    ↓
再写数据文件（16KB，分散写）
-- 如果写数据文件时断电，从 Double Write Buffer 恢复完整页
```

## MVCC 核心原理

### 1. 每行数据隐藏的三列

```sql
-- InnoDB 内部每行数据有三个隐藏列
表结构实际是：
| id | name | age | DB_ROW_ID | TRX_ID | ROLL_PTR | DELETE_FLAG |
|----|------|-----|-----------|--------|-----------|------------|
| 1  | Tom  | 25  | 自动生成  | 事务ID | 回滚指针   | 是否删除    |

-- DB_ROW_ID：行ID（6字节），自增
-- TRX_ID：最近修改这行的事务ID
-- ROLL_PTR：指向 Undo Log 中这条记录的旧版本
```

### 2. Undo Log 链（版本链）

```
事务A 修改 id=1 的 name = 'Tom'
    ↓
Undo Log 记录旧版本：name='Jack', ROLL_PTR=null
    ↓
当前版本：name='Tom', ROLL_PTR=→ Undo Log
    ↓
事务B 修改 id=1 的 name = 'Jerry'
    ↓
Undo Log 记录旧版本：name='Tom', ROLL_PTR=→ 上一条Undo
    ↓
当前版本：name='Jerry', ROLL_PTR=→ 第二条 Undo Log
```

> **版本链**：每行数据通过 `ROLL_PTR` 串联起来，形成一个链表，新事务看不到旧事务的修改。

### 3. ReadView（读视图）

事务开启时生成一个 ReadView，记录"当前活跃的事务 ID 列表"：

```sql
-- 事务5 开启时生成 ReadView
ReadView {
    m_ids: [3, 4, 7]    -- 当前活跃的未提交事务
    min_trx_id: 3       -- 活跃列表中最小的事务ID
    max_trx_id: 8       -- 创建 ReadView 时最大事务ID +1
    creator_trx_id: 5   -- 当前事务自己的ID
}
```

**读取规则**：
```
遍历版本链，每条记录看 TRX_ID：
    ↓
TRX_ID < min_trx_id(3)？  → 这个事务早就提交了，可以读 ✅
TRX_ID in m_ids [3,4,7]？ → 事务未提交，看不见，看上一条 Undo ✅
TRX_ID >= max_trx_id(8)？ → 是我之后开启的事务，看不见，看上一条 ✅
TRX_ID == creator(5)？    → 是我自己改的，可以读 ✅
```

### 4. 两种隔离级别下的 ReadView 生成时机

| 隔离级别 | ReadView 生成时机 | 效果 |
|---------|-----------------|------|
| **Read Committed** | 每次 SELECT 都生成新 ReadView | 每次读到最新已提交版本 |
| **Repeatable Read** | 第一次 SELECT 时生成，之后复用 | 整个事务读到的数据一致 |

```
【RC 级别示例】
事务A（id=100）：UPDATE users SET age=30 WHERE id=1;  -- 未提交
事务B（id=101）：SELECT * FROM users WHERE id=1;      -- 读到 age=25（RC）
   ↓ 事务A提交
事务C（id=102）：SELECT * FROM users WHERE id=1;      -- 读到 age=30（RC）
-- 同一事务内两次 SELECT 结果不同 ❌

【RR 级别示例】
事务B（id=101）：SELECT * FROM users WHERE id=1;      -- 第一次读 age=25，生成 ReadView
   ↓ 事务A提交
事务C（id=102）：SELECT * FROM users WHERE id=1;      -- 看不到这个提交（不在ReadView）
-- 同一事务内多次 SELECT 结果相同 ✅
```

## 🔥 扩展追问

### 追问 1：MVCC 能否完全解决幻读？

**不能完全解决，但能解决大部分**。

```sql
-- 事务A（RR级别）
SELECT * FROM orders WHERE status = 'pending';  -- 读到 5 条
-- 幻读！事务B此时插入了一条新数据
SELECT * FROM orders WHERE status = 'pending';  -- 读到 6 条 ❌

-- InnoDB 的 Next-Key Lock 补刀
SELECT * FROM orders WHERE status = 'pending' FOR UPDATE;
-- 对 status 索引加 Next-Key Lock，其他事务无法插入 status='pending' 的新行
```

**总结**：
- **快照读**（普通 SELECT）：靠 MVCC，基本解决幻读
- **当前读**（SELECT FOR UPDATE / INSERT / UPDATE / DELETE）：靠 Next-Key Lock，彻底解决幻读

### 追问 2：为什么 Read Committed 用 MVCC 无法解决不可重复读？

因为 RC 每次 SELECT 都重新生成 ReadView：

```
事务A（id=100）：BEGIN;
事务B（id=101）：BEGIN;
事务A：SELECT * FROM t WHERE id=1;  -- 读到 age=25
事务B：UPDATE t SET age=30 WHERE id=1; COMMIT;
事务A：SELECT * FROM t WHERE id=1;  -- RC 重新生成 ReadView，看到 age=30 ❌
-- 同一事务内两次读结果不同！
```

RR 第一次 SELECT 就锁定 ReadView，之后不复生，所以可重复。

### 追问 3：MVCC + 间隙锁在 RR 级别下的完整防幻读流程

```
t 表有索引 age：值分布 [10, 20, 30, 40, 50]
    ↓
事务A执行：
SELECT * FROM t WHERE age BETWEEN 20 AND 40 FOR UPDATE;
    ↓
Next-Key Lock 锁住：
- 记录锁：age=20, age=30, age=40 这三行
- 间隙锁： (10,20), (20,30), (30,40), (40,50) 这四个区间
    ↓
其他事务：
- 无法修改 age=20/30/40 的行（记录锁）✅
- 无法插入 age 在 (10,50) 区间的新行（间隙锁）✅
-- 彻底防止幻读！
```

### 追问 4：Redo Log vs Binlog 的区别？

| | Redo Log | Binlog |
|---|---|---|
| **层次** | InnoDB 引擎层 | MySQL Server 层 |
| **内容** | 物理日志（页修改） | 逻辑日志（SQL 语句） |
| **写入时机** | 事务进行中随时写 | 事务提交时统一写 |
| **用途** | 崩溃恢复 | 数据复制、主从同步 |
| **格式** | 固定格式，binlog 有 STATEMENT/ROW/MIXED | 追加写 |

```
写入流程（MySQL 5.7+）：
1. 修改数据 → 写入 Buffer Pool
2. 写入 Redo Log（prepare 状态）
3. 写入 Binlog
4. 提交事务 → Redo Log 改为 commit 状态
   ↓
崩溃恢复：
检查 Redo Log + Binlog 的一致性标记
有 commit → 重做
无 commit 但 Binlog 有记录 → 重做
无 Binlog 记录 → 回滚
```

> **面试加分**：MySQL 5.6 之前 Binlog 和 Redo Log 可能不一致，MySQL 5.6+ 通过 XA 事务保证两阶段提交解决这个问题。

### 追问 5：MySQL 8.0 的 MVCC 有什么改进？

```sql
-- MySQL 8.0 新增：事务 ReadView 可以主动控制
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- MySQL 8.0 新增：事务可以指定 snapshot 读取某个时间点的数据
START TRANSACTION WITH CONSISTENT SNAPSHOT;
-- 传统 RC/RR 都是基于"当前时刻"生成 ReadView

-- MySQL 8.0 新增：CLL（Change Data Capture）支持
-- 可以跟踪事务的提交顺序，变化了哪些行
```

### 追问 6：死锁是如何检测和处理的？

```sql
-- 死锁示例
事务A：UPDATE t1 → 锁住 t1，需要 t2
事务B：UPDATE t2 → 锁住 t2，需要 t1
    ↓
死锁检测（InnoDB 每隔 1ms 检测一次依赖图）：
    发现循环依赖！
    ↓
回滚 undo log 较少的事务（代价最小原则）✅
    ↓
ERROR 1213: Deadlock found when trying to get lock
```

**避免死锁的编码建议**：
1. 按固定顺序访问多个表/行
2. 减少锁持有时间（大事务拆小）
3. 加锁范围尽量精确（不要 `SELECT * FOR UPDATE`，明确列）
4. 合理设计索引，减少锁的粒度

### 追问 7：MVCC 的 undo log 什么时候被清理？

```
MVCC 读取时，沿着版本链往前找 Undo Log
    ↓
当某个 Undo Log 版本对所有活跃事务都不可见时
（即最早活跃事务的 TRX_ID > 该 Undo Log 的 TRX_ID）
    ↓
purge 线程统一清理（后台定期执行）
    ↓
删除无用的 Undo Log 链尾
```

## 一句话收尾

> **ACID 的本质是：原子性靠 Undo Log 回滚，隔离性靠锁 + MVCC 快照读，持久性靠 Redo Log 崩溃恢复，一致性是这三者正确执行后的自然结果；MVCC 则通过 ReadView + 版本链，让读不加锁也能看到一致的数据快照。**
