## 先说结论

```
浅拷贝：复制引用，不复制对象本身
深拷贝：递归复制所有对象，彻底独立
```

---

## 🔥 核心概念

### 浅拷贝（Shallow Copy）

```java
class Person {
    String name;
    int age;
    Address address;  // 引用类型
}

Person p1 = new Person("张三", 18, new Address("北京"));

// 浅拷贝
Person p2 = p1.clone();

// p1 和 p2 的 name、age 是独立的
p2.name = "李四";
p1.name  // 仍是 "张三" ✓

// 但 address 是共享的！
p2.address.city = "上海";
p1.address.city  // 变成 "上海" ✗  ← 问题所在
```

**图示：**

```
浅拷贝后：
┌─────────────────┐     ┌─────────────────┐
│  p1             │     │  p2             │
├─────────────────┤     ├─────────────────┤
│ name: 张三      │     │ name: 李四      │
│ age: 18         │     │ age: 18         │
│ address ────────┼─────┼──→ address      │  ← 指向同一个对象！
└─────────────────┘     └─────────────────┘
                              ↓
                        ┌─────────────┐
                        │ Address     │
                        ├─────────────┤
                        │ city: 上海  │
                        └─────────────┘
```

---

### 深拷贝（Deep Copy）

```java
// 深拷贝后：
┌─────────────────┐     ┌─────────────────┐
│  p1             │     │  p2             │
├─────────────────┤     ├─────────────────┤
│ name: 张三      │     │ name: 李四      │
│ age: 18         │     │ age: 18         │
│ address ────────┼     │ address ────────┼──→ 新的 Address 对象
└─────────────────┘     └─────────────────┘
                              ↓
                        ┌─────────────┐
                        │ Address     │
                        ├─────────────┤
                        │ city: 北京  │  ← 独立副本！
                        └─────────────┘
```

**修改 p2 的 address 不影响 p1，完全独立。**

## 🔥 扩展追问 1：Object 的 clone() 是深拷贝还是浅拷贝？

**默认是浅拷贝。**

```java
// Object.clone() 默认行为
protected native Object clone() throws CloneNotSupportedException {
    // 逐个字段复制
    // 基本类型：值拷贝
    // 引用类型：只复制引用，不复制对象
}
```

**要让一个类支持 clone()，必须：**

```java
class Person implements Cloneable {
    String name;
    Address address;
    
    @Override
    protected Person clone() {
        try {
            return (Person) super.clone();
        } catch (CloneNotSupportedException e) {
            throw new RuntimeException(e);
        }
    }
}
// 此时的 clone() 仍是浅拷贝！
```

## 🔥 扩展追问 2：如何实现深拷贝？

### 方法一：重写 clone()（手动递归）


```java
class Person implements Cloneable {
    String name;
    Address address;
    
    @Override
    protected Person clone() {
        try {
            Person p = (Person) super.clone();
            p.address = this.address.clone();  // 手动拷贝引用对象
            return p;
        } catch (CloneNotSupportedException e) {
            throw new RuntimeException(e);
        }
    }
}

class Address implements Cloneable {
    String city;
    
    @Override
    protected Address clone() {
        try {
            return (Address) super.clone();
        } catch (CloneNotSupportedException e) {
            throw new RuntimeException(e);
        }
    }
}
```

**缺点：**

- 如果有嵌套很深的引用，要层层写 clone
- 对象间有循环引用会很麻烦
- 代码维护成本高

---

### 方法二：序列化（JSON / Hessian / Java Serialization）

**JSON 序列化（最常用）：**

```java
import com.fasterxml.jackson.databind.ObjectMapper;

class Person {
    String name;
    Address address;
    // ... getters/setters
}

class Address {
    String city;
    // ... getters/setters
}

// 深拷贝
ObjectMapper mapper = new ObjectMapper();
Person copy = mapper.readValue(
    mapper.writeValueAsBytes(person), 
    Person.class
);
```

**优点：**

- 简单，一行搞定
- 嵌套多深都能处理
- 循环引用也没问题

**缺点：**

- 对象必须能序列化（implements Serializable）
- 有性能开销（序列化/反序列化）

---

### 方法三：Java 对象序列化

```java
import java.io.*;

public class DeepCloneUtil {
    public static <T extends Serializable> T deepClone(T obj) {
        try {
            ByteArrayOutputStream bos = new ByteArrayOutputStream();
            ObjectOutputStream oos = new ObjectOutputStream(bos);
            oos.writeObject(obj);
            
            ByteArrayInputStream bis = new ByteArrayInputStream(bos.toByteArray());
            ObjectInputStream ois = new ObjectInputStream(bis);
            return (T) ois.readObject();
        } catch (Exception e) {
            throw new RuntimeException("深拷贝失败", e);
        }
    }
}

// 使用
Person copy = DeepCloneUtil.deepClone(person);
```

**缺点：**

- 所有对象必须 implements Serializable
- 性能最差
- 不推荐生产环境用

---

### 方法四：构造器拷贝（拷贝构造器）

```java
class Person {
    String name;
    Address address;
    
    // 拷贝构造器
    public Person(Person other) {
        this.name = other.name;
        this.address = new Address(other.address);  // 拷贝嵌套对象
    }
}

class Address {
    String city;
    
    public Address(Address other) {
        this.city = other.city;
    }
}

// 使用
Person copy = new Person(person);
```

**缺点：**

- 嵌套深的话，每个类都要写拷贝构造器
- 难以扩展

## 🔥 扩展追问 3：实际开发中用什么方案？

**推荐：JSON 序列化**

```java
// Jackson 或 Gson
ObjectMapper mapper = new ObjectMapper();

// Apache Commons Lang 也有序列化工具
SerializationUtils.clone(obj);  // 要求对象 implements Serializable
```

**为什么不用 clone()？**

- clone() 是 Object 的 protected 方法，设计有问题
- 需要层层递归实现深拷贝，代码繁琐
- 违反"优先使用组合而非继承"原则

**实战建议：**

> "实际开发中我一般用 JSON 序列化做深拷贝，简单可靠，第三方库支持好。除非性能特别敏感的场景，再考虑手动 clone 或其他方案。"

## 🔥 扩展追问 4：哪些场景需要深拷贝？

```
1. 缓存对象：用户信息缓存后，修改不应影响缓存
2. 命令模式：命令对象被修改不应影响历史记录
3. 并发场景：多线程可能同时修改对象
4. 业务快照：保存某个时刻的对象状态
5. 设计模式：Prototype 模式用深拷贝创建对象
```

**实际案例：电商订单：**

```java
// 用户下单时，商品信息可能后续会变
OrderItem item = new OrderItem();
item.setProductSnapshot(product.deepClone());  // 深拷贝商品信息
// 即使商品原价改了，订单里记录的还是下单时的价格
```

## 🔥 扩展追问 5：集合的深拷贝？

**集合的 clone() 也是浅拷贝：**

```java
List<Person> original = new ArrayList<>();
original.add(new Person("张三"));

List<Person> shallow = (List<Person>) original.clone();

// shallow 和 original 是不同的 List
// 但里面的 Person 对象是同一个！

shallow.get(0).name = "李四";
original.get(0).name  // 变成 "李四" ✗
```

**集合的深拷贝：**

```java
// 方法一：手动遍历
List<Person> deep1 = new ArrayList<>();
for (Person p : original) {
    deep1.add(p.clone());
}

// 方法二：序列化
ObjectMapper mapper = new ObjectMapper();
List<Person> deep2 = mapper.readValue(
    mapper.writeValueAsString(original),
    mapper.getTypeFactory().constructCollectionType(List.class, Person.class)
);

// 方法三：Java Stream
List<Person> deep3 = original.stream()
    .map(Person::clone)
    .collect(Collectors.toList());
```

## 🔥 扩展追问 6：Cloneable 接口是干什么的？

**Cloneable 是个"标记接口"（Marker Interface）：**

```java
public interface Cloneable {
    // 空接口，没有任何方法
    // 作用：告诉 JVM 这个类允许被 clone()
}
```

**为什么叫标记接口？**

- 里面没有任何方法
- 实现了它，Object.clone() 才会成功
- 没实现会抛 CloneNotSupportedException

**常见标记接口：**

```
Serializable   → 标记可序列化
Cloneable      → 标记可克隆
RandomAccess   → 标记可随机访问（如 ArrayList）
```

## 🔥 扩展追问 7：clone() 相比拷贝构造器有什么问题？

|方面|clone()|拷贝构造器|
|---|---|---|
|语法|protected，需类型转换|public，直接调用|
|类型安全|不保证子类正确克隆|编译期检查|
|继承|子类需重写|父类构造器自动调用|
|final 字段|无法赋值|构造器内可赋值|
|代码可读性|隐晦|清晰|


```java
// clone() 的问题
class Parent implements Cloneable {
    @Override
    protected Object clone() throws CloneNotSupportedException {
        return super.clone();
    }
}

class Child extends Parent {
    @Override
    protected Object clone() throws CloneNotSupportedException {
        return super.clone();  // 返回的是 Parent 类型！
        // 需要手动强转
    }
}
```

**结论：**

> "Effective Java 里 Joshua Bloch 就说过，Cloneable 是设计失败的接口，拷贝构造器是更好的选择。实际开发中尽量避免用 clone()。"

## 🔥 扩展追问 8：拷贝时循环引用怎么处理？


```java
class A {
    B b;
}

class B {
    A a;
}

// A 引用 B，B 引用 A，形成环

A objA = new A();
B objB = new B();
objA.b = objB;
objB.a = objA;

// 序列化天然支持循环引用
ObjectMapper mapper = new ObjectMapper();
// 会自动处理，输出 JSON 后再反序列化即可
```

## 一句话收尾

> "浅拷贝只复制引用，嵌套对象还是共享的；深拷贝递归复制所有对象。实际开发推荐用 JSON 序列化做深拷贝，简单可靠。除非要追求极致性能，再考虑手动 clone 或拷贝构造器。Cloneable 设计有缺陷，不建议过度使用。"
