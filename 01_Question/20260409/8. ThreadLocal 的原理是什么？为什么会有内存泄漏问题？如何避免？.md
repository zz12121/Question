## 先说结论

**ThreadLocal = 线程本地变量，让每个线程拥有自己独立的变量副本，线程之间互不干扰。**

```
核心思想：
不是解决多线程共享数据的问题，而是从根本上避免共享
每个线程一份数据，自然就没有竞争
```

## 🔥 ThreadLocal 的实现原理

### 1. 数据结构

```java
// Thread 类里的属性
public class Thread implements Runnable {
    // 每个线程都有自己的 ThreadLocalMap
    ThreadLocal.ThreadLocalMap threadLocals = null;
}

// ThreadLocalMap 是 ThreadLocal 的静态内部类
static class ThreadLocalMap {
    // 底层是 Entry 数组（哈希表）
    private Entry[] table;
    
    // Entry 是弱引用
    static class Entry extends WeakReference<ThreadLocal<?>> {
        Object value;  // 存的是 ThreadLocal 对应的值
        
        Entry(ThreadLocal<?> k, Object v) {
            super(k);   // 父类 WeakReference，key 是弱引用
            value = v;  // value 是强引用
        }
    }
}
```

**结构图：**

```
Thread-1                    Thread-2
   │                           │
   ▼                           ▼
ThreadLocalMap              ThreadLocalMap
   │                           │
   ├── Entry[0]: tl1 → valueA  ├── Entry[0]: tl1 → valueC
   ├── Entry[1]: tl2 → valueB  ├── Entry[1]: tl2 → valueD
   └── ...                     └── ...
   
tl1 和 tl2 是 ThreadLocal 对象（多个线程共享）
但 valueA/B/C/D 是每个线程独有的
```

---

### 2. 核心方法源码

**set() 方法：**

```java
public void set(T value) {
    // 获取当前线程
    Thread t = Thread.currentThread();
    
    // 获取当前线程的 ThreadLocalMap
    ThreadLocalMap map = getMap(t);
    
    if (map != null) {
        // 直接设置值
        map.set(this, value);
    } else {
        // 第一次使用，创建 Map
        createMap(t, value);
    }
}

ThreadLocalMap getMap(Thread t) {
    return t.threadLocals;
}
```

**get() 方法：**

```java
public T get() {
    Thread t = Thread.currentThread();
    ThreadLocalMap map = getMap(t);
    
    if (map != null) {
        // 以当前 ThreadLocal 为 key 查找
        ThreadLocalMap.Entry e = map.getEntry(this);
        if (e != null) {
            @SuppressWarnings("unchecked")
            T result = (T)e.value;
            return result;
        }
    }
    // 没找到，返回初始值
    return setInitialValue();
}
```

**ThreadLocalMap.set()：**

```java
private void set(ThreadLocal<?> key, Object value) {
    Entry[] tab = table;
    int len = tab.length;
    
    // 计算哈希位置（ThreadLocal 的 threadLocalHashCode）
    int i = key.threadLocalHashCode & (len - 1);
    
    // 线性探测解决哈希冲突
    for (Entry e = tab[i]; e != null; e = tab[i = nextIndex(i, len)]) {
        ThreadLocal<?> k = e.get();
        
        if (k == key) {  // 找到相同的 key，更新 value
            e.value = value;
            return;
        }
        
        if (k == null) {  // key 为 null，说明 ThreadLocal 被回收了
            replaceStaleEntry(key, value, i);  // 清理脏 Entry
            return;
        }
    }
    
    // 找到空槽，插入新 Entry
    tab[i] = new Entry(key, value);
}
```

### 3. 哈希冲突处理

```java
// ThreadLocal 的哈希值
private final int threadLocalHashCode = nextHashCode();

// 每次创建 ThreadLocal，hashCode 增加 0x61c88647
// 这个数是黄金分割数的倒数，能让哈希分布更均匀
private static final int HASH_INCREMENT = 0x61c88647;

// 解决冲突：线性探测
private static int nextIndex(int i, int len) {
    return ((i + 1 < len) ? i + 1 : 0);
}
```

## 🔥 扩展追问 1：为什么会有内存泄漏？

### 根本原因：Key 是弱引用，Value 是强引用

```java
static class Entry extends WeakReference<ThreadLocal<?>> {
    Object value;  // 强引用！
    
    Entry(ThreadLocal<?> k, Object v) {
        super(k);   // key 是弱引用
        value = v;
    }
}
```

**泄漏场景：**

```
1. 创建 ThreadLocal
   ThreadLocal tl = new ThreadLocal();
   
2. 设置值
   tl.set(bigObject);  // bigObject 是 100MB 的大对象
   
3. tl 被置为 null（方法结束或重新赋值）
   tl = null;
   
4. 但是！线程还在运行（比如线程池里的线程）
   
5. Entry 的情况：
   - key（ThreadLocal 引用）是弱引用 → GC 时会被回收 → key 变成 null
   - value（bigObject）是强引用 → 不会被回收！
   
6. 结果：
   Entry = {key=null, value=bigObject}
   这个 Entry 永远访问不到，但 bigObject 一直占着内存！
```

**图示：**

```
Thread（线程池线程，长期存活）
   │
   ▼
ThreadLocalMap
   │
   ├── Entry[0]: key=null, value=100MB大对象  ← 泄漏！访问不到，释放不了
   ├── Entry[1]: key=tl2, value=xxx
   └── ...
   
tl 已经被置为 null，但线程还在，Map 还在，value 就泄漏了
```

---

### 泄漏发生的条件

```
1. 使用线程池（线程长期存活）
2. ThreadLocal 被置为 null（或方法结束，局部变量消失）
3. 没有手动 remove()
4. 后续没有再使用这个 ThreadLocal（没有触发清理）
```

## 🔥 扩展追问 2：如何避免内存泄漏？

### 方案 1：用完及时 remove()（最推荐）

```java
public void process() {
    threadLocal.set(userInfo);
    try {
        // 执行业务逻辑
        doSomething();
    } finally {
        threadLocal.remove();  // 必须！
    }
}
```

**remove() 源码：**

```java
public void remove() {
    ThreadLocalMap m = getMap(Thread.currentThread());
    if (m != null) {
        m.remove(this);  // 从 Map 中删除这个 Entry
    }
}
```

### 方案 2：使用 try-with-resources 模式（封装）

```java
public class AutoCloseableThreadLocal<T> extends ThreadLocal<T> 
    implements AutoCloseable {
    
    @Override
    public void close() {
        this.remove();
    }
}

// 使用
try (AutoCloseableThreadLocal<User> tl = new AutoCloseableThreadLocal<>()) {
    tl.set(user);
    // 执行业务
}  // 自动 remove()
```

### 方案 3：Spring 的解决方案

```java
// Spring 的 RequestContextHolder 使用 transcribe 模式
public static void resetRequestAttributes() {
    requestAttributesHolder.remove();
    inheritableRequestAttributesHolder.remove();
}

// 在拦截器的 finally 里调用
public class RequestContextFilter {
    public void doFilter(request, response, chain) {
        try {
            chain.doFilter(request, response);
        } finally {
            resetRequestAttributes();  // 清理
        }
    }
}
```

## 🔥 扩展追问 3：ThreadLocalMap 的清理机制

### 1. 探测式清理（expungeStaleEntry）

```java
// 在 set()、get()、remove() 时触发
// 从 staleSlot 开始，清理连续的 key=null 的 Entry
private int expungeStaleEntry(int staleSlot) {
    Entry[] tab = table;
    int len = tab.length;
    
    // 清理当前位置
    tab[staleSlot].value = null;
    tab[staleSlot] = null;
    size--;
    
    // 继续向后探测，直到遇到 null
    Entry e;
    int i;
    for (i = nextIndex(staleSlot, len);
         (e = tab[i]) != null;
         i = nextIndex(i, len)) {
        ThreadLocal<?> k = e.get();
        if (k == null) {  // 又遇到一个脏 Entry
            e.value = null;
            tab[i] = null;
            size--;
        } else {
            // 重新哈希，整理位置
            int h = k.threadLocalHashCode & (len - 1);
            if (h != i) {
                tab[i] = null;
                while (tab[h] != null) {
                    h = nextIndex(h, len);
                }
                tab[h] = e;
            }
        }
    }
    return i;
}
```

### 2. 启发式清理（cleanSomeSlots）

```java
// 每次 set() 后，扫描一定数量的槽位
private boolean cleanSomeSlots(int i, int n) {
    boolean removed = false;
    Entry[] tab = table;
    int len = tab.length;
    
    do {
        i = nextIndex(i, len);
        Entry e = tab[i];
        if (e != null && e.get() == null) {
            n = len;  // 发现脏 Entry，扩大扫描范围
            removed = true;
            i = expungeStaleEntry(i);
        }
    } while ((n >>>= 1) != 0);  // log2(n) 次扫描
    
    return removed;
}
```

### 3. 扩容时全量清理

```java
// rehash() 和 resize() 时会遍历整个 table
// 清理所有 key=null 的 Entry
```

**但！这些清理都是"被动"的：**

```
只有调用 set/get/remove 时才会触发清理
如果 ThreadLocal 用完之后再也不访问，就不会触发清理
value 就会一直泄漏！
```

## 🔥 扩展追问 4：父子线程如何共享 ThreadLocal？

### InheritableThreadLocal

```java
// 子线程继承父线程的 ThreadLocal 值
public class InheritableThreadLocal<T> extends ThreadLocal<T> {
    
    // 子线程创建时，复制父线程的值
    protected T childValue(T parentValue) {
        return parentValue;
    }
}

// 使用
private static final InheritableThreadLocal<String> inheritTl = 
    new InheritableThreadLocal<>();

public static void main(String[] args) {
    inheritTl.set("父线程的值");
    
    new Thread(() -> {
        System.out.println(inheritTl.get());  // 输出：父线程的值
    }).start();
}
```

**原理：**

```java
// Thread 类里的属性
ThreadLocal.ThreadLocalMap inheritableThreadLocals = null;

// 创建子线程时，复制父线程的 inheritableThreadLocals
private Thread(ThreadGroup g, Runnable target, String name,
               long stackSize, AccessControlContext acc,
               boolean inheritThreadLocals) {
    // ...
    if (inheritThreadLocals && parent.inheritableThreadLocals != null) {
        this.inheritableThreadLocals =
            ThreadLocal.createInheritedMap(parent.inheritableThreadLocals);
    }
}
```

### 线程池的问题

```java
// 线程池里的线程是复用的！
// InheritableThreadLocal 只在创建线程时复制
// 线程池不会每次都创建新线程，所以值不会更新

ExecutorService pool = Executors.newFixedThreadPool(2);
InheritableThreadLocal<String> tl = new InheritableThreadLocal<>();

tl.set("main-1");
pool.submit(() -> System.out.println(tl.get()));  // main-1

tl.set("main-2");
pool.submit(() -> System.out.println(tl.get()));  // 可能还是 main-1！
```

### 解决方案：TransmittableThreadLocal（阿里）

```java
// 使用 TTL，线程池也能正确传递
TransmittableThreadLocal<String> ttl = new TransmittableThreadLocal<>();

ExecutorService ttlPool = TtlExecutors.getTtlExecutorService(
    Executors.newFixedThreadPool(2)
);

ttl.set("main-1");
ttlPool.submit(() -> System.out.println(ttl.get()));  // main-1

ttl.set("main-2");
ttlPool.submit(() -> System.out.println(ttl.get()));  // main-2
```

## 🔥 扩展追问 6：FastThreadLocal（Netty 优化）

**ThreadLocal 的问题：**

```
1. 哈希冲突需要线性探测，O(n) 复杂度
2. 使用弱引用，需要频繁清理
3. 数组扩容需要 rehash
```

**FastThreadLocal 的优化：**

```java
// 每个 FastThreadLocal 有一个全局唯一的 index
private static final AtomicInteger nextIndex = new AtomicInteger();

public FastThreadLocal() {
    index = nextIndex.getAndIncrement();  // 分配唯一索引
}

// 直接用数组下标访问，O(1)
public V get() {
    InternalThreadLocalMap map = InternalThreadLocalMap.get();
    return map.indexedVariable(index);  // 直接数组访问
}
```

**对比：**

|特性|ThreadLocal|FastThreadLocal|
|---|---|---|
|访问复杂度|O(1) ~ O(n)|O(1)|
|哈希冲突|有|无（直接用 index）|
|弱引用|是|否（需要手动 remove）|
|清理机制|被动清理|无|
|适用场景|通用|高频访问（Netty）|

## 一句话收尾

> "ThreadLocal 通过每个线程维护独立的 ThreadLocalMap 来实现线程隔离，Key 是弱引用的 ThreadLocal，Value 是强引用的实际数据。内存泄漏的原因是：ThreadLocal 被回收后，Key 变成 null，但 Value 还被线程强引用着，如果线程长期存活（如线程池），Value 就泄漏了。避免方法是用完及时 remove()，或者使用 try-finally 确保清理。"
