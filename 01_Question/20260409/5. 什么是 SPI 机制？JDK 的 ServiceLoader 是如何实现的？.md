## 先说结论

**SPI = Service Provider Interface，是一种"反向依赖注入"机制。**

```
传统 DI：框架调用我写的代码
SPI：我写的代码被框架发现并调用

核心思想：
应用的入口通过约定好的接口去"发现"和"加载"实现类，
而不是在代码里写死。
```

---

## 🔥 核心概念

### SPI 解决什么问题？

**场景：JDK 日志门面——SLF4J**

**java**

复制

```java
// 你写代码用 SLF4J 接口
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

Logger logger = LoggerFactory.getLogger(MyClass.class);
logger.info("Hello");
```

**SLF4J 只是接口，谁来实现？**

```
slf4j-api.jar（接口）
      ↓
      发现机制
      ↓
slf4j-log4j12.jar  或  slf4j-simpler.jar  或  logback-classic.jar
（具体实现）
```

**如果没有 SPI：**

- slf4j-api 需要知道所有实现并硬编码依赖
- 每加一个实现就要改 slf4j-api
- 违背开闭原则

**有了 SPI：**

- slf4j-api 只定义接口 + 加载规则
- 各实现按约定放配置文件
- 运行时动态发现，零侵入

## 🔥 扩展追问 1：ServiceLoader 源码解析

**java**

复制

```java
// ServiceLoader 核心源码（简化版）
public final class ServiceLoader<S> implements Iterable<S> {
    
    // 缓存已加载的实现类实例
    private LinkedHashMap<String, S> providers = new LinkedHashMap<>();
    
    // 接口 Class
    private final Class<S> service;
    
    // 类加载器
    private final ClassLoader loader;
    
    // 迭代器，懒加载
    public Iterator<S> iterator() {
        return new Iterator<S>() {
            Iterator<Map.Entry<String, S>> iterator = providers.iterator();
            
            @Override
            public boolean hasNext() {
                if (iterator.hasNext()) {
                    return true;
                }
                // 还没加载过，去扫描配置文件
                return reload();
            }
            
            @Override
            public S next() {
                if (!hasNext()) {
                    throw new NoSuchElementException();
                }
                return iterator.next().getValue();
            }
        };
    }
    
    // 加载配置文件
    private boolean reload() {
        providers.clear();
        // 配置文件路径：META-INF/services/ + 接口全限定名
        String fullName = PREFIX + service.getName();
        
        Enumeration<URL> configs = loader.getResources(fullName);
        
        while (configs.hasMoreElements()) {
            URL url = configs.nextElement();
            parseConfigFile(url);  // 逐行读取类名
        }
        
        return !providers.isEmpty();
    }
    
    // 创建实例
    private void parseConfigFile(URL url) {
        BufferedReader reader = new BufferedReader(
            new InputStreamReader(url.openStream(), StandardCharsets.UTF_8)
        );
        
        String line;
        while ((line = reader.readLine()) != null) {
            // 去掉注释和空行
            int ci = line.indexOf('#');
            if (ci >= 0) line = line.substring(0, ci);
            line = line.trim();
            
            if (!line.isEmpty()) {
                // 用类加载器加载这个类
                Class<?> clazz = Class.forName(line, true, loader);
                // 检查是否是接口的实现类
                if (service.isAssignableFrom(clazz)) {
                    // 实例化并缓存
                    S provider = service.cast(clazz.newInstance());
                    providers.put(line, provider);
                }
            }
        }
    }
}
```

## 🔥 扩展追问 2：加载流程图

```
ServiceLoader.load(Payment.class)
        ↓
    创建 ServiceLoader 实例
        ↓
    首次调用 iterator()
        ↓
    调用 reload()
        ↓
    获取所有 META-INF/services/com.xxx.Payment 文件
        ↓
    逐行解析文件，读到类名：com.example.alipay.AlipayImpl
        ↓
    Class.forName("com.example.alipay.AlipayImpl")
        ↓
    检查 isAssignableFrom() 确认是接口实现
        ↓
    newInstance() 创建实例
        ↓
    放入 LinkedHashMap 缓存
        ↓
    返回迭代器，遍历时逐个返回
```

## 🔥 扩展追问 3：为什么用 LinkedHashMap 而不是 HashMap？

**java**

复制

```java
private LinkedHashMap<String, S> providers = new LinkedHashMap<>();
```

**LinkedHashMap 特点：**

- 保证迭代顺序 = 插入顺序
- 文件里先写的类先加载

**有什么用？**

- 配置文件里靠前的类优先级更高
- 可以控制加载顺序
- 第一个加载的可以设为默认实现

**java**

复制

```java
// 应用可以取第一个作为默认
ServiceLoader<Payment> loader = ServiceLoader.load(Payment.class);
Payment defaultPayment = loader.iterator().next();
```

## 🔥 扩展追问 5：SPI vs API 有什么区别？

|方面|API|SPI|
|---|---|---|
|全称|Application Programming Interface|Service Provider Interface|
|方向|我调用你|我被发现|
|谁提供接口|上游提供|下游提供|
|谁写实现|你写|别人写|
|依赖关系|依赖上游|上游依赖我（反向）|
|典型例子|普通接口调用|JDBC、SLF4J|
|配置文件|不需要|需要 META-INF/services|

**图示：**

```
API（正向依赖）：
┌─────────────┐
│  业务代码    │  ← 我
└──────┬──────┘
       │ 依赖
       ↓
┌─────────────┐
│   框架/库    │  ← 你调用它
└─────────────┘

SPI（反向依赖/插件机制）：
┌─────────────┐
│   框架/库    │  ← 它调用我
└──────┬──────┘
       │ 发现
       ↓
┌─────────────┐
│  业务代码    │  ← 我提供实现
└─────────────┘
```

## 🔥 扩展追问 6：Dubbo 的 SPI 相比 JDK SPI 有什么改进？

**JDK SPI 的问题：**

```
1. 一次性加载所有实现，浪费资源
2. 不能按需加载
3. 没有 IOC 功能
4. 不支持 AOP
5. 获取实现类只能遍历
```

**Dubbo SPI 改进：**

**java**

复制

```java
// Dubbo 用 ExtensionLoader
ExtensionLoader<Protocol> loader = ExtensionLoader.getExtensionLoader(Protocol.class);

// 按名称获取（懒加载，只加载需要的）
Protocol http = loader.getExtension("dubbo");
Protocol hessian = loader.getExtension("hessian");

// 获取自适应（自动包装、懒加载）
Protocol adaptive = loader.getAdaptiveExtension();
```

**Dubbo SPI 注解：**

**java**

复制

```java
// 定义扩展点
@SPI("dubbo")
public interface Protocol {
    @Adaptive
    <T> Exporter<T> export(Invoker<T> invoker);
}

// 实现类
@Activate
public class DubboProtocol implements Protocol { }

// 配置文件中：
dubbo=com.alibaba.dubbo.rpc.protocol.dubbo.DubboProtocol
```

**改进对比：**

|特性|JDK SPI|Dubbo SPI|
|---|---|---|
|加载时机|立即加载全部|懒加载|
|IOC|无|有（自动注入依赖）|
|AOP|无|有（Wrapper 包装）|
|获取方式|遍历|按名称直接获取|
|注解支持|无|@SPI/@Adaptive/@Activate|

## 🔥 扩展追问 7：SPI 的优缺点

**优点：**

```
1. 解耦：接口和实现分离
2. 可扩展：新增实现无需修改原代码
3. 动态加载：运行时决定用哪个实现
4. 插件化：第三方可以提供实现
5. 标准化：统一的发现机制
```

**缺点：**

```
1. 遍历加载慢：首次加载要扫描所有配置文件
2. 占用资源：即使不用也会加载
3. 无法指定优先级：除非自己实现排序
4. 调试困难：配置写错不报错，只是找不到
5. JDK 9+ 模块化：需要在 module-info.java 声明
```

## 🔥 扩展追问 9：Spring 的 DI 和 SPI 有什么区别？

**Spring DI（依赖注入）：**

**java**

复制

```java
@Service
public class MyService {
    @Autowired
    private Payment payment;  // Spring 自动注入
}
```

- Spring 容器主动注入
- 配置 + 注解驱动
- 默认单例

**SPI：**

**java**

复制

```java
ServiceLoader<Payment> loader = ServiceLoader.load(Payment.class);
Payment payment = loader.iterator().next();  // 自己去找
```

- 应用主动发现
- 配置驱动
- 每次 iterator() 可能创建新实例

## 一句话收尾

> "SPI 是一种插件化机制，通过 META-INF/services 配置文件 + ServiceLoader 动态发现实现类，解耦了接口和实现。最典型的例子就是 JDBC 驱动加载，JDK 6 之后不用 Class.forName() 了。Spring Boot 的自动配置、Dubbo 的扩展机制都是 SPI 的思想。相比 JDK SPI，Dubbo 做了改进，支持懒加载和 IOC，实际项目中更常用。"
