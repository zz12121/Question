## 先说结论

| | 缓存穿透 | 缓存击穿 | 缓存雪崩 |
|---|---|---|---|
| **问题本质** | 缓存和数据库都查不到，一直打到 DB | 热点 key 过期瞬间，大量请求打到 DB | 大量 key 同时过期，请求打到 DB |
| **原因** | 恶意请求 / 查询不存在的数据 | 热点数据过期 | 缓存服务重启 / 大量 key 同一时刻过期 |
| **特征** | DB 空结果返回，查不到 | 瞬时 DB 压力暴增 | 大量 key 同时失效，DB 压力暴增 |
| **解决方案** | 布隆过滤器 / 存 null 值 / 严格校验 | 互斥锁 / 逻辑过期 / 永不过期 | 过期时间随机化 / 多级缓存 / 预热 |

## 一图总结三种场景

```
【缓存穿透】
请求 → Redis（miss）→ MySQL（查不到）→ 返回空结果
      ↑ 不断重复
      ↓
MySQL 被打死（因为查不到，所以永不入缓存）


【缓存击穿】
热点key: "product:1" (TTL=0, 刚刚过期)
请求A → Redis → MySQL（查到）→ 写入Redis（恢复热点）
请求B → Redis → MySQL（查到）→ 写入Redis
请求C → Redis → MySQL（查到）→ 写入Redis
      ↑ 同时 N 个请求穿透到 DB
      ↓
MySQL 被打死（热点 key 过期了，但恢复前大量请求并发打过来）


【缓存雪崩】
Redis 内很多 key 同时过期：user:1, user:2, ..., user:10000
时间点 T → 这 N 个 key 全部失效
大量请求 → Redis（全部 miss）→ MySQL
      ↓
MySQL 被打死
```

## 逐个详解

### 场景一：缓存穿透

#### 根本原因

```
恶意攻击：攻击者用大量不存在的 ID（如负数 ID、UUID）请求系统
    ↓
这些 ID 在 Redis 中不存在
    ↓
MySQL 也查不到（数据库本身就没有）
    ↓
每次都返回空，不会写入缓存
    ↓
下次请求还是 miss，继续打 MySQL
    ↓
MySQL 被耗尽
```

#### 方案一：布隆过滤器（推荐）

```java
// 原理：用多个哈希函数，把数据映射到一个位数组
// 查询时：多个位全为 1 才认为可能存在，任一位为 0 一定不存在

// 布隆过滤器数据结构
// 10亿数据量，0.01% 误判率，约需要 1.2GB 内存
布隆过滤器 = bit[0...119,951,163]  （约 120M = 15MB）

// Redis 实现（Redisson 内置布隆过滤器）
RBloomFilter<String> bloomFilter = redisson.getBloomFilter("user:bloom");
bloomFilter.tryInit(1_000_000_000L, 0.01);  // 10亿条数据，1% 误判率

// 初始化时写入所有合法 userId
userIds.forEach(id -> bloomFilter.add("user:" + id));

// 查询时先检查布隆过滤器
public User getUser(String userId) {
    String cacheKey = "user:" + userId;
    
    // 1. 布隆过滤器判断是否存在
    if (!bloomFilter.contains(cacheKey)) {
        return null;  // 一定不存在，直接返回，不查 DB
    }
    
    // 2. 查 Redis 缓存
    String cached = redis.get(cacheKey);
    if (cached != null) {
        return JSON.parseObject(cached, User.class);
    }
    
    // 3. 查 MySQL
    User user = userMapper.selectById(userId);
    if (user != null) {
        redis.setex(cacheKey, 3600, JSON.toJSONString(user));
    }
    return user;
}
```

**布隆过滤器的误判场景**：
```
布隆过滤器说"不存在" → 100% 准确 ✅
布隆过滤器说"可能存在" → 有 1% 概率是误判
    ↓
查询布隆过滤器命中 → 还是要查 Redis/MySQL
    ↓
这 1% 误判会带来额外的 DB 查询，但可控
```

#### 方案二：缓存空值（简单粗暴）

```java
public User getUser(String userId) {
    String cacheKey = "user:" + userId;
    String cached = redis.get(cacheKey);
    
    if (cached != null) {
        if ("NULL".equals(cached)) {
            return null;  // 之前查过，不存在
        }
        return JSON.parseObject(cached, User.class);
    }
    
    User user = userMapper.selectById(userId);
    if (user != null) {
        redis.setex(cacheKey, 3600, JSON.toJSONString(user));
    } else {
        // 查不到，也写入 Redis，防止同一不存在的数据反复打 DB
        redis.setex(cacheKey, 60, "NULL");  // 空值短过期（60秒）
    }
    return user;
}
```

> **注意**：空值过期时间要短（60秒以内），否则合法数据插入后仍返回 null。

### 场景二：缓存击穿

#### 根本原因

```
热点 key：商品详情页 "product:1"（被 10000 QPS 访问）
    ↓
TTL = 300 秒，定时任务正常续期
    ↓
某天凌晨 Redis 故障 10 秒，续期任务停了
    ↓
"product:1" 过期了
    ↓
大量请求同时到达（10000 QPS）
    ↓
每个请求都去查 MySQL
    ↓
MySQL 被 10000 个并发请求打死
```

#### 方案一：互斥锁（分布式锁）

```java
public User getUser(String userId) {
    String cacheKey = "user:" + userId;
    String cached = redis.get(cacheKey);
    if (cached != null) {
        return JSON.parseObject(cached, User.class);
    }
    
    // ========== 互斥锁核心 ==========
    String lockKey = "lock:" + cacheKey;
    String lockValue = UUID.randomUUID().toString();
    
    // SETNX + EXPIRE 原子组合（Redis 6.0+ 简化为 SET ... NX EX）
    Boolean acquired = redis.set(lockKey, lockValue, 
        SetParams.setParams().nx().ex(10));
    
    if (Boolean.TRUE.equals(acquired)) {
        try {
            // 只有一个线程进到这里查 DB
            User user = userMapper.selectById(userId);
            if (user != null) {
                redis.setex(cacheKey, 3600, JSON.toJSONString(user));
            }
            return user;
        } finally {
            // ========== 释放锁 ==========
            // 必须用 Lua 脚本保证原子性：只有锁值匹配才删除
            String script = 
                "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                "  return redis.call('del', KEYS[1]) " +
                "else return 0 end";
            jedis.eval(script, 1, lockKey, lockValue);
        }
    } else {
        // 没抢到锁，短暂等待后重试查缓存
        try {
            Thread.sleep(50);
        } catch (InterruptedException e) {}
        return getUser(userId);  // 递归重试
    }
}
```

> **缺陷**：性能损耗，需要等待锁，性能不如下面方案。

#### 方案二：逻辑过期（推荐，实际生产最常用）

```java
// 核心思路：数据永不过期，但用额外字段标记"逻辑过期时间"
// 查到了也不直接返回，而是检查是否逻辑过期
// 如果逻辑过期，异步重建缓存

// 数据结构：User 对象 + 逻辑过期时间
@Data
public class UserVO {
    private Long id;
    private String name;
    private Integer age;
    
    private long logicExpireTime;  // 逻辑过期时间戳（绝对时间）
}

// 重建缓存：后台异步线程，不阻塞主请求
private static final ScheduledExecutorService rebuildExecutor = 
    Executors.newScheduledThreadPool(10);

// 尝试获取用户（逻辑过期方案）
public User getUser(String userId) {
    String cacheKey = "user:" + userId;
    String cached = redis.get(cacheKey);
    
    if (cached != null) {
        UserVO userVO = JSON.parseObject(cached, UserVO.class);
        
        // 检查是否逻辑过期
        if (userVO.getLogicExpireTime() > System.currentTimeMillis()) {
            return userVO;  // 未过期，直接返回
        }
        
        // ========== 逻辑过期：异步重建缓存 ==========
        // 多个请求同时发现过期，都去重建？
        // 用"抢锁"机制，只有一个人重建，其他人等待
        String lockKey = "rebuild:lock:user:" + userId;
        String lockValue = UUID.randomUUID().toString();
        
        Boolean acquired = redis.set(lockKey, lockValue, 
            SetParams.setParams().nx().ex(10));
        
        if (Boolean.TRUE.equals(acquired)) {
            // 抢到锁，异步重建
            rebuildExecutor.submit(() -> {
                User newUser = userMapper.selectById(userId);
                if (newUser != null) {
                    UserVO newVO = new UserVO();
                    newVO.setId(newUser.getId());
                    newVO.setName(newUser.getName());
                    newVO.setLogicExpireTime(
                        System.currentTimeMillis() + 30 * 1000);  // 30秒后逻辑过期
                    redis.setex(cacheKey, 3600, JSON.toJSONString(newVO));
                }
            });
        }
        
        // 返回旧数据（不阻塞等待重建）
        return userVO;
    }
    
    // 缓存没有，直接查 DB 并写入
    User user = userMapper.selectById(userId);
    // 写入带逻辑过期时间的数据
    UserVO userVO = convert(user);
    userVO.setLogicExpireTime(System.currentTimeMillis() + 30 * 1000);
    redis.setex(cacheKey, 3600, JSON.toJSONString(userVO));
    return user;
}
```

### 场景三：缓存雪崩

#### 根本原因

```
方案一：大量 key 同时过期
Redis 中 10000 个 key，过期时间都设为 3600 秒
    ↓
某个整点时刻，大量 key 同时过期
    ↓
这 10000 个 key 在下一秒全部 miss
    ↓
10000 个请求全部穿透到 MySQL
    ↓
MySQL 被打死

方案二：Redis 宕机
Redis 集群主节点宕机
    ↓
从节点晋升需要几秒到几十秒
    ↓
所有请求直接打到 MySQL
    ↓
MySQL 被打死
```

#### 方案一：过期时间随机化

```java
// ❌ 错误写法：所有 key 同一过期时间
redis.set("user:1", value, 3600);
redis.set("user:2", value, 3600);
// → 3600 秒后同时过期 = 雪崩

// ✅ 正确写法：过期时间 + 随机偏移量
int baseExpire = 3600;
int randomOffset = ThreadLocalRandom.current().nextInt(300);  // 0~300秒随机
int actualExpire = baseExpire + randomOffset;  // 3600~3900秒随机
redis.set("user:1", value, actualExpire);
redis.set("user:2", value, actualExpire + 300);
// → 过期时间分散开，避免同时失效

// 或用 Redis 的 EXPIREAT 指定绝对过期时间
long baseTime = System.currentTimeMillis() + 3600_000;  // 1小时后
redis.setex("user:1", 3600 + ThreadLocalRandom.current().nextInt(300), value);
```

#### 方案二：Redis 高可用架构（防雪崩根本手段）

```
【方案：Redis Cluster / Sentinel + 多级缓存】

                    用户请求
                        ↓
                   Nginx 层
                   (本地缓存 5s)
                        ↓ miss
                   Redis Cluster
                  (主节点 + 从节点)
                        ↓ miss
                   MySQL 主从
                        ↓
                    返回数据
                        ↑
                   Redis 主节点
                   (先写入这里)
                   ← 自动同步到从节点
```

```
防止 Redis 宕机导致雪崩：

方案 A：Redis Sentinel（哨兵模式）
    ↓
主节点宕机 → 哨兵自动选举从节点晋升 → 最多 ~30 秒不可用
    ↓
30 秒内所有请求直接打 MySQL → 需要其他防护

方案 B：Redis Cluster（分片集群）
    ↓
某节点宕机 → 只有 1/N 的 key 不可用（N = 分片数）
    ↓
只有 1/N 的请求打 MySQL → 影响范围大幅缩小 ✅

方案 C：本地缓存兜底（最推荐）
    ↓
Redis 不可用时 → Nginx / JVM 本地缓存兜底
    ↓
MySQL 永远不被直接打
```

#### 方案三：多级缓存（Caffeine + Redis）

```java
// 本地缓存 Caffeine（Guava Cache 替代品，更快）
LoadingCache<String, User> localCache = Caffeine.newBuilder()
    .maximumSize(10_000)           // 最多缓存 1 万条
    .expireAfterWrite(5, TimeUnit.SECONDS)  // 写后 5 秒过期
    .build(key -> null);            // 查不到返回 null

public User getUser(String userId) {
    String cacheKey = "user:" + userId;
    
    // 1. 先查本地缓存（Caffeine，纳秒级）
    User local = localCache.getIfPresent(cacheKey);
    if (local != null) {
        return local;
    }
    
    // 2. 查 Redis
    String cached = redis.get(cacheKey);
    if (cached != null) {
        User user = JSON.parseObject(cached, User.class);
        localCache.put(cacheKey, user);  // 回填本地缓存
        return user;
    }
    
    // 3. 查 MySQL
    User user = userMapper.selectById(userId);
    if (user != null) {
        redis.setex(cacheKey, 3600, JSON.toJSONString(user));
        localCache.put(cacheKey, user);
    }
    return user;
}

// 架构：
// Caffeine（5秒）→ Redis（3600秒）→ MySQL
// 即使 Redis 全量失效，本地缓存顶住 5 秒
// 5 秒内 Redis 恢复 = 雪崩被吸收
```

#### 方案四：缓存预热（定时 + 峰值前）

```java
// 定时预热：每天 0 点（流量高峰前）预加载缓存
@Scheduled(cron = "0 0 * * * ?")  // 每小时整点执行
public void preHeatCache() {
    log.info("开始缓存预热...");
    List<Product> products = productMapper.selectTop100HotProducts();
    for (Product product : products) {
        String key = "product:" + product.getId();

    }
    log.info("预热完成，共 {} 条", products.size());
}

// MySQL 故障恢复后预热
@PostConstruct
public void onStart() {
    preHeatCache();  // 应用启动时预热
}
```

### 追问 1：布隆过滤器的实现原理？误判率如何计算？

```
布隆过滤器 = bit数组 + K个哈希函数

初始状态：bit[0...m-1] = 全部 0

添加元素 x：
    h1(x) = 3 → bit[3] = 1
    h2(x) = 7 → bit[7] = 1
    h3(x) = 12 → bit[12] = 1
    ↓
最终 bit[3] = bit[7] = bit[12] = 1

查询元素 y：
    h1(y) = 3 → bit[3] = 1 ✅
    h2(y) = 7 → bit[7] = 1 ✅
    h3(y) = 12 → bit[12] = 1 ✅
    → 三位全 1 → 布隆过滤器说"可能存在"（误判 1%）
```

**误判率公式**：
```
m = bit数组长度
n = 插入的元素数量
k = 哈希函数数量

最优 k = (m/n) × ln2

误判率 p ≈ (1 - e^(-kn/m))^k

举例：
n = 10亿数据，p=1%，m/n ≈ 10
m ≈ 10亿 / 0.01 / 10 = 10亿 bit = 119.2 MB
```

> **布隆过滤器说"不存在" = 100% 准确，不会误杀；说"存在" = 有 1% 概率误判。

## 一句话收尾

> **缓存穿透靠布隆过滤器拦截不存在的请求，缓存击穿靠互斥锁或逻辑过期保护热点 key，缓存雪崩靠过期时间随机化 + 多级缓存 + Redis 高可用架构三管齐下——穿透是"查不到"，击穿是"一个热点过期了"，雪崩是"一堆同时过期了"，三个问题的防护策略要分别针对。**
