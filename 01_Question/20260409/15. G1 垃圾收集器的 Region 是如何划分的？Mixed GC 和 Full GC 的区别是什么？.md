## 先说结论

**G1 = Garbage First，面向服务端的垃圾收集器。**

```
核心设计理念：
1. 把堆分成等大的 Region（不再是连续的新生代/老年代）
2. 优先回收垃圾最多的 Region（Garbage First！）
3. 可预测的停顿时间（-XX:MaxGCPauseMillis=200）
```

---

## 🔥 Region 划分详解

### 传统收集器 vs G1

```
传统（CMS/Parallel）：
┌──────────────────────┬──────────────────────────────────┐
│    Young Gen         │         Old Gen                  │
│ Eden │ S0 │ S1       │                                  │
└──────────────────────┴──────────────────────────────────┘
   固定大小，物理连续

G1：
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ E  │ O  │ E  │ S  │ H  │ O  │ E  │ S  │
├────┼────┼────┼────┼────┼────┼────┼────┤
│ O  │ E  │ H  │ O  │ S  │ E  │ O  │ E  │
├────┼────┼────┼────┼────┼────┼────┼────┤
│ E  │ S  │ O  │ E  │ O  │ H  │ E  │ O  │
└────┴────┴────┴────┴────┴────┴────┴────┘
  E=Eden  S=Survivor  O=Old  H=Humongous
  
不再物理连续！Region 可以在堆的任意位置
```

### Region 详细参数

```
堆大小：  通过 -Xms -Xmx 设置
Region 数：默认 2048 个（1MB ~ 32MB，必须是 2 的幂）

计算公式：
Region 大小 = 堆大小 / 2048
如果不是 2 的幂，向上取整

例子：
堆 4GB → Region = 4096MB/2048 = 2MB
堆 8GB → Region = 8192MB/2048 = 4MB
堆 32GB → Region = 32768MB/2048 = 16MB

手动设置：
-XX:G1HeapRegionSize=4m
```

### 四种 Region 类型

```
Eden Region（E）：
    新对象分配在这里
    Young GC 后会被清空或晋升
    数量动态调整（不是固定的！）

Survivor Region（S）：
    存放 Young GC 后存活的对象
    对象在这里熬过几次 GC 后晋升到 Old

Old Region（O）：
    存放长期存活对象
    Mixed GC 时才会回收

Humongous Region（H）：
    存放超过 Region 大小 50% 的大对象
    一个对象可以跨多个连续 H Region
    物理上连续，可能有碎片
    JDK 8 后 H 随 Young GC 一起回收
```

---

## 🔥 G1 GC 类型

```
G1 有三种 GC：

1. Young GC（Minor GC）
   - 回收 Eden 和 Survivor Region
   - 完全 STW
   - 时机：Eden 满了

2. Mixed GC
   - 回收全部 Young Region + 部分 Old Region
   - 完全 STW
   - 时机：老年代占堆空间超过阈值（默认 45%）

3. Full GC
   - 回收整个堆
   - 完全 STW，单线程（JDK 10+ 多线程）
   - 时机：Mixed GC 来不及清理，内存耗尽
```

---

## 🔥 Young GC 流程

```
触发条件：Eden Region 填满

Step 1: STW
Step 2: 扫描 GC Roots + Remembered Set
Step 3: 标记所有存活对象
Step 4: 存活对象复制到 Survivor Region
         或晋升到 Old Region（年龄够了）
Step 5: 清空 Eden Region（直接回收）
Step 6: 统计本次停顿时间，调整 Eden 数量

特点：
- 完全 STW
- 停顿时间跟 Eden Region 数量有关
- G1 会动态调整 Eden 数量来满足停顿目标
```

---

## 🔥 Mixed GC 完整流程（重点！）

### Mixed GC 分两大阶段

**阶段一：全局并发标记（与业务线程并发执行）**

```
Step 1: 初始标记（Initial Mark）—— STW，通常随 Young GC 一起做
    标记 GC Roots 直接可达的对象
    顺带做了一次 Young GC

Step 2: 根区域扫描（Root Region Scan）—— 并发
    扫描 Survivor Region 中引用的 Old Region
    必须在下次 Young GC 前完成

Step 3: 并发标记（Concurrent Mark）—— 并发
    从 Step 1 标记的对象开始，递归标记整个堆
    使用 SATB（原始快照）算法处理并发修改
    同时统计每个 Region 的存活对象数量和垃圾占比

Step 4: 最终标记（Remark）—— STW
    处理 SATB 记录的并发期间的引用变化
    清空 SATB 缓冲区

Step 5: 清理（Cleanup）—— STW + 并发
    统计各 Region 存活对象
    按照 Region 垃圾量排序（这就是 Garbage First！）
    回收全空的 Old Region（立即）
    其他 Region 加入"候选回收集合"
```

**阶段二：Mixed GC 回收（STW）**

```
根据全局并发标记的结果，选择：
    所有 Young Region（Eden + Survivor）
  + 部分 Old Region（垃圾最多的那些！）

复制存活对象到空的 Region
释放被选中的 Region

关键参数：
-XX:G1MixedGCLiveThresholdPercent=85
    Old Region 存活对象超过 85% 就不回收（性价比低）

-XX:G1HeapWastePercent=5
    允许堆中 5% 的垃圾不回收（减少 Mixed GC 次数）

-XX:G1MixedGCCountTarget=8
    一轮并发标记后最多做 8 次 Mixed GC
```

## 🔥 Mixed GC vs Full GC 的区别

| 对比项 | Mixed GC | Full GC |
|--------|----------|---------|
| **回收范围** | Young + 部分 Old Region | 全堆（Young + Old + H） |
| **线程** | 多线程并行 | 单线程（JDK 10+ 多线程） |
| **STW 时间** | 短（可控，百毫秒级） | 极长（秒级甚至分钟级） |
| **触发条件** | 老年代占比 > 45% | 内存耗尽、Mixed GC 跟不上 |
| **算法** | 标记-复制 | 标记-整理 |
| **碎片** | 无（复制算法） | 无（整理算法） |
| **频率** | 定期触发 | 尽量避免！|

## 🔥 扩展追问 1：Remembered Set（记忆集）是什么？

**核心问题：**
```
回收 Young Region 时，Old Region 里可能有引用指向 Young Region
必须把这些引用也当作 GC Roots！
否则会把被 Old Region 引用的对象误认为垃圾

但扫描整个 Old Region 太慢了...
```

**Remembered Set（RSet）：**

```
每个 Region 都有一个 RSet
RSet 记录：哪些其他 Region 的对象引用了我

例：
Region A（Old）中的对象 x 引用了 Region B（Young）中的对象 y
→ Region B 的 RSet 会记录：来自 Region A

Young GC 时：
    扫描 Eden/Survivor 的 RSet
    找到所有外部引用
    把这些引用也加入 GC Roots
    就不需要扫描整个 Old Region 了！
```

**写屏障维护 RSet：**

```java
// 每次修改引用时，写屏障触发
obj.field = newRef;  // 写操作
// 写屏障检查：newRef 和 obj 是否在不同 Region？
// 是 → 更新 newRef 所在 Region 的 RSet
```

**RSet 的代价：**
```
每个 Region 都要维护 RSet
写操作时要更新 RSet
额外内存占用 10%~20%
```

## 🔥 扩展追问 2：SATB 算法（原始快照）

**G1 用 SATB 解决并发标记时的对象消失问题：**

```java
// 当删除引用时，写屏障触发
old_ref = B.ref;  // 记录被删除的引用
B.ref = null;

// 把 old_ref 放入 SATB 队列
// 并发标记结束后，在 Remark 阶段处理 SATB 队列
// 把队列里的对象都当作灰色重新扫描
```

**SATB 的思路：**
```
"保留并发标记开始时的存活快照"
只要在并发标记开始时是存活的，就不会被误回收
即使后来引用被删除了，这次也不收
下一次 GC 才会收
这就是"浮动垃圾"，G1 接受这个代价
```

**与 CMS 增量更新的对比：**
```
CMS 增量更新：记录新增的引用（黑→白），重新扫描黑色节点
G1 SATB：记录删除的引用，重新扫描被删除引用的对象

SATB 在 Remark 阶段更快
但引入了更多浮动垃圾
```

## 🔥 扩展追问 3：什么时候触发 Full GC？怎么避免？

**触发 Full GC 的场景：**

```
1. 并发模式失败（Evacuation Failure）
   Mixed GC 时，复制存活对象找不到空的 Region
   
2. Humongous 对象分配失败
   大对象找不到连续的 H Region

3. 元空间满了（MetaSpace OOM）

4. System.gc() 被调用（未禁用的情况）

5. JVM 内部判断 G1 跟不上垃圾产生速度
```

**避免 Full GC 的策略：**

```bash
# 提前触发 Mixed GC（默认 45%，可调低）
-XX:InitiatingHeapOccupancyPercent=35

# 调大堆内存（减少内存压力）
-Xmx8g

# 增加 Mixed GC 次数上限（每轮标记后做更多次清理）
-XX:G1MixedGCCountTarget=16

# 降低 Old Region 的存活率阈值（更激进回收）
-XX:G1MixedGCLiveThresholdPercent=65

# 增大 G1 的 Region 大小（减少 Humongous 对象）
-XX:G1HeapRegionSize=8m
```

## 🔥 扩展追问 4：G1 的停顿预测模型

**G1 最核心的特性：可预测的停顿时间**

```bash
# 设置目标停顿时间（默认 200ms）
-XX:MaxGCPauseMillis=200
```

**实现原理：**

```
G1 给每个 Region 维护历史统计数据：
    - 回收该 Region 预计耗时
    - 该 Region 垃圾量
    
每次 GC 前：
    计算最大停顿时间 → 倒推能回收多少 Region
    优先选垃圾最多、回收最快的 Region
    
这是一个"启发式"算法，不是硬保证
但统计数据越多，预测越准确
```

**动态调整 Eden 数量：**

```
如果上次 Young GC 停顿超过目标：
    G1 减少 Eden Region 数量
    下次 GC 时间自然变短

如果停顿明显低于目标：
    G1 增加 Eden Region 数量
    允许更多对象积累，减少 GC 频率
    
在吞吐量和停顿时间之间动态平衡
```

## 🔥 扩展追问 5：G1 调优关键参数

```bash
# 基础设置
-XX:+UseG1GC                          # 启用 G1（JDK 9+ 默认开启）
-Xms4g -Xmx4g                         # 堆大小
-XX:MaxGCPauseMillis=200              # 最大停顿时间目标

# Region 大小
-XX:G1HeapRegionSize=4m               # 手动设置（2 的幂，1~32MB）

# 并发标记触发时机
-XX:InitiatingHeapOccupancyPercent=45 # 老年代达到 45% 开始并发标记

# Mixed GC 控制
-XX:G1MixedGCLiveThresholdPercent=85  # 存活率超过 85% 不回收
-XX:G1HeapWastePercent=5              # 允许 5% 空间浪费
-XX:G1MixedGCCountTarget=8           # 每轮最多 8 次 Mixed GC

# GC 线程
-XX:ParallelGCThreads=8               # STW 时并行 GC 线程数
-XX:ConcGCThreads=2                   # 并发标记线程数
```

## 🔥 扩展追问 6：G1 和 ZGC 还有哪些区别？

| 对比项 | G1 | ZGC |
|--------|----|----|
| STW 时间 | 百毫秒级 | <10ms（目标 1ms 以内）|
| 堆大小 | 数 GB ~ 数十 GB | 支持 TB 级 |
| 回收算法 | 标记-复制 | 染色指针+读屏障 |
| 并发 | 标记并发，回收 STW | 标记+回收都并发 |
| CPU 开销 | 较低 | 较高（读屏障代价）|
| 内存碎片 | 无 | 无 |
| 适用场景 | 通用，延迟敏感 | 超低延迟，大内存 |
| JDK 版本 | JDK 9 默认 | JDK 15 正式 |

**ZGC 的核心创新：**
```
染色指针：把 GC 状态信息存到指针的高位比特里
读屏障：读对象时自动触发，而不是写时才触发
不需要 Remembered Set（省内存）
回收时对象可以被并发移动（G1 回收时要 STW）
```

## 一句话收尾

> "G1 把堆分成 2048 个等大的 Region，分 Eden/Survivor/Old/Humongous 四种，不再是传统的连续分代。Mixed GC 是 G1 的主力，并发标记后优先回收垃圾最多的 Region，停顿时间可控；Full GC 是兜底手段，单线程 STW，代价极大，要尽量避免。G1 靠 Remembered Set 和 SATB 算法解决跨 Region 引用和并发标记问题，整体上比 CMS 更现代、更稳定。"
