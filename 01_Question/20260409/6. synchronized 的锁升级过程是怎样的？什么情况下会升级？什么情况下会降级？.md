## 先说结论

**synchronized 的锁升级 = 无锁 → 偏向锁 → 轻量级锁 → 重量级锁，是一个从"乐观"到"悲观"的性能优化过程。**

```
核心思想：
无竞争时尽量轻量，有竞争时逐步升级，避免一上来就用操作系统互斥锁
```

---

## 🔥 核心概念

### 锁升级解决什么问题？

**场景：多线程竞争同一把锁**

```
线程 A 执行 synchronized(obj) { ... }
      ↓
线程 B 也执行 synchronized(obj) { ... }
```

**如果没有锁升级：**

- 每次都用重量级锁（OS mutex）
- 线程阻塞/唤醒需要用户态 → 内核态切换
- 即使锁只持有几毫秒，也要挂起线程，性能极差

**有了锁升级：**

- 无竞争时用偏向锁（零开销）
- 轻微竞争时用轻量级锁（CAS 自旋）
- 激烈竞争时才用重量级锁（线程阻塞）
- 根据竞争程度动态调整

---

## 🔥 锁升级的四个阶段

### 阶段 0：无锁

```java
Object obj = new Object();  // 刚创建，无锁状态
```

**Mark Word 存储：**

- hashCode（31位）
- GC 分代年龄（4位）
- 偏向标志位：0
- 锁标志位：01

---

### 阶段 1：偏向锁

```java
synchronized(obj) {
    // 第一个线程 A 来拿锁
}
```

**升级触发条件：** 第一个线程获取锁

**JVM 做了什么：**

```java
// 1. 检查 Mark Word 的偏向标志位
if (biased_lock_bit == 0) {
    // 2. CAS 把线程 A 的 ID 写入 Mark Word
    // 3. 设置偏向标志位 = 1
}
```

**Mark Word 存储：**

- 线程 A 的 ID（54位）
- epoch（2位，批量重偏向用）
- GC 分代年龄
- 偏向标志位：1
- 锁标志位：01

**后续线程 A 再进入：**

```java
// 只需要检查 Mark Word 里的线程 ID 是不是自己
if (markWord.threadId == currentThreadId) {
    // 直接执行，零开销！
}
```

---

### 阶段 2：轻量级锁

```java
// 线程 B 也来竞争这把锁
synchronized(obj) {
    // 偏向锁撑不住了，需要升级
}
```

**升级触发条件：** 有第二个线程竞争

**升级过程（必须在安全点执行）：**

```java
// 1. JVM 在安全点暂停持有偏向锁的线程 A
// 2. 检查线程 A 是否还在同步块内
if (线程A已退出同步块) {
    // 撤销偏向锁，恢复成无锁
} else {
    // 升级成轻量级锁
}
```

**轻量级锁的实现：**

```java
// 线程在自己的栈帧中创建 Lock Record
LockRecord lockRecord = new LockRecord();

// CAS 尝试把 Mark Word 替换成指向 Lock Record 的指针
boolean success = cas(obj.markWord, lockRecord);

if (success) {
    // 获取锁成功
} else {
    // 自旋重试
    for (int i = 0; i < spinCount; i++) {
        if (cas(obj.markWord, lockRecord)) {
            break;
        }
    }
}
```

**Mark Word 存储：**

- 指向栈中 Lock Record 的指针（62位）
- 锁标志位：00

---

### 阶段 3：重量级锁

```java
// 自旋了很多次还是拿不到锁
// 竞争激烈，只能升级
```

**升级触发条件：**

|条件|说明|
|---|---|
|自旋超过阈值|默认 10 次（JDK 6 后自适应）|
|等待线程数超过 CPU 核数一半|JVM 认为自旋不划算|

**重量级锁的实现：**

```java
// 每个对象关联一个 ObjectMonitor（C++ 实现）
ObjectMonitor monitor = new ObjectMonitor();

// Mark Word 指向 monitor
obj.markWord = ptr_to_monitor;

// 获取锁
if (monitor._owner == null) {
    monitor._owner = currentThread;
} else {
    // 进入 _EntryList 阻塞队列，线程挂起
    monitor._EntryList.add(currentThread);
    park(currentThread);  // 真正阻塞
}
```

**ObjectMonitor 结构：**

```c++
class ObjectMonitor {
    Thread* _owner;           // 持有锁的线程
    Queue   _EntryList;       // 等待获取锁的线程队列（阻塞）
    Queue   _WaitSet;         // 调用 wait() 的线程队列
    int     _recursions;      // 重入次数
};
```

**Mark Word 存储：**

- 指向 ObjectMonitor 的指针（62位）
- 锁标志位：10

---

## 🔥 锁升级完整流程图

```
对象创建
   ↓
无锁 (001)
   ↓ 第一个线程获取 synchronized
偏向锁 (101)  ← 线程ID写入Mark Word，后续零开销
   ↓ 第二个线程竞争 → 安全点撤销/升级
   ├─ 线程A已退出 → 恢复无锁
   └─ 线程A还在执行 → 升级轻量级锁
      ↓
轻量级锁 (00) ← CAS + 自旋
   ↓ 自旋失败/竞争激烈
重量级锁 (10) ← OS mutex，线程真正阻塞
```

## 🔥 扩展追问 1：锁会降级吗？

**正常运行中：不会降级，升级是单向的。**

**特殊情况（GC 安全点期间）：**

```java
// JVM 进入安全点，比如 GC 时
if (重量级锁 && 没有线程持有) {
    // 可以恢复 Mark Word，相当于降级
    // 但这是 JVM 内部行为，不暴露给业务
}
```

---

## 🔥 扩展追问 2：为什么偏向锁撤销要在安全点？

**原因：需要检查持有锁线程的栈帧**

```java
// JVM 必须在安全点暂停线程A，才能安全地检查：
if (线程A的栈帧中还有Lock Record) {
    // 还在同步块内，需要升级
} else {
    // 已经退出，可以撤销
}
```

**不在安全点检查的风险：**

- 线程 A 正在修改栈帧
- 线程 B 同时来撤销偏向锁
- 数据不一致，可能崩溃

## 🔥 扩展追问 3：自适应自旋是什么？

**JDK 6 之前：固定自旋 10 次**

**JDK 6 之后：自适应自旋**

```java
// JVM 根据历史记录动态调整
if (上次自旋成功拿到锁) {
    这次多自旋几次（比如 20 次）
} else {
    这次少自旋几次（比如 5 次），甚至直接跳过
}
```

**好处：** 根据锁的实际竞争情况智能调整，不会无谓浪费 CPU

## 🔥 扩展追问 4：epoch 是干什么用的？

**场景：批量撤销偏向锁**

```java
// 类 C 的对象频繁发生偏向锁撤销（超过 20 次）
// JVM 认为这个类不适合用偏向锁了

// 批量重偏向：让这个类的所有对象可以偏向新线程
类C.epoch++;  // epoch 加 1

// 线程再次访问时，发现自己的 epoch 和类的不一样
// 就可以重新偏向，而不需要逐个撤销
```

**批量撤销阈值：40 次**

```java
if (偏向锁撤销次数 > 40) {
    // 这个类以后不再使用偏向锁
    // 直接从无锁 → 轻量级锁
}
```

## 🔥 扩展追问 5：JDK 15 为什么关闭偏向锁？

**原因：**

```
1. 现代应用一启动就是多线程，偏向锁收益有限
2. 偏向锁撤销需要在安全点执行，带来 STW 停顿
3. 维护成本高，代码复杂
```

**JDK 15 变化：**

```java
// 默认关闭
-XX:-UseBiasedLocking

// 可以手动开启
-XX:+UseBiasedLocking

// JDK 18 彻底移除
```

**影响：**

- 对象直接从无锁 → 轻量级锁
- 单线程反复获取同一把锁的场景性能略降
- 但多线程场景性能更稳定

## 🔥 扩展追问 6：Mark Word 在不同锁状态下的存储

|锁状态|存储内容|锁标志位|
|---|---|---|
|无锁|hashCode + GC年龄 + 0 + 01|01|
|偏向锁|线程ID + epoch + GC年龄 + 1 + 01|01|
|轻量级锁|指向 Lock Record 的指针 + 00|00|
|重量级锁|指向 ObjectMonitor 的指针 + 10|10|
|GC标记|空 + 11|11|

**注意：** 偏向标志位 0/1 区分无锁和偏向锁

## 🔥 扩展追问 7：为什么调用 hashCode() 会撤销偏向锁？

```java
Object obj = new Object();
obj.hashCode();  // 调用后，偏向锁就用不了了！
```

**原因：Mark Word 没地方同时存线程ID和hashCode**

```
偏向锁的 Mark Word：
[线程ID:54位][epoch:2位][GC年龄:4位][偏向:1][锁:01]

无锁的 Mark Word：
[hashCode:31位][未使用:25位][GC年龄:4位][偏向:0][锁:01]
```

**调用 hashCode() 后：**

- 需要把 hashCode 写进 Mark Word
- 但偏向锁的 Mark Word 里存的是线程 ID
- 只能撤销偏向锁，恢复成无锁

## 🔥 扩展追问 8：synchronized 锁升级 vs ReentrantLock

|特性|synchronized|ReentrantLock|
|---|---|---|
|锁升级|有（JVM 自动）|无（AQS 直接阻塞）|
|实现|JVM 层|JDK 层（Java 代码）|
|性能|无竞争时更好（偏向锁）|竞争激烈时更可控|
|功能|简单|支持公平锁、中断、超时等|
|灵活性|低|高|

---

## 一句话收尾

> "synchronized 的锁升级是 JVM 为了优化性能设计的，从偏向锁（无竞争零开销）→ 轻量级锁（CAS自旋）→ 重量级锁（OS阻塞），根据竞争程度动态调整。偏向锁在 JDK 15 默认关闭，因为现代多线程应用收益有限。理解 Mark Word 的结构和升级触发条件，是回答这个问题的关键。"
