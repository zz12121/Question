|方面|String|StringBuilder|StringBuffer|
|---|---|---|---|

## 🔥 扩展追问 1：String 为什么不可变？

**String 类的关键实现：**

```java
public final class String {
    // 用 final 修饰，类不能被继承，防止被篡改
    // 用 private 修饰，对外不可见
    // 没有提供 setter
    private final char[] value;  // JDK 8
    private final byte[] value;  // JDK 9+ 为了省内存，用 byte[]
}
```

**不可变的 5 大好处：**

```
1. 线程安全：多线程共享不需要同步
2. 哈希缓存：String 的 hashCode 可以缓存，适合做 HashMap 的 key
3. 安全性：数据库用户名密码、URL、文件路径等都是 String
4. 字符串常量池：省内存
5. 类加载器规范：反射、类加载依赖字符串不可变性
```

**面试加分话术：**

> "String 用 final 修饰保证了类的不可继承性，用 private final char[] 存储数据且不提供修改方法，从设计上保证了不可变。这不是语言限制，是刻意的设计选择。"

## 🔥 扩展追问 2：字符串常量池是什么？

**先看一个现象：**

```java
String s1 = "hello";
String s2 = "hello";
String s3 = new String("hello");

System.out.println(s1 == s2);  // true  ← 地址相同！
System.out.println(s1 == s3);  // false ← new 出来的是新对象
System.out.println(s1.equals(s3));  // true ← 内容相同
```

**字符串常量池原理：**

```
常量池（方法区）
┌─────────────────┐
│ "hello"         │  ← 只存一份
└─────────────────┘
    ↑
    ├── s1 指向它
    └── s2 指向它

new String("hello")：
    ├── 堆内存 new 了一个对象
    └── s3 指向堆内存的对象
```

**字符串拼接：**

```java
String s1 = "a" + "b";  // 编译时优化成 "ab"
String s2 = "ab";
String s3 = new String("ab");

System.out.println(s1 == s2);  // true（都在常量池）
System.out.println(s1 == s3);  // false（s3 在堆）
```

**intern() 的作用（重点来了）：**

```java
String s1 = new String("hello");  // 堆里有一个对象
s1.intern();  // 把 "hello" 放入常量池（如果常量池里没有）

String s2 = "hello";  // 从常量池取
System.out.println(s1 == s2);  // JDK 6: false  JDK 7+: true！

// 为什么 JDK 7+ 不一样？
// 因为 JDK 7 把常量池从方法区移到了堆里
// intern() 不再复制字符串，只是记录引用
```

## 🔥 扩展追问 3：intern() 详细原理（JDK 7+）

```java
// 案例 1
String s1 = new String("a") + new String("b");
// 此时常量池：没有 "ab"（拼接结果不在常量池）
// 堆里：有 "a", "b", "ab"

s1.intern();  // 把堆里 "ab" 的引用放入常量池
String s2 = "ab";
System.out.println(s1 == s2);  // true！都指向堆里的 "ab"
```

```java
// 案例 2（不用 intern 的情况）
String s1 = new String("a") + new String("b");
// 堆里有 "ab"，常量池没有

String s2 = "ab";  // 此时把 "ab" 放入常量池
// s2 指向常量池里的 "ab"

String s3 = s1.intern();  // 返回常量池的引用
System.out.println(s1 == s2);  // false！s1 在堆，s2 在常量池
```

**intern() 总结：**

```
JDK 6：把字符串复制到常量池，返回常量池地址
JDK 7+：把字符串的引用放入常量池，返回常量池引用

使用场景：
- 大量重复字符串，用 intern() 可以省内存
- 但要注意 JDK 6 里常量池在方法区，容易 OOM
```

## 🔥 扩展追问 4：StringBuilder vs StringBuffer 性能差距？

**StringBuffer 的 synchronized：**

```java
// StringBuffer append 方法
@Override
public synchronized StringBuffer append(String str) {
    toStringCache = null;
    super.append(str);
    return this;
}
// 每个操作都要获取锁、解锁，开销不小
```

**性能对比：**

```java
// 单线程下测试 10万次拼接：
StringBuilder:   15ms
StringBuffer:    45ms  ← 慢 3 倍（锁开销）
String:        2000ms ← 慢 100+ 倍（大量 new 对象）

// 多线程下：
StringBuilder: 不安全（可能数据错乱）
StringBuffer:  安全但慢
```

**实战建议：**

> "日常拼接用 StringBuilder，多线程场景下如果要保证线程安全，可以用 StringBuffer，或者考虑更现代的方案比如 Java 8 的 StringJoiner。"

## 🔥 扩展追问 5：字符串拼接的底层原理？

**简单拼接（字面量）：**

```java
String s = "a" + "b" + "c";
// 编译器直接优化成：String s = "abc";
```

**变量拼接：**

```java
String a = "hello";
String b = a + " world";
// 编译后等价于：
String b = new StringBuilder(a).append(" world").toString();
```

**所以循环里别这样写：**

```java
// ❌ 错误写法：每次循环都 new StringBuilder
String result = "";
for (int i = 0; i < 1000; i++) {
    result += i;  // 实际：result = new StringBuilder(result).append(i).toString()
}
// 每轮都要 toString()，创建大量中间对象，GC 压力大
```

**正确写法：**

```java
// ✅ StringBuilder 预分配容量
StringBuilder sb = new StringBuilder(5000);
for (int i = 0; i < 1000; i++) {
    sb.append(i);
}
String result = sb.toString();
```

## 🔥 扩展追问 6：new String("abc") 创建了几个对象？

**这道题常考，看源码：**

```java
// String 构造方法
public String(String original) {
    this.value = original.value;
    this.coder = original.coder;
    this.hash = original.hash;
}
```

```java
String s = new String("abc");
```

**分析：**

```
1. "abc" 本身是字符串字面量 → 先检查常量池有没有
   - 如果没有，在常量池创建一个 "abc"

2. new String("abc") → 在堆里创建一个新 String 对象
   - 把常量池里 "abc" 的 char[] value 复制一份给新对象

结论：
- 常量池里有 "abc" → 1 个对象（堆里）
- 常量池里没有 "abc" → 2 个对象（常量池 + 堆）
```

**实战问题：**

```java
String s1 = new String("ab") + new String("c");
// 此时常量池：没有 "ab", 没有 "c", 没有 "abc"
// 堆里：new String("ab"), new String("c"), "abc"（StringBuilder.toString()）

s1.intern();  // 把堆里 "abc" 的引用放入常量池
String s2 = "abc";

System.out.println(s1 == s2);  // true
```

## 🔥 扩展追问 7：String 能当 HashMap 的 key 吗？为什么？

**能，而且很常见。**

**String 作为 HashMap key 的优势：**

```
1. 不可变 → hashCode 不会变，数据安全
2. hashCode 会被缓存 → 计算一次，反复用，查询快
3. 字符串常用 → 适合做字典、缓存的 key
4. equals 方法已重写，逻辑清晰
```

**反例：如果 key 可变会怎样？**

```java
// 假设有个可变类 MutableKey
MutableKey key = new MutableKey("a", 1);
map.put(key, "value");

key.setA("b");  // 修改了 key 的 hashCode 变了！
// map.get(key) 找不到值了！
// 因为 key 在 map 中的位置变了，但数据还在原位置
```

## 🔥 扩展追问 8：String 在 JDK 9 有什么变化？

**JDK 9+ 把 char[] 改成了 byte[]：**

```java
// JDK 8
private final char[] value;

// JDK 9+
private final byte[] value;
private final coder;  // LATIN1 或 UTF16 编码标识
```

**为什么改？**

- 大多数字符串是 Latin-1 字符（一个字符只需要 1 字节）
- char 在 JVM 里固定 2 字节，造成 50% 的空间浪费
- 用 byte[] + coder 标识，Latin-1 用 1 字节，汉字用 2 字节
- **省内存！String 在 JVM 里占的空间从 48 字节降到了 24 字节**
