## 先说结论

Spring Bean 生命周期核心是 **"创建 → 填充属性 → 初始化 → 使用 → 销毁"**，中间插入了 4 类共 10+ 个扩展点，面试能说出 **BeanFactoryPostProcessor → InstantiationAwareBeanPostProcessor → InitializingBean + init-method → DisposableBean + destroy-method** 这条链路，就已经是中上水平了。

## 完整生命周期流程图

```

   BeanDefinition 准备完成
     ↓
```

## 四类扩展点详解

### 第一类：BeanFactory 级别（最早期，修改 Bean 定义）

#### 1. BeanFactoryPostProcessor

**执行时机**：所有 BeanDefinition 加载完毕后，Bean 实例化之前。

```java
// 场景：动态注册 Bean、修改 Bean 定义、替换占位符
@Component
public class MyBeanFactoryPostProcessor implements BeanFactoryPostProcessor {
    
    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) {
        // 1. 获取某个 Bean 的定义
        BeanDefinition userService = beanFactory.getBeanDefinition("userService");
        
        // 2. 修改 Bean 的作用域
        userService.setScope(BeanDefinition.SCOPE_SINGLETON);
        
        // 3. 修改构造参数（构造函数注入的璯性）
        MutablePropertyValues pv = userService.getPropertyValues();
        pv.add("timeout", 3000);
        
        // 4. 动态注册新的 Bean（不需要 @Component 注解）
        AbstractBeanDefinition newBean = BeanDefinitionBuilder
            .genericBeanDefinition(MyService.class)
            .addPropertyValue("name", "动态注册的Bean")
            .getBeanDefinition();
        beanFactory.registerBeanDefinition("dynamicService", newBean);
        
        // 5. 替换占位符 ${xxx} 为实际值
        // PropertyPlaceholderConfigurer（Spring 3.1 之前）
        // PropertySourcesPlaceholderConfigurer（Spring 3.1 之后，已内置）
    }
}
```

**Spring 内置实现**：

| 实现类 | 作用 |
|--------|------|
| `PropertySourcesPlaceholderConfigurer` | 替换 `${}` 占位符 |
| `ConfigurationClassPostProcessor` | 处理 `@Configuration` 配置类 |
| `CustomAutowireConfigurer` | 自定义自动装配规则 |

### 第二类：Bean 实例化阶段（决定要不要实例化）

#### 2. InstantiationAwareBeanPostProcessor（扩展点中最复杂的一个）

**执行时机**：Bean 实例化前后 + 属性填充前后。

```java
@Component
public class MyInstantiationAwareBeanPostProcessor 
    implements InstantiationAwareBeanPostProcessor {
    
    // ========== 实例化之前 ==========
    // 返回非空 → 直接用返回的实例，不再走构造函数
    // 返回 null → 继续走默认构造函数
    @Override
    public Object postProcessBeforeInstantiation(Class<?> beanClass, String beanName) {
        if (beanClass == UserService.class && "userService".equals(beanName)) {
            // 可以在这里返回一个代理对象，完全绕过构造函数
            return createMockProxy(beanClass);
        }
        return null;  // 继续走正常流程
    }
    
    // ========== 实例化之后，属性填充之前 ==========
    // 返回 false → 跳过属性注入（但后续回调仍会调用）
    @Override
    public boolean postProcessAfterInstantiation(Object bean, String beanName) {
        if (bean instanceof UserService && "userService".equals(beanName)) {
            // 可以在这里阻止属性注入
            // 比如：某些场景下不想让 Spring 注入璯性
            return false;  // 跳过后续的属性填充
        }
        return true;  // 继续正常流程
    }
    
    // ========== 属性填充之前，动态注入璯性 ==========
    // 可以在这里手动注入璯性，覆盖 @Autowired 的效果
    @Override
    public PropertyValues postProcessProperties(
            PropertyValues pvs, Object bean, String beanName) {
        if (bean instanceof UserService) {
            // 动态添加璯性
            MutablePropertyValues mpv = pvs instanceof MutablePropertyValues
                ? (MutablePropertyValues) pvs
                : new MutablePropertyValues(pvs);
            
            mpv.add("dynamicField", "我被动态注入的璯性值");
            return mpv;
        }
        return pvs;
    }
}
```

**@Autowired 注解就是靠这个扩展点实现的**：

```
AutowiredAnnotationBeanPostProcessor extends InstantiationAwareBeanPostProcessor
    ↓
postProcessProperties() 中：
    → 扫描 @Autowired、@Inject、@Value 注解的璯性
    → 通过 BeanFactory 解析依赖
    → 反射注入璯性值
```

### 第三类：Aware 感知接口（让 Bean 感知容器）

#### 3. Aware 系列接口

```java
@Component
public class MyAwareBean implements BeanNameAware,
                                    BeanFactoryAware,
                                    ApplicationContextAware,
                                    EnvironmentAware,
                                    EmbeddedValueResolverAware {
    
    private String beanName;
    private BeanFactory beanFactory;
    private ApplicationContext applicationContext;
    private Environment environment;
    
    @Override
    public void setBeanName(String name) {
        this.beanName = name;
        System.out.println("Bean 名称：" + name);
    }
    
    @Override
    public void setBeanFactory(BeanFactory beanFactory) {
        this.beanFactory = beanFactory;
        // 可以在这里手动 getBean()，但不推荐（璯性注入更好）
    }
    
    @Override
    public void setApplicationContext(ApplicationContext ctx) {
        this.applicationContext = ctx;
        // ApplicationContext 在这里获取，而不是构造函数中
        // 因为构造函数执行时，ApplicationContext 还没准备好
    }
    
    @Override
    public void setEnvironment(Environment environment) {
        this.environment = environment;
        // 获取配置璯性
        String dbUrl = environment.getProperty("spring.datasource.url");
    }
    
    @Override
    public void setEmbeddedValueResolver(StringValueResolver resolver) {
        // 解析 ${} 占位符
        String resolved = resolver.resolveStringValue("${server.port:8080}");
    }
}
```

**Aware 执行顺序**（按继承链深度，从深到浅）：

```
BeanNameAware → BeanFactoryAware → ApplicationContextAware
     ↓
（越接近底层容器，越先执行）
```

### 第四类：初始化和销毁（最后的机会）

#### 4. InitializingBean + init-method

```java
@Component
public class MyService implements InitializingBean {
    
    private String config;
    
    // 璯性注入（由 InstantiationAwareBeanPostProcessor 完成）
    @Value("${my.config:default}")
    public void setConfig(String config) {
        this.config = config;
    }
    
    // ========== 初始化回调 ==========
    // Spring 璯性填充完毕后调用
    // 如果璯性注入失败，这里也会报错
    @Override
    public void afterPropertiesSet() {
        System.out.println("afterPropertiesSet() 被调用，config = " + config);
        if (this.config == null) {
            throw new IllegalStateException("config 璯性未注入！");
        }
        // 做初始化检查
        validateConfig();
    }
}
```

```java
// XML / JavaConfig 配置的 init-method
@Bean(initMethod = "init", destroyMethod = "cleanup")
public MyService myService() {
    return new MyService();
}

// 对应 XML
<bean id="myService" class="com.example.MyService"
      init-method="init" destroy-method="cleanup" />
```

```java
// @PostConstruct（推荐，标准 JSR-250）
// 执行时机：在所有璯性注入完成后，init-method 之前
@Component
public class MyService {
    
    @Value("${my.config}")
    private String config;
    
    @PostConstruct
    public void init() {
        // 第三个执行（postProcessBeforeInitialization 之后）
        System.out.println("@PostConstruct 执行");
    }
    
    // 执行顺序：
    // 1. @PostConstruct 注解的方法
    // 2. InitializingBean.afterPropertiesSet()
    // 3. init-method
}
```

#### 5. DisposableBean + destroy-method

```java
@Component
public class MyService implements DisposableBean {
    
    private Connection connection;
    
    @PreDestroy
    public void releaseConnection() {
        // @PreDestroy，第二个执行
        if (connection != null) {
            try { connection.close(); } catch (Exception e) {}
        }
    }
    
    @Override
    public void destroy() {
        // DisposableBean.destroy()，第一个执行
        System.out.println("DisposableBean.destroy() 被调用");
        // 释放资源、关闭连接
    }
}

// destroy-method 配置
@Bean(destroyMethod = "cleanup")
public DataSource dataSource() {
    return new HikariDataSource(config);
}
```

**销毁顺序**：

```
1. @PreDestroy 注解的方法
2. DisposableBean.destroy()
3. destroy-method
```

## 🔥 扩展追问

### 追问 1：BeanPostProcessor 和 BeanFactoryPostProcessor 的区别？

| | BeanFactoryPostProcessor | BeanPostProcessor |
|---|---|---|
| **执行阶段** | 所有 BeanDefinition 加载后，**实例化前** | **每个 Bean 实例化后**，初始化前后 |
| **处理对象** | BeanDefinition（Bean 的"图纸"） | Bean 实例（Bean 的"成品"） |
| **调用次数** | 每个 FactoryPostProcessor 只调一次 | 每个 Bean 实例调 **两次** |
| **能做什么** | 修改 Bean 定义、动态注册 Bean | AOP 代理、璯性修改、代理替换 |
| **典型实现** | `ConfigurationClassPostProcessor` | `AutowiredAnnotationBeanPostProcessor` |

### 追问 2：Spring 是如何通过 BeanPostProcessor 实现 AOP 的？

```java
// AOP 自动代理创建器
// AnnotationAwareAspectJAutoProxyCreator extends BeanPostProcessor

@Override
public Object postProcessBeforeInitialization(Object bean, String beanName) {
    // 实际上是在这里创建 AOP 代理
    return wrapIfNecessary(bean, beanName);
}

Object wrapIfNecessary(Object bean, String beanName) {
    // 1. 检查是否应该被代理（是否有匹配的切面）
    List<Advisor> advisors = findEligibleAdvisors(bean.getClass(), beanName);
    
    if (!advisors.isEmpty()) {
        // 2. 创建 JDK 动态代理或 CGLIB 代理
        return createProxy(bean.getClass(), bean, advisors);
    }
    
    return bean;  // 无切面匹配，返回原始 Bean
}
```

**关键时机**：BeanPostProcessor 在所有初始化回调（InitializingBean、@PostConstruct）**之前**就创建了代理，所以代理 Bean 调用初始化方法是走代理对象的增强逻辑。

### 追问 4：循环依赖时，这些扩展点是如何被处理的？

```
假设 A 依赖 B，B 依赖 A（构造器循环依赖 → 无法解决）

璯性注入循环依赖（可以解决，三级缓存）：
A 的璯性需要注入 B
    ↓
A 还不完整，但先创建一个"早期暴露的 A"（半成品，放入三级缓存）
    ↓
用半成品 A 注入 B，开始创建 B
    ↓
B 的璯性需要注入 A
    ↓
从三级缓存找到半成品 A，注入 B
    ↓
B 创建完成
    ↓
把 B 注入 A，A 创建完成
```

**三级缓存的作用**：

```java
// 一级缓存（成品 Bean）
Map<String, Object> singletonObjects = new ConcurrentHashMap<>();

// 二级缓存（半成品 Bean，准备升级为成品）
Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>();

// 三级缓存（ObjectFactory，生成半成品 Bean）
Map<String, ObjectFactory<?>> singletonFactories = new ConcurrentHashMap<>();

// getSingleton() 逻辑：
// 1. 先查一级缓存 → 有则返回成品
// 2. 查二级缓存 → 有则返回半成品
// 3. 查三级缓存 → 有则创建半成品 → 放入二级缓存 → 返回
// 4. 都找不到 → 创建新 Bean → 放入一级缓存
```

**循环依赖时，BeanPostProcessor 不生效的场景**：

```java
// 如果 Bean 需要通过 BeanPostProcessor.postProcessBeforeInitialization()
// 做璯性强替换（如 @Autowired），而 Bean 又在循环依赖中
// → Spring 无法处理，抛出异常：Requested bean is currently in creation

// 解决方案：
// 1. 用 @Lazy 延迟加载璯性
@Autowired @Lazy
private AService aService;

// 2. 用 Setter 注入代替构造器注入（璯性注入可以解决循环，构造器注入不能）
```

### 追问 5：Spring 源码中，Bean 生命周期的核心方法调用链

```java
// AbstractAutowireCapableBeanFactory.doCreateBean() 源码逻辑

// 1. 创建实例（通过构造函数或工厂方法）
BeanWrapper instanceWrapper = createBeanInstance(beanName, mbd, args);

// 2. 早期暴露（解决循环依赖）
addSingletonFactory(beanName, () -> getEarlyBeanReference(beanName, mbd, bean));

// 3. 属性填充（璯性注入）
populateBean(beanName, mbd, instanceWrapper);

// 4. 初始化
Object exposedObject = initializeBean(beanName, exposedObject, mbd);

// initializeBean() 内部：
// ① invokeAwareMethods(beanName, bean)        ← Aware 回调
// ② applyBeanPostProcessorsBeforeInitialization ← BeanPostProcessor 前
// ③ invokeInitMethods(beanName, bean)         ← InitializingBean + init-method
// ④ applyBeanPostProcessorsAfterInitialization ← BeanPostProcessor 后
```

### 追问 7：@Bean 和 @Component 的生命周期有什么区别？

| | @Component + @PostConstruct/@PreDestroy | @Bean + initMethod/destroyMethod |
|---|---|---|
| **实例化** | Spring 自动扫描 + 实例化 | 手动在 @Bean 方法中 new |
| **初始化回调** | @PostConstruct / InitializingBean / initMethod | initMethod |
| **销毁回调** | @PreDestroy / DisposableBean / destroyMethod | destroyMethod |
| **生命周期** | Spring 全权管理 | Spring 管理，但实例由 @Bean 方法控制 |
| **使用场景** | 日常业务 Bean | 集成第三方库（无法加注解）|

```java
@Configuration
public class DataSourceConfig {
    
    @Bean(initMethod = "init", destroyMethod = "close")
    public DruidDataSource dataSource() {
        DruidDataSource ds = new DruidDataSource();
        ds.setDriverClassName("com.mysql.jdbc.Driver");
        ds.setUrl("jdbc:mysql://localhost:3306/test");
        return ds;
    }
    
    // 注意：@Bean 方法中的 new 操作不走 BeanPostProcessor
    // 即不会经过 AOP 代理（除非额外配置）
    // 如果需要对 @Bean 创建的对象做 AOP 增强，需要用 @EnableAspectJAutoProxy
}
```

## 一句话收尾

> **Spring Bean 生命周期的本质是"定义 → 实例化 → 注入 → 初始化 → 就绪 → 销毁"，扩展点按执行顺序是 BeanFactoryPostProcessor（改图纸）→ InstantiationAwareBeanPostProcessor（改成品前）→ Aware（感知容器）→ BeanPostProcessor（前后两刀裹上 AOP 代理）→ InitializingBean + @PostConstruct（初始化）→ DisposableBean + @PreDestroy（销毁），掌握了这条链路，Spring 源码阅读就轻松多了。**
