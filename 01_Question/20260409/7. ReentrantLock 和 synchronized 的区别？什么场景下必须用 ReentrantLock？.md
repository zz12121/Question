## 先说结论

**ReentrantLock 是 synchronized 的"功能增强版"，提供了更灵活的锁控制，但使用更复杂。**

```
核心区别：
synchronized = 自动挡，JVM 自动管理，简单但功能有限
ReentrantLock = 手动挡，程序员控制，功能强大但需要手动释放
```

---

## 🔥 核心概念

### 两者解决什么问题？

**相同点：** 都是可重入的互斥锁，保证线程安全

**不同点：** 控制粒度、功能丰富度、使用方式

```
synchronized(obj) {
    // 自动获取锁
    // 自动释放锁（即使抛异常）
}

lock.lock();
try {
    // 手动获取锁
    // 需要手动释放
} finally {
    lock.unlock();  // 必须在 finally 里释放！
}
```

---

## 🔥 详细对比

### 1. 实现层面

|特性|synchronized|ReentrantLock|
|---|---|---|
|实现位置|JVM 层（字节码指令）|JDK 层（Java 代码）|
|底层机制|Monitor + 锁升级|AQS（AbstractQueuedSynchronizer）|
|代码形式|关键字|类（需要手动 lock/unlock）|

**synchronized 字节码：**

```java
public synchronized void method();  // 方法加锁
// 字节码：ACC_SYNCHRONIZED 标志位

synchronized(obj) { }  // 代码块加锁
// 字节码：monitorenter + monitorexit
```

**ReentrantLock 实现：**

```java
public class ReentrantLock implements Lock {
    private final Sync sync;  // 继承 AQS
    
    public void lock() {
        sync.acquire(1);  // AQS 获取锁
    }
    
    public void unlock() {
        sync.release(1);  // AQS 释放锁
    }
}
```

---

### 2. 锁的获取方式

|特性|synchronized|ReentrantLock|
|---|---|---|
|阻塞获取|✅ 支持|✅ `lock()`|
|非阻塞尝试|❌ 不支持|✅ `tryLock()`|
|超时获取|❌ 不支持|✅ `tryLock(timeout, unit)`|
|中断响应|❌ 不支持|✅ `lockInterruptibly()`|

**ReentrantLock 特有功能：**

```java
ReentrantLock lock = new ReentrantLock();

// 1. 尝试获取锁，立即返回结果
if (lock.tryLock()) {
    try {
        // 执行业务
    } finally {
        lock.unlock();
    }
} else {
    // 获取失败，做其他事情
    log.info("获取锁失败，先干别的");
}

// 2. 带超时时间的获取
if (lock.tryLock(3, TimeUnit.SECONDS)) {
    try {
        // 执行业务
    } finally {
        lock.unlock();
    }
} else {
    throw new TimeoutException("获取锁超时");
}

// 3. 可中断的获取
try {
    lock.lockInterruptibly();  // 等待期间可以响应中断
    try {
        // 执行业务
    } finally {
        lock.unlock();
    }
} catch (InterruptedException e) {
    // 被中断了，优雅退出
    Thread.currentThread().interrupt();
}
```

---

### 3. 公平性

|特性|synchronized|ReentrantLock|
|---|---|---|
|公平锁|❌ 非公平（随机竞争）|✅ 可选公平/非公平|

**什么是公平锁？**

```
非公平锁：新来的线程可以和队列里的线程竞争（可能插队）
公平锁：严格按照请求顺序获取锁（FIFO队列）
```

**ReentrantLock 构造：**

```java
// 默认非公平（性能更好）
ReentrantLock lock = new ReentrantLock();

// 指定公平锁
ReentrantLock fairLock = new ReentrantLock(true);
```

**公平锁的代价：**

```java
// 非公平锁：吞吐量大，但可能"饿死"某些线程
// 公平锁：吞吐量低，但每个线程都能轮到
```

---

### 4. 条件变量（Condition）

|特性|synchronized|ReentrantLock|
|---|---|---|
|条件等待|只有一个 `wait/notify`|多个 `Condition`|

**synchronized 的问题：**

```java
synchronized(obj) {
    while (条件A不满足) {
        obj.wait();  // 所有线程都在这里等
    }
    while (条件B不满足) {
        obj.wait();  // 也是在这里等
    }
}

// 唤醒时：obj.notifyAll() 会唤醒所有线程
// 但很多线程被唤醒后发现条件不满足，又继续等待
// 效率低！
```

**ReentrantLock 的解决方案：**

```java
ReentrantLock lock = new ReentrantLock();
Condition conditionA = lock.newCondition();  // 条件A的队列
Condition conditionB = lock.newCondition();  // 条件B的队列

// 线程等待条件A
lock.lock();
try {
    while (条件A不满足) {
        conditionA.await();  // 只在条件A的队列里等
    }
} finally {
    lock.unlock();
}

// 另一个线程唤醒条件A
lock.lock();
try {
    修改条件A;
    conditionA.signal();  // 只唤醒等条件A的线程
} finally {
    lock.unlock();
}
```

**经典应用：生产者-消费者（多条件）**

```java
class BoundedBuffer {
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull = lock.newCondition();   // 队列不满
    private final Condition notEmpty = lock.newCondition();  // 队列不空
    
    public void put(Object x) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length) {
                notFull.await();  // 队列满了，等"不满"信号
            }
            items[putptr] = x;
            if (++putptr == items.length) putptr = 0;
            ++count;
            notEmpty.signal();  // 通知消费者：有数据了
        } finally {
            lock.unlock();
        }
    }
    
    public Object take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) {
                notEmpty.await();  // 队列空了，等"不空"信号
            }
            Object x = items[takeptr];
            if (++takeptr == items.length) takeptr = 0;
            --count;
            notFull.signal();  // 通知生产者：有空间了
            return x;
        } finally {
            lock.unlock();
        }
    }
}
```

### 5. 可重入性

**两者都支持可重入：**

```java
// synchronized
public synchronized void methodA() {
    methodB();  // 同一线程可以再次获取锁
}

public synchronized void methodB() {
    // 可以进入，锁计数 +1
}

// ReentrantLock 同理
public void methodA() {
    lock.lock();
    try {
        methodB();
    } finally {
        lock.unlock();  // 计数 -1，不为0不真正释放
    }
}

public void methodB() {
    lock.lock();
    try {
        // 可以进入
    } finally {
        lock.unlock();
    }
}
```

### 6. 性能对比

|场景|synchronized|ReentrantLock|
|---|---|---|
|无竞争|更快（偏向锁零开销）|稍慢（需要创建对象）|
|低竞争|快（轻量级锁 CAS）|快|
|高竞争|重量级锁，性能下降|AQS 优化，性能更稳定|
|Java 6 之前|慢|快|
|Java 6 之后|优化后接近|接近|

**结论：** Java 6 之后两者性能差距不大，选哪个看功能需求

## 🔥 扩展追问 2：AQS 是什么？ReentrantLock 怎么实现的？

**AQS = AbstractQueuedSynchronizer，JUC 包的核心框架**

```java
public abstract class AbstractQueuedSynchronizer 
    extends AbstractOwnableSynchronizer {
    
    // 核心状态（volatile，保证可见性）
    private volatile int state;
    
    // FIFO 等待队列（CLH 变体）
    private transient volatile Node head;
    private transient volatile Node tail;
}
```

**ReentrantLock 的 AQS 实现：**

```java
abstract static class Sync extends AbstractQueuedSynchronizer {
    
    // 获取锁
    final boolean nonfairTryAcquire(int acquires) {
        final Thread current = Thread.currentThread();
        int c = getState();
        
        if (c == 0) {  // 锁空闲
            if (compareAndSetState(0, acquires)) {
                setExclusiveOwnerThread(current);
                return true;
            }
        } else if (current == getExclusiveOwnerThread()) {  // 重入
            int nextc = c + acquires;
            setState(nextc);
            return true;
        }
        return false;
    }
    
    // 释放锁
    protected final boolean tryRelease(int releases) {
        int c = getState() - releases;
        if (Thread.currentThread() != getExclusiveOwnerThread())
            throw new IllegalMonitorStateException();
            
        boolean free = false;
        if (c == 0) {  // 完全释放
            free = true;
            setExclusiveOwnerThread(null);
        }
        setState(c);
        return free;
    }
}
```

**获取锁流程：**

```
lock.lock()
    ↓
tryAcquire(1)  // 尝试获取
    ↓
成功？→ 执行业务
    ↓ 失败
addWaiter(Node.EXCLUSIVE)  // 加入等待队列
    ↓
acquireQueued()  // 在队列里自旋/阻塞等待
    ↓
被唤醒后再次 tryAcquire
    ↓
成功 → 执行业务
```


## 🔥 扩展追问 4：读写锁（ReadWriteLock）

**ReentrantLock 的兄弟：ReentrantReadWriteLock**

```java
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
Lock readLock = rwLock.readLock();    // 读锁
Lock writeLock = rwLock.writeLock();  // 写锁

readLock.lock();
```

**特性：**

- 读读不互斥
- 读写互斥
- 写写互斥

**synchronized 做不到读写分离！**

## 🔥 扩展追问 5：StampedLock 是什么？

**JDK 8 引入的乐观读锁，比 ReadWriteLock 更快**

```java
StampedLock lock = new StampedLock();

// 乐观读（无锁）
long stamp = lock.tryOptimisticRead();
// 读取数据
if (!lock.validate(stamp)) {  // 检查期间是否有写操作
    // 有冲突，升级为悲观读锁
    stamp = lock.readLock();
    try {
        // 重新读取
    } finally {
        lock.unlockRead(stamp);
    }
}

// 写锁
long stamp = lock.writeLock();
try {
    // 修改数据
} finally {
    lock.unlockWrite(stamp);
}
```

**适用场景：** 读多写少，且读操作可以重复执行（幂等）
