## 基础流程

```
put(key, value)
    ↓
① 计算 hash（key.hashCode() ^ (key.hashCode() >>> 16)）
    ↓
② 判断数组为空 → resize() 初始化（容量16）
    ↓
③ 计算下标：(n-1) & hash  （n 是数组长度）
    ↓
④ 桶为空 → 直接插入
    ↓
⑤ 桶有元素：
   ├─ key 相同 → 覆盖
   └─ key 不同 → 遍历
       - 链表：尾插，>8 且容量>=64 → 树化
       - 红黑树：查找插入
    ↓
⑥ size++ > threshold → resize 扩容
```

## 🔥 扩展追问 1：为什么要用高低位异或算 hash？

```java
// JDK 8 的 hash 计算
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

**原因：**

- `n-1` 只有低位是 1（比如 15 的二进制是 `...00001111`）
- 如果直接用 `hashCode()`，高位信息就浪费了
- 高低位异或后，hash 值的高位信息能参与到下标计算中
- **效果：减少 hash 碰撞，让数据分布更均匀**

## 🔥 扩展追问 2：为什么数组长度必须是 2 的幂次？


```java
// 计算下标
index = (n - 1) & hash

// n = 16 时，n-1 = 15 = 0b00001111
// 任何 hash & 0b00001111，结果都在 0~15 之间
```

**优点：**

1. **位运算比取模快**：`&` 比 `%` 快很多
2. **均匀分布**：只有低位参与计算，配合上面说的高低位异用

**如果是奇数呢？**

```java
// n = 15, n-1 = 14 = 0b00001110
// hash & 0b00001110 结果永远是偶数，奇数下标永远访问不到！
// 分布不均匀，浪费一半空间
```

## 🔥 扩展追问 3：负载因子 0.75 是怎么来的？

```
默认容量：16
负载因子：0.75
扩容阈值：16 × 0.75 = 12
```

**原因：**

- **太小**（如 0.5）：频繁扩容，浪费空间
- **太大**（如 1.0）：碰撞增多，链表/红黑树变长，查询变慢
- **0.75 是时间和空间的平衡点**，经过大量测试得出的最优值
- 源码注释里也写了：_Generally, the default 0.75 offers a good tradeoff between time and space costs_

## 🔥 扩展追问 4：JDK 7 头插法为什么会死循环？

**JDK 7 扩容代码（简化版）：**

```java
void transfer(Entry[] newTable) {
    for (Entry<K,V> e : table) {
        while(null != e) {
            Entry<K,V> next = e.next;  // 记下下一个
            // 头插法！
            e.next = newTable[i];
            newTable[i] = e;
            e = next;
        }
    }
}
```

**并发场景演示：**

```
线程 A 和线程 B 同时 put，触发扩容

初始状态：
链表：A → B → null

线程 A 执行到一半被挂起，此时：
  newTable[下标] = A
  A.next = B
  B.next = null
  e = null

线程 B 正常执行完：
  newTable[下标] = B
  B.next = A
  A.next = null

线程 A 恢复，继续执行：
  e = null（之前保存的）
  不进入循环，直接结束
  
最终结果：newTable[下标] = B → A → null
看起来正常？继续看...
```

**真正的死循环场景：**

```
链表：A → B → C → null

线程 A 执行 transfer，移到 newTable[下标]
  e = A, next = B
  A.next = newTable[下标] = null
  newTable[下标] = A
  e = B

这时线程 A 被挂起，线程 B 也进入 transfer：
  B 也从头插法走一遍
  最终 newTable[下标] = C → B → A → null

线程 A 恢复：
  e = B（还记得吧）
  next = C
  B.next = newTable[下标] = C（指向了线程 B 放进去的 C）
  newTable[下标] = B
  e = C

  C.next = A（因为线程 B 把 A 放后面了）
  C 形成了环！
  下次 get(key) 遍历到这个环 → 死循环！
```

**JDK 8 用尾插法：**

- 每个元素只移动一次
- 不改变原链表的顺序
- 永远不会形成环

## 🔥 扩展追问 5：为什么树化阈值是 8，红黑树有什么特点？

**红黑树特点（4个约束）：**

```
1. 节点非红即黑
2. 根节点是黑色
3. 红节点的子节点必须是黑色（不能有连续红节点）
4. 任意节点到叶子节点的路径黑高相同
```

**为什么是 8？**

- 链表长度 8 的概率极低（< 0.00000006）
- 源码注释说：根据泊松分布，8 是合理的临界值
- 太短（< 8）树化浪费，因为红黑树节点比链表节点大（需要存颜色、父节点等）

**为什么去树化阈值是 6？**

- 设置为 7 的话，8 ↔ 7 来回切换会频繁
- 6 和 8 之间留个缓冲带（7），避免抖动

## 🔥 扩展追问 6：HashMap 线程不安全，有什么替代方案？

|方案|线程安全|并发性能|适用场景|
|---|---|---|---|
|`HashMap`|❌|-|单线程|
|`Collections.synchronizedMap()`|✅|差（全局锁）|低并发|
|`Hashtable`|✅|差（全局锁）|不推荐|
|`ConcurrentHashMap`|✅|好（分段锁/乐观锁）|高并发首选|

**ConcurrentHashMap 进化：**

- **JDK 7**：Segment 分段锁（16 个 Segment，每个管一段）
- **JDK 8**：去掉 Segment，改用 `synchronized` + CAS + 红黑树（类似 HashMap 结构）

## 🔥 扩展追问 7：ConcurrentHashMap JDK 8 是怎么保证线程安全的？

```java
// JDK 8 的实现核心
1. 用 volatile 修饰数组（可见性）
2. put 时：
   - 计算位置
   - 如果该位置为空 → CAS 入
   - 如果不为空 → synchronized 锁住该位置的链表/红黑树
3. 不锁整个表，只锁当前桶，性能高
```

**对比 JDK 7：**

- JDK 7 锁的是 Segment，最多 16 并发
- JDK 8 锁的是单个桶，理论上支持更多并发

## 🔥 扩展追问 8：HashMap 和 Hashtable 的区别？

| 方面         | HashMap        | Hashtable           |     |
| ---------- | -------------- | ------------------- | --- |
| 线程安全       | ❌              | ✅（粗暴加 synchronized） |     |
| null key   | ✅ 支持1个         | ❌ 不支持               |     |
| null value | ✅ 支持多个         | ❌ 不支持               |     |
| 初始容量       | 16             | 11                  |     |
| 扩容方式       | 2倍             | 2倍+1                |     |
| 迭代器        | 快速失败（Iterator） | 安全失败（Enumeration）   |     |
| 速度         | 快              | 慢（锁太重）              |     |

## 一句话总结（面试收尾用）

> "HashMap 通过数组 + 链表/红黑树实现，put 时计算 hash 找位置，碰撞了链式解决，超过 8 个节点就树化。JDK 8 主要优化了扩容策略（不用重新算 hash）和引入了红黑树（查询从 O(n) 变 O(logn)），同时用尾插法避免了 JDK 7 并发下死循环的问题。"
