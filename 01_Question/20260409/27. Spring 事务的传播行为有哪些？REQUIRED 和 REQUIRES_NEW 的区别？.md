## 先说结论

| 传播行为 | 当前有事务？ | 行为 |
|---------|------------|------|
| **REQUIRED**（默认） | ✅ 有 | 加入当前事务 |
| **REQUIRED**（默认） | ❌ 无 | 创建新事务 |
| **REQUIRES_NEW** | ✅ 有 | **挂起当前事务，创建全新事务** |
| **REQUIRES_NEW** | ❌ 无 | 创建新事务 |
| **SUPPORTS** | ✅ 有 | 加入当前事务 |
| **SUPPORTS** | ❌ 无 | 无事务执行 |
| **MANDATORY** | ✅ 有 | 加入当前事务 |
| **MANDATORY** | ❌ 无 | **抛异常** |
| **NOT_SUPPORTED** | ✅ 有 | **挂起当前事务，无事务执行** |
| **NOT_SUPPORTED** | ❌ 无 | 无事务执行 |
| **NEVER** | ✅ 有 | **抛异常** |
| **NEVER** | ❌ 无 | 无事务执行 |
| **NESTED** | ✅ 有 | **嵌套事务（Savepoint）** |
| **NESTED** | ❌ 无 | 创建新事务 |

---

## REQUIRED vs REQUIRES_NEW 核心区别

```
【REQUIRED】
a() ──→ b()
事务A ←──┘
A 调用 B，B 复用 A 的事务
A 和 B 在同一个事务里
→ A 回滚，B 也回滚
→ B 回滚，A 也回滚（因为同一事务）


【REQUIRES_NEW】
a() ──→ b()
事务A ← ┼ ──→ 事务B（挂起A，创建B）
A 和 B 是两个独立事务
→ A 回滚，B 不回滚（已提交）
→ B 回滚，A 不回滚（A 不知道自己被挂起过）
```

---

## 七种传播行为逐个详解

### 1. REQUIRED（默认值，最常用）

```java
@Service
public class OrderService {
    
    @Transactional(propagation = Propagation.REQUIRED)  // 默认，可省略
    public void createOrder(Order order) {
        // 在事务中执行
        orderMapper.insert(order);          // ← 同一事务
        inventoryService.decreaseStock(order.getSkuId());  // ← 同一事务
        paymentService.processPayment(order); // ← 同一事务
    }
}

@Service
public class InventoryService {
    @Transactional(propagation = Propagation.REQUIRED)
    public void decreaseStock(String skuId) {
        // 没有外部事务 → 创建新事务
        // 有外部事务 → 加入外部事务
    }
}
```

**场景**：业务操作需要强一致性，多个 Service 共享同一个事务，一个失败全部回滚。

---

### 2. REQUIRES_NEW（最常用的事务隔离手段）

```java
@Service
public class OrderService {
    
    @Transactional(propagation = Propagation.REQUIRED)
    public void createOrder(Order order) {
        orderMapper.insert(order);  // 订单在事务A中
        
        try {
            // 日志无论如何不能影响主事务
            // REQUIRES_NEW 创建事务B，完全独立
            logService.log("创建订单: " + order.getId());  
        } catch (Exception e) {
            // 日志失败不影响订单创建
            log.warn("日志记录失败", e);
        }
    }
}

@Service
public class LogService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void log(String message) {
        // 独立事务B，提交/回滚与 OrderService 完全无关
        logMapper.insert(message);
    }
}
```

**关键区别图示**：

```
REQUIRED（有问题）：
┌─ OrderService 事务A ─────────────────────────────┐
│  INSERT order                                      │
│  ┌─ LogService (REQUIRED，加入A) ─────────────┐  │
│  │  INSERT log  ← 失败                         │  │
│  └─────────────────────────────────────────────┘  │
│  ← 整个事务A回滚，订单也没了 ❌                    │
└────────────────────────────────────────────────────┘

REQUIRES_NEW（正确）：
┌─ OrderService 事务A ─────────────────┐
│  INSERT order  ← 成功 ✅              │
└──────────────────────────────────────┘
      ↓ 挂起事务A
┌─ LogService 事务B（独立） ──────────┐
│  INSERT log  ← 失败                 │
│  ← 只回滚事务B，不影响A ✅          │
└────────────────────────────────────┘
```

**常见使用场景**：

```java
// 场景1：日志/审计（必须独立）
@Transactional(REQUIRES_NEW)
public void saveAuditLog(User user, String action) {}

// 场景2：发送消息（MQ 发送失败不影响主业务）
@Transactional(REQUIRES_NEW)
public void sendMessage(OrderMessage msg) {}

// 场景3：外部系统调用（需要独立事务，不受主事务影响）
@Transactional(REQUIRES_NEW)
public void callRemoteService(RemoteRequest req) {}
```

---

### 3. NESTED（嵌套事务，Savepoint 机制）

```java
@Service
public class OrderService {
    
    @Transactional
    public void createOrder(Order order) {
        orderMapper.insert(order);  // 主事务
        
        try {
            // NESTED：同一连接，利用 Savepoint
            // 失败时只回滚到这里，内部子操作回滚
            itemService.processItems(order.getItems());  
        } catch (Exception e) {
            // 异常被捕获，主事务继续执行
            // 不会导致整个 createOrder 回滚
        }
        
        notificationService.notify(order);  // 主事务继续
    }
}

@Service
public class ItemService {
    @Transactional(propagation = Propagation.NESTED)
    public void processItems(List<Item> items) {
        // 与主事务同一数据库连接
        // 使用 Savepoint 实现部分回滚
        itemMapper.batchInsert(items);
    }
}
```

**NESTED vs REQUIRES_NEW 对比**：

| | NESTED | REQUIRES_NEW |
|---|---|---|
| 事务数 | 1 个事务 + Savepoint | 2 个独立事务 |
| 数据库连接 | 共用 1 个 | 独立 2 个 |
| 性能 | 好（无额外连接开销）| 差（需要 2 个连接）|
| 提交 | 一起提交 | 分别提交 |
| 回滚 | 可部分回滚（Savepoint）| 各自独立回滚 |
| 外层回滚 | 整个嵌套部分全部回滚 | 内层独立，外层回滚不影响内层 |

> **面试加分**：NESTED 依赖 JDBC 的 Savepoint，如果底层数据库不支持（如 MySQL MyISAM）会退化为 REQUIRED。

---

### 4. SUPPORTS（从属，不强制要求）

```java
@Service
public class ReportService {
    
    @Transactional(propagation = Propagation.SUPPORTS)
    public List<Order> getDailyReport(Date date) {
        // 有外部事务 → 加入（可以读到未提交的脏数据？不，MySQL 默认隔离级别是 RC）
        // 无外部事务 → 无事务执行（纯查询，不需要事务）
        return orderMapper.selectByDate(date);
    }
}

// 通常用于纯查询方法：
// - 有事务时：参与查询，可以读到同一事务内自己改的数据
// - 无事务时：作为普通查询，提升性能（不加事务锁）
```

---

### 5. MANDATORY（强制要求有事务）

```java
@Service
public class PaymentService {
    
    @Transactional(propagation = Propagation.MANDATORY)
    public void doPayment(Order order) {
        // 必须在有事务的情况下调用
        // 如果被没有 @Transactional 的方法直接调用 → 抛异常
        paymentGateway.call(order);
        orderMapper.updateStatus(order.getId(), "PAID");
    }
}

// 正确用法：
@Service
public class OrderService {
    @Transactional  // ← 开启事务
    public void placeOrder(Order order) {
        orderMapper.insert(order);
        paymentService.doPayment(order);  // ✅ 有事务，OK
    }
}

// 错误用法：
public class SomeHelper {
    public void helper() {
        paymentService.doPayment(order);  // ❌ 无事务，抛异常
    }
}
```

> **场景**：确保某个核心方法必须在外层事务中被调用，防止误用。

### 6. NOT_SUPPORTED（挂起当前事务）

```java
@Service
public class FileService {
    
    @Transactional(propagation = Propagation.NOT_SUPPORTED)
    public void uploadFile(File file) {
        // 大文件上传很慢，如果占用事务连接会长时间锁住
        // NOT_SUPPORTED 挂起主事务，避免长时间占用数据库连接
        storageService.uploadToOSS(file);
    }
}

@Service
public class OrderService {
    @Transactional
    public void createOrder(Order order) {
        orderMapper.insert(order);
        try {
            fileService.uploadFile(order.getFile());  // 上传文件
        } catch (Exception e) {
            log.warn("文件上传失败，继续处理订单");
        }
    }
}
```

@Service
public class LegacyService {

### 追问 1：事务传播行为的配置位置（谁决定？）

```java
// 传播行为由"被调用的方法"决定，不是调用者决定

// OrderService 调用 PaymentService
// 传播行为看 PaymentService 的 @Transactional 配置，不是 OrderService 的

OrderService (@Transactional REQUIRED) ──调用──→ PaymentService (@Transactional REQUIRES_NEW)
                                              ↑
                                    PaymentService 说："我要独立事务"
                                              ↓
                                    事务A挂起，事务B创建
                                    两个方法的事务互不影响
```

**所以每个 Service 方法都需要标注自己的传播行为**，而不是调用者决定。

### 追问 2：REQUIRES_NEW 的性能陷阱

```java
// ❌ 过度使用 REQUIRES_NEW 的问题
@Transactional
public void batchImport(List<Order> orders) {
    for (Order order : orders) {
        // 每条订单一个独立事务
        orderService.createOrderWithNewTx(order);  // REQUIRES_NEW
    }
}
// 1000 条订单 = 1000 个独立事务 = 1000 次数据库连接获取/释放
// 性能灾难！

// ✅ 正确做法
@Transactional
public void batchImport(List<Order> orders) {
    for (Order order : orders) {
        orderService.createOrderInCurrentTx(order);  // REQUIRED，同一事务
    }
}
```

> **经验法则**：REQUIRES_NEW 不要在循环中使用，循环场景统一用 REQUIRED 在一个大事务里处理。

### 追问 3：Spring 事务失效的常见场景

```java
// ❌ 场景1：private 方法，Spring 事务基于 AOP，private 方法无法代理
@Transactional
private void save(Order order) {  // ❌ 失效！AOP 代理无法拦截 private 方法
    orderMapper.insert(order);
}

// ✅ 正确：调用同类中的另一个 public 方法
@Transactional
public void savePublic(Order order) {
    doSave(order);  // 走代理对象调用
}

public void doSave(Order order) {  // ← 外部调用走这里，事务生效
    orderMapper.insert(order);
}

// ❌ 场景2：同类内部方法调用（this 调用）
@Service
public class OrderService {
    public void outer() {
        this.inner();  // ❌ this 不是代理对象，事务不生效
    }
    
    @Transactional
    public void inner() {  // ← 不会被代理
    }
}

// ✅ 正确：注入自己（代理注入）
@Service
public class OrderService {
    @Autowired
    private OrderService self;  // 自己注入自己
    
    public void outer() {
        self.inner();  // ✅ 走代理，事务生效
    }
    
    @Transactional
    public void inner() {
    }
}

// ❌ 场景3：异常被 catch 吞掉
@Transactional
public void save(Order order) {
    try {
        doSave(order);
    } catch (Exception e) {
        log.warn("捕获异常", e);
        // ❌ Spring 事务默认只有 RuntimeException/UncheckedException 才回滚
        // checked exception 被 catch 后不抛出 = 不回滚
    }
}

// ✅ 正确
@Transactional(rollbackFor = Exception.class)  // 显式指定所有异常都回滚
public void save(Order order) throws Exception {
    doSave(order);
}

// ❌ 场景4：多数据源，@Transactional 只管主数据源
// 如果 mapper 用的是其他 DataSource，@Transactional 不生效
// 需要用 JTA 或分布式事务（Seata）

// ❌ 场景5：非 Spring 管理的 Bean
// new出来的对象，@Transactional 不生效
OrderService service = new OrderService();  // ← Spring 不会管理这个实例
```

### 追问 4：REQUIRED 和 REQUIRES_NEW 的事务挂起原理

```
事务挂起（suspend）过程：

OrderService 事务A 正在执行中
    ↓
调用 LogService（REQUIRES_NEW）
    ↓
Spring 做的事：
    1. 将当前事务状态保存到 SuspendedResourcesHolder
       → 保存 Connection
       → 保存 Transaction（包含隔离级别、传播行为等）
    2. 释放当前数据库连接（还给连接池）
    3. 创建新 Connection，开启新事务B
    4. 执行 LogService
    ↓
事务B完成（提交或回滚）
    ↓
Spring 恢复事务A：
    → 从 SuspendedResourcesHolder 恢复 Connection
    → 继续使用原来的事务A执行
```

### 追问 5：Spring 事务的隔离级别

```java
@Transactional(isolation = Isolation.READ_COMMITTED)
public void transfer(String from, String to, int amount) {
    // 隔离级别和传播行为是两个独立配置
    // 传播行为：多个方法之间怎么组合事务
    // 隔离级别：事务内部，SQL 之间的可见性
    
    // READ_COMMITTED：
    // → 只能读到已提交的数据
    // → 解决了脏读问题
    // → 但不可重复读、幻读仍可能发生
    
    // 常用隔离级别：READ_COMMITTED（MySQL 默认）
    // 金融场景：REPEATABLE_READ（解决不可重复读）
}

Isolation 枚举值：
  DEFAULT        ← 使用数据库默认（MySQL 默认 RC）
  READ_UNCOMMITTED ← 最低，任何脏读都可能发生
  READ_COMMITTED  ← 只能读到已提交数据 ✅
  REPEATABLE_READ ← 同一事务内多次读一致 ✅
  SERIALIZABLE    ← 完全串行，最严格，性能最差
```

### 追问 6：Spring 事务 + 多线程问题

```java
@Transactional
public void createOrder(Order order) {
    orderMapper.insert(order);
    
    // ❌ 线程池任务中的事务是独立的
    CompletableFuture.runAsync(() -> {
        @Transactional
        notificationService.notify(order);  // ← 新线程，新事务
        // 不受外层 createOrder 事务管理！
    });
    
    // ✅ 正确做法
    // 如果 notify 需要在主事务中执行 → 直接调用，不用线程池
    // 如果需要异步但又要保证一致性 → 用消息队列（最终一致性）
    // 如果需要在子线程中使用主事务 → TransactionTemplate + TransactionSynchronizationManager
    CompletableFuture.runAsync(() -> {
        TransactionSynchronizationManager.registerSynchronization(
            new TransactionSynchronization() {
                @Override
                public void afterCommit() {
                    notificationService.notify(order);  // 主事务提交后才执行
                }
            }
        );
    });
}
```

### 追问 7：@Transactional 和 TransactionTemplate 的区别？

```java
// 方式1：注解方式（@Transactional，推荐）
@Transactional
public void save(Order order) {
    orderMapper.insert(order);
}

// 方式2：编程式（TransactionTemplate）
@Service
public class OrderService {
    
    @Autowired
    private TransactionTemplate transactionTemplate;
    
    public void save(Order order) {
        transactionTemplate.executeWithoutResult(status -> {
            orderMapper.insert(order);
            
            // 可以编程式回滚
            if (order.getAmount() < 0) {
                status.setRollbackOnly();  // 强制回滚
            }
        });
        
        // 编程式事务控制更灵活：
        // - 可以有选择性地回滚（非异常也能回滚）
        // - 适合复杂的事务边界控制
        // - 适合动态决定是否需要事务的场景
    }
}

// 什么时候用 TransactionTemplate？
// - 需要在方法内部动态决定是否开启事务
// - 需要在同一方法内多次开关事务
// - 方法本身不由 Spring 管理（普通类）
```

## 一句话收尾

> **REQUIRED 是"有就用，没有就创建"，多个方法共用一个事务，一个回滚全部回滚；REQUIRES_NEW 是"无论有没有都创建新事务"，挂起外部事务创建独立事务，内层提交了就不受外层回滚影响；嵌套事务 NESTED 用 Savepoint 实现同一连接内的部分回滚，性能优于 REQUIRES_NEW 但能力弱于 REQUIRES_NEW——选对传播行为是分布式系统事务一致性的关键。**
