## 先说结论

**CMS = Concurrent Mark Sweep，并发标记清除收集器。**

```
目标：最短 Stop-The-World 停顿时间
代价：并发执行，占用 CPU；标记清除，产生内存碎片
适用：对响应时间敏感的应用（Web 服务器等）
```

---

## 🔥 核心执行流程（6个阶段）

```
Phase 1: Initial Mark      STW  ← 初始标记（极短暂停）
Phase 2: Concurrent Mark        ← 并发标记（业务线程并行跑）
Phase 3: Concurrent Preclean    ← 并发预清理
Phase 4: Remark            STW  ← 重新标记（短暂停顿）
Phase 5: Concurrent Sweep       ← 并发清除（业务线程并行跑）
Phase 6: Concurrent Reset       ← 并发重置

STW 只在 Phase 1 和 Phase 4，非常短！
```

---

### Phase 1：初始标记（Initial Mark）—— STW

```
Stop The World！但很短（毫秒级）

任务：只标记 GC Roots 直接关联的对象
    GC Roots → 一级直达对象 → 打标记
    
不递归！只标记第一层
所以暂停时间极短
```

```
GC Roots（线程栈变量、类静态变量、JNI 引用...）
    ↓
[对象 A] [对象 B] [对象 C]  ← 只标记这一层
 ↓           ↓
[对象 D] [对象 E]           ← 不标记（下一阶段再做）
```

---

### Phase 2：并发标记（Concurrent Mark）—— 与业务线程同步执行

```
恢复业务线程，GC 线程同步跑

任务：从 Phase 1 标记的对象出发，递归遍历所有可达对象
    沿着引用链往下走，标记所有存活对象

时间最长，但不影响业务（并发）

问题：业务线程同时在修改引用关系！
    ← 这会导致漏标和多标，Phase 4 要修正
```

---

### Phase 3：并发预清理（Concurrent Preclean）—— 并发

```
任务：扫描 Card Table，找出 Phase 2 期间引用关系变化的区域

Card Table：把堆分成很多小块（512字节/块）
每块对应一个 card
若某块有对象被修改，就把 card 标记为 dirty

预清理：处理 dirty card，减少 Phase 4 的工作量
目的：缩短 Phase 4 的 STW 时间
```

---

### Phase 4：重新标记（Remark）—— STW

```
Stop The World！比 Phase 1 稍长，但还是很短

任务：修正 Phase 2 并发期间引用变化导致的标记错误

采用三色标记 + 增量更新 算法

三色标记：
    白色：还没被标记（可能是垃圾）
    灰色：已被标记，但引用对象没全标记完
    黑色：已被标记，所有引用对象都标记完了

白色对象如果没有灰色/黑色对象引用 → 确认是垃圾
```

---

### Phase 5：并发清除（Concurrent Sweep）—— 并发

```
恢复业务线程，GC 线程同步跑

任务：回收所有标记为垃圾的对象，释放内存

用的是"标记-清除"算法（不是标记-整理！）
    → 不移动对象
    → 直接释放内存
    → 产生内存碎片（后面重点说）
```

---

### Phase 6：并发重置（Concurrent Reset）—— 并发

```
重置 CMS 内部数据结构，为下一次 GC 做准备
更新 Card Table、位图等

时间短，并发执行
```

## 🔥 扩展追问 1：CMS 有哪些缺点？

### 缺点一：浮动垃圾（Floating Garbage）

```java
// Phase 2 并发标记时
业务线程：创建对象 X，被 A 引用
GC 线程：标记完 A 了，不会再扫 X
// X 是"存活的"，但没被标记

业务线程：A 引用 Y，但 Y 不再用了，准备被回收
GC 线程：已经把 Y 标记为存活了
// Y 是"垃圾"，但被标记为存活 ← 这就是浮动垃圾

结果：Y 这次 GC 收不掉，等下一次 GC 再收
每次 GC 都有部分垃圾收不干净
```

### 缺点二：内存碎片

```
标记-清除算法不移动对象：

清除前：
[A:活] [B:垃圾] [C:活] [D:垃圾] [E:活]

清除后：
[A:活] [     ] [C:活] [     ] [E:活]
              ↑                ↑
          内存碎片          内存碎片

要分配一个大对象时：
    每个碎片都不够大 → 分配失败 → 触发 Full GC！
```

**解决方案：**

```
-XX:+UseCMSCompactAtFullCollection  # Full GC 时整理内存
-XX:CMSFullGCsBeforeCompaction=N    # 每 N 次 Full GC 执行一次整理
```

### 缺点三：并发模式失败（Concurrent Mode Failure）

```
CMS 并发清除时，业务线程还在产生垃圾
如果老年代内存不够用了怎么办？

场景：
    老年代空间：100%
    CMS 正在并发清除，清了 50%
    业务线程 Young GC 完，往老年代晋升大批对象
    老年代剩余 50% 放不下！

触发：并发模式失败 → 停止并发 → 触发 Serial Old GC！
Serial Old：全程 STW，单线程，特别慢！
```

**这是 CMS 最严重的问题！**

```
预防：
-XX:CMSInitiatingOccupancyFraction=68  
# 老年代占用 68% 就开始 CMS（默认 92%，太晚了）

提前开始，给并发清除留出余量
```

### 缺点四：CPU 资源消耗

```
并发标记和清除阶段，GC 线程和业务线程同时跑

默认 GC 线程数 = (CPU 核数 + 3) / 4
    → 4 核机器：1 个 GC 线程（25% CPU）
    → 8 核机器：2 个 GC 线程（25% CPU）

低于 4 核的机器影响很大！
```

## 🔥 扩展追问 2：三色标记算法详解

**CMS 并发标记的核心算法：**

```
白色：还没被扫描
灰色：自己已标记，但引用的对象还没全部标记
黑色：自己和所有引用对象都标记完了

黑色  → 确定存活，不会再处理
白色  → 扫描结束后确定是垃圾
灰色  → 待处理中间状态
```

**并发时可能出现的问题——对象消失：**

```
初始状态：
A(黑) → B(灰) → C(白)

并发标记进行中，业务线程操作：
1. B 删除对 C 的引用
2. A 增加对 C 的引用

现在：
A(黑) → C(白)   ← 黑色引用了白色
B(灰)            ← 不再引用 C

GC 线程：已经扫完 A 了，不会再扫 A
          B 没有引用 C，不会从 B 扫到 C
结果：C 被标记为垃圾，但实际上存活！← 对象消失问题

触发条件（需同时满足）：
1. 赋值器把一个白色对象的引用写入了黑色对象
2. 删除了从灰色对象出发，到达该白色对象的引用
```

**CMS 的解决方案——增量更新：**

```java
// 写屏障（Write Barrier）
// 当黑色对象 A 增加对白色对象 C 的引用时：
A.ref = C;
// 写屏障触发：把 A 重新标记为灰色
// Phase 4 Remark 时重新扫描 A
```

```
A(黑) → C(白)
写屏障触发 → A 变回灰色

Phase 4 重新扫描 A → 扫到 C → C 标记为存活

解决了对象消失问题！
```

**G1 的解决方案——原始快照（SATB）：**

```java
// 当删除 B 对 C 的引用时：
old_ref = B.ref;  // 保存删除前的引用
B.ref = null;
// 写屏障触发：把 old_ref（C）记录下来
// Phase 4 时重新扫描 C

不管怎么改，Phase 2 开始时存活的对象一定会被标记
（哪怕是浮动垃圾也没事，下次再收）
```

## 🔥 扩展追问 3：CMS 参数调优

```bash
# 开启 CMS
-XX:+UseConcMarkSweepGC

# 触发 CMS 的内存占用比例（默认 92%，建议调低）
-XX:CMSInitiatingOccupancyFraction=68

# 是否根据此比例触发（必须同时设置）
-XX:+UseCMSInitiatingOccupancyOnly

# Full GC 后是否整理内存碎片（JDK 8 默认开）
-XX:+UseCMSCompactAtFullCollection

# 多少次 Full GC 后整理一次（0 = 每次都整理）
-XX:CMSFullGCsBeforeCompaction=0

# CMS 并发线程数
-XX:ConcGCThreads=4

# 是否在 Minor GC 前检查老年代空间是否足够
-XX:+CMSScavengeBeforeRemark
```

| 特性 | CMS | G1 | ZGC |

| 堆大小 | 数 GB | 数 GB ~ 数十 GB | TB 级 |
| 吞吐量 | 好 | 好 | 略低 |

| JDK 14 | ❌ 被移除 | - | - |

## 🔥 扩展追问 5：CMS 为什么会在 JDK 14 被移除？

```
1. 内存碎片问题无法根本解决
2. 并发模式失败时 Full GC 代价太大
3. G1 能提供更好的停顿预测
4. ZGC 更适合现代大内存场景
5. Oracle 不想维护两套收集器代码
```

**JDK 9** CMS 被标记为 deprecated  
**JDK 14** CMS 被正式移除

## 🔥 扩展追问 6：什么时候 CMS 会退化为 Serial Old？

```
以下情况触发 Serial Old（全程 STW，极慢）：
1. 并发模式失败（Concurrent Mode Failure）
   老年代空间不足以支撑并发清除期间的对象晋升

2. 晋升失败（Promotion Failed）
   Minor GC 后，存活对象无法晋升到老年代

3. 内存碎片导致无法分配大对象
   找不到连续内存空间
   
4. 元空间/方法区满了
   触发 Full GC

退化后：
    GC 日志出现：concurrent mode failure
    应用明显卡顿（STW 可能达到秒级！）
```

---

## 一句话收尾

> "CMS 主要分 6 个阶段，只有初始标记和重新标记是 STW，其余都是并发执行的，所以停顿时间很短。但缺点也明显：浮动垃圾导致内存清不干净、标记清除产生内存碎片、并发模式失败时会退化成 Serial Old 造成长时间停顿。JDK 9 就被废弃了，JDK 14 直接移除，生产上现在推荐用 G1 或 ZGC。"
