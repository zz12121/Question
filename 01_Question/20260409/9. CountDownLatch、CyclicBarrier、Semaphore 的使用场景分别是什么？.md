## 先说结论

**三者都是 JUC 包下的同步工具类，但解决的问题完全不同：**

```
CountDownLatch = 倒计时门闩，一个线程等待多个线程完成
CyclicBarrier = 循环栅栏，多个线程互相等待，一起出发
Semaphore = 信号量，控制同时访问的线程数量
```

---

## 🔥 核心概念

### 三者对比一览

|特性|CountDownLatch|CyclicBarrier|Semaphore|
|---|---|---|---|
|核心功能|一个等N个|N个互相等|限流控制|
|计数器方向|递减到0|递增到N|获取/释放许可|
|可重用|❌ 一次性|✅ 可循环|✅ 可循环|
|典型场景|主线程等子线程|多线程分阶段执行|资源池限流|
|异常处理|无|可指定打破屏障的操作|无|

## 🔥 CountDownLatch（倒计时门闩）

### 解决什么问题？

**场景：一个线程需要等待多个线程完成后再继续**

```
老板（主线程）等所有员工（子线程）提交周报后才能下班
```

### 代码示例

```java
public class CountDownLatchDemo {
    // 3个员工
    private static final CountDownLatch latch = new CountDownLatch(3);
    
    public static void main(String[] args) throws InterruptedException {
        // 启动3个员工线程
        for (int i = 1; i <= 3; i++) {
            final int id = i;
            new Thread(() -> {
                try {
                    System.out.println("员工" + id + "开始写周报...");
                    Thread.sleep((long) (Math.random() * 3000));
                    System.out.println("员工" + id + "提交周报！");
                } catch (InterruptedException e) {
                    e.printStackTrace();
                } finally {
                    latch.countDown();  // 完成一个，计数-1
                }
            }).start();
        }
        
        System.out.println("老板等所有员工提交...");
        latch.await();  // 阻塞，直到计数为0
        System.out.println("所有人都提交了，老板下班！");
    }
}
```

**输出：**

```
老板等所有员工提交...
员工1开始写周报...
员工2开始写周报...
员工3开始写周报...
员工3提交周报！
员工1提交周报！
员工2提交周报！
所有人都提交了，老板下班！
```

### 实现原理

```java
public class CountDownLatch {
    private final Sync sync;
    
    // 基于 AQS 实现
    private static final class Sync extends AbstractQueuedSynchronizer {
        Sync(int count) {
            setState(count);  // AQS 的 state = 计数器
        }
        
        // await() 时调用
        protected int tryAcquireShared(int acquires) {
            return (getState() == 0) ? 1 : -1;  // state=0才能获取
        }
        
        // countDown() 时调用
        protected boolean tryReleaseShared(int releases) {
            for (;;) {
                int c = getState();
                if (c == 0) return false;
                int nextc = c - 1;
                if (compareAndSetState(c, nextc)) {
                    return nextc == 0;  // 减到0时唤醒等待线程
                }
            }
        }
    }
    
    public void await() throws InterruptedException {
        sync.acquireSharedInterruptibly(1);
    }
    
    public void countDown() {
        sync.releaseShared(1);
    }
}
```

### 典型应用场景

## 🔥 CyclicBarrier（循环栅栏）

### 解决什么问题？

**场景：多个线程互相等待，达到某个数量后一起继续**

```
旅游团：等所有人都到齐后，导游才带大家出发
而且每天都可以这样（循环使用）
```

### 代码示例

```java
public class CyclicBarrierDemo {
    // 3个人，到齐后执行一个动作
    private static final CyclicBarrier barrier = new CyclicBarrier(3, () -> {
        System.out.println("=== 所有人都到齐了，出发！===");
    });
    
    public static void main(String[] args) {
        for (int i = 1; i <= 3; i++) {
            final int id = i;
            new Thread(() -> {
                try {
                    System.out.println("游客" + id + "到达集合点");
                    barrier.await();  // 等待其他人
                    
                    System.out.println("游客" + id + "开始游玩");
                    Thread.sleep(1000);
                    
                    System.out.println("游客" + id + "到达第二个集合点");
                    barrier.await();  // 再次等待（循环使用！）
                    
                    System.out.println("游客" + id + "结束行程");
                } catch (Exception e) {
                    e.printStackTrace();
                }
            }).start();
        }
    }
}
```

**输出：**

```
游客1到达集合点
游客2到达集合点
游客3到达集合点
=== 所有人都到齐了，出发！===
游客1开始游玩
游客3开始游玩
游客2开始游玩
游客1到达第二个集合点
游客2到达第二个集合点
游客3到达第二个集合点
=== 所有人都到齐了，出发！===
游客1结束行程
游客2结束行程
游客3结束行程
```

### 实现原理

```java
public class CyclicBarrier {
    // 基于 ReentrantLock + Condition 实现
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition trip = lock.newCondition();
    
    private final int parties;      // 需要等待的线程数
    private int count;              // 当前还剩多少线程没到达
    private final Runnable barrierCommand;  // 到达后执行的动作
    
    public int await() throws InterruptedException, BrokenBarrierException {
        final ReentrantLock lock = this.lock;
        lock.lock();
        try {
            final Generation g = generation;
            
            if (g.broken) throw new BrokenBarrierException();
            if (Thread.interrupted()) {
                breakBarrier();
                throw new InterruptedException();
            }
            
            int index = --count;  // 到达一个，计数-1
            if (index == 0) {     // 所有人都到了
                boolean ranAction = false;
                try {
                    final Runnable command = barrierCommand;
                    if (command != null) command.run();  // 执行动作
                    ranAction = true;
                    nextGeneration();  // 重置，准备下一轮
                    return 0;
                } finally {
                    if (!ranAction) breakBarrier();
                }
            }
            
            // 还没齐，阻塞等待
            for (;;) {
                try {
                    trip.await();  // 在 Condition 上等待
                } catch (InterruptedException ie) {
                    // ...
                }
                if (g.broken) throw new BrokenBarrierException();
                if (g != generation) return index;  // 被唤醒，新一轮开始了
            }
        } finally {
            lock.unlock();
        }
    }
    
    private void nextGeneration() {
        trip.signalAll();  // 唤醒所有等待线程
        count = parties;   // 重置计数器
        generation = new Generation();
    }
}
```

## 🔥 Semaphore（信号量）

### 解决什么问题？

**场景：控制同时访问某资源的线程数量**

```
停车场只有10个车位，来了20辆车，只有10辆能进，其他等着
有车出来，才能进去
```

### 代码示例

```java
public class SemaphoreDemo {
    // 3个许可（同时只能有3个线程执行）
    private static final Semaphore semaphore = new Semaphore(3);
    
    public static void main(String[] args) {
        // 10个线程抢3个许可
        for (int i = 1; i <= 10; i++) {
            final int id = i;
            new Thread(() -> {
                try {
                    System.out.println("线程" + id + "尝试获取许可...");
                    semaphore.acquire();  // 获取许可，没有就阻塞
                    
                    System.out.println("线程" + id + "获取成功，开始执行");
                    Thread.sleep(2000);  // 模拟工作
                    
                    System.out.println("线程" + id + "执行完毕，释放许可");
                } catch (InterruptedException e) {
                    e.printStackTrace();
                } finally {
                    semaphore.release();  // 释放许可
                }
            }).start();
        }
    }
}
```

**输出：**

```
线程1尝试获取许可...
线程1获取成功，开始执行
线程2尝试获取许可...
线程2获取成功，开始执行
线程3尝试获取许可...
线程3获取成功，开始执行
线程4尝试获取许可...
线程5尝试获取许可...
...（4-10都在等待）
线程1执行完毕，释放许可
线程4获取成功，开始执行
...
```

### 实现原理

```java
public class Semaphore {
    private final Sync sync;
    
    // 基于 AQS 实现
    abstract static class Sync extends AbstractQueuedSynchronizer {
        Sync(int permits) {
            setState(permits);  // state = 剩余许可数
        }
        
        // 获取许可
        final int nonfairTryAcquireShared(int acquires) {
            for (;;) {
                int available = getState();
                int remaining = available - acquires;
                if (remaining < 0 ||
                    compareAndSetState(available, remaining)) {
                    return remaining;
                }
            }
        }
        
        // 释放许可
        protected final boolean tryReleaseShared(int releases) {
            for (;;) {
                int current = getState();
                int next = current + releases;
                if (compareAndSetState(current, next)) {
                    return true;
                }
            }
        }
    }
    
    public void acquire() throws InterruptedException {
        sync.acquireSharedInterruptibly(1);
    }
    
    public void release() {
        sync.releaseShared(1);
    }
}
```

## 🔥 扩展追问 1：三者的核心区别

```
CountDownLatch：
  主线程 ──wait──┐
                 ├── 子线程1 countDown()
                 ├── 子线程2 countDown()
                 └── 子线程3 countDown()
  单向：子线程不需要等待主线程

CyclicBarrier：
  线程1 ──await──┐
  线程2 ──await──┼── 所有人都到齐后继续
  线程3 ──await──┘
  双向：互相等待，且可循环使用

Semaphore：
  线程1 ──acquire──┐
  线程2 ──acquire──┼── 最多N个同时执行
  线程3 ──acquire──┤
  线程4 ──acquire──┘（阻塞，等释放）
  控制并发数，不是等待关系
```

## 🔥 扩展追问 2：公平模式 vs 非公平模式

```java
// Semaphore 和 CyclicBarrier 都支持公平模式

// 非公平（默认）：先来的不一定先获取
Semaphore unfair = new Semaphore(3);

// 公平：按请求顺序获取（FIFO）

```

**公平性的代价：** 吞吐量降低，因为需要维护队列顺序

## 一句话收尾

> "CountDownLatch 是一个线程等N个线程完成（一次性），CyclicBarrier 是N个线程互相等待一起出发（可循环），Semaphore 是控制同时执行的线程数量（限流）。三者都是基于 AQS 实现，但解决的场景不同：CountDownLatch 用于任务汇总，CyclicBarrier 用于分阶段协作，Semaphore 用于资源池限流。"
