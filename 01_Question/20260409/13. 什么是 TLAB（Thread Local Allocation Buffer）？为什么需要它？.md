## 先说结论

**TLAB = Thread Local Allocation Buffer，是 JVM 为每个线程在堆内存中预先分配的一小块私有内存区域，用于快速分配对象，避免多线程竞争堆内存分配时的锁开销。**

```
核心思想：
对象分配最频繁，如果每次都要竞争锁，性能太差
给每个线程一块"私房钱"，分配对象时先用私房钱，不够了再申请
```

---

## 🔥 核心概念

### 对象分配为什么需要优化？

**场景：高并发下大量创建对象**

```java
// 每个请求都创建很多对象
public void handleRequest() {
    User user = new User();      // 堆上分配
    Order order = new Order();   // 堆上分配
    List<Item> items = new ArrayList<>();  // 堆上分配
    // ...
}
```

**没有 TLAB 时的问题：**

```
多个线程同时 new 对象：
┌─────────┐     ┌─────────┐     ┌─────────┐
│ Thread1 │     │ Thread2 │     │ Thread3 │
└────┬────┘     └────┬────┘     └────┬────┘
     │               │               │
     └───────────────┼───────────────┘
                     ▼
              ┌─────────────┐
              │   堆内存     │  ← 共享区域，需要同步！
              │  (Eden区)   │
              └─────────────┘

问题：
1. 多个线程竞争 Eden 区的指针（bump-the-pointer）
2. 需要 CAS 或加锁保证线程安全
3. 高并发下成为性能瓶颈
```

---

## 🔥 TLAB 是什么？

### 基本原理

```
给每个线程在 Eden 区预先分配一小块私有内存：

┌─────────────────────────────────────────┐
│                Eden 区                 │
├──────────┬──────────┬──────────┬────────┤
│  TLAB    │   TLAB   │   TLAB   │  共享   │
│ Thread1  │ Thread2  │ Thread3  │ 区域   │
│  私有    │   私有   │   私有   │        │
└──────────┴──────────┴──────────┴────────┘

线程在自己的 TLAB 内分配对象：
- 不需要锁！
- 只需要移动指针（bump pointer）
- 和栈上分配一样快！
```

### 分配流程

```java
// 对象分配时的逻辑（简化）：
Object allocate(Class<?> clazz) {
    // 1. 尝试在 TLAB 分配
    if (tlab.top + size <= tlab.end) {
        Object obj = tlab.top;
        tlab.top += size;  // 指针移动，无锁！
        return obj;
    }
    
    // 2. TLAB 不够了，尝试在 Eden 分配（需要锁）
    return allocate_in_eden(clazz);
}

// Eden 分配（慢路径）
Object allocate_in_eden(Class<?> clazz) {
    lock(Eden);  // 需要同步！
    Object obj = eden.top;
    eden.top += size;
    unlock(Eden);
    return obj;
}
```

## 🔥 扩展追问 1：TLAB 的具体实现

### JVM 源码层面（HotSpot）

```cpp
// threadLocalAllocBuffer.hpp
class ThreadLocalAllocBuffer: public CHeapObj<mtThread> {
  HeapWord* _start;   // TLAB 起始地址
  HeapWord* _top;     // 当前分配指针（bump pointer）
  HeapWord* _end;     // TLAB 结束地址
  
  size_t    _desired_size;  // 期望大小
  size_t    _refill_waste_limit;  // 浪费阈值
};

// 分配对象
HeapWord* ThreadLocalAllocBuffer::allocate(size_t size) {
  invariants();
  HeapWord* obj = top();
  if (pointer_delta(end(), obj) >= size) {  // 空间够吗？
    set_top(obj + size);  // 移动指针，无锁！
    invariants();
    return obj;
  }
  return NULL;  // 空间不够
}
```

### 关键参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `-XX:+UseTLAB` | true | 启用 TLAB |
| `-XX:TLABSize` | 动态计算 | 每个 TLAB 的初始大小 |
| `-XX:MinTLABSize` | 2KB | 最小 TLAB 大小 |
| `-XX:MaxTLABSize` | 256KB | 最大 TLAB 大小（Eden的1%） |
| `-XX:TLABWasteTargetPercent` | 1% | TLAB 浪费率目标 |
| `-XX:+ResizeTLAB` | true | 自动调整 TLAB 大小 |

## 🔥 扩展追问 2：TLAB 满了怎么办？

### 慢路径分配

```
TLAB 不够分配当前对象时，有几种选择：
```

**方案 1：在 Eden 区直接分配（需要锁）**

```java
// 当前对象太大，TLAB 装不下
if (size > tlab.remaining()) {
    // 直接在 Eden 分配，需要竞争锁
    return allocate_from_eden(size);
}
```

**方案 2：废弃当前 TLAB，申请新的**

```java
// 计算浪费比例
waste = tlab.remaining();  // 剩余空间被浪费

if (waste < TLABWasteTargetPercent * tlab.size()) {
    // 浪费不多，申请新的 TLAB
    retire_current_tlab();
    create_new_tlab();
    return allocate_from_tlab(size);
} else {
    // 浪费太多，直接在 Eden 分配
    return allocate_from_eden(size);
}
```

**流程图：**

```
对象分配
    │
    ▼
TLAB 够吗？──否──→ 对象 > TLAB 最大大小？──是──→ 直接在 Eden 分配（慢）
    │                  │
    是                 否
    │                  ▼
    ▼            浪费 < 阈值？──是──→ 申请新 TLAB
移动指针分配              │
（快，无锁）              否
                      ▼
                  直接在 Eden 分配
```

## 🔥 扩展追问 3：TLAB 与 GC 的关系

### TLAB 对 GC 的影响

```
1. TLAB 在 Eden 区，属于新生代
2. TLAB 分配的对象和普通对象一样参与 GC
3. Minor GC 时会回收 Eden 区（包括所有 TLAB）
```

### GC 时的处理

```cpp
// 年轻代 GC 时
void YoungGC::collect() {
    // 1. 标记存活对象
    // 2. 复制存活对象到 Survivor 区
    // 3. 清空 Eden 区（所有 TLAB 一起被回收）
    
    // 4. 每个线程重新申请 TLAB
    for (each thread) {
        thread->tlab().clear();  // 清空旧 TLAB
        thread->tlab().initialize();  // 申请新 TLAB
    }
}
```

## 🔥 扩展追问 4：TLAB 的优缺点

### 优点

| 优点 | 说明 |
|------|------|
| 无锁分配 | 线程私有，不需要同步 |
| 速度快 | 只需移动指针，和 C 的 malloc 一样快 |
| 缓存友好 | 线程内连续分配，提高缓存命中率 |
| 减少伪共享 | 不同线程在不同区域分配 |

### 缺点

| 缺点 | 说明 |
|------|------|
| 空间浪费 | TLAB 剩余空间可能浪费（平均 1%） |
| 内存碎片 | 不同大小的对象导致 TLAB 内部碎片 |
| 大对象问题 | 大对象无法使用 TLAB，直接走慢路径 |

## 🔥 扩展追问 5：TLAB vs 栈上分配

| 特性 | TLAB | 栈上分配（逃逸分析） |
|------|------|---------------------|
| 位置 | Eden 区（堆） | 栈帧 |
| 对象生命周期 | 可能逃逸到堆 | 方法结束即销毁 |
| 适用对象 | 所有对象 | 未逃逸的小对象 |
| GC 参与 | 是 | 否（随栈帧弹出销毁） |
| 触发条件 | 默认开启 | 需要逃逸分析判定 |

**两者可以共存：**

```java
public void method() {
    // 1. 先尝试栈上分配（逃逸分析）
    // 如果对象不逃逸，直接在栈上分配
    
    // 2. 栈上分配失败，尝试 TLAB 分配
    // 在 Eden 的线程私有区域分配
    
    // 3. TLAB 失败，Eden 区分配（加锁）
    
    // 4. Eden 不够，触发 Minor GC
    
    // 5. 还是不够，分配到老年代
}
```

## 🔥 扩展追问 6：如何观察和调优 TLAB？

### 查看 TLAB 使用情况

```bash
# 打印 TLAB 统计信息
java -XX:+PrintTLAB MyApp

# 输出示例：
TLAB: gc thread: 0x00007f0b4c001800 [id: 1234] desired_size: 256KB slow allocs: 10  refills: 50 waste: 0.8%
TLAB: gc thread: 0x00007f0b4c002800 [id: 1235] desired_size: 256KB slow allocs: 2   refills: 48 waste: 0.5%

# 说明：
# desired_size: TLAB 大小
# slow allocs: 慢路径分配次数（TLAB 不够，走 Eden 分配）
# refills: 重新申请 TLAB 的次数
# waste: 空间浪费率
```

### 调优建议

```bash
# 如果 slow allocs 很多，说明 TLAB 太小
# 增大 TLAB 大小
-XX:TLABSize=512k

# 或者让 JVM 自动调整（默认开启）
-XX:+ResizeTLAB

# 如果内存紧张，可以减小 TLAB 浪费阈值
-XX:TLABWasteTargetPercent=0.5
```

## 🔥 扩展追问 7：TLAB 与指针碰撞

### 两种分配方式

| 方式 | 说明 | 适用场景 |
|------|------|----------|
| 指针碰撞（Bump the Pointer） | 内存规整，移动指针即可 | Serial、ParNew 等带整理的收集器 |
| 空闲列表（Free List） | 维护空闲内存块列表 | CMS、G1 等不带整理的收集器 |

**TLAB 使用指针碰撞：**

```
Eden 区是连续的，使用标记-复制算法，内存规整
TLAB 内部分配只需：obj = top; top += size;
```

## 一句话收尾

> "TLAB 是 JVM 为每个线程在 Eden 区预先分配的私有内存缓冲区，对象分配时优先在 TLAB 内通过移动指针完成，不需要加锁，极大提高了高并发场景下的对象分配性能。当 TLAB 空间不足时，会走慢路径在 Eden 区直接分配（需要同步）或申请新的 TLAB。TLAB 默认开启，大小动态调整，平均浪费约 1% 的空间换取无锁分配的性能优势。"
