## 先说结论

**OOM 排查核心思路：**

```
稳住 → 收集现场 → 分析根因 → 修复验证

不要上来就重启！
重启会丢失现场，下次还会再 OOM
```

---

## 🔥 第一步：稳住服务（优先级最高）

```bash
# 1. 先看服务是否还活着
ps aux | grep java

# 2. 看当前内存占用
top -p <PID>
free -m

# 3. 如果已经宕机，看是否有 hs_err_pid*.log
ls -la /tmp/hs_err_pid*.log
# 这个文件记录了 JVM Crash 时的完整快照

# 4. 如果服务还在，立刻触发 Heap Dump（不要等它自己挂）
jmap -dump:format=b,file=/tmp/heapdump.hprof <PID>

# 5. 没时间的话，先重启保住服务，
#    但要确保 JVM 参数里配了自动 dump！
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/tmp/heapdump.hprof
```

---

## 🔥 第二步：确认 OOM 类型

```
OOM 不是一种错误，是一类错误！
不同的 OOM 原因完全不同：
```

| OOM 类型 | 错误信息 | 原因 |
|----------|---------|------|
| **堆内存溢出** | `Java heap space` | 对象太多，堆放不下 |
| **元空间溢出** | `Metaspace` | 类太多，动态代理/类加载失控 |
| **直接内存溢出** | `Direct buffer memory` | NIO/Netty 堆外内存泄漏 |
| **栈溢出** | `StackOverflowError` | 递归太深 |
| **GC 开销超限** | `GC overhead limit exceeded` | GC 一直跑但回收不了多少 |
| **线程创建失败** | `unable to create new native thread` | 线程太多，系统资源耗尽 |

**查 OOM 类型：**

```bash
# 看日志
grep "OutOfMemoryError" /path/to/app.log | tail -50

# 看 JVM GC 日志
grep "Full GC" /path/to/gc.log | tail -20
```

## 🔥 第四步：常用命令大全

### jmap —— 内存分析

```bash
# 生成 heap dump（最重要！）
jmap -dump:format=b,file=/tmp/heapdump.hprof <PID>

# 只 dump 存活对象（文件更小）
jmap -dump:live,format=b,file=/tmp/heapdump.hprof <PID>

# 查看堆内存摘要
jmap -heap <PID>

# 查看各类对象数量和内存占用（快速定位）
jmap -histo <PID> | head -30

# 只看存活对象
jmap -histo:live <PID> | head -30
```

### jstat —— GC 统计

```bash
# 每 1 秒输出一次 GC 统计（看 GC 频率）
jstat -gcutil <PID> 1000

# 输出示例：
#  S0     S1     E      O      M     CCS    YGC     YGCT    FGC    FGCT     GCT
#  0.00  95.00  82.75  83.25  96.53  93.01    172    2.456    20    3.890    6.346

# S0/S1: Survivor 使用率
# E: Eden 使用率
# O: Old Gen 使用率  ← 关注！接近 100% 就危险了
# M: Metaspace 使用率
# FGC: Full GC 次数  ← 频繁 Full GC 是问题信号
```

### jstack —— 线程分析（排查线程数 OOM）

```bash
# 输出所有线程堆栈
jstack <PID> > /tmp/thread_dump.txt

# 看线程数量（排查线程 OOM）
jstack <PID> | grep "java.lang.Thread.State" | wc -l

# 找死锁
jstack <PID> | grep -A 20 "deadlock"
```

### jcmd —— 全能命令（JDK 7+）

```bash
# 查看所有 Java 进程
jcmd -l

# 查看 JVM 内存使用
jcmd <PID> VM.native_memory

# 触发 Heap Dump
jcmd <PID> GC.heap_dump /tmp/heapdump.hprof

# 查看 JVM 参数
jcmd <PID> VM.flags

# 查看类加载统计
jcmd <PID> GC.class_histogram | head -30

# 查看元空间
jcmd <PID> VM.metaspace
```

### 系统命令

```bash
# 查看内存使用
free -h

# 查看进程内存
top -p <PID>
cat /proc/<PID>/status | grep -i vm

# 查看 JVM 进程
jps -lvm

# 查看文件句柄（排查文件描述符泄漏）
lsof -p <PID> | wc -l

# 查看线程数（排查线程 OOM）
cat /proc/<PID>/status | grep Threads
```

## 🔥 扩展追问 1：堆内存 OOM 排查流程

```
现象：java.lang.OutOfMemoryError: Java heap space

排查步骤：

1. jmap -histo:live <PID> | head -30
   → 看哪个类的实例最多、占用最大

2. MAT 打开 heapdump.hprof
   → Leak Suspects → 看报告

3. 看对象被谁引用（Dominator Tree）
   → 是 List/Map 里积累了大量对象？
   → 是某个 Cache 没设置过期时间？
   → 是某个请求数据太大没释放？

常见原因：
┌──────────────────────────────────────────┐
│ 1. 内存泄漏（Memory Leak）               │
│    对象用完没释放，越积越多               │
│    典型：静态 Map 不断 put，不删除        │
│                                          │
│ 2. 内存溢出（Memory Overflow）           │
│    正常业务需要的内存超过 JVM 配置        │
│    典型：一次查询返回百万条数据           │
└──────────────────────────────────────────┘
```

## 🔥 扩展追问 2：Metaspace OOM 排查

```
现象：java.lang.OutOfMemoryError: Metaspace

Metaspace 存放：类信息、方法、常量池、静态变量

排查步骤：

1. 看类加载数量
   jcmd <PID> GC.class_histogram | head -30
   jstat -gcmetacapacity <PID>

2. Arthas 看类加载器
   classloader   ← 看哪个 ClassLoader 加载了过多的类

3. jmap 看 ClassLoader
   jmap -clstats <PID>

常见原因：
┌──────────────────────────────────────────────┐
│ 1. 动态代理生成了大量类                       │
│    Cglib/JDK 动态代理每次调用都 new 了新代理？ │
│                                              │
│ 2. JSP 热部署                                │
│    每次修改 JSP 都会生成新的 Servlet 类       │
│                                              │
│ 3. OSGi/热加载框架                            │
│    类加载器泄漏，旧类没被卸载                  │
│                                              │
│ 4. Groovy/Kotlin 脚本动态编译                 │
│    每次编译都产生一个新类                     │
└──────────────────────────────────────────────┘

解决：
-XX:MetaspaceSize=256m
-XX:MaxMetaspaceSize=512m
```

## 🔥 扩展追问 3：直接内存 OOM 排查

```
现象：java.lang.OutOfMemoryError: Direct buffer memory

常见于：Netty、NIO、RPC 框架

排查步骤：

1. 看直接内存使用
   jcmd <PID> VM.native_memory summary

2. 看 ByteBuffer 分配
   jmap -histo:live <PID> | grep ByteBuffer

常见原因：
┌────────────────────────────────────────────┐
│ 1. ByteBuffer.allocateDirect() 分配后没释放 │
│    直接内存不受 GC 管理，要手动释放          │
│                                            │
│ 2. Netty 的 ByteBuf 没有 release()         │
│    Netty 的 Direct ByteBuf 要调用 release() │
│                                            │
│ 3. -XX:MaxDirectMemorySize 设置太小        │
└────────────────────────────────────────────┘

配置：
-XX:MaxDirectMemorySize=1g
```

## 🔥 扩展追问 4：线程数 OOM 排查

```
现象：java.lang.OutOfMemoryError: unable to create new native thread

排查步骤：

1. 看当前线程数
   jstack <PID> | grep "java.lang.Thread.State" | wc -l
   cat /proc/<PID>/status | grep Threads

2. 看线程在干什么
   jstack <PID> > /tmp/thread_dump.txt
   grep -A 5 "Thread-" /tmp/thread_dump.txt | head -100

3. 系统线程限制
   ulimit -u         # 进程可创建的最大线程数
   cat /proc/sys/kernel/threads-max

常见原因：
┌──────────────────────────────────────────┐
│ 1. 线程池配置不当（无界线程池）            │
│    Executors.newCachedThreadPool()        │
│    最大线程数 = Integer.MAX_VALUE         │
│                                          │
│ 2. 每个请求都创建新线程                   │
│    new Thread().start()  ← 没有用线程池  │
│                                          │
│ 3. 线程泄漏                              │
│    线程创建了但没有正常退出               │
└──────────────────────────────────────────┘
```

## 🔥 扩展追问 5：GC overhead limit exceeded

```
现象：java.lang.OutOfMemoryError: GC overhead limit exceeded

触发条件：
    JVM 花了 98% 的时间做 GC
    但只回收了 2% 的内存
    连续多次（默认 5 次）都这样
    → JVM 认为没救了，直接 OOM

本质：内存严重不足，GC 快撑不住了

排查：
1. 看 GC 日志
   grep "Full GC" gc.log
   
2. 看堆内存使用率
   jstat -gcutil <PID> 1000

3. 是"内存泄漏"还是"堆太小"？
   MAT 分析 heapdump

临时方案：
-XX:-UseGCOverheadLimit   ← 关掉这个保护（不建议！）
根本方案：
加堆内存 或 修复内存泄漏
```

## 🔥 扩展追问 6：JVM 参数最佳实践（防患于未然）

```bash
# 必备参数
-Xms4g -Xmx4g                          # 堆大小（初始=最大，避免扩容抖动）
-XX:MetaspaceSize=256m                  # 元空间初始大小
-XX:MaxMetaspaceSize=512m              # 元空间最大

-XX:HeapDumpPath=/home/app/logs/heapdump.hprof

# OOM 时执行脚本（比如自动重启）
-XX:OnOutOfMemoryError="bash /home/app/restart.sh"

# GC 日志（线上必须开）
-Xlog:gc*:file=/home/app/logs/gc.log:time,uptime:filecount=10,filesize=100m

# JMX 远程监控（开发/测试环境）
-Dcom.sun.management.jmxremote
-Dcom.sun.management.jmxremote.port=9999
-Dcom.sun.management.jmxremote.authenticate=false
```

## 🔥 扩展追问 8：内存泄漏 vs 内存溢出的区别？

```
内存泄漏（Memory Leak）：
    对象用完了，但引用没释放
    GC 认为它还活着，不回收
    内存持续增长，最终 OOM
    
    典型场景：
    static Map<String, Object> cache = new HashMap<>();
    往里不停放东西，但不删
    
内存溢出（Memory Overflow）：
    内存确实不够用了
    可能是业务确实需要那么多内存
    也可能是配置太小
    
    典型场景：
    SELECT * FROM orders;  // 返回 1 亿条数据，全加载到内存

区别：
    泄漏 → 修代码
    溢出 → 加内存 或 改算法（分批查询等）
    
判断方法：
    重启后内存一开始正常，慢慢涨 → 泄漏
    一开始就很高 → 溢出
```

## 一句话收尾

> "排查 OOM 第一步先确认类型，堆、元空间、直接内存、线程各不同。核心工具是 jmap + MAT，用 Heap Dump 找大对象和引用链。线上必须配好自动 Dump 参数和 GC 日志，否则等服务挂了现场就没了。最终要分清是内存泄漏还是内存溢出，泄漏改代码，溢出改配置或算法。"
