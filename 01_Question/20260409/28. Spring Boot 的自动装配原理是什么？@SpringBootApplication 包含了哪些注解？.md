## 先说结论

**一句话概括**：`@SpringBootApplication` = `@SpringBootConfiguration` + `@EnableAutoConfiguration` + `@ComponentScan`，自动装配的核心是 `@EnableAutoConfiguration` 里的 `AutoConfigurationImportSelector`，它读取所有 `META-INF/spring.factories`（Spring Boot 2.x）或 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`（Spring Boot 3.x）中的配置类，通过条件注解（`@ConditionalOnClass`、`@ConditionalOnBean` 等）按需注册组件。

---

## @SpringBootApplication 拆解

```java
// @SpringBootApplication 源码
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Configuration                          // ① 本类也是一个配置类
@EnableAutoConfiguration                // ② 启用自动装配
@ComponentScan                          // ③ 组件扫描
public @interface SpringBootApplication {}
```

### 三个子注解各司其职

| 注解 | 作用 | 面试重点 |
|-----|------|---------|
| `@SpringBootConfiguration` | 标记当前类是配置类，等价于 `@Configuration` | 几乎等同于 `@Configuration`，支持 Bean 注册 |
| `@ComponentScan` | 扫描 `@Component` 等注解，注册 Bean | 默认扫描启动类所在包及其子包 |
| `@EnableAutoConfiguration` | **自动装配的核心开关** | 最常问，原理最复杂 |

## 自动装配原理（图解）

```
应用启动
    ↓
@SpringBootApplication
    ↓
@EnableAutoConfiguration
    ↓
@Import(AutoConfigurationImportSelector.class)
    ↓
AutoConfigurationImportSelector.postProcessBeanDefinitionRegistry()
    ↓
从 META-INF/spring.factories 加载所有 AutoConfiguration 类
    ↓
按条件注解过滤（@ConditionalOnClass、@ConditionalOnBean 等）
    ↓
符合条件的配置类 → 注册成 Spring Bean
    ↓
应用启动完成
```

## 逐个拆解核心组件

### 1. @EnableAutoConfiguration — 开启自动装配

```java
// @EnableAutoConfiguration 源码
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@AutoConfigurationPackage   // 自动配置包注册
@Import(AutoConfigurationImportSelector.class)  // ← 关键！
public @interface EnableAutoConfiguration {
    // 指定排除的自动配置类
    Class<?>[] exclude() default {};
    String[] excludeName() default {};
}
```

### 2. AutoConfigurationImportSelector — 真正加载配置的类

```java
// AutoConfigurationImportSelector 核心方法
public class AutoConfigurationImportSelector
    implements DeferredImportSelector, BeanClassLoaderAware,
       ResourceLoaderAware, BeanFactoryAware, EnvironmentAware {

    // 这个方法返回的类，就是 Spring 要加载的自动配置类
    @Override
    public String[] selectImports(AnnotationMetadata annotationMetadata) {
        // 核心逻辑：加载所有 AutoConfiguration
        List<String> configs = SpringFactoriesLoader.loadFactoryNames(
            getBootstrapContext(), getEnvironment()
        );
        // configs ≈ ["DataSourceAutoConfiguration",
        //            "WebMvcAutoConfiguration",
        //            "RedisAutoConfiguration", ...]
        return configs.toArray(new String[0]);
    }
}
```

### 3. SpringFactoriesLoader — 读取配置的引擎

```java
// SpringFactoriesLoader.loadFactoryNames() 核心逻辑

public static List<String> loadFactoryNames(
        Class<?> factoryType, ClassLoader classLoader) {
    
    // 读取 META-INF/spring.factories 文件
    String fullPath = "META-INF/spring.factories";
    
    Enumeration<URL> urls = classLoader.getResources(fullPath);
    
    List<String> result = new ArrayList<>();
    
    while (urls.hasMoreElements()) {
        URL url = urls.nextElement();
        Properties properties = PropertiesLoaderUtils.loadProperties(new UrlResource(url));
        
        // 从 spring.factories 中读取 key = EnableAutoConfiguration
        // value = 所有自动配置类的全限定名（逗号分隔）
        String classNames = properties.getProperty(
            "org.springframework.boot.autoconfigure.EnableAutoConfiguration"
        );
        
        result.addAll(Arrays.asList(classNames.split(",")));
    }
    
    return result;  // 返回 ["A配置", "B配置", "C配置", ...]
}
```

## 条件注解 — 自动装配的"按需加载"机制

```
为什么不需要 DataSource 就不会报 NoSuchBean 错误？
    ↓
因为每个 AutoConfiguration 前面都有条件注解
    ↓
Spring Boot 按需加载，不是全部加载
```

### 常用的条件注解

```java
// 1. @ConditionalOnClass —— 当类路径下存在某个类时才加载
@Configuration
@ConditionalOnClass(DataSource.class)        // 有 DataSource 类才加载
@EnableConfigurationProperties(DataSourceProperties.class)
@Import(DataSourceConfigurationGenerator.class)
public class DataSourceAutoConfiguration {}

// 如果没有引入 Druid/HikariCP 依赖（没有 DataSource.class）
// → 这个配置类不会被加载
// → 不报错


// 2. @ConditionalOnMissingBean —— 容器中没有某个 Bean 时才注册
@Configuration
@ConditionalOnClass(DataSource.class)
@ConditionalOnMissingBean(DataSource.class)  // ← 关键！
public class DataSourceAutoConfiguration {
    @Bean
    public DataSource dataSource() {
        // 只有当用户没有手动定义 DataSource Bean 时才自动配置
        return new HikariDataSource();
    }
}


// 3. @ConditionalOnProperty —— 配置文件中存在某个属性时才加载
@Configuration
@ConditionalOnProperty(prefix = "spring.redis", name = "host")
public class RedisAutoConfiguration {}


// 4. @ConditionalOnBean / @ConditionalOnMissingBean
// 某个 Bean 存在/不存在时才加载


// 5. @AutoConfigureAfter —— 在某个自动配置之后加载
@Configuration
@AutoConfigureAfter(DataSourceAutoConfiguration.class)
public class MyBatisAutoConfiguration {}
```

**配置生效的完整流程**：

```
DataSourceAutoConfiguration 加载检查：
    ↓
@ConditionalOnClass(DataSource.class)
    ↓
项目引入了 Druid 依赖？→ 有 DataSource.class → ✅ 继续
    ↓
@ConditionalOnMissingBean(DataSource.class)
    ↓
用户手动定义了 @Bean DataSource？→ 没有 → ✅ 继续
    ↓
@ConditionalOnProperty(spring.datasource.url)
    ↓
配置文件中写了 spring.datasource.url？→ ✅ 继续
    ↓
注册 HikariCP DataSource Bean ✅
```

## @ComponentScan 的坑 — 启动类必须在根包

```java
@SpringBootApplication
// 默认等价于：@ComponentScan(basePackages = "com.example")
// 即扫描 com.example 及其子包下的所有 @Component
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

// ✅ 正确：启动类在根包
com.example.Application          ← 启动类在根包
com.example.service.UserService   ← 会被扫描到 ✅
com.example.controller.UserController

// ❌ 错误：启动类不在根包
com.example.Application           ← 在子包
com.config.Application           ← 启动类在 com.config

// 结果：com.example 包下的所有 Bean 扫描不到 ❌
// 因为 @ComponentScan 默认只扫描启动类所在包（com.config）
```

### 追问 2：自定义 Starter 的标准流程

```java
// 目标：封装一套"用户画像分析"组件，做成 starter 给其他项目复用

// ========== 1. 引入方：引入依赖 ==========
// pom.xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>user-profile-spring-boot-starter</artifactId>
    <version>1.0.0</version>
</dependency>

// application.yml（可选配置）
user-profile:
  enabled: true
  cache-ttl: 3600
  algorithm: ml

// ========== 2. starter 自身：定义自动配置类 ==========
// src/main/resources/META-INF/spring/
// org.springframework.boot.autoconfigure.AutoConfiguration.imports（Spring Boot 3.x）
// 或 META-INF/spring.factories（Spring Boot 2.x）

// ① 自动配置类（核心）
@Configuration
@ConditionalOnClass(UserProfileService.class)          // 依赖类存在才加载
@ConditionalOnProperty(prefix = "user-profile", havingValue = "true", matchIfMissing = true)
@EnableConfigurationProperties(UserProfileProperties.class)
public class UserProfileAutoConfiguration {
    
    @Bean
    @ConditionalOnMissingBean(UserProfileService.class)  // 没有自定义才注册
    public UserProfileService userProfileService(
            UserProfileProperties properties) {
        return new UserProfileServiceImpl(properties);
    }
}

// ② 配置属性类（@ConfigurationProperties）
@Data
@ConfigurationProperties(prefix = "user-profile")
public class UserProfileProperties {
    private boolean enabled = true;
    private int cacheTtl = 3600;
    private String algorithm = "ml";
}

// ========== 3. starter 的 pom.xml 不需要代码 ==========
// 依赖传递设计：
// user-profile-starter → 依赖 user-profile-core
// 引入 starter 的项目只需依赖 starter，不需要关心 core
```

### 追问 5：@Configuration 和 @SpringBootConfiguration 的区别

```java
// @SpringBootConfiguration 源码
@Configuration
public @interface SpringBootConfiguration {}

// 两者功能完全一样！
// @SpringBootConfiguration 是 @Configuration 的别名
// 只是为了语义更明确（这是 Spring Boot 应用的配置类）

// 唯一区别：@SpringBootConfiguration 只能标注一次
// 但 @Configuration 可以标注多次（需要 @ComponentScan 配合）
```

## 一句话收尾

> **Spring Boot 自动装配的本质是"约定大于配置"——`META-INF/spring.factories`（3.x 用 imports 文件）里列出了所有候选配置类，`@ConditionalOnClass` 判断依赖是否满足，`@ConditionalOnMissingBean` 确保用户手动配置优先，条件都满足才注册成 Bean，`@SpringBootApplication` 只是把 `@Configuration`、`@EnableAutoConfiguration`、`@ComponentScan` 三个注解打包成一个，用起来方便而已。**
