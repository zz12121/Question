## 一、分布式锁的核心要求

```
分布式锁必须满足的 5 个特性：

1. ✅ 互斥性：同一时刻只有一个客户端能持有锁
2. ✅ 不会死锁：持有锁的客户端崩溃后，锁必须能被自动释放
3. ✅ 锁可重入：同一客户端可多次获取同一把锁
4. ✅ 公平性（可选）：按请求顺序依次获取锁
5. ✅ 高性能：获取/释放锁的延迟要低，高并发下不排队
```

## 二、Redis 分布式锁

### 方案一：SETNX + EXPIRE（最简陋，有致命缺陷）

```java
// ❌ 这是错误写法！不要用
public void wrongLock(String lockKey) {
    Boolean result = redis.setnx(lockKey, "1");
    if (result) {
        redis.expire(lockKey, 30);  // ← 两步不是原子的！
        try {
            // 业务逻辑
        } finally {
            redis.del(lockKey);
        }
    }
}

// 致命问题：
// SETNX 成功 → EXPIRE 执行前崩溃 → 锁永远不过期 → 死锁！
```

### 方案二：SET NX EX 原子命令（业界标准雏形）

```java
// ✅ 正确基础版
public boolean tryLock(String lockKey, String requestId, long expireSeconds) {
    // SET + NX + EX 三个操作原子完成
    // NX：不存在才设置
    // EX：设置过期时间
    String result = redis.set(lockKey, requestId, 
        SetParams.setParams().nx().ex(expireSeconds));
    return "OK".equals(result);
}

public void unlock(String lockKey, String requestId) {
    // ❌ 错误写法
    // if (redis.get(lockKey).equals(requestId)) {
    //     redis.del(lockKey);  ← 先查后删，不是原子的，中间可能被其他客户端获取锁
    // }
    
    // ✅ Lua 脚本保证原子性
    String script = 
        "if redis.call('get', KEYS[1]) == ARGV[1] then " +
        "  return redis.call('del', KEYS[1]) " +
        "else return 0 end";
    
    jedis.eval(script, 1, lockKey, requestId);
}

// requestId = UUID.randomUUID().toString() + threadId
// → 用于标识"这把锁是我加的"，防止误删别人的锁
```

### 方案三：Redisson（生产环境推荐）

```java
// Redisson：开箱即用的 Redis 分布式锁库
RLock lock = redisson.getLock("order:lock:" + orderId);

// 加锁（自动续期）
lock.lock();  // 默认 30 秒，自动看门狗续期
// 内部逻辑：
// 1. tryAcquire() → SET lockKey requestId NX EX 30
// 2. scheduleExpirationRenewal() → 每 10 秒续期一次，每次 +30 秒
// 3. unlock() → cancelExpirationRenewal() → DEL

// 指定锁时间
lock.lock(10, TimeUnit.SECONDS);  // 10 秒后自动释放，不续期

// 尝试获取（不等待）
boolean locked = lock.tryLock();
boolean locked2 = lock.tryLock(5, 10, TimeUnit.SECONDS);  // 等5秒，获取到就锁10秒

// 释放
try {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();  // 必须判断持有者，只能自己解锁自己
    }
} finally {
    // 建议用 try-finally 包裹
}

// 可重入
RLock lock1 = redisson.getLock("resource:A");
lock1.lock();    // 第一次
lock1.lock();    // 第二次，同一线程，重入成功
lock1.unlock();
lock1.unlock();  // 必须解锁两次
```

**Redisson 看门狗（Watchdog）机制**：

```
场景：业务执行需要 60 秒，但锁只设置了 30 秒
    ↓
看门狗介入：
lock.lock(30, TimeUnit.SECONDS)
    ↓
加锁成功 → 看门狗启动
    ↓
每 10 秒（internalLockLeaseTime / 3）检查：
  → 锁还在被当前线程持有？
  → 是 → 续期 +30 秒
  → 否 → 不续期
    ↓
业务执行到 30 秒 → 看门狗续期 → 继续执行
    ↓
业务执行到 60 秒 → 业务完成
    ↓
finally: lock.unlock() → 停止看门狗 → DEL 锁
    ↓
锁总共存在 ~60 秒，业务顺利执行完成 ✅
```

### Redisson 的三大问题及应对

```
❌ 问题1：锁自动续期，业务执行完了还在续期？
✅ 解决：用 tryLock(leaseTime, TimeUnit) 指定锁超时，不启用看门狗

❌ 问题2：Redisson 客户端挂了？
✅ 解决：Redisson 集群模式（多个 Redis 节点，需要多数节点都成功才加锁成功）

❌ 问题3：主从切换时锁丢失？
✅ 解决：用 RedLock（红锁）：同时向 N 个独立 Redis 节点加锁，
   N/2+1 个成功才算成功（需 >= 5 个节点）
   注意：RedLock 有争议，Redis 作者 antirez 和分布式系统专家 Martin Kleppmann
   曾公开辩论过，观点对立）
```

## 三、ZooKeeper 分布式锁

### 工作原理：临时顺序节点 + Watch 机制

```
ZooKeeper 数据模型：
/
├── locks/
│   ├── order-0000000001  ← 最小序号，获得锁
│   ├── order-0000000002  ← 等锁
│   └── order-0000000003  ← 等锁

获取锁流程：
1. 在 /locks 下创建临时顺序节点（Ephemeral Sequential）
2. 获取 /locks 下所有子节点
3. 判断自己是否是序号最小的节点
   → 是 → 获取锁成功 ✅
   → 否 → 给比自己小 1 的节点注册 Watch
4. 等比自己小的节点删除（锁释放）→ 被通知 → 重新判断序号
```

```java
// CuratorFramework（ZooKeeper 客户端库）
InterProcessMutex lock = new InterProcessMutex(client, "/locks/order");

// 获取锁
lock.acquire(30, TimeUnit.SECONDS);  // 最多等 30 秒

try {
    // 业务逻辑
} finally {
    lock.release();  // 释放锁
}

// 可重入
lock.acquire(30, TimeUnit.SECONDS);  // 同一线程可再次获取
lock.release();                       // 同样需要解锁两次
```

**ZooKeeper 锁释放机制**：

```
客户端A 持有锁（创建了 /locks/order-0000000001）
    ↓
客户端A 崩溃 / 网络断开
    ↓
ZooKeeper 检测到会话失效（默认 60 秒）
    ↓
ZooKeeper 自动删除临时节点 /locks/order-0000000001
    ↓
客户端B 的 Watch 被触发
    ↓
客户端B 判断自己是最小节点 → 获取锁 ✅
    ↓
完美解决了锁无法自动释放的问题！
```

## 四、Redis vs ZooKeeper 对比

| 对比维度 | Redis 分布式锁 | ZooKeeper 分布式锁 |
|---------|--------------|------------------|
| **可靠性** | 中（主从切换丢锁风险） | 高（ZAB 协议，强一致） |
| **性能** | 极高（纯内存操作） | 中（写本地 + 网络延迟） |
| **锁类型** | TTL 过期，不保证释放 | 临时节点，会话失效自动删除 ✅ |
| **可重入** | 支持（需代码层实现） | 支持（Curator 默认支持） |
| **公平锁** | 不支持普通版，Redisson 支持公平锁 | 支持（顺序节点天然公平） |
| **等待队列** | 不支持（轮询 SETNX） | 支持（Watch 通知，无需轮询 ✅） |
| **锁超时** | 需手动设置 + 看门狗续期 | TTL 无限，等通知 |
| **实现复杂度** | 中（需处理大量边界情况） | 低（Curator 封装完善） |
| **运维成本** | 低（Redis 生态成熟） | 高（ZK 集群部署复杂） |

## 🔥 扩展追问

### 追问 1：Redis 分布式锁的 5 大坑及解决方案

```
坑1：锁被别人误删
原因：业务超时，锁自动释放，另一个客户端拿到锁，原客户端执行完解锁，删了别人的锁

场景：
  线程A：持有锁（超时30秒）
  线程A：业务执行了 35 秒
  线程A：锁自动过期释放了
  线程B：获取到锁
  线程A：执行完成，unlock() → 删了线程B刚拿到的锁！

解决：Lua 脚本 + requestId 判断，只有自己的锁才删
     （但这只解决了"删别人锁"的问题，不解决业务超时问题）
     治本：用 Redisson 看门狗自动续期


坑2：锁无法自动续期（业务超时）
解决：Redisson 看门狗机制


坑3：主从切换丢锁
场景：
  Redis Master 持有锁
  Master → Slave 同步（有延迟）
  Master 宕机，切换到 Slave（新 Master 没有锁数据）
  其他客户端获取到锁（重复获取成功！）

解决：RedLock 向 5 个独立 Redis 节点加锁（N/2+1 成功才算成功）
  但争议很大，实际项目中用 Redisson 集群模式更稳妥


坑4：锁粒度过大
场景：锁了"整个商品分类"而不是"单个商品"
  → 一个商品卖完，整个分类都不能卖了

解决：锁粒度要精确
  ❌ lock("product:category:phone")
  ✅ lock("product:lock:" + skuId)  — 按单个 SKU 加锁


坑5：分布式锁 + Redis 集群脑裂
场景：Redis Cluster 网络分区
  分区1：3个节点，有1个是主
  分区2：2个节点，有1个是主
  两个分区各有1个客户端获取到了同一把锁！

解决：使用 RedLock，且节点分布在不同机房/机架
```

### 追问 2：分布式锁的可重入实现

```java
// Redis 可重入锁（需要维护计数器）
// 基于 ThreadLocal 记录锁的持有次数

@Component
public class RedisReentrantLock {
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    private static final ThreadLocal<Map<String, Integer>> HOLD_COUNT = 
        ThreadLocal.withInitial(HashMap::new);
    
    public boolean tryLock(String lockKey, String requestId, long expireSeconds) {
        Map<String, Integer> counts = HOLD_COUNT.get();
        
        // 如果已经持有这个锁，直接计数 +1
        if (counts.containsKey(lockKey)) {
            counts.put(lockKey, counts.get(lockKey) + 1);
            return true;
        }
        
        // 首次加锁
        String result = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, requestId, expireSeconds, TimeUnit.SECONDS);
        
        if (Boolean.TRUE.equals(result)) {
            counts.put(lockKey, 1);
            return true;
        }
        return false;
    }
    
    public void unlock(String lockKey, String requestId) {
        Map<String, Integer> counts = HOLD_COUNT.get();
        int count = counts.getOrDefault(lockKey, 0);
        
        if (count > 1) {
            counts.put(lockKey, count - 1);  // 重入次数 -1，不真正解锁
        } else {
            counts.remove(lockKey);
            // Lua 脚本原子删除
            String script = 
                "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                "  return redis.call('del', KEYS[1]) " +
                "else return 0 end";
            redisTemplate.execute(
                new DefaultRedisScript<>(script, Long.class),
                Collections.singletonList(lockKey),
                requestId
            );
        }
    }
}
```

### 追问 3：RedLock 的争议（Martin Kleppmann vs antirez）

```
Martin Kleppmann（分布式系统专家，《数据密集型应用》作者）的质疑：

1. 时钟漂移问题：
   RedLock 向 5 台机器发 SET NX EX 命令
   但 5 台机器的时钟可能不同步！
   如果某台机器时钟快了，它的锁会提前过期
   → 锁失效窗口存在

2. RedLock 不是无主架构：
   如果能接受单点故障（Redis 主从），为什么要用 5 个独立机器？
   如果不能接受，用 Raft 协议的 Redis Cluster 更合适

3. RedLock 性能低：
   需要 5 台机器都响应，N/2+1 = 3 才成功
   实际部署 5 台 Redis 成本很高

antirez（Redis 作者）反驳：
   RedLock 的设计前提是 5 台独立机器（不共享内存）
   时钟漂移可以通过合理配置 NTP 解决
   牺牲了部分性能换来了真正的分布式锁

实战建议：
   - 强一致性场景：用 ZooKeeper 或 etcd
   - 高并发缓存场景：用 Redisson 单节点或哨兵模式
   - RedLock：需要 5 台独立机器，成本高，一般公司用不起
```

### 追问 4：ZooKeeper 分布式锁的性能问题

```
ZooKeeper 的性能瓶颈：

问题1：写操作走 ZAB 协议，只能在 Leader 处理
  → 所有写请求都经过 Leader
  → Leader 成为瓶颈

问题2：Watch 机制需要维护 TCP 连接
  → 大量锁请求 → 大量 Watch 连接
  → ZooKeeper 每个连接默认 1MB 内存
  → 万级并发时内存消耗大

问题3：临时节点依赖会话，Session 超时需 60 秒
  → 客户端挂了，60 秒后锁才释放
  → 无法快速故障转移

对比：etcd 的 etcdv3 用 gRPC，性能比 ZK 好 10 倍


解决方案：Curator 提供了 3 种锁实现
- InterProcessMutex：普通分布式可重入锁
- InterProcessSemaphoreMutex：不可重入锁（性能更好）
- InterProcessMultiLock：多锁批量操作
```

### 追问 6：分布式锁的典型应用场景

```java
// 场景1：分布式任务调度（同一时刻只能有一个节点执行）
@Scheduled(cron = "0 0 2 * * ?")
public void dailySettlement() {
    RLock lock = redissonClient.getLock("settlement:daily");
    if (lock.tryLock()) {       // tryLock 不等待，直接判断
        try {
            doSettlement();
        } finally {
            lock.unlock();
        }
    }
    // 获取不到锁的节点直接跳过本次任务
}


// 场景2：库存扣减（超卖问题）
public boolean deductStock(String skuId, int quantity) {
    String lockKey = "stock:lock:" + skuId;
    RLock lock = redissonClient.getLock(lockKey);
    
    try {
        lock.lock(10, TimeUnit.SECONDS);
        
        // Lua 脚本保证原子扣减
        String script = 
            "local stock = redis.call('get', KEYS[1]) " +
            "if tonumber(stock) >= tonumber(ARGV[1]) then " +
            "  redis.call('decrby', KEYS[1], ARGV[1]) " +
            "  return 1 " +
            "else return 0 end";
        
        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(script, Long.class),
            Collections.singletonList("stock:" + skuId),
            String.valueOf(quantity)
        );
        
        return result != null && result == 1;
    } finally {
        lock.unlock();
    }
}


// 场景3：消息幂等（防止消息重复消费）
public void processMessage(String msgId) {
    String lockKey = "msg:lock:" + msgId;
    RLock lock = redissonClient.getLock(lockKey);
    
    if (lock.tryLock(0, 30, TimeUnit.SECONDS)) {
        try {
            // 幂等检查
            if (processedMapper.exists(msgId)) {
                return;  // 已处理过，跳过
            }
            doProcess(msgId);
            processedMapper.insert(msgId);  // 记录已处理
        } finally {
            lock.unlock();
        }
    }
}
```

## 一句话收尾

> **Redis 分布式锁靠 SET NX EX 原子命令 + requestId 防误删 + 看门狗续期，是高并发场景的首选（快但有争议）；ZooKeeper 靠临时顺序节点 + Watch 通知天然实现公平锁和自动释放，可靠性极高但性能略低；生产环境推荐 Redisson（Redis）或 Curator（ZooKeeper），不要自己手写锁逻辑——边界情况太多，社区成熟的轮子永远比自己的更健壮。**
