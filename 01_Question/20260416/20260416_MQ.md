# 理论综合题 - 消息队列（A236-A265）

> 以下为消息队列领域的理论综合题，编号从236起，覆盖 Kafka、RocketMQ、RabbitMQ 三大主流消息中间件的核心原理与机制。

---

## Kafka 核心原理（10题）

236. Kafka 的消息存储结构是怎样的？为什么选择追加写入（Append-Only Log）而不是随机写入？Topic、Partition、Segment 三者的关系是什么？一个 Partition 对应磁盘上的几个文件？（.log / .index / .timeindex）Segment 文件的大小由什么参数控制？（`log.segment.bytes` / `log.roll.ms`）当一个 Segment 写满后会发生什么？

237. Kafka 为什么这么快？从"磁盘读写"、"网络传输"、"操作系统层面"三个维度分析。（顺序写零寻道 / SendFile 零拷贝 + PageCache / 批量压缩 / 分区并行）零拷贝（Zero-Copy）在 Kafka 中具体是怎么实现的？（`java.nio.channels.FileChannel.transferTo()` 底层调用 `sendfile()` 系统调用）零拷贝和普通的文件读取+网络发送有什么区别？少了哪几次数据拷贝？

238. Kafka 的"消费者组"（Consumer Group）机制是怎样的？同一个 Group 内的消费者如何分配 Partition？（RangeAssignor / RoundRobinAssignor / StickyAssignor / CooperativeStickyAssignor）如果消费者数量大于 Partition 数量会怎样？（多出的消费者空闲）一个 Partition 会被同一个 Group 内的多个消费者同时消费吗？消费者组的 Rebalance 是什么？Rebalance 过程中会发生什么？（STW + 重复消费）

239. Kafka 的 Rebalance 机制是怎么触发的？（消费者加入/离开/心跳超时 / 订阅 Topic 数量变化 / Partition 数量变化）Rebalance 期间的"Stop-The-World"问题有多严重？（所有消费者暂停消费，可能造成消息积压）Kafka 2.4 引入的增量协作式 Rebalance（Incremental Cooperative Rebalance / CooperativeStickyAssignor）怎么解决了这个问题？心跳机制（Heartbeat）和会话超时（Session Timeout）分别由什么参数控制？

240. Kafka 如何保证消息"不丢失"？从 Producer 端、Broker 端、Consumer 端三个维度分别说明。Producer 端：`acks=all` + `retries>0` + `min.insync.replicas` 的含义是什么？Broker 端：副本机制（Replication）和 ISR（In-Sync Replicas）列表的作用是什么？Leader 副本和 Follower 副本的同步机制是怎样的？Consumer 端：手动提交 offset（`enable.auto.commit=false`）的意义是什么？"精确一次语义"（Exactly-Once Semantics）在 Kafka 中是怎么实现的？（幂等 Producer + 事务）

241. Kafka 的"幂等 Producer"是怎么工作的？（Producer ID + Sequence Number 去重）为什么需要幂等性？（网络重试可能发送重复消息）幂等 Producer 的作用范围是什么？（单个 Partition 内去重）如果消息发送到了不同的 Partition，幂等性还能保证吗？Kafka 的事务机制（Transactional API）解决了什么问题？（跨多个 Partition 的原子写入 + 消费端的事务隔离 `isolation.level=read_committed`）`transactional.id` 和 `producer.id` 的区别是什么？

242. Kafka 的消息是有序的吗？单个 Partition 内有序、多个 Partition 之间无序——这个结论对吗？如果要保证"全局有序"，有哪些方案？（只有一个 Partition / 全局序号+应用层排序 / 外部排序存储）如果同一个订单的"创建"和"支付"消息被分配到了不同的 Partition，如何保证这两个消息的消费顺序？（相同 Key 路由到相同 Partition / 订单号作为 Key）

243. Kafka 的"水位线"（Watermark / High Watermark）和"Leader Epoch"分别是什么？HW 的作用是什么？（标记哪些消息对所有副本可见，用于保证一致性）LSO（Log Stable Offset）又是什么？Follower 副本的 LEO（Log End Offset）和 HW 是怎么更新的？"Leader Epoch"机制（Kafka 2.x 引入）解决了什么问题？（旧 Leader 恢复后截断不一致的数据导致数据丢失）

244. Kafka 的文件检索机制是怎样的？消息写入后，如何根据 offset 快速定位到具体的 Segment 文件和文件内位置？（二分查找 .index 稀疏索引文件，再通过 base offset + 相对位置定位 .log 文件）稀疏索引为什么比密集索引更适合 Kafka 的场景？（索引文件小、常驻内存）`.timeindex` 文件的作用是什么？（根据时间戳查找 offset，用于按时间重置消费位点）

245. Kafka 的"脏选举"（Unclean Leader Election）是什么？当所有 ISR 副本都宕机了，如果设置 `unclean.leader.election.enable=true`，会发生什么？（从非 ISR 副本中选出新 Leader，可能丢失数据）设置为 `false` 呢？（Partition 不可用，优先保证一致性）在"可用性"和"一致性"之间怎么取舍？Kafka 的 `min.insync.replicas` 参数和 `acks=all` 如何配合使用来保证数据安全？

## RocketMQ 核心原理（8题）

246. RocketMQ 的整体架构是怎样的？NameServer、Broker、Producer、Consumer 四个角色的职责分别是什么？NameServer 和 ZooKeeper 有什么区别？（NameServer 更轻量、各节点独立无通信、无选举机制）Broker 的 Master/Slave 架构是怎样的？ Slave 节点能接收写入吗？ RocketMQ 的 Topic 和 Queue 的关系是什么？（一个 Topic 包含多个 Queue，类似 Kafka 的 Partition）

247. RocketMQ 的"事务消息"是怎么工作的？（半消息 Half Message + 本地事务执行 + 提交/回滚 Commit/Rollback + 回查 Check）具体流程：Producer 先发送半消息到 Broker → Broker 存储到 Half Topic → Producer 执行本地事务 → 根据结果发送 Commit 或 Rollback → 如果 Broker 没收到二次确认，发起回查 → Producer 执行回查逻辑。回查的频率和最大次数分别是什么？（默认回查 15 次，间隔 60s）如果回查 15 次后仍然无法确定事务状态，消息会怎样？（回滚丢弃）

248. RocketMQ 的"延迟消息"是怎么实现的？（修改 Topic 为 `SCHEDULE_TOPIC_XXXX` + 18 个延迟级别对应 18 个 Queue + TimerThread 定时扫描投递到目标 Topic）18 个延迟级别分别是什么？（1s/5s/10s/30s/1m/2m/3m/4m/5m/6m/7m/8m/9m/10m/20m/30m/1h/2h）为什么不支持任意延迟时间？开源版如果要实现任意延迟时间，有哪些方案？（时间轮 Timing Wheel / 分级延迟类似 Hashed Wheel Timer）

249. RocketMQ 的"消息重试"机制是怎样的？消费失败后消息会被投递到哪个队列？（重试队列 `%RETRY%ConsumerGroup`，延迟级别递增）最大重试次数是多少？（默认 16 次）重试间隔是怎样的？（每次间隔递增 10s/30s/1m/2m...）如果超过最大重试次数，消息会去哪里？（死信队列 `%DLQ%ConsumerGroup`）RocketMQ 的"顺序消息"是怎么保证的？（相同 MessageQueue + 单线程消费或 MessageListenerOrderly）

250. RocketMQ 的"消息过滤"有几种方式？（Tag 过滤 / SQL92 表达式过滤）Tag 过滤是在 Broker 端还是 Consumer 端执行的？（Broker 端，基于 Hash 结构高效匹配）Tag 和 SQL92 过滤有什么区别？（Tag 只能做简单匹配，SQL92 支持条件表达式如 `age > 18`）一条消息可以有多个 Tag 吗？（不能，但可以用 `||` 分隔模拟）为什么 RocketMQ 推荐使用 Tag 而不是创建多个 Topic 做区分？

251. RocketMQ 的存储结构是怎样的？CommitLog 和 ConsumeQueue 的关系是什么？（所有 Topic 的消息都顺序写入 CommitLog，ConsumeQueue 是每个 Queue 的索引文件）为什么 RocketMQ 选择"所有消息写一个文件"而不是"每个 Topic 一个文件"？（顺序写效率最高、避免磁盘寻道）ConsumeQueue 中存储了什么？（offset / 消息长度 / tagHashcode）如果根据 ConsumeQueue 找到 CommitLog 中的消息？

252. RocketMQ 和 Kafka 在架构上的核心区别有哪些？（NameServer vs ZooKeeper / CommitLog 统一存储 vs Partition 分散存储 / 延迟消息 vs 不支持 / 事务消息 vs 事务 API / 文件存储模型差异）什么场景下选 RocketMQ 更合适？（电商/金融场景需要事务消息、延迟消息、精细的消费控制）什么场景下选 Kafka 更合适？（大数据量日志采集、流处理、高吞吐量场景）

253. RocketMQ 的"长轮询"（Long Polling）机制是怎么工作的？Consumer 拉取消息时，如果 Broker 上没有新消息，是立即返回空还是等待？长轮询的等待时间由什么参数控制？长轮询和传统的"推模式"（Push）有什么区别和优劣？ RocketMQ 的 Push 模式底层实际是 Pull 还是 Push？（底层是 Pull，通过 `PullMessageService` 不断拉取模拟 Push）

## RabbitMQ 核心原理（7题）

254. RabbitMQ 的整体架构是怎样的？Exchange、Queue、Binding 三者的关系是什么？Exchange 的类型有哪些？（Direct / Fanout / Topic / Headers）Direct Exchange 和 Topic Exchange 的路由规则分别是什么？（Direct 精确匹配 routing key / Topic 支持通配符 `*` 和 `#`）消息从 Producer 到 Consumer 的完整流转路径是怎样的？

255. RabbitMQ 的消息投递保障机制有哪些？（Publisher Confirm / Publisher Return / Consumer ACK / 持久化）Publisher Confirm（确认模式）和 Publisher Return（返回模式）的区别是什么？（Confirm 确认消息是否到达 Exchange / Return 确认消息是否路由到 Queue）如果消息路由不到任何 Queue（routing key 没有匹配的 Binding），会怎样？（根据 `mandatory` 参数决定：true 则回退给 Producer，false 则丢弃）

256. RabbitMQ 的"死信队列"（Dead Letter Exchange, DLX）是怎么工作的？消息在什么情况下会变成死信？（消息被拒绝且 requeue=false / 消息 TTL 过期 / 队列达到最大长度）死信消息会被投递到哪里？（绑定了死信交换器 DLX 的队列）死信队列在实际项目中有什么用途？（延迟队列实现 / 失败消息集中处理 / 异常监控告警）RabbitMQ 原生不支持延迟消息，如何通过"死信队列+TTL"实现延迟投递？

257. RabbitMQ 的消息持久化机制是怎样的？Queue 持久化、Message 持久化、Exchange 持久化三者缺一不可——为什么？（Queue 持久化保证队列元数据不丢失 / Message 持久化保证消息内容不丢失 / Exchange 持久化保证路由关系不丢失）RabbitMQ 的持久化是怎么实现的？（写入磁盘前的消息先缓存在内存 / 镜像队列 Mirror Queue / Quorum Queue）RabbitMQ 3.8 引入的 Quorum Queue（仲裁队列）和 Classic Queue 的区别是什么？（Quorum Queue 基于 Raft 共识、强一致性、但吞吐量略低）

258. RabbitMQ 的消费确认（ACK）机制有几种模式？（自动确认 auto-ack / 手动确认 manual-ack）自动确认在什么场景下会有"消息丢失"的风险？（消费者还没处理完就确认了，消费者宕机后消息丢失）手动确认中 `basic.ack`、`basic.nack`、`basic.reject` 的区别是什么？（ack 确认成功 / nack 拒绝可重入队列 / reject 只能拒绝单条）如果消息一直处理失败（反复 nack requeue），会产生什么问题？（消息堆积、消费阻塞）如何设置重试次数上限？

259. RabbitMQ 的"消费限流"（QoS / Prefetch Count）是怎么工作的？`basic.qos(prefetch_count=N)` 的含义是什么？（限制未确认消息的最大数量为 N）为什么需要消费限流？（消费者处理速度跟不上，限制每次拉取的消息数量，避免消费者 OOM）`prefetch_count` 设置为 1 和 100 分别适合什么场景？RabbitMQ 3.12 的 Quorum Queue 对 `prefetch_count` 的处理和 Classic Queue 有什么不同？

260. RabbitMQ 的集群模式有几种？（普通集群 / 镜像队列集群 / 仲裁队列集群）普通集群中，每个节点都存储所有队列的消息吗？（不是，只有队列所在节点存储完整消息，其他节点只有元数据）镜像队列（Classic Mirrored Queue）的同步方式是什么？（Master-Slave 异步复制）镜像队列有什么问题？（主从切换时可能丢数据 /Slave 不分担写入压力）RabbitMQ 3.13 为什么废弃了镜像队列？（推荐使用 Quorum Queue 替代）

## 消息队列通用 / 选型 / 对比（5题）

261. Kafka、RocketMQ、RabbitMQ 三者的核心对比维度有哪些？从吞吐量、延迟、消息可靠性、消费模式、生态兼容性五个维度分别对比。（Kafka：高吞吐百万级TPS、毫秒级延迟、At-Least-Once 为主 / RocketMQ：十万级TPS、毫秒级延迟、事务消息和延迟消息 / RabbitMQ：万级TPS、微秒级延迟、精细路由和灵活消费控制）在什么业务场景下分别选择哪个？

262. 什么是消息队列的"消息积压"问题？积压的原因通常有哪些？（消费者处理速度慢 / 消费者宕机 / 突发流量洪峰 / Rebalance 导致消费暂停）如何监控消息积压？（Lag 监控 / Consumer Lag 告警）消息积压后如何快速消费？（临时扩容消费者 / 增加分区数 / 降级处理非核心消息 / 批量消费提高吞吐）如果消费者扩容后仍然跟不上，怎么办？（将积压消息转发到新的 Topic + 专用消费者批量处理）

263. 什么是消息队列的"消息幂等性"问题？为什么消息队列本身只保证"至少一次投递"（At-Least-Once）而不保证"精确一次消费"？（网络重试、Rebalance、Consumer 重启都可能导致重复消费）在消费端如何实现幂等？（全局唯一ID+去重表 / Redis SETNX 去重 / 数据库唯一约束 / 业务状态机幂等）幂等性和"消息顺序性"之间有冲突吗？（幂等的去重判断需要时间，可能影响消息处理顺序）

264. 什么是消息队列的"重复消息"和"消息丢失"？从 Producer → Broker → Consumer 全链路分析可能产生重复消息的节点（网络重试、Broker 副本同步、Consumer Rebalance）和可能丢失消息的节点（异步发送未确认、Broker 宕机未刷盘、Consumer 自动提交 ACK）。在实际项目中，你是选择"宁可重复不可丢失"还是"宁可丢失不可重复"？如何用"最大努力交付 + 消费端幂等"来平衡两者？

265. 消息队列的"消息顺序性"如何保证？Kafka 单 Partition 有序，但如果消费者用多线程处理会怎样？（顺序会被打乱）如何在多线程消费场景下保证同一业务实体的消息顺序？（相同 Key hash 到同一个消费线程 / 内存队列按 Key 分组串行处理）RocketMQ 的顺序消息实现原理是什么？（MessageQueueSelector 按 Key 选择固定 Queue + 顺序监听器单线程消费）RabbitMQ 如何保证消息顺序？（单 Consumer + 单 Queue + prefetch_count=1）
