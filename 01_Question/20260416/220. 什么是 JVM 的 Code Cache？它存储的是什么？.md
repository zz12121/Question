**Code Cache 定义**：

```java
/**
 * Code Cache（代码缓存）：
 * 
 * 存储 JIT 编译器将 Java 字节码编译成的本地机器码（Native Code）
 * 
 * 本质：JVM 进程中的一块 Native 内存
 * 不受 -Xmx 控制，属于堆外内存（Off-Heap）
 */
```

## 三、Code Cache 核心参数

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Code Cache 关键参数                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   参数                          默认值（JDK 8）    说明                 │
│   ──────────────────────────────────────────────────────────────────── │
│                                                                         │
│   InitialCodeCacheSize          160KB/160KB       初始大小              │
│   ReservedCodeCacheSize         32MB/48MB        最大预留               │
│   CodeCacheExpansionSize        64KB/128KB       每次扩展量              │
│   CodeCacheMinimumFreeSpace     400KB            最小剩余空间             │
│                                                                         │
│   JDK 8 Tiered：ReservedCodeCacheSize = 48MB                           │
│   JDK 8 Server：ReservedCodeCacheSize = 48MB                            │
│   JDK 11+：ReservedCodeCacheSize = 240MB（更大的默认值）                 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 参数详解

```bash
# InitialCodeCacheSize：JVM 启动时 Code Cache 初始大小
# ──────────────────────────────────────────────────────────────────────
# 太小：启动时就接近满
# 太大：浪费内存
-XX:InitialCodeCacheSize=256KB     # 初始 256KB（必须是 4 的倍数）

# ReservedCodeCacheSize：Code Cache 最大大小（核心参数！）
# ──────────────────────────────────────────────────────────────────────
# 满了之后 JIT 编译被禁用
# 需要根据应用复杂度预估
-XX:ReservedCodeCacheSize=64MB    # 推荐值（默认 48MB 可能不够）

# CodeCacheExpansionSize：每次扩容的增量
# ──────────────────────────────────────────────────────────────────────
-XX:CodeCacheExpansionSize=128KB  # 每次增加 128KB

# 示例：完整的 Code Cache 配置
java -XX:InitialCodeCacheSize=256KB \
     -XX:ReservedCodeCacheSize=96MB \
     -XX:CodeCacheExpansionSize=128KB \
     -XX:+UseCodeCacheFlushing \
     -jar myapp.jar
```

### Code Cache 刷新（Flushing）

```bash
# CodeCacheFlushing：尝试在 Code Cache 满之前清理不常用的代码
# ──────────────────────────────────────────────────────────────────────
-XX:+UseCodeCacheFlushing    # 开启刷新（默认开启）
-XX:-UseCodeCacheFlushing    # 关闭刷新

# 刷新条件（同时满足）：
# 1. Code Cache 使用率 > 2%
# 2. 距离上次刷新 > 30 秒
# 3. 有不常用的已编译方法
```

## 四、如何监控 Code Cache 使用情况

### 4.1 方法一：`-XX:+PrintCodeCache`

```bash
# JVM 退出时打印 Code Cache 使用统计
java -XX:+PrintCodeCache -jar myapp.jar

# 输出示例：
# CodeCache: size=32768Kb, used=25432Kb, free=7336Kb
#   [CodeHeap 'non-nmethods': size=2264Kb used=1187Kb free=1077Kb]
#   [CodeHeap 'nmethods': size=32768Kb used=24345Kb free=8423Kb]
#   [CodeHeap 'profiled nmethods': size=32768Kb used=12456Kb]
#   [CodeHeap 'non-profiled nmethods': size=32768Kb used=11889Kb]
```

### 4.2 方法二：JMX / JConsole

```java
/**
 * 通过 JMX 监控 Code Cache
 */
import java.lang.management.*;
import javax.management.*;

public class CodeCacheMonitor {
    
    public static void monitor() throws Exception {
        // 获取 Code Cache MBean
        MBeanServer mbs = ManagementFactory.getPlatformMBeanServer();
        ObjectName name = new ObjectName("java.lang:type=MemoryPool,name=Code Cache");
        
        MemoryPoolMXBean codeCache = ManagementFactory.newPlatformMXBeanProxy(
            mbs, name.toString(), MemoryPoolMXBean.class);
        
        // 持续监控
        while (true) {
            MemoryUsage usage = codeCache.getUsage();
            long used = usage.getUsed();
            long max = usage.getMax();
            
            double usagePercent = (double) used / max * 100;
            
            System.out.printf("Code Cache: used=%d KB, max=%d KB, usage=%.2f%%%n",
                used / 1024, max / 1024, usagePercent);
            
            if (usagePercent > 90) {
                System.err.println("⚠️ Code Cache 使用率超过 90%！");
            }
            
            Thread.sleep(5000);
        }
    }
}
```

### 5.1 内联参数对比


### 5.2 参数对比表

| 参数 | 控制什么 | 默认值 | 适用场景 |
|-----|---------|-------|---------|
| `-XX:MaxInlineSize` | 所有方法的内联上限 | 35 bytes | 冷方法 |
| `-XX:FreqInlineSize` | 热方法的内联上限 | 325 bytes | 被频繁调用的方法 |
| `-XX:InlineSmallCode` | 小型已编译代码的内联 | ~2000 bytes | 递归内联 |

## 六、Code Cache 调优实战

```java
/**
 * Code Cache 调优的最佳实践
 */
public class CodeCacheTuning {
    
    /**
     * 问题：Code Cache 满了
     * 
     * 症状：
     * - "WARNING: code cache is full"
     * - JIT 编译被禁用
     * - 性能急剧下降
     * 
     * 解决方案：
     */
    
    public void solutions() {
        // 1. 增加 Code Cache 大小
        // -XX:ReservedCodeCacheSize=128MB
        
        // 2. 开启 Code Cache Flushing
        // -XX:+UseCodeCacheFlushing
        
        // 3. 优化应用，减少 JIT 编译需求
        // - 避免大量动态代理
        // - 减少反射调用
        // - 预热应用
        
        // 4. 分层编译调优
        // -XX:+TieredCompilation
        // -XX:TieredStopAtLevel=1  # 只用 C1，快速编译
    }
    
    /**
     * Code Cache 估算公式
     */
    public void estimate() {
        // 估算需要的 Code Cache 大小
        
        // 公式：
        // CodeCache ≈ 方法数 × 平均代码大小 × 编译层级
        
        // 示例：
        // 10000 个方法，平均 100 bytes
        // 分层编译：Tier 1-3 各一份 + Tier 4 一份
        // 总大小 ≈ 10000 × 100 × 2.5 = 2.5 MB
        
        // 保守估算：预留 64MB
        // 激进估算：预留 128MB（大量动态代理）
    }
}
```

## 🔥 扩展追问 1：为什么 Code Cache 不使用 GC？

> **A:** 因为 GC 会引入 STW（Stop-The-World），而 Code Cache 需要持续可用：

```java
/**
 * Code Cache 不 GC 的原因：
 * 
 * 1. JIT 编译是并行的
 *    - GC 会暂停线程
 *    - 暂停期间无法编译新方法
 *    - 影响 JIT 编译器效率
 * 
 * 2. Code Cache 清理策略
 *    - CodeCache Sweeper：后台线程，不 STW
 *    - 只清理不常用的已编译代码
 *    - 保持热点代码活跃
 */
```

## 🔥 扩展追问 2：NMethods vs ProfiledNMethods vs Non-ProfiledNMethods？

> **A:** Code Heap 分区（JDK 9+）：

| Code Heap | 存储内容 | 特点 |
|----------|---------|------|
| `profiled nmethods` | Tier 1-3 编译代码 | 可能去优化 |
| `non-profiled nmethods` | Tier 4 编译代码 | 深度优化，不能去优化 |
| `non-nmethods` | 辅助代码（thunks、vtable） | 基础设施 |

```bash
# JDK 9+ 分区大小控制
-XX:NonProfiledCodeHeapSize=122880KB    # profiled 区
-XX:ProfiledCodeHeapSize=122880KB       # non-profiled 区
```

## 🔥 扩展追问 3：Code Cache 满和 Metaspace 满的区别？

> **A:** 对比表格：

| 对比项 | Code Cache 满 | Metaspace 满 |
|-------|--------------|--------------|
| **存储** | JIT 机器码 | 类元数据 |
| **满的影响** | JIT 禁用，性能下降 | OutOfMemoryError |
| **恢复** | 可恢复（Flushing） | 不可恢复 |
| **GC** | Sweeper 管理 | 类加载器 GC |
| **参数** | ReservedCodeCacheSize | MetaspaceSize |

## 一句话总结

> **Code Cache 是 JIT 编译后的本地机器码存储区（堆外内存），满了之后 JIT 编译被禁用导致性能急剧下降；通过 `-XX:ReservedCodeCacheSize` 控制上限、`-XX:+PrintCodeCache` 监控；`-XX:MaxInlineSize` 控制所有方法内联上限（35字节）、`-XX:FreqInlineSize` 控制热方法内联上限（325字节），热方法可以突破大小限制。**

## 面试加分话术

> "之前压测一个动态代理密集型的服务（MyBatis 插件 + Lombok + 反射），发现 Code Cache 使用率在启动 5 分钟后达到 95%，JIT 编译器被禁用，吞吐量从 8000 TPS 跌到 2000 TPS。最后通过三个方向解决：① 扩容 Code Cache 到 96MB，② 用 `-XX:+UseCodeCacheFlushing` 开启主动清理，③ 关键接口加了 `@FunctionalInterface` 注解让 Lambda 直接内联。这个案例让我理解了 Code Cache 是 JIT 优化的物质基础——Code Cache 不够，再好的优化策略都是空谈。"
