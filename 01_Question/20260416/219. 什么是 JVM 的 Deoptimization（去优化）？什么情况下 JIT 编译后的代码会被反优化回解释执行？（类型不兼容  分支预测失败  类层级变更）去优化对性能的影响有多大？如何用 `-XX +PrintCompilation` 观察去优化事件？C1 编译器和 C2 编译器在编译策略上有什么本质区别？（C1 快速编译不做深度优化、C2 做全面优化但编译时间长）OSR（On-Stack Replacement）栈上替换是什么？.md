## 一、Deoptimization（去优化）是什么？

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    JIT 编译与去优化过程                                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │                                                                 │   │
│   │   Java 源码                                                      │   │
│   │       │                                                          │   │
│   │       ▼                                                          │   │
│   │   ┌─────────────────────────────────────────────────────────┐   │   │
│   │   │                    解释执行阶段                           │   │   │
│   │   │  - 启动阶段（JIT 编译是异步的）                           │   │   │
│   │   │  - 方法被调用一定次数后，进入编译队列                      │   │   │
│   │   └─────────────────────────────────────────────────────────┘   │   │
│   │                      │                                           │   │
│   │                      ▼  JIT 编译完成                             │   │
│   │   ┌─────────────────────────────────────────────────────────┐   │   │
│   │   │                    编译执行阶段                           │   │   │
│   │   │  - 本地代码（Native Code）高速执行                         │   │   │
│   │   │  - 激进优化基于"乐观假设"                                  │   │   │
│   │   └─────────────────────────────────────────────────────────┘   │   │
│   │                      │                                           │   │
│   │                      ▼  ⚠️ 假设被打破                            │   │
│   │   ┌─────────────────────────────────────────────────────────┐   │   │
│   │   │              Deoptimization（去优化）                     │   │   │
│   │   │  - 回退到解释执行                                          │   │   │
│   │   │  - 可能触发重新编译（Recompilation）                       │   │   │
│   │   └─────────────────────────────────────────────────────────┘   │   │
│   │                      │                                           │   │
│   │                      ▼  重新优化                                 │   │
│   │   ┌─────────────────────────────────────────────────────────┐   │   │
│   │   │              重新编译执行（保守优化）                      │   │   │
│   │   └─────────────────────────────────────────────────────────┘   │   │
│   │                                                                 │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│   核心思想：JIT 编译器是"乐观派"——先激进优化，发现错了就回退              │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**去优化定义**：

```java
/**
 * Deoptimization（去优化）：
 * 
 * JIT 编译后的代码被废弃，重新回退到解释执行的过程
 * 
 * 触发原因：
 * - JIT 编译时基于"假设"做了激进优化
 * - 运行时的实际情况打破了这些假设
 * - 继续执行可能导致错误，必须回退
 */
```

## 二、触发去优化的三大场景

### 2.1 场景一：类型不兼容（Monorphic → Megamorphic）

```
┌─────────────────────────────────────────────────────────────────────────┐
│              类型不兼容导致的去优化（Call Site Inline Cache）             │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   JIT 编译优化策略：                                                    │
│   ─────────────────                                                     │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │  单态调用（Monomorphic）：只见过一种类型                           │   │
│   │  ─────────────────────────────────────                           │   │
│   │                                                                 │   │
│   │  callSite(obj):                                                │   │
│   │      if (obj instanceof TypeA) {                               │   │
│   │          call TypeA.method();  ← 内联，无分支，开销 ≈ 0           │   │
│   │      }                                                          │   │
│   │                                                                 │   │
│   │  优化：直接内联，生成更快的机器码                                 │   │
│   │  假设：obj 永远是 TypeA                                          │   │
│   │                                                                 │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │  多态调用（Megamorphic）：见过多种类型                           │   │
│   │  ─────────────────────────────────────                           │   │
│   │                                                                 │   │
│   │  callSite(obj):                                                │   │
│   │      if (obj instanceof TypeA) {                                │   │
│   │          call TypeA.method();                                   │   │
│   │      } else if (obj instanceof TypeB) {                        │   │
│   │          call TypeB.method();                                   │   │
│   │      } else if (...) {                                          │   │
│   │          ...                                                    │   │
│   │      } else {                                                   │   │
│   │          call 虚函数表（运行时绑定）                              │   │
│   │      }                                                          │   │
│   │                                                                 │   │
│   │  触发：见到第 2+ 种类型 → 去优化！                               │   │
│   │  原因：假设被打破，之前的内联代码错误                             │   │
│   │                                                                 │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**代码示例**：

```java
/**
 * 类型不兼容导致去优化的经典案例
 */
public class TypeDeoptExample {
    
    interface Callback {
        void invoke();
    }
    
    static class TypeA implements Callback {
        @Override public void invoke() { /* do A */ }
    }
    
    static class TypeB implements Callback {
        @Override public void invoke() { /* do B */ }
    }
    
    // 第一次调用时，只有 TypeA
    // JIT 编译为：直接内联 TypeA.invoke()，无虚调用开销
    public static void main(String[] args) {
        Callback cb = new TypeA();
        for (int i = 0; i < 100_000; i++) {
            cb.invoke();
        }
        // JIT 认为"只有 TypeA"，做激进内联
        
        // 第二次赋值：TypeB
        cb = new TypeB();  // ← 打破假设！
        cb.invoke();       // ← 去优化！之前的编译代码不可用
    }
}
```

### 2.2 场景二：分支预测失败（Branch Prediction Misfire）

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    分支预测失败导致的去优化                              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   JIT 编译优化策略：                                                    │
│   ─────────────────                                                     │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │                                                                 │   │
│   │  for (int i = 0; i < 1_000_000; i++) {                         │   │
│   │      if (i < 100) {           // ← 分支 A                      │   │
│   │          doFastPath(i);                                        │   │
│   │      } else {                                                  │   │
│   │          doSlowPath(i);       // ← 分支 B                       │   │
│   │      }                                                          │   │
│   │  }                                                              │   │
│   │                                                                 │   │
│   │  JIT 分析：99.99% 走进 A                                         │   │
│   │  假设：A 是热路径，B 几乎不执行                                   │   │
│   │  优化：编译为只执行 A，B 分支视为 unreachable                    │   │
│   │                                                                 │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│   ⚠️ 打破假设：运行到 100000 次后，进入 B 分支                          │
│   ─────────────────────────────────────────────────────                 │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │                                                                 │   │
│   │  预期：100 次 A，900000 次 B                                     │   │
│   │  实际：100 次 A，899900 次 A，100 次 B  ← 分布反转！             │   │
│   │                                                                 │   │
│   │  问题：编译代码只优化了 A，没处理 B                              │   │
│   │  解决：去优化！重新解释执行或保守编译                             │   │
│   │                                                                 │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**代码示例**：

```java
/**
 * 分支预测失败导致去优化
 */
public class BranchDeoptExample {
    
    static boolean flag = false;
    
    public static void main(String[] args) {
        // Warmup：前 10000 次，flag 总是 false
        for (int i = 0; i < 10_000; i++) {
            if (flag) {  // JIT 认为永远不会执行
                doSomething();
            }
        }
        
        // ⚠️ 运行时改变：打破 JIT 的假设
        flag = true;  // JIT 的假设被打破！
        
        // 这个调用会触发去优化
        for (int i = 0; i < 100; i++) {
            if (flag) {
                doSomething();
            }
        }
    }
}
```

### 2.4 第四类：逆优化计数器触发

```java
/**
 * JIT 编译基于计数器触发优化
 * 当优化条件不再满足时，触发逆优化
 */
public class CounterDeopt {
    
    // JIT 编译阈值
    // -XX:CompileThreshold=10000 (默认)
    
    public static void main(String[] args) {
        for (int i = 0; i < 1_000_000; i++) {
            work(i);
        }
        
        // JIT 认为：work() 一直是"热"的
        // 假设：i 总是正数（基于前 100 万次观察）
        
        // ⚠️ 运行到 i=0，work(0) 可能崩溃
        // → 触发 trap-based deoptimization
    }
    
    // -XX:+TraceDeoptimization 查看去优化事件
}
```

## 三、去优化对性能的影响

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    去优化的性能影响                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  性能损失估算：                                                  │   │
│  │  ──────────────                                                  │   │
│  │                                                                 │   │
│  │  编译代码执行：        1x (基准)                                 │   │
│  │  解释执行：            10-100x 慢                                │   │
│  │  去优化开销：          ~1ms (STW + 解释恢复)                      │   │
│  │                                                                 │   │
│  │  ┌─────────────────────────────────────────────────────────┐   │   │
│  │  │  触发瞬间：                                              │   │   │
│  │  │  ───────────                                            │   │   │
│  │  │                                                          │   │   │
│  │  │  1. 找到已编译代码                                       │   │   │
│  │  │  2. 找到解释器帧（Interpreter Frame）                     │   │   │
│  │  │  3. 保存编译代码的寄存器状态                              │   │   │
│  │  │  4. 恢复解释器状态                                        │   │   │
│  │  │  5. 跳转到解释器执行                                      │   │   │
│  │  │                                                          │   │   │
│  │  │  耗时：0.5 - 5ms（需要安全点 SafePoint）                  │   │   │
│  │  │  影响：高并发场景下可能影响吞吐量                          │   │   │
│  │  └─────────────────────────────────────────────────────────┘   │   │
│  │                                                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  典型影响：                                                            │
│  ─────────                                                            │
│  - 单次去优化：影响 < 10ms（通常可接受）                                │
│  - 频繁去优化：累计性能损失可达 30-50%                                   │
│  - 服务延迟毛刺：P99 延迟升高                                            │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**去优化类型**：

| 类型 | 触发条件 | 开销 |
|-----|---------|------|
| **Trapped Deoptimization** | 运行时检查失败（如类型检查） | 低 |
| **Not Entrant** | 代码不再有效（如类卸载） | 中 |
| **Zombie** | 替换后变为僵尸代码 | 低 |
| **Recompilation** | 触发重新编译 | 高 |

## 四、使用 `-XX:+PrintCompilation` 观察去优化

```bash
# 基础用法
java -XX:+PrintCompilation -XX:+UnlockDiagnosticVMOptions MyApp

# 详细输出（包含去优化原因）
java -XX:+PrintCompilation -XX:+UnlockDiagnosticVMOptions \
     -XX:CompileCommand=print,*::method MyApp

# 完整选项
java -XX:+PrintCompilation \
     -XX:+PrintCompilation2 \
     -XX:+PrintInlining \
     -XX:+LogCompilation \
     -XX:LogFile=compilation.log \
     MyApp
```

### 4.1 输出解析

```
# 示例输出
# 格式：timestamp n method [bc] [CompileID] [CompileKind] [Comment]

9834  1       3       java.lang.Object::hashCode (20 bytes)
12545  2       4       com.example.MyClass::compute (45 bytes)   made not entrant
14567  3       4       com.example.MyClass::compute (45 bytes)   COMPILED

解释：
──────────────────────────────────────────────────────────────
n                   ← 方法编译完成（not entrant = 去优化）
method              ← 方法名
bc                  ← bytecode 大小
CompileID           ← 编译ID
CompileKind         ← 编译类型
                    ← made not entrant = 被去优化！
Comment             ← 去优化原因
──────────────────────────────────────────────────────────────
```

### 4.2 去优化标志解读

```bash
# 去优化相关的输出标志
─────────────────────────────────────────────────────────────
made not entrant    ← 代码被标记为不再执行（去优化）
made zombie         ← 代码变为僵尸（已被替换）
uncommon trap       ← 不常见陷阱（分支预测失败）
deoptimize          ← 主动去优化
nmethod removal     ← 编译代码被移除
stack change        ← 栈变化（OSR 相关）
─────────────────────────────────────────────────────────────
```

### 4.3 实战：分析去优化

```bash
# 运行命令
$ java -XX:+PrintCompilation -XX:+UnlockDiagnosticVMOptions \
       -XX:+TraceDeoptimization \
       -jar myapp.jar 2>&1 | grep -i deopt

# 示例输出：
[info][deoptimization] 84532 Thread[main] UncommonTrap 1 @ bci:15 reason=predicate
[info][deoptimization] 84532 Method=com/example/Service::process (100 bytes)
[info][deoptimization] Reason=predicate (分支预测失败)
[info][deoptimization] Action=reinterpret

解释：
──────────────────────────────────────────────────────────────
reason=predicate     ← 分支预测失败
reason=unloaded      ← 类卸载
reason=type_check    ← 类型检查失败
reason=null_check    ← 空指针检查
reason=range_check   ← 范围检查失败
──────────────────────────────────────────────────────────────
```

## 五、C1 vs C2 编译器对比

```

### 5.1 核心对比

| 特性 | C1 (Client) | C2 (Server) |
|-----|------------|-------------|
| **定位** | 客户端，交互式应用 | 服务端，高吞吐量 |
| **编译速度** | 快（秒级） | 慢（分钟级） |
| **优化程度** | 轻量级 | 深度优化 |
| **代码质量** | 中等 | 最优 |
| **启动时间** | 短 | 长 |
| **峰值性能** | 一般 | 最优 |
| **内存开销** | 小 | 大 |

### 5.2 C1 vs C2 优化能力对比

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    C1 vs C2 优化能力对比                                │

│   │   - 常量折叠                                                     │   │

### 5.3 源码级：C1 vs C2 优化示例

```java
/**
 * C1 vs C2 对同一段代码的优化差异
 */
public class CompilerDifference {
    
    // JIT 编译前
    public int sum(int[] arr) {
        int sum = 0;
        for (int i = 0; i < arr.length; i++) {
            sum += arr[i];  // 循环内，可能向量化
        }
        return sum;
    }
    
    /**
     * C1 优化后（Tier 1-3）：
     * - 常量折叠
     * - 简单内联
     * - 基本的循环优化
     * 
     * C2 优化后（Tier 4）：
     * - SIMD 向量化：sum += arr[i] → AVX 指令
     * - 循环展开：每次迭代处理多个元素
     * - 逃逸分析：数组不逃逸，直接栈分配
     * - 激进内联：调用者的 caller 也可能被内联
     */
}
```

### 6.1 OSR 触发条件

```java
/**
 * OSR 通常触发于：
 * 
 * 1. 超长循环（如 for 循环 100 万次）
 *    - 方法很短，但循环很长
 *    - JIT 在方法执行过程中编译完成
 * 
 * 2. 热循环中的代码
 *    - 循环体被识别为热点
 *    - 编译后的循环替换当前栈帧
 */
public class OSRTriggerExample {
    
    public static void main(String[] args) {
        long sum = 0;
        
        // Warmup: 解释执行
        for (int i = 0; i < 10_000; i++) {
            sum += compute(i);
        }
        
        // OSR 触发：循环体很长，JIT 在循环中完成编译
        for (int i = 0; i < 10_000_000; i++) {
            // OSR 点：在这个循环中途切换到编译代码
            sum += compute(i);
        }
    }
    
    // -XX:+PrintCompilation 输出会显示：
    // "OSR compile 5 com.example.OSRTriggerExample::compute @ 123"
    // "OSR code begin ..."
}
```

### 6.2 OSR vs 普通编译

| 特性 | 普通编译 | OSR 编译 |
|-----|---------|---------|
| **触发时机** | 方法入口 | 循环中途 |
| **栈帧替换** | 无 | 在栈中间替换 |
| **复杂度** | 简单 | 高（状态迁移） |
| **使用场景** | 普通方法 | 长循环、热循环 |
| **性能提升** | 正常 | 显著（避免长时间解释执行） |

## 🔥 扩展追问 1：逃逸分析（Escape Analysis）导致的去优化？

> **A:** 是的，逃逸分析是最常见的去优化来源之一：

```java
/**
 * 逃逸分析：判断对象是否逃逸出方法/线程
 * 
 * 优化：对象不逃逸 → 栈上分配 → 减少 GC 压力
 * 
 * 但 JIT 可能判断错误 → 去优化！
 */
public class EscapeDeopt {
    
    public static void main(String[] args) {
        // Warmup：a 不逃逸，栈上分配
        for (int i = 0; i < 100_000; i++) {
            allocateNoEscape(i);
        }
        
        // ⚠️ 去优化点：分配对象被外部引用
        allocateNoEscape(Integer.valueOf(42));
        
        // 如果 JIT 假设错误 → 去优化
    }
    
    static int allocateNoEscape(Integer i) {
        int[] arr = new int[100];  // 假设不逃逸，栈上分配
        arr[0] = i;
        return arr[0];
    }
}
```

## 一句话总结

> **Deoptimization 是 JIT 激进优化失败后的回退机制，触发于类型不兼容/分支预测失败/类层级变更等假设被打破；C1 快速轻量适合启动，C2 深度优化适合峰值性能，分层编译结合两者优点；OSR 栈上替换让长循环中途切换到编译代码避免长时间解释执行；`-XX:+PrintCompilation` 可观察去优化事件。**

## 面试加分话术

> "之前用 JMH 压测一个高性能计算模块时，发现 P99 延迟有个奇怪的毛刺，最后定位是去优化问题——`List<String>` 在循环中途从 `ArrayList` 变成了 `LinkedList`（虽然代码没写，但动态加载的类影响了 JIT 的类型推断）。加了 `-XX:+PrintCompilation -XX:+TraceDeoptimization` 后确认了问题。最后通过重构代码避免泛型类型变化解决。去优化不是 Bug，而是 JIT 乐观优化的代价，关键是让它尽可能少发生。"
