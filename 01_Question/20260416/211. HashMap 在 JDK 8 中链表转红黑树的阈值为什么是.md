为什么是 8？—— 基于泊松分布的概率计算

泊松分布公式：
P(X=k) = (λ^k * e^(-λ)) / k!

其中 λ 是平均每个桶的链表长度（对于 HashMap，λ ≈ 0.5）

原因：HashMap 的负载因子是 0.75
      当数组容量为 n 时，期望的链表长度 λ ≈ 0.5
      （因为 n * (1 - e^(-0.75)) ≈ n * 0.527 ≈ n/2）

final void treeifyBin(Node<K,V>[] tab, int hash) {
    int n, index;
    Node<K,V> e;
    
    // 如果 table 容量 < 64，优先扩容而不是树化
    // 这是为了避免在小数组上树化（得不偿失）
    if (tab == null || (n = tab.length) < MIN_TREEIFY_CAPACITY)

    else if ((e = tab[index = (n - 1) & hash]) != null) {
        // 如果链表长度 >= TREEIFY_THRESHOLD (8)，转红黑树
        TreeNode<K,V> hd = null, tl = null;
        do {
            TreeNode<K,V> p = replacementTreeNode(e, null);
            if (tl == null)
                hd = p;
            else {
                p.prev = tl;
                tl.next = p;
            }
            tl = p;
        } while ((e = e.next) != null);
        // 把链表节点转换为 TreeNode，构建红黑树
        if ((tab[index] = hd) != null)

    }
}

## 🔥 扩展追问 1：为什么退化阈值是 6 而不是 8？两者之间的"间隙"是干嘛用的？

> A: **退化和树化用不同的阈值是为了避免在临界点（8）频繁转换。设置间隙 6 vs 8 意味着链表长度在 7 时不会触发退化，必须降到 6 才退化，这样避免在 8 附近反复树化/退化。**

/*
 * Implementation notes.
 *
 * This map usually acts as a binned (bucketed) hash table, but
 * when bins get too large, they are transformed into bins of
 * TreeNodes, each structured similarly to those in java.util.TreeMap.
 * Most methods try to use normal bins, but on recourse to
 * TreeNode methods (which is relatively uncommon). The bins of
 * TreeTables may be traversed and used like any others, but
 * additionally support faster lookup when they contain more than
 * TREEIFY_THRESHOLD nodes. Because TreeNodes are about twice the
 * size of regular nodes, we use them only when bins contain

## 🔥 扩展追问 4：HashMap 中红黑树的查找是如何利用 Comparable 或 Comparator 的？

> A: **HashMap 通过 `ComparableClassFor()` 和 `compareComparables()` 两个方法，在红黑树节点插入/查找时，利用 key 的自然排序或自定义比较器来决定左右子树的方向。**

```java
// JDK 8 HashMap 源码核心逻辑

// 1. 判断 key 是否实现了 Comparable
static Class<?> comparableClassFor(Object x) {
    if (x instanceof Comparable) {
        // 检查 x 是否实现了 Comparable<K>（而非 raw Comparable）
        if (c == (c = x.getClass()) && c == String.class) {
            return c;  // String 特殊优化
        }
        // 检查 x.getClass() 是否实现了 Comparable<K>
        if (c.getGenericInterfaces() != null) {
            for (Type iface : c.getGenericInterfaces()) {
                if (iface instanceof ParameterizedType &&
                    ((ParameterizedType)iface).getRawType() == Comparable.class &&
                    comparable.compare(
                        ((ParameterizedType)iface).getActualTypeArguments()[0],
                        c) == 0) {
                    return c;
                }
            }
        }
    }
    return null;  // 未实现 Comparable
}

// 2. 比较两个对象（利用 Comparable）
static int compareComparables(Class<?> kc, Object p, Object x) {
    return (x == null ? 0 :
            (p.getClass() == kc ||
             kc == (kc = comparableClassFor(p))) ?
                ((Comparable) p).compareTo(x) : 0);
}

// 3. TreeNode.find() 方法（红黑树查找）
final TreeNode<K,V> find(int h, Object k, Class<?> kc) {
    TreeNode<K,V> p = this;
    do {
        int ph, dir;
        K pk;
        TreeNode<K,V> pl = p.left, pr = p.right, q;
        
        // 根据 hash 大小决定向左还是向右
        if ((ph = p.hash) > h)
            p = pl;  // 当前节点 hash 更大，向左
        else if (ph < h)
            p = pr;  // 当前节点 hash 更小，向右
        else if ((pk = p.key) == k || (k != null && k.equals(pk)))
            return p;  // hash 相等，再用 equals 比较
        
        // hash 相等但 key 不相等，尝试用 Comparable 比较
        else if (pl != null &&
                 (q = pl.find(h, k, kc)) != null)
            return q;  // 左子树找到了
        else if (pr != null)
            p = pr;  // 右子树找
        else
            return null;  // 没找到
    } while (p != null);
    return null;
}

// 4. TreeNode.treeify() 方法（链表转红黑树）
final void treeify(Node<K,V>[] tab) {
    TreeNode<K,V> root = null;
    
    for (Node<K,V> e = this; e != null; e = e.next) {
        TreeNode<K,V> p = new TreeNode<>(...);
        
        // 构建红黑树
        if (root == null) {
            p.parent = null;
            p.red = false;  // 根节点设为黑色
            root = p;
        } else {
            K k = e.key;
            int h = e.hash;
            Class<?> kc = null;
            
            // 寻找插入位置
            TreeNode<K,V> parent;
            do {
                parent = p;
                
                // 先比较 hash
                int dir = (ph = p.hash) > h ? -1 : (ph < h ? 1 : 0);
                
                // hash 相等时，尝试用 Comparable 比较
                if (dir == 0) {
                    if (kc == null)
                        kc = comparableClassFor(k);
                    if (kc != null)
                        dir = compareComparables(kc, k, p.key);
                    else
                        dir = tieBreakOrder(k, p.key);  // 兜底：按 identityHashCode 排序
                }
                
                p = dir < 0 ? p.left : p.right;  // 根据比较结果决定方向
            } while (p != null);
            
            // 插入节点
            e.parent = parent;
            if (dir < 0)
                parent.left = e;
            else
                parent.right = e;
            
            // 平衡红黑树
            root = balanceInsertion(root, e);
        }
    }
    
    // 移动根节点到链表头
    moveRootToFront(tab, root);
}

// 5. tieBreakOrder（hash 相等时的兜底排序）
static int tieBreakOrder(Object a, Object b) {
    int d;
    // 按类名比较
    d = a.getClass().getName().compareTo(b.getClass().getName());
    // 类名相同则按 identityHashCode
    if (d == 0)
        d = System.identityHashCode(a) - System.identityHashCode(b);
    return d;
}

┌────────────────────────────────────────────────────────────────────────────┐
│  HashMap 红黑树查找（TreeNode.find）                                        │
│                                                                            │
│  输入：hash = h, key = k                                                   │
│                                                                            │
│  ┌────────────────────────────────────────────────────────────────────┐   │
│  │  Step 1: 比较 hash 值                                              │   │
│  │       ph = p.hash                                                  │   │
│  │       if ph > h → 向左子树找 (p = p.left)                         │   │
│  │       if ph < h → 向右子树找 (p = p.right)                        │   │
│  │       if ph == h → 进入 Step 2                                     │   │
│  └────────────────────────────────────────────────────────────────────┘   │
│                                    │                                        │
│                                    ▼                                        │
│  ┌────────────────────────────────────────────────────────────────────┐   │
│  │  Step 2: hash 相等时，用 equals 比较 key                           │   │
│  │       if (pk == k || k.equals(pk)) → 找到！返回 p                 │   │
│  │       else → 进入 Step 3                                          │   │
│  └────────────────────────────────────────────────────────────────────┘   │
│                                    │                                        │
│                                    ▼                                        │
│  ┌────────────────────────────────────────────────────────────────────┐   │
│  │  Step 3: hash 相等且 equals 不匹配，尝试 Comparable 比较           │   │
│  │       kc = comparableClassFor(k)                                   │   │
│  │       if kc != null → 用 Comparable.compareTo 比较                │   │
│  │           dir < 0 → 向左子树找                                      │   │
│  │           dir > 0 → 向右子树找                                      │   │
│  │       else → 进入 Step 4（兜底）                                   │   │
│  └────────────────────────────────────────────────────────────────────┘   │
│                                    │                                        │
│                                    ▼                                        │
│  ┌────────────────────────────────────────────────────────────────────┐   │
│  │  Step 4: 兜底排序（tieBreakOrder）                                 │   │
│  │       按类名排序 → 按 identityHashCode 排序                        │   │
│  └────────────────────────────────────────────────────────────────────┘   │
│                                                                            │
│  为什么需要这么多层兜底？                                                  │
│  因为 hash 相等的 key 可能：                                               │
│  ① hashCode 相同但 equals 不相同（哈希冲突）                               │
│  ② 一个实现了 Comparable，一个没有                                        │
│  ③ 两个都实现了 Comparable 但 compareTo 返回 0                           │
└────────────────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 5：红黑树的增删查时间复杂度都是 O(log n) 吗？具体是多少？

## 3. 一句话总结

> **HashMap 链表转红黑树的阈值是 8，基于泊松分布（λ≈0.5）计算得出；退化为 6 而非 8 是为了避免临界点频繁转换抖动；红黑树查找通过 hash 比较 → equals 比较 → Comparable 比较 → identityHashCode 兜底四层逻辑处理哈希冲突。**

## 4. 面试加分话术

> "我之前看 HashMap 源码时对 8 这个数字印象特别深。JDK 团队其实是做了概率计算的——基于泊松分布，在负载因子 0.75 下，链表长度达到 8 的概率不到万分之一，所以 8 本质上是一个安全边界。真正让我觉得设计巧妙的是 6 和 8 的间隙——如果退化阈值也是 8，那在临界点反复增删就会导致树和链表来回切换，白白损耗性能。设置了 6 作为退化阈值后，链表要掉到 6 以下才退化，7 这个中间值只进不退，就稳住了。另外 HashMap 红黑树的查找逻辑我也研究过，treeify 时会先用 comparableClassFor 判断 key 是否实现了 Comparable，找的时候会先比 hash 再比 equals，最后才用 Comparable 的 compareTo，兜底是 identityHashCode，这个层层降级的设计很值得借鉴——既保证了大部分情况走 fast path，又保证了 corner case 不会漏掉。"
