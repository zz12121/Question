## 1. type 列：从好到差的完整排序

```
┌─────────────────────────────────────────────────────────────────────┐
│                 MySQL EXPLAIN type 列 —— 性能天梯                   │
│                                                                     │
│   最优 ──────────────────────────────────────────────────────▶ 最差 │
│                                                                     │
│  system  >  const  >  eq_ref  >  ref  >  ref_or_null  >           │
│    │        │         │        │           │                      │
│    │        │         │        │           │                      │
│    │        │         │        │           └── 加上NULL值搜索       │
│    │        │         │        │                                   │
│    │        │         │        └───────── 索引等值查找              │
│    │        │         │                                             │
│    │        │         └──────────────── 唯一索引等值（每行只返回1条） │
│    │        │                                                       │
│    │        └────────────────────── 主键/唯一索引等值（只匹配1条）   │
│    │                                                                     │
│    └───────────────────────────── 系统表（极少出现，元数据查询）        │
│                                                                     │
│  ─────────────────────────────────────────────────────────────    │
│                                                                     │
│  range  >  index  >  ALL                                              │
│    │        │         │                                              │
│    │        │         └────────── 全表扫描（性能灾难）                │
│    │        │                                                        │
│    │        └────────────────── 全索引扫描（无需读数据文件）            │
│    │                                                                     │
│    └───────────────────────── 索引范围扫描（ BETWEEN/IN/>/< 等）        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.1 每种 type 的含义与示例

| type | 含义 | 出现条件 | SQL 示例 |
|------|------|----------|----------|
| **system** | 系统表，只有1行 | 查询MySQL系统表 | `SELECT * FROM mysql.time_zone` |
| **const** | 主键/唯一索引等值，命中**1行** | WHERE pk = 常量 | `SELECT * FROM users WHERE id = 1` |
| **eq_ref** | JOIN时，被驱动表用主键/唯一索引等值，命中**1行** | JOIN中驱动表匹配1行 | `SELECT * FROM orders o JOIN users u ON u.id = o.user_id` |
| **ref** | 普通索引等值查找，命中**多行** | WHERE idx_col = 常量 | `SELECT * FROM users WHERE name = 'Tom'` |
| **ref_or_null** | ref + 额外扫描NULL值 | WHERE idx_col = x OR idx_col IS NULL | `SELECT * FROM users WHERE name = 'Tom' OR name IS NULL` |
| **range** | 索引范围扫描 | WHERE id > 10 / IN (1,2,3) / BETWEEN | `SELECT * FROM users WHERE id BETWEEN 100 AND 200` |
| **index** | 全索引扫描 | 扫描整个索引树 | `SELECT name FROM users`（覆盖索引） |
| **ALL** | 全表扫描 | 无索引可用或强制走全表 | `SELECT * FROM users WHERE email LIKE '%@gmail.com'` |

**注意：range 和 index 的性能差距**

```
range（有范围条件）：
  → 只扫描索引树的一部分
  → 找到起始位置 → 顺序扫描到结束位置
  → I/O: O(log n + k)，k=结果集行数

index（无范围条件全扫描）：
  → 扫描整个索引 B+Tree 的叶子节点
  → 虽然不用读数据行，但还是要遍历所有索引项
  → I/O: O(n)

"index 只是看起来比 ALL 快，因为不需要回表，但仍然是性能杀手"
```

## 2. Extra 列：各值的含义与性能影响

### 2.1 完整图谱

```
┌──────────────────────────────────────────────────────────────────────┐
│                    Extra 列 —— 性能指示灯                            │
├──────────────────┬───────────────────┬──────────────────────────────┤
│    ✅ 好（推荐）   │   ⚠️ 警告（需优化）  │   ❌ 差（必须优化）           │
├──────────────────┼───────────────────┼──────────────────────────────┤
│ Using index       │ Using index       │ Using filesort               │
│ (覆盖索引)         │ condition (ICP)   │ (文件排序)                   │
│                  │ (索引下推)         │                               │
│                  │                    │ Using temporary              │
│                  │ Using where        │ (创建临时表)                  │
│                  │ (额外过滤)         │                               │
│                  │                    │ Using join buffer            │
│                  │                     │ (BNL/BKA)                   │
└──────────────────┴───────────────────┴──────────────────────────────┘
```

### 2.2 逐项详解

#### Using index（覆盖索引，最优）

```
┌─────────────────────────────────────────────────────────────┐
│  场景：SELECT name, email FROM users WHERE name = 'Tom'    │
│                                                             │
│  索引：(name, email)                                        │
│                                                             │
│  执行过程：                                                 │
│                                                             │
│  name='Tom' ──▶ 命中索引 ──▶ 直接返回 name 和 email          │
│                        │                                    │
│                        └── 无需回表（Using index）           │
│                                                             │
│  ┌─────────┐     ┌─────────┐     ┌─────────────────────┐  │
│  │ 索引树   │────▶│ 索引页   │────▶│ 直接拿到结果（不访问 │  │
│  │ (name,  │     │ name+email│    │ 主表数据页)         │  │
│  │  email) │     │          │    │                     │  │
│  └─────────┘     └─────────┘     └─────────────────────┘  │
│                                                             │
│  优势：I/O次数最少（只读索引树），避免回表随机I/O              │
└─────────────────────────────────────────────────────────────┘
```

**覆盖索引设计原则：**

```sql
-- 慢查询：SELECT id, name, email, phone FROM users WHERE name = 'Tom'
-- 索引：(name)  → 需要回表获取 email, phone, id

-- 优化后：SELECT id, name, email, phone FROM users WHERE name = 'Tom'
-- 索引：(name, email, phone)  → Using index，无需回表

-- 经典场景：分页统计
SELECT COUNT(*) FROM orders WHERE status = 'paid'
-- 索引：(status) → Using index，直接在索引上 COUNT
```

#### Using index condition（ICP，索引下推）

```
┌─────────────────────────────────────────────────────────────┐
│  场景：SELECT * FROM users WHERE name LIKE 'Tom%' AND age = 25 │
│                                                             │
│  索引：(name, age)                                          │
│                                                             │
│  无ICP（MySQL 5.6之前）：                                    │
│  1. 通过 'Tom%' 在索引树定位到 'Tom' 前缀的所有记录          │
│  2. 回表拿到完整行                                           │
│  3. 在服务层过滤 age = 25                                    │
│     → 大量无效回表（age≠25的记录也回表了）                   │
│                                                             │
│  有ICP（MySQL 5.6+）：                                      │
│  1. 在索引树定位到 'Tom' 前缀位置                            │
│  2. **在索引层面过滤 age = 25**（Using index condition）    │
│  3. 只对满足两个条件的记录回表                                │
│     → 回表次数大幅减少                                        │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  索引(name, age)                                     │   │
│  │                                                      │   │
│  │   [Tom,22] [Tom,25] [Tom,30] [Tony,25] [Tony,30]   │   │
│  │            ↑        ↑                               │   │
│  │            用ICP筛选  用ICP筛选                      │   │
│  │            age=25    age=25                         │   │
│  │            ↓        ↓                               │   │
│  │            回表      回表                            │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**ICP 开启条件：**

```sql
-- MySQL 5.6+ 默认开启
SHOW VARIABLES LIKE 'optimizer_switch';
-- index_condition_pushdown=on 表示开启

-- ICP 生效条件：
-- 1. 索引是联合索引 (a, b, c)
-- 2. WHERE 条件使用了索引前缀列
-- 3. WHERE 条件中包含非索引列（需要回表后过滤）
-- 4. 存储引擎支持 ICP（InnoDB 支持，MyISAM 支持）
```

#### Using where（额外过滤）

```
含义：索引定位后，需要在服务层用 WHERE 条件额外过滤

示例：
SELECT * FROM users WHERE name = 'Tom' AND age > 25

索引：(name)    -- 只能用到 name
                                        │
                                        ▼
                               命中 name='Tom' 的所有行
                               (可能100行)
                                        │
                                        ▼
                              在服务层过滤 age > 25
                              (Using where)
                                        │
                                        ▼
                               最终返回10行

⚠️ 单独出现 Using where 不一定是坏事：
   → 关键看是否还有其他Extra（Using filesort / Using temporary）
   → 如果只有 Using where + type=ref，说明索引利用正确
   → 真正的问题是有 Using where + type=ALL（无索引可用）
```

#### Using filesort（文件排序，⚠️警告）

```
┌─────────────────────────────────────────────────────────────┐
│  什么情况下触发？                                            │
│                                                             │
│  需要排序，但索引无法提供有序结果时                            │
│                                                             │
│  案例1：SELECT * FROM orders ORDER BY create_time DESC     │
│         索引：(name, age)  → 无法按 create_time 有序        │
│                                                             │
│  案例2：SELECT * FROM orders WHERE status = 'paid'         │
│         ORDER BY amount DESC                                │
│         索引：(status, create_time)                         │
│         → 虽然status等值，但ORDER BY的是不同列amount         │
│         → 无法利用索引顺序                                   │
│                                                             │
│  案例3：SELECT * FROM users WHERE age > 20 ORDER BY age     │
│         索引：(age)  → age是范围查询，索引有序性被破坏        │
│                                                             │
└─────────────────────────────────────────────────────────────┘

filesort 原理（两种算法）：

┌─────────────────────────────────────────────────────────────┐
│  MySQL 8.0+ 两种模式：                                      │
│                                                             │
│  1. <  mediumtext/tinytext/lob字段>                        │
│     → rowid sort（默认）                                    │
│     → 只在 sort_buffer 中存 (排序列, 主键)                    │
│     → 需要回表                                               │
│     → 优点：sort_buffer 小                                  │
│     → 缺点：回表次数多                                       │
│                                                             │
│  2. 全字段 sort（当所有列都在索引中 or max_length_for_sort_data够大）│
│     → sort_buffer 存完整行                                  │
│     → 无需回表                                               │
│     → 优点：避免回表                                         │
│     → 缺点：sort_buffer 占用大                              │
│                                                             │
│  双路排序 vs 单路排序：                                      │
│  - 双路（< MySQL 4.1）：读行 → 排序 → 读行 → 返回            │
│  - 单路（≥ MySQL 4.1）：读行 → 排序 → 直接返回              │
└─────────────────────────────────────────────────────────────┘
```

**优化 filesort 的策略：**

```sql
-- 策略1：创建合适的索引让结果自然有序
-- 原始：SELECT * FROM orders ORDER BY create_time DESC
-- 索引：ALTER TABLE orders ADD INDEX idx_create_time (create_time DESC);

-- 策略2：用索引覆盖减少排序量
-- 原始：SELECT * FROM orders ORDER BY create_time  -- 100列
-- 优化：SELECT id, create_time FROM orders ORDER BY create_time
--      然后根据 id 回表拿需要的列

-- 策略3：控制排序缓冲区大小
SHOW VARIABLES LIKE 'sort_buffer_size';  -- 默认 256KB
SET GLOBAL sort_buffer_size = 4194304;     -- 4MB（临时生效）

-- 策略4：利用 ASC/DESC 混合索引优化多字段排序
CREATE INDEX idx_status_date ON orders(status, create_time DESC);
SELECT * FROM orders WHERE status = 'paid' ORDER BY create_time DESC;
-- 可以利用索引直接有序返回，无需 filesort
```

#### Using temporary（创建临时表，❌差）

```
触发条件：
1. DISTINCT 查询去重
2. GROUP BY 使用了非索引列
3. UNION / UNION ALL（需要去重）
4. ORDER BY 和 GROUP BY 的列不同

┌─────────────────────────────────────────────────────────────┐
│  案例：SELECT name, COUNT(*) FROM users GROUP BY name      │
│                                                             │
│  索引：(name)                                               │
│                                                             │
│  执行过程：                                                 │
│  1. 扫描 users 表                                          │
│  2. 对每个 name 值创建临时表条目                             │
│  3. 聚合计数                                                │
│  4. 返回结果                                                │
│  5. 删除临时表                                              │
│                                                             │
│  优化方案：                                                 │
│  1. GROUP BY name 加上合适的索引                            │
│     ALTER TABLE users ADD INDEX idx_name (name);           │
│  2. 如果 group by 的列不在 select 中，考虑去掉              │
│  3. 尽量用索引覆盖，避免回表                                 │
└─────────────────────────────────────────────────────────────┘
```

**Using temporary + Using filesort 同时出现 = 最糟糕的情况：**

```sql
-- 灾难SQL：
SELECT * FROM orders
WHERE DATE(create_time) = '2024-01-01'
GROUP BY status
ORDER BY amount DESC;

-- 问题：
-- 1. DATE(create_time) 无法使用索引 → type=ALL
-- 2. GROUP BY status 非索引列 → Using temporary
-- 3. ORDER BY amount DESC 非索引列 → Using filesort

-- 优化后：
ALTER TABLE orders ADD INDEX idx_date_status_amount (create_time, status, amount);
SELECT * FROM orders
WHERE create_time >= '2024-01-01 00:00:00'
  AND create_time < '2024-01-02 00:00:00'
GROUP BY status
ORDER BY amount DESC;
```

## 3. key_len 计算

### 3.1 计算公式速查表

```
┌─────────────────────────────────────────────────────────────────────┐
│                    key_len 计算公式                                  │
│                                                                     │
│  基本类型：                                                          │
│  ─────────────────────────────────────────────────────────────     │
│  TINYINT      →  1  字节                                             │
│  SMALLINT     →  2  字节                                             │
│  INT          →  4  字节                                             │
│  BIGINT       →  8  字节                                             │
│  FLOAT        →  4  字节                                             │
│  DOUBLE       →  8  字节                                             │
│  DATE         →  3  字节                                             │
│  DATETIME     →  8  字节                                             │
│  TIMESTAMP    →  4  字节                                             │
│                                                                     │
│  字符串类型（重要！）：                                               │
│  ─────────────────────────────────────────────────────────────     │
│  CHAR(n)      →  n × 字符集每个字符最大字节数                         │
│                 CHAR(10) utf8mb4 → 10 × 4 = 40 字节                  │
│                                                                     │
│  VARCHAR(n)   →  n × 字符集每个字符最大字节数 + 2（长度前缀）          │
│                 VARCHAR(50) utf8mb4 → 50 × 4 + 2 = 202 字节           │
│                                                                     │
│  NULL 和定长补偿：                                                   │
│  ─────────────────────────────────────────────────────────────     │
│  + 1 字节   →  字段允许 NULL                                        │
│  + 2 字节   →  VARCHAR 长度前缀                                      │
│  CHAR(n)     →  定长，不加长度前缀                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 字符集字节数对照

| 字符集 | 一个字符占字节数 | VARCHAR(50) 纯字符key_len | VARCHAR(50) NOT NULL key_len |
|--------|-----------------|---------------------------|------------------------------|
| latin1 | 1 | 50 + 2 = **52** | 50 |
| utf8 | 3 | 50 × 3 + 2 = **152** | 50 × 3 = **150** |
| utf8mb4 | 4 | 50 × 4 + 2 = **202** | 50 × 4 = **200** |

### 3.3 核心问题：为什么 key_len = 153 而不是 50？

```
┌─────────────────────────────────────────────────────────────────────┐
│  场景：ALTER TABLE users ADD INDEX idx_name (name);                │
│       SELECT * FROM users WHERE name = 'Tom';                     │
│       name 字段定义：name VARCHAR(50) CHARACTER SET utf8mb4        │
│                                                                     │
│  EXPLAIN 输出：key_len = 153                                        │
│                                                                     │
│  为什么不是 50？                                                    │
│  ─────────────────────────────────────────────────────────────     │
│                                                                     │
│  错误的理解：50个字符，每个字符最多4字节，所以 50 × 4 = 200            │
│                                                                     │
│  正确的理解：                                                        │
│                                                                     │
│  VARCHAR(n) 的 key_len = n × 每字符最大字节数 + 长度前缀 + NULL标志  │
│                                                                     │
│  但 key_len = 153 意味着什么？                                      │
│                                                                     │
│  公式反推：153 = n × 4 + 2 + 1                                      │
│           153 = 4n + 3                                              │
│           n = 37.5 → MySQL 只用了索引的前缀                         │
│                                                                     │
│  原因：InnoDB 索引使用 COMPACT/DYNAMIC 行格式                       │
│       行格式只存储 VARCHAR 的前 n 个字符（而不是完整的50字符）        │
│       MySQL 优化器估算：用 name 前 37-38 个字符就能区分大多数记录     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

key_len 的实际含义：索引被使用的字节数

┌─────────────────────────────────────────────────────────────┐
│  key_len = 153 的分解：                                    │
│                                                             │
│  ┌─────────┬─────────┬─────────┬──────────────────────┐   │
│  │ 1字节    │ + 2字节 │ = 3字节 │ 可变长部分（长度前缀）│   │
│  │ NULL标志 │ 长度前缀│         │                      │   │
│  └─────────┴─────────┴─────────┴──────────────────────┘   │
│         │                                    │              │
│         │     ┌────────────────────────────┐ │              │
│         │     │  VARCHAR(50) 前缀字节数    │ │              │
│         │     │  50 × 4 = 200 理论最大     │ │              │
│         │     │  但实际只用了 150 字节     │ │              │
│         │     │  153 - 3 = 150            │ │              │
│         │     └────────────────────────────┘ │              │
│         │                                    │              │
│         │     ┌────────────────────────────┐ │              │
│         │     │  150 / 4 = 37.5 个字符    │ │              │
│         │     │  ≈ 37 个字符被用于索引     │ │              │
│         │     └────────────────────────────┘ │              │
│         │                                                  │
│         ▼                                                  ▼
│  ┌─────────────────────────────────────────────────────┐   │
│  │  name VARCHAR(50) utf8mb4, 允许NULL                 │   │
│  │                                                     │   │
│  │  理论 key_len = 50×4 + 2 + 1 = 203                  │   │
│  │  实际 key_len = 153 = 37×4 + 2 + 1                  │   │
│  │                                                     │   │
│  │  结论：MySQL 只用了 name 的前37个字符做索引          │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 联合索引场景下的 key_len

```sql
-- 联合索引
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id INT NOT NULL,
    status VARCHAR(20) NOT NULL,        -- latin1
    name VARCHAR(100) NOT NULL,          -- utf8mb4
    created_at DATETIME,
    INDEX idx_composite (user_id, status, name)
);

-- 查询
SELECT * FROM orders WHERE user_id = 100 AND status = 'paid';

-- key_len 计算：
user_id (INT, NOT NULL)     →  4  字节
status (VARCHAR(20), latin1, NOT NULL)  →  20×1 + 2 = 22 字节
─────────────────────────────────────────
总 key_len = 4 + 22 = 26 字节

-- 为什么 name 没有算进去？
-- 因为查询只用到了 user_id 和 status
-- 索引 (user_id, status, name) 只被用到了前两列
-- 所以 key_len = 4 + 22 = 26

-- 如果加上 name 条件：
SELECT * FROM orders WHERE user_id = 100 AND status = 'paid' AND name = '商品A';
-- key_len = 4 + 22 + 100×4 + 2 = 428 字节
```

## 🔥 扩展追问 1：联合索引 `(a, b, c)`，查询 `WHERE a = 1 AND c = 3`，能用到索引吗？

> **A:** 能用到索引，但只能用到 a 列。
>
> ```
> ┌─────────────────────────────────────────────────────────────┐
> │  联合索引 (a, b, c) 的 B+Tree 结构：                        │
> │                                                             │
> │                 [主键100]                                    │
> │               /           \                                 │
> │         [a=1, b=1, c=1]    [a=1, b=2, c=3]                 │
> │              │                    │                         │
> │              ▼                    ▼                         │
> │         [a=1, b=1, c=5]      [a=2, b=1, c=2]                │
> │              │                    │                         │
> │              ▼                    ▼                         │
> │         行记录               行记录                         │
> │                                                             │
> │  查询 WHERE a=1 AND c=3：                                   │
> │  1. a=1 可以在索引树定位（a 是第一列）                        │
> │  2. c=3 没办法用索引排序，只能回表后在服务层过滤               │
> │  3. Extra = Using where                                     │
> │                                                             │
> │  如果查询 WHERE b=2：                                        │
> │  → 完全无法利用索引（b 是中间列，断层了）                      │
> │  → type = ALL（全表扫描）                                     │
> │                                                             │
> │  总结：索引最左前缀原则，查询必须从第一列连续往后              │
> └─────────────────────────────────────────────────────────────┘
> ```
>
> **最佳实践：** 如果业务需要频繁按 c 列查询，考虑创建 `(c, a, b)` 或单独的 c 列索引。

## 🔥 扩展追问 2：为什么 `SELECT *` 经常无法 Using index（覆盖索引）？

> **A:** 因为覆盖索引要求查询的所有列都在索引中。
>
> ```sql
> -- 索引：(name, email)
>
> -- ✅ Using index（覆盖索引）
> SELECT name, email FROM users WHERE name = 'Tom'
> -- name+email 都在索引中，无需回表
>
> -- ❌ Using index condition（ICP），需要回表
> SELECT id, name, email, phone, address FROM users WHERE name = 'Tom'
> -- id, phone, address 不在索引中，必须回表
>
> -- ✅ 优化：扩展覆盖索引
> ALTER TABLE users ADD INDEX idx_cover (name, email, phone, address);
> -- 把查询需要的列都放进索引
> ```
>
> **设计覆盖索引的口诀：**
> > "查询什么列，就建什么索引。一查多的场景，多列合并。回表是性能杀手，覆盖索引是解药。"

## 🔥 扩展追问 3：`EXPLAIN` 的 `rows` 列是怎么估算的？

> **A:** rows 是 MySQL 优化器基于统计信息估算的扫描行数，不是精确值。
>
> ```
> 估算原理：
> ┌─────────────────────────────────────────────────────────────┐
> │  MySQL 统计信息存储在 information_schema.tables：           │
> │                                                             │
> │  SELECT table_rows, avg_row_length FROM information_schema.tables  │
> │  WHERE table_name = 'users' AND engine = 'InnoDB';         │
> │                                                             │
> │  返回：                                                      │
> │  table_rows = 1,000,000   （总行数）                        │
> │  avg_row_length = 200      （平均行长）                      │
> │                                                             │
> │  EXPLAIN rows = 1,000,000 × selectivity(name='Tom')        │
> │                                                             │
> │  selectivity = cardinality / total_rows                    │
> │  cardinality ≈ 10,000（name列不同值的数量）                  │
> │  selectivity = 10,000 / 1,000,000 = 0.01                    │
> │                                                             │
> │  估算 rows = 1,000,000 × 0.01 = 10,000                     │
> └─────────────────────────────────────────────────────────────┘
> ```
>
> **估算不准的常见原因：**
> - `ANALYZE TABLE` 未定期执行（统计信息过期）
> - `cardinality` 基于采样，不完全准确
> - 多列组合的选择性无法精确估算

---

> **MySQL EXPLAIN 的 type 列是天梯，Extra 是性能诊断卡——system→ALL 从最优到最差；Using index 是金牌，Using filesort + Using temporary 同时出现是红灯；key_len 是索引利用深度的温度计，153 意味着 MySQL 只用了前 37 个字符就能区分数据。**

---

> "在优化慢查询时，我习惯先看 type 列——只要不是 ALL，就有救。然后看 Extra 中有没有 Using filesort，有的话优先用索引消除排序。比如一个 ORDER BY create_time DESC，我直接加一个 `(create_time DESC)` 索引就能消除。再看 key_len，如果联合索引的 key_len 小于预期，说明索引没被完全利用，很可能是查询条件没有按最左前缀顺序写。有一次，一个 VARCHAR(50) utf8mb4 的 key_len 是 203，但联合索引后面还跟了一列 INT，实际 EXPLAIN 显示 key_len 刚好到 INT 列就停了，说明 WHERE 条件写反了——把 VARCHAR 列写在 INT 列后面了，违反了最左前缀原则。"
