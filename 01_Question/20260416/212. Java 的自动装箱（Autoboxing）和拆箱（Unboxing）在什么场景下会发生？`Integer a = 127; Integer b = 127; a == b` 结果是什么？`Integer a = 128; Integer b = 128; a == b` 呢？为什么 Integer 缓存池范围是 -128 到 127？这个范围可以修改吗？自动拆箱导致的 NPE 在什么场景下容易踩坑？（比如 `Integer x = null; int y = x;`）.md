### 1.2 Integer 缓存池核心机制

```java
// Integer 缓存池源码（JDK 8 Integer.java）
public final class Integer extends Number implements Comparable<Integer> {
    
    // IntegerCache：缓存池实现
    private static class IntegerCache {
        // 缓存范围：[low, high]
        static final int low = -128;
        static final int high;
        
        // 缓存数组
        static final Integer cache[];
        
        static {
            // 默认 high = 127
            // 可以通过 -Djava.lang.Integer.IntegerCache.high=XXX 修改
            int h = 127;
            String integerCacheHighPropValue =
                sun.misc.VM.getSavedProperty("java.lang.Integer.IntegerCache.high");
            
            // 如果设置了系统属性，按设置值来
            if (integerCacheHighPropValue != null) {
                try {
                    int i = parseInt(integerCacheHighPropValue);
                    // 至少是 127，且 Math.max(127, ...) 保证不小于默认值
                    i = Math.max(i, 127);
                    // 最大缓存到 Integer.MAX_VALUE，但通常设到 10000 就很离谱了
                    h = Math.min(i, Integer.MAX_VALUE - (-low) - 1);
                } catch (NumberFormatException nfe) { }
            }
            high = h;
            
            // 创建缓存数组：cache[-128..high]
            cache = new Integer[(high - low) + 1];
            int j = low;
            for (int k = 0; k < cache.length; k++)
                cache[k] = new Integer(j++);
        }
    }
    
    // valueOf：装箱时调用的方法
    public static Integer valueOf(int i) {
        // 范围内直接返回缓存对象，范围外创建新对象
        if (i >= IntegerCache.low && i <= IntegerCache.high)
            return IntegerCache.cache[i + (-IntegerCache.low)];
        return new Integer(i);
    }
}
```

## 🔥 扩展追问 1：自动拆箱导致的 NPE 在什么场景下容易踩坑？

> A: **拆箱 NPE 发生在包装类型为 null 时参与运算，常见场景有：三目运算符、算术运算、集合操作、方法参数传递。**

```java
// 场景1：三目运算符（最常见坑）
Integer a = null;
int b = (a != null) ? a : 0;  // ⚠️ 编译通过，运行 NPE！

// 编译后等价于：
int b = (a != null) ? a.intValue() : 0;
// 三目运算符要求返回类型一致，null 被拆箱为 int → NPE

// 正确写法：
int b = (a != null) ? a.intValue() : 0;  // 手动拆箱
// 或者用 Optional：
int b = Optional.ofNullable(a).orElse(0);


// 场景2：算术运算中的 null
Integer sum = null;
int result = sum + 10;  // ⚠️ NPE
// sum.intValue() + 10 → NPE

// 场景3：集合中的自动拆箱
List<Integer> numbers = new ArrayList<>();
numbers.add(null);
int first = numbers.get(0);  // ⚠️ NPE
// get(0) 返回 null，自动拆箱 → NPE

// 场景4：方法参数自动拆箱
void process(int value) {
    System.out.println(value);
}

Integer param = null;
process(param);  // ⚠️ NPE
// process(param.intValue()) → NPE

// 场景5：HashMap 的 get 返回 null
Map<String, Integer> map = new HashMap<>();
map.put("a", null);
int value = map.get("a");  // ⚠️ NPE（如果 value == null 被返回）
// null 被自动拆箱为 int → NPE

// 场景6：equals 前的自动拆箱（反直觉）
Integer a = null;
boolean result = a.equals(1);  // false（不抛异常）
// a 是 null，但 equals 是方法调用，不涉及拆箱

boolean result2 = 1.equals(a);  // false（不抛异常）
// 1 是 Integer 常量，equals 比较，不涉及拆箱

// ⚠️ 但这样就 NPE：
int result3 = a;  // 隐式拆箱 → NPE
```

## 🔥 扩展追问 2：`==` 比较在 Integer 缓存池中的行为总结？

> A: **`==` 比较的是对象引用（内存地址），不是值。在缓存池范围内返回 true，超出范围返回 false。`equals` 比较的是值。`Integer.compare()` 才是正确的数值比较。**

```java
// 完整对比测试
Integer a = 127;
Integer b = 127;
System.out.println(a == b);           // true（缓存池）
System.out.println(a.equals(b));      // true（值相等）
System.out.println(Integer.compare(a, b));  // 0（值相等）

Integer c = 128;
Integer d = 128;
System.out.println(c == d);           // false（非缓存池，new了两个对象）
System.out.println(c.equals(d));      // true（值相等）
System.out.println(Integer.compare(c, d));  // 0（值相等）

// 自动装箱的坑
Integer e = 1;        // 自动装箱 → valueOf(1) → 缓存池
Integer f = 1;
System.out.println(e == f);           // true（缓存池）

// new Integer 的坑
Integer g = new Integer(127);
Integer h = new Integer(127);
System.out.println(g == h);            // false（两个 new，两个对象）
System.out.println(g.equals(h));      // true（值相等）

// 混合比较
Integer i = 127;
int j = 127;
System.out.println(i == j);            // true！（混合比较，拆箱后比较）
System.out.println(i.equals(j));      // true（自动装箱为 Integer）

// 结论：
// 1. Integer == Integer：比较引用，缓存池内 true
// 2. Integer == int：比较值，拆箱后比较（永远比值）
// 3. Integer.equals(int)：值比较（自动装箱）
// 4. Integer.compare(a, b)：值比较，推荐使用
```

```
最佳实践：

┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│  ❌ 永远不要用 == 比较两个包装类型                                          │
│                                                                            │
│  ✅ 用 equals：                                                            │
│     a.equals(b)                                                           │
│                                                                            │
│  ✅ 用 Integer.compare（推荐）：                                           │
│     Integer.compare(a, b) == 0  // 返回 0 表示相等                        │
│     Integer.compare(a, b) > 0   // a > b                                 │
│     Integer.compare(a, b) < 0   // a < b                                 │
│     // 底层是数值比较，不是引用比较                                        │
│                                                                            │
│  ✅ 在 Map 的 key 中使用 Integer：                                         │
│     HashMap<Integer, String> map = new HashMap<>();                       │
│     map.get(key)  // HashMap 内部用 equals 比较，OK                        │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 3：为什么 Integer 缓存池范围是 -128 到 127？这个范围可以修改吗？

> A: **-128 是固定的最小值（JLS 规范强制），127 是默认值，上限可通过 JVM 参数修改，但过大会有内存问题。**

```
JLS 规范规定：

┌────────────────────────────────────────────────────────────────────────────┐
│  Java Language Specification (JLS) §5.1.7 规定：                          │
│                                                                            │
│  "If the value p being boxed is true, false, a byte, a char in the        │
│  range \u0000 to \u007f, or an int or short number between -128 and 127,  │
│  then let r1 and r2 be the results of any two boxing conversions of p.   │
│  It is always the case that r1 == r2."                                   │
│                                                                            │
│  翻译：                                                                    │
│  如果 p 是以下值之一：                                                     │
│  • true / false                                                          │
│  • byte                                                                  │
│  • char (0x0000 ~ 0x007f, 即 ASCII)                                       │
│  • int 或 short 在 -128 到 127 之间                                       │
│  那么任何两次装箱的结果 r1 和 r2，必须满足 r1 == r2                        │
│                                                                            │
│  ✅ 这是 JLS 强制要求，所有 JVM 实现必须遵守                                │
└────────────────────────────────────────────────────────────────────────────┘

为什么选择 -128 ~ 127？

┌────────────────────────────────────────────────────────────────────────────┐
│                                                                            │
│  1. 历史原因：                                                           │
│    早期 JVM 只缓存 -128 ~ 127，这是经验值                                 │
│                                                                            │
│  2. 统计原因：                                                           │
│    实际编程中，整数在这个范围内使用频率最高                                │
│    • for 循环计数器                                                        │
│    • 数组索引                                                              │
│    • 枚举值                                                                │
│    • 配置参数                                                              │
│                                                                            │
│  3. 空间权衡：                                                            │
│    128 - (-128) + 1 = 256 个对象                                          │
│    每个 Integer 对象约 16 字节 → 256 * 16 = 4KB                          │
│    内存开销可接受                                                          │
│                                                                            │
│  4. 性能收益：                                                            │
│    这个范围内的整数频繁装箱/拆箱，缓存能显著减少对象分配                    │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘

可以修改吗？可以，但有代价：

java -Djava.lang.Integer.IntegerCache.high=10000 MyApp

┌────────────────────────────────────────────────────────────────────────────┐
│  内存代价：                                                               │
│                                                                            │
│  默认：256 个 Integer 对象 ≈ 4KB                                         │
│  扩展到 10000：11229 个 Integer 对象 ≈ 180KB                             │
│  扩展到 100000：100129 个 Integer 对象 ≈ 1.6MB                           │
│  扩展到 1000000：1000129 个 Integer 对象 ≈ 16MB                          │
│                                                                            │
│  ⚠️ 生产环境不建议超过 10000                                               │
└────────────────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 4：Java 编译器如何处理自动装箱/拆箱？（字节码角度）

> A: **编译器在编译阶段自动插入 `valueOf()` 调用（装箱）和 `xxxValue()` 调用（拆箱），可以用 `javap -c` 查看。**

```java
// Java 源代码
```

## 🔥 扩展追问 5：Long/Double/Character/Float 的缓存情况？

> A: **Long 有缓存池（-128~127），Double 和 Float 没有缓存（无界），Character 缓存 0~127，Boolean 只有 TRUE 和 FALSE 两个固定对象。**

```java
// Long 缓存池
Long a = 127L;
Long b = 127L;
System.out.println(a == b);  // true（缓存池）

Long c = 128L;
Long d = 128L;
System.out.println(c == d);  // false（超出缓存范围，new Long）

// Double 没有缓存！
Double e = 127.0;
Double f = 127.0;
System.out.println(e == f);  // false（每次都 new）

// Float 没有缓存！
Float g = 127.0f;
Float h = 127.0f;
System.out.println(g == h);  // false（每次都 new）

// Character 缓存 ASCII 字符
Character i = 'a';  // 自动装箱
Character j = 'a';
System.out.println(i == j);  // true（在缓存范围内）

Character k = '\u0100';  // 超出 ASCII 范围
Character l = '\u0100';
System.out.println(k == l);  // false

// Boolean 只有两个固定对象
Boolean m = true;
Boolean n = true;
System.out.println(m == n);  // true（固定返回 Boolean.TRUE）

Boolean o = Boolean.valueOf("true");
Boolean p = Boolean.valueOf("true");
System.out.println(o == p);  // true
```

## 3. 一句话总结

> **Java 自动装箱通过编译器插入 `valueOf()` 实现，`==` 比较 Integer 时缓存池内返回 true（同一对象），超出范围返回 false（new 的不同对象）；NPE 陷阱集中在三目运算符和算术拆箱；Integer 缓存池范围是 JLS 强制规定的 -128~127，可通过 JVM 参数扩展但不宜过大。**

## 4. 面试加分话术

> "这个问题我之前踩过一个很经典的坑——线上有个接口用 `Integer` 做字典 key，key 值范围是 1~200，用 `==` 判断是否存在，结果 key=200 时永远找不到，因为 200 超出了 Integer 缓存池范围，每次 `valueOf` 都 new 了一个新对象，`==` 比较引用永远 false。后来改成 `Integer.valueOf(key)` 或者直接用 `equals`，问题才解决。另外一个坑是三目运算符：`Integer count = null; int result = count != null ? count : 0;` ——这种写法看着没问题，编译也通过，但运行就 NPE，因为三目运算符统一返回类型，null 被当成需要拆箱的值，正确写法是 `int result = count != null ? count.intValue() : 0`，或者用 `Objects.requireNonNullElse(count, 0)`。面试中我还喜欢补充一点：装箱开销不小，在高频调用的循环里如果频繁装箱，比如 `for (int i = 0; i < 1000000; i++) { list.add(i); }`，会产生大量 Integer 对象，JVM GC 压力会明显增加，所以大循环里尽量用基本类型数组或局部变量缓存。"
