## 二、Happens-Before 八大规则

```java
/**
 * Happens-Before 定义：
 * 
 * "如果操作 A happens-before 操作 B，
 *   那么 A 的执行结果对 B 可见，
 *   且 A 的执行顺序在 B 之前"
 * 
 * 注意：happens-before 只约束 **可见性** 和 **有序性**，
 *       不约束 **原子性**（原子性由 synchronized/volatile/CAS 保证）
 */
```

## 三、其他建立 Happens-Before 的方式

```java
/**
 * 除了 volatile 和 synchronized，还有很多并发工具类建立 happens-before
 * 本质：都基于上述 8 条规则
 */
```

| 并发工具类 | 建立的 Happens-Before 关系 | 底层规则 |
|-----------|--------------------------|---------|
| **CountDownLatch.countDown()** | countDown() HB await() 返回 | 规则 2（锁）+ 规则 8 |
| **CyclicBarrier.await()** | 最后一个 barrier HB 其他线程继续 | 规则 2（锁）|
| **Semaphore.acquire()** | release() HB acquire() 返回 | 规则 2（锁）+ CAS |
| **Future.get()** | 任务完成 HB get() 返回 | 规则 2（锁）+ 规则 5 |
| **Exchanger.exchange()** | T1.exchange() HB T2.exchange() 返回 | 规则 2（锁）|
| **CompletableFuture** | 完成操作 HB 依赖操作 | 规则 2（锁）+ 规则 8 |
| **Phaser.arriveAndAwaitAdvance()** | arrive HB 其他人继续 | 规则 2（锁）|
| **StampedLock.readLock()** | unlockWrite HB lockRead | 规则 2（锁）|
| **ConcurrentHashMap** | 红黑树/链表操作间 | CAS + synchronized |

### 3.1 CountDownLatch 示例

```java
/**
 * CountDownLatch 建立 happens-before 的原理
 */
import java.util.concurrent.CountDownLatch;

public class CountDownLatchDemo {
    
    private int x = 0;
    private CountDownLatch latch = new CountDownLatch(1);
    
    public void writer() {
        x = 42;              // 操作 A
        latch.countDown();   // 操作 B：计数减 1
        /**
         * 规则 2：countDown() 内部释放锁
         * latch.await() 获取锁后可见
         */
    }
    
    public void reader() throws InterruptedException {
        latch.await();       // 操作 C：等待计数为 0
        System.out.println(x);  // 操作 D：一定能读到 x=42
        /**
         * happens-before 链：
         * A HB B（规则 1：程序顺序）
         * B HB C（规则 2 + 规则 8：传递性）
         * A HB D（传递性：A HB B HB C HB D）
         */
    }
}
```

### 3.3 传递性构建复杂 happens-before

```java
/**
 * Happens-before 链的实战：生产者-消费者
 */
public class ProducerConsumer {
    
    private volatile boolean initialized = false;
    private volatile byte[] data = new byte[1024];
    
    public void producer() {
        // 初始化数据
        for (int i = 0; i < data.length; i++) {
            data[i] = (byte) (i % 256);
        }
        
        // 标记初始化完成
        initialized = true;  // volatile 写
        /**
         * happens-before 链：
         * 1. 循环写 data → initialized 写（程序顺序）
         * 2. initialized 写 HB 消费者读取（volatile 规则）
         */
    }
    
    public void consumer() {
        // 等待初始化
        while (!initialized) {
            Thread.yield();
        }
        
        // 读取数据
        byte b = data[0];  // 一定能读到完整的数据
        /**
         * 消费者看到 initialized=true
         * 一定能看到生产者之前的所有写
         */
    }
}
```

## 四、as-if-serial 语义：为什么编译器仍能重排序？

```
┌─────────────────────────────────────────────────────────────────────────┐
│                   as-if-serial vs happens-before                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   as-if-serial 语义：                                                   │
│   ─────────────────                                                    │
│   "编译器和 CPU 可以随意重排序指令，                                        │
│     只要保证 **单线程** 情况下，结果和顺序执行一致即可"                    │
│                                                                         │
│   Happens-Before 规则：                                                 │
│   ──────────────────                                                    │
│   "跨线程时，必须满足 happens-before 关系                                    │
│     违反 happens-before = 可能看到错误的值"                                │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │                                                                  │   │
│   │  as-if-serial 是 **单线程** 优化的自由度                              │   │
│   │  happens-before 是 **多线程** 交互的约束                             │   │
│   │                                                                  │   │
│   │  编译器可以在单线程内重排序：                                        │   │
│   │  - 只要单线程结果正确                                              │   │
│   │  - 无需关心其他线程看到的顺序                                        │   │
│   │                                                                  │   │
│   │  编译器 **不能** 违反 happens-before：                               │   │
│   │  - 如果 A HB B，编译器不能重排 B 到 A 之前                           │   │
│   │  - 否则 B 可能读到 A 的"前值"                                         │   │
│   │                                                                  │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 4.1 编译器重排序 vs Happens-Before

```java
/**
 * 场景：单线程代码，编译器可以重排序
 */
public class CompilerReorder {
    
    public static void main(String[] args) {
        int a = 1;     // A
        int b = 2;     // B
        int c = a + b; // C
        
        // 编译器可能重排序为：
        // int b = 2;     // B（先执行）
        // int a = 1;     // A
        // int c = a + b; // C
        
        // 为什么可以重排？
        // → 单线程下，A 和 B 没有依赖关系
        // → as-if-serial 保证最终 c = 3
        // → 但多线程下可能出问题！
    }
}
```

```java
    private int y = 0;
    
    // Thread A                   // Thread B
    // ─────────                   // ─────────
    // x = 1;                      while (y == 0) { }
    //                            println(x);
    
    /**
     * 问题场景：
     * 
     * 如果编译器重排序：
     * 
     * Thread A：                 Thread B：
     * ──────                     ──────
     * y = 1;  ← 重排序到这里！    while (y == 0) { }  ← y=1，跳出循环
     * x = 1;                      println(x);         ← x 可能还是 0！
     * 
     * 结果：x=0 被打印（错误！）
     * 
     * 为什么能重排？
     * → 单线程下，x 和 y 没有依赖关系
     * → as-if-serial 语义允许
     * 
     * 为什么是错误的？
     * → Thread A: y = 1 HB Thread B: 读取 y（volatile 假设）
     * → 但如果不用 volatile，Thread B 可能读到 y=0
     * → 且 Thread A 的 x=1 还没执行完
     */
}
```

### 4.3 Happens-Before 阻止重排序

```java
/**
 * 添加 synchronized/volatile 阻止重排序
 */
public class HBPreventReorder {
    
    private int x = 0;
    private volatile int y = 0;
    
    // Thread A                   // Thread B
    // ─────────                   // ─────────
    // x = 1;                      while (y == 0) { }
    // y = 1;  ← volatile 写       println(x);  ← 一定打印 1
    //             ↑
    //        happens-before:
    //        1. 程序顺序：A1 HB A2
    //        2. volatile：A2 HB B2
    //        3. 传递性：A1 HB B2
    
    /**
     * volatile 写后的 happens-before：
     * 
     * 1. A: x = 1;          (程序顺序)
     * 2. A: y = 1;          (volatile 写)
     *                 ↓ happens-before
     * 3. B: while(y==0);    (volatile 读)
     * 4. B: println(x);     (程序顺序)
     * 
     * 传递性：A HB B
     * → x=1 一定对 B 可见
     * → 编译器不能重排序！
     */
}
```

## 🔥 扩展追问 1：数据依赖性（Data Dependency）

> **A:** 编译器不能重排有数据依赖的操作：

```java
/**
 * 三种数据依赖关系（编译器不能重排）
 */

// 1. 写后读（RAW: Read After Write）
int a = 1;     // A
int b = a + 1; // B 依赖 A
// B 不能排到 A 前面

// 2. 读后写（WAR: Write After Read）
int a = x;    // A
x = 2;        // B 依赖 A
// B 不能排到 A 前面

// 3. 读后读（RAR: Read After Read）
int a = x;    // A
int b = x;    // B 依赖 A
// B 不能排到 A 前面 ← 这个可以重排！（都是读）

/**
 * 但跨线程没有数据依赖性！
 * 编译器可以自由重排（只要满足 as-if-serial）
 */
```

## 🔥 扩展追问 2：JMM 的三大特性

> **A:** JMM 保证的三大特性：

| 特性 | 说明 | 保证方式 |
|-----|------|---------|
| **原子性** | 操作不可分割 | synchronized + CAS + atomic 类 |
| **可见性** | 修改对其他线程可见 | happens-before 规则 + 内存屏障 |
| **有序性** | 指令顺序符合预期 | happens-before + 内存屏障 |

```java
/**
 * 对比：JMM 三大特性
 */

// 1. 原子性：synchronized
synchronized (lock) {
    x++;  // 原子：lock + unlock 之间完整执行
}

// 2. 可见性：volatile + happens-before
volatile boolean flag = true;  // 写 HB 后续读

// 3. 有序性：happens-before + 内存屏障
// volatile 写前有 StoreStore，写后有 StoreLoad
// 阻止编译器和 CPU 重排序
```

## 🔥 扩展追问 3：双重检查锁（完整 happens-before 分析）

> **A:** 完整分析为什么单例需要 volatile：

```java
public class Singleton {
    private static volatile Singleton instance;  // volatile 关键！
    
    public static Singleton getInstance() {
        if (instance == null) {  // 第一次检查
            synchronized (Singleton.class) {  // 锁 HB
                if (instance == null) {  // 第二次检查
                    instance = new Singleton();
                    /**
                     * instance = new Singleton() 分解为：
                     * 1. 分配内存          (分配空间)
                     * 2. 调用构造方法       (初始化字段)
                     * 3. instance = ref    (写入引用)
                     * 
                     * 没有 volatile：
                     * 2 和 3 可能重排序 → 其他线程看到非 null 但未构造的对象
                     * 
                     * 有 volatile：
                     * StoreStore → 写 → StoreLoad
                     * → 保证 2 完成后才能执行 3
                     * → 保证 instance 对其他线程可见时已构造完成
                     */
                }
            }
        }
        return instance;
    }
}
```

## 🔥 扩展追问 4：JMM 的 happens-before 和分布式一致性协议的区别？

> **A:** 本质相同，但层级不同：

| 对比 | JMM Happens-Before | 分布式一致性（Raft/Paxos） |
|-----|-------------------|--------------------------|
| **层级** | JVM 内存模型（语言层） | 分布式系统（网络层） |
| **范围** | 单 JVM 内多线程 | 多 JVM / 多机器 |
| **延迟** | 纳秒~微秒级 | 毫秒~秒级 |
| **一致性强度** | 单 JVM 内存语义 | 可配置（强一致/最终一致）|
| **实现** | 内存屏障、CAS、锁 | 网络协议、共识算法 |

## 一句话总结

> **JMM 是抽象规范，主内存对应主存、工作内存对应 CPU 缓存/寄存器；happens-before 八大规则（程序顺序/锁/volatile/线程启动终止/传递性）保证跨线程可见性和有序性；as-if-serial 允许编译器在单线程内重排序优化，但必须遵守 happens-before 约束。**
