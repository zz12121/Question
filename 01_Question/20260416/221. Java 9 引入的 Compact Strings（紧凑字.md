## 二、Compact Strings 底层实现

### 2.1 String 源码（JDK 9+）

```java
/**
 * JDK 9+ 的 String 实现
 * 
 * 文件：src/java.base/java/lang/String.java
 */
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {
    
    // ⭐ 核心改变：char[] → byte[]
    private final byte[] value;
    
    // ⭐ 编码标识符（coder）
    // 0 = LATIN1（单字节）
    // 1 = UTF16（双字节，与旧版本兼容）
    private final byte coder;
    
    // 编码常量
    static final byte LATIN1 = 0;
    static final byte UTF16 = 1;
    
    /**
     * 构造函数：根据字符类型选择编码
     */
    String(byte[] bytes, byte coder) {
        this.value = bytes;
        this.coder = coder;
    }
    
    /**
     * 计算字符串哈希值时使用 coder
     */
    @Override
    public int hashCode() {
        int h = 0;
        if (coder == LATIN1) {
            // Latin-1: 逐字节
            for (byte b : value) {
                h = 31 * h + (b & 0xFF);
            }
        } else {
            // UTF-16: 每两个字节（char）
            for (int i = 0; i < value.length; i += 2) {
                h = 31 * h + (char)(value[i] << 8 | value[i+1] & 0xFF);
            }
        }
        return h;
    }
}
```

```java
/**
 * charAt() 的实现（JDK 9+）
 */
public char charAt(int index) {
    if (coder() == LATIN1) {
        // Latin-1: 直接返回（0-255 范围）
        return (char)(value[index] & 0xFF);
    }
    // UTF-16: 组合两个字节
    return (char)(((value[index + 1] & 0xFF) << 8) 
                | (value[index] & 0xFF));
}

/**
 * coder() 返回编码标识
 */
private coder() {
    return coder;
}
```

```
│   原因一：API 兼容性                                                    │
│   ──────────────                                                       │
│   - charAt() 返回 char（UTF-16）                                      │
│   - Character 类的所有方法基于 UTF-16                                   │
│   - StringBuilder / StringBuffer 需要能存储任意字符                     │
│                                                                         │
│   原因二：性能考虑                                                      │
│   ──────────────                                                       │
│   - Latin-1 字符串：charAt() 直接返回（无解码）                         │
│   - 如果都用 UTF-8：每次 charAt() 都需要解码（慢）                       │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐  │
│   │                                                                 │  │
│   │   charAt() 实现（Latin-1）：                                   │  │
│   │   ────────────────────────────                                 │  │
│   │                                                                 │  │
│   │   return (char)(value[index] & 0xFF);                         │  │
│   │   // 直接返回，无解码开销                                       │  │
│   │                                                                 │  │
│   │   charAt() 实现（UTF-8 解码伪代码）：                           │  │
│   │   ────────────────────────────────────                         │  │
│   │                                                                 │  │
│   │   // 1 字节 Latin-1                                            │  │
│   │   if (value[index] < 128) return value[index];                │  │
│   │   // 多字节 UTF-8                                              │  │
│   │   else { ... decode UTF-8 ... }  ← 开销大！                    │  │
│   │                                                                 │  │
│   │   结论：Latin-1 vs UTF-8，Latin-1 更快                         │  │
│   │                                                                 │  │
│   └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│   原因三：Unicode 演进                                                  │
│   ──────────────                                                       │
│   - Java 采用 UTF-16 编码 Unicode（与系统 API 统一）                    │
│   - 虽然 UTF-8 是主流，但 Java 内部保持 UTF-16 一致性                   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 2：String.intern() 在 Compact Strings 下有变化吗？

> **A:** intern() 机制不变，但字符串常量池的组织方式有优化：

```java
/**
 * intern() 仍然基于 StringTable（字符串常量池）
 * 
 * JDK 8+: StringTable 大小可调
 */
public class InternDemo {
    
    public static void main(String[] args) {
        String s1 = new String("hello");
        String s2 = s1.intern();
        String s3 = "hello";
        
        System.out.println(s1 == s2);  // false（s1 是堆对象）
        System.out.println(s2 == s3);  // true（都是常量池）
    }
}

// JDK 8+ 字符串常量池在堆中（JDK 7 从 PermGen 移过来）
// -XX:StringTableSize=600000  # 调优 StringTable 大小
```

## 🔥 扩展追问 3：Compact Strings 对序列化有影响吗？

> **A:** 有影响但向后兼容：

```java
/**
 * Java Serialization 兼容性
 */
public class SerializationDemo implements Serializable {
    
    private static final long serialVersionUID = 1L;
    private String data = "hello";
    
    /**
     * Java 9+ 序列化 String 时：
     * - 写入 byte[] 和 coder
     * - 反序列化时重建 String
     * 
     * Java 8 序列化的 String：
     * - 写入 char[]
     * - Java 9+ 反序列化时自动转为 byte[]
     * 
     * ✅ 向后兼容！
     */
}
```

## 一句话总结

> **Java 9 Compact Strings 将 String 内部从 char[]（2字节/字符）改为 byte[] + coder（0=LATIN1，1=UTF16），纯英文字符串内存节省约50%，Latin-1 字符串每字符占1字节、中文字符占2字节；charAt() 等方法实时解码不影响性能反而因缓存友好更快；混合字符串因 coder 是对象级别会整体使用 UTF16，不会出现 char[] 的"回退"。**

## 面试加分话术

> "之前优化过一个日志采集系统，日志量每天 10GB，大部分是业务日志（英文+时间戳+IP）。升级到 JDK 11 后，堆内存从 12GB 降到 8GB，关键是字符串内存占用从 4GB 降到 2GB。后来踩过一个坑：有一个字段是中文 IP 归属地，混合字符串导致所有这类字符串都使用 UTF16，反而比纯英文时占用更多内存。最后通过把中文归属地单独存一张表，查询时 join，避免在日志字符串里混合中英文解决了。这个经历让我理解了 Compact Strings 的 coder 机制——编码是对象级别的，不是字符级别的。"
