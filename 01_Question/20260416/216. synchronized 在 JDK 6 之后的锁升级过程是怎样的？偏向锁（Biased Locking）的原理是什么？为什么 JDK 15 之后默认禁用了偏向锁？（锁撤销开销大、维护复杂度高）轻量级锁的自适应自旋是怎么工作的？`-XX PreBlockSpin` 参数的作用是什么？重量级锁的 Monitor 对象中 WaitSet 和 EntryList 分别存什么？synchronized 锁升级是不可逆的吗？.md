**锁升级不可逆**：偏向锁 → 轻量级锁 → 重量级锁，**单向通道，不可回退**。

## 二、偏向锁（Biased Locking）原理

```java
/**
 * 偏向锁核心思想：
 * "这个锁只会偏向一个线程，后续该线程无需任何同步开销"
 * 
 * 适用场景：同一线程反复获取同一把锁（如单线程循环）
 */
```

### 2.1 偏向锁工作流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│                       偏向锁获取流程                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  线程 T1 第一次进入 synchronized(obj)：                                  │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  Mark Word：001（无锁状态）                                      │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                              │                                           │
│                              ▼  CAS(001 → 101, Thread=T1)              │
│                              │  成功则偏向 T1，设置 biased_lock=1       │
│                              ▼                                           │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  Mark Word：101 + Thread ID = T1 + Epoch                         │   │
│  │  bias_on=1, lock=01                                              │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                              │                                           │
│                              ▼                                           │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  偏向锁状态（Biased）                                            │   │
│  │                                                                 │   │
│  │  Thread T1 再次进入 synchronized(obj)：                          │   │
│  │  → 检查 Mark Word Thread ID = T1？                              │   │
│  │  → 直接进入，无需 CAS！                                           │   │
│  │  → O(1) 性能，真正的"无锁"                                        │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

// - 或偏向线程不活跃（-XX:BiasedLockingBulkRebiasThreshold = 20 次）

// 场景 B：竞争时撤销（竞争撤销）
void ObjectMonitor::revoke_bias(oop obj, bool bulk, JavaThread* requestor) {
    
    markOop mark = obj->mark();
    
    if (!mark->has_bias_pattern()) {
        return;  // 不是偏向锁，直接返回
    }
    
    // 获取偏向线程
    JavaThread* biased_thread = mark->biased_locker();
    
    if (biased_thread == NULL) {
        // 无偏向线程 → 批量撤销，改为无锁
        return;
    }
    
    // 检查是否是同一个线程
    if (biased_thread == requestor) {
        return;  // 同一个线程，无需撤销

## 三、JDK 15 禁用偏向锁的原因

```java
/**
 * JDK 15+：-XX:+UseBiasedLocking = false（默认）
 * JDK 6-14：-XX:+UseBiasedLocking = true（默认）
 * 
 * 禁用原因（三大痛点）：
 * 1. 锁撤销开销大
 * 2. 维护复杂度高  
 * 3. 现代 JVM G1 GC 的兼容性
 */
```

### 3.1 问题一：批量撤销开销

```
┌─────────────────────────────────────────────────────────────────────────┐
│              批量撤销（Bulk Revocation）的开销                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  场景：N 个对象被偏向线程 T1，后批量 Rebias                             │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  阶段 1：Stop The World（STW）                                    │   │
│  │  需要在安全点（Safe Point）遍历所有线程栈                        │   │
│  │  → 遍历 O(N) 个对象，修改 Mark Word                              │   │
│  │  → 耗时随对象数量线性增长                                         │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                              │                                           │
│                              ▼                                           │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  阶段 2：批量重偏向                                               │   │
│  │  20 次偏向 → 批量 Rebias（改为偏向 T2）                          │   │
│  │  40 次偏向 → 批量 Revoke（彻底禁用偏向）                          │   │
│  │                                                                 │   │
│  │  -XX:BiasedLockingBulkRebiasThreshold=20                        │   │
│  │  -XX:BiasedLockingBulkRevokeThreshold=40                        │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.2 问题二：G1 GC 兼容性问题

```java
/**
 * G1 GC 的 Remembered Set 需要精确追踪引用关系
 * 
 * 偏向锁对象的 Mark Word 包含线程信息
 * 
 * 当 GC 需要扫描对象头时：
 * 1. 偏向锁的 Mark Word 结构复杂
 * 2. 需要额外的处理逻辑
 * 3. G1 的并发标记（Concurrent Marking）与偏向锁冲突
 * 
 * 结果：G1 + 偏向锁 = 额外的同步开销
 */
```

### 3.3 问题三：现代应用场景变化

| 场景 | JDK 6 时代 | JDK 15+ 时代 |
|-----|----------|-------------|
| 线程模型 | 长时间运行的 Server | 短生命周期容器/函数 |
| 锁竞争 | 相对温和 | 微服务瞬时并发高 |
| GC | Serial/ParNew/CMS | G1/ZGC/Shenandoah |
| 偏向锁收益 | 明显 | 降低 |

## 四、轻量级锁与自适应自旋

```java
/**
 * 轻量级锁：避免线程进入内核态（互斥量）
 * 
 * 核心思想：
 * - 线程不阻塞，而是自旋等待
 * - CPU 空转（忙等），比阻塞/唤醒快
 * - 适用于锁竞争时间短的场景
 */
```

### 5.2 EntryList vs WaitSet 对比

| 对比项 | EntryList | WaitSet |
|-------|-----------|---------|
| **线程状态** | BLOCKED（阻塞等待锁） | WAITING / TIMED_WAITING |
| **进入条件** | 竞争锁失败 | 调用 `Object.wait()` |
| **唤醒方式** | 锁释放自动唤醒 | `notify()` / `notifyAll()` |
| **是否持有锁** | ❌ 不持有 | ❌ 不持有 |
| **是否释放锁** | ❌ 未获取锁 | ✅ 已获取锁后调用 wait() 释放 |
| **位置** | Monitor 内部 | Monitor 内部 |

```java
/**
 * EntryList vs WaitSet 的实战理解
 */
public class MonitorDemo {
    private final Object lock = new Object();
    
    public void entryListDemo() {
        // T1 持有锁
        synchronized (lock) {
            // T2、T3 来竞争锁
            // → 进入 EntryList（BLOCKED 状态）
            // → T1 释放锁后，T2/T3 竞争
        }
    }
    
    public void waitSetDemo() {
        synchronized (lock) {
            try {
                // T1 调用 wait()
                // → 释放锁，进入 WaitSet（WAITING 状态）
                // → 可以被 notify() 唤醒
                lock.wait();
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }
    }
    
    public void notifyDemo() {
        synchronized (lock) {
            // T1 调用 notify()
            // → 从 WaitSet 选一个移到 EntryList
            // → 移到 EntryList 的线程继续竞争锁
            lock.notify();
            
            // notifyAll() → 全部移到 EntryList
            // lock.notifyAll();
        }
    }
}
```

## 六、锁升级不可逆性

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    锁升级为什么不可逆？                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   偏向锁 ──▶ 轻量级锁 ──▶ 重量级锁   ✅ 单向，不可回退                    │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │                                                                 │   │
│   │                        原因：                                    │   │
│   │                                                                 │   │
│   │   1. 偏向锁 → 轻量级锁                                           │   │
│   │      - 需要撤销偏向（扫描安全点，修改 Mark Word）                 │   │
│   │      - 已有其他线程竞争，说明不再"单线程"                         │   │
│   │      - 回退偏向成本 > 收益                                       │   │
│   │                                                                 │   │
│   │   2. 轻量级锁 → 重量级锁                                         │   │
│   │      - 已经发生了锁竞争（自旋失败）                                │   │
│   │      - 线程已阻塞，需要 OS mutex                                 │   │
│   │      - 回退到用户态自旋浪费资源                                   │   │
│   │                                                                 │   │
│   │   3. 重量级锁 → 轻量级锁                                          │   │
│   │      - 线程已在内核态，状态复杂                                    │   │
│   │      - 无法无损恢复到轻量级锁状态                                  │   │
│   │                                                                 │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│   ⚠️ 注意：轻量级锁释放后，可以重新升级                                   │
│   ─────────────────────────────────────                                │
│   T1 持有轻量级锁，释放后，Mark Word 恢复无锁状态                         │
│   但下一次竞争可能直接升级为重量级锁                                      │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 1：锁重入（Reentrant）的实现？

> **A:** Monitor 中的 `_recursions` 计数器实现锁重入：

```java
/**
 * synchronized 是可重入锁
 * 
 * 同一个线程多次获取同一把锁：
 * - 第一次：_owner = Thread，_recursions = 0 → 1
 * - 第二次：检查 _owner = 自己，直接 _recursions++ → 2
 * - 退出时：_recursions--，直到 = 0 才真正释放锁
 */
public class ReentrantDemo {
    
    public synchronized void methodA() {
        System.out.println("A 开始");
        methodB();  // 重入！
        System.out.println("A 结束");
    }
    
    public synchronized void methodB() {
        System.out.println("B 开始");
        methodC();  // 重入！
        System.out.println("B 结束");
    }
    
    public synchronized void methodC() {
        System.out.println("C 执行");  // 重入！
    }
}
```

## 🔥 扩展追问 3：synchronized 和 ReentrantLock 区别？

> **A:** 对比表格：

| 对比维度 | synchronized | ReentrantLock |
|---------|--------------|---------------|
| **底层实现** | Monitor（内核态） | AQS + CAS（用户态为主） |
| **是否可重入** | ✅ 是 | ✅ 是 |
| **公平/非公平** | 非公平 | 两者皆可 |
| **tryLock** | ❌ 不支持 | ✅ 支持 |
| **LockInterruptibly** | ❌ 不支持 | ✅ 支持 |
| **Condition** | ❌ 不支持 | ✅ 支持（多条件队列） |
| **超时** | ❌ 不支持 | ✅ 支持（tryLock(timeout)） |
| **性能** | JDK 6+ 优化后相近 | 可中断锁更好 |

## 🔥 扩展追问 4：锁消除（Lock Elision）？

> **A:** JIT 编译器的逃逸分析优化：

```java
/**
 * 锁消除：JIT 分析发现对象不逃逸，直接消除 synchronized
 */
public class LockElisionDemo {
    
    // StringBuffer 是局部变量，不逃逸 → 锁消除
    public String concat(String a, String b) {
        StringBuffer sb = new StringBuffer();
        sb.append(a);
        sb.append(b);
        // JIT 可能消除 synchronized，因为 sb 不会逃逸
        // 相当于：无 synchronized
        return sb.toString();
    }
}

// 对比：sb 逃逸了，不能锁消除
public String concatEscape(String a, String b) {
    StringBuffer sb = new StringBuffer();
    sb.append(a);
    sb.append(b);
    return sb;  // sb 逃逸，JIT 不能消除 synchronized
}
```

## 一句话总结

> **synchronized 锁升级路径：无锁 → 偏向锁 → 轻量级锁 → 重量级锁（JDK 15+ 默认禁用偏向锁），偏向锁记录线程 ID 避免 CAS、撤销开销大被废弃，轻量级锁用 CAS 替换对象头到线程栈、轻量级锁自适应自旋由 JVM 动态调整次数，Monitor 中 EntryList 存 BLOCKED 线程等待竞争锁、WaitSet 存 WAITING 线程等待 notify，锁升级不可逆。**

## 面试加分话术

> "之前优化过一个高并发下单系统，用 JMH 压测发现 synchronized 的锁竞争占 CPU 15%。最后通过三个方向解决：① 把热点锁对象拆分成无锁数组（LongAdder 做库存扣减），② 用 StampedLock 替代 ReentrantReadWriteLock 读多写少场景，③ 禁用了偏向锁（-XX:-UseBiasedLocking），因为服务是短生命周期容器、线程复用率低，偏向锁反而增加开销。压测下来 QPS 从 3000 提到 8000。"
