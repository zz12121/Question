## 二、核心区域详解

### 1. 程序计数器（PC Register）—— 线程私有

```java
/**
 * 程序计数器：记录当前线程正在执行的字节码指令地址
 * 
 * 为什么需要？
 * - Java多线程是通过CPU时间片轮转实现的
 * - 线程切换时需要记录执行位置，以便恢复
 * - 如果是 native 方法，则为空（Undefined）
 */
```

**存储内容**：
- 字节码指令地址（分支、循环、跳转、异常处理都依赖它）
- `native` 方法：`null`（因为不归 JVM 管）

**特点**：
- 唯一 **不会发生 OOM（OutOfMemoryError）** 的区域
- 生命周期与线程一致

---

### 2. 虚拟机栈（VM Stack）—— 线程私有

```java
/**
 * 虚拟机栈：每个方法调用创建一个栈帧（Stack Frame）
 * 方法调用 → 栈帧入栈 → 方法执行 → 栈帧出栈
 */
```

**每个栈帧包含 4 部分**：

```
┌────────────────────────────────────────────────────────────┐
│                      栈帧（Stack Frame）结构                │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                   局部变量表（Local Variables）        │  │
│  │  - 参数列表、局部变量                                  │  │
│  │  - 按变量槽（Slot）存储，1 Slot = 32bit/64bit          │  │
│  │  - long/double 占 2 Slot                              │  │
│  │  - 非基本类型存引用地址                                │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                            │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                   操作数栈（Operand Stack）             │  │
│  │  - 指令操作的数据仓库                                  │  │
│  │  - 最大深度在编译时确定（max_stack）                   │  │
│  │  - 例：iadd 将两个 int 出栈相加，结果入栈              │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                            │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                   动态链接（Dynamic Linking）          │  │
│  │  - 符号引用 → 直接引用（运行时解析）                     │  │
│  │  - 方法区中方法的内存地址                              │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                            │
│  ┌──────────────────────────────────────────────────────┐  │
│  │                   方法返回地址（Return Address）       │  │
│  │  - 正常返回：PC 值恢复                                 │  │
│  │  - 异常返回：异常表查找处理者                          │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**栈帧示意图**：

```
调用 main() → 调用 methodA() → 调用 methodB()

栈顶
┌────────────┐
│ methodB() │  ← 当前执行
│ 栈帧       │
├────────────┤
│ methodA() │
│ 栈帧       │
├────────────┤
│  main()   │
│ 栈帧       │
└────────────┘
栈底
```

### 3. 本地方法栈（Native Method Stack）—— 线程私有

```java
/**
 * 本地方法栈：为 JVM 执行 native（C/C++）方法服务
 * 
 * 与虚拟机栈区别：
 * - 虚拟机栈：执行 Java 方法（字节码）
 * - 本地方法栈：执行 native 方法（JNI 调用底层）
 */
```

**常见 native 方法**：
- `Object.hashCode()` → C++ 实现
- `Thread.start()` → pthread_create
- `System.arraycopy()` → C 实现

---

### 4. 堆（Heap）—— 线程共享

```java
/**
 * 堆：存储所有对象实例和数组
 * 
 * JDK 8+ 内存布局：
 * - Young Generation（年轻代）
 *   - Eden 区（8/10）
 *   - Survivor S0 / S1（1/10，各 1/10）
 * - Old Generation（老年代）（2/3）
 * 
 * JDK 8 之前：PermGen（永久代）
 * JDK 8+：Metaspace（元空间，堆外内存）
 */
```

```
堆内存布局（JDK 8+）：

         堆总大小 = 年轻代 + 老年代
         ┌──────────────────────┬──────────────┐
         │     年轻代 (1/3)       │ 老年代 (2/3) │
         ├─────────┬──────┬─────┤              │
         │ Eden 8/10│ S0 1/10 │ S1 │              │
         │          │ Survivor│    │              │
         │          │  交换区  │    │              │
         └─────────┴──────┴─────┘              │
```

---

### 5. 方法区（Method Area）—— 线程共享

```java
/**
 * 方法区：存储类结构信息（JDK 8+ 改用 Metaspace）
 * 
 * JDK 7：PermGen（永久代）→ 有 OOM 风险
 * JDK 8+：Metaspace → 使用本地内存（OS可用内存）
 */
```

**方法区存储内容**：

| 存储内容 | 说明 |
|---------|------|
| 类型信息 | 类名、父类、修饰符、接口列表 |
| 字段信息 | 字段名、类型、修饰符 |
| 方法信息 | 方法名、参数列表、返回类型、字节码 |
| 静态变量 | static 修饰的变量（JDK 7 还在堆） |
| 运行时常量池 | 符号引用、字面量（String Pool 在堆） |
| JIT 编译缓存 | 方法编译后的本地代码 |

---

### 6. 运行时常量池（Runtime Constant Pool）

```java
/**
 * 运行时常量池：类加载后存放常量和符号引用
 * 
 * 包含：
 * - 字面量：final 常量、字符串字面量
 * - 符号引用：类/方法/字段的全限定名
 * 
 * String Pool（字符串常量池）：
 * - JDK 7 及之前：在方法区（PermGen）
 * - JDK 8+：移到堆中（Interned Strings）
 */
```

## 三、线程私有 vs 线程共享

| 区域 | 线程私有/共享 | 异常类型 | 生命周期 |
|------|-------------|---------|---------|
| **程序计数器** | 私有 | ❌ 无 | 与线程一致 |
| **虚拟机栈** | 私有 | StackOverflowError / OOM | 与线程一致 |
| **本地方法栈** | 私有 | StackOverflowError / OOM | 与线程一致 |
| **堆** | 共享 | OutOfMemoryError | 与 JVM 一致 |
| **方法区** | 共享 | OutOfMemoryError | 与 JVM 一致 |

## 四、递归调用与栈溢出

```java
/**
 * 递归调用 100 层会栈溢出吗？
 * 
 * 答案：取决于 -Xss 参数和每个栈帧的大小
 */
```

**计算示例**：

```
-Xss 默认值：1MB（大多数 JVM）

每个栈帧大小估算：
- 局部变量表：4-8 个 Slot ≈ 32-64 字节
- 操作数栈：2-4 个 Slot ≈ 16-32 字节
- 其他开销：16-32 字节
- 每个栈帧 ≈ 64-128 字节

1MB / 64字节 ≈ 16,384 层
1MB / 128字节 ≈ 8,192 层

结论：100 层递归 不会栈溢出（远未达到上限）
```

**常见 `-Xss` 配置**：

| 配置 | 栈大小 | 100 层递归 |
|-----|-------|-----------|
| `-Xss1m`（默认） | 1MB | ✅ 安全 |
| `-Xss512k` | 512KB | ✅ 安全 |
| `-Xss256k` | 256KB | ✅ 安全（保守估算） |
| `-Xss128k` | 128KB | ⚠️ 可能溢出（栈帧较大时） |

**实际场景**：

```java
// 100 层简单递归：安全
public int sum(int n) {
    if (n <= 1) return n;
    return n + sum(n - 1);
}

// 100 层复杂递归（大数据局部变量）：可能溢出
public void recursiveWithLargeVars(int[] arr) {
    int[] local = new int[10000];  // 大数组入局部变量表
    recursiveWithLargeVars(arr);
}
```

## 🔥 扩展追问 1：为什么 long/double 占 2 个 Slot？

> **A:** 因为 `long`（8字节）和 `double`（8字节）在 32 位 JVM 中无法用 1 个 32bit Slot 存储，所以需要 2 个连续 Slot。这是 **JVM 规范** 规定的，是为了兼容 32 位处理器。

```java
public class SlotExample {
    public void test() {
        long a = 1L;    // 占用 Slot 1 和 Slot 2
        int b = 2;      // 占用 Slot 3
        double c = 1.0; // 占用 Slot 4 和 Slot 5
        int d = 3;      // 占用 Slot 6
    }
}
// 局部变量表：[a(L)][a(L)][b][c(D)][c(D)][d]
```

## 🔥 扩展追问 2：方法区在 JDK 8 后有什么变化？

> **A:** **PermGen → Metaspace 迁移**，这是 JDK 8 最大的内存变化之一：

| 对比项 | JDK 7 PermGen | JDK 8+ Metaspace |
|-------|---------------|------------------|
| 位置 | 堆内（受 -XX:MaxPermSize 限制） | 堆外（本地内存） |
| 默认大小 | 约 64MB | 无上限（受 OS 限制） |
| OOM 风险 | 频繁（类加载多时） | 极少（除非耗尽本地内存） |
| 控制参数 | `-XX:PermSize/MaxPermSize` | `-XX:MetaspaceSize/MaxMetaspaceSize` |

```bash
# JDK 8 配置
-XX:MetaspaceSize=256m      # 初始阈值
-XX:MaxMetaspaceSize=1g     # 最大限制
-XX:MinMetaspaceSize=64m    # 最小保留

# JDK 7 配置（已废弃）
-XX:PermSize=128m
-XX:MaxPermSize=512m
```

## 🔥 扩展追问 3：为什么要分年轻代和老年代？

> **A:** **分代垃圾回收** 的基础假设：
> - **大多数对象朝生夕死**（80% 的对象生命周期很短）
> - **少数对象长期存活**（缓存、连接池等）
> 
> 分代后，Minor GC 只清理年轻代（频率高、速度快），Full GC 才清理老年代。

## 🔥 扩展追问 4：逃逸分析（Escape Analysis）对栈分配的影响？

> **A:** **JDK 6+ 支持栈上分配（Stack Allocation）**：

```java
/**
 * 逃逸分析判断对象是否逃逸出方法/线程
 * 
 * 未逃逸 → 栈上分配（无需 GC）
 * 逃逸   → 堆上分配（需要 GC）
 */
public class EscapeAnalysis {
    
    // 逃逸分析：obj 未逃逸，可以在栈上分配
    public void method() {
        Object obj = new Object();
        obj.hashCode();  // 只在方法内使用
        // 方法结束时 obj 随栈帧一起销毁
    }
    
    // 逃逸：返回对象引用，线程可见
    public Object returnEscape() {
        Object obj = new Object();
        return obj;  // 逃逸！
    }
}
```

**`-XX:+DoEscapeAnalysis`**（默认开启）：启用逃逸分析，触发栈上分配。

## 🔥 扩展追问 5：直接内存（Direct Memory）是什么？

> **A:** **NIO 的零拷贝技术**，使用 `ByteBuffer.allocateDirect()` 分配的堆外内存：

```java
/**
 * 直接内存：不受 -Xmx 控制，属于 OS 本地内存
 * 
 * 常见使用场景：
 * - Netty 高性能网络框架
 * - Kafka 消息队列
 * - Redis（Redis 6+ 使用）
 */
```

```
┌─────────────────────────────────────────────────────────┐
│                      JVM 进程内存                        │
├─────────────────────┬───────────────────────────────────┤
│   堆内存 (-Xmx)     │     直接内存（堆外）              │
│                     │                                   │
│  [    Heap    ]    │  [  Direct Memory  ]             │
│                     │                                   │
│  - 对象实例          │  - NIO ByteBuffer                 │
│  - 数组              │  - Netty Channel Buffers          │
│  - 运行时常量池       │  - 系统 Socket 缓冲区             │
└─────────────────────┴───────────────────────────────────┘
```

**配置**：`‑XX:MaxDirectMemorySize=4g`（默认与 -Xmx 相等）

## 🔥 扩展追问 6：String Pool 在哪？JDK 8 后有变化吗？

> **A:** **JDK 7 之前在方法区（PermGen），JDK 8 移到堆中**：

```
┌────────────────────────────────────────────────────────────┐
│                         JDK 7                              │
├─────────────────────┬──────────────────────────────────────┤
│      方法区         │              堆                       │
│  ┌────────────────┐ │                                      │
│  │  String Pool   │ │   其他对象、数组                      │
│  │  (PermGen)     │ │                                      │
│  └────────────────┘ │                                      │
└─────────────────────┴──────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│                       JDK 8+                               │
├────────────────────────────────────────────────────────────┤
│                      方法区（元空间）                       │
│  - 类信息、字节码、JIT 缓存                                 │
│                                                              │
├────────────────────────────────────────────────────────────┤
│                         堆                                 │
│  ┌──────────────────────────────────────────────────────┐ │
│  │  String Pool（字符串常量池）  ← 从 PermGen 移过来       │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                              │
│  其他对象、数组                                             │
└────────────────────────────────────────────────────────────┘
```

```java
// intern() 方法：将字符串放入 String Pool
String s1 = new String("abc");        // 堆对象
String s2 = s1.intern();              // 取池中引用
String s3 = "abc";                     // 字面量，编译时放入池

System.out.println(s1 == s3);  // false（s1 是堆对象，s3 是池引用）
System.out.println(s2 == s3);  // true（都是池引用）
```

## 一句话总结

> **JVM 运行时数据区 = 线程私有（程序计数器+虚拟机栈+本地方法栈）+ 线程共享（堆+方法区），递归 100 层通常不会栈溢出（默认 1MB 栈空间），-Xss 控制的是每个线程的栈大小。**

## 面试加分话术

> "在之前负责的交易系统压测中，我们遇到过栈溢出问题——业务场景是期权定价的递归计算，深度达到 3000+ 层。排查后发现是因为局部变量表中存了一个大数组（`double[] prices = new double[10000]`），每个栈帧占用了约 80KB，导致默认 1MB 栈空间只够 12 层递归。最终通过三个方向解决：① 改用尾递归优化，② 改用显式栈（循环）替代递归，③ 调大 -Xss 到 2MB。这个场景让我深刻理解了**栈帧结构和局部变量表大小对递归深度的影响**，以及**为什么 JDK 官方推荐用循环替代递归**。"
