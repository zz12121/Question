## 一、Generated Column（生成列）概述

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        MySQL 生成列（Generated Column）                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐  │
│   │                                                                  │  │
│   │  生成列 = 由表达式计算得出的列                                    │  │
│   │  ─────────────────────────────                                   │  │
│   │                                                                  │  │
│   │  CREATE TABLE orders (                                          │  │
│   │      id INT,                                                    │  │
│   │      price DECIMAL(10, 2),                                      │  │
│   │      quantity INT,                                              │  │
│   │      total_price DECIMAL(10, 2) AS (price * quantity),         │  │
│   │      -- ↑ 生成列：自动计算，不需要手动维护                        │  │
│   │      created_at DATETIME                                        │  │
│   │  );                                                             │  │
│   │                                                                  │  │
│   │  INSERT INTO orders (price, quantity) VALUES (99.9, 2);        │  │
│   │  -- total_price 自动为 199.8                                    │  │
│   │                                                                  │  │
│   └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│   为什么需要生成列？                                                    │
│   ───────────────                                                     │
│   1. 避免重复计算（将计算结果存起来）                                   │
│   2. 为表达式结果建立索引                                              │
│   3. 保证数据一致性（计算逻辑集中在一处）                               │
│   4. 简化查询（可以直接 SELECT total_price）                           │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 二、VIRTUAL vs STORED 模式

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    VIRTUAL vs STORED 生成列对比                          │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐  │
│   │                        VIRTUAL（虚拟列）                         │  │
│   │  ────────────────────────────────                              │  │
│   │                                                                 │  │
│   │   特点：                                                        │  │
│   │   - 不存储在磁盘上                                              │  │
│   │   - 读取时实时计算                                             │  │
│   │   - 插入/更新时不增加 IO 开销                                   │  │
│   │   - 磁盘空间 = 0                                               │  │
│   │   - 支持索引（InnoDB）                                         │  │
│   │                                                                 │  │
│   │   ┌──────────────────────────────────────────────────────┐   │  │
│   │   │                                                        │   │  │
│   │   │  SELECT 时：                                           │   │  │
│   │   │  ──────                                                │   │  │
│   │   │                                                        │   │  │
│   │   │  数据行 ──▶ 读取行数据 ──▶ 实时计算 ──▶ 返回结果        │   │  │
│   │   │                                                        │   │  │
│   │   │  磁盘：无额外存储                                         │   │  │
│   │   │  读取：每次都计算                                         │   │  │
│   │   │                                                        │   │  │
│   │   └──────────────────────────────────────────────────────┘   │  │
│   │                                                                 │  │
│   └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐  │
│   │                        STORED（存储列）                          │  │
│   │  ────────────────────────────────                              │  │
│   │                                                                 │  │
│   │   特点：                                                        │  │
│   │   - 存储在磁盘上                                               │  │
│   │   - 插入/更新时计算并写入磁盘                                   │  │
│   │   - 读取时直接返回（无需计算）                                  │  │
│   │   - 增加磁盘空间                                                │  │
│   │   - 必须支持索引                                               │  │
│   │                                                                 │  │
│   │   ┌──────────────────────────────────────────────────────┐   │  │
│   │   │                                                        │   │  │
│   │   │  INSERT/UPDATE 时：                                     │   │  │
│   │   │  ────────────────                                       │   │  │
│   │   │                                                        │   │  │
│   │   │  数据行 + 计算结果 ──▶ 写入磁盘                          │   │  │
│   │   │                                                        │   │  │
│   │   │  SELECT 时：                                            │   │  │
│   │   │  ──────                                                 │   │  │
│   │   │                                                        │   │  │
│   │   │  磁盘读取 ──▶ 直接返回                                   │   │  │
│   │   │  磁盘：占用空间                                          │   │  │
│   │   │  读取：更快（无需计算）                                    │   │  │
│   │   │                                                        │   │  │
│   │   └──────────────────────────────────────────────────────┘   │  │
│   │                                                                 │  │
│   └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.1 详细对比表

| 特性 | VIRTUAL | STORED |
|-----|---------|---------|
| **存储位置** | ❌ 不存储在磁盘 | ✅ 存储在磁盘 |
| **读取时计算** | ✅ 是 | ❌ 否 |
| **写入时计算** | ❌ 否 | ✅ 是 |
| **磁盘空间占用** | 0 | 需要空间 |
| **索引支持** | ✅ 支持 | ✅ 必须支持 |
| **INSERT 性能** | 快（无额外 IO） | 慢（需计算） |
| **SELECT 性能** | 稍慢（需计算） | 快（直接读） |
| **适合场景** | 读取多、计算简单 | 写入少、查询快 |
| **默认值** | VIRTUAL（省略时） | 需显式指定 |

### 2.2 代码示例

```sql
-- 创建测试表
CREATE TABLE employees (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    birth_date DATE,
    
    -- VIRTUAL 列：计算年龄（不存储，读取时计算）
    age INT AS (TIMESTAMPDIFF(YEAR, birth_date, CURDATE())) VIRTUAL,
    
    -- STORED 列：计算薪资等级（存储，插入时计算）
    salary DECIMAL(10,2),
    salary_grade VARCHAR(1) AS (
        CASE 
            WHEN salary < 5000 THEN 'A'
            WHEN salary < 10000 THEN 'B'
            ELSE 'C'
        END
    ) STORED,
    
    -- JSON 字段的生成列
    data JSON,
    -- 从 JSON 中提取字段（VIRTUAL）
    user_id VARCHAR(50) AS (JSON_UNQUOTE(JSON_EXTRACT(data, '$.user_id'))) VIRTUAL
);

-- 插入数据
INSERT INTO employees (name, birth_date, salary, data) 
VALUES ('张三', '1990-05-15', 8000, '{"user_id": "U001"}');

-- 查询时 age 和 user_id 自动计算
SELECT name, age, salary_grade, user_id FROM employees;
-- age = 当前年份 - 1990 = 动态变化
-- salary_grade = 'B'（已存储）
-- user_id = 'U001'（从 JSON 实时提取）
```

## 三、函数索引（Functional Index）

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        MySQL 函数索引（MySQL 8.0+）                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   问题背景：                                                            │
│   ─────────                                                            │
│   普通索引：                                                            │
│   CREATE INDEX idx_name ON users(name);                               │
│   -- 只能匹配：WHERE name = '张三'                                     │
│   -- 无法匹配：WHERE LOWER(name) = '张三'                              │
│   -- 无法匹配：WHERE YEAR(created_at) = 2024                          │
│                                                                         │
│   函数索引：                                                            │
│   ─────────                                                            │
│   CREATE INDEX idx_lower_name ON users(LOWER(name));                  │
│   -- 可以匹配：WHERE LOWER(name) = '张三'                              │
│   -- 本质：对 LOWER(name) 的结果建立索引                                │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐  │
│   │                                                                  │  │
│   │  实现方式：生成列 + 普通索引                                       │  │
│   │  ──────────────────────────                                      │  │
│   │                                                                  │  │
│   │  MySQL 8.0+ 语法（简化写法）：                                   │  │
│   │  CREATE INDEX idx_func ON table(func(column));                 │  │
│   │                                                                  │  │
│   │  MySQL 5.7 等效写法：                                            │  │
│   │  ALTER TABLE table ADD COLUMN col func(...) AS (...);          │  │
│   │  CREATE INDEX idx_func ON table(col);                          │  │
│   │                                                                  │  │
│   └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.1 函数索引使用场景

```sql
-- 场景 1：大小写不敏感的查询
CREATE TABLE users (
    id INT PRIMARY KEY,
    email VARCHAR(255),
    created_at DATETIME
);

-- 方法 1：使用函数索引（MySQL 8.0+）
CREATE INDEX idx_email_lower ON users(LOWER(email));

-- 查询
SELECT * FROM users WHERE LOWER(email) = 'user@example.com';
-- ↑ 使用 idx_email_lower 索引！

-- 方法 2：使用生成列（MySQL 5.7）
ALTER TABLE users 
ADD COLUMN email_lower VARCHAR(255) GENERATED ALWAYS AS (LOWER(email));
CREATE INDEX idx_email_lower ON users(email_lower);


-- 场景 2：日期函数查询
CREATE TABLE orders (
    id INT PRIMARY KEY,
    order_date DATETIME,
    amount DECIMAL(10, 2)
);

-- 函数索引：按年份查询
CREATE INDEX idx_year ON orders(YEAR(order_date));

-- 查询
SELECT * FROM orders WHERE YEAR(order_date) = 2024;
-- ↑ 使用 idx_year 索引！


-- 场景 3：JSON 字段查询
CREATE TABLE products (
    id INT PRIMARY KEY,
    data JSON
);

-- 从 JSON 提取字段建立索引
CREATE INDEX idx_product_type 
ON products((JSON_UNQUOTE(JSON_EXTRACT(data, '$.type'))));

-- 查询
SELECT * FROM products 
WHERE JSON_UNQUOTE(JSON_EXTRACT(data, '$.type')) = 'electronics';
-- ↑ 使用 idx_product_type 索引！


-- 场景 4：字符串前缀查询（MySQL 8.0+ 支持函数）
CREATE TABLE messages (
    id INT PRIMARY KEY,
    content TEXT
);

-- 对前 10 个字符建立索引
CREATE INDEX idx_content_prefix 
ON messages((LEFT(content, 10)));

-- 查询
SELECT * FROM messages WHERE LEFT(content, 10) = '用户反馈：';
-- ↑ 使用 idx_content_prefix 索引！
```

### 3.2 函数索引的限制

```sql
-- ❌ 函数索引的限制

-- 1. 必须是确定性函数（每次输入返回相同输出）
CREATE INDEX idx_now ON orders(NOW());  -- ❌ 报错！NOW() 不是确定性的

-- 2. 不支持自定义函数
CREATE FUNCTION get_grade(salary DECIMAL) RETURNS VARCHAR(1)
    RETURN CASE WHEN salary < 5000 THEN 'A' ELSE 'B' END;
    
CREATE INDEX idx_grade ON orders(get_grade(salary));  -- ❌ 可能不支持

-- 3. 不支持聚合函数
CREATE INDEX idx_sum ON orders(SUM(amount));  -- ❌ 不支持

-- 4. 支持的函数类型
-- - 数学函数：ABS(), CEIL(), FLOOR(), ROUND()...
-- - 字符串函数：LOWER(), UPPER(), LEFT(), RIGHT(), SUBSTRING()...
-- - 日期函数：YEAR(), MONTH(), DAY(), DATE()...
-- - JSON 函数：JSON_UNQUOTE(), JSON_EXTRACT()...
-- - 类型转换：CAST(), CONVERT()...
```

## 四、降序索引（Descending Index）

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        MySQL 8.0 降序索引                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   MySQL 5.7 及之前：                                                    │
│   ──────────────────                                                   │
│   所有索引都是 ASC（升序）                                              │
│   - 即使写 DESC，实际也是 ASC                                          │
│   - 降序扫描需要额外排序                                                │
│                                                                         │
│   MySQL 8.0+：                                                          │
│   ───────                                                              │
│   真正的降序索引                                                        │
│   - 可以指定 ASC 或 DESC                                               │
│   - 优化器可以选择最优方向扫描                                          │
│   - 避免额外的 filesort                                                │
│                                                                         │
│   ┌─────────────────────────────────────────────────────────────────┐  │
│   │                                                                  │  │
│   │  CREATE INDEX idx_score_desc ON students(score DESC);          │  │
│   │  CREATE INDEX idx_name_asc ON students(name ASC);              │  │
│   │                                                                  │  │
│   │  -- 复合索引：不同列可以不同方向                                   │  │
│   │  CREATE INDEX idx_composite ON students(                        │  │
│   │      class_id ASC,     -- 班级升序                               │  │
│   │      score DESC,      -- 成绩降序                               │  │
│   │      name ASC         -- 姓名升序                               │  │
│   │  );                                                             │  │
│   │                                                                  │  │
│   └─────────────────────────────────────────────────────────────────┘  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 4.1 降序索引的用途

```sql
-- 场景：班级排名查询
CREATE TABLE students (
    id INT PRIMARY KEY,
    class_id INT,
    name VARCHAR(50),
    score INT,
    INDEX idx_ranking (class_id, score DESC)  -- 复合降序索引
);

-- 查询：每个班级成绩最好的学生
SELECT class_id, name, score
FROM students
ORDER BY class_id, score DESC
LIMIT 1;

-- EXPLAIN 分析
EXPLAIN SELECT class_id, name, score
FROM students
ORDER BY class_id, score DESC
LIMIT 1;

-- 结果：
-- type: index
-- key: idx_ranking
-- rows: 3
-- Extra: Using index

-- ↑ 使用了索引扫描，不需要额外排序（filesort）
```

### 4.2 为什么 MySQL 5.7 的 DESC INDEX 实际上是 ASC？

```sql
-- MySQL 5.7 中执行：
CREATE INDEX idx_score_desc ON students(score DESC);

-- 问题：MySQL 5.7 会忽略 DESC，直接创建 ASC 索引
-- 即：idx_score_desc 实际上和 idx_score ASC 一样！

-- 验证：
SHOW INDEX FROM students;

-- 输出：
-- Key_name: idx_score_desc
-- Seq_in_index: 1
-- Column_name: score
-- Collation: A  ← A 表示 ASC，不是 D（降序）！

-- 原因：MySQL 5.7 的存储引擎层面不支持降序索引
--       DESC 只是语法糖，实际被忽略

-- 后果：
-- 查询 SELECT * FROM students ORDER BY score DESC;
-- 需要：filesort（额外排序）
-- 
-- MySQL 8.0 修复：
CREATE INDEX idx_score_desc ON students(score DESC);
-- SHOW INDEX 结果：Collation: D（真正的降序）

-- 查询 SELECT * FROM students ORDER BY score DESC;
-- 不需要 filesort，直接使用索引扫描
```

### 4.3 降序索引的工作原理

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        降序索引的物理存储                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   普通 ASC 索引（class_id ASC, score ASC）：                           │
│   ─────────────────────────────────────────                            │
│                                                                         │
│   B-Tree 叶子节点：                                                     │
│   ┌─────────────────────────────────────────────────────────┐          │
│   │  (class_id, score)                                      │          │
│   ├─────────────────────────────────────────────────────────┤          │
│   │  (1, 60) → 张三                                         │          │
│   │  (1, 75) → 李四                                         │          │
│   │  (1, 90) → 王五    ← 分数升序                           │          │
│   │  (2, 55) → 赵六                                         │          │
│   │  (2, 80) → 钱七                                         │          │
│   │  (2, 95) → 孙八    ← 分数升序                           │          │
│   └─────────────────────────────────────────────────────────┘          │
│                                                                         │
│   降序索引（class_id ASC, score DESC）：                               │
│   ─────────────────────────────────────────                            │
│                                                                         │
│   B-Tree 叶子节点：                                                     │
│   ┌─────────────────────────────────────────────────────────┐          │
│   │  (class_id, score)                                      │          │
│   ├─────────────────────────────────────────────────────────┤          │
│   │  (1, 95) → 王五    ← 班级1分数最高                      │          │
│   │  (1, 80) → 李四                                         │          │
│   │  (1, 60) → 张三    ← 班级1分数最低                      │          │
│   │  (2, 98) → 孙八    ← 班级2分数最高                      │          │
│   │  (2, 85) → 钱七                                         │          │
│   │  (2, 55) → 赵六    ← 班级2分数最低                      │          │
│   └─────────────────────────────────────────────────────────┘          │
│                                                                         │
│   关键：索引按 (class_id ASC, score DESC) 排序                          │
│        扫描时直接按这个顺序读取，无需额外排序                              │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 五、综合对比

| 特性 | 生成列 | 函数索引 | 降序索引 |
|-----|--------|---------|---------|
| **MySQL 版本** | 5.7+ | 8.0+ | 8.0+ |
| **存储数据** | VIRTUAL（不存）/STORED（存） | 索引存储 | 索引存储 |
| **建立方式** | 表达式 AS 列类型 | func(col) | col DESC |
| **索引类型** | 普通索引 | 普通索引 | 普通索引 |
| **用途** | 计算列、JSON 提取 | 表达式查询 | 排序优化 |
| **磁盘开销** | VIRTUAL=0 / STORED=有 | 有 | 有 |

## 🔥 扩展追问 1：函数索引在 MySQL 5.7 中如何实现？

> **A:** 用生成列模拟函数索引：

```sql
-- MySQL 5.7：生成列 + 普通索引 = 函数索引

-- 1. 创建生成列
ALTER TABLE users
ADD COLUMN email_lower VARCHAR(255) 
GENERATED ALWAYS AS (LOWER(email));

-- 2. 对生成列建立索引
CREATE INDEX idx_email_lower ON users(email_lower);

-- 3. 查询时使用函数（但实际匹配生成列）
SELECT * FROM users WHERE LOWER(email) = 'test@example.com';
-- 等价于：WHERE email_lower = 'test@example.com'

-- 注意：需要写成 LOWER(email) 才能让优化器识别
-- 有些版本可能需要显式写 email_lower
```

## 🔥 扩展追问 2：为什么函数索引需要生成列？

> **A:** 索引必须存储确定的值：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    为什么函数索引依赖生成列？                             │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   问题：索引需要存储什么？                                              │
│   ───────────────────────                                               │
│                                                                         │
│   普通索引：存储列值                                                    │
│   ──────────────                                                       │
│   CREATE INDEX idx ON users(name);                                     │
│   索引存储：'Alice', 'Bob', 'Carol'...                                 │
│   查询：WHERE name = 'Bob' → 直接定位                                   │
│                                                                         │
│   函数索引：存储函数计算结果                                            │
│   ─────────────────────                                                  │
│   CREATE INDEX idx ON users(LOWER(name));                             │
│   索引存储：'alice', 'bob', 'carol'...                                 │
│   查询：WHERE LOWER(name) = 'bob' → 直接定位                            │
│                                                                         │
│   核心问题：                                                            │
│   ─────────                                                            │
│   索引是 B-Tree 结构，需要存储确定值                                    │
│   - VIRTUAL 列：读取时实时计算，不存储                                  │
│   - 但索引需要存储值！                                                  │
│                                                                         │
│   解决：                                                                │
│   ───                                                                  │
│   MySQL 8.0+：                                                         │
│   - 语法上写 func(col)                                                 │
│   - 内部自动创建隐藏的虚拟列                                            │
│   - 对这个虚拟列建立索引                                                │
│                                                                         │
│   MySQL 5.7：                                                          │
│   - 需要手动创建生成列                                                  │
│   - 再手动创建索引                                                      │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 3：函数索引和表达式索引的区别？

> **A:** 术语问题，MySQL 8.0+ 统称为"函数索引"：

```sql
-- MySQL 8.0+ 支持两种写法（效果相同）

-- 写法 1：函数索引（推荐）
CREATE INDEX idx_func ON users(LOWER(email));

-- 写法 2：括号表达式（等效）
CREATE INDEX idx_expr ON users((LOWER(email)));

-- 复合索引中的表达式
CREATE INDEX idx_complex ON users(
    (LOWER(email)),        -- 表达式
    status,                -- 普通列
    (YEAR(created_at))     -- 表达式
);

-- 注意：表达式索引的括号是必须的！
CREATE INDEX idx_wrong ON users(LOWER(email));  -- ✅ 正确
CREATE INDEX idx_right ON users((LOWER(email))); -- ✅ 正确（推荐）
```

---

## 一句话总结

> **MySQL 生成列是自动计算的列（VIRTUAL 实时计算不占磁盘、STORED 写入时计算占磁盘），函数索引是对表达式结果建立索引（MySQL 8.0+ 直接写 `func(col)`、5.7 用生成列模拟），降序索引（MySQL 8.0+）让复合索引中的列可以独立指定升/降序避免额外排序，MySQL 5.7 的 DESC 语法会被忽略实际创建的是 ASC 索引。**

> "之前重构过一个订单系统，报表查询 `WHERE YEAR(created_at) = 2024 AND MONTH(created_at) = 3` 特别慢，EXPLAIN 显示全表扫描。升级到 MySQL 8.0 后建了 `INDEX idx_ym (YEAR(created_at), MONTH(created_at))`，查询从 3 秒降到 50 毫秒。再比如用户模块的登录查询，原来是 `WHERE email = ? OR username = ?`，后来统一转成小写后建了 `LOWER()` 函数索引，`WHERE LOWER(email) = LOWER(?)` 直接走索引。降序索引的坑也踩过——在 MySQL 5.7 创建了 DESC 索引以为优化了排序，实际 EXPLAIN 还是有 filesort，升级到 8.0 才真正生效。"

## 生产配置清单

```sql
-- 生成列示例
CREATE TABLE orders (
    id INT PRIMARY KEY,
    price DECIMAL(10,2),
    quantity INT,
    
    -- VIRTUAL：自动计算，不占用磁盘
    total_price DECIMAL(10,2) AS (price * quantity) VIRTUAL,
    
    -- STORED：计算后存储，读取更快
    created_at DATETIME,
    order_year INT AS (YEAR(created_at)) STORED
);

-- 函数索引（MySQL 8.0+）
CREATE INDEX idx_lower_email ON users(LOWER(email));
CREATE INDEX idx_year_month ON orders((YEAR(created_at)), (MONTH(created_at)));

-- 降序索引
CREATE INDEX idx_score_desc ON students(class_id, score DESC);

-- JSON 函数索引
CREATE TABLE products (
    data JSON,
    type VARCHAR(50) GENERATED ALWAYS AS (JSON_UNQUOTE(JSON_EXTRACT(data, '$.type'))) STORED
);
CREATE INDEX idx_type ON products(type);

-- 查看索引
SHOW INDEX FROM table_name;

-- 查看查询计划
EXPLAIN SELECT * FROM orders WHERE YEAR(created_at) = 2024;
```
