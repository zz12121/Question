## 2. TRANSACTIONS 段

### 2.1 输出内容解析

```
---TRANSACTIONS---
Trx id counter 283947562            ← 事务ID计数器（下一个新事务ID）
Purge done for trx's n:o < 283947558 ← 已完成purge的事务ID（落后3个）
History list length 4                ← 回滚段中的undo记录数（核心指标）
```

**关键指标含义：**

| 字段 | 含义 | 健康值 |
|------|------|--------|
| `Trx id counter` | 下一个将分配的事务ID | 无直接参考意义 |
| `Purge done for` | Purge线程已经清理到的位置 | 与trx id counter越接近越好 |
| `History list length` | 未被清理的undo记录总数 | **< 1000**，持续增长=purge跟不上 |
| `Read views` | 当前活跃的MVCC读视图数 | 活跃事务数量参考 |

### 2.2 锁等待链（Lock Wait Chain）

```
---TRANSACTIONS---
LIST OF TRANSACTIONS FOR EACH SESSION:
---TRANSACTION 283947560, ACTIVE 32 sec inserting
mysql tables in use 1, locked 1
LOCK WAIT 3 lock struct(s), heap size 1136, 2 row lock(s)
---TRANSACTION 283947558, ACTIVE 108 sec    ← 阻塞者，活跃108秒
mysql tables in use 1, locked 1
lock struct(s), heap size 1136, 8 row lock(s) ← 持有更多行锁
```

**锁等待链分析流程：**

```
发现问题：TRANSACTION 283947560 状态是 LOCK WAIT
         │
         ▼
找到原因：TRANSACTION 283947558 活跃时间更长(108s vs 32s)
         且持有更多锁(8个 vs 2个)
         │
         ▼
判断依据：8 row lock(s) 说明它在多个行上加了X锁
         阻塞了新事务的插入/更新
         │
         ▼
处理方式：KILL 283947558 或等待其提交/回滚
```

**锁类型详解：**

```
lock struct(s): 锁结构数量
  - 每个 UPDATE/INSERT = 1个锁结构
  - 每个唯一索引记录 = 1个锁结构
  - 范围扫描锁 = N个连续记录的锁

heap size 1136: 锁结构占用的堆内存大小
row lock(s): 影响的行锁数量
```

### 2.3 常见锁等待场景

```sql
-- 场景1：唯一索引冲突
Session A: INSERT INTO orders(id, status) VALUES(100, 'pending');
Session B: INSERT INTO orders(id, status) VALUES(100, 'paid');
-- Session B 被 LOCK WAIT，等待 Session A 的X锁释放

-- 场景2：范围锁冲突
Session A: BEGIN;
           SELECT * FROM products WHERE price > 100 FOR UPDATE;
           -- 锁住 price > 100 的所有记录
Session B: INSERT INTO products(id, price) VALUES(201, 150);
           -- 被锁等待，无法插入

-- 场景3：Gap Lock 阻塞插入
Session A: SELECT * FROM users WHERE age BETWEEN 20 AND 30 FOR UPDATE;
Session B: INSERT INTO users(id, age) VALUES(500, 25);
           -- age=25 落在 [20,30] 区间内，被Gap Lock阻塞
```


## 5. PURGE 线程机制

### 5.1 PURGE 的作用

```
┌─────────────────────────────────────────────────────────────┐
│               MVCC + Undo Log 生命周期                      │
│                                                             │
│  时间轴 ──────────────────────────────────────────────────▶ │
│                                                             │
│  T1时刻 事务A：INSERT INTO orders VALUES(1, 'pending')     │
│          │                                                 │
│          │ 创建 undo log record (type = INSERT)            │
│          ▼                                                 │
│  T2时刻 事务B：UPDATE orders SET status='paid' WHERE id=1   │
│          │                                                 │
│          │ 创建 undo log record (type = UPDATE_OLD)         │
│          │ 旧值 'pending' 被记录在 undo 中                  │
│          ▼                                                 │
│  T3时刻 事务C：SELECT * FROM orders (MVCC快照读)            │
│          │                                                 │
│          │ 看到的是事务B提交前的版本                         │
│          │ 因为事务A和B都还未提交                            │
│          ▼                                                 │
│  T4时刻 事务A、事务B 提交                                    │
│                                                             │
│  T5时刻 PURGE 线程登场！                                    │
│          │                                                 │
│          │ 确认没有活跃事务需要这些 undo 记录后：            │
│          │ 1. 删除物理行（UPDATE_OLD的删除标记）             │
│          │ 2. 释放 undo log 页空间                          │
│          ▼                                                 │
│  T6时刻 磁盘空间恢复，History list length 减小               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 undo log truncate vs purge

```
┌────────────────────────────────────────────────────────────────┐
│              Undo Log 的 truncate 和 purge 区别                │
├─────────────────┬──────────────────────┬───────────────────────┤
│ 操作类型         │ truncate (截断)        │ purge (清除)           │
├─────────────────┼──────────────────────┼───────────────────────┤
│ 作用对象         │ undo segment 中的页    │ undo log record 记录  │
│ 操作层级         │ 物理层（页级别）        │ 逻辑层（记录级别）      │
│ 触发时机         │ undo表空间自动扩展后    │ 每次事务提交后          │
│ 效果             │ 释放表空间给操作系统    │ 删除标记的物理行        │
│ 是否可逆         │ 是（空间可重新分配）    │ 否（数据真正删除）       │
│ 对 History length│ 显著减少              │ 逐步减少                │
│ 对性能影响       │ 瞬间大量I/O            │ 持续少量I/O             │
└─────────────────┴──────────────────────┴───────────────────────┘

truncate 过程：
  undo log 页被标记为 reusable（可重用）
  ↓
  当 undo segment 中的所有页都可重用时
  ↓
  truncate 操作将表空间文件末尾截断
  ↓
  物理文件缩小，空间归还操作系统

purge 过程：
  读取 undo log record
  ↓
  检查该记录对应的行是否被标记为删除
  ↓
  如果没有活跃事务还引用这行
  ↓
  物理删除该行
  ↓
  更新相关索引
```

### 5.3 PURGE 性能影响

```
PURGE 跟不上 → History list length 持续增长 → 三重危害：

┌──────────────────────────────────────────────────────────────┐
│  危害1：事务ID回绕加速                                        │
│  ─────────────────────────────────────────────────────────── │
│  InnoDB 事务ID是48位整数，最大值约 281万亿                      │
│  History list 越长 = 未清理事务越多 = ID回绕越快               │
│  一旦回绕到重叠的ID，旧数据可能被错误认为"已提交"                │
│  虽然InnoDB有防护机制，但回绕压力大时系统会强制Purge            │
│  → 造成突发I/O高峰                                             │
├──────────────────────────────────────────────────────────────┤
│  危害2：MVCC 读视图膨胀                                        │
│  ─────────────────────────────────────────────────────────── │
│  每个事务开启时创建一个 Read View，记录"活跃事务ID列表"          │
│  History list 越长 = 可能需要扫描的undo记录越多                 │
│  → UNDO 链变长 = MVCC 查询变慢                                 │
│  → 极端情况下链长上百，查询要遍历上百个undo页                   │
├──────────────────────────────────────────────────────────────┤
│  危害3：锁冲突加剧                                             │
│  ─────────────────────────────────────────────────────────── │
│  长时间未Purge的事务可能持有关键行锁                           │
│  导致新事务被锁等待                                            │
│  严重时引发死锁                                                │
└──────────────────────────────────────────────────────────────┘
```

## 🔥 扩展追问 1：History list length 突然飙升怎么处理？

> **A:** History list length 突然增长，说明 Purge 线程处理不过来。分三步排查：
>
> **第一步：确认是否是正常业务高峰**
> - 大量写入事务同时提交 → Purge 需要时间消化
> - 观察是否持续增长还是短期峰值
>
> **第二步：检查 Purge 线程是否在运行**
> ```sql
> SHOW ENGINE INNODB STATUS\G
> -- 如果看到：
> -- spin waits per second 0
> -- 说明 Purge 线程可能在等待某些资源
> ```
>
> **第三步：加大 Purge 能力**
> ```sql
> SET GLOBAL innodb_purge_threads = 8;
> SET GLOBAL innodb_max_purge_lag = 100000;
> -- innodb_max_purge_lag：延迟新写入（简单粗暴治标）
> ```

## 🔥 扩展追问 2：Buffer Pool 命中率低但内存足够，怎么排查？

> **A:** 这种情况通常是**访问模式问题**，不是内存问题。
>
> **排查方向：**
>
> ```sql
> -- 1. 检查是否有全表扫描
> SELECT * FROM information_schema.tables
> WHERE table_rows > 1000000
>   AND data_length > 1073741824  -- > 1GB
>   AND engine = 'InnoDB';
>
> -- 2. 检查最热的表（Buffer Pool 被谁占满）
> SELECT
>     object_name,
>     index_name,
>     number_of_pages,
>     pages * 16384 / 1024 / 1024 AS 'MB'
> FROM information_schema.innodb_buffer_page_lru
> ORDER BY number_of_pages DESC
> LIMIT 10;
>
> -- 3. 确认索引使用情况
> EXPLAIN SELECT * FROM orders WHERE user_id = 12345;
> -- 如果 type=ALL，说明全表扫描
> ```
>
> **典型场景：** Buffer Pool 60GB，但一个日志表有 800GB 数据
> - 定时任务扫描全表 → 60GB Buffer Pool 全部被污染
> - 热点数据反复被淘汰 → 命中率极低
> - **解决方案：** 将大日志表放到独立实例，或使用黑名单过滤

## 🔥 扩展追问 3：Pending flushes (fsync) 持续非零意味着什么？

> **A:** 意味着磁盘的 fsync 能力跟不上数据写入速度。
>
> **分析流程：**
>
> ```
> 磁盘性能瓶颈判断：
> ┌────────────────────────────────────────────────────────┐
> │  iostat -x 1                                         │
> │                                                      │
> │  Device  r/s    w/s    rkB/s   wkB/s  await  %util  │
> │  sda     120    8900   480     35600   25ms    98%  │
> │                                                      │
> │  %util = 98% → 磁盘几乎满载                          │
> │  w/s = 8900 → 每秒8900次写入                         │
> │  await = 25ms → 平均等待25ms（HDD正常，SSD异常）       │
> │                                                      │
> │  如果是 HDD（%util > 80%）：                         │
> │    → I/O队列积压 → fsync等待                         │
> │    → 必须换 SSD，HDD 无法满足写入负载                 │
> │                                                      │
> │  如果是 SSD（%util > 50% 但 await 很高）：          │
> │    → I/O线程配置不足                                 │
> │    → 增加 innodb_write_io_threads                    │
> │                                                      │
> │  如果 await 很低但 util 100%：                      │
> │    → 并发I/O队列配置不足                             │
> │    → fsync是串行的，瓶颈在单线程刷日志                │
> └────────────────────────────────────────────────────────┘
> ```

## 🔥 扩展追问 4：SHOW ENGINE INNODB STATUS 和 performance_schema 谁更准确？

> **A:** 两者侧重点不同，搭配使用：
>
> | 维度 | SHOW ENGINE INNODB STATUS | performance_schema |
> |------|---------------------------|---------------------|
> | 实时性 | 当前快照，立即可见 | 可查询历史累计值 |
> | 覆盖范围 | InnoDB核心指标 | 细粒度各维度统计 |
> | 适用场景 | 紧急排障第一线 | 趋势分析、慢查询归因 |
>
> **黄金搭档用法：**
> ```sql
> -- 第一步：INNODB STATUS 快速定位问题类型
> SHOW ENGINE INNODB STATUS\G
>
> -- 第二步：performance_schema 定位到具体SQL
> SELECT * FROM performance_schema.events_statements_summary_by_digest
> ORDER BY SUM_TIMER_WAIT DESC LIMIT 5;
>
> -- 第三步：锁定元凶SQL后，单独分析执行计划
> EXPLAIN SELECT ...
> ```

---

> **mysql> SHOW ENGINE INNODB STATUS 输出的是 InnoDB 引擎的"全息快照"——TRANSACTIONS 看锁和事务，BUFFER POOL 看内存效率，FILE I/O 看磁盘能力，History list length 是连接三者的桥梁。**

---

> "在生产环境排查性能问题时，我习惯把 SHOW ENGINE INNODB STATUS 作为'体检报告'——每周定期执行，把 History list length、Buffer pool hit rate 和 Pending I/O 数量画成折线图。一旦某个指标开始劣化，在用户报障之前就能定位问题。有一次 History list length 连续三天从 200 涨到 8000，就是因为一个定时任务开了大事务没提交，Purge 线程一直追不上。优化成小事务 + 调整 innodb_purge_threads 后，三天内恢复正常。"
