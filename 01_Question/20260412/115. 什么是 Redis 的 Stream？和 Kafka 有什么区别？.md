## 先说结论

Redis Stream 是 Redis 5.0 引入的**日志型数据结构**，提供**持久化、消费者组、消息确认**等特性，本质是**内存中的消息队列**；Kafka 是**分布式流处理平台**，基于**磁盘日志**设计，适合**高吞吐、大数据量**场景。Stream 是轻量级队列，Kafka 是企业级流平台。

---

## 🔥 核心概念

| 维度 | Redis Stream | Kafka |
|------|-------------|-------|
| **定位** | 内存消息队列 | 分布式流处理平台 |
| **存储** | 内存为主（可持久化） | 磁盘日志（顺序写） |
| **吞吐量** | 万级 QPS | 百万级 QPS |
| **数据量** | 受内存限制 | TB/PB 级 |
| **消费者模型** | 消费者组（类似 Kafka） | 消费者组 + Partition |
| **消息确认** | XACK 手动确认 | Offset 自动/手动提交 |
| **副本机制** | Redis 主从复制 | ISR 多副本机制 |
| **使用成本** | 低（单 Redis 实例） | 高（ZooKeeper/KRaft + 多 Broker） |

## 🔥 扩展追问 1：核心机制对比

| 特性 | Redis Stream | Kafka |
|------|-------------|-------|
| **消息 ID** | 时间戳-序列号（毫秒精度） | Offset（长整型） |
| **消费进度** | PEL + Last ID（每个消费者组） | Offset（每个 Consumer Group + Partition） |
| **消息保留** | 内存限制 + 手动删除 | 时间/大小策略（可长期保留） |
| **顺序性** | 全局有序 | Partition 内有序，全局无序 |
| **重放能力** | 支持（XRANGE） | 强大（任意 Offset 重放） |
| **分区并行** | 单线程（单 Stream） | 多 Partition 并行消费 |

```java
// Redis Stream 消费者
StreamMessageListenerContainer.create(connectionFactory)
    .receiveAutoAck(
        Consumer.from("group", "consumer-1"),
        StreamOffset.create("stream:orders", ReadOffset.lastConsumed()),
        message -> process(message)
    );

// Kafka 消费者
KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(Arrays.asList("orders"));
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        process(record);
    }
    consumer.commitSync();  // 手动提交 Offset
}
```

## 🔥 扩展追问 2：性能与容量对比

| 指标 | Redis Stream | Kafka |
|------|-------------|-------|
| **单机 QPS** | 5-10 万 | 10-100 万 |
| **延迟** | 亚毫秒级 | 毫秒级 |
| **单条消息大小** | 建议 < 512KB | 默认 1MB，可调 |

| **横向扩展** | 难（需分片） | 易（加 Partition/Broker） |

**选择 Redis Stream**：
- 消息量不大（< 100 万/天）
- 已有 Redis 基础设施
- 需要亚毫秒级延迟
- 团队不想维护 Kafka

**选择 Kafka**：
- 消息量大（> 100 万/天）
- 需要长期保留消息
- 需要流处理（Kafka Streams/Flink）
- 需要高可用多副本

## 🔥 扩展追问 4：混合架构实践

```java
/**
 * 分层消息架构：Redis Stream 做实时，Kafka 做持久化
 */
@Service
public class HybridMessageService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    
    public void sendOrder(Order order) {
        String message = JSON.toJSONString(order);
        
        // 1. 写入 Redis Stream（实时消费）
        RecordId recordId = redisTemplate.opsForStream()
            .add(StreamRecords.newRecord()
                .in("stream:orders")
                .ofObject(message));
        
        // 2. 异步写入 Kafka（持久化 + 离线分析）
        kafkaTemplate.send("orders", order.getOrderId(), message)
            .addCallback(
                success -> log.info("Kafka 发送成功"),
                failure -> log.error("Kafka 发送失败", failure)
            );
    }
}
```

## 🔥 扩展追问 5：Stream 的局限性

| 局限 | 说明 | 解决方案 |
|------|------|---------|
| **内存限制** | 消息堆积受内存限制 | 设置 MAXLEN、定期清理、或转 Kafka |
| **单线程** | 单 Stream 无法并行 | 多 Stream 分片、或使用 Kafka |
| **无分区** | 无法像 Kafka Partition 并行 | 客户端分片到多个 Stream |
| **副本弱** | 主从复制，非强一致 | 启用 AOF 每秒刷盘、或转 Kafka |
| **无事务** | 不支持跨消息事务 | 使用 Lua 脚本、或转 Kafka |

```java
// Stream 分片实现（模拟 Kafka Partition）
public class ShardedStreamProducer {
    
    private int shardCount = 4;
    
    public void send(Order order) {
        // 根据 orderId 取模选择 Stream
        int shard = order.getOrderId().hashCode() % shardCount;
        String streamKey = "stream:orders:" + shard;
        
        redisTemplate.opsForStream().add(
            StreamRecords.newRecord()
                .in(streamKey)
                .ofObject(JSON.toJSONString(order))
        );
    }
}
```

## 一句话收尾

Redis Stream 是**轻量级内存消息队列**，提供类 Kafka 的消费者组和消息确认机制，适合**中小规模、低延迟**场景；Kafka 是**企业级流处理平台**，基于磁盘设计，适合**大数据量、高吞吐、长期存储**场景。
