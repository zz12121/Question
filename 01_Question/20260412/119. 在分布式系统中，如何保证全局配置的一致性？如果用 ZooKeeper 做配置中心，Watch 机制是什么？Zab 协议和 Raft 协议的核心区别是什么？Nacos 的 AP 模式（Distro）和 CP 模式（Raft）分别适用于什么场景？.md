## 先说结论

分布式配置一致性通过**配置中心+推送机制**保证，ZooKeeper 的 Watch 是**客户端监听节点变化**的事件通知机制。Zab 和 Raft 都是**强一致性协议**，Zab 专为 ZooKeeper 设计（崩溃恢复+消息广播），Raft 更通用（Leader 选举+日志复制）。Nacos **AP 模式（Distro）**适合**服务注册**（高可用优先），**CP 模式（Raft）**适合**配置中心**（一致性优先）。

## 🔥 核心概念

| 维度 | 说明 |
|------|------|
| **配置一致性** | 配置变更后，所有节点最终看到相同配置 |
| **ZooKeeper Watch** | 客户端订阅节点事件，变化时服务端推送通知 |
| **Zab 协议** | ZooKeeper 原子广播协议，保证顺序一致性 |
| **Raft 协议** | 通用共识算法，Leader 选举+日志复制 |
| **Nacos AP** | Distro 协议，最终一致，高可用 |
| **Nacos CP** | Raft 协议，强一致，牺牲可用性 |

配置一致性架构：

│                    配置中心（Nacos/ZK）                      │

3. **版本控制**：配置带版本号，防止旧配置覆盖新配置
4. **本地缓存**：客户端本地缓存，配置中心不可用时降级

### 2. ZooKeeper Watch 机制

```java
// ZooKeeper Watch 使用示例
public class ZkConfigWatcher {
    
    private ZooKeeper zk;
    private String configPath = "/config/app";
    
    public void watchConfig() throws Exception {
        // 1. 获取配置并注册 Watch
        byte[] data = zk.getData(configPath, new Watcher() {
            @Override
            public void process(WatchedEvent event) {
                // 2. 配置变化时触发
                if (event.getType() == Event.EventType.NodeDataChanged) {
                    System.out.println("配置变更: " + event.getPath());
                    // 重新获取配置（Watch 是一次性的，需重新注册）
                    refreshConfig();
                }
            }
        }, null);
        
        System.out.println("当前配置: " + new String(data));
    }
    
    private void refreshConfig() {
        try {
            // 重新获取数据并注册 Watch
            byte[] data = zk.getData(configPath, true, null);
            System.out.println("新配置: " + new String(data));
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
```

**Watch 特性**：

| 特性 | 说明 |
|------|------|
| **一次性** | 触发后失效，需重新注册 |
| **轻量级** | 只通知事件类型，不传递数据 |
| **顺序性** | 事件按发生顺序推送 |
| **时效性** | 先得到通知，再获取数据（可能已再次变化） |

**Watch 事件类型**：
- `NodeCreated`：节点创建
- `NodeDeleted`：节点删除
- `NodeDataChanged`：节点数据变化
- `NodeChildrenChanged`：子节点变化

### 3. Zab vs Raft 协议对比

| 维度 | Zab（ZooKeeper） | Raft |
|------|-----------------|------|
| **设计目标** | 专为 ZooKeeper 设计 | 通用共识算法 |
| **Leader 选举** | Fast Leader Election | 随机超时选举 |
| **日志复制** | 崩溃恢复 + 消息广播 | 日志复制 + 提交 |
| **事务 ID** | zxid（64位，高32位epoch，低32位计数） | term + index |
| **崩溃恢复** | 单独阶段，恢复历史数据 | 新 Leader 包含所有已提交日志 |
| **读性能** | 可从 Follower 读（可能读到旧数据） | 只能从 Leader 读（强一致） |

```
Zab 协议流程：
┌─────────────────────────────────────────────────────────────┐
│  1. 崩溃恢复阶段                                             │
│     - 选举具有最大 zxid 的节点为 Leader                      │
│     - Leader 同步数据给 Follower                             │
│     └─────────────────────────────────────────────────────┘ │
│                          ↓                                  │
│  2. 消息广播阶段（正常运行）                                  │
│     - 客户端写请求 → Leader                                  │
│     - Leader 生成 zxid，写入本地日志                          │
│     - 广播给所有 Follower                                     │
│     - 过半 Follower 确认后，Leader 提交                       │
│     - Leader 通知 Follower 提交                               │
└─────────────────────────────────────────────────────────────┘

Raft 协议流程：
┌─────────────────────────────────────────────────────────────┐
│  1. Leader 选举                                              │
│     - 超时后发起选举，term + 1                               │
│     - 得票过半成为 Leader                                    │
│     └─────────────────────────────────────────────────────┘ │
│                          ↓                                  │
│  2. 日志复制                                                 │
│     - 客户端请求 → Leader                                    │
│     - Leader 追加日志，term + index                          │
│     - 发送 AppendEntries 给 Follower                         │
│     - 过半确认后提交，通知 Follower 提交                      │
└─────────────────────────────────────────────────────────────┘
```

### 4. Nacos AP vs CP 模式

```java
// Nacos 服务注册（默认 AP 模式）
spring:
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848
        # AP 模式：服务注册优先可用性，最终一致

// Nacos 配置中心（CP 模式）
spring:
  cloud:
    nacos:
      config:
        server-addr: localhost:8848
        # CP 模式：配置变更强一致
```

| 模式 | 协议 | 适用场景 | 特点 |
|------|------|---------|------|
| **AP（Distro）** | 自研 Distro 协议 | 服务注册发现 | 高可用、最终一致、分区容错 |
| **CP（Raft）** | Raft 协议 | 配置中心 | 强一致、牺牲可用性 |

```
Nacos 双模式架构：
┌─────────────────────────────────────────────────────────────┐
│                      Nacos Server                           │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Distro 协议（AP）                                   │   │
│  │  - 服务注册表（Service Registry）                     │   │
│  │  - 每个节点独立处理写请求                              │   │
│  │  - 异步同步到其他节点                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Raft 协议（CP）                                     │   │
│  │  - 配置数据（Configuration）                         │   │
│  │  - Leader 处理写请求                                  │   │
│  │  - 强一致同步到 Follower                              │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**场景选择**：

| 场景 | 推荐模式 | 原因 |
|------|---------|------|
| 服务注册发现 | **AP** | 服务上下线频繁，可用性优先，短暂不一致可接受 |
| 配置管理 | **CP** | 配置必须一致，不能读到旧配置 |
| 分布式锁 | **CP** | 强一致要求，必须知道谁是真正的锁持有者 |
| 元数据管理 | **CP** | 元数据一致性要求高 |

## 🔥 扩展追问 1：ZooKeeper 的局限性

| 局限 | 说明 | 解决方案 |
|------|------|---------|
| **单点写入** | 只有 Leader 能处理写请求 | 读多写少场景适用 |
| **Watch 一次性** | 触发后需重新注册 | 客户端封装自动重注册 |
| **会话超时敏感** | 网络抖动导致会话失效 | 调整 sessionTimeout |
| **数据量限制** | 单节点数据 < 1MB | 只存元数据，不存业务数据 |
| **扩容复杂** | 新增节点需全量同步 | 提前规划集群规模 |
