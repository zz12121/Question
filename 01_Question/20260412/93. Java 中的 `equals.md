## 1. 核心架构图

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                       equals() 与 hashCode() 契约                               │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│   【契约关系】                                                                   │
│   ┌─────────────────────────────────────────────────────────────────────────┐   │
│   │                                                                          │   │
│   │   Contract（约定）：                                                      │   │
│   │                                                                          │   │
│   │   1. 一致性：如果 a.equals(b) == true                                    │   │
│   │           则 a.hashCode() 必须 == b.hashCode()                            │   │
│   │                                                                          │   │
│   │   2. 反之不一定：a.hashCode() == b.hashCode()                           │   │
│   │              不一定 a.equals(b)                                          │   │
│   │              （hash 碰撞是允许的）                                        │   │
│   │                                                                          │   │
│   │   3. 自反性：a.equals(a) 必须为 true                                     │   │
│   │                                                                          │   │
│   │   4. 对称性：a.equals(b) == b.equals(a)                                  │   │
│   │                                                                          │   │
│   │   5. 传递性：a.equals(b) && b.equals(c) → a.equals(c)                    │   │
│   │                                                                          │   │
│   └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                 │
│   【HashMap 查找流程】                                                          │
│   ┌───────────────────────────────────────────────────────────────────────┐   │
│   │                                                                        │   │
│   │   map.get(key) 怎么找？                                                 │   │
│   │                                                                        │   │
│   │   Step 1: 计算 hash = key.hashCode()                                    │   │
│   │           │                                                             │   │
│   │           ▼                                                             │   │
│   │   ┌─────────────────────────────────────────────────┐                   │   │
│   │   │              hash = 31 的桶位置                   │                   │   │
│   │   │   [0] [1] [2] [3] ... [31] ... [n-1]             │                   │   │
│   │   │        └── 链表/红黑树（hash 碰撞的元素）          │                   │   │
│   │   └─────────────────────────────────────────────────┘                   │   │
│   │           │                                                             │   │
│   │           ▼                                                             │   │
│   │   Step 2: equals() 逐一比较，命中则返回                                   │   │
│   │                                                                        │   │
│   │   【关键】：                                                             │   │
│   │   hash 不同 → 直接跳过（O(1)）                                           │   │
│   │   hash 相同 → 遍历链表/红黑树，用 equals() 比较（O(n) 或 O(logn)）        │   │
│   │                                                                        │   │
│   └───────────────────────────────────────────────────────────────────────┘   │
│                                                                                 │
│   【违反契约的后果】                                                            │
│   ┌───────────────────────────────────────────────────────────────────────┐   │
│   │                                                                        │   │
│   │   如果 equals() 返回 true，但 hashCode() 不等：                           │   │
│   │                                                                        │   │
│   │   key1.equals(key2) = true                                             │   │
│   │   key1.hashCode() = 31                                                 │   │
│   │   key2.hashCode() = 99    ← 违反契约！                                 │   │
│   │                                                                        │   │
│   │   结果：                                                                │   │
│   │   ┌─────────────────┐  ┌─────────────────┐                            │   │
│   │   │ HashMap.put()   │  │ HashMap.get()  │                            │   │
│   │   │ 存到 bucket[31]  │  │ 去 bucket[99] 找 │  ← 找不到！              │   │
│   │   └─────────────────┘  └─────────────────┘                            │   │
│   │                                                                        │   │
│   │   ⚠️ HashMap 逻辑错乱，无法正确存取                                      │   │
│   │                                                                        │   │
│   └───────────────────────────────────────────────────────────────────────┘   │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

## 2. 核心代码示例

### 2.1 Object 类的默认实现

```java
// ========== Object 类的默认实现 ==========

public class Object {
    
    // 默认 equals：引用比较（==）
    public boolean equals(Object obj) {
        return (this == obj);
    }
    
    // 默认 hashCode：根据对象内存地址计算
    public native int hashCode();
    
    // 注意：hashCode() 是 native 方法，由 JVM 实现
}

// ========== 默认行为示例 ==========
public class DefaultDemo {
    public static void main(String[] args) {
        String s1 = new String("hello");
        String s2 = new String("hello");
        
        // s1 == s2：引用不同 → false
        System.out.println(s1 == s2);              // false
        
        // s1.equals(s2)：内容相同 → true
        System.out.println(s1.equals(s2));         // true
        
        // hashCode 不同？但 String 重写了！
        System.out.println(s1.hashCode());          // 99162322
        System.out.println(s2.hashCode());          // 99162322（相同！）
    }
}
```

### 2.2 典型重写场景：User 对象

```java
// ========== 没有重写 equals/hashCode 的 User ==========
@Getter
@Setter
public class User {
    private Long id;
    private String name;
    private Integer age;
}

// 测试
public class UserTest {
    public static void main(String[] args) {
        User u1 = new User();
        u1.setId(1L);
        u1.setName("张三");
        
        User u2 = new User();
        u2.setId(1L);
        u2.setName("张三");
        
        System.out.println(u1.equals(u2));  // false（引用不同！）
        System.out.println(u1.hashCode() == u2.hashCode());  // false（地址不同！）
        
        // ❌ 问题：用 User 做 HashMap 的 key
        Map<User, String> map = new HashMap<>();
        map.put(u1, "用户1");
        
        System.out.println(map.get(u2));  // null！（找不到！）
        // 原因：u2 和 u1 是不同的对象，hashCode 不同，被存在不同桶
    }
}

// ========== 正确重写 ==========
@Getter
@Setter
public class User {
    private Long id;
    private String name;
    private Integer age;
    
    @Override
    public boolean equals(Object o) {
        // 1. 自反性
        if (this == o) return true;
        
        // 2. 非空检查
        if (o == null || getClass() != o.getClass()) return false;
        
        // 3. 强制类型转换
        User user = (User) o;
        
        // 4. 比较关键字段
        return id != null && id.equals(user.id);
    }
    
    @Override
    public int hashCode() {
        // 必须用 equals 中参与比较的字段来计算
        // 否则违反契约！
        return Objects.hash(id);
    }
}

// ✅ 现在 u1.equals(u2) = true，u1.hashCode() == u2.hashCode()
```

### 2.4 String 和 Integer 的实现参考

```java
// ========== String 的实现（参考） ==========
public final class String {
    private final char[] value;  // 存储字符
    private int hash;           // 缓存 hashCode
    
    @Override
    public boolean equals(Object anObject) {
        if (this == anObject) return true;
        if (anObject instanceof String) {
            String anotherString = (String) anObject;
            int n = value.length;
            if (n == anotherString.value.length) {
                // 逐字符比较
                char[] v1 = value;
                char[] v2 = anotherString.value;
                int i = 0;
                while (n-- != 0) {
                    if (v1[i] != v2[i]) return false;
                    i++;
                }
                return true;
            }
        }
        return false;
    }
    
    @Override
    public int hashCode() {
        // 懒计算：第一次调用时计算，之后缓存
        int h = hash;
        if (h == 0 && value.length > 0) {
            h = value.length;
            for (char c : value) {
                h = 31 * h + c;  // 经典算法
            }
            hash = h;
        }
        return h;
    }
}

// ========== 为什么 String 用 31 作为乘数？ ==========
// 1. 31 是质数，散布均匀
// 2. 31 * i = (i << 5) - i，JVM 会优化为位移运算
// 3. 历史原因：早期 Java 性能考量
```

### 2.5 hashCode 的计算策略

```java
// ========== 常见 hashCode 计算方式 ==========

// 方式一：Objects.hash()（Java 7+，推荐）
@Override
public int hashCode() {
    return Objects.hash(field1, field2, field3);
}

// 方式二：手动计算
@Override
public int hashCode() {
    int result = 17;                    // 起始值（质数）
    result = 31 * result + (field1 == null ? 0 : field1.hashCode());
    result = 31 * result + (field2 == null ? 0 : field2.hashCode());
    result = 31 * result + field3;     // 基本类型直接加
    return result;
}

// 方式三：Lombok @AutoConstant（更高效）
// 适用于不可变对象，hashCode 可以是常量
public final class Status {
    public static final int HASH = Objects.hash("status");
}
```

## 🔥 扩展追问 1：HashMap 为什么要先比较 hash 再比较 equals？

> **A:** 这是性能优化的考量——hashCode 定位桶 O(1)，equals 逐一比较是 O(n)。

```
HashMap.get(key) 的执行流程：

1. 计算 key.hashCode() → 得到桶下标
   └── 纯计算，无遍历，O(1)

2. 找到对应桶，遍历链表/红黑树
   └── 如果 hash 分布均匀，链表长度很短
   └── 红黑树 O(logn)，链表 O(n)

为什么需要两步？
├── 如果直接用 equals() 遍历所有桶，需要 O(n²) 或更差
├── hashCode() 快速过滤掉 99%+ 的不匹配元素
└── equals() 只在少量候选元素中精确比较

最佳情况：hash 完美分布，每个桶 0-1 个元素
   └── get 操作 ≈ O(1)
```

## 🔥 扩展追问 2：为什么 hash 碰撞时 HashMap 会变慢？

> **A:** hash 碰撞意味着多个元素落在同一个桶里，需要用 equals() 逐一比较。

```java
// 最坏情况：所有 key 的 hashCode 都相同
Map<String, String> map = new HashMap<>();

// 放入 10000 个元素，但 hash 都在 bucket[31]
// 查找时：
// 1. hashCode() → 31 ✓ 快
// 2. 遍历链表 10000 次，每次 equals() 比较字符串
//    └── 10000 次 equals，每次 O(n) 字符串比较
//    └── 总复杂度 O(n²)

System.out.println(map.get("target"));  // 巨慢无比！

// JDK 8+ 的优化：链表超过 8 个转红黑树
// 链表：O(n)
// 红黑树：O(logn)
// 但最坏情况仍然慢
```

**HashMap 的优化措施：**
```java
// 负载因子 0.75 = 当桶使用超过 75% 时扩容
// 扩容后桶数量翻倍，hash 分布更均匀
// 如果数据量确定，可以预创建容量避免扩容
Map<String, String> map = new HashMap<>(expectedSize * 4 / 3);
// 或者 Guava Maps.newHashMapWithExpectedSize()
```

## 🔥 扩展追问 3：equals() 用 `instanceof` 还是 `getClass()`？

> **A:** 有两种设计思路，要根据业务场景选。

| 场景 | 推荐方式 | 原因 |
|------|---------|------|
| 父子类比较 | instanceof | 父类.equals(子类) 应该返回 true（子类的字段可能更多） |
| 严格相等 | getClass() | 精确类型匹配，防止父类equals被子类破坏 |
| 不可继承类 | 都可以 | String、Integer 等 final 类 |

```java
// 方式一：instanceof（更宽松，推荐用于业务对象）
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof User)) return false;  // 允许子类比较
    
    User user = (User) o;
    return Objects.equals(this.id, user.id);
}

// ⚠️ 注意：使用 instanceof 要保证对称性和传递性！
// 如果子类加了新字段，可能违反传递性

// 方式二：getClass()（更严格）
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    // 必须完全相同的类
    return o != null && getClass() == o.getClass() && Objects.equals(id, ((User) o).id);
}

// 最佳实践：final 类用 instanceof，非 final 类考虑 getClass()
public final class Status { }  // String 模式
public class User { }           // 业务类，可以用 getClass()
```

## 🔥 扩展追问 4：hashCode 返回固定值有什么后果？

> **A:** 所有元素都 HashMap 同一桶，变成纯链表，性能退化到 O(n)。

```java
// ❌ 错误实现：hashCode 永远返回 1
public class BadKey {
    private String id;
    
    @Override
    public int hashCode() {
        return 1;  // 所有对象都在 bucket[1]
    }
    
    @Override
    public boolean equals(Object o) {
        // 正确实现
        return o instanceof BadKey && Objects.equals(id, ((BadKey) o).id);
    }
}

// 结果：
// put 1000 个元素 → 全存在 bucket[1] 的链表里
// get 查找 → 每次遍历整个链表 → O(n)
// 1000 个元素 → 最多比较 1000 次
// 10000 个元素 → 最多比较 10000 次
// ⚠️ HashMap 退化为线性搜索

// ✅ 正确实现：让 hashCode 尽可能分散
@Override
public int hashCode() {
    return Objects.hash(id);
}
```

## 🔥 扩展追问 5：为什么 Set 的 add 方法也要检查 equals？

> **A:** Set 底层是 HashMap，value 用固定对象 PRESENT，key 的查找完全依赖 equals()。

```java
// HashSet 内部实现（简化）
public class HashSet<E> {
    private final Map<E, Object> map = new HashMap<>();
    
    public boolean add(E e) {
        return map.put(e, PRESENT) == null;  // 如果 key 已存在，put 返回旧值
    }
    
    public boolean contains(Object o) {
        return map.containsKey(o);  // containsKey 也要用 hashCode + equals
    }
}

// 过程：
// set.add(user1) → map.put(user1, PRESENT) → null（成功）
// set.add(user2) → map.put(user2, PRESENT) → null（成功）
// set.add(user1) → map.put(user1, PRESENT) → PRESENT（失败，已存在）

// 如果 user1.equals(user2) == true 但 hashCode() 不同：
// set.add(user1) 存在 bucket[A]
// set.add(user2) 存在 bucket[B]（不同桶！）
// set.contains(user2) 在 bucket[B] 找，找不到！→ false
// ⚠️ Set 逻辑错乱
```

## 一句话总结

> **equals() 和 hashCode() 必须保持一致契约：相等的对象必须有相同的 hashCode，这保证了 HashMap/HashSet 在 O(1) 时间复杂度内通过 hash 定位桶、再用 equals 精确命中；如果违反契约，对象作为 key 时 put 能存进去但 get 找不到；如果 hash 碰撞严重，HashMap 会退化为链表/红黑树遍历，性能从 O(1) 退化到 O(n) 或 O(logn)。**

> "之前项目里出过线上事故：有个枚举类实现了 equals() 做业务匹配，但没重写 hashCode()，导致这个枚举对象作为 HashMap 的 key 时出现逻辑错乱——put 进去了但 get 不到。最后排查半天才定位到是 hashCode 不一致的问题。教训是：如果业务上需要自定义 equals 规则（比如按 id 判断两个对象相等），那 hashCode 必须也用 id 来算，否则 HashSet 的 contains、HashMap 的 get 全部会出问题。另外建议用 Lombok 的 @EqualsAndHashCode，或者直接让 IDE 自动生成，保证两个方法用同一套字段参与计算。"
