## 先说结论

意向锁（Intention Lock）是 MySQL InnoDB 的**表级锁**，用于表明事务**稍后要对表中的某些行加行锁**，目的是实现表锁与行锁的**共存兼容**，避免加锁时的全表遍历检查。

---

## 🔥 核心概念

| 维度 | 说明 |
|------|------|
| **锁类型** | 表级锁，分为意向共享锁（IS）和意向排他锁（IX） |
| **触发时机** | 事务对行加 S 锁前，先对表加 IS；加 X 锁前，先对表加 IX |
| **核心作用** | 让表锁快速判断能否加锁，无需遍历检查每一行 |
| **兼容性** | IS/IX 之间互相兼容，但与表级 S/X 锁有冲突规则 |

---

## 🔥 详细展开

### 1. 意向锁的两种类型

```sql
-- 表结构
CREATE TABLE account (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance DECIMAL(10,2)
) ENGINE=InnoDB;

-- 场景1：对某行加共享锁（SELECT ... LOCK IN SHARE MODE）
BEGIN;
SELECT * FROM account WHERE id = 1 LOCK IN SHARE MODE;
-- 自动对 account 表加 IS 锁，对 id=1 的行加 S 锁
```

```sql
-- 场景2：对某行加排他锁（UPDATE/DELETE/SELECT ... FOR UPDATE）
BEGIN;
UPDATE account SET balance = balance - 100 WHERE id = 1;
-- 自动对 account 表加 IX 锁，对 id=1 的行加 X 锁
```

| 意向锁 | 触发语句 | 含义 |
|--------|---------|------|
| **IS（Intention Shared）** | `SELECT ... LOCK IN SHARE MODE` | 事务打算对某些行加 S 锁 |
| **IX（Intention Exclusive）** | `UPDATE/DELETE/INSERT / SELECT ... FOR UPDATE` | 事务打算对某些行加 X 锁 |

### 2. 锁兼容性矩阵

```
                    已持有锁
                  IS    IX    S     X
                ┌────┬────┬────┬────┐
    请求 IS     │ ✓  │ ✓  │ ✓  │ ✗  │
    请求 IX     │ ✓  │ ✓  │ ✗  │ ✗  │
    请求 S      │ ✓  │ ✗  │ ✓  │ ✗  │
    请求 X      │ ✗  │ ✗  │ ✗  │ ✗  │
                └────┴────┴────┴────┘
```

**关键结论**：
- IS 与 IX **互相兼容**：多个事务可以同时打算对表加行锁
- IX 与表级 S 锁**不兼容**：有人要改行时，不能加表级读锁
- IX 与表级 X 锁**不兼容**：有人要改行时，不能加表级写锁

### 3. 为什么需要意向锁？

**没有意向锁的问题**：
```sql
-- 事务A：对 id=1 的行加了 X 锁
BEGIN;
UPDATE account SET balance = 100 WHERE id = 1;

-- 事务B：想对整个表加表级 X 锁（如 ALTER TABLE）
BEGIN;
LOCK TABLES account WRITE;
-- 问题：B 怎么知道表里有行被锁了？
-- 必须遍历所有行检查，性能极差！
```

**有意向锁的解决方案**：
```sql
-- 事务A：先对表加 IX，再对行加 X
-- account 表上有了 IX 锁

-- 事务B：想加表级 X 锁
-- 检查：表上有 IX 锁 → 与 X 不兼容 → 直接阻塞，无需遍历行！
```

### 4. 实际案例演示

```sql
-- 会话1
BEGIN;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 加锁顺序：表级 IX → 行级 X（id=1）

-- 会话2
BEGIN;
SELECT * FROM account WHERE id = 2 FOR UPDATE;
-- 可以执行！表级 IX 与 IX 兼容
-- 最终：表上有两个 IX，id=1 和 id=2 分别有 X 锁

-- 会话3
BEGIN;
LOCK TABLES account READ;  -- 尝试加表级 S 锁
-- 阻塞！因为表上有 IX，与 S 不兼容
```

**查看意向锁**：
```sql
-- MySQL 8.0
SELECT * FROM performance_schema.data_locks 
WHERE OBJECT_NAME = 'account';

-- 输出示例
-- LOCK_TYPE: TABLE, LOCK_MODE: IX
-- LOCK_TYPE: RECORD, LOCK_MODE: X, LOCK_DATA: 1
```

## 🔥 扩展追问 1：意向锁与行锁的加锁顺序

```sql
-- 完整加锁流程
BEGIN;
UPDATE account SET balance = 100 WHERE id = 1;
```

**加锁顺序**：
1. **表级**：对 `account` 表加 **IX 锁**（意向排他）
2. **行级**：对 `id=1` 的索引记录加 **X 锁**
3. 如果 id=1 不存在（RR 级别）：对间隙加 **Gap Lock**

**释放顺序**（两阶段锁）：
- 锁在事务结束时统一释放（COMMIT/ROLLBACK）
- 不是语句结束就释放

## 🔥 扩展追问 2：意向锁与表锁的冲突场景

### 场景1：ALTER TABLE 被阻塞

```sql
-- 会话1：业务正常更新
BEGIN;
UPDATE account SET balance = 100 WHERE id = 1;
-- 持有：表 IX + 行 X

-- 会话2：DBA 想加字段
ALTER TABLE account ADD COLUMN remark VARCHAR(100);
-- 需要表级 X 锁
-- 阻塞！因为会话1持有 IX，与 X 不兼容
```

**解决方案**：
```sql
-- 1. 查看阻塞源
SELECT * FROM sys.innodb_lock_waits;

-- 2. 杀会话或等待
KILL <blocking_thread_id>;

-- 3. 使用 pt-online-schema-change 在线 DDL
pt-online-schema-change --alter "ADD COLUMN remark VARCHAR(100)" \
  D=test,t=account --execute
```

### 场景2：LOCK TABLES 与行锁冲突

```sql
-- 会话1
BEGIN;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 表 IX + 行 X

-- 会话2
LOCK TABLES account WRITE;
-- 阻塞！IX 与 X 不兼容
```

## 🔥 扩展追问 3：意向锁与 MDL 锁的区别

| 维度 | 意向锁（Intention Lock） | MDL 锁（Metadata Lock） |
|------|-------------------------|------------------------|
| **层级** | 存储引擎层（InnoDB） | Server 层 |
| **作用** | 协调表锁与行锁 | 保护表结构（元数据） |
| **触发** | DML 操作（SELECT/UPDATE/INSERT） | 任何表访问（包括 SELECT） |
| **类型** | IS、IX | 共享读、共享写、排他 |
| **查看** | `performance_schema.data_locks` | `performance_schema.metadata_locks` |

```sql
-- 同时存在两种锁
BEGIN;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- MDL 锁：表级共享读锁（保护表结构）
-- 意向锁：表级 IX + 行级 X
```

## 🔥 扩展追问 4：意向锁的性能影响与优化

### 问题：大量行锁导致 IX 冲突

```sql
-- 长事务持有大量行锁
BEGIN;
UPDATE account SET status = 1 WHERE create_time > '2024-01-01';
-- 更新了 100万行，持有表 IX + 100万个行 X 锁

-- 其他会话想加表锁（如备份）
LOCK TABLES account READ;
-- 必须等长事务结束！
```

### 优化方案

```sql
-- 方案1：分批处理，缩短事务
SET @batch_size = 1000;
REPEAT
    UPDATE account SET status = 1 
    WHERE create_time > '2024-01-01' AND status = 0
    LIMIT @batch_size;
    COMMIT;
    SLEEP(0.1);
UNTIL ROW_COUNT() = 0 END REPEAT;

-- 方案2：使用 READ COMMITTED 减少锁持有时间
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 方案3：在线备份改用 --single-transaction（不锁表）
mysqldump --single-transaction -u root -p test account > backup.sql
```

## 一句话收尾

意向锁是 InnoDB 实现表锁与行锁**共存**的关键机制，事务加行锁前必须先对表加意向锁（IS/IX），让表锁快速判断兼容性而无需遍历全表，是**多粒度锁协议**的核心实现。
