## 2. 核心代码示例

### 2.1 JDK 动态代理

```java
// ========== 步骤一：定义接口 ==========
public interface UserService {
    void addUser(String name);
    User getUser(Long id);
}

// ========== 步骤二：实现类（目标对象）==========
@Service
public class UserServiceImpl implements UserService {
    @Override
    public void addUser(String name) {
        System.out.println("真正执行业务：添加用户 " + name);
    }
    
    @Override
    public User getUser(Long id) {
        System.out.println("真正执行业务：查询用户 " + id);
        return new User(id, name);
    }
}

// ========== 步骤三：定义 InvocationHandler ==========
public class JdkProxyHandler implements InvocationHandler {
    
    // 目标对象
    private final Object target;
    
    public JdkProxyHandler(Object target) {
        this.target = target;
    }
    
    /**
     * proxy: 代理对象（一般不用）
     * method: 被调用的方法
     * args: 方法参数
     */
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        String methodName = method.getName();
        
        // 前置增强
        long start = System.currentTimeMillis();
        System.out.println("[JDK Proxy] 执行方法前: " + methodName);
        
        try {
            // 执行业务方法
            Object result = method.invoke(target, args);
            return result;
        } finally {
            // 后置增强
            long cost = System.currentTimeMillis() - start;
            System.out.println("[JDK Proxy] 执行方法后: " + methodName + ", 耗时: " + cost + "ms");
        }
    }
}

// ========== 步骤四：创建代理对象 ==========
public class JdkProxyDemo {
    public static void main(String[] args) {
        UserService target = new UserServiceImpl();
        
        // 生成代理对象
        UserService proxy = (UserService) Proxy.newProxyInstance(
            target.getClass().getClassLoader(),      // 类加载器
            target.getClass().getInterfaces(),       // 接口数组（JDK代理必须面向接口）
            new JdkProxyHandler(target)              // InvocationHandler
        );
        
        // 调用方法，实际走的是 invoke()
        proxy.addUser("张三");
        // 输出:
        // [JDK Proxy] 执行方法前: addUser
        // 真正执行业务：添加用户 张三
        // [JDK Proxy] 执行方法后: addUser, 耗时: 5ms
    }
}
```

### 2.2 CGLIB 动态代理

```java
// ========== 步骤一：CGLIB 不需要接口 ==========
@Service
public class UserService {  // 普通类，没有实现接口
    
    public void addUser(String name) {
        System.out.println("真正执行业务：添加用户 " + name);
    }
    
    public User getUser(Long id) {
        System.out.println("真正执行业务：查询用户 " + id);
        return new User(id, name);
    }
}

// ========== 步骤二：定义 MethodInterceptor ==========
public class CglibProxyInterceptor implements MethodInterceptor {
    
    @Override
    public Object intercept(Object obj, Method method, Object[] args, 
                           MethodProxy methodProxy) throws Throwable {
        String methodName = method.getName();
        
        // 前置增强
        System.out.println("[CGLIB Proxy] 调用方法前: " + methodName);
        
        // 两种调用方式
        // 方式一：调用父类的方法（推荐，性能更好）
        Object result = methodProxy.invokeSuper(obj, args);
        
        // 方式二：通过反射调用（性能稍差）
        // Object result = method.invoke(target, args);
        
        // 后置增强
        System.out.println("[CGLIB Proxy] 调用方法后: " + methodName);
        
        return result;
    }
}

// ========== 步骤三：创建代理对象 ==========
public class CglibProxyDemo {
    public static void main(String[] args) {
        // 创建增强器
        Enhancer enhancer = new Enhancer();
        
        // 设置要代理的类（生成它的子类）
        enhancer.setSuperclass(UserService.class);
        
        // 设置回调
        enhancer.setCallback(new CglibProxyInterceptor());
        
        // 创建代理对象（是 UserService 的子类）
        UserService proxy = (UserService) enhancer.create();
        
        // 调用方法
        proxy.addUser("李四");
        // 输出:
        // [CGLIB Proxy] 调用方法前: addUser
        // 真正执行业务：添加用户 李四
        // [CGLIB Proxy] 调用方法后: addUser
    }
}

// ========== 步骤四：Spring AOP 集成方式 ==========
@Configuration
@EnableAspectJAutoProxy
public class AopConfig {
    // proxyTargetClass = true 强制使用 CGLIB
    // proxyTargetClass = false（默认）则根据目标类是否有接口来选择
}
```

### 2.3 Spring AOP 中的选择策略

```java
// Spring 默认行为（@EnableAspectJAutoProxy 默认配置）
@EnableAspectJAutoProxy(proxyTargetClass = false)

// 实际源码（Spring AOP 内部逻辑）
// org.springframework.aop.framework.DefaultAopProxyFactory

public AopProxy createAopProxy(AdvisedSupport config) {
    // 条件1：有接口 且 proxyTargetClass=false  → JDK代理
    // 条件2：有接口 且 proxyTargetClass=true  → CGLIB
    // 条件3：没接口 → CGLIB
    
    if (config.isProxyTargetClass()) {
        return new CglibAopProxy(config);  // CGLIB
    }
    
    Class<?>[] interfaces = config.getProxiedInterfaces();
    if (interfaces.length > 0 && !hasNoUserSpecifiedDeclaredInterfaces(config)) {
        return new JdkDynamicAopProxy(config);  // JDK代理
    }
    
    return new CglibAopProxy(config);  // CGLIB兜底
}
```

## 🔥 扩展追问 1：两者性能差异有多大？

> **A:** JDK 动态代理在 JDK 8 之前性能比 CGLIB 差很多（约慢 10-20%），但 JDK 8 以后做了优化，差距缩小到 10% 以内。

**关键性能因素：**

| 因素 | JDK 代理 | CGLIB 代理 |
|------|---------|-----------|
| 反射调用 | `method.invoke()` | `methodProxy.invokeSuper()` |
| 字节码生成 | ProxyGenerator 生成的类 | ASM 直接操作字节码 |
| 运行时开销 | 每次调用都要反射 | 生成子类后直接调用 |
| 首次创建 | 快 | 慢（要生成字节码） |
| 多次调用 | 一般 | 更好 |

**Spring Boot 2.x 默认选择**：CGLIB（因为 `proxyTargetClass=true`）

## 🔥 扩展追问 2：JDK 代理为什么要强制面向接口？

> **A:** 这是 JDK 动态代理的实现机制决定的。

```
Proxy.newProxyInstance() 做了什么：
1. 根据接口数组，生成一个 $Proxy0 类（字节码）
2. 这个类继承 java.lang.reflect.Proxy
3. 实现你传入的所有接口
4. 内部持有 InvocationHandler 引用

生成的代理类大致长这样：
public final class $Proxy0 extends Proxy implements UserService {
    private InvocationHandler h;
    
    public void addUser(String name) {
        h.invoke(this, method, args);  // 委托给 Handler
    }
}
```

**因为 Java 单继承的限制，代理类已经继承了 Proxy，所以不能再继承其他类。** 如果目标类没有接口，就无法用 JDK 代理。

## 🔥 扩展追问 3：CGLIB 为什么不能代理 private/final 方法？

> **A:** CGLIB 是通过生成目标类的子类来实现代理的。

```
CGLIB 生成的是子类：
public class UserService$$EnhancerByCGLIB$$abc123 
    extends UserService {
    
    public void addUser(String name) {
        MethodInterceptor.intercept(...);
    }
}
```

**不能代理的原因：**

| 限制 | 原因 |
|------|------|
| private 方法 | 子类不能重写父类的 private 方法 |
| final 方法 | final 方法不能被重写 |
| static 方法 | static 方法不属于实例，不能被继承 |

## 🔥 扩展追问 4：Spring AOP 和 AspectJ 是什么关系？

> **A:** Spring AOP 是 AspectJ 的简化版，AspectJ 是全量 AOP 框架。

| 特性 | Spring AOP | AspectJ |
|------|-----------|---------|
| 织入时机 | 运行时（动态代理） | 编译时/加载时 |
| 切面支持 | 方法级别 | 字段、构造函数、静态块 |
| 代理方式 | JDK/CGLIB | 字节码修改（编译时织入） |
| 性能 | 运行时开销 | 无运行时开销 |
| 依赖 | 轻量 | 需要 AspectJ 编译器（ajc） |
| 配置 | Spring IOC 集成 | 需要额外配置 |

**实际选择：**
- 普通业务用 Spring AOP（足够，集成方便）
- 框架层面追求极致性能用 AspectJ（比如 Spring 框架本身）

## 🔥 扩展追问 5：MyBatis 的 `@Mapper` 注解是怎么用代理的？

> **A:** MyBatis 使用 JDK 动态代理来生成 Mapper 接口的实现类。

```java
// MyBatis 源码核心逻辑（MapperProxyFactory）
public class MapperProxyFactory<T> {
    
    private final Class<T> mapperInterface;
    private final Map<Method, MapperMethod> methodCache = new ConcurrentHashMap<>();
    
    // 创建代理对象
    public T newInstance(SqlSession sqlSession) {
        final MapperProxy<T> proxy = new MapperProxy<>(sqlSession, mapperInterface, methodCache);
        return (T) Proxy.newProxyInstance(
            mapperInterface.getClassLoader(),
            new Class[]{mapperInterface},
            proxy
        );
    }
}

// MapperProxy 实现了 InvocationHandler
public class MapperProxy<T> implements InvocationHandler {
    
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        // 关键：如果是 Object 方法（toString/equals等），直接跳过
        if (Object.class.equals(method.getDeclaringClass())) {
            return method.invoke(this, args);
        }
        
        // 缓存方法
        final MapperMethod mapperMethod = resolveMethod(method);
        return mapperMethod.execute(sqlSession, args);  // 真正执行SQL
    }
}
```

**为什么用 JDK 而不用 CGLIB？** 因为 Mapper 是接口，不需要 CGLIB。JDK 代理更轻量。

---

## 一句话总结

> **JDK 动态代理基于接口通过 Proxy + InvocationHandler 实现，CGLIB 基于继承通过 Enhancer + MethodInterceptor 生成子类实现，Spring AOP 默认 CGLIB（proxyTargetClass=true），两者都不能代理 final/private/static 方法。**

**面试加分话术：**

> "之前项目里排查过一个 MyBatis 懒加载慢的问题，定位到是 CGLIB 代理的副作用——因为 CGLIB 生成的是子类，所以 MyBatis 返回的对象实际上是代理对象，字段访问会有额外开销。解决方式是在 Mapper 里用 `@Fetch(FetchType.LAZY)` 精确控制，而不是全量开懒加载。另外 Spring Boot 2.x 默认强制用 CGLIB 是为了解决循环依赖问题，因为 CGLIB 可以代理不需要接口的类。如果是 SSM 项目用 XML 配置的 Spring AOP，默认是 JDK 代理，这时候如果 Service 没用接口就会报错。"
