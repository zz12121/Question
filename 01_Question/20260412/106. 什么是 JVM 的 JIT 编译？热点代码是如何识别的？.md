## 先说结论

JIT（Just-In-Time）是 JVM **运行时把热点字节码编译成机器码**的技术，让 Java 从解释执行变成编译执行；热点代码通过**方法调用计数器**和**回边计数器**识别，达到阈值后触发 C1/C2 分层编译。

---

## 🔥 核心概念

| 概念 | 说明 |
|------|------|
| **解释执行** | 逐条解释字节码，启动快，执行慢 |
| **JIT 编译** | 热点代码编译成机器码，启动慢，执行快 |
| **热点代码** | 被频繁执行的方法或循环体 |
| **方法调用计数器** | 统计方法被调用次数 |
| **回边计数器** | 统计循环体执行次数 |
| **分层编译** | C1 快速编译 + C2 激进优化，平衡启动和峰值性能 |

---

## 🔥 详细展开

### 1. 为什么需要 JIT？

```java
// Java 执行方式演进：
// 1. 纯解释执行（早期 JVM）
//    - 启动快，无需等待编译
//    - 执行慢，每条字节码都解释执行

// 2. 纯 JIT 编译（如 HotSpot 早期）
//    - 首次执行就编译，启动慢
//    - 编译耗时，用户感知卡顿

// 3. 解释 + JIT 混合（现代 JVM）
//    - 先解释执行，收集统计信息
//    - 热点代码触发 JIT 编译
//    - 平衡启动速度和峰值性能

// 4. 分层编译（JDK 7+ 默认）
//    - 0层：解释执行
//    - 1-3层：C1 编译器（快速生成代码）
//    - 4层：C2 编译器（激进优化）
```

---

### 2. 热点代码识别——两种计数器

```java
// 1. 方法调用计数器（Invocation Counter）
//    - 统计方法被调用的次数
//    - 默认阈值：Client 模式 1500 次，Server 模式 10000 次
//    - 可通过 -XX:CompileThreshold 调整

// 2. 回边计数器（Back Edge Counter）
//    - 统计循环体执行次数（循环"回边"指跳回循环开头的分支）
//    - 默认阈值：Client 模式 13995，Server 模式 10700
//    - 公式：CompileThreshold × (OnStackReplacePercentage / 100)

// 计数器位置：每个方法对应的栈帧里，与方法绑定
```

**计数器的工作机制：**

```java
public class HotSpotDemo {
    
    // 方法调用计数器场景
    public static int add(int a, int b) {
        return a + b;
    }
    
    public static void main(String[] args) {
        // 场景1：方法调用触发 JIT
        for (int i = 0; i < 100000; i++) {
            add(i, i);  // 调用 10 万次，触发 JIT 编译
        }
        
        // 场景2：循环触发 OSR（On-Stack Replacement）
        int sum = 0;
        for (int i = 0; i < 100000; i++) {  // 回边计数器累计
            sum += i;  // 循环体执行次数达到阈值，触发 OSR 编译
        }
    }
}
```

---

### 3. 分层编译模型详解

```java
// JDK 8+ 默认开启分层编译（-XX:+TieredCompilation）

// 0层：解释执行（Tier 0）
// - 启动时所有代码在此层执行
// - 收集方法调用次数、分支跳转频率等统计信息

// 1层：C1 简单编译（Tier 1）
// - 方法调用计数 > 1500
// - 快速编译，只做简单优化（方法内联、常量传播）
// - 无性能分析

// 2层：C1 有限性能分析（Tier 2）
// - 增加性能分析（分支预测、类型继承分析）
// - 为 C2 编译收集信息

// 3层：C1 完全性能分析（Tier 3）
// - 完整性能分析
// - 如果 C2 队列繁忙，在此层多停留

// 4层：C2 编译（Tier 4）
// - 激进优化（逃逸分析、锁消除、循环展开等）
// - 生成高质量机器码
// - 峰值性能接近 C++

// 触发路径：
// 解释执行 → C1（1/2/3层） → C2（4层）
//     ↑___________________________|
//           （去优化后回退）
```

---

### 4. OSR（On-Stack Replacement）—— 循环的 JIT

```java
// 问题：main 方法只执行一次，但里面的循环执行 100 万次
// 方法调用计数器 = 1，永远不会触发 JIT
// 解决方案：OSR 栈上替换

public static void main(String[] args) {
    long start = System.currentTimeMillis();
    
    int sum = 0;
    for (int i = 0; i < 10000000; i++) {  // 回边计数器累计
        sum += i;
    }
    // 当回边计数达到阈值，JVM 会在循环体内"切开"：
    // - 仍在解释执行的老栈帧继续
    // - 同时编译循环体的机器码
    // - 下次循环跳转到新编译的机器码执行
    // - 老栈帧逐渐退出
    
    System.out.println("sum=" + sum);
    System.out.println("耗时=" + (System.currentTimeMillis() - start));
}

// OSR 编译的入口不是方法开头，而是循环头
// 所以 OSR 编译的方法通常不能内联到其他方法里
```

### 5. 热点代码衰减与溢出

```java
// 方法调用计数器的衰减机制：
// - 默认情况下，计数器统计的是"绝对调用次数"
// - 如果方法很久没被调用，应该降低热度
// - 开启 -XX:-UseCounterDecay 关闭衰减（默认开启）

// 衰减时机：
// - GC 时检查
// - 如果方法在一定时间内没被调用，计数器减半

// 计数器溢出：
// - 计数器是有限位数的（如 16 位）
// - 达到最大值后不再增加，但方法仍是热点

// 查看 JIT 编译日志：
// -XX:+PrintCompilation（JDK 8）
// -XX:+LogCompilation（详细日志，需 hsdis 支持）
```

## 🔥 扩展追问 2：JIT 编译的触发条件有哪些？

```java
// 1. 方法调用计数器达到阈值
//    -XX:CompileThreshold=10000（Server 模式默认）

// 2. 回边计数器达到阈值（OSR）
//    -XX:OnStackReplacePercentage=140
//    -XX:CompileThreshold=10000
//    实际阈值 = 10000 × (140/100) = 14000

// 3. 强制编译（开发调试）
//    -XX:+CompileCommand=compileonly,java.lang.String::hashCode

// 4. 后台编译队列
//    - 编译线程异步执行，不阻塞应用线程
//    - 编译队列长度：-XX:CICompilerCount=3（默认）

// 禁止 JIT 编译（仅解释执行）：
// -Xint

// 强制所有方法编译（首次调用就编译）：
// -Xcomp（不推荐，启动极慢）
```

## 🔥 扩展追问 3：C1 vs C2 编译器的区别

| 维度 | C1（Client Compiler） | C2（Server Compiler） |
|------|----------------------|----------------------|
| 编译速度 | **快**（毫秒级） | **慢**（秒级） |
| 优化程度 | 简单优化 | 激进优化 |
| 生成代码质量 | 一般 | 接近 C++ |
| 适用场景 | 客户端、启动速度敏感 | 服务端、长期运行 |
| 分析信息 | 无或有限 | 完整性能分析 |
| 典型优化 | 方法内联、常量传播 | 逃逸分析、锁消除、循环展开 |

```java
// 分层编译的优势：
// 1. 启动阶段：解释 + C1 快速编译，快速达到可接受性能
// 2. 运行阶段：C2 渐进式优化，最终达到峰值性能
// 3. 避免 C2 编译慢导致的启动卡顿

// 历史背景：
// JDK 6 之前：-client 用 C1，-server 用 C2，必须二选一
// JDK 7+：分层编译，两者结合
// JDK 8+：分层编译默认开启，-client/-server 参数废弃
```

## 🔥 扩展追问 4：JIT 编译的副作用与注意事项

```java
// 1. 编译线程 CPU 占用
//    - C2 编译是 CPU 密集型
//    - 大方法编译可能导致短暂 CPU 飙升
//    - 可通过 -XX:CICompilerCount 控制编译线程数

// 2. Code Cache 溢出
//    - 编译后的机器码存放在 Code Cache（默认 240MB）
//    - 满了后停止 JIT 编译，退化为解释执行
//    - 参数：-XX:ReservedCodeCacheSize=512m

// 3. 去优化（Deoptimization）开销
//    - 激进优化假设被打破时回退
//    - 需要恢复解释执行状态，有性能损耗

// 4. 调试困难
//    - JIT 后的代码与源码行号映射复杂
//    - 堆栈跟踪可能显示 [Compiled Code]
//    - 使用 -XX:-OmitStackTraceInFastThrow 保留完整堆栈

// 5. 预热问题
//    - 刚启动时都是解释执行，性能差
//    - 需要"预热"让热点代码触发 JIT
//    - 生产环境：启动后先跑一波流量预热
//    - 或使用 GraalVM Native Image AOT 编译
```

## 一句话收尾

> JIT 是 Java **"先解释收集信息，再编译热点代码"** 的聪明策略，通过**方法调用计数器 + 回边计数器**识别热点，**分层编译（C1 快速 + C2 激进）**平衡启动和峰值性能，让 Java 既能秒启动，又能跑出接近 C++ 的速度。

---
几个关键点再强调一下：

**两种计数器的区别：**
| 计数器 | 统计对象 | 触发结果 |
|--------|---------|---------|
| **方法调用计数器** | 方法被调用次数 | 整个方法 JIT 编译 |
| **回边计数器** | 循环体执行次数 | OSR 栈上替换（只编译循环） |

**分层编译的 5 个层级：**
```
0层：解释执行（收集统计信息）
1层：C1 简单编译（快速，无分析）
2层：C1 有限分析（加性能分析）
3层：C1 完全分析（等 C2 队列）
4层：C2 激进优化（峰值性能）
```

**OSR 的核心价值：**
- `main` 方法只执行一次，但里面的循环执行 100 万次
- 方法调用计数器 = 1，永远不会触发 JIT
- OSR 在**循环头**切入，让长时间运行的循环也能享受 JIT 优化

**面试高频考点：**
- 为什么需要分层编译？→ 平衡启动速度和峰值性能
- C1 和 C2 的区别？→ C1 快但优化少，C2 慢但优化激进
- 如何观察 JIT 编译？→ `-XX:+PrintCompilation`
