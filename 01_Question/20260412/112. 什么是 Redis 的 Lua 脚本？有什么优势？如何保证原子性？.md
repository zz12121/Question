## 先说结论

Redis Lua 脚本是在 Redis 服务端执行的脚本语言，**天然原子性**（脚本执行期间不处理其他命令），优势是**减少网络往返**、**实现复杂逻辑**、**替代事务和分布式锁**。原子性由 Redis 的单线程模型保证，脚本执行期间不会插入其他客户端命令。

| 维度 | 说明 |
|------|------|
| **本质** | 嵌入 Redis 的 Lua 解释器，脚本在服务端执行 |
| **核心优势** | 原子性、减少 RTT、实现复杂业务逻辑 |
| **原子性保证** | Redis 单线程模型，脚本执行期间不处理其他命令 |
| **适用场景** | 库存扣减、限流计数、分布式锁、复杂聚合计算 |
| **限制** | 默认 5 秒超时、脚本需尽量短小、避免阻塞 |

// Java 中使用 Jedis/Lettuce
String luaScript =

### 2. 核心优势详解

#### 优势 1：减少网络往返（RTT）

```java
// ❌ 传统方式：3 次网络往返
String stock = jedis.get("stock:1001");           // RTT 1
if (Integer.parseInt(stock) > 0) {
    jedis.decr("stock:1001");                      // RTT 2
    jedis.sadd("order:user:1001", "1001");         // RTT 3
}

// ✅ Lua 脚本：1 次网络往返，服务端执行
String luaScript = 
    "local stock = redis.call('get', KEYS[1]) " +
    "if tonumber(stock) > 0 then " +
    "    redis.call('decr', KEYS[1]) " +
    "    redis.call('sadd', KEYS[2], ARGV[1]) " +
    "    return 1 " +
    "end " +
    "return 0";

jedis.eval(luaScript, 2, "stock:1001", "order:user:1001", "1001");
```

#### 优势 2：天然原子性

```lua
-- 库存扣减 + 记录订单（原子执行）
local stock = redis.call('get', KEYS[1])
if tonumber(stock) <= 0 then
    return 0  -- 库存不足
end

redis.call('decr', KEYS[1])           -- 扣库存
redis.call('sadd', KEYS[2], ARGV[1])  -- 记录购买用户
redis.call('incr', KEYS[3])           -- 销量统计
return 1
```

**执行期间**：
- 其他客户端的命令**全部排队等待**
- 脚本执行完后才处理其他命令
- 不存在"读到中间状态"的问题

#### 优势 3：替代 Redis 事务

```lua
-- Lua 脚本实现类似 MULTI/EXEC 的效果，但更灵活
local balance = redis.call('get', 'user:1001:balance')
if tonumber(balance) < tonumber(ARGV[1]) then
    return {-1, "余额不足"}  -- 条件判断，可回滚逻辑
end

redis.call('decrby', 'user:1001:balance', ARGV[1])
redis.call('incrby', 'user:1001:spent', ARGV[1])
return {1, "扣款成功"}
```

### 3. 原子性保证机制

```
┌─────────────────────────────────────────────────────────────┐
│                      Redis 单线程模型                        │
├─────────────────────────────────────────────────────────────┤
│  客户端A: EVAL script1  ──┐                                 │
│  客户端B: GET key         │  ← 排队等待                      │
│  客户端C: SET key value   │                                 │
│                           ▼                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  执行 Lua script1（原子性：期间不处理其他命令）       │   │
│  │  1. redis.call('get', key)                          │   │
│  │  2. 业务逻辑计算                                     │   │
│  │  3. redis.call('set', key, newValue)                │   │
│  └─────────────────────────────────────────────────────┘   │
│                           │                                 │
│                           ▼                                 │
│  继续处理客户端B、C的命令...                                │
└─────────────────────────────────────────────────────────────┘
```

**关键点**：
- Redis 是**单线程处理命令**（6.0 后是多线程 IO，但命令执行仍是单线程）
- Lua 脚本作为一个"大命令"整体执行
- 脚本内部的所有 Redis 命令**不会被其他客户端打断**

## 🔥 扩展追问 1：redis.call vs redis.pcall 的区别

```lua
-- redis.call：命令出错时，脚本终止并返回错误给客户端
local result = redis.call('incr', 'string_key')  -- 如果 key 是 list，报错终止

-- redis.pcall：命令出错时，返回错误信息，脚本继续执行
local result = redis.pcall('incr', 'string_key')
if type(result) == 'table' and result.err then
    -- 处理错误，脚本继续
    redis.call('set', 'error:log', result.err)
    return 0
end
```

| 方法 | 错误处理 | 使用场景 |
|------|---------|---------|
| `redis.call` | 抛出错误，脚本终止 | 简单脚本，错误即失败 |
| `redis.pcall` | 返回错误，脚本继续 | 需要错误处理的复杂逻辑 |

## 🔥 扩展追问 2：Lua 脚本的限制与注意事项

### 限制 1：执行超时

```bash
# 默认配置
lua-time-limit 5000  # 5 秒，脚本执行超过此时间会被标记

# 超过后，Redis 只接受 SCRIPT KILL 和 SHUTDOWN NOSAVE
```

**解决方案**：
```lua
-- 脚本尽量短小，复杂逻辑拆分到多个小脚本
-- 或使用 Redis 函数（Redis 7.0+）
```

### 限制 2：随机性命令

```lua
-- ❌ 以下命令在 Lua 脚本中执行会报错（随机性）
redis.call('keys', '*')        -- 返回结果不确定
redis.call('smembers', 'set')  -- 集合无序，结果不确定

-- ✅ 解决方案：使用替代命令
redis.call('scan', 0, 'match', 'prefix:*')  -- SCAN 替代 KEYS
-- 或使用 redis.replicate_commands()（Redis 3.2+）
```

### 限制 3：数据一致性（主从复制）

```lua
-- 问题：Lua 脚本在主库执行，从库重放的是脚本结果
-- 如果脚本依赖当前时间、随机数等，主从不同步

-- 解决方案：Redis 3.2+ 默认将脚本转换为命令序列传播
# redis.conf
lua-replicate-commands yes  # 默认开启
```

## 🔥 扩展追问 3：实战案例 - 分布式锁 + 可重入

```lua
-- 加锁脚本（支持可重入）
-- KEYS[1]: lock:key, ARGV[1]: threadId, ARGV[2]: expireTime

local lockValue = redis.call('get', KEYS[1])

if lockValue == false then
    -- 锁不存在，直接加锁
    redis.call('set', KEYS[1], ARGV[1], 'PX', ARGV[2])
    return 1
elseif string.find(lockValue, ARGV[1]) == 1 then
    -- 锁存在且是当前线程持有（可重入）
    local count = tonumber(string.sub(lockValue, #ARGV[1] + 2)) or 0
    redis.call('set', KEYS[1], ARGV[1] .. ':' .. (count + 1), 'PX', ARGV[2])
    return count + 1
else
    -- 锁被其他线程持有
    return 0
end
```

```lua
-- 解锁脚本（原子性保证）
-- KEYS[1]: lock:key, ARGV[1]: threadId

local lockValue = redis.call('get', KEYS[1])
if not lockValue then
    return 1  -- 锁已释放
end

if string.find(lockValue, ARGV[1]) == 1 then
    local count = tonumber(string.sub(lockValue, #ARGV[1] + 2)) or 1
    if count > 1 then
        -- 可重入次数减一
        redis.call('set', KEYS[1], ARGV[1] .. ':' .. (count - 1))
        return count - 1
    else
        -- 彻底释放
        redis.call('del', KEYS[1])
        return 0
    end
else
    -- 不是当前线程持有的锁
    return -1
end
```

**为什么必须用 Lua？**
- 判断锁归属 + 删除锁 必须是原子操作
- 如果用 `GET` + `DEL` 两条命令，中间可能被其他线程插入

## 🔥 扩展追问 4：Redis 7.0 Functions vs Lua 脚本

| 特性 | Lua 脚本（EVAL） | Redis Functions（7.0+） |
|------|-----------------|------------------------|
| **持久化** | 客户端每次发送 | 服务端持久化存储 |
| **加载** | 每次传输执行 | 一次加载，多次调用 |
| **命名** | 无，通过脚本内容识别 | 有函数名，如 `mylib.myfunc` |
| **管理** | 难管理 | 可列出、删除、备份 |
| **适用** | 临时逻辑 | 常驻业务逻辑 |

```bash
# Redis 7.0 Functions 示例
FUNCTION LOAD REPLACE "#!lua name=mylib\n redis.register_function('incrby', function(keys, args) return redis.call('incrby', keys[1], args[1]) end)"

FUNCTION CALL mylib.incrby mykey 10
```

## 一句话收尾

Redis Lua 脚本利用**单线程模型**实现天然原子性，将多个命令打包在服务端一次执行，既减少网络往返又保证操作不可分割，是**库存扣减、分布式锁、限流计数**等场景的核心利器。
