## 先说结论

ZGC 和 Shenandoah 是**低延迟垃圾收集器**，目标都是 **<10ms 的停顿时间**；ZGC 用**染色指针**和**读屏障**实现并发整理，Shenandoah 用**转发指针**实现并发复制；相比 G1 的**分区 + 增量回收**，它们实现了**全并发**（标记、转移、重定位都并发），适合大堆内存、低延迟要求的场景。

## 🔥 核心概念

| 维度 | G1 | ZGC | Shenandoah |
|------|-----|-----|------------|
| 设计目标 | 平衡吞吐和延迟 | **超低延迟** (<10ms) | **超低延迟** (<10ms) |
| 最大堆 | 几十 GB | **TB 级** (16TB) | **TB 级** |
| 停顿时间 | 几十~几百 ms | **<10ms** | **<10ms** |
| 核心机制 | 分区 + 增量回收 | 染色指针 + 读屏障 | 转发指针 + Brooks 屏障 |
| 并发阶段 | 标记并发，整理 STW | **全并发**（标记/转移/重定位） | **全并发** |
| 内存开销 | 较低 | 较高（染色指针占用高位） | 较高（转发指针） |
| 适用 JDK | JDK 7+ | JDK 11+ (生产)，JDK 15+ 正式 | JDK 12+ (实验)，JDK 15+ 正式 |
| 平台支持 | 全平台 | Linux x64 (JDK 11)，全平台 (JDK 14+) | 全平台 |

## 🔥 详细展开

### 1. G1 的局限——为什么需要新一代 GC

```java
// G1 的设计：分区（Region）+ 增量回收
// - 堆划分为多个 1~32MB 的 Region
// - 优先回收垃圾最多的 Region
// - 但：最终标记和筛选回收需要 STW（Stop-The-World）

// G1 的停顿来源：
// 1. 初始标记：STW，标记 GC Roots（短暂）
// 2. 并发标记：并发执行
// 3. 最终标记：STW，处理 SATB 队列（几十 ms）
// 4. 筛选回收：STW，复制存活对象（最耗时，几百 ms）

// 问题：大堆时，存活对象多，筛选回收的 STW 不可接受
// 100GB 堆，10GB 存活对象 → 复制 10GB 需要几百 ms STW
```

---

### 2. ZGC —— 染色指针 + 读屏障

```java
// ZGC 的核心设计：染色指针（Colored Pointers）

// 64 位指针的高 4 位存储元数据（ZGC 只使用 42 位寻址）：
// [16位未使用][4位元数据][42位地址]
//            │  │  │  │
//            │  │  │  └── Finalizable：对象是否仅通过 finalize 访问
//            │  │  └───── Remapped：指针是否已重映射
//            │  └──────── Marked1/Marked0：标记位（交替使用）
//            └─────────── 保留

// 染色指针的优势：
// 1. 标记信息在指针里，无需访问对象头
// 2. 多视图映射：同一物理地址可映射到多个虚拟地址（Remapped/Marked）
// 3. 转移对象无需修改引用，只需切换视图

// 读屏障（Load Barrier）：
// 访问对象时检查指针状态，如需重定向则转发
Object readBarrier(Object* ref) {
    if (ref & MARKED_MASK) {  // 检查染色位
        return slow_path(ref); // 需要重映射，进入慢路径
    }
    return ref;  // 快速路径，直接返回
}
```

**ZGC 的执行周期：**

```java
// ZGC 周期（全并发，无 STW 整理）：

// 1. 并发标记（Concurrent Mark）
//    - 从 GC Roots 出发，遍历对象图
//    - 标记存活对象（染色指针设置 Marked 位）
//    - 应用线程通过读屏障协助标记

// 2. 并发预备重分配（Concurrent Prepare for Relocate）
//    - 选择需要清理的 Region
//    - 创建转发表（Forwarding Table）

// 3. 并发重分配（Concurrent Relocate）
//    - 复制存活对象到新 Region
//    - 更新转发表
//    - 旧对象保留，转发指针指向新对象
//    - STW：仅初始迁移roots（<1ms）

// 4. 并发重映射（Concurrent Remap）
//    - 修正所有指向旧对象的引用
//    - 与下一次标记合并，避免单独遍历

// 关键：重分配阶段，应用线程访问对象时：
// - 读屏障检查指针是否指向旧 Region
// - 如果是，通过转发表转发到新对象
// - 同时自愈（Self-Healing）：修正引用，下次直接访问新对象
```

---

### 3. Shenandoah —— 转发指针 + Brooks 屏障

```java
// Shenandoah 的核心设计：转发指针（Brooks Pointer）

// 每个对象头增加一个额外字段：转发指针
// 正常状态：转发指针指向自己
// 转移状态：转发指针指向新对象地址

class ObjectHeader {
    mark_word mark;           // 常规对象头
    object_ref brooks_ptr;    // 转发指针（Shenandoah 特有）
    // ...
};

// 访问对象的流程：
Object* read_ref(Object* ref) {
    Object* brooks = ref->brooks_ptr;
    if (brooks == ref) {
        return ref;  // 未转移，直接访问
    } else {
        return brooks;  // 已转移，转发到新对象
    }
}
```

**Shenandoah 的执行周期：**

```java
// Shenandoah 周期（与 ZGC 类似，实现细节不同）：

// 1. 初始标记（Init Mark）：STW，标记 GC Roots（<1ms）

// 2. 并发标记（Concurrent Mark）：遍历对象图

// 3. 最终标记（Final Mark）：STW，处理 SATB 队列（<1ms）

// 4. 并发清理（Concurrent Cleanup）：回收无存活对象的 Region

// 5. 并发转移（Concurrent Evacuation）：
//    - 复制存活对象到新 Region
//    - 更新转发指针指向新对象
//    - 应用线程通过 Brooks 屏障转发访问

// 与 ZGC 的主要区别：
// 1. ZGC 用染色指针，Shenandoah 用转发指针
// 2. ZGC 指针染色在硬件层面（MMU 支持），Shenandoah 需要额外对象头字段
// 3. ZGC 支持更大堆（16TB），Shenandoah 通常到几百 GB
```

## 🔥 扩展追问 1：ZGC 的染色指针详解

```java
// 染色指针的内存布局（64位）：
// ┌─────────────────────────────────────────────────────────────┐
// │ 0000 0000 │ 0000 │ 0000 0000 0000 0000 0000 00 │ 00 0000 0000 │
// │  16位保留  │ 4位染色 │        42位地址（4TB寻址）       │  低位对齐  │
// │           │ 0000=Remapped │                              │            │
// │           │ 0001=Marked0  │                              │            │
// │           │ 0010=Marked1  │                              │            │
// │           │ 0100=Finalizable│                            │            │
// └─────────────────────────────────────────────────────────────┘

// 多视图映射（Multi-Mapping）：
// 同一物理内存映射到多个虚拟地址，仅染色位不同
// 应用看到的指针：Marked0 视图
// GC 看到的指针：Remapped 视图
// 转移时无需修改引用，只需切换视图映射

// 自愈（Self-Healing）：
// 应用线程访问旧对象时：
// 1. 读屏障发现指针是旧视图（Marked0）
// 2. 查询转发表找到新地址
// 3. 返回新对象，并**修正引用**（更新为 Remapped 视图）
// 4. 下次访问直接走快速路径
```

## 🔥 扩展追问 2：什么时候选 ZGC / Shenandoah / G1？

| 场景 | 推荐 GC | 原因 |
|------|---------|------|
| 堆 < 8GB，常规应用 | G1 | 开销低，足够用 |
| 堆 8-32GB，低延迟要求 | G1 / ZGC | G1 调优后可接受，ZGC 更保险 |
| 堆 > 32GB，低延迟要求 | **ZGC** / Shenandoah | 大堆低延迟唯一选择 |
| 堆 > 100GB | **ZGC** | ZGC 支持更大堆（16TB） |
| 容器环境，内存受限 | G1 / Shenandoah | ZGC 需要更多内存开销 |
| Windows/非 Linux | Shenandoah | ZGC JDK 11 仅 Linux，JDK 14+ 全平台 |
| 极致延迟要求（<1ms） | ZGC | 实测停顿更稳定 |

## 🔥 扩展追问 3：ZGC 的调优参数

```java
// ZGC 设计哲学：自动调优，参数极少

// 基本配置：
-XX:+UseZGC                    // 启用 ZGC
-Xms<size> -Xmx<size>          // 堆大小（建议固定，避免动态调整）
-XX:SoftMaxHeapSize=<size>     // 软最大堆（ZGC 会尽量保持在此以下）

// 并发线程数（默认自动，基于 CPU 数）
-XX:ConcGCThreads=<n>          // 并发 GC 线程数
-XX:ParallelGCThreads=<n>      // 并行 GC 线程数（STW 阶段用）

// GC 触发时机：
-XX:ZCollectionInterval=<sec>  // 强制 GC 间隔（默认 0，不强制）
-XX:ZAllocationSpikeTolerance=<n> // 分配速率容忍度（默认 2）

// 诊断参数：
-XX:+ZGenerational              // JDK 21+ 启用分代 ZGC（实验性）
-XX:+ZVerifyViews               // 验证染色指针视图（调试）
-XX:+ZDebugGV                   // 详细 GC 日志

// ZGC 的调优原则：
// 1. 给足堆内存（ZGC 需要更多空间用于并发转移）
// 2. 关注分配速率，避免突发分配导致 GC 跟不上
// 3. 大部分场景无需调优，默认即可
```

## 🔥 扩展追问 4：ZGC 的弱点和注意事项

```java
// 1. 内存开销
// - 染色指针需要 64 位 JVM（压缩指针失效）
// - 小堆（<4GB）时内存效率不如 G1
// - 建议堆 >= 8GB 再考虑 ZGC

// 2. 吞吐量略低于 G1
// - 读屏障有轻微开销（~5%）
// - 并发阶段占用 CPU
// - 如果不需要低延迟，G1 吞吐量更好

// 3. 平台支持
// - JDK 11：仅 Linux x64
// - JDK 14：支持 macOS 和 Windows
// - JDK 15：正式特性（非实验性）

// 4. 与某些特性不兼容
// - 压缩 OOP（UseCompressedOops）自动关闭
// - 某些 JVMTI 代理可能不兼容

// 5. 分代 ZGC（JDK 21+）
// - 传统 ZGC 不分代，年轻对象和老对象一起处理
// - JDK 21 引入分代 ZGC（-XX:+ZGenerational）
// - 分代后吞吐量更好，延迟依然优秀
```

## 一句话收尾

> ZGC 和 Shenandoah 是**大堆低延迟场景**的救星，通过**染色指针/转发指针 + 读屏障**实现全并发 GC，停顿时间稳定在 **<10ms** 且与堆大小无关；G1 适合常规场景，ZGC 适合**百 GB 级大堆 + 极致延迟要求**，JDK 21 的分代 ZGC 更是未来趋势。

---
几个关键点再强调一下：

**ZGC vs Shenandoah 的核心区别：**
| 维度 | ZGC | Shenandoah |
|------|-----|------------|
| 核心技术 | **染色指针**（指针里存元数据） | **转发指针**（对象头加字段） |
| 指针修复 | 多视图映射，无需修改引用 | 需更新转发指针 |
| 最大堆 | **16TB** | 几百 GB |
| 平台 | JDK 11 仅 Linux，JDK 14+ 全平台 | 全平台 |

**染色指针的 4 位元数据：**
- `Marked0/Marked1`：交替标记，区分不同 GC 周期
- `Remapped`：指针已重映射（正常状态）
- `Finalizable`：仅通过 finalize 访问

**自愈（Self-Healing）机制：**
- 应用线程访问旧对象 → 读屏障拦截 → 查转发表 → 返回新对象 → **修正引用**
- 下次访问直接走快速路径，无需转发

**选择建议（面试必答）：**
- 堆 < 8GB → **G1**（简单高效）
- 堆 8-32GB，低延迟 → G1 调优 或 **ZGC**
- 堆 > 32GB，低延迟 → **ZGC** / Shenandoah
- 堆 > 100GB → **ZGC**（唯一选择）
- JDK 21+ → 尝试 **分代 ZGC**（`-XX:+ZGenerational`）
