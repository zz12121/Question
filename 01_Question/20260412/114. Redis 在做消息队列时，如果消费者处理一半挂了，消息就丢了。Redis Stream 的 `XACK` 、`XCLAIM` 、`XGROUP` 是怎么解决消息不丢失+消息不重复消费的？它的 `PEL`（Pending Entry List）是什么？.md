## 先说结论

Redis Stream 通过 **Consumer Group（消费者组）+ PEL（待处理列表）+ XACK 确认机制** 解决消息不丢失问题，通过 **XCLAIM 转移所有权** 处理消费者宕机，通过 **消息 ID 去重** 保证不重复消费。PEL 是记录已投递但未确认消息的列表，是 Stream 实现可靠消费的核心数据结构。

---

## 🔥 核心概念

| 概念 | 作用 |
|------|------|
| **Consumer Group** | 消费者组，组内多个消费者竞争消费，消息只被消费一次 |
| **PEL** | Pending Entry List，记录已投递但未 XACK 确认的消息 |
| **XACK** | 确认消息已处理，从 PEL 中移除 |
| **XCLAIM** | 转移消息所有权，处理宕机消费者的未确认消息 |
| **消息 ID** | 时间戳+序列号，全局唯一，天然去重依据 |

### 2. 核心命令详解

```bash
# 1. 创建消费者组（从开头消费）
XGROUP CREATE stream:orders group:order-processors 0

# 2. 消费者读取消息（阻塞等待）
XREADGROUP GROUP group:order-processors consumer-a COUNT 1 BLOCK 5000 STREAMS stream:orders >

# 3. 确认消息处理完成
XACK stream:orders group:order-processors 1713000000-0

# 4. 查看 PEL（待处理列表）
XPENDING stream:orders group:order-processors - + 10

# 5. 转移消息所有权（ConsumerB 宕机，ConsumerC 接管）
XCLAIM stream:orders group:order-processors consumer-c 3600000 1713000001-0
```

### 3. Java 代码实现

```java
// Lettuce 实现可靠消费
public class StreamConsumer {
    
    private StatefulRedisConnection<String, String> connection;
    private String streamKey = "stream:orders";
    private String groupName = "group:order-processors";
    private String consumerName = "consumer-" + UUID.randomUUID();
    
    public void startConsume() {
        RedisAsyncCommands<String, String> async = connection.async();
        
        // 1. 创建消费者组（幂等）
        async.xgroupCreate(XReadArgs.StreamOffset.from(streamKey, "0"), 
            groupName, XGroupCreateArgs.Builder.mkstream())
            .exceptionally(e -> {
                // 组已存在会报错，忽略
                return "OK";
            });
        
        // 2. 启动消费循环
        while (!Thread.interrupted()) {
            try {
                // 读取消息（> 表示读取新消息）
                List<StreamMessage<String, String>> messages = async.xreadgroup(
                    XReadArgs.GroupOffset.consumer(groupName, consumerName),
                    XReadArgs.StreamOffset.latest(streamKey)
                ).get(5, TimeUnit.SECONDS);
                
                for (StreamMessage<String, String> msg : messages) {
                    processMessage(msg);
                    // 3. 确认消息
                    async.xack(streamKey, groupName, msg.getId());
                }
            } catch (Exception e) {
                log.error("消费异常", e);
            }
        }
    }
    
    private void processMessage(StreamMessage<String, String> msg) {
        // 业务处理...
        String orderId = msg.getBody().get("order_id");
        processOrder(orderId);
    }
}
```

## 🔥 扩展追问 1：PEL 详解 - 可靠消费的核心

```
PEL 数据结构（Redis 内部实现）：
┌─────────────────────────────────────────────────────────────┐
│  PEL (Pending Entry List) - 每个消费者组一个                  │
├─────────────────────────────────────────────────────────────┤
│  Key: stream 消息 ID                                          │
│  Value: {                                                     │
│    consumer: "consumer-a",      // 当前负责的消费者           │
│    time: 1713000000000,         // 最后投递时间（毫秒）        │
│    deliveries: 3                // 投递次数（用于重试统计）    │
│  }                                                            │
└─────────────────────────────────────────────────────────────┘
```

**PEL 的作用**：
1. **防止消息丢失**：消息被读取后进入 PEL，未 XACK 前不会消失
2. **支持重试**：消费者宕机后，其他消费者可通过 XCLAIM 接管
3. **监控消费延迟**：通过 PEL 中的时间戳计算消息处理耗时

```bash
# 查看 PEL 详情
XPENDING stream:orders group:order-processors - + 10

# 返回结果：
1) (integer) 2                    # PEL 中消息总数
2) "1713000001-0"                 # 最小 ID
3) "1713000002-0"                 # 最大 ID
4) 1) 1) "consumer-b"             # 消费者
      2) "2"                      # 该消费者的待处理消息数
   5) 1) "consumer-a"
      2) "1"
```

## 🔥 扩展追问 2：宕机恢复与消息转移

```java
/**
 * 死信检测与转移（定时任务执行）
 */
public class PendingMessageHandler {
    
    @Scheduled(fixedDelay = 30000)  // 每 30 秒执行
    public void checkPendingMessages() {
        RedisAsyncCommands<String, String> async = connection.async();
        
        // 1. 获取所有待处理消息
        List<PendingMessage> pendingList = async.xpending(
            streamKey, groupName, 
            Range.unbounded(), 
            Limit.from(100)
        ).get();
        
        for (PendingMessage pending : pendingList) {
            String messageId = pending.getId();
            String consumer = pending.getConsumer();
            long idleTime = pending.getMillisSinceLastDelivery();
            
            // 2. 超过 60 秒未确认，认为是死信
            if (idleTime > 60000) {
                // 3. 转移给当前消费者处理
                async.xclaim(streamKey, groupName, 
                    consumerName, 
                    XClaimArgs.Builder.minIdleTime(60000),
                    messageId);
                
                log.warn("消息 {} 从 {} 转移给 {} 处理", 
                    messageId, consumer, consumerName);
            }
        }
    }
}
```

```bash
# XCLAIM 命令详解
XCLAIM stream:orders group:order-processors consumer-c 60000 1713000001-0

# 参数说明：
# 60000 = min-idle-time，消息空闲超过 60 秒才能被转移
# 防止两个消费者同时争抢同一条消息
```

## 🔥 扩展追问 3：消息去重机制

```java
/**
 * 幂等消费实现
 */
public class IdempotentConsumer {
    
    private RedisCommands<String, String> redis;
    private String processedSet = "processed:message:ids";
    
    public void consume(StreamMessage<String, String> msg) {
        String messageId = msg.getId();
        
        // 1. 检查是否已处理（Set 去重）
        Long added = redis.sadd(processedSet, messageId);
        if (added == 0) {
            log.info("消息 {} 已处理，跳过", messageId);
            return;  // 已存在，重复消费
        }
        
        // 2. 设置过期时间（7天）
        redis.expire(processedSet, 7 * 24 * 3600);
        
        try {
            // 3. 业务处理
            processBusiness(msg);
            
            // 4. 确认消息
            redis.xack(streamKey, groupName, messageId);
            
        } catch (Exception e) {
            // 处理失败，从 Set 中移除，允许重试
            redis.srem(processedSet, messageId);
            throw e;
        }
    }
}
```

**去重策略对比**：

| 策略 | 实现 | 适用场景 |
|------|------|---------|
| **消息 ID 去重** | Stream ID 天然唯一 | 基础去重 |
| **业务 ID 去重** | 订单号等业务唯一键 | 业务幂等 |
| **Redis Set 去重** | 已处理 ID 存入 Set | 精确去重，需内存 |
| **数据库唯一索引** | 插入时利用 DB 约束 | 最终一致性 |

## 🔥 扩展追问 4：Stream vs List / PubSub / 专业 MQ

| 特性 | List (BLPOP) | PubSub | Stream | Kafka/RabbitMQ |
|------|-------------|--------|--------|----------------|
| **持久化** | ✅ | ❌ | ✅ | ✅ |
| **消费者组** | ❌ | ❌ | ✅ | ✅ |
| **消息确认** | ❌ | ❌ | ✅ (XACK) | ✅ |
| **消息回溯** | ❌ | ❌ | ✅ | ✅ |
| **消息堆积** | 内存限制 | 内存限制 | 内存+磁盘 | 磁盘 |
| **性能** | 高 | 极高 | 高 | 极高 |
| **使用场景** | 简单队列 | 实时广播 | 可靠队列 | 企业级 MQ |

```java
// Stream 适用场景判断
if (需要消息不丢失 && 需要消费者组 && 消息量 < 百万/天) {
    使用 Redis Stream;
} else if (需要企业级特性 || 消息量巨大) {
    使用 Kafka/RabbitMQ;
} else if (简单队列 && 允许丢失) {
    使用 List;
} else if (实时推送 && 允许丢失) {
    使用 PubSub;
}
```

## 一句话收尾

Redis Stream 通过 **Consumer Group 实现负载均衡**、**PEL 记录未确认消息**、**XACK 确认消费**、**XCLAIM 转移死信**，配合**消息 ID 天然唯一性**，实现了轻量级可靠消息队列，适合中小规模的消息不丢失场景。
