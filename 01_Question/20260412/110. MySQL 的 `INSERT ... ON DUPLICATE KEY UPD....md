## 先说结论

`INSERT ... ON DUPLICATE KEY UPDATE`（简称 IODKU）的加锁行为取决于冲突类型和隔离级别：
- **主键冲突**：对主键索引加**排他记录锁（X Record Lock）**
- **唯一索引冲突**：对唯一索引加**排他记录锁**，同时回表对主键也加**排他记录锁**
- **RR vs RC**：RR 可能额外加 Gap 锁防幻读，RC 只加记录锁

---

## 🔥 核心概念

| 维度 | 说明 |
|------|------|
| **语句本质** | 先尝试 INSERT，冲突则转 UPDATE，是"原子性"的 UPSERT 操作 |
| **加锁时机** | 检测到重复键时立即加锁，UPDATE 阶段持有锁不放 |
| **冲突类型差异** | 主键冲突只锁主键；唯一索引冲突需锁唯一索引 + 主键（回表） |
| **隔离级别影响** | RR 加 Next-Key Lock（记录+间隙），RC 只加 Record Lock |

---

## 🔥 详细展开

### 1. 执行流程与加锁分析

```sql
-- 表结构
CREATE TABLE user (
    id INT PRIMARY KEY AUTO_INCREMENT,
    phone VARCHAR(20) UNIQUE KEY,  -- 唯一索引
    name VARCHAR(50),
    age INT,
    INDEX idx_name (name)
) ENGINE=InnoDB;

-- IODKU 语句
INSERT INTO user (id, phone, name, age) 
VALUES (1, '13800138000', '张三', 20)
ON DUPLICATE KEY UPDATE age = age + 1;
```

#### 场景 A：主键冲突（id=1 已存在）

```sql
-- 事务1
BEGIN;
INSERT INTO user (id, phone, name, age) 
VALUES (1, '13900139000', '李四', 25)  -- phone不同，但id冲突
ON DUPLICATE KEY UPDATE age = age + 1;
-- 加锁：对主键 id=1 加 X Record Lock
```

**加锁情况**：
- 主键索引：`id=1` 的 **X 记录锁**
- 唯一索引：无锁（因为 INSERT 阶段就被主键冲突拦截了）

#### 场景 B：唯一索引冲突（phone 已存在）

```sql
-- 事务1
BEGIN;
INSERT INTO user (id, phone, name, age) 
VALUES (2, '13800138000', '王五', 30)  -- id不同，但phone冲突
ON DUPLICATE KEY UPDATE age = age + 1;
-- 加锁：phone唯一索引 + 主键id 都要加锁
```

**加锁情况**（RR 隔离级别）：
- 唯一索引 `phone='13800138000'`：**X Record Lock**
- 回表找到对应主键 `id=1`，对主键 `id=1`：**X Record Lock**
- RR 额外：唯一索引和主键的 **Gap Lock**（防止幻读）

### 2. RR vs RC 隔离级别对比

```sql
-- 会话1（RR级别）
BEGIN;
INSERT INTO user (id, phone, name, age) 
VALUES (2, '13800138000', '王五', 30)
ON DUPLICATE KEY UPDATE age = age + 1;
-- 加锁：唯一索引 Next-Key Lock + 主键 Record Lock
```

```sql
-- 会话2（RR级别）
BEGIN;
INSERT INTO user (id, phone, name, age) 
VALUES (3, '13800138001', '赵六', 35);  -- phone=13800138001 是新的
-- 如果 13800138001 落在 Gap 区间内，会被阻塞！
```

| 隔离级别 | 唯一索引加锁 | 主键加锁 | Gap Lock |
|---------|-------------|---------|----------|
| **RR** | X Record Lock + Gap Lock | X Record Lock | ✅ 有，防幻读 |
| **RC** | X Record Lock | X Record Lock | ❌ 无 |

### 3. 死锁场景演示

```sql
-- 死锁场景：两个事务交叉插入相同唯一键
-- 时间线：
T1: BEGIN;
T1: INSERT INTO user VALUES (1, 'A', ...) ON DUPLICATE KEY UPDATE ...;
    -- 发现 phone='A' 不存在，获得 phone='A' 的插入意向锁
    -- 但 phone='A' 之前被 T2 持有 Gap Lock，等待...

T2: BEGIN;
T2: INSERT INTO user VALUES (2, 'B', ...) ON DUPLICATE KEY UPDATE ...;
    -- 发现 phone='B' 不存在，获得 phone='B' 的插入意向锁
    -- 但 phone='B' 之前被 T1 持有 Gap Lock，等待...

-- 结果：死锁！
```

**死锁原因**：
- RR 级别下，IODKU 先对唯一索引加 Gap Lock
- 两个事务互相持有对方需要的间隙，形成循环等待

## 🔥 扩展追问 1：主键冲突 vs 唯一索引冲突，加锁差异的本质

```sql
-- 主键冲突：只访问主键索引
INSERT INTO user (id, ...) VALUES (1, ...) 
ON DUPLICATE KEY UPDATE age=age+1;
-- 索引访问路径：聚簇索引 → 发现冲突 → 加锁 → UPDATE

-- 唯一索引冲突：需要回表
INSERT INTO user (phone, ...) VALUES ('138...', ...) 
ON DUPLICATE KEY UPDATE age=age+1;
-- 索引访问路径：二级唯一索引 → 发现冲突 → 加锁 → 回表查主键 → 对主键加锁 → UPDATE
```

**本质区别**：
- 主键是聚簇索引，数据在一起，只需一次索引访问
- 唯一索引是二级索引，找到后需回表定位主键，因此要对两个索引都加锁

## 🔥 扩展追问 2：高并发下的死锁规避方案

### 方案 1：降低隔离级别到 RC

```sql
-- 如果业务允许不可重复读，用 RC 避免 Gap Lock
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

### 方案 2：先 SELECT 后 INSERT/UPDATE（非原子，不推荐）

```sql
-- 不推荐：非原子，并发下仍有竞态条件
SELECT * FROM user WHERE phone = '13800138000' FOR UPDATE;
-- 不存在则 INSERT，存在则 UPDATE
```

### 方案 3：使用分布式锁（Redis/ZK）

```java
// 推荐：高并发场景下，先加分布式锁再操作DB
String lockKey = "user:phone:" + phone;
RLock lock = redissonClient.getLock(lockKey);
try {
    lock.lock();
    // 执行 IODKU
    userMapper.upsert(phone, name, age);
} finally {
    lock.unlock();
}
```

### 方案 4：调整 SQL 顺序，减少冲突概率

```sql
-- 批量插入时，按主键/唯一键排序，减少死锁
INSERT INTO user VALUES (1, 'A', ...), (2, 'B', ...), (3, 'C', ...)
ON DUPLICATE KEY UPDATE age=VALUES(age);
```

## 🔥 扩展追问 3：IODKU 与 REPLACE INTO 的加锁差异

| 特性 | INSERT ... ON DUPLICATE KEY UPDATE | REPLACE INTO |
|------|-----------------------------------|--------------|
| **冲突处理** | 保留原记录，UPDATE 指定字段 | 删除原记录，插入新记录 |
| **主键变化** | 主键不变 | 主键重新生成（AUTO_INCREMENT） |
| **加锁范围** | 只锁冲突记录 | 先 DELETE（可能锁更多）再 INSERT |
| **性能** | 较好 | 较差（两次写操作） |
| **使用场景** | 增量更新（如计数器） | 完全替换记录 |

```sql
-- REPLACE INTO 实际执行：DELETE + INSERT
REPLACE INTO user (id, phone, name, age) VALUES (1, '138...', '张三', 20);
-- 等价于：
-- DELETE FROM user WHERE id=1;  -- 可能触发级联删除
-- INSERT INTO user VALUES (1, '138...', '张三', 20);
```

## 🔥 扩展追问 4：如何查看实际加锁情况

```sql
-- 开启锁监控
SHOW ENGINE INNODB STATUS;

-- 查看当前锁等待
SELECT * FROM information_schema.INNODB_LOCKS;
SELECT * FROM information_schema.INNODB_LOCK_WAITS;

-- MySQL 8.0 性能库
SELECT * FROM performance_schema.data_locks 
WHERE OBJECT_NAME = 'user';

-- 查看事务
SELECT * FROM information_schema.INNODB_TRX;
```

**锁信息解读**：
```
RECORD LOCKS space id 58 page no 3 n bits 72 index PRIMARY of table `test`.`user`
trx id 12345 lock_mode X locks rec but not gap
-- X locks rec but not gap = 记录锁，无间隙锁
-- X locks gap before rec = 间隙锁
-- X = 排他锁，S = 共享锁
```

## 一句话收尾

`INSERT ... ON DUPLICATE KEY UPDATE` 在检测到重复键时对冲突索引加排他记录锁，唯一索引冲突需额外回表锁主键；RR 隔离级别会加 Gap 锁防幻读但也增加死锁风险，高并发场景建议用分布式锁或降隔离级别到 RC。
