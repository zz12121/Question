## 先说结论

Redis Pipeline 是**客户端技术**，通过**一次发送多个命令、一次读取多个结果**来减少网络 RTT，但**不保证原子性**；批量命令（如 MGET/MSET）是**服务端原子操作**，但命令类型受限。Pipeline 适合**大量独立命令**的场景，批量命令适合**同类多键操作**。

---

## 🔥 核心概念

| 维度 | Pipeline | 批量命令（MGET/MSET 等） |
|------|----------|-------------------------|
| **实现位置** | 客户端 | 服务端 |
| **原子性** | ❌ 非原子，命令穿插执行 | ✅ 原子执行 |
| **命令类型** | 任意命令混合 | 固定批量命令 |
| **RTT 优化** | ✅ 一次往返 | ✅ 一次往返 |
| **内存占用** | 客户端缓存结果 | 服务端处理 |
| **使用场景** | 大量独立操作 | 同类多键读写 |

## 🔥 详细展开

### 1. Pipeline 工作原理

```
┌─────────────────────────────────────────────────────────────┐
│  无 Pipeline（3 次 RTT）                                     │
│  Client ──GET k1──→ Server                                  │
│  Client ←─v1─────── Server  ← RTT 1                        │
│  Client ──GET k2──→ Server                                  │
│  Client ←─v2─────── Server  ← RTT 2                        │
│  Client ──GET k3──→ Server                                  │
│  Client ←─v3─────── Server  ← RTT 3                        │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  有 Pipeline（1 次 RTT）                                     │
│  Client ──GET k1──→                                         │
│  Client ──GET k2──→  缓冲发送                                │
│  Client ──GET k3──→                                         │
│           ↓                                                 │
│  Client ←─v1───────                                         │
│  Client ←─v2───────  批量读取  ← RTT 1                      │
│  Client ←─v3───────                                         │
└─────────────────────────────────────────────────────────────┘
```

### 2. Java 代码示例

```java
// Jedis Pipeline 示例
try (Jedis jedis = jedisPool.getResource()) {
    Pipeline pipeline = jedis.pipelined();
    
    // 批量发送命令（不等待结果）
    for (int i = 0; i < 1000; i++) {
        pipeline.get("key:" + i);
        pipeline.incr("counter:" + i);
    }
    
    // 同步获取所有结果
    List<Object> results = pipeline.syncAndReturnAll();
    
    // 处理结果
    for (int i = 0; i < results.size(); i += 2) {
        String value = (String) results.get(i);      // get 结果
        Long count = (Long) results.get(i + 1);      // incr 结果
    }
}
```

## 🔥 扩展追问 1：Pipeline vs 批量命令深度对比

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 读取 1000 个不相关的 key | Pipeline + MGET | MGET 限制在一条命令，太多 key 会阻塞 |

## 🔥 扩展追问 2：Pipeline 的注意事项

### 注意 1：非原子性

```java
// ❌ 错误：用 Pipeline 实现原子性
Pipeline pipeline = jedis.pipelined();
pipeline.decr("stock:1001");        // 可能被执行
pipeline.sadd("orders:1001", "1");  // 可能失败
pipeline.sync();

// 问题：两个命令之间可能插入其他客户端的操作
// 解决：用 Lua 脚本或 MULTI/EXEC
```

### 注意 2：内存溢出风险

```java
// ❌ 危险：Pipeline 缓存太多结果
Pipeline pipeline = jedis.pipelined();
for (int i = 0; i < 1000000; i++) {
    pipeline.get("key:" + i);  // 100 万个命令缓冲在内存
}
List<Object> results = pipeline.syncAndReturnAll();  // 内存爆炸！

// ✅ 正确：分批 Pipeline
int batchSize = 1000;
for (int i = 0; i < 1000000; i += batchSize) {
    Pipeline batchPipeline = jedis.pipelined();
    for (int j = i; j < i + batchSize && j < 1000000; j++) {
        batchPipeline.get("key:" + j);
    }
    List<Object> batchResults = batchPipeline.syncAndReturnAll();
    // 处理本批结果，释放内存
}
```

### 注意 3：错误处理

```java
Pipeline pipeline = jedis.pipelined();
pipeline.get("string_key");
pipeline.hget("hash_key", "field");  // 类型错误，但 Pipeline 不会立即抛异常
pipeline.get("another_key");

List<Object> results = pipeline.syncAndReturnAll();
// results 中可能包含 JedisDataException 对象
for (Object result : results) {
    if (result instanceof JedisDataException) {
        // 处理错误
    }
}
```

## 🔥 扩展追问 3：Pipeline 与事务的区别

```java
// MULTI/EXEC：事务，原子性但非隔离
jedis.multi();
jedis.decr("stock:1001");
jedis.sadd("orders:1001", "1");
jedis.exec();  // 原子执行，但中间状态可见

// Pipeline：非原子，只是批量发送
Pipeline pipeline = jedis.pipelined();
pipeline.decr("stock:1001");
pipeline.sadd("orders:1001", "1");
pipeline.sync();  // 命令可能穿插执行

// Lua 脚本：真正原子 + 隔离
jedis.eval(luaScript, 2, "stock:1001", "orders:1001", "1");
```

## 🔥 扩展追问 4：Pipeline 性能实测

```java
@Test
public void performanceTest() {
    int count = 10000;
    String[] keys = new String[count];
    for (int i = 0; i < count; i++) keys[i] = "key:" + i;
    
    // 1. 普通循环
    long start1 = System.currentTimeMillis();
    try (Jedis jedis = jedisPool.getResource()) {
        for (String key : keys) {
            jedis.get(key);
        }
    }
    System.out.println("普通循环: " + (System.currentTimeMillis() - start1) + "ms");
    // 结果：约 5000-10000ms（取决于网络延迟）
    
    // 2. Pipeline
    long start2 = System.currentTimeMillis();
    try (Jedis jedis = jedisPool.getResource()) {
        Pipeline pipeline = jedis.pipelined();
        for (String key : keys) {
            pipeline.get(key);
        }
        pipeline.syncAndReturnAll();
    }
    System.out.println("Pipeline: " + (System.currentTimeMillis() - start2) + "ms");
    // 结果：约 50-100ms（提升 100 倍！）
}
```

**性能提升原理**：
- 本地测试：提升不明显（RTT 本来就小）
- 远程测试：提升 10-100 倍（减少 RTT）
- 批量越大，提升越明显（但注意内存）

## 一句话收尾

Redis Pipeline 是**客户端批量发送技术**，通过减少网络 RTT 提升性能，但**不具备原子性**；批量命令是**服务端原子操作**，适合同类多键场景；需要原子性时选择 Lua 脚本或 MULTI/EXEC。
