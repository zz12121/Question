## 先说结论

UPDATE 语句在 InnoDB 内部走**"查 Buffer Pool → 改内存页 → 写 Undo Log → 写 Redo Log → 返回成功 → 异步刷脏页"**的流程；脏页通过 **Checkpoint 机制** 异步刷盘，**Sharp Checkpoint** 是关闭时全量刷盘（慢但一致），**Fuzzy Checkpoint** 是运行时增量刷盘（快但可能有脏页），InnoDB 主要用 Fuzzy + Redo Log 保证崩溃恢复。

---

## 🔥 核心概念

| 组件 | 作用 |
|------|------|
| **Buffer Pool** | 内存中的数据页缓存，减少磁盘 IO |
| **Undo Log** | 记录修改前的数据，用于回滚和 MVCC |
| **Redo Log** | 记录物理修改，用于崩溃恢复 |
| **脏页（Dirty Page）** | 内存中被修改但未刷盘的页 |
| **Checkpoint** | 将脏页刷回磁盘，推进可恢复点 |
| **LSN（Log Sequence Number）** | 日志序列号，标记日志位置 |

---

## 🔥 详细展开

### 1. UPDATE 语句的执行流程

```sql
UPDATE account SET balance = balance - 100 WHERE id = 1;
```

**执行步骤：**

```
1. 连接器 → 解析器 → 优化器 → 执行器
   └─ 生成执行计划：主键索引查找 id=1

2. 存储引擎层（InnoDB）：
   
   a. 查 Buffer Pool
      └─ 数据页在 Pool 中？直接读内存
      └─ 不在？从磁盘读入 Buffer Pool（可能触发 LRU 淘汰）
   
   b. 加锁
      └─ 对 id=1 的记录加排他锁（X Lock）
      └─ 防止其他事务并发修改
   
   c. 写 Undo Log（逻辑日志）
      └─ 记录：UPDATE account SET balance = 原值 WHERE id = 1
      └─ 用于：事务回滚、MVCC 读旧版本
   
   d. 修改内存页（脏页产生）
      └─ balance = balance - 100
      └─ 页状态变为 Dirty，加入 Flush List
   
   e. 写 Redo Log（物理日志）
      └─ 记录：页号 + 偏移量 + 修改内容
      └─ 先写 Redo Log Buffer，再刷盘（WAL 原则）
   
   f. 返回执行器
      └─ 告知"修改成功"

3. 执行器 → 返回客户端
   └─ 此时数据还在内存，磁盘仍是旧值！
```

**关键：数据修改后，磁盘和内存不一致，靠 Redo Log 保证崩溃不丢数据。**

### 2. Buffer Pool 的结构

```java
// Buffer Pool 是 InnoDB 的内存核心，默认 128MB（可配置 innodb_buffer_pool_size）

// 三大链表：
// 1. Free List：空闲页，可立即使用
// 2. LRU List：最近使用的页（热数据在前）
// 3. Flush List：脏页，等待刷盘

// 页（Page）结构：
┌─────────────────────────────────────┐
│ File Header (38 bytes)              │
│  - 页号、表空间 ID、前后页指针         │
├─────────────────────────────────────┤
│ Page Header (56 bytes)              │
│  - 记录数、空闲空间、最后插入位置       │
├─────────────────────────────────────┤
│ Infimum + Supremum Records          │
│  - 虚拟最小/最大记录                  │
├─────────────────────────────────────┤
│ User Records（实际数据行）            │
│  - id=1, balance=900 ...            │
├─────────────────────────────────────┤
│ Free Space                          │
├─────────────────────────────────────┤
│ Page Directory（页目录，加速查找）     │
├─────────────────────────────────────┤
│ File Trailer (8 bytes)              │
│  - 校验和（刷盘时写入，读取时校验）     │
└─────────────────────────────────────┘

// 修改后：
// - 页被加入 Flush List（按最早修改时间排序）
// - 页头部记录 oldest_modification LSN
```

### 3. Redo Log 的 WAL 机制

```sql
-- WAL（Write-Ahead Logging）：先写日志，再刷脏页
-- 保证：只要 Redo Log 落盘，数据就不会丢

-- Redo Log 文件（ib_logfile0, ib_logfile1）：
-- - 固定大小，循环写入
-- - 写满后覆盖最旧的日志（需确保对应脏页已刷盘）

-- LSN（Log Sequence Number）：
-- - 单调递增的 8 字节整数
-- - 标记 Redo Log 的位置
-- - 关键 LSN：
--   * checkpoint_lsn：已刷盘的脏页对应的 LSN
--   * current_lsn：当前写入的 LSN
--   * 差距 = 未刷盘的脏页对应的日志量
```

**Redo Log 刷盘时机：**

```java
// 1. 事务提交时（默认）
//    innodb_flush_log_at_trx_commit = 1
//    - 0：每秒刷盘，可能丢 1 秒数据
//    - 1：每次提交都 fsync（默认，最安全）
//    - 2：每次提交写 OS Buffer，每秒 fsync

// 2. Log Buffer 满时（默认 16MB）
//    innodb_log_buffer_size = 16777216

// 3. 每秒后台刷盘
//    即使事务未提交，也可能刷盘

// 4. Checkpoint 时
//    推进 checkpoint_lsn，释放旧日志空间
```

---

### 4. Checkpoint 机制详解

```java
// Checkpoint 目的：
// 1. 将脏页刷回磁盘，释放 Buffer Pool 空间
// 2. 推进 checkpoint_lsn，回收 Redo Log 空间
// 3. 缩短崩溃恢复时间（只需恢复 checkpoint 之后的日志）

// Checkpoint 触发时机：
// 1. 定时触发（每秒/每 10 秒）
// 2. Redo Log 快满时（超过 75%）
// 3. Buffer Pool 脏页比例过高（超过 90%）
// 4. 数据库关闭时
// 5. 手动执行（FLUSH TABLES WITH READ LOCK）
```

### 5. Sharp Checkpoint vs Fuzzy Checkpoint

| 维度 | **Sharp Checkpoint** | **Fuzzy Checkpoint** |
|------|---------------------|----------------------|
| 触发时机 | 数据库关闭 | 运行时持续进行 |
| 刷盘范围 | **所有脏页** | **部分脏页**（增量） |
| 阻塞性 | 阻塞所有 DML | 不阻塞，异步进行 |
| 耗时 | 长（脏页多可能几分钟） | 短（每次少量） |
| 一致性 | 强（所有数据一致） | 弱（部分新部分旧） |
| 使用场景 | 正常关闭、备份 | 正常运行时 |
| 崩溃恢复 | 无需恢复 | 需 Redo Log 恢复 |

```sql
-- Sharp Checkpoint 过程（正常关闭）：
-- 1. 停止接受新连接
-- 2. 等待所有事务提交或回滚
-- 3. 将 Buffer Pool 中所有脏页刷盘
-- 4. 更新 checkpoint_lsn 到最新
-- 5. 关闭文件，退出

-- Fuzzy Checkpoint 过程（运行时）：
-- 1. 从 Flush List 尾部取一批脏页（最旧的）
-- 2. 异步写入磁盘（不阻塞用户线程）
-- 3. 更新 checkpoint_lsn
-- 4. 重复直到脏页比例正常

-- 查看 Checkpoint 信息：
SHOW ENGINE INNODB STATUS;

-- Log sequence number 123456789          -- 当前 LSN
-- Log flushed up to   123456700          -- 已刷盘 LSN
-- Pages flushed up to 123456500          -- 脏页刷盘 LSN
-- Last checkpoint at  123456400          -- Checkpoint LSN
-- 差距 = 123456789 - 123456400 = 389，表示有 389 字节日志对应的脏页未刷盘
```

## 🔥 扩展追问 1：Double Write Buffer 是什么？

```java
// 问题：脏页刷盘时，如果写到一半断电，页会损坏（部分新部分旧）
// 解决：Double Write Buffer（双写机制）

// 过程：
// 1. 脏页先写入 Double Write Buffer（共享表空间中的连续区域）
// 2. Double Write Buffer 写满或定时刷盘（顺序写，快）
// 3. 再将脏页写入实际数据文件（随机写，慢）
// 4. 如果步骤 3 中断，用 Double Write Buffer 恢复

// 恢复时：
// - 检查数据页校验和
// - 如果损坏，从 Double Write Buffer 拷贝完整页覆盖

// 参数：
// innodb_doublewrite = ON（默认开启，可关闭但风险极高）
// 关闭后可用原子写（Atomic Write）替代（需硬件支持）
```

## 🔥 扩展追问 2：Change Buffer 是什么？

```java
// 场景：更新非唯一二级索引
// 问题：如果索引页不在 Buffer Pool，需要磁盘 IO 读入，很慢

// 解决：Change Buffer（写缓冲）
// - 记录二级索引的修改到 Change Buffer（也是 Buffer Pool 的一部分）
// - 等索引页被读入 Buffer Pool 时，再合并应用修改
// - 减少随机磁盘 IO

// 适用：非唯一二级索引的插入、删除、更新
// 不适用：唯一索引（需要立即检查唯一性）

// 参数：
// innodb_change_buffering = all（默认，缓冲所有操作）
// innodb_change_buffering = inserts（仅缓冲插入）
// innodb_change_buffering = none（关闭）

// 合并时机：
// 1. 索引页被读入 Buffer Pool 时
// 2. 后台线程定期合并
// 3. 数据库关闭时
// 4. Checkpoint 时
```

## 🔥 扩展追问 3：崩溃恢复过程

```sql
-- 数据库启动时的崩溃恢复：

1. 读取 Redo Log，找到 last checkpoint LSN
   └─ 从此位置开始恢复

2. 扫描 Redo Log，按 LSN 顺序重放
   └─ 页不在 Buffer Pool？读入
   └─ 应用 Redo Log 中的修改

3. 处理 Undo Log
   └─ 未提交事务：回滚（用 Undo Log 恢复旧值）
   └─ 已提交事务：无需处理（Redo 已保证持久化）

4. 恢复完成，接受连接

-- 恢复时间取决于：
-- - Checkpoint 频率（Fuzzy Checkpoint 越频繁，恢复越快）
-- - Redo Log 大小（越大，可能未刷盘的日志越多）
```

## 一句话收尾

> InnoDB 的 UPDATE 是**"内存修改 + 日志先行"**的异步架构：改 Buffer Pool 脏页立即返回，Redo Log 保证崩溃不丢，Checkpoint 异步刷盘释放空间；**Sharp Checkpoint** 是关库时全量刷盘保证一致，**Fuzzy Checkpoint** 是运行时增量刷盘保证性能，两者配合让 MySQL 既能高性能运行，又能安全恢复。

---
几个关键点再强调一下：

**UPDATE 执行的 6 步流程：**
```
1. 查 Buffer Pool（磁盘页读入内存）
2. 加排他锁（X Lock）
3. 写 Undo Log（回滚用）
4. 改内存页（产生脏页）
5. 写 Redo Log（WAL 保证持久化）
6. 返回成功（此时磁盘还是旧数据！）
```

**Sharp vs Fuzzy Checkpoint 对比：**
| | Sharp | Fuzzy |
|--|-------|-------|
| 触发 | 数据库关闭 | 运行时持续 |
| 范围 | 所有脏页 | 部分脏页 |
| 阻塞 | 是 | 否 |
| 恢复 | 无需恢复 | 需 Redo Log |

**LSN（Log Sequence Number）的三个关键值：**
- `current_lsn`：当前写入位置
- `flushed_lsn`：已刷盘位置  
- `checkpoint_lsn`：已刷盘的脏页对应的日志位置

**崩溃恢复只需恢复 `checkpoint_lsn` 之后的日志**，所以 Checkpoint 越频繁，恢复越快，但刷盘开销越大。
