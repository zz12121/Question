## 先说结论

`execute()` 是"发射后不管"，`submit()` 把任务包进 `FutureTask` 可以拿结果、捕异常；`Future.get()` 会把任务异常**包装成 `ExecutionException` 抛出**，必须 `.getCause()` 才能拿到原始异常；线程命名用 `ThreadFactory` 自定义，是生产排查的标配。

---

## 🔥 核心概念

| 维度 | `execute()` | `submit()` |
|------|------------|------------|
| 参数 | `Runnable` | `Runnable` / `Callable<T>` |
| 返回值 | `void` | `Future<T>` |
| 异常处理 | 抛给 `UncaughtExceptionHandler`，线程终止 | 异常被捕获存入 `Future`，`get()` 时才抛出 |
| 任务结果 | 无法获取 | 可通过 `Future.get()` 获取 |
| 底层实现 | 直接运行 `Runnable` | 包装成 `FutureTask` 再调 `execute()` |
| 线程是否终止 | 异常后线程销毁，新建替补 | 线程不终止，异常存在 Future 里 |

## 🔥 详细展开

### 1. execute() —— 异常直接暴露，线程会死

```java
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    2, 4, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(10),
    r -> new Thread(r, "worker-thread"), // 先用简单命名
    new ThreadPoolExecutor.AbortPolicy()
);

// execute 提交任务
pool.execute(() -> {
    System.out.println("执行任务: " + Thread.currentThread().getName());
    throw new RuntimeException("任务执行失败！"); // 异常直接抛出
    // ↑ 这个异常会：
    // 1. 触发 UncaughtExceptionHandler（如果设置了的话）
    // 2. 导致当前线程终止
    // 3. 线程池自动创建新线程替补
});

// 结果：任务异常 → 调用方完全不知道 → 默认打印到 stderr，不被日志系统捕获
// 生产环境极其危险：任务静默失败
```

### 2. submit() —— 异常被装进 Future，get() 时才爆

```java
// 关键：submit 内部把 Runnable/Callable 包成 FutureTask
// FutureTask.run() 内部 try-catch 了所有异常，存入 outcome 字段

Future<String> future = pool.submit(() -> {
    System.out.println("执行: " + Thread.currentThread().getName());
    if (someCondition) {
        throw new IllegalArgumentException("非法参数");
    }
    return "任务结果";
});

// 异常在这里才真正暴露！
try {
    String result = future.get(); // 阻塞等待结果
} catch (ExecutionException e) {
    // 原始异常被包装进 ExecutionException
    Throwable cause = e.getCause(); // ← 必须 getCause() 才能拿到真实异常
    log.error("任务执行失败，原始异常: {}", cause.getMessage(), cause);
} catch (InterruptedException e) {
    Thread.currentThread().interrupt(); // 恢复中断标志
    log.warn("等待任务结果被中断");
} catch (CancellationException e) {
    log.warn("任务已被取消");
}
```

### 3. FutureTask 内部原理（为什么异常不立即抛）

```java
// FutureTask.run() 伪代码（JDK 源码简化版）
public void run() {
    try {
        Callable<V> c = callable;
        V result = c.call();          // 执行任务
        set(result);                   // 正常：保存结果到 outcome
    } catch (Throwable ex) {
        setException(ex);              // 异常：保存异常到 outcome，不向上抛！
    }
}

// FutureTask.get() 伪代码
public V get() throws InterruptedException, ExecutionException {
    // 阻塞等待任务完成...
    Object x = outcome;
    if (x instanceof AltResult) {      // 是异常结果
        Throwable cause = ((AltResult) x).ex;
        throw new ExecutionException(cause); // 包装成 ExecutionException 抛出
    }
    return (V) x;                      // 是正常结果直接返回
}

// 所以：submit() 提交的任务无论成功失败，线程都不会终止
// 异常被"冷冻"在 Future 里，等 get() 时"解冻"
```

### 4. 自定义线程命名 —— 生产排查必备

```java
// 方案1：实现 ThreadFactory 接口（最灵活）
public class NamedThreadFactory implements ThreadFactory {
    private final String namePrefix;
    private final AtomicInteger threadNumber = new AtomicInteger(1);
    private final boolean daemon;

    public NamedThreadFactory(String namePrefix) {
        this(namePrefix, false);
    }

    public NamedThreadFactory(String namePrefix, boolean daemon) {
        this.namePrefix = namePrefix;
        this.daemon = daemon;
    }

    @Override
    public Thread newThread(Runnable r) {
        Thread t = new Thread(r, namePrefix + "-" + threadNumber.getAndIncrement());
        t.setDaemon(daemon);
        t.setPriority(Thread.NORM_PRIORITY);
        // 设置全局异常处理器，execute() 的异常也能被捕获
        t.setUncaughtExceptionHandler((thread, ex) ->
            log.error("线程 [{}] 未捕获异常", thread.getName(), ex)
        );
        return t;
    }
}

// 使用
ThreadPoolExecutor orderPool = new ThreadPoolExecutor(
    8, 16, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000),
    new NamedThreadFactory("order-process"), // ← 线程名：order-process-1, 2, 3...
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

### 6. 批量任务的完整最佳实践

```java
// 场景：批量调用下游接口，部分失败要知道哪些失败了
@Service
public class BatchService {

    @Resource(name = "ioTaskExecutor")
    private ThreadPoolExecutor executor;

    public BatchResult batchProcess(List<String> ids) {
        List<Future<ProcessResult>> futures = ids.stream()
            .map(id -> executor.submit(() -> processOne(id)))
            .collect(Collectors.toList());

        List<String> successIds = new ArrayList<>();
        List<String> failedIds  = new ArrayList<>();

        for (int i = 0; i < futures.size(); i++) {
            String id = ids.get(i);
            try {
                ProcessResult result = futures.get(i).get(5, TimeUnit.SECONDS); // 设超时
                successIds.add(id);
            } catch (ExecutionException e) {
                log.error("任务[{}]执行失败: {}", id, e.getCause().getMessage());
                failedIds.add(id);                // 记录失败，不影响其他
            } catch (TimeoutException e) {
                log.warn("任务[{}]执行超时，取消任务", id);
                futures.get(i).cancel(true);       // 超时取消
                failedIds.add(id);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                break;                             // 中断则退出循环
            }
        }

        return new BatchResult(successIds, failedIds);
    }

    // 更简洁的写法：CompletableFuture（推荐）
    public CompletableFuture<BatchResult> batchProcessAsync(List<String> ids) {
        List<CompletableFuture<ProcessResult>> futures = ids.stream()
            .map(id -> CompletableFuture
                .supplyAsync(() -> processOne(id), executor)
                .exceptionally(ex -> {
                    log.error("任务[{}]失败: {}", id, ex.getMessage());
                    return ProcessResult.failed(id); // 失败返回默认值，不中断整体
                }))
            .collect(Collectors.toList());

        return CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]))
            .thenApply(v -> aggregateResults(futures));
    }
}
```

## 🔥 扩展追问 1：execute() 里的异常怎么捕获？

```java
// 方法1：任务内部 try-catch（最直接）
pool.execute(() -> {
    try {
        doWork();
    } catch (Exception e) {
        log.error("任务执行失败", e);
        // 可以发告警、写重试队列等
    }
});

// 方法2：UncaughtExceptionHandler（兜底）
threadFactory.setUncaughtExceptionHandler((thread, ex) -> {
    log.error("线程[{}]未捕获异常", thread.getName(), ex);
    alertService.sendAlert("线程池异常: " + ex.getMessage());
});

// 方法3：重写 ThreadPoolExecutor.afterExecute（最强，能拦截所有）
ThreadPoolExecutor pool = new ThreadPoolExecutor(...) {
    @Override
    protected void afterExecute(Runnable r, Throwable t) {
        super.afterExecute(r, t);
        // execute() 的异常直接在 t 里
        if (t != null) {
            log.error("execute 任务异常", t);
        }
        // submit() 的异常需要从 FutureTask 里取
        if (t == null && r instanceof Future<?>) {
            try {
                ((Future<?>) r).get(0, TimeUnit.NANOSECONDS); // 非阻塞取结果
            } catch (ExecutionException ee) {
                log.error("submit 任务异常", ee.getCause()); // ← 这里捕获
            } catch (TimeoutException | InterruptedException ignore) {}
        }
    }
};
```

## 🔥 扩展追问 2：Future.get() 的超时和取消

```java
// 设置超时，避免无限等待
try {
    String result = future.get(3, TimeUnit.SECONDS); // 最多等 3 秒
} catch (TimeoutException e) {
    future.cancel(true);  // 取消任务（true=中断正在执行的线程）
    log.warn("任务超时，已取消");
    return fallbackResult; // 降级返回默认值
}

// cancel(true) vs cancel(false)
// true：向任务线程发送中断信号（任务需要响应 interrupt 才能真正停止）
// false：仅标记取消，如果任务已在运行则等它跑完

// 判断任务状态
future.isDone();      // 完成（正常/异常/取消 都算）
future.isCancelled(); // 是否被取消
```

## 一句话收尾

> `execute()` 适合"发出去不管结果"的场景，异常要靠 `UncaughtExceptionHandler` 兜底；`submit()` 适合"要结果要感知异常"的场景，记住 `get()` 拿到的是 `ExecutionException`，真正的病因在 `getCause()` 里；线程命名是排查问题的眼睛，**`pool-1-thread-3` 是盲盒，`order-process-3` 才是线索**。

---
几个关键点再强调一下：

**execute() vs submit() 的本质区别：**
- `execute()` → 直接扔给线程池跑，异常直接抛，**线程会死**
- `submit()` → 包装成 `FutureTask`，异常被 `try-catch` 存起来，**线程不死**，等 `get()` 时才抛

**Future.get() 异常处理的标准写法：**
```java
try {
    result = future.get();
} catch (ExecutionException e) {
    // 必须 getCause() 才能拿到真实异常
    log.error("失败", e.getCause()); 
}
```

**线程命名生产必做：**
```java
// 一眼能看出业务归属，排查时 jstack 直接定位
new ThreadFactoryBuilder().setNameFormat("payment-worker-%d").build()
// 输出：payment-worker-1, payment-worker-2...
```
