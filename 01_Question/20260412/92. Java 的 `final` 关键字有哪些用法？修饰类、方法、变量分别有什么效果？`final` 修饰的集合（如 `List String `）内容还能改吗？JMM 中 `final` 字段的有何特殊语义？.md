## 2. 核心代码示例

### 2.1 final 修饰类

```java
// ========== 禁止继承 ==========
public final class String {
    // String 类被 final 修饰，不能被继承
    // 这样保证 String 的行为是确定的，不会被恶意篡改
}

// ❌ 编译报错：cannot inherit from final class String
class MyString extends String { }

// ========== 为什么 String 要 final？ ==========
// 场景：如果可以继承 String
class EvilString extends String {
    @Override
    public char charAt(int index) {
        // 恶意篡改：返回错误的字符
        return 'X';
    }
}

// 任何依赖 String.charAt() 的代码都会被破坏
// 登录校验、加密解密、日志系统...全部会出问题
```

### 2.2 final 修饰方法

```java
// ========== 父类 ==========
public class Parent {
    
    // final 方法：子类不能重写
    public final void finalMethod() {
        System.out.println("final 方法，不能被重写");
    }
    
    // 普通方法：子类可以重写
    public void normalMethod() {
        System.out.println("普通方法，可以被重写");
    }
    
    // private 方法：隐式 final
    private final void privateMethod() {
        // private 方法等价于隐式 final，子类看不到，所以无法重写
    }
}

// ========== 子类 ==========
public class Child extends Parent {
    
    // ❌ 编译报错：cannot override the final method
    @Override
    public final void finalMethod() { }
    
    // ✅ 可以重写
    @Override
    public void normalMethod() { }
}

// ========== 为什么要用 final 方法？ ==========
// 模板方法模式：父类定义算法骨架，某些步骤由子类实现
public abstract class AbstractService {
    
    // final：模板步骤不能被修改
    public final void execute() {
        before();
        doProcess();    // 子类实现
        after();
    }
    
    protected abstract void doProcess();
    
    private void before() { System.out.println("前置处理"); }
    private void after() { System.out.println("后置处理"); }
}
```

### 2.3 final 修饰变量

```java
// ========== 基本类型：值不可变 ==========
public class FinalBasic {
    
    final int a = 1;           // ✅ 正确：定义时赋值
    final int b;               // ✅ 正确：空白 final，必须在构造函数中赋值
    
    public FinalBasic(int b) {
        this.b = b;            // 构造函数中赋值一次
    }
    
    public void test() {
        // ❌ 编译报错：cannot assign a value to final variable a
        a = 2;
        // ❌ 编译报错：cannot assign a value to final variable b
        b = 3;
    }
}

// ========== 引用类型：引用不可变 ==========
public class FinalReference {
    
    final StringBuilder sb = new StringBuilder("hello");
    
    public void test() {
        // ✅ 可以：修改引用类型的内容
        sb.append(" world");        // 内容可以改
        sb.setLength(0);            // 内容可以改
        
        // ❌ 编译报错：cannot assign a value to final variable sb
        sb = new StringBuilder("new");  // 引用不能改
    }
}

// ========== static final：常量 ==========
public class Constants {
    
    // static final：编译期常量（内联优化）
    public static final int MAX_RETRY = 3;
    
    // 只有 final 而没有 static：每实例一份
    public final int PAGE_SIZE = 20;
    
    // blank final + 构造函数赋值
    public final int timeout;
    public Constants(int timeout) {
        this.timeout = timeout;
    }
}
```

### 2.4 final 修饰集合（重点）

```java
// ========== final + Collections.unmodifiableList ==========
public class ImmutableListDemo {
    
    // ❌ 常见误区：以为加了 final 就能防修改
    public final List<String> list1 = new ArrayList<>();
    list1.add("a");  // ✅ 可以！final 只锁引用，不锁内容
    
    // ✅ 正确做法：创建真正不可变的列表
    public final List<String> list2 = List.of("a", "b", "c");
    list2.add("d");  // ❌ 运行时抛 UnsupportedOperationException
    
    // ✅ 或者手动包装
    public final List<String> list3 = Collections.unmodifiableList(new ArrayList<>());
}

// ========== 项目中的正确用法 ==========
public class UserService {
    
    // ✅ 推荐：final + Collections.unmodifiableList
    private final List<String> SUPPORTED_ROLES = List.of("ADMIN", "USER", "GUEST");
    
    // ✅ 推荐：copy 模式（防御性复制）
    private final List<String> cache = new ArrayList<>();
    
    public List<String> getCache() {
        // 返回副本，防止外部修改
        return new ArrayList<>(cache);
    }
    
    public void addToCache(String item) {
        // 内部可以修改
        cache.add(item);
    }
}
```

### 2.5 JMM 中 final 的特殊语义

```java
// ========== 问题：普通字段在多线程下可能看到默认值 ==========
public class FinalFieldDemo {
    
    // 普通字段：构造函数赋值后，其他线程可能看到默认值 0
    String normalField;
    
    // final 字段：构造函数完成后，对其他线程【安全可见】
    final String safeField;
    
    public FinalFieldDemo() {
        this.safeField = "constructed";  // 构造函数中赋值
        this.normalField = "constructed";
    }
}

// ========== 安全发布模式 ==========
public class SafePublication {
    
    // 方式一：final 字段 + 构造函数完成后才发布对象
    public final String data;
    
    public SafePublication() {
        this.data = "important data";
        // 构造函数完成后，data 对其他线程可见
    }
    
    // 方式二：static 初始化（类加载时完成，安全发布）
    public static final SafePublication INSTANCE = new SafePublication();
    
    // 方式三：volatile / AtomicReference
    private static final AtomicReference<SafePublication> INSTANCE2 = 
        new AtomicReference<>(new SafePublication());
}

// ========== 不安全发布会导致的问题 ==========
public class UnsafeDemo {
    
    private final String value;
    
    public UnsafeDemo() {
        this.value = "init";
        // 问题：如果对象引用被其他线程看到时，构造函数还没执行完
        // 那个线程可能看到 value = null（默认值）
    }
    
    // ✅ 解决方案：使用 final + 正确的发布方式
    // JMM 保证：只要构造函数没有 this 逃逸，final 字段就是安全的
}
```

## 🔥 扩展追问 1：为什么 String 要设计成 final？

> **A:** 三个原因：安全、性能、不可变对象的好处。

```java
// 1. 安全：防止恶意篡改
// 如果 String 可以继承，攻击者可以重写 charAt() 篡改返回结果
// 银行系统、登录校验、加密解密全部依赖 String 的确定性行为

// 2. 性能：字符串常量池
String s1 = "hello";       // 放在常量池
String s2 = "hello";       // 复用 s1 的引用
System.out.println(s1 == s2);  // true

// 3. 不可变对象的好处
// - 天然线程安全，不需要同步
// - hashCode 缓存（String 的 hashCode 只计算一次）
// - 可以安全作为 Map 的 key
// - 安全性：用户名、密码、URL、文件路径...这些都不应该被修改
```

## 🔥 扩展追问 2：final 能解决什么问题？解决不了什么？

> **A:** final 能解决的是"意外修改"，但防不住所有问题。

| 场景 | final 能解决 | 说明 |
|------|-------------|------|
| 变量重新赋值 | ✅ | `a = 2` 编译报错 |
| 子类重写方法 | ✅ | `@Override` 编译报错 |
| 类被继承 | ✅ | `extends` 编译报错 |
| 集合内容修改 | ❌ | `list.add()` 仍然可以 |
| 反射修改字段 | ❌ | `field.setAccessible(true)` 可以绕过 |
| 对象状态修改 | ❌ | `sb.append()` 可以 |

**真正不可变的方式：**
```java
// 不可变对象模式
public final class ImmutableUser {
    
    private final String name;      // final 字段
    private final int age;
    private final List<String> tags;  // 引用也 final
    
    public ImmutableUser(String name, int age, List<String> tags) {
        this.name = name;           // 构造函数中赋值
        this.age = age;
        // 防御性复制：防止外部修改
        this.tags = Collections.unmodifiableList(new ArrayList<>(tags));
    }
    
    // 没有 setter，所有字段只读
}
```

## 🔥 扩展追问 3：内部类访问局部变量为什么要 final？

> **A:** 这是 Java 的闭包机制决定的。

```java
public class Outer {
    
    public Runnable createRunnable(final int count) {
        // ✅ 局部变量加 final（Java 7 之前必须加，Java 8 可以不加但仍是隐式 final）
        
        return new Runnable() {
            @Override
            public void run() {
                // 内部类引用外部的局部变量
                // Java 会在创建 Runnable 时，把 count 的值拷贝一份
                System.out.println(count);
            }
        };
    }
    
    public Runnable createRunnableJava8(int count) {
        // Java 8+: 可以不加 final，但仍是 effectively final
        return () -> System.out.println(count);
    }
}
```

**原理：**
```
内部类为什么需要 copy？
├── 局部变量在栈上，方法结束后就销毁
├── 内部类的生命周期可能超过方法（被返回、异步执行）
└── 所以要把变量 copy 一份到堆上（作为内部类对象的字段）

为什么需要 final？
├── copy 之后，如果原变量还能改，就造成数据不一致
├── final 保证 copy 和原变量值一致
└── Java 8 的 effectively final 只是编译器帮你加 final
```

## 🔥 扩展追问 4：try-with-resources 的变量为什么要 final？

> **A:** 同样是闭包问题，资源必须有效。

```java
// ❌ 编译报错：variable resource in try-with-resources must be final or effectively final
public void demo() {
    BufferedReader reader = new BufferedReader(new FileReader("test.txt"));
    reader = new BufferedReader(...);  // 重新赋值了
    
    try (reader) {  // 这里 reader 不是 final
        // 编译器不知道该 close 哪个
    }
}

// ✅ 正确写法
public void demo() {
    final BufferedReader reader = new BufferedReader(new FileReader("test.txt"));
    try (reader) {
        // 编译器保证只用这一个 reader
    }
}

// ✅ 或者用另一种方式（不重新赋值）
public void demo() {
    try (BufferedReader r1 = new BufferedReader(...);
         BufferedReader r2 = new BufferedReader(...)) {
        // 两个资源
    }
}
```

## 🔥 扩展追问 5：final 和 JMM 的 happens-before 关系？

> **A:** final 字段有特殊的内存语义。

```
普通字段的写入：
Thread A                      Thread B
─────────────────────────────────────────────
obj.x = 1;     ──────────────▶  System.out.println(obj.x);
                                       │
                                       │ 可能看到 0（默认值）！
                                       │ 因为没有 happens-before 保证

final 字段的写入：
Thread A                      Thread B
─────────────────────────────────────────────
构造函数完成         happens-before    读取对象引用
      │                          │
      ▼                          ▼
obj.finalX = 1;    ──────────────▶  System.out.println(obj.finalX);
                                       │
                                       │ 一定看到 1！
                                       │ JMM 保证 final 字段的写入在构造函数之前
                                       │ 对其他线程可见
```

**前提条件：**
```java
// ✅ 正确使用 final 安全发布的场景
public class SafeClass {
    public final int x;
    public SafeClass() {
        this.x = 1;
    }
}

// ❌ 错误：this 逃逸（final 语义失效）
public class UnsafeClass {
    public final int x;
    
    public UnsafeClass() {
        this.x = 1;
        // ❌ 致命错误：在构造函数中把 this 传出去
        new Thread(() -> System.out.println(this.x)).start();
    }
}
```

## 一句话总结

> **final 修饰类禁止继承保证安全，修饰方法禁止重写保证模板方法语义，修饰变量禁止重新赋值保证状态稳定，但修饰引用类型只锁引用不锁内容（需配合 Collections.unmodifiableList），final 字段在 JMM 下有特殊语义——构造函数完成后对其他线程安全可见，前提是没有 this 逃逸。**

---

**面试加分话术：**

> "之前项目里遇到过一个大坑：队友写了个缓存类，用 final List 存数据，以为加了 final 就不可变了，结果线上有人直接调 list.clear() 把整个缓存清空。后来改成 `List.of()` 创建真正不可变的列表，彻底杜绝了这种误操作。另外关于 JMM 的 final 语义，有个经典问题——如果对象在构造函数里还没完成初始化就把 this 传给其他线程（比如启动新线程），即使字段都加了 final，其他线程也可能看到默认值。这叫 this 逃逸，是严重的安全问题，正确做法是等构造函数执行完再发布对象。"
