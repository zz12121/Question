## 先说结论

`ScheduledThreadPoolExecutor` 是**线程池版定时任务**，支持多线程并发执行、异常隔离、任务取消；`Timer` 是单线程串行执行，一个任务抛异常或耗时过长会拖垮整个定时器，生产环境**绝对不能用 `Timer`**。

---

## 🔥 核心概念

| 维度 | `Timer` | `ScheduledThreadPoolExecutor` |
|------|---------|------------------------------|
| 线程模型 | **单线程** | **线程池**（可配置） |
| 任务执行 | 串行排队 | 并发执行（多任务） |
| 异常处理 | 一个任务抛异常，**整个 Timer 挂掉** | 异常只影响当前任务，其他继续 |
| 任务取消 | `TimerTask.cancel()` | `Future.cancel()` |
| 精度 | 一般 | 更高（纳秒级调度） |
| 线程安全 | 无 | 有 |
| 生产可用性 | ❌ **禁用** | ✅ **推荐** |

## 🔥 详细展开

### 1. Timer 的致命缺陷（生产禁用）

```java
// Timer 是 JDK 1.3 的老古董，设计有根本缺陷
Timer timer = new Timer("my-timer");

// 致命问题1：单线程串行
timer.schedule(new TimerTask() {
    @Override
    public void run() {
        System.out.println("任务1开始: " + new Date());
        try {
            Thread.sleep(5000); // 模拟耗时5秒
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        System.out.println("任务1结束");
    }
}, 0, 1000); // 期望每秒执行

timer.schedule(new TimerTask() {
    @Override
    public void run() {
        System.out.println("任务2: " + new Date()); // ← 永远执行不到！
    }
}, 0, 1000);

// 输出：只有任务1在执行，任务2被阻塞
// 因为 Timer 只有一个线程，任务1占着，任务2只能等

// 致命问题2：异常导致整个 Timer 崩溃
timer.schedule(new TimerTask() {
    @Override
    public void run() {
        throw new RuntimeException("出错了！"); // ← Timer 线程终止！
    }
}, 0);

// 后续所有任务都不会再执行，且没有任何通知！
// 系统"静默死亡"，监控都发现不了
```

### 2. ScheduledThreadPoolExecutor —— 生产标准方案

```java
// 创建：指定核心线程数（通常 2-5 就够了，定时任务不会太多）
ScheduledThreadPoolExecutor scheduler = new ScheduledThreadPoolExecutor(
    3, // 核心线程数
    new ThreadFactoryBuilder()
        .setNameFormat("scheduled-task-%d")
        .setUncaughtExceptionHandler((t, e) -> 
            log.error("定时任务异常 [{}]", t.getName(), e))
        .build(),
    new ThreadPoolExecutor.DiscardPolicy() // 满了丢弃，定时任务一般不阻塞
);

// 特性1：多线程并发执行，互不阻塞
scheduler.scheduleAtFixedRate(() -> {
    System.out.println("任务A: " + Thread.currentThread().getName());
    try { Thread.sleep(3000); } catch (InterruptedException e) {}
}, 0, 1, TimeUnit.SECONDS);

scheduler.scheduleAtFixedRate(() -> {
    System.out.println("任务B: " + Thread.currentThread().getName()); // 能执行！
}, 0, 1, TimeUnit.SECONDS);

// 输出：任务A和任务B在不同线程并发执行
// scheduled-task-0 执行 任务A
// scheduled-task-1 执行 任务B

// 特性2：异常隔离，一个任务崩不影响其他
scheduler.scheduleAtFixedRate(() -> {
    throw new RuntimeException("任务C异常");
}, 0, 1, TimeUnit.SECONDS);

// 任务C抛异常，但调度器继续运行，任务A和B不受影响
// 异常会被捕获并记录，不会终止调度线程
```

### 3. 四种调度方式对比

```java
ScheduledThreadPoolExecutor scheduler = new ScheduledThreadPoolExecutor(2);

// 1. schedule() —— 延迟执行一次
ScheduledFuture<?> future = scheduler.schedule(() -> {
    System.out.println("3秒后执行一次");
}, 3, TimeUnit.SECONDS);

// 2. scheduleAtFixedRate() —— 固定频率（以开始时间为基准）
// 任务开始时间间隔固定，不管执行多久
scheduler.scheduleAtFixedRate(() -> {
    System.out.println("每2秒执行，不管上次跑多久");
    try { Thread.sleep(5000); } // 即使执行5秒，下次2秒后还是触发（并发执行）
} catch (Exception e) {}
}, 0, 2, TimeUnit.SECONDS);
// ⚠️ 注意：如果任务执行时间 > 间隔，会并发执行多个实例！

// 3. scheduleWithFixedDelay() —— 固定延迟（以结束时间为基准）
// 上次结束 + 延迟 = 下次开始
scheduler.scheduleWithFixedDelay(() -> {
    System.out.println("上次结束后等3秒再执行");
    try { Thread.sleep(2000); } // 执行2秒 + 等待3秒 = 5秒周期
} catch (Exception e) {}
}, 0, 3, TimeUnit.SECONDS);

// 4. 组合使用（更灵活的场景）
// 比如：每天凌晨2点执行
long delay = calculateDelayTo2AM(); // 计算到下次凌晨2点的毫秒数
scheduler.schedule(() -> {
    doDailyJob();
    // 递归调度下一天
    scheduler.schedule(this, 1, TimeUnit.DAYS); 
}, delay, TimeUnit.MILLISECONDS);
```

---

### 4. 固定频率 vs 固定延迟的选择

```
scheduleAtFixedRate（固定频率）：
时间轴：|----任务1(2s)----|----任务2(2s)----|----任务3----|
        0s              2s              4s
适用：心跳检测、定时上报（必须按固定间隔）

scheduleWithFixedDelay（固定延迟）：
时间轴：|----任务1(2s)----|等待3s|----任务2(2s)----|等待3s|----任务3----|
        0s              2s     5s              7s
适用：数据同步、批量处理（任务执行完休息固定时间再开始）

⚠️ 重要：如果任务执行时间 > 固定频率间隔
- scheduleAtFixedRate：任务会并发执行（可能堆积）
- scheduleWithFixedDelay：不会并发，上次结束才开始算延迟
```

### 5. 任务取消和优雅关闭

```java
// 取消单个任务
ScheduledFuture<?> future = scheduler.scheduleAtFixedRate(() -> {
    // 任务逻辑
}, 0, 10, TimeUnit.SECONDS);

// 取消任务（true=中断正在执行的线程）
future.cancel(true); // 或 false 仅标记取消

// 检查状态
future.isCancelled(); // 是否已取消
future.isDone();      // 是否已完成（正常结束/异常/取消都算）

// 优雅关闭整个调度器
public void gracefulShutdown() {
    scheduler.shutdown(); // 停止接收新任务
    try {
        // 等待已有任务完成（最多等60秒）
        if (!scheduler.awaitTermination(60, TimeUnit.SECONDS)) {
            scheduler.shutdownNow(); // 强制中断
        }
    } catch (InterruptedException e) {
        scheduler.shutdownNow();
        Thread.currentThread().interrupt();
    }
}
```

## 🔥 扩展追问 1：Spring 的 @Scheduled 底层用的是什么？

```java
// Spring @Scheduled 默认使用 ThreadPoolTaskScheduler
// 底层就是 ScheduledThreadPoolExecutor 的包装

@Configuration
@EnableScheduling
public class ScheduleConfig implements SchedulingConfigurer {
    
    @Override
    public void configureTasks(ScheduledTaskRegistrar taskRegistrar) {
        // 自定义线程池（默认只有1个线程！）
        ThreadPoolTaskScheduler scheduler = new ThreadPoolTaskScheduler();
        scheduler.setPoolSize(10);              // 多个 @Scheduled 方法并发执行
        scheduler.setThreadNamePrefix("spring-scheduled-");
        scheduler.setAwaitTerminationSeconds(60);
        scheduler.setWaitForTasksToCompleteOnShutdown(true);
        scheduler.initialize();
        
        taskRegistrar.setTaskScheduler(scheduler);
    }
}

// 使用
@Component
public class MyJobs {
    
    @Scheduled(fixedRate = 5000)  // 固定频率
    public void job1() { /* ... */ }
    
    @Scheduled(fixedDelay = 10000) // 固定延迟
    public void job2() { /* ... */ }
    
    @Scheduled(cron = "0 0 2 * * ?") // Cron表达式
    public void dailyJob() { /* ... */ }
}
```

## 🔥 扩展追问 3：定时任务执行时间超过间隔怎么办？

```java
// 问题：任务执行5秒，但设置每秒执行一次

// scheduleAtFixedRate 的问题：并发堆积
scheduler.scheduleAtFixedRate(() -> {
    System.out.println("开始: " + System.currentTimeMillis());
    try { Thread.sleep(5000); } catch (InterruptedException e) {}
}, 0, 1, TimeUnit.SECONDS);
// 结果：任务并发执行，线程池耗尽，OOM

// 解决方案1：用 scheduleWithFixedDelay（推荐简单场景）
scheduler.scheduleWithFixedDelay(() -> {
    // 上次结束 + 1秒延迟 = 下次开始
}, 0, 1, TimeUnit.SECONDS);

// 解决方案2：任务内部判断，跳过本次
AtomicBoolean running = new AtomicBoolean(false);
scheduler.scheduleAtFixedRate(() -> {
    if (!running.compareAndSet(false, true)) {
        log.warn("上次任务还在执行，跳过本次");
        return;
    }
    try {
        doWork();
    } finally {
        running.set(false);
    }
}, 0, 1, TimeUnit.SECONDS);

// 解决方案3：使用 Resilience4j 的 RateLimiter / 信号量限流
Semaphore semaphore = new Semaphore(1);
scheduler.scheduleAtFixedRate(() -> {
    if (!semaphore.tryAcquire()) {
        log.warn("任务堆积，跳过");
        return;
    }
    try {
        doWork();
    } finally {
        semaphore.release();
    }
}, 0, 1, TimeUnit.SECONDS);
```

## 一句话收尾

> `Timer` 是单线程定时炸弹，一个异常全炸光，生产环境**绝对禁用**；`ScheduledThreadPoolExecutor` 是多线程并发调度，异常隔离、任务可控，是 Java 定时任务的**唯一标准答案**；简单场景直接用，复杂分布式场景上 Quartz/XXL-JOB。

---
几个关键点再强调一下：

**Timer 为什么生产禁用（面试必答）：**
1. **单线程** → 一个任务耗时，其他任务全阻塞
2. **异常传染** → 一个任务抛异常，整个 Timer 线程终止，后续任务全挂，且**静默无通知**

**固定频率 vs 固定延迟的选择：**
- `scheduleAtFixedRate` → 以**开始时间**为基准，任务执行慢会**并发堆积**
- `scheduleWithFixedDelay` → 以**结束时间**为基准，天然防并发，上次跑完再等固定时间

**防止任务堆积的三种方案：**
1. 用 `scheduleWithFixedDelay` 替代（简单场景）
2. 任务内部用 `AtomicBoolean` 或 `Semaphore` 做执行中标记
3. 分布式场景用 Quartz / XXL-JOB 的集群锁
