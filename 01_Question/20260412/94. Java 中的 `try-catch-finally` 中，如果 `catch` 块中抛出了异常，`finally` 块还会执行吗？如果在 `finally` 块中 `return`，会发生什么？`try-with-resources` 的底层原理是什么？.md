## 先说结论

`finally` **一定执行**（除非 JVM 崩溃或线程被强杀）；`finally` 中 `return` 会**吞掉 catch 里的异常**；`try-with-resources` 底层是编译器糖，自动插入 `addSuppressed` 链式关闭逻辑。

---

## 🔥 核心概念

| 场景 | 结论 |
|------|------|
| catch 抛出异常 → finally | **finally 仍然执行**，执行完后继续向上抛 catch 的异常 |
| finally 中 return | **吞掉所有异常**，方法直接返回 finally 的值 |
| finally 中抛出新异常 | **原始异常丢失**，只传播 finally 的异常 |
| try-with-resources | 编译器生成 `addSuppressed`，资源关闭异常不会丢失 |
| finally 不执行的情况 | `System.exit()` / JVM crash / 死循环/线程被 kill |

## 🔥 详细展开

### 1. catch 抛异常后，finally 还执行吗？

```java
public static void demo1() {
    try {
        System.out.println("try");
        throw new RuntimeException("try异常");
    } catch (RuntimeException e) {
        System.out.println("catch: " + e.getMessage());
        throw new IllegalStateException("catch里抛的新异常"); // ← 抛出新异常
    } finally {
        System.out.println("finally: 我一定执行！"); // ← 仍然执行！
    }
}

// 输出：
// try
// catch: try异常
// finally: 我一定执行！
// Exception: IllegalStateException: catch里抛的新异常  ← catch的异常向上传播
```

**结论：finally 必执行，执行完后传播 catch 里的新异常（原始 try 异常已被 catch 处理，不再传播）。**

### 2. finally 中 return 会怎样？⚠️ 危险行为

```java
public static int dangerousReturn() {
    try {
        System.out.println("try");
        return 1;                          // ← 准备返回 1
    } catch (Exception e) {
        System.out.println("catch");
        throw new RuntimeException("catch异常"); // ← 准备抛异常
    } finally {
        System.out.println("finally");
        return 999; // ← 直接覆盖 return 1，并吞掉 catch 异常！！
    }
}

// 输出：
// try
// finally
// 返回值：999  ← try 的 return 1 被覆盖
// 异常被完全吞掉，调用方感知不到任何问题！！
```

```java
// 更危险的场景：异常被静默吞掉
public static String silentBug() {
    try {
        // 假设这里发生了 NPE
        String s = null;
        return s.toUpperCase(); // NullPointerException
    } catch (NullPointerException e) {
        throw new RuntimeException("NPE发生了！", e);
    } finally {
        return "假装成功"; // ← 吞掉了 RuntimeException，返回正常值
        // 调用方以为一切OK，实际上业务逻辑完全没跑
    }
}
```

> ⚠️ **生产禁忌：finally 里永远不要写 return / throw**，这是 CheckStyle 和 SonarQube 的默认警告规则。

### 3. finally 中修改返回值（基本类型 vs 引用类型）

```java
// 基本类型：finally 改不了 try 里已经 return 的值
public static int primitiveTest() {
    int result = 0;
    try {
        result = 1;
        return result;   // 此时 result=1 已被压栈保存
    } finally {
        result = 999;    // 改了局部变量，但不影响已压栈的返回值
        // 注意：这里没有 return，所以不会覆盖
    }
    // 返回 1，不是 999
}

// 引用类型：finally 可以修改对象内容（但不能改引用本身）
public static List<String> referenceTest() {
    List<String> list = new ArrayList<>();
    try {
        list.add("try");
        return list;      // 返回的是引用（地址）
    } finally {
        list.add("finally"); // 通过引用修改对象内容，调用方能看到！
        // list = new ArrayList<>(); ← 这样改引用没用
    }
    // 返回的 list 包含 ["try", "finally"]
}
```

### 4. try-with-resources 底层原理

**语法糖形式：**
```java
// 你写的代码
try (InputStream is = new FileInputStream("file.txt");
     BufferedReader br = new BufferedReader(new InputStreamReader(is))) {
    return br.readLine();
} catch (IOException e) {
    log.error("读取失败", e);
    throw e;
}
```

**编译器展开后（javap 反编译）：**
```java
// 编译器自动生成的等价代码（伪代码）
InputStream is = new FileInputStream("file.txt");
Throwable primaryException = null;

try {
    BufferedReader br = new BufferedReader(new InputStreamReader(is));
    Throwable brException = null;
    try {
        return br.readLine();           // 业务逻辑
    } catch (Throwable t) {
        brException = t;               // 记录业务异常
        throw t;
    } finally {
        if (br != null) {
            if (brException != null) { // 业务有异常
                try {
                    br.close();
                } catch (Throwable suppressed) {
                    // 关闭失败的异常不丢失，附加到主异常上！
                    brException.addSuppressed(suppressed); // ← 核心！
                }
            } else {
                br.close();            // 业务无异常，正常关闭
            }
        }
    }
} catch (Throwable t) {
    primaryException = t;
    throw t;
} finally {
    if (is != null) {
        if (primaryException != null) {
            try {
                is.close();
            } catch (Throwable suppressed) {
                primaryException.addSuppressed(suppressed); // ← 关闭异常不丢
            }
        } else {
            is.close();
        }
    }
}
```

**关键点 `addSuppressed` 的价值：**
```java
// 传统 finally 写法：关闭异常会吞掉业务异常
try {
    // 抛出 IOException: 网络中断
} finally {
    stream.close(); // 又抛出 IOException: 连接已关闭
    // 最终调用方只看到 "连接已关闭"，"网络中断" 丢了！
}

// try-with-resources：两个异常都保留
try (Socket s = new Socket(...)) {
    // 抛出 IOException: 网络中断  ← 主异常
    // close() 抛出 IOException: 连接已关闭 ← addSuppressed 附加到主异常
}

// 捕获时可以获取所有异常
try {
    // ...
} catch (IOException e) {
    System.out.println("主异常: " + e.getMessage());
    for (Throwable suppressed : e.getSuppressed()) {
        System.out.println("附加异常: " + suppressed.getMessage()); // 不丢失
    }
}
```

### 5. AutoCloseable 接口实现

```java
// try-with-resources 要求资源实现 AutoCloseable
public class DatabaseConnection implements AutoCloseable {
    private final Connection conn;
    
    public DatabaseConnection(String url) throws SQLException {
        this.conn = DriverManager.getConnection(url);
        System.out.println("连接已建立");
    }
    
    public ResultSet query(String sql) throws SQLException {
        return conn.createStatement().executeQuery(sql);
    }
    
    @Override
    public void close() throws SQLException {
        // 保证幂等：多次 close 不报错
        if (conn != null && !conn.isClosed()) {
            conn.close();
            System.out.println("连接已释放");
        }
    }
}

// 使用：无需手动 close，编译器保证关闭
try (DatabaseConnection db = new DatabaseConnection(URL)) {
    ResultSet rs = db.query("SELECT * FROM users");
    // 处理结果
} // ← 自动调用 db.close()
```

```java
// 情况1：System.exit() - JVM 直接退出，finally 不执行
try {

try {
    while (true) {} // 永远不退出 try

    System.out.println("这行也不会打印");

// 情况3：线程被 kill -9 / OOM Killer - OS 级别强杀

// 情况4：JVM 崩溃（SIGSEGV、内存溢出等）

Thread daemon = new Thread(() -> {

        // ...

daemon.start();
// 主线程结束 → JVM 退出 → 守护线程 finally 不保证执行
```

## 🔥 扩展追问 2：多资源关闭顺序是怎样的？

```java
// 多资源按照声明的逆序关闭！
try (A a = new A();    // 第1个声明
     B b = new B();    // 第2个声明
     C c = new C()) {  // 第3个声明
    // 业务逻辑
}
// 关闭顺序：C → B → A （LIFO，类似栈）
// 这符合资源依赖关系：b 可能依赖 a，所以先关 b 再关 a

// 验证：
class Resource implements AutoCloseable {
    String name;
    Resource(String name) { 
        System.out.println("打开: " + name); 
        this.name = name; 
    }
    @Override
    public void close() { 
        System.out.println("关闭: " + name); 
    }
}

try (Resource r1 = new Resource("R1");
     Resource r2 = new Resource("R2");
     Resource r3 = new Resource("R3")) {
}
// 输出：
// 打开: R1
// 打开: R2
// 打开: R3
// 关闭: R3
// 关闭: R2
// 关闭: R1
```

## 🔥 扩展追问 3：try-with-resources 和 try-finally 的性能对比？

```java
// 性能基本一致，都是编译期展开
// try-with-resources 的额外开销：
// 1. addSuppressed 调用（仅在有异常时才触发，正常路径无开销）
// 2. Suppressed 异常链的内存（极少情况）

// 推荐优先使用 try-with-resources：
// ✅ 代码更简洁
// ✅ 不会忘记关闭资源
// ✅ 异常不丢失（addSuppressed）
// ✅ 支持多资源有序关闭
// ✅ 编译器静态检查

// 唯一需要 try-finally 的场景：
// 资源不实现 AutoCloseable（极少见，一般可包装）
class LegacyResource {
    void release() {} // 非标准接口
}

// 包装成 AutoCloseable
try (AutoCloseable wrapper = () -> legacy.release()) {
    legacy.doWork();
}
```

## 🔥 扩展追问 4：面试高频陷阱题

```java
// 陷阱1：finally 中 return 覆盖 try 中 return
public int trap1() {
    try { return 1; }
    finally { return 2; } // 返回 2，不是 1
}

// 陷阱2：finally 改基本类型无效
public int trap2() {
    int x = 1;
    try { return x; }   // 压栈保存 x=1
    finally { x = 2; }  // 改局部变量无效，返回 1
}

// 陷阱3：catch 和 finally 都抛异常，catch 的异常丢失
public void trap3() throws Exception {
    try {
        throw new Exception("A");
    } catch (Exception e) {
        throw new Exception("B"); // ← 这个会被丢失！
    } finally {
        throw new Exception("C"); // ← 调用方只看到 C
    }
}

// 陷阱4：try-with-resources 的 close 异常 vs 业务异常
// 业务异常 > close 异常（close 被 addSuppressed，不会覆盖主异常）
// 与传统 finally 正好相反！
```

## 一句话收尾

> `finally` 是 JVM 级别的保证，`catch` 里抛异常挡不住它；`finally` 里写 `return` 是异常吞噬器，生产代码严禁使用；`try-with-resources` 通过编译器 + `addSuppressed` 实现了**资源不泄漏、异常不丢失**的最优解，这才是资源管理的正确姿势。

---
几个关键点整理一下，方便你背：

**面试必背三条规律：**

1. **finally 必执行** → `catch` 里抛再狠的异常也拦不住 `finally`，除非 `System.exit()` 或 JVM 崩
2. **finally return = 异常黑洞** → 会把 `try`/`catch` 里的异常全吞掉，SonarQube 直接报高危，生产代码碰都别碰
3. **try-with-resources 优于 finally 手动 close** → 因为 `addSuppressed` 能保留关闭时的附加异常，而传统 `finally` 关闭异常会覆盖掉业务异常

**容易踩的坑（高频出题点）：**
- `finally` 改基本类型 → 无效（返回值已压栈）
- `finally` 改引用对象内容 → 有效（操作的是堆上的对象）
- 多资源关闭顺序 → **逆序**（LIFO，后声明的先关闭）
