## 先说结论

`FOR UPDATE` 在 RR 级别下会加**行锁（X 锁）+ 间隙锁（Gap Lock）**；如果 `age` 有索引，只锁符合条件的记录及间隙；如果 `age` 无索引，会触发**锁升维**，退化为**表锁**或锁住全表所有记录及间隙；避免方法是**确保查询条件走索引**。

---

## 🔥 核心概念

| 锁类型 | 作用 | 场景 |
|--------|------|------|
| **行锁（Record Lock）** | 锁住单条记录 | 命中索引，精确匹配 |
| **间隙锁（Gap Lock）** | 锁住索引间隙，防幻读 | RR 级别范围查询 |
| **临键锁（Next-Key Lock）** | 行锁 + 间隙锁 | RR 默认锁模式 |
| **表锁（Table Lock）** | 锁住整张表 | 无索引、锁升维 |
| **锁升维（Lock Escalation）** | 行锁变表锁 | 无索引、大量行锁 |

---

## 🔥 详细展开

### 1. age 有索引时的加锁情况

```sql
-- 表结构：user(id PK, age INDEX, name)
-- 数据：id=1,age=10; id=2,age=20; id=3,age=25; id=4,age=30

SELECT * FROM user WHERE age > 18 FOR UPDATE;

-- 执行计划：走 age 索引（Index Range Scan）

-- 加锁范围（临键锁 Next-Key Lock）：
-- 1. 锁住 age=20 的行（X 锁）
-- 2. 锁住 (18, 20] 的间隙（Gap Lock）
-- 3. 锁住 age=25 的行（X 锁）
-- 4. 锁住 (20, 25] 的间隙（Gap Lock）
-- 5. 锁住 age=30 的行（X 锁）
-- 6. 锁住 (25, 30] 的间隙（Gap Lock）
-- 7. 锁住 (30, +∞) 的间隙（Gap Lock，防止插入 age>30 的记录）

-- 结果：
-- - 已存在的 age=20,25,30 记录被锁（不能修改/删除）
-- - age∈(18,20)、(20,25)、(25,30)、(30,+∞) 区间不能插入新记录
-- - age≤18 的记录不受影响（可以插入 age=15）
```

**锁的示意图：**
```
age 索引：  10    20    25    30    +∞
           |     |     |     |     |
           ↑     [=====|=====|=====|====>
                 ↑     ↑     ↑     ↑
               行锁+间隙锁组合（临键锁）
               
[===== 表示被锁住的范围（Gap Lock）
↑      表示被锁住的记录（Record Lock）
```

---

### 2. age 无索引时的锁升维

```sql
-- 表结构：user(id PK, age, name)  -- age 无索引

SELECT * FROM user WHERE age > 18 FOR UPDATE;

-- 执行计划：全表扫描（ALL，遍历每一行）

-- 问题：InnoDB 行锁是基于索引的，无索引时怎么办？
-- 答案：锁升维！所有记录都加锁

-- 加锁情况：
-- 1. 遍历全表每一行
-- 2. 对每一行加 X 锁（行锁）
-- 3. 对所有间隙加 Gap Lock

-- 实际效果 = 锁住了整张表！
-- - 任何记录的修改/删除都被阻塞
-- - 任何新记录的插入都被阻塞（所有间隙都被锁）
-- - 其他事务的任何写操作都被阻塞
```

**锁升维的本质：**

```java
// InnoDB 行锁的实现依赖索引
// 无索引时，InnoDB 不知道"哪些行符合条件"
// 只能：遍历全表 → 逐行判断 → 逐行加锁

// 优化：MySQL 5.6+ 有 Index Condition Pushdown
// 但锁还是在存储引擎层加，无法避免

// 极端情况：
// 1000 万行表，age>18 有 900 万行
// 需要加 900 万个行锁 + 无数间隙锁
// 内存消耗巨大，性能极差，甚至 OOM
```

---

### 3. 锁升维的解决方案

```sql
-- 方案1：给查询列加索引（根本解决）
CREATE INDEX idx_age ON user(age);

-- 方案2：优化查询条件，使用主键或已有索引
-- 避免：WHERE 无索引列 = xxx
-- 改用：WHERE 主键 = xxx 或 WHERE 有索引列 = xxx

-- 方案3：降低隔离级别（权衡一致性和性能）
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- RC 级别：没有 Gap Lock，只有 Record Lock
-- 但会引入幻读问题

-- 方案4：使用覆盖索引，减少回表
-- 如果 SELECT 的列都在索引里，避免回表加锁

-- 方案5：分批处理，减少单次加锁范围
-- 避免：一次锁住大量记录
-- 改用：LIMIT 分批处理

-- 方案6：使用乐观锁替代悲观锁
-- 避免：SELECT ... FOR UPDATE
-- 改用：UPDATE ... WHERE version = old_version
```

---

### 4. 不同隔离级别的锁差异

```sql
-- READ UNCOMMITTED (RU)
-- - 不加锁，直接读最新数据（可能脏读）
SELECT * FROM user WHERE age > 18 FOR UPDATE; -- 几乎不加锁

-- READ COMMITTED (RC)
-- - 加行锁（Record Lock），不加间隙锁
-- - 每次查询重新生成 ReadView，解决不可重复读
SELECT * FROM user WHERE age > 18 FOR UPDATE;
-- 只锁 age=20,25,30 的行，不锁间隙
-- 其他事务可以插入 age=22 的记录（幻读可能发生）

-- REPEATABLE READ (RR) - 默认
-- - 加临键锁（Next-Key Lock = Record + Gap）
-- - 事务开始时生成 ReadView，保证可重复读
SELECT * FROM user WHERE age > 18 FOR UPDATE;
-- 锁行 + 锁间隙，防止幻读

-- SERIALIZABLE
-- - 所有 SELECT 都加共享锁（S Lock）
-- - 相当于所有读都变成 SELECT ... LOCK IN SHARE MODE
-- - 最严格，性能最差
```

## 🔥 扩展追问 1：Gap Lock 的精确范围

```sql
-- 数据：id=1,age=10; id=2,age=20; id=3,age=25; id=4,age=30

-- 场景1：age > 18
-- 锁住：(18,20], (20,25], (25,30], (30,+∞)
-- 注意：左开右闭，18 本身不锁，20 被锁

-- 场景2：age >= 20
-- 锁住：[20,20], (20,25], (25,30], (30,+∞)
-- 注意：[20,20] 是行锁，不是间隙

-- 场景3：age = 20（等值查询）
-- 如果 age 是唯一索引：只锁 age=20 的行
-- 如果 age 是普通索引：锁 (10,20] 和 (20,25]（防止幻读）

-- 场景4：age BETWEEN 20 AND 25
-- 锁住：[20,20], (20,25], [25,25]
-- 或：(15,20], (20,25], (25,30] 取决于优化器
```

## 🔥 扩展追问 2：死锁案例分析

```sql
-- 事务A：锁住 age>18 的所有记录
BEGIN;
SELECT * FROM user WHERE age > 18 FOR UPDATE;
-- 锁住：age=20,25,30 及间隙

-- 事务B：想插入 age=22 的记录
BEGIN;
INSERT INTO user(age, name) VALUES (22, 'test');
-- 阻塞！等待事务A释放 (20,25) 间隙锁

-- 事务A：想插入 age=22 的记录
INSERT INTO user(age, name) VALUES (22, 'test');
-- 死锁！事务B 在等待事务A，事务A 又在等事务B
-- MySQL 检测死锁，回滚其中一个事务

-- 解决方案：
-- 1. 按固定顺序加锁（如按主键排序后加锁）
-- 2. 降低隔离级别到 RC（无 Gap Lock）
-- 3. 使用 INSERT IGNORE 或 ON DUPLICATE KEY UPDATE
```

## 🔥 扩展追问 3：如何查看当前锁情况

```sql
-- 查看当前锁等待
SELECT * FROM information_schema.INNODB_TRX;        -- 活跃事务
SELECT * FROM information_schema.INNODB_LOCKS;      -- 当前锁（MySQL 8.0 移除）
SELECT * FROM performance_schema.data_locks;        -- MySQL 8.0+ 替代
SELECT * FROM performance_schema.data_lock_waits;   -- 锁等待关系

-- 查看锁等待详情
SELECT 
    r.trx_id waiting_trx_id,
    r.trx_mysql_thread_id waiting_thread,
    r.trx_query waiting_query,
    b.trx_id blocking_trx_id,
    b.trx_mysql_thread_id blocking_thread,
    b.trx_query blocking_query
FROM information_schema.innodb_lock_waits w
JOIN information_schema.innodb_trx b ON b.trx_id = w.blocking_trx_id
JOIN information_schema.innodb_trx r ON r.trx_id = w.requesting_trx_id;

-- 查看锁等待超时时间
SHOW VARIABLES LIKE 'innodb_lock_wait_timeout';  -- 默认 50 秒
```

## 🔥 扩展追问 4：FOR UPDATE 的优化技巧

```sql
-- 技巧1：使用主键或唯一索引精确锁定
-- 好：
SELECT * FROM user WHERE id = 1 FOR UPDATE;  -- 只锁一行

-- 不好：
SELECT * FROM user WHERE age = 20 FOR UPDATE;  -- 可能锁多行+间隙

-- 技巧2：减少 SELECT 范围
-- 不好：
SELECT * FROM user WHERE age > 18 FOR UPDATE;  -- 锁大量记录

-- 好：
SELECT * FROM user WHERE age > 18 AND age < 25 FOR UPDATE;  -- 缩小范围

-- 技巧3：使用 LIMIT 限制
SELECT * FROM user WHERE status = 'pending' FOR UPDATE LIMIT 10;
-- 只锁 10 条，而不是所有 pending

-- 技巧4：NOWAIT 和 SKIP LOCKED（MySQL 8.0）
SELECT * FROM user WHERE status = 'pending' FOR UPDATE NOWAIT;
-- 立即返回，不等待，拿不到锁报错

SELECT * FROM user WHERE status = 'pending' FOR UPDATE SKIP LOCKED;
-- 跳过被锁的行，拿其他行，实现"抢单"模式
```

## 一句话收尾

> `FOR UPDATE` 在 RR 级别下是**行锁 + 间隙锁**的组合拳，有索引时精准打击，无索引时**锁升维**变成表锁灾难；避免锁升维的核心是**确保查询走索引**，同时注意 Gap Lock 可能引发的死锁，必要时用 RC 级别或乐观锁替代。

---
几个关键点再强调一下：

**RR 级别 `FOR UPDATE` 的加锁规则：**

| 条件 | 加锁类型 | 范围 |
|------|---------|------|
| 唯一索引精确匹配 | Record Lock | 单行 |
| 普通索引等值 | Next-Key Lock | 行 + 相邻间隙 |
| 范围查询 | Next-Key Lock | 范围内所有行 + 间隙 |
| 无索引 | 锁升维 → 表锁效果 | 全表 |

**锁升维的后果：**
- 全表扫描逐行加锁
- 所有记录被锁，所有间隙被锁
- 其他事务任何写操作都被阻塞
- 性能极差，可能 OOM

**避免锁升维的方法：**
1. **给查询列加索引**（根本解决）
2. 使用主键或已有索引查询
3. 降低隔离级别到 RC（无 Gap Lock，但有幻读风险）
4. 使用乐观锁替代 `FOR UPDATE`

**MySQL 8.0 新特性：**
```sql
FOR UPDATE NOWAIT;      -- 拿不到锁立即报错，不等待
FOR UPDATE SKIP LOCKED; -- 跳过被锁的行，实现抢单模式
```
