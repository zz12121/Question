## 先说结论

binlog 是 MySQL **Server 层的逻辑日志**，记录所有**数据变更的 SQL 语句**，用于**主从复制**和**数据恢复**；redo log 是 **InnoDB 层的物理日志**，记录**页级别的物理修改**，用于**崩溃恢复**；两者是**"逻辑 vs 物理"、"Server vs 引擎"、"复制 vs 恢复"**的分层设计。

---

## 🔥 核心概念

| 维度 | **binlog** | **redo log** |
|------|-----------|--------------|
| **层级** | MySQL Server 层 | InnoDB 存储引擎层 |
| **日志类型** | **逻辑日志**（SQL 语句/行数据） | **物理日志**（页偏移量修改） |
| **记录内容** | 原始 SQL 或 变更前后的行数据 | 页号 + 偏移量 + 修改内容 |
| **主要用途** | **主从复制**、数据恢复、审计 | **崩溃恢复**（ACID 持久性） |
| **写入时机** | 事务提交时 | 事务执行中持续写 |
| **刷盘策略** | `sync_binlog` 控制 | `innodb_flush_log_at_trx_commit` 控制 |
| **文件形式** | 多个文件，追加写，不会覆盖 | 固定大小（如 2 个 1GB 文件），循环写 |
| **崩溃恢复** | 不直接参与 | 核心机制 |
| **主从复制** | 核心机制 | 不参与 |

---

## 🔥 详细展开

### 1. binlog 详解

```sql
-- binlog 记录格式（三种）：

-- 1. STATEMENT（语句模式）
--    记录原始 SQL 语句
--    优点：日志小
--    缺点：某些语句主从结果不一致（如 UUID()、NOW()、RAND()）

-- 示例：
UPDATE account SET balance = balance - 100 WHERE id = 1;
-- binlog 记录：上述 SQL 原文

-- 2. ROW（行模式，MySQL 5.7+ 默认）
--    记录变更前后的行数据
--    优点：主从一致，精确
--    缺点：日志大（批量更新可能爆炸）

-- 示例：
-- 变更前：id=1, balance=1000
-- 变更后：id=1, balance=900
-- binlog 记录：完整的行前像和行后像

-- 3. MIXED（混合模式）
--    默认 STATEMENT，不安全时自动转 ROW
```

**binlog 文件结构：**

```java
// mysql-bin.000001, mysql-bin.000002 ...

// 每个 binlog 文件包含多个 Event：
┌─────────────────────────────────────┐
│ Event Header (19 bytes)             │
│  - timestamp、type_code、server_id   │
│  - event_length、next_position       │
├─────────────────────────────────────┤
│ Event Data                          │
│  - FORMAT_DESCRIPTION_EVENT：文件头   │
│  - QUERY_EVENT：BEGIN、DDL           │
│  - TABLE_MAP_EVENT：表结构映射        │
│  - WRITE_ROWS_EVENT：INSERT          │
│  - UPDATE_ROWS_EVENT：UPDATE         │
│  - DELETE_ROWS_EVENT：DELETE         │
│  - XID_EVENT：事务提交                │
└─────────────────────────────────────┘

// 查看 binlog：
SHOW BINARY LOGS;
SHOW MASTER STATUS;
SHOW BINLOG EVENTS IN 'mysql-bin.000001' LIMIT 10;

// 解析工具：
// mysqlbinlog mysql-bin.000001 --base64-output=DECODE-ROWS -v
```

### 2. redo log 详解

```java
// redo log 是物理日志，记录"在某个页的某个偏移量修改了什么"

// 示例：UPDATE balance = 900 WHERE id = 1
// 假设 id=1 的记录在页号 1234，偏移量 567 处

// redo log 记录（简化）：
{
    type: MLOG_WRITE_STRING,  // 写操作类型
    space_id: 10,             // 表空间 ID
    page_no: 1234,            // 页号
    offset: 567,              // 页内偏移
    len: 8,                   // 修改长度
    data: "00000384"          // 新值（900 的十六进制）
}

// 特点：
// 1. 物理精确：直接定位到磁盘位置
// 2. 幂等：重复应用多次结果相同
// 3. 顺序写：性能高
```

---

### 3. 两者写入流程对比

```sql
-- 事务执行流程（以 UPDATE 为例）：

BEGIN;
UPDATE account SET balance = 900 WHERE id = 1;
COMMIT;

-- 执行时：
-- 1. 查询 Buffer Pool，加载数据页
-- 2. 加锁、写 Undo Log
-- 3. 修改内存页（产生脏页）
-- 4. 【持续写】redo log buffer ← redo log 先写内存
-- 5. 【提交时】binlog cache ← binlog 先写内存

-- COMMIT 时（两阶段提交）：
-- 阶段1：准备阶段
--   - redo log 刷盘（prepare 状态）
--   - binlog 刷盘

-- 阶段2：提交阶段  
--   - redo log 标记为 commit 状态
--   - 返回客户端成功

-- 为什么需要两阶段提交？
-- 保证 binlog 和 redo log 的一致性
-- 崩溃恢复时，根据两者状态判断事务是否提交
```

---

### 4. 两阶段提交详解

```java
// 两阶段提交（2PC）保证 binlog 和 redo log 一致性

// 正常流程：
//         redo log          binlog
// BEGIN    [prepare]         [未写入]
//          [fsync]           [fsync]
// COMMIT   [commit]          [已写入]

// 崩溃场景分析：

// 场景1：redo log prepare 后，binlog 写入前崩溃
// - redo log 有 prepare，binlog 没有
// - 恢复时：回滚事务（binlog 没有，不能复制到从库）

// 场景2：binlog 写入后，redo log commit 前崩溃
// - redo log 有 prepare，binlog 有记录
// - 恢复时：提交事务（binlog 有，必须复制到从库）

// 场景3：redo log commit 后崩溃
// - 事务已完成，无需处理

// 判断依据：
// - redo log prepare + binlog 存在 = 提交
// - redo log prepare + binlog 不存在 = 回滚
```

## 🔥 扩展追问 1：binlog 的刷盘策略

```sql
-- sync_binlog 参数控制 binlog 刷盘时机：

-- sync_binlog = 0（默认）
-- - 由 OS 决定何时刷盘
-- - 性能最好，但可能丢数据（OS 崩溃）

-- sync_binlog = 1
-- - 每次事务提交都 fsync 刷盘
-- - 最安全，性能最差

-- sync_binlog = N（N > 1）
-- - 每 N 次事务提交刷盘一次
-- - 平衡性能和安全

-- 生产建议：
-- 主库：sync_binlog = 1（配合 innodb_flush_log_at_trx_commit = 1）
-- 从库：sync_binlog = 0 或 100（提升性能）
```

## 🔥 扩展追问 2：binlog 与主从复制

```sql
-- 主从复制流程：

-- 1. Master 写入 binlog
-- 2. Slave IO Thread 连接 Master，拉取 binlog
-- 3. Slave 写入 relay log（中继日志）
-- 4. Slave SQL Thread 重放 relay log，应用到从库

-- 复制模式：
-- 1. 异步复制（默认）
--    - Master 不等待 Slave 确认
--    - 性能最好，可能丢数据

-- 2. 半同步复制（semi-sync）
--    - Master 等待至少一个 Slave 确认收到 binlog
--    - 平衡性能和安全
--    - 插件：rpl_semi_sync_master_enabled

-- 3. 组复制（Group Replication）
--    - 基于 Paxos 协议，多数派确认
--    - 强一致，自动故障转移

-- 复制延迟排查：
SHOW SLAVE STATUS;
-- Seconds_Behind_Master：延迟秒数
-- 原因：大事务、从库性能差、网络延迟
```

## 🔥 扩展追问 4：redo log 与 binlog 的持久化顺序

```sql
-- 关键问题：如果先写 redo log，再写 binlog，中间崩溃会怎样？

-- 场景：
-- 1. redo log 已刷盘（prepare）
-- 2. binlog 未刷盘
-- 3. 崩溃

-- 恢复：
-- - redo log 有 prepare 记录
-- - 检查 binlog，发现没有对应事务
-- - 判定为未提交，回滚

-- 结果：主库回滚，从库没有这条记录 → 一致 ✓

-- 反过来：先写 binlog，再写 redo log
-- 1. binlog 已刷盘
-- 2. redo log 未刷盘
-- 3. 崩溃

-- 恢复：
-- - redo log 没有记录
-- - 不知道这个事务是否存在
-- - 但 binlog 有，从库会复制

-- 结果：主库没有，从库有 → 不一致 ✗

-- 结论：必须先 redo log prepare，再 binlog，最后 redo log commit
```

## 一句话收尾

> binlog 是 MySQL **"对外说话的嘴巴"**（逻辑日志，用于复制和恢复），redo log 是 InnoDB **"自我保护的盾牌"**（物理日志，用于崩溃恢复）；两者通过**两阶段提交**保证一致性，binlog 让数据能复制到从库，redo log 让数据崩溃后不丢，是 MySQL **高可用 + 高可靠**的双保险。

---
几个关键点再强调一下：

**binlog vs redo log 的核心区别：**

| 维度 | binlog | redo log |
|------|--------|----------|
| 层级 | **Server 层** | **InnoDB 层** |
| 类型 | **逻辑日志**（SQL/行数据） | **物理日志**（页偏移量） |
| 用途 | **复制** + 恢复 | **崩溃恢复** |
| 文件 | 多个，追加写 | 固定大小，循环写 |

**两阶段提交的必要性：**
```
prepare(redo log) → 写 binlog → commit(redo log)
```
- 保证崩溃恢复时，binlog 和 redo log 状态一致
- 避免主从不一致（主库回滚，从库复制）

**binlog 三种格式：**
| 格式 | 记录内容 | 优点 | 缺点 |
|------|---------|------|------|
| STATEMENT | SQL 语句 | 日志小 | 主从不一致风险 |
| ROW | 行前后像 | 精确一致 | 日志大 |
| MIXED | 混合 | 平衡 | 复杂 |

**生产配置建议：**
```ini
# 主库（数据安全优先）
sync_binlog = 1
innodb_flush_log_at_trx_commit = 1
binlog_format = ROW

# 从库（性能优先）
sync_binlog = 0
innodb_flush_log_at_trx_commit = 2
```
