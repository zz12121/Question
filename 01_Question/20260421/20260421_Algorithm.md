### A326."LRU缓存的实现原理是什么？HashMap+双向链表如何配合完成O(1)的get和put操作？"

**参考答案：**

LRU（Least Recently Used）缓存淘汰策略的核心思想是：当缓存满时，淘汰最近最少使用的数据。

**数据结构组合：**
- **HashMap**：Key → 双向链表节点指针，实现O(1)查找
- **双向链表**：维护访问顺序，最近访问的移到头部，尾部即是最久未访问

**操作流程：**
- **get(key)**：HashMap查找节点 → 存在则将节点移到链表头部 → 返回值；不存在返回-1
- **put(key, value)**：
  - key已存在：更新值 → 移到头部
  - key不存在：创建新节点 → 插入头部 → 若超出容量，删除尾部节点 → 同时从HashMap删除对应Key

**Java实现关键：**
```java
class LRUCache {
    // 使用LinkedHashMap的重写方式最简洁
    // accessOrder=true 表示按访问顺序排序
    super(capacity, 0.75f, true);
    protected boolean removeEldestEntry(Map.Entry eldest) {
        return size() > capacity;
    }
}
```

**时间复杂度：** get和put均为O(1)，因为HashMap查找O(1)，链表节点移动O(1)

### A327."LFU缓存与LRU的区别是什么？LFU的O(1)实现需要哪些数据结构？"

**参考答案：**

**LRU vs LFU：**
- LRU按**访问时间**排序，淘汰最久未访问的
- LFU按**访问频率**排序，淘汰访问次数最少的；同频率时按LRU淘汰

**LFU的O(1)实现需要三层结构：**
1. **KeyMap**：key → Node（包含value、freq）
2. **FreqMap**：freq → 双向链表（该频率的所有节点，按LRU顺序）
3. **minFreq**：记录当前最小频率，用于O(1)找到淘汰对象

**操作流程：**
- **get(key)**：从KeyMap找到Node → 从旧freq链表移除 → 加入freq+1链表 → 更新minFreq
- **put(key, value)**：
  - 已存在：更新值 → 频率+1（同get的频率提升）
  - 不存在：若满容量，从minFreq链表尾部删除 → 新Node加入freq=1链表 → minFreq=1

**关键难点：** 同频率下按LRU淘汰，所以FreqMap的每个freq对应一个双向链表而非单个节点

### A328."布隆过滤器的原理是什么？为什么会有误判率？如何估算误判率？"

**参考答案：**

**原理：**
- 一个长度为m的位数组（初始全0）+ k个独立哈希函数
- **添加元素**：用k个哈希函数计算k个位置 → 全部置1
- **查询元素**：用k个哈希函数计算k个位置 → 全部为1则"可能存在"，任一为0则"一定不存在"

**误判原因：**
不同元素可能映射到相同的k个位置（哈希冲突），导致不存在的元素被误判为存在。但**不存在的一定不会漏判**（不会出现假阴性）。

**误判率公式：**
```
p ≈ (1 - e^(-kn/m))^k
```
- n：已插入元素数量
- m：位数组长度
- k：哈希函数个数

**最优k值：** k = (m/n) × ln2

**工程实践：**
- Guava BloomFilter：默认3%误判率，可指定预期元素数和误判率
- Redis + BloomFilter模块：分布式场景
- Counting BloomFilter：用计数器替代位，支持删除（但空间翻4倍）

### A329."跳表的原理是什么？与平衡树相比有哪些优势和劣势？Redis为什么用跳表实现ZSet？"

**参考答案：**

**跳表原理：**
- 在有序链表基础上，增加多级索引层
- 每个节点以概率p（通常1/2或1/4）晋升到上一层
- 查找时从最高层开始，逐层向下，类似二分查找
- 空间复杂度O(n)，查找/插入/删除平均O(log n)

**跳表 vs 平衡树：**

| 维度 | 跳表 | 红黑树/AVL |
|------|------|-----------|
| 实现复杂度 | 简单，无需旋转 | 复杂，需要旋转/变色 |
| 范围查询 | 天然支持，沿底层链表遍历 | 需要中序遍历 |
| 并发友好 | 局部修改，CAS友好 | 旋转涉及多节点，锁粒度大 |
| 空间开销 | 指针较多O(n)额外空间 | 每节点2个指针+颜色位 |
| 最坏复杂度 | O(n)（极端退化） | O(log n)严格保证 |

**Redis选择跳表的原因：**
1. ZSet核心操作ZRANGEBYSCORE是范围查询，跳表底层链表天然支持
2. 实现简单，易维护
3. 并发场景下更友好（Redis 6.0多线程IO）
4. 通过调整概率参数可平衡空间与时间

### A330."一致性哈希的原理是什么？虚拟节点解决什么问题？与取模哈希相比有什么优势？"

**参考答案：**

**原理：**
- 将整个哈希值空间组织成虚拟圆环（0 ~ 2^32-1）
- 每个节点和数据都用相同的哈希函数映射到环上
- 数据顺时针找到的第一个节点就是其归属节点

**虚拟节点：**
- 问题：节点少时数据分布不均匀
- 方案：每个物理节点对应多个虚拟节点（如150~200个），均匀分布在环上
- 效果：虚拟节点越多，数据分布越均匀

**一致性哈希 vs 取模哈希：**

| 维度 | 一致性哈希 | 取模哈希 hash%N |
|------|-----------|----------------|
| 节点增减 | 只影响相邻节点数据 | 几乎所有数据重新映射 |
| 数据迁移量 | O(K/N) K为总key数 | 接近O(K) |
| 均匀性 | 依赖虚拟节点 | 天然均匀 |
| 复杂度 | 较高 | 极简 |

**Java实现：** TreeMap模拟哈希环，ceilingKey找后继节点

**工程应用：** Redis Cluster的Hash Slot（16384个槽）是一致性哈希的变体，预分区避免环维护开销

### A331."前缀树（Trie）的原理和适用场景是什么？与HashMap相比有什么优劣？"

**参考答案：**

**原理：**
- 根节点为空，每条边代表一个字符
- 从根到某节点的路径就是该节点代表的前缀/字符串
- 每个节点包含子节点映射（通常26个字母或更大字符集）和结束标记

**适用场景：**
1. 前缀匹配：搜索框自动补全
2. 词频统计：大量字符串的词频统计
3. 字符串排序：按字典序遍历
4. 敏感词过滤：AC自动机（Trie + KMP失败指针）

**Trie vs HashMap：**

| 维度 | Trie | HashMap |
|------|------|---------|
| 前缀查询 | O(L)天然支持 | 不支持，需遍历 |
| 空间效率 | 公共前缀共享，前缀多时省空间 | 每个Key独立存储 |
| 精确查找 | O(L) | O(1)平均 |
| 空间开销 | 稀疏时浪费大（每节点数组/Map） | 相对紧凑 |
| 插入/删除 | O(L) | O(1)平均 |

**L为字符串长度**

**优化方向：** 双数组Trie（用两个数组压缩存储，空间降低90%）、Radix Tree（合并单分支路径）

### A332."红黑树的5条性质是什么？为什么工程实践中多选择红黑树而非AVL树？"

**参考答案：**

**红黑树5条性质：**
1. 每个节点是红色或黑色
2. 根节点是黑色
3. 每个叶子节点（NIL）是黑色
4. 红色节点的两个子节点必须是黑色（不能有连续红节点）
5. 从任一节点到其所有叶子节点的路径，黑色节点数量相同（黑高相同）

**红黑树 vs AVL树：**

| 维度 | 红黑树 | AVL树 |
|------|--------|-------|
| 平衡标准 | 近似平衡（黑高相同） | 严格平衡（高度差≤1） |
| 查找性能 | O(log n)，略慢 | O(log n)，略快 |
| 插入/删除 | 最多3次旋转/2次旋转 | 最多O(log n)次旋转 |
| 旋转开销 | 小 | 大 |
| 存储开销 | 1位颜色 | 平衡因子（int） |

**工程选择红黑树的原因：**
1. 增删操作更频繁时，红黑树旋转次数少，综合性能更优
2. 近似平衡已经保证O(log n)，对查找性能影响很小
3. 实现相对简单（Java TreeMap、Linux CFS调度器、epoll红黑树）

**AVL适用场景：** 查找远多于增删（如数据库内存索引、只读字典）

### A333."B+树与B树的区别是什么？为什么MySQL InnoDB选择B+树而非B树作为索引结构？"

**参考答案：**

**B+树 vs B树：**

| 维度 | B+树 | B树 |
|------|------|------|
| 数据存储 | 只在叶子节点存储数据 | 所有节点都可能存储数据 |
| 叶子节点链接 | 叶子节点有双向链表 | 无链表连接 |
| 非叶节点 | 只存索引键（路由信息） | 存键+数据+子指针 |
| 查找路径 | 一定走到叶子节点 | 可能在中间节点命中 |
| 范围查询 | 沿叶子链表遍历，高效 | 需要中序遍历整棵树 |

**InnoDB选择B+树的原因：**
1. **磁盘IO更少**：非叶节点只存键，同样页大小（16KB）能存更多索引项，树更矮。3层B+树可存2000万+行（假设索引键12字节，一个页存~1300个键，3层 = 1300×1300×1300）
2. **范围查询高效**：SELECT * FROM t WHERE id BETWEEN 100 AND 200，沿叶子链表遍历即可
3. **查询稳定**：每次都走到叶子节点，路径长度一致，性能可预测
4. **页利用率高**：节点满时才分裂，保证至少50%利用率

## 二、排序算法（5题）

### A334."快排、归并排序、堆排的时间复杂度和稳定性分别是什么？各自适用场景是什么？"

**参考答案：**

| 算法 | 平均时间 | 最坏时间 | 空间 | 稳定性 | 适用场景 |
|------|---------|---------|------|--------|---------|
| 快排 | O(nlogn) | O(n²) | O(logn) | 不稳定 | 通用排序，内存排序首选 |
| 归并 | O(nlogn) | O(nlogn) | O(n) | 稳定 | 大数据外部排序、需稳定性 |
| 堆排 | O(nlogn) | O(nlogn) | O(1) | 不稳定 | TopK问题、空间受限场景 |

**深入对比：**
- **快排**最坏O(n²)出现在已排序数组+固定选pivot，可用三数取中或随机pivot规避
- **归并**稳定但需O(n)额外空间，适合链表排序（归并链表O(1)空间）
- **堆排**不稳定但空间O(1)，缓存不友好（跳跃访问），实际比快排慢2~5倍
- **Java Arrays.sort()**：基础类型用双轴快排，对象用TimSort（归并+插入混合）

### A335."排序算法的稳定性是什么意思？哪些排序是稳定的？什么场景必须要求稳定？"

**参考答案：**

**稳定性定义：** 对于相等的元素，排序后相对顺序保持不变。

**稳定性分类：**
- **稳定排序**：冒泡、插入、归并、计数、桶、基数、TimSort
- **不稳定排序**：选择、快排、堆排、Shell排序

**不稳定的根本原因：**
- 选择排序：交换可能跨越相等元素
- 快排：partition交换不保证相等元素顺序
- 堆排：向下调整可能交换相等元素的相对位置

**必须要求稳定的场景：**
1. 多关键字排序：先按成绩排序，再按班级排序——要求同班级内成绩顺序保持
2. 数据库ORDER BY：先按A列排序，再按B列排序，要求B列相同时A列顺序保持
3. 业务语义：支付记录先按金额排序，金额相同按时间排序

**工程技巧：** 不稳定排序可通过"添加原始下标作为次要排序键"来模拟稳定

### A336."计数排序、桶排序、基数排序这三种非比较排序的原理和适用条件是什么？"

**参考答案：**

**计数排序：**
- 原理：统计每个值出现次数，然后按次数输出
- 时间O(n+k)，空间O(k)，k为值域范围
- 适用：值域小且为整数（如年龄0~150、分数0~100）
- 不适用：值域大或浮点数

**桶排序：**
- 原理：将数据分到有限个桶中，每个桶内用比较排序
- 时间O(n)平均（数据均匀分布时），最坏O(n²)
- 适用：数据均匀分布，范围已知
- 典型场景：1亿个浮点数，范围[0,1)

**基数排序：**
- 原理：按位排序（先个位→十位→百位），每趟用稳定排序（通常计数排序）
- 时间O(d×(n+k))，d为位数，k为基数（十进制k=10）
- 适用：等长整数或字符串排序
- 不适用：长度差异大或含负数（需偏移处理）

**核心限制：** 三者都依赖数据的特定性质（值域小/分布均匀/等长），不是通用排序

### A337."二分查找的变种有哪些？分别适用于什么场景？常见易错点是什么？"

**参考答案：**

**二分查找变种：**

1. **标准二分**：查找目标值，存在返回索引，不存在返回-1
2. **查找第一个等于目标**：`nums[mid] >= target` 时right=mid，最后检查`nums[left]==target`
3. **查找最后一个等于目标**：`nums[mid] <= target` 时left=mid，需注意mid取上取整
4. **查找第一个大于等于目标**：`nums[mid] >= target` 时right=mid，返回left
5. **查找最后一个小于等于目标**：`nums[mid] <= target` 时left=mid
6. **旋转数组查找**：先判断哪半边有序，再决定搜索方向

**常见易错点：**
1. **死循环**：`left=mid`时mid必须取上取整`left+(right-left+1)/2`，否则left不动死循环
2. **溢出**：`(left+right)/2` → 改用`left+(right-left)/2`
3. **边界条件**：空数组、单元素、全部相同元素
4. **开闭区间**：`[left, right]` 还是 `[left, right)`，影响while条件和mid更新

**工程应用：**
- `Arrays.binarySearch()`
- 数据库B+树索引查找
- 求解单调函数的零点（如：最小化最大值、最大化最小值）

### A338."TopK问题的三种解法是什么？分别适用于什么场景？时间复杂度各是多少？"

**参考答案：**

**三种解法：**

| 解法 | 时间复杂度 | 空间 | 适用场景 |
|------|-----------|------|---------|
| 排序取前K | O(nlogn) | O(1) | n较小时通用 |
| 小顶堆 | O(nlogK) | O(K) | n大，K小，流式数据 |
| 快速选择 | O(n)平均 | O(1) | n大，不需要有序，允许最坏O(n²) |

**小顶堆法：**
- 维护大小为K的小顶堆
- 新元素 > 堆顶时替换堆顶并下沉
- 适合流式数据（不知道总量）

**快速选择法：**
- 基于快排的partition
- partition后pivot位置为p：
  - p == K-1：前K个就是答案
  - p < K-1：只递归右半边
  - p > K-1：只递归左半边
- BFPRT算法可保证最坏O(n)，但常数大，工程中不常用

**工程实践：**
- Java `PriorityQueue`（小顶堆）
- `Arrays.sort()` 后取前K（n小时更快，因为快排和堆排的常数差异）
- 流式TopK：Flink KeyedProcessFunction + 堆

### A339."二叉树的前中后序遍历非递归实现的核心思路是什么？层序遍历如何实现？"

**参考答案：**

**非递归核心思路：** 用栈模拟递归调用栈

**前序（根左右）：**
- 方案1：栈，先压右子再压左子
- 方案2：统一写法，访问时输出

**中序（左根右）：**
- 一路向左压栈 → 弹出访问 → 转右子树
- 关键：先走到最左，再弹出处理

**后序（左右根）：**
- 最复杂的非递归遍历
- 方案1：前序（根右左）→ 反转结果
- 方案2：用prev指针记录上一个访问的节点，判断右子树是否已访问
- 方案3：统一写法，null标记法

**层序遍历：** 队列实现
```java
Queue<TreeNode> q = new LinkedList<>();
q.offer(root);
while (!q.isEmpty()) {
    int size = q.size();
    for (int i = 0; i < size; i++) {
        TreeNode node = q.poll();
        // 处理node
        if (node.left != null) q.offer(node.left);
        if (node.right != null) q.offer(node.right);
    }
}
```
**层序关键：** 用size记录当前层节点数，实现按层分组

### A340."图的BFS和DFS的实现方式是什么？拓扑排序用什么算法？如何检测有向图中的环？"

**参考答案：**

**BFS：**
- 队列实现，逐层扩展
- 适合最短路径（无权图）、层序遍历
- 需要visited数组防重复访问

**DFS：**
- 栈或递归实现，深入到底再回溯
- 适合路径搜索、连通分量、拓扑排序
- 需要visited数组和递归栈（可检测环）

**拓扑排序（Kahn算法 - BFS）：**
1. 计算所有节点入度
2. 入度为0的入队
3. 出队节点加入结果，其邻居入度-1，入度为0再入队
4. 若结果长度 < 节点数 → 存在环

**拓扑排序（DFS后序逆序）：**
- DFS后序序列的逆序即为拓扑序
- 需要三色标记：0=未访问、1=访问中、2=已完成

**环检测：**
- **有向图**：DFS三色标记，遇到"访问中"（灰色）节点说明有环
- **无向图**：DFS记录parent，遇到已访问且非parent的节点说明有环
- **并查集**：无向图中，若合并时发现已在同一集合 → 有环

### A341."Dijkstra和Bellman-Ford最短路径算法的区别是什么？Floyd算法适用于什么场景？"

**参考答案：**

| 维度 | Dijkstra | Bellman-Ford | Floyd |
|------|----------|--------------|-------|
| 适用图 | 无负权边 | 可有负权边 | 可有负权边 |
| 时间 | O((V+E)logV) | O(VE) | O(V³) |
| 空间 | O(V) | O(V) | O(V²) |
| 单源/多源 | 单源 | 单源 | 多源 |
| 负环检测 | 不支持 | 支持 | 不直接支持 |
| 核心思想 | 贪心+优先队列 | 松弛V-1轮 | 动态规划 |

**Dijkstra核心：** 优先队列每次取最近未访问节点，松弛其邻居。贪心正确性依赖无负权边。

**Bellman-Ford核心：** 对所有边松弛V-1轮，第V轮还能松弛则存在负环。

**Floyd核心：**
```
dp[k][i][j] = min(dp[k-1][i][j], dp[k-1][i][k] + dp[k-1][k][j])
```
枚举中转点k，可优化为二维滚动数组。

**Java工程场景：**
- 地图导航：Dijkstra（路网无负权）
- 网络路由：Bellman-Ford（如RIP协议）
- 所有点对最短路径：Floyd（小图）

### A342."并查集的原理是什么？路径压缩和按秩合并如何影响时间复杂度？"

**参考答案：**

**原理：**
- 用树形结构维护不相交集合
- find(x)：沿parent指针找到根（代表元）
- union(x, y)：将x的根连到y的根下

**路径压缩：**
- find时将沿途节点直接连到根
- 实现方式：递归或迭代
```java
int find(int x) {
    if (parent[x] != x) parent[x] = find(parent[x]); // 路径压缩
    return parent[x];
}
```

**按秩合并：**
- 维护每个集合的秩（近似高度）
- 合并时将秩小的连到秩大的下面
- 防止树退化成链

**时间复杂度：**
- 无优化：find最坏O(n)
- 只路径压缩：均摊O(α(n))
- 只按秩合并：find O(log n)
- **两者结合**：均摊O(α(n))，α为反阿克曼函数，实际≤4

**工程应用：**
- 最小生成树Kruskal算法
- 等式方程的可满足性（LeetCode 990）
- 省份数量（连通分量）
- 岛屿数量（网格合并）

### A343."最小生成树的Prim和Kruskal算法的原理和适用场景分别是什么？"

**参考答案：**

**Prim算法：**
- 原理：从一个点出发，每次选距离已选集合最近的点加入（贪心）
- 实现：优先队列，O(ElogV)
- 适用：稠密图（边多），邻接矩阵

**Kruskal算法：**
- 原理：将所有边按权排序，依次选最小边，若不构成环则加入（贪心+并查集）
- 实现：排序+并查集，O(ElogE)
- 适用：稀疏图（边少），边列表

**对比：**

| 维度 | Prim | Kruskal |
|------|------|---------|
| 时间 | O(ElogV) | O(ElogE) |
| 数据结构 | 优先队列 | 并查集 |
| 稠密图 | 更优 | 较慢 |
| 稀疏图 | 较慢 | 更优 |
| 增量构建 | 天然支持 | 不支持 |
| 并行化 | 困难 | 可并行排序 |

**Java工程场景：** 网络布线、管道铺设、集群节点连通最小成本

### A344."动态规划解题的核心要素是什么？如何判断一个问题是否适合用DP？状态转移方程如何推导？"

**参考答案：**

**DP核心要素：**
1. **最优子结构**：问题的最优解包含子问题的最优解
2. **重叠子问题**：子问题被重复计算（与分治的区别）
3. **无后效性**：当前状态确定后，未来决策与过去路径无关

**判断是否适合DP：**
- 能否将问题拆解为子问题？
- 子问题是否有重叠？
- 是否满足最优子结构？
- 状态是否有无后效性？

**状态转移方程推导步骤：**
1. **定义状态**：dp[i]或dp[i][j]代表什么
2. **找最后一步**：最后一步有哪些选择？
3. **推导转移**：dp[i] = min/max/dp[i-1], dp[i-2]...
4. **初始条件**：dp[0] = ? dp[1] = ?
5. **计算顺序**：从底向上，确保依赖已计算

**经典模型：**
- 线性DP：最长递增子序列、打家劫舍
- 区间DP：矩阵链乘法、戳气球
- 背包DP：0-1背包、完全背包
- 树形DP：树的最大独立集
- 状态压缩DP：旅行商问题

### A345."0-1背包和完全背包的状态转移方程有什么区别？空间优化如何实现？"

**参考答案：**

**0-1背包**（每个物品最多选一次）：
```
dp[i][j] = max(dp[i-1][j], dp[i-1][j-w[i]] + v[i])
```
含义：第i个物品，容量j时的最大价值。不选 → dp[i-1][j]，选 → dp[i-1][j-w[i]]+v[i]

**完全背包**（每个物品可选无限次）：
```
dp[i][j] = max(dp[i-1][j], dp[i][j-w[i]] + v[i])
```
关键区别：选第i个物品时，状态是dp[**i**][j-w[i]]（不是i-1），因为选完后还可以继续选第i个

**空间优化为一维：**
- 0-1背包：**逆序遍历**j（从大到小）
  ```
  for j from W downto w[i]:
      dp[j] = max(dp[j], dp[j-w[i]] + v[i])
  ```
  逆序保证dp[j-w[i]]是上一轮（i-1）的值

- 完全背包：**正序遍历**j（从小到大）
  ```
  for j from w[i] to W:
      dp[j] = max(dp[j], dp[j-w[i]] + v[i])
  ```
  正序使得dp[j-w[i]]可能已包含当前物品（即允许重复选）

**常见变种：**
- 多重背包：二进制拆分优化
- 分组背包：每组只能选一个
- 混合背包：以上三种混合

### A346."贪心算法与动态规划的区别是什么？贪心正确性如何证明？举例说明贪心不适用的场景。"

**参考答案：**

**贪心 vs DP：**

| 维度 | 贪心 | 动态规划 |
|------|------|---------|
| 决策方式 | 每步选局部最优 | 枚举所有可能取全局最优 |
| 是否回溯 | 不回溯 | 考虑所有子问题 |
| 时间 | 通常O(nlogn) | 通常O(n²)或更高 |
| 正确性 | 需要证明 | 天然正确（枚举了所有） |
| 适用范围 | 窄 | 宽 |

**贪心正确性证明方法：**
1. **交换论证法**：假设最优解不包含贪心选择，将贪心选择替换进去，结果不更差
2. **贪心选择性质**：局部最优选择可以导致全局最优
3. **拟阵理论**：满足拟阵性质的问题贪心一定正确

**贪心适用的经典问题：**
- 活动选择（区间调度）
- Huffman编码
- 最小生成树（Prim/Kruskal）
- Dijkstra最短路径

**贪心不适用的场景：**
- 0-1背包：贪心按性价比排序可能错过全局最优（如：容量10，物品A(6,6) B(5,5) C(5,5)，贪心选A但BC更优）
- 找零钱问题：面值[1,3,4]，金额6，贪心4+1+1=3枚，最优3+3=2枚

### A347."最长递增子序列（LIS）的O(nlogn)解法原理是什么？与O(n²)解法的区别在哪里？"

**参考答案：**

**O(n²)解法：**
```
dp[i] = max(dp[j] + 1)  for all j < i and nums[j] < nums[i]
```
每个位置枚举前面所有更小元素，O(n²)

**O(nlogn)解法（耐心排序）：**
- 维护数组tails，tails[k]表示长度为k+1的递增子序列的**最小末尾元素**
- 对每个nums[i]：
  - 若nums[i] > tails最后一个元素：追加到tails末尾（LIS长度+1）
  - 否则：在tails中二分查找第一个≥nums[i]的位置，替换之

**为什么正确？**
- tails始终递增（可二分查找）
- 替换操作不影响LIS长度，但为后续元素创造更多机会（末尾更小更容易接上）
- 最终tails长度即为LIS长度

**注意：** tails数组本身不是LIS，只是长度正确。要还原LIS需额外记录前驱。

**Java实现关键：**
```java
int[] tails = new int[n];
int len = 0;
for (int num : nums) {
    int pos = Arrays.binarySearch(tails, 0, len, num);
    if (pos < 0) pos = -(pos + 1); // 找到插入位置
    tails[pos] = num;
    if (pos == len) len++;
}
```

### A348."编辑距离问题的状态转移方程是什么？如何优化空间复杂度？"

**参考答案：**

**问题定义：** 将字符串A转换成字符串B所需的最少操作次数（插入、删除、替换）

**状态定义：** dp[i][j] = A的前i个字符转换成B的前j个字符的最少操作数

**状态转移方程：**
```
if A[i-1] == B[j-1]:
    dp[i][j] = dp[i-1][j-1]  // 字符相同，不需要操作
else:
    dp[i][j] = 1 + min(
        dp[i-1][j],     // 删除A[i-1]
        dp[i][j-1],     // 在A中插入B[j-1]
        dp[i-1][j-1]    // 替换A[i-1]为B[j-1]
    )
```

**初始条件：**
- dp[0][j] = j（A为空，插入j个字符）
- dp[i][0] = i（B为空，删除i个字符）

**空间优化：**
- 二维→两行滚动：O(min(m,n))
```
只保留上一行dp[i-1][*]和当前行dp[i][*]
```
- 一维：需要一个临时变量保存dp[i-1][j-1]

**变种问题：**
- 只允许插入/删除：替换代价设为无穷大
- 不同操作有不同代价：加权编辑距离
- 回文最少插入次数：s与s.reverse()的编辑距离/2

**参考答案：**

**常见贪心应用：**
1. 活动选择/区间调度：选最多不重叠区间

### A350."滑动窗口算法的适用条件是什么？如何确定窗口的扩张和收缩条件？"

**参考答案：**

**适用条件：**
1. 问题可转化为"满足某条件的连续子数组/子串"
2. 窗口扩张和收缩具有单调性（不会来回摇摆）

- 最短满足条件的子数组 → 窗口收缩为主，满足时尝试收缩

**窗口操作框架：**
```java
int left = 0;
for (int right = 0; right < n; right++) {
    // 1. 将right元素加入窗口
    window.add(nums[right]);
    
    // 2. 判断是否需要收缩
    while (window不满足条件) {
        window.remove(nums[left]);
        left++;
    }
    
    // 3. 更新答案
    ans = 更新(ans, window);
}
```

**关键设计点：**
- 窗口状态用什么数据结构维护？HashMap（字符频率）、计数器、单调队列
- 收缩条件如何判断？取决于题目约束
- 答案在何时更新？最长→每次循环更新，最短→收缩后更新

| 技巧 | 表达式 | 含义 |
|------|--------|------|
| 判断第k位 | n & (1 << k) | 第k位是否为1 |

```java
int add(int a, int b) {

}
```

```java
boolean isPowerOfTwo(int n) {
```

```java
int countOnes(int n) {

    return count;
}
```

### A352."单调栈和单调队列分别解决什么问题？核心实现思路是什么？"

**参考答案：**

**单调栈：**
- 解决：寻找元素左右第一个比它大/小的元素
- 类型：递增栈（找更小）、递减栈（找更大）
- 典型问题：
  - 每日温度（找右边第一个更大）
  - 柱状图最大矩形（找左右更小边界）
  - 接雨水（按行/列计算）

**实现框架：**
```java
Deque<Integer> stack = new ArrayDeque<>();
int[] result = new int[n];
for (int i = 0; i < n; i++) {
    while (!stack.isEmpty() && nums[stack.peek()] < nums[i]) {
        int idx = stack.pop();
        result[idx] = i - idx; // idx右边第一个更大是i
    }
    stack.push(i);
}
```

**单调队列：**
- 解决：滑动窗口内的最值问题
- 类型：递减队列（窗口最大值）、递增队列（窗口最小值）
- 典型问题：滑动窗口最大值

**实现框架：**
```java
Deque<Integer> deque = new ArrayDeque<>(); // 存下标
for (int i = 0; i < n; i++) {
    // 移除窗口外的元素
    while (!deque.isEmpty() && deque.peekFirst() < i - k + 1) 
        deque.pollFirst();
    // 维护单调性
    while (!deque.isEmpty() && nums[deque.peekLast()] < nums[i]) 
        deque.pollLast();
    deque.offerLast(i);
    if (i >= k - 1) result[i - k + 1] = nums[deque.peekFirst()];
}
```

**核心区别：** 单调栈看"历史"（左右边界），单调队列看"当前窗口"（滑动范围）

### A353."字符串匹配的KMP算法原理是什么？next数组如何计算？与BF匹配的区别？"

**参考答案：**

**BF暴力匹配：** 每次失配，模式串回到开头，主串回退。最坏O(m×n)

**KMP核心思想：** 失配时，利用已匹配的前缀信息，模式串尽可能少回退，主串不回退

**next数组（部分匹配表）：**
- next[i]：模式串前i个字符中，最长相等前后缀的长度
- 例如："ABABC"的next = [-1, 0, 0, 1, 2]

**next计算：**
```java
void computeNext(String pattern, int[] next) {
    next[0] = -1;
    int i = 0, j = -1;
    while (i < pattern.length() - 1) {
        if (j == -1 || pattern.charAt(i) == pattern.charAt(j)) {
            next[++i] = ++j;
        } else {
            j = next[j]; // 回退到更短的前缀
        }
    }
}
```

**匹配过程：**
```java
int i = 0, j = 0; // i主串，j模式串
while (i < main.length() && j < pattern.length()) {
    if (j == -1 || main.charAt(i) == pattern.charAt(j)) {
        i++; j++;
    } else {
        j = next[j]; // 模式串滑动，主串不动
    }
}
```

**时间复杂度：** O(m+n)，m为主串长度，n为模式串长度

**KMP的变种应用：** AC自动机（多模式匹配）= Trie + KMP失败指针

### A354."分治法的核心思想是什么？归并排序和快速排序如何体现分治思想？与动态规划的区别？"

**参考答案：**

**分治核心：**
1. **分解**：将原问题分解为若干子问题
2. **解决**：递归求解子问题（足够小则直接求解）
3. **合并**：将子问题的解合并为原问题的解

**归并排序体现分治：**
- 分解：将数组从中间一分为二
- 解决：递归排序左半和右半
- 合并：O(n)合并两个有序数组

**快速排序体现分治：**
- 分解：选pivot，将数组分为< pivot和> pivot两部分（partition）
- 解决：递归排序左半和右半
- 合并：无需额外合并（in-place）

**分治 vs 动态规划：**

| 维度 | 分治 | 动态规划 |
|------|------|---------|
| 子问题 | 相互独立 | 相互重叠 |
| 重复计算 | 无 | 有（需记忆化） |
| 合并方式 | 需要合并 | 状态转移 |
| 典型代表 | 归并排序、快排 | 最长公共子序列、背包 |

**分治适用场景：**
- 最近点对问题
- 大整数乘法（Karatsuba）
- 矩阵乘法（Strassen）
- 求逆序对（归并排序变体）

### A355."回溯法的框架是什么？如何进行剪枝优化？N皇后问题如何用回溯求解？"

**参考答案：**

**回溯框架：**
```java
void backtrack(路径, 选择列表) {
    if (满足结束条件) {
        收集结果;
        return;
    }
    for (选择 : 选择列表) {
        做选择;
        backtrack(路径, 选择列表);
        撤销选择; // 回溯
    }
}
```

**核心要素：**
- 路径：已做出的选择
- 选择列表：当前可做的选择
- 结束条件：到达决策树底层

**剪枝优化策略：**
1. **约束剪枝**：提前判断不满足约束的条件
2. **排序剪枝**：先排序，遇到不可能的分支提前终止
3. **去重剪枝**：同一层相同选择只做一次（i > 0 && nums[i]==nums[i-1] && !used[i-1]）
4. **可行性剪枝**：当前路径不可能达到目标（如剩余和不够/超出）
5. **最优性剪枝**：当前路径不可能比已知最优解更好

**N皇后问题：**
```java
void solve(int row, char[][] board) {
    if (row == n) { 收集结果; return; }
    for (int col = 0; col < n; col++) {
        if (!isValid(board, row, col)) continue; // 剪枝
        board[row][col] = 'Q';
        solve(row + 1, board);
        board[row][col] = '.'; // 回溯
    }
}
```

**N皇后剪枝：** 用Set记录已占用的列、正对角线、反对角线，O(1)判断冲突

**工程场景：** 排列组合问题、子集问题、数独求解、括号生成、正则匹配
